# 自我内部循环实现方案 20260512

对应规范：

```text
规范/自我内部循环实现规范20260512.md
规范/自我内部循环最小生命闭环验收规范20260510.md

规范/自我线程与任务管理线程权责边界规范20260512.md
```

本方案是落地方案，不替代规范；实施时以规范为准。

## 1. 目标

把内部循环落实成可运行链路：

```text
自我状态
-> 根需求 / 需求树
-> 当前主需求
-> 任务发起
-> 任务筹办
-> 方法查找 / 学习缺口
-> 任务执行
-> 状态回写
-> 需求结算
-> 下一轮主需求重判
```

当前不能把“安全任务已经建立”误判为内部循环完成。

完成标准至少是：

```text
任务执行结果能改变自我核心状态；
来源需求能结算；
下一轮主需求选择受到上轮结果影响。
```

## 2. 当前代码已经具备的骨架

### 2.1 自我线程主循环

已有入口：

```cpp
自我线程类::执行主循环一轮_(...)
```

现有阶段：

```text
步骤_冻结本轮消息批次_
步骤_规范化消息并落一次特征账_
步骤_重算受影响二次特征_
步骤_刷新线程上下文并生成治理帧_
步骤_整理需求并做根层重判_
步骤_生成并执行主派发决议_
执行休眠期自检报告修复门控_
```

评估：

```text
主循环骨架可保留。
需要继续确保每段只做自身职责，不把任务域写入混回自我线程。
```

### 2.2 需求树入树

已有入口：

```cpp
需求类::执行更新指令(...)
私有_收到派生需求消息时完善需求树(...)
私有_确保派生需求节点(...)
```

已具备：

```text
新增 / 更新 / 满足 / 删除需求。
同父直接子需求按目标重叠合并。
结构角色刷新。
活跃阻塞子需求权重平分。
派生需求缺关键字段时弹错并拒绝入树。
```

缺口：

```text
明确值 > 区间 > 方向 的细粒度优先级还需补强。
需求域动作动态方法化尚未完全闭合。
```

### 2.3 自我线程任务发起

已有入口：

```cpp
私有_提交任务发起请求到任务管理界面线程_线程侧(...)
```

现状：

```text
自我线程在无当前主任务时，会向任务管理界面线程提交发起任务请求。
任务管理界面线程支持请求类型 = 发起任务。
```

评估：

```text
方向符合新边界。
后续需要继续清理旧名和旧路径，避免直接创建任务头的旧逻辑回流。
```

### 2.4 任务管理界面线程承接

已有入口：

```cpp
任务管理界面线程::提交任务治理请求(...)
私有_按需求创建任务并初始化(...)
私有_确保任务虚拟存在(...)
私有_用需求初始化任务虚拟存在(...)
私有_应用工作结果到任务壳(...)
```

已具备：

```text
发起任务请求允许任务信息节点为空。
界面线程可以按来源需求创建任务信息节点。
界面线程负责绑定任务虚拟存在。
界面线程统一应用工作结果到任务壳。
任务状态变化处会生成任务状态动作动态。
```

缺口：

```text
任务状态提交动作动态仍需确保动作来源最终方法化。
任务状态推进请求里仍有直接写入痕迹，需要后续迁移到提交类本能方法。
```

### 2.5 任务筹办 / 任务执行本能方法

已有入口：

```cpp
自我动作实现模块::任务筹办(...)
自我动作实现模块::任务执行(...)
注册本能函数执行闭环(...)
```

已具备：

```text
任务筹办注册为本能方法。
任务执行注册为本能方法。
两者都先确保自身方法环境。
两者都通过运行存在返回任务层回执。
任务执行调用当前方法，并把世界事实交给输出结果场景。
```

缺口：

```text
任务执行存在“无显式需求满足证据时，以本能动作运行成功作为完成建议”的过渡保底。
最终必须改成显式需求满足证据。
```

## 3. 实现总链路

### 3.1 第一段：自我状态到当前主需求

目标：

```text
自我能读取安全值 / 服务值，并选择当前主需求。
```

使用入口：

```cpp
私有_从需求树构建需求列表(...)
私有_选择当前主需求候选_按根权重(...)
自我类::设置当前主需求(...)
```

实施要求：

```text
1. 安全 / 服务根需求必须存在。
2. 需求列表刷新不得创建任务。
3. 当前主需求必须属于当前自我需求树。
4. 当前主需求无效或已截止时不得派发任务。
```

验证日志：

```text
服务需求数
安全需求数
主导需求候选
主导任务候选
```

完成条件：

```text
改变安全值或服务值后，当前主需求跟随压力变化。
```

### 3.2 第二段：当前主需求到任务发起

目标：

```text
当前主需求有效时，自我线程只提交任务发起意图。
```

使用入口：

```cpp
私有_提交任务发起请求到任务管理界面线程_线程侧(...)
任务管理界面线程::提交任务治理请求(...)
```

实施要求：

```text
1. 请求类型 = 发起任务。
2. 来源需求 = 当前主需求。
3. 允许创建任务信息节点 = true。
4. 允许创建任务虚拟存在 = true。
5. 自我线程不得调用任务类创建任务头。
```

完成条件：

```text
任务管理界面线程返回任务信息节点；
需求通过 需求类::绑定对应任务 关联到任务信息节点；
任务虚拟存在生成并初始化。
```

### 3.3 第三段：任务创建到任务筹办

目标：

```text
任务管理线程把未启动 / 待重筹办任务推进到筹办。
```

使用入口：

```cpp
私有_构造任务工作包(...)
任务管理工作线程::提交任务工作包(...)
自我动作实现模块::任务筹办(...)
任务模块::筹办::筹办任务(...)
```

实施要求：

```text
1. 工作包推进模式根据任务状态确定。
2. 未启动 / 待重筹办 -> 只筹办。
3. 任务筹办只返回运行回执。
4. 任务筹办不得直接写任务虚拟存在。
5. 任务管理界面线程读取回执后同步当前方法、候选方法、可执行输入参数场景、任务状态。
```

关键规则：

```text
筹办只根据需求目标查找方法。
筹办不自带方向。
找到候选方法后先绑定到任务虚拟存在的候选集。
候选集中只有一个方法时可直接赋给当前方法；多个方法按方法状态、条件匹配、结果能力和稳定性规则选择。
```

完成条件：

```text
任务虚拟存在能看到候选方法集和当前方法；
若当前方法可用且参数可绑定，任务状态建议为就绪；
若不可用，产生明确缺口回执。
```

### 3.4 第四段：筹办缺口到需求树生长

目标：

```text
筹办发现缺口时，不把它当失败，而是转成更细需求。
```

使用入口：

```cpp
私有_构造执行缺失原子需求消息(...)
私有_落任务管理派生需求到自我内部世界(...)
私有_收到派生需求消息时完善需求树(...)
需求类::执行更新指令(...)
```

实施要求：

```text
1. 缺口必须带父需求。
2. 缺口必须带当前状态和目标状态。
3. 缺口必须带目标特征类型。
4. 方法不可用时，目标应是 方法状态 = 可用。
5. 缺动作时才生成动作骨架相关需求。
6. 不允许用宽泛“学习需求”掩盖具体目标。
```

完成条件：

```text
需求树能从根需求长出方法状态、条件结果对、参数补齐等子需求；
同目标需求不会重复风暴；
子需求会触发自己的任务发起。
```

### 3.5 第五段：学习链补齐方法

目标：

```text
方法从只有头节点，逐步达到可用。
```

职责划分：

```text
定向学习：
    缺动作时发生。

尝试学习：
    有动作后，建立条件结果对。

练习：
    在可用基础上提升稳定性和参数覆盖度。
```

实施要求：

```text
1. 方法状态 = 可用 的最低条件是至少一组有效条件结果对。
2. 尝试学习应产出真实条件结果对。
3. 练习不负责让方法从无到可用；它负责让方法更好用。
4. 学习方法自身必须先确保自身方法环境和规格。
```

完成条件：

```text
根任务筹办发现候选方法不可用；
派生 方法状态 = 可用 子需求；
子需求任务推进学习链；
目标方法具备有效条件结果对；
父任务重筹办后可进入就绪。
```

### 3.6 第六段：任务执行到世界事实

目标：

```text
就绪任务调用当前方法，产生世界事实输出。
```

使用入口：

```cpp
自我动作实现模块::任务执行(...)
调用任务当前方法(...)
改变特征值(...)
```

实施要求：

```text
1. 任务状态必须允许执行。
2. 当前方法必须存在。
3. 可执行输入参数场景必须存在。
4. 当前方法必须有本能函数或动作骨架入口。
5. 当前方法负责写输出结果场景。
6. 任务执行只写运行回执。
```

完成条件：

```text
输出结果场景包含真实特征变化或结果状态；
运行存在包含当前方法运行存在、目标是否达成、建议任务状态。
```

### 3.7 第七段：任务结果到需求结算

目标：

```text
任务结果改变来源需求，并影响下一轮判断。
```

使用入口：

```cpp
私有_落任务管理结果到自我内部世界(...)
私有_按成功结果满足来源需求并唤醒父任务_任务管理新链(...)
私有_处理任务满足反馈_线程侧(...)
```

实施要求：

```text
1. 任务完成不等于需求满足，需求必须复核。
2. 需求满足由自我线程处理。
3. 自我线程不得写任务折算值和任务状态。
4. 需求结算应更新累计安全 / 服务结算。
5. 自我安全 / 服务核心值变化后，下一轮主需求选择必须受影响。
```

完成条件：

```text
服务值低 -> 服务需求 -> 服务任务 -> 提升服务值 -> 服务需求结算 -> 下一轮服务压力下降。
```

## 4. 当前主要差距

### 4.1 需求层真实生长仍需持续验证

现状：

```text
派生需求入树入口已经存在。
```

差距：

```text
需要用日志确认根任务筹办缺口能稳定生成第一层子需求；
子需求任务化后能继续筹办；
同目标需求不会重复生成。
```

### 4.2 方法学习链尚未证明闭合

现状：

```text
尝试学习、定向学习、练习已作为本能方法存在。
```

差距：

```text
还需要验证三者是否能按方法状态链，为目标方法产出真实条件结果对。
```

### 4.3 任务执行完成判断仍有过渡保底

现状：

```text
判断任务来源需求是否被输出满足(...) 会在无显式证据时使用本能动作运行成功。
```

风险：

```text
方法执行成功但需求未满足时，可能误建议任务完成。
```

修订方向：

```text
必须让输出结果场景或当前方法运行存在提供明确需求满足证据。
```

### 4.4 动作动态方法化未完全收口

现状：

```text
任务状态动作动态已经在界面线程提交点生成。
```

差距：

```text
任务域、需求域、方法域提交动作最终都应由对应本能方法作为动作来源。
```

### 4.5 休眠期修复只记录缺失

现状：

```text
自检报告可以记录缺失。
```

当前边界：

```text
自检线程不能生成修复需求。
某些缺失只能等自我休眠时修复。
```

后续：

```text
休眠期修复策略另行细化。
```

## 5. 推荐实施顺序

### P0：日志闭环确认

目标：

```text
先用日志确认当前链路卡在哪一段。
```

必须覆盖：

```text
当前安全值 / 服务值
当前主需求
任务发起请求
任务虚拟存在
任务筹办候选方法集
当前方法
筹办缺口类型
派生需求消息
新需求节点
任务执行回执
需求结算结果
```

### P1：需求层闭合

目标：

```text
安全 / 服务压力能稳定产生当前主需求。
```

完成条件：

```text
同一状态多轮不重复生成同目标需求；
压力变化能改变当前主需求。
```

### P2：任务发起闭合

目标：

```text
当前主需求能通过任务管理界面线程生成任务信息节点和任务虚拟存在。
```

完成条件：

```text
自我线程不直接创建任务头；
任务虚拟存在具备任务状态、任务目标、候选方法集、当前方法槽位。
```

### P3：任务筹办闭合

目标：

```text
任务筹办能根据需求目标查找候选方法，并写回候选集 / 当前方法。
```

完成条件：

```text
安全需求能找到改变安全值的方法候选；
服务需求能找到改变服务值的方法候选；
候选不可用时派生 方法状态 = 可用 需求。
```

### P4：方法可用学习链闭合

目标：

```text
方法状态 = 可用 的子需求能通过学习链满足。
```

完成条件：

```text
目标方法从只有头节点推进到至少一组条件结果对；
父任务重筹办时该方法被视为可用。
```

### P5：任务执行闭合

目标：

```text
就绪任务能调用当前方法并输出世界事实。
```

完成条件：

```text
改变特征值方法能根据输入参数场景修改安全值或服务值；
输出结果场景包含需求满足证据。
```

### P6：需求结算闭合

目标：

```text
任务完成后来源需求被结算，并影响下一轮主需求选择。
```

完成条件：

```text
服务值 20 -> 40 后，服务压力下降；
服务根需求累计服务结算增加；
下一轮主需求不再重复创建同一任务。
```

### P7：连续多轮稳定

目标：

```text
内部循环连续多轮无任务风暴、无派生需求风暴、无空转。
```

完成条件：

```text
至少 10 轮稳定；
状态有合理变化或进入待机；
无空指针主链；
无无来源派生需求。
```

## 6. 最小验收剧本

初始条件：

```text
安全值 = 100
服务值 = 20
目标服务值 = 100
待机状态 = false
服务根需求有效
改变特征值本能方法已注册
```

第 1 轮期望：

```text
当前主需求 = 服务根需求
提交任务发起请求
任务管理线程创建服务任务
任务虚拟存在创建
```

第 2 轮期望：

```text
任务筹办查找改变服务值相关方法
候选方法进入任务方法候选集
若方法不可用，派生 方法状态 = 可用 子需求
```

第 N 轮期望：

```text
学习链让候选方法可用
父任务重筹办进入就绪
任务执行调用改变特征值
服务值上升
服务需求结算
下一轮服务压力下降
```

验收通过：

```text
状态影响需求；
需求影响任务；
任务影响方法；
方法影响状态；
状态再次影响需求。
```

## 7. 暂不纳入本轮闭合的内容

```text
1. 外界探索需求。
2. 复杂因果模板沉淀。
3. 长期记忆压缩。
4. 休眠期修复策略。
5. 复杂风险评估。
```

这些属于自我苏醒后或初步成熟阶段，不作为当前内部循环第一步完成条件。

## 8. 最终建议

优先顺序必须是：

```text
需求层闭合
-> 任务发起闭合
-> 任务筹办闭合
-> 方法可用学习链闭合
-> 任务执行闭合
-> 需求结算闭合
-> 多轮稳定
```

不要先追外界探索，也不要先追复杂学习。

第一阻塞点仍然是：

```text
需求树真实生长。
```

只有根任务筹办能稳定发现缺口、派生细需求、让细需求任务化并回头支持父任务，内部循环才开始真正转起来。
