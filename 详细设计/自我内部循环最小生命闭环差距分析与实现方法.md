# 自我内部循环最小生命闭环差距分析与实现方法

日期：2026-05-10

对应计划：

```text
计划/20260510_自我内部循环最小生命闭环阶段总计划_v0.1.md
```

对应规范：

```text
规范/自我内部循环最小生命闭环验收规范20260510.md
```

## 1. 分析结论

当前代码已经具备最小闭环的主体骨架：

```text
自我初始化
→ 安全 / 服务根需求
→ 当前主需求选择
→ 需求任务化
→ 任务管理界面线程承接
→ 任务虚拟存在绑定
→ 任务工作包推进
→ 任务结果回流
→ 自我线程价值结算
```

但还缺一个统一的“内部循环完成度诊断面”。也就是说，闭环的许多链路已经能跑，工程上还不能稳定回答：

```text
当前自我内部循环断在哪一级？
最近一次是否真的回写了状态？
最近一次是否真的结算了来源需求？
连续多轮是否存在重复任务风暴、无限派生需求或空转？
```

因此本阶段不应先大改闭环逻辑，而应先补：

```text
P0 诊断结构
P1-P6 代码链路验收点
P7 多轮稳定轨迹
P8 控制面板展示
```

### 2.2 当前最大缺口

```text
闭环执行能力已有雏形，
闭环验收和多轮稳定诊断不足。
```

具体表现：

```text
1. 没有统一的内部循环完成度枚举和诊断结构。
2. 主需求选择没有公开压力评估结果和选择原因。
3. 任务化没有专门的重复任务风暴诊断。
4. 自我线程仍保留部分“确保任务虚拟存在”路径，和界面线程唯一承接边界存在重叠。
5. 状态回写和需求结算已有代码，但缺少面向闭环等级的最近回写 / 最近结算标记。
6. 连续多轮稳定性尚无环形轨迹和判定规则。
7. 控制面板尚未显示内部循环等级和最近断点。
```

## 3. P0：验收仪表和诊断基线

### 现状

已有：

```text
结构_自我初始化结果
步骤记录
缺失项
冷启动完成 / 治理入口就绪 / 任务治理就绪 / 完全运行就绪
```

缺少：

```text
枚举_自我内部循环完成度
结构_自我内部循环诊断
统一诊断函数
每一级最早断点说明
```

### 实现方法

在 `自我类.ixx` 增加公共诊断结构：

```cpp
enum class 枚举_自我内部循环完成度 : std::uint8_t {
    未初始化 = 0,
    冷启动完成 = 1,
    需求树生成成立 = 2,
    任务化可用 = 3,
    本能动作可用 = 4,
    结果回写可用 = 5,
    需求结算可用 = 6,
    连续多轮稳定 = 7,
};

struct 结构_自我内部循环诊断 {
    枚举_自我内部循环完成度 完成度 =
        枚举_自我内部循环完成度::未初始化;

    bool 自我锚点有效 = false;
    bool 核心状态可读写 = false;
    bool 根需求有效 = false;
    bool 当前主需求有效 = false;
    bool 当前主任务有效 = false;
    bool 当前主任务虚拟存在有效 = false;
    bool 方法或本能动作可调用 = false;
    bool 最近结果已回写 = false;
    bool 最近需求已结算 = false;
    bool 最近多轮稳定 = false;

    I64 安全压力 = 0;
    I64 服务压力 = 0;
    std::string 当前主需求主键{};
    std::string 当前主任务主键{};
    std::string 当前任务虚拟存在主键{};
    std::string 最近断点{};
    std::vector<std::string> 缺失项{};
    std::string 摘要{};
};
```

建议提供：

```cpp
结构_自我内部循环诊断 诊断自我内部循环(
    const 自我类& 自我对象
) noexcept;
```

第一版只读取自我类、需求、任务、最近结算快照。多轮轨迹由 P7 再补。


## 4. P1：冷启动验收

### 现状

初始化模块已阶段化。`自我类.初始化.cpp` 中已经执行：

```text
数据层
语素层
概念层
自我锚点
核心特征
本能方法
根需求与结构根按需入口
主链镜像
运行态同步
验收
```

`结构_自我初始化结果` 已能表达冷启动和治理入口状态。

### 差距

```text
1. 初始化结果没有映射为内部循环完成度 1 级。
2. 冷启动验收还没有作为独立诊断函数复用。
3. 当前主任务 / 当前主方法允许延后，这一点需要在诊断里明确，避免误报冷启动失败。
```

### 实现方法

诊断函数里将以下条件判为 1 级：

```text
自我存在有效
自我内部世界有效
自我现实场景有效
安全值 / 服务值 / 风险安全值可读
安全根需求 / 服务根需求有效
默认本能方法或本能动作运行时已注册
```

当前主任务、当前主方法只进入 P3/P4，不纳入 P1。

补充口径：

```text
需求根 / 任务根 / 方法根 是存在信息结构字段，默认可以为空。
冷启动诊断不再把三类根字段非空作为硬条件。
当流程需要创建需求、任务或方法时，由对应创建链确保根字段；
请求创建失败才属于非正常逻辑错误。
```

### 计划标记

```text
P1 主要是诊断映射，不是重写初始化。
```

## 5. P2：需求树生成成立

### 阶段定位

需求树正常生成属于内部循环建立阶段的“需求层闭合”，不是初始化阶段，也不是自我苏醒阶段。

```text
初始化阶段：
    只要求需求系统入口、安全根需求、服务根需求存在。

需求树生成成立：
    自我能根据安全值、服务值、内部状态、任务缺口和方法缺口，
    自动生成 / 更新 / 组织需求树，并选出当前主需求。
```

当前已完成的压力评估和主需求选择，只是 P2 第一段，不能完整代表需求树正常生成。

### 现状

已有两处压力选择口径：

```text
自我类::选择当前主需求候选_()
自我线程模块.impl.cpp::私有_选择当前主需求候选_按根权重()
```

核心逻辑是：

```text
安全压力 = 安全差值 * 安全根权重
服务压力 = 服务差值 * 服务根权重
安全压力 >= 服务压力 选安全根需求
否则选服务根需求
```

### 差距

```text
1. 压力计算结果没有公共结构输出。
2. 当前主需求选择原因没有进入快照。
3. 自我类和自我线程各有一份选择逻辑，长期看容易出现口径漂移。
4. 需求树动态生成、子需求组织、去重合并、管理需求 / 执行需求角色刷新尚未纳入内部循环 2 级诊断。
5. 任务缺条件、方法不可用、结果不确定等内部缺口到需求树的入口，需要按父需求归属和原子目标去重重新验收。
```

### 实现方法

新增压力评估结构：

```cpp
struct 结构_自我根需求压力评估 {
    I64 安全差值 = 0;
    I64 服务差值 = 0;
    I64 安全压力 = 0;
    I64 服务压力 = 0;
    需求类::节点类* 选中需求 = nullptr;
    std::string 选择原因{};
};
```

建议放在 `自我类` 公共功能侧：

```cpp
结构_自我根需求压力评估 评估根需求压力() const noexcept;
```

自我线程后续改为调用这一个口径，或至少把两处实现都落到同一个内部 helper。

### 计划标记

```text
P2 第一段可和 P0/P1 同一分支完成。
完整 P2 由 `计划/20260510_自我内部循环P2b需求树生成成立计划_v0.1.md` 承接。
```

### P2b 实现方法

新增或补充诊断字段：

```cpp
bool 需求树生成成立 = false;
bool 动态需求生成入口可用 = false;
bool 需求去重合并可用 = false;
bool 需求结构角色已刷新 = false;
std::string 最近需求生成原因{};
std::string 最近需求去重合并摘要{};
std::string 需求树生成断点{};
```

验收口径：

```text
安全 / 服务压力能激活根需求或生成对应修复需求。
任务缺少条件能生成或复用补条件需求。
方法结构缺口能生成或复用补方法 / 学习需求。
同目标需求不会每轮重复生成。
有子需求的需求刷新为管理需求，叶子需求刷新为执行需求。
需求树生成后能选出当前主需求。
```

## 6. P3：主需求任务化

### 现状

自我线程已经在整理需求和主派发阶段做了：

```text
刷新需求列表
确保需求相关任务
选择当前主需求
选择当前主任务
必要时按需求创建任务头
写回当前主任务
```

需求侧已经有：

```text
对应任务
任务列表
绑定对应任务统一入口
任务初始化上下文
```

### 差距

```text
1. 任务化是否复用旧任务，缺少诊断输出。
2. 同一需求的活动任务数量缺少显式风暴检测。
3. 当前主任务与来源需求之间的关系没有进入内部循环诊断。
```

### 实现方法

在 P0 诊断中加入：

```text
当前主需求是否有对应任务
对应任务是否未终结
任务列表中未终结任务数量
当前主任务是否属于自我任务根
当前主任务是否绑定来源需求
```

判定规则：

```text
同一需求未终结任务数量 > 1
    -> 记录“重复任务风险”

当前主需求有效但无任务，且自我线程本轮未发起任务治理请求
    -> P3 不成立
```

### 计划标记

```text
P3 已有主链，先补诊断和重复任务检测。
```

## 7. P4：任务语义和方法入口

### 现状

任务语义链已经存在：

```text
任务主信息类::任务虚拟存在
任务管理界面线程::私有_确保任务虚拟存在
任务管理界面线程::私有_用需求初始化任务虚拟存在
任务管理工作线程::提交任务工作包
```

界面线程会写入：

```text
任务发起存在
任务作用对象
任务所属上下文
任务当前状态
任务目标状态
任务目标
任务完成条件目标状态
任务满足关系
任务安全权重
任务服务权重
任务综合优先级
任务语义状态
```

工作线程已经要求工作包带任务虚拟存在，缺失会返回错误：

```text
缺少任务虚拟存在，界面线程未完成绑定
```

### 差距

```text
1. 自我线程仍有若干“确保任务虚拟存在” helper，会在派发前和治理主体解析时补建任务虚拟存在。
2. 最新边界要求任务虚拟存在创建 / 初始化归任务管理界面线程承接，因此自我线程侧应收口为“解析已有”或“请求界面线程确保”。
3. 任务虚拟存在是否具有任务概念集和关键特征槽，缺少统一验收函数。
4. 方法 / 本能动作可调用性尚未进入内部循环诊断。
```

### 实现方法

分两步：

```text
第一步：诊断先识别任务虚拟存在是否有效、概念集是否包含任务、任务目标 / 完成条件 / 任务状态特征是否存在。

第二步：边界清理，把自我线程中的任务虚拟存在 helper 拆成：
    解析任务虚拟存在_只读
    确保任务虚拟存在_界面线程专用
```

自我线程中用于快照和治理落账的路径，应改为：

```text
任务主键 -> 任务信息节点 -> 已绑定任务虚拟存在
```

如果不存在：

```text
不在自我线程补建；
记录“任务语义绑定缺失”；
交由主派发或界面线程承接请求处理。
```

方法 / 本能动作可调用性诊断第一版可用保守标准：

```text
最近任务推进回执为 方法可执行 / 目标方法头待重入 / 已执行方法
或 最近工作结果.已执行方法 = true
或 本能动作运行时已注册且当前任务已有候选方法 / 绑定方法
```

### 需要确认

```text
是否立即删除自我线程侧“确保任务虚拟存在”的创建能力？
```

建议：

```text
按最新边界应删除创建能力，但保留只读解析能力。
治理落账如果解析不到任务虚拟存在，应退化为缺失诊断，不应在自我线程补建。
```

### 计划标记

```text
P4 既有能力基本完成，但边界有残留，需要列为专项清理。
```

## 8. P5：动作执行与状态回写

### 现状

本能动作和价值回写已有可用路径：

```text
自我类::应用自我需求安全值增加
自我类::应用自我需求服务值增加
自我类::应用安全服务闭环安全推进
自我类::应用安全服务闭环服务推进
自我类::应用服务归零待机保护
自我类::应用定时衰减服务值
自我类::应用按双值重算服务衰减步长
自我类::应用安全低位回升
自我类::应用安全高位回落
自我类::应用风险安全回归
```

自我线程价值结算会调用：

```text
私有_应用自我价值增量_线程侧
```

该函数会读取旧安全 / 服务值，调用自我类先天函数，再计算实际增量。

任务工作线程会输出：

```text
已执行方法
存在方法执行数据
方法运行结果
上行消息集
推进事件列表
```

### 差距

```text
1. 代码具备回写动作，但没有“最近结果已回写”的统一闭环标志。
2. 诊断无法直接判断最近一轮是否导致安全 / 服务 / 待机 / 时序步长变化。
3. 工作结果到自我价值反馈的触发路径需要在 P5/P6 验收中显式追踪。
```

### 实现方法

新增最近核心状态回写快照：

```cpp
struct 结构_自我核心状态回写快照 {
    bool 已生成 = false;
    时间戳 时间 = 0;
    I64 旧安全值 = 0;
    I64 新安全值 = 0;
    I64 旧服务值 = 0;
    I64 新服务值 = 0;
    I64 安全净变化 = 0;
    I64 服务净变化 = 0;
    std::string 来源任务主键{};
    std::string 来源需求主键{};
    std::string 摘要{};
};
```

落点：

```text
自我线程处理任务满足反馈 / 价值反馈时刷新。
自我类直接先天函数可选刷新更底层快照，但第一阶段建议放自我线程，避免所有先天函数都改签名。
```

P5 诊断：

```text
最近核心状态回写快照.已生成
且 安全净变化或服务净变化非零
```

### 需要确认

```text
当前所有“任务执行成功”是否都会进入任务满足反馈处理？
```

从代码看结算函数已存在，但需要在 P5 分支里继续追踪：

```text
任务管理上行消息 / 结果状态消息
→ 自我线程消息批次
→ 处理任务满足反馈
```

如果发现有结果路径未触发结算，应在自我线程消息处理器补桥。

### 计划标记

```text
P5 第一轮已完成“闭环回写诊断”。
P5 第二轮继续处理“结果触发验收”和真实价值回写路径。
```

### 2026-05-10 实施记录

已落地：

```text
1. 自我类新增最近核心状态回写快照。
2. 自我线程任务价值反馈路径记录旧安全值 / 新安全值、旧服务值 / 新服务值和净变化。
3. 自我类内部循环诊断将最近核心状态回写快照纳入 P5 判断。
4. 控制面板摘要展示最近回写是否有效，以及安全 / 服务净变化。
```

动态验证结果：

```text
--summary 能显示最近回写字段。
当前运行仍显示最近回写 = 否。
```

判断：


## 11.2 标准原子状态词典补充

## 13. 需讨论或确认的问题

### 问题一：自我线程是否允许继续补建任务虚拟存在

当前代码里自我线程仍有任务虚拟存在确保路径。最新边界是：

```text
任务管理界面线程负责绑定任务虚拟存在。
自我线程只提交任务治理请求。
```

建议：

```text
删除自我线程侧创建能力；
保留解析已有任务虚拟存在的只读 helper；
解析不到时记录诊断缺失，交给界面线程承接。
```

### 问题二：所有工作结果是否都进入需求结算

结算函数存在且完整，但需要追踪：

```text
任务工作结果 / 上行结果状态消息
→ 自我线程消息处理
→ 处理任务满足反馈
```

建议在 P5-P6 分支中加日志和诊断钩子确认。如果存在成功结果未结算路径，再补消息处理桥。

### 问题三：连续多轮稳定的默认阈值

建议默认：

```text
N = 5
同一需求活动任务上限 = 1
连续空转阈值 = 3

```
