# 需求满足程度与权重分摊结算详细设计

## 1. 背景

需求来自方向。

一个需求表达为：

`被需求对象 当前状态 -> 目标状态`

满足需求依靠任务。

需求自身带有安全权重和服务权重，这两个权重表示：

`该需求被完全满足时，最多可以结算给自我安全值/服务值的总增量`

但任务常常不会一次完全满足需求，因此需要按照本轮实际推进程度分摊需求权重。

本文定义“满足程度 -> 安全/服务实际增量”的结算算法。

## 2. 总原则

1. 需求权重是完全满足时的总价值。
2. 任务结果只结算本轮实际推进的那一段。
3. 不使用 Q10000 或百分比特征。
4. 所有可比较、可累积值统一使用 `I64`。
5. 数值型和布尔型需求可以直接按差额结算。
6. 有序枚举需求可以映射为 I64 阶梯后结算。
7. 无序枚举需求只用于路由或命中判定，不参与满足程度分摊结算。
8. 必须记录已结算状态，防止重复结算。

## 3. 基础概念

### 3.1 需求状态

每个可结算需求至少需要：

| 字段 | 含义 |
| --- | --- |
| 初始状态值 | 需求建立或首次结算时的起点值 |
| 目标状态值 | 需求希望达到的目标值 |
| 当前实际状态值 | 当前读到的真实状态值 |
| 已结算状态值 | 上次已经结算到的状态值 |
| 安全权重 | 完全满足时最多结算的安全增量 |
| 服务权重 | 完全满足时最多结算的服务增量 |
| 方向 | 增加、减少、保持、枚举达成 |

### 3.2 任务结果状态

每个任务结果至少需要：

| 字段 | 含义 |
| --- | --- |
| 执行前状态值 | 本轮任务执行前读到的状态 |
| 执行后状态值 | 本轮任务执行后读到的状态 |
| 结果方向 | 增加、减少、保持、反向 |
| 命中需求 | 是否服务于该需求 |

任务结果不直接决定安全值或服务值。

它只能提供可结算状态变化证据。

## 4. 数值型需求结算

### 4.1 增加方向

需求：

`初始状态值 < 目标状态值`

公式：

```text
总目标差额 = 目标状态值 - 初始状态值
已结算终点 = min(已结算状态值, 目标状态值)
当前可结算终点 = min(当前实际状态值, 目标状态值)
有效满足贡献 = max(0, 当前可结算终点 - 已结算终点)
```

结算：

```text
安全增量 = 安全权重 * 有效满足贡献 / 总目标差额
服务增量 = 服务权重 * 有效满足贡献 / 总目标差额
```

完成后：

```text
已结算状态值 = 当前可结算终点
累计安全结算 += 安全增量
累计服务结算 += 服务增量
```

### 4.2 减少方向

需求：

`初始状态值 > 目标状态值`

公式：

```text
总目标差额 = 初始状态值 - 目标状态值
已结算终点 = max(已结算状态值, 目标状态值)
当前可结算终点 = max(当前实际状态值, 目标状态值)
有效满足贡献 = max(0, 已结算终点 - 当前可结算终点)
```

结算：

```text
安全增量 = 安全权重 * 有效满足贡献 / 总目标差额
服务增量 = 服务权重 * 有效满足贡献 / 总目标差额
```

完成后：

```text
已结算状态值 = 当前可结算终点
累计安全结算 += 安全增量
累计服务结算 += 服务增量
```

### 4.3 保持方向

保持方向不是“越变越满足”，而是“没有偏离即满足”。

第一版处理：

| 情况 | 结算 |
| --- | --- |
| 当前实际状态值 == 目标状态值 | 可一次性结算剩余权重 |
| 当前实际状态值 != 目标状态值 | 不结算 |

保持方向不做连续差额分摊。

## 5. 布尔型需求结算

布尔型需求视为数值型 `0/1`。

例如：

- 方法是否可用：`0 -> 1`
- 引用是否有效：`0 -> 1`
- 方法结构是否完整：`0 -> 1`

公式等同增加方向：

```text
总目标差额 = 1
有效满足贡献 = 1 或 0
```

结果：

| 当前状态 | 结算 |
| --- | --- |
| 仍为 `0` | 不结算 |
| 变为 `1` | 一次性结算剩余权重 |

## 6. 枚举型需求结算

### 6.1 有序枚举

有序枚举必须先映射为 I64 阶梯。

例如：

```text
可调度 = 0
执行中 = 1
已回流 = 2
```

然后按数值型需求结算。

要求：

1. 映射表必须稳定。
2. 阶梯方向必须明确。
3. 不能把无序分类强行映射为有序阶梯。

### 6.2 无序枚举

无序枚举只用于路由，不参与满足程度分摊。

例如：

```text
任务停滞原因 = 候选方法待补齐 / 等待学习 / 等待结果 / 连续失败 / 未知
```

这类状态只能决定后续任务如何处理，不能直接计算安全/服务增量。

如确实需要结算，必须另设可结算的布尔或数值需求，例如：

```text
任务推进状态：停滞 -> 可继续推进
```

## 7. 防重复结算

必须引入结算锚。

### 7.1 最小结算锚字段

| 字段 | 含义 |
| --- | --- |
| 已结算状态值 | 已经结算到的状态值 |
| 累计安全结算 | 该需求累计结算出的安全增量 |
| 累计服务结算 | 该需求累计结算出的服务增量 |
| 最近结算任务 | 最近一次贡献结算的任务 |
| 最近结算时间 | 最近一次结算时间 |

### 7.2 重复结算禁止

不得用“当前状态已经满足”反复结算。

每次结算必须从 `已结算状态值` 开始计算新增贡献。

若：

```text
有效满足贡献 <= 0
```

则本轮安全增量和服务增量均为 `0`。

## 8. 超额完成处理

超额完成不得超额结算。

增加方向：

```text
当前可结算终点 = min(当前实际状态值, 目标状态值)
```

减少方向：

```text
当前可结算终点 = max(当前实际状态值, 目标状态值)
```

这样最终累计结算不会超过需求安全权重和服务权重。

## 9. 反向结果处理

如果任务结果方向与需求方向相反：

1. 本轮不产生正向安全/服务结算。
2. 是否产生负面后果，按现有零增量与负面后果边界规范处理。
3. 反向结果可以成为结果不符合次数增加、任务重筹办或学习需求的证据。

## 10. 整数除法余数处理

使用 I64 整数分摊时会有余数。

第一版规则：

1. 中间轮次使用整数除法向零截断。
2. 当需求最终完全满足时，将剩余未结算权重一次性补齐。

例如：

```text
安全权重 = 100
总目标差额 = 3
每次推进 1
```

前两轮：

```text
100 * 1 / 3 = 33
```

第三轮完全满足时：

```text
补齐剩余 = 100 - 已累计安全结算
```

服务权重同理。

## 11. 示例

### 11.1 数值型增加需求

需求：

```text
自我安全值 0 -> 10
安全权重 = 100
服务权重 = 50
```

第一轮：

```text
当前实际状态 = 3
有效满足贡献 = 3
安全增量 = 100 * 3 / 10 = 30
服务增量 = 50 * 3 / 10 = 15
已结算状态值 = 3
```

第二轮：

```text
当前实际状态 = 8
有效满足贡献 = 8 - 3 = 5
安全增量 = 100 * 5 / 10 = 50
服务增量 = 50 * 5 / 10 = 25
已结算状态值 = 8
```

第三轮：

```text
当前实际状态 = 12
当前可结算终点 = min(12, 10) = 10
有效满足贡献 = 10 - 8 = 2
安全增量 = 剩余 20
服务增量 = 剩余 10
已结算状态值 = 10
```

总计：

```text
安全结算 = 100
服务结算 = 50
```

### 11.2 布尔需求

需求：

```text
方法是否可用 0 -> 1
安全权重 = 20
服务权重 = 80
```

若方法仍不可用：

```text
安全增量 = 0
服务增量 = 0
```

若方法变为可用：

```text
安全增量 = 20
服务增量 = 80
```

## 12. 接入点

建议接入在自我线程最终结算链中。

输入来自：

- 需求节点。
- 任务结果。
- 状态变化事实。
- 任务执行记录。
- 方法执行结果。

输出写入：

- 自我安全值。
- 自我服务值。
- 需求结算锚。
- 双值结算账。
- 最近结算摘要。

任务管理工作线程不得直接结算安全值或服务值。

工作线程只能上报状态变化、结果事实和执行记录。

## 13. 待补字段

需求侧建议补：

| 字段 | 类型 |
| --- | --- |
| 初始状态值 | `I64` |
| 已结算状态值 | `I64` |
| 累计安全结算 | `I64` |
| 累计服务结算 | `I64` |
| 最近结算任务 | 任务引用 |
| 最近结算时间 | 时间戳 |

任务结果侧建议补：

| 字段 | 类型 |
| --- | --- |
| 执行前状态值 | `I64` |
| 执行后状态值 | `I64` |
| 命中需求 | 需求引用 |
| 结果方向 | 枚举或方向词 |

## 14. 完成判据

1. 数值型需求能按实际推进量分摊安全/服务权重。
2. 布尔型需求能按 `0 -> 1` 一次性结算。
3. 有序枚举能通过稳定阶梯映射后结算。
4. 无序枚举不参与分摊结算。
5. 已结算状态值能防止重复结算。
6. 超额完成不会超额结算。
7. 最终完全满足时能补齐整数除法余数。
8. 任务管理工作线程不直接写安全值或服务值。
