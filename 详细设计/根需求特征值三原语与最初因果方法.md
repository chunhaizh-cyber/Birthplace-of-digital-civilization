# 根需求特征值三原语与最初因果方法

本文用于固定一条更底层的自我成长起点：

`安全值 / 服务值` 的根需求，最终必须落到 `特征值如何变化` 这件事上。

## 1. 根需求目标分层

自我的一级根需求仍固定为：

- `安全根需求`
- `服务根需求`

这两个根需求都可以拆成两层目标：

- `终极目标`：安全值和服务值分别达到各自最大目标值。
- `过程目标`：安全值和服务值朝最大目标值持续增长。

因此，对根需求来说：

- `最大值` 是终点。
- `增长` 是逼近终点的过程。
- `减少` 是与过程目标相反的方向，应进入拒止、规避或负面因果链。

这不会新增第三根需求树；它只是把安全、服务两个根需求的数值目标结构说清楚。

## 2. 三个最小执行原语

围绕任意可数值化特征，最初必须具备三个执行原语：

| 原语 | 语义 | 对根需求的意义 |
| --- | --- | --- |
| `增加特征值` | 让目标特征当前值按正向增量上升 | 默认是安全/服务根需求的正向过程动作 |
| `减少特征值` | 让目标特征当前值按负向增量下降 | 默认是安全/服务根需求的反向过程动作，需进入拒止或风险链 |
| `改变特征值` | 让目标特征当前值从旧值变为新值，并显式记录方向 | 是更泛化的数值变化动作，可归约为增加、减少或零变化 |

必须固定：

- `增加` 和 `减少` 是方向明确的最小动作。
- `改变` 是通用动作壳，执行后必须通过 `新值 - 旧值` 判定实际方向。
- 零变化不是增长，也不是减少，应作为 `无直接增量` 或 `待观察` 处理。

## 3. 安全值与服务值的首批实例

这三个原语的第一批正式实例应落在：

- `增加安全值`
- `减少安全值`
- `改变安全值`
- `增加服务值`
- `减少服务值`
- `改变服务值`

其中：

- `增加安全值 / 增加服务值` 是正向根因果的最小动作样本。
- `减少安全值 / 减少服务值` 是拒止目标、风险记忆、禁止项或负向因果的最小动作样本。
- `改变安全值 / 改变服务值` 是动作动态与状态变化之间的通用桥，可在回读后归入增加、减少或零变化。

## 4. 最初因果生成

最初因果不应先从复杂任务或复杂方法开始，而应先从以下最小形态开始：

```text
条件：目标特征当前值低于目标值
动作：增加特征值
结果：目标特征值上升
方向：正向逼近根需求
```

对应负向样本为：

```text
条件：目标特征当前值需要增长
动作：减少特征值
结果：目标特征值下降
方向：背离根需求，形成拒止目标或负向因果
```

通用变化样本为：

```text
条件：目标特征存在旧值
动作：改变特征值
结果：目标特征出现新值
方向：由新旧差值判定为增加、减少或零变化
```

这三类因果是后续所有复杂方法因果的底座。

## 5. 最初方法生成

最初方法应从三类方法头开始：

- `增加单特征当前值`
- `减少单特征当前值`
- `改变单特征当前值`

它们的条件和结果最少应包含：

- `目标主体`
- `目标特征类型`
- `旧值`
- `候选新值` 或 `增量`
- `方向`
- `是否触及上界 / 下界`
- `动作前后回读结果`

执行后，必须生成动作边界：

```text
动作开始
-> 写入或尝试写入目标特征
-> 回读目标特征
-> 动作结束
-> 生成状态变化与实例因果
```

## 6. 与现有代码的关系

当前代码已经具备一部分基础：

- `自我类::安全值增加 / 安全值减少`
- `自我类::服务值增加 / 服务值减少`
- `自我_查找特征类型`
- `自我_配对参数`
- `自我_改变特征值`
- 动作开始 / 结束记录由执行桥和动态 / 因果事实写入承担

本轮补充的口径是：

- 任务筹办和任务执行是任务系统普通函数，不作为本能方法注册。
- 不再新增或调用 `管理_*` 伪本能方法。
- 特征值改变必须通过可执行方法产生事实状态，再进入条件结果对归并。

历史 `管理_读取/写入/记录/增减/改变单特征当前值` 口径已废弃，不能再作为默认方法头、执行桥白名单或任务筹办兜底路径。

## 7. 后续落地顺序

建议后续按以下顺序继续：

1. 先让三原语稳定注册为本能方法头。
2. 再给三原语补条件场景与结果场景的统一字段。
3. 然后让执行桥在动作后回读目标特征，归类为增加、减少或零变化。
4. 最后把这些动作动态转成最初实例因果，并允许方法学习在此基础上组合出更复杂方法。

验收口径：

`任一安全值/服务值变化，都能回溯到增加、减少或改变特征值中的一个原语动作，并能据此生成正向因果、负向因果或零变化记录。`
