# 2026-04-12 抽象动态生成详细设计

## 1. 目标

本文用于固定 `抽象动态` 的生成方式，并与 [20260412_抽象状态生成详细设计.md](D:\鱼巢\详细设计\待完成\20260412_抽象状态生成详细设计.md) 接平。

本文重点回答：

- 什么是抽象动态
- 抽象动态与实例动态如何区分
- 抽象动态为什么“和抽象状态类似，但比抽象状态多一层动态特征”
- 动态的初始状态、结果状态与时间因素如何进入抽象动态
- 抽象动态在世界树中应挂到哪里
- 抽象动态如何命中、升格、衰减和淘汰


## 2. 设计结论

抽象动态与抽象状态的关系，可以压成一句话：

`抽象动态 = 抽象状态链 + 动态差分链 + 时间二次特征链`

也就是说，抽象动态的生成方式和抽象状态类似，但比抽象状态多的部分主要是：

1. `初始状态` 的二次特征
2. `结果状态` 的二次特征
3. `初始状态 -> 结果状态` 的变化二次特征
4. `时间因素` 抽象后的二次特征

因此抽象动态不是只看：

- “谁做了什么动作”

而是至少要同时看：

- “从什么抽象状态出发”
- “变到了什么抽象状态”
- “变化方向和差分特征是什么”
- “这个变化在时间上属于什么模式”


## 3. 基本定义

### 3.1 实例动态

`实例动态` 指的是：

- 某个主体
- 在某个具体时间段
- 发生了一次具体动作或具体变化

它对应当前工程里的 `动态节点`。

### 3.2 抽象动态

`抽象动态` 指的是：

- 从多个实例动态中归并出来的可复用动态模式
- 它描述的是“某类变化是如何发生的”，而不是“某次变化什么时候发生”

必须固定：

- 抽象动态不是某次单独事件
- 抽象动态不是某个具体主体私有的动态
- 具体主体只能命中抽象动态，不能定义抽象动态


## 4. 与抽象状态的关系

抽象动态不应脱离抽象状态单独生成。

推荐固定依赖顺序：

```text
实例状态
-> 抽象状态
-> 实例动态
-> 抽象动态
```

原因是：

1. 动态本身天然包含 `初始状态` 与 `结果状态`
2. 若初始状态和结果状态还停在实例层，则抽象动态身份会过度依赖具体样本
3. 先有抽象状态，才能把动态稳定压缩到“状态模式之间的迁移”

因此：

- 抽象状态是抽象动态的输入底座
- 抽象动态是对“状态模式变化 + 时间模式”的进一步抽象


## 5. 抽象动态的额外输入面

相比抽象状态，抽象动态多出来的部分固定为两大组。

## 5.1 状态差分二次特征组

这是抽象动态最核心的第一组额外输入。

至少包括：

1. `初始状态二次特征`
- 该动态开始时，主体所处的抽象状态特征

2. `结果状态二次特征`
- 该动态结束时，主体所处的抽象状态特征

3. `状态迁移二次特征`
- 从初始状态到结果状态的方向、差分、迁移类型

例如：

- `低位 -> 中位`
- `高位 -> 稳定带`
- `空指针 -> 已绑定`
- `未启动 -> 运行中`

必须固定：

- 抽象动态不能只保留动作名
- 必须保留“从哪到哪”的抽象状态迁移面

## 5.2 时间因素二次特征组

这是抽象动态比抽象状态多出的第二组关键输入。

抽象动态在删除单次实例时间后，仍要保留“时间模式的抽象结果”。

至少建议包括：

1. `持续时间区间`
- 极短
- 短窗
- 中窗
- 长窗

2. `起止节奏`
- 立即完成
- 持续推进
- 长时拖延
- 中断失败

3. `发生频率`
- 高频
- 中频
- 低频
- 偶发

4. `重复模式`
- 单次型
- 周期型
- 连续型
- 间歇型

5. `相位模式`
- 只有开始边界
- 有开始有完成
- 有开始有失败
- 多阶段推进

必须固定：

- 抽象动态中保留的是“时间因素的二次特征”
- 不是实例动态的原始时间戳本身


## 6. 抽象动态的生成总链

推荐固定为：

```text
实例动态
-> 取初始状态、结果状态
-> 初始状态与结果状态分别命中抽象状态
-> 生成状态迁移二次特征
-> 生成时间因素二次特征
-> 删除单次实例时间身份
-> 形成抽象动态候选键
-> 更新或创建抽象动态候选
-> 达阈值后升格为正式抽象动态
```

这条主链里，和抽象状态相比新增的部分就是：

- `状态迁移二次特征`
- `时间因素二次特征`


## 7. 世界树挂载结构

建议抽象动态进入 `概念域::动态`，而不是继续挂在运行期动态节点链上。

推荐挂载结构：

```text
世界树根
-> 概念域根
  -> 动态域根
    -> 动作/变化类型根
      -> 抽象动态
        -> 更细抽象动态
    -> 复合抽象动态分支
```

### 7.1 动态域下的主干

主干优先表达：

- 同一类动作或同一类变化模式的逐层细分

例如：

- `变化`
  - `上升型变化`
  - `下降型变化`
  - `切换型变化`
- `管理动态`
  - `治理推进动态`
  - `边界开始动态`
  - `边界结束动态`

### 7.2 动态域下的聚合分支

聚合分支优先表达：

- 多个抽象动态共同构成的更高层动态模式

第一阶段可以先不做复杂聚合，只保留单条抽象动态。


## 8. 数据结构建议

建议新增：

- `抽象动态节点主信息类`

### 8.1 最小字段

至少应包含：

- `概念域`
  - 固定为 `动态`
- `抽象动态类型`
  - `单动态`
  - `复合动态`
- `动作核心类型`
  - 动词核或动作语义类别
- `来源业务语义类型`
  - 可复用当前 `动态业务语义类型`
- `父抽象动态主键`
- `初始抽象状态主键`
- `结果抽象状态主键`
- `状态迁移二次特征主键列表`
- `时间因素二次特征主键列表`
- `适用条件主键`
- `命中次数`
- `首次命中时间`
- `最近命中时间`
- `稳定度`
- `活跃度`
- `置信度`
- `当前状态`
  - `候选`
  - `稳定`
  - `冷却`
  - `待淘汰`

### 8.2 不得进入身份字段的内容

以下内容不得作为抽象动态身份字段：

- 单次实例动态的 `起始时间`
- 单次实例动态的 `结束时间`
- 单次实例动态的唯一主键
- 单次实例动态的来源错误码
- 单次实例主体的唯一主键

必须固定：

- 实例时间可以参与生成“时间二次特征”
- 但不得直接作为抽象动态身份


## 9. 动态二次特征的拆法

## 9.1 初始状态二次特征

建议至少回答：

- 起始状态属于哪个抽象状态
- 起始状态在当前维度上的方向落点
- 起始状态是否满足某条件

例如：

- `起始状态=安全低位`
- `起始状态=服务中位区`

## 9.2 结果状态二次特征

建议至少回答：

- 结果状态属于哪个抽象状态
- 结果状态在当前维度上的方向落点
- 结果状态是否满足某条件

例如：

- `结果状态=安全稳定带`
- `结果状态=已绑定`

## 9.3 状态迁移二次特征

建议至少回答：

- 方向
  - 上升
  - 下降
  - 回升
  - 回落
  - 切换
  - 迁移
- 差分强度
  - 小幅
  - 中幅
  - 大幅
- 迁移类别
  - 同层微调
  - 跨区间跃迁
  - 终态进入

### 9.3.1 迁移身份建议

建议用：

- `初始抽象状态`
- `结果抽象状态`
- `方向`
- `差分类别`

共同生成迁移签名。

## 9.4 时间因素二次特征

建议拆成 4 类：

1. `时长类`
- 极短 / 短 / 中 / 长

2. `频率类`
- 高频 / 中频 / 低频 / 偶发

3. `节奏类`
- 立即完成 / 持续推进 / 缓慢拖尾 / 中断失败

4. `相位类`
- 开始型 / 完成型 / 失败型 / 完整边界型

这些二次特征都来自实例动态的时间字段与相位字段，但一旦进入抽象动态，就只保留抽象后的类别结果。


## 10. 抽象动作的正确口径

和抽象状态一样，抽象动态也必须来自“抽象动作”，而不是自然出现。

建议新增统一入口：

- `尝试从实例动态生成抽象动态候选(...)`

### 10.1 内部步骤

至少执行以下步骤：

1. 校验实例动态合法
2. 读取 `初始状态 / 结果状态`
3. 让初始状态、结果状态分别命中对应抽象状态
4. 计算 `状态迁移二次特征`
5. 计算 `时间因素二次特征`
6. 丢弃单次实例时间作为身份字段
7. 生成候选键
8. 更新或创建抽象动态候选

### 10.2 删除时间的正确实现

这里的“删除时间”必须固定为：

- 不把实例动态的具体开始时间、结束时间写进抽象动态身份
- 只把它们转成：
  - 持续时间类别
  - 时间窗类别
  - 频率类别
  - 节奏类别

也就是说：

```text
实例时间 -> 时间二次特征
实例时间 -/-> 抽象动态身份原值
```


## 11. 候选键建议

抽象动态候选键建议至少由以下字段组合：

- `概念域`
- `动作核心类型`
- `来源业务语义类型`
- `初始抽象状态主键`
- `结果抽象状态主键`
- `状态迁移签名`
- `时间因素签名`
- `适用条件主键`

例如：

```text
动态|回升|被动记账动态|安全低位|安全稳定带|低位到稳定带_回升|短窗_立即完成_高频|普通运行
```

这才足以区分：

- 同样叫“回升”
- 但起点、终点、节奏完全不同

的两类抽象动态。


## 12. 命中关系设计

建议保留三层关系：

1. `实例动态 -> 抽象动态`
- 表示某次实例动态被归到哪个抽象动态

2. `抽象动态 -> 初始抽象状态 / 结果抽象状态`
- 表示该动态模式依赖哪些状态模式

3. `存在 -> 抽象动态`
- 表示某个存在命中过哪些动态模式

这样后续可以同时支持：

- 轨迹分析
- 因果提炼
- 学习样本聚类
- 控制面板解释


## 13. 升格规则

抽象动态候选的升格，建议至少满足：

1. `命中次数 >= 最小阈值`
2. `最近命中时间` 仍在活跃窗口内
3. `初始状态 -> 结果状态` 的迁移模式相对稳定
4. `时间因素模式` 没有频繁漂移
5. 相比父抽象动态具备独立区分价值

### 13.1 复合抽象动态

第一阶段不必优先做，但若后续需要，可建立在多个已稳定抽象动态的共同模式之上。


## 14. 低频衰减与淘汰

抽象动态和抽象状态一样，需要活跃度管理。

### 14.1 衰减输入

建议至少观察：

- 最近命中时间差
- 最近窗口内命中频率
- 时间模式是否漂移
- 状态迁移模式是否漂移

### 14.2 淘汰条件

满足以下任一条件时，可进入淘汰流程：

1. 长期未命中
2. 命中频率持续低于阈值
3. 状态迁移模式不再稳定
4. 时间因素模式持续漂移
5. 相比父抽象动态已失去独立区分意义

### 14.3 淘汰处理

建议分三档：

1. `稳定 -> 冷却`
2. `冷却 -> 候选`
3. `候选 -> 清理`


## 15. 与当前现有系统的对接点

当前仓内可直接复用的基础有：

1. `动态节点`
- 已有 `初始状态 / 结果状态 / 时间 / 相位 / 来源动作`

2. `动态业务语义类型`
- 已有 `动作事实动态 / 状态变化动态 / 被动记账动态 / 治理过程动态 / 边界结束动态`

3. `二次特征节点`
- 已有通用承载

4. `抽象状态设计`
- 已经明确了状态域下的抽象挂载方式

因此第一阶段最合适的实现顺序是：

1. 先让实例动态能够引用初始/结果抽象状态
2. 再补状态迁移二次特征
3. 再补时间因素二次特征
4. 最后生成抽象动态候选与稳定节点


## 16. 第一阶段最小落地范围

建议第一阶段只接入最稳定的 3 类动态：

1. `状态变化动态`
2. `被动记账动态`
3. `边界结束动态`

原因是：

- 这三类动态最容易拿到明确的初始状态和结果状态
- 时间因素也更容易抽成稳定类别

首批最适合接入的候选动态：

- `安全值回升`
- `安全值回落`
- `服务值衰减`
- `待机保护触发`
- `当前主任务切换`


## 17. 一句话收口

可收成一句话：

`抽象动态的生成方式和抽象状态类似，但它不能只看静态状态模式；它必须额外吸收“初始状态与结果状态的二次特征”和“时间因素的二次特征”，最终形成“某类状态模式在某类时间模式下如何迁移”的可复用动态概念。`
