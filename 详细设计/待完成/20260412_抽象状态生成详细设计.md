# 2026-04-12 抽象状态生成详细设计

## 1. 目标

本文用于把 [0605_抽象状态生成规范.md](D:\鱼巢\规范\0605_抽象状态生成规范.md) 落成可实现的详细设计，固定以下内容：

- 抽象状态在世界树中的挂载位置
- 抽象状态与实例状态的结构边界
- 单特征抽象状态与复合抽象状态的生成流程
- 命中、升格、衰减、淘汰的最小实现规则


## 2. 设计结论

抽象状态不作为普通 `状态节点` 直接长期复用。

原因固定为：

1. `状态节点` 当前天然表达的是状态事实
2. `状态节点主信息类` 当前字段天然偏向实例语义
3. 抽象状态虽然在抽象后不再保留单次实例时间，但仍需要独立的统计字段与概念身份

因此本设计固定：

- `实例状态` 继续使用当前 `状态节点`
- `抽象状态` 进入 `概念域::状态`
- 具体存在与抽象状态之间，通过“命中关系”连接


## 3. 世界树挂载结构

推荐挂载结构固定为：

```text
世界树根
-> 概念域根
  -> 状态域根
    -> 特征类型状态概念根
      -> 单特征抽象状态
        -> 更细单特征抽象状态
    -> 复合抽象状态分支
      -> 复合抽象状态
```

具体要求：

1. `特征类型状态概念根`
- 每个可抽象的特征类型一个根节点

2. `单特征抽象状态`
- 挂在对应特征类型根节点下
- 表达某个稳定区间、稳定类别或稳定值域

3. `复合抽象状态`
- 不直接塞进单一特征的细分主干
- 挂在状态域的聚合分支下
- 与其组成状态之间保留组成关系


## 4. 数据结构建议

当前代码里还没有专用抽象状态节点主信息类，因此建议新增一类专用主信息。

建议命名：

- `抽象状态节点主信息类`

### 4.1 最小字段

最少应包含：

- `概念域`
  - 固定为 `状态`
- `抽象状态类型`
  - `单特征`
  - `复合`
- `来源特征类型`
  - 单特征抽象状态必填
- `父状态概念主键`
  - 上层细分时使用
- `值域 / 区间 / 类别`
  - 单特征抽象状态的核心身份
- `组成状态主键列表`
  - 复合抽象状态必填
- `适用条件主键`
  - 允许为空，但稳定后应补齐
- `命中次数`
- `首次命中时间`
- `最近命中时间`
- `稳定度`
- `活跃度`
- `置信度`
- `当前状态`
  - `候选`
  - `稳定`
  - `冷却`
  - `待淘汰`

### 4.2 明确不进入身份字段的内容

以下字段不得作为抽象状态身份的一部分：

- 某次实例状态的 `发生时间`
- 某次实例状态的 `收到时间`
- 某次实例状态的唯一主键
- 某次实例状态的变化原因说明
- 某个具体存在主键


## 5. 实例状态到抽象状态的转换

## 5.1 单特征抽象状态

标准流程固定为：

```text
实例状态
-> 取特征类型
-> 取状态值
-> 将状态值映射到稳定区间 / 稳定类别
-> 删除单次实例时间语义
-> 形成抽象状态候选键
-> 查找是否已有对应候选
-> 没有则创建，有则更新命中统计
```

### 5.1.1 候选键建议

候选键建议由以下字段组合：

- `概念域`
- `来源特征类型`
- `区间/类别签名`
- `适用条件主键`

例如：

```text
状态|安全值|低位|普通运行
状态|服务值|中位区|普通运行
```

### 5.1.2 区间映射要求

区间映射必须由特征类型自己定义，不允许全域只用一种切法。

也就是说：

- `安全值` 的低位/高位阈值可以有自己规则
- `服务值` 的中位/高位规则可以有自己规则
- 指针类特征可以走类别映射，不一定走数值区间


## 5.2 复合抽象状态

标准流程固定为：

```text
多个单特征抽象状态被同一存在在相近观察窗口内共同命中
-> 提取组成状态集合
-> 生成复合候选键
-> 统计共同命中次数
-> 达阈值后升格为复合抽象状态
```

### 5.2.1 复合候选键建议

建议由以下字段组合：

- `概念域`
- `组成状态集合签名`
- `适用条件主键`

例如：

```text
状态|{安全值稳定带,服务值中位区,风险安全稳定}|普通运行
```

### 5.2.2 共同命中窗口

建议固定一个“共同命中观察窗口”：

- 同一存在在一个有限观察窗口内命中同一组单特征抽象状态
- 才允许计为一次复合命中

这样可以避免把跨很长时间、偶然拼接出来的状态误当作稳定复合状态。


## 6. 抽象动作实现口径

这里必须把“抽象时删除时间”收成工程动作，而不是自然语言描述。

建议新增统一入口：

- `尝试从实例状态生成抽象状态候选(...)`
- `尝试从命中组合生成复合抽象状态候选(...)`

### 6.1 抽象动作的内部步骤

单特征抽象动作至少做以下步骤：

1. 校验实例状态合法
2. 提取 `特征类型`
3. 提取 `状态值`
4. 映射到稳定区间/类别
5. 丢弃单次实例时间作为身份字段
6. 生成候选键
7. 更新或创建抽象状态候选

### 6.2 “删除时间”的正确实现

这里的“删除时间”不是把所有时间字段都物理清零，而是：

- 不把实例状态的 `发生时间 / 收到时间` 带入抽象状态身份
- 改为更新抽象状态自己的：
  - `首次命中时间`
  - `最近命中时间`

也就是说实现上应是：

```text
实例时间 -> 只用于更新统计
实例时间 -/-> 抽象状态身份
```


## 7. 命中关系设计

抽象状态建好后，需要让具体存在能够“命中”它。

建议保留两层关系：

1. `实例状态 -> 抽象状态`
- 表示某条实例状态被归到哪个抽象状态

2. `存在 -> 抽象状态`
- 表示某个存在当前或历史上命中过哪些抽象状态

这样后续可以同时支持：

- 审计与回放
- 概念统计
- 任务管理上下文压缩
- 学习和因果引用


## 8. 升格规则

抽象状态候选不是创建出来就稳定。

建议至少按下面规则升格：

### 8.1 单特征抽象状态升格

至少满足：

1. `命中次数 >= 最小阈值`
2. `最近命中时间` 仍在活跃窗口内
3. 区间命中没有频繁漂移
4. 与父状态相比具备独立区分价值

### 8.2 复合抽象状态升格

至少满足：

1. `共同命中次数 >= 最小阈值`
2. 组成状态集合相对稳定
3. 该组合在相近条件下可重复成立
4. 不是单次偶然拼接

### 8.3 升格后的状态

升格后应至少做：

1. 绑定正式概念名称
2. 固化概念树挂点
3. 允许被二次特征、因果和后续学习引用


## 9. 低频衰减与淘汰

抽象状态需要活跃度管理。

### 9.1 衰减输入

建议每轮周期或每次相关命中检查时，更新：

- `最近命中时间差`
- `最近观察窗口内命中频率`
- `稳定度`

### 9.2 衰减逻辑

若长期未命中或频率过低，则：

1. `活跃度下降`
2. `稳定度下降`
3. `推荐度下降`

### 9.3 淘汰逻辑

当同时满足以下情况之一时，进入淘汰态：

1. 长期未命中
2. 命中频率持续低于阈值
3. 与父状态概念相比没有独立区分价值
4. 复合组成关系已经失稳

### 9.4 淘汰处理建议

建议分三档：

1. `稳定 -> 冷却`
2. `冷却 -> 候选`
3. `候选 -> 清理`

不要一上来直接物理删除，优先保留成长轨迹。


## 10. 与现有系统的对接点

当前仓内已具备的可复用基础有：

1. `状态节点`
- 已能稳定产生实例状态

2. `二次特征主信息类`
- 已有 `概念名称`、`标量区间` 等可复用字段基础

3. `概念树`
- 规范层已明确存在 `概念域::状态`

因此第一阶段不建议重做整个概念系统，而建议：

1. 先补 `抽象状态候选结构`
2. 再补 `实例状态 -> 抽象状态候选` 的统一入口
3. 再补 `命中统计 / 升格 / 淘汰`
4. 最后再补控制面板和语言层显示


## 11. 第一阶段最小落地范围

建议第一阶段只做：

1. `单特征抽象状态`
2. 可量化 `I64` 特征
3. 已有稳定区间定义的特征

首批最适合接入的候选特征：

- `安全值`
- `服务值`
- `风险安全`

原因是：

- 已经有相对明确的区间语义
- 命中样本容易积累
- 也最容易被任务管理、根层重判和解释层消费


## 12. 一句话收口

实现上最重要的口径只有一句：

`抽象状态不是“时间为空的状态节点”，而是“实例状态经过归桶、归并、删除实例时间身份、保留概念统计后生成的状态概念节点”；它应挂到概念域，而不是继续伪装成普通状态事实节点。`
