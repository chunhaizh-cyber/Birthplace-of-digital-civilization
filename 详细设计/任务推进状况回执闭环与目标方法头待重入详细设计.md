# 任务推进状况回执闭环与目标方法头待重入详细设计

## 1. 目标

本文用于细化任务推进闭环的工程落地口径，重点固定以下规则：

```text
推进执行单元产出状况；
任务管理线程解释状况；
自我线程改树、结账、重调度；
子任务完成后父任务待重筹办。
```

同时修正“方法查找未命中”的处理：

```text
方法查找未命中
    ≠ 立即创建候选方法存在状态子需求

方法查找未命中
    = 筹办阶段创建或复用待学习目标方法头，并返回待重入
```

一句话定型：

```text
找不到方法 -> 建方法头；
方法头不可执行 -> 转成方法状态目标，后续筹办按结果能力选择承接方法；
方法可执行 -> 父任务执行目标。
```

## 3. 总体闭环

完整任务推进闭环如下：

```text
父需求 D
    ↓
父任务 T
    ↓
筹办
    ↓
返回筹办状况
    ↓
任务管理线程解释状况
    ↓
必要时上交自我线程改需求树 / 任务树
    ↓
创建或复用子需求 / 子任务
    ↓
子任务执行
    ↓
子任务完成
    ↓
父任务标记为待重筹办
    ↓
父任务重新筹办
    ↓
若方法、条件、参数均闭合，进入执行
    ↓
执行返回执行状况
    ↓
任务管理线程归并事实和条件结果对
    ↓
自我线程根据任务目标结果结算需求满足、价值账和风险账
```

该闭环不是固定两轮，而是按缺口多轮推进：

```text
第 1 轮：让目标方法头存在
第 2 轮：让方法有动作骨架
第 3 轮：让方法有条件结果对
第 4 轮：让方法有可靠运行账
第 5 轮：执行父任务目标方法
```

每个子任务完成后，父任务都必须重新筹办。

## 4. 三层职责边界

### 4.1 本能函数

本能函数负责最小动作和状况产生。任务筹办普通函数、任务执行普通函数、学习 / 练习 / 尝试学习函数也按同一推进执行单元边界处理。

职责：

```text
1. 检查或执行一个最小动作。
2. 返回本轮状况。
3. 产生事实、状态变化、动态记录。
4. 产生条件结果对样本。
5. 明确说明当前缺口。
6. 返回执行回执。
```

不负责：

```text
1. 不直接创建父需求的子需求。
2. 不直接给父需求结算安全值 / 服务值。
3. 不判断整个需求树下一步如何生长。
4. 不直接把父任务标记为最终完成。
5. 不替父需求做全局结算。
```

本能函数的定位是：

```text
产生状况，不解释全局意义。
```

因此必须固定：

```text
动作执行成功 ≠ 任务目标达成
任务目标达成 ≠ 治理结果可接受
任务目标达成 ≠ 父需求可直接结算
子任务完成 ≠ 父任务完成
```

### 4.2 任务管理线程

任务管理线程负责解释本能函数返回的状况，并推进任务局部状态。

职责：

```text
1. 判断当前状况是否可在任务内部继续处理。
2. 判断任务是否需要等待子任务。
3. 判断父任务是否需要重筹办。
4. 判断是否需要上交自我线程生成子需求。
5. 判断是否可以从筹办阶段进入执行阶段。
6. 解释执行结果。
7. 归并条件结果对。
8. 推进任务状态和任务阶段。
```

任务管理线程不直接做全局需求结算，也不直接修改全局价值账。它只把状况解释成任务层动作或自我线程消息。

### 4.3 自我线程

自我线程负责全局结构变化和治理。

职责：

```text
1. 新增、更新、满足、删除需求节点。
2. 子需求去重。
3. 创建或复用子任务。
4. 唤醒父任务。
5. 归类任务结果。
6. 结算安全值 / 服务值。
7. 治理禁止结果和风险结果。
8. 触发根层重判。
9. 触发主派发决议。
10. 重调度任务队列。
```

自我线程是唯一能够改变需求树、任务树和全局账本的治理层。

## 5. 任务推进阶段和状况

建议用统一阶段枚举承载筹办和执行：

```cpp
enum class 枚举_任务推进阶段 : std::uint8_t {
    未定义 = 0,
    筹办 = 1,
    执行 = 2,
};
```

建议用统一状况枚举承载本轮推进结果：

```cpp
enum class 枚举_任务推进状况 : std::uint8_t {
    未定义 = 0,

    父需求已满足 = 1,
    存在活动子需求 = 2,

    方法查找未命中 = 10,
    已建立方法头待重入 = 11,
    方法缺动作骨架 = 12,
    方法缺可执行入口 = 13,
    方法缺条件节点 = 14,
    方法缺结果节点 = 15,
    方法缺条件结果对 = 16,
    参数不可配 = 17,
    方法运行账不足 = 18,
    方法结果不稳定 = 19,
    方法可执行 = 20,

    动作执行失败 = 30,
    动作执行成功无可比较结果 = 31,
    目标结果符合 = 32,
    目标结果不符合 = 33,
    出现需要附带结果 = 34,
    出现不需要附带结果 = 35,
    出现禁止结果 = 36,
};
```

其中 `方法查找未命中` 是筹办内部事实，外部稳定回执应优先返回 `已建立方法头待重入`，表示未命中已被转化为目标方法头承载点。

## 6. 结构化回执

本能函数可以保留旧返回码，但外层必须形成结构化回执：

```cpp
struct 结构_任务推进回执 {
    枚举_任务推进阶段 阶段 = 枚举_任务推进阶段::未定义;
    枚举_任务推进状况 状况 = 枚举_任务推进状况::未定义;

    可解析引用<任务节点类> 当前任务{};
    可解析引用<需求节点类> 对应需求{};
    可解析引用<方法节点类> 当前方法{};

    const 词性节点类* 目标特征类型 = nullptr;
    三向关系掩码 目标满足关系 = 关系_等于;

    const 词性节点类* 缺口特征类型 = nullptr;
    三向关系掩码 缺口满足关系 = 关系_等于;

    std::vector<可解析引用<状态节点类>> 实际结果状态集{};
    std::vector<可解析引用<状态节点类>> 禁止结果状态集{};
    std::vector<可解析引用<状态节点类>> 需要附带结果状态集{};

    bool 需要自我线程 = false;
    bool 父任务需要重筹办 = false;
    bool 需要创建或复用子需求 = false;
    bool 当前任务目标达成 = false;
    bool 治理结果可接受 = true;

    std::string 摘要{};
};
```

该结构的作用：

```text
1. 保留本轮阶段。
2. 保留具体状况。
3. 指向当前任务、需求、方法。
4. 标明目标特征。
5. 标明缺口特征。
6. 携带实际结果、禁止结果、附带结果。
7. 明确是否需要自我线程。
8. 明确父任务是否需要重筹办。
9. 明确是否需要创建或复用子需求。
```

## 7. 修正后的筹办流程

父任务筹办应按以下顺序推进：

```text
父任务 T
    ↓
读取对应父需求 D 的单目标
    ↓
判断父需求是否已满足
    ↓
判断是否存在活动子需求
    ↓
查找能满足 D 目标结果的方法
    ↓
如果查找未命中：
        创建或复用待学习方法头 M
        把 D 的目标特征类型 / 目标状态关系写入 M 的结果能力
        标记 M 为待学习 / 不可执行
        父任务 T 标记为待重筹办
        返回：已建立方法头待重入
    ↓
如果命中方法头：
        检查动作骨架
        检查可执行入口
        检查条件节点
        检查结果节点
        检查条件结果对
        检查参数绑定
        检查运行账和可靠度
    ↓
如果方法可执行：
        返回：方法可执行
        任务管理线程切换到执行阶段
```

关键约束：

```text
创建方法头 ≠ 找到可执行方法
创建方法头 ≠ 父需求满足
创建方法头 ≠ 当前任务完成
```

创建方法头只是把“缺少方法”变成一个可学习、可补结构、可练习的对象。

## 8. 目标方法头写入内容

当父需求是：

```text
目标主体 = A
目标特征类型 = X
当前状态 = S0
目标状态 = S1
满足关系 = 等于 / 大于 / 小于组合
```

筹办创建或复用的方法头 `M` 至少应写入：

```text
M.来源需求 = D
M.来源任务 = T
M.结果能力.目标特征类型 = X
M.结果能力.目标状态关系 = D.满足关系
M.结果能力.目标状态 = S1
M.当前成熟度 = 待学习
M.当前可执行状态 = 不可执行
```

下一轮查找方法时，系统不是按名字找到它，而是按：

```text
这个方法头声明自己能产生父需求需要的目标结果
```

找到它。

命中后仍必须继续检查：

```text
1. 有没有动作骨架。
2. 有没有可执行入口。
3. 有没有条件节点。
4. 有没有结果节点。
5. 有没有条件结果对。
6. 参数能不能配。
7. 运行账是否足够。
8. 结果是否稳定。
```

## 9. 方法查找未命中的处理边界

状态处理分界如下：

| 状况 | 处理者 | 动作 | 是否创建子需求 |
| --- | --- | --- | --- |
| 方法查找未命中 | 筹办本能函数 / 筹办阶段 | 创建或复用待学习方法头，写入目标结果能力 | 否 |
| 已建立方法头待重入 | 任务管理线程 | 将父任务标记为待重筹办，等待下一轮筹办 | 否 |
| 方法头存在但缺动作骨架 | 任务管理线程 + 自我线程 | 创建或复用方法动作骨架状态子需求 / 子任务 | 是 |
| 方法头存在但缺可执行入口 | 任务管理线程 + 自我线程 | 创建或复用方法可执行入口状态子需求 / 子任务 | 是 |
| 方法头存在但缺条件结果对 | 任务管理线程 + 自我线程 | 创建或复用方法条件结果配对状态子需求 / 子任务 | 是 |
| 方法头存在且可执行 | 任务管理线程 | 进入执行阶段 | 否 |

核心边界：

```text
没有方法头：筹办造方法头。
方法头不可执行：学习子任务补方法。
方法可执行：父任务执行目标。
```

这里没有绕过方法查找：

```text
查找方法返回未命中，这个结果是真的。
筹办基于未命中事实创建待学习方法头。
下一轮命中的不是可执行方法，而是待学习方法头。
待学习方法头仍必须通过可执行性检查。
```

## 10. 筹办状况路由

筹办阶段的状况路由如下：

| 筹办状况 | 说明 | 处理者 | 后续 |
| --- | --- | --- | --- |
| 父需求已满足 | 父需求目标状态已经达成 | 任务管理线程、自我线程 | 父任务目标达成，交自我线程结算 |
| 存在活动子需求 | 仍有未完成子需求阻塞父任务 | 任务管理线程 | 父任务等待或调度子任务 |
| 方法查找未命中 | 没有方法头或可执行方法覆盖目标结果 | 筹办阶段 | 创建或复用待学习方法头 |
| 已建立方法头待重入 | 待学习方法头已出生 | 任务管理线程 | 父任务待重筹办 |
| 方法缺动作骨架 | 方法有结果声明但不能执行 | 自我线程 | 创建或复用动作骨架子需求 |
| 方法缺可执行入口 | 方法没有可调用动作入口 | 自我线程 | 创建或复用可执行入口子需求 |
| 方法缺条件节点 | 方法结构缺条件节点 | 自我线程 | 创建或复用条件节点子需求 |
| 方法缺结果节点 | 方法结构缺结果节点 | 自我线程 | 创建或复用结果节点子需求 |
| 方法缺条件结果对 | 方法缺少可验证经验 | 自我线程 | 创建或复用条件结果配对子需求 |
| 参数不可配 | 实际场景缺少方法所需参数特征，当前任务输入未闭合 | 自我线程或任务管理线程 | 创建或复用参数配对状态 / 任务参数绑定状态目标；不落到方法是否可用 |
| 方法运行账不足 | 方法执行样本不足 | 自我线程 | 创建或复用运行账完整性子需求，后续由筹办按结果能力选择承接方法 |
| 方法结果不稳定 | 方法结果复现不稳定 | 自我线程 | 创建或复用稳定状态子需求，后续由筹办按结果能力选择承接方法 |
| 方法可执行 | 方法、条件、参数均闭合 | 任务管理线程 | 进入执行阶段 |

## 11. 缺口到子需求目标的映射

“方法查找未命中”不再映射为子需求，而是映射为目标方法头出生。

| 返回状况 | 后续目标 |
| --- | --- |
| 方法查找未命中 | 创建或复用待学习目标方法头 |
| 方法缺动作骨架 | 方法动作骨架状态 = 已形成 |
| 方法缺可执行入口 | 方法可执行入口状态 = 已具备 |
| 方法缺条件节点 | 方法条件节点存在状态 = 已存在 |
| 方法缺结果节点 | 方法结果节点存在状态 = 已存在 |
| 方法缺条件结果对 | 方法条件结果配对状态 = 已配对 |
| 参数不可配 | 参数配对状态 / 任务参数绑定状态 = 已配对 |
| 方法运行账不足 | 方法运行账完整性 = 达标 |
| 方法结果不稳定 | 方法可靠度 / 稳定状态 = 达标 |

筹办返回的可执行性缺口必须转换成“让目标方法头可执行”的子需求，不能复制父需求原目标。

方法结构补齐任务必须把“被补方法”和“当前执行方法”分开：

```text
被补方法 = 缺动作骨架 / 条件结果对 / 运行账的目标方法头
当前执行方法 = 能产出该缺口目标的方法，可由学习 / 练习 / 尝试学习承接，但必须由筹办查找命中
```

学习承接不是硬性通道。底层规则保持一致：筹办查找方法，执行按任务推进闭环分解子任务；学习 / 练习只是声明了可产出多种方法状态，因此可能被普通匹配选中。

不得把被补方法本身绑定为当前执行方法；否则会把“修方法”的子任务误当成“执行父需求目标”的业务方法。

示例：

```text
父需求：
    改变 A 的 X 特征

第一轮筹办：
    找不到方法
    创建方法头 M，M 声明能改变 X
    父任务待重筹办

第二轮筹办：
    命中方法头 M
    M 缺动作骨架

子需求：
    主体 = 方法头 M
    特征 = 方法动作骨架状态
    目标 = 已形成
```

子需求主体是 `方法头 M`，不是父需求目标主体 `A`。

## 12. 待学习方法头去重

必须加入硬约束：

```text
同一方法树下，不得重复创建相同目标结果能力的待学习方法头。
```

去重判断至少包含：

```text
1. 目标特征类型。
2. 目标状态关系 / 方向。
3. 目标状态。
4. 目标主体类型或目标主体范围。
5. 来源需求或适用范围。
```

流程：

```text
查找可执行方法未命中
    ↓
查找是否已有待学习方法头覆盖该目标
    ↓
若已有：
        绑定该方法头
        父任务待重筹办
        返回：已建立方法头待重入
    ↓
若没有：
        创建方法头
        写入目标结果能力
        父任务待重筹办
        返回：已建立方法头待重入
```

这不是用名称查找，而是仍然按结果能力查找，只是候选范围包括：

```text
待学习方法头
```

## 13. 活动子需求判定

父任务是否等待，不能看子节点数量。

错误判断：

```text
父需求是否存在子节点
```

正确判断：

```text
父需求是否存在活动子需求
```

活动子需求定义：

```text
1. 未满足。
2. 未截止。
3. 未取消。
4. 仍然阻塞父任务执行。
```

如果只看子节点数量，则第一轮子需求完成后，历史子节点仍然存在，父任务会被错误卡住，无法进入后续筹办或执行。

## 14. 子任务完成后的父任务重筹办

子任务完成后，父任务不得直接完成。

固定流程：

```text
子任务完成
    ↓
父任务不直接完成
    ↓
父任务标记为待重筹办
    ↓
父任务重新进入筹办
```

原因是：

```text
子任务只解决一个阻塞点，不能证明父任务目标已经达成。
```

例如：

```text
子任务：补齐方法动作骨架
```

完成后只能说明：

```text
方法动作骨架可能已经具备
```

不能说明：

```text
父需求目标已经满足
```

所以父任务必须重新检查：

```text
1. 重新查找方法。
2. 重新检查方法可执行性。
3. 重新检查条件结果对。
4. 重新配对参数。
5. 重新判断是否还有活动子需求。
6. 重新判断父需求是否已满足。
```

## 15. 执行状况路由

执行阶段也必须返回精确状况。

| 执行状况 | 说明 | 是否更新方法成功/失败统计 | 后续 |
| --- | --- | ---: | --- |
| 动作执行失败 | 动作函数未成功产生可比较结果 | 否 | 记录运行失败账，不记结果符合/不符合 |
| 动作执行成功无可比较结果 | 有事实，但无法验证目标 | 否 | 归并事实，父任务重筹办或继续检查 |
| 目标结果符合 | 实际结果匹配当前任务目标 | 是，符合次数 +1 | 任务目标达成，需求满足检查 |
| 目标结果不符合 | 执行成功，但结果不覆盖目标 | 是，不符合次数 +1 | 归并条件结果对，可能生成学习子需求 |
| 出现需要附带结果 | 非当前目标，但命中其他有效需求 | 不计入当前目标统计 | 路由到对应需求账 |
| 出现不需要附带结果 | 非当前目标、非禁止结果 | 不计入当前目标统计 | 只沉淀事实 |
| 出现禁止结果 | 触犯禁止项或风险约束 | 可单独记风险账 | 自我线程风险治理 |

执行统计硬规则：

```text
只有在方法可执行、动作函数执行成功、产生可比较结果之后，
才用“结果是否符合预期”更新方法成功次数或失败次数。
```

可比较目标结果定义为：

```text
实际结果状态集里存在与当前任务目标特征类型相同、
可由特征比较服务比较、
且可按当前任务满足关系判断符合 / 不符合的状态变化。
```

动作函数执行成功但没有可比较目标结果时，只能返回 `动作执行成功无可比较结果`。该状况不得把任务标记为完成，不得触发父需求结算，不得更新目标结果符合 / 不符合统计。

## 16. 任务状态迁移

任务状态建议支持：

```text
未启动
运行中
等待中
待重筹办
完成
失败
```

任务阶段建议支持：

```text
筹办
执行
已完成
阻塞
```

核心迁移：

```text
未启动 -> 筹办 / 运行中
筹办 -> 方法查找未命中 -> 创建或复用方法头 -> 待重筹办
待重筹办 -> 筹办
筹办 -> 存在活动子需求 -> 等待中
筹办 -> 方法头可执行性缺口 -> 等待中
子任务完成 -> 父任务待重筹办
筹办 -> 方法可执行 -> 执行
执行 -> 目标结果符合 -> 完成
执行 -> 目标结果不符合 -> 待重筹办或失败
执行 -> 禁止结果 -> 阻塞 / 风险治理
```

## 17. 与现有实现的衔接点

当前工程已经具备的基础：

```text
1. 需求主信息已经单目标化。
2. 需求类已有同父子链目标重叠检查。
3. 需求类已有直接子需求目标视图收集逻辑。
4. 任务状态已有待重筹办。
5. 任务阶段已有筹办 / 执行 / 已完成 / 阻塞。
6. 自我线程消息处理器已有统一结算、需求树更新、任务控制态更新等入口。
7. 本能方法已有任务筹办、任务执行一步、查找方法、配对参数、改变特征值、检查需求是否满足等方法。
8. 本能函数已有 `已建立方法头待重入` 语义。
```

仍需重点确认或补齐：

```text
1. 筹办 / 执行是否都返回结构化任务推进回执。
2. 任务管理线程是否有统一状况路由表。
3. 方法查找未命中是否只创建或复用方法头，不创建子需求。
4. 方法头是否写入父需求目标结果能力。
5. 待学习方法头是否按目标结果能力去重。
6. 下一轮筹办是否能按结果能力命中待学习方法头。
7. 方法头不可执行时，是否只生成方法可执行性子需求。
8. 子需求目标主体是否为方法头、方法虚拟存在、任务或参数绑定场景。
9. 子任务完成后父任务是否必定进入待重筹办。
10. 父任务等待是否使用活动子需求，而不是子节点数量。
11. 执行结果是否完成四类归类。
12. 成功 / 失败统计是否只在结果验证阶段更新。
```

## 18. 最终系统原则

最终定型原则如下：

```text
筹办负责发现能不能执行；
找不到方法时，筹办负责让目标方法头出生；
方法状态子任务负责让目标方法头可执行，学习 / 练习是否承接由筹办和执行选择；
执行负责产生事实结果；
推进执行单元负责返回状况；
任务管理线程负责解释状况和推进任务状态；
自我线程负责改需求树、任务树、价值账和风险账；
每个子任务完成后，父任务必须重新筹办。
```

一句话：

```text
筹办和执行都不直接结束问题，它们只暴露状况；
状况被任务管理线程和自我线程吸收后，
才推动下一轮筹办、子任务承接、方法执行或价值结算。
```
