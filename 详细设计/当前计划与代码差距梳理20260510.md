# 当前计划与代码差距梳理 20260510

## 0. 总判断

当前代码已经具备很多结构骨架：

- 自我冷启动、根需求、根结构、核心状态读取诊断已经存在。
- `需求类` 已经具备需求更新指令、目标去重合并、结构角色刷新、阻塞子需求分权、任务初始化上下文和任务绑定接口。
- 自我线程已经能选择根需求、为需求创建任务信息节点、提交任务管理界面线程。
- 任务管理界面线程已经能绑定任务虚拟存在，并按需求初始化任务语义特征。
- 任务管理工作线程已经要求工作包必须携带任务虚拟存在，不再由工作线程补建。
- 因果补完目前已经有诊断和学习目标草案，但仍停在预览 / 闸门阶段。

但按最新阶段定义，第一道门槛还没有完成：

```text
自我同一目标特征的特征值差距
→ 根需求激活
→ 子需求自动生成 / 更新 / 合并
→ 执行需求任务化
```

也就是说，代码现在更多是：

```text
根需求存在
当前主需求可选
任务壳可创建
派生需求可由任务消息入树
```

还不是：

```text
需求树能从自我安全 / 服务这类同一目标特征的特征值差距中正常生长。
```

本文件中的“差距”只指同一目标特征类型上的当前值 / 目标值未满足。
它不能表达不同特征类型之间的关系；跨特征关系应进入状态动态、二次特征 / 关系特征、因果实例 / 因果模板或方法结果能力。

因此，`内部循环建立`、`自我苏醒`、`因果补完期`、`初步成熟` 均不能标为完成。

## 2. 关键代码证据

### 2.1 需求类有树操作骨架

`需求类.h` 中 `结构_需求树更新指令` 已经携带：

```text
需求主体
需求场景
被需求存在
当前状态
目标状态
满足关系
安全 / 服务权重
派生归因
```

`执行更新指令_已加锁` 支持新增、更新、满足、删除；新增时会补齐目标主体和特征缓存，并通过目标重叠查找合并直接子需求。

结论：

```text
需求树“被命令更新”的能力存在。
```

缺的不是底层插入函数，而是：

```text
谁在自我状态变化后构造这个更新指令。
```

### 2.2 自我线程的根需求权重口径仍不一致

`自我线程模块.impl.cpp` 中 `私有_从需求树构建需求列表` 当前把根需求权重写成当前值：

```cpp
安全根需求->主信息.安全权重 = 当前安全值;
服务根需求->主信息.服务权重 = 当前服务值;
```

而 `私有_选择当前主需求候选_按根权重` 使用的是压力差：

```text
安全压力 = 安全方向差值 * 根安全权重
服务压力 = (目标服务值 - 当前服务值) * 根服务权重
```

结论：

```text
主需求选择使用压力；
需求列表和子需求分权却使用当前值。
```

这会让“需求树生长权重”和“主需求选择压力”不是同一套口径。需求树正常生长阶段应统一成压力 / 差距驱动。

### 2.3 派生需求入树已经有硬入口

`私有_确保派生需求节点` 会从任务管理派生需求消息中解析父需求、当前状态、目标状态、目标特征类型、目标主体和派生归因，再用 `需求类::执行更新指令` 入树。

它还会拒绝：

```text
缺上级需求
上级需求是需求树结构根
缺当前状态或目标状态
缺目标特征类型
同目标复制父需求
```

结论：

```text
任务 / 方法阻塞 -> 派生需求入树 这条路径已经接近规范。
```

但它依赖任务管理消息，不解决：

```text
自我特征值差距 -> 原生需求树生长。
```

### 2.4 控制面板把根需求存在误判为“需求树生成成立”

`控制面板类.cpp` 当前：

```cpp
快照.需求树生成成立 =
    需求根节点
    && 快照.需求数 > 0
    && (自我.获取安全根需求() != nullptr || 自我.获取服务根需求() != nullptr);
```

这只说明：

```text
需求根和根需求存在。
```

不能说明：

```text
需求树已经根据特征值差距正常生长。
```

结论：

```text
控制面板命名和验收口径需要修订。
```

建议拆成：

```text
需求根初始化成立
根需求成立
特征值差距需求生成成立
子需求去重合并成立
执行需求任务化成立
需求树正常生长成立
```

### 2.5 任务虚拟存在链已具备，但不是需求树生长的证明

`任务模块.管理界面线程.impl.cpp` 中，界面线程会：

```text
确保任务虚拟存在
给任务虚拟存在写入发起存在、作用对象、上下文、当前状态、目标状态、目标、完成条件、满足关系、权重和任务语义状态
```

`任务模块.管理工作线程.ixx` 中，工作线程缺少任务虚拟存在会返回失败。

结论：

```text
任务信息节点 -> 任务虚拟存在 的新边界基本成立。
```

但这仍然只是：

```text
需求任务化以后能绑定语义身份。
```

不能替代：

```text
需求树能从自我状态中生长。
```

### 2.6 因果补完仍是诊断和草案

自我线程已有因果补完诊断快照、缺口类型、学习候选类型、学习目标主体 / 特征 / 当前状态 / 目标状态等字段。

控制面板也明确显示：


结论：

```text

```

## 3. 当前最重要的实现缺口

### 缺口 1：自我特征值差距没有生成执行子需求

目标链路应为：

```text
安全 / 服务值变化
→ 计算压力差距
→ 找到安全根需求 / 服务根需求
→ 生成或更新子需求
→ 子需求目标指向自我核心特征状态
→ 子需求角色为执行需求
```

当前缺少一个正式入口，例如：

```cpp
需求节点* 根据自我特征值差距确保根需求子需求(
    自我类& 自我对象,
    需求节点* 根需求,
    枚举_自我根需求类型 类型,
    时间戳 now);
```

该入口应构造 `结构_需求树更新指令`，而不是直接改需求节点字段。

### 缺口 2：根需求权重应使用压力，不应使用当前值

安全 / 服务根需求权重应来自：

```text
目标值 - 当前值
根需求权重
压力阈值
```

而不是直接把当前安全值 / 服务值写入权重。

### 缺口 3：控制面板需求树指标要改名和分级

`需求树生成成立` 应降级或改名为：

```text
需求根初始化成立
```

真正的 `需求树正常生长成立` 至少需要：

```text
有由自我特征值差距生成的活动子需求；
该子需求不是根需求；
该子需求具备当前状态、目标状态、满足关系、目标特征类型；
重复运行不会生成重复同目标子需求；
可任务化。
```

### 缺口 4：内部循环验收需要真实轮次剧本

现在诊断结构能判断很多字段，但还缺一个连续场景：

```text
服务值低
→ 生成服务修复子需求
→ 创建任务信息节点
→ 绑定任务虚拟存在
→ 选择提升服务值本能动作
→ 执行回写服务值
→ 结算来源需求
→ 下一轮压力降低
```

没有这个剧本，不能把内部循环标成建立。

### 缺口 5：自检线程边界要保持

最新口径：

```text
自检线程只能发现缺失，并报送自我线程。
自检线程不生成修复需求。
有些缺失只能等自我休眠时才能修复。
```

因此苏醒阶段的“自我能生成自检 / 修复需求”要修订为：

```text
自检线程发现缺失 -> 报送自我线程；
自我线程按运行态、待机/休眠条件、风险门控决定是否生成修复需求。
```

休眠期修复动作的决策规则当前放入待完成项，不应在本阶段凭空实现。

## 4. 建议的修订后实现顺序

### P1：口径修正和仪表修正

1. 控制面板把 `需求树生成成立` 拆成根初始化与正常生长两级。
2. 计划索引中明确：当前未完成项首位是 `P2b 需求树生成成立`。
3. 禁止把因果补完诊断、学习目标草案、任务虚拟存在绑定误记为内部循环完成。

### P2：需求树特征值差距生长

1. 增加自我特征值差距转需求更新指令的本能转换函数。
2. 服务值不足时，在服务根需求下创建或更新服务修复执行需求。
3. 安全值不足时，在安全根需求下创建或更新安全修复执行需求。
4. 重复运行走目标重叠合并，不重复长同目标需求。
5. 根需求与子需求权重统一使用压力口径。

### P3：执行需求任务化

1. 对 P2 生成的执行需求调用现有任务化入口。
2. 确保任务信息节点绑定任务虚拟存在。
3. 控制面板展示该任务来自特征值差距子需求，而不是根需求占位。

### P4：本能动作执行与回写

1. 让服务 / 安全修复任务能选到安全服务推进或改变特征值本能方法。
2. 执行结果回写自我核心状态。
3. 任务反馈写入来源需求结算账本。

### P5：连续轮次验收

1. 增加最小内部循环剧本。
2. 验证服务值低时生成服务子需求并任务化。
3. 验证执行后服务压力下降。
4. 验证重复轮次不产生任务风暴。

### P6：苏醒阶段

只有 P2-P5 跑通后再推进：

```text
连续多轮稳定
待机
自检报送
自我报告
故障收口
```

### P7：因果补完与初步成熟

只有苏醒后再推进：

```text
因果缺口正式生成学习需求
学习需求任务化
内部学习不足转外界探索需求
探索安全门控
探索结果回写因果
```

## 6. 下一步最小落地点

下一步不应继续推进苏醒或初步成熟，而应回到第一阻塞点：

```text
实现“自我特征值差距 -> 根需求子需求”的生长函数。
```

最小验收：

```text
服务值低于目标值
→ 服务根需求下出现一个活动执行子需求
→ 子需求具备当前状态、目标状态、满足关系、目标特征类型
→ 再跑一轮不会重复创建同目标子需求
→ 当前主需求能落到该子需求或其任务化结果
```

只有这一步成立，后续任务化、执行、结算、苏醒才有真实承载对象。
