# 任务管理任务详细设计

## 1. 目标

本文用于对当前工程中的 `任务管理任务` 重新做一版可落地的详细设计，统一回答以下问题：

- `任务管理任务` 在主运行闭环中的正式职责到底是什么
- 哪些职责必须留在 `自我线程主循环`
- `任务管理请求 / 任务管理上下文 / 一步治理结果 / 实例堆 / 宿主任务虚拟存在 / 恢复快照` 应如何拆层
- `执行一步治理` 应按什么顺序推进，哪些环节必须显式化
- `任务管理任务` 与 `任务执行器 / 学习承接链 / 方法树 / 任务树` 的边界是什么
- 如何从当前 `V0` 单函数实现，收口到后续可持续扩展的模块化实现

本文是工程详细设计，不替代上位规范；若与规范冲突，应以上位规范为准。

## 3. 当前实现现状

当前工程已经具备 `任务管理任务` 的最小可运行闭环，主要能力包括：

- 读取最小任务管理上下文
- 选择宿主任务
- 绑定或回填当前方法挂点
- 推进宿主任务状态
- 组织治理步骤节点与治理结果节点
- 创建或复用业务子任务、补条件子任务、学习子任务
- 写回治理稳定字段、动态和因果证据
- 生成学习方法骨架并同步到学习账本

当前 `V0` 的主要优点：

- 主链已经统一成 `主循环 -> 任务管理一步治理 -> 外层再消费`
- 任务树、方法树、学习树已经不再完全割裂
- 结果面、上下文字段和治理快照已经有正式结构承载

当前基线的主要剩余问题：

- `执行一步治理` 仍然承担了过多职责，结构上接近“大总管函数”
- `判域 / 缺口提炼 / 方法挂点 / 状态迁移 / 子任务创建 / 学习转接 / 证据写回` 交叉在同一条流程里，难以局部替换
- `执行一步治理` 仍然偏大，虽然已切出桥接阶段和四层壳，但内部还没有完全机械分段
- `mailbox / 显式唤醒队列` 已经落地，但事件优先级和更细粒度队列观测还可继续加强
- `延迟事实车道` 已经升级成正式待消费事实，但持久化与更细粒度归因还可继续加强
- `最终去向` 已统一由主循环裁决，后续重点转为让业务链和学习链更充分利用这条统一裁决面

因此，当前阶段不再是“先把请求、学习承接和恢复快照从零立起来”，而是在 `M1-M6` 已完成基线上继续做模块机械拆分、业务链深化和回归自动化。

## 4. 设计总原则

必须固定以下原则：

1. `任务管理任务` 是治理承接者，不是根层裁决者。
2. `任务管理任务` 每轮最多只推进 `一步`，不得在内部连续推进多轮闭环。
3. `任务管理任务` 可以组织结构、挂点、等待态、学习承接，但不能越权给出“需求已满足 / 主链已结束 / 允许正式回流”。
4. `任务管理任务` 的正式输入不是黑箱，而是 `请求 + 最小稳定上下文 + 本轮补充材料`。
5. `任务管理任务` 的正式输出不是日志串，而是 `一步治理结果`。
6. `实例堆` 承载瞬时运行态，`宿主任务虚拟存在` 承载治理状态面，`恢复快照` 承载跨退出最小恢复集。
7. `来源归属` 与 `树结构归属` 必须分开，尤其是学习子任务。
8. `主循环` 是统一协调者，所有影响继续/停止/收束的大结论，都必须回到主循环再判。
9. `每个宿主任务` 最多只允许一个任务管理运行时实例，但 `任务管理主体` 允许跨宿主、跨候选路径、跨学习回流沉淀成长特征。
10. `任务管理主体虚拟存在` 不替代 `实例堆 / 宿主任务虚拟存在 / 恢复快照` 三层，而是位于其上的跨轮成长汇总层。
11. `方法需求位` 是成长需求到具体方法枝之间的正式桥层，不得跳过。
12. `任务管理方法因果` 必须晚于 `状态镜像完成 + 动作动态完成`，不得在半成品状态上抢跑。
13. `任务完成` 是任务管理主体的局部运行目标，`安全值 / 服务值` 仍然是上位方向约束。

## 5. 主链中的正式位置

任务管理任务在最新主链中的正式位置固定为：

```text
主循环消费消息
-> 先天机制执行
-> 根层重判
-> 需求治理与主需求刷新
-> 装配任务管理上下文
-> 生成任务管理请求
-> 任务管理任务单步推进
-> 返回一步治理结果
-> 主循环再次重判与决定去向
-> 若允许继续，再提交下一轮治理/执行/学习
```

这里必须明确：

- `主循环` 负责根层重判、双值结算、需求整理、最终去向决定。
- `任务管理任务` 负责宿主任务的一步治理承接。
- `任务执行器` 负责叶子执行。
- `学习承接链` 负责缺口补齐。
- `任务管理任务` 不能直接替代主循环做最终拍板。

## 6. 重新定义任务管理任务的职责面

### 6.1 应承担的职责

`任务管理任务` 后续正式只承担以下 8 类职责：

| 职责面 | 说明 | 输出 |
| --- | --- | --- |
| `宿主承接` | 确认本轮围绕哪个宿主任务继续治理 | 宿主锚点 |
| `判域提缺` | 判当前功能域，提炼主缺口 | 功能域、缺口 |
| `去向生成` | 生成本轮下一步去向候选 | 去向候选 |
| `方法组织` | 召回、绑定、切换或出生方法挂点 | 当前方法挂点、方法来源 |
| `任务树组织` | 推进宿主状态、步骤节点、结果节点、业务子任务、补条件子任务 | 结构写回结果 |
| `学习承接` | 仅在确有学习缺口时提出学习任务与学习对象 | 学习承接决策 |
| `治理写回` | 把治理状态面、证据、摘要写回稳定载体 | 治理镜像、动态、因果、摘要 |
| `单步返回` | 将本轮最小稳定结果返回外层 | 一步治理结果 |

### 6.2 不应承担的职责

以下职责必须留在主循环或其它承接层：

- 根层重判
- 安全值 / 服务值统一结算
- 需求集合整理、合并、去重、权重调整
- 最终继续/停止/待机/收束拍板
- 叶子方法长时间执行
- 外设阻塞等待
- 学习回流最终允许裁决
- 方法成熟度正式升格裁决

## 7. 六层数据面重构

在进入这六层之前，先补一条主体化边界：

- 六层数据面回答的是“单轮单步治理怎样承接”
- 但 `任务管理主体虚拟存在` 回答的是“到底是谁在跨轮成长”

因此，后续实现顺序不能直接从“六层解耦”开始，而应先把主体层立住，再做六层重构。

当前工程已经以以下六层数据面为正式主线，并在 `M1-M4` 中落下了最小可运行版本：

### 7.1 `结构_任务管理请求`

定位：本轮单步治理的输入载体。  
归属：实例堆内部。

建议字段：

| 字段 | 作用 |
| --- | --- |
| `请求ID` | 本轮请求唯一标识 |
| `宿主任务` | 本轮宿主锚点 |
| `提交时间` | 本轮快照时间 |
| `提交序号` | 防重放、防乱序 |
| `上下文` | 最小稳定任务管理上下文 |
| `补充材料` | 本轮可选材料，如场景、快照、原始反馈摘要 |
| `触发事件类型` | 是普通推进、执行回流、学习回流还是外部反馈 |

说明：

- 请求是 `一次输入材料`，不是长期真值。
- 请求允许带补充材料，但补充材料不得直接冒充稳定上下文字段。
- 当前代码已正式挂带 `触发来源 / 来源最小原语位图 / 来源主观察特征 / 影子验证状态 / 是否允许正式资产提交 / 恢复重建摘要`。
- `M5` 的补层目标不是重写请求壳，而是让这些字段优先来自显式 `mailbox` 事件消费，而不是只由局部上下文反推。

### 7.2 `结构_任务管理上下文`

定位：稳定治理中间态。  
归属：请求核心、宿主任务虚拟存在镜像、恢复快照来源。

当前头文件里的上下文结构可以保留大方向，但后续按以下语义收口：

| 分层 | 字段 |
| --- | --- |
| `根层上位输入` | `最近根层重判结果`、`最近执行前门控结果` |
| `治理锚点` | `当前主需求`、`当前管理任务`、`当前宿主任务` |
| `任务侧真值镜像` | `当前步骤`、`当前方法`、`最近结果` |
| `治理中间态` | `当前缺口类型`、`当前下一步去向`、`最近功能域`、`最近总控结果` |
| `反馈摘要层` | `最近反馈类型`、`最近反馈摘要`、`最近上层反馈摘要` |

说明：

- 上下文服务承接，不服务调试炫技。
- 上下文字段必须可拆、可镜像、可恢复。

### 7.3 `结构_任务管理单步决策`

定位：本轮治理核心判定结果。  
归属：`执行一步治理` 内部中间结构，建议新增。

建议字段：

| 字段 | 作用 |
| --- | --- |
| `当前功能域` | 筹办 / 维护 / 纠偏 / 等待 / 收束 |
| `当前缺口类型` | 本轮主缺口 |
| `当前下一步去向` | 本轮候选去向 |
| `当前治理状态迁移` | 本轮应落的治理状态 |
| `当前方法来源` | 当前方法挂点来源 |
| `当前步骤位类型` | 执行位 / 等待位 / 回接位 / 补条件位 |
| `当前步骤语义类型` | 治理步骤 / 业务推进步骤 / 等待维持步骤 / 回接派生步骤 / 补条件步骤 |
| `当前后果治理` | 双面对齐后应如何继续 |

说明：

- 这层的目的是把“决策”从“大结果结构”里单独拎出来。
- 以后 `判域提缺` 与 `树写回` 应围绕这层解耦。

### 7.4 `结构_任务管理写回结果`

定位：本轮对任务树、方法树、学习树真正做出的结构动作集合。  
归属：一步治理内部产物，建议新增。

建议字段：

| 字段 | 作用 |
| --- | --- |
| `当前步骤节点` | 本轮结束后的步骤节点 |
| `最新结果节点` | 本轮结束后的结果节点 |
| `当前业务子任务` | 本轮新建或复用的业务子任务 |
| `当前补条件子任务` | 本轮新建或复用的补条件子任务 |
| `当前学习子任务` | 本轮新建或复用的学习子任务 |
| `当前学习方法` | 本轮学习资产挂点 |
| `宿主目标状态` | 宿主目标状态镜像 |
| `宿主结果状态` | 宿主结果状态镜像 |
| `双面摘要` | 目标/结果对齐摘要 |
| `结构动作摘要` | 本轮结构写回摘要 |

说明：

- 当前 `结构_任务管理结果` 中大量 `已创建X / 已复用X` 布尔位，后续应逐步下沉到这层。
- 最终对外返回时不需要暴露全部写细节，只需挂必要摘要与锚点。

### 7.5 `结构_任务管理一步结果`

定位：任务管理任务对外的正式返回结构。  
归属：外层主循环消费对象。

后续应固定成“核心稳定返回 + 可选挂带”两部分：

| 返回核心 | 作用 |
| --- | --- |
| `宿主任务` | 本轮返回锚点 |
| `当前功能域` | 本轮落在哪个治理域 |
| `当前缺口类型` | 本轮主缺口 |
| `当前下一步去向` | 本轮下一步候选 |
| `当前治理状态迁移` | 宿主状态迁移 |
| `当前步骤节点` | 当前步骤镜像 |
| `当前方法节点` | 当前方法镜像 |
| `最新结果节点` | 最近结果镜像 |
| `最近总控结果` | 最近总控摘要 |
| `最近反馈类型` | 本轮反馈类型 |
| `最近反馈摘要` | 本轮反馈摘要 |

可选挂带：

- 学习承接决策
- 收束建议
- 重筹办建议
- 管理证据摘要
- 动态与因果锚点
- 审计摘要
- 主体回并摘要
- 当前方法需求位
- 当前本能能力缺口
- 输入条件主键
- 治理态型

### 7.6 `实例堆 / 宿主任务虚拟存在 / 恢复快照`

这三层必须彻底分工：

| 层 | 保存什么 | 不保存什么 |
| --- | --- | --- |
| `实例堆` | 运行状态、等待原因、最近请求、mailbox、持有推进权 | 稳定任务真值 |
| `宿主任务虚拟存在` | 当前功能域、下一步去向、当前方法、当前步骤、最近总控结果、最近反馈摘要等治理状态面 | 线程对象、锁、队列 |
| `恢复快照` | 重建实例所需最小恢复集 | 整个请求、整个实例、函数执行中间栈 |

必须额外补一句：

- 上表三层只回答“每个宿主任务的一轮治理怎样跑、怎样记、怎样续”
- 它们还不负责“多个宿主、多条候选路径、多次学习回流之后，任务管理主体怎样汇总成长”

因此本设计补充新增一个上层承载：

### 7.7 `任务管理主体虚拟存在 / 分身继承面 / 方法需求位特征面 / 本能能力缺口面`

这四层不属于 `1219` 所定义的“每宿主一实例”运行时纪律的替身，而是位于其上的主体化补层。

它们分别回答：

| 层 | 回答什么 | 不回答什么 |
| --- | --- | --- |
| `任务管理主体虚拟存在` | 谁在跨轮成长，最近沉淀了什么推进模式、失败模式、价值证据 | 单个宿主本轮线程态细节 |
| `分身继承面` | 某条候选推进链/学习回流链这次继承了哪些宿主特征与方法特征 | 运行时实例调度队列 |
| `方法需求位特征面` | 当前缺的是哪类治理能力位，最少需要哪些状态/动态/因果原料 | 直接替代具体方法枝 |
| `本能能力缺口面` | 命中的方法需求位还缺哪些最小动作原语才能落地 | 高层治理目标本身 |

这四层的加入，目标不是把旧三层推翻，而是把“治理镜像系统”升级成“主体成长系统”。

## 8. 单步治理正式流程

后续 `执行一步治理` 必须重构为以下 13 段固定流程。

这里最关键的顺序变化不是“多几个判断”，而是：

- `主体状态面 / 分身继承面` 必须前置
- `方法需求位 / 本能能力缺口` 必须在正式决策链中显式出现
- `动态 / 因果` 必须建立在状态镜像已经稳定之后

不能再把这些内容留到流程尾部统一顺手补写。

### 8.1 受理请求

输入：`结构_任务管理请求`。  
动作：

- 校验 `请求ID / 宿主任务 / 上下文`
- 冻结本轮快照时间
- 标记本轮触发来源

产出：

- `本轮请求快照`

### 8.2 宿主锚点确认

动作：

- 若当前治理任务不存在，先补齐治理任务存在面
- 若当前宿主任务失效，按锚点规则重新选择宿主
- 优先沿当前宿主树继续推进，不默认全局重扫

宿主选择优先级建议固定为：

1. 当前宿主任务仍有效，则优先沿用
2. 若存在待回接的已完成直接业务子任务，则优先回到该父宿主
3. 若存在最深未终结派生业务子任务，则优先推进该派生链
4. 若以上都无，再回到锚点根任务重新展开

### 8.3 上位输入约束应用

动作：

- 消费 `最近根层重判结果`
- 消费 `最近执行前门控结果`
- 若上位仅允许等待 / 收束 / 停止，则本轮不得进入普通扩张治理

说明：

- 该阶段只消费上位结论，不重做上位裁决。

### 8.4 确保主体与分身存在

动作：

- 确保 `任务管理主体虚拟存在` 已存在
- 为当前宿主承接链创建或复用一个 `分身继承面`
- 绑定 `主体ID / 分身ID / 宿主任务虚拟存在引用 / 方法虚拟存在引用`
- 为本轮后续回并预留 `本地增量摘要 / 回并策略 / 最近锚点`

必须固定：

- 这里补的是主体级承载，不是运行时推进权。
- `分身` 是逻辑继承切面，不是并行运行时实例。

### 8.5 继承宿主/方法特征并刷新治理特征面

动作：

- 从 `宿主任务虚拟存在` 继承当前治理状态面
- 从 `方法虚拟存在` 继承当前方法资产和方法特征
- 生成或刷新 `输入条件主键 / 治理态型 / 结构完备度 / 方法可用度 / 风险门控`
- 先把小而硬的治理特征写成正式字段，再决定后续判域和桥层判断

产出：

- `治理特征面`
- `继承特征面`

必须固定：

- 这一阶段先形成稳定状态面，再进入动态、因果和方法桥层。
- 大快照、原始反馈整包和调试材料仍只留在请求补充层。

### 8.6 记录主体/分身动作动态并形成主体因果锚点

动作：

- 记录本轮主体/分身承接开始动态
- 在治理特征面稳定后，记录本轮最小承接动态
- 若需要生成本地因果锚点，也必须晚于 `状态镜像完成 + 动作动态完成`

必须固定：

- 这里只形成 `主体/分身` 的最小动态和局部因果锚点，不在此阶段越权生成完整方法长链。
- 该阶段的主要目的是保证后续 `方法需求位 / 本能能力缺口` 有合法的主体承接基础。

### 8.7 判域、提缺并命中当前方法需求位

输出：

- `当前功能域`
- `当前缺口类型`
- `当前方法需求位`

判域规则建议固定为：

| 条件 | 功能域 |
| --- | --- |
| 治理任务缺失、宿主缺失、方法挂点缺失 | `筹办` |
| 宿主运行中且仍可继续 | `维护` |
| 宿主挂起、待启动、需要从等待态重入 | `纠偏` |
| 宿主已终结或上位要求收束 | `收束` |
| 当前应等待执行、学习、外部反馈或稳定观察 | `等待` |

缺口提炼必须显式，不允许靠最终摘要反推。  
当前工程建议继续沿用现有缺口枚举，并把来源固定为：

- 上位约束型缺口
- 宿主缺失型缺口
- 方法挂点型缺口
- 状态推进型缺口
- 回接等待型缺口
- 学习补齐型缺口

同时必须固定：

- `方法需求位` 是成长需求和具体方法枝之间的桥层。
- 这一步只命中能力位，不直接伪造成具体方法已经成熟。

### 8.8 评估当前本能能力缺口并生成下一步去向候选

输出：

- `当前本能能力缺口类型`
- `缺失原语位图`
- `当前下一步去向`
- `当前治理状态迁移`

必须固定：

- 这一步要区分“当前缺的是方法枝”还是“当前缺的是支撑该方法需求位落地的最小动作原语”。
- `本能能力缺口` 是正式桥层，不得只停留在流程暗含逻辑。

去向候选只回答“本轮更像该往哪走”，不回答“外层最终是否批准”。

### 8.9 组织方法挂点

动作：

- 先沿用当前宿主已挂方法
- 再尝试当前主方法回填
- 再从最近结果回看方法
- 再从宿主方法树首节点候选筛选
- 最后才允许默认本能兜底

方法组织阶段必须输出：

- 当前方法节点
- 当前方法来源
- 方法可用但不可执行时的挂起原因

并且必须固定：

- 若 `方法需求位` 已命中，但 `本能能力缺口` 仍未补齐，则允许返回挂起/转学习，不得伪装成“方法已可执行”。

### 8.10 组织任务树结构

动作：

- 推进宿主任务状态
- 组织治理步骤节点与治理结果节点
- 视情况创建或复用业务子任务
- 视情况创建或复用补条件子任务
- 回接已完成业务子任务的结果

必须固定：

- `步骤节点` 解决当前治理位
- `结果节点` 记录本轮治理产物
- `业务子任务` 用于继续推进宿主目标
- `补条件子任务` 用于补当前可执行条件

### 8.11 组织学习承接

学习承接必须从“流程尾部顺手补”改成“显式决策后再创建”。

后续增加 `结构_学习承接决策`，至少包含：

| 字段 | 作用 |
| --- | --- |
| `是否触发学习` | 本轮是否进入学习承接 |
| `学习类型` | 定向学习 / 广泛学习 |
| `来源根任务` | 来源锚点 |
| `来源父任务` | 来源宿主 |
| `来源功能域` | 来源治理域 |
| `来源缺口类型` | 为什么要学 |
| `来源方法需求位` | 补哪段方法 |
| `来源本能能力缺口类型` | 当前缺的是哪类最小原语能力 |
| `来源最小动作原语位图` | 仍缺哪些原语 |
| `来源主观察特征` | 学习后要回读哪个主观察特征 |
| `回流目标` | 学完回哪 |
| `回流方法位/方法枝` | 回流后优先补到哪一位/哪一枝 |

学习触发规则建议固定为：

1. 有明确方法/条件/观察/可靠性缺口时，才允许定向学习。
2. 仅 `无方法` 或 `方法不可用`，先形成待学习记录，不自动等于立刻执行学习。
3. 只有待学习方法已具备 `动作骨架 + 条件侧`，才允许把它推入正式学习承接。
4. 学习任务结构上统一挂到 `任务管理学习树` 下，来源靠字段追溯，不靠树位置推断。
5. 当当前缺的是 `本能能力缺口` 而不是完整方法枝时，学习任务允许直接补齐最小动作原语或治理本能组合。

### 8.12 写回治理状态面、主体回并摘要与恢复信息

动作：

- 写回宿主任务虚拟存在上的稳定治理字段
- 写回 `主体回并摘要 / 当前方法需求位 / 当前本能能力缺口 / 输入条件主键 / 治理态型`
- 记录最小治理动态
- 记录最小治理因果锚点
- 更新治理实例堆与恢复快照

写回顺序固定为：

1. 先写宿主任务与主体层的小而硬状态面
2. 再回并本轮分身增量与主体摘要
3. 再写证据、动态与因果锚点
4. 最后更新恢复快照

必须固定：

- `状态面 -> 动态 -> 因果 -> 恢复` 的顺序不得倒置。
- 不能再把“先跑完流程，再在尾部统一补主体状态”当成正式实现口径。

### 8.13 组装一步治理结果并返回外层

动作：

- 组装稳定返回核心字段
- 挂带可选证据摘要
- 挂带可选 `主体回并摘要 / 当前方法需求位 / 当前本能能力缺口 / 输入条件主键 / 治理态型`
- 返回主循环

必须固定：

- 本轮返回后即结束本轮承接
- 不允许在 `任务管理任务` 内部继续连跑第二轮治理

## 9. 运行态与等待模型

本节不再按“未来完整线程池”来写，而按当前已落地和下一步待推进两栏收口。

### 9.1 已落地部分（M4 基线）

当前工程已经正式具备：

- `实例堆 / 宿主任务虚拟存在 / 恢复快照` 三层运行承载
- `恢复重建` 的一次性消费语义
- `等待学习回流 / 等待外部反馈` 的恢复快照投影
- 恢复后首轮必须回到 `主循环 -> 根层重判 -> 再提交` 的纪律
- 学习正式提交闸门等待态与等待原因显式化

这意味着：

- 恢复快照重建不再是未来式，而是当前正式主链的一部分
- 等待态已经可以跨退出保留最小治理语义
- 线程恢复后不会越权连续推进第二轮治理

### 9.2 已落地扩展（M5 / M6）

当前工程已经正式具备：

1. 最小 `mailbox / 显式唤醒队列`
2. `学习回流 / 外部反馈 / 执行结果 / 恢复重建 / 普通推进` 五类显式事件源
3. `延迟事实车道`
4. 显式“最终去向裁决”函数

当前已固定的运行纪律：

- `学习回流 / 外部反馈 / 执行结果 / 恢复重建 / 普通推进` 已统一进入显式事件源
- `同宿主一轮只推进一步` 的纪律保持不变
- `恢复重建` 保持一次性消费，不会因为邮箱而重复触发
- “稍后再处理”的事实已经从摘要占位升级成正式延迟事实
- 最终 `继续 / 待机 / 停止 / 收束 / 转入学习 / 再提交` 已由主循环统一拍板

### 9.3 当前等待纪律

必须继续固定：

- 等待通过状态建模，不通过阻塞物理线程实现
- 实例释放线程，但不丢治理语义
- 唤醒后必须重新回到 `主循环 -> 根层重判 -> 再提交`
- 即使引入 `mailbox`，也不得让同一宿主绕过主循环连续跑两轮

## 14. 最终结论

这版详细设计要收口的不是“再给当前代码补几个判断”，而是把 `任务管理任务` 从当前 `V0` 的大一统函数，重构成一套边界清楚、数据层明确、单步推进可审计、后续可升级到实例堆与事件唤醒模型的正式治理承接系统。

一句话收口：

`任务管理任务后续应被实现为：在主循环提交的稳定上下文之上，围绕宿主任务只推进一步、显式产出决策与写回结果、并把最终去向交还外层再判的治理承接器。`
