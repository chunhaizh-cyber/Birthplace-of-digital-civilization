# 存在-特征双根树

## 1. 接入规则

- 本文是在 [特征双根树.md](</D:/鱼巢/详细设计/特征双根树.md>) 基础上继续加一层：把 `特征所在的存在` 插回主树。
- 仍然保持 `双根` 不变：
  - `安全值`
  - `服务值`
- 但现在树节点允许出现两类对象：
  - `存在节点`
  - `特征节点`
- 记法统一为两种：
  - `自我存在::安全值`
  - `任务管理任务侧承载（当前宿主）`、`方法虚拟存在（当前方法）`、`任务虚拟存在（写回对象族）`
- 本树仍只统计 `49` 个特征类型，不按实例数展开。
- 其中进入主树的存在族按建模应然分成 4 类：
  - `自我存在`
  - `任务管理任务侧承载`
  - `任务虚拟存在`
  - `方法虚拟存在`
- `需求虚拟存在`、`学习账本实体` 当前还没有进入这棵树。

## 1.1 重要纠偏

- `自我只能挂自我特征`，这是一条归属边界，不因当前代码怎么写回而改变。
- 前一版把 `任务管理_*` 直接挂在 `自我存在`，那是按当前代码的写回宿主画出来的。
- 但按 `1219` 和 `1228` 口径，这批特征不应被解释为 `自我特征`，而应属于 `任务管理任务侧承载`：
  - 稳定治理字段优先落 `宿主任务虚拟存在`
  - 主体/分身/方法位/本能缺口桥接字段再落 `任务管理主体虚拟存在 / 分身继承面 / 方法需求位特征面`
- 历史实现曾把它们写进 `自我存在`，这也是前一版误挂的来源。
- 当前活跃读写已按任务侧宿主收口；若旧世界树或旧日志仍显示为 `自我存在 / 自我内部世界`，应按“历史实现残留 + 观测场景表现”解释，不改变本文的建模归属。
- 历史宿主兼容解释见 [任务管理历史宿主兼容说明.md](</D:/鱼巢/详细设计/任务管理历史宿主兼容说明.md>)。


## 2. 存在分布

| 存在族 | 当前进入主树的特征类型数 | 主树中的角色 |
| --- | ---: | --- |
| 自我存在 | 18 | 双根承载者，且只承载自我本体特征 |
| 任务管理任务侧承载 | 28 | 承载全部 `任务管理_*` 特征；优先对应宿主任务治理面，局部桥接项再下沉到主体/分身面 |
| 任务虚拟存在 | 2 | 承载 `任务目标状态 / 任务结果状态` |
| 方法虚拟存在 | 1 | 承载 `方法类型` |

## 4. 四个关键存在切换点

### 4.1 `自我存在 -> 任务管理任务侧承载`

- 切换位置：`自我存在::当前主任务`
- 含义：
  - `自我存在` 负责给出根方向、当前主需求和当前主任务
  - `任务管理任务侧承载` 负责围绕这个宿主展开具体治理状态面
- 所以这里是 `根方向面 -> 宿主治理面` 的切换点。

### 4.2 `任务管理任务侧承载 -> 方法虚拟存在`

- 切换位置：`任务管理_当前方法指针`
- 切过去以后当前进入的方法侧特征只有 1 个：
  - `方法虚拟存在（当前方法） -> 方法类型`
- 含义：
  - `任务管理任务侧承载` 负责决定“当前挂到哪个方法”
  - `方法虚拟存在` 负责承载“这个方法本身是什么类型”
- 所以这里是 `宿主治理面 -> 方法资产面` 的切换点。

### 4.3 `任务管理任务侧承载 -> 任务虚拟存在`

- 切换位置：`任务管理_当前治理状态迁移`
- 切过去以后当前进入的任务侧特征有 2 个：
  - `任务目标状态`
  - `任务结果状态`
- 含义：
  - `任务管理任务侧承载` 负责做治理判断、生成迁移结论
  - `任务虚拟存在` 负责把这个结论沉淀成任务账本双面状态
- 所以这里是 `宿主治理面 -> 任务账本面` 的切换点。

### 4.4 `任务虚拟存在 -> 任务管理任务侧承载`

- 切回位置：`任务管理_当前宿主目标结果比较`
- 含义：
  - 任务账本侧产出了 `目标状态 / 结果状态`
  - 任务管理侧再把它们拿回来，做 `已对齐 / 待补齐 / 存在差额` 的治理比较
- 所以这里是 `任务账本面 -> 宿主治理面` 的回流点。


## 5. 为什么 `任务虚拟存在` 在树里只出现一次

- 实际运行时，`任务目标状态 / 任务结果状态` 可能分别落在：
  - 宿主任务虚拟存在
  - 当前步骤任务虚拟存在
  - 最近结果任务虚拟存在
- 但本文统计的是 `特征类型树`，不是 `实例树`。
- 所以这里把它们统一收成一个：
  - `任务虚拟存在（写回对象族）`
- 这样做的好处是：
  - 不破坏“双根树”单次挂接结构
  - 仍然保留“这两个特征不属于自我存在，而属于任务虚拟存在”的关键信息


## 6. 任务完成在存在树里的位置

- `任务完成` 当前仍不是独立特征或独立存在。
- 它在存在树里的承载路径是：
  - `任务虚拟存在（写回对象族）::任务结果状态=完成`
  - `任务管理任务侧承载::任务管理_当前宿主目标结果比较=已对齐`
  - `任务管理任务侧承载::任务管理_当前后果治理=无额外后果/回接后保持等待`
- 也就是说：
  - `完成事实` 先在 `任务虚拟存在` 上成立
  - 然后回流到 `任务管理任务侧承载`，再反投到双根方向判断
- 所以 `任务完成` 是当前存在树里的一个 `跨存在组合概念`，而不是单一节点。

## 8. 下一步最值得分析的对象

- `任务管理任务侧承载`
  - 因为 `28` 个 `任务管理_*` 特征都聚在这里，说明真正重的不是 `自我存在`，而是任务管理治理面本身
- `自我存在`
  - 因为它只保留 `18` 个根侧/本体特征，更适合继续分析“哪些才是真正的自我恒定特征”
- `任务虚拟存在`
  - 因为它是“治理判断真正落到账本”的唯一出口
- `方法虚拟存在`
  - 因为它现在只承载 `方法类型`，明显偏薄，后续很可能继续长出更多方法侧特征
