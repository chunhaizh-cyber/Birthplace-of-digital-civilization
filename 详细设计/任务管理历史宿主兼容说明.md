# 任务管理历史宿主兼容说明

## 1. 当前判定规则

- `自我只能挂自我特征`。
- `任务管理_*` 特征当前应按 `宿主任务虚拟存在 / 任务管理主体虚拟存在 / 分身继承面 / 方法需求位特征面` 解释。
- 看到旧世界树、旧日志或旧状态节点里出现 `自我存在 / 自我内部世界`，不能直接把它判成当前语义宿主。

## 2. 为什么还会看到旧宿主

- 历史实现曾通过旧 helper 把大量 `任务管理_*` 治理证据写到 `自我存在`。
- 当前活跃读写已经迁出 `自我存在`，但旧日志、旧状态节点、旧世界树快照仍可能保留当时的宿主显示。
- `状态观测场景` 和 `语义宿主` 不是一回事；即使某条状态仍出现在 `自我内部世界`，也不代表它属于 `自我存在`。

## 3. 解释优先级

1. 先按特征语义判宿主：`任务管理_*` 默认归任务侧承载，不归 `自我存在`。
2. 再按当前活跃读写链判承载：优先看 `任务账本 / 主体桥接 / 分身继承 / 方法需求位特征面`。
3. 最后再解释旧显示：若旧世界树或旧日志仍落在 `自我存在 / 自我内部世界`，统一标成 `历史实现残留`。

## 4. 观察到旧宿主时怎么解释

| 看到的显示 | 应解释为 | 分析时怎么用 |
| --- | --- | --- |
| `自我存在::任务管理_*` | 历史实现残留 | 不拿它作为当前语义归属依据，只作历史回看 |
| `自我内部世界` 下的任务管理状态节点 | 旧快照或观测场景表现 | 继续回到任务账本、主体桥接字段判断真实宿主 |
| 控制面板仍能看见旧世界树节点 | 兼容展示保留 | 允许保留，但需要加“旧宿主残留”解释 |

## 5. 当前分析结论

- 当前活跃口径已经是 `任务账本 / 主体桥接`。
- 历史宿主说明的目的不是否认旧数据存在，而是防止把旧显示误判成当前语义。
- 控制面板世界树页现在会默认过滤已识别的 `历史宿主残留`，并单独列出残留清单，方便把“当前活跃树”与“旧实现遗留”分开看。
- 已补命令级入口：`--legacy-host-residue-summary` 用于输出当前残留摘要，`--archive-legacy-host-residue` 用于单独写出清理前归档快照，`--cleanup-legacy-host-residue` 用于执行安全清理。
- `--cleanup-legacy-host-residue` 若检测到残留，会先自动写出归档 JSON；若归档失败，则中止清理，避免“先删后无证”。
- 已补自动审计入口：`--acceptance-report` 现在会包含 `场景28 历史宿主残留自动审计`；若检测到残留，会先自动归档，再按 `提示 / 告警 / 阻断` 分级给出结论。
- 当前分级规则固定为：
  - `提示`：只剩因果模板/因果实例这类尾迹残留，未发现活跃链继续引用。
  - `告警`：残留已进入特征/状态/动态层，但当前未发现活跃链继续引用，属于“可清理但未阻断”。
  - `阻断`：仍有活跃节点引用旧宿主残留，说明旧宿主痕迹仍在活跃链上，必须阻断验收。
- 因此当前验收口径是：`提示/告警` 会保留归档并继续通过，`阻断` 或 `归档失败` 才会让 `场景28` 与整体验收失败。
- 清理顺序固定为 `因果模板 -> 因果实例 -> 动态 -> 状态 -> 特征`；若节点仍被活跃链引用，则保留为阻塞项，不做强删。
- 后续如果继续做世界树清洗，应优先处理“旧节点保留但无新写入”的识别与标注，而不是回头恢复旧宿主写法。
