# 任务线程请求承接与状态提交收口详细设计 20260512

## 1. 设计结论

按最新线程边界，任务线程模块要收成三层：

```text
自我线程：
    发任务发起 / 调度 / 重筹办 / 等待 / 取消 / 权重更新请求。
    不创建任务信息节点。
    不写任务状态。

任务管理界面线程：
    接收请求。
    创建任务信息节点和任务虚拟存在。
    初始化任务语义。
    派发工作包。
    统一提交任务状态、任务阶段、当前方法、可执行输入参数场景和任务虚拟存在镜像。
    生成任务状态动作动态。
    上行消息给自我线程。

任务管理工作线程：
    只执行一轮任务推进。
    调用任务筹办 / 任务执行本能方法。
    读取运行存在中的任务层回执。
    产出任务工作结果和治理结果。
    不直接写任务状态。
```

硬规则：

```text
任务域真实提交只有一个入口：任务管理界面线程。
```

其中任务域包括：

```text
任务状态
任务阶段
任务虚拟存在上的任务状态镜像
当前方法
任务方法候选集
可执行输入参数场景
筹办 / 执行缺口
等待原因
任务折算值
任务承接权重
任务状态动作动态
```

本文件是后续实施计划的详细设计依据。实施计划必须逐项对照本文件。

## 3. 请求协议设计

### 3.1 请求类型枚举

在 `任务管理线程协议` 中新增：

```cpp
enum class 枚举_任务管理请求类型 : std::uint8_t {
    未定义 = 0,

    发起任务 = 1,
    调度任务 = 2,
    重筹办任务 = 3,
    等待任务 = 4,
    取消任务 = 5,
    更新权重 = 6,
    同步回执 = 7,
};
```

语义：

```text
发起任务：
    来源需求有效，任务信息节点可以为空。
    界面线程负责创建任务信息节点和任务虚拟存在。

调度任务：
    任务信息节点必须有效。
    界面线程根据任务状态构造筹办或执行工作包。

重筹办任务 / 等待任务 / 取消任务：
    任务信息节点必须有效。
    界面线程验证状态迁移并提交状态。

更新权重：
    任务信息节点必须有效。
    界面线程固化任务承接权重。

同步回执：
    工作线程或外部回执进入任务域的专用同步入口。
```

### 3.2 治理请求扩展

现有 `结构_任务治理请求` 保留，但加入请求类型：

```cpp
struct 结构_任务治理请求 {
    枚举_任务管理请求类型 请求类型 =
        枚举_任务管理请求类型::调度任务;

    std::uint64_t 请求ID = 0;
    std::string 幂等键{};

    任务节点类* 任务信息节点 = nullptr;
    任务节点类* 上级任务信息节点 = nullptr;
    需求节点类* 来源需求 = nullptr;

    存在节点类* 发起存在 = nullptr;
    存在节点类* 作用对象 = nullptr;
    场景节点类* 所属场景 = nullptr;

    时间戳 当前时间 = 0;

    bool 允许创建任务信息节点 = false;
    bool 允许创建任务虚拟存在 = true;
    bool 允许初始化任务语义 = true;
    bool 允许派发工作线程 = true;
    bool 允许复用等待任务 = true;

    枚举_任务状态 建议初始任务状态 = 枚举_任务状态::未启动;

    I64 需求承接权重 = 0;
    I64 任务初始权重 = 0;

    const 语素入口节点类* 任务完成判据 = nullptr;
    const 语素入口节点类* 需求回写规则 = nullptr;
    const 语素入口节点类* 调度原因 = nullptr;

    std::string 请求摘要{};
};
```

注意：

```text
请求摘要只允许作为日志文本，不得作为系统语义判断依据。
```

## 4. 幂等键设计

现有幂等键不能只依赖任务节点，因为发起任务请求可能尚无任务节点。

新增：

```cpp
std::string 私有_构造任务治理幂等键(
    const 任务管理线程协议::结构_任务治理请求& 请求) noexcept;
```

规则：

```text
发起任务：
    发起任务|需求=<需求主键>

调度任务：
    调度任务|任务=<任务主键>|需求=<需求主键>

重筹办任务：
    重筹办任务|任务=<任务主键>

等待任务：
    等待任务|任务=<任务主键>

取消任务：
    取消任务|任务=<任务主键>

更新权重：
    更新权重|任务=<任务主键>|需求=<需求主键>

同步回执：
    同步回执|任务=<任务主键>|请求=<请求ID或回执主键>
```

发起任务的幂等键必须以需求为主，防止同一个需求多轮重复创建任务。

## 5. 界面线程请求承接流程

`任务管理界面线程::提交任务治理请求` 按请求类型分发。

### 5.1 发起任务

校验：

```text
来源需求必须有效。
任务信息节点允许为空。
若任务信息节点为空，必须 `允许创建任务信息节点=true`。
```

流程：

```text
1. 构造发起任务幂等键。
2. 查幂等索引，已有未完成请求则返回已有承接结果。
3. 如果任务信息节点为空：
       调用 `私有_按需求创建任务并初始化`。
4. 写入需求对应任务和任务列表。
5. 确保任务虚拟存在。
6. 初始化任务语义。
7. 若请求允许派发工作线程：
       根据任务状态构造工作包。
   否则：
       只返回任务创建 / 语义初始化承接结果。
```

发起任务不必立即筹办。是否创建后立即筹办由请求的 `允许派发工作线程` 决定。

### 5.2 调度任务

校验：

```text
任务信息节点必须有效。
来源需求建议有效；若缺失，可从任务反查来源需求，但不得在工作线程里补写。
```

流程：

```text
1. 确保任务虚拟存在。
2. 根据任务状态构造推进模式。
3. 构造工作包。
4. 提交工作线程或进入等待工作包队列。
```

### 5.3 状态推进类请求

`重筹办任务`、`等待任务`、`取消任务` 不经过工作线程。

流程：

```text
1. 校验任务节点。
2. 读取旧状态。
3. 校验状态迁移是否合法。
4. 由界面线程提交状态和阶段。
5. 同步任务虚拟存在镜像。
6. 生成任务状态动作动态。
7. 推送上行消息。
```

## 6. 工作包推进模式

新增：

```cpp
enum class 枚举_任务推进模式 : std::uint8_t {
    自动 = 0,
    只创建 = 1,
    只筹办 = 2,
    只执行 = 3,
    只同步回执 = 4,
    只状态推进 = 5,
};
```

`结构_任务工作包` 增加：

```cpp
枚举_任务推进模式 推进模式 = 枚举_任务推进模式::自动;
```

构造规则：

```text
请求类型 = 发起任务 且不允许派发工作线程：
    推进模式 = 只创建
    允许筹办 = false
    允许执行方法 = false

任务状态 = 未启动 / 待重筹办 / 无法执行：
    推进模式 = 只筹办
    允许筹办 = true
    允许执行方法 = false

任务状态 = 就绪 / 排队中 / 执行中：
    推进模式 = 只执行
    允许筹办 = false
    允许执行方法 = true

其他状态：
    推进模式 = 只状态推进
    允许筹办 = false
    允许执行方法 = false
```

禁止默认同一工作包同时允许筹办和执行，除非显式 `推进模式=自动` 并有独立验收理由。

## 7. 工作线程边界

工作线程职责：

```text
1. 校验工作包。
2. 按推进模式调用任务筹办或任务执行本能方法。
3. 读取运行存在中的任务层回执。
4. 读取输出结果场景中的世界事实摘要。
5. 填充工作结果和治理结果。
6. 生成上行消息建议。
```

工作线程禁止：

```text
1. 调用 `任务类::写入任务状态`。
2. 调用 `任务类::写入任务当前阶段`。
3. 调用 `任务类::同步任务运行镜像特征`。
4. 写任务虚拟存在上的任务状态、当前方法、可执行输入参数场景。
5. 生成已经提交的任务状态动作动态。
```

当前需迁移点：

```text
`任务模块.管理工作线程.ixx` 中 `私有_同步任务层回执到任务壳` 仍直接写任务状态和阶段。
```

目标改名：

```cpp
void 私有_填充任务层回执到治理结果(
    任务管理线程协议::结构_任务工作结果& 工作结果,
    const 任务模块::筹办::结构_任务推进回执& 回执,
    枚举_任务状态 推导状态,
    时间戳 now) noexcept;
```

它只写：

```text
工作结果.治理结果.宿主任务旧状态
工作结果.治理结果.宿主任务新状态
工作结果.治理结果.当前方法
工作结果.治理结果.可执行输入参数场景
工作结果.治理结果.筹办缺口类型
工作结果.治理结果.执行缺口类型
工作结果.治理结果.任务层回执运行存在
工作结果.治理结果.等待原因
工作结果.治理结果.派生需求提示
```

## 8. 回执读取边界

三通道规则：

```text
运行存在：
    任务层含义。读取建议任务状态、目标是否达成、缺口、等待原因、派生需求提示、当前方法、可执行输入参数场景。

输出结果场景：
    世界事实。读取真实特征变化、状态节点、动态事实、因果实例、结果状态。

方法虚拟存在：
    方法记忆。读取或更新运行账、最近运行、学习产物、条件结果对索引。
```

工作线程必须拆成两类读取函数：

```cpp
const 语素入口节点类* 读取任务层回执语素入口(
    const 存在节点类* 运行存在,
    const 语素入口节点类* 特征类型) noexcept;

const 语素入口节点类* 读取世界结果语素入口(
    const 场景节点类* 输出结果场景,
    const 语素入口节点类* 特征类型) noexcept;
```

任务层字段不得从输出结果场景读取。

## 9. 状态推导规则

工作线程可以推导，但只作为治理结果建议，不提交。

优先级：

```text
1. 运行存在 / 任务层回执中的 建议任务状态。
2. 推进模式对应的确定性兜底。
3. 旧执行结果推导兜底。
```

确定性兜底：

```text
只筹办：
    筹办成功 -> 就绪
    无候选方法 / 候选不可用 / 参数缺失 -> 待重筹办或等待中
    任务目标不可解析 -> 无法执行

只执行：
    当前方法运行成功且需求满足 -> 完成
    当前方法运行成功但需求未满足 -> 待重筹办
    当前方法运行失败 -> 失败或待重筹办
```

禁止恢复旧规则：

```text
本能动作运行成功 -> 待重筹办
```

## 10. 界面线程统一提交工作结果

`私有_应用工作结果到任务壳` 是唯一工作结果提交点。

目标流程：

```text
1. 读取任务旧状态。
2. 从工作结果治理结果读取建议新状态、阶段、当前方法、可执行输入参数场景、缺口、等待原因。
3. 校验状态迁移。
4. 提交任务状态 / 阶段。
5. 同步任务虚拟存在任务状态镜像、当前方法、可执行输入参数场景和缺口。
6. 生成任务状态动作动态。
7. 同步任务运行镜像。
8. 推送上行消息。
```

动作动态只在第 6 步生成。工作线程不得生成“已提交”的状态动作动态。

## 11. 回执结构通用化

当前 `任务模块::筹办::结构_任务推进回执` 可以先保留兼容，但后续应迁到任务管理协议：

```cpp
struct 结构_任务层回执 {
    枚举_任务状态 建议任务状态 = 枚举_任务状态::未定义;
    bool 目标是否达成 = false;

    const 语素入口节点类* 任务层执行回执 = nullptr;
    const 语素入口节点类* 筹办缺口类型 = nullptr;
    const 语素入口节点类* 执行缺口类型 = nullptr;
    const 语素入口节点类* 等待原因 = nullptr;
    const 语素入口节点类* 派生需求提示 = nullptr;

    方法节点类* 当前方法 = nullptr;
    场景节点类* 可执行输入参数场景 = nullptr;
    存在节点类* 当前方法运行存在 = nullptr;
    存在节点类* 运行回执存在 = nullptr;
};
```

迁移顺序：

```text
第一阶段先禁止工作线程提交任务状态。
第二阶段再把筹办命名空间的推进回执迁为通用任务层回执。
```

## 12. 验收规则

### 12.1 静态反查

工作线程中不得出现：

```powershell
rg -n "任务类::写入任务状态|任务类::标记任务待重筹办|任务类::写入任务当前阶段|任务类::同步任务运行镜像特征|任务类::应用任务状态|任务类::写入任务已折算" 任务模块.管理工作线程.ixx
```

预期：

```text
0 个命中。
```

界面线程允许出现：

```text
任务类::写入任务状态
任务类::写入任务当前阶段
任务类::应用任务状态
任务类::同步任务运行镜像特征
```

因为界面线程是任务域提交者。

### 12.2 行为验收

1. 自我线程发 `发起任务` 请求且不提供任务信息节点，界面线程创建任务信息节点和任务虚拟存在。
2. 同一需求重复发起任务请求时，幂等键阻止重复任务。
3. 任务状态为 `未启动` 或 `待重筹办` 时，工作包只筹办，不执行。
4. 任务状态为 `就绪` 或 `排队中` 时，工作包只执行，不筹办。
5. 工作线程产出工作结果后，任务状态只由界面线程提交一次。
6. 任务层字段只从运行存在读取；输出结果场景不承载任务状态。
7. Debug x64 构建通过。

## 13. 与其他计划的顺序

本详细设计承接：

```text
自我线程任务域写入收口
任务筹办本能方法
任务执行本能方法
```

必须先于：

```text
需求树真实生长剧本
内部循环闭合验收
自我苏醒验收
因果补完学习需求正式入树
```

原因：

```text
若工作线程和界面线程仍双写任务状态，后续需求树生长、任务执行和需求结算无法确认状态变化的权威来源。
```
