# 自我线程职责梳理

本文根据现有规范，把 `自我线程` 的主要职责重新收口成一份面向后续实现的职责说明。

主要依据：

- `规范/0010_自我实现逻辑与运行闭环规范.md`
- `规范/0013_自我线程运行时实现规范.md`
- `规范/0014_自我线程主循环最小职责表规范.md`
- `规范/0015_自我线程模块接口总表规范.md`
- `规范/0027_自我线程与任务管理工作线程状态流转与事件协议_v0.1规范.md`
- `规范/0037_自我线程消息后治理八步实现表_v0.1.md`
- `规范/0567_最终裁决层与外层重判边界规范.md`

## 1. 总定位

自我线程不是万能执行器，也不是叶子任务的执行者。

它的定位应固定为：

`面向需求信息的控制面治理中枢 + 主循环运行时承载器 + 任务管理工作线程调度者`

一句话收口：

`自我线程负责统一收件、统一重判、统一调度、统一派发、统一路由和统一结算。`

因此，自我线程直接治理的是本轮控制面，而不是任务步骤展开、方法本体执行或局部流程判断。

## 2. 直接职责

### 2.1 生命周期与运行边界

自我线程必须负责：

- 初始化自我运行承载，确保自我、根需求、根任务、任务管理任务、方法根等基础入口可用。
- 启动、停止、等待停止和故障收口。
- 维护线程生命周期状态、最近运行阶段、Tick 计数、故障摘要和恢复摘要。
- 在停止、收束、待机、故障等边界触发时形成可观测的阶段事件和日志。

它不负责：

- 宿主进程监督。
- WebView2、控制面板窗口或外设运行时。
- 反向控制宿主生命周期。

### 2.2 消息首收件人与轮次边界

自我线程必须作为以下消息的首收件人：

- 控制消息。
- 全局环境消息。
- 结果回流消息。
- 会改变控制面判断的关键推进事件。
- 恢复重建一次性事件。

一轮主循环必须遵守：

`冻结一批，清完这一批，再做一次决议。`

也就是说：

- 每轮先冻结本轮开始时已经可见的消息批次。
- 本轮处理冻结批次及其同步内部闭包。
- 本轮中新产生的异步消息进入下一轮。
- 高优先级控制消息可以按抢占规则打断当前轮。

自我线程不能为了清空活队列而在同一轮无限追消息。

### 2.3 一次事实落账与二次特征重算

自我线程收到消息后，不能直接拿消息字段做最终裁决。

它必须先完成：

- 将治理消息规范化。
- 把消息落成一次特征变化、状态变化、动态记录或最小事实账。
- 对本轮事实做归并、去重、对账和冲突标记。
- 基于一次事实重算受影响的二次特征。
- 把线程状态变化、结果变化、阻塞变化、派生需求等事实翻译成“对自我意味着什么”的判定材料。

当前代码中对应的运行骨架是：

`冻结本轮消息批次 -> 规范化消息并落一次特征账 -> 重算受影响二次特征`

### 2.4 需求治理与主需求选择

自我线程直接治理的核心对象是需求信息。

它必须负责：

- 刷新需求树和线程持有的需求列表。
- 按需求类型聚拢需求项。
- 维护需求类型组的安全权重、服务权重和汇总权重。
- 选择本轮主导需求和主导任务候选。
- 对消息驱动的新增需求、派生需求和需求状态变化做增量刷新。
- 在需求适合任务化但缺少任务头时，创建对应任务入口。

这里的重点是“需求治理”和“任务入口实例化”，不是直接展开任务步骤。

### 2.5 根层重判与最终裁决

自我线程是当前运行时里最接近最终裁决层的工程承载者。

它必须负责：

- 基于一次事实、二次特征、需求状态、当前值和门控边界发起根层重判。
- 形成本轮整体去向，例如继续提交、待机、收束、停止、回到重筹办、转入学习。
- 把承接层给出的候选去向升级或压回为外层最终裁决。
- 在每次承接层回流后，重新进入 `根层重判 -> 上下文刷新 -> 再提交`。

任务管理工作线程、学习线程、动作线程可以给出候选结果，但不能替代自我线程做最终裁决。

### 2.6 任务入口实例化与工作线程调度

自我线程必须负责：

- 根据主导需求确保对应任务头存在。
- 只信任属于当前自我需求树和任务树的当前主链节点。
- 设置或刷新当前主需求、当前主任务。
- 装配任务管理工作线程需要的最小上下文。
- 将当前主任务提交给任务管理工作线程推进。
- 管理可运行、等待、回流、收束停止等全局队列或其过渡形态。

规范边界是：

`自我线程决定让谁推进，工作线程负责推进一步。`

自我线程不能替工作线程展开步骤，也不能在同一轮连续推进多轮治理闭环。

### 2.7 结果回流、路由与结算

规范目标态中，自我线程必须统一接收：

- 子结果。
- 局部结果。
- 最终结果。
- 异常结果。
- 派生需求。
- 学习反馈。

它再决定：

- 结果是否影响全局优先级、运行队列或最终去向。
- 子结果是否需要路由回父层任务。
- 派生需求是否实例化为新需求或新任务。
- 本轮是否触发学习、否定项、恢复或故障收口。

当前代码已将旧的独立 `结果路由` 和 `本轮结算` 步骤物理移出主路径，结果吸收正在向“工作线程上报事实，自我线程吃事实”的最小主链收缩。

后续补回结果路由和结算时，应保持它们作为自我线程控制面职责，而不能把旧的大治理链原样搬回。

### 2.8 可观测性与快照

自我线程必须把关键控制面状态暴露给控制面板、摘要和日志，包括：

- 生命周期状态。
- 最近运行阶段。
- 最近最终去向。
- 最近运行摘要。
- 最近需求与权重治理输出。
- 最近主派发输出。
- 最近队列治理输出。
- 最近结算治理输出。
- 最近双值结算账。
- 最近学习恢复接口。
- 治理 mailbox 与冻结批次镜像。

这些快照是观察面，不应反向成为新的真值层。

## 3. 不应承担的职责

自我线程不应直接承担：

- 叶子步骤的具体执行。
- 任务步骤的局部展开。
- 父任务内部局部流程完成与否判断。
- 方法逻辑本体。
- 方法匹配、策略沙盘、方案演练这类任务管理内部策略子模块。
- 学习阶段内部每一步细节。
- 长时间全量扫描需求树或需求队列。
- 绕过工作线程直接改写任务运行态。
- 绕过外层重判直接连续推进多轮主闭环。
- 宿主进程监督、外设运行时、前端服务线程职责。

判断标准：

`凡是回答“现在全局准不准、派给谁、结果回哪里、最后怎么算”的，留在自我线程。`

`凡是回答“这一步内部怎么做、方法怎么执行、局部流程是否完成”的，下沉到任务管理工作线程、动作线程、学习线程或具体执行模块。`

## 4. 与任务管理工作线程的分工

自我线程负责：

- 拍板。
- 排队。
- 派发。
- 路由。
- 结算。
- 持有控制态真值。

任务管理工作线程负责：

- 在界面线程封装的任务工作包约束下推进一步任务。
- 更新局部任务运行态。
- 上报阻塞、解阻、局部失败、异常、派生需求和结果状态。
- 返回当前一步的推进事实。

必须固定：

- 需求根 / 任务根 / 方法根 是存在信息结构字段，默认可以为空；创建需求、任务或方法时由对应创建链按需确保，创建失败才是非正常逻辑错误。
- 任务工作线程不能创建或修补任务根，也不能创建任务虚拟存在；任务虚拟存在由任务建立 / 界面线程承接链路负责。
- 派生需求可以由工作线程提出，但只能由自我线程决定是否实例化。
- 子结果必须先回自我线程，再由自我线程路由给父层任务。
- 优先级、预算、运行许可和最终去向由自我线程维护。

## 5. 当前代码主链口径

截至当前代码，`自我线程模块.impl.cpp` 中主循环骨架已经收为：

```text
执行主循环一轮_
-> 步骤_冻结本轮消息批次_
-> 步骤_规范化消息并落一次特征账_
-> 步骤_重算受影响二次特征_
-> 步骤_刷新线程上下文并生成治理帧_
-> 步骤_整理需求并做根层重判_
-> 步骤_生成并执行主派发决议_
```

近期清理后，旧的以下步骤不再属于运行代码主路径：

- `步骤_生成主派发判定二次特征_`
- `步骤_路由结果并协调父任务_`
- `步骤_本轮结算学习触发故障收口_`

当前主派发最小职责已经收为：

- 读取当前主需求。
- 判断需求是否适合任务化。
- 缺任务时按需求补任务。
- 设置当前主任务。
- 生成线程裁决结果。
- 提交当前主任务给任务管理工作线程推进。
- 吸收工作线程返回的上行消息与推进事件。

这说明当前实现正在从规范完整八步，过渡为更窄的“消息判据 + 需求任务提交”主链。

## 6. 后续梳理方向

后续继续完善自我线程时，建议按以下顺序推进：

1. 先稳定 `消息 -> 一次事实 -> 二次特征 -> 需求刷新 -> 主派发` 这条当前主链。
2. 再把工作线程上报的 `线程状态消息 / 结果状态消息 / 派生需求消息` 完整接入自我线程事实吸收。
3. 结果路由和结算只补“新主链需要的最小版本”，不要恢复旧的大治理帧长链。
4. 将任务控制态、任务运行态、任务根语义继续分层，避免自我线程和工作线程混写。
5. 保持自我线程只做控制面，不再把方法选择、步骤展开、学习细节拉回一级主线。

最终验收口径：

`自我线程每轮能稳定冻结消息、吸收事实、刷新需求、选择主任务、提交工作线程、吃回结果事实，并把下一轮该怎么走收成一个可观测的最终去向。`
