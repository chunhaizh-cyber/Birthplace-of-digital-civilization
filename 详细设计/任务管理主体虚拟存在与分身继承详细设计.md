# 任务管理主体虚拟存在与分身继承详细设计

## 1. 目标

本文用于把“任务管理任务作为独立主体、可有分身、最终汇总回主体”正式落成结构层，回答以下问题：

- `任务管理主体` 与 `宿主任务虚拟存在`、`任务管理实例堆` 到底是什么关系
- `很多分身` 应如何解释，才不撞 `每宿主最多一个运行时实例`
- `成长需求 -> 方法需求位 -> 本能能力缺口 -> 最小原语` 这条桥应怎样落字段
- 后续 `P0` 的 `请求 / 决策 / 写回 / 返回` 应建立在什么新前提之上

本文是对 [`任务管理任务详细设计.md`](./任务管理任务详细设计.md) 的主体化补层。  
它不替代 `1219` 的三层运行时分工，而是补足“谁在成长”这条主线。

本设计主要对齐以下规范：

- [`../规范/1140_任务管理任务方法成长驱动特征规范.md`](../规范/1140_任务管理任务方法成长驱动特征规范.md)
- [`../规范/1190_任务管理方法因果生成规范.md`](../规范/1190_任务管理方法因果生成规范.md)
- [`../规范/1195_任务管理方法需求位与因果到方法映射规范.md`](../规范/1195_任务管理方法需求位与因果到方法映射规范.md)
- [`../规范/1196_任务管理方法需求位最小状态动态因果清单.md`](../规范/1196_任务管理方法需求位最小状态动态因果清单.md)
- [`../规范/1219_任务管理实例堆与恢复快照规范.md`](../规范/1219_任务管理实例堆与恢复快照规范.md)

## 3. 核心判断

必须先固定以下 6 条：

1. `任务管理任务` 自身也是任务，它有自己的成长轨迹，不应只作为“宿主治理镜像”的搬运层。
2. `任务管理主体虚拟存在` 不是 `宿主任务虚拟存在` 的替身，而是位于其上的跨轮成长汇总层。
3. `很多分身` 不是“多个运行时实例并发抢同一个宿主”，而是“同一主体在不同宿主、不同候选路径、不同学习回流链上的多个继承分身”。
4. `每个宿主任务` 仍然只允许 `最多一个任务管理运行时实例`，这一条不被主体化设计改写。
5. 成长需求不能直接等价成具体方法，必须先压缩成 `方法需求位`。
6. `任务管理方法因果` 必须晚于 `状态镜像完成 + 动作动态完成`，再由因果支持方法需求位下的方法出生、验证和升格。

一句话收口：

`旧三层回答“每宿主的一轮治理怎样跑”，新增主体层回答“跨多宿主、多路径、多回流之后，到底是谁在成长”。`

## 4. 与现有三层的关系

当前 `1219` 固定的三层仍然成立：

| 层 | 作用 |
| --- | --- |
| `实例堆` | 跑当前宿主的一轮运行时实例 |
| `宿主任务虚拟存在` | 记当前宿主的治理状态面 |
| `恢复快照` | 在退出或恢复时续当前宿主实例 |

本文新增的不是“第四个运行时实例层”，而是“主体成长补层”：

| 新层 | 作用 |
| --- | --- |
| `任务管理主体虚拟存在` | 汇总跨宿主、跨分身、跨学习回流的稳定成长特征 |
| `任务管理分身继承面` | 表达某条候选治理链本轮继承了什么、本地增量是什么、怎样回并 |
| `任务管理方法需求位特征面` | 表达当前缺的是哪类治理能力位 |
| `任务管理本能能力缺口面` | 表达命中的方法位还缺哪些最小动作原语 |

因此，后续结构关系应理解为：

```text
任务管理主体虚拟存在
  -> 挂多个分身继承面
      -> 每个分身对应某个宿主上的一次候选治理链
          -> 每个宿主仍只有一个运行时实例
              -> 运行时实例继续使用 实例堆 / 宿主任务虚拟存在 / 恢复快照
```

## 5. 四个新增结构面

## 5.1 `结构_任务管理主体虚拟存在`

定位：

- 任务管理主体自己的稳定特征承载层
- 负责长期方向、稳定模式、失败模式、完成模式、学习吸收与价值吸收

不是：

- 运行时实例对象
- 单宿主治理镜像
- 线程调度容器

建议字段如下：

| 字段 | 作用 |
| --- | --- |
| `主体ID` | 全局唯一主体标识 |
| `主体版本` | 主体结构版本或成长版本 |
| `总目标类型` | 默认固定为“推进任务完成” |
| `安全服务取向` | 安全优先、服务优先或平衡 |
| `预算策略` | 节省、平衡、积极等预算风格 |
| `风险偏好` | 保守、中性、激进 |
| `稳定推进次数` | 多轮中确认有效推进的累计次数 |
| `完成次数` | 宿主进入完成或终态的累计次数 |
| `失败次数` | 主体判定失败或宿主失败的累计次数 |
| `失败模式簇摘要` | 主失败模式的压缩摘要 |
| `方法位覆盖位图` | 七个方法需求位的覆盖情况 |
| `高价值证据次数` | 形成价值证据的累计次数 |
| `高置信轻量因果链次数` | 形成高置信轻量因果链的累计次数 |
| `当前活跃分身数` | 当前仍未回并的分身数量 |
| `最近高置信推进链摘要` | 最近一次强推进链摘要 |
| `最近收敛结论` | 最近一次主体层收敛判断 |
| `最近主失败原因` | 最近一次主失败原因 |
| `最近融合来源摘要` | 最近吸收了哪些宿主/方法/学习回流 |
| `最近汇总时间` | 最近一次主体汇总时间 |

这一层最关键的边界是：

- 它只记录“主体成长结果”
- 不直接记录某个宿主实例的锁、mailbox、等待队列、持有推进权

## 5.2 `结构_任务管理分身继承面`

定位：

- 某条候选推进链、候选宿主链、候选学习回流链的局部继承面

不是：

- 独立线程实例
- 独立宿主拥有者

建议字段如下：

| 字段 | 作用 |
| --- | --- |
| `分身ID` | 当前分身唯一标识 |
| `主体ID` | 归属哪个任务管理主体 |
| `宿主任务指针` | 当前继承面围绕哪个宿主 |
| `当前方法指针` | 当前分身命中的方法或候选方法 |
| `来源学习回流标识` | 是否来自某条学习回流链 |
| `继承宿主特征位图` | 从宿主任务虚拟存在带入了哪些特征 |
| `继承方法特征位图` | 从方法虚拟存在带入了哪些特征 |
| `本地二次特征摘要` | 本分身本轮压缩出的局势摘要 |
| `本地增量摘要` | 本轮新增了哪些推进/失败/收束增量 |
| `本地最近动作动态锚点` | 本分身最近动作动态 |
| `本地最近实例因果锚点` | 本分身最近实例因果 |
| `命中方法需求位` | 当前命中哪个能力位 |
| `回并序号` | 回并到主体时的排序序号 |
| `最后汇总时间` | 上次回并时间 |
| `汇总策略` | 覆盖、累加、弱覆盖、仅追加证据等 |
| `当前状态` | 活跃、待回并、已回并、已废弃 |

这里必须固定：

- 分身可以很多
- 但分身不等于运行时实例
- 每个分身只表达“主体在某条候选链上的局部继承与局部增量”

## 5.3 `结构_任务管理方法需求位特征面`

定位：

- 把“当前缺哪类治理能力”显式化
- 把成长需求压缩成稳定的能力位，而不是直接跳到具体方法枝

最小方法需求位继续沿用 `1195` 的七位：

1. `补结构方法位`
2. `选择推进方法位`
3. `分解补条件方法位`
4. `求证判定方法位`
5. `门控转域方法位`
6. `失败策略方法位`
7. `收束复盘方法位`

建议字段如下：

| 字段 | 作用 |
| --- | --- |
| `当前命中方法需求位` | 当前最需要哪类治理能力 |
| `来源功能域` | 这次命中来自哪个治理域 |
| `来源缺口类型` | 这次命中来自哪个主缺口 |
| `来源下一步去向` | 当前下一步去向如何压缩到了该能力位 |
| `输入条件主键` | 当前能力位的条件压缩锚点 |
| `结构完备度` | 当前结构是否足够承接该位 |
| `方法可用度` | 当前方法资源是否足够 |
| `风险门控` | 当前风险是否允许进入该位 |
| `状态底座位图` | 该位所需状态是否已齐 |
| `动态底座位图` | 该位所需动作动态是否已齐 |
| `实例因果位图` | 该位所需单跳实例因果是否已齐 |
| `轻量因果链位图` | 该位所需轻量因果链是否已齐 |
| `当前准备度` | 未就绪、可尝试、可验证、可稳定化 |
| `建议调用原函数` | 默认承接原函数，如 `管理_选择器` |
| `候选方法摘要` | 当前位下已有候选方法的压缩摘要 |
| `是否允许出生候选枝` | 当前是否允许在该位下出生候选方法 |
| `是否建议学习扩选` | 是否需要靠学习补能力覆盖 |

这一层的正式作用是：

- 把 `成长需求` 压缩成 `能力位`
- 再由 `能力位` 去看是否已有候选方法、是否要学习、是否缺最小原语

## 5.4 `结构_任务管理本能能力缺口面`

定位：

- 把“方法位命中了，但还缺哪些最小动作原语才能真正落地”显式化

不是：

- 高层方法名清单
- 根层目标集合

建议字段如下：

| 字段 | 作用 |
| --- | --- |
| `所属方法需求位` | 这份缺口面服务哪个方法需求位 |
| `所需最小动作原语集合` | 本位最少需要哪些动作原语 |
| `已具备原语` | 当前已经具备哪些最小动作 |
| `缺失原语` | 当前仍缺哪些原语 |
| `主观察特征` | 动作后要回读的主观察特征 |
| `写回目标` | 结果写回到主体、宿主、方法还是学习链 |
| `动作边界要求` | 是否必须有开始/结束双边界 |
| `动作后回读要求` | 是否必须回读前后状态再做派生 |
| `是否需学习补齐` | 当前缺口是否需要学习链补齐 |
| `学习优先级` | 若需学习，优先级多少 |
| `缺口摘要` | 对本位原语缺口的可读摘要 |

本层默认最小动作原语至少包括：

- `确保单承载存在`
- `读取单特征当前值`
- `写入单特征当前值`
- `写入单挂点指针`
- `记录动作开始`
- `记录动作结束`
- `动作后回读主观察特征并派生状态变化/实例因果`

这意味着：

- 高层方法位不直接等于最小动作
- 但高层方法位必须最终能追溯回这些最小原语

## 6. 主体化后的新前半链

在引入主体层后，`执行一步治理` 的前半段不应再只是“读宿主上下文 -> 判域 -> 写回”，而应先变成：

```text
受理请求
-> 确保任务管理主体虚拟存在存在
-> 确保当前分身存在
-> 从宿主任务虚拟存在 / 方法虚拟存在继承特征
-> 刷新主体状态面
-> 记录主体动作动态
-> 生成主体实例因果
-> 命中当前方法需求位
-> 展开本能能力缺口
-> 再进入单步决策 / 树写回 / 一步结果
```

其中必须继续固定：

- `主体实例因果` 的生成晚于 `状态镜像完成 + 动作动态完成`
- `方法需求位` 的命中晚于主体状态和分身继承完成
- `本能能力缺口` 的展开晚于方法需求位命中

## 7. 对现有头文件的落点建议

如果后续开始落代码，第一落点建议仍然是 [`../任务管理任务模块.h`](../任务管理任务模块.h)。

建议先只补结构壳，不立即大改实现：

```cpp
struct 结构_任务管理主体虚拟存在;
struct 结构_任务管理分身继承面;
struct 结构_任务管理方法需求位特征面;
struct 结构_任务管理本能能力缺口面;
```

同时继续保留当前 `结构_任务管理上下文` 和 `结构_任务管理结果` 作为过渡载体。  
不要在主体层刚引入时就急着删除旧结构。

## 8. 与 P0/P1/P2 的关系

引入主体层后，后续顺序应固定为：

### P-1 主体化

- 先定义主体虚拟存在
- 再定义分身继承面
- 再定义方法需求位特征面
- 再定义本能能力缺口面

### P0 解耦化

- 再拆 `请求 / 单步决策 / 写回结果 / 一步结果`
- 但这次由主体层和方法需求位驱动，而不是只由宿主镜像驱动

### P1 学习化

- 让学习承接变成“命中了方法位，但本能能力仍不足”的有因学习

### P2 运行时化

- 最后再补 mailbox、等待态、恢复快照和实例重建

## 9. 最终结论

如果不先引入 `任务管理主体虚拟存在`，后续即使把 `执行一步治理` 切得更整齐，也仍然会默认：

- 成长只属于某个宿主的最近治理镜像
- 方法需求仍然只是流程中间态
- 学习承接仍然更像尾部补丁

而主体层一旦立住，后面的 `请求 / 决策 / 写回 / 返回` 才真正有了新的脊梁。

一句话收口：

`下一步应先把“谁在成长、怎样分身、缺哪类治理能力、缺哪些最小原语”写成正式结构，再去拆单步治理流程。`
