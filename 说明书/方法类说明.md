# 方法类说明

更新时间：2026-05-07

## 1. 当前定位

`方法类` 是方法树结构管理器，负责维护方法首节点、条件节点和结果节点之间的树关系，并提供方法注册、条件/结果子节点创建、本能方法补齐和能力签名查找能力。

当前方法查找口径已经固定：

```text
方法不是因为它叫什么被找到，
而是因为它声明的结果能力能覆盖需求关心的目标特征类型和目标方向。
```

动作名、动作句柄和本能 ID 只用于注册、执行和调试，不作为需求结果查找的命中依据。

## 2. 主信息结构

方法节点主信息使用：

```cpp
class 方法主信息类 {
public:
    结构_方法节点公共信息 公共{};
    方法角色信息 角色{};
};
```

`方法角色信息` 是唯一角色承载：

```cpp
using 方法角色信息 = std::variant<
    std::monostate,
    结构_方法首节点信息,
    结构_方法条件节点信息,
    结构_方法结果节点信息>;
```

旧的 `方法信息基类` 和三种派生提示类已经删除。不要再按旧“大字段基类”理解方法节点。

## 3. 三种节点角色

| 节点角色 | 职责 |
| --- | --- |
| 方法首节点 | 动作骨架、方法虚拟存在、来源/作用域、来源任务、能力签名 |
| 方法条件节点 | 条件场景、条件主键、条件面、条件判定索引、时间窗口 |
| 方法结果节点 | 结果场景、结果主键、结果包、结果初始状态 |

角色边界是硬约束：

```text
首节点不承载条件场景。
首节点不承载结果场景。
条件节点不承载动作句柄。
结果节点不承载动作句柄。
结果初始状态只属于结果节点。
```

写入路径必须先取得对应角色指针：

```cpp
取首节点信息()
取条件节点信息()
取结果节点信息()
```

不要把 `首节点信息()`、`条件节点信息()`、`结果节点信息()` 当作跨角色通用写入口。

## 4. 方法能力结构

当前权威能力表达是：

```text
方法能力签名 = 条件面 -> 结果包
```

对应结构：

```cpp
struct 结构_方法能力签名 {
    std::string 能力键{};
    结构_方法条件面 条件面{};
    结构_方法结果包 结果包{};
    I64 成熟度阶段 = 0;
    bool 可被方法查找命中 = true;
    bool 是否根写入原语 = false;
};
```

旧 `结果能力集` 已删除。方法能力的权威来源只有能力签名；方法查找必须读取 `能力签名.结果包`，不能回退到动作名、动作句柄、本能 ID 或旧索引。

## 5. 条件集合

方法条件不是单个特征，而是条件面：

```text
条件面
└── 可用条件组集
    ├── 条件组1：条件A 且 条件B
    └── 条件组2：条件C 且 条件D
```

组内条件全部满足，组间任意一组满足即可。条件节点可继续保留 `条件判定索引` 作为二次特征过渡索引。

## 6. 结果集合

方法结果不是单结果槽，而是结果包：

```text
结果包
├── 结果项1
├── 结果项2
└── 结果项3
```

例如 `练习` 的能力签名结果包包含：

```text
方法稳定状态 更稳定
方法运行账完整性 更完整
方法是否可用 趋向可用
方法成熟度阶段 上升
方法新用途发现状态 发现候选
```

`尝试学习` 的结果包包含：

```text
候选方法存在状态 从无到有
方法动作骨架状态 已形成
方法条件结果配对状态 已配对
方法结构是否完整 更完整
```

`定向学习` 的结果包保留学习方向、目标特征类型识别和目标变化方向识别，但它属于外界信息或未归因现象处理链，不属于自我内部方法补齐闭环。

## 7. 方法查找规则

方法查找只回答：

```text
有没有方法能把需求关心的特征推向目标方向？
```

单目标需求可以命中某个结果项。多目标需求必须由同一个能力签名的同一个结果包覆盖全部必需结果，单次查找不偷偷拼多个方法。

```text
需求结果集 ⊆ 方法能力签名.结果包
```

找不到就是找不到，含义是当前方法树中没有任何方法声明自己能造成这个结果变化。不能改按名称、动作句柄或本能 ID 取方法。

候选排序不再使用旧评分率作为主排序。结果能力命中后，优先使用方法树事实样本数、匹配距离、能力包宽度和运行账。结果符合/不符合账只在以下口径下更新：

```text
方法条件满足
参数配对成功
动作函数执行成功
实际结果状态有效且可比较
同条件下存在可比较的已有结果分支
```

同条件下复现已有等价结果时记录结果符合次数；同条件下出现新的非等价结果分支时记录结果不符合次数。条件不满足、参数失败、动作失败、中断、超时、未执行、没有可比较结果，都只能进入查找诊断、参数配对记录、运行账或任务执行记录，不能进入结果符合/不符合账。

执行成功后不会直接改方法树统计，而是先形成条件结果对样本。样本进入条件结果对归并：

```text
添加：没有等价条件节点或结果节点时新增结构。
融合：已有等价条件节点或结果节点时更新统计和证据。
```

归并只沉淀实际条件和实际结果，更新样本次数和最近样本时间；结果符合/不符合来自同条件复现/分裂，需求是否满足由自我线程按需求目标另行判断。

## 8. 本能方法补齐

本能方法补齐流程：

```text
创建或复用方法首节点
写入动作名、动作句柄、来源和作用域
写入能力签名
创建条件子节点
创建结果子节点
初始化方法虚拟存在特征
```

补齐时不把条件/结果字段写回首节点。条件场景归条件节点，结果场景和结果初始状态归结果节点。

## 9. 内部方法补齐闭环

自我内部方法补齐闭环为：

```text
方法需要可用
    -> 查找能提升方法可用性的能力
    -> 练习
    -> 若没有候选方法，生成候选方法存在需求
    -> 查找能形成候选方法的能力
    -> 尝试学习
    -> 从内部因果样本形成候选方法
    -> 练习验证
    -> 方法可用
```

内部因果样本已经包含动作主体、动作函数、动作前状态和动作后状态，因此尝试学习不应再派生定向学习。定向学习只服务于外界信息上下文或动作主体/动作函数未知的未归因现象。

## 10. 根写入原语

`改变特征值` 是根写入原语，可以执行任意特征类型的值改变，但当前阶段不作为普通方法查找命中。

它的能力签名可以记录根能力性质，并必须标记：

```cpp
是否根写入原语 = true
可被方法查找命中 = false
```

这样系统不会用根写入原语绕过学习和练习闭环。

## 11. 当前验收口径

方法树健康摘要应满足：

```text
结构缺口 = 0
能力包缺失 = 0
条件结果对缺失 = 0
```

代码扫描中不应再出现对方法首节点读写条件/结果角色的业务路径。条件和结果角色访问必须发生在已确认节点角色后。
