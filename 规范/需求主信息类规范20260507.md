# 需求主信息类规范 20260507

## 1. 总原则

需求主信息只是一张目标关系索引卡，不是值类型仓库。

当前模型固定为：

```text
需求：一个目标特征类型
任务：一个目标特征类型
方法：可以产生多个结果特征类型变化
```

因此需求主信息只表达一个目标特征类型。多目标需求必须通过管理需求的直接子链展开，不在需求主信息中保存目标包。

需求主信息只保存：

```text
谁提出需求
要求哪个对象的哪个状态满足目标状态
满足关系是什么
由哪个任务承接
权重与结算账本
派生来源记录
```

需求主信息不保存：

```text
I64 / VecIU64 / 枚举 / 布尔 / 指针等目标值类型
集合比较规则
区间比较规则
枚举解释规则
二次特征解释规则
复杂目标值包
```

具体怎么比较，由特征类或状态比较服务完成。需求类只接收比较结果。

## 2. 三向关系

需求方向只保留三向关系：

```cpp
enum class 枚举_三向关系 : std::int8_t {
    小于 = -1,
    等于 = 0,
    大于 = 1,
};
```

组合关系用掩码表达：

```cpp
using 三向关系掩码 = std::uint8_t;

inline constexpr 三向关系掩码 关系_小于 = 0b001;
inline constexpr 三向关系掩码 关系_等于 = 0b010;
inline constexpr 三向关系掩码 关系_大于 = 0b100;

inline constexpr 三向关系掩码 关系_保持 = 关系_等于;
inline constexpr 三向关系掩码 关系_改变 = 关系_小于 | 关系_大于;
inline constexpr 三向关系掩码 关系_大于等于 = 关系_大于 | 关系_等于;
inline constexpr 三向关系掩码 关系_小于等于 = 关系_小于 | 关系_等于;
```

概念固定如下：

```text
保持 = 等于
改变 = 小于 或 大于
存在 / 可用 / 完整 / 已形成 = 特征值或状态，不是方向
区间 = 特征类解释的状态比较兼容范围，不是需求字段
```

## 3. 原子需求节点

单个执行需求节点只表达一个原子目标：

```text
目标主体
目标特征类型
当前状态
目标状态
满足关系
```

当前 `需求主信息类` 中的：

```text
被需求存在
被需求当前状态
需求状态
```

就是执行需求的核心目标关系。

一个任务只承接一个需求，所以任务也只能有一个目标特征类型。任务目标可以缓存，但权威来源仍然是对应需求的单目标关系。

后续可以在目标面中补充：

```cpp
三向关系掩码 满足关系 = 关系_等于;
const 词性节点类* 目标特征类型缓存 = nullptr;
```

`目标特征类型缓存` 只用于去重和快速查找，不是权威语义源。权威来源仍然是状态节点及其状态特征。

## 4. 管理需求与执行需求

需求节点分为两类：

```text
管理需求：
    组织子需求
    分配权重
    等待子需求满足
    汇总子需求状态

执行需求：
    推动某个对象的某个状态满足目标状态
```

多目标需求必须表达为：

```text
父管理需求
├── 原子执行需求 1
├── 原子执行需求 2
└── 原子执行需求 3
```

不得表达为：

```text
一个需求节点
└── 持久化目标包
```

## 5. 子链就是目标包

同一父需求的直接子链，就是该父需求的目标展开集合。

因此：

```text
目标包不是持久化结构。
目标包最多是运行时视图。
```

允许存在：

```text
候选需求展开包：创建需求前的临时输入。
直接子需求目标视图：任务筹办时临时收集父需求子链。
方法查找输入包：由需求子链临时生成。
```

运行时视图只用于调度和展开；单次方法查找应落到某个具体执行需求的单目标，不把管理需求直接当成多目标查找包。

不允许存在：

```text
需求主信息类.目标包
需求主信息类.条件包
```

原因是它会和需求树父子结构重复，导致权威来源不清。

## 6. 子链去重

需求去重不看派生归因，不看缺口签名作为身份。

同父直接子链去重依据是：

```text
目标主体
目标特征类型
方向 / 满足关系
```

目标键可表达为：

```cpp
struct 结构_需求目标键 {
    std::string 目标主体主键{};
    std::string 目标特征类型主键{};
    三向关系掩码 方向掩码 = 0;
};
```

重叠判断：

```cpp
bool 需求目标重叠(
    const 结构_需求目标键& a,
    const 结构_需求目标键& b
) noexcept {
    if (a.目标主体主键 != b.目标主体主键) {
        return false;
    }
    if (a.目标特征类型主键 != b.目标特征类型主键) {
        return false;
    }
    return (a.方向掩码 & b.方向掩码) != 0;
}
```

如果当前阶段还无法稳定解析目标主体，可以先用：

```text
目标特征类型 + 方向掩码
```

作为过渡键，但长期应补上目标主体。

## 7. 派生归因

派生归因只用于：

```text
记录来源
自检归因
控制面板观察
重复触发统计
```

派生归因不用于：

```text
决定需求身份
决定同父子需求去重
替代目标特征类型和方向
```

现有注释应统一为：

```cpp
// 派生需求归因：用于记录来源、自检归因、控制面板观察和重复触发统计。
// 需求去重不使用派生归因。
// 需求去重以同父子链上的目标特征类型和方向重叠为准。
```

## 8. 特征比较服务

需求类不比较值，只调用特征类或状态比较服务。

比较结果：

```cpp
struct 结构_特征比较结果 {
    bool 可比较 = false;
    枚举_三向关系 关系 = 枚举_三向关系::等于;
    std::string 说明{};
};
```

接口：

```cpp
struct 特征比较服务 {
    static 结构_特征比较结果 比较状态(
        const 状态节点类* 当前状态,
        const 状态节点类* 目标状态
    ) noexcept;
};
```

需求满足判断只做关系掩码接受：

```cpp
bool 三向关系被接受(
    枚举_三向关系 关系,
    三向关系掩码 掩码
) noexcept;
```

## 9. 方法补齐需求表达

方法补齐入口是管理需求，不是一个带目标包的执行需求。

例如：

```text
方法补齐管理需求
├── 候选方法存在状态 == 存在
├── 方法动作骨架状态 == 已形成
├── 方法条件结果配对状态 == 已配对
├── 方法运行账完整性 == 目标完整性
└── 方法是否可用 == 目标可用状态
```

每个子需求都是一个原子执行需求。父需求只负责组织、等待和汇总。

这会自然接到：

```text
尝试学习 -> 形成候选方法
练习 -> 验证候选方法、补运行账并推动方法可用状态
```

## 10. 一句话定型

需求树的子链本身就是包，没必要再在需求节点里藏一个包。

需求主信息只保存目标关系，不保存值解释；值解释全部下放到特征类，需求类只接收三向比较结果。

任务只认对应需求的一个目标；方法可以产生多个实际结果，但只有匹配当前任务目标的结果可以进入当前任务满足和结算。
