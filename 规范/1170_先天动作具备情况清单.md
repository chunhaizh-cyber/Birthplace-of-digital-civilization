# 1170 先天动作具备情况清单

## 1. 目标

本文用于回答一个具体问题：

- 当前系统里的 `先天动作` 是否已经具备
- 哪些已经按规范落成
- 哪些只是语义存在、尚未完全收口
- 哪些后续必须继续补齐

本文不重新定义 `先天动作`，而是在以下规范基础上做一次现状清点：

- [1130_先天记账动作与被动动作动态规范.md](D:\鱼巢\规范\1130_先天记账动作与被动动作动态规范.md)
- [0013_自我线程运行时实现规范.md](D:\鱼巢\规范\0013_自我线程运行时实现规范.md)


## 2. 判定标准

按 `1120` 与 `1130`，一个动作要被判定为“先天动作已经具备”，至少应同时满足：

1. 只能由运行时被动触发，不能进入普通任务/方法的主动调用链。
2. 由 `自我线程` 每个周期自动调用一次，或已经明确接入等价周期调用入口。
3. 有真实执行入口，而不只是枚举名或语义预留。
4. 运行期会留下明确的 `动作动态信息`。
5. 每次调用都能生成一条对应的本能动作动态。
6. 该动作动态能够回带主结果状态，或至少能追溯到本次被动写回结果。

如果只满足其中一部分，则统一分为：

- `已具备`：六条都成立，或已达到运行时可闭环验收的程度。
- `部分具备`：真实动作已经发生，但还未完全落成显式动作动态，或调用边界还没收口。
- `未具备`：当前只有预留名字、历史语义，或旧链路已移除。


## 3. 现状总判断

当前结论是：

- `先天维持动作`：基础骨架已经具备，但按新双值时序规则核对后仍缺多个关键动作。
- `先天记账动作`：大量已经发生真实记账，但还没有全部升格成显式被动动作动态。
- 因此整体不能判定为 “先天动作已经全部具备”。

更准确地说：

- `维持类` 已基本落地。
- 但旧版只把 `安全时序回归` 当成一个粗粒度总称；按当前 `0311` 新规则重新核对后，必须拆成多条更细的内部先天动作。
- `记账类` 只完成了一部分。
- `边界收口` 还差最后一步。

## 7. 优先补齐顺序

建议后续按以下顺序继续收口：

1. 补上 `按双值重算服务衰减步长` 的真实运行时入口与动作动态。
2. 补 `按服务值重算安全增加步长 / 按服务值重算安全减少步长` 的独立内部先天动作动态。
3. 补 `安全低位回升 / 安全高位回落` 的真实运行时入口与动作动态。
4. 再补 `方法最近成功标记写入 / 最近耗时写入 / 平均耗时EMA更新`。
5. 再补 `学习任务来源优先级继承 / 显式学习请求写入 / 显式学习请求清除`。
6. 最后补齐条件桶和结果桶的被动动作动态。
