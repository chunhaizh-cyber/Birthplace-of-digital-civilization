# 1190 任务管理方法因果生成规范

## 1. 目标

本文用于在 `1285` 的上位因果约束下，专门固定 `任务管理任务` 的因果生成边界，回答以下问题：

- `任务管理方法` 的实例因果应在什么时机生成
- 单跳实例因果应取哪些 `因 / 动作 / 果`
- `轻量因果链` 应如何组织，不能如何组织
- 它们与 `根层裁决 / 任务管理上下文 / 单步治理返回` 的顺序关系是什么
- 这层规范与 `1150 / 1160 / 1290` 的关系是什么

本文要解决的是“任务管理方法专用因果怎么生成”，不是重新定义全域因果体系。

至于“这些因果最终如何支撑某类任务管理方法位出生、验证和升格”，以 [1195_任务管理方法需求位与因果到方法映射规范.md](D:\鱼巢\规范\1195_任务管理方法需求位与因果到方法映射规范.md) 为准。

如果要先单独查看“当前到底需要哪些因果信息”，以 [1197_任务管理方法所需因果信息清单.md](D:\鱼巢\规范\1197_任务管理方法所需因果信息清单.md) 为准。

如果要查看“这些因果应按什么模式生成、怎样从临时走向稳定”，以 [1198_任务管理方法因果生成模式与稳定化规范.md](D:\鱼巢\规范\1198_任务管理方法因果生成模式与稳定化规范.md) 为准。

因此必须固定：

## 3. 基本判断

必须先固定以下 10 条：

1. `任务管理方法因果` 属于 `实例因果信息 + 轻量因果链` 的第一阶段落地对象，不属于抽象因果模板。
2. 单跳实例因果仍必须满足 `同锚点 + 同层级 + 时间有序`。
3. `任务管理方法` 的单跳因果，默认优先生成 `存在因果`，锚点优先取 `宿主任务虚拟存在`。
4. 如果条件状态和结果状态全部属于同一公开场景层，允许额外生成对应 `场景因果`，但不得把场景态和存在态混写成一条单跳因果。
5. `任务管理方法` 的单跳因果，只描述 `治理层内部` 的条件、动作和结果，不得把后续叶子执行、结果节点成立、根值结算硬压成同一跳。
6. 跨层解释必须走 `轻量因果链`。
7. 因果生成必须晚于 `状态镜像完成 + 动作动态完成`，不得在半成品状态上抢跑。
8. `二次特征` 负责压缩局势，`实例因果` 负责记录这次为什么有效，二者不能互相替代。
9. 若因果动作证据来自 `自我的动作动态`，则必须继续服从 `内部先天动作 > 自我内部动作 > 外部动作` 的可信度分层。
10. 轻量因果链整体可信度不得高于其中最弱的一跳。
11. 普通治理方法因果的生成，必须建立在 `根层已允许继续推进`、且任务管理上下文已形成的前提下。
12. 因果生成后的结果，只能作为治理证据和成长证据，不替代外层后续重判。

### 3.1 在统一主链中的位置

按当前最新体系，`任务管理方法因果` 必须处在下面这条链中：

```text
根层方向裁决
-> 形成任务管理上下文
-> 提交一轮治理请求
-> 单步治理执行并完成状态/动态写回
-> 生成实例因果
-> 组织轻量因果链
-> 返回治理结果与成长证据
-> 外层重判
```

必须固定：

1. `实例因果` 生成晚于本轮状态镜像和动作动态完成。
2. `轻量因果链` 组织晚于实例因果生成。
3. 因果层不反向替代根层裁决，也不替代任务调度结论。


## 4. 生成前提

`任务管理方法` 的专用因果生成，只能在以下前提满足后进行：

### 4.1 已准备好的条件输入

至少应已经具备：

- `根层允许继续推进`
- 一级治理输入主字段
- `输入条件主键`
- 二次特征状态
- 当前功能域

其中：

- `根层允许继续推进` 用于保证本轮仍属于普通治理路径，而不是纯收束或停机边界处理
- 一级治理输入用于补充“这次为什么进入该治理域”
- `输入条件主键` 用于做条件压缩锚点
- 二次特征用于做可复用局势标签

### 4.2 已准备好的动作信息

至少应已经具备：

- 当前管理方法的 `开始运行` 动态
- 当前管理方法的 `完成` 或 `失败` 动态

并且必须固定：

- 单跳因果中的 `动作`，优先取显式 `方法调用动态`
- 不得在已有显式动作动态时，再拿某个结果状态名回退冒充动作名
- 若动作来自 `内部先天动作动态`，则动作事实无需再次校验真实性
- 若动作来自 `外部动作动态`，则外部结果仍必须依赖结果状态或反馈确认

### 4.3 已准备好的结果状态

至少应已经具备以下三组中的一组：

1. `管理派生产出状态`
2. `任务结构推进状态`
3. `宿主任务状态变化`

推荐最小结果面为：

- `管理任务_条件缺口分类`
- `管理任务_可执行性判断`
- `管理任务_推荐分解方向`
- `管理任务_是否需要补条件任务`
- `管理任务_是否建议等待学习`
- `管理任务_结果节点主键`
- `任务_当前步骤`
- `任务_当前方法`
- `任务_当前子任务`
- `任务_状态`

### 4.4 可选辅助证据

以下信息不是单跳合法性的硬前提，但应作为可挂接辅助证据保留：

- `管理任务_失败原因码`
- `管理任务_二次特征证据`
- 成长证据四元组
- `方法尝试次数 / 方法稳定出现次数 / 方法可靠度_Q10000`


## 5. 单跳实例因果规范

## 5.1 默认锚点

`任务管理方法` 的单跳实例因果，默认锚点固定为：

- `宿主任务虚拟存在`

理由：

- 当前治理输入、二次特征、管理派生产出、结构推进状态大多落在 `任务虚拟存在`
- 这样更容易满足 `同锚点 + 同层级`
- 也更符合“管理方法在治理宿主任务自身”的语义

### 5.2 标准表达

`任务管理方法` 的单跳实例因果，标准表达固定为：

```text
因：
  一级治理输入裁剪集
  + 二次特征
  + 输入条件主键

动作：
  当前管理方法调用动态

果：
  管理派生产出
  + 任务结构推进变化
  + 宿主任务状态变化
```

其中：

- `因` 不是要求把所有一级输入全量塞进一条因果
- `因` 应优先保留对本次治理判定真正有贡献的最小裁剪集
- `果` 不是要求三组结果都必须同时变化
- 只要存在真实结果变化，就允许生成单跳因果

### 5.3 最小条件裁剪规则

当前建议固定以下最小条件优先级：

1. `当前功能域`
2. `输入条件主键`
3. `治理态型`
4. `结构完备度`
5. `方法可用度`
6. `风险门控`
7. 直接触发本次治理分支的一级门控字段

第 7 项常见候选包括：

- `依赖缺失`
- `需要新子任务或新步骤`
- `关键方法不再可用`
- `已满足完成判据`
- `已确定无法继续`
- `服务收益不足以继续`
- `安全要求必须停止`

必须固定：

- `输入条件主键` 是条件压缩锚点，不是条件全集的替代品
- `二次特征名称词指针` 只是显示名，不得替代对应的二次特征值

### 5.4 动作选择规则

动作选择按以下优先级固定：

1. `当前管理方法完成动态`
2. `当前管理方法失败动态`
3. 与本次方法调用同批次的显式方法动作事件

必须固定：

- 若已存在 `完成 / 失败` 边界动态，应优先取结束边界作为因果动作主事件
- `开始运行` 动态应保留在过程动态列表中，不应替代主动作事件

### 5.5 结果选择规则

结果选择按以下优先级固定：

1. `宿主任务状态变化`
2. `任务结构推进变化`
3. `管理派生产出状态`
4. `管理任务_结果节点主键`
5. `管理任务_失败原因码`

必须固定：

1. 若 `完成 / 失败` 动态本身挂有明确主结果状态，优先使用该主结果状态。
2. 若 `宿主任务状态` 发生了真实变化，应优先把它作为主结果。
3. `结果主键` 和 `失败原因码` 默认是摘要证据，不应在有更具体结果时抢占主结果。
4. `管理任务_结果节点主键` 在本规范中只承担 `结果回指字段 / 链桥接辅助字段 / 候选聚类字段` 职责，不得直接冒充“任务管理方法本轮真实主结果”。

### 5.6 不得生成单跳因果的情况

遇到以下情况，必须禁止生成单跳实例因果：

- 条件状态与结果状态不在同一锚点层
- 只有 `输入条件主键哈希`，却没有任何可校验条件状态
- 只有 `结果摘要哈希`，却没有任何可校验结果状态
- 没有显式动作动态
- 结果仍处于中间态，尚未完成最终镜像写回


## 6. 过程动态与主结果规则

`任务管理方法` 的因果提炼，必须继续服从 `1285` 的事件顺序：

1. `方法开始运行`
2. `中间状态变化或派生产出写回`
3. `方法完成或失败`

因此：

- `开始运行` 动态应进入过程动态列表
- 中间写回产生的关键治理动态也应尽量进入过程动态列表
- `完成 / 失败` 动态优先作为因果主动作事件
- 主结果状态优先从结束边界动态回带的结果状态里选择

不得记录成：

1. 只留下某个结果状态
2. 再回退解释“这个状态就是动作”


## 7. 生成时机与写回顺序

`任务管理方法` 的专用因果生成，推荐固定为以下顺序：

1. 外层完成根层裁决并装配任务管理上下文
2. 生成并镜像一级治理输入、二次特征、管理派生产出、结构推进状态
3. 记录当前管理方法 `开始运行` 动态
4. 执行管理方法
5. 记录中间治理动态与结果状态
6. 记录当前管理方法 `完成 / 失败` 动态
7. 生成单跳实例因果
8. 组织轻量因果链
9. 再做后续查询、学习、模板沉淀或升格判定
10. 把结果交回外层继续重判和调度

必须固定：

- `实例因果` 生成必须晚于第 5 步
- `轻量因果链` 组织必须晚于第 6 步
- `抽象因果模板` 必须继续后置，不得在本阶段抢跑


## 8. 轻量因果链规范

## 8.1 作用

`任务管理方法` 的轻量因果链，当前只负责：

- 把多条合法单跳因果按桥接关系串起来
- 提供“这条治理为什么最后产生了结构推进或价值结果”的多跳解释

它不负责：

- 直接替代单跳实例因果
- 直接替代学习线程或方法升格算法
- 直接生成抽象因果模板

## 8.2 最小桥接键

当前建议固定以下最小桥接键：

- `宿主任务`
- `当前功能域`
- `输入条件主键`
- `当前管理方法首节点`
- `管理任务_结果节点主键`

必须补充理解：

- 这里的 `管理任务_结果节点主键` 只用于把相近治理结果挂到同类桥接面上。
- 它只回答“这几条链在摘要层是否相近”，不回答“这次治理是否已经成立”。
- 一旦存在更具体的 `结果状态列表 / 宿主状态变化 / 结构推进变化 / 二次特征结果`，应优先使用这些更具体结果。

若已补齐原文，可优先使用：

- `输入条件主键原文`

### 8.3 最小链形态

当前最小应先组织出一条 `治理链`：

```text
输入条件
-> 命中管理方法
-> 管理产出
-> 任务结构推进
```

当后续原料可用时，再允许延长成：

```text
输入条件
-> 命中管理方法
-> 管理产出
-> 任务结构推进
-> 叶子任务执行
-> 结果节点成立
-> 根值或价值结算
```

### 8.4 链组织约束

必须固定：

1. 因果链允许跨层，但每一跳都必须先是合法单跳因果。
2. 不得把“尚未生成单跳实例”的内容直接塞进因果链。
3. 不得用一条长字符串摘要替代链中的桥接关系。
4. 不得用 `二次特征证据` 或 `结果主键` 直接冒充整条因果链。
5. 不得把“外部动作已发出”直接桥接成“外部结果已确认”的强因果链。


## 9. 最小接口建议

为了便于后续代码落地，建议先固定以下最小接口形态：

```cpp
struct 结构_任务管理方法因果生成输入_v0 {
    任务节点类* 宿主任务 = nullptr;
    基础信息节点类* 宿主任务虚拟存在 = nullptr;
    方法节点类* 当前管理方法首节点 = nullptr;
    I64 当前功能域 = 0;
    std::string 输入条件主键原文{};
    I64 输入条件主键哈希 = 0;
    std::vector<状态节点主信息类> 条件状态列表{};
    std::vector<运行时动态主信息记录*> 过程动态列表{};
    运行时动态主信息记录* 主动作动态 = nullptr;
    std::vector<状态节点主信息类> 结果状态列表{};
    I64 结果主键 = 0; // 仅作结果回指与链桥接字段，不得替代真实结果状态
    I64 失败原因码 = 0;
    I64 二次特征证据 = 0;
};

基础信息节点类* 生成任务管理方法实例因果(
    const 结构_任务管理方法因果生成输入_v0& 输入);

基础信息节点类* 组织任务管理方法轻量因果链(
    const std::vector<基础信息节点类*>& 输入因果集,
    任务节点类* 宿主任务,
    方法节点类* 当前管理方法首节点);
```

当前阶段必须固定：

- 先实现 `实例因果`
- 再实现 `轻量因果链`
- 不要先做任务管理专用 `抽象因果模板`


## 10. 禁止事项

当前阶段必须禁止：

- 把 `任务管理方法` 的治理层结果和叶子执行结果混写成一跳
- 把 `二次特征名称` 当成因果条件真值
- 只用哈希值，不保留可校验状态列表
- 只用动作名文本，不挂真实动作动态
- 让 `失败原因码` 或 `结果主键` 在有真实结果状态时冒充主结果
- 让 `结果主键命中` 直接冒充“这条治理链已经成立”
- 在状态写回未完成时抢先生成因果
- 跳过 `实例因果`，直接做模板沉淀


## 11. 最小落地顺序

建议后续按以下顺序落地：

1. 先生成 `宿主任务虚拟存在` 锚点下的任务管理方法单跳实例因果。
2. 再用 `宿主任务 + 输入条件主键 + 当前管理方法首节点` 组织最小治理链。
3. 然后再补 `输入条件主键原文` 的任务侧镜像，提升链的可解释性。
4. 最后视需要延长到 `叶子执行 -> 结果节点 -> 价值结算` 的长链。


## 12. 一句话收口

可收敛为一句话：

`任务管理方法的因果生成，应先在宿主任务虚拟存在上，以“一级治理输入裁剪集 + 二次特征 + 输入条件主键 -> 当前管理方法调用动态 -> 管理产出/结构推进/宿主状态变化”的单跳实例因果为核心，再按桥接键组织轻量因果链，严禁跨层偷写成单跳总因果。`
