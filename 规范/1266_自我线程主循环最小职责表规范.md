# 1266 自我线程主循环最小职责表规范

## 1. 目标

本文用于固定以下问题：

- `自我线程主循环` 最少必须承担哪些职责
- 哪些职责必须留在主循环，不能下沉给动作线程、学习线程或叶子执行层
- `需求 / 任务 / 方法 / 学习 / 任务管理任务` 的最小治理闭环，在主循环层应如何串起来
- 主循环与 `根层重判`、`任务管理上下文`、`消息回流`、`双值结算` 的关系是什么

本文只收口 `自我线程主循环` 的最小职责表，不展开线程池、锁顺序或叶子执行细节。

## 3. 核心结论

必须固定以下 11 条：

1. `自我线程主循环` 就是当前运行时中的 `自我主循环`。
2. 主循环首先负责 `根层重判`，而不是先盲目推进任务或学习。
3. 主循环必须承接线程回流消息，并把它们统一转成：状态变化、结果特征、条件判定结果、因果引用面和新的治理输入。
4. 主循环不直接凭空发明因果；因果必须建立在动作/方法执行后的变化、结果消息和状态比较之上。
5. 主循环必须统一做 `双值结算`，不能把安全/服务结算散落给各执行线程自行决定。
6. 主循环必须统一整理需求、合并权重和刷新当前主需求。
7. 主循环必须确保根任务、任务管理任务和必要的学习承接任务处于可被下一轮使用的存在状态。
8. 主循环必须装配 `任务管理上下文`，再决定是否提交治理请求。
9. 主循环必须统一消费显式治理事件与延迟事实，再决定 `继续 / 待机 / 停止 / 收束 / 转入学习 / 再提交` 这类最终运行去向。
10. 动作线程和学习线程只负责消费与推进，不负责替代主循环做最终运行裁决。
11. 若缺少本文职责中的任一关键项，规范层就不能宣称“需求-任务-方法-学习-任务管理任务”已经形成真正最小治理闭环。

一句话收口：

`主循环不是简单心跳器，而是根层重判、消息归并、双值结算、需求治理、任务管理上下文装配和最终去向决定的统一承载面。`

具体到 `任务管理任务` 自身最小必须承担哪些治理职责、哪些不应继续留在主循环里，按 [1115_任务管理任务最小职责表规范.md](D:\鱼巢\规范\1115_任务管理任务最小职责表规范.md) 执行。


## 4. 主循环最小职责表

| 职责 | 回答的问题 | 为什么必须留在主循环 |
| --- | --- | --- |
| `消息消费与结果归并` | 上一轮线程推进到底产生了什么 | 只有主循环能把多线程结果统一回收成下一轮输入 |
| `固定先天机制执行` | 服务衰减、待机保护、风险回归等是否该触发 | 这些属于统一主体底线规则，不应散落给叶子线程 |
| `根层重判` | 当前是否允许继续推进，还是应待机/停止/收束 | 这是所有承接层之前的上位输入 |
| `需求治理` | 当前真正最该推进哪个需求，哪些需求该合并/降权/删除 | 需求集是统一主体的治理对象，不应分散裁判 |
| `根任务与治理任务确保存在` | 当前根任务、任务管理任务、必要学习任务是否就绪 | 没有治理任务存在面，后面就无闭环入口 |
| `显式事件消费` | 这一轮到底是普通推进、恢复重建、学习反馈、执行结果还是外部反馈驱动 | 只有主循环能把事件统一转成下一轮治理输入 |
| `任务管理上下文装配` | 这一轮治理请求需要哪些统一中间态 | 这些中间态跨线程共享，不能由单个执行器私造 |
| `延迟事实归并` | 当前还有哪些“已到达但需稍后裁决”的事实要保留 | 这些事实会直接影响最终去向，不应只留在局部摘要 |
| `双值结算` | 本轮结果如何影响安全值和服务值 | 双值属于根控制面，只能统一结算 |
| `最终去向决定` | 继续提交、待机、收束、转入学习还是停止 | 这属于最终裁决层的工程承载面 |


## 5. 统一主循环顺序

从规范层看，主循环至少应按以下顺序运行：

```text
1. 消费动作线程 / 学习线程 / 宿主线程回流消息
2. 执行固定先天机制
3. 基于当前值与需求值做根层重判
4. 刷新与整理需求
5. 确保根任务、任务管理任务、必要学习承接任务存在
6. 消费显式治理事件并归并延迟事实
7. 装配任务管理上下文
8. 依据任务结果与需求权重统一结算安全值 / 服务值
9. 决定本轮最终去向
10. 若允许继续推进，再提交下一轮治理或学习请求
```

必须固定：

- 不得把第 `7` 步结算下放给动作线程自行决定最终真值。
- 不得在第 `3` 步之前先默认“允许继续推进”。
- 不得在第 `8` 步之前让动作线程或学习线程自行连跑下一轮主闭环。


## 6. 主循环必须直接完成的事项

### 6.1 消费回流消息

主循环必须直接消费至少以下类型的回流：

- `任务执行返回消息`
- `学习线程阶段结果`
- `等待/唤醒结果`
- `宿主输入或外部反馈`
- `恢复重建一次性事件`
- `mailbox` 中待消费的显式治理事件

必须固定：

- 这些回流首先属于主循环输入，不属于执行线程本地真值。
- 回流要先归并，再进入根层重判与双值结算。
- 正式 `负反馈` 也只能在这一步之后，由主循环统一生成。

### 6.2 固定先天机制执行

主循环必须直接承担：

- `定时衰减服务值`
- `按双值重算服务衰减步长`
- `按服务值重算安全增加步长`
- `按服务值重算安全减少步长`
- `安全低位回升`
- `安全高位回落`
- `服务归零待机保护`
- `风险安全回归`
- 其他已固定为镶嵌态的底线机制

必须固定：

- 这些机制是主循环统一底线，不属于普通候选方法。
- 对纳入固定清单的每个先天本能函数，自我线程都必须在每个周期自动调用一次。
- 每次调用都必须生成一条对应的 `本能动作动态`。
- 该动作动态应回带本轮主结果状态；若本轮无变化，也仍应保留动作动态事实。

### 6.4 需求治理

主循环必须直接完成：

- 新需求登记
- 重复需求合并
- 需求权重整理
- 已完成需求删除或降权
- 内部学习需求与派生需求的维护

必须固定：

- 需求治理属于统一主体内部整理面，不应放到叶子执行器里各自处理。

### 6.5 根任务与治理任务确保存在

主循环必须确保至少以下存在面：

- `安全根任务`
- `服务根任务`
- `任务管理任务`
- 必要时的 `内部学习任务`

必须固定：

- 这里的“确保存在”不等于每轮重建。
- 主循环负责检查、唤醒、补建与继续使用。

主循环必须装配至少以下治理中间态：

- `最近根层裁决结果`
- `当前主需求`
- `当前管理对象`
- `当前缺口类型`
- `当前下一步去向`

必须固定：

- 这些中间态是承接层共用输入，不得由单个执行线程私造为最终真值。

### 6.7 双值结算

主循环必须按以下材料统一结算安全值与服务值：

- 任务结果
- 该任务服务的需求
- 需求权重
- 结果方向与需求方向是否一致

必须固定：

- 不能只按“成功/失败”粗暴结算。
- 必须显式结合需求位与结果方向。
- 更细的 `实际结果方向` 与 `需求方向` 单特征对齐规则，按 [0321_实际结果方向与需求方向对齐判定表规范.md](D:\鱼巢\规范\0321_实际结果方向与需求方向对齐判定表规范.md) 执行。

### 6.8 最终去向决定

主循环必须统一决定：

- `继续提交`
- `回到重筹办`
- `转入学习`
- `进入收束`
- `待机`
- `停止`

必须固定：

- 这是最终裁决层的工程承接动作。
- 动作线程和学习线程只能提供证据，不能越权代决。
- 即使事件来自 `mailbox`，也必须先回到主循环统一拍板，不得让同一宿主越权连续推进两轮。


## 7. 主循环不直接做的事

以下事项不应直接放在主循环里长期阻塞执行：

1. 叶子方法长时间执行
2. 复杂任务树细节推进
3. 大块学习搜索与验证
4. 阻塞式外设或宿主 I/O

必须固定：

- 主循环负责“决定与归并”，不负责长期占用执行时间片的重活。


## 8. 与因果的边界

必须固定：

1. 主循环不直接凭空生成因果。
2. 主循环负责基于以下材料刷新或引用因果：
   - 动作后的状态变化
   - 方法返回结果
   - 执行线程回流消息
   - 前后状态比较结果
3. 没有变化，就没有新的变化因果；但动作事实动态仍可存在。
4. 因果仍然只作为主循环后续判断与组织的证据面，不直接替代根层裁决。


## 9. 收口结论

后续统一按以下口径执行：

1. 自我线程主循环必须同时承担：消息归并、固定机制、根层重判、需求治理、治理任务存在维护、上下文装配、双值结算和最终去向决定。
2. 若这些职责没有在规范层被统一收口，就不能宣称最小治理闭环已经成立。
3. 主循环是“统一主体”的运行核心，不是简单定时器。

最终统一收口为：

`自我线程主循环真正要做的，不只是选需求和收消息，而是把回流结果重新组织成下一轮可裁决、可结算、可治理的统一输入。`
