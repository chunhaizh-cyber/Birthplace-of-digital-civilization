# 1196 任务管理方法需求位最小状态动态因果清单

## 1. 目标

本文用于把 [1195_任务管理方法需求位与因果到方法映射规范.md](D:\鱼巢\规范\1195_任务管理方法需求位与因果到方法映射规范.md) 里的 7 个 `方法需求位`，进一步落成一张可直接对照实现的最小清单，回答以下问题：

- 每个方法需求位最少需要哪些 `状态`
- 每个方法需求位最少需要哪些 `动态`
- 每个方法需求位最少需要什么 `单跳实例因果`
- 每个方法需求位最少需要什么 `轻量因果链`
- 当前这些原料大致准备到什么程度

本文不是重新定义方法需求位，而是把它们变成一张可落地、可审计的实现清单。

如果只想单独查看“后续到底要生成哪些因果信息”，以 [1197_任务管理方法所需因果信息清单.md](D:\鱼巢\规范\1197_任务管理方法所需因果信息清单.md) 为准。

- 帮助后续按表补代码
- 帮助区分“状态已就绪”和“专用因果还没接上”

### 2.1 在统一主链中的位置

本文默认处于以下主链中段：

`根层方向裁决`
-> `形成任务管理上下文`
-> `选择当前方法需求位`
-> `按本文检查最小状态/动态/因果原料`
-> `执行对应管理方法`
-> `返回外层重判`

必须固定：

1. 本文只在 `根层已允许继续推进` 且 `任务管理上下文已形成` 时生效。
2. 若 `最近根层裁决结果` 已要求 `停止 / 收束 / 待机`，则普通 7 个方法需求位不再按常规路径继续扩展。
3. 本文列出的最小状态、最小动态和最小因果，只是任务管理治理证据面，不得替代根层方向裁决本身。


## 3. 使用口径

为了避免把不同层次的原料混成一类，本文把每个方法需求位拆成四面：

1. `最小状态面`
- 进入该方法位判定和结果沉淀，至少要看见哪些显式状态

2. `最小动态面`
- 这类方法运行时，至少要留下哪些动作动态

3. `最小实例因果`
- 这类方法第一次能成立什么单跳治理因果

4. `最小轻量因果链`
- 这类方法如果继续成长，至少要能接成哪类多跳解释链

另外，本文中的 `当前准备度` 只做粗粒度判断：

- `状态已大体具备`
- `动态已大体具备`
- `专用实例因果待补`
- `专用轻量因果链待补`


## 4. 共性底座

7 个方法需求位共享的最小公共底座固定为：

### 4.1 共用状态底座

- `最近根层裁决结果`
- `当前功能域`
- `当前缺口类型`
- `当前下一步去向`
- `输入条件主键`
- `治理态型`
- `结构完备度`
- `方法可用度`
- `风险门控`
- `结构推进成立`
- `治理闭环成立`
- `管理任务_结果节点主键`
- `管理任务_失败原因码`
- `管理任务_二次特征证据`
- `任务_状态`
- `任务_当前方法`
- `任务_最近方法`
- `任务_当前步骤`
- `任务_当前子任务`

补充固定：

- 若系统启用了主体层，允许在消费这些共用状态底座时稳定挂带 `主体ID / 分身ID / 当前本能能力缺口类型` 作为继承标识。
- 这些主体层标识不替代本文列出的状态底座，只是额外的桥接字段。

### 4.2 共用动态底座

- 当前管理方法 `开始运行`
- 当前管理方法 `完成 / 失败`
- `任务管理上下文刷新完成`
- `方法运行次数递增`
- `方法成功次数递增 / 方法失败次数递增`
- `方法可靠度更新`
- `任务最近调度时间更新`
- `任务当前方法切换`
- `任务最近方法切换`

### 4.3 共用因果底座

每个方法需求位都至少应满足：

```text
因：
  最近根层裁决结果
  + 当前缺口类型
  + 当前下一步去向
  条件状态裁剪集
  + 二次特征
  + 输入条件主键

动作：
  当前管理方法调用动态

果：
  管理派生产出
  + 结构推进变化
  + 宿主任务状态变化
```

### 4.4 共用轻量因果链约束

下表中所有 `最小轻量因果链`，默认都额外服从以下约束：

1. `链整体可信度 <= 链最弱跳可信度`
2. 若某一跳使用 `外部动作动态` 作为主动作，则该跳必须补齐：
- `外部结果确认状态`
- `外部结果确认来源`
3. 没有 `外部结果确认状态` 时，只能说明 `外部动作已发出`，不得把该跳记为强确认果跳
4. 若最弱跳属于“未确认外部结果跳”，则整条链不得记为高可信稳定链

可直接理解为：

- 本表里的 `最小轻量因果链`，默认先按内部治理链解释
- 一旦后续向外部动作或外部结果延长，就必须自动挂上 `外部确认` 与 `最弱跳上限` 两条约束

### 4.5 与外层重判的边界

必须继续固定：

1. `方法需求位` 的最小状态/动态/因果，只回答“当前治理位是否具备承接与解释原料”。
2. 它们不能直接回答“根层方向是否仍允许继续推进”。
3. 每次方法位执行后，即使已经生成专用实例因果或轻量因果链，也必须回到外层重新做 `根层裁决 -> 上下文刷新 -> 再判当前方法位`。

## 6. 分位说明

### 6.1 补结构方法位

这类方法位是 `任务管理任务` 的出生口。

若它不具备：

- 任务就容易停在“知道缺东西，但不会把结构搭出来”

因此它的首要检查顺序应固定为：

1. `依赖缺失 / 任务链未成形 / 无可试方法` 是否已显式化
2. 结构建立类动态是否已留痕
3. `补结构 -> 结构成形` 的实例因果是否已可生成

### 6.2 失败策略方法位

这类方法位和学习等待链耦合最紧。

它的特殊点在于：

- 状态和动态原料相对最全
- 但 `失败策略专用实例因果` 和 `等待学习链的轻量因果链` 仍未正式接上

因此它通常会成为因果接线时最先能落地的一组。

### 6.3 收束复盘方法位

这类方法位和 `价值因果` 耦合最紧。

它的特殊点在于：

- 二次特征里的 `治理价值成立 / 价值闭环类型 / 条件价值映射已出现` 已经为它准备好了较强状态底座
- 但真正的 `收束 -> 价值结算` 长链，还必须等轻量因果链组织接上


## 7. 当前总体判断

按这张清单看，当前可以收成一句话：

`7 个方法需求位的状态底座已经大体具备，任务/方法侧共用记账动态也已经大体具备；真正还没补上的，主要不是“状态和动态”，而是“每个方法需求位专用的实例因果”和“继续把它们串成受最弱跳可信度约束的轻量因果链”的路径。`


## 8. 最小补齐顺序

建议下一步按以下顺序推进：

1. 先补 `补结构方法位` 和 `失败策略方法位` 的专用实例因果。
2. 再补 `门控转域方法位` 和 `收束复盘方法位` 的专用实例因果。
3. 然后把 `补结构 / 失败纠偏 / 门控转域 / 收束价值` 四条最小轻量因果链接上。
4. 最后再补 `选择推进 / 分解补条件 / 求证判定` 这三组更细的专用链。


## 9. 一句话收口

可收敛为一句话：

`任务管理任务的 7 个方法需求位，现在已经有了大体可用的状态面和动态面；下一阶段真正要补的核心，不再是继续堆状态，而是把每个方法需求位对应的专用实例因果和受最弱跳可信度约束的最小轻量因果链正式接进代码。`
