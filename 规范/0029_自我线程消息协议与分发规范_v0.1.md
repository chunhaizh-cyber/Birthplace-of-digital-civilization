# 0029 自我线程消息协议与分发规范 v0.1

## 1. 目标

本文用于把前序消息总表正式落成“可直接编码”的消息合同，固定以下内容：

- 消息协议如何定义
- 首收件人如何判定
- 哪些消息必须上浮到自我线程
- 自我线程如何按大类分发处理
- 一轮消息批次如何执行
- 消息处理后允许修改哪些状态面

本文的目标只有一个：

`把“会收到什么消息、谁先收、怎么分发、怎么处理、处理后改什么状态”固定成代码合同。`

本文不负责：

- 新增一级职责
- 展开任务管理工作线程内部算法
- 展开方法内部执行细节
- 先行定义任务三层实体字段全集

## 3. 核心结论

必须固定以下结论：

1. `0028` 负责定义“自我线程收什么消息”；`0029` 负责定义“这些消息怎么编码、怎么分发、怎么落主循环”。
2. 消息协议至少拆成四个实现模块：
   - `自我线程消息协议模块`
   - `自我线程消息路由模块`
   - `自我线程消息处理器模块`
   - `自我线程消息批次执行器模块`
3. 消息协议模块负责定义 `枚举 / 消息头 / 载荷结构`，不负责处理逻辑。
4. 消息路由模块只负责：
   - 判定首收件人
   - 判定是否需要上浮到自我线程
   - 判定是否需要路由回父任务
5. 消息处理器模块只负责：
   - 改哪些状态
   - 是否触发根层重判
   - 是否触发新任务实例化
   - 是否触发结果路由
6. 消息批次执行器模块只负责：
   - 冻结本轮消息批次
   - 先处理抢占消息
   - 处理本轮普通消息
   - 处理本轮同步闭包
   - 做一次根层重判
   - 做一次主派发/收束决议
7. `控制消息 / 全局环境消息` 必须先到 `自我线程`。
8. `局部外部反馈` 可先到相关工作线程；若升级为控制面变化，再上浮线程。
9. `结果回流消息` 必须先到线程，再由线程路由给父层任务。
10. `任务三层实体` 应在 `0029` 之后再单独定义；消息协议先解决“怎么说话”，任务实体再解决“话改到哪”。
11. 若需要把 `0028 + 0029` 进一步压成可直接对照代码字段与写权限的矩阵表，应按 `0030` 执行。

一句话收口：

`0029 不是继续扩职责，而是把消息世界正式写成协议、路由和批次执行骨架。`


## 4. 四个实现模块

### 4.1 自我线程消息协议模块

该模块只负责定义消息合同，不负责处理逻辑。

最小内容至少包括：

- `枚举_消息大类`
- `枚举_消息子类`
- `枚举_首收件人`
- `结构_治理消息头`
- `结构_治理消息载荷_*`
- `结构_治理消息`

推荐模块名：

`自我线程消息协议模块.ixx`

### 4.2 自我线程消息路由模块

该模块只负责决定“谁先收、是否上浮、是否回父任务”。

推荐模块名：

`自我线程消息路由模块.ixx`

### 4.3 自我线程消息处理器模块

该模块按七类消息提供处理入口，但每个入口只负责控制面动作，不展开局部工序。

推荐模块名：

`自我线程消息处理器模块.ixx`

### 4.4 自我线程消息批次执行器模块

该模块把“冻结一批，清完一批，再做一次决议”写成统一轮次骨架。

推荐模块名：

`自我线程消息批次执行器模块.ixx`


## 5. 消息协议模块规范

### 5.1 消息大类

```cpp
enum class 枚举_消息大类 : std::uint8_t {
    控制消息 = 0,
    全局环境消息 = 1,
    工作线程状态消息 = 2,
    任务推进关键事件消息 = 3,
    派生需求消息 = 4,
    结果回流消息 = 5,
    学习与审计消息 = 6
};
```

### 5.2 消息子类

必须固定：

1. `消息子类` 必须按 `消息大类` 分区，不得跨大类混用。
2. 同一子类必须能唯一映射到一类处理函数。
3. 子类枚举允许后续扩展，但不得改变既有子类的语义。

### 5.3 首收件人枚举

```cpp
enum class 枚举_首收件人 : std::uint8_t {
    自我线程 = 0,
    任务管理工作线程 = 1
};
```

### 5.4 治理消息头

```cpp
struct 结构_治理消息头 {
    主键 消息主键 = 0;
    枚举_消息大类 大类{};
    枚举_消息子类 子类{};

    主键 来源线程主键 = 0;
    主键 来源任务主键 = 0;
    主键 目标任务主键 = 0;
    主键 父任务主键 = 0;
    主键 关联场景主键 = 0;

    时间戳 产生时间 = 0;

    bool 可抢占 = false;
    bool 仅局部有效 = false;

    std::uint64_t 追踪签名 = 0;
};
```

必须固定：

1. 所有治理消息都必须带 `消息头`。
2. `可抢占 = true` 只允许用于控制类或故障升级类消息。
3. `仅局部有效 = true` 的消息，默认不直接进入全局重判，除非经路由器升级。

### 5.5 治理消息载荷

推荐先采用“分类型结构体”，不强制一开始引入复杂 `variant`：

```cpp
struct 结构_控制消息载荷;
struct 结构_全局环境消息载荷;
struct 结构_工作线程状态消息载荷;
struct 结构_任务推进关键事件载荷;
struct 结构_派生需求消息载荷;
struct 结构_结果回流消息载荷;
struct 结构_学习与审计消息载荷;
```

统一治理消息外壳建议为：

```cpp
struct 结构_治理消息 {
    结构_治理消息头 头{};
    std::string 载荷键{};
};
```

必须固定：

1. 载荷的真实结构，应由 `头.大类 + 头.子类` 决定。
2. `载荷键` 只作为轻量索引键或分发表达，不替代正式载荷结构。


## 6. 消息路由模块规范

路由模块只回答三件事：

1. `谁先收件`
2. `是否需要上浮到自我线程`
3. `是否需要路由回父任务`

必须至少提供以下接口：

```cpp
枚举_首收件人 判定首收件人(const 结构_治理消息&);
bool 是否需要上浮到自我线程(const 结构_治理消息&);
bool 是否需要路由回父任务(const 结构_治理消息&);
```

### 6.1 判定首收件人规则

必须固定：

1. `控制消息` 首收件人恒为 `自我线程`。
2. `全局环境消息` 首收件人恒为 `自我线程`。
3. `工作线程状态消息` 首收件人恒为 `自我线程`。
4. `任务推进关键事件消息 / 派生需求消息 / 结果回流消息` 默认由工作线程产生并上浮线程，因此首收件人可直接判为 `自我线程`。
5. `明确绑定到具体任务且仅影响局部流程的普通外部反馈`，可判为 `任务管理工作线程`。

### 6.2 上浮线程规则

必须固定：

以下情形必须上浮到 `自我线程`：

- 改变控制态
- 改变优先级
- 改变队列状态
- 请求实例化新任务
- 需要结果路由
- 需要统一结算

以下情形原则上不上浮：

- 步骤内部小进度
- 局部临时变量变化
- 每一次内部小重试
- 纯中间态缓存更新

### 6.3 路由回父任务规则

必须固定：

以下消息必须允许路由回父任务：

- 子任务结果
- 需要父层任务判断的局部结果
- 派生需求被否决结果
- 子结果导致的局部解阻结果


## 7. 消息处理器模块规范

消息处理器模块至少提供以下入口：

```cpp
void 处理控制消息_(const 结构_治理消息&);
void 处理全局环境消息_(const 结构_治理消息&);
void 处理工作线程状态消息_(const 结构_治理消息&);
void 处理任务推进关键事件_(const 结构_治理消息&);
void 处理派生需求消息_(const 结构_治理消息&);
void 处理结果回流消息_(const 结构_治理消息&);
void 处理学习与审计消息_(const 结构_治理消息&);
```

必须固定：

1. 处理器模块不直接展开局部工序。
2. 处理器模块不直接替代工作线程做步骤推进。
3. 每个处理函数只负责：
   - 改哪些状态
   - 是否触发根层重判
   - 是否触发新任务实例化
   - 是否触发结果路由


## 8. 消息批次执行器模块规范

消息批次执行器模块必须把“一轮怎么吃消息”固定成统一骨架。

至少提供以下函数：

```cpp
void 冻结本轮消息批次_();
void 处理抢占控制消息_();
void 处理普通消息批次_();
void 处理本轮同步闭包_();
void 根层重判_();
void 生成主派发决议_();
```

轮次顺序必须固定为：

`冻结本轮消息集 -> 先处理抢占控制消息 -> 处理本轮全部冻结消息 -> 处理本轮同步闭包 -> 做一次根层重判 -> 做一次主派发 / 收束决议`

必须固定：

1. 每轮只做一次 `根层重判`。
2. 每轮最多做一次 `主派发 / 收束决议`。
3. 抢占消息可以中断普通消息，但不得跳过结算。
4. 本轮异步新产生的消息必须进入下一轮。


## 9. 三张表

### 9.1 消息类型表

字段必须至少包括：

- 消息大类
- 消息子类
- 来源
- 首收件人
- 是否可抢占
- 是否影响优先级
- 是否触发根层重判
- 是否可能实例化新任务
- 是否需要路由父任务

### 9.2 状态改动表

字段必须至少包括：

- 消息子类
- 会修改的对象
  - 线程控制态
  - 任务控制态
  - 任务局部运行态摘要
  - 工作线程状态表
  - 结果缓存
  - 学习候选池
- 谁可写
- 是否立即持久化

### 9.3 消息闭包表

字段必须至少包括：

- 触发消息
- 本轮同步会引发什么内部消息
- 是否必须本轮处理完
- 是否允许延后到下一轮

必须固定：

1. `消息闭包表` 的意义，是把“冻结一批，清完这一批，再做决议”从口号变成规则。
2. 本轮同步闭包必须在本轮内清完。
3. 异步后果不得为了“减少轮次”而强行追入当前闭包。


## 10. 与任务实体的前后顺序

必须固定：

1. `消息协议` 先解决 `怎么说话`。
2. `任务实体` 再解决 `话改到哪`。
3. 因此 `任务实体模块` 应排在 `0029` 之后，而不应先于 `0029` 落地。

推荐下一份规范为：

`任务实体模块 v0.1`

其最小三层应为：

- `结构_任务根节点`
- `结构_任务控制态`
- `结构_任务局部运行态`
