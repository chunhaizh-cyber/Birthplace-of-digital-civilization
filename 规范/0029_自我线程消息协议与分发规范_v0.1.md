## 3. 核心结论

必须固定以下结论：

1. `0028` 负责定义“自我线程正式收什么消息”；`0029` 负责定义“这些消息怎么编码、怎么分发、怎么进批次执行器”。
2. 自我线程正式输入消息只保留两级：
   - `一级来源链`
   - `二级处理类型`
3. `学习 / 审计 / 复盘` 不进入正式输入协议枚举；若升级为正式事实，必须重新包装成 7 类中的一种。
4. 协议层只表达 `事实`，不直接表达 `需求成立 / 权重变化 / 否定项成立` 这类结论。
5. 自我线程在消费这些消息时，应统一经过：
   `一次特征 -> 二次特征 -> 影响项 -> 根层重判 -> 主派发`
6. `工作线程状态消息` 当前保留的是兼容名；协议语义上应按“`其它线程状态变化消息`”理解。
7. 发件线程只负责构造并投递消息；线程状态真值、状态节点和动作动态，只能由 `自我线程` 在消费消息阶段统一写回。
8. `自我线程` 自身的关键生命周期变化属于协议例外：它们应落 `关键日志 / 宿主级审计 / 恢复判断材料`，不强制作为普通治理输入继续进入“线程状态变化消息”主链。


## 4. 协议枚举

### 4.1 消息来源链

```cpp
enum class 枚举_消息来源链 : std::uint8_t {
    线程直达消息 = 0,
    任务管理工作线程回流消息 = 1
};
```

### 4.2 消息处理类型

```cpp
enum class 枚举_消息处理类型 : std::uint8_t {
    控制域事实消息 = 0,
    业务域事实消息 = 1,
    延迟事实消息 = 2,
    工作线程状态消息 = 3,
    任务推进关键事件消息 = 4,
    派生需求上报消息 = 5,
    结果与关键中间状态回流消息 = 6
};
```

### 4.3 首收件人

正式进入自我线程输入集合的消息，其首收件人恒为 `自我线程`。

若某些局部反馈仍先在工作线程内部消化，它们不属于本文正式协议范围。


## 5. 协议结构

### 5.1 消息头

```cpp
struct 结构_治理消息头 {
    主键 消息主键 = 0;

    枚举_消息来源链 一级分类{};
    枚举_消息处理类型 二级类型{};

    主键 发出者主键 = 0;
    主键 动作主体主键 = 0;
    主键 作用对象主键 = 0;
    主键 父任务主键 = 0;
    主键 关联场景主键 = 0;

    时间戳 产生时间 = 0;
    std::uint64_t 路径签名 = 0;

    bool 来源成功标志 = false;
    I64 错误码 = 0;
    I64 置信度 = 0;
};
```

### 5.2 特征变化项

```cpp
struct 结构_特征变化项 {
    主键 变化主体主键 = 0;
    主键 特征主键 = 0;
    特征值 变化前值{};
    特征值 变化后值{};

    bool 是否确认生效 = false;
    bool 是否关键中间状态 = false;
    bool 是否已恢复 = false;

    std::string 变化原因键{};
};
```

### 5.3 消息本体

```cpp
struct 结构_治理消息 {
    结构_治理消息头 头{};
    std::string 动作语义键{};
    std::vector<结构_特征变化项> 变化项集{};
};
```

必须固定：

1. `动作语义键 + 变化项集` 才是正式语义承载面。
2. `一级分类 / 二级类型` 只回答来源链和处理入口，不替代正式语义。


## 6. 分发接口

协议层建议至少提供以下处理入口：

```cpp
void 处理控制域事实消息_(const 结构_治理消息&);
void 处理业务域事实消息_(const 结构_治理消息&);
void 处理延迟事实消息_(const 结构_治理消息&);
void 处理工作线程状态消息_(const 结构_治理消息&);
void 处理任务推进关键事件消息_(const 结构_治理消息&);
void 处理派生需求上报消息_(const 结构_治理消息&);
void 处理结果与关键中间状态回流消息_(const 结构_治理消息&);
```

必须固定：

1. 7 类处理入口都只做 `控制面处理`，不展开工作线程内部算法。
2. 任何处理函数都不得直接跳过 `二次特征 / 影响项` 链去拍根层结论。
3. `处理工作线程状态消息_(...)` 当前是兼容入口名；其语义应统一承接除 `自我线程` 关键日志例外外的任意线程状态变化消息，并由 `自我线程` 在消费阶段完成正式落账。


## 7. 批次执行骨架

协议层批次执行必须固定为：

```text
冻结本轮消息集
-> 规范化
-> 落一次特征账
-> 重算二次特征
-> 生成影响项
-> 根层重判
-> 主派发决议
-> 本轮结算
```

必须固定：

1. 一轮只处理本轮冻结消息及其同步内部闭包。
2. 本轮中新产生的异步消息，不得追入本轮。
3. 控制域事实消息允许按规则抢占，但不能绕过最终结算。
4. `落一次特征账` 发生在 `自我线程` 的消息消费阶段；发件线程不得借构造消息之名直接写回线程状态真值。
5. 其它线程状态变化在 `落一次特征账` 之后，不得停在“只记账”；还应继续进入 `二次特征 / 影响项` 链，评估它对自我的影响。


## 8. 不再保留的旧口径

以下旧口径在协议层不再保留：

- 把 `控制消息 / 全局环境消息 / 学习与审计消息` 当作顶层正式输入枚举
- 把 `学习反馈 / 审计结论 / 复盘结论` 当作正式输入消息子类
- 让 `需求 / 任务权重 / 否定项成立` 直接进入正式消息协议


## 9. 完成标准

达到下面 6 条，即可视为协议层已经对齐：

1. 已经把 `一级来源链 + 二级处理类型` 收成正式枚举。
2. 已经把 `关键中间状态 / 已恢复 / 已确认` 补进变化项合同。
3. 已经把 7 类正式输入消息对应到 7 个处理入口。
4. 已经把旧的 `学习与审计消息` 顶层输入口径移除。
5. 已经把协议层重新接回 `一次特征 -> 二次特征 -> 影响项 -> 重判` 主链。
6. 已经明确线程状态变化消息只授予发件线程上报权限，不授予直接落账权限。
