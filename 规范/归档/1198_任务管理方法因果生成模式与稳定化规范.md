# 1198 任务管理方法因果生成模式与稳定化规范

## 1. 目标

本文用于固定 `任务管理方法` 的两种因果生成模式，以及它们如何从 `候选因果` 逐步进入 `稳定因果信息` 与 `方法成长`。

本文重点回答以下问题：

- 因果生成到底有哪两种模式
- 每种模式的起点、窗口边界和生成顺序是什么
- 临时因果怎样才能升格为较稳定因果
- 什么时候允许继续尝试学习
- 因果如何继续支撑方法需求位与候选方法成长

本文不直接改写 `1285` 的全域因果硬约束，也不直接替代 `1190` 的任务管理方法单跳实例因果规范。

## 3. 基本判断

必须先固定以下 15 条：

1. `任务管理方法因果` 的生成模式，当前统一只有两种：`目标驱动回溯式` 与 `自然归纳式`。
2. 两种模式的第一产物都只能是 `候选实例因果` 或 `临时因果`，不得直接记为稳定因果。
3. 两种模式都必须继续服从 `同锚点 + 同层级 + 时间有序`。
4. 任何模式都不得跳过显式 `动作动态`，直接把状态共现写成因果。
5. `二次特征` 只负责压缩局势，不得直接替代原始条件状态或原始结果状态。
6. `结果主键`、`失败原因码`、`二次特征证据` 默认都是辅助证据，不得在存在更具体结果时抢占主结果。
7. `自然归纳式` 的重复，不能只看“特征类型相同”，还必须看 `锚点桶 + 层级桶 + 功能域桶 + 果方向桶` 是否一致。
8. 一个时间窗口若已经跨出单跳边界，必须拆成多条单跳实例因果，再组织成轻量因果链。
9. 候选因果必须允许出现 `反例`、`结果分裂` 和 `稳定度下降`，不得只累计正例。
10. 只有在出现一定次数的稳定复现后，才允许把因果继续用于 `尝试学习` 与 `候选方法出生`。
11. 若动作证据来自 `内部先天动作动态`，则动作事实可直接可信，不再重复校验动作真实性。
12. 这套动作证据可信度分层，同样适用于后续组织出的所有因果链。
13. 两种模式都只能在 `最近根层裁决结果=允许继续推进` 的前提下启动。
14. 若 `当前缺口类型` 或 `当前下一步去向` 尚未显式化，则只允许生成局部候选因果，不得直接升格为稳定因果信息。
15. 因果稳定化后的产物，仍然只是外层重判、方法成长和学习尝试的证据，不得直接替代最终继续/停止裁决。


## 4. 两种生成模式

### 4.1 目标驱动回溯式因果生成

`目标驱动回溯式` 的语义是：

- 先有一个想确认的 `目标果`
- 再回溯这条果是由什么条件和动作导致的

这个模式适合：

- 已经知道当前某个方法需求位最需要什么结果
- 已经知道要验证某个特定治理果是否稳定成立
- 已经知道某个结果方向是成长关键证据

固定入口是：

- `目标果状态`

并且默认还必须已有：

- `最近根层裁决结果=允许继续推进`
- `当前缺口类型`
- `当前下一步去向`

固定回溯顺序是：

1. `目标果状态`
2. `结束动作动态`
3. `开始动作动态`
4. `过程动态`
5. `条件状态`

必须固定：

- 这里的“目标果状态”是第一锚点，不能先用任意动作动态做模糊搜索
- 若找不到与该果同批次的显式结束动态，则本次回溯不得生成实例因果
- 条件状态只取对该果真正有贡献的最小裁剪集，不取全量噪声状态

### 4.2 自然归纳式临时因果生成

`自然归纳式` 的语义是：

- 没有预设目标果
- 先从重复出现的同类结果里归纳出临时因果
- 再在后续样本中继续验证或淘汰

这个模式适合：

- 系统还不知道应该先学哪种治理方法
- 已出现大量重复治理片段
- 希望从运行历史中自然发现可复用治理关系

固定入口是：

- `重复出现的同类果桶`

并且默认还必须已有：

- `最近根层裁决结果=允许继续推进`
- `当前缺口类型`
- `当前下一步去向`

这里的“同类果桶”至少必须同时满足：

- `同锚点`
- `同层级`
- `同功能域` 或 `同方法需求位桶`
- `同结果状态主键`
- `同方向桶`

其中方向桶至少按以下几类归类：

- `0 -> 1`
- `0 -> 非0`
- `低 -> 中/高`
- `未定义 -> 已定义`
- `旧值 -> 新值`
- `n -> n + 1`

必须固定：

- 不能只凭“特征类型相同、方向一致”就认定为同类果
- 若结果状态主键不同，即使方向相同，也不得进同一果桶


## 5. 窗口截取规则

### 5.1 目标驱动模式的窗口

目标驱动模式的最小窗口固定为：

`开始动态 -> 中间治理动态 -> 结束动态 -> 目标果`

推荐截取顺序是：

1. 先定位 `目标果`
2. 向前找最近的同批次 `结束动态`
3. 再向前找与之配对的 `开始动态`
4. 收集这个边界窗口中的 `过程动态`
5. 再补入这个窗口开始时刻已成立的关键条件状态

不得改成：

- 只按两个同名状态之间的全部内容硬截窗口
- 把多个结束动态混在同一条单跳因果里

### 5.2 自然归纳模式的窗口

自然归纳模式生成临时因果时，也必须优先按 `动作边界` 截窗口，而不是按“两个相同果之间的所有内容”粗暴截取。

推荐顺序是：

1. 先在同类果桶中找到一个样本果
2. 找到它对应的 `结束动态`
3. 找到该结束动态配对的 `开始动态`
4. 收集边界内的 `过程动态`
5. 收集窗口开始时已成立的关键条件状态
6. 形成一条 `临时实例因果`

必须固定：

- 同一个窗口只能先产出一条主单跳实例因果
- 若窗口内存在多个阶段性结果变化，应拆成多条单跳，再考虑组织轻量因果链


## 6. 动作与条件的抽象收敛规则

### 6.1 动作收敛

对多条临时因果做归纳时，不得直接对完整动态列表做交集。

动作侧应优先收敛为：

- `主动作事件`
- `当前管理方法首节点`
- `方法需求位`
- `动作相位骨架`

原因是：

- 细节动态经常会随实现变化
- 但 `方法首节点 + 结束动态 + 方法需求位` 更稳定

### 6.2 条件收敛

条件侧做交集时，应按以下优先级保留：

1. `当前功能域`
2. `输入条件主键`
3. `二次特征`
4. 直接门控字段
5. 经过验证的稳定辅助条件

必须固定：

- 交集只在 `同桶临时因果` 内进行
- 不能跨功能域、跨锚点、跨方法需求位直接取交集
- 条件交集变少以后，必须继续用后续样本验证，不能直接认定条件越少越真


## 7. 动作动态可信度分层

两种生成模式在抽取动作证据时，都必须继续服从以下分层：

### 7.1 内部先天动作动态

- 动作事实直接可信
- 动作边界直接可信
- 若果本身就是该动作直接写回的内部结果，则可作为高可信果侧证据

因此：

- 目标驱动模式可直接从该动作回溯候选实例因果
- 自然归纳模式可优先把这类样本放入高可信桶

### 7.2 自我内部动作动态

- 动作事实直接可信
- 动作边界直接可信
- 果侧仍需校验 `同锚点 + 同层级 + 时间有序 + 同批次`

因此：

- 这类动作可直接作为主动作事件候选
- 但整条因果仍需通过结果对齐校验

### 7.3 外部动作动态

- 直接可信的只是：`动作已发出`
- 不直接等于：`外部结果已成立`

因此：

- 外部动作若没有观测、反馈、回执或结果状态确认，只能形成弱候选因果
- 不得直接进入高置信稳定因果

### 7.4 对因果链的约束

必须固定：

1. 因果链中的每一跳，都按其动作类别分别应用上述规则。
2. 整条链的可信度上限，不得高于最弱的一跳。
3. 若某一跳只证明“外部动作已发出”，则该跳不得被桥接成“外部结果已确认”。
4. 若某一跳依赖的动作事实本身不成立，则整条链不得继续向后稳定化。


## 8. 反例、分裂与置信规则

### 8.1 必须记录的负向信息

每条候选或临时因果，至少应记录：

- `样本数`
- `正例数`
- `反例数`
- `结果分裂次数`
- `稳定复现次数`
- `当前置信度`

### 8.2 反例判定

至少以下情况应记为反例或分裂：

- 同类条件下没有出现预期果
- 同类条件下出现了不同方向的果
- 同类条件下出现了不同主结果状态
- 同类条件下动作边界不一致，导致无法支持同一动作语义

### 8.3 置信度语义

`当前置信度` 只表示这条因果在当前证据下的稳定程度，不表示绝对真值。

必须固定：

- 反例会降低置信度
- 结果分裂会降低置信度
- 稳定复现会提升置信度
- 长时间未再出现，可按窗口老化规则降低置信度


## 9. 从临时因果到稳定因果

当前建议固定以下阶段：

1. `候选实例因果`
- 单次合法窗口生成
- 仅说明“这次可能是因果”

2. `临时因果`
- 至少两次同桶出现
- 允许开始做条件和动作收敛

3. `待验证稳定因果`
- 已有多次稳定复现
- 反例和分裂仍在可接受范围
- 允许进入 `尝试学习`

4. `稳定因果信息`
- 经多轮验证仍稳定
- 已能支撑方法需求位上的候选方法出生、验证或升格

必须固定：

- `尝试学习` 应发生在 `待验证稳定因果` 阶段之后，而不是第一次临时因果出现时
- `稳定因果信息` 仍然不是最终抽象因果模板
- `稳定因果信息` 一旦被用于当前治理轮，也必须在外层重新经过 `根层重判 -> 上下文刷新`


## 10. 与方法成长的衔接

稳定化后的因果，不应直接被当成方法本体。

正确顺序固定为：

`候选/临时因果`
-> `待验证稳定因果`
-> `稳定因果信息`
-> `映射到方法需求位`
-> `候选方法出生`
-> `候选方法验证`
-> `来源升格`

必须固定：

- 因果是证据
- 方法是能力
- `果` 只是结果，不得直接等同为方法


## 11. 与 1190 的分工边界

可收成一句话：

- `1190` 回答“单跳实例因果和轻量因果链长什么样”
- `1198` 回答“这些因果是从哪里开始生成、如何自然稳定下来”

因此后续实现顺序应为：

1. 先由外层完成 `根层重判`
2. 再形成任务管理上下文，并显式化 `最近根层裁决结果 / 当前缺口类型 / 当前下一步去向`
3. 按 `1197` 确认所需因果信息
4. 按 `1198` 选择生成模式并截取合法窗口
5. 按 `1190` 生成合法单跳实例因果
6. 再按 `1190` 组织轻量因果链
7. 再按 `1195` 和 `1196` 支撑方法需求位成长
8. 最后返回外层重新做 `根层重判 -> 上下文刷新`


## 12. 一句话收口

`任务管理方法的因果生成，当前应固定为“目标驱动回溯式”和“自然归纳式”两种模式；二者都必须先产出候选或临时因果，再经过同桶归纳、反例约束、稳定复现与置信提升，逐步进入可尝试学习的稳定因果信息，严禁把状态共现直接当成稳定因果或直接当成方法。`
