## 1. 目标

本文用于固定以下问题：

- `学习方法` 在当前体系中是如何生成的

## 3. 核心结论

### 3.0 2026-04 最新口径修正

若本文下文旧表述与本节冲突，以本节为准。

必须新增固定以下 12 条：

1. 方法成长的主口径统一改为：`有没有`、`能不能用`、`好不好用`。
2. `稳定度` 不再作为学习生成链的显性主概念；若继续保留，只允许退为后台统计证据，不得再作为主判词。
3. `任务管理任务` 发现当前任务没有可直接命中需求的可用方法时，应先复用已有但未可用的方法头；若还没有方法头，必须先创建一个“仅含结果(需求)”的 `待学习的方法首节点`，再围绕该方法头衍生学习目标、学习需求与候选学习程序。
4. `方法不存在` 与 `方法已存在但不可用`，都应统一进入 `方法补齐学习链`；不得先把 `默认本能兜底` 挂成当前普通执行方法来代替真实方法选择。
5. `学习任务` 不是每个方法各建一条长期任务；应优先视为一个可复用的执行壳，真正排队的是绑定到不同方法头的学习工单。
6. `学习工作线程` 的职责，是在方法列表中查找符合当前学习目标的 `可执行学习方法`，并推动当前学习工单达到可执行。
7. 当学习工单达到可执行后，当前轮学习方法应直接带入 `待学习的方法首节点`，开始对该方法头做补条件、补动作、补结果、补调用资格的学习。
8. 抽象因果现在可以直接补方法骨架，因此学习链应优先利用 `抽象因果 + 可执行学习方法 + 二次特征裁判` 来完善方法树。
9. `候选学习程序` 仍保留，但其职责应更偏向“当前轮如何学”；方法头本体与学习工单本体不得混写。
10. `任务管理任务` 生成的真正学习需求，不是“样本收集/差异比较”等动作名，而是：`让当前待学习方法头变为可用方法`。
11. `样本收集 / 差异比较 / 共同条件提取 / 候选动作生成 / 结果预测 / 沙箱验证 / 风险评估 / 边界标记` 这些都应下沉为满足该主需求的 `学习动作` 或 `原语级内部最小方法组合`，不再冒充顶层需求本身。
12. 对 `任务管理任务` 来说，本文中的 `原函数` 应优先理解为 `原语级内部最小方法`；它们作用于 `任务虚拟存在`，而不是直接改世界树。

必须固定以下 16 条：

1. `学习方法` 不是一开始就以完整形态出现。
2. `学习方法` 的生成链，统一固定为：

```text
学习目标
    ↓
学习需求
    ↓
原函数集合
    ↓
任务管理任务组合原函数 / 元函数
    ↓
二次特征判断组合效果
    ↓
获得候选学习方法
    ↓
满足可用判据后获得正式学习方法节点
```

3. `学习目标` 只描述最终方法状态，不直接等于可执行需求。
4. `学习需求` 是为了达到目标状态必须满足的中间条件集合；对任务管理学习主链而言，顶层学习需求应优先收口为 `让方法可用`。
5. `学习需求` 不直接生成完整学习方法，而应先生成 `原函数需求集合`。
6. `原函数` 是最小学习动作单元，不等于完整学习方法。
7. `元函数` 是原函数的组合控制规则，不直接替代原函数做具体学习动作。
8. `任务管理任务` 在这一层中的职责是 `学习程序编排者`，而不是“万能学习者”。
9. `候选学习程序` 是一次学习任务在当前轮临时组合出的执行程序草案。
10. `候选学习方法节点` 是已经具备部分可用性、可复用、但尚未进入正式优先调用层的方法节点。
11. `正式学习方法节点` 必须先满足 `有无 -> 可用`，再根据风险、边界、适用范围与效果进入 `优选/默认执行` 层。
12. `二次特征` 不是展示用摘要，而是学习程序的正式裁判输入。
13. `二次特征` 必须直接驱动 `继续执行 / 补样本 / 拆分候选 / 降权 / 试运行 / 转正 / 暂停 / 废弃` 等程序改写决策。
14. `方法头节点` 允许先以 `目标特征类型` 为核心字段占位；学习模块后续负责将其补全为 `可用、可执行、可迁移、可优选` 的方法节点。
15. 长期反复有效的原函数组合允许压缩沉淀为新的 `沉淀原函数`，从而扩大学习工具箱。
16. 整条链的最终产物不是单一结果，而至少包括：`候选学习程序`、`候选学习方法节点`、`正式学习方法节点` 三层成果。

可收敛为一句话：

`学习目标规定方向，学习需求规定“让方法可用”这一主缺口，原语级内部最小方法提供动作，元函数提供组合，任务管理任务负责编排，二次特征负责判断，可执行组合逐步沉淀为可用方法，再进一步筛出优选方法。`


## 4. 统一生成链

后续统一按以下主链理解学习方法生成：

```text
学习目标
-> 目标状态描述
-> 学习需求集合
-> 原函数需求集合
-> 可用原函数匹配
-> 元函数组合
-> 候选学习程序
-> 执行并记录过程
-> 提取二次特征
-> 判断程序效果
-> 调整程序
-> 沉淀为候选学习方法节点
-> 满足可用判据后进入正式方法存在面
-> 再经效果比较进入优选方法层
```

必须固定：

1. `学习目标` 在前，`学习需求` 在后。
2. `原函数匹配` 在前，`元函数组合` 在后。
3. `二次特征裁判` 在程序执行之后，并对下一轮程序改写有直接约束力。
4. `候选学习程序` 与 `候选学习方法节点` 不得混写为同一个层级对象。
5. `正式学习方法节点` 只能来自已经满足最小可用判据的候选节点，不得跳过候选层直接登记。


## 5. 学习目标与学习需求

### 5.1 学习目标

`学习目标` 回答的是：

- 想让哪一个方法头或方法片区变成什么状态
- 想让哪一类学习结果达标
- 想让哪一类失败率、风险值、适用范围或默认执行资格发生怎样的变化

典型学习目标包括：

- `把某个方法头节点补全为可用方法`
- `让某类新特征拥有可用处理方式`
- `把某个候选方法补到可执行`
- `把某个已可用方法推进到优选`
- `让某个方法可以迁移到相似场景`
- `降低某类学习任务的失败率`

必须固定：

- `学习目标 = 最终方法状态`
- 学习目标是方向层，不直接等于执行层需求。

### 5.2 学习需求

`学习需求` 回答的是：

- 为了达到该学习目标，系统还必须补足哪些中间条件

例如：

```text
学习目标：
    把方法头_处理新特征A 补全为可用方法

顶层学习需求：
    1. 让方法头_处理新特征A 变为可用方法

为满足该主需求而触发的学习动作/原语级需求：
    1. 收集新特征A的正样本
    2. 收集新特征A的负样本
    3. 比较成功样本和失败样本差异
    4. 生成候选条件
    5. 生成候选动作
    6. 预测候选动作结果
    7. 执行沙箱验证
    8. 判断是否已可用、风险值、适用边界
```

必须固定：

- `学习需求 = 为达到目标状态而必须满足的中间条件集合`
- `任务管理任务` 对外生成的顶层学习需求，应优先收口为 `让当前待学习方法头可用`
- 若当前方法为空，或当前方法头尚未具备 `动作 + 条件 + 结果` 的最小可用骨架，也应继续收口为 `让当前待学习方法头可用`
- 学习需求应可被继续拆解为 `原函数需求集合`
- 学习需求不得直接跳过原函数层生成完整学习方法


## 6. 原函数分层

### 6.1 总定义

`原函数` 是最小学习动作单元，是可被任务管理任务拿来组合的最小齿轮。

典型原函数包括：

- `收集样本`
- `读取特征`
- `比较特征`
- `提取共同点`
- `提取差异点`
- `生成候选条件`
- `生成候选动作`
- `生成候选结果`
- `执行沙箱验证`
- `计算成功率`
- `计算失败率`
- `计算可用证据`
- `计算迁移度`
- `标记边界`
- `回滚候选方法`
- `申请进入可用层`
- `申请进入优选层`

必须固定：

- 原函数不等于完整学习方法。
- 学习方法是原函数经过元函数组织后的可执行程序结果；后续再逐步进入可用层与优选层。

### 6.2 根原函数

`根原函数` 是系统内置、不能再依赖学习产生的最小学习能力。

例如：

- `读取节点`
- `读取特征`
- `比较两个特征值`
- `记录样本`
- `统计次数`
- `计算比例`
- `判断是否超过阈值`
- `创建候选节点`
- `标记失败`
- `标记成功`

### 6.3 需求原函数

`需求原函数` 是根原函数在当前学习需求下形成的参数化版本。

例如当前目标是学习 `新特征A` 的处理方法，则可构造：

- `读取新特征A样本`
- `比较新特征A正负样本差异`
- `提取新特征A变化区间`
- `统计新特征A导致任务失败的频率`

必须固定：

- 需求原函数本质上应优先视作根原函数的参数化展开。
- 若某个“原函数”完全从零构造且无法回落到根原函数，则它应被视作尚未成熟的新学习方法，而不是正式原函数。

### 6.4 沉淀原函数

当某些原函数组合长期稳定时，允许压缩沉淀为新的 `沉淀原函数`。

例如：

```text
读取样本
-> 比较正负样本
-> 提取共同条件
```

若该组合长期稳定，则允许沉淀为：

```text
从正负样本提取候选条件
```

必须固定：

- 沉淀原函数来源于长期稳定组合，而不是手工任意命名。
- 沉淀后应保留来源链，确保其可追溯到原函数组合历史。


## 7. 元函数与任务管理任务的职责

### 7.1 元函数

`元函数` 负责组合，不直接替代原函数做事。

元函数最少应支持：

- `顺序组合`
- `条件分支`
- `循环迭代`
- `并行尝试`
- `权限门控`
- `失败回滚`
- `候选选择`
- `转入可用层`
- `转入优选层`

例如：

```text
顺序组合(
    收集正样本,
    收集负样本,
    比较正负样本,
    生成候选条件,
    沙箱验证,
    计算可用证据
)
```

```text
循环组合(
    动作 = 收集更多样本,
    退出条件 = 样本覆盖度 >= 阈值
)
```

```text
分支组合(
    如果 冲突度过高 -> 拆分方法头节点,
    如果 样本不足 -> 继续收集样本,
    如果 已满足可用判据 -> 申请试运行
)
```

### 7.2 任务管理任务

`任务管理任务` 在本文中的职责，统一固定为：

1. 接收学习目标
2. 衍生学习需求
3. 查找已有学习方法
4. 若无合适方法，则选择或构造原函数
5. 使用元函数组合原函数
6. 生成候选学习程序
7. 执行候选学习程序
8. 提取并读取二次特征
9. 调整组合
10. 满足可用判据后沉淀为学习方法节点，并继续竞争优选层

必须固定：

- `任务管理任务` 是学习程序编排者，不是“亲自学习的一切动作总和”。
- `原函数` 负责做事，`元函数` 负责组织，`任务管理任务` 负责选择、编排、改写与收口。


## 8. 二次特征的裁判职责

### 8.1 总定义

`二次特征` 不是原始特征，而是对样本关系、方法结构、验证效果、收敛趋势的判断性特征。

例如原始特征可能是：

- `特征A的值`
- `状态B的区间`
- `动作C的结果`
- `样本D是否成功`

而二次特征可能是：

- `样本覆盖度`
- `样本冲突度`
- `条件可用证据`
- `动作有效率`
- `失败集中度`
- `边界清晰度`
- `迁移相似度`
- `过拟合风险`
- `学习收敛度`
- `方法退化趋势`

必须固定：

- 二次特征负责回答“当前组合是否有效、是否需要改写”。
- 二次特征不得只作为展示字段存在。

### 8.2 七类二次特征

后续至少固定以下七类：

#### 8.2.1 样本类

- `正样本数量`
- `负样本数量`
- `边界样本数量`
- `样本覆盖度`
- `样本新鲜度`
- `样本可信度`
- `样本重复度`
- `样本污染风险`

用途：

- 判断是否可以开始形成假设
- 判断是否还需要继续收集样本
- 判断是否足以支持试运行或转正

#### 8.2.2 假设类

- `假设冲突度`
- `假设复杂度`
- `假设覆盖度`
- `假设可解释度`
- `假设压缩度`

用途：

- 判断候选条件是否过宽或过窄

#### 8.2.3 方法结构类

- `边界完整度`
- `回滚完整度`
- `依赖复杂度`

用途：

- 判断方法头节点是否已经补全
- 判断候选方法是否允许进入沙箱

#### 8.2.4 验证类

- `验证次数`
- `成功率`
- `失败率`
- `连续成功次数`
- `连续失败次数`
- `可用证据`
- `退化率`

用途：

- 判断是否允许低权限试运行
- 判断是否允许转正
- 判断是否需要回退

#### 8.2.5 迁移类

- `相似场景命中率`
- `迁移成功率`

用途：

- 判断迁移范围应扩大还是收缩

## 9. 二次特征驱动的程序改写

后续统一固定：

```text
二次特征 -> 决策 -> 原函数增删改 -> 学习程序变化
```

例如：

```text
如果 样本覆盖度低：
    添加“收集样本”或“收集边界样本”原函数

如果 假设冲突度高：
    添加“拆分样本簇”或“拆分候选方法”原函数

如果 条件完整度低：
    添加“提取共同条件”或“补条件模板”原函数

如果 失败集中度高：
    添加“失败原因分类”原函数

如果 稳定度高 且 风险值低：
    添加“申请试运行”或“申请转正”原函数

如果 迁移度高 且 边界清晰：
    添加“迁移验证”原函数

如果 风险值高：
    添加“降低权限”或“回滚候选”原函数
```

必须固定：

- `二次特征` 的价值不在于被展示，而在于直接驱动程序改写。
- `候选学习程序` 不是一次性画死的固定流程，而应允许边执行边调形。


## 10. 三层成果

### 10.1 候选学习程序

`候选学习程序` 是一次学习任务当前轮为补全某个方法头节点而临时组合出的流程草案。

它的特点是：

- 任务相关
- 当前轮有效
- 还不能直接视为通用方法

### 10.2 候选学习方法节点

当某个候选学习程序在多轮、多次、多个相似头节点上持续有效时，可升级为 `候选学习方法节点`。

它的特点是：

- 已表现出可复用性
- 仍需继续验证边界、风险与迁移性
- 已可进入方法链，但仍处于候选态

### 10.3 正式学习方法节点

只有当候选方法满足以下条件后，才允许转正为 `正式学习方法节点`：

- `稳定有效`
- `边界清楚`
- `失败率可接受`
- `风险可控`
- `具备迁移能力或具备明确禁止迁移边界`
- `可回滚`

必须固定：

- 正式学习方法节点必须来自候选层，不得越级产生。


## 11. 节点与结构要求

后续至少允许并推荐显式化以下节点类型：

- `学习目标节点`
- `学习需求节点`
- `原函数节点`
- `元函数节点`
- `学习程序节点`
- `二次特征节点`
- `候选学习方法节点`
- `正式学习方法节点`

它们之间的关系统一按以下口径理解：

```text
学习目标节点
    └── 衍生多个学习需求节点

学习需求节点
    └── 需要多个原函数节点

任务管理任务
    └── 调用元函数组合原函数节点

学习程序节点
    └── 执行后产生过程记录

过程记录
    └── 提取二次特征节点

二次特征节点
    └── 判断学习程序是否有效

有效程序
    └── 沉淀为候选学习方法节点

稳定候选
    └── 转正为正式学习方法节点
```

## 12. 一个标准例子

假设学习目标为：

```text
把方法头_处理新特征A 补全为稳定方法
```

则可按以下链条推进：

1. 衍生学习需求：

```text
样本收集需求
差异比较需求
共同条件提取需求
候选动作生成需求
结果预测需求
验证需求
风险评估需求
```

2. 匹配原函数：

```text
收集新特征A样本
读取样本状态
比较成功样本与失败样本
提取共同条件
生成候选动作
预测结果状态
沙箱验证
计算稳定度
计算风险值
```

3. 用元函数组合：

```text
顺序组合：
    收集样本
    比较样本
    提取条件
    生成动作
    预测结果
    沙箱验证
    计算二次特征

条件分支：
    如果样本不足 -> 回到收集样本
    如果冲突过高 -> 拆分候选方法
    如果稳定度达标 -> 申请试运行
    如果风险过高 -> 降低权限或废弃
```

4. 执行后得到二次特征：

```text
样本覆盖度 = 40
假设冲突度 = 75
稳定度 = 30
风险值 = 60
```

5. 系统判断：

```text
样本不足，冲突过高，不允许试运行
```

6. 程序调整为：

```text
增加边界样本收集
增加样本聚类
增加失败原因分类
生成多个候选方法
```

7. 再次执行后：

```text
样本覆盖度 = 85
假设冲突度 = 20
稳定度 = 78
风险值 = 25
```

8. 任务管理任务可判定：

```text
允许进入低权限试运行
```

9. 若多次试运行稳定，则允许沉淀为：

```text
候选学习方法节点 -> 正式学习方法节点
```


## 13. 对当前工程的收口要求

当前工程后续应按以下方向逐步收口：

1. `学习任务模块` 应显式区分 `学习目标` 与 `学习需求`。
2. `学习调度模块` 应增加 `原函数需求 -> 原函数匹配 -> 元函数组合` 的中间层，而不是只维护粗粒度学习阶段。
3. `学习执行模块` 应允许 `候选学习程序` 被二次特征驱动动态改写，而不是只跑固定流程。
4. `任务管理任务模块` 应显式承担 `学习程序编排者` 的职责。
5. `方法类 / 方法节点` 应允许区分 `方法头节点`、`候选学习方法节点`、`正式学习方法节点` 三层状态。
6. `二次特征应用模块` 应优先服务于学习程序裁判，而不是只输出观察摘要。
7. `学习写回` 应区分写回 `候选程序过程记录`、`候选学习方法节点` 与 `正式学习方法节点` 三类落账对象，避免混账。


## 14. 收口结论

后续统一按以下定义执行：

`学习方法的生成，不是由学习模块凭空产生，而是由任务管理任务从学习目标出发，衍生学习需求，根据学习需求选择或构建原函数，再通过元函数组合成候选学习程序，并用二次特征持续判断、修正、验证，最终沉淀为稳定的正式学习方法节点。`

更简洁地说：

`目标生需求，需求取原函数，任务管理编程序，二次特征做判断，稳定程序成方法。`
