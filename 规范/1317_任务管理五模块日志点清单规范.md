## 2. 核心结论

必须固定以下结论：

1. `任务管理任务` 的日志验收，不得只看最后一个结论词，必须覆盖 `入口点 -> 分支点 -> 产出点 -> 收口点`。
2. 对五模块做逻辑验收前，必须先预测“这一轮按规范应该出现什么运行时数据”，再去读日志。
3. 若日志看不出分支是否命中，默认先补正式链路日志，不得先补场景化验收。
4. 每个模块都至少要留下：`触发信息`、`关键输入`、`关键中间态`、`关键产出/结论`。
5. `日志清单` 的作用是证明“五模块流程是否按规范命中”，不是替代业务摘要或 UI 提示。

一句话收口：

`五模块日志验收 = 先预测每段应该冒出什么真实数据，再把这些数据打到正式运行链上，最后按模块顺序核对是否命中。`

## 3. 总体日志链

任务管理任务的最小日志观察链固定为：

```text
入口日志
-> 筹办日志
-> 可执行判断日志
-> 能执行判断日志
-> 执行日志
-> 反馈日志
-> 一步治理结果日志
```

### 3.1 固定要求

必须固定：

1. `入口日志` 必须先于五模块日志出现。
2. `能执行判断 = 不允许` 时，`执行日志` 可以只留下“未进入执行模块”的边界记录，但不能没有 `反馈日志`。
3. `反馈日志` 必须能回指本轮前面模块的关键结论，否则无法判断收口是否来自真实运行链。
4. `一步治理结果日志` 必须能看出它来自哪一轮任务管理治理，而不是孤立输出一段摘要文本。

## 4. 五模块关键点清单

### 4.1 模块一：筹办

筹办模块最少要验证四类关键点：

1. `判域点`
2. `主缺口点`
3. `找方法点`
4. `找条件/分流点`

#### 4.1.1 预测要求

至少要先预测：

- 本轮宿主任务是谁
- 本轮落在哪个治理域
- 本轮主缺口最可能是什么
- 本轮应该走 `绑定方法 / 补结构 / 补条件 / 重筹办 / 观察试探` 中哪条分流

#### 4.1.2 最小日志字段

- `轮次ID` 或等价可关联标识
- `宿主任务ID`
- `当前步骤ID`
- `当前方法ID`
- `当前功能域`
- `主缺口类型`
- `候选方法摘要`
- `条件缺口摘要`
- `筹办分流结论`

#### 4.1.3 足够与不足

足够：

- 能看出本轮到底是 `缺方法` 还是 `缺条件`
- 能看出最终分流到了哪一类筹办结论

不足：

- 只写“进入筹办”
- 只写“筹办完成”
- 只写一个泛化摘要，却看不出分流依据

### 4.2 模块二：可执行判断

可执行判断模块最少要验证四类关键点：

1. `方法需求位命中点`
2. `方法存在层点`
3. `方法可用层点`
4. `本能能力缺口点`

#### 4.2.1 预测要求

至少要先预测：

- 当前主缺口会被压成哪个 `方法需求位`
- 当前局部最可能是 `局部可执行` 还是 `局部不可执行`
- 若不可执行，最可能缺的是 `方法挂点 / 条件 / 原语 / 学习承接`

#### 4.2.2 最小日志字段

- `轮次ID`
- `当前方法需求位`
- `方法存在层结果`
- `方法可用层结果`
- `当前本能能力缺口类型`
- `缺失原语位图` 或等价摘要
- `是否建议转学习`
- `局部可执行结论`

#### 4.2.3 足够与不足

足够：

- 能看出“局部能不能做”的依据来自哪里
- 能区分 `缺条件` 与 `缺原语/缺学习`

不足：

- 只写“不可执行”
- 没有需求位、缺口位、学习建议这些中间态

### 4.3 模块三：能执行判断

这里的 `能执行判断` 按规范等价于 `允许执行判断`。

能执行判断模块最少要验证三类关键点：

1. `根层门控点`
2. `上下文门控点`
3. `最终允许/拒绝点`

#### 4.3.1 预测要求

至少要先预测：

- 本轮根层是 `允许普通执行` 还是 `仅等待/仅收束/禁止继续`
- 如果不允许，拒绝应该主要来自哪一层
- 本轮是否应直接跳过执行模块，转入反馈模块

#### 4.3.2 最小日志字段

- `轮次ID`
- `最近根层重判结果`
- `上下文门控摘要`
- `局部可执行结论`
- `是否允许进入普通执行链`
- `拒绝来源层级`
- `主拒绝原因类型`
- `回流建议`

#### 4.3.3 足够与不足

足够：

- 能判断本轮是“局部可执行但不被放行”，还是“局部本身不可执行”
- 能判断“不允许”后是否正确跳过执行模块

不足：

- 只写“门控拒绝”
- 没有来源层级、主拒绝原因、回流建议

### 4.4 模块四：执行

执行模块最少要验证四类关键点：

1. `执行承接方式点`
2. `步骤/结果落点`
3. `子任务派生点`
4. `等待态落点`

#### 4.4.1 预测要求

至少要先预测：

- 本轮是 `直接执行 / 进入等待 / 派生补条件子任务 / 派生学习子任务`
- 若允许执行，最关键的步骤或结果应该落到哪里
- 若不允许执行，本模块是否只留下边界日志而不真正推进

#### 4.4.2 最小日志字段

- `轮次ID`
- `执行承接方式`
- `当前步骤ID`
- `当前结果ID`
- `执行下发摘要`
- `等待原因`
- `补条件子任务ID`
- `学习子任务ID`
- `状态迁移摘要`

#### 4.4.3 足够与不足

足够：

- 能看出到底是“真的执行了”，还是“只转等待/派生”
- 能看出子任务是不是补条件还是学习

不足：

- 只写“执行完毕”
- 看不出步骤、结果、子任务、等待原因的真实变化

### 4.5 模块五：反馈

反馈模块最少要验证四类关键点：

1. `反馈来源点`
2. `结果写回点`
3. `下一步去向点`
4. `一步治理结果收口点`

#### 4.5.1 预测要求

至少要先预测：

- 本轮反馈主要来自 `执行反馈 / 学习反馈 / 外部反馈 / 门控拒绝`
- 本轮更新后最可能的 `当前缺口类型`
- 本轮收口后的 `当前下一步去向候选`
- 一步治理结果里最关键的字段应该回成什么方向

#### 4.5.2 最小日志字段

- `轮次ID`
- `反馈来源类型`
- `最近结果ID`
- `最近反馈类型`
- `最近反馈摘要`
- `当前缺口类型`
- `当前下一步去向候选`
- `最近总控结果`
- `一步治理结果摘要`

#### 4.5.3 足够与不足

足够：

- 能把反馈来源和最终收口关联起来
- 能看出下一轮为什么会继续/等待/补条件/转学习

不足：

- 只有最终摘要，没有来源和写回链

## 6. 字段流转观察要求

为了证明 `1214 输入 -> 五模块 -> 1213 输出` 真正走通，日志观察至少要覆盖下面这些字段的流转：

1. `宿主任务ID`
   必须从入口持续可追到反馈收口。

2. `当前方法ID / 当前步骤ID`
   必须能看出是否在筹办或执行阶段发生了切换、复用或新建。

3. `当前缺口类型`
   必须能看出它在 `筹办 -> 可执行判断 -> 反馈` 三段之间是否发生变化。

4. `当前下一步去向候选`
   必须能看出它是否来自真实反馈收口，而不是直接抄入输入上下文。

5. `最近反馈类型 / 最近反馈摘要`
   必须能看出它是由 `执行/学习/外部/门控拒绝` 哪一类反馈形成的。

## 7. 一条最小日志验收示例

假设本轮输入是：

- `宿主任务 = T100`
- `当前方法挂点缺失`
- `根层允许继续`

则最小预测应是：

1. 入口日志出现 `T100`、当前方法为空、根层允许继续。
2. 筹办日志出现 `主缺口 = 缺方法挂点`，并分流到 `绑定方法/补结构`。
3. 可执行判断日志出现 `局部不可执行`，原因偏向 `缺方法挂点`，而不是 `缺条件`。
4. 能执行判断日志可以显示“暂不进入普通执行链”或“等待先补结构”，但拒绝原因必须与前面缺口一致。
5. 执行日志不能误生成 `补条件子任务`。
6. 反馈日志要能看出：本轮收口成 `补结构/绑定方法方向`，而不是 `补条件方向`。

如果日志里只看到了：

- “筹办完成”
- “执行完成”
- “反馈完成”

则默认视为 `日志不足，不能据此判定规范已命中`。

## 8. 禁止的日志做法

后续默认禁止把以下做法当成足够的任务管理日志验收：

1. 只在最外层打一条“任务管理开始/结束”。
2. 只记字符串摘要，不记关键 ID、关键中间态和分流结论。
3. 只看最后一步治理结果，不核对前面五模块的真实生成链。
4. 只靠 UI 摘要或控制面板词语推断内部已按规范运行。
