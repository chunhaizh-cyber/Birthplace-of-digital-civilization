# 0960 四域方法树自动生长闭环规范 v0

## 1. 目标

本文用于把四域方法树的“自动生长”收成一条可执行闭环。

必须固定：

- 四域方法树仍然可用，但其定位是 `任务管理的工程方法树 / 经验复用树 / 本域兜底写回树`
- 四域方法树不是根语义裁决树，不得替代安全/服务根方向判断
- 运行时先匹配本域方法树，不先调用兜底
- 匹配不到时，才进入本域兜底
- 本域兜底成功后，允许把成功路径写回本域方法树
- 写回后的新枝允许先作为低成熟度方法头存在
- `方法来源` 与 `方法成熟度阶段` 必须分开落账
- 只有在相似任务上稳定复现后，才允许把成熟度阶段从 `运行期临时 / 反推候选` 推进到 `已验证稳定 / 组合成熟`
- 筹办树的自动生长，是任务层经验积累与智力进化的主要工程落点

本文直接回答：

- 四棵树在运行时如何匹配
- 匹配失败时如何发芽
- 发芽后如何记录稳定复现
- 何时升格、何时只保留临时枝

## 3. 运行时总闭环

四域方法树的自动生长，统一固定成以下 4 步：

```text
1. 根层先裁决当前方向是否允许继续推进
2. 允许推进后，再用任务管理上下文匹配本域树
3. 匹配不到 -> 走本域兜底
4. 已被外层确认有效的成功路径，才允许写回并继续稳定复现
```

可直接收成一句话：

```text
根层先裁决 -> 再匹配本域树 -> 匹配不到就走本域兜底 -> 外层确认有效后再写回成新枝 -> 稳定复现后再升格
```

其中在任务控制语义上应继续固定：

```text
筹办不只是铺下一步路，也是把成功筹办经验沉淀进方法树、推动任务层智力进化的主域
```

并且必须补充固定：

```text
四域方法树回答的是“当前域里优先复用什么管理路径”；它不回答“根层什么方向才是最终正确”。
```


## 4. 匹配规则

### 4.1 运行时主入参

四域方法树的运行时主入参，当前工程上仍然以 `任务虚拟存在` 为核心承载面；
但在最新体系下，必须理解为：

```text
根方向裁决通过后的任务管理上下文
-> 以任务虚拟存在 / 方法虚拟存在 / 当前上下文字段作为工程载体
-> 进入四域方法树匹配
```

因此必须固定：

1. 没有经过根层方向裁决的路径，不得直接进入四域树当成“可匹配经验”。
2. `任务虚拟存在` 仍是当前工程主承载面，但后续输入重心应逐步转向 `任务管理任务最小一次特征组`。
3. 当前匹配输入里使用的预算、稳定性、依赖缺失、完成判据等字段，只能视为工程代理量，不得冒充根语义本体。

最小匹配上下文至少包括：

- 当前功能域
- 任务虚拟存在
- 当前方法首节点（如果有）
- 当前安全值 / 服务值 / 需求值 / 预算
- 当前步骤、当前方法、最近结果、当前缺口、当前下一步去向等任务管理上下文
- 最近结果稳定性、最近失败、风险变化、检查点状态

### 4.2 匹配单位

匹配单位固定为：

```text
首节点 -> 条件节点 -> 结果节点 -> 下一首节点
```

不允许引入第二套树边结构。

### 4.3 最小匹配流程

最小可执行匹配流程固定为：

```text
域根默认结果节点
-> 扫描其下所有候选首节点
-> 在候选首节点下扫描条件节点
-> 用条件主键命中某个条件节点
-> 命中后取对应结果节点
-> 返回命中的首节点作为建议执行首节点
```

默认条件节点和默认续接首节点只承担“占位骨架”和“默认挂载点”职责，不视为具体命中的成熟枝条。


## 5. 发芽规则

当本域匹配失败时，必须先走本域兜底。

若本域兜底在某个具体任务上走通一条局部路径，则只在以下前提同时成立时，才允许把成功路径写回本域方法树：

1. 该路径没有违背安全/服务根方向约束
2. 该路径的有效性已经被外层任务执行与结果判断确认
3. 写回的是“本域管理路径经验”，不是“根层最终结论”

写回后的最小结构固定为：

```text
父结果节点 -> 新首节点 -> 条件节点 -> 结果节点
```

### 5.1 挂载位置

写回位置固定为：

- 若已有“最接近的父结果节点”，则优先挂到该父结果节点下面
- 若没有更近的父结果节点，则挂到域根默认结果节点下面

### 5.2 动作句柄白名单

自动生长过程中，新枝的动作句柄只允许来自：

1. `枚举_动作句柄类型::本能函数ID`
2. `枚举_动作句柄类型::动作序列主键`

不允许长出第三种句柄类型。

### 5.3 初次落账口径

新枝第一次写回时，必须把“来源”和“成熟度阶段”拆开处理，不能再把二者混写成一个字段。

最小纪律：

- 来源字段仍写到 `方法首节点主信息类::来源`
- 成熟度阶段、可信度与权限优先写到 `方法虚拟存在` 特征
- 若当前工程仍使用 `枚举_方法沉淀来源::运行期临时 / 反推 / 组合` 作为兼容来源值，也不得再把它们当成成熟度阶段的唯一真值

建议的初次落账组合：

- 本域兜底直接长出的新枝
  - 来源：兼容现实现可先写 `运行期临时`
  - 成熟度阶段：`运行期临时`
- 自我动作动态强证据直接支持出生的新枝
  - 来源：兼容现实现可先写 `运行期临时`
  - 成熟度阶段：`运行期临时` 或 `反推候选`
- 本能直接沉淀的路径
  - 来源：`本能`
  - 成熟度阶段：`已验证稳定`
- 由复杂因果链压出的新枝
  - 来源：兼容现实现可先写 `反推` 或 `运行期临时`
  - 成熟度阶段：`反推候选`

### 5.4 具体字段名建议与写回位置

四域自动生长在工程落地时，优先按以下字段与位置写回：

1. 方法首节点身份字段
   - `方法首节点主信息类::来源`
   - `方法首节点主信息类::作用域`
2. 方法虚拟存在标准特征
   - 复用：`特征_方法沉淀来源_`
   - 复用：`特征_方法可靠度_Q10000_`
   - 复用：`特征_方法稳定出现次数_`
   - 新增优先建议：`特征_方法成熟度阶段_`
   - 新增优先建议：`特征_方法允许自动召回_`
   - 新增优先建议：`特征_方法允许默认执行_`
   - 若需要和“稳定出现”拆语义，再补：`特征_方法稳定复现次数_`
3. 写回位置清单
   - `四域方法树注册模块_v0.ixx::创建子首节点_`：写来源字段，初始化方法虚拟存在
   - `方法类::初始化方法虚拟存在信息(...)`：初始化来源镜像、成熟度与权限缺省值
   - `方法类::同步方法沉淀来源标准特征(...)`：同步来源镜像并回补成熟度/权限缺省值
   - `方法类::同步方法成熟度与权限标准特征(...)`：稳定复现后推进成熟度、自动召回和默认执行权限
   - `方法类::同步方法可靠度标准特征(...)`：同步可信度Q
   - `任务控制总控模块_v0.ixx`：决定“优先回补既有方法”还是“允许新生低成熟度方法头”，并在判定时优先读 `读取方法成熟度阶段标准特征(...)`
   - `自我线程类::私有_接入学习结果到方法成长判据_v0(...)`：把学习成长后的来源、成熟度、权限与可靠度同步到学习任务事实，作为展示与回放口径


## 6. 结构动作

四域方法树的自动生长，只允许以下 5 类结构动作：

### 6.1 发芽

条件：

- 本域无命中路径
- 兜底成功一次

动作：

- 新增一组 `首 + 条 + 果`

### 6.2 抽枝

条件：

- 已有路径前半段常成功
- 后半段经常需要补同一步

动作：

- 在已有结果节点下面追加新的下一首节点
- 再给这个下一首追加条件节点与结果节点

### 6.3 分叉

条件：

- 同一首节点在不同条件下应走不同后续路

动作：

- 在同一首节点下增加多个条件节点

### 6.4 归并

条件：

- 多条枝条主干相同，只是条件或局部步骤不同

动作：

- 抽出共同首节点
- 把原枝条退成条件分支

### 6.5 修剪

条件：

- 长期不稳定
- 几乎不用
- 安全上已不再适合

动作：

- 不物理删除
- 只降权、隐藏或退到备用区


## 7. 稳定复现与升格规则

方法树成长的真正驱动力，不是“成功过一次”，而是“结果是否稳定复现”。

这里必须额外固定：

- `动作动态复现` 是高质量证据来源，不等于成熟度已经稳定
- `稳定复现` 才是成熟度升格的主裁判
- 成熟度升格优先改 `方法虚拟存在` 特征，不强制改历史来源字段

必须固定：

- 兜底成功让新枝出生
- 稳定复现让新枝长稳
- 有适配旧方法时，应优先回补旧方法，不必强行出生平行新枝

最小升格阈值固定为：

- `1` 次成功写回：允许出生 `运行期临时`
- `3` 次稳定复现：达到候选枝阈值，可从 `运行期临时` 升到 `反推候选`
- `5` 次稳定复现且结果确认、安全边界稳定：可升到 `已验证稳定`
- `已验证稳定` 后，若可复用结构被长期压缩和复现，再升到 `组合成熟`

阈值后续允许配置化，但在未配置前，默认使用 `3 / 5`。


## 8. 四域边界

自动生长必须遵守“本域优先、本域兜底、本域沉淀”。

并且必须继续固定：

- 不得跨过根方向裁决，直接把四域树命中当成语义正确
- 不得把尚未通过外层方向确认的局部成功路径直接沉淀为稳定经验
- 不得让四域树为了复用成功率而反向修改安全/服务根约束

### 8.1 筹办树只吸收

- 新的细分方式
- 新的补条件方式
- 新的初始绑定方式
- 新的等待学习挂接方式
- 新的初始检查点布置方式

### 8.2 维护树只吸收

- 稳定的一步推进链
- 检查点插入模式
- 可靠度更新模式
- 继续维护 / 转纠偏 / 转收束 / 回筹办 的判别套路

### 8.3 纠偏树只吸收

- 替代方法模式
- 回退模式
- 补条件任务模式
- 直接收束模式

### 8.4 收束树只吸收

- 完成判定模式
- 失败收口模式
- 最终反馈模式
- 清理与归档顺序

禁止跨域混长。


## 9. 工程落地纪律

### 9.1 模块职责

自动生长相关能力按以下模块分层：

- `四域方法树匹配器_v0`：负责条件主键生成与树内匹配
- `四域方法树写回器_v0`：负责发芽、抽枝、分叉的最小写回
- `四域方法树生长判定器_v0`：负责稳定复现统计与成熟度升格
- `任务控制总控模块_v0`：负责把匹配、兜底、写回、升格串成运行时闭环

### 9.2 注册模块要求

[四域方法树注册模块_v0.ixx](D:\鱼巢\四域方法树注册模块_v0.ixx) 中的 `创建子首节点_` 必须允许显式携带 `枚举_方法沉淀来源`，不得把所有新枝写死为 `组合`；同时应在创建后走 `方法虚拟存在` 初始化链，补齐成熟度与权限特征的最小落账。

### 9.3 运行时闭环要求

[任务控制总控模块_v0.ixx](D:\鱼巢\任务控制总控模块_v0.ixx) 必须至少完成以下闭环：

```text
匹配本域树
-> 若命中则优先复用
-> 若未命中则允许继续执行本域兜底
-> 本域执行成功后优先判断是否回补已有方法，否则写回低成熟度方法头
-> 命中或写回后的方法枝都要累计稳定复现
-> 满足阈值后再升级成熟度阶段与执行权限
```


## 10. 最小验收标准

视为满足本规范的最小实现，应至少做到：

1. 域根默认结果节点下面可以新增非默认子枝
2. 本域匹配器可以区分“默认骨架”与“已沉淀具体枝条”
3. 写回器可以把成功路径回写为 `首 + 条 + 果`
4. 写回器可以区分“来源字段”与“成熟度阶段字段”的落账位置
5. 生长判定器可以按稳定复现次数完成最小成熟度升格
6. 当已有适配方法头存在时，运行时闭环可以优先回补旧方法


## 11. 一句话收束

> 四域方法树的自动生长，不是凭空发明新方法，而是在“本域匹配失败”时调用既有兜底函数，走通一条局部路径后，把它按 `首节点 + 条件节点 + 结果节点` 的结构写回对应域树里；第一次只要求出生为低成熟度方法头，之后再靠相似任务上的稳定复现逐步提升成熟度与执行权限。
