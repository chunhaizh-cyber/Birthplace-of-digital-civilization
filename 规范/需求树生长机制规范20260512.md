规范编号：020003

父级规范：

```text

020001 需求主信息类规范20260507.md

```

## 1. 核心定义

需求树生长不是初始化，也不是任务执行失败后的兜底。

需求树生长是：

```text
自我状态、任务筹办缺口、任务执行回执、方法缺口、结算结果
    -> 形成更细的需求目标
    -> 通过需求树更新指令入树
    -> 触发任务发起或父任务重筹办
    -> 让下一轮需求选择和任务推进发生变化。
```

一句话：

```text
需求树生长 = 自我把“父需求无法一步满足”的事实，展开成更细、可任务化、可结算的子需求。
```

根需求、安全根需求、服务根需求被初始化出来，只表示自我具备需求系统骨架，不表示需求树已经正常生长。

## 2. 权责边界

需求树归自我线程管理。

```text
自我线程：
    写需求树。
    生成 / 合并 / 满足 / 删除需求节点。
    刷新需求权重。
    根据需求树发出任务发起意图和调度意图。

任务管理线程：
    不直接写需求树。
    通过任务回执、筹办缺口、执行缺口、派生需求消息，把任务域发现的事实上报自我线程。

任务筹办 / 任务执行普通函数：
    不直接写需求树。
    只返回运行存在中的任务层回执，或通过任务管理工作线程形成派生需求消息。

自检线程：
    不直接生成修复需求。
    只发现缺失并报送自我线程；是否入树由自我线程按阶段和策略决定。
```

需求树变化若需要生成方法动作动态，动作来源必须是需求域本能方法，例如：

```text
本能方法_派生需求入树
本能方法_回写需求满足
```

线程只是调度者，不是动作来源。

## 3. 需求节点的最小语义

单个执行需求只表达一个原子目标：

```text
目标宿主 / 被需求存在
目标特征类型
当前状态
目标状态
满足关系
```

这里的“目标宿主 / 被需求存在”不是要被整体改变的对象，而是目标特征依附的存在。
需求改变的不是存在本身，而是：

```text
目标宿主.目标特征类型 -> 目标状态
```

因此，完整需求目标必须能回答：

```text
谁需要：需求主体
特征挂在哪个存在上：目标宿主 / 被需求存在
要改变哪个特征：目标特征类型
该特征要达到什么状态：目标状态
```

规范中历史出现的“目标主体”，若不是指 `需求主体`，均按“目标宿主 / 被需求存在”理解。

对应代码字段：

```text
需求主信息类::被需求存在
需求主信息类::被需求当前状态
需求主信息类::需求状态
需求主信息类::满足关系
需求主信息类::目标特征类型缓存
```

需求节点不保存目标值包，也不把多目标塞进一个主信息。

多目标必须表达为：

```text
父管理需求
├─ 原子执行需求 A
├─ 原子执行需求 B
└─ 原子执行需求 C
```

结构角色由子链决定：

```text
有子需求 = 管理需求
无子需求 = 执行需求
```

当前代码映射：

```cpp
需求类::刷新需求结构角色(...)
需求类::刷新需求及祖先结构角色(...)
```

## 4. 生长入口

需求树只通过统一入口生长：

```cpp
需求类::执行更新指令(
    需求根节点,
    结构_需求树更新指令)
```

允许的更新类型：

```text
新增需求
更新需求
满足需求
删除需求
```

不得绕过更新指令直接拼接需求树，除非是冷启动创建根需求这类初始化动作。

当前代码中的关键步骤：

```text
执行更新指令
    -> 添加子需求节点
    -> 从状态补齐目标宿主和特征缓存
    -> 检查是否与父需求目标重叠
    -> 查找同父直接子需求目标重叠
    -> 合并或新增
    -> 刷新子需求权重
    -> 刷新需求及祖先结构角色
```

## 5. 生长触发源

### 5.1 自我状态触发

自我线程读取：

```text
安全值
服务值
目标安全值
目标服务值
待机状态
时序步长
```

然后刷新安全根需求、服务根需求权重，并选择当前主需求。

当前代码映射：

```cpp
私有_从需求树构建需求列表(...)
私有_选择当前主需求候选_按根权重(...)
```

这一步只让根需求活跃，不一定新增子需求。

### 5.2 任务筹办缺口触发

任务筹办的目标是：

```text
任务状态 -> 就绪
```

筹办不能一步让任务就绪时，不是失败，而是需求树继续生长的入口。

例如：

```text
无确定候选方法
候选方法不可用
候选方法缺执行入口
候选方法条件不满足
参数配对失败
```

这些缺口应由任务管理线程上报为任务层回执或派生需求提示，自我线程再决定是否入树。

筹办自身不得凭空决定生长方向：

```text
一切以当前需求目标和方法查找 / 条件匹配的确定结果为准。
```

### 5.3 任务执行缺口触发

任务执行调用当前方法后，如果输出结果没有满足来源需求，或当前方法执行输出明确给出派生需求特征，则形成派生需求消息。

当前代码映射：

```cpp
私有_构造执行缺失原子需求消息(...)
私有_派生需求执行输出完整(...)
```

执行缺失派生需求必须携带：

```text
父需求
当前状态
目标状态
目标特征类型
目标宿主 / 被需求存在
派生归因
```

缺少当前状态、目标状态或目标特征类型时，属于逻辑错误，应拒绝入树并记录日志 / 弹错。

### 5.4 子需求结算触发

子需求满足后，会影响父需求：

```text
子需求有效截止
子需求不再参与活跃子需求权重分配
父需求权重重新分配
父任务可被请求重筹办
```

父任务状态推进请求由自我线程发给任务管理线程；自我线程不得直接写任务状态。

## 6. 入树前闸门

派生需求入树前必须通过这些检查：

```text
1. 必须存在上级需求。
2. 上级需求不能是需求树结构根。
3. 必须能解析当前状态和目标状态。
4. 必须能解析目标特征类型。
5. 必须能解析目标宿主 / 被需求存在；第一阶段可从状态主体补齐。
6. 必须明确满足关系；缺省为等于。
7. 必须带派生归因，至少能说明来自哪个任务 / 方法 / 缺口。
```

当前代码映射：

```cpp
私有_确保派生需求节点(...)
私有_提示派生需求逻辑错误(...)
```

闸门失败时不得兜底生成模糊需求。

## 7. 去重与合并

同一父需求下，直接子需求去重依据是：

```text
目标宿主 / 被需求存在
目标特征类型
满足关系 / 方向掩码
```

当前代码映射：

```cpp
需求类::查找直接子需求_按目标重叠(...)
需求类::合并需求主信息_已加锁(...)
```

派生归因不是身份键。

```text
同一个目标可以有不同派生来源；
不同派生来源不应导致同父直接子链重复长出多个同目标需求。
```

需求细化优先级：

```text
明确特征值 > 区间 > 方向
```

这个优先级不表示需求主信息保存值包；值和区间仍由状态节点、特征节点和状态比较服务表达。它表示：

```text
当同一目标宿主、同一目标特征类型下存在多个候选需求时，
更具体的目标应优先用于筹办、方法查找和满足判断。
```

实现要求：

```text
1. 不得把明确值需求合并丢失成宽泛方向需求。
2. 宽泛方向需求遇到明确值需求时，应保留更具体目标状态。
3. 当前代码已有方向重叠合并；明确值 / 区间优先级是需要持续补强的规则。
```

## 8. 权重刷新

父需求权重只分配给活跃且阻塞父任务执行的子需求。

活跃子需求条件：

```text
需求有效截止 == 0
是否阻塞父任务执行 == true
```

当前代码映射：

```cpp
需求类::刷新直接子需求权重_已加锁(...)
```

非活跃子需求权重应归零，不继续拉动主需求选择。

## 9. 任务化关系

需求树生长到执行需求后，执行需求可以进入任务承接。

权威关系是：

```text
需求节点 D
    -> 任务信息节点 N
        -> 任务虚拟存在 T
```

不得改成：

```text
需求节点 D -> 任务虚拟存在 T
```

自我线程只发任务发起意图；任务信息节点和任务虚拟存在由任务管理线程提交。

当前代码中需求与任务的绑定统一使用：

```cpp
需求类::绑定对应任务(...)
```

长期要求：

```text
绑定动作由任务管理界面线程在任务创建提交时完成。
```

## 10. 筹办缺口与子需求生长

筹办缺口应生长成能被任务化的原子需求。

示例：

```text
父需求：提高安全值
父任务筹办：找到候选方法，但候选方法不可用
派生子需求：方法状态 = 可用
子任务：让该方法达到可用
```

如果缺口是方法不可用，需求目标应表达为：

```text
方法状态 = 可用
```

而不是模糊表达为：

```text
方法不可用
方法动作骨架状态
```

除非筹办 / 执行回执明确证明真正缺的是动作入口，才可以派生：

```text
方法动作骨架状态 = 已形成
```

方法补齐需求应按缺口事实细化，不能由筹办自带方向。

## 11. 学习需求与探索需求的位置

学习需求不是初始化内容。

学习需求出现的条件是：

```text
内部需求仍有效
任务筹办或执行发现内部信息 / 方法结构 / 条件结果对不足
已有内部记忆无法满足该缺口
```

外界探索需求出现的条件是：

```text
学习需求无法仅靠内部记忆满足
外部信息可能降低不确定性
探索风险可接受
有可执行探索方法
```

因此阶段顺序是：

```text
初始化
-> 需求树正常生长
-> 内部任务闭环
-> 自我苏醒
-> 因果补完
-> 学习需求形成
-> 外界探索需求形成
-> 初步成熟
```

需求树正常生长属于内部循环建立前期，不等同于自我苏醒。

## 12. 满足与结算

需求满足由自我线程提交，任务完成由任务管理线程提交。

两者不能混为一个状态。

```text
任务完成：
    任务管理线程根据任务执行回执推进任务状态。

需求满足：
    自我线程根据任务完成回执、输出事实和需求目标，回写需求当前状态或标记需求满足。
```

需求满足后：

```text
需求有效截止 = 当前时间
累计安全结算 / 累计服务结算更新
最近结算任务主键更新
最近结算时间更新
父级需求重新汇总
下一轮主需求选择受到影响
```

若需求状态变化生成动作动态，动作来源必须是需求域本能方法：

```text
本能方法_回写需求满足
```

## 13. 正常生长闭环

最小闭环：

```text
自我状态
-> 安全 / 服务压力
-> 根需求活跃
-> 当前主需求
-> 任务发起意图
-> 任务管理线程创建任务
-> 任务筹办
-> 发现缺口
-> 派生更细子需求
-> 子需求任务化
-> 子任务完成
-> 父任务重筹办
-> 父任务就绪
-> 父任务执行
-> 来源需求满足
-> 自我状态 / 需求账本改变
-> 下一轮主需求重判
```

需求树越长，不表示系统失败。

```text
需求树变长 = 系统正在把模糊目标拆成更细、可执行、可结算的目标。
```

## 14. 非正常逻辑错误

以下情况不得兜底生成需求：

```text
1. 派生需求缺上级需求。
2. 派生需求上级是需求树结构根。
3. 缺当前状态或目标状态。
4. 缺目标特征类型。
5. 无法确定目标宿主 / 被需求存在且不能从状态主体补齐。
6. 需求直接指向任务虚拟存在而绕过任务信息节点。
7. 自检线程直接生成修复需求。
8. 筹办在没有确定缺口事实时凭空派生学习方向。
```

处理方式：

```text
弹错 / 记录日志 / 拒绝入树 / 标记待完成项。
```

不得用宽泛需求掩盖逻辑错误。

## 16. 已知差距

当前实现已经具备需求树生长骨架，但仍有差距：

```text
1. 明确值 > 区间 > 方向 的细粒度优先级还需要继续补强。
2. 部分旧函数名仍包含“确保需求相关任务”，需按任务管理线程权责边界持续收口。
3. 需求域动作动态方法化尚未完全覆盖所有直接写入点。
4. 学习需求和外界探索需求仍处阶段建设中，不能作为自我苏醒或初步成熟完成证据。
```

这些差距不改变本规范的权威口径。

## 17. 最终定稿

```text
需求树正常生长，是自我线程把内部状态压力和任务回执缺口，转成更细的原子需求，并通过统一更新指令入树、去重、分权、任务化、结算和回写的过程。

任务筹办不能一步让任务就绪时，不是失败，而是生长入口。
任务执行不能满足来源需求时，不是直接闭环失败，而是生成更细需求或重筹办的依据。
需求树越长，表示目标正在被拆解；只有缺口无来源、目标不可解析、状态不完整时，才是逻辑错误。
```
