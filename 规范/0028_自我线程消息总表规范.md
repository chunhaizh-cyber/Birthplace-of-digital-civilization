# 0028 自我线程消息总表与处理规范

## 1. 目标

本文用于把 `自我线程` 应接收哪些消息、这些消息谁先接收、哪些必须上浮到控制面，以及自我线程应如何处理它们，正式写成一张总表。

本文只回答以下问题：

- 自我线程应接受哪些消息类型
- 哪些消息必须先到自我线程，哪些可先到工作线程
- 哪些消息会改变优先级、控制态和任务入口实例化
- 哪些消息可以抢占当前轮次
- 哪些消息原则上不应直接送到自我线程

本文不负责：

- 新增一级职责
- 展开任务管理工作线程内部步骤算法
- 展开方法内部执行细节
- 取代 [0027_自我线程与任务管理工作线程状态流转与事件协议_v0.1规范.md](D:\鱼巢\规范\0027_自我线程与任务管理工作线程状态流转与事件协议_v0.1规范.md) 的状态流转与双状态机定义

必须先固定一句话：

`自我线程接受的消息，只覆盖控制面、根层判断面和全局调度面；不吞步骤细节和方法内部噪声。`

## 3. 核心结论

必须固定以下结论：

1. 自我线程接收的消息，应限定为 `控制消息 / 全局环境消息 / 工作线程状态消息 / 任务推进关键事件消息 / 派生需求消息 / 结果回流消息 / 学习与审计消息` 七类。
2. `控制消息` 和 `全局环境消息` 必须先进入 `自我线程`。
3. `明确绑定到具体任务且仅影响局部流程的外部反馈`，可先进入对应任务管理工作线程；若升级为 `派生需求 / 阻塞变化 / 异常 / 需要改优先级 / 需要实例化新任务`，必须再上报 `自我线程`。
4. `任务推进关键事件消息` 只上浮会改变控制面判断的部分；步骤内部小进度、局部临时变量变化、内部小重试和纯中间状态不得默认上浮。
5. `派生需求消息` 由任务管理工作线程或父层任务管理实例提出，但是否将其转成新任务入口，只能由 `自我线程` 判断。
6. `结果回流消息` 必须先回 `自我线程`，再由 `自我线程` 路由给父层任务。
7. `学习与审计消息` 只吃结算后的结果，不得抢占当前业务轮次。
8. 自我线程持有 `控制态真值`、`全局优先级`、`任务入口实例化权` 和 `最终结算权`；工作线程只处理局部流程语义。
9. 自我线程每轮必须先冻结本轮开始时已可见的消息批次，处理完该冻结批次及其同步内部闭包，再做一次根层重判与主派发/收束决议。
10. 高优先级控制消息可抢占当前轮次。

一句话收口：

`控制和全局环境先到自我线程，局部外部反馈先到相关工作线程，关键推进与结果再回自我线程。`


## 4. 一轮消息处理总框架

自我线程每轮不是“随来随判”，而应固定为：

`冻结本轮已可见消息 -> 先处理抢占类控制消息 -> 处理普通消息 -> 处理这些消息同步引发的内部闭包 -> 做一次根层重判 -> 做一次主派发 / 收束决议 -> 生成本轮结算与学习输入`

因此线程内部至少应保留以下处理骨架：

```cpp
void 冻结本轮消息批次_();
void 处理抢占控制消息_();
void 处理普通消息批次_();
void 处理本轮同步闭包_();
void 根层重判_();
void 生成主派发决议_();
void 生成本轮结算_();
void 后置处理学习审计_();
```

必须固定：

1. 一轮只做一次 `根层重判`。
2. 一轮最多做一次 `主派发 / 收束决议`。
3. 抢占类控制消息可以中断普通消息处理，但不能绕过最终结算。
4. 本轮处理中异步新产生的消息，必须进入下一轮，不得为了“清空活队列”而追入本轮。


## 5. 统一消息结构建议

为了直接落代码，自我线程治理入口建议统一收成一个总壳：

```cpp
struct 结构_治理消息 {
    主键 消息主键 = 0;
    枚举_消息大类 大类{};
    枚举_消息子类 子类{};
    主键 来源线程主键 = 0;
    主键 来源任务主键 = 0;
    主键 目标任务主键 = 0;
    主键 父任务主键 = 0;
    主键 关联场景主键 = 0;
    时间戳 产生时间 = 0;
    bool 可抢占 = false;
    bool 仅局部有效 = false;
    std::string 载荷键{};
};
```

必须固定：

1. `大类` 决定线程走哪条处理分支。
2. `来源任务主键 / 目标任务主键 / 父任务主键` 用于结果路由、派生需求回送与局部外部反馈定向投递。
3. `可抢占` 只允许控制类或故障升级类消息置位。
4. `仅局部有效` 为 `true` 的消息，原则上应先由对应工作线程处理，再决定是否升级上报线程。

## 7. 七类消息的处理规范

### 7.1 控制消息

典型内容至少包括：

- 启动
- 暂停
- 继续
- 打断
- 收束
- 停止
- 预算刷新
- 优先级刷新
- 配置热更新
- 运行模式切换

必须固定：

1. `控制消息` 改的是控制面，不经任务管理任务转译。
2. `控制消息` 的首收件人必须是 `自我线程`。
3. `打断 / 收束 / 停止` 到达后，线程应要求相关工作线程尽快回传局部结果或中断结果。
4. 必要时可直接触发生命周期状态迁移，而不必先转成需求。

处理逻辑：

1. 校验消息来源是否合法。
2. 更新线程控制态。
3. 决定是否抢占当前轮次。
4. 向受影响的任务管理工作线程下发对应控制事件。
5. 若是 `收束 / 停止`，要求相关工作线程尽快回传局部结果或中断结果。
6. 标记本轮需要重新结算或直接迁移生命周期状态。

线程内部方法建议：

```cpp
void 处理控制消息_(const 结构_治理消息&);
void 更新线程控制态_();
void 下发控制事件_到工作线程_();
void 触发生命周期迁移_();
```

改动对象：

- 线程控制态
- 任务控制态
- 工作线程状态表

是否触发根层重判：

- 通常 `是`
- 纯粹的心跳配置刷新可只改控制态，不必立刻重判

### 7.2 全局环境消息

满足任一条件，就属于 `全局环境消息`：

- 没有明确绑定到某个任务
- 可能形成新的根需求
- 会影响多个任务
- 会改变全局优先级
- 会触发安全接管或收束
- 会改变资源总量或运行模式

典型内容至少包括：

- 场景状态整体变化
- 全局资源变化
- 全局风险上升
- 关键对象状态突变
- 运行规则变化
- 安全阈值变化
- 外部高优先级事件

必须固定：

1. `全局环境消息` 若先发给局部工作线程，会造成自我线程失明，因此禁止这样做。
2. 该类消息必须参与本轮状态刷新、根需求形成和全局重判。

处理逻辑：

1. 把环境变化并入本轮 `世界变化集`。
2. 更新上下文快照。
3. 识别是否形成新的根需求。
4. 刷新全局优先级和队列。
5. 必要时创建新的任务入口。

线程内部方法建议：

```cpp
void 处理全局环境消息_(const 结构_治理消息&);
void 合并环境变化_();
void 刷新线程上下文快照_();
void 识别根需求_();
void 重算全局优先级_();
void 必要时创建任务入口_();
void 转发局部外部反馈_到工作线程_(const 结构_治理消息&);
```

改动对象：

- 线程轮次上下文
- 根需求候选集
- 全局任务队列

是否触发根层重判：

- `一定触发`

补充固定：

- 若外部反馈 `明确绑定某个任务主键，而且只影响该任务当前步骤`，则它不算全局环境消息，应先转发给对应工作线程。

### 7.3 工作线程状态消息

典型内容至少包括：

- 已启动
- 已绑定任务
- 心跳
- 空闲
- 忙碌
- 阻塞
- 超时
- 故障
- 已收束
- 已停止

必须固定：

1. 这类消息反映的是执行载体健康与控制状态，不是任务语义。
2. 自我线程只用它们更新线程管理表和任务控制态，不深入工作线程内部工序。
3. 超时、故障、失联都可触发故障收口、重建或队列调整。

处理逻辑：

1. 更新工作线程注册表。
2. 把对应任务的控制态更新为 `已派发 / 等待 / 故障 / 收束中` 等。
3. 若线程超时或故障，进入故障收口。
4. 必要时重建工作线程并重新入队。

线程内部方法建议：

```cpp
void 处理工作线程状态消息_(const 结构_治理消息&);
void 更新工作线程状态表_();
void 更新任务控制态_从工作线程状态();
void 故障收口_();
void 恢复重建入队_();
```

改动对象：

- 工作线程状态表
- 任务控制态
- 故障摘要
- 恢复摘要

是否触发根层重判：

- `心跳 / 空闲 / 已绑定` 一般不立即触发
- `超时 / 故障 / 失联 / 强阻塞` 必须触发

### 7.4 任务推进关键事件消息

典型内容至少包括：

- 当前步骤可执行
- 当前步骤被阻塞
- 当前步骤解除阻塞
- 等待前序步骤
- 等待条件
- 等待子结果
- 等待确认
- 局部失败
- 局部重试上限触发
- 当前任务局部完成

必须固定：

1. 这类消息只上报 `前沿变化`，不上报整个工序流水。
2. `步骤先后和条件满足` 决定 `能不能跑`；`优先级` 由自我线程决定 `先让谁跑`。
3. 若父任务被阻塞，线程必须把它移出可运行队列。
4. 若步骤解除阻塞，线程必须把相关任务重新放回可运行队列。

处理逻辑：

1. 更新任务局部状态摘要。
2. 同步修改任务控制态中的队列状态。
3. 若父任务被阻塞，则从可运行队列移出。
4. 若某任务解除阻塞，则放回可运行队列。
5. 若只影响局部流程，则不必立即根层重判；若影响全局竞争关系，则标记本轮重判。

线程内部方法建议：

```cpp
void 处理任务推进关键事件_(const 结构_治理消息&);
void 更新任务前沿摘要_();
void 更新任务队列状态_();
void 移出可运行队列_();
void 放回可运行队列_();
```

改动对象：

- 任务控制态
- 可运行队列
- 等待队列
- 局部运行态摘要缓存

是否触发根层重判：

- 单纯局部推进一般 `否`
- `阻塞 / 解阻 / 大失败` 应触发

### 7.5 派生需求消息

典型内容至少包括：

- 缺资源
- 缺确认
- 缺信息
- 缺方法
- 缺依赖结果
- 风险处置需求
- 异常恢复需求
- 补偿需求

必须固定：

1. `任务管理任务可以提需求，不能自己生子任务。`
2. 线程接收派生需求后，必须先判断其是否仅局部可消化。
3. 只有确需外部任务支撑时，线程才有权把它实例化为新的任务入口。
4. 若线程否决该派生需求，必须把拒绝结果路由回父层任务。

处理逻辑：

1. 判断它是不是局部等待即可解决。
2. 如果只是局部等待，则不实例化新任务，只更新父任务等待原因。
3. 如果确需外部支撑，则评估它对根层权重和全局优先级的影响。
4. 若批准，则由自我线程创建新的依赖任务入口。
5. 把新任务派发给新的任务管理工作线程。
6. 若不批准，则把拒绝结果路由回父层任务。

线程内部方法建议：

```cpp
void 处理派生需求消息_(const 结构_治理消息&);
bool 是纯局部等待_(const 结构_治理消息&);
void 评估派生需求_对根层影响();
void 实例化依赖任务_();
void 驳回派生需求_并回送父任务_();
```

改动对象：

- 根需求候选集
- 任务根节点
- 新任务入口
- 任务控制态
- 优先级队列

是否触发根层重判：

- `一定触发`

### 7.6 结果回流消息

典型内容至少包括：

- 子任务结果
- 局部结果
- 最终结果
- 成功
- 失败
- 中断结果
- 收束结果
- 异常结果
- 可学习样本候选

必须固定：

1. `子结果先回线程，线程再回父任务；父任务判断局部影响，线程判断全局影响。`
2. 线程接收结果后，必须先判断其对全局优先级、阻塞解除、新根需求、收束/降级的影响。
3. 结果路由与总账结算必须都留在自我线程。

处理逻辑：

1. 接收结果并写入结果缓存。
2. 判断它对全局的影响，包括：
   - 是否改变全局优先级
   - 是否解除父任务阻塞
   - 是否引发新的根需求
   - 是否触发降级、收束、停止
3. 把结果路由给父层任务。
4. 父层任务只判断“这对我当前步骤有什么影响”。
5. 自我线程再根据父层返回的局部状态，更新全局队列与本轮结算。

线程内部方法建议：

```cpp
void 处理结果回流消息_(const 结构_治理消息&);
void 计算结果全局影响_();
void 路由子结果_到父任务_();
void 更新父任务队列状态_();
void 纳入本轮结算候选_();
```

改动对象：

- 结果缓存
- 任务控制态
- 父任务等待状态
- 全局优先级队列
- 本轮结算集

是否触发根层重判：

- 通常 `是`

### 7.7 学习与审计消息

典型内容至少包括：

- 学习反馈
- 因果候选
- 样本验证结果
- 误学习回滚建议
- 审计记录
- 运行诊断结果
- 复盘结论

必须固定：

1. 该类消息不用于即时派发。
2. 必须先挂到结算之后的学习/审计队列。
3. 若复盘结果升级为真正的新环境事实或新控制要求，才允许转成 `全局环境消息` 或 `控制消息` 重新入队。

处理逻辑：

1. 先挂到后置队列。
2. 等待本轮结算完成。
3. 消费这些消息，更新样本池、因果候选、诊断摘要。
4. 默认不直接触发新任务。
5. 如果复盘结果升级成真正的新环境事实或控制要求，再转成全局环境消息或控制消息重新入队。

线程内部方法建议：

```cpp
void 处理学习与审计消息_(const 结构_治理消息&);
void 缓存后置学习审计消息_();
void 结算后消费学习审计_();
void 更新样本池与审计摘要_();
void 必要时回灌为新消息_();
```

改动对象：

- 学习候选池
- 审计日志
- 健康摘要
- 复盘摘要

是否触发根层重判：

- 一般不立即触发
- 只有被回灌成新的控制消息或全局环境消息时才触发


## 8. 原则上不应直接送到自我线程的消息

以下“局部噪声”原则上不应直接送到自我线程：

- 步骤内部的小进度
- 局部临时变量变化
- 方法内部中间态
- 每一次小重试
- 纯局部缓存更新
- 明确绑定某任务且只影响该任务当前步骤的普通外部反馈

必须固定：

1. 上述消息应先给相关任务管理工作线程。
2. 只有当它们升级成 `阻塞变化 / 派生需求 / 异常 / 结果回流 / 优先级变化 / 控制态变化` 时，才允许上浮到自我线程。


## 9. 共用支撑方法

除了七类消息各自的处理函数，自我线程还应保留以下共用支撑方法：

```cpp
void 记录阶段事件_已加锁();
void 刷新快照_已加锁();
void 重算全局优先级_();
void 生成线程轮次上下文包_();
void 生成回流结算包_();
```

这些方法分别回答：

- 记录发生了什么
- 刷新线程外可读的最小运行态
- 统一计算优先级
- 把本轮冻结消息集结成一个治理帧
- 把本轮处理结果结成总账


## 10. 自我线程处理消息的统一规则

自我线程处理上述消息时，统一遵守以下 5 条规则：

1. `统一入口`
   所有可上浮到控制面的消息，统一进入治理邮箱或线程消息入口。

2. `冻结批次`
   每轮先冻结本轮消息集，处理完本轮看见的全部消息，以及它们在本轮内引发的同步内部闭包；异步新消息留到下一轮。

3. `控制抢占`
   控制消息可抢占普通轮次。

4. `结果先线程后父层`
   任何子结果、局部结果、最终结果，先回自我线程，再由线程路由给父层任务。

5. `学习后置`
   学习和审计只吃结算后的结果，不直接抢占当前任务派发。


## 11. 主循环伪代码骨架

```cpp
void 自我线程::执行主循环一轮_() {
    冻结本轮消息批次_();

    处理抢占控制消息_();

    while (存在未处理普通消息_()) {
        auto 消息 = 取下一条普通消息_();

        switch (消息.大类) {
        case 枚举_消息大类::控制:
            处理控制消息_(消息);
            break;
        case 枚举_消息大类::全局环境:
            处理全局环境消息_(消息);
            break;
        case 枚举_消息大类::工作线程状态:
            处理工作线程状态消息_(消息);
            break;
        case 枚举_消息大类::任务推进关键事件:
            处理任务推进关键事件_(消息);
            break;
        case 枚举_消息大类::派生需求:
            处理派生需求消息_(消息);
            break;
        case 枚举_消息大类::结果回流:
            处理结果回流消息_(消息);
            break;
        case 枚举_消息大类::学习与审计:
            处理学习与审计消息_(消息);
            break;
        }
    }

    处理本轮同步闭包_();
    根层重判_();
    生成主派发决议_();
    生成本轮结算_();
    后置处理学习审计_();
    刷新快照_已加锁();
}
```

必须固定：

1. `结果回流消息` 的处理早于 `后置处理学习审计_()`。
2. `学习与审计消息` 的即时接收与真正消费是两回事；接收可以发生在本轮，但消费必须后置。
3. 线程不得在一轮里同时做多次主派发决议。


## 12. 最小验收口径

当前若要承认 `0028` 已成立，至少必须满足：

1. 自我线程接受的消息类型已正式限定为七类。
2. `控制消息 / 全局环境消息 / 工作线程状态消息 / 任务推进关键事件消息 / 派生需求消息 / 结果回流消息 / 学习与审计消息` 都有明确首收件人。
3. `局部噪声不上浮线程` 已写成正式规则。
4. `结果先线程后父层` 已写成正式规则。
5. `学习后置` 已写成正式规则。
6. `消息类型 / 首收件人 / 是否影响优先级 / 是否可抢占 / 是否可触发新任务` 的总表已存在。

最终收口：

`0028 的意义，是把“自我线程到底该收什么消息、这些消息怎么分流、哪些必须进入控制面、线程收到后怎么处理、哪些方法名和处理骨架可直接落代码”正式写成消息处理规范。`
