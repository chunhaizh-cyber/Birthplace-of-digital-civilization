# 0028 自我线程消息总表与处理规范

## 1. 目标

本文用于把 `自我线程` 的正式输入消息重新收口成一份更稳定的总表，回答以下问题：

- 自我线程一级上到底只收哪两条来源链
- 在线程内部应再拆成哪 7 类具体处理类型
- 一条正式消息至少必须带哪些字段
- 哪些消息会直接影响服务 / 安全
- 哪些消息会触发根层重判、主派发或新任务实例化
- 哪些东西不应作为外来消息直接塞给线程

本文只收口 `自我线程正式输入消息`，不展开：

- 工作线程内部小进度和方法内部噪声
- 学习与审计后置队列的内部算法
- 需求权重和优先级的完整计算公式

必须先固定一句话：

`自我线程只收事实消息，不直接收结论消息。`

## 3. 核心结论

必须固定以下结论：

1. 自我线程一级上只收两条来源链：
   - `线程直达消息`
   - `任务管理工作线程回流消息`
2. 为了真正处理清楚，线程内部必须再拆成 7 类具体处理类型：
   - `控制域事实消息`
   - `业务域事实消息`
   - `延迟事实消息`
   - `工作线程状态消息`
   - `任务推进关键事件消息`
   - `派生需求上报消息`
   - `结果与关键中间状态回流消息`
3. 自我线程收的是 `事实消息`，不是 `结论消息`。
4. `需求` 不是外来消息类型；它是线程处理事实消息后的内部整理产物。
5. `任务权重变化` 不是外来消息类型；它是线程重算二次特征和影响项后的内部控制结果。
6. `否定项成立` 不是外来消息类型；它应由线程根据 `关键中间状态变化 + 二次特征 + 负向影响项` 自行抽取。
7. 所有正式消息都必须统一表达为：
   `谁发出 -> 谁做了什么动作 -> 导致谁的哪些特征发生了什么变化`
8. `控制域`、`业务域`、`工作线程状态` 和 `结果/关键中间状态` 都可以直接影响服务和安全。
9. `派生需求` 本身更偏间接影响；直接影响服务和安全的，通常是触发它的关键中间状态和事实变化。
10. `学习 / 审计 / 复盘` 不再作为自我线程正式输入消息类型；它们属于本轮结算后的后置队列。若升级为正式控制事实或业务事实，应重新包装成前述 7 类中的某一类再入队。
11. `工作线程状态消息` 当前保留的是兼容名，语义上应按“`其它线程状态变化消息`”理解；它既可以来自 `线程直达消息`，也可以来自 `任务管理工作线程回流消息`。
12. 发件线程只负责上报状态变化事实，不得直接改写 `自我内部世界` 中的线程状态特征、状态节点或动作动态；这些真值只能由 `自我线程` 在收到消息后统一落账。
13. `自我线程` 自身的关键生命周期变化属于例外：应进入 `关键日志 / 宿主级审计 / 恢复判断材料`，不作为本文所说“其它线程状态变化消息”的主处理对象。
14. 从 `任务管理任务 -> 自我线程` 的上行消息视角再收一层时，长期正式收为三类：
   - `线程状态消息`
   - `派生需求消息`
   - `结果消息`
15. 其中：
   - `线程状态消息` 在实现上直接按 `工作线程状态消息 / 线程状态消息` 落；`当前步骤 / 等待原因 / 是否可继续推进` 等只作为补充特征类型
   - `派生需求消息` 用于表达任务推进过程中发现的本地不可消化缺条件、外部支撑需求或会阻塞当前步骤推进的事实
   - `结果消息` 兼容承接 `结果与关键中间状态回流消息`，负责上报阶段完成后的结果事实、服务对象状态与关键中间状态
16. 从 `推进一步` 原语的视角看，这三类消息就是 `任务管理任务` 的正式外送口；每次 `推进一步` 只允许产生其中一种或小组合，不要求三类齐发。
17. `局部运行态更新` 本身不等于消息；只有当其需要被自我线程正式消费时，才应封装进上述三类之一。

一句话收口：

`两大来源链够用；线程内部按 7 类兼容处理；而从任务管理任务上行视角又正式收成“线程状态 / 派生需求 / 结果”三类；线程只收事实，不收结论；其它线程状态真值由自我线程在收消息后统一落账，而自我线程自身关键状态变化走关键日志例外。`


## 4. 一级来源链

| 一级来源链 | 定义 | 典型来源 | 是否先到自我线程 |
| --- | --- | --- | --- |
| `线程直达消息` | 不经任务管理工作线程转译，直接进入自我线程的事实消息 | 控制面、环境监测、场景变化、延迟到期事实、其它线程状态变化直接上报 | 是 |
| `任务管理工作线程回流消息` | 由任务管理工作线程在局部推进后回流给自我线程的事实消息 | `线程状态消息`、`派生需求消息`、`结果消息`；兼容细分上仍可见 `工作线程状态 / 关键推进事件 / 派生需求 / 结果与关键中间状态` | 是 |

必须固定：

1. 只有进入 `自我线程` 的正式消息，才属于本文范围。
2. `明确绑定某任务且只影响当前步骤的普通局部反馈`，若仍停留在工作线程内部，就不属于本文的正式输入消息。
3. 这类局部反馈只有在升级为 `阻塞变化 / 派生需求 / 异常 / 结果回流 / 控制面变化` 后，才重新包装成本文定义的正式输入消息。

### 4.1 任务管理任务上行消息三分法

从 `任务管理任务 -> 自我线程` 这一条上行链单独看，推荐再收成三类：

| 上行大类 | 当前兼容细分 | 回答的问题 | 最小语义 |
| --- | --- | --- | --- |
| `线程状态消息` | `工作线程状态消息` | `我现在怎么跑，能不能继续跑？` | 当前运行状态，加上步骤前沿、等待原因、预算/异常/控制响应等补充特征 |
| `派生需求消息` | `派生需求上报消息` | `我这里缺什么，是否需要外部支撑，是否会阻塞父任务？` | 归属段、派生需求本体段、触发事实段、影响提示段 |
| `结果消息` | `结果与关键中间状态回流消息` | `这次推进以后，实际留下了哪些执行事实？` | 结果头、结果节点引用、快速补充字段、后续支撑补充 |

必须固定：

1. `任务管理任务` 或 `工作线程` 不得替 `自我线程` 自身上报“自我线程运行状态”。
2. `任务推进关键事件` 若只是在表达 `当前步骤 / 等待原因 / 可继续推进与否`，长期应规范化为 `线程状态消息` 的补充特征类型，而不是独立上行大类。
3. `派生需求消息` 是独立正式上行类，但它表达的是候选事实和外部支撑需求，不等于正式需求树节点。
4. 作为 `任务管理任务 -> 自我线程` 的业务上行消息，三类已经封顶，不建议再新增第四类一级类型；新增内容必须先判断应归入 `线程状态 / 派生需求 / 结果` 中的哪一类。
5. `关键中间状态 / 未满足项 / 恢复重建事实` 都应归并进这三类，不另立第四类。
6. `确认消息` 若未来存在，应来自另一条验证来源链，不属于 `任务管理任务` 上行消息三分类。
7. `ACK / 重复消费 / 幂等冲突 / 反序到达 / 协议异常` 属于传输层与基础设施层处理，不属于业务上行一级类型。
8. 当前 7 类二级处理类型仍继续保留，主要是为了兼容路由和渐进改造。
9. 一次 `推进一步` 若只是表达 `当前前沿 / 等待原因 / 是否可继续推进`，默认优先规范化为 `线程状态消息`，而不是额外新立事件大类。


## 5. 统一消息公共结构

### 5.1 公共头

正式消息至少应包含：

- `消息主键`
- `一级分类`
  - `线程直达消息`
  - `任务管理工作线程回流消息`
- `二级类型`
- `发出者主键`
- `动作主体主键`
- `动作语义键`
- `作用对象主键`
- `父任务主键（可空）`
- `关联场景主键`
- `产生时间`
- `路径签名`
- `来源成功标志`
- `错误码`
- `置信度`

### 5.2 特征变化项集

每个变化项至少应包含：

- `变化主体主键`
- `特征类型主键`
- `变化前值`
- `变化后值`
- `是否确认生效`
- `是否关键中间状态`
- `是否已恢复`
- `变化原因键`

### 5.3 一句话要求

必须固定：

`消息不是“一个名字”，而是“一个动作 + 一组特征变化项”。`


## 6. 七类消息总表

| 二级类型 | 一级来源链 | 必带字段 | 是否直接影响服务 | 是否直接影响安全 | 是否触发重判 | 是否可实例化新任务 | 最小处理要求 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `控制域事实消息` | `线程直达消息` | 控制意图/运行状态/预算/优先级策略变化项 | 是 | 是 | 是 | 否 | 先更新线程控制态与任务控制态，再决定是否抢占本轮 |
| `业务域事实消息` | `线程直达消息` | 业务对象、资源、场景、风险、条件等变化项 | 是 | 是 | 是 | 条件成立时是 | 先落一次特征账，再重算依赖满足、可执行性、风险和影响项 |
| `延迟事实消息` | `线程直达消息` | 到期主体、到期特征、生效时刻、是否自动生效 | 是 | 是 | 是 | 条件成立时是 | 把延迟到期事实实体化为一次特征变化，再进入重算与重判 |
| `工作线程状态消息` | `线程直达消息 / 任务管理工作线程回流消息` | 工作线程主键、绑定任务主键、运行状态变化、错误码、最近心跳时间 | 是 | 是 | 条件成立时是 | 否 | 只上报线程状态变化事实；由自我线程收到后统一更新线程状态账与对应任务控制态，必要时进入故障收口 |
| `任务推进关键事件消息` | `任务管理工作线程回流消息` | 任务管理任务主键、所属任务主键、当前步骤主键、局部运行状态变化、等待原因变化 | 是 | 是 | 条件成立时是 | 否 | 只上浮会改变控制面判断的推进事件；阻塞退队，解阻再入队 |
| `派生需求上报消息` | `任务管理工作线程回流消息` | 发起任务管理任务主键、所属父任务主键、当前步骤主键、派生需求类型、触发它的关键状态变化 | 间接 | 间接 | 是 | 是 | 对应正式上行类 `派生需求消息`；只上报候选事实，不授予直接生子任务权限 |
| `结果与关键中间状态回流消息` | `任务管理工作线程回流消息` | 结果头、结果节点引用段、快速补充段、后续支撑补充段 | 是 | 是 | 是 | 条件成立时是 | 先解析结果节点，再判断全局影响，再路由父任务，再结算和抽取否定项候选 |


## 7. 七类消息处理规范

### 7.1 控制域事实消息

典型内容：

- `控制意图 = 请求启动 / 请求暂停 / 请求继续 / 请求打断 / 请求收束 / 请求停止`
- `运行模式切换`
- `预算刷新`
- `优先级策略刷新`

必须固定：

1. `启动 / 暂停 / 继续 / 打断 / 收束 / 停止` 不再是裸消息名，而是 `控制域特征值变化`。
2. `请求收束` 和 `已收束` 不得混写到同一个值桶。
3. 这类消息先改 `控制面`，再由结果回流和状态确认去改 `实际运行状态`。

### 7.2 业务域事实消息

典型内容：

- `资源可用性变化`
- `场景状态变化`
- `风险等级变化`
- `关键对象状态突变`
- `某业务条件满足 / 不满足`

必须固定：

1. 业务域事实消息可以直接影响服务和安全。
2. 它们不是“需求消息”；需求是线程处理事实后的内部产物。

### 7.3 延迟事实消息

典型内容：

- `超时到期`
- `等待窗口到期`
- `确认超时`
- `预算刷新时点到达`

必须固定：

1. 它表达的是“到这个时间点才正式生效的事实”。
2. 到期后必须先实体化成一次特征变化，再进入二次特征和影响项链。

### 7.4 工作线程状态消息

典型内容：

- `已启动`
- `已绑定任务`
- `忙碌`
- `阻塞`
- `超时`
- `故障`
- `已收束`
- `已停止`

必须固定：

1. `工作线程状态消息` 当前是兼容名；语义上应按“线程状态变化消息”理解。
2. 这类消息反映的是 `执行载体状态`，不是任务语义本身。
3. 但它仍可直接影响服务和安全，并可能触发故障收口和恢复重建。
4. 发件线程不得在上报前直接改写 `自我内部世界` 中的线程状态特征、状态节点或动作动态。
5. 线程状态真值、状态节点与对应动作动态，只能由 `自我线程` 在消费该消息后统一落账。
6. 从 `任务管理任务` 的上行视角看，它本身就是 `运行状态消息` 的正式实现形态。

### 7.5 任务推进关键事件消息

典型内容：

- `当前步骤可执行`
- `当前步骤阻塞`
- `当前步骤解除阻塞`
- `等待前序步骤`
- `等待条件`
- `等待子结果`
- `局部失败`
- `局部完成`

必须固定：

1. 线程不收局部小进度流水账，只收 `会改变控制面判断` 的关键事件。
2. 这类消息先影响 `任务管理任务局部推进态`，再经线程改写 `任务控制态`。
3. 若这些事件本质上只是在表达 `当前步骤 / 等待原因 / 是否可继续推进` 等线程运行面事实，长期应规范化为 `线程状态消息` 的补充特征类型，不再单独作为上行大类存在。

### 7.6 派生需求上报消息

典型内容：

- `缺资源`
- `缺确认`
- `缺信息`
- `缺方法`
- `风险处置需求`
- `异常恢复需求`
- `补偿需求`

必须固定：

1. `派生需求` 是上报，不是结论，更不是直接的新任务实例。
2. 线程必须先判断它是局部等待，还是需要升格为新的任务入口。
3. 当前它保留为兼容路由入口；长期语义应并入 `结果状态消息` 的 `未满足项 / 后续支撑请求段`。

### 7.7 结果状态消息（兼容入口：结果与关键中间状态回流消息）

典型内容：

- `结果头`
- `最终结果`
- `服务对象当前状态`
- `局部结果`
- `子任务结果`
- `中断结果`
- `收束结果`
- `异常结果`
- `关键中间状态变化集`
- `未满足项 / 后续支撑请求`

必须固定：

1. 这类消息的核心不是“成功/失败标签”，而是一次任务推进后的 `执行事实包`。
2. 这类执行事实的主承载面，默认是 `任务结果节点` 里的场景信息。
3. 消息层最小只需返回 `结果节点` 和少量补充信息，不必把服务对象状态、关键中间状态、未满足项整包重复平铺。
4. 若结果节点尚未稳定落成，才允许临时在消息层展开更多细节。
5. `派生需求` 与 `未满足项` 长期应按本类消息内部段落理解，不再扩成第三大上行类。
6. 自我线程收到后，应先解析结果节点，再判断全局影响，再路由父任务，最后统一结算。


## 8. 不应作为外来消息直接上报的东西

以下内容不得作为自我线程正式输入消息：

- `需求已经成立`
- `这个任务优先级应该更高`
- `这个否定项已经成立`
- `任务权重变化`
- `步骤内部小进度`
- `纯局部变量变化`
- `学习/审计结论本身`

必须固定：

1. `需求` 是线程内部整理产物。
2. `任务权重` 是线程内部重算产物。
3. `否定项成立` 是线程基于关键中间状态和负向影响项抽取的结果。
4. `学习 / 审计 / 复盘` 只有升级成正式控制事实或业务事实时，才允许重新包装入队。


## 9. 自我线程统一处理链

无论进入哪一类消息，自我线程都必须统一遵守：

1. `冻结本轮消息批次`
2. `规范化为动作语义 + 特征变化项集`
3. `落一次特征账`
4. `重算二次特征`
5. `生成影响项`
6. `整理需求和权重`
7. `根层重判`
8. `主派发决议`

必须固定：

1. 线程不是“见一条判一条”。
2. 一轮只处理本轮冻结消息及其同步内部闭包。
3. 本轮中新产生的异步消息，必须进入下一轮。
4. 任意线程状态变化消息都只能先进入这条统一处理链；发件线程不得绕过该链直接写入线程状态真值。


## 10. 完成标准

达到下面 7 条，即可视为这轮消息总表已经站稳：

1. 已经把自我线程正式输入收敛成 `两大来源链`。
2. 已经把线程内部处理类型收敛成 `7 类`。
3. 已经固定“线程只收事实，不收结论”。
4. 已经明确 `需求 / 权重 / 否定项` 不属于外来消息。
5. 已经明确 7 类消息各自是否直接影响服务 / 安全、是否触发重判、是否可实例化新任务。
6. 已经把消息正式接回 `冻结 -> 特征 -> 二次特征 -> 影响项 -> 重判 -> 派发` 的统一治理链。
7. 已经明确线程状态变化只能先上报消息，再由 `自我线程` 在消费阶段统一落账。
