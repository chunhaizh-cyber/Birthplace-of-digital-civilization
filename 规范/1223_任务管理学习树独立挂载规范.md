本文用于固定以下问题：

- 学习任务是否必须物理挂在 `安全根任务 / 服务根任务` 下面

- 如何区分 `来源归属` 与 `树结构归属`
- 学习任务独立成树后，怎样仍然保持对安全/服务来源的可追溯性

本文要收口的是 `学习任务的树结构归属`，不是学习算法实现。

3. 本文不是新的根裁决层，而是学习任务树结构挂载的承接层。


## 3. 核心结论

必须固定以下 11 条：

1. `来源归属` 与 `树结构归属` 必须分开。
2. 学习任务在语义上仍然来源于 `安全/服务根任务` 及其任务管理衍生需求。
3. 但在结构上，学习任务可以不直接挂在 `安全根任务 / 服务根任务` 下面。
4. 当 `任务管理任务` 被视为安全和服务的底层治理系统时，学习任务应优先挂在 `任务管理学习树` 下。
5. `任务管理学习树` 不是新的原生根需求树，而是 `任务管理系统` 的内部治理子树。
6. 无论学习任务挂在哪里，都必须保留：
   - `来源根任务`
   - `来源父任务`
   - `来源功能域`
   - `来源方法需求位`
   - `来源本能能力缺口类型`
   - `来源最小动作原语位图`
   - `来源主观察特征`
   - `方法服务目标`
   - `服务方法位/方法枝`
7. `任务子任务生成的学习任务` 与 `任务管理任务生成的独立学习任务` 都可以挂在 `任务管理学习树` 下，只是来源字段和反馈后的消费边界不同。
8. 以后不得再用“物理挂在根任务下面”来代替“来源追溯”。
9. 若当前实现保留共享学习执行器，则它默认应以 `任务管理学习树` 为执行载体。
10. `任务管理学习树` 承载的是独立学习任务实体；统一学习调度器只能消费这棵树中的任务，不得用共享执行器替代任务实体本身。
11. 任一挂在学习树上的学习任务执行结果在当前治理轮被消费后，仍必须先回到外层重新做 `根层重判 -> 上下文刷新`，再决定是否继续调度或消费学习情况。

一句话收口：

`学习任务在语义上来源于安全/服务，在结构上挂到任务管理学习树；来源靠字段追溯，不靠树位置推断。`

并且：

`任务管理学习树负责承载独立学习任务实体，统一学习调度器负责消费这棵树。`


## 4. 为什么独立成树是合理的

若 `任务管理任务` 是安全和服务的底层治理系统，则：

- `学习任务` 本质上是在修复和扩充治理能力
- 它处理的是方法资产、条件模板、结果模板、边界模板、可验证性
- 这些都属于 `任务管理系统` 的内部治理资产

因此从结构上看，把学习任务直接挂在业务根任务下面，会带来三个问题：

1. 混淆 `业务树` 与 `治理树`
2. 容易把“学习执行器”误判为业务子任务
3. 多来源共享学习时，父子关系会变得很别扭

所以更合理的结构是：

```text
安全根任务 / 服务根任务
-> 任务管理任务
-> 任务管理学习树
   -> 定向学习子任务
   -> 任务管理广泛学习子任务
```

## 6. 任务管理学习树定义

`任务管理学习树` 是：

- 任务管理系统专门承载学习任务、学习阶段、学习情况反馈承接的内部治理树

它不是：

- 第三根需求树
- 与安全/服务并列的业务根任务树
- 独立语义上的学习根任务

它至少应有：

- `学习树根节点`
- `任务子任务学习分支`
- `任务管理独立学习分支`
- 各学习任务头
- 各学习阶段步骤


## 7. 两类学习任务在独立树中的挂法

### 7.1 任务子任务生成的学习任务

结构上：

- 挂在 `任务管理学习树/任务子任务学习分支`

语义上：

- 来源于某个父任务当前缺口

最少字段：

- `来源根任务头指针`
- `来源父任务头指针`
- `来源方法需求位`
- `来源本能能力缺口类型`
- `来源最小动作原语位图`
- `来源主观察特征`
- `学习目标类型`
- `缺口类型`
- `缺失方法数量`
- `服务方法位/方法枝`


### 7.2 任务管理任务生成的独立学习任务

结构上：

- 挂在 `任务管理学习树/任务管理独立学习分支`

语义上：

- 来源于任务管理任务长期识别出的储备学习动机

最少字段：

- `来源根任务头指针`
- `来源任务管理任务头指针`
- `来源功能域`
- `来源方法需求位`
- `来源本能能力缺口类型`
- `来源最小动作原语位图`
- `来源主观察特征`
- `上浮压力类型`
- `学习目标类型`
- `服务方法位/方法枝`


## 8. 反馈后的消费规则

学习任务独立成树后，反馈消费边界必须更明确：

- `任务子任务生成的学习任务`
  - 默认先反馈给当前 `任务管理任务`
  - 再由任务管理任务决定是否让 `来源方法位/方法枝` 参与当前父任务推进

- `任务管理任务生成的独立学习任务`
  - 默认先反馈给当前 `任务管理任务`
  - 再由任务管理任务决定是否进入 `来源根任务的方法资产` 或按需求位服务同类任务

因此必须禁止：

- 因为学习任务不挂在父任务下面，就丢掉来源追溯和后续消费关系


## 9. 当前实现对照

当前实现已经具备：

- 共享学习执行器
- 来源根任务头指针
- 来源父任务头指针
- 来源方法需求位
- 显式学习子任务

当前实现尚未具备：

- 独立的 `任务管理学习树`
- `任务子任务学习分支 / 任务管理独立学习分支`
- `来源任务管理任务头指针`
- `来源功能域`
- `来源压力类型`
- `来源本能能力缺口类型`
- `来源最小动作原语位图`
- `来源主观察特征`
- `服务方法位/方法枝`

所以当前实现仍处于：

- `来源字段已部分具备`
- `结构挂载仍未正式独立`


## 10. 后续实现顺序

建议顺序固定为：

1. 先创建 `任务管理学习树根节点`
2. 再创建 `定向学习分支 / 广泛学习分支`
3. 将当前显式学习子任务迁入 `定向学习分支`
4. 再新增 `任务管理广泛学习子任务`
5. 最后让共享学习执行器按学习树分支消费

并且始终保持：

- `学习树挂载 -> 调度 -> 执行 -> 反馈` 只能作为承接层运行。
- 每轮消费结果后，仍必须回到外层重新做 `根层重判 -> 上下文刷新`。


## 11. 结论

对你现在这套系统来说：

`学习任务在树结构上不直接挂在安全和服务两个根任务下面，而是挂在任务管理学习树下独立承载`

是合理的，而且比继续把学习任务直接挂在业务根任务下面更稳。

必须同时保留一条边界：

`结构独立，不等于来源独立；来源与后续消费仍必须追溯到安全/服务及其父任务。`
