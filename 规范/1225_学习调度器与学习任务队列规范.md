# 1225 学习调度器与学习任务队列规范

## 1. 目标

本文用于固定以下问题：

- 学习调度首先应拥有哪些调度对象
- 是否可以把所有学习任务统一进入一个学习队列
- 学习优先级应如何参与调度
- 学习任务是否应提交给通用任务管理任务直接执行
- 学习调度器、学习任务实体、共享学习执行器三者的职责边界是什么

本文收口的是 `学习调度运行模型`，不是具体学习算法细节。

### 2.1 在统一主链中的位置

本文默认处于以下主链中段：

`根层方向裁决`
-> `形成任务管理上下文`
-> `学习任务进入学习任务表`
-> `学习调度器判定就绪/等待/合并`
-> `共享学习执行器执行`
-> `学习任务直接完善学习对象方法并形成学习情况`
-> `反馈任务管理任务`
-> `返回外层重判与再筹办`

必须固定：

1. 本文只在 `根层已允许继续推进` 且 `任务管理上下文已形成` 时进入常规学习调度路径。
2. 若 `最近根层裁决结果` 已要求 `停止 / 收束 / 待机`，则普通学习队列调度不得按常规治理路径继续推进。
3. 本文不是新的根裁决层，而是学习任务表、队列和执行器的承接层。


## 3. 核心结论

必须固定以下 14 条：

1. 学习调度首先要有 `独立学习任务实体`，不能直接只对“学习方法候选”做裸调度。
2. 所有独立学习任务都必须先进入 `学习任务表`，再由调度器决定是否进入就绪队列。
3. 学习调度运行时至少分成：`学习任务表`、`就绪队列`、`等待表`、`共享学习执行器` 四层。
4. `统一一个学习调度` 是对的，但它调度的是 `独立学习任务实体`，不是替代这些任务实体。
5. 学习任务不应脱离独立任务实体重新揉回普通执行链；它仍由 `任务管理任务` 推进，并在就绪后选择一条学习方法执行，其中 `尝试学习方法` 只是默认本能选项之一。
6. 更合理的职责划分是：`任务管理任务生成/更新/推进学习任务并判定是否就绪与如何消费学习情况`，`学习调度器排序/分派`，`共享学习执行器承载被选中学习方法的实际执行`。
7. 单一 FIFO 裸队列不足以表达学习调度，必须至少补 `等待表` 与 `去重/合并` 机制。
8. 每个独立学习任务都必须有自己的阶段状态，不允许只用一个全局学习工作流状态替代全部学习任务生命周期。
9. `任务子任务学习任务` 与 `任务管理独立学习任务` 都统一进入学习任务表，由同一个调度器统一排序。
10. 调度优先级至少要同时考虑：`学习来源类型`、`父任务优先级`、`来源根任务类型`、`需求位压力/缺口量`、`等待时长`。
11. 学习任务完成后必须先形成 `学习情况摘要` 并反馈给 `任务管理任务`；但 `待学习方法` 执行后得到的 `条件-结果对` 应在学习任务执行过程中直接融合写入方法侧。
12. 后续实现不得把“共享学习执行器存在”误写成“学习任务不需要独立排队和独立状态”。
13. 当学习调度器服务于 `任务管理方法` 分支时，外层仍须显式化 `最近根层裁决结果`、`当前缺口类型`、`当前下一步去向`。
14. 任一学习任务队列结果在当前治理轮被消费后，仍必须先回到外层重新做 `根层重判 -> 上下文刷新`，再决定是否继续入队、调度、召回或升格。

一句话收口：

`学习统一调度是对的，但必须建立在独立学习任务实体和学习任务队列之上。`


## 4. 最小运行模型

最小运行模型固定为：

`任务管理推进`
-> `生成学习需求`
-> `创建独立学习任务头节点`
-> `写入学习任务表`
-> `返回外层根层重判/上下文刷新`
-> `下一轮任务管理任务进入新的筹办阶段`
-> `筹办找到学习任务兜底方法`
-> `兜底方法生成可执行学习方法选项`
-> `任务管理任务推进学习任务并判定是否就绪`
-> `进入就绪队列`
-> `学习调度器排序/分派`
-> `共享学习执行器承载被选中的学习方法`
-> `执行当前待学习方法并完善学习对象方法`
-> `条件-结果对融合写入方法`
-> `学习情况摘要反馈任务管理任务`
-> `任务管理任务决定继续学习/挂起/使用已完善方法/再派生学习任务`

不得简化成：

`有学习需求`
-> `直接让共享学习执行器自己选一个东西学`

并且必须继续固定：

- `学习结果写回学习任务`
- `返回外层根层重判/上下文刷新`
- `再决定是否重新入队或继续调度`

三步不能互相跳过。


## 5. 四层职责

### 5.0 与任务管理治理分支的边界

必须固定：

1. 本文描述的是全域“学习任务表 + 就绪队列 + 等待表 + 共享学习执行器”的共性运行结构。
2. 若服务对象是 `任务管理方法` 分支，则必须额外携带：
- `最近根层裁决结果`
- `当前缺口类型`
- `当前下一步去向`
3. 这三项属于治理上下文，不属于所有学习队列节点的统一强制字段。

### 5.1 学习任务表

学习任务表回答的是：

- 现在总共有哪几条学习任务
- 每条学习任务的来源、目标、反馈消费边界是什么
- 哪些任务处于就绪、等待、执行中、完成、失败、取消

学习任务表至少应承载：

- `学习任务ID`
- `学习任务类型`
- `来源根任务`
- `来源父任务 / 来源任务管理任务`
- `来源方法需求位`
- `学习目标类型`
- `学习类型`
- `目标约束强度`
- `缺口量 / 压力强度`
- `当前阶段`
- `当前状态`
- `当前优先级`
- `去重键`
- `可合并键`
- `方法服务目标`
- `最近结果摘要`


### 5.2 就绪队列

就绪队列回答的是：

- 现在有哪些学习任务已经可运行
- 下一个该由执行器取走谁

就绪队列不负责：

- 保存全部学习语义
- 替代学习任务表

就绪队列至少应保存：

- `学习任务ID`
- `当前调度优先级`
- `入队时间`
- `来源类型`
- `父任务优先级摘要`


### 5.3 等待表

等待表回答的是：

- 哪些学习任务目前不能继续执行
- 它们在等什么
- 何时应该被重新放回就绪队列

等待表至少应保存：

- `学习任务ID`
- `等待原因`
- `等待开始时间`
- `唤醒条件`
- `最近失败原因`
- `重试次数`

等待原因至少包括：

- `等待学习对象`
- `等待下一轮筹办生成尝试学习方法`
- `等待观察结果`
- `等待写回补偿`
- `等待父任务消费学习情况`
- `等待下一轮调度窗口`


### 5.4 共享学习执行器

共享学习执行器回答的是：

- 当前这一轮实际执行哪条学习任务的哪条 `待学习方法`
- 被选中的学习方法执行到哪个学习阶段
- 这一轮产出了什么

共享学习执行器至少负责：

- `承载被选中的学习方法`
- `执行当前待学习方法`
- `把本轮得到的条件-结果对融合写入方法侧`
- `采样`
- `更新`
- `验证`
- `提交`
- `回滚`

共享学习执行器不负责：

- 决定学习任务是否存在
- 决定学习任务是否就绪
- 绕过学习任务表直接长期持有全部学习状态
- 直接替代学习任务实体完成反馈消费
- 绕过外层重判直接决定正式召回、正式执行或成熟度升格


## 6. 为什么不能只用一个裸队列

如果只有一个裸 FIFO 队列，会缺少：

1. `去重`
- 相同来源、相同需求位、相同目标的学习任务会无限堆积

2. `合并`
- 相近的广泛学习任务无法合并执行

3. `等待语义`
- 不能区分“就绪可跑”和“暂时不能跑”

4. `反馈消费边界`
- 无法把学习情况准确交给正确的任务管理承接链消费

5. `阶段状态`
- 无法给每条学习任务保留自己独立的 `采样/更新/验证/提交/回滚`

因此：

`学习统一调度` 不等于 `只保留一个 FIFO 队列`。


## 7. 为什么不应交给通用任务管理任务直接执行

如果把学习任务重新当成普通任务交回通用任务管理任务直接执行，会带来：

- `业务任务管理` 与 `学习任务管理` 相互递归
- 学习任务阶段机与普通任务阶段机混杂
- 去重、合并、等待补偿难以表达
- 学习对象扫描与学习写回边界混乱

因此应固定：

- `任务管理任务`
  - 负责生成、更新、消费学习任务结果
- `学习调度器`
  - 负责选择下一条学习任务
- `共享学习执行器`
  - 负责真正执行学习

一句话：

`学习任务由任务管理任务生成，但不直接交给通用任务管理任务执行。`

并且必须继续固定：

- 共享学习执行器执行结束后，不得直接在本层内部连跑下一轮治理闭环。
- 正确顺序应为：`执行完成 -> 学习任务实体接结果并反馈任务管理任务 -> 返回外层根层重判/上下文刷新 -> 再决定是否继续调度`。


## 8. 学习优先级

学习调度优先级至少由以下因素合成：

- `来源根任务类型`
  - 默认 `安全 > 服务`
- `学习任务类型`
  - 默认 `任务子任务学习任务 > 任务管理独立学习任务`
- `学习类型`
  - 默认 `定向 > 弱定向 > 泛化`
- `来源方法需求位压力`
- `缺口量`
  - 例如 `缺失方法数量`
- `父任务优先级`
- `等待时长`
- `是否已接近完成验证/提交`

必须固定：

`安全上的定向补缺`
应默认高于
`服务上的广泛储备学习`


## 9. 去重与合并

### 9.1 去重键

学习任务至少应支持如下去重键：

- `来源根任务`
- `来源父任务或来源任务管理任务`
- `来源方法需求位`
- `学习目标类型`
- `学习类型`

若去重键完全相同，则不得无限重复创建新学习任务。


### 9.2 可合并键

学习任务至少应支持如下合并键：

- `学习目标类型`
- `来源方法需求位`
- `学习对象族`
- `结果消费边界`

满足合并条件时，允许共享一次执行，但必须：

- 保留原始学习任务列表
- 分开记录各自结果与反馈消费摘要


## 10. 两类学习任务如何进入调度器

### 10.1 任务子任务学习任务

来源于：

- 具体父任务
- 具体缺口
- 具体需求位

默认特点：

- 优先级更高
- 使用时更靠近父任务
- 目标更定向


### 10.2 任务管理独立学习任务

来源于：

- 任务管理压力上浮
- 方法需求位长期缺口
- 扩选/覆盖/健壮/可验证等压力

默认特点：

- 目标更泛化
- 可被统一调度延后
- 使用时更偏向方法资产和同需求位任务族

这两类任务都必须：

- 独立建实体
- 进入学习任务表
- 统一由学习调度器排序


## 11. 每个学习任务必须拥有自己的阶段状态

必须固定：

- 每个独立学习任务都应有自己的：
  - `当前阶段`
  - `最近阶段时间`
  - `最近学习对象`
  - `最近学习目标`
  - `最近结果`

禁止：

- 仅用一个全局学习工作流状态替代所有独立学习任务的阶段机

如果当前实现尚未达到这点，应视为：

- `共享学习执行器暂时复用单一工作流状态`
- 但这只是一种过渡实现
- 不得上升为最终架构


## 12. 最小落账要求

最小应落账如下状态：

- `学习任务_当前状态`
- `学习任务_当前阶段`
- `学习任务_当前优先级`
- `学习任务_去重键`
- `学习任务_可合并键`
- `学习任务_等待原因`
- `学习任务_最近调度时间`
- `学习任务_最近执行时间`
- `学习任务_最近结果摘要`


## 13. 当前实现收口建议

当前实现若要继续朝本规范靠拢，建议按以下顺序落地：

1. 先建立 `学习任务表`
2. 再建立 `就绪队列 + 等待表`
3. 再把当前共享学习执行链接到学习调度器
4. 最后把全局 `学习工作流状态` 拆成“每个学习任务一份状态”

并且始终保持：

- `学习任务表/队列/执行器` 只能作为承接层运行。
- 每轮消费结果后，仍必须回到外层重新做 `根层重判 -> 上下文刷新`。

不建议的顺序：

- 先继续堆更多学习目标和学习对象
- 但底层仍然只有一个全局工作流状态


## 14. 验收标准

满足以下条件，才可认为本规范落地：

1. 任意一条学习需求都能在任务树或学习树中找到独立学习任务实体。
2. 学习调度日志能明确显示：进入任务表、进入就绪队列、进入等待表、被执行器取走。
3. 不同来源的学习任务能统一进入同一调度器排序。
4. 学习结果能明确回写到对应学习任务实体，再反馈给任务管理任务决定如何服务父任务或方法资产。
5. 不再把“共享学习执行器正在跑”误当成“学习任务已经独立实现完毕”。
