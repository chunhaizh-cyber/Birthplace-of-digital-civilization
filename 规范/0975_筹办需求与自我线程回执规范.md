# 0975 筹办需求与自我线程回执规范

## 1. 目标

本文用于固定一条统一口径：

`筹办不直接硬编码“补什么”，而是先形成需求，再等待自我线程回执。`


## 2. 核心结论

必须固定以下结论：

1. 所有任务的筹办，都应尽量收成同一个通用规则：`少什么就提什么需求`。
2. 当前任务本地不直接裁定“这个需求一定能满足”；`需求是否可满足` 统一交给 `自我线程`。
3. 上述规则只属于筹办层，不得上升成系统总模型；更上游仍然是 `理解环 -> 特征空间生长 -> 可见缺口 -> 需求形成`。
4. `自我线程` 的回执，至少要回答：
   - 这个需求是否可满足
   - 需要等待还是可以继续
   - 返回哪个步骤、子任务、方法或结果节点
5. `任务管理任务` 的职责，是形成筹办需求、提交给自我线程、再把回执挂回任务树。
6. `可执行判断`、`允许执行判断` 不再作为一级主阶段保留；它们只是 `执行步骤入口前的筹办闸门`。
7. 任务树上真正长出来的步骤和子任务，应该来自“需求回执后的挂树动作”，而不是各处分散的特化硬编码。


## 3. 通用筹办规则

通用筹办规则固定为：

```text
读取当前目标
-> 读取当前树和虚拟存在特征
-> 判断当前缺口
-> 少什么就形成什么需求
-> 提交给自我线程
-> 等待需求回执
-> 把回执返回的节点挂回任务树
```

上面代码块只表示语义顺序，不表示直接可调用的实现签名。

这里必须区分两件事：

1. `缺口判断`
   它回答：当前到底少了什么。
2. `可满足性判断`
   它回答：这个缺口是否能在当前轮被满足，以及应由谁承接。

前者属于筹办；后者统一交给自我线程。


## 4. 需求回执的最小内容

任何筹办需求回执，至少要提供：

1. `需求集合`
2. `是否可满足`
3. `是否需要等待`
4. `返回步骤节点`
5. `返回结果节点`
6. `返回子任务头`
7. `返回方法头`
8. `等待原因`

若当前没有直接返回节点，也必须明确说明：

- 是继续等待
- 还是回到重筹办


## 5. 任务管理任务的特别约束

对 `任务管理任务` 来说，筹办必须固定成：

1. 读取宿主任务、当前步骤、当前结果、任务虚拟存在、方法虚拟存在。
2. 形成 `筹办需求集合`。
3. 把需求提交给自我线程。
4. 消费自我线程返回的 `可满足性 + 返回节点`。
5. 把这些节点挂回：
   - `任务管理任务树`
   - `宿主任务树`

因此以下动作不应再被理解成彼此独立的一级阶段：

- `可执行判断`
- `允许执行判断`

它们应统一降为：

`执行当前步骤前的筹办闸门`

也就是：

- 筹办把门条件交给执行步骤
- 执行步骤只消费这些门条件


## 6. “少什么就提什么需求”的最小映射

当前统一至少允许以下映射：

1. 没宿主任务：提 `宿主锚定需求`
2. 没当前步骤：提 `补步骤需求`
3. 没预测结果：提 `补结果需求`
4. 没方法：提 `方法补齐需求`
5. 方法不可用：提 `学习需求`
6. 缺条件：提 `补条件需求`
7. 缺结果侧：提 `补结果侧需求`
8. 缺前置承接对象：提 `前置子任务需求`
9. 实际结果偏离预测：提 `重筹办需求`


## 7. 禁止项

禁止以下做法：

1. 让 `任务管理任务` 本地直接硬编码决定“这个需求已经能满足”。
2. 把 `可执行判断`、`允许执行判断` 继续保留成和 `目标 / 筹办 / 执行` 并列的一级主阶段。
3. 在没有自我线程回执的情况下，直接把“推测能补的节点”当成正式返回节点。
4. 让多个特化 helper 分散决定补条件、补结构、补学习，而没有统一的筹办需求入口。
