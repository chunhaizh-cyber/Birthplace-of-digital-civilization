# 存在根概念元特征规范 20260509

## 1. 总原则

“存在”根概念只声明所有存在共有的元特征槽，不声明业务特征槽。

存在根概念的共性是：

```text
所有存在都可以被识别、被归类、被放入上下文、被追踪来源、被判断有效、被扩充记忆。
```

不是：

```text
所有存在都有位置、速度、目标、输入、输出、执行主体、占用状态。
```

根层如果挂太多业务特征，会迫使颜色、方法、任务、机位、航班、抽象概念都带上一堆不相干字段。根层必须轻，只作为概念树的骨架钉。

当前前提保持不变：

```text
1. 存在只有一种结构：存在节点主信息类。
2. 不新增抽象存在主信息类。
3. 不新增存在形态枚举。
4. 概念存在和实例存在靠所在树区分。
5. 实例可以对应多个概念存在。
6. 第一阶段用 概念集 作为实例概念归属的权威入口。
```

## 2. 存在根概念定义

存在根概念可以定义为：

```text
存在 = 可被命名、可被归类、可被放入上下文、可被追踪来源、可被判断有效、可被更新确认的对象。
```

存在根概念只提供元特征槽：

```text
存在
├─ 所属上下文
├─ 来源
├─ 存在状态
├─ 创建时间
├─ 更新时间
├─ 最近确认时间
├─ 可信度
└─ 描述 / 备注
```

其中前 7 个为推荐根特征槽，描述 / 备注为可选槽。

## 3. 必选根特征槽

### 3.1 所属上下文

表示这个存在位于哪个范围里。

```text
所属上下文 = 世界根 / 场景 / 内部世界 / 概念根 / 某个宿主存在
```

示例：

```text
B-1234飞机.所属上下文 = 当前机场运行场景
101机位.所属上下文 = 当前机场运行场景
加油任务T001.所属上下文 = 当前机场运行场景
飞机概念.所属上下文 = 存在概念根
```

如果特征值指向另一个存在、场景或基础信息节点，按“存在作为特征值”规范使用：

```text
特征值::指针句柄
```

### 3.2 来源

表示这个存在从哪里来。

推荐来源值：

```text
系统预置
自然语言解析
外部数据
人工创建
方法生成
观察生成
```

示例：

```text
飞机概念.来源 = 系统预置
B-1234飞机.来源 = 外部航班数据
加油任务T001.来源 = 方法生成
某未知对象X.来源 = 观察生成
```

来源用于支撑后续整理、归类和可信度治理。尤其在“实例在先、概念在后”的过程中，系统需要知道一个存在是观察出来的、语言解析出来的，还是规则 / 方法生成的。

### 3.3 存在状态

表示这个存在当前是否仍在系统认识范围内。

第一阶段只使用少量状态：

```text
未知
候选
有效
失效
已删除
```

示例：

```text
某雷达目标.存在状态 = 候选
B-1234飞机.存在状态 = 有效
过期任务T0001.存在状态 = 失效
```

注意：

```text
存在状态 != 飞机运行状态
存在状态 != 任务执行状态
存在状态 != 设备工作状态
```

区分：

```text
存在状态：所有存在都有。
任务状态：只有任务类存在有。
运行状态：飞机、航班、设备等下层概念才有。
```

### 3.4 创建时间

表示这个存在节点什么时候被系统创建。

```text
创建时间 = I64 时间戳
```

示例：

```text
飞机概念.创建时间 = 系统初始化时间
B-1234飞机.创建时间 = 第一次接入时间
任务T001.创建时间 = 任务生成时间
```

### 3.5 更新时间

表示这个存在的记忆最近什么时候被修改。

```text
更新时间 = I64 时间戳
```

以下更新都可以刷新更新时间：

```text
特征更新
概念集更新
需求更新
任务更新
方法更新
因果更新
描述更新
```

### 3.6 最近确认时间

表示系统最近一次观察或验证到该存在仍然成立的时间。

```text
最近确认时间 = I64 时间戳
```

它和更新时间不同：

```text
更新时间：系统改了它。
最近确认时间：系统又观察 / 验证到它还成立。
```

示例：

```text
B-1234飞机.更新时间 = 修改了当前位置
B-1234飞机.最近确认时间 = 最新一帧数据仍然命中它
```

一个对象没有被删除，不代表它仍被确认存在。长期存在的实例必须能区分“未删除”和“最近仍确认”。

### 3.7 可信度

表示系统对这个存在整体可信程度。

```text
可信度 = I64
```

推荐第一阶段区间：

```text
0 ~ 10000
```

示例：

```text
人工创建的机位概念.可信度 = 10000
外部系统同步的航班实例.可信度 = 9000
视觉识别出来的未知对象.可信度 = 5000
```

当前代码已有全局可信度比较痕迹，因此可信度可以作为根层共性元特征。

## 4. 可选根特征槽

### 4.1 描述

用于保存存在的语言说明。

```text
描述 = 指向文本节点 / 语言信息节点 / 备注节点
```

适合概念存在：

```text
飞机概念.描述 = “可飞行交通工具”
任务概念.描述 = “需要被完成的目标性虚拟存在”
方法概念.描述 = “用于完成任务的过程性虚拟存在”
```

实例存在不要求一定填写描述。

### 4.2 备注

用于保存人工补充、临时说明或调试说明。

```text
备注 = 指向文本节点 / 语言信息节点 / 备注节点
```

备注不是权威语义来源，不参与默认推理。

### 4.3 标签

标签可以作为轻量分类辅助，但不替代概念集。

```text
标签 = 文本 / 语言信息 / 标签存在
```

标签适合 UI、检索和临时分组，不作为实例概念归属的第一真相。

## 5. 不属于存在根的业务特征

以下不要挂在“存在”根概念上：

```text
位置
速度
尺寸
颜色
角色
执行主体
作用对象
输入
输出
目标
方法体
运行状态
占用状态
优先级
```

原因：

```text
飞机有位置，颜色概念不一定有位置。
任务有目标，机位不一定有目标。
方法有输入输出，旅客不一定有输入输出。
机位有占用状态，红色不一定有占用状态。
```

这些业务特征应放到更深的概念节点上。

示例：

```text
存在
├─ 实物存在
│  ├─ 位置
│  ├─ 尺寸
│  └─ 物理状态
│
├─ 任务
│  ├─ 任务目标
│  ├─ 执行主体
│  ├─ 作用对象
│  ├─ 完成条件
│  └─ 当前方法
│
├─ 方法
│  ├─ 输入
│  ├─ 输出
│  ├─ 前置条件
│  ├─ 执行步骤
│  └─ 方法体
│
└─ 机场资源
   ├─ 可调度
   ├─ 占用状态
   └─ 资源状态
```

层级越深，特征越细；根层只放元特征。

## 6. 结构字段不复制成根特征

以下虽然与所有存在有关，但第一阶段不作为根概念特征扩充。

### 6.1 名称、类型

`基础信息基类` 已有：

```text
名称
类型
```

它们是结构字段，不必复制为根概念特征。

规则：

```text
名称、类型 = 结构字段
名称特征、类型特征 = 可选镜像，不作为第一阶段必需
```

### 6.2 概念归属

第一阶段继续使用：

```cpp
std::vector<可解析引用<存在节点类>> 概念集{};
```

作为权威结构。

不要把“概念归属”强行降级成普通根特征。

可以允许镜像：

```text
B-1234.概念 = 飞机
B-1234.概念 = 交通工具
```

但第一阶段权威入口仍是：

```text
B-1234.概念集 = [飞机, 交通工具]
```

### 6.3 内部世界、需求根、任务根、方法根

这些是存在节点的结构能力，不是普通特征槽。

所有存在可以拥有：

```text
内部世界
需求根
任务根
方法根
因果实例
```

但这不等于“存在”根概念要给每个实例创建这些特征节点。

规则：

```text
内部世界、需求根、任务根、方法根、实例因果 = 结构能力
不作为根概念元特征扩充
```

## 7. 实例创建时的扩充顺序

根据概念创建实例时，应沿概念路径扩充。

示例一：

```text
存在
└─ 虚拟存在
   └─ 任务
      └─ 加油任务
```

创建“加油任务T001”时，扩充顺序：

```text
存在根特征
+ 虚拟存在特征
+ 任务特征
+ 加油任务特征
```

结果：

```text
加油任务T001
├─ 所属上下文
├─ 来源
├─ 存在状态
├─ 创建时间
├─ 更新时间
├─ 最近确认时间
├─ 可信度
├─ 任务目标
├─ 执行主体
├─ 作用对象
├─ 完成条件
└─ 当前方法
```

示例二：

```text
存在
└─ 实物存在
   └─ 交通工具
      └─ 飞机
```

创建“B-1234飞机”时，扩充顺序：

```text
存在根特征
+ 实物存在特征
+ 交通工具特征
+ 飞机特征
```

结果：

```text
B-1234飞机
├─ 所属上下文
├─ 来源
├─ 存在状态
├─ 创建时间
├─ 更新时间
├─ 最近确认时间
├─ 可信度
├─ 位置
├─ 速度
├─ 航向
├─ 运行状态
└─ 占用机位
```

第一阶段扩充只复制 / 实例化概念特征，不做动态继承和复杂冲突治理。

## 8. 根元特征建议命名

推荐使用以下词性入口作为根元特征类型：

```text
所属上下文
来源
存在状态
创建时间
更新时间
最近确认时间
可信度
描述
备注
标签
```

这些词性入口可进一步对应基础信息层中的存在概念或抽象特征模板，但命名本身不得混入业务含义。

## 9. 禁止项

禁止：

```text
1. 在“存在”根概念上挂位置、速度、颜色、目标、输入、输出等业务特征。
2. 给所有存在强制创建任务目标、执行主体、作用对象、方法体等任务 / 方法特征。
3. 把名称、类型、概念集、内部世界、需求根、任务根、方法根复制成根概念特征。
4. 用存在根特征替代下层概念特征。
5. 在根层引入存在形态、抽象存在类或复杂归属记录。
```

允许：

```text
1. 在根层声明所属上下文、来源、存在状态、创建时间、更新时间、最近确认时间、可信度。
2. 可选声明描述、备注、标签。
3. 下层概念继续扩充自己的业务特征。
4. 创建实例时沿概念路径逐层扩充特征。
```

## 10. 验收标准

实现和数据应满足：

```text
1. 存在根概念只含元特征槽。
2. 存在根概念不含位置、速度、输入、输出、目标、占用状态等业务特征。
3. 名称、类型仍为结构字段。
4. 概念归属仍以 概念集 为第一阶段权威入口。
5. 内部世界、需求根、任务根、方法根仍为结构能力。
6. 创建实例时会获得存在根元特征。
7. 创建下层实例时会叠加下层概念业务特征。
8. 重复特征按特征类型或抽象特征模板去重。
```

## 11. 最终定稿

```text
“存在”根概念的特征不是业务特征，而是元特征。

所有存在共有：
1. 所属上下文
2. 来源
3. 存在状态
4. 创建时间
5. 更新时间
6. 最近确认时间
7. 可信度
8. 描述 / 备注，可选

名称、类型、概念集、内部世界、需求根、任务根、方法根，属于结构字段或结构能力，不必作为根概念特征复制。

位置、速度、目标、输入、输出、执行主体、占用状态等，属于下层概念特征，不属于“存在”根概念。
```
