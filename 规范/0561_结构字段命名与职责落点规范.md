# 0561 结构字段命名与职责落点规范

## 1. 目标

本文用于统一回答以下问题：

- 新增字段时，什么时候该命名为 `特征类型`，什么时候该命名为 `名称 / 摘要 / 主键 / 节点主键 / ID`
- 为什么不能继续使用 `索引键 / 归桶键 / 摘要键 / 稳定主键 / 候选键` 这类过泛名字
- 规范、代码、日志、控制面板中，字段名应如何与真实职责对齐
- 当一个对象既要可读、又要可检索、又要可回指时，哪些字段应并存，哪些字段不能互相偷换

## 2. 统一结论

后续新增字段时，必须先判定它属于哪一类职责，再决定名字。

统一只允许先做这一层判断：

1. 它表达的是 `可比较的维度类别`，还是 `具体对象身份`
2. 它表达的是 `人工可读解释`，还是 `系统回指与检索`
3. 它表达的是 `节点身份`，还是 `节点指针`
4. 它表达的是 `长期稳定引用`，还是 `运行期临时缓存`

因此必须固定：

1. `特征类型` 只表达“比较维度或作用维度是什么”。
2. `名称` 只表达“给人看时怎么叫”。
3. `摘要` 只表达“压缩解释或审计提示”。
4. `主键` 只表达“这个对象、关系、条件、结果或路径是谁”。
5. `节点主键` 只表达“某个已落节点对象的正式身份回指”。
6. `ID` 只表达“跨线程、跨模块、跨快照、跨重启仍需稳定引用的编号型身份”。

统一优先规则是：

- 先用具体领域名词，再加字段职责后缀
- 不得先写一个抽象名字，再靠注释解释真实职责

例如：

- `需求特征类型列表`
- `输入条件主键`
- `最近结果节点主键`
- `动态路径主键`
- `学习任务ID`
- `最近反馈摘要`

而不应优先写成：

- `需求索引键`
- `结果归桶键`
- `状态摘要键`
- `稳定主键`
- `候选键`


## 3. 字段后缀的固定语义

| 后缀 | 固定语义 | 典型示例 |
| --- | --- | --- |
| `名称` | 人工可读名称，不承担唯一性 | `动作名称`、`主体名称` |
| `类型` | 枚举或分类，不一定可比较 | `来源类型`、`语义类别` |
| `特征类型` | 可比较、可归纳、可映射的维度类别 | `主结果特征类型` |
| `特征类型列表` | 多个维度类别的并列表达 | `需求特征类型列表` |
| `主键` | 对象、条件、结果、路径、关系的身份字段 | `任务主键`、`条件主键` |
| `节点主键` | 某个已正式落节点对象的身份回指 | `结果节点主键`、`方法首节点主键` |
| `ID` | 长期稳定编号 | `学习任务ID`、`因果ID` |
| `摘要` | 解释、提示、审计文本 | `最近反馈摘要`、`结果摘要原文` |
| `原文` | 未压缩的原始表达 | `结果摘要原文` |
| `列表` | 多个同类元素集合 | `条件状态列表`、`过程动态列表` |
| `数量` | 计数值 | `特征类型数量`、`观察次数` |
| `时间` | 时间戳或时间点 | `最近发生时间`、`创建时间` |
| `指针` | 运行期内存引用，不承担持久身份 | `当前方法首节点指针` |

必须额外固定：

- `类型` 和 `特征类型` 不是一回事
- `主键` 和 `ID` 不是一回事
- `名称` 和 `摘要` 不是一回事
- `节点主键` 和 `节点指针` 不是一回事


## 4. 各类字段的命名边界

### 4.1 什么时候用 `特征类型`

当字段表达的是：

- 当前对象参与比较的维度
- 需求、方法、结果、状态的作用面
- 召回、映射、筛选时使用的维度类别

统一命名为：

- `特征类型`
- `主结果特征类型`
- `目标特征类型列表`
- `影响特征类型列表`

必须固定：

1. `特征类型` 不直接表示某个具体结果已经成立。
2. `特征类型` 不替代 `状态`、`结果节点` 或 `二次特征`。
3. 若字段里存的其实是具体对象身份，就不能叫 `特征类型`。

### 4.2 什么时候用 `名称`

当字段只负责：

- 对外展示
- 人工阅读
- UI 文案
- 日志解释

统一命名为：

- `名称`
- `主体名称`
- `来源动作名`

必须固定：

1. `名称` 不承担唯一性。
2. `名称` 不直接承接并头、去重、归桶。
3. 只有名称、没有主键或结构字段时，不得视为身份稳定。

### 4.3 什么时候用 `摘要`

当字段只负责：

- 把复杂结果压成可读提示
- 审计、恢复、调试时给人快速看懂
- 作为候选解释镜像

统一命名为：

- `摘要`
- `最近反馈摘要`
- `结果摘要原文`
- `主结果状态摘要`

必须固定：

1. `摘要` 不是正式身份字段。
2. `摘要` 不是最终判定依据。
3. 若内部需要把摘要压成哈希，只能视为实现细节，暴露名仍优先写 `主键` 或 `摘要原文`。

### 4.4 什么时候用 `主键`

当字段表达的是：

- 某个对象、条件、结果、路径、锚点、关系的正式身份
- 去重、回指、查询、关联所依赖的唯一身份

统一命名为：

- `任务主键`
- `输入条件主键`
- `结果主键`
- `动态路径主键`
- `动作句柄主键`
- `锚点主键`

必须固定：

1. `主键` 必须带领域前缀，不得裸写一个孤立的 `主键` 作为跨模块字段名。
2. 若该字段只在某个具体节点层有效，应优先写成 `节点主键`。
3. `主键命中` 只表示找到了候选，不表示结论成立。

### 4.5 什么时候用 `节点主键`

当字段表达的是：

- 某个已经正式落在节点层的对象身份
- 恢复快照、任务镜像、控制面板、审计链需要回指到具体节点

统一命名为：

- `结果节点主键`
- `方法首节点主键`
- `条件节点主键`

必须固定：

1. `节点主键` 优先用于“回到具体节点”。
2. 若只是抽象结果桶或抽象条件桶身份，不强制带 `节点` 二字。
3. `节点主键` 和 `节点指针` 应尽量成对理解，但不能互相偷换。

### 4.6 什么时候用 `ID`

当字段表达的是：

- 明确采用编号语义
- 需要跨线程、跨模块、跨快照、跨重启长期稳定引用
- 当前工程里已形成稳定 `ID` 传统

统一命名为：

- `学习任务ID`
- `因果ID`

必须固定：

1. 规范可以讨论 `稳定ID` 这类职责概念，但实际字段命名应优先落成 `具体领域名 + ID`。
2. 不建议新增字段名 `稳定主键`。
3. 若某字段本质是对象主键而不是编号，不要为了“显得稳定”改叫 `ID`。

### 4.7 什么时候用 `主键字段组合`

当字段真的在存：

- 一个显式拼好的组合身份
- 多个身份字段拼出的可回溯组合

才允许命名为：

- `主键字段组合`
- `候选主键字段组合`

必须固定：

1. 若系统里已经分别保留了各个原字段，优先保留原字段，不额外新建平行组合字段。
2. 若只是“运行时拿几个字段一起比”，但没有独立存这个组合，不要把逻辑过程写成字段名。
3. `字段组合` 只能是显式结构，不得拿来包装含糊缩写。


## 5. 统一判定顺序

新增字段时，默认按以下顺序命名：

1. 先问：这是不是 `给人看的解释文本`
   - 是：优先用 `名称 / 摘要 / 原文`
2. 再问：这是不是 `比较维度`
   - 是：优先用 `特征类型 / 特征类型列表`
3. 再问：这是不是 `对象或关系身份`
   - 是：优先用 `主键 / 节点主键`
4. 再问：这是不是 `长期稳定编号`
   - 是：优先用 `具体领域名 + ID`
5. 再问：这是不是 `运行期引用`
   - 是：优先用 `指针 / 句柄`

若以上都不满足，才允许回到更具体的场景里另行补名。

不得反过来先起：

- `索引键`
- `归桶键`
- `摘要键`
- `稳定主键`
- `候选键`

再让阅读者猜它实际是 `名称`、`主键` 还是 `摘要`。


## 6. 各域推荐命名口径

| 域 | 优先字段名 |
| --- | --- |
| 需求 | `需求类型`、`需求主体主键`、`被需求状态主键`、`需求特征类型列表` |
| 任务 | `任务主键`、`最近条件主键`、`最近结果节点主键`、`最近反馈摘要`、`最近下一步去向` |
| 方法 | `动作键`、`动作句柄主键`、`条件主键`、`结果主键`、`主结果特征类型`、`方法首节点主键` |
| 因果 | `条件主键`、`过程主键`、`结果主键`、`因果ID` |
| 学习 | `学习任务ID`、`来源任务主键`、`来源方法主键`、`学习对象类型` |
| 动态与路径 | `动态路径主键`、`步骤链主键`、`子树主键`、`主结果状态摘要` |
| 概念 | `概念主键`、`关系类型`、`适用条件主键`、`失效条件主键` |

必须额外固定：

- 同一职责在跨域出现时，优先复用同一后缀语义
- 不得在任务域叫 `结果节点主键`，到控制面板又改叫 `稳定主键`
- 不得在方法域叫 `主结果特征类型`，到学习域又改叫 `结果标签`


## 7. 禁止事项

以下做法统一视为违规：

1. 把 `名称` 当 `主键` 使用。
2. 把 `摘要` 当 `最终结果` 使用。
3. 把 `特征类型` 当 `具体结果状态` 使用。
4. 把 `节点指针` 当 `长期身份字段` 使用。
5. 新增字段名 `索引键 / 归桶键 / 摘要键 / 稳定主键 / 候选键 / 命中键`。
6. 只有内部哈希，没有原字段或原文镜像，却把它当成跨模块正式字段。
7. 一个字段同时承担 `名称 + 主键 + 摘要 + 最终裁决` 多重职责。


## 8. 迁移与兼容要求

后续规范与实现统一按以下口径迁移：

1. 旧字段若还能保留，只能作为兼容层，不再作为新增命名模板。
2. 新增字段优先先落具体领域名，再落职责后缀。
3. 规范正文、代码字段、日志字段、控制面板字段，应尽量保持同名或可直接映射。
4. 当已有 `主键` 或 `节点主键` 时，不得再平行新增 `稳定主键` 之类泛名。
5. 当已有 `摘要原文` 时，不得只对外暴露一个不可回读的摘要哈希。


## 9. 最终结论

以后统一按“具体领域名词 + 明确职责后缀”命名字段。

一句话固定为：

`维度类字段叫特征类型，解释类字段叫名称或摘要，身份类字段叫主键或节点主键，编号类字段叫具体领域名 + ID。`
