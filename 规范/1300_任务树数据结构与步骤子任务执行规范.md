### 2.1 生命周期裁判边界

任务主生命周期职责固定为：

- `任务类`：只负责任务头/步骤/结果节点的数据结构与原子操作。
- `任务执行器`：只负责单任务推进，不直接裁决任务头主生命周期。
- `任务管理器_v0`：负责任务头生命周期迁移、子任务创建绑定、结果写回，以及根任务治理时的统一裁判入口。

不再允许把“任务头状态裁决”和“单步执行推进”分散写在多个模块里长期并存。

本文负责：

- 把任务树的结构层和运行层拆清楚
- 固定“步骤”和“子任务”的定义边界
- 固定“执行”和“等待”的最小树上表达
- 固定第一层必须遵守的结构约束

- 方法评分算法

- 安全值、服务值的具体数值公式

## 3. 核心结论

必须固定以下结论：

1. 当前任务树的核心节点种类只有三类：`任务头结点`、`任务步骤节点`、`任务结果节点`。
2. `子任务` 不是第四种节点种类；`子任务` 本质上仍然是一个新的 `任务头结点`。
3. `步骤` 表示“当前任务内部的一个执行位或等待位”。
4. `子任务` 表示“从当前任务向下分解出的一个独立治理单元”。
5. “等待子任务完成”本身可以是一个步骤节点；但被等待的那件独立工作，仍然必须是子任务。
6. `任务头结点 -> 步骤节点 -> 结果节点` 是当前任务树的最小主链。
7. `步骤节点` 可以挂并发子任务，`结果节点` 可以挂派生子任务。
8. 任务树不是纯物理树，还同时包含“逻辑回指关系”和“运行账本层”。
9. 主信息中的字段负责运行期镜像与热路径读取；任务虚拟存在中的特征负责统一写回、状态记录和后续检索。
10. 真正执行的是方法绑定在步骤上的一次运行；真正向下扩张的是子任务头结点。

可收束为一句话：

`任务树 = 头结点主干 + 步骤执行位 + 结果分支位 + 子任务分解位 + 任务虚拟存在账本。`


## 4. 任务树的两层结构

当前任务树必须明确区分两层：

### 4.1 物理节点层

物理节点层就是 `任务链` 上真实存在的节点结构。

当前物理节点只允许承载三种主信息：

- `任务头结点信息`
- `任务步骤节点信息`
- `任务结果节点信息`

物理层负责：

- 父子挂接
- 节点创建与删除
- 基本遍历顺序

### 4.2 运行语义层

运行语义层不只由“父子指针”构成，还包括：

- `父任务头结点`
- `来源父步骤节点`
- `来源父结果节点`
- `所属任务头结点`
- `当前步骤节点`
- `当前子任务头结点`
- `最近实际结果节点`
- `并发子任务头结点列表`
- `派生子任务头结点列表`
- `任务虚拟存在`
- `任务事实/方向/执行账特征`

因此必须固定：

- 任务树是“物理树 + 逻辑回指 + 运行账本”的复合结构
- 不能只看 `父/子/下` 指针来理解任务树语义


## 5. 节点模型

### 5.1 任务信息基类

`任务信息基类` 是所有任务节点的公共底座，至少承载以下内容：

- 基本标识：名称、描述、类型、节点种类
- 运行状态：状态、创建时间
- 优先级：基准优先级、局部优先级偏移、调度优先级
- 场景锚点：当前节点对应的条件/执行/结果场景
- 任务虚拟存在：运行账和特征写回锚点
- 任务树类型：`编排任务` / `叶子任务`
- 执行流水账：运行次数、成功次数、失败次数、连续失败次数、转入学习次数、最近错误码、启动时间、完成时间、最近调度时间、执行方法

必须固定：

- 基类字段解决的是“所有任务节点都要带的运行期共性”
- 不允许把头结点专属、步骤专属、结果专属字段继续堆进基类

### 5.2 任务头结点

`任务头结点` 负责表达一个完整任务实例的主干。

当前必须承载：

- `需求`
- `是否真根任务`
- `是否常驻任务`
- `父任务头结点`
- `来源父步骤节点`
- `来源父结果节点`
- `当前步骤节点`
- `当前子任务头结点`
- `最近实际结果节点`
- `直属子任务头结点列表`
- `目标特征类型`

头结点回答的问题是：

- 这个任务是谁在负责
- 这个任务现在做到哪一步
- 这个任务最近产出了什么结果
- 这个任务已经分解出了哪些下级子任务

### 5.3 步骤节点

`步骤节点` 负责表达“当前任务内部的一个执行位”。

其中更细的步骤字段合同，当前统一按 [1311_任务步骤结构规范_v0.1.md](D:\鱼巢\规范\1311_任务步骤结构规范_v0.1.md) 执行；本文只保留任务树视角下的节点职责与边界。

当前必须承载：

- `所属任务头结点`
- `当前选中方法首节点`
- `备用方法首节点列表（按需）`
- `并发子任务头结点列表`
- `当前子任务头结点`
- `步骤序号`
- `已重试次数`
- `允许重试次数`
- `超时截止时间`
- `允许切换方法`

步骤节点回答的问题是：

- 当前这一步准备用哪条步骤方案推进、当前采用什么方法做
- 这一步是否可切换到备用方法、可重试、已超时
- 这一步有没有挂出补条件子任务或并发子任务
- 当前父任务是否应停在这里等待

必须补充固定：

- 并列步骤才是同层候选方案的主组织面，不再要求把所有候选方法继续堆进单一步骤的列表里。
- 不同条件链、不同结果链或不同收口去向的候选方法，必须拆成并列步骤节点。
- 同一步骤若允许局部换法，只保留 `当前选中方法首节点` 与少量 `备用方法首节点列表`。

### 5.4 结果节点

`结果节点` 负责表达“步骤下产出的一个结果位”。

必须补充固定：

- 步骤节点建立后，应立即至少建立一个结果节点，作为本步骤的初始化结果承接面。
- 这个初始化结果节点默认按 `预测结果` 或 `占位结果` 理解，用于承接后续命中、转步、派生和回填，不等于本步已经完成。
- 初始化结果节点必须同时给出 `结果方向二次特征`；它是结果特征状态的二次特征，初始化时按 `步骤目标状态` 对 `当前状态` 生成，后续再按更明确的预测/实际结果状态刷新。

当前必须承载：

- `结果角色`：预测结果 / 实际结果
- `所属任务头结点`
- `来源步骤节点`
- `派生子任务头结点列表`
- `来源方法首节点`
- `对应方法结果节点`
- `结果方向二次特征`
- `命中预测结果节点`
- `命中后动作`
- `是否预测外结果`
- `置信度_Q10000`
- `错误码`
- `评估摘要存在`

结果节点回答的问题是：

- 这一步预期会出现什么结果
- 实际执行后出现了什么结果
- 实际结果是否命中了某个预测结果
- 命中后下一步该继续、回跳、补条件、转学习还是挂起


## 6. 步骤与子任务的边界

### 6.1 步骤的定义

步骤必须固定理解为：

`当前任务内部的一个执行位、等待位、检查位或回接位`

步骤节点本身不等于独立任务，它只是父任务内部的一个过程位置。

### 6.2 子任务的定义

子任务必须固定理解为：

`从父任务分解出来的、拥有自己任务头、自己步骤、自己结果、可继续被治理的独立任务实例`

因此子任务在结构上仍然是新的 `任务头结点`。

### 6.3 什么时候长步骤

以下情况应优先建模为步骤：

- 当前任务内部的单步执行
- 当前任务内部的方法切换位
- 当前任务内部的检查点
- 当前任务内部的等待位
- 当前任务内部的回接位
- “发起子任务并等待其完成”这一父任务过程位

### 6.4 什么时候必须长子任务

以下情况必须建模为子任务，而不能只停留在步骤：

- 该工作有独立目标
- 该工作需要自己选择方法
- 该工作需要自己的结果链
- 该工作可能继续细分
- 该工作失败后需要独立纠偏或收束
- 该工作完成后会回写给父任务继续推进

### 6.5 等待子任务的规范

必须固定：

- “等待子任务完成”可以作为步骤节点
- “被等待的独立工作”必须是子任务头结点
- 父步骤在等待期间应进入 `等待中` 或 `挂起`
- 父任务的 `当前子任务头结点` 与该步骤的 `当前子任务头结点` 应同步指向被等待的子任务

可收束为一句话：

`等待是步骤语义，分解才是子任务语义。`


## 7. 标准结构形态

### 7.1 最小主链

当前标准最小主链固定为：

```text
任务头结点
-> 步骤节点
-> 结果节点
```

这里的含义是：

- 头结点负责“这是什么任务”
- 步骤节点负责“现在做哪一步”
- 结果节点负责“这一步出现了什么结果”

### 7.2 补条件结构

当前补条件或补前置的标准结构固定为：

```text
父任务头结点
-> 父步骤节点（等待位）
   -> 并发子任务头结点
```

语义固定为：

- 父步骤负责发起并等待
- 子任务负责独立补齐缺口

### 7.3 结果派生结构

当前结果驱动下级任务的标准结构固定为：

```text
步骤节点
-> 结果节点
   -> 派生子任务头结点
```

语义固定为：

- 步骤先产出结果
- 某个结果命中后决定“派生哪个子任务”

### 7.4 下一步挂接结构

当前下一步标准挂接口径固定为：

```text
步骤A
-> 结果A
   -> 步骤B
```

必须固定：

- 下一步不是直接挂在上一个步骤后面
- 下一步是挂在“哪个结果被命中”之后
- 这样才能保留分支语义


## 8. 执行规范

### 8.1 执行单元

当前最小执行单元必须固定为：

`当前选中的步骤节点 + 该步骤的当前选中方法首节点`

并列步骤的选择先于方法执行；不是头结点直接执行，也不是结果节点直接执行。

### 8.2 当前指针

运行期至少维护以下三个“当前指针”：

- 头结点的 `当前步骤节点`
- 头结点或步骤节点的 `当前子任务头结点`
- 头结点的 `最近实际结果节点`

它们分别回答：

- 当前主链走到哪里
- 当前被哪个子任务接管
- 最近一次真实产出了什么

### 8.3 执行开始

执行开始时，至少应同步以下内容：

- `运行次数`
- `启动时间`
- `执行方法`
- `当前步骤节点`
- `最近调度时间`

### 8.4 执行结束

执行结束时，至少应同步以下内容：

- `成功次数 / 失败次数 / 连续失败次数`
- `最近错误码`
- `完成时间`
- `执行方法`
- `最近实际结果节点`
- 是否 `转入学习`

### 8.5 执行与结果的关系

必须固定：

- 步骤执行后产生的是结果节点
- 结果节点用于命中预测、判断分支和挂接下一步
- 不能让“状态变化”跳过结果节点直接切下一步


## 9. 状态规范

### 9.1 头结点状态

头结点状态回答的是：

- 这个任务整体现在处于什么阶段

它允许进入：

- `就绪`
- `筹办中`
- `排队中`
- `运行中`
- `等待中`
- `挂起`
- `待重筹办`
- `完成`
- `失败`
- `取消`
- `超时`
- `无法执行`

### 9.2 步骤状态

步骤状态回答的是：

- 这一执行位当前是否可做、正在做、在等待，还是已经结束

当前必须允许用步骤状态表达：

- 就绪步骤
- 执行中步骤
- 等待子任务的步骤
- 挂起等待外部条件的步骤
- 已完成步骤
- 已失败步骤

### 9.3 等待与挂起

必须固定区分：

- `等待中`：已知要等谁、等什么，后续条件满足后应继续回到当前步骤
- `挂起`：当前步骤被显式暂停，恢复条件不一定已经明确


## 10. 任务虚拟存在与账本规范

### 10.1 任务虚拟存在

每个任务节点都允许拥有自己的 `任务虚拟存在`。

其职责是：

- 承载运行账
- 承载任务事实
- 承载任务方向
- 承载状态写回
- 为后续检索、因果归纳和治理调度提供统一入口

### 10.2 三类写回口径

任务树上的写回必须统一分为三类：

- `执行账`
  - 运行次数
  - 成功次数
  - 失败次数
  - 连续失败次数
  - 转入学习次数
  - 最近错误码
  - 最近开始/结束/调度时间
  - 当前方法、最近方法、最近结果

- `任务事实`
  - 当前业务事实
  - 当前资源事实
  - 当前环境事实
  - 当前等待对象
  - 当前结算动作回写

- `任务方向`
  - 状态是否向目标接近
  - 某个关键值的最近变化方向
  - 某个动作对根需求是否正向

### 10.3 主信息与虚拟存在的关系

必须固定：

- 主信息字段是热路径镜像
- 任务虚拟存在特征是统一写回面
- 后续查询、因果、调制和回放，应优先从任务虚拟存在取值


## 11. 创建与绑定规范

### 11.1 创建头结点

`需求 -> 任务头结点` 是唯一合法的任务成立入口。

必须固定：

- 一个需求默认只保一个关联头结点
- 头结点可自动回写到 `需求.相关任务`

### 11.2 创建步骤节点

步骤节点可挂在：

- 任务头结点下，表示当前阶段的首步或并列步骤
- 结果节点下，表示命中某个结果后的下一步

必须固定：

- 步骤是“当前任务内部”的节点
- 步骤不能脱离所属任务头结点而独立存在
- 若候选方案对应不同前置条件、不同结果链或不同完成去向，必须拆成并列步骤，而不是只塞到单一步骤的方法集合里
- 步骤节点创建后，应立即在其下建立至少一个结果节点，并完成初始化结果字段写入

### 11.3 创建结果节点

结果节点必须挂在步骤节点下。

必须固定：

- 结果不允许直接挂在头结点下
- 结果节点必须能回指出处于哪个步骤、由哪个方法产出
- 主链是否收口，应看头结点完成判据是否成立，而不是要求结果节点与头结点目标字段字面相同
- 初始化结果节点至少应写入：`结果角色`、`所属任务头结点`、`来源步骤节点`、`来源方法首节点（若已选定）`、`对应方法结果节点（若可推导）`、`结果方向二次特征`、`命中后动作`、`是否预测外结果`、`置信度_Q10000`、`错误码`

### 11.4 绑定子任务

子任务绑定必须至少经过以下两种之一：

- `步骤 -> 并发子任务`
- `结果 -> 派生子任务`

不允许出现：

- 只有父任务知道，步骤和结果都不知道的“悬空子任务”


## 12. 结构硬约束

后续代码与规范校验必须至少满足以下约束：

1. 任务树核心节点种类只允许 `头结点 / 步骤节点 / 结果节点` 三类。
2. 子任务不是新节点种类，子任务必须仍然是 `任务头结点`。
3. 一个步骤必须只属于一个任务头结点。
4. 一个结果必须只来源于一个步骤节点。
5. 等待子任务完成可以作为步骤，但不能把子任务本体压扁成步骤。
6. 需要独立目标、独立方法或独立结果链的工作，必须拆成子任务。
7. 下一步默认挂在结果节点之后，不直接跳过结果节点。
8. 执行单元固定为 `步骤 + 当前方法`，不是头结点直接执行。
9. 结构差异明显的候选推进路径，必须拆成并列步骤，不得长期塞在单一步骤的方法列表里。
10. 任务是否正式收口，以头结点完成判据成立为准，不以结果节点与头结点目标字段字面相同为准。
11. 实际结果必须能回指预测结果或明确标注为预测外结果。
12. 任务运行账、任务事实、任务方向应统一写入任务虚拟存在，不得继续分散为无入口的临时字段。


## 13. 一句话收束

`步骤` 是任务内部的执行位和等待位，`子任务` 是任务向下分解出的独立治理单元；任务树由 `头结点 -> 步骤节点 -> 结果节点` 主链，加上 `步骤挂并发子任务`、`结果挂派生子任务` 两类扩展结构共同组成。`
