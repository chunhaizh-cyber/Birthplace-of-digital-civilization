# 1300 任务树数据结构与步骤子任务执行规范

## 1. 目标

本文用于把所有任务统一收口成同一棵树结构。


## 2. 核心结论

必须固定以下结论：

1. 所有任务都遵循 `头结点 -> 步骤节点 -> 结果节点` 的基本链。
2. `头结点` 表达目标。
3. `步骤节点` 表达当前准备执行或等待执行的中间步骤。
4. `结果节点` 表达当前步骤的结果，以及是否需要长下一步。
5. `子任务` 不是新节点种类；子任务仍然是新的头结点。
6. 任务与方法各自的 `虚拟存在` 是主逻辑特征面，树节点只保存当前实例锚点与运行镜像。
7. `步骤任务` 由筹办建立，属于主步骤链。
8. `前置任务` 由执行方法运行中提出需求，再由自我线程实例化并挂回当前步骤，不属于主步骤链。


## 3. 头结点

头结点必须至少承载：

1. 标的
2. 目标
3. 当前步骤节点
4. 最近实际结果节点
5. 步骤数量

对三类任务分别表现为：

- 普通任务：目标存在、目标特征、目标值
- 任务管理任务：宿主任务、完成判据
- 学习任务：目标方法、可用判据


## 4. 步骤节点

步骤节点必须至少承载：

1. 所属任务头结点
2. 当前方法首节点
3. 方法实例条件节点
4. 当前活动结果节点
5. `并发前置任务头结点列表`

步骤节点的职责只有一个：

`表达当前任务接下来到底执行哪一步`

补充固定：

1. 步骤节点属于主步骤链，不等同于前置任务。
2. 步骤节点若需要外部支撑，应把这些支撑任务挂在 `并发前置任务头结点列表`，而不是把它们写成下一步主步骤。
3. 当前步骤只有在前置任务全部满足后，才允许继续执行。


## 5. 结果节点

结果节点必须至少承载：

1. 来源步骤节点
2. 方法实例结果节点
3. 本步结果状态
4. `下一跳步骤节点`
5. 结果角色（预测 / 实际）

结果节点的职责只有一个：

`承接当前步骤的产物，并决定是否回到筹办继续长树`

必须固定：

1. 一个步骤节点下面允许存在多个结果节点。
2. 这些结果节点可以同时包含预测结果和实际结果。
3. 若结果节点的 `下一跳步骤节点` 非空，则它是下一步的权威枝路；为空时才回退到结果节点的子步骤。
4. 若实际结果与命中的预测结果不一致，则任务状态转入 `待重筹办`，并回到筹办阶段改写任务树。
5. 结果节点不得再复用 `后续任务` 承载“下一步步骤”；该语义只允许落在 `下一跳步骤节点`。


## 6. 前置任务与派生任务

任务树里需要区分两类“头结点形式的子任务”：

### 6.1 前置任务

前置任务的定义固定为：

- 由执行方法在运行中发现额外缺口后提出
- 由自我线程实例化
- 挂在当前步骤节点的 `并发前置任务头结点列表`
- 用于让当前步骤变得可继续执行

因此：

- 补条件
- 补材料
- 补依赖
- 补学习样本

只要是在执行方法运行中才显现出来的支撑需求，都属于前置任务，而不属于主步骤链。

### 6.2 派生任务

派生任务的定义固定为：

- 由某个结果节点触发
- 挂在结果节点的 `派生子任务头结点列表`
- 表达“这个结果导致系统决定再开一项独立任务”

### 6.3 主步骤链与前置链不得混用

必须固定：

1. 筹办产生的是主步骤任务，不是前置任务。
2. 执行方法产生的是前置任务需求，不是下一步主步骤。
3. 前置任务完成后，应回到当前步骤继续执行；不应直接把前置任务当作主步骤链的一环。


## 7. 执行

当前统一执行链固定为：

```text
头结点读取目标
-> 当前步骤执行方法
-> 结果写入结果节点
-> 比对目标
-> 达成则完成
-> 未达成则回筹办
```
