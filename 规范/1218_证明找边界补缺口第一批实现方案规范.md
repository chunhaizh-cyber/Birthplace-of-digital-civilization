# 1218 证明找边界补缺口第一批实现方案规范

## 1. 目标

本文用于把 `验证与完善型学习` 里最先应落地的三组能力收成第一批实现方案：

- `证明`
- `找边界`
- `补缺口`

本文回答的是：

- 这三组能力在工程上最少应该怎样实现
- 每组能力的输入、处理中间量、最小产出、默认写回位置是什么
- 当前代码已经有什么原料
- 后续应按什么顺序落地

本文不展开：

- `扩覆盖`
- `降成本`
- `替代路径完整编排`
- `全量验证与完善调度器`

## 3. 核心结论

必须固定以下 9 条：

1. 第一批实现不追求“完整验证与完善系统”，而是先打通 `证明 / 找边界 / 补缺口` 三条最短闭环。
2. `证明` 先围绕 `可靠度 / 稳定复现 / 结果分裂 / 结果确认` 落账。
3. `找边界` 先围绕 `禁用条件 / 回退条件 / 未确认外部结果 / 失败条件桶` 落账。
4. `补缺口` 先围绕 `条件模板补全 / 结果模板补全` 落账。
5. 第一批实现允许“先计数、先摘要、先写任务事实”，不要求一步到位长出完整模板系统。
6. 第一批实现必须优先复用现有特征、现有任务事实、现有学习阶段，不另起第二套学习主循环。
7. `证明` 的主裁判在方法账本；`找边界` 的主裁判在失败样本与未确认样本；`补缺口` 的主裁判在当前学习目标和当前候选缺口标记。
8. 第一批实现必须允许“先只反馈学习情况与方法增量，不立即使用已完善方法”，不得强迫每轮学习都直接唤醒原任务。
9. 第一批实现完成后，才进入 `替代路径 / 扩覆盖 / 降成本 / 可验证回执链补全器`。

一句话收口：

`第一批先把“这方法靠不靠谱、什么时候别用、当前缺什么”三件事真正写进账，而不是先做更大的泛化系统。`


## 4. 第一批三组能力

## 4.1 证明

### 目标

回答：

- 这方法当前是否值得继续保留
- 它的可信度有没有提升
- 它是不是已经从“候选”进入“待验证稳定”

### 最小输入

- `方法尝试次数`
- `方法稳定出现次数`
- `方法结果分裂次数`
- `结果确认状态`
- `结果确认来源`
- `学习目标_本轮已达成`
- `学习目标_可用方法已增加`
- `学习目标_方法稳定性已提升`

### 最小处理中间量

- `可靠度_Q10000`
- `稳定复现增量`
- `分裂增量`
- `本轮是否形成可确认成功样本`

### 最小产出

- `方法可靠度增量`
- `稳定复现次数增量`
- `结果分裂次数增量`
- `结果确认状态`
- `结果确认来源`

### 默认写回位置

- `方法虚拟存在`
- `方法可靠度标准特征`
- `学习任务结果字段`

### 当前已有原料

- `方法_可靠度_Q10000`，见 [任务控制总控模块_v0.ixx](D:\鱼巢\任务控制总控模块_v0.ixx#L1721)
- `同步方法可靠度标准特征`，见 [方法类.cpp](D:\鱼巢\方法类.cpp#L954)
- `方法_结果分裂次数`，见 [自我线程模块.ixx](D:\鱼巢\自我线程模块.ixx#L88)
- 外部确认与回执链，见 [自我线程模块.ixx](D:\鱼巢\自我线程模块.ixx#L2574)


## 4.2 找边界

### 目标

回答：

- 什么条件下不要再用这个方法
- 什么条件下需要快速回退
- 当前失败更像“偶然失败”还是“边界命中”

### 最小输入

- `失败样本`
- `取消样本`
- `超时原因`
- `结果分裂样本`
- `未确认外部结果样本`
- `来源错误码`
- `当前学习目标`
- `当前学习对象条件摘要`

### 最小处理中间量

- `失败条件桶`
- `未确认条件桶`
- `分裂条件桶`
- `禁用候选数`
- `回退候选数`

### 最小产出

- `禁用条件数量`
- `回退条件数量`
- `未确认外部结果次数`
- `失败边界摘要`

### 默认写回位置

- `学习任务结果字段`
- `方法虚拟存在`
- `任务管理方法服务匹配层`

### 当前已有原料

- `来源错误码` 和显式学习请求链，见 [任务执行类.ixx](D:\鱼巢\任务执行类.ixx)
- `结果分裂次数`，见 [自我线程模块.ixx](D:\鱼巢\自我线程模块.ixx#L1741)
- `外部结果确认状态 / 来源`，见 [因果信息类.ixx](D:\鱼巢\因果信息类.ixx)

### 第一批落地方式

- 不急着直接写复杂条件树
- 先把边界写成：
  - `数量`
  - `最近错误码`
  - `最近对象主键`
  - `最近目标需求位`
  - `最近失败理由`


## 4.3 补缺口

### 目标

回答：

- 当前方法缺的是条件还是结果
- 这轮学习到底补上了没有
- 补上的缺口应该怎样被任务管理任务消费

### 最小输入

- `当前学习目标类型`
- `当前学习对象`
- `当前候选.缺条件`
- `当前候选.缺结果`
- `观察成功`
- `结果场景`
- `提炼方法首节点`

### 最小处理中间量

- `条件模板补全数量`
- `结果模板补全数量`
- `本轮补缺是否达成`
- `补缺消费强度`

### 最小产出

- `条件模板补全数量`
- `结果模板补全数量`
- `本轮补缺已达成`
- `默认学习情况消费策略`

### 默认写回位置

- `学习任务结果字段`
- `学习任务目标字段`
- `必要时回写到方法节点结构`

### 当前已有原料

- 学习目标分类已能区分 `补齐条件 / 补齐结果`，见 [自我线程模块.ixx](D:\鱼巢\自我线程模块.ixx#L5618)
- 学习目标任务事实同步已接通，见 [自我线程模块.ixx](D:\鱼巢\自我线程模块.ixx#L5717)
- 学习更新阶段已能刷新对象和结果场景，见 [自我线程模块.ixx](D:\鱼巢\自我线程模块.ixx#L10448)

### 第一批落地方式

- 先不直接改完整方法树结构
- 先把“补了多少条件 / 补了多少结果 / 是否命中当前缺口”显式写进学习任务结果面

建议第一批先补以下字段：

- `学习结果_方法可靠度增量`
- `学习结果_稳定复现次数增量`
- `学习结果_结果分裂次数增量`
- `学习结果_禁用条件数量`
- `学习结果_回退条件数量`
- `学习结果_条件模板补全数量`
- `学习结果_结果模板补全数量`
- `学习结果_本轮补缺已达成`
- `学习结果_结果确认状态`
- `学习结果_结果确认来源`

说明：

- 第一批先全部允许落在 `任务事实` 上
- 后续再决定哪些升级成正式模板结构或正式方法特征


## 6. 第一批阶段接入点

建议接入到现有学习阶段：

1. `采样阶段`
- 确认当前是 `证明 / 找边界 / 补缺口` 中哪类目标

2. `更新阶段`
- 收集本轮样本、刷新对象、刷新目标

3. `验证阶段`
- 结算可靠度、分裂、确认状态、补缺是否达成

4. `提交阶段`
- 按学习情况消费策略决定：
  - `继续学习`
  - `挂起观察`
  - `使用已完善方法`
  - `再派生学习任务`


## 7. 当前代码对齐结论

按现状对齐，当前可分成三档：

- `证明`
  - 已有较强原料和部分实现
- `补缺口`
  - 已有目标分类和对象刷新原料
- `找边界`
  - 只有原料和规范，缺专门结算层

因此第一批真正的实现顺序应固定为：

1. `证明`
2. `补缺口`
3. `找边界`


## 8. 后续扩展边界

第一批之后，再进入：

- `替代路径`
- `组合桥接片段`
- `扩覆盖`
- `降成本`
- `可验证回执链补全器`

这些都不应抢在第一批之前实现。

`1218` 的职责，就是把 `证明 / 找边界 / 补缺口` 从“概念上要做”收成“第一批应该怎样落、先写哪些字段、挂在哪个阶段”的可执行实现清单。
