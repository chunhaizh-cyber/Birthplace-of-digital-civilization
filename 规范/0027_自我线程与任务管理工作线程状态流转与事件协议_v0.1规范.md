- `任务根节点` 与 `任务运行态` 如何彻底分开

## 3. 核心结论

必须固定以下结论：

1. `自我线程` 管的是控制面，不管局部工序面。
2. `任务管理工作线程` 只负责推进、等待、上报、返回和更新局部状态，不负责根层判断。
3. `任务根节点` 必须只读；`任务运行态` 必须可变；两者不得混写。
4. `派生需求` 可以由任务管理工作线程上报，但不能直接生成子任务实例。
5. `子结果` 必须先回 `自我线程`，再由 `自我线程` 路由给父层任务。
6. `父层任务` 只判断子结果对自身步骤和局部流程的影响；`自我线程` 只判断其对全局优先级、运行队列和最终去向的影响。
7. `优先级` 统一由 `自我线程` 维护；任务管理工作线程不得私自改全局优先级。
8. `父任务` 一旦因依赖阻塞，必须退出 `可运行队列`。
9. 真正获得加权的不是抽象“子任务”，而是 `能解开当前阻塞链的依赖任务`。
10. 协议层里的 `判断` 一词只保留给 `自我线程` 和 `父层任务`；任务管理工作线程不使用“判断”一词。
11. 自我线程每轮必须先冻结本轮开始时已可见的消息批次，并处理完这批消息及其同步内部闭包，再做一次根层重判与主派发/收束决议。
12. 本轮处理中异步新产生的消息必须进入下一轮；高优先级控制消息可按抢占规则打断当前轮次。
13. `控制消息` 与 `全局环境消息` 的首收件人必须是 `自我线程`。
14. `明确绑定到具体任务且仅影响局部流程的外部反馈`，可先进入对应的 `任务管理工作线程`；若升级为 `派生需求 / 阻塞变化 / 异常 / 优先级变化 / 新任务实例化请求`，必须再上报 `自我线程`。
15. `结果回流消息` 必须回 `自我线程`；`推进消息` 只上报会改变控制面判断的部分，不得把纯局部噪声全部推给线程。
16. `控制态真值` 由 `自我线程` 持有；任务管理工作线程只接收与自身相关的 `控制投影`，不得反向定义全局控制态。

一句话收口：

`自我线程负责拍板、排队、派发、路由和结算；任务管理工作线程负责在只读任务根节点约束下推进步骤、上报阻塞、返回结果。`


## 3.1 轮次消息语义

协议层必须固定以下轮次消息语义：

1. 每轮开始时，自我线程先冻结本轮开始时已可见的消息批次。
2. 本轮必须处理完该冻结批次，以及这些消息同步引发的内部消息闭包。
3. 本轮处理中异步新产生的消息，不得为了“清空活队列”而继续追入本轮。
4. 高优先级控制消息，如 `打断 / 收束 / 停止 / 强制优先级刷新 / 强制预算刷新`，可按抢占规则中断本轮。
5. 每轮在消息闭包完成后，只做一次根层重判，并最多做一次主派发或收束决议。

工程推荐口径：

- `双缓冲邮箱`
- `本轮内部同步工作队列`

一句话收口：

`一轮清一箱，不追下一箱。`


## 3.2 消息来源、首收件人与最终知晓人

协议层必须先区分三个概念：

- `谁产生消息`
- `谁先接收消息`
- `谁最终需要知道`

这三者不必是同一个对象。

必须固定以下分流规则：

1. `控制消息` 先进入 `自我线程`，再由自我线程向相关任务管理工作线程下发控制事件。
2. `全局环境消息` 先进入 `自我线程`，不得先塞给某个局部工作线程私下消化。
3. `明确绑定到某任务且仅影响局部流程的外部反馈`，可先进入对应的 `任务管理工作线程`。
4. 上述局部外部反馈一旦升级成 `派生需求 / 阻塞变化 / 异常 / 需要改优先级 / 需要实例化新任务`，必须再上报 `自我线程`。
5. `结果回流消息` 必须先回 `自我线程`，再由自我线程决定全局路由与最终结算。
6. `推进消息` 只把会改变控制面判断的部分上报 `自我线程`；纯局部中间态不得把线程淹没成日志收集器。
7. `控制真值` 由 `自我线程` 持有，工作线程只接收自身可见的 `控制投影`；不得共享一个可被双方随意改写的全局控制对象。

### 3.3 消息分流总表

| 消息类型 | 典型来源 | 首收件人 | 是否必须上报自我线程 | 是否影响优先级 / 控制面 |
| --- | --- | --- | --- | --- |
| `控制面消息` | 外部控制源、宿主边界控制、人工控制、线程边界模块 | `自我线程` | `是` | `是` |
| `全局环境消息` | 环境整体变化、全局风险变化、全局资源变化、运行模式变化 | `自我线程` | `是` | `是` |
| `任务绑定外部反馈` | 指定任务的外部确认、局部条件满足、局部资源到位 | `相关任务管理工作线程` | `条件成立时是` | `条件成立时是` |
| `局部推进消息` | 步骤内部小进度、临时变量变化、局部小重试、中间状态 | `相关任务管理工作线程` | `否` | `否` |
| `关键推进事件` | 阻塞、解除阻塞、派生需求、局部失败、异常 | `相关任务管理工作线程` | `是` | `是` |
| `结果回流消息` | 子结果、局部结果、最终结果、异常结果 | `自我线程` 或先到工作线程后立即回线程 | `是` | `是` |

### 3.4 控制消息与环境消息分流

必须固定：

1. `打断 / 收束 / 停止 / 预算刷新 / 优先级刷新 / 运行模式切换` 都属于控制消息。
2. 控制消息改的是控制面，不是工序面，因此必须先进入 `自我线程`。
3. `没有明确绑定到某个任务`、`可能形成新的根需求`、`会影响多个任务`、`会改变全局优先级`、`会触发安全接管或收束`、`会改变资源总量或运行模式` 的消息，都属于 `全局环境消息`。
4. 全局环境消息若先发给某个局部工作线程，会造成自我线程失明，因此禁止这样做。

最小链条固定为：

`控制源 / 全局环境 -> 自我线程 -> 相关工作线程`

### 3.5 任务绑定外部反馈分流

以下条件同时满足时，外部反馈可先进入对应任务管理工作线程：

- 已明确绑定到某个任务实例
- 只影响该任务当前步骤或局部流程
- 不直接改变根层优先级
- 不直接形成新的全局需求

例如：

- 某一步等待的外部确认回来了
- 某个局部条件已满足
- 某个依赖资源已到位
- 某个指定任务收到局部执行反馈

最小链条固定为：

`局部外部反馈 -> 相关工作线程 -> 自我线程`

但必须补一条升级规则：

局部外部反馈若导致以下任一后果，必须再上报 `自我线程`：

- `派生需求`
- `阻塞变化`
- `异常`
- `需要改优先级`
- `需要实例化新任务`

### 3.6 推进消息与结果回流分流

必须固定：

1. 不是所有推进细节都直接发给 `自我线程`。
2. 只有会改变 `调度 / 结算 / 路由 / 优先级 / 是否实例化任务` 的推进消息，才需要上报 `自我线程`。

应直接上报 `自我线程` 的推进消息至少包括：

- `当前步骤阻塞`
- `当前步骤解除阻塞`
- `派生需求上报`
- `局部失败`
- `异常`
- `当前任务完成`
- `最终结果返回`
- `需要父层任务判断的子结果`

不必直接上报 `自我线程` 的细粒度推进消息至少包括：

- `步骤内部的小进度`
- `局部临时变量变化`
- `每一次内部小重试`
- `纯工作线程内部的中间状态`

必须进一步固定：

- `结果回流消息` 必须回 `自我线程`
- `推进消息` 只把影响控制面判断的部分回 `自我线程`

一句话收口：

`控制和全局环境先到自我线程，局部外部反馈先到相关工作线程，关键推进与结果再回自我线程。`

更细的七类消息定义、首收件人与处理方法，按 [0028_自我线程消息总表规范.md](D:\鱼巢\规范\0028_自我线程消息总表规范.md) 执行。


## 4. 只读任务根节点与可变任务运行态

### 4.1 任务根节点（只读）

`任务根节点` 必须只读，至少固定以下字段面：

- `来源需求`
- `方向`
- `父任务引用`
- `创建者`
- `根约束`
- `初始授权边界`

必须固定：

1. `任务根节点` 只能由 `自我线程` 创建。
2. `任务根节点` 创建后，局部线程不得改写其方向、来源需求和根约束。
3. `任务管理工作线程` 只能引用 `任务根节点`，不能把它当作运行时草稿结构。

### 4.2 任务运行态（可变）

`任务运行态` 必须与 `任务根节点` 分离，至少固定以下可变字段面：

- `当前可执行步骤前沿`
- `当前等待原因`
- `子结果缓存`
- `局部完成度`
- `当前流程状态`
- `局部错误 / 异常`
- `运行中优先级快照`

必须固定：

1. `任务运行态` 由 `任务管理工作线程` 推进和刷新。
2. `任务运行态` 可以更新局部状态，但不能回写 `任务根节点` 的只读字段。
3. `父任务` 的步骤推进依据 `任务运行态` 变化，而不是改写 `任务根节点` 本身。


## 5. 事件协议

### 5.1 自我线程 -> 任务管理工作线程

| 事件名 | 发送方 | 接收方 | 含义 | 最小载荷 | 禁止越权 |
| --- | --- | --- | --- | --- | --- |
| `启动任务实例` | 自我线程 | 任务管理工作线程 | 正式授予某任务入口的运行资格 | `任务入口主键 / 任务根节点主键 / 上下文摘要 / 优先级快照 / 预算摘要` | 不得隐含允许改写任务根节点 |
| `优先级刷新` | 自我线程 | 任务管理工作线程 | 通知该任务当前运行中优先级快照已更新 | `任务入口主键 / 新优先级快照 / 刷新原因` | 不得让工作线程借机改全局队列 |
| `预算刷新` | 自我线程 | 任务管理工作线程 | 通知本轮资源/时间预算变化 | `任务入口主键 / 资源预算 / 时间预算 / 并发约束` | 不得隐式扩权 |
| `打断` | 自我线程 | 任务管理工作线程 | 要求立刻停止当前推进并返回中断结果 | `任务入口主键 / 打断原因 / 生效时点` | 不得被工作线程忽略为“建议” |
| `收束` | 自我线程 | 任务管理工作线程 | 要求停止扩张、清理局部运行态并返回收束结果 | `任务入口主键 / 收束原因 / 当前边界状态` | 不得继续派生新需求 |
| `停止` | 自我线程 | 任务管理工作线程 | 要求终止该任务实例 | `任务入口主键 / 停止原因` | 不得继续保留活跃步骤 |
| `子结果送达` | 自我线程 | 父层任务管理工作线程 | 把已路由的子结果送达父层任务实例 | `父任务主键 / 子结果摘要 / 路由原因 / 是否解阻` | 不得把未结算裸结果直接塞入父层步骤逻辑 |

### 5.2 任务管理工作线程 -> 自我线程

| 事件名 | 发送方 | 接收方 | 含义 | 最小载荷 | 禁止越权 |
| --- | --- | --- | --- | --- | --- |
| `当前步骤可执行` | 任务管理工作线程 | 自我线程 | 当前步骤前沿已满足执行条件 | `任务入口主键 / 当前步骤主键 / 局部状态摘要` | 不得把“可执行”扩写成“已完成” |
| `当前步骤阻塞` | 任务管理工作线程 | 自我线程 | 当前步骤因依赖/条件缺失而阻塞 | `任务入口主键 / 阻塞原因 / 缺失对象 / 是否需要外部解阻` | 不得直接把阻塞对象长成子任务 |
| `派生需求上报` | 任务管理工作线程 | 自我线程 | 上报在推进过程中发现的派生需求 | `任务入口主键 / 派生需求摘要 / 来源步骤 / 建议理由` | 不得直接实例化依赖任务 |
| `局部结果返回` | 任务管理工作线程 | 自我线程 | 返回一步推进后的局部结果 | `任务入口主键 / 当前步骤结果 / 局部完成度 / 异常摘要` | 不得宣称全局成功 |
| `最终结果返回` | 任务管理工作线程 | 自我线程 | 返回该任务实例当前轮次的最终结果 | `任务入口主键 / 最终结果摘要 / 是否完成 / 是否失败 / 后续建议` | 不得跳过线程直接回父层任务 |
| `异常/故障上报` | 任务管理工作线程 | 自我线程 | 上报异常、故障、超时、中断等失稳事实 | `任务入口主键 / 异常类型 / 错误码 / 故障摘要 / 当前局部状态` | 不得自行收口为最终去向 |

### 5.3 事件协议硬规则

必须固定：

1. 自我线程只发 `控制事件`，不发局部步骤细节。
2. 任务管理工作线程只回 `局部状态事件`、`阻塞事件`、`结果事件` 和 `异常事件`。
3. 任一 `派生需求上报` 都只能产生 `候选依赖任务`，不能直接产生 `已实例化子任务`。
4. 任一 `最终结果返回` 都必须先进入 `自我线程` 的路由与结算链。
5. `派生需求上报 / 局部结果返回 / 最终结果返回` 若属于本轮冻结消息同步引发的内部闭包，应在本轮内继续处理。
6. 上述事件若由异步新执行产生，则进入下一轮。
7. `局部步骤小进度 / 临时变量变化 / 内部小重试 / 纯中间态` 不属于必须上报线程的协议事件。
8. 工作线程不得把所有推进噪声都上报给线程，导致线程退化为日志收集器。
9. 自我线程持有 `控制态真值`；工作线程只接收相关 `控制事件副本` 或 `控制投影`，不得反向改写全局控制态。


## 6. 双状态机

### 6.1 自我线程侧状态

自我线程在单任务交互面上，至少固定以下状态：

- `可运行`
- `等待`
- `已派发`
- `等待子结果`
- `收束中`
- `已停止`
- `已结算`

### 6.2 任务管理工作线程侧状态

任务管理工作线程在单任务交互面上，至少固定以下状态：

- `已受理`
- `展开步骤中`
- `等待前序步骤`
- `等待条件`
- `等待子结果`
- `当前步骤可执行`
- `局部完成`
- `局部失败`
- `已中断`
- `已收束`
- `已停止`

### 6.3 状态机最小转移口径

#### 自我线程侧

| 当前状态 | 触发事件 | 下一状态 |

### 6.4 术语限制

必须固定：

1. `判断` 只保留给 `自我线程` 和 `父层任务`。
2. `任务管理工作线程` 在协议里只使用：
   - `推进`
   - `等待`
   - `上报`
   - `返回`
   - `更新局部状态`
3. 不得在任务管理工作线程协议中写“判断当前全局优先级”“判断是否应生成子任务”“判断最终去向”。


## 7. 标准流转链

### 7.1 父任务阻塞 -> 依赖任务先跑 -> 结果回流

标准流转链固定为：

`父任务当前步骤不可执行`
-> `任务管理工作线程上报阻塞原因 / 派生需求`
-> `自我线程判断是否实例化依赖任务`
-> `自我线程创建依赖任务入口并调度其工作线程`
-> `依赖任务完成后结果先回自我线程`
-> `自我线程更新全局优先级并路由给父层任务`
-> `父层任务判断该结果对自身步骤的影响`
-> `若当前步骤已解除阻塞，自我线程再把父任务放回可运行队列`

### 7.2 协议图

```mermaid
flowchart LR
    A["父任务步骤阻塞"] --> B["工作线程上报阻塞/派生需求"]
    B --> C["自我线程判断是否实例化依赖任务"]
    C --> D["自我线程创建依赖任务入口"]
    D --> E["调度依赖任务工作线程"]
    E --> F["依赖任务返回结果到自我线程"]
    F --> G["自我线程更新全局优先级并路由结果"]
    G --> H["父层任务判断对子步骤的影响"]
    H --> I["自我线程决定父任务是否重回可运行队列"]
```

### 7.3 标准流转链硬规则

必须固定：

1. 父任务阻塞时，先上报再裁决，不得局部自生子任务。
2. 依赖任务结果必须先回自我线程，不得直接塞回父层任务。
3. 父层任务只能判断“对我当前步骤有没有影响”，不能反向改全局队列。
4. 自我线程只能判断“是否重新入队、如何排序、是否收束”，不能替父层任务做局部步骤完成判断。


## 8. 优先级规则

### 8.1 统一维护权

必须固定：

1. `优先级` 统一由 `自我线程` 维护。
2. `任务管理工作线程` 只能回传局部状态和阻塞信息，不能私自改全局优先级。

### 8.2 阻塞退出规则

必须固定：

1. 父任务一旦因依赖阻塞，必须退出 `可运行队列`。
2. 退出 `可运行队列` 不等于任务失败，只表示当前不具备继续推进资格。

### 8.3 解阻加权规则

必须固定：

1. 获得加权的不是抽象“子任务永远高于父任务”。
2. 获得加权的是 `能解开当前阻塞链的依赖任务`。
3. `解阻加权` 必须与当前根需求方向、双值趋势和全局队列状态共同裁决，不能机械按父子辈分排序。


## 9. 当前阶段最小验收口径

当前若要承认 `0027 v0.1` 已成立，至少必须满足：

1. `任务根节点` 与 `任务运行态` 已被明确分离，且只读/可变边界已经写死。
2. 自我线程到任务管理工作线程、任务管理工作线程回自我线程的事件表已经完整。
3. 自我线程侧和任务管理工作线程侧两套状态机已经分开定义。
4. `父任务阻塞 -> 依赖任务先跑 -> 结果回流 -> 父任务解阻` 这条标准流转链已经固定。
5. `优先级统一由自我线程维护`、`父任务阻塞退出可运行队列`、`解阻任务获得加权` 三条规则已经固定。
6. 规范中未重新扩写自我线程职责，只把既有边界翻译成状态和事件合同。

最终收口：

`0027 的意义，不是继续发明功能，而是把“谁发事件、谁回事件、谁处于什么状态、谁在什么时刻有权拍板”正式写成交通规则。`
