- 哪些属于 `局部条件拒绝`

## 3. 核心结论

必须固定以下结论：

1. `拒绝原因类型` 是执行前门控的正式输出字段，不允许只返回一个模糊的“不能执行”。
2. `拒绝原因类型` 必须先分三层：
   - `根层拒绝`
   - `上下文拒绝`
   - `局部条件拒绝`
3. 同一轮执行前门控若同时命中多类原因，必须遵守优先级：
   - `根层拒绝 > 上下文拒绝 > 局部条件拒绝`
4. `拒绝原因类型` 只能建立在前序比较、门控和条件判定结果之上，不允许直接拿一次特征或主键字段当最终拒绝原因。
5. `建议回流路径` 必须建立在 `拒绝原因类型` 之上，不允许脱离原因单独拍脑袋指定。
6. `拒绝原因类型` 只是门控拒绝解释，不等于最终裁决本体。

一句话收口：

`执行前门控若拒绝执行，必须至少说清楚“是根层拦下、上下文拦下，还是局部条件拦下”，并给出对应建议回流路径。`


## 4. 定义

`拒绝原因类型` 指的是：

- 执行前门控在判定“当前不允许进入普通执行链”后，为解释拒绝来源和回流路径所给出的结构化原因类型

它回答的是：

- 为什么当前不能执行
- 这份拒绝主要来自哪一层
- 当前更适合回到哪条治理路径

它不是：

- 根层重判本体
- 任务管理上下文本体
- 条件判定本体
- 最终裁决全部内容


## 5. 三层原因总表

### 5.1 根层拒绝原因类型

| 原因类型值 | 含义 | 默认建议回流路径 | 说明 |
| --- | --- | --- | --- |
| `命中致命性安全边界` | 当前已触及或将触及 `安全值 -> 0` 的致命边界 | `进入收束/待机/停止边界` | 这是最强拒绝类型。 |
| `命中违规违法禁止` | 当前执行将触发违规、违法或根层禁止项 | `进入收束/待机/停止边界` | 不允许转成普通执行尝试。 |
| `当前仅允许边界处理` | 根层已判定本轮只允许收束、待机或停止链 | `进入收束/待机/停止边界` | 与局部失败不同。 |
| `根层未放行执行` | 根层虽未命中强禁止，但当前仍未允许普通执行 | `回到重筹办` 或 `转入学习` | 具体由外层再解释更细路径。 |

### 5.2 上下文拒绝原因类型

| 原因类型值 | 含义 | 默认建议回流路径 | 说明 |
| --- | --- | --- | --- |
| `当前管理对象缺失` | 本轮没有明确管理对象 | `回到重筹办` | 先补治理锚点。 |
| `当前步骤缺失` | 当前任务没有可承接的步骤位置 | `回到重筹办` | 需要先补结构。 |
| `当前方法挂点缺失` | 当前步骤或任务没有已挂接的方法首节点 | `回到重筹办` | 应先补治理结构或显性化方法位，不应直接把“挂点缺失”当成可执行学习。 |
| `当前去向要求非执行路径` | 当前下一步去向已明确不是执行链 | `按去向返回` | 例如重筹办、学习、等待、收束。 |
| `治理状态未到可执行态` | 当前仍处于等待学习、等待反馈、等待恢复等状态 | `挂起等待` | 不是“不可执行”，而是“现在还不能执行”。 |

### 5.3 局部条件拒绝原因类型

| 原因类型值 | 含义 | 默认建议回流路径 | 说明 |
| --- | --- | --- | --- |
| `当前方法不存在` | 当前上下文下还没有合法方法挂点 | `挂起等待` 或 `回到重筹办` | 默认先挂起待学习方法位并等待再次观察；若连方法位都未显性化，则先回重筹办。 |
| `当前方法不可用` | 方法存在，但当前不可用 | `挂起等待` | 默认先挂起对应待学习方法，等待条件变化、再次观察或再次命中。 |
| `当前条件未满足` | 条件缺口已显性化，但当前不能直接执行 | `回到重筹办` 或 `转入学习` | 仅当已存在动作骨架时，才允许进一步进入学习补条件。 |
| `当前无权限` | 当前上下文下没有权限使用该方法或条件 | `转入学习` 或 `进入边界处理` | 取决于权限是否可补。 |
| `当前不可获取` | 关键条件无法获取 | `挂起等待` 或 `转入学习` | 当前执行应拦下。 |
| `当前值未知需观察` | 关键值未知，需要先观察 | `挂起等待` 或 `回到重筹办` | 先补观察链。 |


## 6. 原因优先级与归并规则

必须固定：

1. 若命中任一 `根层拒绝原因类型`，则最终 `拒绝来源层级 = 根层`。
2. 若未命中根层，但命中任一 `上下文拒绝原因类型`，则最终 `拒绝来源层级 = 上下文层`。
3. 只有在前两层都未命中时，局部条件拒绝才允许成为最终拒绝来源。
4. 同层内若命中多个原因，当前阶段应先提炼一个 `主拒绝原因类型`，详细清单可作为证据挂带。

当前主拒绝原因提炼优先级建议固定为：

```text
命中致命性安全边界
> 命中违规违法禁止
> 当前仅允许边界处理
> 当前管理对象缺失
> 当前步骤缺失
> 当前方法挂点缺失
> 当前去向要求非执行路径
> 治理状态未到可执行态
> 当前无权限
> 当前不可获取
> 当前值未知需观察
> 当前条件未满足
> 当前方法不可用
> 当前方法不存在
```


## 7. 建议回流路径规范表

| 建议回流路径值 | 适用场景 | 说明 |
| --- | --- | --- |
| `回到重筹办` | 结构缺失、步骤缺失、路径需要重组 | 适合重新组织任务树和方法挂点。 |
| `转入学习` | 已存在待学习方法骨架，且条件侧已存在或可由学习任务补给 | 适合进入补条件面、补可靠性链与受控方法学习，不适合把“纯无方法”直接转成学习执行。 |
| `挂起等待` | 等待外部反馈、等待观察、等待条件可获取 | 适合保留当前治理状态，等待后续事件。 |
| `进入收束/待机/停止边界` | 命中根层禁止、仅允许边界处理 | 适合直接退出普通执行链。 |
| `按上下文既定去向返回` | 当前下一步去向已明确指向非执行路径 | 保持与任务管理上下文一致。 |

必须固定：

1. `建议回流路径` 是门控输出建议，不等于外层最终已批准。
2. 但它必须和 `拒绝原因类型` 保持可解释映射，不能乱跳。
3. `当前方法不存在 / 当前方法不可用` 默认优先 `挂起等待`；只有当外层已经组织出 `待学习方法` 的动作骨架，且条件侧已存在或可由学习任务补给时，才允许进一步裁成 `转入学习`。


## 8. 严格禁止的混层

以下做法必须禁止：

1. 只返回 `不能执行`，不返回拒绝原因类型。
2. 同时命中根层禁止，却把拒绝原因写成局部条件不足。
3. 当前上下文已要求非执行路径，却把拒绝原因写成方法不可用。
4. 不区分 `方法不存在` 与 `方法不可用`。
5. 不区分 `当前条件未满足` 与 `当前值未知需观察`。
6. 让 `建议回流路径` 与 `拒绝原因类型` 彼此脱钩。
