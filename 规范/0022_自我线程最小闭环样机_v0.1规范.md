## 3. 样机定位与硬边界

### 3.1 样机定位

当前样机必须固定为：

`单线程 + 单场景 + 一轮主循环 + 一个上下文包 + 一个活跃任务包 + 一个结算包 + 一条回写链`

它当前验证的是：

- 能不能看见差额
- 能不能把差额翻成需求
- 能不能把需求压成一步治理
- 能不能匹配出建议型方法
- 能不能生成 `解释 / 预警 / 建议`
- 能不能把结果结算并回写成动态和因果候选

### 3.2 当前只开的动作边界

第一版只允许以下输出类型：

- `解释`
- `预警`
- `建议`

第一版真正允许落地的动作只先开：

- `建议`

因此必须固定：

- `解释` 和 `预警` 只产生诊断性输出，不触发现实动作
- `建议` 只形成可供人工确认的建议结果，不直接改写外部世界
- `执行 / 回滚 / 自动联动改写` 本版全部关闭

### 3.3 安全边界

第一版安全侧只保留一个硬阈值打断：

- 风险安全差额超阈值时，直接终止服务推进，转入止损式结算

必须固定：

- 安全守卫能打断，但不接管长期方向
- 被打断路径同样必须结算、回写动态、落因果候选

- 第一版先验证 `自我线程内核闭环`

### 6.3 最小状态面

## 7. 一轮最小主链

当前最小主链固定为：

`唤醒 -> 状态刷新 -> 差额评估 -> 根需求生成 -> 任务管理输出一步治理 -> 方法选择 -> 生成解释/预警/建议 -> 回流结算 -> 写回动态节点和因果候选`

把它压到更细顺序，即：

1. 生成 `线程轮次上下文包`
2. 封口上下文包
3. 生成 `一步治理任务包`
4. 方法选择并封口任务包
5. 生成 `回流结算包`
6. 封口结算包
7. 写回 `动态节点`
8. 写回 `因果候选`

必须固定：

- 一轮里不允许派生第二个活跃任务包
- 一轮结束后不自动续跑下一轮
- 本版样机只证明“一轮闭环成立”


## 8. 最小规则表

### 8.1 当前规则只保留四类

`自我线程最小规则表.ixx` 当前只保留：

1. `无差额待机规则`
   - 服务差额和安全差额都不成立时，直接结束本轮

2. `服务主导规则`
   - 风险硬阈值未命中时，允许服务差额驱动建议生成

3. `风险硬阈值打断规则`
   - 风险安全差额一旦超阈值，立即禁止服务推进

4. `授权拒绝规则`
   - 人工确认不可用或建议输出被禁时，不允许输出正式建议，但仍要结算

### 8.2 当前不做

本版不做：

- 复杂权重竞争
- 多目标联立仲裁
- 连续多轮预算回归
- 学习驱动阈值自调


## 9. 最小方法表

### 9.1 当前只放三个建议型方法

| 方法名 | 命中条件 | 输出类型 | 作用 |
| --- | --- | --- | --- |
| `方法_状态解释建议` | 服务差额存在且场景就绪度不足 | `解释 + 建议` | 解释当前缺口，并给出保守建议 |
| `方法_风险预警建议` | 风险安全差额接近硬阈值但未越界 | `预警 + 建议` | 给出风险预警和保守建议 |
| `方法_保守建议输出` | 其它建议型方法未命中但允许输出 | `建议` | 兜底给出最小保守建议 |

### 9.2 当前禁止

方法表当前禁止：

- 真实控制外部设备
- 直接改写场景状态
- 直接写已验证因果
- 直接宣布需求满足

### 9.3 方法未命中的处理

必须固定：

- 方法未命中不等于本轮无事发生
- 方法未命中必须进入结算
- 方法未命中必须落 `动态节点`
- 方法未命中必须落 `因果候选`

对于方法未命中路径：

- `结算包.选定方法主键 = 0`
- `执行成功 = false`
- `错误码 = 方法未命中预留错误码`
- `来源输出场景 = 来源输入场景`


## 10. 最小回写纪律

### 10.1 动态节点最小写回面

无论成功、失败、授权拒绝还是方法未命中，结算后都必须落 `1` 个动态节点，且至少带以下字段：

- `来源动作`
- `来源输入场景`
- `来源输出场景`
- `执行成功`
- `错误码`
- `动态路径签名`

### 10.2 因果候选最小写回面

同一轮中还必须落 `1` 个因果候选，且至少带以下字段：

- `来源任务主键`
- `来源方法主键`
- `置信度`
- `已验证 = false`

### 10.3 失败路径的写回口径

对于授权拒绝、方法未命中和安全打断路径，必须固定：

- 仍然生成 `结算包`
- 仍然生成 `动态节点`
- 仍然生成 `因果候选`
- `来源方法主键` 若不存在，则写 `0`
- `样本类别` 默认写 `负样本` 或 `待验证`


## 11. 失败路径同样要闭环

当前样机至少要覆盖四条路径：

1. `成功建议路径`
   - 服务差额存在
   - 方法命中
   - 输出建议
   - 完成结算和回写

2. `授权拒绝路径`
   - 差额存在
   - 一步治理已生成
   - 因人工确认不可用或输出被禁而拒绝
   - 仍然结算并回写

3. `方法未命中路径`
   - 差额存在
   - 授权通过
   - 方法未命中
   - 仍然结算并回写

4. `安全硬阈值打断路径`
   - 风险安全差额超阈值
   - 服务推进被中断
   - 仍然结算并回写

必须固定：

`失败路径不是日志分支，而是正式账路径。`

## 13. 完成标准

当前样机完成标准至少必须满足以下六条：

1. `一轮只能生成 1 个上下文包。`
2. `一个上下文包最多派生 1 个活跃任务包。`
3. `无论成功、失败、授权拒绝还是方法未命中，都必须生成结算包。`
4. `结算后必须落 1 个动态节点，且至少带上 来源动作 / 来源输入场景 / 来源输出场景 / 执行成功 / 错误码 / 路径签名。`
5. `结算后必须同时落 1 个因果候选，且至少带上 来源任务主键 / 来源方法主键 / 置信度，并且默认 已验证 = false。`
6. `场景侧只挂 状态 / 动态 / 关系 / 因果索引，不往场景里塞大对象副本。`

在此基础上，再补两条附加完成标准：

7. `第一版只允许 解释 / 预警 / 建议 三类输出。`
8. `真正落地动作先只开 建议，安全侧先只开一个硬阈值打断。`
