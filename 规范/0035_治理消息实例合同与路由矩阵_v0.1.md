## 3. 核心结论

必须固定以下结论：

1. 一条治理消息必须至少包含：`消息头 + 动作语义 + 变化项集`。
2. `消息头` 里必须同时有：
   - `一级来源链`
   - `二级处理类型`
3. `变化项集` 不再只有前后值，还必须带：
   - `是否关键中间状态`
   - `是否已恢复`
   - `是否确认生效`
4. 自我线程正式输入消息的首收件人，恒为 `自我线程`。
5. `学习 / 审计 / 复盘` 不进入本文的正式实例合同；它们若升级为正式事实，应先重新包装成 7 类之一。
6. 线程状态变化消息的发件线程只负责上报事实；线程状态特征、状态节点和动作动态的正式落账，只能由 `自我线程` 在消费消息后执行。


## 4. 表 1：治理消息字段表

### 4.1 消息公共字段

| 字段段落 | 字段 | 含义 | 备注 |
| --- | --- | --- | --- |
| 识别段 | `消息主键` | 一条消息的唯一标识 | 用于追踪、幂等、审计 |
| 路由段 | `一级来源链` | `线程直达消息 / 任务管理工作线程回流消息` | 是来源链，不是业务语义本体 |
| 路由段 | `二级处理类型` | 7 类正式处理类型之一 | 是线程内部正式处理入口 |
| 行为段 | `发出者主键` | 谁把消息送进线程 | 可与动作主体不同 |
| 行为段 | `动作主体主键` | 谁做了这件动作 | 是动态节点首要落点 |
| 行为段 | `动作语义键` | 这次动作是什么 | 例如 `下发收束指令 / 提交结果 / 确认条件满足` |
| 行为段 | `作用对象主键` | 这次动作主要作用到谁 | 可为空 |
| 路由补充段 | `父任务主键` | 若需要回父任务，父任务是谁 | 便于结果回路由 |
| 场景段 | `关联场景主键` | 这次动作关联到哪个场景 | 便于动态节点回指 |
| 时间段 | `产生时间` | 消息进入线程世界的时间 | 不是唯一生效时间 |
| 追踪段 | `路径签名` | 过程路径签名 | 用于调试与回放 |
| 来源段 | `来源成功标志` | 这条动作或观测链是否成功 | 与动态节点对齐 |
| 来源段 | `错误码` | 来源错误信息 | 与动态节点对齐 |
| 来源段 | `置信度` | 这条消息整体可信度 | 默认允许 `0-10000` |
| 变化段 | `变化项集` | 这次动作导致的全部一次特征变化 | 不允许偷简成单值字段 |

### 4.2 必须固定

1. `一级来源链` 回答“它从哪条来源链进线程”。
2. `二级处理类型` 回答“它在线程内部走哪条处理分支”。
3. `动作语义键 + 变化项集` 才是正式业务语义承载面。


## 5. 表 2：特征变化项表

| 字段 | 含义 | 备注 |
| --- | --- | --- |
| `变化主体主键` | 谁的特征变了 | 不默认等于动作主体 |
| `特征主键` | 改的是哪个一次特征 | 应能映射到一次特征字典 |
| `变化前值` | 变更前值 | 允许为空 |
| `变化后值` | 变更后值 | 不可省略 |
| `是否确认生效` | 是否已成为正式真值 | `false` 时只能作候选事实 |
| `是否关键中间状态` | 是否属于关键中间状态 | 结果回流时尤其重要 |
| `是否已恢复` | 后续是否已被拉回 | 反向中间态抽否定项时使用 |
| `变化原因键` | 为什么发生这次变化 | 便于状态节点原因字段和审计 |

必须固定：

1. 一条消息可带 `1..N` 个变化项。
2. 一个动作若同时改多个特征，必须写成多个变化项。
3. `控制域变化`、`业务域变化`、`关键中间状态变化` 都必须通过变化项表达，不得偷塞进消息名里。
4. 线程状态变化消息中的变化项，只描述“已上报的状态变化事实”，不授予发件线程直接改写自我内部世界的权限。

## 8. 结果与关键中间状态回流消息的额外合同

这类消息必须至少拆成三段：

1. `最终结果段`
2. `关键中间状态段`
3. `提示段`

| 段落 | 最小字段 | 作用 |
| --- | --- | --- |
| `最终结果段` | `任务主键 / 父任务主键 / 最终结果状态 / 最终结果值 / 是否成功 / 完成时间` | 说明“最后到了哪” |
| `关键中间状态段` | `关键中间状态变化项列表` | 说明“路上出现过哪些关键反向态、阻塞态、副作用态” |
| `提示段` | `可能存在反向推进 / 可能形成阻塞 / 可能影响父任务 / 可能形成否定项候选` | 只做提示，不替代线程最终判断 |

必须固定：

1. 这类消息不得只提交最终结果。
2. 工作线程只交事实和提示，不直接裁定“否定项成立”。
3. 自我线程收到后，要先判断全局影响，再路由父任务，再统一结算。


## 9. 推荐最小接口

```cpp
export enum class 枚举_消息来源链 : std::uint8_t {
    线程直达消息 = 0,
    任务管理工作线程回流消息 = 1
};

export enum class 枚举_消息处理类型 : std::uint8_t {
    控制域事实消息 = 0,
    业务域事实消息 = 1,
    延迟事实消息 = 2,
    工作线程状态消息 = 3,
    任务推进关键事件消息 = 4,
    派生需求上报消息 = 5,
    结果与关键中间状态回流消息 = 6
};

export struct 结构_特征变化项 {
    主键 变化主体主键 = 0;
    主键 特征主键 = 0;
    特征值 变化前值{};
    特征值 变化后值{};
    bool 是否确认生效 = false;
    bool 是否关键中间状态 = false;
    bool 是否已恢复 = false;
    std::string 变化原因键{};
};

export struct 结构_治理消息头 {
    主键 消息主键 = 0;
    枚举_消息来源链 一级分类{};
    枚举_消息处理类型 二级类型{};
    主键 发出者主键 = 0;
    主键 动作主体主键 = 0;
    主键 作用对象主键 = 0;
    主键 父任务主键 = 0;
    主键 关联场景主键 = 0;
    时间戳 产生时间 = 0;
    std::uint64_t 路径签名 = 0;
    bool 来源成功标志 = false;
    I64 错误码 = 0;
    I64 置信度 = 0;
};
```
