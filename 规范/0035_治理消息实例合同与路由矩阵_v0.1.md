## 3. 核心结论

必须固定以下结论：

1. 一条治理消息必须至少包含：`消息头 + 动作语义 + 变化项集`。
2. `消息头` 里必须同时有：
   - `一级来源链`
   - `二级处理类型`
3. `变化项集` 不再只有前后值，还必须带：
   - `是否关键中间状态`
   - `是否已恢复`
   - `是否确认生效`
4. 自我线程正式输入消息的首收件人，恒为 `自我线程`。
5. `学习 / 审计 / 复盘` 不进入本文的正式实例合同；它们若升级为正式事实，应先重新包装成 7 类之一。
6. 线程状态变化消息的发件线程只负责上报事实；`线程状态特征` 与 `治理侧状态节点` 的正式落账，只能由 `自我线程` 在消费消息后执行。现实世界的动作动态本体，则由 `任务管理任务 / 任务 / 本能动作` 三类执行体直接管理；`自我线程` 只沉淀其上推事实后的 `治理侧动作摘要 / 审计材料 / 因果材料`。
7. 从 `任务管理任务 -> 自我线程` 的上行语义看，现有实例合同应再收成三类：
   - `线程状态消息`
   - `派生需求消息`
   - `结果消息`
8. 其中 `派生需求上报消息` 不再只作为兼容路由入口；它对应正式上行类 `派生需求消息`，用于表达本地不可消化缺条件与外部支撑需求事实。
9. `线程状态消息` 在实现上直接按 `线程状态消息` 落；`当前步骤 / 等待原因 / 是否可继续推进 / 是否已响应控制意图` 等，只作为线程状态消息里的补充特征变化项表达；`结果消息` 则继续承接结果与关键中间状态回流。
10. 作为 `任务管理任务 -> 自我线程` 的业务上行消息，这三类已经封顶，不建议再新增第四类一级类型；`关键中间状态 / 未满足项 / 恢复重建事实` 都应归并进这三类。
11. `确认消息` 若未来存在，应来自另一条验证来源链；`ACK / 重复消费 / 幂等冲突 / 反序到达` 等属于传输层处理，不属于业务上行一级分类。
12. 对这三类正式上行消息，`自我线程` 的底层治理应统一只做两件事：`沉淀治理侧动作摘要/因果材料`，或 `沉淀状态变化`。
13. 三类消息的区别不在“能不能提供动作摘要或状态变化”，而在“何者为主，以及后续进入哪条判定链”：
    - `线程状态消息` 主走 `控制态 / 局部推进态` 判定链
    - `派生需求消息` 主走 `缺口事实 / 需求候选 / 根层重判` 判定链
    - `结果消息` 主走 `结果事实 / 关键中间状态 / 影响项 / 服务安全归结` 判定链
14. `直接管理` 不等于 `自己闭环到底`；下层执行体直接管理现实动作动态后，仍必须把关键事实通过正式上行消息回流给 `自我线程`。
15. 三类消息先落成 `状态变化` 与 `治理侧动作摘要/因果材料` 时，只是完成 `入账`；真正的线程治理工作仍在其后。
16. 入账之后，`自我线程` 还必须继续执行 `归并与对账、二次特征、影响项、根值/需求树/任务权重、根层重判、派发与路由、结算/否定项/学习触发` 这条治理主链。


## 4. 表 1：治理消息字段表

### 4.1 消息公共字段

| 字段段落 | 字段 | 含义 | 备注 |
| --- | --- | --- | --- |
| 识别段 | `消息主键` | 一条消息的唯一标识 | 用于追踪、幂等、审计 |
| 路由段 | `一级来源链` | `线程直达消息 / 任务管理工作线程回流消息` | 是来源链，不是业务语义本体 |
| 路由段 | `二级处理类型` | 7 类正式处理类型之一 | 是线程内部正式处理入口 |
| 行为段 | `发出者主键` | 谁把消息送进线程 | 可与动作主体不同 |
| 行为段 | `动作主体主键` | 谁做了这件动作 | 是动态节点首要落点 |
| 行为段 | `动作语义键` | 这次动作是什么 | 例如 `下发收束指令 / 提交结果 / 确认条件满足` |
| 行为段 | `作用对象主键` | 这次动作主要作用到谁 | 可为空 |
| 路由补充段 | `父任务主键` | 若需要回父任务，父任务是谁 | 便于结果回路由 |
| 场景段 | `关联场景主键` | 这次动作关联到哪个场景 | 便于动态节点回指 |
| 时间段 | `产生时间` | 消息进入线程世界的时间 | 不是唯一生效时间 |
| 追踪段 | `路径签名` | 过程路径签名 | 用于调试与回放 |
| 来源段 | `来源成功标志` | 这条动作或观测链是否成功 | 与动态节点对齐 |
| 来源段 | `错误码` | 来源错误信息 | 与动态节点对齐 |
| 来源段 | `置信度` | 这条消息整体可信度 | 默认允许 `0-10000` |
| 变化段 | `变化项集` | 这次动作导致的全部一次特征变化 | 不允许偷简成单值字段 |

### 4.2 必须固定

1. `一级来源链` 回答“它从哪条来源链进线程”。
2. `二级处理类型` 回答“它在线程内部走哪条处理分支”。
3. `动作语义键 + 变化项集` 才是正式业务语义承载面。


## 5. 表 2：特征变化项表

| 字段 | 含义 | 备注 |
| --- | --- | --- |
| `变化主体主键` | 谁的特征变了 | 不默认等于动作主体 |
| `特征主键` | 改的是哪个一次特征 | 应能映射到一次特征字典 |
| `变化前值` | 变更前值 | 允许为空 |
| `变化后值` | 变更后值 | 不可省略 |
| `是否确认生效` | 是否已成为正式真值 | `false` 时只能作候选事实 |
| `是否关键中间状态` | 是否属于关键中间状态 | 结果回流时尤其重要 |
| `是否已恢复` | 后续是否已被拉回 | 反向中间态抽否定项时使用 |
| `变化原因键` | 为什么发生这次变化 | 便于状态节点原因字段和审计 |

必须固定：

1. 一条消息可带 `1..N` 个变化项。
2. 一个动作若同时改多个特征，必须写成多个变化项。
3. `控制域变化`、`业务域变化`、`关键中间状态变化` 都必须通过变化项表达，不得偷塞进消息名里。
4. 线程状态变化消息中的变化项，只描述“已上报的状态变化事实”，不授予发件线程直接改写自我内部世界的权限。

| 二级处理类型 | 一级来源链 | 是否回父任务 | 是否进入二次特征重算 | 是否进入影响项生成 | 是否进入根层重判 | 是否可能实例化新任务 |
| --- | --- | --- | --- | --- | --- | --- |
| `控制域事实消息` | `线程直达消息` | 否 | 是 | 是 | 是 | 否 |
| `业务域事实消息` | `线程直达消息` | 否 | 是 | 是 | 是 | 条件成立时是 |
| `延迟事实消息` | `线程直达消息` | 否 | 是 | 是 | 是 | 条件成立时是 |
| `工作线程状态消息` | `线程直达消息 / 任务管理工作线程回流消息` | 否 | 是 | 是 | 条件成立时是 | 否 |
| `任务推进关键事件消息` | `任务管理工作线程回流消息` | 否 | 是 | 是 | 条件成立时是 | 否 |
| `派生需求上报消息` | `任务管理工作线程回流消息` | 是 | 是 | 是 | 是 | 是 |
| `结果与关键中间状态回流消息` | `任务管理工作线程回流消息` | 是 | 是 | 是 | 是 | 条件成立时是 |

必须固定：

1. `工作线程状态消息 / 派生需求上报消息 / 结果与关键中间状态回流消息` 这三类来自任务管理上行链的正式业务消息，都必须进入 `二次特征` 链，不能停在“只记账”。
2. `派生需求上报消息` 进入 `根层重判`，不意味着它可直接改正式需求树；正式需求树更新必须晚于候选评估与线程裁决。
3. `结果与关键中间状态回流消息` 进入 `影响项生成`，不意味着它可直接按原始结果字段汇总上级 `服务值 / 安全值`；这些归结必须经 `二次特征 + 影响项` 完成。


## 7. 表 4：消息落位表

| 二级处理类型 | 一次特征变化落位 | 治理侧动作摘要落位 | 二次特征落位 | 影响项落位 | 备注 |
| --- | --- | --- | --- | --- | --- |
| `控制域事实消息` | `状态节点` | `动态节点（治理侧摘要）` | `二次特征` | `影响项` | 控制链先改控制意图，再等事实状态确认 |
| `业务域事实消息` | `状态节点` | `动态节点（治理侧摘要）` | `二次特征` | `影响项` | 业务事实可直接形成服务/安全影响 |
| `延迟事实消息` | `状态节点` | `动态节点（治理侧摘要）` | `二次特征` | `影响项` | 延迟生效事实进入线程后先实体化 |
| `工作线程状态消息` | `状态节点` | `动态节点（治理侧摘要）` | `二次特征` | `影响项` | 当前兼容名沿用；状态节点与治理侧动态摘要由自我线程收到消息后统一落位，故障、超时、失联可直接形成负向影响项 |
| `任务推进关键事件消息` | `状态节点` | `动态节点（治理侧摘要）` | `二次特征` | `影响项` | 先表达局部推进变化，再由线程改任务控制态 |
| `派生需求上报消息` | `状态节点` | `动态节点（治理侧摘要）` | `二次特征` | `影响项` | 主用途是表达缺口事实并形成需求候选；线程审批其是否升格为正式新任务 |
| `结果与关键中间状态回流消息` | `状态节点` | `动态节点（治理侧摘要）` | `二次特征` | `影响项` | 主用途是表达结果事实与关键中间状态；关键中间状态必须保留，不得只交最终结果 |

必须固定：

1. 表 4 不是在区分“三类消息能不能落状态节点或动态节点”，而是在强调它们都会形成 `状态变化` 与 `治理侧动作摘要` 这两类治理材料。
2. `工作线程状态消息` 以 `控制态 / 局部推进态` 更新为主。
3. `派生需求上报消息` 以 `缺口事实 -> 需求候选` 为主，不直接落正式需求树。
4. `结果与关键中间状态回流消息` 以 `结果事实 -> 影响项 -> 服务安全归结` 为主，不直接裸着改上级 `服务值 / 安全值`。
5. 表中的 `动态节点` 属于 `知识沉淀层` 的治理侧摘要与因果材料，不表示 `自我线程` 直接代管下层执行体的现实动作动态本体。
6. 表 4 中的 `状态节点 / 动态节点` 只表示 `一次入账`；后续仍必须继续进入 `对账 / 二次特征 / 影响项 / 根层治理` 链，不能把“已落账”误当成“已裁决”。


## 8. 结果状态消息的额外合同（兼容入口：结果与关键中间状态回流消息）

这类消息必须按“执行事实包”理解，而不是按“成功/失败摘要”理解。

但在传输层，不要求把整包事实全部重复平铺到消息体；默认应由 `任务结果节点` 及其场景信息承载主要事实。

因此消息最小应拆成四段：

1. `结果头`
2. `结果节点引用段`
3. `快速补充段`
4. `后续支撑补充段`

| 段落 | 最小字段 | 作用 |
| --- | --- | --- |
| `结果头` | `发出者主键 / 任务管理任务主键 / 任务主键 / 父任务主键 / 关联场景主键 / 时间戳 / 幂等键` | 说明“这包执行事实是谁在什么时候提交的” |
| `结果节点引用段` | `结果节点主键或指针 / 结果节点所属场景主键 / 是否已正式写入结果节点` | 说明“主要执行事实去哪里读” |
| `快速补充段` | `结果状态 / 完成度或结果值 / 是否成功 / 错误码 / 完成时间` | 说明“最后到底交出了什么”，便于快速路由 |
| `后续支撑补充段` | `是否存在关键中间状态 / 是否仍有未满足项 / 是否需要外部支撑 / 是否需要父任务判断 / 是否需要自我线程重判 / 是否疑似需要新任务入口` | 说明“还缺什么、是否需要继续支撑” |

必须固定：

1. 这类消息的中心不是“最终成没成”，而是“本次执行留下了哪些已知事实”。
2. 主体事实应优先写进 `结果节点` 的场景信息，而不是要求消息层重复整包平铺。
3. `服务对象当前状态 / 关键中间状态 / 未满足项` 等信息，可由自我线程在解析 `结果节点` 后正式读取。
4. `派生需求消息` 仍保持为独立正式上行类；`未满足项 / 后续支撑请求段` 只承接阶段完成后仍残留的支撑缺口，不替代过程内独立上报的 `派生需求消息`。
5. 工作线程只交事实和提示，不直接裁定“否定项成立”。
6. 自我线程收到后，要先解析结果节点，再判断全局影响，再路由父任务，再统一结算。

```cpp

export struct 结构_治理消息头 {

    主键 动作主体主键 = 0;
    主键 作用对象主键 = 0;

    I64 错误码 = 0;
    I64 置信度 = 0;
};
```


## 10. 完成标准

1. 已经把消息头中的 `一级来源链 + 二级处理类型` 正式写进合同。
2. 已经把变化项中的 `关键中间状态 / 已恢复 / 已确认` 三个位补齐。
3. 已经把 7 类消息的回父任务、重判、实例化和影响项生成条件写成矩阵。
