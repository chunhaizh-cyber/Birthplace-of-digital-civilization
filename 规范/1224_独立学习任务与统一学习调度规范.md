# 1224 独立学习任务与统一学习调度规范

## 1. 目标

本文用于固定以下问题：

- 学习需求是否都必须显式落成独立学习任务
- 学习任务实体是否必须显式承载 `学习目标`、`学习需求` 与当前 `候选学习程序`
- 是否应把所有学习语义都揉进一条共享学习任务
- 独立学习任务与统一学习调度之间是什么关系
- 共享学习执行器应承担哪些职责，不应承担哪些职责

本文要收口的是 `学习任务实体` 与 `学习调度执行` 的关系，不是具体学习算法实现。

## 3. 核心结论

必须固定以下 12 条：

1. 每一条学习需求都应显式落成 `独立学习任务实体`。
2. 不允许把所有学习语义都直接揉进一条无来源、无目标、无归属的大一统学习任务。
3. `独立学习任务实体` 负责承载语义、来源、目标、学习需求、当前候选学习程序、学习情况与审计。
4. `统一学习调度器` 负责排队、去重、合并、优先级排序与执行分派。
5. `共享学习执行器` 负责承载实际执行，但真正执行的是当前学习任务当前轮被选中的 `候选学习程序 / 学习方法`；`尝试学习方法` 只是其中一个可选本能入口，不负责替代学习任务实体本身。
6. `独立学习任务实体` 与 `统一学习调度器` 必须同时存在，二者不可互相替代。
7. 即使多个学习任务共用一次学习执行，它们也必须先各自存在，再由调度器决定是否合并执行。
8. 学习结果摘要必须先回写到对应学习任务实体，再反馈给任务管理任务消费；但被选中的学习方法在本轮执行后得到的 `条件-结果对` 允许先直接融合写入方法侧。
9. `任务子任务学习任务` 与 `任务管理独立学习任务` 都适用这条规则。
10. 后续实现不得再把“共享学习执行器存在”误写成“学习任务不需要独立创建”。
11. 当学习任务调度服务于 `任务管理方法` 分支时，外层仍须显式化 `最近根层裁决结果`、`当前缺口类型`、`当前下一步去向`。
12. 任一学习任务执行结果在当前治理轮被消费后，仍必须先回到外层重新做 `根层重判 -> 上下文刷新`，再决定是否继续调度、召回或升格。

一句话收口：

`学习需求必须独立成任务，学习执行允许统一调度。`


## 4. 三层职责划分

### 4.1 独立学习任务实体

回答的是：

- 这次为什么学
- 为谁学
- 学什么
- 学出来以后回给谁

它至少应承载：

- `来源根任务`
- `来源父任务 / 来源任务管理任务`
- `来源方法需求位`
- `学习目标类型`
- `学习需求摘要`
- `学习类型`
- `目标约束强度`
- `缺口量或压力强度`
- `当前候选学习程序ID或版本`
- `方法服务目标`
- `当前状态`
- `学习结果`


### 4.2 统一学习调度器

回答的是：

- 现在该先学谁
- 哪些学习任务可以合并
- 哪些学习任务应暂缓
- 该把哪条任务交给执行器

它至少应负责：

- `排队`
- `优先级排序`
- `去重`
- `相近任务合并`
- `候选学习程序版本切换`
- `执行分派`
- `执行完成后的任务唤醒`


### 4.3 共享学习执行器

回答的是：

- 当前这一轮实际执行哪条学习任务的哪条 `待学习方法`
- 执行到哪一阶段
- 这一轮产出了什么

它至少应负责：

- `承载被选中的候选学习程序或学习方法`
- `执行当前待学习方法`
- `把本轮得到的条件-结果对融合写入方法侧`
- `记录本轮二次特征摘要`
- `采样`
- `更新`
- `验证`
- `提交`
- `回滚`

它不应负责：

- 决定这条学习需求是否存在
- 决定这条学习需求应拆成哪些原函数
- 决定这条学习任务是否进入就绪
- 替代学习任务实体保存来源语义
- 绕过学习任务实体直接完成全部反馈消费
- 绕过外层重判直接决定正式召回、正式执行或成熟度升格

### 4.4 与任务管理治理分支的边界

必须固定：

1. 本文描述的是全域“独立学习任务实体 + 统一学习调度器 + 共享学习执行器”的共性运行结构。
2. 若服务对象是 `任务管理方法` 分支，则必须额外携带：
- `最近根层裁决结果`
- `当前缺口类型`
- `当前下一步去向`
3. 这三项属于治理上下文，不属于所有学习任务实体和所有学习调度状态的统一必填字段。


## 5. 为什么不能只保留一个共享学习任务

如果只保留一条共享学习任务，会丢掉：

1. `来源语义`
- 这轮学习到底是为哪个父任务或哪个根任务而生

2. `目标语义`
- 这轮是为了补缺口，还是为了广泛储备

3. `反馈后的消费边界`
- 学完后应继续学习、挂起观察、使用已完善方法，还是再派生学习任务

4. `审计能力`
- 以后无法清楚回答“这轮学习到底解决了哪条需求”

因此：

`共享学习执行器` 可以保留，`共享学习任务` 不能替代全部学习任务实体。


## 6. 为什么又不能让每个学习任务都独占执行器

如果每个学习任务都各自独占执行器，会带来：

- 重复学习
- 重复扫描学习对象
- 重复写回
- 资源争抢
- 难以去重和合并

因此：

`独立学习任务实体` 不等于 `独立学习执行器实例`。

更合理的方式是：

`多个独立学习任务实体`
-> `统一学习调度器`
-> `共享学习执行器`

并且必须继续固定：

- 共享学习执行器执行结束后，不得直接在本层内部连跑下一轮治理闭环。
- 正确顺序应为：`执行完成 -> 学习任务实体接结果 -> 返回外层根层重判/上下文刷新 -> 再决定是否继续调度`。


## 7. 两类学习任务如何适用本规范

### 7.1 任务子任务学习任务

必须：

- 独立创建
- 保留来源父任务
- 保留来源根任务
- 保留缺口量

默认：

- 优先级更高
- 目标更定向
- 使用时更靠近当前父任务


### 7.2 任务管理独立学习任务

必须：

- 独立创建
- 保留来源任务管理任务
- 保留来源功能域
- 保留来源方法需求位
- 保留上浮压力类型

默认：

- 可被统一调度延后
- 目标更泛化
- 使用时更偏向方法资产与同需求位任务族


## 8. 调度器合并规则

允许合并执行，但必须满足至少以下条件：

- `学习类型` 相同或兼容
- `来源方法需求位` 相同
- `学习目标类型` 相同
- `学习需求摘要` 相近或兼容
- `候选学习程序结构` 可共享
- `反馈消费边界` 不冲突

禁止合并的情况：

- 一个是 `任务子任务学习任务`，一个是 `任务管理独立学习任务`，且目标语义明显不同
- 一个要求 `立即用于当前父任务`，另一个要求 `先挂起观察`
- 一个是 `补条件/补结果`，另一个是 `广泛扩选`

即使合并执行，也必须：

- 保留原始任务列表
- 分开写回各自结果


## 9. 最小落账要求

### 9.1 独立学习任务实体最小字段

- `学习任务ID/头指针`
- `学习任务类型`
- `来源根任务头指针`
- `来源父任务头指针` 或 `来源任务管理任务头指针`
- `来源功能域`
- `来源方法需求位`
- `学习目标类型`
- `学习需求摘要`
- `学习类型`
- `目标约束强度`
- `缺口量或压力强度`
- `当前候选学习程序ID或版本`
- `方法服务目标`
- `当前状态`
- `本轮结果摘要`


### 9.2 学习调度器最小运行态

- `待调度学习任务数`
- `正在执行的学习任务`
- `合并执行组`
- `最近调度时间`
- `最近调度原因`


## 10. 当前实现收口方向

对当前工程，最合理的收口方向是：

1. 保留共享学习执行器。
2. 把所有学习需求先落成独立学习任务。
3. 由学习调度器统一选择：
   - 先执行哪条
   - 是否可合并执行
   - 执行完成后反馈给哪条任务管理承接链消费
4. `任务管理学习树` 负责承载这些独立学习任务。


## 11. 后续实现顺序

建议顺序固定为：

1. 先把现有学习需求全部显式落成独立学习任务实体
2. 再统一补 `学习任务类型 / 方法服务目标 / 调度状态`
3. 再把共享学习执行器前面补成正式 `学习调度器`
4. 最后补“可合并执行、分开回写”的执行组逻辑

并且始终保持：

- `学习任务创建/调度/执行/反馈` 只能作为承接层运行。
- 每轮消费结果后，仍必须回到外层重新做 `根层重判 -> 上下文刷新`。


## 12. 结论

对你当前这套系统来说，最稳的结构不是：

- `只有一个学习任务`

也不是：

- `每个学习任务都各跑一套执行器`

而是：

`独立学习任务实体 + 统一学习调度器 + 共享学习执行器`

这条应作为后续所有学习任务实现的总约束。
