# 方法规范 20260425

状态：当前权威规范。

适用范围：方法主信息、方法树、方法虚拟存在、本能函数、任务筹办、任务执行、学习任务触发、任务管理工作线程与自我线程之间的缺口消息。

本规范覆盖 2026-04-25 收敛后的新逻辑。旧上下文镜像、旧摘要、旧四域方法树隐式直调、工作线程直接建任务等逻辑均不再作为实现依据。

## 1. 总原则

方法是“为了让某个存在的某类特征状态发生目标变化而可被执行的能力单元”。

方法不直接等同于任务。任务的目标来自需求，方法负责尝试达成该目标特征状态。

需求树是任务组织的权威来源。任务只通过任务列表和需求关系被组织，不再依赖旧的任务树步骤结构作为唯一权威。

任务可以引用方法，方法不能自行新建任务。方法执行中发现缺口时，只返回缺口事实和状态需求，由自我线程按需求创建任务。

筹办和执行都会生成缺口。通常筹办生成方法缺口，执行生成条件缺口。

缺口必须落成“带时间的状态信息”。不再单独补一个“需求目标特征类型”字段；特征类型从当前状态和目标状态中读取。

方法运行结果分两类保存。函数功能结果写入结果场景，函数运行情况写入方法虚拟存在。

每次真正调用方法函数时，都必须生成一条动作动态。动作动态记录“自我主动调用某个方法”这一整体动作事实，写入自我所在场景，不写入自我内部世界。

轻量返回结构只用于桥接本轮调用，不是权威数据源。

## 2. 方法的最小结构

方法首节点是一个方法的入口和身份节点。

方法首节点必须承载以下核心信息：

| 字段 | 用途 |
| --- | --- |
| 名称 | 标识方法语义，供控制面板和调试读取。 |
| 节点种类 | 必须能区分方法首节点、方法条件节点、方法结果节点等结构角色。 |
| 动作句柄 | 指向本能函数 ID 或未来可学习动作入口。 |
| 方法虚拟存在 | 记录方法运行状态和可用性状态。 |
| 条件场景 | 记录方法需要哪些输入条件和参数特征。 |
| 结果场景 | 记录方法执行产生的功能事实、缺口和状态变化。 |
| 主结果特征类型 | 表示该方法主要能够改变哪类特征状态。 |
| 结果初始状态 | 作为方法可产生结果方向的证据。 |
| 来源任务 | 记录该方法最初为哪个任务或需求生成，便于追踪。 |

方法首节点的子链用于挂载方法结构样本。当前规范允许并鼓励以下节点挂在方法首节点子链上：

| 子节点类型 | 用途 |
| --- | --- |
| 方法条件节点 | 保存一次或一类条件场景，用于实例化执行参数。 |
| 方法结果节点 | 保存一次或一类结果场景，用于记录结果方向和可达状态。 |
| 学习生成样本节点 | 保存学习任务生成的条件和结果样本，后续可融合为正式条件或结果节点。 |

每次本能函数运行都应该生成或复用一组条件与结果记录，并按条件方向和结果方向进行融合。

融合不是覆盖。相同方向、相同特征类型、相近条件的样本可以合并权重；不同方向或不同结果特征类型的样本应保留为不同候选。

## 3. 方法虚拟存在

方法虚拟存在记录“方法自身的运行情况和可用性”，不记录函数功能结果的细账。

方法虚拟存在是学习任务的直接作用对象。学习任务的目标通常是让某个方法虚拟存在达到 `方法是否可用=1`。

方法虚拟存在至少应记录以下运行特征：

| 特征 | 类型 | 用途 |
| --- | --- | --- |
| 方法是否可用 | I64 或布尔状态 | 0 表示不可用，1 表示可用。学习任务负责产生该状态。 |
| 方法条件节点数量 | I64 | 表示方法首节点下可用于实例化条件的条件节点数量。 |
| 方法结果节点数量 | I64 | 表示方法首节点下可证明结果方向的结果节点数量。 |
| 方法_最近运行状态 | 枚举或 I64 | 记录最近一次运行成功、失败、缺条件、缺方法等状态。 |
| 方法_最近运行阶段 | 枚举或 I64 | 记录最近一次停在确保、解析绑定、执行、输出哪个阶段。 |
| 方法_最近错误码 | I64 | 记录最近一次本能函数或执行桥错误码。 |
| 方法_最近质量 | I64 | 记录最近一次质量评分。 |
| 方法_最近成本 | I64 | 记录最近一次成本评分或耗时换算。 |
| 方法_最近开始时间 | 时间戳 | 记录最近一次运行开始。 |
| 方法_最近结束时间 | 时间戳 | 记录最近一次运行结束。 |
| 方法_最近耗时 | I64 | 记录最近一次运行耗时。 |
| 方法_最近输入场景 | 指针句柄 | 指向本轮输入或条件场景。 |
| 方法_最近结果场景 | 指针句柄 | 指向本轮功能结果账本。 |
| 方法_最近动作动态 | 指针句柄 | 指向本轮真实方法调用生成的动作动态。 |
| 方法_最近目标存在 | 指针句柄 | 指向本轮实际被作用的存在。 |
| 方法_最近宿主任务 | 指针句柄 | 指向触发本轮运行的任务。 |

方法虚拟存在不得写入旧式自然语言摘要作为结构依据。

可读文本只能作为调试显示。权威判断必须来自状态、特征、场景、指针和枚举字段。

## 4. 条件场景

条件场景定义方法需要哪些参数，不负责保存任务上下文镜像。

方法真正执行时才使用条件场景实例化输入参数。

参数来源按任务层级区分：

| 层级 | 被管理对象 | 参数来源 |
| --- | --- | --- |
| 管理自身的任务管理任务 | 任务管理任务自身 | 任务管理任务的虚拟存在。 |
| 管理学习的任务管理任务 | 学习对象方法 | 被学习方法的虚拟存在。 |
| 管理普通任务的任务管理任务 | 被管理任务 | 被管理任务的虚拟存在。 |
| 普通任务 | 需求对应的现实目标 | 任务场景或任务绑定的现实场景。 |

条件场景中的每个条件应尽量表达以下信息：

| 信息 | 用途 |
| --- | --- |
| 所需特征类型 | 方法需要从参数存在或场景中读取哪类特征。 |
| 当前状态约束 | 方法要求该特征当前处于什么状态。 |
| 方向约束 | 方法适用于状态上升、下降、保持或其他方向。 |
| 参数主体类型 | 说明该条件绑定任务虚拟存在、方法虚拟存在还是现实场景。 |
| 缺失时处理 | 说明缺少该条件时应生成什么需求。 |

执行方法无法根据条件节点生成条件时，产生条件缺口。

条件缺口的规范状态需求是：

```text
被需求存在 = 当前方法的虚拟存在
需求特征 = 方法条件节点数量
当前状态 = 当前条件节点数量
目标状态 = 1 或更高
需求去向 = 自我线程
```

## 5. 结果场景

结果场景记录函数功能结果，是本能函数功能事实的权威账本。

结果场景至少记录以下内容：

| 内容 | 用途 |
| --- | --- |
| 执行动作 ID | 表示本轮运行的是哪个本能函数或动作。 |
| 输入条件快照 | 记录本轮实际绑定到的条件。 |
| 输出状态变化 | 记录被改变的状态和特征。 |
| 缺口事实 | 记录缺方法、缺条件、缺结果、缺权限等功能缺口。 |
| 需求当前状态 | 缺口对应的当前状态。 |
| 需求目标状态 | 缺口对应的目标状态。 |
| 结果方向 | 当前状态到目标状态形成的方向。 |
| 结果质量和成本 | 记录功能结果质量和成本。 |
| 相关方法虚拟存在 | 指向本轮方法虚拟存在。 |
| 相关宿主任务 | 指向本轮任务。 |

功能执行结果，例如“状态已改变”“产生新条件需求”“产生补方法需求”，必须写入结果场景。

运行情况，例如“函数成功”“函数失败”“错误码”“耗时”“成本”“质量”，必须写入方法虚拟存在。

返回结构只携带结果场景、方法虚拟存在、成功码、需求当前状态、需求目标状态和方向等桥接字段。

## 6. 动作动态

动作动态记录“自我主动调用方法函数”这一整体变化。

动作动态不是普通状态转换动态的别名。

普通动态只记录某个特征从旧状态变到新状态。它可以说明状态转换顺序，但不能说明导致转换的主动动作。

动作动态必须携带动作来源，至少能回答：

| 问题 | 字段或结构 |
| --- | --- |
| 谁在动作 | 动态主体，通常为自我存在。 |
| 做了什么 | 来源动作、来源动作名或动作句柄对应方法。 |
| 何时发生 | 动态时间段，包含开始和结束时间。 |
| 调用了哪个方法 | 来源方法或方法首节点引用。 |
| 输入来自哪里 | 来源输入场景。 |
| 输出写到哪里 | 来源输出场景。 |
| 是否成功 | 来源执行成功。 |
| 为什么失败 | 来源错误码。 |

动作动态的落点固定为自我所在场景。

```text
动作动态场景 = 世界树.自我所在场景
动态主体 = 自我存在
动态特征 = 方法调用动作动态
来源类型 = 方法调用
来源动作 = 当前被真实调用的方法首节点
来源动作名 = 当前方法动作名；缺失时使用“方法调用”
来源动作相位 = 完成 或 失败
来源输入场景 = 本轮条件场景或输入场景
来源输出场景 = 本轮结果场景或输出场景
```

动作动态由调用方法的函数主动生成。

工作线程、任务管理任务、自我线程不能在没有真实调用方法函数时伪造动作动态。

每次真正调用方法函数都必须生成一条动作动态。即使方法没有改变任何目标特征，也要保留本轮动作事实动态。

真实调用的边界是：方法已经通过可用性检查、条件场景已经能够实例化、执行桥已经进入动作函数或本能函数调用。

以下情况不得生成真实动作动态：

| 情况 | 原因 |
| --- | --- |
| 方法不可用 | 尚未真实调用方法函数。 |
| 缺条件节点 | 尚未实例化执行条件。 |
| 只做筹办查找 | 筹办不是被筹办方法的真实执行。 |
| 只处理线程消息 | 消息落账是治理侧摘要，不是下层真实方法调用。 |

状态信息、普通动态和动作动态的关系如下：

| 信息 | 记录内容 | 落点 |
| --- | --- | --- |
| 状态信息 | 某存在某特征在某时间的值。 | 该特征所属存在的内部世界。 |
| 普通动态 | 同一主体同一特征的旧状态到新状态转换。 | 该特征所属存在的内部世界。 |
| 状态顺序因果 | 旧状态之后出现新状态的转换顺序。 | 该特征所属存在的内部世界。 |
| 动作动态 | 自我真实调用某方法函数的动作事实。 | 自我所在场景。 |
| 动作因果 | 动作动态与后续状态变化之间的因果证据。 | 由后续因果生成逻辑桥接动作动态和结果状态。 |

动作动态可以作为高可信内部因果证据。

动作动态本身不等于方法成熟。它只能证明“自我确实调用过这个方法”。方法是否稳定、条件是否充分、结果是否可复现，仍由学习和稳定复现逻辑判断。

## 7. 因果信息

因果信息由动态信息生成，但不同动态生成的因果可信度不同。

当前规范把因果信息先分为两大类：

| 因果类型 | 来源动态 | 主要用途 |
| --- | --- | --- |
| 动作因果 | 自我真实调用方法形成的动作动态。 | 完善自我的方法，判断某方法在某条件下是否能导致目标结果。 |
| 状态因果 | 状态变化形成的普通动态。 | 发现状态前置链、特征联动关系、任务筹办路径和世界规律。 |

动作因果回答“自我执行了什么方法，导致了什么变化”。

状态因果回答“哪些状态变化稳定地先于、伴随或约束另一些状态变化”。状态因果本身不证明主动动作责任。

### 7.1 动作因果

动作因果的最小形式是：

```text
条件状态集合 + 动作动态 -> 结果状态或结果动态
```

动作因果必须有真实动作动态作为动作侧证据。

动作因果由自我执行方法时产生，主要用于：

| 用途 | 说明 |
| --- | --- |
| 方法可用性判断 | 若某方法多次在相同条件下产生目标结果，可提升方法可用性证据。 |
| 方法条件补全 | 从成功动作因果中提取稳定前置条件，补成方法条件节点。 |
| 方法结果补全 | 从成功动作因果中提取稳定结果状态，补成方法结果节点。 |
| 方法失败学习 | 从失败动作因果中提取缺条件、缺权限、缺资源等失败前置。 |
| 组合方法学习 | 多条动作因果按结果到条件串联后，可形成组合方法候选。 |

动作因果不得在以下情况下生成：

| 情况 | 原因 |
| --- | --- |
| 只有状态变化，没有动作动态 | 无法证明自我执行了动作。 |
| 只有消息落账，没有真实方法调用 | 治理消息不是方法动作。 |
| 方法不可用或缺条件导致未执行 | 尚无动作事实。 |

### 7.2 状态因果

状态因果由普通动态生成。

普通动态至少表达：

```text
某主体 / 某场景 / 某特征: 状态A -> 状态B
```

状态因果的最小生成单位不是“文字摘要”，而是状态动态中的主体、特征类型、状态值、时间段和场景。

状态因果至少包含以下基础类型：

| 基础因果 | 生成来源 | 用途 |
| --- | --- | --- |
| 同特征顺序因果 | 同一主体、同一特征，状态 A 稳定先于状态 B。 | 记录状态演化顺序。 |
| 同特征前置因果 | 同一主体、同一特征，需要 B 时 A 总是先出现。 | 生成需求前置链；要 B 先要 A。 |
| 同特征阶段因果 | 同一主体、同一特征，A -> B -> C 稳定出现。 | 形成任务状态、方法状态、学习状态的阶段链。 |
| 同特征循环因果 | 同一主体、同一特征，A -> B -> A 或多状态循环稳定出现。 | 识别周期、恢复、回退和可逆状态。 |
| 跨特征联动因果 | 同一主体或同一场景，特征 C 的变化稳定先于特征 D 的变化。 | 发现“调 C 可以影响 D”的间接筹办路径。 |
| 跨主体关系因果 | 同一场景内，主体 X 的特征变化稳定先于主体 Y 的特征变化。 | 发现对象间影响关系。 |
| 场景约束因果 | 场景状态变化稳定改变某主体状态变化的可发生性。 | 识别环境条件、资源条件和安全服务边界。 |
| 动态阶段因果 | 某动态的开始、过程、完成稳定构成阶段顺序。 | 识别动作或过程的内部阶段。 |
| 聚合动态因果 | 多个低层动态稳定组成一个高层动态。 | 形成抽象动作、复合事件和组合方法候选。 |
| 阻断因果候选 | 某状态出现后，原本常见的后续状态不再出现。 | 发现禁止项、风险约束和失败原因。 |

其中 `同特征前置因果` 和 `跨特征联动因果` 是当前任务筹办最优先需要的两类状态因果。

### 7.3 状态前置链

状态前置链用于从目标状态倒推前置状态。

示例：

```text
任务状态: 未筹办 -> 已筹办 -> 可执行 -> 已完成
```

当需求目标是 `任务状态=已完成` 时，筹办可以生成前置需求：

```text
需要 已完成
-> 先需要 可执行
-> 先需要 已筹办
```

该链只能作为筹办依据，不得直接宣称需求已经满足。

状态前置链的生成条件：

| 条件 | 说明 |
| --- | --- |
| 同主体或同一可比主体类型 | 避免把不同对象的状态乱连。 |
| 同特征类型 | 前置链第一阶段只处理同一特征。 |
| 时间顺序稳定 | A 的时间必须稳定早于 B。 |
| 反例低于阈值 | B 经常不需要 A 时，不能升格为必要前置。 |
| 可重复出现 | 单次出现只能是候选。 |

### 7.4 跨特征联动链

跨特征联动链用于发现不同特征类型之间的关系。

示例：

```text
方法条件节点数量: 0 -> 1
稳定先于
方法是否可用: 0 -> 1
```

当需求目标是 `方法是否可用=1` 时，筹办可以先生成或选择能让 `方法条件节点数量>0` 的学习任务。

跨特征联动链的生成条件：

| 条件 | 说明 |
| --- | --- |
| 同主体或同一场景 | 优先从同一方法虚拟存在、同一任务虚拟存在、同一现实对象内部发现关系。 |
| C 的变化早于 D 的变化 | 必须有明确时间顺序。 |
| 时间窗口合理 | D 不能离 C 太远，否则只保留弱候选。 |
| D 的变化具有稳定复现 | 偶然共现不能升格。 |
| 存在反例统计 | C 出现但 D 不出现时要降低置信度。 |

跨特征联动因果可以服务间接方法查找。

如果没有直接改变目标特征 D 的方法，但存在稳定联动 `C -> D`，筹办可以先找能改变 C 的方法。

### 7.5 生成与升格规则

由动态生成因果时，必须先生成候选，再逐步升格。

```text
动态样本
-> 候选因果
-> 临时因果
-> 待验证稳定因果
-> 稳定因果
-> 抽象因果模板
```

当前阶段优先实现到候选因果和临时因果即可。

升格必须依赖：

| 证据 | 说明 |
| --- | --- |
| 出现次数 | 同类动态重复出现。 |
| 反例次数 | 前因出现但后果不出现，或后果出现但前因缺失。 |
| 时间窗口 | 前后变化之间的时间间隔是否稳定。 |
| 主体一致性 | 是否同主体、同场景或同类型主体。 |
| 特征一致性 | 是否同特征或稳定跨特征映射。 |
| 动作边界 | 动作因果必须有动作动态边界。 |

状态因果不得直接提升方法成熟度。

状态因果只能帮助筹办拆需求、发现前置条件、发现间接路径和组织学习任务。

动作因果才可以进入方法可用性、方法条件、方法结果和方法成熟度的学习链。

### 7.6 与任务筹办的关系

任务筹办查找方法的顺序应逐步扩展为：

| 顺序 | 查找依据 |
| --- | --- |
| 1 | 直接能产生目标特征类型结果的方法。 |
| 2 | 方法结果节点中已有同特征目标方向的方法。 |
| 3 | 状态前置链中能先达成前置状态的方法。 |
| 4 | 跨特征联动链中能间接影响目标特征的方法。 |
| 5 | 动作因果链中可组合到目标结果的方法组合候选。 |
| 6 | 仍无方法时，生成缺方法或学习需求。 |

因果只负责提供候选路径。

最终是否执行，仍必须经过需求方向、方法可用性、条件满足、根层安全服务边界和当前任务状态判断。

### 7.7 使用位置与消费方式

因果信息不是独立运行的调度器。

因果信息的使用位置固定为：筹办、执行后结算、学习任务、方法维护、自我线程治理和展示诊断。

| 使用位置 | 使用动作因果 | 使用状态因果 | 输出结果 |
| --- | --- | --- | --- |
| 任务筹办 | 查找能直接或组合达成目标结果的方法。 | 查找前置状态链和跨特征间接路径。 | 候选方法、前置需求、间接需求、缺方法需求。 |
| 方法执行前 | 检查当前条件是否接近历史成功动作因果。 | 检查目标状态是否存在必要前置未满足。 | 允许执行、先补条件、先补前置需求。 |
| 方法执行后 | 把本轮动作动态和结果状态生成动作因果候选。 | 把本轮状态动态并入状态顺序、前置和联动候选。 | 候选因果、方法学习证据、状态规律证据。 |
| 学习任务 | 用成功/失败动作因果补方法条件、方法结果和可用性。 | 用状态因果发现应学习哪个前置或联动方法。 | 方法条件节点、方法结果节点、方法是否可用状态。 |
| 方法维护 | 用动作因果统计方法稳定性、失败条件和反例。 | 不直接提升方法成熟度，只提供外部规律提示。 | 方法虚拟存在运行特征、成熟度候选。 |
| 自我线程治理 | 接收因果候选，按需求生成任务或学习任务。 | 把前置链和联动链转成需求树上的子需求。 | 需求树更新、任务创建、等待任务唤醒。 |
| 控制面板 | 展示某任务、需求、方法为什么被选择或阻塞。 | 展示状态前置链、跨特征联动链和反例。 | 可解释诊断，不参与权威判断。 |

任务筹办是状态因果的第一消费方。

筹办面对一个目标需求时，按以下顺序使用因果：

```text
目标需求状态
-> 直接找目标特征方法
-> 查状态前置链，生成前置子需求
-> 查跨特征联动链，生成间接子需求
-> 查动作因果链，生成组合方法候选
-> 仍无路径，生成缺方法需求
```

方法执行后结算是动作因果的第一生成方，也是状态因果的重要生成方。

执行后必须先把结果写入状态和动态，再由因果生成器消费这些动态：

```text
真实方法调用
-> 动作动态
-> 结果状态变化
-> 普通状态动态
-> 动作因果候选
-> 状态因果候选
```

学习任务是动作因果的第一学习消费方。

学习任务不得只因状态因果稳定就直接把方法标为可用。

学习任务可以使用状态因果来决定“应该补哪个前置状态或联动特征”，但必须使用动作因果或实际执行复现来证明“哪个方法能做到”。

自我线程是因果转需求的权威入口。

工作线程和方法执行体不能因为发现因果路径就直接创建任务。它们只能返回因果候选、缺口事实或需求状态，由自我线程把它们转成需求树更新和任务创建。

### 7.8 两类因果的边界

动作因果偏向“方法学习”。

状态因果偏向“需求拆解和世界规律”。

| 问题 | 优先使用 |
| --- | --- |
| 这个方法是否真的能改变目标状态？ | 动作因果。 |
| 这个方法缺哪些稳定条件？ | 动作因果为主，状态因果辅助。 |
| 要达到状态 B 前是否必须先有状态 A？ | 状态因果。 |
| 改变特征 C 是否会带动特征 D？ | 状态因果。 |
| 没有直接方法时能不能走间接路径？ | 状态因果先给路径，动作因果再证明路径上的方法。 |
| 方法是否可以升格为可用？ | 动作因果和执行复现，状态因果不能单独升格。 |
| 当前任务为什么暂停？ | 状态因果解释前置缺口，动作因果解释方法失败或缺条件。 |

如果两类因果冲突，按以下规则处理：

| 冲突 | 处理 |
| --- | --- |
| 状态因果认为 A 必要，但动作因果显示方法可绕过 A 达成 B。 | 降低 A 的必要前置置信度，保留为常见前置。 |
| 动作因果显示方法成功，但状态因果显示后续经常引发负面状态。 | 方法可用性不直接否定，但执行前门控必须看到风险链。 |
| 状态因果显示 C 会带动 D，但动作因果复现失败。 | C -> D 保留为世界候选规律，不作为当前方法证据。 |
| 动作因果样本少，状态因果样本多。 | 可用于筹办候选，不可用于方法升格。 |

### 7.9 存储形态

动作因果和状态因果都存为因果节点。

短期不新增两套完全不同的数据结构，而是复用 `因果主信息类`，用字段约定区分来源、动作侧、状态迁移侧和证据样本。

当前 `因果主信息类` 的核心字段可对应为：

| 字段 | 动作因果用法 | 状态因果用法 |
| --- | --- | --- |
| 形态 | 候选或临时阶段可先用实例；多次归并后升为抽象。 | 同左。 |
| 适用锚点类型 | 自我、方法虚拟存在、任务虚拟存在、现实对象或场景。 | 同主体、同场景、跨主体关系所归属的锚点。 |
| 适用层级 | 动作动态或结果状态所在动态层级。 | 状态动态层级或聚合动态层级。 |
| 条件模板 | 动作执行前稳定条件特征。 | 前置状态、前置特征或场景条件。 |
| 动作模板 | 指向方法首节点、动作语义节点或动作动态语义。 | 空或指向“状态转移/联动/阻断”等状态因果语义节点。 |
| 动作名称 | 方法动作名。 | `状态前置`、`跨特征联动`、`场景约束` 等状态因果类型名。 |
| 动作语义键 | 方法调用语义键或动作句柄键。 | 状态因果类型键，例如 `state.precondition`、`state.cross_feature`。 |
| 状态迁移模板 | 本方法导致的结果状态迁移。 | A -> B、C变化 -> D变化、场景变化 -> 主体变化等迁移模板。 |
| 主状态迁移 | 主要结果状态迁移。 | 主要后继状态或主要被影响特征。 |
| 来源类型 | 方法调用、观察生成、学习生成等。 | 通常为观察生成或状态动态生成。 |
| 置信度 | 动作因果稳定度。 | 状态规律稳定度。 |
| 已验证 | 需要真实复现后才能为真。 | 只能表示状态规律已稳定，不表示方法已验证。 |
| 证据动态样本 | 真实动作动态及其结果动态样本。 | 普通状态动态、聚合动态、阻断样本动态。 |
| 成立次数 | 动作后目标结果出现次数。 | 前置或联动关系成立次数。 |
| 失败次数 | 动作后目标结果未出现或反向结果次数。 | 前置缺失仍出现后果、前因出现但后果未出现等反例次数。 |
| 最近命中时间 | 最近一次动作因果成立时间。 | 最近一次状态规律命中时间。 |
| 最近失败时间 | 最近一次动作因果反例时间。 | 最近一次状态规律反例时间。 |

动作因果的推荐存储形态：

```text
因果节点
  形态 = 实例 或 抽象
  来源类型 = 方法调用 / 学习生成
  动作模板 = 方法首节点 或 动作语义节点
  动作名称 = 方法动作名
  动作语义键 = 方法动作键
  条件模板 = 方法执行前条件特征集合
  状态迁移模板 = 动作后产生的结果状态迁移集合
  主状态迁移 = 本轮主结果迁移
  证据动态样本 = 动作动态 + 相关结果动态
  成立次数 / 失败次数 / 置信度 / 已验证
```

状态因果的推荐存储形态：

```text
因果节点
  形态 = 实例 或 抽象
  来源类型 = 观察生成 / 状态动态生成
  动作模板 = 空，或状态因果语义节点
  动作名称 = 状态前置 / 跨特征联动 / 场景约束 / 阻断候选
  动作语义键 = state.precondition / state.cross_feature / state.scene_gate / state.block
  条件模板 = 前置状态或前置特征集合
  状态迁移模板 = 后继状态迁移或被影响特征迁移集合
  主状态迁移 = 主要后继状态迁移
  证据动态样本 = 普通状态动态或聚合动态
  成立次数 / 失败次数 / 置信度
```

状态因果虽然复用 `动作模板` 字段，但不得把该字段解释成真实动作。

当 `动作语义键` 以 `state.` 开头时，`动作模板 / 动作名称` 表示状态因果语义，不表示自我做了动作。

### 7.10 存储位置与索引

因果节点应挂在最能解释该因果的锚点下。

| 因果类型 | 推荐挂载位置 | 索引位置 |
| --- | --- | --- |
| 自我方法调用动作因果 | 自我所在场景或自我存在内部世界中的因果区。 | 场景因果索引、方法虚拟存在因果索引。 |
| 方法学习动作因果 | 被学习方法的虚拟存在内部世界。 | 方法虚拟存在、学习任务、相关方法节点。 |
| 普通任务动作因果 | 被管理任务虚拟存在或任务场景。 | 任务虚拟存在、来源需求、结果场景。 |
| 同特征状态前置因果 | 该特征所属存在的内部世界。 | 特征节点、状态节点、存在内部世界。 |
| 跨特征联动因果 | 同一主体内部世界；跨主体时挂场景。 | 两个特征节点、主体存在、场景。 |
| 场景约束因果 | 场景节点。 | 场景因果索引、受影响主体索引。 |
| 聚合动态因果 | 高层动态所属场景。 | 高层动态、低层动态样本、场景。 |
| 阻断因果候选 | 被阻断结果所属主体或场景。 | 目标特征、阻断状态、场景。 |

存储原则：

1. 因果节点本体只存结构引用和统计，不复制状态、动态、方法、任务的大对象。
2. 具体证据必须通过 `证据动态样本` 回到动态节点，再从动态节点回到状态节点。
3. 可查询索引可以出现在场景、存在、特征、方法虚拟存在或任务虚拟存在中，但索引只保存引用。
4. 同一因果重复出现时应合并统计，不应无限新建同义因果节点。
5. 若状态因果和动作因果共享同一个结果状态迁移，也应存为两条不同因果，因为它们回答的问题不同。

去重键建议：

| 因果类型 | 去重键 |
| --- | --- |
| 动作因果 | 锚点类型 + 动作语义键 + 条件模板签名 + 主状态迁移签名 + 层级。 |
| 同特征状态前置因果 | 主体类型或主体主键 + 特征类型 + 前置状态签名 + 后继状态签名 + 层级。 |
| 跨特征联动因果 | 主体类型或场景主键 + 前置特征类型 + 后继特征类型 + 前置迁移签名 + 后继迁移签名。 |
| 场景约束因果 | 场景类型或场景主键 + 场景状态签名 + 受影响主体类型 + 受影响特征类型。 |
| 阻断因果候选 | 阻断状态签名 + 原预期后继状态签名 + 时间窗口签名。 |

存储阶段建议：

```text
实例样本：保留具体证据动态样本，方便追溯。
临时因果：合并同类样本，累计成立/失败次数。
稳定因果：保留代表样本和统计，不再保存无限样本。
抽象模板：删除单次主体身份，只保留可比较条件、动作或状态迁移模式。
```

## 8. 方法查找与筹办

筹办的核心职责是根据需求找方法。

筹办输入是任务，任务的目标来自需求。

筹办查找方法时，先从需求状态提取目标特征类型。

方法候选必须满足以下至少一项：

| 条件 | 说明 |
| --- | --- |
| 主结果特征类型等于需求目标特征类型 | 最直接匹配。 |
| 方法结果节点中存在同类型结果状态 | 可作为候选。 |
| 方法结果方向与需求方向同向 | 可作为弱候选。 |

当找到多个候选方法时，优先选择更精准的方法。

精准度排序应考虑以下因素：

| 因素 | 优先规则 |
| --- | --- |
| 目标特征类型完全一致 | 高于泛化方法。 |
| 结果状态更接近目标状态 | 高于只同向但距离较远的方法。 |
| 条件约束更匹配当前状态 | 高于条件宽泛或条件未知的方法。 |
| 方法是否可用为 1 | 高于不可用方法。 |
| 方法已有条件节点和结果节点 | 高于只有占位的方法。 |
| 存在稳定因果路径 | 高于没有因果路径的间接候选。 |

任务状态类需求可以按状态出现顺序生成子需求。例如从未启动到完成，可以按筹办中、就绪、执行中、完成等顺序拆分。

筹办发现没有方法时，生成缺方法需求。

筹办发现有目标方法占位但缺动作骨架时，也生成缺方法需求。

目标方法缺动作骨架时，规范需求是：

```text
被需求存在 = 目标方法的虚拟存在
需求特征 = 方法是否可用
当前状态 = 0
目标状态 = 1
需求去向 = 任务管理任务组织学习
```

筹办不得直接创建学习任务或普通子任务。

筹办只返回缺口，任务创建只能由自我线程根据需求完成。

## 9. 方法执行

执行的核心职责是运行当前已选方法。

执行入口是 `执行任务`。

`执行任务` 的参数是任务，也就是自我线程创建的某个任务管理任务实例或普通任务实例。

工作线程执行一轮时，应返回两类数据：

| 数据 | 含义 |
| --- | --- |
| 任务执行数据 | 本轮任务状态、记账结果、当前方法、结果场景、任务状态变化等。 |
| 方法实际数据 | 本轮方法返回的功能结果、缺口状态、动作动态、方法虚拟存在、成功码和解释。 |

执行前不应由工作线程自行拆四段式。

本能函数运行一次时，内部自动完成四段式：

| 阶段 | 职责 |
| --- | --- |
| 确保 | 确保方法首节点、方法虚拟存在、输入场景、结果场景可用。 |
| 解析绑定 | 根据条件场景从参数存在或任务场景中绑定实际值。 |
| 执行 | 执行方法动作，改变目标状态或形成缺口。 |
| 输出 | 将功能事实写结果场景，将运行情况写方法虚拟存在，返回轻量桥接结果。 |

执行桥在真正调用动作函数或本能函数时，必须生成动作动态。

动作动态生成不替代结果场景。结果场景仍记录函数功能事实，动作动态只记录“本轮确实发生了方法调用”。

执行发现条件节点缺失时，生成缺条件需求。

执行发现方法虚拟存在未达到 `方法是否可用=1` 时，可以生成方法可用性需求，但这通常应由筹办提前发现。

执行不得直接创建任务。

执行不得恢复旧上下文镜像逻辑。

执行不得依赖旧摘要判断。

## 10. 本能函数

本能函数是方法函数的最小运行单元。

所有本能函数必须遵守内部四段式执行壳。

外部执行桥只负责调用本能函数并接收轻量返回，不再理解四段式细节。

本能函数必须满足以下返回规范：

| 返回信息 | 存储位置 |
| --- | --- |
| 函数功能结果 | 结果场景。 |
| 缺条件、缺方法等功能缺口 | 结果场景。 |
| 改变了哪些特征类型和值 | 结果场景。 |
| 成功、失败、错误码 | 方法虚拟存在。 |
| 成本、质量、耗时 | 方法虚拟存在。 |
| 动作动态入口 | 轻量返回结构和方法虚拟存在。 |
| 结果场景入口 | 轻量返回结构和方法虚拟存在。 |
| 方法虚拟存在入口 | 轻量返回结构和上行消息。 |

每个本能函数执行后，都应该在方法树中生成或融合一组条件与结果记录。

条件按条件方向融合，结果按结果方向融合。

如果缺条件，本能函数或执行桥应在结果场景记录“当前方法需要条件节点”。

如果缺方法，本能函数或筹办应在结果场景记录“目标方法需要可用”。

## 11. 三层任务共用同一套方法逻辑

当前任务执行逻辑统一覆盖三层任务。

| 层级 | 目标 | 方法作用对象 | 典型需求 |
| --- | --- | --- | --- |
| 任务管理任务管理自身 | 完善任务管理任务自身能力 | 任务管理任务虚拟存在 | 任务状态变为完成。 |
| 任务管理任务管理学习 | 完善被学习方法 | 方法虚拟存在 | 方法是否可用变为 1。 |
| 任务管理任务管理普通任务 | 完成普通任务需求 | 被管理任务虚拟存在或任务场景 | 需求状态被满足。 |

这三层不使用三套执行逻辑。

差异只在参数来源、需求目标和结果回写对象。

任务管理任务默认固定两个顶层步骤：

| 步骤 | 对应本能方法 | 职责 |
| --- | --- | --- |
| 筹办 | 自我_任务筹办 | 根据需求找方法，发现缺方法。 |
| 执行 | 自我_任务执行一步 | 执行当前方法，发现缺条件或写入结果。 |

反馈是任务执行闭环中的结果回流逻辑，不要求作为任务管理任务默认第三个顶层步骤。

## 12. 学习任务触发

学习任务不是工作线程直接创建的任务。

学习任务由自我线程根据需求创建。

创建学习任务的典型需求是：

```text
被需求存在 = 具体方法的虚拟存在
需求特征 = 方法是否可用
当前状态 = 0
目标状态 = 1
```

有了该需求后，调度逻辑再查找能满足 `方法是否可用=1` 的方法。

学习过程可以进一步派生更细需求，例如：

```text
被需求存在 = 具体方法的虚拟存在
需求特征 = 方法结果节点数量
当前状态 = 0
目标状态 = 1
```

学习任务应补齐方法的条件节点、结果节点、动作骨架或结果方向证据。

学习任务成功后，应更新方法虚拟存在的 `方法是否可用=1`，并唤醒等待该方法的父任务重新筹办。

## 13. 消息与线程边界

自我线程是唯一能够正式创建任务的线程。

工作线程不能直接创建任务。

工作线程只执行当前任务方法，并返回任务执行数据和方法实际数据。

任务管理界面线程负责接收任务管理工作线程的上行消息，并改变任务管理任务状态。

工作线程可以把消息交给自我线程，但持久收取和状态处理应由任务管理界面线程承担，因为工作线程可能退出。

自我线程消费消息时可以沉淀治理侧动态摘要，但不得把治理侧摘要冒充真实动作动态。

真实动作动态只能由真实调用方法函数的执行体生成，并落在自我所在场景。

派生需求消息至少应携带：

| 字段 | 用途 |
| --- | --- |
| 任务主键 | 说明哪个任务产生缺口。 |
| 当前方法主键 | 能定位方法节点时填写。 |
| 方法虚拟存在主键 | 方法节点主键缺失时仍可定位学习对象。 |
| 派生需求类型 | 缺方法、缺条件、任务步骤等。 |
| 需求特征类型 | 从需求当前状态或目标状态反推。 |
| 需求当前状态 | 缺口当前状态。 |
| 需求目标状态 | 缺口目标状态。 |
| 当前等待原因 | 调试文本，不作为权威结构依据。 |

当前日志的有效形态示例：

```text
任务管理派生需求 | 类型=缺方法 | 方法虚拟存在=8d | 主观察=方法是否可用 | 当前状态=KF | 目标状态=KG
```

未来缺条件日志应呈现类似形态：

```text
任务管理派生需求 | 类型=缺条件 | 方法虚拟存在=<方法虚拟存在主键> | 主观察=方法条件节点数量 | 当前状态=<0> | 目标状态=<1>
```

## 14. 废弃项

以下逻辑不得作为新实现依据：

| 废弃项 | 原因 |
| --- | --- |
| 旧上下文镜像 | 新执行参数由条件场景和目标存在绑定，不再复制上下文。 |
| 旧摘要字段 | 摘要不是结构合同，不能驱动判断。 |
| 工作线程直接建任务 | 任务只能由自我线程根据需求创建。 |
| 任务管理任务直接在当前主需求下长子任务 | 最新规范要求先提需求。 |
| 隐式四域方法树直调 | 旧四域函数应转成本能方法，由筹办组织。 |
| 结果节点作为任务必需节点 | 任务主信息已收束，结果事实进入结果场景。 |
| 步骤节点强制组织任务树 | 当前以需求树和任务列表组织任务，步骤只保留为方法执行过程中的可选结构。 |
| 在自我内部世界伪造真实动作动态 | 动作动态属于自我所在场景中的真实方法调用事实。 |

## 15. 验收标准

方法规范实现完成时，应满足以下检查项：

| 检查项 | 通过标准 |
| --- | --- |
| 筹办缺方法 | 日志能看到 `类型=缺方法`、`主观察=方法是否可用`、当前状态和目标状态。 |
| 执行缺条件 | 日志能看到 `类型=缺条件`、`主观察=方法条件节点数量`、当前状态和目标状态。 |
| 方法虚拟存在 | 派生需求和结果状态消息能带回方法虚拟存在主键。 |
| 结果场景 | 本能函数功能结果能从结果场景读取。 |
| 动作动态 | 每次真实方法调用都能在自我所在场景找到一条方法调用动作动态。 |
| 动作动态边界 | 方法不可用、缺条件、消息落账等未真实调用方法的路径不会生成真实动作动态。 |
| 状态因果候选 | 同特征状态顺序和跨特征联动能先形成候选或临时因果，不直接冒充动作因果。 |
| 动作因果候选 | 真实动作动态能与结果状态形成候选动作因果，用于方法学习链。 |
| 运行账 | 方法运行成功、失败、阶段、耗时和错误码能从方法虚拟存在读取。 |
| 学习触发 | `方法是否可用=1` 的需求能派生学习任务。 |
| 任务创建边界 | 新任务只由自我线程创建。 |
| 旧逻辑清理 | 旧上下文和旧摘要不参与新判断。 |
