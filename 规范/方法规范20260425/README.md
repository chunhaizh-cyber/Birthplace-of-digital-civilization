状态：当前权威规范。

适用范围：方法主信息、方法树、方法虚拟存在、本能函数、需求树、任务筹办、任务执行、缺口需求生成与回传。

## 0. 最新定版：需求树与两阶段任务

需求树是运行主结构。任务不是管理者，而是需求节点的运行壳；每个需求节点都应有一个对应任务。任务只有两个核心阶段：

| 阶段 | 职责 |
| --- | --- |
| 筹办 | 读取需求结果、上下文条件包与限制包，执行或等价执行 `查找方法`，获得结果能力命中的方法配对结果；若找不到能力，则生成方法补齐入口子需求。 |
| 执行 | 根据方法配对结果校验并解析方法首节点，配对参数，生成可执行输入参数场景并调用方法入口；若条件不闭合，则生成条件缺口子需求。 |

不再把 `任务管理任务` 或 `学习任务` 作为长期结构权威。旧代码中的同名角色只作为过渡兼容壳存在；学习、候选方法补齐、能力包补齐、特征父链补齐、参数补齐等都表达为需求树上的原子需求节点。

核心规则：

1. 需求节点表达“要满足什么”，任务表达“为满足该需求而运行的过程壳”。
2. 缺口不是失败终点；方法补齐入口和条件缺口都必须尽量生长为子需求。
3. 叶子需求是当前方法可执行且条件满足的需求，不是静态“无子节点”的结构概念。
4. 叶子需求执行成功后，满足状态向父需求回传，父需求重新检查自身缺口是否闭合。
5. `查找方法` 的权威输出是按结果能力命中的 `方法配对结果存在`；`选中自我能力值` 只在命中本能方法时作为执行入口缓存，不能替代结果查找。

## 1. 总原则

方法是“为了让某个存在的某类特征状态发生目标变化而可被执行的能力单元”。

方法不直接等同于任务。任务的目标来自需求，方法负责尝试达成该目标特征状态。

需求树是任务组织的权威来源。任务只通过需求节点的对应关系被组织，不再依赖旧的任务树步骤结构，也不依赖独立任务管理任务作为权威调度者。

任务可以引用方法，方法不能自行新建任务。方法执行中发现缺口时，只返回缺口事实和状态需求，由自我线程按需求创建任务。

筹办和执行都会生成缺口。通常筹办生成方法补齐入口，执行生成条件缺口。

缺口必须落成“带时间的状态信息”。不再单独补一个“需求目标特征类型”字段；特征类型从当前状态和目标状态中读取。

方法运行结果分两类保存。函数功能结果写入结果场景，函数运行情况写入本轮运行虚拟存在，并可把最近运行摘要同步到方法虚拟存在。

每次真正调用方法函数时，都必须生成一条动作动态。动作动态记录“自我主动调用某个方法”这一整体动作事实，写入自我所在场景，不写入自我内部世界。

轻量返回结构只用于桥接本轮调用，不是权威数据源。

## 2. 方法的最小结构

方法首节点是一个方法的入口和身份节点。

方法首节点必须承载以下核心信息：

| 字段 | 用途 |
| --- | --- |
| 名称 | 标识方法语义，供控制面板和调试读取。 |
| 节点种类 | 必须能区分方法首节点、方法条件节点、方法结果节点等结构角色。 |
| 动作句柄 | 指向本能函数 ID 或未来可学习动作入口。 |
| 方法虚拟存在 | 记录方法运行状态和可用性状态。 |
| 能力签名集 | 记录该方法在若干条件组成立时能产生哪些结果变化包。 |
| 方法级条件/结果索引 | 只作为能力签名、条件节点和结果节点的派生索引，不作为单槽结果声明。 |
| 来源任务 | 记录该方法最初为哪个任务或需求生成，便于追踪。 |

方法首节点的子链用于挂载方法结构样本。当前规范允许并鼓励以下节点挂在方法首节点子链上：

| 子节点类型 | 用途 |
| --- | --- |
| 方法条件节点 | 保存一次或一类条件场景，用于实例化执行参数。 |
| 方法结果节点 | 保存一次或一类结果场景，用于记录结果方向和可达状态。 |
| 学习生成样本节点 | 保存学习型子需求生成的条件和结果样本，后续可融合为正式条件或结果节点。 |

每次本能函数运行都应该先产生条件结果对样本，再进入方法树的条件结果对归并。

条件结果对归并包含两种动作：

```text
添加：方法树中没有等价的条件节点或结果节点，需要新增结构。
融合：方法树中已有等价的条件节点或结果节点，只更新统计和证据。
```

执行阶段只负责产生样本，不直接更新结果符合/不符合统计，也不直接推动方法可用性。样本进入归并阶段后，再根据是否已有等价条件、等价结果决定添加还是融合。

条件结果对样本至少包含：

```text
来源方法
来源任务
执行前条件状态集
执行后实际结果状态集
执行结束时间
```

归并只吸收实际条件和实际结果：

```text
实际结果负责增加样本次数和最近样本时间。
同条件下复现已有等价结果，记录结果符合次数。
同条件下出现新的非等价结果分支，记录结果不符合次数。
```

样本本身不携带旧隔离学习验证字段。学习任务的直接产物是方法树变化；普通任务是否满足需求，由需求满足检查和自我线程下一轮筹办判断，不写回为学习样本的执行前目标字段。

融合不是覆盖。相同方向、相同特征类型、等价条件下的样本可以融合统计；不同条件、不同方向或不同结果特征类型的样本应保留为不同候选。

四种归并组合固定为：

| 条件节点 | 结果节点 | 操作 |
| --- | --- | --- |
| 不存在 | 不存在 | 添加条件节点，添加结果节点。 |
| 不存在 | 结果在别的条件下存在 | 添加条件节点，并在新条件下添加结果节点。 |
| 存在 | 不存在 | 融合条件节点，添加结果节点。 |
| 存在 | 存在 | 融合条件节点，融合结果节点。 |

结果节点必须挂在具体条件节点下。同一个结果出现在不同条件下，不能因为结果相同就把统计混在一起。

不能融合的情况：

```text
条件主体不同
条件特征类型不同
条件状态不可比较
条件关系不重叠
结果主体不同
结果特征类型不同
结果状态不可比较
结果方向或目标关系不重叠
```

实际结果是事实，不天然等于任务成功或失败。结果符合/不符合账只回答两个问题：同一条件下结果是否复现；或普通任务结果在需求满足检查中是否满足目标状态。它不要求执行样本携带执行前目标字段。

## 2.1 方法能力签名

方法不是“一个条件对应一个结果”，而是：

```text
方法能力 = 条件面 -> 结果包
方法 = 能力签名集
```

条件面由多个条件组构成：

```text
同一条件组内：条件项全部满足。
多个条件组之间：任意一个条件组满足即可。
```

结果包由多个结果项构成：

```text
同一结果包内可以同时声明多个特征变化。
```

因此一个方法可以表达：

```text
条件组A -> 结果包A
条件组B -> 结果包B
条件组C -> 结果包C
```

条件项和结果项必须支持 `作用对象` 与 `绑定名`。绑定名用于保证条件和结果作用于同一个对象，避免“条件说的是方法A，结果却写到方法B”。

示例：

```text
条件：
  目标方法.候选方法存在状态 = 存在
  目标方法.运行账可生成 = 是

结果：
  目标方法.方法运行账完整性 ↑
```

这里 `目标方法` 是绑定名。查找、配参、执行都必须保持该绑定。

## 3. 方法虚拟存在

方法虚拟存在记录“方法自身的运行情况和可用性”，不记录函数功能结果的细账。

方法虚拟存在是学习型子需求的直接作用对象。学习目标应拆成候选方法存在、动作骨架、条件结果配对、运行账完整性、可用性等原子状态，而不是只用 `方法是否可用=1` 单槽承载。

初始化方法虚拟存在时必须同时写入两类信息：

1. 通用特征信息：所有方法都应具备，用于运行、治理、学习、自检和统计。
2. 区分特征信息：用于说明这个方法和其他方法有什么不同。

这两类信息缺一不可。只写通用特征会导致多个方法虚拟存在不可区分；只写区分信息会导致方法无法进入统一运行、学习和治理链路。

每个方法虚拟存在的特征信息都必须能区分该方法。方法虚拟存在不能只是一组完全相同的通用运行字段；它至少要包含能说明“这是什么方法、作用于什么目标、凭什么条件产生什么结果”的身份特征或结构标识特征。

通用运行特征可以同名，例如 `方法是否可用`、`最近运行状态`、`结果符合次数`、`结果不符合次数`。但这些通用特征不能成为方法虚拟存在的全部内容。两个不同方法即使当前都不可用、同条件结果复现统计相同，也必须在身份、动作、目标、条件或结果结构上有差异。

方法虚拟存在至少应具备以下区分信息中的若干项。动作骨架方法应具备动作侧区分信息；需求位方法应具备目标侧区分信息；学习生成方法应尽快补齐条件和结果侧区分信息。

| 区分信息 | 类型 | 用途 |
| --- | --- | --- |
| 方法标识键 | 文本或主键特征 | 稳定标识该方法，可由动作名、动作句柄、目标特征和结果方向组合生成。 |
| 动作入口类型 | I64 或枚举 | 区分本能函数、外部实现、动作序列、学习生成动作等入口来源。 |
| 动作语义键 | 文本或主键特征 | 表示方法做什么，避免多个方法只有通用运行账而无法分辨。 |
| 本能动作ID | I64 | 本能函数方法必须写入，用于区分不同本能动作。 |
| 能力签名键 | 文本或结构键 | 表示一组 `条件面 -> 结果包` 的稳定身份。 |
| 结果项集合标识 | 文本、I64 或二次特征引用集合 | 表示该方法能同时造成哪些特征变化。 |
| 条件结构标识 | 文本或结构引用 | 表示方法执行所需条件类型集合。 |
| 结果结构标识 | 文本或结构引用 | 表示方法可产生的结果类型集合。 |
| 来源范围 | 指针句柄或主键 | 区分该方法属于哪个主体、任务、学习对象或方法根。 |

如果两个方法虚拟存在的区分信息完全一致，系统应优先复用同一个方法或合并统计，不应新建两个只在节点主键上不同的方法虚拟存在。若它们确实需要共存，必须补充一个真实语义差异，例如不同条件结构、不同结果方向、不同作用范围或不同动作入口。

方法虚拟存在至少应初始化以下通用特征信息：

| 特征 | 类型 | 用途 |
| --- | --- | --- |
| 方法是否可用 | I64 或布尔状态 | 0 表示不可用，1 表示可用。由练习或学习型本能按结果能力推动。 |
| 方法类型 | I64 或枚举 | 表示内部方法、外部方法等通用类型。 |
| 动作入口类型 | I64 或枚举 | 表示本能函数、外部实现、动作序列、学习生成动作等入口来源。 |
| 成熟度阶段 | I64 或枚举 | 表示候选、可学习、可召回、可默认执行等阶段。 |
| 允许自动召回 | I64 或布尔状态 | 表示调度和筹办是否可自动召回该方法。 |
| 允许默认执行 | I64 或布尔状态 | 表示满足条件后是否允许默认执行。 |
| 特征账本生命周期 | I64 或枚举 | 表示候选、活跃、保留、淘汰等生命周期。 |
| 方法条件节点数量 | I64 | 表示方法首节点下可用于实例化条件的条件节点数量。 |
| 方法结果节点数量 | I64 | 表示方法首节点下可证明结果方向的结果节点数量。 |
| 方法_最近运行状态 | 枚举或 I64 | 记录最近一次运行成功、失败、缺条件、候选方法补齐等状态。 |
| 方法_最近运行阶段 | 枚举或 I64 | 记录最近一次停在确保、解析绑定、执行、输出哪个阶段。 |
| 方法_最近错误码 | I64 | 记录最近一次本能函数或执行桥错误码。 |
| 方法_最近质量 | I64 | 记录最近一次质量评分。 |
| 方法_最近成本 | I64 | 记录最近一次成本评分或耗时换算。 |
| 方法_最近开始时间 | 时间戳 | 记录最近一次运行开始。 |
| 方法_最近结束时间 | 时间戳 | 记录最近一次运行结束。 |
| 方法_最近耗时 | I64 | 记录最近一次运行耗时。 |
| 方法_最近输入场景 | 指针句柄 | 指向本轮输入或条件场景。 |
| 方法_最近结果场景 | 指针句柄 | 指向本轮功能结果账本。 |
| 方法_最近动作动态 | 指针句柄 | 指向本轮真实方法调用生成的动作动态。 |
| 方法_最近目标存在 | 指针句柄 | 指向本轮实际被作用的存在。 |
| 方法_最近宿主任务 | 指针句柄 | 指向触发本轮运行的任务。 |

方法虚拟存在不得写入旧式自然语言摘要作为结构依据。

可读文本只能作为调试显示。权威判断必须来自状态、特征、场景、指针和枚举字段。

## 4. 条件场景

条件场景定义方法需要哪些参数，不负责保存任务上下文镜像。

方法真正执行时才使用条件场景实例化输入参数。

参数来源按任务层级区分：

| 层级 | 被管理对象 | 参数来源 |
| --- | --- | --- |
| 管理自身的任务管理任务 | 任务管理任务自身 | 任务管理任务的虚拟存在。 |
| 管理学习的任务管理任务 | 学习对象方法 | 被学习方法的虚拟存在。 |
| 管理普通任务的任务管理任务 | 被管理任务 | 被管理任务的虚拟存在。 |
| 普通任务 | 需求对应的现实目标 | 任务场景或任务绑定的现实场景。 |

条件场景中的每个条件应尽量表达以下信息：

| 信息 | 用途 |
| --- | --- |
| 所需特征类型 | 方法需要从参数存在或场景中读取哪类特征。 |
| 当前状态约束 | 方法要求该特征当前处于什么状态。 |
| 方向约束 | 方法适用于状态上升、下降、保持或其他方向。 |
| 参数主体类型 | 说明该条件绑定任务虚拟存在、方法虚拟存在还是现实场景。 |
| 缺失时处理 | 说明缺少该条件时应生成什么需求。 |

执行方法无法根据条件节点生成条件时，产生条件缺口。

条件缺口的规范状态需求是：

```text
被需求存在 = 当前方法的虚拟存在
需求特征 = 方法条件节点数量
当前状态 = 当前条件节点数量
目标状态 = 1 或更高
需求去向 = 自我线程
```

## 5. 结果场景

结果场景记录函数功能结果，是本能函数功能事实的权威账本。

结果场景至少记录以下内容：

| 内容 | 用途 |
| --- | --- |
| 执行动作 ID | 表示本轮运行的是哪个本能函数或动作。 |
| 输入条件快照 | 记录本轮实际绑定到的条件。 |
| 输出状态变化 | 记录被改变的状态和特征。 |
| 缺口事实 | 记录候选方法补齐、缺条件、缺结果、缺权限等功能缺口。 |
| 需求当前状态 | 缺口对应的当前状态。 |
| 需求目标状态 | 缺口对应的目标状态。 |
| 结果方向 | 当前状态到目标状态形成的方向。 |
| 结果质量和成本 | 记录功能结果质量和成本。 |
| 相关方法虚拟存在 | 指向本轮方法虚拟存在。 |
| 相关宿主任务 | 指向本轮任务。 |

功能执行结果，例如“状态已改变”“产生新条件需求”“产生候选方法补齐需求”，必须写入结果场景。

运行情况，例如“函数成功”“函数失败”“错误码”“耗时”“成本”“质量”，必须写入方法虚拟存在。

返回结构只携带结果场景、方法虚拟存在、成功码、需求当前状态、需求目标状态和方向等桥接字段。

## 6. 动作动态

动作动态记录“自我主动调用方法函数”这一整体变化。

动作动态不是普通状态转换动态的别名。

普通动态只记录某个特征从旧状态变到新状态。它可以说明状态转换顺序，但不能说明导致转换的主动动作。

动作动态必须携带动作来源，至少能回答：

| 问题 | 字段或结构 |
| --- | --- |
| 谁在动作 | 动态主体，通常为自我存在。 |
| 做了什么 | 来源动作、来源动作名或动作句柄对应方法。 |
| 何时发生 | 动态时间段，包含开始和结束时间。 |
| 调用了哪个方法 | 来源方法或方法首节点引用。 |
| 输入来自哪里 | 来源输入场景。 |
| 输出写到哪里 | 来源输出场景。 |
| 是否成功 | 来源执行成功。 |
| 为什么失败 | 来源错误码。 |

动作动态的落点固定为自我所在场景。

```text
动作动态场景 = 世界树.自我所在场景
动态主体 = 自我存在
动态特征 = 方法调用动作动态
来源类型 = 方法调用
来源动作 = 当前被真实调用的方法首节点
来源动作名 = 当前方法动作名；缺失时使用“方法调用”
来源动作相位 = 完成 或 失败
来源输入场景 = 本轮条件场景或输入场景
来源输出场景 = 本轮结果场景或输出场景
```

动作动态由调用方法的函数主动生成。

工作线程、任务管理任务、自我线程不能在没有真实调用方法函数时伪造动作动态。

每次真正调用方法函数都必须生成一条动作动态。即使方法没有改变任何目标特征，也要保留本轮动作事实动态。

真实调用的边界是：方法已经通过可用性检查、条件场景已经能够实例化、执行桥已经进入动作函数或本能函数调用。

以下情况不得生成真实动作动态：

| 情况 | 原因 |
| --- | --- |
| 方法不可用 | 尚未真实调用方法函数。 |
| 缺条件节点 | 尚未实例化执行条件。 |
| 只做筹办查找 | 筹办不是被筹办方法的真实执行。 |
| 只处理线程消息 | 消息落账是治理侧摘要，不是下层真实方法调用。 |

状态信息、普通动态和动作动态的关系如下：

| 信息 | 记录内容 | 落点 |
| --- | --- | --- |
| 状态信息 | 某存在某特征在某时间的值。 | 该特征所属存在的内部世界。 |
| 普通动态 | 同一主体同一特征的旧状态到新状态转换。 | 该特征所属存在的内部世界。 |
| 状态顺序因果 | 旧状态之后出现新状态的转换顺序。 | 该特征所属存在的内部世界。 |
| 动作动态 | 自我真实调用某方法函数的动作事实。 | 自我所在场景。 |
| 动作因果 | 动作动态与后续状态变化之间的因果证据。 | 由后续因果生成逻辑桥接动作动态和结果状态。 |

动作动态可以作为高可信内部因果证据。

动作动态本身不等于方法成熟。它只能证明“自我确实调用过这个方法”。方法是否稳定、条件是否充分、结果是否可复现，仍由学习和稳定复现逻辑判断。

## 7. 因果信息

因果信息由动态信息生成，但不同动态生成的因果可信度不同。

当前规范把因果信息先分为两大类：

| 因果类型 | 来源动态 | 主要用途 |
| --- | --- | --- |
| 动作因果 | 自我真实调用方法形成的动作动态。 | 完善自我的方法，判断某方法在某条件下是否能导致目标结果。 |
| 状态因果 | 状态变化形成的普通动态。 | 发现状态前置链、特征联动关系、任务筹办路径和世界规律。 |

动作因果回答“自我执行了什么方法，导致了什么变化”。

状态因果回答“哪些状态变化稳定地先于、伴随或约束另一些状态变化”。状态因果本身不证明主动动作责任。

### 7.1 动作因果

动作因果的最小形式是：

```text
条件状态集合 + 动作动态 -> 结果状态或结果动态
```

动作因果必须有真实动作动态作为动作侧证据。

动作因果由自我执行方法时产生，主要用于：

| 用途 | 说明 |
| --- | --- |
| 方法可用性判断 | 若某方法多次在相同条件下产生目标结果，可提升方法可用性证据。 |
| 方法条件补全 | 从成功动作因果中提取稳定前置条件，补成方法条件节点。 |
| 方法结果补全 | 从成功动作因果中提取稳定结果状态，补成方法结果节点。 |
| 方法失败学习 | 从失败动作因果中提取缺条件、缺权限、缺资源等失败前置。 |
| 组合方法学习 | 多条动作因果按结果到条件串联后，可形成组合方法候选。 |

动作因果不得在以下情况下生成：

| 情况 | 原因 |
| --- | --- |
| 只有状态变化，没有动作动态 | 无法证明自我执行了动作。 |
| 只有消息落账，没有真实方法调用 | 治理消息不是方法动作。 |
| 方法不可用或缺条件导致未执行 | 尚无动作事实。 |

### 7.2 状态因果

状态因果由普通动态生成。

普通动态至少表达：

```text
某主体 / 某场景 / 某特征: 状态A -> 状态B
```

状态因果的最小生成单位不是“文字摘要”，而是状态动态中的主体、特征类型、状态值、时间段和场景。

状态因果至少包含以下基础类型：

| 基础因果 | 生成来源 | 用途 |
| --- | --- | --- |
| 同特征顺序因果 | 同一主体、同一特征，状态 A 稳定先于状态 B。 | 记录状态演化顺序。 |
| 同特征前置因果 | 同一主体、同一特征，需要 B 时 A 总是先出现。 | 生成需求前置链；要 B 先要 A。 |
| 同特征阶段因果 | 同一主体、同一特征，A -> B -> C 稳定出现。 | 形成任务状态、方法状态、学习状态的阶段链。 |
| 同特征循环因果 | 同一主体、同一特征，A -> B -> A 或多状态循环稳定出现。 | 识别周期、恢复、回退和可逆状态。 |
| 跨特征联动因果 | 同一主体或同一场景，特征 C 的变化稳定先于特征 D 的变化。 | 发现“调 C 可以影响 D”的间接筹办路径。 |
| 跨主体关系因果 | 同一场景内，主体 X 的特征变化稳定先于主体 Y 的特征变化。 | 发现对象间影响关系。 |
| 场景约束因果 | 场景状态变化稳定改变某主体状态变化的可发生性。 | 识别环境条件、资源条件和安全服务边界。 |
| 动态阶段因果 | 某动态的开始、过程、完成稳定构成阶段顺序。 | 识别动作或过程的内部阶段。 |
| 聚合动态因果 | 多个低层动态稳定组成一个高层动态。 | 形成抽象动作、复合事件和组合方法候选。 |
| 阻断因果候选 | 某状态出现后，原本常见的后续状态不再出现。 | 发现禁止项、风险约束和失败原因。 |

其中 `同特征前置因果` 和 `跨特征联动因果` 是当前任务筹办最优先需要的两类状态因果。

### 7.3 状态前置链

状态前置链用于从目标状态倒推前置状态。

示例：

```text
任务状态: 未筹办 -> 已筹办 -> 可执行 -> 已完成
```

当需求目标是 `任务状态=已完成` 时，筹办可以生成前置需求：

```text
需要 已完成
-> 先需要 可执行
-> 先需要 已筹办
```

该链只能作为筹办依据，不得直接宣称需求已经满足。

状态前置链的生成条件：

| 条件 | 说明 |
| --- | --- |
| 同主体或同一可比主体类型 | 避免把不同对象的状态乱连。 |
| 同特征类型 | 前置链第一阶段只处理同一特征。 |
| 时间顺序稳定 | A 的时间必须稳定早于 B。 |
| 反例低于阈值 | B 经常不需要 A 时，不能升格为必要前置。 |
| 可重复出现 | 单次出现只能是候选。 |

### 7.4 跨特征联动链

跨特征联动链用于发现不同特征类型之间的关系。

示例：

```text
方法动作骨架状态: 0 -> 1
稳定先于
方法是否可用: 0 -> 1
```

当需求目标是 `方法是否可用=1` 时，筹办应先按结果能力查找能推动方法可用性的本能方法。若目标候选方法结构不完整，则由执行函数报告实际缺失，再由自我线程创建候选方法存在、动作骨架、条件结果配对、运行账完整性等原子子需求。

跨特征联动链的生成条件：

| 条件 | 说明 |
| --- | --- |
| 同主体或同一场景 | 优先从同一方法虚拟存在、同一任务虚拟存在、同一现实对象内部发现关系。 |
| C 的变化早于 D 的变化 | 必须有明确时间顺序。 |
| 时间窗口合理 | D 不能离 C 太远，否则只保留弱候选。 |
| D 的变化具有稳定复现 | 偶然共现不能升格。 |
| 存在反例统计 | C 出现但 D 不出现时要降低置信度。 |

跨特征联动因果可以服务间接方法查找。

如果没有直接改变目标特征 D 的方法，但存在稳定联动 `C -> D`，筹办可以先找能改变 C 的方法。

### 7.5 生成与升格规则

由动态生成因果时，必须先生成候选，再逐步升格。

```text
动态样本
-> 候选因果
-> 临时因果
-> 待验证稳定因果
-> 稳定因果
-> 抽象因果模板
```

当前阶段优先实现到候选因果和临时因果即可。

升格必须依赖：

| 证据 | 说明 |
| --- | --- |
| 出现次数 | 同类动态重复出现。 |
| 反例次数 | 前因出现但后果不出现，或后果出现但前因缺失。 |
| 时间窗口 | 前后变化之间的时间间隔是否稳定。 |
| 主体一致性 | 是否同主体、同场景或同类型主体。 |
| 特征一致性 | 是否同特征或稳定跨特征映射。 |
| 动作边界 | 动作因果必须有动作动态边界。 |

状态因果不得直接提升方法成熟度。

状态因果只能帮助筹办拆需求、发现前置条件、发现间接路径和组织学习型子需求。

动作因果才可以进入方法可用性、方法条件、方法结果和方法成熟度的学习链。

### 7.6 与任务筹办的关系

任务筹办查找方法的顺序应逐步扩展为：

| 顺序 | 查找依据 |
| --- | --- |
| 1 | 直接能产生目标特征类型结果的方法。 |
| 2 | 方法结果节点中已有同特征目标方向的方法。 |
| 3 | 状态前置链中能先达成前置状态的方法。 |
| 4 | 跨特征联动链中能间接影响目标特征的方法。 |
| 5 | 动作因果链中可组合到目标结果的方法组合候选。 |
| 6 | 仍无路径时，创建或复用候选方法节点，并展开候选方法补齐子需求。 |

因果只负责提供候选路径。

最终是否执行，仍必须经过需求方向、方法可用性、条件满足、根层安全服务边界和当前任务状态判断。

### 7.7 使用位置与消费方式

因果信息不是独立运行的调度器。

因果信息的使用位置固定为：筹办、执行后结算、学习型子需求、方法维护、自我线程治理和展示诊断。

| 使用位置 | 使用动作因果 | 使用状态因果 | 输出结果 |
| --- | --- | --- | --- |
| 任务筹办 | 查找能直接或组合达成目标结果的方法。 | 查找前置状态链和跨特征间接路径。 | 候选方法、前置需求、间接需求、候选方法补齐子需求。 |
| 方法执行前 | 检查当前条件是否接近历史成功动作因果。 | 检查目标状态是否存在必要前置未满足。 | 允许执行、先补条件、先补前置需求。 |
| 方法执行后 | 把本轮动作动态和结果状态生成动作因果候选。 | 把本轮状态动态并入状态顺序、前置和联动候选。 | 候选因果、方法学习证据、状态规律证据。 |
| 学习型子需求 | 用成功/失败动作因果补方法条件、方法结果和可用性。 | 用状态因果发现应学习哪个前置或联动方法。 | 方法条件节点、方法结果节点、方法可用性相关状态。 |
| 方法维护 | 用动作因果统计方法稳定性、失败条件和反例。 | 不直接提升方法成熟度，只提供外部规律提示。 | 方法虚拟存在运行特征、成熟度候选。 |
| 自我线程治理 | 接收因果候选，按需求生成子需求或任务壳。 | 把前置链和联动链转成需求树上的子需求。 | 需求树更新、任务创建、等待任务唤醒。 |
| 控制面板 | 展示某任务、需求、方法为什么被选择或阻塞。 | 展示状态前置链、跨特征联动链和反例。 | 可解释诊断，不参与权威判断。 |

任务筹办是状态因果的第一消费方。

筹办面对一个目标需求时，按以下顺序使用因果：

```text
目标需求状态
-> 直接找目标特征方法
-> 查状态前置链，生成前置子需求
-> 查跨特征联动链，生成间接子需求
-> 查动作因果链，生成组合方法候选
-> 仍无路径，创建或复用候选方法节点，并生成候选方法补齐子需求
```

方法执行后结算是动作因果的第一生成方，也是状态因果的重要生成方。

执行后必须先把结果写入状态和动态，再由因果生成器消费这些动态：

```text
真实方法调用
-> 动作动态
-> 结果状态变化
-> 普通状态动态
-> 动作因果候选
-> 状态因果候选
```

学习型子需求是动作因果的第一学习消费方。

学习型子需求不得只因状态因果稳定就直接把方法标为可用。

学习型子需求可以使用状态因果来决定“应该补哪个前置状态或联动特征”，但必须使用动作因果或实际执行复现来证明“哪个方法能做到”。

自我线程是因果转需求的权威入口。

工作线程和方法执行体不能因为发现因果路径就直接创建任务。它们只能返回因果候选、缺口事实或需求状态，由自我线程把它们转成需求树更新和任务创建。

### 7.8 两类因果的边界

动作因果偏向“方法学习”。

状态因果偏向“需求拆解和世界规律”。

| 问题 | 优先使用 |
| --- | --- |
| 这个方法是否真的能改变目标状态？ | 动作因果。 |
| 这个方法缺哪些稳定条件？ | 动作因果为主，状态因果辅助。 |
| 要达到状态 B 前是否必须先有状态 A？ | 状态因果。 |
| 改变特征 C 是否会带动特征 D？ | 状态因果。 |
| 没有直接方法时能不能走间接路径？ | 状态因果先给路径，动作因果再证明路径上的方法。 |
| 方法是否可以升格为可用？ | 动作因果和执行复现，状态因果不能单独升格。 |
| 当前任务为什么暂停？ | 状态因果解释前置缺口，动作因果解释方法失败或缺条件。 |

如果两类因果冲突，按以下规则处理：

| 冲突 | 处理 |
| --- | --- |
| 状态因果认为 A 必要，但动作因果显示方法可绕过 A 达成 B。 | 降低 A 的必要前置置信度，保留为常见前置。 |
| 动作因果显示方法成功，但状态因果显示后续经常引发负面状态。 | 方法可用性不直接否定，但执行前门控必须看到风险链。 |
| 状态因果显示 C 会带动 D，但动作因果复现失败。 | C -> D 保留为世界候选规律，不作为当前方法证据。 |
| 动作因果样本少，状态因果样本多。 | 可用于筹办候选，不可用于方法升格。 |

### 7.9 存储形态

动作因果和状态因果都存为因果节点。

短期不新增两套完全不同的数据结构，而是复用 `因果主信息类`，用字段约定区分来源、动作侧、状态迁移侧和证据样本。

当前 `因果主信息类` 的核心字段可对应为：

| 字段 | 动作因果用法 | 状态因果用法 |
| --- | --- | --- |
| 形态 | 候选或临时阶段可先用实例；多次归并后升为抽象。 | 同左。 |
| 适用锚点类型 | 自我、方法虚拟存在、任务虚拟存在、现实对象或场景。 | 同主体、同场景、跨主体关系所归属的锚点。 |
| 适用层级 | 动作动态或结果状态所在动态层级。 | 状态动态层级或聚合动态层级。 |
| 条件模板 | 动作执行前稳定条件特征。 | 前置状态、前置特征或场景条件。 |
| 动作模板 | 指向方法首节点、动作语义节点或动作动态语义。 | 空或指向“状态转移/联动/阻断”等状态因果语义节点。 |
| 动作名称 | 方法动作名。 | `状态前置`、`跨特征联动`、`场景约束` 等状态因果类型名。 |
| 动作语义键 | 方法调用语义键或动作句柄键。 | 状态因果类型键，例如 `state.precondition`、`state.cross_feature`。 |
| 状态迁移模板 | 本方法导致的结果状态迁移。 | A -> B、C变化 -> D变化、场景变化 -> 主体变化等迁移模板。 |
| 主状态迁移 | 主要结果状态迁移。 | 主要后继状态或主要被影响特征。 |
| 来源类型 | 方法调用、观察生成、学习生成等。 | 通常为观察生成或状态动态生成。 |
| 置信度 | 动作因果稳定度。 | 状态规律稳定度。 |
| 已验证 | 需要真实复现后才能为真。 | 只能表示状态规律已稳定，不表示方法已验证。 |
| 证据动态样本 | 真实动作动态及其结果动态样本。 | 普通状态动态、聚合动态、阻断样本动态。 |
| 成立次数 | 动作后目标结果出现次数。 | 前置或联动关系成立次数。 |
| 失败次数 | 动作后目标结果未出现或反向结果次数。 | 前置缺失仍出现后果、前因出现但后果未出现等反例次数。 |
| 最近命中时间 | 最近一次动作因果成立时间。 | 最近一次状态规律命中时间。 |
| 最近失败时间 | 最近一次动作因果反例时间。 | 最近一次状态规律反例时间。 |

动作因果的推荐存储形态：

```text
因果节点
  形态 = 实例 或 抽象
  来源类型 = 方法调用 / 学习生成
  动作模板 = 方法首节点 或 动作语义节点
  动作名称 = 方法动作名
  动作语义键 = 方法动作键
  条件模板 = 方法执行前条件特征集合
  状态迁移模板 = 动作后产生的结果状态迁移集合
  主状态迁移 = 本轮主结果迁移
  证据动态样本 = 动作动态 + 相关结果动态
  成立次数 / 失败次数 / 置信度 / 已验证
```

状态因果的推荐存储形态：

```text
因果节点
  形态 = 实例 或 抽象
  来源类型 = 观察生成 / 状态动态生成
  动作模板 = 空，或状态因果语义节点
  动作名称 = 状态前置 / 跨特征联动 / 场景约束 / 阻断候选
  动作语义键 = state.precondition / state.cross_feature / state.scene_gate / state.block
  条件模板 = 前置状态或前置特征集合
  状态迁移模板 = 后继状态迁移或被影响特征迁移集合
  主状态迁移 = 主要后继状态迁移
  证据动态样本 = 普通状态动态或聚合动态
  成立次数 / 失败次数 / 置信度
```

状态因果虽然复用 `动作模板` 字段，但不得把该字段解释成真实动作。

当 `动作语义键` 以 `state.` 开头时，`动作模板 / 动作名称` 表示状态因果语义，不表示自我做了动作。

### 7.10 存储位置与索引

因果节点应挂在最能解释该因果的锚点下。

| 因果类型 | 推荐挂载位置 | 索引位置 |
| --- | --- | --- |
| 自我方法调用动作因果 | 自我所在场景或自我存在内部世界中的因果区。 | 场景因果索引、方法虚拟存在因果索引。 |
| 方法学习动作因果 | 被学习方法的虚拟存在内部世界。 | 方法虚拟存在、学习型子需求、相关方法节点。 |
| 普通任务动作因果 | 被管理任务虚拟存在或任务场景。 | 任务虚拟存在、来源需求、结果场景。 |
| 同特征状态前置因果 | 该特征所属存在的内部世界。 | 特征节点、状态节点、存在内部世界。 |
| 跨特征联动因果 | 同一主体内部世界；跨主体时挂场景。 | 两个特征节点、主体存在、场景。 |
| 场景约束因果 | 场景节点。 | 场景因果索引、受影响主体索引。 |
| 聚合动态因果 | 高层动态所属场景。 | 高层动态、低层动态样本、场景。 |
| 阻断因果候选 | 被阻断结果所属主体或场景。 | 目标特征、阻断状态、场景。 |

存储原则：

1. 因果节点本体只存结构引用和统计，不复制状态、动态、方法、任务的大对象。
2. 具体证据必须通过 `证据动态样本` 回到动态节点，再从动态节点回到状态节点。
3. 可查询索引可以出现在场景、存在、特征、方法虚拟存在或任务虚拟存在中，但索引只保存引用。
4. 同一因果重复出现时应合并统计，不应无限新建同义因果节点。
5. 若状态因果和动作因果共享同一个结果状态迁移，也应存为两条不同因果，因为它们回答的问题不同。

去重键建议：

| 因果类型 | 去重键 |
| --- | --- |
| 动作因果 | 锚点类型 + 动作语义键 + 条件模板结构标识 + 主状态迁移结构标识 + 层级。 |
| 同特征状态前置因果 | 主体类型或主体主键 + 特征类型 + 前置状态结构标识 + 后继状态结构标识 + 层级。 |
| 跨特征联动因果 | 主体类型或场景主键 + 前置特征类型 + 后继特征类型 + 前置迁移结构标识 + 后继迁移结构标识。 |
| 场景约束因果 | 场景类型或场景主键 + 场景状态结构标识 + 受影响主体类型 + 受影响特征类型。 |
| 阻断因果候选 | 阻断状态结构标识 + 原预期后继状态结构标识 + 时间窗口标识。 |

存储阶段建议：

```text
实例样本：保留具体证据动态样本，方便追溯。
临时因果：合并同类样本，累计成立/失败次数。
稳定因果：保留代表样本和统计，不再保存无限样本。
抽象模板：删除单次主体身份，只保留可比较条件、动作或状态迁移模式。
```

## 8. 方法查找与筹办

筹办的核心职责是根据需求找方法。

筹办输入是任务，任务的目标来自需求。

筹办查找方法时，必须先把任务目标还原为 `需求结果` 和上下文条件，再调用或等价执行 `查找方法` 的匹配逻辑。

`查找方法` 不按名称、动作句柄、本能 ID 或固定编号查找方法，不直接执行方法，不补全执行参数。它只回答一个问题：

```text
有没有一个方法的同一个结果包，能够覆盖需求要求的结果变化集合？
```

正式定义固定为：

```text
需求结果集 = 若干需求结果项
方法能力签名 = 条件面 -> 结果包
匹配核心 = 需求结果集 ⊆ 方法结果包

匹配成功：
    找到方法

匹配失败：
    找不到
```

方法不是因为“它是谁”被找到，而是因为“它能把需求关心的特征推向目标方向”被找到。

结果匹配依据只能是：

| 依据 | 说明 |
| --- | --- |
| 需求结果项集合 | 需求要改变的特征、作用对象、绑定名、方向和值域集合。 |
| 方法结果包 | 某个能力签名声明的一包结果变化。 |
| 集合覆盖 | 每个必需需求结果项都必须被同一个结果包中的结果项覆盖。 |
| 当前上下文条件包 | 只用于过滤方法条件是否满足，不替代结果能力匹配。 |
| 查找限制包 | 限制最高允许目标特征、宿主类型、抽象层数、值类型和值域。 |

结果项覆盖需求项必须至少满足：

```text
特征类型一致或在允许的抽象关系内覆盖。
作用对象一致或可通过绑定名绑定。
方向一致或值域覆盖。
结果项没有被标记为风险结果。
```

`特征类型` 的值域是抽象特征树。任意抽象特征或实例特征都可以作为目标；实例特征参与匹配时先归一到它引用的抽象特征。方法查找不得把特征类型降级为方法名、动作名或语素文本。

不得使用：

```text
方法名称
动作名称
显示文本
语素文本
函数名称
动作句柄
本能 ID
固定编号
```

名称只能用于显示、日志和调试。

找不到时的解释也固定：

```text
查找方法失败
  =
当前方法树中没有任何方法声明自己能造成这个需求所要求的结果变化。
```

这只能说明闭环未建立、缺少能够满足该结果需求的本能方法，或已有本能方法缺少结果声明。查找失败不得改走“直接取某个本能方法”的绕路。

标准输入结构：

```text
输入参数场景
  输入条件包
    需求条件包
    查找限制包
    候选范围包，可选
```

需求条件包示例：

```text
需求条件包
  目标宿主场景 = 自我现实场景
  目标抽象特征 = 抽象特征_自我安全值
  目标值类型 = 类型_I64
  执行差值区间 = [1,I64_MAX]
```

这里的“增加”必须还原为：

```text
执行后值 - 执行前值 > 0
```

即：

```text
执行差值 ∈ [1,I64_MAX]
```

`不等于零` 不是单一区间，应表达为组合抽象特征 `任一命中(小于零, 大于零)`，或在匹配时展开为两个区间。

每个候选方法应在方法首节点声明 `能力签名集`。候选是否命中的核心判断为：

```text
存在某个能力签名：
    需求结果集 ⊆ 能力签名.结果包
```

候选条件是否可用再由能力签名的条件面与当前上下文条件包判断：

```text
存在某个条件组：
    条件组内全部必需条件项被当前上下文条件包满足
```

其中包含关系只用于过滤已按结果命中的候选方法，由以下规则共同决定：

1. 条件项特征类型相同，或需求特征类型沿抽象父链能到达能力特征类型。
2. 需求 I64 点值视作 `[v,v]`，必须被能力区间包含。
3. 需求 I64 区间必须被能力区间包含。
4. 抽象特征引用、存在引用或场景引用应先转成抽象类型口径，再在限制包内判断父链覆盖。
5. 任一必需条件项无法被当前上下文满足，则该候选不可执行，但这不改变“结果包是否覆盖需求”的事实。

如果需求有多个必需结果项，单次 `查找方法` 只允许返回“同一个方法的同一个结果包覆盖全部必需结果”的候选。不得在查找阶段偷偷把多个方法拼接起来满足需求；多方法组合属于任务筹办或方法序列构造职责。

当找到多个候选方法时，先按可执行性过滤，再按结果能力和事实样本排序。

方法选择排序应考虑以下因素：

| 因素 | 优先规则 |
| --- | --- |
| 当前条件 | 必须满足；不满足只记录条件检查，不进入条件结果对事实账。 |
| 参数配对 | 必须成功；配对失败只记录参数配对记录，不进入条件结果对事实账。 |
| 根写入原语 | 普通方法查找中不作为优先候选。 |
| 方法事实样本数 | 更多者优先；来自方法树条件结果对归并样本。 |
| 同条件结果复现证据 | 同条件下结果符合次数更多、不符合次数更少者优先；这是结果稳定性证据，不是任务成功率。 |
| 总匹配距离 | 更小者优先。 |
| 类型抽象距离 | 更小者优先。 |
| 值域抽象距离 | 更小者优先。 |
| 能力包宽度 | 更窄者优先。 |
| 平均运行时长 | 更短者优先。 |
| 最近执行情况 | 最近正常者优先。 |

方法结果符合/不符合统计的正式口径：

```text
同一条件下，动作执行成功后，实际结果是否复现已有等价结果；
或者普通任务完成后，实际状态是否满足来源需求目标状态。
```

它不是动作函数跑完率，也不是学习动作自身的成功率。学习动作改变的是方法树；后续是否可用、能不能满足需求，由自我线程按需求和方法树事实重新筹办判断。

条件结果对归并中，只有走到以下阶段，才允许更新同条件结果符合/不符合账：

```text
1. 方法条件满足
2. 参数配对成功
3. 方法动作函数执行成功
4. 产生实际结果状态
5. 存在等价条件节点
6. 特征/状态比较服务能比较实际结果与该条件下已有结果
```

同条件下复现已有等价结果时写入 `结果符合次数 +1`；同条件下出现新的非等价结果分支时写入 `结果不符合次数 +1`。不得再把旧 `成功次数 / 失败次数` 作为方法结果账字段使用。

以下情况不得进入该账本：

```text
方法未查到
方法查到但条件不满足
参数配对失败
任务没有实际执行方法
动作函数执行失败
执行被中断或超时
没有产生可比较结果
结果状态缺失
```

这些前置问题只能进入方法查找诊断、条件检查记录、参数配对记录、方法运行账或任务执行记录。方法条件信息节点虚拟存在只回答一个问题：

```text
在这组条件下，动作已经成功执行后，结果是否复现或分裂。
```

`查找方法` 的输出不是可执行调用帧，而是 `方法配对结果存在`。

方法配对结果至少保存：

| 字段 | 说明 |
| --- | --- |
| 配对状态 | 配对成功、配对失败、超出限制或输入绑定失败。 |
| 来源需求条件包 | 原始需求条件包引用。 |
| 使用限制包 | 本次查找使用的限制包。 |
| 选中自我能力值 | 权威选择结果，使用 `枚举_本能方法ID` 的 I64 值表示“我会什么”。 |
| 缓存方法首节点 | 仅为运行期缓存；执行前必须校验其仍匹配 `选中自我能力值`。 |
| 缓存方法虚拟存在 | 仅为运行期缓存；不作为任务长期语义主键。 |
| 匹配方法能力包 | 命中的能力包。 |
| 原始需求绑定包 | 原始需求口径，供下一步执行参数绑定使用。 |
| 匹配路径记录 | 类型和值域抽象路径。 |
| 类型抽象距离 / 值域抽象距离 / 总匹配距离 | 选择候选的解释依据。 |
| 待配对形参表 | 下一步 `配对参数` 需要消费的形参表。 |

当没有任何方法满足 `目标特征类型 + 目标方向` 时，输出配对失败和结构化失败原因。不得因为目标像某个学习场景，就直接写入 `自我_练习`、`自我_尝试学习` 或 `自我_定向学习` 的能力值。

最终链路固定为：

```text
任务需求
  -> 查找方法
  -> 结果能力命中的方法配对结果存在
  -> 任务虚拟存在保存配对结果和执行入口缓存
  -> 运行期按配对结果校验并解析方法首节点
  -> 配对参数
  -> 可执行输入参数场景
  -> 执行选中方法
```

查找方法不得跳过 `配对参数` 直接执行。

任务状态类需求可以按状态出现顺序生成子需求。例如从未启动到完成，可以按筹办中、就绪、执行中、完成等顺序拆分。

筹办按结果能力未命中方法时，直接新建或复用候选方法节点，再围绕该候选方法展开原子补齐需求。

筹办发现候选方法缺动作骨架时，生成“方法动作骨架状态 -> 已形成”的子执行需求。

目标方法缺动作骨架时，规范需求是：

```text
被需求存在 = 目标方法的虚拟存在
需求特征 = 方法动作骨架状态
当前状态 = 0
目标状态 = 1
需求去向 = 需求树派生方法能力缺口子需求
```

筹办不得直接创建学习任务或普通任务；它只提出原子子需求，由需求树和任务壳承接。

筹办只返回缺口，缺口应落成当前需求的子需求；任务只作为该子需求的运行壳被创建或取得。

## 9. 方法执行

执行的核心职责是运行当前已选方法。

执行入口是 `执行任务`。

`执行任务` 的参数是任务，也就是某个需求节点的对应任务。

执行阶段不得重新解释方法名称。它必须读取任务虚拟存在或任务主信息中的 `选中自我能力值`，运行期再用该能力值解析方法首节点、方法虚拟存在和执行入口。

在配对参数或调用目标方法前，执行阶段必须先做方法结构自检。自检至少检查：

```text
方法虚拟存在
自我能力值
方法能力包
形参包
结果包
条件结果对
试运行限制包
可调用执行入口
```

如果目标方法只有方法头节点，或缺少可调用执行入口，则执行阶段必须写：

```text
任务虚拟存在
  当前阶段 = 阻塞
  缺口类型 = 执行缺失

方法虚拟存在 / 输出结果场景
  方法自检结果
  执行入口或动作骨架缺失
```

然后输出对应原子需求。该子需求的筹办阶段可以查到 `练习` 或 `尝试学习`，由它们补候选能力包、形参包、结果包、条件结果对和试运行限制包。

工作线程执行一轮时，应返回两类数据：

| 数据 | 含义 |
| --- | --- |
| 任务执行数据 | 本轮任务状态、记账结果、当前方法、结果场景、任务状态变化等。 |
| 方法实际数据 | 本轮方法返回的功能结果、缺口状态、动作动态、方法虚拟存在、成功码和解释。 |

执行前不应由工作线程自行拆四段式。

本能函数运行一次时，内部自动完成四段式：

| 阶段 | 职责 |
| --- | --- |
| 确保 | 确保方法首节点、方法虚拟存在、本轮运行虚拟存在、输入场景、结果场景和运行参数声明可用。 |
| 解析绑定 | 根据方法虚拟存在中的运行参数声明，从输入参数场景或任务场景中绑定实际值。 |
| 执行 | 执行方法动作，改变目标状态或形成缺口；无法继续执行时写入返回参数信息和返回场景信息。 |
| 输出 | 将功能事实写结果场景，将运行情况写本轮运行虚拟存在，返回本轮运行虚拟存在。 |

确保阶段不得因为方法头、方法虚拟存在或运行参数声明已经存在就直接返回成功。它只是把后续阶段所需结构补齐；本能函数仍必须继续经过解析绑定、执行和输出。

当解析绑定失败、执行条件不足或动作无法执行时，本能函数不应返回单个结果特征，也不应提前绕过输出阶段。它应在本轮运行虚拟存在中写入至少以下信息：

| 信息 | 用途 |
| --- | --- |
| 结果值或错误码 | 表示停在哪类失败或成功状态。 |
| 返回参数信息 | 表示哪些参数已绑定、哪些参数缺失或不满足。 |
| 返回场景信息 | 指向输出结果场景、缺口场景或临时补位场景。 |
| 方法虚拟存在入口 | 指回本轮运行所属方法。 |

执行桥在真正调用动作函数或本能函数时，必须生成动作动态。

动作动态生成不替代结果场景。结果场景仍记录函数功能事实，动作动态只记录“本轮确实发生了方法调用”。

执行发现条件节点缺失时，生成缺条件需求。

执行发现目标候选方法结构或可用性不足时，可以返回候选方法补齐事实；正式需求仍由自我线程展开为原子子需求。

执行不得直接创建任务。

执行不得恢复旧上下文镜像逻辑。

执行不得依赖旧摘要判断。

## 10. 本能函数

本能函数是方法函数的最小运行单元。

所有本能函数必须遵守内部四段式执行壳。

外部执行桥只负责调用本能函数并接收运行虚拟存在，不再理解四段式细节。

本能函数必须满足以下返回规范：

| 返回信息 | 存储位置 |
| --- | --- |
| 函数功能结果 | 结果场景。 |
| 缺条件、候选方法补齐等功能缺口 | 结果场景。 |
| 改变了哪些特征类型和值 | 结果场景。 |
| 成功、失败、错误码 | 本轮运行虚拟存在，并同步到方法虚拟存在的最近运行特征。 |
| 成本、质量、耗时 | 本轮运行虚拟存在。 |
| 动作动态入口 | 本轮运行虚拟存在和方法虚拟存在。 |
| 结果场景入口 | 本轮运行虚拟存在和方法虚拟存在。 |
| 方法虚拟存在入口 | 本轮运行虚拟存在和上行消息。 |

本能函数返回值应是本轮运行虚拟存在。这个虚拟存在可看作任务虚拟存在的同类结构，用于承载本轮方法调用的运行账本；结果值只是其中一个特征，不再由单个结果特征承担全部返回语义。

本能函数目标函数形态固定为：

```cpp
using 本能动作函数 = std::function<存在节点类*(
    方法类::节点类* 方法首节点,
    场景节点类* 输入参数场景,
    场景节点类* 输出结果场景)>;
```

返回值为 `存在节点类*`，表示本轮运行虚拟存在。旧式单特征结果不再作为本能函数标准返回值。

每个本能函数执行后，都应该在方法树中生成或融合一组条件与结果记录。

条件按条件方向融合，结果按结果方向融合。

如果缺条件，本能函数或执行桥应在结果场景记录“当前方法需要条件节点”。

如果需要补齐候选方法，本能函数或筹办应在结果场景记录“目标候选方法需要动作骨架、条件结果配对、运行账完整性或可用性推进”等原子目标。

## 11. 过渡兼容：旧三层任务共用同一套方法逻辑

本节描述旧实现的兼容壳，不再是长期结构权威。新逻辑中，任务管理、学习、普通执行都收敛为需求树上的需求节点及其对应任务；差异只体现在需求条件包、缺口类型和回传规则。

当前任务执行逻辑统一覆盖三层任务。

| 层级 | 目标 | 方法作用对象 | 典型需求 |
| --- | --- | --- | --- |
| 任务管理任务管理自身 | 完善任务管理任务自身能力 | 任务管理任务虚拟存在 | 任务状态变为完成。 |
| 任务管理任务管理学习 | 完善被学习方法 | 方法虚拟存在 | 候选方法存在、动作骨架、条件结果配对、运行账完整性或方法可用性提升。 |
| 任务管理任务管理普通任务 | 完成普通任务需求 | 被管理任务虚拟存在或任务场景 | 需求状态被满足。 |

这三层不使用三套执行逻辑。

差异只在参数来源、需求目标和结果回写对象。

过渡期旧任务管理壳曾固定两个顶层步骤；当前新口径中，筹办和执行都已经迁出本能方法，成为任务系统普通函数：

| 阶段 | 承载方式 | 职责 |
| --- | --- | --- |
| 筹办 | 普通任务筹办函数 | 根据执行需求结果查找方法；未命中时创建或复用带目标结果声明的候选方法头。 |
| 执行 | 普通任务执行函数 | 读取任务头绑定的当前方法并调用；发现动作入口、参数或条件等实际缺失时输出对应原子需求，成功时写入实际结果。 |

反馈是任务执行闭环中的结果回流逻辑，不要求作为任务管理任务默认第三个顶层步骤。

## 12. 知识缺口需求触发

学习不再是独立任务类型的权威概念。学习表示需求树生长到知识缺口时，该缺口节点的对应任务会执行学习型或补齐型方法。

旧代码中的学习任务只作为兼容壳存在。

自检可以形成执行缺失或候选方法补齐事实。方法只有头节点时，不生成“缺方法需求”；筹办已经创建了目标方法头，后续缺什么就展开什么原子需求：

```text
方法头节点
  -> 方法自检结果
  -> 执行入口 / 动作骨架 / 条件结果对等实际缺失
  -> 原子需求：补齐该具体缺失
  -> 对应任务筹办
  -> 查找方法
  -> 按结果需求查找练习或尝试学习
```

自我内部方法补齐闭环不触发定向学习。内部因果信息已经有动作主体、动作函数、动作前状态、动作后状态、结果特征类型和结果变化方向，因此尝试学习阶段已经具备形成方法候选所需的方向材料。

内部闭环固定为：

```text
方法补齐入口
  -> 查找能提升方法可用性的方法
  -> 练习
  -> 发现没有候选方法
  -> 查找能形成候选方法的方法
  -> 尝试学习
  -> 读取内部因果信息
  -> 形成候选方法
  -> 练习验证候选方法
  -> 候选方法变成可用方法
  -> 方法补齐入口满足
```

内部闭环分派口径：

| 缺口类型 | 对应学习方法 | 行为 |
| --- | --- | --- |
| 没有候选方法 | 尝试学习 | 读取内部因果样本，从动作函数、动作主体、动作前状态、动作后状态中抽取动作骨架、条件面和结果包，形成候选方法能力签名。 |
| 已有候选方法但可用性、稳定性或运行账不足 | 练习 | 在已有动作分支上生成或优化参数，验证条件结果对，发现同一动作的新用途，更新成功/失败统计和稳定性特征。 |
| 内部因果样本缺失或不可用 | 不命中定向学习 | 输出尝试学习未形成候选方法；原因只能是内部样本缺失、样本与缺口不匹配、结果特征类型不对应、条件不足、动作函数无法转成动作骨架或结果方向无法稳定归纳。 |

定向学习只属于外界信息处理链：

```text
外界信息
  -> 定向学习
  -> 观察需求 / 确认需求 / 信息需求
  -> 外界方向材料
```

当外界信息里动作主体未知、动作函数未知，或只有未归因现象时，才需要定向学习形成方向、观察目标或确认目标。

`练习` 的主目标是让已有动作分支的条件结果对逐步稳定：

```text
方法虚拟存在
自我能力枚举值
方法能力包
形参包
结果包
已有条件结果对
试运行限制包
历史运行事实
稳定性统计
候选区间 / 稳定区间
```

实参生成不由单独模式开关驱动，而由任务传入参数完整度判定：

```text
目标特征类型 + 练习方向 / 执行差值 / 目标执行后值 齐备
  -> 指定练习

目标特征类型或方向约束缺失
  -> 随机练习
```

随机练习由 `练习` 自己生成未出现过的候选实参，并把候选实参表、参数来源、运行事实写入方法虚拟存在和已有条件结果对。

候选结构不得直接成为稳定事实。条件结果对必须经过运行事实验证后，才能从候选升格为稳定。

如果目标方法没有自我能力枚举值、外部实现句柄或动作序列句柄，就没有可调用执行入口。此时 `练习` 只能写执行入口或动作骨架缺失事实、叶子需求状态、失败原因和建议下一步能力，不能伪造试运行结果。

典型知识缺口需求是：

```text
被需求存在 = 具体方法的虚拟存在
需求特征 = 候选方法存在状态 / 方法动作骨架状态 / 方法条件结果配对状态 / 方法运行账完整性 / 方法是否可用
当前状态 = 0
目标状态 = 1
```

有了这些原子子需求后，调度逻辑再按结果能力查找能满足对应特征方向的方法，例如尝试学习或练习。

学习链可以通过缺口特征触发更细需求，例如：

```text
被需求存在 = 具体方法的虚拟存在
需求特征 = 方法结果节点数量
当前状态 = 0
目标状态 = 1
```

知识缺口需求的对应任务应按缺口类型补齐方法的条件节点、结果节点、动作骨架或结果方向证据。生成需求的权责属于自我线程、任务筹办或允许生成子需求的学习方法；`练习` 只负责输出缺口特征和完善已有动作分支。

知识缺口需求满足后，应更新对应的候选方法结构状态、运行账完整性或可用性状态，并向父需求回传，使父需求的对应任务重新筹办。

## 13. 消息与线程边界

自我线程是唯一能够正式创建需求节点对应任务的线程。

工作线程不能直接创建任务。

工作线程只执行当前任务方法，并返回任务执行数据和方法实际数据。

任务管理界面线程属于过渡兼容层，只能接收上行消息并展示或镜像需求/任务状态，不再作为长期权威调度者。

工作线程可以把消息交给自我线程，但持久收取和状态处理应由任务管理界面线程承担，因为工作线程可能退出。

自我线程消费消息时可以沉淀治理侧动态摘要，但不得把治理侧摘要冒充真实动作动态。

真实动作动态只能由真实调用方法函数的执行体生成，并落在自我所在场景。

派生需求消息至少应携带：

| 字段 | 用途 |
| --- | --- |
| 任务主键 | 说明哪个任务产生缺口。 |
| 当前方法主键 | 能定位方法节点时填写。 |
| 方法虚拟存在主键 | 方法节点主键缺失时仍可定位学习对象。 |
| 派生需求类型 | 候选方法补齐、缺条件、任务步骤等。 |
| 需求特征类型 | 从需求当前状态或目标状态反推。 |
| 需求当前状态 | 缺口当前状态。 |
| 需求目标状态 | 缺口目标状态。 |
| 当前等待原因 | 调试文本，不作为权威结构依据。 |

当前日志的有效形态示例：

```text
任务管理派生需求 | 类型=候选方法补齐 | 方法虚拟存在=8d | 主观察=方法动作骨架状态 | 当前状态=KF | 目标状态=KG
```

未来缺条件日志应呈现类似形态：

```text
任务管理派生需求 | 类型=缺条件 | 方法虚拟存在=<方法虚拟存在主键> | 主观察=方法条件节点数量 | 当前状态=<0> | 目标状态=<1>
```

## 14. 废弃项

以下逻辑不得作为新实现依据：

| 废弃项 | 原因 |
| --- | --- |
| 旧上下文镜像 | 新执行参数由条件场景和目标存在绑定，不再复制上下文。 |
| 旧摘要字段 | 摘要不是结构合同，不能驱动判断。 |
| 工作线程直接建任务 | 任务只能由自我线程根据需求创建。 |
| 任务管理任务直接在当前主需求下长子任务 | 最新规范要求先提需求。 |
| 隐式四域方法树直调 | 旧四域函数应转成本能方法，由筹办组织。 |
| 结果节点作为任务必需节点 | 任务主信息已收束，结果事实进入结果场景。 |
| 步骤节点强制组织任务树 | 当前以需求树和任务列表组织任务，步骤只保留为方法执行过程中的可选结构。 |
| 在自我内部世界伪造真实动作动态 | 动作动态属于自我所在场景中的真实方法调用事实。 |

## 15. 验收标准

方法规范实现完成时，应满足以下检查项：

| 检查项 | 通过标准 |
| --- | --- |
| 候选方法补齐 | 日志能看到候选方法虚拟存在、主观察特征、当前状态和目标状态。 |
| 执行缺条件 | 日志能看到 `类型=缺条件`、`主观察=方法条件节点数量`、当前状态和目标状态。 |
| 方法虚拟存在 | 派生需求和结果状态消息能带回方法虚拟存在主键。 |
| 结果场景 | 本能函数功能结果能从结果场景读取。 |
| 动作动态 | 每次真实方法调用都能在自我所在场景找到一条方法调用动作动态。 |
| 动作动态边界 | 方法不可用、缺条件、消息落账等未真实调用方法的路径不会生成真实动作动态。 |
| 状态因果候选 | 同特征状态顺序和跨特征联动能先形成候选或临时因果，不直接冒充动作因果。 |
| 动作因果候选 | 真实动作动态能与结果状态形成候选动作因果，用于方法学习链。 |
| 运行账 | 方法运行成功、失败、阶段、耗时和错误码能从方法虚拟存在读取。 |
| 学习触发 | 候选方法存在、动作骨架、条件结果配对、运行账完整性或方法可用性需求能按结果能力命中学习/练习本能。 |
| 任务创建边界 | 新任务只由自我线程创建。 |
| 旧逻辑清理 | 旧上下文和旧摘要不参与新判断。 |
