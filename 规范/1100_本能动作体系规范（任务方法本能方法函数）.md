# 1100 本能动作体系规范（任务方法本能方法函数）

本文是“本能动作体系”的工程规范，用于避免循环依赖、避免各模块 I/O 不一致、避免外设控制权冲突，并支撑：
- Step1~Step4 四步本能方法函数（确保/解析绑定/执行/输出）
- 任务→方法→本能方法函数的分层
- 外设（真实世界）与自我组织（虚拟存在）双世界一致性

> 术语：
> - 世界树：真实世界映射（事实）
> - 方法虚拟存在：方法的组织信息（意图、经验、预算、锁、统计）
> - 输入场景：一次调用的“实参/上下文快照”
> - 条件模板：方法首节点主信息 `场景根节点`（形参约束树）


## 1. 模块分层与职责边界

### 1.1 本能动作类（仅通用函数）
**模块名建议**：`本能动作通用函数模块`
**只允许包含**：本能动作实现会复用、且不依赖具体外设/具体自我逻辑的通用工具，例如：
- Step2 场景模式匹配（形参约束 ↔ 实参绑定）
- 统一回执写入助手（写 success/诊断/耗时/关键指标）
- 统一“形参模板写入助手”（把所需参数类型写入方法首节点条件模板）
- 统一“统计更新助手”（写入方法虚拟存在的计数/率/最近值等）

**禁止包含**：任何具体动作（相机启动、识别、生成存在…）的业务实现。


### 1.2 本能动作管理类（注册与运行）
**模块名建议**：`本能动作管理模块`
**只负责**：
- 注册/注销/查询/枚举本能动作
- 运行本能动作（按四步或按统一入口）
- 提供“执行环境对象”（env）并保证不持锁执行

**禁止包含**：任何外设调用、世界树写入、相机识别逻辑。


### 1.3 本能方法实现模块（具体业务）
示例：
- `外设本能方法_相机模块`
- `自我本能方法_相机认知模块`

**负责**：每个具体本能方法的 Step1~Step4 业务实现。
**允许 import**：其确实需要的外设模块、相机工作流、世界树环境等。
**必须遵守**：统一 I/O 协议（第 2 节）、锁协议（第 4 节）、回执协议（第 3 节）。


## 2. 统一 I/O 协议（输入从哪来，输出写到哪）

### 2.1 允许的输入来源（只读）
每个本能方法在 Step2/Step3 允许读取的输入仅来自：
1) 输入场景（本次调用实参）
2) 方法首节点主信息 `场景根节点`（条件模板/形参约束树）
3) 方法虚拟存在（组织信息：目标/约束/预算/锁/历史 best/样本）
4) 世界树真实存在（事实：相机当前参数、状态、存在事实等）

> 约束：输入场景不用于存长期事实；长期事实写世界树，长期意图/经验写方法虚拟存在。


### 2.2 允许的输出落点（写入）
Step4 允许写入：
1) 输出场景（回执/诊断/关键摘要）
2) 方法虚拟存在（统计、best、样本、锁状态、预算消耗）
3) 世界树真实存在（事实更新：相机当前参数/存在状态等）


## 3. 统一回执规范（Step4 必写）

### 3.1 最小回执字段（必须）
每个本能方法 Step4 至少输出：
- `成功/失败`（bool）
- `错误码`（enum/int）
- `诊断字符串`（可累积）
- `耗时`（us 或 ms）
- `时间戳/帧序号`（如有）

并建议输出：
- 关键指标摘要（识别率、unknownRate、稳定度 score、FPS、深度有效比例等）
- 绑定摘要（Step2 绑定表是否歧义、总分）


### 3.2 回执写入位置
- 输出场景：用于调用方（任务/方法调度）即时消费
- 方法虚拟存在：用于长期学习（成功率、失败原因、best 更新、样本池）

### 3.3 显式本能动作的事件边界回执

当前规范新增硬约束：

1. 每个显式本能动作都应被视为一条事件。
2. 每条显式本能事件都必须具备 `开始` 与 `结束` 两个边界。
3. 本能动作产生的状态变化，必须落在这两个边界之间。
4. 若动作失败，也必须保留事件边界，只是结束相位改为 `失败`。

最小记录顺序固定为：

1. `动作开始运行`
2. `中间状态变化/结果写入`
3. `动作完成` 或 `动作失败`

不得反过来写成：

1. 先写结果状态
2. 再由结果状态名回退冒充动作名

例如：

- 正确：`定时衰减服务值开始运行 -> 服务值下降 -> 定时衰减服务值完成`
- 错误：`动作{服务值}`

### 3.4 动作名与结果状态的边界

必须固定：

1. `动作名` 表示“做了什么”
2. `结果状态` 表示“变成了什么”

因此下列字段只能视为结果状态，不得直接当作动作事件名：

- `任务_状态`
- `任务_绑定全局容器指针`
- `服务值`
- `安全值`

更准确的表达应为：

- `绑定任务全局容器 -> 任务_绑定全局容器指针变化`
- `执行步骤 -> 任务_状态变化`
- `定时衰减服务值 -> 服务值下降`

### 3.5 显式动作的主结果回带规则

当前规范继续固定：

1. 显式动作完成或失败时，仍必须继续记录结果状态节点。
2. 若该动作存在显式结束相位动态，则该结束相位动态应优先挂回本次动作的 `主结果状态`。
3. 因果提炼时，不应再主要依赖“动作后状态列表猜主结果”，而应优先读取结束相位动作动态中回带的主结果状态。

因此：

- `动作动态` 负责说明“这次动作结束后，哪一个状态是本次主结果”
- `状态列表` 负责说明“这次动作一共带来了哪些结果状态”

对 `定时衰减服务值` 这类由 `自我线程` 周期自动调用的 `内部先天动作`，当前正式最小记录单位应固定为：

1. `自我线程` 在本周期自动调用一次对应先天本能函数
2. 动作本体执行并得到本轮主结果
3. 生成一条 `本能动作动态`

并且必须固定：

- 每周期一次调用，对应一条动作动态
- 这条动作动态默认记为本轮动作事实的完成态记录
- 这条动作动态应回带本轮 `主结果状态`
- 若本轮没有引起数值变化，也仍应保留这条动作动态，只是结果状态可标为 `无变化`

另外，凡是需要为 `本能动作名 / 动作动态来源动作名 / 动作补全自然语言名` 新增或补全自然语言名称的地方，统一应通过 `语素集.添加词性词(...)` 获取词性词指针，不得另建独立自然语言缓冲区。详细统一规则见 [2000_自然语言词性词统一获取规则.md](D:\鱼巢\规范\2000_自然语言词性词统一获取规则.md)。


## 4. 外设控制权（锁）规范

### 4.1 锁放置位置
外设调参/模式切换等控制权锁，必须挂在**方法虚拟存在**下（因为调参意图属于方法）。

### 4.2 锁字段（最小）
- `持有者ID`
- `租约到期时间`
- `优先级`
- `抢占计数/原因（可选）`

### 4.3 锁规则（硬性）
- 无锁：Step3 禁止写外设参数（只能读状态/取帧）
- 高优先级可抢占：抢占必须写事件回执
- 任务只通过“目标/约束/预算/锁”影响自组织回路，不直接调用“设置参数”动作入口


## 5. Step1~Step4 四步规范（统一语义）

### Step1：确保
- 确保方法树中存在本动作的**方法首节点**
- 确保方法首节点的 `场景根节点`（条件模板）中写入本动作所需的**参数类型（形参类型）**
- 确保方法虚拟存在根存在，并准备好必要的组织特征（目标/约束/预算/锁/统计入口）
- 幂等：重复调用不得重复造节点/重复写同类型参数

### Step2：解析与绑定（模式匹配）
- 输入：输入场景（实参） + 条件模板（形参）
- 产出：绑定表（形参节点→实参节点）+ 匹配度总分 + 是否歧义 + 未满足约束列表
- 规则：
  - 默认 AND；OR 用“特征节点自身无值且有子节点”表示（子约束满足其一）
  - 值相等采用“算法相等”（二次特征比较/差值阈值），不是数学 ==
  - 同类型多候选：取匹配度最高者，并列按主键最小；top1-top2 小于阈值 => 标记歧义
  - 多余实参：忽略（不影响成功），但可用于后续细化

### Step3：执行
- 唯一允许产生副作用的步骤
- 外设类动作：执行外设指令（需锁）；自我类动作：写入世界树事实/生成存在/更新存在等

### Step4：输出
- 写输出场景回执 + 方法虚拟存在经验沉淀 + 必要的世界树事实更新


## 5.5 最小动作清单与 I/O 特征类型

本节用于回答两个问题：

- 当前系统最小应固定哪些动作原语
- 每个最小动作最少读哪些输入特征、写哪些输出特征

本节只讨论 `最小动作`，不讨论组合任务动作、管理类原函数和完整方法。

### 5.5.1 最小动作的边界

必须固定：

1. `最小动作` 是最小变化原语，不等于方法，不等于任务。
2. `方法` 由一个或多个最小动作组织而成；`任务` 消费的是方法返回结果，而不是动作自身。
3. 最小动作当前先只支持 `单观察面`：一次动作默认只围绕一个主观察特征做前后取证。
4. 最小动作可以分为两类：
   - `独立原语`：允许独立命名、独立调用、独立记录动作边界
   - `固定镶嵌机制`：只能嵌入其他功能函数中运行，方向固定，不参与改向
5. `组合任务动作`、`管理类原函数`、`四域兜底动作` 不属于最小动作层。
6. 动作执行后，必须以重新读取观察特征的结果作为是否发生变化的依据；不得只凭“已写入候选值”认定变化成立。

### 5.5.2 独立原语最小清单

| 动作名 | 直接输入特征类型 | 直接输出特征类型 | 用途与边界 |
| --- | --- | --- | --- |
| `读取单特征当前值` | `主体指针特征`、`目标特征类型` | `同类型带值特征` | 最基础观察动作；用于需求值/当前值/返回值比较，也用于动作后的再次观察。 |
| `写入单特征当前值` | `主体指针特征`、`目标特征类型`、`候选值特征` | `同类型带值特征` | 最基础改单值动作；动作后的真实结果仍以后读值为准。 |
| `写入单挂点指针` | `挂点主体指针特征`、`挂点类型特征`、`目标节点指针特征` | `同类型指针特征` | 用于设置 `当前任务`、`当前步骤`、`当前子任务`、`当前方法首节点` 等挂点。 |
| `确保单承载存在` | `存在类型特征`、必要的`归属指针特征` | `存在指针特征` | 用于确保 `自我存在`、`任务虚拟存在`、`方法虚拟存在`、`场景存在` 等承载面。 |
| `记录动作开始` | `动作名特征`、`动作主体指针特征`、`输入场景指针特征` | `动作事实动态（开始相位）` | 只记录动作开始事实，不代表一定发生了状态变化。 |
| `记录动作结束` | `动作名特征`、`动作主体指针特征`、`输出场景指针特征`、可选`错误码特征` | `动作事实动态（完成/失败相位）` | 只记录动作结束事实；完成或失败都必须保留边界。 |

### 5.5.3 固定镶嵌机制最小清单

以下动作当前应视为 `固定镶嵌机制`：

| 动作名 | 直接输入特征类型 | 直接输出特征类型 | 说明 |
| --- | --- | --- | --- |
| `按时下降服务值` | `服务值`、`服务时序衰减步长` | `服务值` | 时序维护动作；由自我线程每周期自动调用一次，每次调用生成一条本能动作动态。 |
| `按双值重算服务衰减步长` | `服务值`、`安全值` | `服务时序衰减步长` | 把服务下降放大与安全低位压制统一收口为一个内部先天动作；由自我线程每周期自动调用一次，每次调用生成一条本能动作动态。 |
| `按服务值重算安全增加步长` | `服务值` | `安全增加步长` | 负责把服务高低映射到安全低位回升能力；属于内部先天动作，由自我线程每周期自动调用一次，每次调用生成一条本能动作动态。 |
| `按服务值重算安全减少步长` | `服务值` | `安全减少步长` | 负责把服务高低映射到安全高位回落速度；属于内部先天动作，由自我线程每周期自动调用一次，每次调用生成一条本能动作动态。 |
| `安全低位回升` | `安全值`、`安全增加步长` | `安全值` | 仅当安全值低于 `50%` 时触发，推动安全值向 `50%` 回升；由自我线程每周期自动调用一次，每次调用生成一条本能动作动态。 |
| `安全高位回落` | `安全值`、`安全减少步长` | `安全值` | 仅当安全值高于 `80%` 时触发，推动安全值向 `80%` 回归；由自我线程每周期自动调用一次，每次调用生成一条本能动作动态。 |
| `服务归零待机保护` | `服务值`、`安全值` | `待机状态`、`安全值` | 服务归零后触发的底线保护动作；由自我线程每周期自动调用一次，每次调用生成一条本能动作动态。 |
| `按服务值重算时序步长` | `服务值` | `时序正向步长`、`时序反向步长` | 属于服务与安全耦合的内嵌调节机制，应保持镶嵌态。 |

当前不应把 `风险安全回归` 直接写死为绝对固定机制；是否纳入固定镶嵌清单，待风险安全轴的根层口径完全稳定后再定。

### 5.5.4 不属于最小动作层的对象

下列对象不得直接降格为“最小动作”：

- `推进一步`
- `自动筹办任务`
- `推进步骤子任务`
- `执行步骤`
- `管理_筹办器`
- `管理_选择器`
- `管理_分解器`
- `管理_求证器`
- `管理_门控器`
- `管理_复盘器`
- `管理_失败策略器`

这些对象要么是 `组合任务动作`，要么是 `管理类原函数`，都高于最小动作层。

### 5.5.5 最小动作后的固定派生规则

最小动作执行后的取证与派生顺序必须固定为：

1. 记录动作开始事实
2. 执行动作本体
3. 再次读取主观察特征
4. 记录动作结束事实
5. 若前后值不同，再生成新的 `前状态 / 后状态 / 二次特征方向 / 实例因果`

并且必须固定：

1. 没有变化，不得伪造新的变化状态。
2. 没有变化，不得伪造新的变化因果。
3. 没有变化，不等于动作没有发生；动作事实动态仍然成立。
4. 当前最小动作层默认只围绕 `一个主观察特征` 取证；多特征联合观察留待上层方法和任务层组织。

### 5.5.6 动作 ID / 代码落点对照表

本表只给出当前最小动作层的 `稳定对照关系`。

必须固定：

1. 表中若写 `沿用调用方动作 ID`，表示该项不是独立本能动作，而是某个动作的事件相位。
2. 表中若写 `已定义未注册`，表示枚举 ID 已保留，但当前默认启动链尚未把它作为独立本能函数注册。
3. 表中若写 `无独立本能 ID`，表示当前该能力仍保持镶嵌态，不应误当成可自由调度的候选动作。
4. 一个最小动作允许复用 `世界元动作 ID` 作为底层原语 ID；这不等于上层方法名和任务名也要复用同一个名字。

| 最小动作名 | 动作 ID | 当前状态 | 主要代码落点 | 说明 |
| --- | --- | --- | --- | --- |

## 6. 本能动作 ID 分段规范（建议）
- 1xxx：自我核心（识别并组织存在等）
- 5xxx：外设类（相机启动/关闭/取帧/读写参数）
- 6xxx：自我认知类（读帧特征/识别率/查找/生成/更新存在）
- 9xxx：调试/测试动作（仅开发期）

要求：
- 每个 ID 必须绑定唯一“函数名称”
- 删除动作：直接移除注册并清理旧路径（不做兼容桥接）


## 7. 依赖边界（防循环引用）

### 7.1 推荐 import 方向（从低到高）
- 主信息定义/基础类型（最低层）
- 本能动作通用函数模块（可选依赖更高层，但要谨慎）
- 本能动作管理模块（只依赖最低层）
- 具体实现模块（外设/自我认知），可以依赖外设模块、世界树环境、相机工作流

### 7.2 严禁
- 本能动作管理模块 import 具体外设/相机工作流/世界树环境
- 通用函数模块反向 import 本能动作管理模块


## 8. 注册入口规范
每个实现模块必须导出：
- `注册默认本能动作_XXX()`
自我线程启动后，必须在 `自我线程类::自我初始化()` 阶段统一调用这些注册入口，避免散落注册代码。

本节新增硬约束如下：

1. 本能函数注册时机必须固定在 `自我线程类::自我初始化()`。
2. 任务执行热路径、学习线程推进路径、周期维护路径不得承担“首次注册”职责。
3. 注册阶段只负责“函数注册 + 方法首节点创建/复用 + 方法信息回填”，不等价于执行本能函数本体。
4. 注册流程必须幂等：重复初始化注册时不得重复造方法头，必须复用既有本能方法首节点。
5. 初始化完成前，至少应完成“默认本能动作集”的注册，作为后续任务/学习执行的前置条件。


## 9. 最小自检规范（强烈建议）
每个本能方法至少提供一组“最小输入场景样例”用于单元自检：
- 能跑通 Step1~Step4
- 输出回执包含最小字段
- 对外设动作：无锁时不得写参数（测试锁门控）
