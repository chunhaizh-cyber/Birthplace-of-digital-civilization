# 0040 任务管理上行消息三分结构与派生需求消息规范 v0.1

## 1. 目标

本文用于把 `任务管理任务 -> 自我线程` 这条上行链的消息边界重新收口，重点回答以下问题：

- 任务管理任务向自我线程正式上报的非控制面消息到底分几类
- 为什么三类已经足够，且不建议再新增第四类业务一级类型
- `派生需求消息` 为什么不能再并入 `结果消息`
- 什么叫“本地不可消化的缺条件事实”
- 任务管理任务和自我线程在“派生需求 -> 正式需求 -> 子任务入口”这条链上各自有什么权限
- 自我线程收到派生需求消息后最少应按什么顺序处理
- `恢复重建 / 确认消息 / ACK / 重复消费` 这些内容应归到哪一层，而不误长成第四类业务消息

本文只负责收口上行消息边界与派生需求语义，不负责：

- 重写完整 `自我线程` 主循环实现
- 重写完整 `任务管理任务` 五模块算法
- 穷举全部派生需求类型字典
- 直接替代结果消息或线程状态消息的专门合同

## 3. 核心结论

必须固定以下结论：

1. 自我线程应接收三类正式上行消息：`线程状态消息`、`派生需求消息`、`结果消息`。
2. `线程状态消息` 回答“我现在怎么跑”；`派生需求消息` 回答“我这里缺什么、是否需要外部支撑”；`结果消息` 回答“我跑完之后结果是什么、服务对象现在怎样、过程里出现过哪些关键中间状态”。
3. `派生需求消息` 不等于正式需求；它只表达 `任务管理任务` 在本地推进中发现的 `本地不可消化缺条件`、`外部支撑依赖` 或 `会阻塞当前步骤推进的事实`。
4. `任务管理任务` 只能提出派生需求，不能直接创建子任务，也不能直接把派生需求写进正式需求树。
5. 是否将派生需求升格为正式需求、是否创建子任务入口、是否发起新的任务管理任务实例，只能由 `自我线程` 在根层重判和统一调度中决定。
6. `派生需求消息` 不是“过程中的结果消息”，也不是“新需求信息”；正式协议名推荐固定为 `派生需求消息`，`需求候选消息` 只保留为解释性别名。
7. 不是所有局部缺条件都要上报为派生需求；只有 `本地不可消化 / 会实质阻塞 / 需要外部支撑 / 会影响父任务或服务安全 / 可能需要新任务入口` 的情况，才升级为派生需求消息。
8. `结果消息` 仍应承接 `最终结果 + 服务对象当前状态 + 关键中间状态 + 未满足项`；但它不替代过程中的派生需求上报。
9. `未满足项` 仍可出现在结果消息里，但它回答的是“阶段完成后还留下什么未满足项”；`派生需求消息` 回答的是“当前推进过程中已发现什么本地不可消化缺口”。
10. 自我线程收到派生需求消息后，应先把它当事实入账，再重算二次特征、分类、决定是否升格，而不是一收到就直接写需求树或直接生子任务。
11. 作为 `任务管理任务 -> 自我线程` 的业务上行消息，三类已经覆盖 `状态 / 缺口 / 产出` 三种本质情况，不建议再新增第四类业务一级类型。
12. `关键中间状态` 不应单列第四类；它属于 `结果消息` 的正式组成部分。
13. `恢复重建` 不应单列第四类；它只是在三类中的不同语义内容：
    - 汇报“我故障了 / 需要重建 / 已恢复”时，归 `线程状态消息`
    - 汇报“恢复需要额外资源或外部支撑”时，归 `派生需求消息`
    - 汇报“恢复后世界和服务对象变成什么样”时，归 `结果消息`
14. `确认消息` 若未来存在，应来自另一条验证路径，例如外部观测、环境事实回流、人工确认或服务对象状态复核；它不属于 `任务管理任务` 的这三类上行消息。
15. `ACK / 重复消费 / 幂等冲突 / 反序到达 / 协议异常` 等属于传输层和基础设施层处理问题，应通过幂等键、消费结果对象、对账器解决，不得升格成第四类业务一级消息。
16. 现实世界的动作动态，不由 `自我线程` 直接管理；它们由 `任务管理任务 / 任务 / 本能动作` 三类执行体直接管理。
17. `自我线程` 只接收这些执行体上推的事实消息，并据此执行 `冻结批次 / 状态变化落账 / 二次特征重算 / 需求候选整理 / 根层重判 / 任务入口实例化 / 结果路由 / 总账结算 / 学习触发`。
18. 对 `自我线程` 而言，三类消息进入线程后统一只有两种基础治理用法：`提取并沉淀治理侧动作摘要/因果材料`，以及 `提取并沉淀状态变化`。
19. 三类消息的区别不在“能不能提供动作摘要或状态变化”，而在“以哪一种为主，以及后续进入哪条判定链”。
20. `线程状态消息` 主走 `控制态 / 局部推进态 / 可继续推进性` 判定链；`派生需求消息` 主走 `缺口事实 / 需求候选 / 根层重判` 判定链；`结果消息` 主走 `结果事实 / 关键中间状态 / 影响项 / 服务安全归结` 判定链。
21. 因此，三类消息共享同一套底层治理框架，不应被理解成三套彼此割裂的处理器。
22. `直接管理` 不等于 `自己闭环到底`；下层执行体直接管理动作动态后，仍必须把关键事实通过三类上行消息回流给 `自我线程`，才能形成完整闭环。
23. 三类消息先被提取成 `治理侧动作摘要/因果材料` 与 `状态变化`，这一步只叫 `入账`，不等于线程已经完成治理判断。
24. `自我线程` 的主责发生在 `入账之后`：它要继续把账本归并成可裁决材料，再完成 `判、算、派、结`。
25. 因此，`消息 -> 状态/动态` 只是治理入口；真正的线程治理主链至少还包括 `归并与对账、生成二次特征、生成影响项、更新根值/需求树/任务权重、根层重判、派发与路由、结算/否定项/学习触发` 七步。


## 4. 三类上行消息总表

| 上行消息类 | 回答的问题 | 典型触发时机 | 能否直接创建子任务 | 自我线程最小处理 |
| --- | --- | --- | --- | --- |
| `线程状态消息` | `我现在怎么跑，是否结束，为什么停在这里` | 启动、推进、等待、阻塞、故障、收束、停止 | 否 | 更新线程状态账、任务控制态候选、必要时重判 |
| `派生需求消息` | `我这里缺什么，是否需要外部支撑，是否会阻塞父任务` | 推进到方法或步骤时发现本地不可消化缺口 | 否 | 先入事实账，再分类，再决定是否升格为正式需求或依赖任务入口 |
| `结果消息` | `我跑完之后结果是什么，服务对象现在怎样，过程中出现过哪些关键中间状态` | 一次阶段性推进结束、局部结果返回、最终结果返回 | 否 | 解析结果事实、重算二次特征、路由父任务、统一结算 |

必须固定：

1. 三类消息都是“事实上报”，都不授予发件方最终裁决权。
2. `线程状态消息` 偏执行面，`派生需求消息` 偏缺口面，`结果消息` 偏产出面。
3. 这三类消息互补，不互相替代。
4. `状态 / 缺口 / 产出` 之外不再新增第四类业务一级类型；若出现新内容，必须先判断它应并入三类中的哪一类。
5. 对 `自我线程` 而言，三类消息都只有两种基础治理用法：`沉淀治理侧动作摘要/因果材料` 或 `沉淀状态变化`。
6. 三类消息的区别不在是否能提供这两类材料，而在何者为主，以及后续走哪条判定链。

### 4.1 为什么三类已经够

因为 `任务管理任务` 往上汇报时，本质上只会遇到三种情况：

- 在汇报自身推进状态
- 在汇报当前推进缺口
- 在汇报推进产出

一句话收口：

`线程状态消息` 负责“怎么跑”，`派生需求消息` 负责“缺什么”，`结果消息` 负责“跑完后变成什么样”。  

再新增第四类一级类型，通常只是把这三种再切碎，接口会更花，但边界不会更清楚。

### 4.1A 三层分工

必须把这条链正式分成三层：

#### A. 现实动作层

现实世界的动作在这一层直接发生，由三类执行体直接管理动作动态：

- `任务管理任务`
  - 管过程型动作动态，例如 `步骤推进 / 等待 / 阻塞 / 解阻 / 局部重试 / 提交结果`
- `任务`
  - 管目标语义上的动作动态，例如 `任务完成 / 中断 / 收束 / 结果状态变化 / 父子依赖变化`
- `本能动作`
  - 管快速反射型动作动态，例如 `避险 / 止损 / 保命 / 立即让渡 / 强制打断`

#### B. 线程治理层

`自我线程` 不直接代管上述现实动作动态本体；它负责：

- 收消息
- 冻结本轮消息
- 把消息提取成 `状态变化` 与 `治理侧动作摘要`
- 重算二次特征
- 整理需求候选
- 根层重判
- 任务入口实例化
- 结果路由
- 总账结算
- 学习触发

#### C. 知识沉淀层

这一层负责留痕、抽象和后续判断底座：

- `状态节点`
  - 记录 `谁的哪个特征变成了什么值`
- `动态节点`
  - 记录 `谁做了什么动作、成功与否、路径如何`
- `二次特征`
  - 记录 `依赖满足 / 当前步骤可执行 / 反向推进 / 风险升高` 等判断结果
- `因果实例`
  - 记录 `条件 - 动作 - 结果` 链

必须固定：

1. 现实世界动作动态的 `直接管理权`，属于 `任务管理任务 / 任务 / 本能动作` 三类执行体。
2. `自我线程` 不直接管理这些现实动作动态本体，只管理它们上推事实后的治理判断、调度和结算。
3. 知识沉淀层里的 `动态节点` 属于 `治理侧摘要 / 审计材料 / 因果沉淀`，不等于 `自我线程` 代替下层执行体直接执行业务动作。
4. 下层执行体必须在直接管理动作动态后，把关键事实通过三类上行消息回流给 `自我线程`；否则闭环不完整。

### 4.1B 入账后的治理七步

必须固定：

1. 三类消息被提取成 `状态变化` 与 `治理侧动作摘要/因果材料` 后，只能算 `一次入账完成`，不能算线程已经得出治理结论。
2. `自我线程` 的真正主责，是在入账之后把事实继续翻译成判断、调度与结算材料。
3. 从治理视角看，落账之后至少还要继续走 7 步：

| 步骤 | 主要回答的问题 | 最小输出 |
| --- | --- | --- |
| `1. 归并与对账` | `这批账是不是同一轮、同一次执行、同一对象的稳定事实` | `执行尝试汇总 / 冲突标记 / 去重结果 / 关键中间状态归并结果` |
| `2. 生成二次特征` | `这些一次事实意味着什么` | `依赖满足 / 当前步骤可执行 / 阻塞或解阻 / 反向推进 / 风险变化 / 方法可调用性 / 有效优先级候选` |
| `3. 生成影响项` | `这些变化对服务和安全的方向、强度、时效与可逆性是什么` | `直接服务影响 / 准备性服务影响 / 物理安全影响 / 风险安全影响` |
| `4. 更新根值、需求树与任务权重` | `这些影响应怎样归结到根层结构` | `安全值 / 服务值` 更新、`需求候选或需求树增量`、`任务权重与队列候选` |
| `5. 根层重判` | `当前主导需求是什么，系统整体下一步应如何裁决` | `主导需求 / 正式需求升格结论 / 队列裁决 / 收束或停止判定` |
| `6. 派发与路由` | `这轮裁决要落实成哪些外发动作和结果去向` | `任务入口实例化 / 新任务管理任务实例 / 控制指令 / 结果路由动作` |
| `7. 结算、否定项、学习触发` | `这轮到底推进了什么，沉淀了什么，下一轮要带走什么` | `结算摘要 / 否定项候选 / 学习触发项 / 故障收口材料` |

一句话收口：

`落状态、动态只是开始；自我线程后面真正做的是“对账、算二次特征、算影响项、改根层结构和权重、重判、派发、结算”。`

### 4.2 `线程状态消息` 应装什么

`线程状态消息` 用来汇报执行载体和局部推进状态，不直接下业务结论。

它的主定位是：`驱动和完善任务管理任务 / 工作线程的控制态与局部推进态`。

按统一底层治理框架，它至少有两种基础落法：

- `提取并沉淀治理侧动作摘要/因果材料`
  - 摘要 `谁响应了什么控制 / 谁进入阻塞或解阻 / 谁完成收束 / 谁故障退出`
  - 用于 `因果链 / 动作轨迹摘要 / 审计 / 恢复重建依据`
- `提取并沉淀状态变化`
  - 更新 `工作线程运行状态 / 任务控制态 / 当前等待原因 / 当前步骤可执行性候选 / 当前阻塞或解阻状态`
  - 后续进入 `二次特征重算`，判断 `当前步骤可执行 / 是否允许继续推进 / 是否需要收束 / 是否应退出可运行队列`

必须固定：

1. `线程状态消息` 主要用于完善 `控制态与局部推进态`，不是自我对外的业务结果表达。
2. 它可以提供 `治理侧动作摘要/因果材料`，也可以提供 `状态变化`，但后续主走的是 `控制链 / 推进链`，而不是 `正式需求树更新链` 或 `服务安全结果归结链`。
3. 它不应直接更新正式需求树，不应直接汇总上级服务值或安全值，也不应直接充当自我的对外业务结果。

应至少包括：

- `任务管理任务主键 / 绑定任务主键 / 父任务主键`
- `当前运行状态`
  - 例如：`已启动 / 推进中 / 等待中 / 阻塞 / 超时 / 故障 / 收束中 / 已停止`
- `当前步骤 / 当前步骤前沿`
- `当前等待原因`
  - 例如：`等待前序 / 等待条件 / 等待子结果 / 等待确认 / 等待资源`
- `是否还能继续推进`
- `是否已响应打断 / 收束 / 停止`
- `当前预算消耗 / 时间消耗`
- `错误码 / 路径签名 / 幂等键`

能收进去的典型语义包括：

- 阻塞了
- 解阻了
- 超时了
- 故障了
- 收束完成了
- 已停止了

必须固定：

1. 这些都不需要另立新类，归 `线程状态消息` 即可。
2. `恢复重建` 若表达的是“线程载体当前处于故障、待重建、已恢复”等状态，也归 `线程状态消息`。

### 4.3 `派生需求消息` 应装什么

`派生需求消息` 用来汇报“本地消化不了的缺口”。

它的主定位是：`上报缺口，让自我线程判断要不要升格成正式需求`。

按统一底层治理框架，它至少有两种基础落法：

- `提取并沉淀治理侧动作摘要/因果材料`
  - 摘要 `哪个任务管理任务 / 在哪个步骤 / 因为什么原因 / 上报了什么派生需求`
  - 用于 `因果实例 / 缺口模式识别 / 复盘为什么会开新任务`
- `提取并沉淀状态变化`
  - 更新 `当前方法前置条件不满足 / 当前任务等待原因变化 / 当前步骤由可执行变为不可执行 / 局部阻塞成立`
  - 后续进入 `二次特征重算`，判断 `当前步骤可执行 / 依赖满足 / 是否形成反向推进 / 是否需要外部支撑`

它的正式后续判定链应固定为：

1. 先入账为 `缺口事实`
2. 再转成 `状态变化`
3. 再重算 `二次特征`
4. 再由 `自我线程` 评估其对 `服务 / 安全 / 根值` 的影响
5. 再形成 `需求候选`
6. 只有通过 `根层重判` 后，才允许写入正式需求树
7. 再决定是否创建子任务并发起新的任务管理任务实例

应至少包括：

- `所属任务 / 父任务 / 当前步骤 / 当前方法`
- `缺口类型`
  - 例如：`缺资源 / 缺确认 / 缺信息 / 缺方法 / 风险处置 / 异常恢复 / 补偿动作`
- `为什么本地无法消化`
- `是否阻塞当前步骤`
- `是否影响父任务推进`
- `触发该缺口的事实`
  - 也就是：`谁做了什么动作，导致哪个条件特征不满足`
- `对服务 / 安全的影响提示`
- `是否建议线程重判`

能收进去的典型语义包括：

- 方法前置条件缺资源
- 当前步骤需要外部确认
- 局部无法恢复，需要外部补偿
- 风险升高，需要额外处置

必须固定：

1. 它只是候选事实，不是正式需求，更不是直接创建子任务。
2. 是否升格为正式需求、是否新建任务入口，由 `自我线程` 判断。
3. `派生需求消息` 可以提供 `治理侧动作摘要/因果材料`，也可以提供 `状态变化`，但它不直接改正式需求树；它只负责把“缺口事实”送入 `需求候选 -> 根层重判` 这条链。
4. 它主要是自我对内暴露的 `缺口状态`，不应直接当成对外业务结果。

### 4.4 `结果消息` 应装什么

`结果消息` 用来汇报推进产出，不只是成功/失败。

它的主定位是：`汇报任务执行产出，是三类里最接近自我外露业务状态的一类`。

按统一底层治理框架，它至少有两种基础落法：

- `提取并沉淀治理侧动作摘要/因果材料`
  - 摘要 `谁提交了结果 / 哪个任务做了什么动作 / 在什么场景里完成 / 成功或失败或异常 / 路径签名 / 错误码`
  - 用于 `因果关系 / 动作闭环摘要 / 审计与复盘`
- `提取并沉淀状态变化`
  - 更新 `服务对象当前状态 / 任务结果状态 / 关键中间状态 / 是否已恢复 / 未满足项`
  - 后续进入 `二次特征重算`，判断 `依赖满足 / 当前步骤可执行 / 反向推进标记 / 风险升高或下降 / 有效优先级候选`

它的正式后续判定链应固定为：

1. 先更新 `结果事实`
2. 再更新 `关键中间状态事实`
3. 再生成 `负面信息候选 / 否定项候选`
4. 再生成 `影响项`
5. 再通过 `二次特征与影响项`，把这些事实归结到上级 `安全值 / 服务值`
6. 再调整 `需求权重 / 任务权重 / 全局调度权重`

应至少包括：

- `最终结果段`
  - `成功 / 失败 / 部分完成 / 中断 / 收束完成 / 异常结束 / 完成度 / 错误码`
- `服务对象当前状态段`
  - `服务对象主键`，以及哪些特征从什么值变成了什么值
- `关键中间状态段`
  - `反向推进 / 阻塞与解阻 / 副作用 / 风险变化 / 会影响父任务语义的中间状态`
- `未满足项段`
  - 哪些还没闭合，是否建议线程重判

能收进去的典型语义包括：

- 任务完成了，服务对象状态更新了
- 虽然最后成功，但中途曾经阻塞、风险升高、后来恢复
- 结果只完成一半，还遗留未满足项

必须固定：

1. `关键中间状态` 不需要单列第四类；它属于 `结果消息` 的组成部分。
2. `未满足项` 也不需要另立第四类；它属于 `结果消息` 的后续支撑段。
3. `结果消息` 可以提供 `治理侧动作摘要/因果材料`，也可以提供 `状态变化`；但它不能裸着直接按结果字段去汇总上级服务值或安全值。
4. 上级 `服务值 / 安全值 / 权重` 的更新，只能由 `自我线程` 在 `二次特征与影响项` 层完成归结，不能由结果消息原始字段直接拍板。

### 4.5 `恢复重建` 为什么不单列第四类

`恢复重建` 不是第四类业务上行消息，它只是三类中的不同语义内容：

- 若在回答“我故障了 / 我需要重建 / 我已经恢复”，归 `线程状态消息`
- 若在回答“恢复需要额外资源、确认、信息或补偿”，归 `派生需求消息`
- 若在回答“恢复后世界和服务对象变成什么样”，归 `结果消息`

必须固定：

1. `恢复重建` 是跨三类可出现的语义主题，不是第四类一级类型。

### 4.6 `确认消息` 为什么不算第四类

`确认消息` 若未来存在，也不属于 `任务管理任务 -> 自我线程` 这条业务上行链。

它应来自另一条验证路径，例如：

- 外部观测
- 环境事实回流
- 人工确认
- 服务对象状态复核

必须固定：

1. `任务管理任务 -> 自我线程` 这条链仍然只保留三类。
2. `确认消息` 属于另一条来源链，而不是三类中的第四类。

### 4.7 `ACK / 重复消费 / 协议异常` 为什么不算第四类

以下内容不属于业务一级类型：

- `ACK`
- `重复消费`
- `幂等冲突`
- `反序到达`
- `协议异常`

这些属于传输层或基础设施层，应通过以下机制处理：

- `幂等键`
- `消费结果对象`
- `对账器`

必须固定：

1. 这些问题不应升格成第四类业务消息。
2. 业务上行仍然只保留 `线程状态 / 派生需求 / 结果` 三类。


## 5. 为什么 `派生需求消息` 不能并入 `结果消息`

因为 `结果消息` 太晚了。

很多需要外层立即处理的情况，不是等本轮任务结束以后才出现，而是在推进到某方法或某步骤时已经成立，例如：

- 前置条件不满足
- 当前步骤被实质阻塞
- 本地没有可切换的替代方法
- 必须依赖外部资源、确认、信息或另一个任务
- 父任务若继续留在可运行队列，会造成错误推进

此时如果不立即上报，自我线程就无法及时决定：

- 是否调整全局优先级
- 是否把父任务移出可运行队列
- 是否需要补资源、等确认、等外部信息
- 是否需要实例化新的依赖任务入口
- 是否应降级、等待或收束

一句话收口：

`结果消息` 负责汇报“已经发生了什么”，`派生需求消息` 负责汇报“现在缺什么、需不需要外部支撑”。  


## 6. 什么情况下应发 `派生需求消息`

### 6.1 不需要上报为派生需求的情况

以下情况只留在 `任务管理任务` 局部运行态，不升级为派生需求消息：

- 只是短暂等待
- 仍可本地重试
- 有备用方法可切换
- 已有结果马上回流
- 不影响父任务推进
- 不影响服务或安全判断

这些情况本质上只是 `局部缺条件`，还不是需要外层接管的 `派生需求`。

### 6.2 需要上报为派生需求的情况

满足任一条，就应升级为 `派生需求消息`：

- 本地无法消化
- 当前步骤被实质阻塞
- 需要外部资源、确认、信息或方法能力
- 会影响父任务推进
- 会影响服务或安全判断
- 可能需要新任务入口

必须固定：

1. `派生需求消息` 的上报门槛是“需要外层处理”，不是“出现任何局部缺条件”。
2. 发件方只能上报“这个缺口存在且本地无法消化”，不能顺手裁定“因此必须创建子任务”。


## 7. `派生需求消息` 最小合同

推荐正式固定为：

```cpp
struct 结构_派生需求消息 {
    主键 发出者主键 = 0;
    主键 任务管理任务主键 = 0;
    主键 当前任务主键 = 0;
    主键 父任务主键 = 0;
    主键 当前步骤主键 = 0;
    主键 当前方法主键 = 0;
    主键 关联场景主键 = 0;
    时间戳 产生时间 = 0;
    std::string 幂等键{};

    std::string 派生需求类型{};
    I64 阻塞级别 = 0;
    bool 是否本地不可消化 = false;
    bool 是否影响当前步骤可执行 = false;
    bool 是否影响父任务推进 = false;

    主键 动作主体主键 = 0;
    std::string 动作语义键{};
    主键 变化主体主键 = 0;
    主键 特征主键 = 0;
    特征值 变化前值{};
    特征值 变化后值{};
    std::string 当前等待原因{};
    std::string 条件缺口摘要{};

    bool 可能影响服务 = false;
    bool 可能影响安全 = false;
    bool 可能需要新任务入口 = false;
    bool 可能需要线程重判 = false;

    std::string 摘要{};
};
```

### 7.1 字段分组解释

| 段落 | 最小字段 | 含义 |
| --- | --- | --- |
| 归属段 | `任务管理任务主键 / 当前任务主键 / 父任务主键 / 当前步骤主键 / 当前方法主键 / 关联场景主键 / 时间戳 / 幂等键` | 回答“谁在什么上下文里上报这个派生需求事实” |
| 派生需求本体段 | `派生需求类型 / 阻塞级别 / 是否本地不可消化 / 是否影响当前步骤可执行 / 是否影响父任务推进` | 回答“缺的是什么、严重到什么程度、影响到哪一层” |
| 触发事实段 | `动作主体 / 动作语义 / 变化主体 / 特征变化 / 等待原因 / 条件缺口摘要` | 回答“为什么会出现这个派生需求” |
| 影响提示段 | `可能影响服务 / 可能影响安全 / 可能需要新任务入口 / 可能需要线程重判` | 只给提示，不给裁决 |

### 7.2 推荐类型口径

首批建议至少覆盖：

- `缺资源`
- `缺确认`
- `缺信息`
- `缺方法`
- `异常恢复`
- `补偿动作`

必须固定：

1. 正式协议名推荐固定为 `派生需求消息`。
2. `需求候选消息` 可以作为解释性同义名出现，但不建议作为正式协议名与代码名并行扩张。
3. `可能需要新任务入口` 只能是提示位，不能等价于“已批准创建子任务”。


## 8. 自我线程收到 `派生需求消息` 后的最小处理顺序

自我线程至少按 5 步处理：

### 8.1 先当事实入账

先把派生需求消息落成：

- 一次特征变化
- 状态节点
- 治理侧动作摘要或因果材料

不得一收到就直接写需求树。

### 8.2 重算二次特征

至少重算：

- `当前步骤可执行`
- `依赖满足`
- `父任务是否阻塞`
- `是否形成反向推进`
- `对服务 / 安全的影响候选`

### 8.3 分类

把派生需求分为三类：

| 类别 | 含义 | 处理 |
| --- | --- | --- |
| `A. 纯局部等待` | 仍可局部消化或仅需短暂等待 | 不创建新任务，只更新等待状态 |
| `B. 可并入已有需求 / 已有任务链` | 已有需求树节点或已有依赖任务可承接 | 不新建需求树节点，只并入已有承接对象 |
| `C. 需要升格为正式需求` | 当前缺口已需要外层正式治理与调度 | 才允许进入需求树 / 需求池 / 根层重判 |

### 8.4 决定是否创建子任务入口

这一步只归 `自我线程`：

- 若批准：
  - 创建任务根节点
  - 初始化任务控制态
  - 建立父子关系
  - 发起新的任务管理任务实例
- 若不批准：
  - 把拒绝、延后或并入已有链的结果回写给父任务

### 8.5 更新父任务状态

父任务通常会进入：

- `等待依赖`
- `等待确认`
- `等待资源`
- `等待外部结果`

并从可运行队列中移出，直到依赖解除。


## 9. 子任务创建边界

必须固定：

1. `任务管理任务` 只能提出派生需求，不能直接创建子任务。
2. `任务管理任务` 不能把派生需求直接写成正式需求树节点。
3. `任务管理任务` 不能直接发起新的任务管理任务实例。
4. `自我线程` 才能决定：
   - 是否升格为正式需求
   - 是否实例化新的依赖任务入口
   - 是否建立与父任务的父子关系
   - 是否发起新的任务管理任务实例

推荐正式表述：

`由自我线程根据派生需求，决定是否实例化新的依赖任务入口，并建立其与父任务的父子关系。`


## 10. `派生需求消息` 与 `结果消息` 的区别

| 消息类 | 主要回答的问题 | 典型时间点 |
| --- | --- | --- |
| `派生需求消息` | `我现在缺什么，需不需要外部支撑` | 过程内触发 |
| `结果消息` | `我跑完之后结果是什么，服务对象现在怎样，中间出现过什么关键状态` | 阶段完成后汇报 |

必须固定：

1. `派生需求消息` 偏“缺口”。
2. `结果消息` 偏“产出”。
3. 二者都可带未满足项相关信息，但语义焦点不同。
4. `结果消息` 中的 `未满足项` 不应拿来替代过程中的派生需求上报。


## 11. 推荐一句话规范表述

推荐正式写成：

`任务管理任务发给自我线程的业务上行消息，一级上只保留线程状态消息、派生需求消息、结果消息三类。现实世界的动作动态由任务管理任务、任务、本能动作三类执行体直接管理；自我线程不直接代管这些动作动态本体，而是接收它们上推的事实消息，并将其先提取为治理侧动作摘要/因果材料与状态变化。该步骤只构成一次入账；入账之后，自我线程还必须继续执行归并与对账、二次特征生成、影响项生成、根值与需求树归结、根层重判、派发与路由、结算与学习触发。线程状态消息负责汇报“怎么跑”，主走控制态与局部推进态判定链；派生需求消息负责汇报“缺什么”，主走缺口事实、需求候选与根层重判链；结果消息负责汇报“跑完后变成什么样”，主走结果事实、关键中间状态、影响项与服务安全归结链。关键中间状态、未满足项、恢复重建事实都应收编进这三类，而不再新增第四类一级类型；确认消息若存在，应视为另一条来源链，不属于任务管理任务上行消息。任务管理任务只能提出派生需求，不能直接创建子任务；是否升格为正式需求、是否创建子任务入口并发起新的任务管理任务实例，由自我线程在根层重判与统一调度中决定。`


## 12. 完成标准

达到下面 14 条，即可视为这轮规范已经站稳：

1. 已经把 `线程状态消息 / 派生需求消息 / 结果消息` 三类上行消息正式区分开。
2. 已经明确 `派生需求消息` 是候选事实，不是正式需求。
3. 已经明确 `任务管理任务` 不能直接创建子任务。
4. 已经明确 `自我线程` 是派生需求升格、依赖任务入口创建和父子关系建立的唯一裁决者。
5. 已经明确 `派生需求消息` 与 `结果消息` 的时间点和语义焦点不同。
6. 已经把旧口径中“只保留两大类上行消息”的结论修订为三分类口径。
7. 已经明确 `关键中间状态 / 未满足项 / 恢复重建事实` 都必须并入三类，而不新增第四类一级类型。
8. 已经明确 `确认消息` 若存在，属于另一条验证来源链，不属于任务管理任务上行消息。
9. 已经明确 `ACK / 重复消费 / 幂等冲突 / 反序到达 / 协议异常` 属于传输层与基础设施层，不属于业务一级分类。
10. 已经明确现实世界动作动态由 `任务管理任务 / 任务 / 本能动作` 三类执行体直接管理，而不是由 `自我线程` 直接代管。
11. 已经明确 `自我线程` 对三类消息的统一基础治理用法，是提取 `治理侧动作摘要/因果材料` 与 `状态变化`，而不是接管下层现实动作动态本体。
12. 已经明确三类消息的区别在 `主用途与后续判定链`，而不在是否能提供治理材料。
13. 已经明确 `消息转成状态、动态` 只是 `入账`，不等于线程已经完成治理判断。
14. 已经明确入账之后，`自我线程` 还必须继续执行 `归并与对账 -> 二次特征 -> 影响项 -> 根值/需求树/任务权重 -> 根层重判 -> 派发与路由 -> 结算/否定项/学习触发` 这条七步治理链。
