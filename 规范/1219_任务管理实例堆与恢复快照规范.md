# 1219 任务管理实例堆与恢复快照规范

## 1. 目标

本文用于固定 `任务管理任务` 的状态承载分层，回答以下问题：

- `任务管理任务` 是否应有独立的运行时实例堆
- `任务管理状态` 是否应直接写进宿主任务虚拟存在
- 线程退出、宿主停机、异常重启时，到底该保存什么
- 哪些字段属于瞬时调度态，哪些字段属于可恢复治理态
- `根层裁决 -> 任务管理上下文 -> 请求/实例/快照` 之间如何接平
- 为什么不应把整个任务管理实例原样持久化

本文要固定的是：

- `实例堆 / 宿主任务虚拟存在 / 恢复快照` 这组 `运行承载三层`
- 以及允许外挂但不持有推进权的 `任务管理主体虚拟存在` 汇总载体边界

## 3. 核心结论

必须固定以下 16 条：

1. `每个宿主任务` 仍然只允许对应 `最多一个任务管理运行时实例`。
2. `任务管理运行时实例` 必须保存在独立的 `实例堆/实例表` 中。
3. `宿主任务虚拟存在` 必须保存可恢复、可观察、可审计的 `任务管理治理状态`。
4. `线程退出/宿主停机` 时，不得原样持久化整个任务管理实例对象。
5. 退出时允许保存的只能是 `恢复快照`，而不是 `线程态对象镜像`。
6. `实例堆` 负责瞬时调度态，`任务虚拟存在` 负责治理状态面，`恢复快照` 负责跨退出恢复。
7. `锁状态 / 条件变量等待态 / 正在执行的函数体 / 持有推进权` 这类内容，严禁写入恢复快照。
8. `当前功能域 / 下一功能域 / 最近总控结果 / 最近原因说明 / 当前管理方法` 这类内容，应镜像到宿主任务虚拟存在。
9. `恢复快照` 应理解为“重新生成任务管理实例所需的最小状态”，而不是“恢复某条物理线程”。
10. 重启后应优先从 `宿主任务虚拟存在 + 恢复快照` 重建任务管理实例，而不是假定旧实例仍然存在。
11. `任务管理请求` 必须建立在 `根层已完成裁决`、且 `任务管理上下文` 已形成之后。
12. `宿主任务虚拟存在` 镜像的是治理状态面与任务管理上下文，不是整包请求材料。
13. `恢复快照` 只保存“下一轮重新装配上下文并继续推进”所需的最小字段，不保存根语义来源本体。
14. 上述三层是 `运行承载三层`，不包含主体级成长汇总；主体级汇总必须按 [1228_任务管理主体虚拟存在与分身继承规范.md](D:\鱼巢\规范\1228_任务管理主体虚拟存在与分身继承规范.md) 另行承载。
15. 允许存在一个 `任务管理主体虚拟存在` 作为非运行时汇总载体，用于跨宿主、跨分身、跨学习回流汇总成长特征，但它不持有推进权。
16. “很多分身” 只能理解为主体级逻辑分身；同宿主内部仍只有一个活实例持有推进权。

可收敛为一句话：

`任务管理任务必须采用“实例堆承载瞬时调度态，任务虚拟存在承载治理状态面，恢复快照承载跨退出最小恢复信息”的三层结构。`

### 3.1 最新统一主链中的位置

按当前最新体系，本文必须服从以下主链：

```text
根层方向裁决
-> 形成任务管理上下文
-> 生成任务管理请求
-> 实例堆中的运行时实例消费请求并单步推进
-> 结果、治理状态面与恢复信息分层写回
-> 外层重判后再进入下一轮请求
```

必须固定：

1. `实例堆` 不承载根层裁决来源，只承载已提交的一轮运行时输入和瞬时状态。
2. `宿主任务虚拟存在` 承载的是“外部可见的治理状态面”和最小任务管理上下文镜像。
3. `恢复快照` 只服务“下一轮如何继续”，不服务“把旧线程原样复活”。
4. `任务管理主体虚拟存在` 若存在，也只做跨轮成长汇总，不应被误写成第四个运行时线程层。

更细的 `任务管理上下文` 最小稳定字段表，以 [1214_任务管理上下文最小字段表规范.md](D:\鱼巢\规范\1214_任务管理上下文最小字段表规范.md) 为准；更细的 `一步治理结果` 返回字段表，以 [1213_任务管理一步治理结果最小返回面规范.md](D:\鱼巢\规范\1213_任务管理一步治理结果最小返回面规范.md) 为准；本文只负责说明这些字段如何分层落到 `实例堆 / 宿主任务虚拟存在 / 恢复快照`。


## 4. 运行承载三层定义

### 4.1 任务管理实例堆

`任务管理实例堆` 指的是：

- 当前进程内存中的任务管理实例表、队列、mailbox、等待信息集合

它的职责是：

- 保存运行时瞬时状态
- 保存事件驱动调度状态
- 保存等待与唤醒过程中的内部信息

它不是：

- 可长期持久化的业务状态账本
- 跨进程/跨退出的稳定恢复源

### 4.2 宿主任务虚拟存在

`宿主任务虚拟存在` 指的是：

- 宿主任务在任务虚拟存在中持有的、可被任务管理读取和写回的治理状态面

它的职责是：

- 表达当前治理进度
- 表达任务管理最近结论
- 表达最近一次根层裁决与任务管理上下文中的关键镜像
- 为 UI、审计、因果、成长和恢复提供统一可见面

它不是：

- 线程调度容器
- 任务管理实例内部队列

### 4.3 恢复快照

`恢复快照` 指的是：

- 线程退出、宿主停机、异常重启时，从运行时实例提炼出的最小恢复信息

它的职责是：

- 帮助重建任务管理实例
- 指示下一轮从哪里继续
- 保证恢复后语义正确

它不是：

- 整个运行时实例对象的二进制转储
- 物理线程恢复点

### 4.4 主体级汇总载体（非运行时）

允许额外存在：

- `任务管理主体虚拟存在`

它的职责是：

- 汇总跨宿主、跨分身、跨学习回流的稳定成长特征
- 承接 `主体ID / 分身ID / 方法需求位 / 本能能力缺口` 这类主体级桥接信息

它不是：

- 运行时实例堆
- 第四个线程层
- 同宿主并行推进权限


## 5. 实例堆必须保存的内容

以下字段必须归入 `任务管理实例堆`：

- `宿主任务头指针`
- `实例运行状态`
- `等待原因`
- `最近提交请求`
- `最近反馈快照`
- `最近处理序号`
- `最近开始时间`
- `最近结束时间`
- `最近唤醒时间`
- `当前等待开始时间`
- `持有推进权`
- `是否已在待运行队列`
- `mailbox/待消费事件`

这些字段的共同特点是：

- 强依赖当前进程内调度
- 强依赖线程与队列时序
- 退出后直接失效

因此：

- 可以在运行时堆中改得很频繁
- 不应直接当成稳定任务事实


## 6. 任务虚拟存在必须保存的内容

以下字段应镜像进 `宿主任务虚拟存在`：

- `管理任务_最近根层裁决结果`
- `管理任务_当前功能域`
- `管理任务_下一功能域`
- `管理任务_当前步骤`
- `管理任务_当前方法`
- `管理任务_当前子任务`
- `管理任务_当前缺口类型`
- `管理任务_主体ID`
- `管理任务_分身ID`
- `管理任务_输入条件主键`
- `管理任务_治理态型`
- `管理任务_结构完备度`
- `管理任务_方法可用度`
- `管理任务_风险门控`
- `管理任务_当前方法需求位`
- `管理任务_当前本能能力缺口类型`
- `管理任务_当前下一步去向`
- `管理任务_最近总控结果`
- `管理任务_最近原因说明`
- `管理任务_最近反馈类型`
- `管理任务_最近反馈摘要`
- `管理任务_最近结果节点主键（若存在结果节点）`
- `管理任务_条件主键`
- `管理任务_二次特征状态`
- `管理任务_等待学习标记`
- `管理任务_等待外部反馈标记`
- `管理任务_最近实例因果锚点`
- `管理任务_最近动作动态锚点`

这些字段的共同特点是：

- 对任务管理结论有解释价值
- 可用于恢复、显示、审计、因果和成长
- 不依赖具体线程对象

必须额外固定：

- 最近治理结果应由 `最近总控结果 / 最近反馈摘要 / 最近下一步去向 / 最近结果节点主键（若存在）` 共同表达。
- 其中的 `最近结果节点主键` 只承担 `结果定位 / 检索辅助 / 审计回指` 职责。
- 读取宿主任务虚拟存在时，不得把 `最近结果节点主键` 直接当成“任务当前已经满足需求”的依据。

因此：

- 应优先理解为宿主任务的治理状态面
- 允许比实例堆更新得更慢，但必须保持语义正确


## 7. 恢复快照必须保存的内容

恢复快照只应保存“重建实例所需的最小恢复信息”，建议至少包括：

- `宿主任务头指针`
- `快照版本`
- `最近根层裁决结果`
- `最近功能域`
- `最近总控结果`
- `最近原因说明`
- `最近等待原因`
- `最近提交时间`
- `最近反馈摘要`
- `主体ID`
- `分身ID`
- `最近条件主键`
- `最近治理态型`
- `最近方法需求位`
- `最近本能能力缺口类型`
- `最近结果节点主键（若存在）`
- `最近下一步去向`
- `当前恢复点类型`
- `是否存在待消费学习回流`
- `是否存在待消费外部反馈`

恢复快照应满足：

- 读出来后能重建一个新的任务管理实例
- 能让新实例知道“下一轮从哪里继续”
- 但不要求恢复到“某条线程刚执行到第几条指令”

必须额外固定：

- `最近结果节点主键` 写入恢复快照时，只承担 `恢复回指 / 检索辅助` 职责。
- 它不能单独决定恢复后“上一轮是否已经成功”或“当前目标是否已经完成”。
- 恢复后的真实判断，仍必须回到宿主任务虚拟存在里的更具体治理状态面，并重新走外层重判。


## 8. 严禁写入恢复快照的内容

以下内容严禁写入恢复快照：

- `mutex / lock_guard / condition_variable` 状态
- `持有推进权`
- `是否正在线程池某工作线程上运行`
- `线程 ID`
- `队列内部游标`
- `lambda / function 对象`
- `临时迭代器`
- `原样复制的最近请求结构体中的悬空指针`

原因是：

- 这些内容不是治理语义
- 退出后不可直接复原
- 强行保存会制造伪恢复


## 9. 退出规则

### 9.1 正常退出

正常退出时应执行：

1. 清理实例堆中的 `持有推进权`
2. 清空等待中的线程态资源
3. 写入恢复快照
4. 保留任务虚拟存在中的治理状态面
5. 释放实例堆对象

### 9.2 异常退出

异常退出时至少应保证：

- 不把损坏的实例对象直接回写成任务事实
- 能写多少恢复快照就写多少恢复快照
- 宿主任务虚拟存在仍保留最后一次可信治理状态

### 9.3 线程退出

线程退出时应理解为：

- 结束某一轮运行时承载

而不是：

- 任务管理任务本体被销毁

因此线程退出后：

- 实例堆对象可以重建
- 任务虚拟存在和恢复快照必须仍可支撑下一轮继续推进


## 10. 恢复规则

恢复时应按以下顺序：

1. 先读取宿主任务虚拟存在中的治理状态面。
2. 再读取恢复快照。
3. 用两者合成新的运行时实例。
4. 把实例置为 `待提交`、`待运行` 或某种明确等待态。
5. 下一轮由自我线程重新驱动。

不允许：

- 假定旧实例对象仍然有效
- 跳过恢复快照直接把实例表当成持久层

## 12. 与当前实现的直接要求

对当前实现必须收口成：

1. 当前工程中的 `治理实例堆快照 / 最近请求 / 恢复快照` 已经形成正式最小承载；`M4` 的恢复重建一次性消费、`M5` 的显式 `mailbox` 事件消费均已落地。
2. `任务控制总控模块_v0.ixx` 输出的 `最近根层裁决结果 / 当前功能域 / 下一功能域 / 最近总控结果 / 原因说明 / 当前缺口类型 / 当前下一步去向 / 管理派生产出 / 二次特征` 必须继续镜像写入宿主任务虚拟存在。
3. 当前已禁止把整个 `结构_任务管理实例` 直接序列化到任务事实中；恢复只允许通过 `恢复快照` 重建。
4. 下一阶段应继续深化实例堆内 `mailbox/待消费事件` 的优先级、观测面和持久化，但仍保持“每宿主一实例、一轮一步、一旦消费后即失效”的纪律。
5. 后续若把 `结构_任务管理请求` 继续细化，优先接平的应是 `显式事件消费结果` 与 `任务管理上下文关键字段`，而不是先扩张线程态字段。


## 13. 最终结论

最终必须固定成：

`任务管理实例堆` 用来跑，
`宿主任务虚拟存在` 用来记，
`恢复快照` 用来续，
`任务管理主体虚拟存在` 用来跨宿主汇总成长。

其中前 3 者构成 `运行承载三层`，后一者是 `非运行时主体汇总层`；两组职责不得互相替代。
