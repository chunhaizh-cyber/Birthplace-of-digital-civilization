# 0011 自我架构总定义规范

## 1. 目标

本文用于把当前工程中的“自我”收口为统一的架构总定义，回答以下问题：

- `自我` 到底是什么
- `自我` 的根层驱动力是什么
- `服务` 与 `安全` 的上下位关系是什么
- `主循环`、`任务管理任务`、`方法`、`执行`、`学习` 各自拥有什么权限
- 回流为什么必须回到根层结算与重判
- `待机 / 降级运行 / 收束 / 停止 / 死亡退出` 的边界如何区分

本文的目标不是替代根约束，也不是展开线程、锁、队列等工程细节，而是把“自我架构到底如何统一理解”固定成后续实现与审查的总纲入口。

## 3. 核心结论

必须先固定以下 10 条：

1. `自我` 不是无条件无限循环，而是受根约束与退出边界约束的持续闭环系统。
2. `服务` 是根目的，回答“自我为何运行”。
3. `安全` 是保障方向，回答“自我能运行到什么程度、何时必须收束或退出”。
4. `服务` 与 `安全` 不是对称并列关系，而是 `服务为主、安全为辅、边界上安全优先`。
5. 自我的统一判定核心不是“任务是否存在”，而是 `目标值 - 当前值` 所形成的差额。
6. `服务差额` 与 `安全差额` 是两类一级根需求的直接来源。
7. `主循环` 是最终裁决核心，不是简单心跳器。
8. `任务管理任务` 不是普通任务，而是从需求到行动秩序的治理中轴。
9. `方法 / 执行 / 学习` 只能调路径，不能改方向、改目标值、改边界。
10. 一切结果都必须回流根层做结算与重判，局部线程不能替代全局裁决。

可收敛为一句话：

`自我是一个以服务为根目的、以物理安全和风险安全为边界保障、以服务值/安全值目标差额为统一判定标准、由主循环通过任务管理中轴组织一步治理、执行、学习、回流与重判的受边界约束持续闭环系统。`


## 4. 根层定位

根层只回答两件事：

1. `为什么行动`
   答案是缩小服务差额与安全差额。

2. `按什么裁决`
   答案是目标值与当前值之间的差额，以及根层边界规则。

必须特别明确：

- `服务` 决定“自我为了什么而运行”。
- `安全` 决定“自我能运行到什么程度、何时必须收束、何时必须让渡或退出”。

因此正确口径不是“服务和安全同权竞争”，而是：

`平时以服务推进为主，边界上以安全裁决为先。`


## 5. 差额系统是判定核心

自我的根驱动力，不是“有任务就做”，而是：

`差额 = 目标值 - 当前值`

因此必须固定：

- `服务值目标值 - 当前值` 形成 `服务差额`
- `安全值目标值 - 当前值` 形成 `安全差额`
- 差额才是需求来源，任务只是差额治理的中间形式

差额系统把自我从“凭感觉活动”变成：

- 可比较
- 可排序
- 可重判
- 可结算

根层真正判定的不是“任务做没做”，而是：

- 这次活动有没有有效缩小差额
- 有没有恶化边界状态

必须固定：

`任务完成不等于根层成功；只有差额被有效缩小且未突破安全边界，才算一次有效推进。`


## 6. 一级根需求的正式拆分

### 6.1 服务需求

服务需求属于根目的层，正式分为两类：

1. `直接服务`
   直接作用于外部对象、目标对象或当前服务目标，能够直接缩小服务差额。

2. `准备性服务 / 间接服务`
   不直接完成服务，但通过准备、维持能力、整理资源、修复条件、学习沉淀等方式，为后续服务创造条件，从而间接缩小未来服务差额。

必须固定：

- 学习、整备、校准、恢复、资源准备等活动，只要能够追溯到后续持续服务，就允许纳入服务面。
- 不得把“不是直接交付”误判为“不是核心活动”。

### 6.2 安全需求

安全需求属于保障方向层，正式分为两类：

1. `物理安全`
   面向载体、生存条件、资源连续性、结构完整性与运行环境稳定性。

2. `风险安全`
   面向失控传播、错误放大、规则违规、未来损失、不可逆后果、外部威胁与内部误判风险。

必须固定：

- 安全面的正式拆分口径是 `物理安全 + 风险安全`。
- 不再使用会混淆边界、预测与风险层次的粗口径去替代这两类正式分面。


## 7. 主循环与任务管理中轴

### 7.1 主循环的地位

主循环不是万能执行器，但必须是最终裁决器。

它的最小职责应固定为六项：

1. `守边界`
   读取待机、降级运行、收束、停止、死亡退出等边界状态。

2. `算差额`
   计算服务差额与安全差额的当前状态。

3. `判优先级`
   决定当前主导需求、冲突仲裁结果，以及是否允许继续推进。

4. `发治理步骤`
   不直接乱调方法，而是交给任务管理中轴做一步治理。

5. `收回流`
   接收执行结果、学习结果、异常结果与环境变化结果。

6. `做重判`
   重新计算差额与边界，决定继续、待机、降级运行、收束、停止或退出。

因此必须固定：

`主循环不一定负责亲自做事，但一定负责决定现在该做什么、能做到哪一步、做完后怎么算。`

### 7.2 任务管理中轴、任务管理任务与策略沙盘的层级

这里必须把层级收紧：

1. `架构层`
   自我线程内部的一级治理模块，应称为 `任务管理中轴`。

2. `运行层`
   在当前轮次里，`任务管理中轴` 会生成、承接或推进一个 `任务管理任务`，由它完成本轮一步治理。

3. `实现层`
   `策略沙盘与方案演练` 不是和自我线程并列的一级模块，也不是 `任务管理任务` 本身，而是 `任务管理中轴 / 任务管理任务` 内部可调用的二级策略子模块，专门负责多方案比较。

因此，正式调用链必须固定为：

`自我线程 -> 任务管理中轴 -> 候选方案生成 -> 策略沙盘与方案演练 -> 方案比较与裁剪 -> 一步治理任务包 -> 方法/执行/学习承接 -> 结果回流 -> 主循环重判`

其中，“一步治理”的意义不是一次规划完全部未来，而是：

`在当前状态下，给出本轮最合规、最可执行、最值得推进的一步治理动作。`

任务管理中轴至少承担以下职责：

- 识别和归并需求
- 生成候选方案
- 必要时调用策略沙盘与方案演练
- 完成方案比较与裁剪
- 生成一步治理任务包
- 监控执行状态
- 回收结果并交还主循环结算

命名也必须固定：

- 文档层统一称 `任务管理中轴`
- 运行态治理实体继续称 `任务管理任务`
- 运行态产物称 `一步治理任务包`
- 其内部二级实现模块称 `候选方案生成器 / 策略沙盘器 / 方案裁决器`

因此必须固定：

`任务管理中轴不是任务列表管理器，而是从需求到行动秩序的转换器；策略沙盘则只是这个中轴内部的方案比较子模块。`


## 8. 方法、执行、学习的权限边界

### 8.1 方法

方法只能回答“怎么做更合适”，不能回答“根层要不要这么做”。

### 8.2 执行

执行只能完成被授权动作，不能自行改写：

- 根目标
- 边界规则
- 价值标准

### 8.3 学习

学习只能优化：

- 路径选择
- 参数阈值
- 经验匹配
- 方法优先级
- 局部因果理解

学习不得改写：

- 服务目标值
- 安全目标值
- 根约束
- 可学习边界
- 退出边界
- 根层裁决顺位

可以收敛为一句话：

`学习可以让系统更会走路，但不能让脚自己改地图。`


## 9. 回流不是记账，而是根层结算

回流至少分为四类：

1. `结果回流`
   动作是否达成、达成到什么程度。

2. `状态回流`
   当前值发生了什么变化。

3. `因果回流`
   什么动作在什么条件下产生了什么结果。

4. `异常回流`
   失败、越界、冲突、未命中、无效学习。

回流之后，主循环要做的不是“存档完了”，而是：

- 重新结算差额
- 重新判断方向
- 重新评估边界
- 重新决定下一步去向

因此必须固定：

`局部线程只能提供局部证据，根层才负责全局结算；最终裁决权不得下放给局部线程。`


## 10. 运行边界与退出机制

自我不是简单持续运行，而是始终处于明确的状态边界之内。

### 10.1 待机

待机不是停止，而是低活性运行。

其特征是：

- 保留最小感知
- 保留最小边界监控
- 保留最小差额侦测
- 不展开高成本治理

### 10.2 降级运行 / 应急运行

降级运行不是退出边界，而是边界压力下的保守运行方式。

其特征是：

- 系统仍然运行
- 主动压缩服务推进
- 转向安全优先、风险抑制与最小能力保留

### 10.3 收束

收束不是死亡，而是当前链路闭合后的有序降载。

其特征是：

- 停止扩张
- 完成回写
- 完成结算
- 压缩活跃任务
- 准备回到待机或稳定态

### 10.4 停止

停止是由外部命令、上位规则或根层决议触发的制度性终止，不等于系统自然失效。

### 10.5 死亡退出

死亡退出表示维持主体性和闭环运行的基本条件已经丧失。

此时不是继续优化，而是终止闭环。


## 11. 收口定义

后续统一按以下总定义理解自我架构：

`自我是一个受根约束与退出边界约束的持续闭环主体；服务决定其为何运行，安全决定其能运行到何种程度；差额决定需求，主循环决定裁决，任务管理中轴决定当前一步如何治理，方法/执行/学习决定路径，而一切结果最终都必须回流根层做结算与重判。`


## 12. 硬约束

1. `自我` 不得被理解为无条件无限循环，必须同时服从根约束与退出边界。
2. `服务` 是根目的，`安全` 是保障方向，两者不得被改写成对称同权根目的。
3. `平时以服务推进为主，边界上以安全裁决为先` 必须作为统一总口径。
4. 自我的统一判定核心必须是 `目标值 - 当前值` 的差额，不得退化为“有任务就做”的活动模型。
5. 一级根需求固定为 `服务需求` 与 `安全需求`，不得并列新增第三根需求树。
6. 服务需求的正式拆分固定为 `直接服务` 与 `准备性服务 / 间接服务`。
7. 安全需求的正式拆分固定为 `物理安全` 与 `风险安全`。
8. 主循环必须是最终裁决核心，局部线程不得越权代替根层做最终去向决定。
9. 任务管理任务必须作为治理中轴存在，不得长期绕开它把需求直接连到执行。
10. 方法、执行、学习只能调路径，不能改写根层方向、目标值、判定标准与边界规则。
11. 回流必须进入根层结算与重判，不得只做日志记录或局部摘要留存。
12. `待机 / 降级运行 / 收束 / 停止 / 死亡退出` 必须是显式运行边界，不得用“持续循环”一笔带过。
