# 信息入口类型与信息存储树路由规范 20260510

## 0. 核心原则

系统不使用人类词性直接决定节点类型或存储位置。

人类词性只是自然语言解析结果，是语言数字化工具。它负责把词、短语、句式转换成候选语义信号，例如：

```text
这个词更像名词
这个词更像动词
这个短语更像状态描述
这个结构更像因果表达
```

系统内部真正用于建节点、选择主信息类型、选择存储树的是：

```text
信息入口类型
```

因此原来的：

```cpp
枚举_程序词性
```

应改名为：

```cpp
枚举_信息入口类型
```

一句话：

```text
人类词性是语言进入系统的翻译器；
信息入口类型才是世界树里的通行证。
```

---

## 1. 最终分层

```text
自然语言
  ↓
人类词性层：语言数字化
  ↓
信息入口类型层：系统语义入口
  ↓
基础信息类型层：节点主信息类型
  ↓
存储树路由层：模板树 / 现实世界树 / 内部世界树 / 需求根树 / 任务根树 / 方法根树 / 语言记录树
```

各层职责：

```text
自然语言：
    原始输入。

人类词性层：
    把自然语言转换成人类语言结构候选。

信息入口类型层：
    把语言候选转换成系统内部可处理的语义入口。

基础信息类型层：
    决定节点主信息结构。

存储树路由层：
    决定节点应该进入哪棵树、哪个根节点。
```

简化表达：

```text
人类词性负责候选。
上下文负责消歧。
信息入口类型负责落树。
基础信息类型负责节点结构。
```

---

## 2. 三类类型的边界

### 2.1 人类词性

人类词性来自分词、词性标注、语法分析、语言模型判断。

它包括：

```text
名词
专有名词
动词
形容词
副词
介词
连词
数词
量词
短语
句子结构
因果表达
状态描述
```

人类词性不直接决定：

```text
节点放哪棵树
节点是不是模板
节点是不是实例
节点是不是任务
节点是不是方法
节点是不是虚拟存在
```

它只负责提供候选。

### 2.2 信息入口类型

信息入口类型是系统内部语义入口。

它决定：

```text
1. 当前语言对象在系统中被当成什么；
2. 使用哪种主信息类型；
3. 默认进入哪棵存储树；
4. 默认挂在哪个根节点；
5. 是否需要主体、场景、时间、任务、方法等上下文限定。
```

例如：

```text
飞机        -> 存在概念入口
B-1234      -> 存在实例入口
位置        -> 特征模板入口
B-1234.位置 -> 特征实例入口
移动        -> 动态模板入口
B-1234移动  -> 动态实例入口
加油任务    -> 任务概念入口
T001任务    -> 任务虚拟存在入口
```

### 2.3 基础信息类型

基础信息类型是节点内部主信息结构。

例如：

```text
存在
抽象特征
特征
状态
动态
关系
因果
场景
任务
方法
语言记录
```

它不负责解释自然语言，也不直接负责路由。

它只回答：

```text
这个节点的主信息结构是什么？
```

## 3. 信息入口类型枚举

建议定义：

```cpp
enum class 枚举_信息入口类型 : std::uint16_t {
    未定义 = 0,

    存在概念入口,
    存在实例入口,

    特征模板入口,
    特征实例入口,

    状态模板入口,
    状态实例入口,

    动态模板入口,
    动态实例入口,

    关系模板入口,
    关系实例入口,

    因果模板入口,
    因果实例入口,

    场景模板入口,
    场景实例入口,

    需求概念入口,
    需求节点入口,

    任务概念入口,
    任务信息节点入口,
    任务虚拟存在入口,

    方法概念入口,
    方法信息节点入口,
    方法虚拟存在入口,

    语言记录入口
};
```

如后续需要表达更细的语言辅助结构，可以扩展：

```cpp
二次特征模板入口,
修饰关系入口,
逻辑连接入口,
数值入口,
单位入口,
特征值辅助入口
```

但建议先不要急着塞入主枚举，除非系统已经明确需要这些入口独立建节点。

## 4. 存储树枚举

```cpp
enum class 枚举_信息存储树 : std::uint8_t {
    未定义 = 0,

    基础信息模板树,
    现实世界树,
    内部世界树,

    需求根树,
    任务根树,
    方法根树,

    语言记录树,
    不入树
};
```

说明：

```text
基础信息模板树：
    存放概念、模板、抽象结构。

现实世界树：
    存放现实中的实例、特征实例、动态实例、场景实例。

内部世界树：
    存放系统内部推理、假设、虚拟状态、模拟对象。

需求根树：
    存放需求相关节点。

任务根树：
    存放任务相关节点。

方法根树：
    存放方法、流程、策略相关节点。

语言记录树：
    存放原始语言记录、表达痕迹、语料追踪。

不入树：
    只作为临时解析结构或辅助信号，不形成正式节点。
```

## 6. 语素入口主信息

原来的“词性节点”不应再理解为纯粹的人类词性节点。

更准确的名称建议是：

```text
语素入口节点
```

或者：

```text
语言入口节点
```

如果保留旧名“词性节点”，也应明确说明：

```text
词性节点不是单纯记录人类词性，而是记录语言对象进入系统语义结构的入口信息。
```

建议结构：

```cpp
class 词性主信息类 {
public:
    // 人类词性：来自分词、标注、语言解析。
    // 只作为语言数字化结果，不直接决定存储树。
    枚举_词性 人类词性 = 枚举_词性::未定义;

    // 信息入口类型：系统内部真正使用的语义入口。
    枚举_信息入口类型 信息入口类型 = 枚举_信息入口类型::未定义;

    // 由 信息入口类型 推导。
    枚举_主信息类型 对应基础信息类型 = 枚举_主信息类型::未定义;

    // 当前入口绑定的概念节点、实例节点、特征节点、动态节点等基础信息节点。
    可解析引用<基础信息节点类> 对应基础信息{};
};
```

如果任务节点、方法节点、需求节点不是 `基础信息节点类`，则不要强行塞进 `对应基础信息`。

可以拆成：

```cpp
class 词性主信息类 {
public:
    枚举_词性 人类词性 = 枚举_词性::未定义;
    枚举_信息入口类型 信息入口类型 = 枚举_信息入口类型::未定义;
    枚举_主信息类型 对应基础信息类型 = 枚举_主信息类型::未定义;

    可解析引用<基础信息节点类> 对应基础信息{};
    可解析引用<运行节点类> 对应运行节点{};
};
```

规则：

```text
基础信息节点进入 对应基础信息。
任务、方法、需求等运行结构进入 对应运行节点。
不要把不同生命周期、不同树体系的节点混进同一个字段。
```

## 7. 人类词性到信息入口类型

人类词性与信息入口类型不是一对一，而是一对多候选关系。

| 人类词性 | 可能的信息入口类型 |
| --- | --- |
| 名词 | 存在概念入口、特征模板入口、状态模板入口、场景模板入口、任务概念入口、方法概念入口 |
| 专有名词 | 存在实例入口、场景实例入口、任务虚拟存在入口、方法虚拟存在入口 |
| 动词 | 动态模板入口、动态实例入口、方法概念入口 |
| 形容词 | 特征模板入口、状态模板入口、特征实例入口 |
| 副词 | 二次特征模板入口、修饰关系入口 |
| 介词 | 关系模板入口、关系实例入口 |
| 连词 | 逻辑连接入口、因果模板入口、语言记录入口 |
| 数词 / 量词 | 数值入口、单位入口、特征值辅助入口 |

判定规则：

```text
裸词通常进入模板 / 概念入口。
带主体、时间、场景、任务上下文时，才进入实例入口。
同一词面可以有多个人类词性。
同一人类词性可以生成多个信息入口候选。
最终入口由上下文消歧决定。
```

---

## 8. 上下文消歧规则

信息入口类型的最终选择依赖上下文。

常见上下文包括：

```text
主体
时间
地点
场景
任务
方法
因果结构
关系结构
属性归属
值绑定
实例标识
```

示例：“位置”

```text
词面：位置
人类词性：名词
候选入口：
    特征模板入口
    状态模板入口
    存在概念入口
```

根据上下文消歧：

```text
“位置”
    -> 特征模板入口
    -> 基础信息模板树 / 抽象特征根 / 位置

“B-1234 的位置”
    -> 特征实例入口
    -> 现实世界树 / B-1234.位置

“当前位置为 101机位”
    -> 特征实例入口
    -> 主体的当前位置特征实例，值为 101机位
```

## 9. 示例规范

### 9.1 “飞机”

```text
词面：飞机
人类词性：名词
信息入口类型：存在概念入口
主信息类型：存在
存储树：基础信息模板树
对应节点：存在概念根 / 飞机
```

说明：

```text
存在概念只作为词义锚点和实例创建模板。
不新增抽象存在类。
不新增存在形态。
现实实例通过 概念集 对应一个或多个存在概念。
```

### 9.2 “B-1234”

```text
词面：B-1234
人类词性：专有名词
信息入口类型：存在实例入口
主信息类型：存在
存储树：现实世界树
对应节点：现实世界树 / B-1234
```

说明：

```text
B-1234 是现实或业务世界中的具体实例。
它可以通过 概念集 指向 飞机、航空器、资产 等多个概念。
```

### 9.3 “位置”

```text
词面：位置
人类词性：名词
信息入口类型：特征模板入口
主信息类型：抽象特征
存储树：基础信息模板树
对应节点：抽象特征根 / 位置
```

### 9.4 “B-1234.位置”

```text
词面：B-1234.位置
人类词性：名词短语
信息入口类型：特征实例入口
主信息类型：特征
存储树：现实世界树
对应节点：现实世界树 / B-1234 的位置特征实例
```

### 9.5 “当前位置 = 101机位”

```text
主体：B-1234
特征类型：当前位置
特征值：101机位
```

如果 `101机位` 本身是存在节点或基础信息节点，则特征值继续使用：

```text
指针句柄
```

不新增：

```text
当前值信息
```

也就是说：

```text
特征实例持有值。
值可以是数值、文本、枚举，也可以是其他节点的句柄。
```

## 10. 词面相同但入口不同

### 10.1 “安全”

```text
安全
├─ 人类词性：名词
│  └─ 信息入口类型：存在概念入口
│     └─ 表示“安全”这个概念存在
│
└─ 人类词性：形容词
   └─ 信息入口类型：特征模板入口
      └─ 表示“安全性”这个抽象特征
```

### 10.2 “移动”

```text
移动
├─ 人类词性：动词
│  └─ 信息入口类型：动态模板入口
│     └─ 移动这个动态模板
│
└─ 带具体主体、时间、起点、终点
   └─ 信息入口类型：动态实例入口
      └─ 某次移动动态实例
```

### 10.3 “加油任务”

```text
加油任务
├─ 信息入口类型：任务概念入口
│  └─ 任务概念树 / 加油任务
│
└─ T001加油任务
   └─ 信息入口类型：任务虚拟存在入口
      └─ 某个任务信息节点绑定的任务虚拟存在
```

## 11. 语素层结构

语素层不再是：

```text
词 + 人类词性 -> 基础信息
```

而应改成：

```text
词 + 人类词性 + 信息入口类型 -> 基础信息 / 运行节点
```

推荐结构：

```text
词节点
  ├─ 词面
  ├─ 原始文本位置
  ├─ 人类词性记录
  │    ├─ 名词
  │    ├─ 动词
  │    └─ 形容词
  │
  └─ 信息入口节点
       ├─ 信息入口类型
       ├─ 主信息类型
       ├─ 存储树
       ├─ 默认根节点
       ├─ 上下文限定
       └─ 对应节点
```

如果一个词有多个解释，应允许多个信息入口节点：

```text
词节点：安全
  ├─ 信息入口节点：存在概念入口
  └─ 信息入口节点：特征模板入口
```

## 12. 路由流程

标准流程：

```text
1. 接收自然语言。
2. 分词、标注、人类词性分析。
3. 根据人类词性生成信息入口候选。
4. 读取上下文。
5. 选择最终信息入口类型。
6. 根据入口定义表取得主信息类型。
7. 根据入口定义表取得存储树和默认根。
8. 创建或引用对应节点。
9. 记录词面到入口节点的映射。
```

伪代码：

```cpp
信息入口候选集 生成入口候选(
    const 词节点类& 词节点,
    const 语言上下文类& 上下文
);

枚举_信息入口类型 选择入口类型(
    const 信息入口候选集& 候选集,
    const 语言上下文类& 上下文
);

节点引用 路由到信息节点(
    枚举_信息入口类型 入口类型,
    const 语言上下文类& 上下文
);
```

## 13. 定稿规则

```text
1. 人类词性是语言数字化工具，不直接决定系统节点类型。

2. 信息入口类型是系统语义入口，负责决定主信息类型、默认存储树、默认根节点。

3. 基础信息类型只表示节点的主结构，不承担自然语言解释职责。

4. 存储树路由由信息入口类型决定，而不是由人类词性决定。

5. 同一个词面可以有多个人类词性，也可以有多个信息入口类型。

6. 裸词通常指向模板入口或概念入口。

7. 带主体、时间、场景、任务、方法上下文的词组，才指向实例入口。

8. 任务、方法、需求等运行节点如果不是基础信息节点，应使用独立引用字段，不混入普通基础信息引用。

9. 词性节点如继续保留命名，应解释为语素入口节点，而不是严格的人类词性节点。

10. 系统内部所有落树行为，应通过信息入口定义表统一路由。
```

## 14. 最终一句话

```text
人类词性负责把语言变成候选；
信息入口类型负责把候选变成系统入口；
基础信息类型负责决定节点结构；
存储树路由负责决定节点落点。
```

这版规范的目标是：自然语言解析仍然能接入系统，但系统不会被人类语法结构绑死。词性只是入口前的翻译层，真正支配世界树结构的是信息入口类型。
