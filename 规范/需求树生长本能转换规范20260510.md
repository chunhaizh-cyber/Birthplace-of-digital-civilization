# 需求树生长本能转换规范 20260510

## 1. 总原则

需求树正常生长中的转换，原则上属于自我内核本能。

```text
本能转换函数 = C++ 内置的确定性转换函数。
本能转换方法 = 将本能转换函数包装成方法系统可识别、可调度、可记录的本能方法。
学习转换方法 = 后续由学习、因果补完或探索形成的普通方法 / 学习方法。
```

工程口径：

```text
需求树正常生长中的转换，先实现为本能转换函数；
其中需要被任务系统调用、记录、调度、复盘的，再注册成本能转换方法。
```

一句话：

```text
本能转换函数保证内部循环能跑；
本能转换方法让这些转换进入任务 / 方法 / 因果记录体系。
```

## 2. 为什么属于本能层

需求树生长的核心转换不是后天学习技能，而是数字生命的基础生理机制。

例如：

```text
自我状态 -> 根需求压力
根压力 -> 当前主需求
特征值差距 -> 需求节点
父需求 -> 子需求权重
重复目标 -> 合并需求
执行需求 -> 任务信息节点
任务反馈 -> 自我安全 / 服务值
任务反馈 -> 需求结算账本
结算结果 -> 下一轮需求选择
```

如果这些转换依赖普通方法或后天学习，会形成死锁：

```text
方法系统尚未建立 -> 需求树不能生长
需求树不能生长 -> 任务和方法不能正常建立
```

因此：

```text
需求树能不能生长：靠本能。
需求树长得聪不聪明：靠后续学习和因果。
```

边界修订：

```text
这里的“差距”只表示同一目标特征类型上的特征值变化或未满足状态。
它不能表达不同特征类型之间的关系。

不同特征类型之间的联动、影响、先后关系和因果关系，应由状态动态、二次特征 / 关系特征、因果实例 / 因果模板、方法结果能力承载。
```

## 3. 三层转换分类

### 3.1 内核本能转换函数

这是最底层 C++ 函数，直接保证需求树能工作。

允许直接由：

```text
自我类
自我线程
需求类
任务管理界面线程
任务管理工作线程
```

在各自职责边界内调用。

示例：

```text
评估根需求压力
特征值差距转需求节点
刷新子需求权重
合并重复需求
刷新需求结构角色
执行需求转任务信息节点
任务反馈转需求结算
刷新父级需求结算
```

### 3.2 本能转换方法

这是把内核本能转换函数注册到方法系统中的包装。

用途：

```text
任务调用
运行记录
因果沉淀
方法虚拟存在绑定
后续复盘
控制面板观察
```

本能转换方法内部实际调用对应的本能转换函数。

示例：

```text
本能方法_自我状态转根需求压力
本能方法_特征值差距转需求节点
本能方法_刷新子需求权重
本能方法_合并重复需求
本能方法_执行需求转任务信息节点
本能方法_任务反馈转需求结算
```

### 3.3 学习转换方法

这是后续学习或探索阶段形成的方法。

例如：

```text
通过外部观察判断某状态。
通过历史因果推断方法有效性。
通过外界反馈修正需求权重。
通过多次任务结果学习更好的分解策略。
```

这些不属于需求树最小生长必需项。

## 4. 必须是本能层的转换

以下转换必须先有本能转换函数。

其中需要进入任务系统调度或复盘的，再注册本能转换方法。

| 转换 | 本能转换函数 | 本能转换方法 | 说明 |
| --- | --- | --- | --- |
| 自我状态 -> 根需求压力 | 必须 | 建议 | 内部循环起点 |
| 根压力 -> 当前主需求 | 必须 | 建议 | 决定当前处理方向 |
| 特征值差距 -> 需求节点 | 必须 | 建议 | 同一目标特征的当前值 / 目标值未满足时生成需求 |
| 状态节点 -> 目标特征缓存 | 必须 | 可选 | 需求去重和合并依赖 |
| 父需求 -> 子需求权重 | 必须 | 可选 | 层级调度核心 |
| 重复目标 -> 合并需求 | 必须 | 可选 | 防止需求风暴 |
| 有子 / 无子 -> 管理 / 执行需求 | 必须 | 可选 | 需求结构角色判断 |
| 执行需求 -> 任务信息节点 | 必须 | 建议 | 需求进入行动链 |
| 任务反馈 -> 自我安全 / 服务值 | 必须 | 建议 | 结果回写核心 |
| 任务反馈 -> 需求结算账本 | 必须 | 建议 | 闭环核心 |
| 子需求结算 -> 父需求结算 | 必须 | 可选 | 管理需求汇总 |
| 方法阻塞 -> 派生需求 | 必须，但可后置 | 建议 | 因果补完和学习需求前置 |

## 5. 不属于需求树转换方法、但可属于任务层本能方法的内容

以下不属于本规范中的“需求树生长本能转换方法”：

```text
1. 任务管理工作线程调度。
2. 任务管理界面线程承接。
3. 自我线程主循环。
```

这些是任务系统或自我线程的治理流程，不是本能方法。

最新口径修订：

```text
任务筹办和任务执行不是任务层本能方法，已改为任务管理生命周期普通函数。

它们不是需求树生长转换方法；
也不是自我线程主循环、任务管理界面线程或工作线程调度。
```

边界要求：

```text
任务筹办 / 任务执行普通函数只返回任务层过程回执；
输出结果场景只承载世界事实；
任务虚拟存在的任务状态、当前方法、候选方法集，由任务管理层读取运行存在后同步。
```

不得恢复旧的：

```text
自我_任务执行一步 步骤壳；
工作线程直接修改任务账；
任务筹办 / 执行把任务状态写入输出结果场景；
自我线程主循环、任务管理界面线程承接、工作线程调度注册成本能方法。
```

## 6. 暂不属于本能转换方法的内容

以下可以后续由普通方法、学习方法或因果沉淀方法承担：

```text
复杂因果推断
外界探索策略
学习资料选择
多方法竞争排序
复杂风险评估
长期记忆压缩
因果模板升格
外界反馈解释
探索对象选择
```

## 7. 建议命名

### 7.1 本能转换函数命名

建议建立命名空间：

```cpp
namespace 自我本能转换函数 {

bool 自我状态转根需求压力(...);
需求节点类* 特征值差距转需求节点(...);
std::size_t 刷新子需求权重(...);
需求节点类* 合并或新增同目标需求(...);
void 刷新需求结构角色(...);
任务节点类* 执行需求转任务信息节点(...);
bool 任务反馈转需求结算(...);
bool 刷新父级需求结算(...);

}
```

第一阶段不强制迁移已有函数到该命名空间。

可以先建立映射表：

```text
本能转换名称 -> 当前实现函数 -> 所属模块 -> 是否需要包装成本能方法
```

### 7.2 本能转换方法命名

建议本能方法名称：

```text
本能方法_自我状态转根需求压力
本能方法_根压力转当前主需求
本能方法_特征值差距转需求节点
本能方法_刷新子需求权重
本能方法_合并重复目标需求
本能方法_刷新需求结构角色
本能方法_执行需求转任务信息节点
本能方法_任务反馈转需求结算
本能方法_子需求结算转父需求结算
本能方法_方法阻塞转派生需求
```

代码枚举名应按项目现有 `枚举_本能方法ID` 风格再定，不在规范中直接占用枚举值。

## 8. 注册条件

不是所有本能转换函数都必须立刻注册成本能方法。

注册成本能方法应至少满足：

```text
1. 该转换需要被任务系统作为可执行方法调用。
2. 该转换需要记录方法虚拟存在、运行账或条件结果对。
3. 该转换的输入 / 输出可以表达为方法能力签名。
4. 该转换不破坏模块职责边界。
```

暂不注册的情况：

```text
1. 只是需求类内部维护函数，例如刷新结构角色。
2. 只是线程内调度裁决，例如自我线程主循环选择。
3. 输入输出尚未能稳定表达为状态 / 特征 / 需求目标。
4. 注册后会引入任务系统依赖自身启动的循环依赖。
```

## 9. 方法能力签名要求

注册成本能方法后，必须声明能力签名。

能力签名仍遵守现有方法查找规则：

```text
查找方法只按结果能力匹配；
不得按方法名称、动作名称、动作句柄、本能 ID 或固定编号命中。
```

示例：

```text
本能方法_特征值差距转需求节点
条件面：
    当前状态 = 有效
    目标状态 = 有效
    满足关系 = 有效
结果包：
    需求节点存在状态 -> 已建立
    需求目标缓存状态 -> 已就绪
```

```text
本能方法_任务反馈转需求结算
条件面：
    来源任务 = 有效
    来源需求 = 有效
    任务反馈 = 有效
结果包：
    需求结算状态 -> 已更新
    自我核心状态回写状态 -> 已回写
```

## 10. 调用边界

本能转换函数可以直接调用，但必须守住模块职责：

```text
需求类：
    维护需求树结构、目标、权重、去重、合并、任务绑定字段。

自我线程：
    决策、调度、消息归并、需求树更新、任务治理请求提交和价值结算。

任务管理界面线程：
    绑定任务虚拟存在、初始化任务语义、生成工作包、归并任务结果。

任务管理工作线程：
    只处理任务工作包，产生工作结果和上行消息。
```

禁止：

```text
1. 本能转换函数绕过需求类直接破坏需求树结构。
2. 本能转换方法直接替父需求结算。
3. 工作线程直接修改自我线程内部世界。
4. 自我线程重新补建任务虚拟存在。
5. 因业务缺口缺少上下文而静默落默认根。
```

## 11. 错误处理

本能转换层同样遵守全局错误处理口径。

非正常逻辑错误：

```text
显式父需求为空。
父需求解析为需求树结构根。
执行需求任务化后无对应任务信息节点。
任务建立完成后仍无任务虚拟存在。
需求同时指向任务信息节点和任务虚拟存在。
实例入口缺主体或场景却试图落树。
```

处理：

```text
弹错 / 报错。
记录日志。
中断当前转换。
不静默兜底。
```

业务缺口：

```text
方法缺动作骨架。
方法缺条件结果对。
参数不可配。
方法运行账不足。
方法结果不稳定。
因果链缺关键节点。
```

处理：

```text
返回结构化状况。
由任务管理线程解释状况。
必要时由自我线程决定是否入需求树。
```

## 12. 与需求树生长计划的关系

需求树正常生长计划中的“转换使用的方法”统一改为：

```text
本能转换函数 / 本能转换方法
```

落地顺序：

```text
1. 先核对已有 C++ 转换函数。
2. 再补统一本能转换函数门面或映射表。
3. 最后按需要注册成本能方法。
```

不得反过来：

```text
先注册一批本能方法，
再让需求树依赖方法系统才能生长。
```

## 13. 最终定稿

```text
需求树生长的转换是自我的生理反射，不是后天技能。

因此：
1. 最小闭环必须由本能转换函数保证。
2. 需要进入任务 / 方法 / 因果记录体系的转换，再包装为本能转换方法。
3. 学习方法和外界探索方法只能增强需求树，不得成为需求树最小生长前提。
```
