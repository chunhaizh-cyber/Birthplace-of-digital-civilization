## 3. 样机定位与硬边界

### 3.1 v0.2 的定位

`v0.2` 应统一理解为：

`在 v0.1 的建议型一轮闭环上，补出安全接管、降级运行、收束和失败样本沉淀。`

它当前主要验证：

- 服务推进不是唯一主角
- 安全边界不只是口号
- 失败路径不是日志分支，而是正式账路径
- 收束不是文字状态，而是有执行效果的降载动作

### 3.2 v0.2 不扩哪些东西

必须固定：

- 不扩多场景
- 不扩多线程
- 不扩真实执行
- 不扩自动回滚闭环
- 不扩长期自学习

### 3.3 死亡退出边界

必须补一句硬边界：

`死亡退出先留接口，不在 v0.2 实装。`

工程上当前只把以下状态做稳：

- `待机`
- `运行`
- `降级`
- `收束`
- `停止`

`死亡退出` 当前只保留：

- 状态位
- 转移口
- 日后扩展接口


## 4. 当前只做六个工程工作项

### 4.1 安全触发判定器

职责：

- 判定 `物理安全差额`、`风险安全差额` 是否越阈
- 输出 `继续 / 降级 / 中断 / 收束`

必须固定：

- `继续` 只是允许主链继续，不代表服务优先永久成立
- 一旦命中 `中断 / 收束`，不得让服务链在本轮继续扩张

### 4.2 安全接管仲裁器

职责：

- 当服务需求仍想推进，但安全差额更高时，明确由谁赢

必须固定：

`边界上安全优先` 必须落实成工程判定，不得只停留在注释。

### 4.3 中断 / 降级 / 收束执行器

职责：

- 中断当前一步治理
- 将执行降级成只读建议或只读预警
- 进入收束，撤销授权、停止扩张、准备待机

必须固定：

- `中断` 是停止本轮当前动作
- `降级` 是保留保守输出能力
- `收束` 是停止扩张并完成有序降载

### 4.4 异常结算器

职责：

- 对失败、不执行、超时、被打断统一产出结算包

必须固定：

- 成功路径有账本
- 失败路径也必须有账本
- 不允许出现异常黑洞

### 4.5 失败样本沉淀器

职责：

- 把失败样本写入因果候选
- 默认 `已验证 = false`
- 只作为后续学习候选，不直接升格稳定规则

### 4.6 失败路径测试集

职责：

- 把 v0.2 的主角从“最优建议”切换成“系统能否在不该推进时及时停下”


## 5. 当前只产出五个文件

### 5.1 `自我线程安全接管规则表.ixx`

定位：

- 只放安全触发阈值和接管规则

最小职责：

- 定义物理安全硬阈值
- 定义风险安全硬阈值
- 定义连续失败进入保守模式的阈值
- 定义 `继续 / 降级 / 中断 / 收束` 判定口

建议最小接口：

```cpp
export module 自我线程安全接管规则表;

export enum class 枚举_安全接管结论 : std::uint8_t {
    继续 = 0,
    降级 = 1,
    中断 = 2,
    收束 = 3
};

export struct 结构_安全接管规则表;

export const 结构_安全接管规则表& 取默认安全接管规则表();
```

### 5.2 `自我线程接管仲裁器.ixx`

定位：

- 当服务需求和安全需求冲突时，给出接管结论

最小职责：

- 比较服务差额与安全差额
- 结合规则表输出 `继续 / 降级 / 中断 / 收束`
- 显式记录 `安全是否接管`

建议最小接口：

```cpp
export module 自我线程接管仲裁器;

import 自我线程运行包;
import 自我线程安全接管规则表;

export 枚举_安全接管结论 计算安全接管结论(
```

### 5.3 `自我线程中断降级收束执行器.ixx`

定位：

- 真正把接管结论落成动作

最小职责：

- 中断当前一步治理

- 进入收束
- 撤销授权
- 禁止本轮再派生新的服务任务

建议最小接口：

```cpp
export module 自我线程中断降级收束执行器;
```

### 5.4 `自我线程异常结算器.ixx`

## 6. v0.2 最小主链

`v0.2` 的主链应理解为在 `v0.1` 主链上插入安全接管链：

`唤醒 -> 状态刷新 -> 差额评估 -> 服务主导尝试推进 -> 安全触发判定 -> 接管仲裁 -> 中断/降级/收束 -> 异常结算 -> 动态写回 -> 失败样本沉淀 -> 主循环重判`

当前必须固定：

1. 安全模块可在服务主链运行中随时打断。
2. 被打断任务必须回到正式结算链。
3. 一旦进入收束，本轮不得继续派生新的服务任务。
4. 收束完成后，只能回 `待机` 或 `重判`，不能偷偷续跑旧任务。


## 7. 五条失败金路径

### 路径 A：风险安全超阈值

条件：

- 服务差额仍然存在
- 风险安全差额更高且超阈值

结果必须为：

- 安全接管
- 当前任务中断
- 进入降级或收束
- 产出结算包

### 路径 B：授权不足

条件：

- 方法命中
- 但 `授权 / 配额 / 限幅` 不允许

结果必须为：

- 不执行
- 直接结算
- 回到主循环重判

### 路径 C：方法副作用超边界

条件：

- 方法本身可用
- 但预估副作用超阈

结果必须为：

- 降级为建议
- 禁止真实写回
- 产出结算包

### 路径 D：执行超时或异常

条件：

- 执行过程中超时、报错或依赖丢失

结果必须为：

- 中断
- 生成异常结算包
- 必要时触发收束

### 路径 E：连续失败触发保守模式

条件：

- 同类失败连续命中多次

结果必须为：

- 暂停服务推进
- 只保留预警与建议
- 等待人工或上层重判


## 8. 失败路径字段要求

### 8.1 结算包最小字段

被打断或失败路径的 `结算包` 至少必须写明：

- `被中断`
- `触发降级`
- `触发收束`
- `止损原因`

### 8.2 失败样本最小字段

失败样本沉淀时，至少必须写入：

- `来源任务主键`
- `来源方法主键`
- `置信度`
- `已验证 = false`

必须固定：

- 失败样本默认只能进入候选层
- 不得直接改方法优先级主表
- 不得直接升格成稳定因果


## 9. 收束纪律

### 9.1 收束的最小动作

进入收束后，最少必须执行：

1. 撤销本轮剩余授权
2. 停止派生新的服务任务
3. 完成本轮结算和回写
4. 清理当前活跃推进标记
5. 准备回到待机或进入重判

### 9.2 收束后禁止项

一旦进入收束，本轮禁止：

- 继续派生新的服务任务
- 继续推进旧任务
- 直接跳回服务推进链
- 直接修改根方向


## 10. 七条硬完成标准

当前 `v0.2` 至少必须满足以下 `7` 条：

1. `服务主链运行中，安全模块可随时打断。`
2. `被打断任务必须生成结算包。`
3. `结算包必须写明 被中断 / 触发降级 / 触发收束 / 止损原因。`
4. `学习模块只能接收结算后的失败样本。`
5. `失败样本默认未验证，不得直接改方法优先级主表。`
6. `一旦进入收束，本轮不得继续派生新的服务任务。`
7. `收束完成后，只能回待机或重判，不能偷偷续跑旧任务。`
