# 2020 因果信息自然语言表达规范

## 1. 目标

本文用于固定当前工程中：

- 因果信息如何转换为自然语言
- 实例因果与抽象因果应如何区分表达
- 因果文本如何同时保留来源、证据与可信度线索

本文重点回答：

- 因果信息转自然语言时先抽什么、后说什么
- 什么情况下可以按事实说，什么情况下只能按归纳说
- 成立次数、失败次数如何影响表达强弱
- 控制面板、日志、自检输出应如何统一口径

本文服从以下硬边界：

1. 因果节点及其结构化字段是真值层。
2. 自然语言表达只是解释层和显示层。
3. 自然语言不得反向新增、删改或伪造因果真值。
4. 抽象因果不是当前事实，不得按当前事实语气直说。


## 3. 定位与边界

### 3.1 本文负责

本文负责：

- 固定 `因果信息 -> 因果表达单元 -> 因果句型 -> 文本` 的主链
- 固定实例因果与抽象因果的不同表达口径
- 固定条件、动作、结果的抽取顺序
- 固定成立次数、失败次数对表达强弱的影响方式


### 3.2 本文不负责

本文不负责：

- 发明新的因果真值字段
- 取代 [1285_因果分层建模与实例生成规范.md](D:\鱼巢\规范\1285_因果分层建模与实例生成规范.md) 中的实例因果生成边界
- 把抽象因果改写成当前事实
- 把自然语言显示层变成新的推理层


### 3.3 总原则

必须固定：

1. 因果信息不直接拼句子，先生成因果表达单元。
2. 因果表达单元固定围绕 `条件 + 动作 + 结果` 组织。
3. 实例因果按事实语气表达。
4. 抽象因果按归纳语气表达。
5. 成立次数、失败次数只影响可信度和语气强弱，不改变真值性质。

一句话固定：

`因果先抽成 条件-动作-结果 的表达单元，再按 实例/抽象 选择事实语气或归纳语气。`


## 4. 因果自然语言主链

因果信息转换为自然语言的主链固定为：

`因果节点`
-> `识别因果类型`
-> `抽取条件 / 动作 / 结果`
-> `生成因果表达单元`
-> `选择因果句型`
-> `线性化`
-> `输出文本 + 来源 + 证据 + 次数`

必须固定：

1. 先识别 `实例因果 / 抽象因果`，再选语气。
2. 先抽结构化槽位，再生成文本。
3. 不允许跳过表达单元层直接在各模块手拼字符串。


## 5. 因果表达单元最小字段

当前工程里，一条可显示的因果表达单元最少应包含：

```text
因果类型
锚点类型
锚点名称
条件短语列表
动作短语
结果短语列表
主结果短语
来源节点列表
证据节点列表
成立次数
失败次数
可信度提示
```

其中必须固定：

- `因果类型` 只分 `实例因果` 与 `抽象因果`
- `条件短语列表` 用来表达为什么会这样
- `动作短语` 用来表达中间发生了什么
- `主结果短语` 用来表达这条因果最核心的果
- `成立次数 / 失败次数` 用来表达当前可参考程度


## 6. 因果槽位抽取规则

### 6.1 标准骨架

单条因果的标准表达骨架固定承接：

`同层条件状态集合 + 动作/事件 -> 同层结果状态或结果动态`

来源见 [1285_因果分层建模与实例生成规范.md](D:\鱼巢\规范\1285_因果分层建模与实例生成规范.md:94)。


### 6.2 条件抽取

条件短语抽取顺序固定为：

1. 优先取 `条件状态列表`
2. 若条件状态为空，可回退取条件动态或条件锚点中的稳定状态
3. 不允许凭文本方便性临时编造条件

若有多条条件，线性化时默认使用：

- `且`
- `并且`


### 6.3 动作抽取

动作短语抽取顺序固定为：

1. 优先取显式动作动态中的 `动作语义`
2. 若动作动态已补自然语言名，优先使用该名称
3. 若无显式动作语义，不得拿结果名冒充动作名

动作名称统一服从 [2000_自然语言词性词统一获取规则.md](D:\鱼巢\规范\2000_自然语言词性词统一获取规则.md)。


### 6.4 结果抽取

结果短语抽取顺序固定为：

1. 优先取 `主结果状态`
2. 若无主结果状态，再取 `结果状态列表`
3. 若结果状态仍为空，再取 `结果动态`

必须固定：

- 若已有 `主结果状态`，自然语言应优先围绕主结果组织
- 其他结果只作为补充结果显示


### 6.5 锚点抽取

锚点表达规则固定为：

1. `场景因果` 优先说场景上下文
2. `存在因果` 优先说存在自身上下文
3. `抽象因果` 默认视为世界通用模式，来源于世界树根链下 `因果信息` 子链
4. 抽象因果默认不强调单次锚点，而强调“这类情况”


## 7. 实例因果表达规则

### 7.1 语气

实例因果是运行时真实发生的因果事实，因此默认使用事实语气。

允许句型：

- `在{条件}下，{动作}导致了{结果}`
- `因为{条件}，所以{动作}产生了{结果}`
- `{动作}使{结果}成立`


### 7.2 最小输出要求

实例因果自然语言最少应包含：

1. 至少一个条件短语，或明确标记“无显式条件”
2. 一个动作短语
3. 一个主结果短语

推荐示例：

- `因为当前主任务为安全根任务，且当前方法为停止当前试探并进入保守态，所以当前下一步去向变为保守处理。`


## 8. 抽象因果表达规则

### 8.1 语气

抽象因果是归纳出的世界通用因果信息，不是当前事实。

因此必须使用归纳语气：

- `通常`
- `往往`
- `一般会`
- `在这类条件下`
- `可能`

不得写成：

- `现在就是`
- `必然已经`
- `本次已经发生`


### 8.2 最小输出要求

抽象因果自然语言最少应包含：

1. 条件短语或条件类型
2. 动作短语
3. 主结果短语
4. 成立次数与失败次数中的至少一项可见提示

并且必须继续固定：

- 抽象因果中的标量状态不得直接输出单次实例里的具体数值
- 抽象条件优先表达为 `高位 / 低位 / 稳定带 / 某一数值状态`
- 抽象结果优先表达为 `上升 / 下降 / 回升 / 回落 / 增大 / 减小 / 切换 / 变化` 这类方向短语
- 实例因果可以保留事实值，抽象因果不得把单次事实值伪装成世界通用模式

推荐句型：

- `通常在{条件}下，{动作}会导致{结果}`
- `这类情况下，{动作}往往会产生{结果}`

推荐示例：

- `通常在当前主任务为安全根任务，且当前方法为停止当前试探并进入保守态时，系统会把当前下一步去向调整为保守处理。`


## 9. 成立次数与失败次数的表达规则

当前阶段固定：

1. 抽象因果可信度只参考 `成立次数` 与 `失败次数`
2. 不额外引入新的复杂升格层
3. `成立次数 / 失败次数` 影响表达强弱和排序，不改变“它只是抽象因果”这一性质

推荐显示方式：

- `已成立 12 次，失败 1 次`
- `成立较多，失败较少`
- `目前证据仍少`

推荐语气对应关系：

1. `成立次数明显高于失败次数`：可用 `通常 / 往往`
2. `成立次数与失败次数接近`：可用 `可能 / 有时`
3. `失败次数高于成立次数`：应降级为 `暂不稳定 / 仅作参考`

必须固定：

- 即使 `成立次数` 很高，抽象因果也仍不是当前事实
- 即使 `失败次数` 很低，也不得把抽象因果表述成绝对真理


## 10. 线性化规则

### 10.1 条件拼接

若条件只有一条，直接表达：

- `因为{条件}`
- `在{条件}下`

若条件有多条，默认按以下顺序拼接：

1. 主条件
2. 补充条件
3. 环境条件

默认连接词为：

- `且`
- `并且`


### 10.2 结果拼接

结果表达优先级固定为：

1. `主结果短语`
2. 补充结果短语

若结果过多：

- 文本层只显示主结果
- 其他结果放到证据展开层


### 10.3 省略规则

允许做以下省略：

1. 当前页上下文已经明确锚点时，可省略重复锚点名称
2. 条件为空时，可省略“因为”结构，直接用 `{动作}使{结果}成立`

不允许做以下省略：

1. 把动作省掉，只剩“条件 -> 结果”
2. 把实例因果和抽象因果说成同一种语气
3. 把来源和证据完全省没


## 11. 输出层要求

控制面板、日志、自检输出中的因果自然语言，最少应保留：

```text
文本
因果类型
来源节点
证据节点
成立次数
失败次数
```

推荐显示分三层：

1. `标题层`：主结果或动作名
2. `单句层`：完整因果句
3. `证据层`：条件、动作、结果、来源、次数


## 12. 当前实现建议

当前代码实现建议统一走以下最小接口思路：

```text
因果节点
-> 生成因果表达单元
-> 生成因果自然语言文本
```

其中：

1. 状态节点先复用已有状态自然语言生成能力
2. 动作名称统一复用 `语素集.添加词性词(...)`
3. 抽象因果默认从世界树根链下 `因果信息` 子链读取
4. 控制面板与日志使用同一套因果文本生成器，不各自复制规则


## 13. 当前统一口径

当前阶段，因果信息自然语言表达统一收口为：

1. 先抽 `条件 / 动作 / 结果`
2. 再生成 `因果表达单元`
3. 实例因果按事实语气表达
4. 抽象因果按归纳语气表达
5. 抽象因果的可信度只看 `成立次数 / 失败次数`
6. 自然语言只做解释，不改写因果真值
