# 0000 规范总览与优先级

## 1. 总则

### 1.1 目标与作用

本手册收录 `鱼巢` 所有根约束、语义规则、工程实现规范。

其核心作用是回答：

- 先看哪份规范？
- 多份规范冲突时谁优先？
- 哪些属于不可学习、不可改写的根层？
- 哪些属于可落地、可成长的实现层？

所有规范文件统一按编号排序，预留 `100` 间隔，便于后续插入。

- `0000`：总览与裁决入口，不直接定义具体字段和算法。
- `0100–0999`：语义与行为主轴的上位根层与治理层。
- `1000–1198`：语义与行为主轴的实现承接层。
- `1200–1290`：工程与结构主轴。
- `2000+`：自然语言与表达专题。

也就是说：

- 从目录排序看，`0000–0999` 主要承载根目的、安全、需求、任务、方法、因果等上位语义。
- 从主轴理解看，真正的 `语义与行为主轴` 覆盖 `0100–1198`，其中 `1000–1198` 是更靠近代码实现的承接段，但仍然服从上位语义。
- `1200–1290` 单独构成 `工程与结构主轴`，负责把上位语义稳定落到运行时和工程结构中。

### 1.2 两条主轴（核心设计哲学）

`语义与行为主轴（0100–1198）`

解决“做什么”和“为什么”：

- 目标
- 方向
- 概念
- 需求
- 任务
- 方法
- 决策

必须严格遵守根约束，不可被实现方便性反向修改。

`工程与结构主轴（1200–1290）`

解决“如何稳定实现”：

- 模块分层
- 线程模型
- 锁策略
- 写回机制
- 宿主/内核分离

必须服务于语义主轴，不能以“能编译”为由违反根约束。

两条主轴必须同时成立：

- 行为正确的东西，必须工程上可长期稳定运行。
- 工程上能跑的东西，绝不能违反语义根约束。

### 1.3 优先级裁决规则（扩写重点）

编号越小，优先级越高，越接近根约束层。

因此必须固定以下原则：

- 编号越小 = 优先级越高。
- 上位规范不可被下位规范改写。
- 下位规范必须以上位规范为前提。
- 不得以“实现方便”或“临时 hack”否定上位规则。

冲突处理流程固定为：

1. 以上位规范为准。
2. 下位规范立即进入修订状态。
3. 记录冲突原因。
4. 提交 `规范/ Issue`。
5. 在修订完成前，不得以临时实现替代上位规则。

其中：

- `0500_根约束不变量清单.md` 属于最高优先级检查面。
- 任何模块、任何学习过程都不得触碰 `0500` 中已经固定的不变量。
- 每次重大重构前，必须逐条对照 `0100–0500` 根约束清单。

### 1.4 本文件定位

`0000` 的职责是：

- 给出全局阅读顺序。
- 给出上下位关系与冲突裁决口径。
- 区分根层、语义层、承接层、工程层。
- 说明哪些文档是稳定规范，哪些文档只是研究材料。

`0000` 不负责：

- 直接替代 [0100_根约束与可学习边界规则.md](D:\鱼巢\规范\0100_根约束与可学习边界规则.md)
- 直接替代 [0200_服务根目的与安全保障规则.md](D:\鱼巢\规范\0200_服务根目的与安全保障规则.md)
- 直接替代 [1200_模块分层规则与改造说明.md](D:\鱼巢\规范\1200_模块分层规则与改造说明.md)
- 直接发明新的字段、状态机、线程接口或算法参数

### 1.5 当前统一主链（总入口）

经过当前阶段规范收口，后续默认统一按以下主链理解系统：

`根层方向裁决`
-> `形成任务管理上下文`
-> `四域/调度/学习/因果/方法成长等承接层运行`
-> `结果写回对应实体`
-> `返回外层重新做根层重判与上下文刷新`

必须固定：

1. `四域方法树`、`任务调度`、`学习调度`、`因果生成`、`方法来源判定`、`方法成熟度升格` 都属于承接层，不是新的根裁决层。
2. 这些承接层只能回答“怎么组织当前治理轮”，不能越权替代根层方向裁决。
3. 任一承接层结果一旦会影响 `继续 / 停止 / 收束 / 待机 / 升格 / 回流`，都必须先回到外层重新做 `根层重判 -> 上下文刷新`。
4. 任一承接层中的最终判定，只能基于比较后形成的 `二次特征`；一次特征、原始状态、原始数值、方法返回值、各类主键/条件主键/摘要字段都只能作为输入真值面或辅助定位键，不得直接充当最终裁判依据。

当前阶段最关键的三个治理中间态固定为：

- `最近根层裁决结果`
- `当前缺口类型`
- `当前下一步去向`

必须继续固定：

- 这三项不是所有全域状态的统一强制字段。
- 但在 `任务管理方法`、`学习任务`、`学习调度`、`方法成长` 这条治理主链里，它们已经成为当前最小必备中间态。

### 2.4 工程与结构主轴：1200–1290

这一层回答“怎样长期稳定地把上位规则运行起来”。

它主要覆盖：

- 模块分层
- 线程池与线程拥有树
- 链表锁、线程锁、条件变量
- 类职责与函数归位
- 学习任务队列与学习调度
- 学习写回
- 宿主层与核心内核分层
- 宿主通道、设备运行时、自我线程运行时
- 因果模块实现

这一层 `可以改造、可以工程演进`，但必须满足两条硬边界：

- 不能反向改写语义主轴。
- 不能拿“当前实现省事”去突破根层。

### 2.5 扩展与草案带：1300 以后

`1300+` 可用于后续结构草案、下一阶段规范或大型扩展主题。

其中当前固定：

- `2000+` 预留给 `自然语言与表达专题`
- 该编号段用于承接“结构化世界如何外显为自然语言”的专门规范
- 该编号段仍然服从 `0000–1290` 的上位规则，不得反向改写真值层、根层和工程边界

这类文档默认不高于 `0100–1290` 已稳定规范。

若与稳定规范冲突：

- 以已稳定编号规范为准。
- 草案进入补证、修订或拆分流程。

## 4. 使用方式与维护流程（新增扩写）

### 4.1 日常开发

日常开发时，任何新功能、新模块、新字段、新流程修改，都必须先查规范，再写实现。

必须固定：

- 先判断本次修改属于哪一类问题：根层约束、语义解释、任务方法承接，还是工程运行时实现。
- 一旦定位到对应类别，必须从该类别中 `编号最低` 的规范开始查。
- 若问题跨越多类规范，则必须先看上位语义，再看承接实现，最后看工程落地。

可以收敛为一条开发纪律：

- `先查规范，后写代码；先看低编号，后看高编号。`

### 4.2 代码审查

代码审查时，PR 不应只说明“改了什么”，还必须说明“遵守了哪些规范”。

必须固定：

- 每个 PR 都应附带 `符合 XXX 规范` 声明。
- 声明中至少应列出本次改动直接依赖的上位规范编号。
- 若本次实现存在临时缺口、部分对齐或后续待补项，也必须在 PR 中显式说明，不得用沉默代替说明。

推荐最小写法：

- `符合 0100 / 0200 / 0550 / 1200 规范`

若存在例外情况，必须补充：

- 当前例外点是什么。
- 为什么尚未完全收口。
- 后续准备回收到哪一份规范或哪一条 Issue。

### 4.3 学习写回

学习写回属于高风险环节，因为它会把运行中成长结果沉淀回正式结构。

因此必须固定：

- 任何成长结果都不得违反 `0100–0500` 根约束。
- 任何写回前，都应先做一次根层对照，确认未触碰根目的、安全边界、生死规则与不变量。
- 一旦学习结果与根层冲突，不得直接写回为正式稳定结构。

冲突时只能进入以下路径之一：

- 降级为候选结果。
- 保留为局部假设。
- 记录为反例或边界样本。
- 直接丢弃，不进入正式写回。

### 4.4 冲突上报

当实现、学习结果、规范之间出现冲突时，不能只在代码里临时绕过，必须进入显性上报流程。

必须固定：

- 在 `规范/` 目录新建 Issue 记录。
- 标题格式统一为：`冲突上报-XXXX_规范名称`

其中：

- `XXXX` 为冲突涉及的主规范编号。
- `规范名称` 为对应规范短名。

Issue 内容至少应包含：

- 冲突涉及的规范编号与条目。
- 当前触发场景。
- 当前实现影响面。
- 暂行处理方式。
- 建议修订方向或待确认问题。

### 4.5 版本同步

每次规范更新后，不应只改当前规范文件，还必须同步更新对外编号清单。

必须固定：

- 每次规范更新后，都要在白皮书 `附录 A` 同步最新编号清单。
- 若新增、废弃、拆分、重命名规范，也必须同步反映到 `附录 A`。
- `附录 A` 应保持“当前稳定规范编号总表”的地位，避免白皮书与规范目录脱节。

### 4.6 最小维护闭环

整个维护流程可收敛为以下闭环：

1. 开发前先查对应低编号规范。
2. 实现后在 PR 中声明符合哪些规范。
3. 学习写回前再次校验 `0100–0500`。
4. 发现冲突后立即按格式上报。
5. 规范更新后同步白皮书 `附录 A`。

## 5. 根层与实现层的边界

### 5.1 不可学习、不可改写的根层

以下内容属于根层，不得交给学习过程决定，也不得由实现层反向改写：

- 是否以服务为根目的
- 安全为何是保障边界而不是替代根目的
- 根方向如何从目标值与当前值差异中推出
- 根需求为何是持续逼近型
- 哪些条目属于不变量

一句话说：

- `根层决定“允许什么、不允许什么、最终朝哪里对齐”。`

### 5.2 可落地、可成长的语义层

以下内容允许在证据积累、运行验证、经验沉淀中成长：

- 概念是否稳定成立
- 哪些因果仅局部成立
- 哪种任务分解更优
- 哪种方法更谨慎或更可靠
- 哪些四域方法树枝条值得升格

但这些成长结果只能：

- 细化路径
- 提高效率
- 提高稳定性

不能：

- 改写根目的
- 改写安全边界的地位
- 改写不变量

### 5.3 可落地、可演进的工程层

以下内容允许随架构演进而调整：

- 模块分层与依赖收敛
- 线程调度模型
- 锁顺序与条件变量用法
- 学习写回流程
- 宿主/内核分离方式

但工程调整必须满足：

- 上位语义不变。
- 行为闭环不丢。
- 运行时安全性不下降。

## 6. 阅读顺序与查阅入口

### 6.1 遇到新问题时先问什么

先问四个问题：

1. 这件事有没有碰到根目的、安全边界或不变量？
2. 这件事是在解释语义，还是在设计执行路径？
3. 这件事是在落任务/方法，还是在改线程/锁/模块结构？
4. 这次修改会不会让下位实现反向改写上位规则？

### 6.2 默认查阅顺序

默认按以下顺序阅读：

1. 先看 `0100–0500`，确认这件事是否被根层允许。
2. 再看 `0550–0999`，确认术语、表示、控制、证据、任务治理口径，以及当前问题处在统一主链的哪一段。
3. 再看 `1000–1226`，确认任务管理上下文、四域承接、学习承接、方法成长和因果承接怎么落地。
4. 最后看 `1200–1295`，确认模块、线程、锁、写回、宿主结构、因果模块和方法升格怎样稳定实现。

可收敛为一句话：

- `先判是否允许，再判语义怎么成立，再判执行如何承接，最后判工程如何稳定落地。`

## 7. 优先级裁决的正式口径

### 7.1 基本裁决顺序

当多份规范同时涉及同一问题时，按以下顺序裁决：

1. 先比是否为上位编号。
2. 再比是否属于根层。
3. 再比是否为同主题下更直接的专门规范。
4. 最后才讨论实现方便性。

固定原则是：

- `上位优先于下位。`
- `根层优先于成长层。`
- `语义合法性优先于实现方便性。`
- `工程稳定性优先于临时拼接。`

### 7.2 跨主轴冲突的裁决口径

若 `语义与行为主轴` 和 `工程与结构主轴` 出现张力，必须这样处理：

- 不能用“能编译、能运行、能上线”否定上位语义。
- 也不能用“语义上想这样”豁免线程、锁、宿主边界的工程硬约束。
- 正确做法是回到承接层与工程层重新设计，让两条主轴同时成立。

因此：

- 行为正确但工程不稳定，不算完成。
- 工程能跑但违反根约束，也不算完成。

### 7.3 正式冲突处理流程

发现冲突后，统一按以下流程处理：

1. 标出冲突涉及的规范编号与条目。
2. 判定上位规范与下位规范。
3. 以上位规范为准，立即冻结下位结论的继续扩散。
4. 记录冲突原因、触发场景、当前实现影响面。
5. 提交 `规范/ Issue`，进入修订流程。
6. 修订完成前，不得以临时注释、临时分支判断或临时 hack 作为长期替代。

### 7.4 重大重构前检查点

每次重大重构前，必须至少完成以下检查：

1. 逐条对照 `0100–0500`。
2. 明确本次重构是否触碰 `0500` 不变量。
3. 若触碰不变量，重构方案直接退回，不进入实现阶段。
4. 若不触碰不变量，再进入 `0550–1290` 的承接与工程设计。

## 8. 新增与修订规范的纪律

### 8.1 编号纪律

所有规范文件按编号排序，默认预留 `100` 间隔。

例如：

- 需要补根层细化规则，可插入 `0150`、`0250`、`0450`。
- 需要补任务控制细化规则，可插入 `0860`、`0955`。
- 需要补工程分层细化规则，可插入 `1212`、`1285`。

### 8.2 修订纪律

新增或修订下位规范时，必须显式说明：

- 它以上哪几份上位规范为前提。
- 它补的是哪一段空白，而不是改写哪一条上位规则。
- 它若与现有规范冲突，准备如何修订而不是如何绕过。

### 8.3 非编号文档的地位

`深研报告`、`差距分析`、`设计草案`、`实现说明` 等非稳定编号文档：

- 可以作为分析材料。
- 可以暴露问题和提出方案。
- 不可以直接覆盖稳定规范。

若其结论需要进入主体系，必须：

1. 回写到对应编号规范。
2. 明确编号位置。
3. 通过上位规范一致性检查。

## 9. 本文件的最终结论

后续所有规范编写、代码实现、重构裁决，都应默认遵守以下结论：

1. `0000` 是全局总览、阅读顺序和冲突裁决入口。
2. `0100–0500` 是根层，属于不可学习、不可改写区。
3. `0550–0999` 是语义展开层，负责把根层展开成概念、表示、控制和治理语义。
4. `1000–1198` 是语义承接层，负责把语义主轴落成任务执行、动作体系和方法成长结构。
5. 当前统一主链固定为：`根层方向裁决 -> 任务管理上下文 -> 各承接层运行 -> 结果返回外层根层重判`。
6. `最近根层裁决结果 / 当前缺口类型 / 当前下一步去向` 是当前治理主链最关键的三个中间态。
7. `1200–1295` 是工程与结构主轴，负责把上位语义稳定实现为模块、线程、锁、写回、宿主结构、因果模块与方法成长承接。
8. 编号越小，优先级越高；下位规范不得反向改写上位规范。
9. `0500` 根约束不变量属于最高优先级检查面，任何模块、任何学习过程不得触碰。
10. 每次重大重构前，必须逐条对照 `0100–0500` 根约束清单。
11. 不得以“实现方便”“临时 hack”“先跑起来再说”否定上位规则。
12. 两条主轴必须同时成立：`行为必须正确，工程必须稳定；工程可以落地，但不能违背根约束。`
