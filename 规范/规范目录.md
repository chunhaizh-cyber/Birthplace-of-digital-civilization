# 规范目录

## 编号规则

```text
一级规范编号：000-100。
    用于最基础、最原则性的系统规范。

二级规范编号：父级三位编号 + 子级三位编号。
    例如：001 号规范的第 1 个二级解释文件编号为 001001。

三级规范编号：父级六位编号 + 子级三位编号。
    例如：001001 号规范的第 1 个三级细则编号为 001001001。
```

优先级规则：

```text
1. 编号越短，层级越高。
2. 同一主题发生冲突时，父级规范优先于子级规范。
3. 同级规范发生冲突时，编号越小优先级越高。
4. 000-100 只放基础原则和最高层边界，后续细则必须挂到对应父级下面。
5. 历史入口、兼容说明、详细设计不得覆盖父级权威规范。
6. 上级规范修改后，必须全量检查并修订其全部下级规范。
7. 下级规范修订必须以上级规范为准，不得保留与上级冲突的旧口径。
```

本目录只建立优先级索引，不改现有文件名。

## 冲突处理规则

```text
1. 000-100 是最基础、最原则性的一级规范编号段。
2. 父级规范优先于子级规范。
3. 同级规范编号越小，优先级越高。
4. 历史入口规范只保留索引价值，不作为最新实现依据。
5. 详细设计类文档不得覆盖同主题的权威规范。
6. 子级解释文件必须挂到对应父级下面，避免重新平铺编号。
7. 上级规范发生任何实质修改后，必须从目录中找出全部直接和间接下级规范，逐份检查、修订或标记废止。
8. 下级规范如暂未完成修订，必须在文首注明“待按上级规范修订”，不得继续作为实现依据。
```

## 上级规范变更处理流程

```text
1. 修改上级规范。
2. 从本目录查找该编号下所有直接下级和间接下级。
3. 逐份检查下级规范是否仍与上级一致。
4. 一致：记录已检查。
5. 不一致：按上级规范修订。
6. 暂不能修订：在下级规范文首标记待修订，并说明冲突点。
7. 修订完成后更新计划或完成记录。
```
