# 0321 实际结果方向与需求方向对齐判定表规范

## 1. 目标

本文用于把 `实际结果方向` 与 `需求方向` 的最小对齐判定表单独固定下来。

本文要回答的是：

- 单特征条件下，执行后返回的 `实际结果方向` 与 `需求方向` 如何逐项对照
- 哪些组合允许判为本轮推进成立
- 哪些组合只能判为不成立、无推进或反向

## 3. 核心结论

必须固定以下结论：

1. `需求方向` 只能来自 `需求值 - 当前值`。
2. `实际结果方向` 只能来自 `返回值 - 当前值`。
3. 两者都只能落在：
   - `大于`
   - `小于`
   - `等于`
4. 单特征条件下，只有 `实际结果方向` 与 `需求方向` 对齐时，才允许判为本轮推进成立。
5. 当 `需求方向 = 等于` 时，`实际结果方向` 也只能是 `等于`。
6. 当 `需求方向 ≠ 等于` 时，`实际结果方向` 必须等于 `需求方向`。
7. `实际结果方向 = 等于` 且 `需求方向 ≠ 等于` 时，只能判为 `无推进`。
8. `实际结果方向` 与 `需求方向` 相反时，只能判为 `反向`，不得判为成立。

一句话收口：

`目标已完成时，只允许保持不变；目标未完成时，只允许向需求方向推进。`


## 4. 判定前提

在进入本表之前，至少必须先满足：

1. 当前比较的是 `同一特征类型`。
2. `需求方向` 与 `实际结果方向` 使用的是同一份 `当前值` 基准。
3. `当前值` 默认取该轮动作或方法执行前、用于形成需求方向的那一份比较基准值。
4. `需求方向` 已合法生成。
5. `实际结果方向` 已合法生成。

若上述任一条件不满足，则当前不允许进入正式对齐判定。


## 5. 单特征最小判定表

设：

- `需求方向 = sign(需求值 - 当前值)`
- `实际结果方向 = sign(返回值 - 当前值)`

则单特征最小判定表如下：

| 需求方向 | 实际结果方向 | 含义 | 判定 |
| --- | --- | --- | --- |
| `等于` | `等于` | 该特征目标已完成，返回结果也保持不变 | 成立 |
| `等于` | `大于` | 该特征原本已完成，但返回结果向上偏离 | 不成立 |
| `等于` | `小于` | 该特征原本已完成，但返回结果向下偏离 | 不成立 |
| `大于` | `大于` | 当前需求要求上升，返回结果也上升 | 成立 |
| `大于` | `等于` | 当前需求要求上升，但返回结果无推进 | 不成立 |
| `大于` | `小于` | 当前需求要求上升，但返回结果反向下降 | 不成立 |
| `小于` | `小于` | 当前需求要求下降，返回结果也下降 | 成立 |
| `小于` | `等于` | 当前需求要求下降，但返回结果无推进 | 不成立 |
| `小于` | `大于` | 当前需求要求下降，但返回结果反向上升 | 不成立 |


## 6. 三类不成立结果的固定解释

### 6.1 已完成但被执行结果破坏

- 条件：`需求方向 = 等于`，且 `实际结果方向 ≠ 等于`
- 含义：原本已对齐，但实际执行把其改偏

### 6.2 未完成但无推进

- 条件：`需求方向 ≠ 等于`，且 `实际结果方向 = 等于`
- 含义：实际执行没有把该特征向需求方向推进

### 6.3 未完成且方向相反

- 条件：`需求方向 ≠ 等于`，且 `实际结果方向 ≠ 需求方向` 且 `实际结果方向 ≠ 等于`
- 含义：实际执行把该特征往反方向推

必须固定：

1. 上述三类都不得判为本轮推进成立。
2. `无推进` 不等于 `可继续默认复用当前方法`。
3. `反向` 必须被视为更强的负反馈信号。


## 7. 与双值结算的关系

双值结算若消费执行结果，则必须先按本表完成单特征对齐判断。

结算层允许直接消费的不是原始返回描述，而是：

- `单特征对齐成立/不成立结果`
- 必要时的 `不成立类型`

必须固定：

1. 不能只因为“方法返回了结果”就判定任务推进成立。
2. 不能只因为“执行完成了”就判定服务或安全应正向加分。
3. 双值结算必须显式结合 `需求权重 + 实际结果方向 + 需求方向`。

更细的 `双值结算` 最小输入、结算顺序与禁止混层，按 [0322_双值结算最小输入结算顺序与禁止混层规范.md](D:\鱼巢\规范\0322_双值结算最小输入结算顺序与禁止混层规范.md) 执行。


## 8. 与回收型根层重判的关系

回收型根层重判若消费 `实际返回结果`，当前至少应先得到：

- `需求方向`
- `实际结果方向`
- `单特征对齐结果`

然后才允许把它继续作为上位输入，与：

- `安全方向`
- `服务方向`
- `根层禁止项`

一起进入外层重判。

因此必须固定：

1. 本表只负责 `实际结果方向` 与 `需求方向` 的单特征对齐。
2. 本表不是多特征综合裁决表。
3. 本表也不是回收型根层重判本身。


## 9. 多特征情况的当前口径

当一个需求由多个特征组成时，当前必须：

1. 先逐特征按本表判定。
2. 先标出 `否决项特征`。
3. 否决项先裁，普通特征再按权重和综合。
4. 再把综合结果提交更高层做最终裁决。

当前不得：

1. 跳过逐特征判定，直接做总判断。
2. 用普通权重和抵消否决项失败。
3. 把安全特征与服务特征混成单层黑箱比较。

更细的多特征综合规则，按 [0327_多特征需求权重汇总与否决项裁决规范.md](D:\鱼巢\规范\0327_多特征需求权重汇总与否决项裁决规范.md) 执行。


## 10. 严格禁止的混层

以下做法必须禁止：

1. 用 `方法执行成功` 直接代表推进成立。
2. 用 `结果主键` 直接代表结果方向对齐。
3. 用 `动作已执行` 直接代表结果已推进需求。
4. 用 `实际结果方向` 直接代替最终裁决。
5. 在未保证 `同特征类型 + 同当前值基准` 的情况下做方向对齐比较。
