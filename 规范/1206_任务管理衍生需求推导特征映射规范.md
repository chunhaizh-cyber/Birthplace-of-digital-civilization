# 1206 任务管理衍生需求推导特征映射规范

## 1. 目标

本文用于把 `任务管理任务` 中的 `派生需求` 重新收口到新的总模型上。

本文回答：

- 派生需求到底是什么
- 派生需求与特征空间是什么关系
- 为什么不能再把“从缺口反推特征”当成总模型
- 任务管理任务在什么阶段只负责上报派生需求


## 2. 核心结论

必须固定：

1. `派生需求` 不是特征的来源。
2. `派生需求` 是当前特征空间中已经显现出来、且具有行动价值的一类局部可见缺口。
3. `派生需求` 不是新的根需求，也不是任务管理任务可以直接落账的正式需求节点。
4. `任务管理任务` 只能上报派生需求，不得因为识别到派生需求就越权实例化正式子任务或正式需求树。
5. 新特征的发现应优先理解为 `理解驱动的特征空间生长`；派生需求只是这套特征空间里已经可见的局部行动凸起。

一句话收口：

`派生需求不是“特征的前因”，而是“特征空间中已经显现出来的局部行动缺口”。`


## 3. 总链位置

后续统一按以下顺序理解：

```text
经验/现象
-> 一阶/二阶/高阶特征
-> 可见缺口
-> 派生需求
-> 上报自我线程
-> 回执是否满足与由谁承接
```

必须固定：

1. `派生需求` 晚于 `高阶特征显现`。
2. `派生需求` 不得替代 `特征空间生长`。
3. `从缺口反推特征` 只允许作为局部筹办表达，不得再作为整套系统的总定义。


## 4. 任务管理任务的最小职责

`任务管理任务` 在本文中的职责只有三件事：

1. 读取当前宿主、当前步骤、当前结果以及任务/方法虚拟存在上的已有特征。
2. 识别哪些局部差异已经足够可见，可以形成派生需求。
3. 把派生需求上报给 `自我线程`，等待回执。

因此必须固定：

- 任务管理任务负责 `需求上报`
- 自我线程负责 `是否可满足 + 是否需要等待 + 返回谁承接`


## 5. 允许上浮派生需求的事实面

当前允许直接触发 `派生需求候选` 的事实，至少包括：

- 当前目标状态所需前置条件缺失
- 当前方法不存在或不可用
- 当前步骤所需资源未到位
- 当前步骤所需确认未到位
- 当前外部依赖结果未回流，且已超过本地可等待边界
- 当前局部失败已显式表明需要外部支撑
- 当前步骤被阻塞，且阻塞原因不属于本地可重试或本地可等待

必须补充固定：

1. 这些都是 `局部治理运行事实`。
2. 它们只回答“本地为什么过不去”，不直接回答“系统现在必须生成什么正式对象”。


## 6. 派生需求与局部等待的边界

必须继续固定：

- 已有承接、只差回流 -> `局部等待`
- 没有承接、缺口仍在外面 -> `派生需求`

因此：

1. 若当前只是等待已有外部结果短时回流，应优先落为 `局部等待`。
2. 只有当“缺口不能靠本地继续推进，也不能靠已有等待态自然吸收”时，才应上浮 `派生需求`。


## 7. 与特征空间生长的边界

必须固定：

1. `派生需求` 不负责发明新特征。
2. 但 `派生需求` 可以反向刺激特征空间生长。
3. 当系统发现同类派生需求反复出现时，应进一步上浮为：
   - 新的一阶区分维度
   - 新的二阶治理解释
   - 新的三阶成长优先级特征

例如：

- 反复出现 `方法不可用`，不应只重复发 `缺方法` 需求，还应促使系统长出 `method_availability`、`method_reuse_confidence` 等特征类型。
- 反复出现“知道要做但不清楚验收”，不应只重复发局部补条件需求，还应促使系统长出 `acceptance_clarity`。


## 8. 禁止项

禁止以下做法：

1. 把 `派生需求` 当成新特征的唯一来源。
2. 把 `派生需求` 当成任务管理任务本地可直接落账的新正式任务。
3. 让任务管理任务绕过自我线程，自行决定“这个派生需求一定能满足”。
4. 继续把 `从缺口反推特征` 写成总模型。


## 9. 最终收口

`1206` 现在只收口“哪些局部可见缺口可以被上浮成派生需求”。特征空间的发现、命名、组合与生长，统一以上位的理解环规范为准；派生需求只是其中已经进入可行动层的一段，不再承担全部特征生成职责。`
