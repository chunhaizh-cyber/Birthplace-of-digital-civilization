## 3. 核心结论

必须固定以下结论：

1. `派生需求` 不是新的根需求，也不是任务管理任务可以自行落账的正式需求节点。
2. `派生需求` 当前应只理解为：`当前步骤要达成目标状态，但其所需前置状态无法在本地满足时，上报出去的缺口表达`。
3. `任务管理任务` 可以从 `局部缺条件 / 局部失败 / 局部阻塞 / 方法不可用 / 外部依赖未回流 / 缺确认 / 缺资源` 等事实中，上浮 `派生需求消息`。
4. `派生需求` 只能先上浮到外层，由 `自我线程` 或上位协调层决定是否升格、是否创建新任务入口、是否转入学习承接。
5. `任务管理任务` 自己不能因为识别到 `派生需求`，就直接实例化正式子任务、改正式需求树、或越权判定“必须学习”。
6. 若当前局面只需要等待已有外部结果，不应硬上浮 `派生需求`；应优先落为 `局部等待 + 线程状态消息`。
7. 若根层已经判定当前路径应停止、收束或保持边界，普通的扩选、覆盖、健壮等增长型派生需求不得继续按常规上浮。
8. 当前 `1206` 只保留“能上浮派生需求，但仍回外层决定”的窄版结论；旧版 `八维压力系统` 不再作为当前 `P0` 主驱动内容。

一句话收口：

`当前衍生需求只是一种“本地无法消化的目标状态缺口”上报，不是任务管理任务可直接落账的新正式任务。`


## 4. 最小来源事实面

当前允许直接触发 `派生需求候选` 的事实，至少包括以下几类：

- `当前目标状态所需前置条件缺失`
- `当前方法不可调用，且本地无可替代方法`
- `当前步骤所需资源未到位`
- `当前步骤所需确认未到位`
- `当前外部依赖结果未回流，且不是短时等待即可吸收`
- `当前局部失败已显式表明需要外部支撑`
- `当前步骤被阻塞，且阻塞原因不属于本地可重试或本地可等待`

必须补充固定：

1. 这些都是 `局部治理运行事实`，不是根层最终裁决。
2. 这些事实只回答“为什么本地这一步过不去”，不直接回答“系统现在必须创建什么正式对象”。


## 5. 最小派生需求模板

当前 `派生需求消息` 最少应能表达以下内容：

```text
目标存在 E
目标特征 F
目标状态 G

当前缺口事实 D
缺失前置状态集 P
本地为何无法继续 K
建议外部支撑方向 S
```

建议进一步固定：

1. `D` 回答“当前卡在什么局部事实上”。
2. `P` 回答“缺哪组前置状态”。
3. `K` 回答“为什么不能继续当成本地等待或本地重试处理”。
4. `S` 只是一种支撑方向建议，例如 `补条件 / 补资源 / 补确认 / 补方法 / 转学习`，不是正式落账指令。


## 6. 与局部等待的边界

当前必须先区分 `局部等待` 与 `派生需求`：

- 若外部依赖已存在，只是尚未回流，应优先落为 `局部等待`。
- 若当前只是等待执行结束、等待学习结果、等待外部确认短时回流，也应优先落为 `局部等待`。
- 只有当“缺口不能靠本地继续推进、也不能靠已有等待态自然吸收”时，才应上浮 `派生需求`。

可压成一句话：

`已有承接、只差回流 -> 等待；没有承接、缺口仍在外面 -> 派生需求。`
