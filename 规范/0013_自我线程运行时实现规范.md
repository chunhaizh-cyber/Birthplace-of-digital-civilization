## 2. 定位

当前 `自我线程类` 的定位应固定为：

`面向需求信息的控制面治理中枢 + 主循环运行时承载器 + 任务管理工作线程调度者`

它在总架构中的工程地位，应统一理解为：

- `主循环` 的运行时承载者
- `根需求形成`、`需求树 / 需求队列 / 方法禁止项集` 的统一整理、排序与权重修正入口
- `任务根节点 / 任务实例入口` 的统一创建者
- `任务管理任务工作线程` 的调度者与刹车位
- `子结果回线程 -> 父任务结果路由 -> 最终结算` 这条链的工程协调器
- `线程状态事件协议` 的唯一拍板方

它不是：

- 根约束定义器
- 宿主监督线程
- 后台前端服务线程
- 相机或外设运行时

### 2.1 最新统一主链中的位置

按当前最新体系，自我线程必须服从以下主链：

```text
自我线程心跳
-> 消费状态 / 事件 / 结果回流消息
-> 形成根需求并刷新需求树与最小根层判断输入
-> 按父需求权重调制子需求权重
-> 生成需求队列并计算综合权重
-> 根据安全值 / 服务值变化趋势生成和更新方法禁止项集
-> 刷新当前主需求
-> 创建任务根节点和任务实例入口
-> 装配任务管理上下文并调度任务管理工作线程
-> 任务管理任务在只读任务根节点约束下推进步骤
-> 返回局部结果 / 派生需求 / 方法结果
-> 自我线程统一接收结果并决定是否实例化新任务
-> 自我线程更新全局优先级、路由结果给父层任务并做最终结算
-> 自我线程再次重判根层与当前功能域
```

必须固定：

1. 自我线程可以触发和承载运行时中的 `根层重判`，但它不是根语义定义来源。
2. 自我线程直接治理的对象是 `需求信息`，其最小运行时承载至少包括 `需求树 / 需求队列 / 方法禁止项集`，而不是叶子任务、叶子方法或单个工作线程。
3. 自我线程承担的是 `控制面职责`，而不是 `任务完成面职责`。
4. `任务管理任务` 只负责在 `只读任务根节点` 约束下推进步骤并返回结果；它可以提出 `派生需求`，但不能直接生成子任务实例。
5. 自我线程负责把根层结论、需求队列摘要、方法禁止项摘要和任务管理上下文装配到请求里，再交给后续任务管理工作线程承接。
6. 子结果必须先回到自我线程，再由自我线程路由给父层任务；父层任务判断局部影响，自我线程判断全局影响。
7. 自我线程不应绕过 `外层重判 -> 再提交` 这条链，直接在同一轮里替后续线程连续推进多轮治理闭环。
8. 自我线程每个循环必须先冻结“本轮开始时已经可见的消息批次”，并把这批消息及其同步内部闭包处理完，再做一次根层重判与主派发/收束决议。
9. 本轮处理中异步新产生的消息不得追入当前轮次，而应进入下一轮；高优先级控制消息可按抢占规则打断本轮。
10. `控制消息` 与 `全局环境消息` 必须先进入 `自我线程`；明确绑定到具体任务且仅影响局部流程的外部反馈，可先进入对应任务管理工作线程。
11. `结果回流消息` 必须先回 `自我线程`；`推进消息` 只上报会改变控制面判断的部分。
12. `控制态真值` 由 `自我线程` 持有；任务管理工作线程只接收与自身相关的 `控制投影`。
13. `需求树 / 需求队列` 的消息驱动增量刷新，属于 `自我线程` 内部可直接完成的最重控制面工作。
14. `需求树 / 需求队列` 的定时全局刷新，不得在线程主循环内长时间执行，而必须由 `自我线程` 发起为一个任务，并交给任务管理工作线程完成。
15. `需求列表` 是 `需求队列` 的线程持有形态；它由自我线程从 `需求树` 的 `链上 / 链下` 生成，并按 `需求类型聚拢 + 类型组权重汇总 + 权重排序` 规则维护。

### 2.2 任务管理的三层口径

当前运行时文档里，必须统一以下命名层级：

- `架构层`：自我线程内部的一级治理模块，统一称 `任务管理中轴`
- `运行层`：这一轮由中轴承接的治理实体，继续称 `任务管理任务`
- `实现层`：若代码里落为类或模块，统一建议称 `任务管理模块类`

补充固定：

- `策略沙盘与方案演练` 只属于实现层可调用的二级策略子模块
- 它负责多方案比较，不直接拥有根方向裁决权
- 当前工程中的 `任务管理任务模块.*` 可视为过渡承载，不改变上面这三层口径


## 3. 当前运行时车道

当前实现中，自我线程至少包含以下几条运行时车道：

- `主消息/心跳车道`
  - 处理主队列
  - 驱动双值刷新、根层重判、根需求形成和根任务就绪判断
  - 维护 `可运行队列 / 等待队列 / 回流队列 / 收束停止队列`
  - 装配最小任务管理上下文并决定是否实例化任务入口
- `任务管理工作线程车道`
  - 推进普通任务和管理任务
  - 在 `V0` 形态下同步承接“提交后的一步治理或执行落地”
  - 返回局部结果、步骤状态和派生需求候选
- `学习队列车道`
  - 推进常驻学习任务
  - 执行 `采样 -> 更新 -> 验证 -> 提交 / 回滚`
- `延迟任务事实车道`
  - 暂存热点路径产生的任务事实
  - 在低频路径统一整理并写回
- `任务树与结果路由车道`
  - 串行保护任务根节点 / 任务入口实例化
  - 串行保护子结果路由给父层任务

这些车道之间更细的状态流转和事件交通规则，按 [0027_自我线程与任务管理工作线程状态流转与事件协议_v0.1规范.md](D:\鱼巢\规范\0027_自我线程与任务管理工作线程状态流转与事件协议_v0.1规范.md) 执行。

这几条车道必须理解成：

- 同属一个运行时
- 但职责必须分层
- 不得因为“都在自我线程里”就互相直接越权改结构

### 3.1 轮次与消息批次语义

自我线程的轮次语义必须固定为：

`冻结一批，清完这一批，再做一次决议。`

具体规则如下：

1. 每轮开始时，自我线程必须先冻结“本轮开始时已经产生并被本轮看见的全部消息”。
2. 本轮冻结消息集必须在本轮内处理完，不得把这批消息的人为尾巴拖到下一轮。
3. 本轮还必须处理完这些冻结消息在本轮内同步引发的内部消息闭包，例如：
   - 子结果到线程后的结果路由
   - 父层任务因收到子结果而产生的同步局部状态更新
   - 由同步路由直接引发的派生需求上报
4. 本轮处理中异步新产生的消息，不属于当前轮次闭包，应进入下一轮。
5. 本轮完成“冻结消息 + 同步内部闭包”后，主循环只做一次根层重判，并最多做一次主派发或收束决议。
6. 高优先级控制消息，如 `打断 / 收束 / 停止 / 强制优先级刷新 / 强制预算刷新`，可按抢占规则打断当前轮次。
7. `控制消息` 和 `全局环境消息` 先进入 `自我线程`；明确绑定到具体任务且仅影响局部流程的外部反馈，可先进入相关任务管理工作线程。
8. `局部推进消息` 留在工作线程内部，只有会改变 `调度 / 路由 / 结算 / 优先级 / 是否实例化任务` 的关键推进事件，才需要上报线程。

更细的消息分类、首收件人和处理方法，按 [0028_自我线程消息总表规范.md](D:\鱼巢\规范\0028_自我线程消息总表规范.md) 执行。

若进一步把这些消息正式落成 `协议 / 路由 / 处理入口 / 批次执行骨架` 四块代码合同，按 [0029_自我线程消息协议与分发规范_v0.1.md](D:\鱼巢\规范\0029_自我线程消息协议与分发规范_v0.1.md) 执行。

若进一步需要把消息世界压成四张代码对照矩阵，直接回答“是什么消息 / 谁先收 / 会改谁 / 会触发什么大动作”，按 [0030_自我线程消息类型表与路由矩阵_v0.1.md](D:\鱼巢\规范\0030_自我线程消息类型表与路由矩阵_v0.1.md) 执行。

若进一步需要把这组消息规范真正翻成当前仓库的实现顺序、桥接策略和模块改造计划，按 [0031_自我线程消息模块实现方案与计划_v0.1.md](D:\鱼巢\规范\0031_自我线程消息模块实现方案与计划_v0.1.md) 执行。

工程实现上，推荐采用：

- `双缓冲邮箱`
- `本轮内部同步工作队列`

也就是：

- `收件箱_A` 持续接收新异步消息
- `收件箱_B` 作为本轮冻结消息集
- 主循环先清空 `B`
- 再清空本轮同步内部队列
- 两者都为空后才允许进入本轮根层重判与收尾


## 4. 主循环职责

当前主循环的职责应固定为：

- 处理主队列消息
- 维持心跳驱动
- 刷新最小根层判断输入
- 形成根需求
- 更新需求树
- 按父需求权重调制子需求权重
- 生成需求队列并刷新当前主需求
- 从需求树生成并维护线程持有的需求列表
- 对消息驱动的新增需求、需求状态变化做增量刷新
- 按需求队列综合权重评估安全值 / 服务值变化趋势
- 生成和更新方法禁止项集
- 执行运行时中的根层重判
- 创建任务根节点和任务实例入口
- 装配和刷新任务管理上下文
- 决定是否提交 `任务管理请求`
- 调度任务管理工作线程、学习线程启动
- 接收派生需求并决定是否实例化新任务
- 路由子任务结果给父层任务
- 协调低频整理、消息回投与学习唤醒
- 完成本轮最终结算与学习触发判断

必须补充固定：

- 自我线程内联完成的需求治理，只允许是 `受影响范围增量刷新`，不得扩张为长时间全量扫描。
- 定时或周期性的 `需求树 / 需求队列全局刷新`，必须被实例化为 `刷新任务`，由任务管理工作线程执行。
- 自我线程对全局刷新任务只负责：`判断是否需要刷新 -> 创建任务入口 -> 赋予优先级和预算 -> 接收刷新结果 -> 决定是否替换当前需求列表/需求队列`。

更细的主循环职责项、统一顺序和“哪些必须留在主循环”的边界，按 [0014_自我线程主循环最小职责表规范.md](D:\鱼巢\规范\0014_自我线程主循环最小职责表规范.md) 执行。

必须继续固定：

- 这里的“决定是否提交 `任务管理请求`”，应理解为当前运行时里最接近 `最终裁决层` 的工程承载动作。
- 它必须建立在 `条件判定结果 + 二次特征 + 根层重判结果` 之上。
- 其统一边界按 [0567_最终裁决层与外层重判边界规范.md](D:\鱼巢\规范\0567_最终裁决层与外层重判边界规范.md) 执行。

主循环不应直接承担：

- 叶子步骤的具体执行
- 任务步骤的局部展开
- 局部流程完成与否的判断
- 沿步骤树直接生成子任务实例
- 方法逻辑本体
- 需求树或需求队列的长时间全量遍历与重建
- 改写只读任务根节点
- 学习阶段内部每一步的细节
- 宿主重启监督
- 在线程承载层内部偷偷连续推进多轮治理闭环


## 4.1 初始化阶段（新增硬约束）

`自我线程类::自我初始化()` 必须承担以下顺序化动作：

1. 初始化自我与基础特征承载。
2. 统一注册默认本能动作集（含方法首节点创建/复用与方法信息回填）。
3. 初始化初始需求与任务。
4. 刷新初始化完成状态并进入运行态。

必须固定：

1. 本能函数“首次注册”属于初始化阶段职责，不得延迟到任务执行/学习执行/周期维护中再补做。
2. 注册完成是“可执行前置条件”，未完成注册不得标记运行时初始化完成。
3. 注册不等于执行：初始化阶段不要求逐个执行本能函数本体。


## 5. 动作线程职责

当前动作线程入口的职责应固定为：

- 从执行队列中取任务
- 将任务置为 `执行中`
- 在结构调整锁保护下调用当前 `V0` 入口，在 `只读任务根节点` 约束下推进一步
- 接收 `结构_任务返回消息`
- 返回局部步骤结果、局部完成状态和派生需求候选
- 不直接生成子任务实例

必须固定：

1. 动作线程只消费 `执行队列`。
2. 动作线程的结构改动入口，统一收束到 `推进一步 + 返回消息`。
3. 动作线程可以把事实排队到延迟写回车道，但不应在热点路径里做重整理。
4. 动作线程不负责根层方向裁决；它只承接已由主循环装配好的请求或已入队的执行工作。
5. 当前若仍通过 `根任务筹办器_.推进一步` 同步直调，也只能视为 `1205` 所述线程池模型的 `V0 单线程退化承载`。
6. `继续提交下一轮请求 / 停止提交 / 转入学习 / 进入收束 / 允许回流 / 允许升格` 这类最终裁决，不属于动作线程职责，统一由主循环在外层重判后决定。
7. `是否局部完成 / 是否局部失败 / 是否局部重试 / 是否继续下一步骤` 这类局部流程判断，属于父层任务管理实例职责，不属于自我线程职责。
8. `派生需求` 必须先返回给自我线程，再由自我线程判断是否实例化为新任务。


## 6. 学习线程职责

当前学习线程的职责应固定为：

- 只消费 `学习队列`
- 特判常驻 `学习任务节点_`
- 推进学习阶段机
- 在更新阶段调用 `私有_尝试学习一步`
- 在提交阶段唤醒等待学习的方法和任务

当前学习线程应理解为：

- 已经能真正执行
- 但仍属于专用学习工作流
- 还不是完全并入普通叶子步骤执行器的最终形态

因此必须固定：

1. 学习线程内部的阶段推进，是学习专用运行时逻辑。
2. 学习线程的产出，主要是方法树更新、候选补全和等待任务唤醒。
3. 学习线程不直接替代任务调度器。
4. 学习线程回流后，仍应回到主循环做 `根层重判 -> 上下文刷新 -> 再提交`，不得自己越权续推主闭环。


## 7. 延迟事实与低频整理

当前实现已经存在“动作线程只记最小事实，扩展整理移到低频路径”的口径。

因此应固定：

- 热路径允许只写最小回执、关键事实和最小方向特征
- 复杂投影、对齐、账本补齐和大规模同步，应由低频整理路径完成

延迟事实队列的职责应固定为：

- 承接热点路径不宜立刻落地的任务事实
- 在更低争用的时间点统一冲刷到任务虚拟存在


## 8. 本模块锁与顺序

当前模块至少存在以下锁：

- `故障锁_`
- `mtx_`
- `执行队列锁_`
- `学习队列锁_`
- `延迟任务事实锁_`
- `根任务结构调整锁_`

本模块的默认规则应固定为：

- 默认单锁化
- 能不双持就不双持
- `故障锁_` 视为独立故障账本锁，默认不与其它锁同持

若确实无法避免多锁同持，本模块的局部顺序应固定为：

1. `mtx_`
2. `执行队列锁_ / 学习队列锁_`
3. `延迟任务事实锁_`
4. `根任务结构调整锁_`

必须固定：

1. 队列等待时，只允许持有对应队列锁。
2. 持有 `根任务结构调整锁_` 时，不得再去等待条件变量。
3. `try_to_lock` 只能作为受限的拥塞退避手段，不得作为主正确性机制。


## 9. 启停与拥有关系

当前自我线程必须服从以下拥有关系：

- `自治宿主` 拥有 `自我线程`
- `自我线程` 拥有 `动作线程` 与 `学习线程`

因此必须固定：

1. 自我线程只负责启动和回收自己的直属子线程。
2. 动作线程和学习线程不得反向控制宿主。
3. 停机应遵守 [1230_多线程嵌套启停管理规范.md](D:\鱼巢\规范\1230_多线程嵌套启停管理规范.md)。


## 10. 自我线程的硬约束

1. 主循环负责“决定与派发”，动作线程和学习线程负责“消费与推进”。
2. 任何任务结构改动，都应尽量通过统一的返回消息和结构调整入口落地。
3. 热路径事实回写应最小化，重整理必须延后。
4. 学习线程是专用学习运行时，不得误当成普通动作线程的别名。
5. 任何队列等待都必须带谓词，并遵守 [1240_线程内部锁与条件变量使用规范.md](D:\鱼巢\规范\1240_线程内部锁与条件变量使用规范.md)。
6. 本模块的本地锁顺序必须可见，不能依赖“写代码的人自己记住”。
7. 线程生命周期状态变化和关键运行时状态变化必须形成日志。
8. 主循环每轮在提交任务管理请求前，应先完成最小根层重判与任务管理上下文刷新。
9. 动作线程或后续线程池每轮只承接一步，不得在承载层内部连跑多轮治理。
10. 任何一步推进后的结果，都应先回到主循环或外层协调器，再进入下一轮判定与提交。
