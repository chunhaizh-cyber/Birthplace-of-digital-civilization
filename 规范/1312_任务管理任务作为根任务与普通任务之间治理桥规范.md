# 1312 任务管理任务作为根任务与普通任务之间治理桥规范

## 1. 目标

本文用于固定当前系统中 `任务管理任务` 的桥接定位，回答以下问题：

- `任务管理任务` 与 `安全根任务 / 服务根任务 / 普通任务` 的关系是什么
- `任务管理任务` 为什么既不是根任务，也不是系统外的管理超体
- 从根需求到普通任务执行、再到特征变化与双值结算，中间到底由谁承接
- `任务状态改变 / 方法状态改变 / 因果证据生成 / 特征值变化` 分别应落在哪一层
- `任务管理任务` 哪些能直接写，哪些只能形成候选与证据并上浮外层

## 3. 核心结论

必须固定以下结论：

1. `任务管理任务` 仍然是任务，不是系统外的管理超体，也不是新的根层。
2. `任务管理任务` 的标的是 `宿主任务`，它回答的是：`当前怎样把宿主任务继续推进到更接近完成。`
3. 在当前统一主链中，`任务管理任务` 的正式定位可以收口为：`安全/服务真根任务` 与 `普通任务执行链` 之间的 `治理桥与过程承接层`。
4. `安全根任务 / 服务根任务` 负责承载总方向、常驻分解与持续重筹办；`任务管理任务` 负责把这种方向落实为当前宿主任务上的具体治理步骤、方法挂点、前置任务、结果节点与因果证据。
5. `任务管理任务` 的主执行面，当前只保留两个正式承接段：`筹办` 与 `执行`；其中 `目标` 负责定标，不再展开成另一套中间过程层。
6. `任务管理任务` 的 `筹办` 负责完善任务执行流程，也就是把宿主任务补到 `可继续执行`：补步骤、补结果锚点、补方法挂点、补前置任务承接、补执行入口闸门。
7. `任务管理任务` 的 `执行` 负责消费当前已回执的治理步骤与执行入口闸门，推动宿主任务前进一步，并把治理结果写入结果节点。
8. `普通任务` 负责承载实际业务推进链；其正式结构仍是 `头结点 -> 步骤节点 -> 结果节点`，前置任务与派生任务继续按 [1300_任务树数据结构与步骤子任务执行规范.md](D:\鱼巢\规范\1300_任务树数据结构与步骤子任务执行规范.md) 执行。
9. `学习任务` 是 `任务管理任务` 派生并继续治理的内部治理子任务；它的标的是 `方法`，其职责是 `完善方法`，而不是替代宿主任务本身的执行链。
10. 因此，`任务管理任务` 主要负责 `完善任务执行流程`，`学习任务` 主要负责 `完善方法本体`；两者分工固定，不得混写。
11. `任务管理任务` 可以推动 `任务状态改变`、`当前步骤改变`、`当前方法挂点改变`、`最近结果改变`、`局部等待与回筹办状态改变`，也可以生成中间因果证据与方法候选，但这些变化多数属于 `治理中间态 / 过程层影响 / 候选证据`。
12. `任务管理任务` 不得直接把 `安全需求` 或 `服务需求` 写成 `安全值 / 服务值` 的最终增量；双值变化仍必须经 `叶子需求首次满足 -> 外层正式结算` 这条链成立后才能落账。
13. `任务管理任务` 生成的因果、方法建议、下一步去向、收束建议、重筹办建议，都只是 `候选与证据`，不得直接越权冒充最终裁决。
14. `方法状态改变` 在当前阶段主要指 `当前方法挂点 / 可用候选 / 替代建议 / 学习承接建议 / 可靠度候选证据 / 谨慎度候选` 这类治理相关方法状态；`方法已验证 / 方法成熟 / 方法允许升格` 仍须走学习与验证链路，不得由任务管理任务单独拍板。
15. 除学习任务的专门例外外，绝大多数普通业务任务都应理解为 `安全根任务` 或 `服务根任务` 子树上的任务；学习任务在语义上仍来源于这两棵根任务，但结构上可按 [1223_任务管理学习树独立挂载规范.md](D:\鱼巢\规范\1223_任务管理学习树独立挂载规范.md) 独立挂到 `任务管理学习树`。

一句话收口：

`任务管理任务` 负责把 `根需求/根任务` 对应的总方向，翻译成 `宿主任务当前这一轮该怎样筹办与执行`；其中方法本体的完善继续交给 `学习任务`，而双值结算和最终裁决仍属于外层正式规则链。


## 4. 统一链条

### 4.1 从根需求到双值变化的统一主链

当前必须把整条链固定理解为：

```text
安全根需求 / 服务根需求
-> 安全根任务 / 服务根任务
-> 任务管理任务
-> 宿主任务树上的步骤 / 方法 / 前置任务 / 结果节点
-> 叶子需求首次满足
-> 外层主循环做结果回挂与正式结算
-> 安全值 / 服务值变化
```

必须固定：

1. 根需求定义总方向，不直接执行。
2. 真根任务持续分解并承载常驻方向，不直接充当叶子执行节点。
3. `任务管理任务` 承接的是从“总方向”到“当前局部推进”的翻译层。
4. 在这条翻译层内部，`任务管理任务` 主要分为 `筹办` 与 `执行` 两部分。
5. 正式双值结算的真值单位仍是 `叶子需求首次满足`，不是 `任务管理任务跑过一次`，也不是 `某个方法被调用过一次`。

### 4.2 从治理过程到因果与方法候选的桥接链

当前还必须把治理侧内部链条理解为：

```text
宿主任务当前状态
-> 任务管理任务显性化缺口
-> 长出治理步骤 / 方法挂点 / 前置任务需求
-> 必要时派生学习任务以完善方法
-> 执行当前治理步骤
-> 生成结果节点与过程证据
-> 形成候选因果 / 方法候选 / 下一步去向候选
-> 外层重判、学习、验证或再次提交
```

这条链回答的是：

- 当前到底缺什么
- 该先补结构、补条件、补方法，还是继续执行
- 当前方法为什么被选中或被替换
- 当前是否已经需要派生学习任务去补方法
- 当前结果为什么支持继续、等待、回筹办或转学习

### 4.3 执行停滞时的统一收口

当前必须再固定一个更底层的判断：

`任务执行不下去时，通常不是先抽象成“问题很多”，而是先抽象成“当前因果链未成”。`

必须固定：

1. `形成因果链` 是 `筹办` 的正式本能职责之一。
2. 因此，只要工作线程执行卡住，第一承接面都应先回到 `筹办`，而不是直接把“等待”当最终结论。
3. `筹办` 解决问题的统一方式，是把“目标 -> 当前条件 -> 可用方法 -> 中间结果 -> 下一步去向”补成可继续推进的因果链。
4. 若缺口属于宿主任务内部可补的步骤、方法、条件、结果锚点或学习承接，则记为 `虚拟存在缺口`，由 `筹办` 直接补链。
5. 若缺口属于现实世界支撑不足，例如外部资源、外部确认、现实信息或现实场景尚未具备，则仍先记为“因果链未成”，但处理动作应是 `筹办外化现实支撑缺口 -> 上报自我线程 -> 由自我线程创景/建新需求/建新任务入口 -> 再回到筹办链`。
6. 因此可以说 `筹办` 是所有执行停滞问题的统一治理入口；但这不等于 `筹办线程自己直接完成所有现实支撑动作`。

一句话收口：

`筹办` 负责把一切“执行不下去”重新翻译成“因果链哪里还没接上”，内部缺口就地补链，现实缺口则外化并交给自我线程补齐后再回链。


## 5. 四层分工

| 层级 | 回答的问题 | 主要承载者 | 典型内容 | 明确不包含 |
| --- | --- | --- | --- | --- |
| `根需求 / 真根任务层` | `系统总方向是什么、这一侧长期还缺什么` | `安全根需求 / 服务根需求` 与 `安全根任务 / 服务根任务` | 总方向、常驻分解、根面观察、持续重筹办 | 局部执行细节、单步等待原因、方法临时替换细节 |
| `任务管理任务层` | `当前怎样推进宿主任务` | `任务管理任务树` 与 `任务管理上下文` | 当前缺口、治理步骤、当前方法挂点、前置任务外化、结果节点、因果证据、一步治理结果 | 根方向最终裁决、双值最终结算、方法成熟度最终拍板 |
| `宿主任务 / 普通任务层` | `这个具体任务当前执行到哪一步、有哪些前置任务和结果` | `宿主任务树` | 头结点、步骤节点、结果节点、前置任务、派生任务、当前步骤、最近结果 | 根层方向裁决、全局调度预算、双值结算 |
| `外层主循环 / 结算层` | `是否继续、如何路由结果、是否结算双值` | `自我线程 / 主循环 / 正式结算链` | 根层重判、结果回挂、需求树定位、双值结算、调度与派发许可 | 具体治理步骤内部细节 |

必须固定：

1. `任务管理任务` 是桥，不是顶替上下两层。
2. 它上接 `根需求 / 根任务` 的方向与分解，下接 `宿主任务 / 方法 / 结果 / 因果证据` 的局部推进。
3. 只要结论会影响 `继续 / 停止 / 收束 / 待机 / 升格 / 双值结算`，最终都必须回到外层重判与正式规则链。


## 5.1 任务管理任务与学习任务的固定分工

当前必须把二者的分工固定为：

```text
任务管理任务
-> 负责完善任务执行流程
-> 主要承接：筹办、执行

学习任务
-> 作为任务管理任务的内部治理子任务
-> 负责完善方法
```

必须固定：

1. `任务管理任务` 先判断当前宿主任务是否缺方法、缺到什么程度、是否需要学习承接。
2. 一旦确认需要补方法，本轮应由 `任务管理任务` 创建、复用或承接相应 `学习任务`。
3. `学习任务` 的直接工作对象是 `方法`，不是宿主现实对象。
4. `学习任务` 完成后，结果必须继续回到 `任务管理任务` 与外层主循环消费，不能脱离宿主任务独立冒充最终裁决。


## 5.2 自我线程与任务管理运行时的交接

当前必须把线程交接链固定为：

```text
自我线程
-> 创建或实例化任务入口
-> 通过消息/请求提交给任务管理承接层或实例 mailbox
-> 任务管理运行时按任务创建或唤醒工作线程
-> 工作线程单步推进任务
-> 结果、阻塞、异常、派生需求再回到自我线程
```

必须固定：

1. `自我线程` 负责根层重判、决定是否创建任务、创建哪类任务，以及把任务或任务请求正式提交下去。
2. `自我线程` 不直接长期持有局部推进权；它负责 `创建 / 提交 / 派发 / 路由 / 结算`，不负责在本层展开步骤细节。
3. 任务一旦被正式创建，后续应以 `消息 / 请求 / mailbox` 形式交给 `任务管理运行时承接层`，而不是靠全局盲扫被动发现。
4. `任务管理运行时承接层` 负责根据任务创建、复用或唤醒对应工作线程，并让该任务获得一次 `推进一步` 的执行机会。
5. 工作线程只负责局部推进，不负责根层判断、最终裁决或最终结算。
6. 工作线程产生的 `局部结果 / 最终结果 / 阻塞事实 / 异常事实 / 派生需求候选`，都必须先回 `自我线程`，再由自我线程统一路由和消费。

### 5.2.1 术语边界

必须额外固定：

1. 若某线程负责 `展示任务管理信息 / 展示工作线程列表 / 汇总消息 / 承接人工控制意图`，它应继续叫 `任务管理界面线程`，且不得持有推进权。
2. 若某线程或承接层负责 `接收任务消息 / 建立实例 / 创建或唤醒工作线程 / 安排推进`，它不应再叫 `界面线程`，而应理解为 `任务管理承接层 / 实例调度层 / 任务管理运行时承接线程`。
3. 不得把 `界面线程` 与 `任务推进承接线程` 混成同一个术语，否则会把观察层和运行层混写。
4. 当前若口头上说“把任务发给任务管理界面线程，由它创建工作线程并推进任务”，规范落账时应改写为：
   - `把任务发给任务管理运行时承接层`
   - `由承接层创建或唤醒工作线程并推进任务`


## 6. 任务管理任务必须承接的桥接面

### 6.1 任务状态桥接

`任务管理任务` 必须显式承接以下任务状态桥接：

1. `当前步骤前沿`
2. `当前等待原因`
3. `条件满足状态`
4. `依赖等待状态`
5. `派生需求挂起状态`
6. `局部结果缓存`
7. `局部重试计数`
8. `当前治理状态迁移`
9. `当前步骤 / 当前方法 / 最近结果` 的镜像更新

这些字段回答的都是：

`宿主任务现在推进到哪里了，还卡在哪，下一步更像往哪走。`

### 6.2 方法状态桥接

`任务管理任务` 在当前阶段至少要承接以下方法相关状态：

1. `当前方法挂点`
2. `备用方法候选`
3. `方法替换建议`
4. `是否建议转学习`
5. `方法可靠度候选证据`
6. `方法谨慎度候选`
7. `当前方法需求位`
8. `当前本能能力缺口类型`

必须固定：

1. 这些多数属于 `治理证据与候选`，不是正式成熟度真值。
2. `任务管理任务` 可以推动“当前该用哪个方法”的局部变化，也可以决定“是否转学习任务补方法”，但不能单独宣布“该方法已被长期验证”为正式成熟方法。
3. `学习任务` 才是当前正式负责“补动作 / 补条件 / 补结果 / 验证 / 尝试学习”的方法完善子任务。

### 6.3 因果桥接

`任务管理任务` 必须承担“过程证据到候选因果”的桥接责任。

它至少要能把以下材料串起来：

1. 输入条件
2. 当前缺口
3. 当前治理动作
4. 当前方法
5. 当前结果
6. 目标对齐状态
7. 最近根层裁决结果
8. 当前下一步去向候选

必须固定：

1. 这里生成的是 `候选因果证据`、`链桥接信息` 或 `治理专用实例因果原料`。
2. 因果模块当前定位仍是 `证据仓 + 检索索引 + 方法生成期筛选原料层`，不直接给出最终继续/停止结论。
3. 因果命中只能支持 `候选方法 / 候选解释 / 候选去向`，不得直接升格成“已验证方法”。

### 6.4 特征变化桥接

`任务管理任务` 必须把“某个方法最终改变了什么”桥接清楚，但它桥接的是 `变化链`，不是直接跳写最终值。

当前必须按以下顺序理解：

```text
治理步骤
-> 方法执行或治理动作执行
-> 结果节点成立
-> 宿主状态 / 宿主特征 / 方法相关特征出现变化
-> 外层根据需求树与结果回挂做正式消费
-> 必要时进入叶子需求首次满足与双值结算
```

因此必须固定：

1. `任务管理任务` 可以把“方法最终改变了哪些状态、哪些特征、哪些结果节点”显性化。
2. 但它不能直接把这种变化越权写成 `安全值 +n` 或 `服务值 +n`。
3. 若某个方法改变的是世界上的一般特征值、状态值或宿主状态，`任务管理任务` 负责把这类变化与当前需求、当前任务、当前结果节点对应起来。
4. 若这种变化来自尚不成熟的方法缺口，则应优先交给 `学习任务` 去补齐该方法，而不是让 `任务管理任务` 自己长期兼任方法完善器。
5. 若这种变化进一步导致叶子需求首次满足，则外层才能据此进入正式结算链。


## 7. 一个安全侧示例

例如：

`安全根需求` 需要某个 `安全特征状态` 从 `不满足` 变成 `满足`。

当前统一链应理解为：

```text
安全根需求
-> 安全根任务确认仍存在安全缺口
-> 任务管理任务接管当前宿主任务
-> 显性化当前缺口与当前步骤
-> 绑定或补齐当前方法
-> 若方法运行中发现额外支撑缺口，则外化前置任务
-> 执行当前治理步骤
-> 写回结果节点与候选因果证据
-> 外层根据需求树与结果回挂确认叶子需求是否首次满足
-> 若首次满足成立，再进入安全值正式结算
```

这里必须明确：

1. `任务管理任务` 负责中间治理过程。
2. `安全特征状态` 的变化可以由方法执行带出，但是否足以代表 `安全根需求` 当前被满足，仍须经外层按正式规则判断。
3. 即使治理结果很强，也不能直接绕过 `叶子需求首次满足` 与 `正式结算` 这两道链条。


## 8. 严格禁止的混层

以下做法必须禁止：

1. 把 `任务管理任务` 说成系统外的总控超体。
2. 把 `任务管理任务` 直接等同于 `安全根任务` 或 `服务根任务`。
3. 把 `任务管理任务` 单轮推进后的结果，直接写成 `安全值 / 服务值` 的最终增量。
4. 把 `当前下一步去向候选 / 收束建议 / 重筹办建议 / 学习建议` 写成已获批准的最终裁决。
5. 把 `候选因果` 直接写成 `已验证方法`。
6. 把 `任务管理任务` 的局部推进态与 `任务控制态`、`任务真值层`、`双值结算层` 混写。
7. 用“某个方法执行过一次”替代“叶子需求首次满足”作为结算依据。
8. 用“某个结果节点主键存在”替代“需求已满足 / 双值应变化 / 主链已结束”。


## 9. 当前阶段工程落点

当前阶段工程上应按以下方式落地：

1. `普通任务树` 继续按 `头结点 -> 步骤节点 -> 结果节点` 组织，步骤由筹办阶段生成，前置任务由执行方法暴露缺口后经自我线程回挂。
2. `任务管理任务` 自己也必须显式长成一棵真实治理任务树，记录治理步骤、治理结果和下一治理步骤。
3. `一步治理结果` 应继续作为 `任务管理任务` 交给外层主循环的最小稳定返回面，而不是最终裁决。
4. `任务管理任务` 对 `宿主任务`、`当前步骤`、`当前方法`、`最近结果`、`候选因果证据`、`下一步去向候选` 的桥接，应优先显式化为结构字段或稳定返回字段，不再隐藏在临时调用链里。
5. 当前若要说明“某个任务管理步骤最终改变了哪个特征值”，应优先通过 `目标状态 / 当前状态 / 结果节点 / 结果方向二次特征 / 相关宿主特征` 这一组桥接字段表达，而不是直接用自由文本跳写结论。
6. 当前若要表达线程交接，应优先写成 `自我线程创建并提交任务 -> 任务管理运行时承接层创建或唤醒工作线程 -> 工作线程单步推进 -> 结果回自我线程`，不再把 `界面线程` 当成推进线程使用。


## 10. 结论

可以把本文压成下面这句话：

`任务管理任务` 是 `根任务方向` 与 `普通任务执行链` 之间的正式治理桥；它负责把缺口、步骤、方法、前置任务、结果、因果证据和下一步候选连起来，但最终裁决、结果回挂与双值结算仍属于外层主循环和正式规则链。
