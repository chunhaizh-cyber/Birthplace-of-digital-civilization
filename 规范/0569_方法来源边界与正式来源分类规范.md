# 0569 方法来源边界与正式来源分类规范

## 1. 目标

本文用于固定以下问题：

- `方法来源` 到底回答什么，不回答什么
- 当前正式允许的 `方法来源` 应分成哪几类
- `自我动作动态 / 外界因果 / 观察结果 / 学习过程` 与正式 `方法来源` 的边界是什么
- 旧口径中的 `运行时学习沉淀` 还能不能继续作为正式来源使用

本文只收口 `方法来源` 的正式分类与边界，不直接讨论方法成熟度、可用性、可执行性和最终裁决。

## 3. 核心结论

必须固定以下 13 条：

1. `方法来源` 回答的是“这个方法主要怎么来的”。
2. `方法来源` 不等于 `方法存在`、`方法可用`、`方法可执行`、`方法成熟度`。
3. `方法来源` 是正式身份标签，不是证据桶标签。
4. 一个方法可以同时有多种证据来源，但正式来源标签默认只写 `主出生来源` 或 `主提炼来源`。
5. `自我动作动态`、`外界因果`、`外部回执`、`观察结果` 属于 `证据来源层`，不是正式方法来源标签。
6. 当前正式允许的 `方法来源` 至少包括：`本能`、`外部导入`、`因果抽象`、`观察抽象`、`路径提炼`。
7. `运行时学习沉淀` 只允许作为过渡期兼容总类或旧字段兼容值，不再作为新规范中的首选正式来源标签。
8. `因果抽象` 指方法头的主出生依据来自较明确的因果证据抽象。
9. `观察抽象` 指方法头的主出生依据来自重复观察、对比、现象模式抽象，而不是来自已闭合的稳定因果。
10. `路径提炼` 指方法头的主出生依据来自稳定步骤链、稳定任务路径或稳定任务子树的结构提炼。
11. 学习过程默认不再单独构成新的正式方法头出生来源；学习主要负责对已出生方法头做补条件、补结果、验证和升格。
12. 正式来源一旦写入，默认不因成熟度上升而自动改写；只有初次分类明显错误，或从旧兼容值迁移到精确来源值时，才允许受控修订。
13. `方法来源` 不是重点成长对象；规范上不单独建设一条独立的“来源升格链”。

一句话收口：

`方法来源回答“怎么来的”，证据来源回答“靠什么知道的”；正式来源必须精确分类，不能再把所有运行时后天方法都混写成“运行时学习沉淀”。`


## 4. 正式方法来源总表

| 正式方法来源 | 含义 | 典型原料 | 主要边界 |
| --- | --- | --- | --- |
| `本能` | 系统初始化即具备、先天注册的方法 | 本能动作、本能方法头 | 回答“先天就有”，不回答成熟度 |
| `外部导入` | 外部显式提供并注册的方法定义 | 开发者导入、宿主导入、外部配置 | 回答“外部明确注入”，不回答当前是否适用 |
| `因果抽象` | 以较明确的因果证据为主，抽象出方法结构 | 因果实例、稳定因果对、轻量因果链 | 主导依据是因果，不是单纯观察或任务路径 |
| `观察抽象` | 以重复观察、对比结果、现象模式为主，抽象出方法结构 | 多次观察结果、对比模式、回执模式 | 允许当前未形成稳定因果闭合 |
| `路径提炼` | 以稳定步骤链、任务路径、任务子树为主，提炼出方法结构 | 成功片段、稳定路径、稳定任务子树 | 主导依据是结构和顺序，不是单点因果 |
必须补充固定：

- `需求驱动 / 目标驱动` 若只是表示“先有目标或治理缺口，再去状态列表中按同类特征筛选因果/观察/路径证据，再组织候选方法出生”，它属于方法生成的触发条件与桥接路径，不单独构成新的正式方法来源标签。
- 这类方法最终仍必须按主导出生依据，归入 `因果抽象 / 观察抽象 / 路径提炼` 之一。


## 5. 相邻概念边界

### 5.1 正式方法来源与证据来源

必须固定：

- `正式方法来源` 回答“方法头的主出生归因是什么”。
- `证据来源` 回答“支撑这次出生或修订的原料来自哪里”。
- 一个方法只有一个正式来源标签，但可以保留多个证据来源记录。

例如：

- `自我动作动态`
- `外界因果`
- `外部回执`
- `状态比较结果`

这些都更适合写在证据侧、学习侧或审计侧，而不是直接冒充正式来源。

### 5.2 正式方法来源与方法成熟度

必须固定：

- `来源` 回答“怎么来的”。
- `成熟度` 回答“现在有多可信、允许怎样被默认使用”。
- 来源不会因为从 `运行期临时` 升到 `已验证稳定` 就自动改变。

### 5.3 正式方法来源与方法可用 / 可执行

必须固定：

- `来源=外部导入` 的方法，当前也可能不可用或不可执行。
- `来源=观察抽象` 的方法，当前也可能已经可执行。
- 因此来源不承担“当前能不能调用”的语义。

### 5.4 正式方法来源与方法存在

必须固定：

- 只要已经形成正式方法头，就可以写正式来源。
- 当前不可用、不可执行，不影响方法来源存在。
- 正式来源的写入前提是“方法头已正式存在”，不是“当前已能执行”。

### 5.5 正式方法来源与后续成长

必须固定：

- `方法来源` 主要服务身份归因、学习归档、回写解释、审计追踪和粗粒度成长方向参考。
- `方法来源` 不参与根层裁决，不直接决定需求是否满足，也不直接决定是否继续推进。
- 因此 `方法来源` 默认应保持稳定，而不是被设计成持续成长的主链对象。
- 后续若只是补证、补边界、补稳定性，应优先保留原来源，不额外改源。


## 6. 旧口径兼容与迁移

当前工程里若仍存在 `运行时学习沉淀` 这类旧字段或旧展示口径，应按以下方式理解：

1. 它只代表“后天运行期出生”，不再代表精确正式来源。
2. 新规范新增或重写来源时，应优先写成以下精确值之一：
- `因果抽象`
- `观察抽象`
- `路径提炼`
3. 无法立即判断精确来源时，可暂时保留旧兼容值，但应进入 `待精确化修订` 状态。

必须进一步固定：

- 这里的 `修订` 只指必要修订，不等于独立的来源成长链。
- 当前优先允许的修订只有两类：
  - 原来写错了
  - 旧兼容值需要迁到精确来源值
- 除此之外，默认只补证、不改源。

建议的最小迁移口径：

- 以因果实例、稳定因果对、轻量因果链为主时，优先迁到 `因果抽象`
- 以重复观察、对比现象、外部回执模式为主时，优先迁到 `观察抽象`
- 以成功片段、稳定步骤链、任务路径、任务子树为主时，优先迁到 `路径提炼`
- 若旧描述曾写成“学习过程综合生成新头”，应回溯其主导出生依据，优先迁到 `因果抽象 / 观察抽象 / 路径提炼`


## 7. 收口结论

后续统一按以下口径执行：

1. 正式方法来源至少分为 `本能 / 外部导入 / 因果抽象 / 观察抽象 / 路径提炼` 五类。
2. `自我动作动态 / 外界因果 / 观察结果 / 回执` 只作为证据来源，不直接充当正式方法来源。
3. `运行时学习沉淀` 不再作为新规范中的首选正式来源值，只保留过渡兼容意义。
4. 正式来源写在方法身份侧，成熟度、可用性和可执行性仍按各自分层单独判定。
5. 正式来源默认稳定，只允许必要修订，不单独建设来源升格链。

最终统一收口为：

`来源要写精确出生归因，证据要写支撑原料；方法从哪里来，不等于方法现在能不能用、能不能执行、是不是已经成熟。`
