# 0970 任务筹办规则

## 1. 目标

本文用于固定一条唯一口径：

`筹办的目的，就是完善任务树。`


## 2. 核心结论

必须固定以下结论：

1. 所有任务都遵循 `目标 -> 筹办 -> 执行`。
2. `筹办` 不是整个任务控制，也不是执行的别名。
3. `筹办` 的职责只有一个：让当前任务拥有一棵可执行的任务树。
4. `筹办` 的产物是主步骤节点、结果节点和方法实例准备度；前置任务不在筹办阶段直接生成。
5. `筹办` 不直接宣布目标达成；目标达成只能来自执行后的结果比对。
6. `筹办` 只负责识别缺口并提出需求；`需求是否可满足` 统一交给 `自我线程` 回执。
7. `少什么就提什么需求` 只是筹办层规则；更上游仍然是 `理解环 -> 特征空间生长 -> 可见缺口 -> 需求形成`。
8. `步骤任务` 在筹办时产生；`前置任务` 在执行方法运行时产生，并由自我线程实例化后挂回当前步骤。


## 3. 筹办的最小动作

筹办阶段最少应完成：

1. 明确当前任务目标。
2. 选择或创建当前步骤节点。
3. 为当前步骤创建结果节点。
4. 绑定或创建当前步骤的方法头。
5. 依据方法模板创建任务实例条件节点和结果节点。
6. 当当前推进需求已经可见时，把它长成主步骤链上的下一步骤，而不是塞成杂项子任务。
7. 尽量把以上动作统一收成：`少什么就提什么需求 -> 等待自我线程回执 -> 把返回节点挂回任务树`。
8. 执行方法运行中若发现额外前置缺口，只负责提出前置任务需求，不在筹办阶段提前伪造前置任务头。

必须补充固定：

- 上面这条只描述筹办层动作，不代表“需求是特征的唯一来源”。


## 4. 三类任务的筹办区别

### 4.1 普通任务

筹办要补的是：

- 指向现实存在的步骤树
- 围绕当前目标持续生长的主步骤任务链

### 4.2 任务管理任务

筹办要补的是：

- 任务管理任务本体步骤树
- 宿主任务的待治理步骤树
- 筹办需求集合，以及由自我线程回执返回的步骤、方法和结果

### 4.3 学习任务

筹办要补的是：

- 学习任务自己的步骤树
- 待学习方法的最小可用骨架

补充固定：

- 学习任务中的主步骤，同样由筹办建立。
- 学习方法执行过程中若发现还缺样本、边界或验证材料，这些才属于前置任务需求。


## 5. 学习任务的特别约束

学习任务的最后一步才是 `尝试学习`。

在此之前，筹办必须先让学习对象可以执行，至少补齐：

1. 动作骨架
2. 条件侧
3. 结果侧
4. 当前学习步骤与当前学习结果节点

因此：

- “方法不可用” 不直接进入空转学习
- 必须先由筹办把学习任务树和待学习方法骨架补到可执行


## 6. 何时回到筹办

满足任一条件就应回到筹办：

1. 当前没有步骤。
2. 当前步骤没有结果节点。
3. 当前步骤没有方法。
4. 执行后目标未达成。
5. 当前路径已失效。
6. 自我线程尚未明确回执当前需求是否可满足。
7. 执行阶段产出的前置任务尚未完成，导致当前步骤无法继续推进。


## 7. 禁止项

禁止以下做法：

1. 把 `维护 / 纠偏 / 收束` 当成与筹办并列的任务主流程。
2. 把 `找方法 / 找条件` 当成比任务树更高的筹办定义。
3. 把“方法存在”直接等同“当前步骤可执行”。
4. 在没有步骤节点和结果节点时直接进入执行。
5. 让当前任务本地直接裁定“当前需求已经可满足”，绕过自我线程。
6. 在筹办阶段提前把执行中才会显现的前置任务硬生成为主步骤。
