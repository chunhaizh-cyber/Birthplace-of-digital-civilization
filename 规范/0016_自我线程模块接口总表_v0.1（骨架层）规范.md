本文用于把“主循环骨架”正式落成一份当前可执行的详细设计版 `接口总表 v0.1`。

这版只回答以下问题：

## 3. 当前底座与设计前提

当前 `v0.1` 骨架层必须建立在以下已存在承载面之上：

1. `链级承载` 已存在：  
   `世界链 -> 特征值链 -> 需求链 -> 任务链 -> 方法链`

2. `存在级挂点` 已存在：  
   `需求根 / 任务根 / 方法根` 已经可以挂在统一主体上，适合承接需求出生、任务挂接和方法声明。

3. `场景级快照面` 已存在：  
   `状态索引 / 动态索引 / 关系索引 / 实例因果索引` 已经可以承接“本轮世界看到了什么、变化了什么、关联了什么、留下了什么因果证据”。

4. `回流证据面` 已存在：  
   当前动态与因果实例结构已经能承接 `来源动作、输入场景、输出场景、来源任务主键、来源方法主键、置信度、已验证状态` 等回流所需字段。

因此，当前最应该补的不是新概念，而是：

- 主循环骨架的模块边界
- 读写权限
- 触发条件
- 停止 / 让渡 / 回流条件

必须固定：

`挂钩点已经在，当前阶段优先补接口边界，而不是继续扩概念。`


## 4. v0.1 的范围与边界

### 4.1 本版只覆盖骨架层

`v0.1` 只覆盖以下 `7` 个模块：

1. `生命周期与边界控制`
2. `世界与状态刷新`
3. `服务/安全差额评估`
4. `根需求生成`
5. `优先级与冲突仲裁`
6. `任务管理一步治理`
7. `回流结算与主循环重判`

### 4.2 本版先不展开承接层

本版暂不把以下模块单独展开：

- `方法匹配与选择`
- `执行调度与过程监督`
- `安全守卫与止损`
- `学习与因果沉淀`

这不表示它们不重要，而是表示：

- 当前阶段先证明骨架层能闭环
- 承接层先以 `任务管理一步治理` 的输出需求位保留挂点
- 不让自我线程在这一版被写成一锅沸腾的总线汤

### 4.3 本版的最小闭环目标

本版至少要把以下主链做实：

`需求识别 -> 一步治理 -> 结果回流 -> 主循环重判`

并把它落到底座上：

`世界链 / 场景索引 -> 需求根 -> 任务根 -> 方法根声明 -> 动态/因果回流 -> 重判`


## 5. 核心结论

必须固定以下结论：

1. 主循环骨架先不负责具体算法，而先负责治理顺序和接口边界。
2. `世界与状态刷新` 只负责把世界照清楚，不负责裁决。
3. `差额评估` 只负责算清差额和趋势，不负责直接发动作。
4. `根需求生成` 只负责把差额翻成一级根需求，不负责任务拆解。
5. `优先级与冲突仲裁` 是当前骨架层里最核心的治理裁决位，负责“谁主导、谁让位、是否接管、是否收束”。
6. `任务管理一步治理` 只负责把主导根需求翻译成当前唯一有效的一步治理，不负责长期规划替代主循环。
7. `回流结算与主循环重判` 不是日志存档，而是统一结算、更新差额并决定下一轮入口。
8. 本版虽然不单列 `方法匹配 / 执行监督 / 学习沉淀` 模块，但必须在 `任务管理一步治理` 的输出里显式保留 `所需方法描述 / 授权需求 / 回流要求`。
9. 安全接管在 `v0.1` 中先由 `生命周期与边界控制` 和 `优先级与冲突仲裁` 共同承接，不得因为尚未拆出独立安全模块，就把“安全优先”降成口号。

一句话收口：

`v0.1 先把主循环骨架写实，不先把承接层写满。`

## 7. 骨架层的当前挂接口径

### 7.1 世界看见面挂在场景索引

`世界与状态刷新` 当前优先挂接：

- `状态索引`
- `动态索引`
- `关系索引`
- `实例因果索引`

这意味着本轮“看见世界”的真值来源，应优先来自场景侧已经挂住的事实和变化，而不是临时拼接的局部缓存。

### 7.2 需求与任务挂在统一主体根节点

`根需求生成` 和 `任务管理一步治理` 当前优先挂接：

- `需求根`
- `任务根`
- `方法根`

必须固定：

- `需求根` 承接“这一轮为什么要管”
- `任务根` 承接“这一轮怎么管一步”
- `方法根` 承接“这一步预计需要什么方法承接”

### 7.3 回流优先写回动态与因果

`回流结算与主循环重判` 当前优先使用：

- 动态中的来源动作、输入场景、输出场景、成功/失败与错误信息
- 因果实例中的来源任务主键、来源方法主键、置信度、已验证状态

必须固定：

`回流不是记一条“做过了”，而是留下足以支撑下轮重判的状态变化和因果证据。`


## 8. 骨架层主循环顺序

当前 `v0.1` 骨架层的主循环顺序固定为：

`生命周期与边界控制 -> 世界与状态刷新 -> 服务/安全差额评估 -> 根需求生成 -> 优先级与冲突仲裁 -> 任务管理一步治理 -> 回流结算与主循环重判`

这条顺序的意义是：

- `生命周期与边界控制` 先回答“这一轮有没有资格继续”
- `世界与状态刷新` 再回答“这一轮到底看到了什么”
- `服务/安全差额评估` 接着回答“世界相对目标差了多少”
- `根需求生成` 把差额翻成可治理对象
- `优先级与冲突仲裁` 回答“这轮到底先管谁”
- `任务管理一步治理` 回答“只往前走哪一步”
- `回流结算与主循环重判` 回答“这一步算不算推进，以及下一轮从哪重新进入”


## 9. 两条最小验证路径

### 9.1 服务主导闭环

```text
生命周期与边界控制
-> 世界与状态刷新
-> 服务/安全差额评估
-> 根需求生成（服务需求主导）
-> 优先级与冲突仲裁
-> 任务管理一步治理
-> 回流结算与主循环重判
-> 下一轮入口
```

这条路径至少要证明：

- 系统能从世界快照里提炼出服务差额
- 服务需求能进入 `需求根`
- 能形成当前唯一有效的一步治理
- 结果能回到动态/因果面结算
- 下一轮能据此重判，而不是停在局部线程

### 9.2 安全接管闭环

```text
生命周期与边界控制
-> 世界与状态刷新
-> 服务/安全差额评估
-> 根需求生成（安全需求显性化）
-> 优先级与冲突仲裁（安全接管）
-> 回流结算与主循环重判
-> 下一轮入口 = 降级运行 / 收束 / 停止
```

这条路径至少要证明：

- 风险安全或物理安全差额超阈值时，服务不再默认主导
- 安全接管可以在未展开独立安全模块前，先通过仲裁与生命周期模块落地
- 接管结果会进入下一轮入口状态，而不是只留在局部判断里


## 10. 本版不负责的内容

- `方法匹配与选择` 的独立模块化
- `执行调度与过程监督` 的独立模块化
- `安全守卫与止损` 的独立模块化
