# 0015 自我线程模块接口总表规范

## 1. 目标

本文用于在继续细分模块内部实现之前，先把 `自我线程` 的模块接口边界固定成一份总表。

当前阶段不先拆内部算法，而先回答以下问题：

- 自我线程主线上到底分成哪 `12` 个模块
- 每个模块最小只负责什么
- 每个模块读什么、写什么、能调用谁、不能越权改什么
- 各模块在什么条件下触发、在什么条件下必须停止 / 让渡 / 回流
- 用哪两条金路径证明“服务主导闭环”和“安全接管闭环”已经成立

本文服务的工程目标是：

- 先把 `需求识别 -> 方法调用 -> 结果回写` 连成最小强闭环
- 先把“谁负责什么、谁能改什么、谁不能改什么”钉死
- 先证明主循环能完成 `判 -> 发一步治理 -> 收回流 -> 重判`
- 在接口边界稳定之前，不提前摊开各模块内部算法

## 4. 核心结论

必须固定以下结论：

1. 当前阶段应走 `小切口、强闭环` 路线，先把最小治理闭环跑通，再展开模块内部。
2. `自我线程` 的价值首先是 `优先级判断、资源分配、纠偏和收束`，它是治理框架，不是算法杂货铺。
3. `12` 个模块应先按接口分工固定，再进入实现细分；接口边界先于内部细节。
4. `任务、方法、学习` 都只能调路径，不能改根方向；根方向、双值结算和最终去向仍属于主循环与根层重判。
5. `任务管理一步治理` 是治理中轴，不是全能裁决器；它只承接本轮最值得推进的一步。
6. `安全守卫与止损` 必须是显式模块，而不是隐藏在各处的零散 if；否则“安全优先”无法落成工程事实。
7. `学习与因果沉淀` 当前只先占接口位，不先深挖内部；它只能补路径，不得提前冲入根层改地图。
8. 若没有 `服务主导闭环` 与 `安全接管闭环` 两条金路径，本规范簇不能宣称“自我线程主循环已经活起来”。

一句话收口：

`先固定模块接口总表，再验证两条金路径，最后才逐步拆模块内部。`

## 6. 调用顺序与权力边界总纪律

必须固定以下总纪律：

1. 常规服务推进链默认按以下顺序运行：  
   `生命周期与边界控制 -> 世界/状态刷新 -> 服务值/安全值差额评估 -> 根需求生成 -> 优先级与冲突仲裁 -> 任务管理一步治理 -> 授权/配额/限幅 -> 方法匹配与选择 -> 执行调度与过程监督 -> 结果回流、结算与重判（先结算） -> 学习与因果沉淀 -> 主循环下一轮正式重判`

2. 安全接管链默认允许从任一阶段切入：  
   `任一模块 -> 安全守卫与止损 -> 结果回流、结算与重判（先结算） -> 学习与因果沉淀（仅记录止损证据） -> 生命周期与边界控制 / 主循环下一轮正式重判`

3. 学习补路径链默认只作为承接链存在：  
   `任务管理一步治理 / 方法匹配与选择 / 已结算结果证据包 -> 学习与因果沉淀 -> 主循环下一轮正式重判`

4. 只有以下模块拥有轮级裁决权：
   - `生命周期与边界控制`
   - `优先级与冲突仲裁`
   - `结果回流、结算与重判`

5. `任务管理一步治理` 拥有一步治理权，但不拥有根方向改写权和最终真值裁决权。

6. `授权/配额/限幅`、`方法匹配与选择`、`执行调度与过程监督`、`学习与因果沉淀` 全部属于路径承接层，只能回答“怎么推进更合适”，不能回答“根层要不要这么做”。

7. 任何模块一旦命中以下情况，必须停止本地扩张并让渡或回流：
   - 输入不足
   - 权限不足
   - 风险超阈值
   - 需要等待外部条件
   - 已形成一步结果
   - 已到达本模块边界

8. 表中未列入的 `写回对象` 一律视为默认禁止写回；若后续要新增写回面，必须增补规范。


## 7. 两条金路径

### 7.1 金路径 A：服务主导闭环

这条路径用于证明系统能在服务主导条件下真实推进服务：

```text
边界检查
-> 状态刷新
-> 差额评估
-> 生成服务需求
-> 优先级与冲突仲裁（服务主导）
-> 任务管理一步治理
-> 授权/配额/限幅
-> 方法匹配与选择
-> 执行调度与过程监督
-> 结果回流与结算
-> 学习与因果沉淀
-> 主循环重判
```

必须观察到：

- `服务需求` 能被显性生成
- `任务管理一步治理` 能产出本轮推进动作
- `授权` 真正卡住“允许做到哪”
- `方法` 能被命中并下发执行
- `结果` 能回到根层结算
- `学习` 只消费已结算结果并形成路径证据
- 下一轮能据此重判，而不是停留在局部线程真值

### 7.2 金路径 B：安全接管闭环

这条路径用于证明安全在边界上真的具有优先裁决权：

```text
边界检查
-> 状态刷新
-> 差额评估
-> 风险差额超阈值
-> 生成安全需求
-> 优先级与冲突仲裁（安全主导）
-> 任务管理生成止损一步治理
-> 最小必要授权
-> 安全守卫与止损
-> 打断 / 降级运行 / 回滚 / 收束
-> 结果回流与结算
-> 学习与因果沉淀（仅记录止损证据）
-> 主循环重判
```

必须观察到：

- 风险超阈值后，不再沿原服务路径盲推
- `安全守卫与止损` 能显式打断或降级
- 止损动作只拿最小必要授权
- 打断结果能够回流主循环，而不是停留在局部线程
- 安全链完成后不能直接跳回服务链
- 下一轮去向由根层重判统一决定，而不是由执行层私下续跑
