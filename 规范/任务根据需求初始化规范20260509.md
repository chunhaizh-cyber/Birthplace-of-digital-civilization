# 任务根据需求初始化规范 20260509

## 1. 总原则

任务根据需求初始化时，采用以下口径：

```text
需求由存在发起。
任务信息节点由该存在为了满足需求而创建。
任务信息节点挂入发起存在的任务根节点。
任务信息节点内包含 / 引用一个任务虚拟存在。
任务虚拟存在是一个存在节点。
任务虚拟存在通过概念集归属于“任务”及具体任务概念。
任务虚拟存在创建后，按任务概念扩充特征槽，再由需求填充初始值。
```

核心修正：

```text
任务信息节点不要被任务虚拟存在替代。
任务信息节点内应包含 / 引用一个任务虚拟存在。
```

也就是：

```text
任务信息节点 = 任务系统里的运行壳、调度壳、树节点
任务虚拟存在 = 任务的语义身份、记忆对象、自然语言对象
```

这两者是一对组合，不是二选一。

---

## 2. 推荐最终关系

```text
存在 E
└─ 任务根节点
   └─ 任务信息节点 N
      ├─ 任务虚拟存在 = T_exist
      ├─ 来源需求 = D
      ├─ 运行状态
      ├─ 调度信息
      ├─ 执行反馈
      └─ 子任务 / 方法运行信息
```

其中：

```text
T_exist 是一个存在节点。
T_exist.概念集 = [存在, 虚拟存在, 任务, 具体任务概念]
```

例如：

```text
B-1234飞机
└─ 任务根节点
   └─ N001：任务信息节点
      └─ 任务虚拟存在 = T001：加油任务虚拟存在
         ├─ 概念集 = [任务, 保障任务, 加油任务]
         ├─ 特征槽：任务目标
         ├─ 特征槽：作用对象
         ├─ 特征槽：完成条件
         └─ 特征槽：当前方法
```

这里：

```text
N001 是任务壳。
T001 是任务作为“存在”的语义对象。
```

---

## 3. 为什么任务信息节点不能删

现有结构已经把任务树作为存在自身的结构能力：

```text
存在节点主信息类 有 需求根节点 / 任务根节点 / 方法根节点。
存在析构时会释放自己的任务树。
```

这说明任务链不是临时装饰，而是存在生命周期的一部分。

需求侧也已经把任务节点当作“当前权威运行壳”：

```text
需求主信息类::对应任务
需求主信息类::任务列表
```

所以不能改成：

```text
需求 -> 任务虚拟存在
```

更合适的是：

```text
需求 -> 任务信息节点 -> 任务虚拟存在
```

---

## 4. 层次分工

| 层 | 作用 | 是否保留 |
| --- | --- | --- |
| 任务信息节点 | 任务树、调度、运行状态、执行反馈、子任务组织 | 保留 |
| 任务虚拟存在 | 任务的语义身份、概念集、特征槽、自然语言引用 | 新增 / 绑定 |
| 任务概念 | 创建任务虚拟存在时的模板 | 保留在存在概念树 |
| 需求节点 | 任务产生的动因和目标记录 | 保留 |

一句话：

```text
任务信息节点负责“这个任务怎么被系统管理”。
任务虚拟存在负责“这个任务作为一个对象是什么”。
```

---

## 5. 任务主信息类字段

任务信息节点主信息里需要有：

```cpp
可解析引用<存在节点类> 任务虚拟存在{};
```

当前 `任务主信息类` 已经有该字段，应保留。它的规范语义是：

```text
任务作为虚拟存在的语义对象。
```

示意：

```cpp
class 任务主信息类 {
public:
    // 任务作为虚拟存在的语义对象。
    可解析引用<存在节点类> 任务虚拟存在{};

    // 任务信息节点服务的需求。
    可解析引用<需求节点类> 对应需求{};

    // 任务运行适用场景。
    可解析引用<场景节点类> 场景{};

    // 原有任务状态、统计、执行记录、调度字段继续由任务信息节点承载。
};
```

可以后续补充辅助字段：

```cpp
可解析引用<存在节点类> 发起存在{};
可解析引用<存在节点类> 作用对象{};
```

但第一阶段不强制新增；也可以先写在任务虚拟存在特征里。

注意：

```text
如果可解析引用<需求节点类> 当前可用，来源需求优先留在任务信息节点主信息。
不要为了追求统一，把需求节点类* 硬塞进任务虚拟存在的特征值里。
```

特征值里的 `指针句柄` 第一阶段主要用于存在 / 基础信息节点等语义对象引用。任务信息节点和需求节点这类运行壳关系，应优先保留结构引用。

---

## 6. 对需求部分的修正

需求主信息继续保留：

```cpp
需求主信息类::对应任务
需求主信息类::任务列表
```

它们仍然指向：

```text
任务信息节点
```

不新增并列权威字段：

```cpp
对应任务实例
```

否则会出现两个入口：

```text
需求 -> 任务信息节点
需求 -> 任务虚拟存在
```

后续同步会变复杂。

推荐关系：

```cpp
需求节点->主信息.对应任务 = 任务信息节点;
任务信息节点->主信息.任务虚拟存在 = 任务虚拟存在节点;
```

如果为了查询方便，可以加只读辅助函数：

```cpp
存在节点类* 取需求对应任务虚拟存在(需求节点类* 需求);
```

内部走：

```text
需求.对应任务 -> 任务主信息.任务虚拟存在
```

## 7. 初始化输入

任务初始化至少需要：

```text
发起存在 E
来源需求 D
任务概念 C_task
```

可选输入：

```text
作用对象
所属场景
候选方法概念
任务名称
初始优先级
初始时限
调用方传入的额外概念集
方法归属存在
```

其中 `C_task` 可以是通用任务概念，也可以是更具体的任务概念：

```text
任务
保障任务
加油任务
检查任务
调度任务
修复任务
查询任务
```

如果暂时无法判断具体任务类型，则使用通用：

```text
任务
```

## 8. 初始化流程

任务初始化流程应为：

```text
1. 需求 D 触发任务创建。
2. 找到需求主体 / 发起存在 E。
3. 在 E.任务根节点 下创建任务信息节点 N。
4. 根据任务概念创建任务虚拟存在 T_exist。
5. T_exist.概念集 = [任务, 具体任务概念]。
6. 按任务概念扩充 T_exist 的特征槽。
7. 将 N.任务虚拟存在 = T_exist。
8. 将 D.对应任务 = N，并把 N 加入 D.任务列表。
9. 根据需求内容填充 T_exist 的任务语义特征。
10. N 负责后续调度、执行、反馈和子任务组织。
```

伪结构：

```text
D001：加油需求
└─ 对应任务 = N001

N001：任务信息节点
└─ 任务虚拟存在 = T001

T001：加油任务虚拟存在
├─ 概念集 = [任务, 保障任务, 加油任务]
├─ 发起存在 = B-1234飞机
├─ 作用对象 = B-1234飞机
├─ 任务目标 = 完成加油
├─ 完成条件 = 油量达到目标值
└─ 任务状态 = 待规划
```

任务虚拟存在仍然遵守存在概念规范：

```text
存在只有一种结构。
实例通过概念集对应多个概念。
新建实例时把概念特征复制 / 实例化到实例上。
```

---

## 9. 任务概念集规则

任务虚拟存在的概念集建议包含：

```text
存在
虚拟存在
任务
具体任务概念
```

例如：

```text
T001：加油任务虚拟存在
概念集 = [
    存在,
    虚拟存在,
    任务,
    保障任务,
    加油任务
]
```

如果不想现在建立“虚拟存在”概念，也可以简化为：

```text
概念集 = [
    任务,
    加油任务
]
```

这只是概念树分层，不是新增数据结构。

## 10. 任务通用语义特征槽

“任务”概念节点上建议至少声明这些语义特征槽：

```text
发起存在
任务目标
任务状态
作用对象
执行主体
所属上下文
前置条件
完成条件
失败条件
输入信息
输出结果
候选方法
当前方法
优先级
时限
结果评价
```

这些槽属于任务虚拟存在的语义面，不是任务信息节点运行调度字段的替代品。

第一阶段最小必填：

```text
发起存在
任务状态
任务目标
```

推荐填充：

```text
作用对象
优先级
时限
完成条件
候选方法
```

后续再填：

```text
执行主体
当前方法
输入信息
输出结果
结果评价
```

## 11. 需求到任务的映射

需求初始化任务时，核心是把需求内容分配到两层：

```text
任务信息节点 N：保存运行壳关系和调度权威状态。
任务虚拟存在 T_exist：保存语义特征和自然语言可见信息。
```

推荐映射表：

| 需求内容 | 目标位置 | 说明 |
| --- | --- | --- |
| 需求本身 | `N.对应需求` | 结构引用，权威 |
| 需求发起者 | `T_exist.发起存在` | 语义特征 |
| 需求目标 | `T_exist.任务目标` | 任务要达成什么 |
| 需求对象 | `T_exist.作用对象` | 任务作用到谁 |
| 需求期望状态 | `T_exist.完成条件` | 满足什么状态算完成 |
| 需求约束 | `T_exist.前置条件 / 失败条件` | 约束可拆到不同槽 |
| 需求优先级 | `N.调度优先级` + `T_exist.优先级镜像` | N 权威，T_exist 可查询 |
| 需求截止时间 | `N.调度时限` + `T_exist.时限镜像` | N 权威，T_exist 可查询 |
| 推荐方案 | `T_exist.候选方法` | 指向方法概念或方法虚拟存在 |
| 需求上下文 | `N.场景` + `T_exist.所属上下文` | N 权威，T_exist 可查询 |

例如：

```text
B-1234飞机产生需求：需要加油
```

初始化后：

```text
N001：任务信息节点
├─ 对应需求 = D001加油需求
├─ 运行状态 = 待规划 / 未启动
├─ 调度优先级 = 普通 / 由需求决定
└─ 任务虚拟存在 = T001

T001：加油任务虚拟存在
├─ 发起存在 = B-1234飞机
├─ 任务目标 = 完成加油
├─ 作用对象 = B-1234飞机
├─ 完成条件 = 油量达到目标值
├─ 任务状态 = 待规划
├─ 候选方法 = 加油方法概念
└─ 当前方法 = 空
```

引用型语义特征值，例如：

```text
发起存在
作用对象
候选方法
当前方法
所属上下文
```

统一使用 `特征值::指针句柄`。

不建议把需求信息节点本身作为任务虚拟存在普通特征值。需求到任务的权威关系已经在：

```text
需求.对应任务
任务信息节点.对应需求
```

---

## 12. 特征槽初始化规则

任务虚拟存在的特征槽初始化分两步：

```text
第一步：按任务概念补槽。
第二步：按需求内容填值。
```

补槽来自概念路径：

```text
存在
→ 虚拟存在
→ 任务
→ 具体任务概念
```

填值优先级建议：

```text
1. 调用方显式传入值
2. 来源需求中的值
3. 具体任务概念默认值
4. 通用任务概念默认值
5. 存在根概念默认值
```

不要让通用概念默认值覆盖需求给出的值。需求是这次任务的动因，它的内容优先。

## 13. 任务状态放哪里

任务状态分成两层：

```text
任务信息节点里的状态 = 运行权威状态
任务虚拟存在里的“任务状态”特征 = 语义镜像 / 查询入口
```

例如：

```text
N001.运行状态 = 执行中
T001.任务状态 = 执行中
```

调度系统看：

```text
任务信息节点 N001
```

自然语言和特征查询看：

```text
任务虚拟存在 T001
```

更新时由任务系统同步镜像：

```cpp
更新任务运行状态(N001, 执行中);
同步任务虚拟存在状态(N001);
```

这样执行系统不会被特征查询拖着跑，语义系统也能读懂任务当前是什么状态。

---

## 14. 子任务规则

第一阶段：

```text
子任务结构以任务信息节点树为权威。
```

推荐：

```text
N001：任务信息节点
├─ 子任务 N001-1
├─ 子任务 N001-2
└─ 子任务 N001-3

T001：任务虚拟存在
└─ 只保存任务特征、概念集、语义记忆
```

不要同时写：

```text
任务信息节点子链
任务虚拟存在.任务根节点
```

原因：

```text
存在节点主信息类的任务根是拥有生命周期的结构字段。
任务节点树也有自己的释放逻辑。
不要把同一棵裸指针树同时挂到两个拥有者下面。
```

如果以后要让 `T001.任务根节点` 也能查到子任务，可以做：

```text
非拥有索引
查询函数
镜像视图
```

不要直接共享所有权指针。

---

## 15. 方法规则

方法也采用同样分层：

```text
方法信息节点 = 方法运行壳
方法虚拟存在 = 方法语义对象
```

任务信息节点可以关联方法信息节点；方法信息节点内部再引用方法虚拟存在。

```text
任务信息节点 N001
└─ 当前方法信息节点 MNode001
   └─ 方法虚拟存在 MExist001
```

方法根节点应该在存在下，不应该在任务根节点下。

也就是说：

```text
某个存在.方法根节点
└─ 方法信息节点 MNode001
   └─ 方法虚拟存在 = MExist001
```

任务侧只保存引用：

```text
N001.当前方法信息节点 = MNode001
T001.当前方法 = MExist001
```

不要直接让任务虚拟存在承担方法运行结构。语义壳和执行壳分开，系统更稳。

## 16. 去重和复用规则

### 16.1 同一需求是否重复建任务

如果需求 D 已经有未结束的任务信息节点 N：

```text
D.对应任务 = N
且 N.运行状态 不属于 [已完成, 失败, 取消, 超时]
```

则优先复用 N，不再新建任务。

如果原任务已结束，但需求再次变为待满足，可以新建任务：

```text
D001 -> N001 已失败
D001 仍未满足
创建 N002
```

### 16.2 同一存在可以有多个同名任务

允许：

```text
B-1234飞机
└─ 任务根
   ├─ N001：加油任务信息节点
   ├─ N002：加油任务信息节点
   └─ N003：加油任务信息节点
```

实例身份不靠名称唯一，而靠节点身份、创建时间、来源需求、上下文区分。

## 17. 伪代码规范

接口应返回任务信息节点，而不是任务虚拟存在：

```cpp
任务类::节点类* 根据需求初始化任务(
    存在节点类* 发起存在,
    需求节点类* 来源需求,
    const std::vector<存在节点类*>& 任务概念集,
    场景节点类* 所属场景 = nullptr
);
```

伪流程：

```cpp
任务类::节点类* 根据需求初始化任务(
    存在节点类* 发起存在,
    需求节点类* 来源需求,
    const std::vector<存在节点类*>& 任务概念集,
    场景节点类* 所属场景
) {
    if (!发起存在 || !来源需求) {
        return nullptr;
    }

    // 1. 如果来源需求已有未结束任务信息节点，优先复用。
    if (auto* 已有任务信息节点 = 查找需求未结束任务(发起存在, 来源需求)) {
        return 已有任务信息节点;
    }

    // 2. 在发起存在任务根下创建任务信息节点。
    auto* N = 创建任务信息节点(发起存在, 来源需求, 所属场景);
    if (!N) {
        return nullptr;
    }

    // 3. 创建任务虚拟存在。
    auto* T_exist = 创建存在实例(
        生成任务名称(来源需求),
        任务概念集,
        所属场景
    );
    if (!T_exist) {
        return nullptr;
    }

    // 4. 绑定任务信息节点与任务虚拟存在。
    N->主信息.任务虚拟存在 = 引用(T_exist);

    // 5. 按概念扩充任务虚拟存在特征槽。
    根据概念扩充实例记忆(T_exist, 任务概念集);

    // 6. 填充语义特征。
    设置指针特征(T_exist, "发起存在", 发起存在);
    设置任务目标特征(T_exist, 来源需求);
    设置完成条件特征(T_exist, 来源需求);
    设置优先级镜像特征(T_exist, 来源需求);
    设置时限镜像特征(T_exist, 来源需求);

    if (所属场景) {
        设置指针特征(T_exist, "所属上下文", 所属场景);
    }

    // 7. 设置运行壳状态并同步语义镜像。
    设置任务信息节点状态(N, 未启动);
    同步任务虚拟存在状态(N);

    // 8. 回写需求。
    来源需求->主信息.对应任务 = 引用(N);
    来源需求->主信息.任务列表.push_back(引用(N));

    return N;
}
```

方法选择另走方法信息节点：

```cpp
方法类::节点类* 为任务选择或创建方法(
    存在节点类* 方法归属存在,
    任务类::节点类* 任务信息节点,
    存在节点类* 方法概念
);
```

返回方法信息节点，并由它引用方法虚拟存在。

## 18. 禁止项

任务根据需求初始化时，禁止：

```text
1. 删除任务信息节点，用任务虚拟存在直接替代。
2. 让需求直接以任务虚拟存在作为对应任务权威入口。
3. 新增需求.对应任务实例 作为并列权威字段。
4. 为任务新增独立的任务存在类。
5. 为任务新增存在形态枚举。
6. 默认把任务信息节点挂到内部世界而不是任务根节点。
7. 把任务概念当作任务虚拟存在实例使用。
8. 让概念默认值覆盖需求显式值。
9. 初始化任务时强制创建方法实例。
10. 为同一需求重复创建多个未结束任务信息节点。
11. 把方法信息节点挂入任务根节点。
12. 把方法根节点设计成任务根节点的子容器。
13. 让任务信息节点子链和任务虚拟存在.任务根节点共享同一棵裸指针任务树。
```

允许：

```text
1. 任务信息节点继续使用 任务主信息类。
2. 任务信息节点挂入发起存在的 任务根节点。
3. 任务信息节点引用一个任务虚拟存在。
4. 任务虚拟存在使用 存在节点主信息类。
5. 任务虚拟存在通过 概念集 归属于多个任务概念。
6. 按概念路径补齐任务虚拟存在特征槽。
7. 根据需求填充任务虚拟存在初始语义值。
8. 用任务信息节点树表达子任务权威结构。
9. 方法信息节点挂入某个存在的 方法根节点。
10. 任务信息节点通过当前方法信息节点引用方法运行壳。
11. 任务虚拟存在通过当前方法特征引用方法虚拟存在。
```

---

## 19. 验收标准

实现本规范时，至少满足：

```text
1. 根据需求创建任务时，发起存在的任务根节点下新增的是任务信息节点。
2. 任务信息节点主信息中绑定任务虚拟存在。
3. 任务虚拟存在拥有任务概念集。
4. 任务虚拟存在含发起存在、任务状态、任务目标等语义特征。
5. 任务信息节点状态是运行权威状态。
6. 任务虚拟存在任务状态是语义镜像。
7. 需求.对应任务 仍指向任务信息节点。
8. 同一需求已有未结束任务信息节点时不重复创建。
9. 子任务结构以任务信息节点树为权威。
10. 任务虚拟存在的任务根不与任务信息节点子树共享裸指针所有权。
11. 后续创建的方法信息节点出现在某个存在的方法根节点下。
12. 任何方法信息节点都不作为任务根节点的子节点。
```

## 20. 最终定稿

```text
任务信息节点仍然保留。
任务信息节点是任务树、运行状态、调度反馈的权威结构。
任务虚拟存在是任务作为“存在”的语义身份和记忆对象。
任务信息节点包含一个任务虚拟存在引用。
需求节点仍然通过 对应任务 指向任务信息节点。
任务虚拟存在通过任务信息节点间接关联需求。
任务虚拟存在按任务概念初始化概念集和特征槽。
子任务结构第一阶段以任务信息节点树为权威，不与任务虚拟存在的任务根共享裸指针所有权。
```

一句话：

```text
任务信息节点是“任务机器”，任务虚拟存在是“任务身份证 + 记忆体”。
机器要保留，身份证也要有；二者绑定，不互相替代。
```
