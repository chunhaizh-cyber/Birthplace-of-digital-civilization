# 0323 双值结算增量合法来源与最小类型表规范

## 1. 目标

本文用于把 `双值结算增量` 的合法来源和最小类型表单独固定下来。

本文要回答的是：

- 哪些来源才有资格生成直接结算增量
- 最小稳定类型表里，到底允许哪些增量类型
- 哪些东西只能判为 `零增量` 或 `不具结算资格`

## 3. 核心结论

必须固定以下结论：

1. 直接 `双值结算增量` 的合法来源，不是“任务做过了”或“方法成功了”，而是 `具备结算资格的叶子需求首次满足`。
2. 直接结算增量必须按 `叶子需求所属根树` 归属：
   - `安全树叶子首次满足 -> 安全正向增量`
   - `服务树叶子首次满足 -> 服务正向增量`
3. 当前最小稳定类型表只固定三类：
   - `安全正向增量`
   - `服务正向增量`
   - `零增量`
4. `跨树直接增量` 必须禁止。
5. `不具结算资格`、`无推进`、`反向`、`仅局部成功`，都不得直接生成正向结算增量。
6. 当前若需要更复杂的负面后果链，应另立规则；不得把它们直接偷混进“合法直接结算增量”类型表。

更细的 `零增量 / 负面后果 / 禁止项后果 / 经验项后果` 边界，按 [0324_零增量与负面后果边界规范.md](D:\鱼巢\规范\0324_零增量与负面后果边界规范.md) 执行。

一句话收口：

`直接结算增量只能来自具备资格的叶子需求首次满足；当前最小稳定类型只有安全正向、服务正向和零增量。`


## 4. 直接增量的合法来源

直接结算增量的合法来源，当前必须同时满足：

1. 存在明确 `当前需求`
2. 该需求是 `叶子需求`
3. 该叶子需求 `所属根树明确`
4. 该叶子需求从 `未满足` 转为 `已满足`
5. 这是一次真正完成，而不是等待、重筹办、阶段切换或仅局部成功
6. 已按 [0321_实际结果方向与需求方向对齐判定表规范.md](D:\鱼巢\规范\0321_实际结果方向与需求方向对齐判定表规范.md) 完成结果对齐判断

若上述任一条件不满足，则当前不得直接生成正向结算增量。


## 5. 最小类型表

### 5.1 安全正向增量

定义：

- `安全树叶子需求首次满足` 后，允许直接加到 `安全值` 的结算增量

最小来源条件：

- 来源需求属于 `安全树`
- 需求具备直接结算资格
- 使用 `叶子承接总权值` 形成候选增量

作用对象：

- `安全值`

明确禁止：

- 直接加到 `服务值`

### 5.2 服务正向增量

定义：

- `服务树叶子需求首次满足` 后，允许直接加到 `服务值` 的结算增量

最小来源条件：

- 来源需求属于 `服务树`
- 需求具备直接结算资格
- 使用 `叶子承接总权值` 形成候选增量

作用对象：

- `服务值`

明确禁止：

- 直接加到 `安全值`

### 5.3 零增量

定义：

- 本轮回流虽已完成比较和结算检查，但当前不允许直接产生正向结算增量

典型来源：

- 当前需求不具直接结算资格
- 当前需求不是叶子需求
- 当前只是局部成功，不是叶子首次满足
- 单特征或组内结果判为 `无推进`
- 单特征或组内结果判为 `反向`
- 所属根树不明确

作用对象：

- 当前不直接增加 `安全值`
- 当前不直接增加 `服务值`


## 6. 最小类型总表

| 增量类型 | 合法来源 | 直接作用对象 | 是否允许跨树 | 当前状态 |
| --- | --- | --- | --- | --- |
| `安全正向增量` | `安全树叶子需求首次满足` | `安全值` | 否 | 正式稳定 |
| `服务正向增量` | `服务树叶子需求首次满足` | `服务值` | 否 | 正式稳定 |
| `零增量` | 不具资格 / 无推进 / 反向 / 根树不明 / 非叶子 | 无直接正向回写 | 不涉及 | 正式稳定 |


## 7. 当前不纳入最小稳定类型表的内容

以下内容当前不纳入“合法直接结算增量”的最小稳定类型表：

1. `安全负向直接结算增量`
2. `服务负向直接结算增量`
3. `跨树直接结算增量`
4. `只因方法成功而产生的增量`
5. `只因任务完成而产生的增量`
6. `只因结果主键命中而产生的增量`

必须固定：

1. 这些内容若后续要正式引入，必须另立规则。
2. 在另立规则之前，不得把它们偷偷并入当前最小稳定类型表。


## 8. 与承接总权值的关系

合法直接结算增量当前必须以：

- `叶子承接总权值`

作为权值来源。

必须固定：

1. 不得把 `根权重 + 叶子权重` 重新相加当增量。
2. 不得跳过承接传播，直接用上位抽象权重结算。
3. 更细的承接传播和叶子结算规则，按 [0310_安全值服务值与任务系统耦合规则.md](D:\鱼巢\规范\0310_安全值服务值与任务系统耦合规则.md) 执行。


## 9. 与双值结算顺序的关系

本表回答的是：

- `当前允许形成什么增量类型`

但它不替代 [0322_双值结算最小输入结算顺序与禁止混层规范.md](D:\鱼巢\规范\0322_双值结算最小输入结算顺序与禁止混层规范.md) 中的完整顺序。

固定顺序应理解为：

```text
结果回流
-> 对齐判断
-> 判断是否具备结算资格
-> 判定落入哪种增量类型
-> 形成增量候选
-> 回写双值
-> 外层根层重判
```


## 10. 严格禁止的混层

以下做法必须禁止：

1. `方法执行成功 -> 直接生成正向增量`
2. `任务完成 -> 直接生成正向增量`
3. `结果主键命中 -> 直接生成正向增量`
4. `非叶子需求 -> 直接生成正向增量`
5. `根树不明确 -> 仍生成正向增量`
6. `服务树叶子首次满足 -> 直接生成安全正向增量`
7. `安全树叶子首次满足 -> 直接生成服务正向增量`
8. `不区分资格与类型，直接回写双值`
