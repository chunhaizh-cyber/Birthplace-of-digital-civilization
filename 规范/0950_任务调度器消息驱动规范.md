## 3. 核心结论

必须固定以下结论：

1. `任务调度器` 是消息驱动的任务管理器，不是需求真值引擎，也不是执行器。
2. `需求状态` 应独立管理，任务调度器只读取其结果，不直接裁决需求是否满足。
3. `执行器` 必须按单步执行工作，每一步结束都必须返回消息。
4. 没有返回消息时，任务调度器不得凭空推进到下一步。
5. 真根任务固定只有两个：`安全根任务` 与 `服务根任务`。
6. 真根任务不得进入终局完成态；伪根任务、阶段任务与普通子任务可以完成。
7. `子任务清零` 指当前已不存在可执行子任务，且所有子任务只落在 `完成 / 失败 / 无法执行` 三种状态。
8. 子任务清零后若父任务对应需求仍未满足，则必须重新筹办，而不是停在无后续动作的中间状态。

可收敛为一句话：

`任务调度器只消费消息、更新任务可管理状态、维护执行队列，并在需求未满足时触发重新筹办。`

### 3.1 最新统一主链中的位置

按当前最新体系，任务调度器必须服从以下主链：

```text
根层方向裁决
-> 形成任务管理上下文
-> 调度器选择当前应推进的任务
-> 执行器单步执行并返回消息
-> 调度器消费消息并更新任务可管理状态
-> 外层重判根方向与当前功能域
-> 再决定是否重新入队、重筹办或收束
```

必须固定：

1. `任务调度器` 不负责根层方向裁决。
2. `任务调度器` 不替代四域判域器，只承接已形成的任务管理上下文与返回消息。
3. `任务调度器` 处理消息后的结果，仍要交回外层做 `根层重判 -> 判域 -> 再调度`。


## 4. 三层职责划分

### 4.1 需求管理

需求管理负责：

- 维护需求节点
- 判断需求是否满足
- 维护需求失效、撤销、保留等语义

需求管理不负责：

- 推进任务步骤
- 直接决定下一个执行哪个任务

### 4.2 任务调度器

任务调度器负责：

- 接收任务返回消息
- 更新任务可管理状态
- 维护可执行任务队列
- 在子任务清零后判断是否需要重新筹办
- 选择下一个可执行任务
- 把步骤结果、最近结果、任务状态等任务侧真值变化整理回任务管理上下文

任务调度器不负责：

- 直接计算需求真值
- 直接裁决根层方向
- 直接执行方法
- 无消息时主动猜测执行现场状态

### 4.3 执行器

执行器负责：

- 当前步骤的筹办
- 当前步骤的实际执行
- 当前步骤结束后生成返回消息

执行器不负责：

- 决定任务是否终局完成
- 决定是否进入下一轮筹办
- 独自推进跨多步闭环


## 5. 状态主权划分

后续应把任务状态分成两类：

### 5.1 执行现场状态

这类状态由执行器负责更新：

- `筹办中`
- `执行中`
- `等待回复`
- `等待子任务回复`

它们表达的是：

- 当前一步正在发生什么
- 当前是否仍处于执行现场

### 5.2 可管理状态

这类状态由任务调度器负责更新：

- `就绪`
- `排队中`
- `完成`
- `失败`
- `无法执行`
- `待重筹办`

它们表达的是：

- 该任务当前是否还能被继续管理
- 是否应该重新进入筹办
- 是否已经阶段性收口

若当前代码中的任务状态枚举尚未完整覆盖这些状态，则后续重构应补齐枚举或提供等价承载字段。


## 6. 返回消息规则

### 6.1 基本原则

必须固定：

- 每一个步骤执行结束后，都必须产生一条返回消息
- 返回消息是任务调度器管理任务状态的唯一可信入口
- 没有返回消息时，任务调度器只允许等待，不允许自动推进下一步

### 6.2 V1 标准消息类型

V1 先固定以下七类：

1. `步骤完成`
2. `步骤失败`
3. `步骤无法执行`
4. `生成子任务`
5. `子任务完成`
6. `子任务失败`
7. `子任务无法执行`

后续若需要新增消息类型，可以在不违背本文主逻辑的前提下扩展。

### 6.3 最小消息载荷

为保证 `任务调度器 -> 外层重判 -> 再调度` 这条链不断，返回消息至少应能承载：

- 来源任务头
- 当前步骤
- 当前方法首节点（若本步存在）
- 最近结果节点或结果特征摘要
- 当前步骤后的任务状态建议
- 当前下一步去向建议
- 是否生成了子任务

必须固定：

1. 返回消息首先服务于调度器与外层判定，不是给执行器自己做闭环。
2. 若当前消息无法回指 `任务头 / 当前步骤 / 最近结果`，则调度器不得据此擅自推进后续流程。
3. `当前下一步去向建议` 只是建议，最终仍由外层 `根层重判 + 判域` 共同解释。
4. 若需要更细化到 `实践结果特征列表 / 命中需求回挂提示 / 来源承接类型`，按 [0955_实践结果消息最小结构规范.md](D:\鱼巢\规范\0955_实践结果消息最小结构规范.md) 执行。


## 7. 子任务清零与重筹办

### 7.1 子任务清零的定义

`子任务清零` 指的是：

- 当前已没有可执行的子任务

这里“没有可执行”具体表达为：

- 所有子任务都处于 `完成`
- 或 `失败`
- 或 `无法执行`

### 7.2 子任务清零后的判定顺序

当父任务收到子任务终结类消息，并发现子任务已清零时，必须按以下顺序处理：

1. 先汇总父任务当前任务侧真值与最新结果
2. 将其提交外层进行根层重判与功能域重判
3. 若外层已判定不允许继续推进，则父任务转收束或停机边界处理
4. 若根层仍允许继续，且父任务绑定需求已满足，则父任务可以进入 `完成`
5. 若根层仍允许继续，但需求未满足，则父任务进入 `待重筹办`
6. 由调度器重新把该父任务送回筹办链

必须固定：

- 父任务不直接“执行父需求”
- 父任务只通过重新筹办来生成下一阶段任务


## 8. 所有任务的统一闭环

后续应固定所有任务都遵守：

- `局部阶段结束 -> 检查对应需求 -> 已满足则完成 -> 未全满足则重新筹办 -> 生成下一阶段任务`

其中：

- 真根任务由于对应持续逼近型根需求，不会真正走到终局完成
- 伪根任务、阶段任务和普通子任务则可能在需求满足后结束


## 9. 真根任务与伪根任务

### 9.1 真根任务

当前系统中真根任务固定只有两个：

- `安全根任务`
- `服务根任务`

必须固定：

- 真根任务常驻
- 真根任务的局部阶段结束后，应先检查根需求
- 根需求由于属于持续逼近型需求，结构上不会真正被“全满足”
- 因此真根任务会自然回到 `重新筹办`

### 9.2 伪根任务

伪根任务指的是：

- 在某一局部阶段承担类似顶层职责
- 但其绑定需求仍然是可终结的阶段性需求

因此：

- 伪根任务允许进入 `完成`


## 10. 删除约束

物理删除需求节点或任务节点之前，必须先严格判定：

- 对应任务已经不存在
- 或对应任务已不再执行
- 或对应任务已经脱离调度链与执行链

如果这一点无法严格保证，则：

- 不应直接物理删除
- 应优先使用逻辑取消、逻辑失效或逻辑解绑


## 11. 结论

后续任务系统应固定为：

- 需求真值独立管理
- 执行器单步执行并返回消息
- 调度器按消息管理任务状态与可执行队列
- 子任务清零但需求未满足时，必须重新筹办
- 真根任务只允许是安全与服务两个，且结构上永不真正完成
