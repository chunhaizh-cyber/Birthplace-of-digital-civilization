# 0900 任务类实现规范（当前任务节点与状态管理）

本文用于固定当前 `任务类.ixx` 的实现边界，说明：
- 任务类已经实现了哪些功能
- 这些功能分别落在什么层次
- 任务侧真值与任务管理上下文在这里如何分工
- 哪些能力仍然不属于任务类

本文对应当前工程实现：
- `任务类.ixx`
- `主信息定义模块.ixx`
- `任务执行类.ixx`


## 1. 任务类的定位

当前 `任务类` 的定位是：

- 任务节点的创建与结构维护
- 任务虚拟存在初始化与特征写回
- 任务头结点需求/条件场景维护
- 任务状态字段承载与内部状态写回
- 任务删除与逻辑取消
- 具体任务侧真值的结构落点维护

当前 `任务类` 不负责：

- 方法召回
- 方法执行
- 步骤推进
- 预测结果自动生成
- 预测命中判断
- 跨任务调度
- 根层方向裁决
- 任务管理上下文总判

这些能力应由：

- `任务执行类.ixx` 负责执行推进
- `任务调度类.ixx` 负责调度
- 后续筹办逻辑负责方法召回与预测结果生成

### 1.1 最新统一主链中的位置

按当前最新体系，`任务类` 应处在下面这条链里：

```text
根层方向裁决
-> 形成任务管理上下文
-> 任务类承载和维护具体任务侧真值
-> 调度器与执行器围绕这些真值推进
-> 结果再回到外层重判
```

因此必须固定：

1. `任务类` 是任务侧结构真值与运行锚点的承载层，不是根层裁决层。
2. `任务类` 可以被任务管理上下文读取和镜像，但不替代任务管理上下文本身。
3. `任务类` 写入的是真实任务结构与任务侧字段，不是“当前这一轮管理策略”。


## 2. 当前任务模型

当前任务侧以三类核心节点组织：

- `任务头结点`
- `任务步骤节点`
- `任务结果节点`

### 2.1 头结点语义

头结点负责：
- 持有 `需求`
- 持有 `任务虚拟存在`
- 持有 `任务全局条件场景`
- 缓存 `当前步骤节点`
- 缓存 `最近实际结果节点`

必须补充固定：

- `需求`、`当前步骤节点`、`最近实际结果节点` 属于具体任务头的任务侧真值。
- 管理侧若需要快速裁决，可以保留镜像；但任务头上的值才是任务侧直接真值归属点。

### 2.2 步骤节点语义

步骤节点负责：
- 表示任务中的一个待执行步骤
- 持有步骤自己的条件场景
- 表达一条独立的局部推进方案
- 持有 `当前选中方法首节点`
- 视需要持有 `备用方法首节点列表`
- 持有 `步骤序号`
- 持有 `允许重试次数`
- 持有 `超时截止时间`
- 持有 `允许切换方法`

必须补充固定：

- 不同条件链、不同结果链或不同收口去向的候选方法，不得继续塞在同一步骤里，必须拆成并列步骤节点。
- `当前选中方法首节点` 属于具体任务推进路径上的任务侧真值。
- 并列步骤具体选谁，应由该步骤自己的前置状态、风险门控、优先级和历史效果共同决定，而不是先在单一步骤里堆满方法再临时挑选。
- `当前缺口类型`、`当前下一步去向` 当前不应直接塞进步骤节点主信息里冒充同层真值；若后续需要持久化，也应先与上位任务管理上下文建立清晰映射。

### 2.3 结果节点语义

结果节点负责：
- 同时表达 `预测结果` 和 `实际结果`
- 记录来源步骤
- 记录来源方法
- 记录 `结果方向二次特征`
- 记录命中预测结果
- 记录命中后动作
- 记录是否为预测外结果
- 记录错误码、置信度、评估摘要锚点


## 3. 任务类已经实现的功能

### 3.1 内部辅助能力

`任务模块_detail` 已经实现了一批底层辅助函数，用于支撑任务类公开接口：

- 时间标准化
- 任务虚拟存在创建
- 任务内部状态场景创建
- 任务状态特征槽初始化
- 任务内部状态写回
- 头结点/步骤节点/结果节点主信息提取
- 从任意任务节点推导所属任务头结点
- 确保任务头结点条件场景

这些函数的意义是：

- 把运行期状态尽量写回 `任务虚拟存在`
- 避免继续把运行态字段无限堆进主信息
- 保证头结点、步骤节点、结果节点都有一致的底层写入路径


### 3.2 创建任务节点

当前已经实现三套创建参数结构：

- `结构_创建任务头参数`
- `结构_创建步骤参数`
- `结构_创建结果参数`

并提供四个创建入口：

- `新建任务节点`
- `创建任务头结点`
- `创建步骤节点`
- `创建结果节点`

#### 3.2.1 新建任务节点

`新建任务节点` 是统一的底层建点入口，负责：

- 为新节点规范化创建时间
- 检查头结点是否绑定需求
- 初始化任务虚拟存在
- 将节点挂到任务链根层或父节点子链
- 写入一次“创建”内部状态

它不关心该节点是头、步骤还是结果，只负责最低层结构挂接。

#### 3.2.2 创建任务头结点

`创建任务头结点` 已实现：

- 根据需求构造头结点主信息
- 从需求同步名称、描述、类型、需求主键
- 设置初始状态、优先级、任务树类型
- 绑定初始场景
- 任务头、步骤、结果节点的 `基准优先级` 都以主任务头结点为最终来源
- 可自动把任务回填到 `需求.相关任务`
- 可自动调用 `进入筹办阶段`

必须额外固定：

- 这里同步的 `需求主键` 只承担 `结构身份同步 / 粗召回入口 / 索引辅助` 职责
- 不得把 `需求主键命中` 直接当成 `任务已满足需求` 或 `任务方向正确`
- 任务是否满足需求，仍必须回到 `需求值 / 当前值 / 返回值` 的比较，并由对应 `二次特征` 判定

因此，当前任务成立的最小闭环已经具备：

`需求 -> 创建任务头结点 -> 初始化任务虚拟存在`

#### 3.2.3 创建步骤节点

`创建步骤节点` 已实现：

- 在指定父节点下创建步骤节点
- 自动推导所属任务头结点
- 绑定步骤场景
- 选择当前方法
- 写入步骤序号、重试次数、超时、是否允许切换方法
- 步骤建立后，立即在其下创建并初始化结果节点
- 如果头结点还没有当前步骤，则自动把该步骤设为当前步骤

这意味着：

- 当前任务类已经能表达“这一步准备用什么方法做”
- 步骤节点与结果节点应视为成对出生的局部结构，而不是先有步骤、长期没有结果锚点
- 步骤节点已经成为方法绑定的主要位置

#### 3.2.4 创建结果节点

`创建结果节点` 已实现：

- 在步骤节点下创建结果节点
- 区分 `预测结果` / `实际结果`
- 自动记录所属任务头结点、来源步骤节点
- 自动继承来源方法
- 可记录命中预测结果节点

必须补充固定：

- 步骤节点创建后，应立即至少创建一个结果节点作为初始化结果锚点。
- 初始化结果节点默认按 `预测结果` 或 `占位结果` 理解，它的存在只表示“本步骤的结果承接面已建好”，不等于本步已经执行完成。
- 初始化结果节点必须给出 `结果方向二次特征`；它是 `结果特征状态` 的二次特征，初始化时按 `步骤目标状态` 对 `当前状态` 生成，后续若补入更明确的预测/实际结果状态，应再按该结果状态刷新。
- 初始化时至少应写入：`结果角色`、`所属任务头结点`、`来源步骤节点`、`来源方法首节点（若已选定）`、`对应方法结果节点（若可推导）`、`结果方向二次特征`、`命中后动作`、`是否预测外结果`、`置信度_Q10000`、`错误码`。
- 可记录命中后动作
- 可记录是否预测外结果
- 可记录错误码、置信度
- 对于“实际结果”，自动写回头结点的最近实际结果缓存

这意味着：

- 任务类已经能表达“步骤下挂预测结果和实际结果”
- 也已经能表达“实际结果回指预测结果”


### 3.3 任务虚拟存在写回

当前任务类已经提供统一的运行态写回接口：

- `记录任务虚拟存在特征`
- `写任务虚拟存在_I64特征`
- `写任务虚拟存在_指针特征`

在此基础上，已经实现的快捷写回包括：

- `标记任务最近调度`
- `设置任务当前步骤`
- `设置任务当前分支动作`
- `设置任务最近结果节点`
- `绑定任务全局容器`

这里的实际语义是：

- 任务运行态会同步写回到任务虚拟存在
- 任务虚拟存在承担观察、写回和运行镜像
- 任务头/步骤主信息仍是 `当前步骤 / 当前方法 / 最近结果` 这类任务侧真值的直接结构落点
- 若任务虚拟存在镜像与任务头/步骤主信息不一致，必须优先以任务头/步骤主信息为准，再修订镜像

注意：

`设置任务当前分支动作` 目前名称还沿用了旧叫法，但在新模型下更接近“当前处置动作提示”。


### 3.4 头结点维护与筹办入口

当前头结点维护已经实现四个关键入口：

- `更新任务头结点需求`
- `确保任务头结点场景`
- `更新任务头结点场景`
- `进入筹办阶段`

#### 3.4.1 更新任务头结点需求

已实现：

- 替换头结点绑定的需求
- 同步目标状态主键或目标特征类型
- 在需要时补齐描述与类型
- 可自动更新 `需求.相关任务`
- 自动标记最近调度时间

#### 3.4.2 确保任务头结点场景

已实现：

- 如果头结点还没有条件场景，则自动创建
- 创建完成后，自动把该场景作为任务全局容器写回任务虚拟存在

这里的语义已经固定为：

`头结点场景 = 当前任务实例的全局运行条件场景`

#### 3.4.3 更新任务头结点场景

已实现：

- 将外部准备好的条件场景绑定到头结点
- 同步将其写为任务绑定的全局容器

#### 3.4.4 进入筹办阶段

已实现：

- 检查头结点是否具备需求
- 同步需求到头结点
- 确保或更新头结点条件场景
- 将头结点状态切换到 `筹办中`

这说明当前任务类已经具备了：

`头结点成立 -> 条件场景到位 -> 进入筹办阶段`

但它还没有进一步负责：

- 召回方法
- 生成预测结果节点


### 3.5 状态管理

当前统一状态入口是：

- `设置任务状态`

它已经实现：

- 写主信息镜像状态
- 计算是否变化
- 将变化同步写入任务虚拟存在内部状态

因此当前状态模型是：

- 主信息中的 `状态` 当前仍用于运行期快速读取
- 任务虚拟存在中的状态特征承担同步写回与观察记录

但后续重构应固定为：

- `任务类` 只提供状态字段承载与写回能力
- `任务类` 不拥有任务状态的最终裁决权
- 主信息中的 `状态` 应被视为 `过渡期缓存字段`
- 当调度器与执行器拥有统一状态访问接口后，主信息中的 `状态` 可以继续弱化，直至删除


### 3.6 删除与取消

当前已实现：

- `删除任务节点`
- `逻辑取消任务`
- `取任务主信息指针_不加锁`

#### 3.6.1 删除任务节点

已实现：

- 如果删除的是头结点，会先清理需求上的反向引用
- 从任务链中物理删除该节点

后续还应固定：

- 物理删除前，必须先严格判定该任务已不存在或不再执行
- 若任务仍可能被调度、执行或被其它节点引用，则不得直接物理删除
- 在无法确认安全删除时，应优先使用逻辑取消或逻辑失效

#### 3.6.2 逻辑取消任务

已实现：

- 不物理删除节点
- 只将任务状态切换为 `取消`

这使得当前系统已经具备：

- 结构维护意义上的删除
- 运行意义上的取消


## 4. 当前任务类没有实现的能力

以下能力目前不在任务类中：

- 自动方法召回
- 自动生成预测结果节点
- 用二次特征进行预测命中
- 步骤执行
- 执行失败后的重试策略推进
- 命中结果后的下一步骤切换逻辑
- 跨任务优先级调度

这些能力目前应理解为：

- `任务类` 只负责“任务节点和运行锚点管理”
- `任务执行器` 负责“怎么推进一步”
- `任务调度器` 负责“下一帧先推进谁”


## 5. 当前已形成的最小工作流

基于当前实现，已经能跑通的最小流程是：

必须先补充一个外层前提：

`根层方向已允许继续推进 -> 已形成任务管理上下文 -> 再进入任务类承载与维护结构真值`

### 5.1 建立任务

`需求 -> 创建任务头结点 -> 初始化任务虚拟存在`

### 5.2 进入筹办

`更新头结点需求 -> 确保头结点条件场景 -> 进入筹办中`

### 5.3 补步骤

`在头结点下创建步骤节点 -> 绑定可用方法列表 -> 选择当前方法`

### 5.4 补结果

`在步骤节点下创建预测结果节点 / 实际结果节点 -> 实际结果回指命中预测`

也就是说，当前 `任务类` 已经能支撑：

`任务结构建立`

但还没有独立完成：

`任务结构自动筹办`

更准确地说：

- `任务类` 已能承接“具体任务是什么、当前步骤在哪、最近结果是什么”这类任务侧真值
- 但它不直接承接“当前缺口类型是什么、当前下一步去向是什么、当前应进入哪个功能域”这类管理上下文真值


## 6. 当前实现的边界结论

一句话总结当前 `任务类`：

`它已经是任务节点与任务运行锚点的管理器，但还不是任务筹办器，也不是任务执行器。`

进一步说：

- 它已经实现了任务头、步骤、结果三类节点的创建和维护
- 已经实现了任务虚拟存在与状态写回
- 已经实现了头结点进入筹办阶段的最小入口
- 但尚未实现方法召回、预测生成、预测命中和执行推进本身

### 6.1 V1 重构约束

针对后续消息驱动任务系统，还应补充固定：

1. `任务类` 只管理节点结构、虚拟存在写回与必要缓存，不直接裁决任务是否完成或重筹办。
2. `任务信息基类::状态` 当前是过渡字段；新逻辑不应继续把它当成最终语义源。
3. 真正的任务状态裁决应由 `任务调度器` 与 `任务执行器` 按消息驱动分工完成。
4. 真根任务只允许是 `安全根任务` 与 `服务根任务` 两个头任务。
5. 删除任务节点前，必须先确认它已脱离调度链与执行链，不能留下悬空引用。
6. `当前步骤 / 当前方法首节点 / 最近结果` 这类字段，后续仍应作为任务侧真值保留明确归属点。
7. `当前缺口类型 / 当前下一步去向 / 当前进入哪一域` 这类字段，不得偷偷混入任务类里冒充任务侧真值；它们首先属于任务管理上下文。
