# 0017 承接层四模块接口总表 v0.1规范

## 1. 目标

本文用于把 `承接层四模块` 正式固定成一份 `v0.1` 接口总表。

本文只回答以下问题：

- 在 `任务管理一步治理` 已产出本轮一步治理后，承接层应拆成哪 `4` 个模块
- 这 `4` 个模块各自最小负责什么、读什么、写什么、能调用谁、不能越权改什么
- 当前如何把 `需求识别 -> 方法调用 -> 结果回写` 这条链安全接平到底层动作
- 为什么必须把 `授权`、`选法`、`执行`、`止损` 四种权力拆开，而不能混写成一个大函数

本文对承接层的统一定位固定为：

`承接层像离合器，不是方向盘。`

也就是说：

- 根层负责判方向
- 骨架层负责决定本轮主导需求和一步治理
- 承接层只负责把这一步治理安全咬合到具体动作上

## 3. 当前范围与边界

### 3.1 本版只覆盖四模块

`v0.1` 只覆盖以下 `4` 个承接层模块：

1. `授权 / 配额 / 限幅`
2. `方法匹配与选择`
3. `执行调度与过程监督`
4. `安全守卫与止损`

### 3.2 本版不负责

本文不负责以下内容：

- 根需求生成与主导需求裁决
- 主循环最终去向决定
- 学习与因果沉淀的独立模块化
- 方法内部算法和复杂评分公式
- 安全守卫的复杂预测模型
- 线程池、锁顺序和具体代码级函数切分

必须固定：

`这一版先把承接层四模块钉死，不把学习模块和根层裁决揉进来。`


## 4. 核心结论

必须固定以下结论：

1. 承接层的职责是把 `一步治理` 翻译成受控动作，不是重新定义本轮方向。
2. `授权 / 配额 / 限幅` 回答的是“这轮允许做到哪”，不是“怎么做”。
3. `方法匹配与选择` 回答的是“用什么做”，不是“允不允许做”。
4. `执行调度与过程监督` 回答的是“实际发生了什么”，不是“应不应该继续做”。
5. `安全守卫与止损` 回答的是“什么时候必须停”，不是“之后长期应该往哪走”。
6. `能看见` 不等于 `能处置`，`能处置` 也不等于 `能改写`；四模块必须严格分权。
7. 若把这四种权力揉在一起，最容易发生的失稳就是：局部线程自扩权、自续跑、自解释成功，最后把根层裁决掏空。
8. 承接层所有模块都只能服务于 `结果回流、结算与重判`，不得绕开它自行连跑下一轮主闭环。
9. 当前最先要补硬的是：
   - `授权 / 配额 / 限幅`
   - `执行调度与过程监督`
10. `方法匹配与选择` 当前可以先做轻量版，`安全守卫与止损` 当前可以先做硬阈值版，但两者的权力边界必须先固定。

一句话收口：

`承接层负责把治理命令咬合到动作上，但没有权力接管根方向。`

## 6. 承接层内部主链

承接层内部主链固定为：

`任务管理一步治理 -> 授权 / 配额 / 限幅 -> 方法匹配与选择 -> 执行调度与过程监督 -> 结果回流、结算与重判`

并行旁路固定为：

`方法匹配与选择 / 执行调度与过程监督 ↘ 安全守卫与止损（并行监测，可随时打断）`

必须固定：

1. `授权 / 配额 / 限幅` 先于方法选择。
2. `方法匹配与选择` 先于执行调度。
3. `安全守卫与止损` 从候选阶段起就在线，而不是等执行失败后才补救。
4. 任一承接模块都不能绕过 `结果回流、结算与重判` 自行续跑下一轮。


## 7. 为什么这四模块必须分开

必须把以下四种权力显式拆开：

- `授权模块` 决定“允许做到哪”
- `方法模块` 决定“用什么做”
- `执行模块` 决定“实际做了什么”
- `安全模块` 决定“何时必须停”

若不分开，最容易混掉的就是：

1. 把局部方法可用误当成正式授权。
2. 把执行开始误当成根层放行。
3. 把局部成功误写成全局成功。
4. 把安全打断偷换成新的根方向。

因此必须固定：

`承接层四模块不是为了拆得好看，而是为了防止局部线程自扩权。`


## 8. 和现有底座的直接挂接

这一步不是新造大陆，而是给现有挂点安上器官。

### 8.1 治理到方法的传递面

以下承载面直接服务承接层主链：

- `需求链 / 任务链 / 方法链`
- `需求根 / 任务根 / 方法根`

其中：

- `任务管理一步治理 -> 授权` 主要沿 `任务链` 与 `任务根` 传递
- `授权 -> 方法匹配` 主要沿 `方法链` 与 `方法根` 传递
- `方法匹配 -> 执行监督` 主要沿 `方法链 + 场景输入` 传递

### 8.2 执行监督与回流观察面

以下承载面直接服务执行监督与结果回流：

- `状态索引`
- `动态索引`
- `关系索引`
- `实例因果索引`

必须固定：

- `状态索引` 负责回答“当前事实是什么”
- `动态索引` 负责回答“执行过程发生了什么”
- `关系索引` 负责回答“当前关键关系与条件是什么”
- `实例因果索引` 负责回答“这次方法/任务与结果之间已有哪类证据”

### 8.3 因果实例当前先服务于方法选择与结果校验

当前因果实例中的以下字段，优先服务于承接层：

- `来源任务主键`
- `来源方法主键`
- `置信度`
- `已验证`

当前阶段先不把它们直接升级为稳定学习规则，而先用于：

- 方法选择时的证据比较
- 执行后结果的有效性校验
- 主循环回流时的证据包组装


## 9. 与现有约束规范的对齐口径

### 9.1 与执行前门控的关系

`授权 / 配额 / 限幅` 是承接层当前最接近工程化 `执行前门控` 的承载面，但它不是根层重判本体。

必须固定：

1. `授权 / 配额 / 限幅` 至少应共同消费：
   - `根层重判结果`
   - `任务管理上下文`
   - `局部条件结果`
2. 若 [0317_执行前门控最小输入与判定顺序规范.md](D:\鱼巢\规范\0317_执行前门控最小输入与判定顺序规范.md) 的任一关键门未通过，承接层不得继续进入普通执行链。
3. `授权成功` 不等于 `根层重新裁决通过`；它只是本轮一步治理在当前包络中的工程放行结果。

### 9.2 与动作分类边界的关系

`方法匹配与选择` 和 `执行调度与过程监督` 在调用具体动作时，必须服从 [1120_自我动作分类与调用边界规范.md](D:\鱼巢\规范\1120_自我动作分类与调用边界规范.md)。

必须固定：

- 不得把内部治理动作伪装成外部服务动作
- 不得把内部域和外部域主结果混成一个黑盒动作
- 外部动作结果若无观测、回执或状态确认，只能形成弱结果证据

### 9.3 与动态回写的关系

`执行调度与过程监督` 的结果记录，必须服从 [1130_先天记账动作与被动动作动态规范.md](D:\鱼巢\规范\1130_先天记账动作与被动动作动态规范.md)。

必须固定：

- 执行过程中的关键相位切换、状态切换和账本更新，不能只改静态字段
- 这些变化必须留下 `动态节点` 或等价的被动动作动态
- 动态记录不是附属品，而是后续回流、结算和因果比较的正式输入


## 10. 当前实现优先级

当前最应该优先补硬的是两个模块：

1. `授权 / 配额 / 限幅`
   - 这是当前最缺的，因为权限体系还没有真正贴到协同场景上。
   - 它也是防止承接层自扩权的第一道硬边界。

2. `执行调度与过程监督`
   - 这是当前最容易先闭起来的，因为 `动态 / 场景 / 因果` 挂点已经在。
   - 它能最快形成“真有动作、真有回写、真可追踪”的链。

当前可采用的保守实现策略是：

- `方法匹配与选择` 先做轻量版
- `安全守卫与止损` 先做硬阈值版

但即便是轻量版 / 硬阈值版，也不得跨越本文定义的权力边界。

若当前继续补完最后一个 `学习与因果沉淀` 模块，并把 `12` 模块用两条金路径完整闭起来，按 [0018_学习与因果沉淀模块接口总表_v0.1规范.md](D:\鱼巢\规范\0018_学习与因果沉淀模块接口总表_v0.1规范.md) 执行。
