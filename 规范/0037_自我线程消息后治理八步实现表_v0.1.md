本文用于把 `自我线程` 在“收到消息之后”真正必须承担的处理链，正式压成一份可落代码的 `实现表`。

## 3. 核心结论

必须固定以下结论：

1. `自我线程` 拿到消息后，不得“见一条判一条”；必须先把消息翻译成 `一次特征变化`，再把这些变化翻译成 `二次特征与判定依据`，并继续归结成 `影响项与双值影响摘要`，最后才允许进入 `根层重判 / 主派发 / 结果路由 / 结算`。
2. 一轮只处理 `本轮冻结消息批次` 及其 `同步内部闭包`；本轮中新产生的异步消息，一律留待下一轮。
3. `根层重判` 只允许在 `治理帧` 或等价统一判定材料形成后执行，不得在一次特征尚未落账、二次特征尚未重算时提前裁决。
4. `主派发` 与 `结果路由` 不是一回事；一轮最多只有一个 `主派发决议`（即 `主调度决议`），但该决议下可以有 `0..N 个彼此不冲突的并发派发动作`，同时也可以有多个受控的 `结果路由动作`。
5. `步骤展开 / 方法内部执行 / 父任务局部步骤判断` 仍属于下游承接层，自我线程只负责控制面裁决与路由。
6. `学习触发` 只能后置登记，不得在本轮结算前把学习链拉成新的主循环。
7. 与任务相关的读写，必须区分 `任务根语义`、`任务控制态` 和 `任务管理任务局部推进态`；第 `5/6` 步优先改 `任务控制态`，第 `7` 步才协调局部推进摘要和语义结果回写。
8. 发件线程尤其是线程状态变化发件线程，只负责把事实送入本轮冻结消息；`自我内部世界` 中的线程状态特征、状态节点和动作动态，只能由 `自我线程` 在第 `2` 步统一落账。
9. 其它线程状态变化进入第 `2` 步后，不能只停在“已落账”；还必须在第 `3/4` 步继续被翻译成“这对自我意味着什么”的判断依据与影响项。
10. `自我线程` 自身的关键生命周期变化属于例外：它们应落 `关键日志 / 宿主级审计 / 恢复判断材料`，而不是作为本表里的普通线程状态消息继续扩张。
11. `消息转成状态、动态` 只叫 `入账`；`自我线程` 的治理主责应从入账之后开始，继续完成 `对账、算二次特征、算影响项、改根层结构与权重、重判、派发、结算`。


## 4. 范围与硬边界

本版只固定 `收到消息后` 的两层顺序：

1. 冻结本轮消息批次
2. 规范化消息并落一次特征账
3. 入账之后再继续进入七步治理链

必须固定：

1. `是否触发重判` 列表示“本步骤是否直接执行或正式进入本轮根层重判”，不是“是否会影响下游未来重判”。
2. `是否触发派发` 列表示“本步骤是否会向外形成本轮对外动作”，其中 `结果路由` 也算对外动作，但不等于 `主派发`。
3. 第 `7` 步产生的父任务局部影响，不得再在同一轮反向拉起第二次 `根层重判`；若需要重新裁决，必须进入下一轮。
4. 第 `8` 步可以登记 `学习候选 / 否定项候选 / 恢复候选`，但不得在同一轮继续展开学习执行。
5. 任何线程状态变化都必须先以正式消息进入第 `1` 步冻结批次；不得由发件线程在批次外直接改写线程状态真值。
6. 第 `3/4` 步处理线程状态变化时，关注点不是“线程自己变了”这一件事，而是“这对自我的执行能力、风险、等待关系和治理节奏意味着什么”。

## 7. 表 2：八步与三包写入时机对照表

| 步骤 | 主要写入包 | 写入性质 | 封口要求 |
| --- | --- | --- | --- |
| `1` | 无 | 只切批次，不写三包正文 | 不适用 |
| `2` | 无 | 只产出一次特征账与动作记录 | 不适用 |
| `3` | 无 | 只产出二次特征账与判断候选 | 不适用 |
| `4` | `线程轮次上下文包` | 写入 `世界变化 / 结果变化 / 延迟事实 / 队列 / 预算 / 判定依据` 草稿段 | 不封口 |
| `5` | `线程轮次上下文包` | 写入 `需求治理 / 仲裁 / 队列调整 / 方法禁止项`，形成本轮共识 | `根层重判完成后封口` |
| `6` | `一步治理任务包（0..N）` | 按动作逐包写入 `来源段 / 任务入口段 / 主派发动作段` | `主派发决议生成后封口各包线程侧字段` |
| `7` | `回流结算包` | 写入 `结果回流 / 父任务路由 / 中间状态 / 否定项候选` 草稿段 | 不封口 |
| `8` | `回流结算包` | 写入 `双值结算 / 学习候选 / 故障收口 / 下一轮唤醒原因` | `本轮结算完成后封口` |

必须固定：

1. `线程轮次上下文包` 的正式封口点在第 `5` 步之后，不得拖到 `主派发` 之后再改根层共识。
2. 每个 `一步治理任务包` 只承接同一份 `主派发决议` 下的一个 `派发动作`；同一决议可生成 `0..N` 个彼此不冲突的任务包，但不得把多个动作硬塞进同一包。
3. `回流结算包` 必须覆盖 `最终结果段 + 关键中间状态段 + 提示段`，且必须在第 `8` 步统一封口。
4. 线程状态变化消息的发件线程只负责上报事实；真正的线程状态落账、状态节点写入和动作动态沉淀，都必须发生在第 `2` 步。


## 8. 推荐的最小接口骨架

建议把这条治理链先压成如下最小接口：


必须固定：

1. `规范化消息_为特征变化` 只负责翻译和落账，不得偷做根层裁决。
2. `重算受影响二次特征_` 应优先走增量重算，不要求每轮全量扫描。
3. `处理结果回流_并路由父任务_` 只负责 `全局影响判断 + 回父任务路由`，不替代父任务完成局部步骤裁决。
4. 若当前暂不单拆“影响项函数”，则允许把 `影响项生成 + 服务/安全影响摘要聚合` 先折叠进第 `4` 步，但语义上不得跳过这一层。


## 9. 三条样板链在治理表中的落位

### 9.1 控制链

`自我线程下发收束指令`
-> 第 `2` 步落成 `工作线程.控制意图 = 请求收束`
-> 第 `5` 步进入边界判断与主导去向
-> 第 `6` 步形成 `收束主派发`
-> 工作线程回报 `运行状态 = 已收束`
-> 下一轮第 `2/3/7/8` 步完成回流与结算

### 9.2 结果回流链

`子任务提交结果`
-> 第 `2` 步落成 `子任务.结果状态 = 成功`
-> 第 `3` 步重算 `依赖满足 = true`
-> 第 `7` 步回路由给父任务
-> 父任务局部摘要更新为 `当前步骤可执行 = true`
-> 第 `8` 步写入回流结算包

### 9.3 环境链

`环境监测器上报资源到位`
-> 第 `2` 步落成 `资源.可用性 = 可用`
-> 第 `3` 步重算 `依赖满足 = true`
-> 第 `4` 步进入治理帧
-> 第 `5` 步重算 `主导需求 / 可运行队列 / 有效优先级`
-> 第 `6` 步决定是否派发新的任务入口

### 9.4 单决议、多执行最小例子

若本轮根层重判后得到：

- `任务 A 可运行`
- `任务 B 可运行`
- `任务 C 需等待 A 的结果`
- `任务 D 因风险超阈值必须收束`

则本轮的 `主派发决议` 可以是：

- `并发派发 A、B`
- `向 D 下发收束`
- `C 留在等待队列`

必须固定：

- 这里仍然只有一份 `主派发决议`
- 但该决议下包含了多个 `可并发派发动作`
- `C` 虽未被派发，也仍然属于同一份决议的正式处理结果


## 10. 完成标准

达到下面 7 条，就可视为这轮“消息后治理链”已经站稳：

1. 任意正式消息都能先落成 `一次特征变化 + 动作记录`，而不是被直接拿去做最终裁决。
2. `依赖满足 / 当前步骤可执行 / 反向推进标记 / 有效优先级候选` 至少能从一次特征变化增量重算。
3. `治理帧` 已成为 `根层重判` 的统一输入，不再直接拿散落消息字段做裁决。
4. 一轮只生成一个 `主派发决议`，但该决议下可以包含 `0..N` 个彼此不冲突的派发动作，并与 `结果路由` 明确分层。
5. `线程轮次上下文包 / 一步治理任务包 / 回流结算包` 的写入时机与封口点已经固定。
6. `学习触发` 和 `故障收口` 已被放回第 `8` 步后置处理，不再抢在根层重判之前展开。
7. 已经明确线程状态变化只能由发件线程先上报，再由 `自我线程` 在第 `2` 步统一落账。


## 11. 一句话收口

`自我线程` 收到消息后的实现，不应再理解为“收消息然后直接发任务”，而应固定理解为：

`冻结消息批次`
-> `消息翻译为一次特征变化`
-> `归并与对账`
-> `一次特征变化重算二次特征`
-> `生成影响项`
-> `更新根值/需求树/任务权重`
-> `生成治理帧`
-> `根层重判`
-> `主派发`
-> `结果路由`
-> `本轮结算与学习后置`

只有这条链固定住，`0035/0036` 的消息合同才真正有线程承接面，`0020/0021` 的三包合同也才真正有稳定写入时机；线程状态真值也才真正只会在 `自我线程` 的消费阶段落账。
