## 1. 目标

本文用于把 `方法来源` 与 `方法成熟度阶段` 的概念口径继续收成代码侧可直接落地的规则表，回答以下问题：

- 方法头的宽松出生、低成熟度落账与严格可信升级，到底由哪些字段和哪些模块主裁判

本文的定位不是重新定义全部实现细节，而是把现有口径翻成：

- `统一方法头出生入口`
- `统一来源判定字段`
- `统一成熟度判定字段`
- `统一升格阈值`
- `统一写回位置`

## 3. 核心结论

必须固定以下 19 条：

1. 只要已经正式生成了 `方法首节点`，就必须进入方法树。
2. 还只是 `候选想法 / 元层候选 / 任务实例 / 任务路径原料` 时，不得假装已经是方法树正式节点。
3. `是否进入方法树` 的主裁判是“有没有正式 `方法首节点`”，不是“这个想法是不是看起来像方法”。
4. `方法头是否出生` 与 `方法是否可信/是否成熟` 必须分开判定。
5. `初次落账` 与 `后续成熟度升格` 必须分开判定。
6. `初次落账` 的主裁判，是出生路径、原料主导类型，以及是否已有适配方法树可以承接这批新证据。
7. 运行时新方法头可以宽松出生，但首次应默认落到低成熟度阶段，如 `运行期临时` 或 `反推候选`。
8. `后续成熟度升格` 的主裁判，仍然是 `稳定复现`；原料类型只决定更适合往哪里成熟，不替代稳定复现阈值。
9. `方法来源` 主要回答“怎么来的”；`方法成熟度阶段` 主要回答“多可信、能否默认召回/执行”；当前展示层里的 `运行期临时 / 反推 / 组合` 可视作过渡期混合标签。
10. `本能` 与 `外部导入` 是初始化或显式注入来源，不应被运行时自动升格链随意改写；工程落地时也应优先复用现有 `方法首节点主信息 + 方法虚拟存在特征类型`，而不是另起一套独立成熟度数据结构。
11. 任一新方法头的出生、来源写回和成熟度升格，都只能发生在根层仍允许继续推进的前提下。
12. 任一候选方法或升格结果在当前治理轮被消费后，若会影响继续/停止/收束路径，仍必须回到外层重新做 `根层重判 -> 上下文刷新`。
13. `成熟度升格结果` 在本文中首先属于承接层结果；只有外层最终裁成 `允许升格`，它才算正式改写主闭环的默认可用性与后续走向。
14. `方法存在`、`方法可用`、`方法可执行`、`方法成熟/升格` 必须分开，不得因为当前不可用或不可执行就否定方法头出生。
15. 正式 `方法来源` 的精确分类按 [0569_方法来源边界与正式来源分类规范.md](D:\鱼巢\规范\0569_方法来源边界与正式来源分类规范.md) 执行；旧字段中的 `运行时学习沉淀` 只保留兼容总类意义，不再作为首选正式来源值。
16. 所有由 `自我动作动态信息` 生成的因果，都允许先进入 `正式方法首节点出生判定`；若不存在适配旧方法，则可直接出生低成熟度方法首节点；学习不负责方法头出生，只负责后续补全和验证。
17. `自我动作动态因果` 的“先出生方法头，再进入学习补全/验证”是一条正式允许的承接路径；它不要求先完成正式学习，先出生的方法头默认仍应落在 `运行期临时` 或 `反推候选`。
18. 对可执行方法树，若 `动作不同` 或 `条件组合不同`，则应独立生成另一棵方法树；不得仅因为结果同类就强制并树。
19. `主结果特征类型` 在工程上优先承担索引桶职责；同一特征类型下允许存在多棵方法树，真正的唯一性仍须回到 `动作 + 条件组合 + 主结果语义` 判定。

一句话收口：

`先判断这个对象是否已经成为正式方法头，再把“来源”和“成熟度阶段”分开写入；方法头可以宽松出生，但可信度只能靠稳定复现和结果确认严格裁判。`

并且必须继续固定：

- `升格证据成立` 不等于 `已经允许升格`
- `允许升格` 的最终边界，按 [0567_最终裁决层与外层重判边界规范.md](D:\鱼巢\规范\0567_最终裁决层与外层重判边界规范.md) 执行
- `方法头出生` 的最小条件与四层分工，按 [0568_方法存在可用可执行成熟分层规范.md](D:\鱼巢\规范\0568_方法存在可用可执行成熟分层规范.md) 执行
- `方法头出生` 的最小必要信息组，按 [0571_方法头出生最小必要信息规范.md](D:\鱼巢\规范\0571_方法头出生最小必要信息规范.md) 执行
- 学习每次尝试后的结果允许先直接写入方法证据面，但这类直接写入不自动等于成熟度升格


## 4. 三个硬边界

### 4.1 什么必须进入方法树

以下对象一旦已经有 `方法首节点`，就必须进入方法树：

- 本能方法头
- 运行期写回的新首节点
- 从成功片段提炼出的正式方法头
- 从因果证据正式生成的候选方法头
- 从稳定任务路径或任务子树抽象出的正式方法头
- 外部显式导入并已注册的方法头

必须固定：

- 这里的“进入方法树”指的是进入现有 `方法链`
- 而不是只在元层留一条“方法实体”记录
- 这里不额外要求“当前自我已可执行该方法”；方法一旦已经能被结构化表达为正式方法头，就应允许进入方法树
- 但进入方法树前，仍必须先做一次 `动作键 + 条件主键 + 主结果特征类型 + 主结果方向或主结果语义` 的唯一性检查；若命中旧头，则走回补旧头，而不是新建平行头

### 4.2 什么暂时不进入方法树

以下对象在未生成正式 `方法首节点` 前，不得直接算作方法树节点：

- 仅存在于元层的候选方法描述
- 学习对象原料
- 单次任务路径实例
- 尚未抽象完成的任务树结构
- 未验证的外界因果链
- 仅在调度器或展示层里的建议候选

必须固定：

- `方法树正式节点`
- `元层候选`
- `任务实例`
- `因果原料`

这四者不能混写成一个概念。

### 4.3 什么是方法来源，什么是方法成熟度阶段

必须固定：

- `方法来源` 记录“这个方法主要怎么来的”
- `方法成熟度阶段` 记录“这个方法现在多可信、允许怎样被使用”
- `来源` 与 `成熟度阶段` 可以组合记录，不能互相冒充

例如：

- `来源=本能，成熟度阶段=已验证稳定`
- `来源=因果抽象，成熟度阶段=反推候选`
- `来源=观察抽象，成熟度阶段=运行期临时`
- `来源=路径提炼，成熟度阶段=运行期临时`

### 4.4 与任务管理治理分支的边界

必须固定：

1. 本文描述的是全域“方法来源与成熟度阶段判定”的共性链路。
2. 若服务对象是 `任务管理方法` 分支，则必须额外携带：
- `最近根层裁决结果`
- `当前缺口类型`
- `当前下一步去向`
3. 这三项属于治理上下文，不属于所有方法头统一必填身份字段。


## 5. 统一判定字段

代码侧做来源判定与成熟度升格时，至少应统一收集以下字段：

### 5.1 身份字段

- `方法首节点指针`
- `动作句柄类型`
- `动作句柄主键`
- `方法首节点主信息类::来源`
- `方法主作用表达`
- `方法最小结构承载类型`
- `方法虚拟存在::特征_成熟度阶段`
- `方法虚拟存在::特征_当前可信度Q10000`
- `方法虚拟存在::特征_是否允许自动召回`
- `方法虚拟存在::特征_是否允许默认执行`

### 5.2 出生路径字段

- `是否本能注册`
- `是否外部显式导入`
- `是否来自成功片段提炼`
- `是否来自四域方法树写回`
- `是否来自因果学习生成`
- `是否来自观察结果/对比模式抽象`
- `是否来自稳定任务路径/任务子树抽象`
- `是否来自目标/方法需求位驱动检索`
- `命中的状态列表同类特征键`
- `是否命中已有适配方法`

### 5.3 证据主导字段

- `是否主导原料为自我动作动态`
- `是否主导原料为外界因果`
- `是否主导原料为轻量因果链`
- `是否主导原料为稳定任务路径`
- `是否主导原料为稳定任务子树`
- `动作动态同桶复现次数`

### 5.4 升格裁判字段

- `方法成熟度阶段`
- `方法连续稳定次数`
- `方法可靠度_Q10000`
- `方法结果分裂次数`
- `外部结果确认状态`
- `因果链最弱跳是否未确认`
- `是否已形成可复用通用步骤骨架`
- `是否已证明能压缩筹办/搜索/推演成本`
- `条件归桶一致性`
- `结果归桶一致性`

### 5.5 学习写回辅助字段

- `学习结果_已接入方法成长`
- `学习结果_学习成长后来源名称指针`
- `学习目标类型`
- `学习对象类型`
- `来源方法需求位`

### 5.6 存储落点约束

必须固定：

1. `方法来源` 优先写在 `方法首节点主信息`，因为它回答的是身份归因。
2. `方法成熟度阶段`、`当前可信度Q`、`自动召回权限`、`默认执行权限` 优先写成 `方法虚拟存在` 的特征类型。
3. 若现有热路径已经能同步场景特征到 `方法虚拟存在`，则应复用该特征同步链，不另造平行账本。
4. 除非现有结构无法表达，否则默认禁止为“方法成熟度”单独引入新的独立结构体、独立链或独立账本。

## 7. 后续成熟度升格规则表

### 7.1 总原则

必须固定：

- `正式成熟度升格` 的主裁判是 `稳定复现`
- `证据主导类型` 只决定“更适合往哪里升”
- 不替代稳定复现阈值本身
- 不得把 `来源改写` 和 `成熟度升格` 混写成一个动作

| 当前成熟度阶段 | 目标成熟度阶段 | 最低稳定复现门槛 | 证据主导要求 | 额外约束 |
| --- | --- | --- | --- | --- |
| `运行期临时` | `反推候选` | `>= 3` | 主导证据偏稳定因果、因果链或自我动作动态桥接 | 不得存在 `未确认外部结果` 作为最弱跳 |
| `运行期临时` | `已验证稳定` | `>= 3` | 主导证据已完成结果确认，条件/结果归桶一致 | 不得只靠动作记账次数直接升格 |
| `反推候选` | `已验证稳定` | `>= 3` | 主导证据偏稳定因果、最弱跳已确认 | 必须证明不是一次偶然成功 |
| `已验证稳定` | `组合成熟` | `>= 5` | 主导证据偏稳定任务路径或任务子树 | 必须证明已从“证据推导”收敛到“结构复用” |
| `本能/外部导入` | 保持原来源 | 不适用 | 不适用 | 来源不属于运行时成熟度升格链对象 |

### 7.3 组合成熟的附加条件

除稳定复现阈值外，升格到 `组合成熟` 还必须同时满足：

1. 该结构能脱离当前宿主任务复用。
2. 该结构能显著减少显式筹办、搜索或推演。
3. 该结构中的关键动作都能展开到既有本能、既有元函数或已沉淀方法片段。
4. 该结构不是只因为路径长、步骤多就被误判为高层组合。


## 8. 禁止规则表

以下情况必须禁止直接落账或禁止直接升格：

| 场景 | 必须禁止的动作 | 原因 |
| --- | --- | --- |
| 只有任务实例，没有正式方法头 | 直接写来源 | 任务树不是方法树 |
| 只有元层候选，没有方法头 | 直接进方法树 | 元层候选不等于结构真值 |
| 一次偶然成功片段 | 直接写 `组合成熟` | 缺稳定复现 |
| 只有动作动态重复记账，缺结果确认 | 直接写 `已验证稳定` | 动作动态复现不等于稳定复现 |
| 只有外界因果命中，未确认外部结果 | 直接写高置信成熟枝 | 最弱跳未确认 |
| 因果链很长，但结构未稳定 | 直接写 `组合成熟` | 长因果链不等于稳定结构 |
| 已有适配方法树可承接同批证据 | 强制并行新生方法树 | 仅当 `动作 + 条件组合 + 主结果语义` 同键时，才应优先回补旧方法树 |
| 外部导入方法在运行中被普通学习链命中 | 自动改写来源为运行时来源 | 导入来源应保留原始来源标签 |
| 本能方法仅因为补了条件或结果 | 改写来源为运行时来源 | 本能来源应保留，本体可成长但来源不应被抹掉 |


## 9. 模块职责分配

### 9.1 出生判定入口

代码侧来源初次落账，应优先集中在以下入口：

- [方法类.cpp](/D:/鱼巢/方法类.cpp) 的 `注册方法首节点`
- [方法类.cpp](/D:/鱼巢/方法类.cpp) 的 `查找或创建_本能方法首节点`
- [四域方法树写回器_v0.ixx](/D:/鱼巢/四域方法树写回器_v0.ixx) 的写回请求来源字段

必须固定：

- 不允许多个热点模块各自发明一套独立来源判定逻辑

### 9.2 升格主裁判

代码侧后续升格，应优先集中在：

- `四域方法树生长判定器_v0`
- 学习结果接入方法成长判据的统一入口

必须固定：

- 来源改写与成熟度升格都不由展示层裁判
- 不由元层候选评分器裁判
- 不由因果模块单独裁判

### 9.3 元层职责

元层可以：

- 镜像来源结果
- 记录学习回写和适配关系
- 做查询和展示

但必须固定：

- 元层不得反向替代方法树结构真值
- 元层方法实体不应先于方法首节点真值宣布来源

## 11. 最小伪代码口径

建议统一按以下顺序实现：

```text
if (最近根层裁决结果 != 允许继续推进):
    不新生方法头
    不做普通成熟度升格
    返回外层

if (!已有正式方法头):
    不进入方法树
    只保留在元层/学习层/任务层

if (显式本能注册):
    来源 = 本能
    成熟度阶段 = 已验证稳定
elif (显式外部导入):
    来源 = 外部导入
    成熟度阶段 = 已验证稳定
elif (已有适配方法可承接当前证据):
    不新生方法头
    回补原方法的适配边界/条件模板/结果模板
else:
    if (主导证据偏稳定因果
        and 条件结果边界较清晰):
        来源 = 因果抽象
    elif (主导证据偏重复观察/对比模式
          and 当前未形成稳定因果闭合):
        来源 = 观察抽象
    elif (主导证据偏稳定路径/任务结构/成功片段):
        来源 = 路径提炼
    else:
        来源 = 旧兼容值_运行时学习沉淀

    if (来源 == 因果抽象
        and 条件结果边界较清晰):
        成熟度阶段 = 反推候选
    else:
        成熟度阶段 = 运行期临时

写方法头来源
写方法虚拟存在特征: 成熟度阶段/可信度/召回权限
写方法沉淀来源特征

if (成熟度阶段 in [运行期临时, 反推候选]
    and 稳定复现 >= 3
    and 条件结果归桶一致
    and 最弱跳已确认):
    升格到 已验证稳定

if (成熟度阶段 == 已验证稳定
    and 稳定复现 >= 5
    and 主导证据偏稳定任务结构
    and 已证明可抽象成通用骨架
    and 已证明能简化思考成本):
    升格到 组合成熟

写回后:
    返回外层做 根层重判 -> 上下文刷新
```


## 12. 最小验收标准

视为满足本文的最小实现，应至少做到：

1. 所有正式生成的 `方法首节点` 都能进入方法树。
2. 本能方法的来源稳定写为 `本能`，不被普通学习链抹掉。
3. 运行时后天新头默认先落为低成熟度阶段，如 `运行期临时` 或 `反推候选`。
4. 正式来源应优先写成 `因果抽象 / 观察抽象 / 路径提炼` 之一；旧的 `运行时学习沉淀` 只保留兼容意义。
5. `已验证稳定` 与 `组合成熟` 的成熟度升格统一由稳定复现主裁判。
6. 学习任务镜像、方法虚拟存在特征、方法头来源三处写回语义一致，且不另起平行成熟度结构。


## 13. 一句话收口

`代码侧的正确做法不是“看到像方法的东西就只给一个混合标签”，而是先确认是否已经形成正式方法头，再把来源和成熟度阶段分开写入，最后只让稳定复现统一裁判它是否足以成为可信成熟方法。`
