## 4. v1 的范围与不范围

### 4.1 v1 只做到哪里

`v1` 当前只要求把 `状态/事件驱动 + 根需求形成 + 需求树/需求队列治理 + 方法禁止项更新 + 任务入口实例化 + 结果路由 / 结果回流` 这条最小闭环，真正落到当前仓库代码里。

也就是说，`v1` 的完成线只到：

`单线程 / 单轮闭环 / 建议型输出 / 成功与失败都结算 / 动态与因果候选都能回写`

### 4.2 v1 先不做到哪里

`v1` 当前明确不做：

- 多线程并发治理扩展
- 真实自动执行闭环
- 自动回滚与自动联动改写
- 完整权限链、安全接管链和多角色协同链实装
- 大规模重写 `任务管理任务模块` 全部旧流程
- 先把整个仓库迁成 `.ixx` 物理文件结构

### 4.3 v1 的总目标

必须固定：

`v1` 的总目标不是把所有规范一次性写进代码，而是让当前仓库先真实跑出一条可验证的最小治理闭环主线。


## 5. 当前代码现实与工程判断

### 5.1 当前最稳的切入口

当前最稳的切入口不是全面重构，而是：

1. 先把 `三包` 落成可编译结构
2. 再把主循环切薄，并显性化 `根需求 -> 需求树 -> 需求队列 -> 方法禁止项集 -> 全局队列`
3. 再把 `任务入口实例化 + 任务管理工作线程调度` 从主循环里抽成稳定桥位
4. 最后再把 `派生需求转任务 + 结果路由 + 最小回写` 接上

### 5.2 当前最重要的工程现实

当前仓库现实必须承认：

1. 当前主锚点仍在 [自我线程类.cpp](D:\鱼巢\自我线程类.cpp) 和 [任务管理任务模块.cpp](D:\鱼巢\任务管理任务模块.cpp)。
2. 当前规范中建议的若干 `.ixx` 模块，代码里并未完整物理存在。
3. 因此 `v1` 落地时，应优先沿现有 `.h / .cpp` 体系增量接入，而不是先做文件系统级迁移。
4. 逻辑模块名可以先与规范对齐，但物理文件名允许先按现有仓库过渡。

### 5.3 当前命名的落地口径

本计划统一采用以下口径：

- 文档层：`任务管理中轴`
- 代码层过渡承载：`任务管理任务模块.*`
- 下一步建议显式导出的代码入口：`任务管理模块类::生成一步治理(...)`
- 运行态产物：`结构_一步治理任务包`

必须固定：

`先统一逻辑入口，再谈是否迁文件名。`


## 6. 总体落地原则

必须固定以下 `9` 条总原则：

1. `先小切口，后全链扩展`
2. `先单轮闭环，后多轮调度`
3. `先建议型输出，后真实执行`
4. `先成功/失败都结算，后谈成长`
5. `先把根需求 / 需求树 / 需求队列 / 方法禁止项治理从主循环里显性化，再往下交给任务管理任务`
6. `先把任务入口实例化、工作线程调度和结果路由从主循环里抽清，再往里塞策略沙盘与方法过滤`
7. `先沿现有 cpp/h 承载落地，不强制先改物理文件后缀`
8. `先桥接，不暴力重写`
9. `先让代码按规范跑起来，再考虑风格统一和模块美化`
