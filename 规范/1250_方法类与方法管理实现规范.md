# 1250 方法类与方法管理实现规范

## 1. 目标

本文用于固定当前 `方法类.cpp` 与 `方法管理模块_v0.ixx` 的实现边界，回答以下问题：

- 当前方法系统已经实现了哪些能力
- 方法首节点、条件节点、结果节点与方法虚拟存在分别负责什么
- 方法索引、运行证据与方法管理应如何分层
- 哪些事情属于方法系统，哪些不属于

本文对应当前工程实现：

- `方法类.cpp`
- `方法管理模块_v0.ixx`
- `主信息定义模块.ixx`


## 2. 定位

当前方法系统的定位不是单纯的“方法仓库”，而是四层组合：

- `方法结构层`：创建和维护方法首节点、条件节点、结果节点
- `方法账本层`：维护方法虚拟存在、内部状态场景与运行证据
- `方法索引层`：维护动作名、动作句柄主键、主作用特征类型、作用方向、结果 token、主信息指针等索引
- `方法管理层`：基于元层实体和关系做候选查询、绑定、替换与结果记录

当前方法系统不负责：

- 根需求和根方向定义
- 全局任务调度
- 叶子步骤的直接执行推进
- 把学习闭环直接替代成另一套方法树系统

必须额外固定：

- `动作名 / 动作句柄主键 / 主作用特征类型 / 作用方向 / 条件主键 / 结果主键 / token` 统一属于 `方法索引与归桶辅助字段`
- 它们负责 `粗召回 / 去重 / 归桶 / 索引 / 证据摘要`
- 它们不直接承担 `方法语义真值`，也不直接判定 `需求是否满足 / 方法是否正确 / 任务是否成立`
- `二次特征 / 条件模板 / 结果摘要` 在方法系统里统一视为运行时证据面；真正的运行时放行、切换、结束和升阶，按 [0577_原语层与选择函数规范.md](D:\鱼巢\规范\0577_原语层与选择函数规范.md) 经 `原语层 -> 选择函数` 执行


## 3. 当前方法模型

当前实现中的方法模型，应统一理解为：

`方法首节点 -> 条件节点 -> 结果节点`

同时并行挂接两套运行期承载：

- `方法虚拟存在`
- `方法内部状态场景`

三层节点的职责应固定为：

- `方法首节点`
  - 承载动作句柄、本能 ID、来源、主作用特征类型、作用方向与作用域
  - 作为方法召回与绑定的统一锚点
- `条件节点`
  - 作为当前方法树下这组具体条件组合的真值承载
  - 以条件主键承载场景模板与条件判定索引
- `结果节点`
  - 作为结果桶
  - 以结果主键、结果摘要字段与结果 token 承接结果模板、召回入口和运行证据

方法虚拟存在的职责应固定为：

- 挂方法运行状态
- 挂历史运行次数、成功次数、失败次数、最近错误码、最近调用时间
- 挂方法成熟度阶段、当前可信度Q、自动召回权限、默认执行权限等运行期判断特征
- 作为条件桶虚拟存在、结果桶虚拟存在的父锚点

这里必须进一步固定：

- `方法来源` 仍属于方法首节点身份信息，优先落在 `方法首节点主信息类::来源`
- `方法成熟度阶段 / 当前可信度Q / 自动召回权限 / 默认执行权限` 属于运行期判断信息，优先落在 `方法虚拟存在` 的特征类型
- 若当前实现仍保留 `枚举_方法沉淀来源::运行期临时 / 反推 / 组合` 这样的兼容来源值，也不得再把它们同时当成“成熟度阶段”的唯一真值
- `方法存在 / 方法可用 / 方法可执行 / 方法成熟(升格)` 的统一分层，按 [0568_方法存在可用可执行成熟分层规范.md](D:\鱼巢\规范\0568_方法存在可用可执行成熟分层规范.md) 执行
- 正式 `方法来源` 的精确分类，按 [0569_方法来源边界与正式来源分类规范.md](D:\鱼巢\规范\0569_方法来源边界与正式来源分类规范.md) 执行
- `方法头出生` 的最小必要信息，按 [0571_方法头出生最小必要信息规范.md](D:\鱼巢\规范\0571_方法头出生最小必要信息规范.md) 执行

必须额外固定：

- `方法首节点` 首先承载的是“方法存在面”。
- 当前可用、当前可执行、当前成熟与否，都不是方法头出生的必要条件。
- 因此不能因为“当前自我还不能执行它”，就拒绝为该方法建立正式头节点。
- 新生方法头至少应能落下：`身份锚点 + 主作用表达 + 来源归因 + 最小结构承载`。
- 工程组织上，一棵可执行方法树默认只承接：`一个动作骨架 + 一组具体条件组合 + 一类主结果语义`。
- 对可执行方法树，唯一性主键至少应覆盖：`动作句柄主键或动作名 + 条件主键 + 主结果特征类型 + 主结果方向或主结果语义`。
- 若 `动作不同` 或 `条件主键不同`，应独立生成另一棵方法树，不再强求并到同一方法头下。
- `结果特征类型` 优先承担索引桶职责；它用于快速找候选方法树，不单独承担并头职责。
- 若出现更好的方法候选，允许切换当前方法挂点；切换不等于删除旧方法，旧方法仍应保留为已存在方法与证据面。

### 3.1 运行时原语绑定

方法节点在运行时至少应带以下绑定：

- `guard_primitives`
  - 解释前提/条件是否满足
- `direction_primitives`
  - 解释方向是否连续一致
- `result_primitives`
  - 解释当前结果是否落入目标范围
- `conflict_primitives`
  - 解释当前结果是否与目标方向或安全边界冲突
- `escalation_primitives`
  - 解释当前是否必须从 `2次特征` 升到 `3次 / n次特征`
- `selection_policy`
  - 根据原语输出选择下一执行原语、下一方法节点或结束去向

必须固定：

1. `条件节点` 与 `结果节点` 继续负责承载 `条件判定索引 / 结果摘要索引 / 场景模板 / 结果模板`。
2. 这些模板和索引是方法运行时证据输入，不是最终执行控制器。
3. 方法系统可以召回候选、组织模板、保存证据，但不得裸读二次特征就直接判 `可执行 / 已达标 / 应切换下一步`。


## 4. 当前已经实现的能力

### 4.1 本能方法首节点出生

当前已经实现：

- 按本能动作 ID 查找或创建方法首节点
- 以动作句柄和本能 ID 为主键生成先天方法头
- 初始化方法虚拟存在与方法内部状态场景
- 由自我线程初始化阶段统一触发默认本能注册与方法头建档

这意味着：

- 本能函数并不是绕过方法树直接裸调用
- 本能函数一旦进入方法系统，就应尽量拥有稳定的方法头锚点
- 本能方法头的“首次建档”应在初始化阶段完成，运行期只做复用或证据写回

### 4.2 场景事实同步到方法虚拟存在

当前已经实现：

- 从输入场景同步特征到方法虚拟存在
- 从输出场景同步特征到方法虚拟存在
- 标记方法调用开始、结束
- 在结束阶段写入运行证据

这里的实现口径必须固定为：

- `输入/输出场景` 是证据来源
- `方法虚拟存在` 是方法运行账本
- 新增的运行期判断信息，优先沉淀为 `方法虚拟存在` 的特征类型
- 热路径上允许保留最小账本，重写回和扩展整理可延后到低频路径

建议优先复用或补齐以下字段/特征名：

- 方法头身份字段
  - `方法首节点主信息类::来源`
  - `方法首节点主信息类::作用域`
- 方法虚拟存在特征
  - 复用 `特征_方法沉淀来源_`
  - 复用 `特征_方法可靠度_Q10000_`
  - 复用 `特征_方法稳定出现次数_` 作为最小稳定计数
  - 新增优先建议：`特征_方法成熟度阶段_`
  - 新增优先建议：`特征_方法允许自动召回_`
  - 新增优先建议：`特征_方法允许默认执行_`
  - 若后续确认“稳定出现”和“稳定复现”语义不足以共用，再补：`特征_方法稳定复现次数_`

命名纪律必须与现实现风格对齐：

- 词面建议分别为 `方法_成熟度阶段`、`方法_允许自动召回`、`方法_允许默认执行`、`方法_稳定复现次数`
- 不得为这些判断信息再起平行 `struct/class/账本`

### 4.3 条件节点与结果节点的桶化创建

当前已经实现：

- 按场景模板生成条件主键并取或创建条件节点
- 按结果主键或结果摘要字段取或创建结果节点
- 在命中已有桶时融合模板内容
- 吸收条件判定索引与结果摘要索引

因此当前条件/结果节点应理解为：

- 不是一次性实例结果
- 而是供召回、复用、统计和写证据使用的“方法桶”

### 4.4 运行证据写回

当前已经实现的证据写回层级是：

`方法虚拟存在 -> 条件桶虚拟存在 -> 结果桶虚拟存在`

这条写回链承担：

- 运行次数累计
- 成败次数累计
- 最近错误码更新
- 样本数累计

因此必须固定：

- 方法统计不是只记在方法首节点
- 条件桶和结果桶也要独立沉淀局部证据

### 4.5 写回位置清单

在当前实现里，方法出生、运行、学习回补至少应按以下位置写回：

1. 方法首次注册时
   - 先补齐 `方法头出生最小必要信息`
   - 写 `方法首节点主信息类::来源`
   - 调 `初始化方法虚拟存在信息(...)`
   - 调 `同步方法沉淀来源标准特征(...)`
   - 通过 `同步方法成熟度与权限标准特征(...)` 初始化 `特征_方法成熟度阶段_ / 特征_方法允许自动召回_ / 特征_方法允许默认执行_`
   - 通过 `同步方法可靠度标准特征(...)` 初始化 `特征_方法可靠度_Q10000_`
2. 运行开始与结束时
   - 继续走 `记录方法调用开始(...) / 记录方法调用结束(...)`
   - 写 `特征_方法运行状态_ / 特征_运行次数_ / 特征_成功次数_ / 特征_失败次数_ / 特征_最近成功标记_ / 特征_方法最后调用时间_`
3. 因果回补到已有方法时
   - 优先把本轮学习或因果结果直接写入已有方法的条件节点 / 结果节点 / 方法虚拟存在
   - 同步更新该方法头对应的 `方法虚拟存在` 上的成熟度、可信度和权限特征
   - 不额外创建平行“候选方法账本”
4. 新生低成熟度方法头时
   - 仍通过方法注册链创建方法首节点
   - 来源字段按当前工程兼容值写入
   - 成熟度阶段优先写到 `方法虚拟存在`，不得再只靠来源字段冒充
5. 稳定复现后升格时
   - 优先调 `同步方法成熟度与权限标准特征(...)`
   - 再调 `同步方法可靠度标准特征(...)`
   - 管理判定或展示读取时，优先走 `读取方法成熟度阶段标准特征(...)`
   - 默认不强制篡改历史来源字段

### 4.6 当前方法挂点的切换纪律

必须固定：

1. 若存在更好的方法候选，允许替换 `当前方法首节点挂点`。
2. 这里的“更好”当前至少包括：
   - 否决项更少
   - 同方向覆盖更高
   - 条件更容易满足
   - 可靠度或稳定复现更高
3. 切换当前方法挂点，不得删除旧方法。
4. 旧方法应继续保留为：
   - 已存在方法
   - 运行证据来源
   - 后续可再次比较的候选

### 4.5 索引维护与召回

当前已经实现的索引至少包括：

- 动作名到方法首节点
- 主作用特征类型与作用方向到方法首节点
- 结果 token 到结果节点
- 结果节点到方法首节点
- 主信息指针到方法节点
- 后续应补或统一成两层：
  - `主结果特征类型 -> 方法树列表`
  - `动作句柄主键或动作名 + 条件主键 + 主结果特征类型 + 主结果方向或主结果语义 -> 方法首节点`

当前也已经实现：

- 索引脏标记
- 按需重建索引
- 在读方法链时构建临时索引，再一次性交换

因此索引层的职责应固定为：

- 只负责加速查找
- 不能替代方法链真值
- 不能在索引不一致时改写方法链内容
- `主作用特征类型 / 作用方向命中` 只能表示“值得进入候选”
- `结果 token / 结果主键命中` 只能表示“值得进入结果比对或因果查询”
- `主结果特征类型` 命中，优先用于找到候选方法树列表
- `动作句柄主键或动作名 + 条件主键 + 主结果特征类型 + 主结果方向或主结果语义` 命中，优先用于判定“是否已存在同键方法树”，而不是直接并行新生头
- 最终是否可执行、是否命中需求、是否继续推进，仍必须由外层 `二次特征` 判定

### 4.6 方法管理模块的职责

`方法管理模块_v0` 当前已经实现一层独立的元层方法管理能力，主要负责：

- 注册方法元实体
- 查询候选方法
- 为任务选择最佳方法
- 绑定、替换任务与方法
- 记录执行结果和适配关系

它的评分依据已包含：

- 任务类型与方法适用类型
- 标签匹配
- 安全值投影风险
- 服务值投影收益
- 历史成功率
- 绑定关系中的累计修正

因此必须固定：

- `方法类` 负责方法树和方法账本
- `方法管理模块_v0` 负责元层候选与管理决策
- 二者互补，不应混成单一巨型模块


## 5. 锁与一致性边界

当前方法系统至少存在以下锁：

- `方法链.链表锁`
- `元缓存_mtx_`
- `本能_mtx_`
- `idx_mtx_`

它们的职责应固定为：

- `方法链.链表锁`
  - 保护方法树真值
  - 保护节点结构、模板融合与 `_已加锁` 写操作
- `元缓存_mtx_`
  - 保护方法虚拟存在相关缓存与内部元数据缓存
- `本能_mtx_`
  - 保护本能动作注册与本能查找缓存
- `idx_mtx_`
  - 保护索引缓存与脏标记

必须固定以下实现口径：

1. 方法链是真值，索引只是缓存。
2. 方法链发生结构或摘要变化后，应标记索引脏，而不是在热点路径立刻全量重建。
3. 重建索引时，应优先读方法链构建临时索引，再短时持有 `idx_mtx_` 交换结果。
4. 默认不允许在持有 `idx_mtx_` 时再去补拿方法链锁。
5. 如果已在方法链写锁内，只允许走专门的 `_已加锁` 路径，不得再反向嵌套拿 `idx_mtx_`。


## 6. 方法系统的硬约束

1. 方法首节点是统一锚点，任何后天条件、结果、证据都只能挂在既有方法头下扩展。
2. 条件节点和结果节点必须按“桶”理解，不得误写成一次性执行实例。
3. 方法虚拟存在是方法运行账本，不是任务调度状态机。
4. 新增的成熟度、可信度、召回权限等运行期判断信息，应优先挂在 `方法虚拟存在` 的特征类型，不得轻易另起平行独立数据结构。
5. 方法索引只负责召回加速，不得被当成真值来源。
6. 索引重建必须遵守“扫描与交换分离”，避免长时间持有缓存锁。
7. 本能方法注册、方法树写入、索引刷新必须分层，不得在同一热点路径里全部叠加。
8. `方法管理模块_v0` 的评分和绑定属于管理决策，不得反向替代方法树结构定义。
9. 方法系统可以提供候选与证据，但不得直接推进任务状态机。
10. 若已有适配方法树能承接同一 `动作句柄或动作名 + 条件主键 + 主结果语义` 的新因果证据，应优先回补既有方法树，而不是强行再生并行树。
11. 若 `动作不同` 或 `条件主键不同`，则它们应视为不同方法树；即使主结果特征类型相同，也不得仅因结果同类而强制并树。
12. 方法节点、条件节点、结果节点不得直接以裸 `二次特征` 作为最终执行条件；运行时必须判与其绑定的原语输出。
