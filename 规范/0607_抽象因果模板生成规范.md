- 抽象因果模板是什么
- 抽象因果模板与实例因果如何区分
- 抽象因果模板为什么允许跨时间、跨具体场景/存在

## 3. 基本定义

### 3.1 实例因果

`实例因果` 指的是：

- 一次真实发生过的
- 具有明确锚点、层级和时间顺序的因果事实

它的标准骨架是：

```text
条件状态集合 + 动作/事件 -> 结果状态或结果动态
```

### 3.2 抽象因果模板

`抽象因果模板` 指的是：

- 由不同时间、不同具体场景或不同具体存在中反复出现的同类实例因果
- 在删除单次时间身份和具体锚点身份后
- 按 `抽象条件模式 + 抽象动作模式 + 抽象结果模式`
- 归并形成的可复用因果模板

它回答的是：

- 这类变化通常怎么发生
- 哪些条件模式往往会导向哪些结果模式

必须固定：

- 抽象因果模板不是当前事实
- 具体场景或具体存在只能命中模板，不能定义模板
- 抽象因果模板允许跨时间、跨具体场景/存在
- 但不允许跳过实例因果合法性校验直接出生


## 4. 核心边界

### 4.1 跨时间、跨场景/存在只属于模板层

必须固定：

`跨时间、跨具体场景/存在的归并，只属于抽象因果模板层，不属于单条实例因果层。`

因此：

- `实例因果` 继续服从：
  - 同锚点
  - 同层级
  - 时间有序
- `抽象因果模板` 才允许：
  - 跨时间
  - 跨具体场景
  - 跨具体存在

### 4.2 每条样本必须先是合法实例因果

必须固定：

`每条来源样本，都必须先是合法实例因果。`

这意味着：

1. 先校验单条实例因果是否合法
2. 再决定它能否进入抽象模板候选桶

不得反过来：

- 先把若干片段并成模板
- 再回头补实例合法性

### 4.3 抽象因果不是忽略场景，而是抽象场景

必须固定：

`抽象因果不是忽略场景，而是把具体场景/存在抽象成可比较的条件模式。`

这意味着：

- 具体场景主键不进入模板身份字段
- 但场景中的关键条件结构必须进入 `条件模式`
- 若具体存在层的内部条件参与了因果，也必须进入 `条件模式`

因此：

- 抽象因果删除的是 `具体锚点身份`
- 不是删除 `锚点类型和条件结构`

### 4.4 场景模板与存在模板分通道

必须固定：

1. `场景模板` 与 `存在模板` 分通道维护
2. 不同 `锚点类型` 不得混桶
3. 即使动作名相同、结果名相近，也不能让 `场景模板` 和 `存在模板` 直接并成一个模板

### 4.5 结果模式分裂时必须拆桶

必须固定：

若不同时间、不同具体场景/存在中的样本，在条件模式和动作模式相近时，仍持续导向明显不同的结果模式，
则必须拆成不同抽象因果模板。

不得做：

- 只因动作名相同就强行合并
- 只因条件看起来相近就忽略结果分裂


## 5. 抽象因果模板的身份字段

抽象因果模板的最小身份建议由以下字段共同决定：

- `模板域 = 因果`
- `锚点类型`
- `状态层级`
- `状态面类型`
- `条件模式主键`
- `动作模式主键`
- `结果模式主键`
- `根需求投影结果`
- `证据窗口类型`

必须固定：

- `具体锚点主键` 不进入模板身份字段
- `单次时间戳` 不进入模板身份字段
- 名称字符串不承担模板身份职责

一句话收口：

`抽象因果模板的身份，应靠结构字段和模式主键，不靠某次实例或某个名字。`


## 6. 同类证据判定

一个实例因果要想进入某个抽象因果候选桶，至少应满足：

1. `锚点类型` 相同
2. `状态层级` 可比
3. `状态面类型` 可比
4. `条件模式` 同类
5. `动作模式` 同类
6. `结果模式` 同类
7. `根需求投影结果` 可比
8. `证据窗口类型` 可比

必须继续固定：

- 这些字段只负责 `归桶 / 候选聚类 / 去重`
- 不自动证明模板已成立
- 模板是否成立，仍要依赖命中次数、反例、稳定度和最近窗口表现


## 7. 标准生成链

抽象因果模板的标准生成链固定为：

```text
不同时间、不同场景/存在的合法实例因果
-> 提取条件 / 动作 / 结果
-> 将具体场景或具体存在抽象成条件模式
-> 将具体动作抽象成动作模式
-> 将具体结果抽象成结果模式
-> 删除单次时间身份与具体锚点ID
-> 按 锚点类型 + 状态层级 + 状态面类型 + 条件模式 + 动作模式 + 结果模式 + 根需求投影 + 证据窗口 归桶
-> 累计证据、冲突证据与反例
-> 形成抽象因果候选
-> 达阈值后升格为抽象因果模板
```

### 7.1 生成前提

必须至少满足：

1. 至少 `2` 条及以上同类合法实例因果
2. 这些实例因果处于可比较窗口内
3. 条件、动作、结果都已能抽成可比较模式

### 7.2 生成结果

生成结果至少应保存：

- `命中次数`
- `反例次数`
- `最近命中时间`
- `最近失败时间`
- `稳定度`
- `最近失效原因`


## 8. 命中增强、拆桶与升格

### 8.1 命中增强

当新的合法实例因果命中某个模板候选时，应至少更新：

- `命中次数`
- `最近命中时间`
- `稳定度`

### 8.2 反例与冲突

若新样本满足以下任一情况，应记入冲突或反例，而不是盲目命中：

1. 条件模式相近，但结果模式相反
2. 动作模式相近，但根需求投影结果相反
3. 时间窗已不可比
4. 同一模板桶内出现持续结果分裂

### 8.3 拆桶条件

满足以下任一条件时，应优先拆桶：

1. `结果模式` 持续分裂
2. `条件模式` 内部已出现稳定子类
3. `场景模板` 与 `存在模板` 被错误混合
4. 不同证据窗口下的表现长期不一致

### 8.4 升格条件

一个抽象因果候选要升格为稳定模板，至少应满足：

1. `命中次数 >= 最小阈值`
2. `反例次数` 未压过支持证据
3. `最近命中时间` 仍在活跃窗口内
4. `条件模式 + 动作模式 + 结果模式` 没有持续漂移
5. 相比父模板或相邻模板具备独立区分价值


## 9. 与任务管理主链的边界

必须继续固定：

1. 当前任务管理主链优先消费：
- `实例因果`
- `轻量因果链`

2. 抽象因果模板更适合服务：
- `长期策略`
- `学习反馈`
- `概念成长`
- `方法稳定化`

3. 不得把抽象因果模板直接当成本轮单步治理的硬判据

也就是说：

- `实例层` 负责回答“这轮为什么这样”
- `模板层` 负责回答“这类情况通常怎样”


## 10. 自然语言边界

抽象因果模板的自然语言表达继续服从 [2020_因果信息自然语言表达规范.md](D:\鱼巢\规范\2020_因果信息自然语言表达规范.md)：

- 使用 `通常 / 往往 / 一般会 / 可能`
- 不得写成当前事实
- 应显示 `成立次数 / 失败次数` 或同类提示

必须继续固定：

- 即使模板命中次数很高，也仍不是当前事实
- 即使反例次数很低，也不得表述成绝对真理


## 11. 一句话收口

可收敛为一句话：

`实例因果继续要求同锚点、同层级、时间有序；抽象因果模板允许跨时间、跨具体场景/存在，但只能在抽象后的条件模式、动作模式、结果模式同类时归并，而且若结果模式持续分裂，就必须拆成不同模板。`
