# 1290 因果信息模块实现规范

## 1. 目标

本文用于固定当前 `因果信息类.ixx` 的实现边界，回答以下问题：

- 当前因果模块已经实现了哪些最小能力
- 因果如何创建、累计、排序和查询
- 因果稳定度 Q 的语义是什么，不是什么
- 因果如何服务于学习与反推，又不越权替代方法验证

本文对应当前工程实现：

- `因果信息类.ixx`
- [0980_原生学习兜底函数集规范（现有实现对齐版）.md](D:\鱼巢\规范\0980_原生学习兜底函数集规范（现有实现对齐版）.md)

同时必须固定：

- 本文描述的是 `当前实现现状`
- 下一阶段因果重构的目标边界，以 [1285_因果分层建模与实例生成规范.md](D:\鱼巢\规范\1285_因果分层建模与实例生成规范.md) 为准
- `动作动态因果 -> 反推候选 -> 组合方法升格` 这条分层口径，以 [1295_动作动态因果与反推-组合方法升格规范.md](D:\鱼巢\规范\1295_动作动态因果与反推-组合方法升格规范.md) 为准
- `任务管理方法` 这条专用因果生成链，按 [1190_任务管理方法因果生成规范.md](D:\鱼巢\规范\归档\1190_任务管理方法因果生成规范.md) 继续细化
- 运行时若要用因果驱动方法、任务或执行选择，必须按 [0577_原语层与选择函数规范.md](D:\鱼巢\规范\0577_原语层与选择函数规范.md) 经 `原语层 -> 选择函数` 转译
- 当“当前实现现状”和“下一阶段建模目标”不一致时，不得把现状误写成长期上位原则


## 2. 定位

当前因果模块的定位应固定为：

`运行时因果证据仓 + 反向检索索引 + 方法生成期筛选原料层`

但必须额外固定：

- 该定位对应的是当前未完成分层重构前的运行态定位
- 后续若重构到 `实例因果信息 / 抽象因果模板 / 按需因果链筛选视图` 体系，本文需要按新实现继续修订
- `自我动作动态` 可视为当前最小高可信内部因果证据
- `外界因果` 默认属于待验证外部证据，不能只靠命中就直接产出已验证方法
- `因果链` 在当前口径下更适合作为生成方法头时按需组织的筛选/倒推视图，不应误写成常驻成熟知识层

### 2.1 在统一主链中的位置

本文对应的当前实现，默认处于以下主链中后段：

`根层方向裁决`
-> `形成具体运行上下文`
-> `状态/动态落账`
-> `因果模块创建、累计、查询候选因果`
-> `外层学习、反推、任务管理或重判消费`

必须固定：

1. 当前因果模块不负责根层方向裁决。
2. 当前因果模块返回的是运行时证据和候选，不是最终继续/停止结论。
3. `任务管理方法` 这条专用因果分支，在使用当前因果模块时，仍须先由外层显式化 `最近根层裁决结果`、`当前缺口类型`、`当前下一步去向`，再进入 [1190_任务管理方法因果生成规范.md](D:\鱼巢\规范\归档\1190_任务管理方法因果生成规范.md) 那套更细的治理专用生成链。
4. `因果条件模板 / 因果结果模板` 在运行时统一视为证据模板；“当前因果是否仍稳定成立、是否足以继续执行”必须交给正式原语解释。

它不是：

- 已验证方法库
- 任务调度器
- 跨进程持久化因果数据库


## 3. 当前因果模型

当前实现中的因果模型，最少由以下部分构成：

- `条件状态列表`
- `动作信息`
- `过程动态列表`
- `结果状态列表`
- `观察次数`
- `首次发生时间`
- `最近发生时间`
- `稳定度Q`

当前因果键由以下内容组合：

- `动作信息指针`
- `条件主键`
- `过程主键`
- `结果主键`

因此当前因果应理解为：

- 同一进程内的运行时聚合证据
- 不是跨进程稳定 ID
- `条件主键 / 过程主键 / 结果主键` 在当前实现中都只承担 `归桶定位 / 查询定位 / 候选聚类` 职责
- 它们不能直接充当 `因果成立真值`，也不能直接拿来判定 `需求是否满足` 或 `任务是否应继续`


## 4. 创建与累计规则

当前 `创建或累计因果信息` 的前提应固定为：

1. 条件状态列表非空
2. 结果状态列表非空
3. 过程动态列表非空
4. 动作信息非空
5. 条件主键、过程主键、结果主键都非零

若这些前提不满足，当前实现应理解为：

- 直接不创建
- 返回空结果

若命中相同因果键，则当前实现会：

- 复用已有因果节点
- 刷新因、果和锚点动态
- 刷新条件、过程、结果列表
- 更新最近发生时间
- 累加观察次数
- 重算稳定度 Q

若未命中相同因果键，则当前实现会：

- 新建因果节点
- 初始化首发时间和最近时间
- 初始化观察次数为 1
- 记录到正向索引和反向索引

### 4.1 任务管理专用因果的外层前提

当前模块本身仍是全域运行时证据仓，因此必须额外固定：

1. 若调用方要生成 `任务管理方法` 专用因果，仅满足本节的最小创建前提还不够。
2. 调用方还必须先在外层准备好：
- `最近根层裁决结果`
- `当前缺口类型`
- `当前下一步去向`
3. 这三项仍应由 `任务管理上下文` 承载，不应强行塞回当前全域因果键本体。
4. 因此当前因果模块可继续做“证据仓”，而治理专用因果解释仍应落在 [1190_任务管理方法因果生成规范.md](D:\鱼巢\规范\归档\1190_任务管理方法因果生成规范.md) 到 [1198_任务管理方法因果生成模式与稳定化规范.md](D:\鱼巢\规范\归档\1198_任务管理方法因果生成模式与稳定化规范.md) 这一带。


## 5. 稳定度 Q 的当前语义

当前 `稳定度Q` 的实现是启发式排序分，不是概率真值。

当前构成至少包括：

- 观察次数分
- 是否有动作信息
- 是否有条件主键
- 是否有过程主键
- 是否有结果主键

因此必须固定：

1. `稳定度Q` 只用于当前实现中的排序和粗筛。
2. `稳定度Q` 不能直接当成“因果正确概率”。
3. 后续若要升级稳定度算法，必须保证“同样本更稳定优先、更新样本更近优先”的排序语义不被破坏。


## 6. 查询与排序规则

当前模块已经实现：

- 按结果主键查询因果
- 按动作信息查询因果

两类查询的排序语义都应固定为：

1. 先按 `稳定度Q` 降序
2. 再按 `最近发生时间` 降序
3. 最后按上限截断

因此必须固定：

- 查询接口返回的是“候选因果证据”
- 不是“已经可直接执行的方法”
- `结果主键命中` 只表示“候选结果面相近”
- `条件主键命中` 只表示“候选前提面相近”
- 最终是否成立，仍必须回到更具体的状态、动态、二次特征，并经 `因果稳定原语` 或同级正式原语解释后，再进入外层重判或选择链

### 6.1 与外层重判的边界

必须继续固定：

1. 无论是按结果主键查询还是按动作信息查询，返回值都只是 `候选因果证据列表`。
2. 即使某条候选证据 `稳定度Q` 很高，也不能直接替代外层 `根层重判 -> 上下文刷新`。
3. 若查询结果被 `任务管理方法`、`反推学习` 或 `候选方法生成` 消费，也必须回到各自外层链路继续验证。
4. 因果证据若要进入运行时继续/切换/结束判断，必须先经 `guard_primitives / result_primitives / conflict_primitives / escalation_primitives` 等绑定原语转译，不得裸判模板命中。


## 7. 与反推学习的边界

当前因果模块与反推学习的关系应固定为：

- 因果模块负责沉淀和检索证据
- 反推学习负责把因果证据提炼为候选方法或回补已有方法，必要时按需组织单步或多步因果链筛选轨迹

因此必须固定：

1. 因果命中只能产出候选，不得直接产出“已验证方法”。
2. 因果模块不直接决定方法升格。
3. 任何因果到方法的转化，都必须经过学习或验证链路。
4. 若已有适配方法头能承接当前因果，应优先回补既有方法的条件/结果模板，而不是强制新生并行方法头。
5. `自我动作动态` 更适合作为 `运行期临时` 或 `反推候选` 方法头的强证据起点。
6. `动作动态复现` 只是高质量证据来源，不等于成熟度已经稳定。
7. `稳定复现` 才是方法成熟度与权限升格的主裁判。
8. `外界因果` 在未完成确认、复现或结果验证前，不得直接抬成正式方法来源。
9. `因果链` 的主要职责，是在生成方法树时服务 `反推` 与筛选，帮助长出较复杂的方法树候选，但不应被当成常驻成熟知识层。
10. `组合方法` 的收敛主裁判应是稳定任务结构与长期复现，而不是因果模块单独拍板。
11. 因果条件和因果结果继续保留为模板证据，但运行时是否满足、是否稳定、是否冲突，必须通过原语层输出后再交给选择函数消费。

### 7.1 具体字段名建议

按当前工程风格，因果到方法的落账优先建议以下字段与特征名：

- 方法头身份字段
  - `方法首节点主信息类::来源`
  - `方法首节点主信息类::作用域`
- 方法虚拟存在特征
  - 复用：`特征_方法沉淀来源_`
  - 复用：`特征_方法可靠度_Q10000_`
  - 复用：`特征_方法稳定出现次数_`
  - 新增优先建议：`特征_方法成熟度阶段_`
  - 新增优先建议：`特征_方法允许自动召回_`
  - 新增优先建议：`特征_方法允许默认执行_`
  - 若确认需要把“稳定复现”从“稳定出现”里拆出来，再补：`特征_方法稳定复现次数_`

这里必须额外固定：

- `当前可信度Q` 优先复用 `特征_方法可靠度_Q10000_`
- 不要围绕候选方法、成熟度或权限再新起平行独立数据结构

### 7.2 写回位置清单

因果证据进入方法系统时，最小写回顺序应固定为：

1. 判断是否已有适配方法头
   - 若有：优先回补该方法头下的条件节点 / 结果节点
   - 若无：才允许走新生低成熟度方法头链路
2. 新生方法头时
   - 通过方法注册链创建 `方法首节点`
   - 写 `方法首节点主信息类::来源`
   - 初始化 `方法虚拟存在`
3. 同步标准特征
   - 调 `同步方法沉淀来源标准特征(...)` 镜像来源
   - 调 `同步方法成熟度与权限标准特征(...)` 初始化或升级 `特征_方法成熟度阶段_ / 特征_方法允许自动召回_ / 特征_方法允许默认执行_`
   - 调 `同步方法可靠度标准特征(...)` 初始化或升级 `特征_方法可靠度_Q10000_`
4. 后续稳定复现时
   - 优先更新 `方法虚拟存在` 上的成熟度、可信度与权限特征
   - 读取时优先走 `读取方法成熟度阶段标准特征(...)`
   - 默认不强制修改历史来源字段
   - 学习成长回写链应同时把以下任务事实写回到学习任务头，便于展示与调试：
     - `学习结果_学习成长后来源名称指针`
     - `学习结果_学习成长后成熟度阶段`
     - `学习结果_学习成长后成熟度名称指针`
     - `学习结果_学习成长后允许自动召回`
     - `学习结果_学习成长后允许默认执行`
     - `学习结果_学习成长后可靠度_Q10000`
5. 整个过程中
   - 不新增独立“候选方法账本”
   - 不把因果模块本身变成成熟度真值仓


## 8. 线程与身份边界

当前因果键与反向索引使用了动作信息指针和内存态索引，因此必须固定：

1. 当前因果模块默认是进程内运行时结构，不是可直接持久化协议。
2. 任何跨进程、跨重启复用，都不能直接依赖当前指针键。
3. 若要引入持久化或跨进程因果 ID，必须新增稳定标识层，不能篡改现有运行时键语义。

另外，当前模块没有独立暴露细粒度内部线程同步，因此必须固定：

- 因果模块的写入与查询，应由上层调用方保证时序串行或更外层同步


## 9. 硬约束

1. 因果模块的职责是证据累计和候选检索，不是最终决策。
2. 因果必须同时具备因、动作、过程、果的最小闭环才能成立。
3. `稳定度Q` 只是一阶排序分，不得误读成概率。
4. 当前因果键是运行时键，不得误当成跨进程稳定ID或持久化主键。
5. 因果到方法的转化必须继续经过学习和验证，不得跳过。
6. 因果模块可以支持低成熟度方法头出生，但不得直接单方面判定“已验证稳定”或“组合成熟”。
7. 当已有方法能够承接同类证据时，应优先回补已有方法，而不是让抽象因果无限并行长头。
8. 因果链在当前实现中应优先理解为派生的筛选/倒推视图；默认先做单步倒推，只有单步不够时才组织多步桥接链。
9. 若因果被 `任务管理方法` 分支消费，外层仍必须先具备 `最近根层裁决结果 / 当前缺口类型 / 当前下一步去向`，不得让当前全域因果模块越权冒充治理上下文。
