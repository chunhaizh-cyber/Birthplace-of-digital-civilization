# 0032 存在-特征-动作-消息-二次特征映射规范 v0.1

## 1. 目标

本文用于先冻结一份统一语义底座，回答以下问题：

- 系统里哪些对象算合法 `存在`
- 每类存在有哪些可直接记录的 `一次特征`
- 哪些存在能执行哪些 `动作`
- 一条 `消息` 最少必须带哪些语义字段
- `二次特征` 应由哪些一次特征和动作结果重算得出
- 消息消费完成后，事实、动态、判断和因果应分别写回哪里

本文只做一件事：

`把“存在 -> 一次特征 -> 动作 -> 消息 -> 二次特征”这条语义主链固定成统一映射规范。`

本文不负责：

- 直接扩写 `自我线程` 运行逻辑
- 直接改 C++ 结构体实现
- 展开任务管理工作线程内部步骤算法
- 展开方法执行细节与学习算法

## 3. 核心结论

必须固定以下结论：

1. `存在` 是一切状态、动作、消息和二次特征的承载主体。
2. `一次特征` 是可直接观测、直接写回、可作为状态真值记录的特征。
3. `动作` 是某存在在某时刻、某场景下对对象执行的一次过程事实。
4. `消息` 不是“一个名字”，而是“一个动作导致的一组一次特征变化”。
5. `二次特征` 不是原始事实，而是消息消费后基于一次特征比较、关系整理和条件归并得到的结构化判断结果。
6. `控制域特征` 必须拆成 `控制意图` 与 `实际运行状态` 两组；请求值与事实值不得混写。
7. 自我线程后续的判断依据，应优先围绕 `一次特征 + 二次特征` 运转，而不是围绕消息名字运转。
8. `状态节点` 负责记真值，`动态节点` 负责记动作过程，`二次特征` 负责记判断结果，`因果实例` 负责记条件-动作-结果链。
9. 长期目标不是把“所有一次特征”直接硬连到 `服务值 / 安全值`，而是逐步建立 `关键特征变化 -> 二次特征 -> 影响项 -> 服务值/安全值聚合` 的可解释映射。
10. 与任务相关的一次特征，不得把 `任务根语义`、`任务控制态` 和 `任务管理任务局部推进态` 混写；三者必须按 `0039` 明确归属。

一句话收口：

`消息是动作造成的一次特征变化集合；二次特征是线程消费这些变化后得到的关系判断；影响项再把这些判断归结到服务和安全的正式影响层。`

## 4. 七层主链与硬边界

推荐固定主链为：

```text
存在
-> 一次特征
-> 动作
-> 消息（变化项集）
-> 二次特征
-> 影响项
-> 服务值/安全值聚合
-> 因果实例 / 派发 / 结算 / 学习写回
```

必须固定：

1. `一次特征` 是原始真值层，不直接冒充最终裁判。
2. `二次特征` 是比较后或关系化后的结构结果，不直接冒充原始事实。
3. 一条消息允许一次携带多个变化项，不允许偷简成“只改一个特征”的单值通知。
4. `场景` 继续只保留索引与引用，不嵌整包状态副本。
5. `因果实例` 记录的是 `条件 + 动作 + 结果` 的结构链，不替代状态节点和动态节点本身。
6. `影响项` 是在二次特征之后、双值结算之前的正式归结层，不得与原始事实或最终结算值混写。

必须额外固定：

1. 所有线程子存在至少共享 `生命周期状态 / 控制意图 / 实际运行状态` 三类核心标准特征。
2. `自我线程` 与 `工作线程` 是在该通用线程标准面上的不同特化，而不是两套互不相关的状态世界。

### 5.2 控制域特征取值表

控制域特征必须至少拆成两组：

1. `控制意图特征`
2. `实际运行状态特征`

| 控制域特征 | 典型取值 | 含义 |
| --- | --- | --- |
| 控制意图特征 | `请求启动 / 请求暂停 / 请求继续 / 请求打断 / 请求收束 / 请求停止 / 请求恢复重建 / 请求刷新预算 / 请求刷新优先级` | 表示“想让它变成什么” |
| 实际运行状态特征 | `待机 / 启动中 / 运行中 / 已暂停 / 已打断 / 收束中 / 已收束 / 已停止 / 故障中` | 表示“它现在实际上是什么” |

必须固定：

1. `请求收束` 和 `已收束` 不得写成同一个值。
2. 控制消息通常只改 `控制意图特征`。
3. 结果回流、工作线程状态回报、状态确认通常改 `实际运行状态特征`。
4. 对线程子存在而言，最小标准特征面 = `生命周期状态 + 控制意图特征 + 实际运行状态特征`。

## 7. 表 3：消息字段合同表

### 7.1 消息定义

必须先固定一句话：

`消息不是一个名字，而是一个动作导致的一组特征变化。`

### 7.2 语义消息最小合同表

| 字段段落 | 字段 | 含义 | 备注 |
| --- | --- | --- | --- |
| 识别段 | `消息主键` | 这条消息的唯一标识 | 用于幂等、追踪、审计 |
| 行为段 | `发出者` | 谁把消息送进线程 | 不等于动作主体 |
| 行为段 | `动作主体` | 谁做了动作 | 是动态节点首要落点 |
| 行为段 | `动作语义` | 做了什么 | 推荐用 `动作语义键` 固定 |
| 行为段 | `作用对象` | 动作主要作用到谁 | 可为空 |
| 场景段 | `关联场景` | 这件事发生在哪个场景或链路里 | 方便映射输入输出场景 |
| 时间段 | `产生时间` | 何时进入消息世界 | 不等于所有结果值的唯一生效时间 |
| 来源段 | `来源是否成功` | 这条动作或观测链是否成功 | 对齐动态节点 |
| 来源段 | `错误码` | 来源错误信息 | 对齐动态节点 |
| 来源段 | `置信度` | 这条消息的来源可信度 | 便于审计与学习 |
| 追踪段 | `路径签名` | 过程路径签名 | 便于调试与回放 |
| 变化段 | `变化项集` | 这次动作导致的全部一次特征变化项 | 不允许缩成单项字段 |

### 7.3 变化项最小合同表

| 字段 | 含义 | 备注 |
| --- | --- | --- |
| `变化主体` | 谁的特征变了 | 不默认等于动作主体 |
| `特征类型` | 改的是哪个一次特征 | 应能映射状态节点特征 |
| `变化前值` | 变更前的值 | 可为空 |
| `变化后值` | 变更后的值 | 不可省略 |
| `是否确认生效` | 该变化是否已成为正式真值 | `false` 时只能作候选事实 |

### 7.4 线程投递补充段

当消息进入 `自我线程` 时，允许在语义消息外再叠一层线程投递头，用于：

- 路由大类
- 路由子类
- 是否局部有效
- 是否可抢占
- 优先级
- 目标任务 / 父任务 / 来源线程等定向投递信息

必须固定：

1. `0032` 冻结的是语义消息合同。
2. `0029` 冻结的是线程投递头和线程路由合同。
3. 线程路由头不得反过来替代语义消息本体。

### 7.5 推荐最小结构草案

```cpp
struct 结构_特征变化项 {
    主键 变化主体主键 = 0;
    主键 特征主键 = 0;
    特征值 变化前值{};
    特征值 变化后值{};
    bool 已确认生效 = false;
};

struct 结构_语义消息 {
    主键 消息主键 = 0;
    时间戳 产生时间 = 0;
    主键 关联场景主键 = 0;

    主键 发出者主键 = 0;
    主键 动作主体主键 = 0;
    std::string 动作语义键{};
    主键 作用对象主键 = 0;
    主键 记录者主键 = 0;

    bool 来源执行成功 = false;
    I64 来源错误码 = 0;
    I64 置信度 = 0;
    std::uint64_t 路径签名 = 0;

    std::vector<结构_特征变化项> 变化项集{};
};
```

必须固定：

1. 这是 `语义消息` 的最小骨架，不含线程路由头。
2. 线程实际投递时，可由 `0029` 在外层再叠加 `路由 / 优先级 / 是否抢占 / 是否局部有效` 等控制字段。


## 8. 表 4：二次特征判定表

这张表回答：

`哪些判断不应散落在线程 if-else 里，而应收敛成可复用的二次特征。`

| 二次特征名称 | 主体 / 客体 / 左右对象 | 依赖的一次特征列表 | 判定公式摘要 | 输出形式 | 谁负责重算 | 何时重算 |
| --- | --- | --- | --- | --- | --- | --- |
| 当前步骤可执行 | 主体=`任务管理任务` | 当前步骤前沿、前序完成状态、依赖等待状态、阻塞状态、控制意图、实际运行状态 | `前沿有效 && 前序已完成 && 依赖已满足 && 非阻塞 && 非停止/收束禁止态` | `是否满足 + 置信度` | 自我线程或任务管理实例 | 消费推进事件、结果回流、控制意图变化后 |
| 依赖满足 | 主体=`父任务`，客体=`子任务或资源` | 子任务结果状态、资源可用性、当前等待原因 | `等待对象可用或结果成功，且等待原因被覆盖` | `是否满足 + 标量值(1/0)` | 自我线程 | 消费结果回流、环境资源消息后 |
| 父任务阻塞 | 主体=`父任务`，客体=`依赖对象` | 当前等待原因、阻塞状态、依赖满足 | `存在等待原因 && 依赖未满足` | `是否满足 + 置信度` | 自我线程 | 派生需求上报、推进阻塞上报后 |
| 父任务解阻 | 主体=`父任务`，客体=`依赖对象` | 父任务阻塞、依赖满足、当前等待原因 | `上一时刻阻塞 && 当前依赖满足` | `是否满足 + 标量值(1/0)` | 自我线程 | 结果回流、资源到位、确认链回报后 |
| 风险升高 | 主体=`自我 / 任务 / 外部对象` | 风险等级、风险阈值、异常结果、审计结论 | `风险后值高于前值或越过阈值` | `标量值 + 是否满足 + 置信度` | 自我线程 | 环境突变、异常结果、审计结论后 |
| 预算紧张 | 主体=`自我线程` 或 `任务` | 当前预算、剩余时间、时效压力、资源窗口 | `剩余预算低于阈值或时间压力高于阈值` | `标量值 + 是否满足` | 自我线程 | 预算刷新、轮次推进、超时上报后 |
| 是否需要收束 | 主体=`自我线程` | 控制意图、实际运行状态、风险升高、预算紧张、异常结果 | `请求收束/停止 || 风险失控 || 预算紧张严重 || 异常越界` | `是否满足 + 置信度` | 自我线程 | 控制消息、故障结果、风险和预算变化后 |
| 有效优先级 | 主体=`任务控制态` | 根层权重、时效压力、风险压力、解阻价值、预算压力 | `根权重 + 时效加权 + 风险加权 + 解阻价值 - 预算惩罚` | `标量值 + 置信度` | 自我线程 | 根层重判前、环境变化、结果回流、解阻后 |

必须固定：

1. 这些二次特征是线程判断的正式中间层，不得继续散写成大量匿名 if-else。
2. 二次特征只能建立在 `一次特征真值 + 合法动作结果` 之上。
3. 最终裁决应优先消费二次特征，而不是直接消费消息名字。


## 9. 表 5：一次特征变化到节点落位表

这张表回答：

`消息消费后，变化事实、动作事实、判断结果和因果链分别写到哪里。`

| 来源 | 写入目标 | 主写入者 | 场景保留方式 | 说明 |
| --- | --- | --- | --- | --- |
| 单个 `变化项` | `状态节点` | 消息消费器或回写器 | 场景保存状态索引 | 表达“谁的哪个特征变成了什么值” |
| 整条 `动作消息` | `动态节点` | 消息消费器或回写器 | 场景保存动态索引 | 表达“谁做了什么动作，输入输出和成功与否如何” |
| `二次特征判定结果` | `二次特征节点` | 二次特征判定器 | 场景保存二次特征索引 | 表达“这些状态变化推出了什么关系判断” |
| `条件 + 动作 + 结果` | `因果实例` | 因果生成器 | 场景保存实例因果索引 | 表达“什么条件下做了什么动作，导致了什么结果” |
| 所有节点引用集合 | `场景索引` | 场景维护器 | 只保存索引，不保存大副本 | 保证场景是引用层，不是副本层 |

必须固定：

1. `一次特征变化项 -> 状态节点`
2. `整条动作消息 -> 动态节点`
3. `判断结果 -> 二次特征节点`
4. `条件 + 动作 + 结果 -> 因果实例`


## 10. 三条样板链

为避免规范继续漂浮，先固定三条样板链。

### 10.1 样板 1：控制链

```text
自我线程
-> 执行动作“下发收束指令”
-> 工作线程.控制意图特征 = 请求收束
-> 工作线程回报“完成收束”
-> 工作线程.实际运行状态特征 = 已收束
-> 任务.局部运行状态 = 已收束
-> 自我线程重算“是否需要收束 / 有效优先级 / 父任务阻塞状态”
```

### 10.2 样板 2：结果回流链

```text
子任务
-> 执行动作“提交结果”
-> 子任务.结果状态 = 成功
-> 子任务.局部完成度 = 100
-> 自我线程重算“依赖满足 = true”
-> 自我线程重算“父任务解阻 = true”
-> 自我线程重算“当前步骤可执行 = true”
-> 必要时路由给父任务并更新有效优先级
```

### 10.3 样板 3：环境链

```text
环境监测器
-> 执行动作“上报资源到位”
-> 资源.可用性 = 可用
-> 自我线程重算“依赖满足 = true”
-> 自我线程重算“当前步骤可执行 = true”
-> 自我线程重算“有效优先级”
-> 再决定是否把相关任务放回可运行队列
```
