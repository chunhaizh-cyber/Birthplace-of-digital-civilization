# 1195 任务管理方法需求位与因果到方法映射规范

## 1. 目标

本文用于在 `任务管理任务成长需求` 与 `任务管理方法因果生成` 之间，补上一层正式桥接规范，回答以下问题：

- `任务管理任务` 需要哪些 `方法需求位`
- `方法需求位` 与 `实际方法`、`管理类原函数`、`因果证据` 各是什么关系
- 因果信息如何支持方法 `出生 / 验证 / 升格`
- 它们在 `根层裁决 -> 任务管理上下文 -> 单步治理` 主链中的位置是什么
- 为什么不能直接把 `果` 当成方法

本文要解决的是“任务管理任务需要什么方法位，以及因果如何支持它们长出来”，不是直接定义某条具体方法实现。

具体到 7 个方法需求位各自最少需要哪些 `状态 / 动态 / 实例因果 / 轻量因果链`，以 [1196_任务管理方法需求位最小状态动态因果清单.md](D:\鱼巢\规范\1196_任务管理方法需求位最小状态动态因果清单.md) 为准。

## 3. 核心结论

必须先固定以下 13 条：

1. `任务管理任务` 的成长需求，不能直接等价为某个已存在方法。
2. `果` 不是方法，`果` 只是一次运行后留下的结果状态或结果动态。
3. `因果` 不是方法，`因果` 只是支持方法出生、验证和升格的证据。
4. 在 `成长需求` 与 `实际方法` 之间，必须存在一层 `方法需求位`。
5. `方法需求位` 表示“系统当前缺哪类治理能力”，不表示“已经有哪条具体方法枝”。
6. 一条具体方法枝，必须最终能分解回既有管理类原函数、既有元函数，或它们的合法组合片段。
7. `因果到方法` 的正确路径不是“因果直接变方法”，而是“因果支撑某个方法需求位下的候选方法出生、验证、升格”。
8. 因此倒推顺序必须固定为：

```text
成长需求
-> 方法需求位
-> 方法位所需因果证据
-> 候选方法出生
-> 稳定复现
-> 方法升格
```
9. 若某条因果链包含 `未确认外部结果跳`，则该因果链不能直接作为高置信方法出生证据。
10. 因果链可继续作为弱证据保留，但必须待 `外部结果确认` 或后续稳定复现后，才允许进入高置信出生/升格判定。
11. `方法需求位` 只能在 `根层已允许继续推进`、且 `任务管理上下文` 已形成的前提下被命中和求解。
12. `方法需求位` 只回答“当前治理链缺哪类能力位”，不替代根层方向裁决。
13. 命中的 `方法需求位` 允许继续下钻出 `本能能力缺口`，以判断当前缺的是方法枝本身，还是支撑该位落地所需的最小动作原语；更细承载按 [1228_任务管理主体虚拟存在与分身继承规范.md](D:\鱼巢\规范\1228_任务管理主体虚拟存在与分身继承规范.md) 执行。


## 4. 三层桥接定义

### 4.0 在统一主链中的位置

按当前最新体系，`方法需求位` 必须处在下面这条链中：

```text
根层方向裁决
-> 形成任务管理上下文
-> 命中或上浮某个方法需求位
-> 收集该方法位所需因果证据
-> 候选方法出生 / 验证 / 升格
-> 返回治理结论
-> 外层继续重判与调度
```

必须固定：

1. `方法需求位` 的来源是任务管理上下文中的治理缺口和成长压力，不是根层直接发明的方法名。
2. `候选方法枝` 的结果，仍要回到任务管理层和外层重判，不得把“某方法位已命中”直接当成根层成立。

### 4.1 方法需求位

`方法需求位` 指的是：

- 任务管理任务为了稳定治理，必须拥有的一类治理能力位

它回答的是：

- 当前到底缺哪类方法
- 这类方法最少要能解决什么治理问题

它不是：

- 具体方法首节点
- 具体动作句柄
- 一条单次因果

### 4.2 候选方法位

`候选方法位` 指的是：

- 某个方法需求位下，已经出现但尚未稳定升格的一条候选方法枝

它回答的是：

- 这类能力当前有没有初步可用路径
- 这条路径还处于出生、验证还是稳定复现阶段

### 4.3 因果到方法映射

`因果到方法映射` 指的是：

- 用实例因果和轻量因果链，去支持某个方法需求位下的候选方法枝出生、验证和升格

它不是：

- 把单条因果硬转换成一个正式方法 ID
- 把某个结果状态直接当成方法

### 4.4 只有目标时的需求位驱动候选出生

`只有目标时的需求位驱动候选出生` 指的是：

- 当前还没有现成方法枝，只有某个明确的治理目标、缺口或 `方法需求位`
- 先按该需求位要求的最小状态面，在状态列表中查找同类型特征
- 再用这些同类型特征去筛选相关 `实例因果 / 轻量因果链 / 观察证据`
- 最后组织出该需求位下的候选方法枝

这里的 `同类型特征` 至少应满足：

- 作用对象类型可比
- 所处治理阶段或角色可比
- 条件/结果对位关系可比
- 不能只凭字面相似或单个结果名相似

必须固定：

1. 这是一条 `需求位驱动 -> 证据筛选 -> 候选出生` 的桥接路径，不是“目标直接生成方法”。
2. 若已存在可承接这些证据的旧方法枝，应优先回补旧枝，而不是平行新生。
3. 若状态列表中找不到同类型特征，或筛不出合法因果证据，则只保留该需求位，不得强行出生方法。
4. 这条路径生成出的正式来源，仍应按主导证据类型落到 `因果抽象 / 观察抽象 / 路径提炼`，而不是另造 `需求驱动来源`。


## 5. 为什么不能直接把果当方法

必须固定：

1. 同一个 `果`，可能由多个不同方法达成。
2. 同一条方法，在不同条件下，也可能产生不同 `果`。
3. `果` 只回答“发生了什么结果”，不回答“这是哪类可复用治理能力”。

例如：

```text
果 = 新建了一个补条件子任务
```

它可能来自：

- `管理_筹办器` 的补结构路径
- `管理_分解器` 的细分路径
- `管理_失败策略器` 的恢复补条件路径

因此：

- `补条件子任务被创建` 不能直接等价成某条方法
- 它最多只能作为某个方法需求位下的一类结果证据


## 6. 方法需求位的最小集合

当前 `任务管理任务` 最小应先固定以下 7 个方法需求位：

1. `补结构方法位`
2. `选择推进方法位`
3. `分解补条件方法位`
4. `求证判定方法位`
5. `门控转域方法位`
6. `失败策略方法位`
7. `收束复盘方法位`

它们与 [1110_管理类原函数与四域方法树接口规则.md](D:\鱼巢\规范\1110_管理类原函数与四域方法树接口规则.md) 中的七个管理类原函数默认对齐如下：

| 方法需求位 | 默认主要承接原函数 | 主要回答的问题 |
| --- | --- | --- |
| 补结构方法位 | `管理_筹办器` | 当前缺什么结构，怎样先把任务搭起来 |
| 选择推进方法位 | `管理_选择器` | 多个候选里当前应推进哪个 |
| 分解补条件方法位 | `管理_分解器` | 需要拆什么步骤、子任务、补条件任务 |
| 求证判定方法位 | `管理_求证器` | 是否满足目标、是否满足完成判据 |
| 门控转域方法位 | `管理_门控器` | 当前应继续维护、转纠偏、回筹办还是进收束 |
| 失败策略方法位 | `管理_失败策略器` | 失败后该替代、回退、恢复还是等待学习 |
| 收束复盘方法位 | `管理_复盘器` | 当前应如何定终态、写反馈、沉淀经验 |

必须固定：

- 这里的“对齐”是治理职责对齐，不是强制一一等于一个函数实现。
- 一条具体方法枝可以由一个原函数承接，也可以由多个合法组合片段承接。
- 这些方法需求位首先属于任务管理层能力位，不属于根层目的位。


## 7. 方法需求位与实际方法的关系

这层关系必须固定成：

```text
一个方法需求位
-> 可以长出多条具体方法枝

一条具体方法枝
-> 必须归属于至少一个方法需求位
```

因此：

- `方法需求位` 比单条方法更稳定
- `具体方法枝` 比方法需求位更具体

例如：

`补结构方法位` 下面，可以同时长出：

- “先补条件再绑定方法”的临时枝
- “先创建观察任务再回筹办”的反推枝
- “先分解再插检查点”的组合枝

它们都属于同一类能力位，但不是同一条方法。


## 8. 每个方法需求位所需的最小因果

每个方法需求位，至少都应有 4 类因果支持：

### 8.1 出生因果

回答：

- 在什么条件下，第一次出现这类方法候选是合理的

标准表达：

```text
条件局势
-> 调用某类管理方法
-> 出现该方法位的第一次有效结果
```

### 8.2 验证因果

回答：

- 相似条件下，这条方法是否反复得到同类结果

标准表达：

```text
输入条件主键 K
-> 命中同类方法位候选
-> 重复出现相似结果主键
```

### 8.3 升格因果

回答：

- 这条候选方法是否已经从偶然成功，变成稳定复现

标准表达：

```text
同桶条件
-> 多条合法单跳实例因果
-> 组织成轻量因果链
-> 支撑来源升格
```

必须继续服从：

- 若轻量因果链的 `最弱跳` 属于 `未确认外部结果跳`
- 则该链不得直接进入高置信 `升格因果`
- 只能先保留为待验证链证据

### 8.4 价值因果

回答：

- 这类方法不只是能推进治理，还是否真的带来价值闭环

标准表达：

```text
治理方法位命中
-> 结构推进
-> 叶子结果成立
-> 可结算价值出现
```


## 9. 各方法需求位的最小因果需求表

| 方法需求位 | 最小条件因果 | 最小动作因果 | 最小结果因果 | 最小链级因果 |
| --- | --- | --- | --- | --- |
| 补结构方法位 | `依赖缺失 / 任务链未成形 / 无可试方法` | `管理_筹办器` 或等价补结构路径 | `新步骤 / 新子任务 / 新方法绑定` | `输入条件 -> 补结构 -> 结构成形` |
| 选择推进方法位 | `已有候选 / 候选评分可比` | `管理_选择器` 或等价选择路径 | `当前方法切换 / 当前步骤切换` | `输入条件 -> 选择 -> 推进当前分支` |
| 分解补条件方法位 | `需要新子任务或新步骤 / 条件缺口存在` | `管理_分解器` 或等价分解路径 | `补条件任务建立 / 子任务建立` | `输入条件 -> 分解 -> 结构细化` |
| 求证判定方法位 | `检查点通过候选 / 结果稳定候选 / 完成判据待判` | `管理_求证器` 或等价求证路径 | `满足/不满足判定 / 完成判据成立` | `输入条件 -> 求证 -> 终态或续行建议` |

| 失败策略方法位 | `最近失败 / 连续失败 / 关键方法失效` | `管理_失败策略器` 或等价失败处理路径 | `替代方法 / 回退 / 等待学习 / 恢复任务` | `输入条件 -> 纠偏 -> 回到可继续态` |
| 收束复盘方法位 | `已满足完成判据 / 已确定无法继续 / 终态待判` | `管理_复盘器` 或等价收束路径 | `任务终态 / 经验摘要 / 反馈写回` | `输入条件 -> 收束 -> 终态与经验沉淀` |


## 10. 因果如何支持方法出生、验证、升格

这条映射必须固定为：

### 10.1 出生

若某个方法需求位长期被命中，但当前没有稳定方法枝，而某次运行形成了合法 `出生因果`，则：

- 允许在该方法需求位下出生一条 `运行期临时` 候选枝
- 若当前只有治理目标或 `方法需求位`，还没有现成方法枝，允许先按 `需求位 -> 状态列表同类特征 -> 合法出生因果 -> 候选方法出生` 这条路径补出候选枝

但必须补一句：

- 若该 `出生因果` 依赖的轻量因果链包含 `未确认外部结果跳`，则只允许记为 `弱出生证据`
- 不得直接记为 `高置信出生`

### 10.2 验证

若该候选枝在相似 `输入条件主键` 下，重复形成合法 `验证因果`，则：

- 允许累计 `稳定出现次数`
- 允许进入候选观察态

### 10.3 升格

若该候选枝已经形成多条同桶实例因果，并能组织成轻量因果链，则：

- 允许进入 `反推`
- 继续稳定复现后，允许升格到 `组合`

但必须继续服从：

- 轻量因果链整体可信度，不得高于 `最弱跳可信度`
- 若最弱跳属于 `未确认外部结果跳`，则该链不得直接支撑高置信升格
- 只能继续保留为观察态链证据

### 10.4 价值叠加

若该方法位下某条方法枝还能进一步形成 `价值因果`，则：

- 允许挂上价值叠加态
- 但不得把价值叠加态误写成新的来源枚举


## 11. 不得直接从因果跳到方法的情况

以下情况必须禁止直接生成或升格方法：

- 只有一个 `果`，没有合法 `因` 和 `动作`
- 只有单条偶然成功，没有稳定复现
- 只有 `结果摘要哈希`，没有可校验结果状态
- 只有 `输入条件主键哈希`，没有可校验条件状态
- 没有明确归属到某个 `方法需求位`
- 该路径无法分解回既有管理类原函数或合法组合片段
- 因果链包含 `未确认外部结果跳`，却被直接当成高置信出生或高置信升格证据


## 12. 倒推顺序规范

以后如果要从 `任务管理任务成长需求` 倒推“需要哪些因果信息”，顺序必须固定为：

1. 先确认根层仍允许继续推进，且当前任务管理上下文已经稳定。
2. 再确认当前成长目标缺的是哪个 `方法需求位`。
3. 再确认该方法需求位最小需要哪些 `出生因果 / 验证因果 / 升格因果 / 价值因果`。
4. 再确认这些因果各自最少需要哪些 `条件状态 / 动作动态 / 结果状态 / 链桥接键`。
5. 最后才去检查代码里这些因果原料和生成路径是否已经具备。

不得倒着做成：

```text
看到一个果
-> 直接把它当方法
-> 再倒推出成长需求
```


## 13. 一句话收口

可收敛为一句话：

`任务管理任务的成长需求，必须先压缩成“方法需求位”，再由每个方法需求位倒推出所需因果；因果不能直接等于方法，果也不能直接等于方法，因果只能作为候选方法出生、验证、升格与价值叠加的证据层。`
