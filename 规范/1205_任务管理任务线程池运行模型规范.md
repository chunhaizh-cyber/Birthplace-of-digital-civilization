# 1205 任务管理任务线程池运行模型规范

## 1. 目标

本文用于固定 `任务管理任务` 的运行模型，回答以下问题：

- `任务管理任务` 是否应由自我线程循环驱动
- `每个任务对应一个任务管理实例` 应如何理解
- 为什么应采用 `线程池 + 逻辑实例`，而不是 `一任务一永久物理线程`
- `任务管理任务界面线程` 应处于什么位置、是否持有推进权
- `任务管理任务` 每次运行到底允许推进多少
- 根层方向、任务管理上下文、线程池承载与消息回投之间如何接平
- 如何进入 `执行中 / 等待学习 / 等待外部反馈 / 等待执行结束`
- 等待与唤醒应如何建模

本文要固定的是 `任务管理任务` 的运行时语义，不是某个具体模块的临时代码写法。

## 3. 核心结论

必须固定以下 18 条：

1. `任务管理任务` 必须由 `自我线程` 的循环调度驱动。
2. `任务管理任务` 的输入，不应来自全局盲扫，而应由自我线程将 `待处理宿主任务` 连同本轮参数快照显式提交。
3. `每个宿主任务` 必须对应 `最多一个任务管理实例`。
4. `任务管理实例` 是逻辑工作线程体，不要求等于一个永久占用的物理线程。
5. 运行时应采用 `有限上限的任务管理线程池` 承载实例执行。
6. `任务管理实例` 每次运行只允许推进 `宿主任务前进一步`。
7. `前进一步` 可以包括：选择管理步骤、选择管理方法、写回一步状态转移、进入等待态、回收一步结果。
8. `前进一步` 不允许等价于“一次运行内连续完成多轮治理闭环”。
9. `任务管理实例` 可以让宿主任务进入 `就绪 / 排队中 / 执行中 / 等待学习 / 等待外部反馈 / 收束相关状态`。
10. `等待执行结束`、`等待学习反馈`、`等待外部反馈` 都必须实现为 `等待态`，不得阻塞占住物理线程死等。
11. `唤醒` 必须由事件驱动，不得依赖实例内部长时间轮询。
12. `同一宿主任务同一时刻` 只能有一个任务管理实例持有推进权。
13. `任务管理任务` 可以修改宿主任务的治理状态，但不能伪造叶子执行结果、外部成功结果或需求满足结论。
14. 当前 `自我线程 -> 任务控制总控模块_v0::处理任务入口(...)` 的同步调用形态，可视为本文模型的 `V0 单线程退化实现`，后续可平滑升级为真正的线程池模型。
15. `任务管理请求` 必须建立在 `根层已完成裁决`、且 `任务管理上下文` 已形成的前提下。
16. 每轮单步管理结束后，结果必须回到外层执行 `根层重判 -> 判域/调度 -> 再提交`，不得在线程池内部私自连跑多轮治理闭环。
17. 从最小可调度单元看，线程池承载的不是“一个会自己长时间跑的大循环”，而是 `任务树 + 局部运行态` 上的一次 `推进一步`。
18. `自我线程` 不直接展开任务树，只负责决定是否再次批准该实例继续 `推进一步`。
19. 允许存在 `任务管理界面线程`，用于展示 `任务管理任务界面信息`、`工作线程列表` 与 `消息汇总`，并承接控制意图输入。
20. `任务管理界面线程` 不得持有推进权，不得直接修改宿主任务事实、任务树或根层裁决；它只能读取快照并把控制意图交回运行时承接层。

可收敛为一句话：

`任务管理任务应采用“一任务一实例、线程池承载、每轮单步推进、等待靠事件唤醒”的运行模型。`

### 3.1 最新统一主链中的位置

按当前最新体系，本文必须服从以下主链：

```text
根层方向裁决
-> 形成任务管理上下文
-> 自我线程或外层协调器提交任务管理请求
-> 任务管理实例在线程池中单步推进
-> 返回一步结果、消息与上下文更新
-> 外层重判根方向与当前功能域
-> 再决定是否继续提交、重筹办、等待或收束

任务管理界面线程
-> 只读任务管理界面快照与工作线程快照
-> 可发送控制意图
-> 不直接持有推进权
```

必须固定：

1. `线程池` 承载的是管理实例的单步执行，不承载根层裁决。
2. `任务管理实例` 可以做域内分析与一步治理动作，但不能跳过外层重判，自行闭环多轮推进。
3. `自我线程` 或其它外层协调器，负责把根层结论和最小任务管理上下文带入本轮请求。
4. 更细的 `一步治理结果` 最小返回面，按 [1213_任务管理一步治理结果最小返回面规范.md](D:\鱼巢\规范\1213_任务管理一步治理结果最小返回面规范.md) 执行。
5. `任务管理界面线程` 只消费快照与汇总，不得被误当成另一条推进线程。


## 4. 核心定义

### 4.1 任务管理实例

`任务管理实例` 指的是：

- 专门负责某一个宿主任务治理推进的逻辑执行体

它至少应持有：

- `宿主任务标识`
- `任务树主体`
- `局部运行态`
- `实例状态`
- `等待原因`
- `恢复点`
- `最近提交请求`
- `最近管理步骤`
- `最近管理方法`

它不是：

- 宿主任务本身
- 叶子执行线程
- 永久占用的系统线程

必须补充固定：

- `任务管理实例` 运行时至少应理解成：`任务树主体 + 局部运行态 + mailbox/最近请求`。
- 物理线程只负责执行其中的一次 `推进一步`，不长期占有这份实例状态。

### 4.2 任务管理请求

`任务管理请求` 指的是：

- 自我线程提交给任务管理实例的一轮执行输入

它至少包含：

- `根层允许继续推进`
- `宿主任务头结点`
- `当前步骤结点`
- `当前结果结点`
- `当前方法首节点`
- `当前缺口类型`
- `当前下一步去向`
- `当前场景`
- `宿主任务虚拟存在特征快照`
- `当前功能域判定输入`
- `最近反馈`
- `控制上下文`
  - `打断`
  - `收束`
  - `停止`
  - `预算刷新`
  - `优先级刷新`
- `已路由回来的子结果 / 依赖结果`
- `调度时间`

更细的 `任务管理上下文` 最小稳定字段表，按 [1214_任务管理上下文最小字段表规范.md](D:\鱼巢\规范\1214_任务管理上下文最小字段表规范.md) 执行；本节列出的 `当前场景 / 虚拟存在快照 / 调度时间` 等仍可视为请求补充材料，不等于稳定上下文字段本体。

必须补充固定：

- 这里的 `当前功能域判定输入` 属于任务管理上下文的一部分，不等于根层方向本身。
- 若 `根层允许继续推进 = false`，则本轮请求只允许承载收束或停机边界处理，不允许普通筹办、维护或纠偏扩张。

### 4.3 任务管理线程池

`任务管理线程池` 指的是：

- 执行任务管理实例“单步运行”的有限工作线程集合

它负责：

- 取出可运行实例
- 执行一轮单步管理
- 执行结束后立刻释放物理线程

它不负责：

- 替代任务管理实例保存状态
- 长期阻塞等待外部事件

必须补充固定：

- `任务管理线程池` 本身应在 `自我的内部世界` 中拥有一个正式 `线程池存在`。
- 线程池中的每个具体工作线程在物理线程创建时，都应同步创建对应 `工作线程子存在`。
- 若某工作线程属于某线程池，则该工作线程存在必须挂在其 `所属线程池存在` 下面，不再直接平铺到 `自我存在` 或匿名日志层。
- 工作线程子存在至少应初始化 `生命周期状态 / 控制意图 / 实际运行状态` 三类核心标准特征；其余如 `健康状态 / 绑定任务 / 阻塞状态 / 心跳健康` 按适用性补齐。

### 4.4 任务管理界面线程

`任务管理界面线程` 指的是：

- 用于展示任务管理任务界面信息、工作线程列表、消息汇总，并承接控制意图输入的观察/交互线程

它负责：

- 读取 `任务管理任务界面快照`
- 读取 `任务管理工作线程快照`
- 展示汇总、实体数量、状态数量、收发消息情况
- 把暂停、恢复、刷新、切换等控制意图回传给运行时承接层

它不负责：

- 直接消费 `任务管理请求`
- 直接推进 `任务管理实例`
- 直接修改宿主任务树、步骤树、结果树
- 直接改写根层裁决或双值真值

必须补充固定：

- `任务管理界面线程` 不是 `任务管理线程池` 的一部分。
- `任务管理界面线程` 不持有推进权，只持有观察权与控制意图发起权。
- 若界面线程需要显示运行态，应优先读取 `宿主任务虚拟存在镜像 + 工作线程快照 + 恢复快照摘要`，不得直接改写运行时实例堆。

### 4.5 等待态

`等待态` 指的是：

- 本轮已完成一步管理，但后续推进依赖尚未返回的事件，因此实例主动挂起、释放工作线程的状态

至少包括：

- `等待执行结果`
- `等待学习反馈`
- `等待外部反馈`
- `等待下一次调度窗口`


## 5. 提交模型

### 5.1 自我线程提交规则

必须固定：

- 自我线程每轮先完成根层重判与最小任务管理上下文整理
- 再选出 `当前待处理宿主任务`
- 再把该任务的本轮状态快照打包成 `任务管理请求`
- 最后将请求投递给对应的 `任务管理实例`

不允许：

- 任务管理实例自己无边界扫全局任务树决定“今天想管谁”
- 任务管理实例直接读取未经提交的漂移现场替代本轮输入

### 5.2 一任务一实例

必须固定：

- 同一个宿主任务，只允许存在一个 `任务管理实例`
- 若实例已存在，新请求应进入该实例的 `mailbox` 或覆盖其 `最近提交请求`
- 不允许为同一宿主任务并行创建多个管理实例

### 5.3 输入快照边界

每轮输入必须理解为：

- `提交时快照`

因此一轮运行中应优先使用：

- 提交时提供的任务状态
- 提交时提供的最近反馈
- 提交时提供的功能域输入

只有在规则明确允许时，才可补读最新状态。

默认不允许：

- 在同一轮里多次重读宿主任务并据此改变本轮结论
- 在线程池内部绕过外层根层重判，直接把上一轮局部结论延长成下一轮总结论


## 6. 单步推进规则

### 6.1 总原则

必须固定：

- `任务管理任务每次运行只推进宿主任务前进一步`

这里的“前进一步”指：

- 完成一次管理决策
- 完成一次合法状态转移
- 完成一次结果回收

而不是：

- 一次运行内连续跑完整个 `筹办 -> 维护 -> 纠偏 -> 收束`

### 6.2 合法步型

每次运行只允许完成以下步型中的 `一种`：

1. `域判定步`
   - 根据输入快照判断本轮进入 `筹办 / 维护 / 纠偏 / 收束 / 等待学习处理`
   - 这里的域判定只属于任务管理层，不替代根层方向裁决

2. `方法选择步`
   - 选择或切换本轮管理方法
   - 更新当前步骤、当前方法、建议方法、二次特征

3. `结构推进步`
   - 创建一个步骤
   - 或补齐一个结果锚点
   - 或形成一个派生需求候选
   - 或绑定一个当前方法

4. `执行下发步`
   - 让宿主任务进入 `就绪 / 排队中 / 执行中`
   - 将任务送入后续执行链

5. `等待挂起步`
   - 让宿主任务进入 `等待学习 / 等待外部反馈 / 等待执行结果`
   - 为实例写入等待原因与恢复点

6. `结果回收步`
   - 回收一次执行结果、学习结果或外部反馈
   - 形成 `下一域建议 / 下一状态建议 / 当前下一步去向建议`

7. `终态收口步`
   - 将宿主任务推进到 `完成 / 失败 / 取消 / 无法继续`

必须补充固定：

- 合法步型默认建立在 `根层已允许继续推进` 的前提下；若根层已不允许继续推进，则普通步型应让位于 `终态收口步` 或停机边界处理。
- 即使完成了 `域判定步` 或 `结果回收步`，后续是否继续推进，仍应回到外层再做 `根层重判 -> 判域/调度`。
- 若本轮通过 `结构推进步` 形成了 `学习型派生需求候选`，则它必须先返回外层，由自我线程或学习调度层决定是否正式创建对应的依赖学习任务。
- 被外层批准后的依赖学习任务，才会在下一轮重新进入 `任务管理任务` 的支撑链，并落到新的 `筹办` 阶段处理。
- 这轮新的 `筹办` 阶段，应优先为该依赖学习任务找到 `学习任务兜底方法`。
- `学习任务兜底方法` 的职责之一，是为当前依赖学习任务生成一组可执行学习方法选项；在这些方法选项尚未生成前，不得跳过筹办直接进入 `执行下发步`。

### 6.3 非法步型

一次运行中默认不允许：

- 连续创建多个步骤
- 连续实例化多个正式子任务
- 下发执行后在同轮内等待执行结束再继续推进
- 在同轮里同时完成 `筹办 + 维护`
- 在同轮里同时完成 `纠偏 + 收束`


## 7. 任务管理实例状态机

任务管理实例至少应具有以下状态：

1. `未创建`
2. `待提交`
3. `待运行`
4. `分析中`
5. `已下发等待执行结果`
6. `等待学习反馈`
7. `等待外部反馈`
8. `回收结果中`
9. `已完成`
10. `已失败`
11. `已取消`

推荐的主路径是：

```text
待提交
-> 待运行
-> 分析中
-> 已下发等待执行结果 / 等待学习反馈 / 等待外部反馈 / 已完成 / 已失败 / 已取消
-> 回收结果中
-> 待运行
```

必须固定：

- `等待` 状态不占用物理线程
- `回收结果中` 仍然只允许回收一次结果并推进一步


## 8. 线程池模型

### 8.1 为什么使用线程池

必须固定：

- 任务管理实例数量可能远大于可接受的系统线程数
- 很多实例会长期处于等待态
- 因此不应采用 `一任务一永久系统线程`

### 8.2 正确模型

正确模型应为：

```text
一任务一管理实例
+ 有上限的任务管理线程池
+ 每次只执行一次推进一步
+ 等待时释放物理线程
```

### 8.3 线程池上限

必须固定：

- 任务管理线程池必须有并发上限
- 上限应与 `动作线程 / 学习线程 / 自我线程` 分离评估
- 不允许让任务管理线程池无限扩张

推荐原则：

- 若当前实现尚处于保守阶段，可先退化为 `上限 = 1`
- 这等价于“单线程串行执行所有任务管理实例”

### 8.4 与自我线程的关系

必须固定：

- 自我线程负责 `发现待处理任务 + 提交请求 + 投递唤醒`
- 任务管理线程池负责 `执行单步管理`
- 自我线程不是任务管理线程池的替代物
- 但在 `V0` 阶段，自我线程可同步直调任务管理入口，视为线程池大小为 `1` 的特例
- 自我线程或外层协调器还负责把 `根层裁决结果 + 任务管理上下文` 装配进请求，再交给线程池承载
- 自我线程不直接展开任务树；它只负责决定该实例当前是否继续获得下一次 `推进一步` 的资格

一句话收口：

`线程池跑一步，自我线程批下一步。`


## 9. 等待与唤醒规则

### 9.1 等待不是阻塞

必须固定：

- 任务管理实例进入等待态后，必须保存：
  - `等待原因`
  - `等待目标`
  - `最近下发动作`
  - `恢复点`
- 然后立刻释放物理线程

### 9.2 唤醒事件

至少以下事件可唤醒任务管理实例：

- `步骤执行完成`
- `步骤执行失败`
- `步骤无法执行`
- `依赖任务完成`
- `依赖任务失败`
- `学习提交完成`
- `学习回滚完成`
- `外部结果已确认`
- `超时`
- `人工中断`

### 9.3 唤醒后的规则

唤醒后必须重新形成一轮新请求：

- 旧等待态只提供恢复上下文
- 新一轮开始前，仍应先由外层完成根层重判与最小任务管理上下文刷新
- 新一轮仍然只允许推进一步


## 10. 宿主任务状态推进责任

任务管理任务必须允许推动宿主任务进入以下治理状态：

- `就绪`
- `排队中`
- `执行中`
- `待重筹办`
- `等待学习`
- `等待外部反馈`
- `收束相关状态`
- `完成 / 失败 / 取消 / 无法执行`

但必须固定：

- `任务管理任务` 负责的是 `让宿主任务进入相应状态`
- 不等于它自己直接完成叶子动作执行

因此运行语义应是：

```text
选择管理步骤与管理方法
-> 推动宿主任务进入对应运行状态
-> 等待执行结束或反馈回流
-> 结果回到外层重判
-> 下一轮继续管理
```


## 11. 状态修改权限边界

### 11.1 允许修改

任务管理任务允许修改：

- `当前步骤`
- `当前方法`
- `当前依赖任务引用`
- `宿主任务当前治理状态`
- `等待学习标记`
- `等待外部反馈标记`
- `管理状态面`
- `管理动作动态`
- `管理实例因果`

### 11.2 不允许修改

任务管理任务不允许：

- 伪造叶子执行成功结果
- 伪造外部动作成功结果
- 绕过执行器直接把外部动作记成已完成
- 绕过学习提交直接宣称学习已成功
- 直接篡改需求满足真值


## 12. 并发、去重与幂等

必须固定：

1. 同一宿主任务同一时刻只能有一个活跃推进者。
2. 同一实例的请求 mailbox 默认按时间顺序消费。
3. 重复投递同一轮请求时，应支持 `去重` 或 `覆盖旧快照`。
4. 以下操作必须具备幂等保护：
   - `创建步骤`
   - `形成派生需求候选`
   - `进入等待学习`
   - `绑定当前方法`
   - `切换到等待外部反馈`


## 13. 与当前代码的关系

当前代码更接近：

```text
自我线程
-> 根层重判与任务管理上下文装配
-> 直接调用 任务控制总控模块_v0::处理任务入口(...)
-> 同步写回任务管理任务镜像树
```

这意味着：

- `任务管理任务镜像树` 已存在
- `任务管理任务` 的管理步骤、管理状态、管理方法、管理因果也已存在
- 但当前仍主要是 `自我线程同步驱动`
- 还没有完全拆成独立的 `任务管理实例 mailbox + 线程池`

因此必须把当前实现理解为：

- `本文模型的 V0 退化实现`

而不是：

- 最终形态已经完全到位

并且必须补充理解为：

- 当前同步直调并不意味着根层、任务管理上下文、四域承接和线程承载可以混成一层。
- 只是暂时由 `自我线程 + 同步入口` 承担了后续线程池与 mailbox 还没拆开的职责。


## 14. 推荐落地顺序

后续实现应按以下顺序推进：

1. 固定 `根层裁决结果 + 任务管理上下文 -> 任务管理请求` 的装配顺序。
2. 固定 `一任务一实例` 的实例索引。
3. 固定 `任务管理请求` 结构。
4. 固定 `实例状态机` 与 `等待原因`。
5. 固定 `单步合法步型表`。
6. 接入 `任务管理线程池`，初始并发上限可为 `1`。
7. 将当前 `自我线程同步直调` 收敛为 `向实例投递请求`。
8. 再把 `等待学习 / 外部反馈 / 执行完成` 的唤醒统一成事件回投。


## 15. 禁止事项

禁止以下实现：

1. 一个宿主任务同时被多个任务管理实例并行推进。
2. 一个任务管理实例在一次运行里连续推进多步。
3. 任务管理实例在物理线程里长时间阻塞等待执行结束。
4. 把“已发出外部动作”直接当成“外部结果已成立”。
5. 未经返回消息或回流事件，就直接推进到下一步。
6. 让任务管理任务直接替代叶子执行器。
