## 3. 核心结论

必须固定以下结论：

1. 最终结果不足以支撑完整判断，因为很多否定项来自“曾经出现、后来又被恢复的反向中间态”。
2. 任务结果回流不得只提交最终结果；凡出现关键中间状态变化，必须连同最终结果一起提交。
3. 关键中间状态必须表达为“动作主体—动作语义—变化主体—特征类型—变化前值—变化后值”的特征变化项或其专用细化结构。
4. 工作线程只提交中间状态与提示，不直接裁定否定项是否成立。
5. 否定项候选由自我线程基于中间状态变化和二次特征判定结果抽取。
6. 关键中间状态在进入线程后，还应继续通过 `0038` 生成对应的正向/负向影响项；负向影响项是稳定否定项来源之一。


## 4. 为什么只交最终结果不够

以下类型的信息，即使最后被恢复，也仍然是关键否定项来源：

- 目标是 `依赖满足`，过程中却出现过 `依赖断开`
- 目标是 `风险下降`，过程中却先出现 `风险升高`
- 目标是 `步骤可执行`，过程中却先经历 `阻塞`
- 目标是 `资源可用`，过程中却先出现 `资源被占满`

必须固定：

1. 最终结果告诉你“最后到了哪”。
2. 关键中间状态告诉你“路上曾掉进哪些坑”。
3. 否定项主要藏在这些关键中间状态里，而不只藏在最终失败里。


## 5. 什么叫关键中间状态

不要求任务执行过程把所有微状态都上报，只上报 `关键中间状态`。

### 5.1 关键中间状态的 5 条判定标准

只要满足任一条，就应进入结果回流：

1. 与目标方向相反
2. 形成阻塞或解阻
3. 改变控制面判断
4. 产生副作用或风险
5. 改变父任务语义

一句话收口：

`上报的是语义关键帧，不是原始流水账。`


## 6. 关键中间状态专用结构

建议使用如下最小结构：

```cpp
struct 结构_关键中间状态变化项 {
    主键 动作主体主键 = 0;
    std::string 动作语义键{};

    主键 变化主体主键 = 0;
    主键 特征主键 = 0;
    特征值 变化前值{};
    特征值 变化后值{};

    时间戳 发生时间 = 0;
    bool 已确认生效 = false;
    bool 已恢复 = false;
};
```

必须固定：

1. `已恢复 = true` 代表该反向中间态后来被拉回，但仍然不能丢。
2. `已确认生效 = false` 代表它仍处于候选事实层，不得直接当最终状态真值。


## 7. 结果回流消息三段式

建议正式固定为：

```cpp
struct 结构_任务结果回流消息 {
    主键 任务主键 = 0;
    主键 父任务主键 = 0;

    枚举_结果状态 最终结果状态{};
    特征值 最终结果值{};

    std::vector<结构_关键中间状态变化项> 关键中间变化集{};

    bool 可能存在反向中间态 = false;
    bool 可能形成阻塞 = false;
    bool 可能影响父任务 = false;
};
```

### 7.1 三段解释

| 段落 | 最小内容 | 含义 |
| --- | --- | --- |
| 最终结果段 | `最终结果状态 + 最终结果值` | 说明最后到了哪 |
| 关键中间状态段 | `关键中间变化集` | 说明推进过程中出现过哪些关键状态变化 |
| 提示段 | `可能存在反向中间态 / 可能形成阻塞 / 可能影响父任务` | 只做提示，不替代最终裁决 |

必须固定：

1. 工作线程可以提交“可能”提示，但不能直接把“否定项成立”当最终结论提交。
2. 关键中间状态段是正式字段，不是日志附注。


## 8. 自我线程收到结果回流后的最小处理顺序

自我线程至少做 5 步：

1. 落一次特征变化  
   把最终结果和关键中间状态分别落到 `状态节点 / 动态节点`

2. 重算二次特征  
   当前最小必须至少重算：
   - `依赖满足`
   - `当前步骤可执行`

3. 抽取否定项候选  
   把那些 `反向推进 / 形成阻塞 / 产生副作用` 的中间态转成否定项候选

4. 路由给父任务  
   父任务只判断：这对我当前步骤有什么影响

5. 更新全局控制面  
   自我线程再判断是否影响优先级、收束、实例化新任务、学习候选池


## 9. 值得优先扩展的 4 个二次特征

当前正式最小集仍以 [0034_最小二次特征判定表_v0.1.md](D:\鱼巢\规范\0034_最小二次特征判定表_v0.1.md) 的两项为准，但结果回流链最值得优先扩展的候选二次特征固定为：

1. `相对目标偏移`
2. `反向推进标记`
3. `依赖满足`
4. `当前步骤可执行`

### 9.1 相对目标偏移

对标量特征，可先采用：

```text
相对目标偏移 = |变化后值 - 目标值| - |变化前值 - 目标值|
```

解释：

- `< 0`：更接近目标
- `= 0`：无明显变化
- `> 0`：更远离目标

### 9.2 反向推进标记

它是对 `相对目标偏移` 的离散化：

- 接近目标
- 中性
- 远离目标

必须固定：

1. `相对目标偏移` 和 `反向推进标记` 在当前版是结果回流链的下一批直接扩展候选。
2. 它们当前还不替代 `依赖满足` 和 `当前步骤可执行` 的最小正式地位。


## 10. 否定项候选如何从中间状态里长出来

否定项候选最少可从 4 类情况里抽取：

1. 反向推进但最终恢复
2. 形成阻塞
3. 造成副作用
4. 反复出现的失败性中间态

一句话收口：

`中间状态提供原始负样本，二次特征把它们提纯成否定项候选。`


## 11. 规范表述

推荐正式表述如下：

`在任务向目标推进的过程中，凡出现与目标方向相反、形成阻塞、产生副作用或会改变父任务语义的关键中间状态，任务在完成后向自我线程提交结果时，必须将这些关键中间状态变化一并提交。`

`关键中间状态应表达为“动作主体—动作语义—变化主体—特征类型—变化前值—变化后值”的特征变化项。`

`自我线程基于这些变化项重算二次特征，如相对目标偏移、反向推进标记、依赖满足和当前步骤可执行性，并据此抽取否定项候选、更新全局控制态并路由结果给父任务。`


## 12. 对下游模块的直接约束

必须固定：

1. `治理消息模块.ixx` 应允许正式定义 `结构_关键中间状态变化项` 与 `结构_任务结果回流消息`。
2. `治理消息路由模块.ixx` 应继续把结果回流消息视为必须上浮到自我线程的正式类型。
3. 现有线程逻辑在真正接线前，可以先只消费 `最终结果段 + 提示段`，但不得删除 `关键中间状态段`。

一句话收口：

`结果回流消息从现在起不再只是“最后成功没成功”，而是“最后结果 + 关键中间状态轨迹 + 提示段”的正式合同。`
