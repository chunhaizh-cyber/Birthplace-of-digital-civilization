## 3. 核心结论

必须固定以下 16 条：

1. `自我线程主循环` 就是当前运行时中的 `自我主循环`。
2. 主循环首先负责 `根层重判`，而不是先盲目推进任务或学习。
3. 主循环必须承接线程回流消息，并把它们统一转成：状态变化、结果特征、条件判定结果、因果引用面和新的治理输入。
4. 主循环不直接凭空发明因果；因果必须建立在动作/方法执行后的变化、结果消息和状态比较之上。
5. 主循环必须统一做 `双值结算`，不能把安全/服务结算散落给各执行线程自行决定。
6. 主循环必须统一刷新 `需求树`、按父需求权重调制子需求权重、生成 `需求队列` 并刷新当前主需求。
7. 主循环必须依据 `需求队列` 的综合权重以及对 `安全值 / 服务值` 的变化趋势，统一生成和更新 `方法禁止项集`。
8. 主循环必须统一管理 `可运行队列 / 等待队列 / 回流队列 / 收束停止队列` 这组全局运行队列。
9. 主循环必须根据根需求创建 `任务根节点 / 任务实例入口`，再把入口交给任务管理工作线程推进。
10. 主循环必须接收 `派生需求`，并决定是否将其实例化为新的依赖任务。
11. 子任务结果必须先回主循环，再由主循环路由给父层任务；父层任务判断局部影响，主循环判断全局影响。
12. 主循环每轮必须先冻结本轮开始时已可见的消息批次，并把这批消息及其同步内部闭包处理完，再进入根层重判。
13. 本轮处理中异步新产生的消息必须延后到下一轮；高优先级控制消息可按抢占规则中断本轮。
14. 主循环必须统一消费显式治理事件与延迟事实，再决定 `继续 / 待机 / 停止 / 收束 / 转入学习 / 再提交` 这类最终运行去向。
15. 动作线程和学习线程只负责消费与推进，不负责替代主循环做最终运行裁决。
16. 若缺少本文职责中的任一关键项，规范层就不能宣称“需求-任务-方法-学习-任务管理任务”已经形成真正最小治理闭环。
17. `控制消息` 与 `全局环境消息` 的首收件人必须是 `主循环 / 自我线程`，不得先塞给局部工作线程。
18. `明确绑定到具体任务且仅影响局部流程的外部反馈`，可先进入对应任务管理工作线程；若升级为 `派生需求 / 阻塞变化 / 异常 / 需要改优先级 / 需要实例化新任务`，必须再上报主循环。
19. `结果回流消息` 必须先回主循环；`推进消息` 只上报会改变控制面判断的部分。
20. 主循环内部最重的控制面工作，是收到需求增量消息后对 `需求树 / 需求队列` 做一次有边界的增量刷新，而不是完成任务本身。
21. `需求树 / 需求队列` 的定时全局刷新必须任务化，由主循环发起为刷新任务并交给任务管理工作线程执行。

一句话收口：

`主循环不是简单心跳器，而是根层重判、需求形成、全局优先级、任务入口实例化、工作线程调度、结果路由、双值结算和最终去向决定的统一承载面。`

## 4. 主循环最小职责表

| 职责 | 回答的问题 | 为什么必须留在主循环 |
| --- | --- | --- |
| `消息消费与结果归并` | 上一轮线程推进到底产生了什么 | 只有主循环能把多线程结果统一回收成下一轮输入 |
| `固定先天机制执行` | 服务衰减、待机保护、风险回归等是否该触发 | 这些属于统一主体底线规则，不应散落给叶子线程 |
| `根层重判` | 当前是否允许继续推进，还是应待机/停止/收束 | 这是所有承接层之前的上位输入 |
| `根需求形成与需求树 / 需求队列 / 方法禁止项治理` | 当前真正最该推进哪个需求，父需求如何调制子需求，队列如何排序，哪些方法当前必须禁止 | 需求信息及其运行时约束是统一主体的直接治理对象，不应分散裁判 |
| `全局运行队列治理` | 当前谁可运行、谁应等待、谁在回流、谁应进入收束/停止 | 优先级和运行资格属于统一控制面，不应由局部线程各自决定 |
| `任务入口实例化与工作线程调度` | 当前应创建哪个任务入口、交给哪个任务管理工作线程推进 | 任务根节点方向只读，实例化和调度资格必须由主循环统一掌握 |
| `显式事件消费与结果路由` | 这一轮到底是普通推进、恢复重建、学习反馈、执行结果还是外部反馈驱动；子结果应路由给谁 | 只有主循环能把事件统一转成下一轮治理输入并处理父子任务之间的全局路由 |
| `延迟事实归并` | 当前还有哪些“已到达但需稍后裁决”的事实要保留 | 这些事实会直接影响最终去向，不应只留在局部摘要 |
| `双值结算` | 本轮结果如何影响安全值和服务值 | 双值属于根控制面，只能统一结算 |
| `最终去向决定` | 继续提交、待机、收束、转入学习还是停止 | 这属于最终裁决层的工程承载面 |


## 5. 统一主循环顺序

从规范层看，主循环至少应按以下顺序运行：

```text
1. 冻结本轮开始时已可见的消息批次
2. 清空本轮冻结消息集
3. 清空这些消息在本轮内引发的同步内部闭包
4. 执行固定先天机制
5. 基于当前值与需求值做根层重判
6. 形成根需求并刷新需求树
7. 按父需求权重调制子需求权重
8. 生成需求队列、计算综合权重并刷新当前主需求
9. 依据需求队列综合权重与双值变化趋势生成 / 更新方法禁止项集
10. 必要时创建 `需求树 / 需求队列全局刷新任务入口`
11. 刷新可运行 / 等待 / 回流 / 收束停止队列
12. 创建任务根节点与任务入口，并装配任务管理上下文
13. 调度任务管理工作线程
14. 接收派生需求并决定是否实例化新任务；接收子结果并路由给父层任务
15. 依据任务结果、需求队列综合权重与方法禁止项命中情况统一结算安全值 / 服务值
16. 决定本轮最终去向
17. 若允许继续推进，再提交下一轮治理或学习请求
```

必须固定：

- 不得把第 `14` 步结算下放给动作线程自行决定最终真值。
- 不得在第 `5` 步之前先默认“允许继续推进”。
- 不得在第 `14` 步之前让动作线程或学习线程自行连跑下一轮主闭环。
- 本轮冻结消息集处理完之前，不得先行做本轮最终派发或最终去向裁决。
- 本轮中新产生的异步消息，不得为了“清空活队列”而硬追入当前轮次。


## 6. 主循环必须直接完成的事项

### 6.1 消费回流消息

主循环必须直接消费至少以下类型的回流：

- `任务执行返回消息`
- `学习线程阶段结果`
- `等待/唤醒结果`
- `宿主输入或外部反馈`
- `恢复重建一次性事件`
- `mailbox` 中待消费的显式治理事件

必须固定：

- 这些回流首先属于主循环输入，不属于执行线程本地真值。
- 回流要先冻结成“本轮消息集”，并处理其同步内部闭包，再进入根层重判与双值结算。
- 正式 `负反馈` 也只能在这一步之后，由主循环统一生成。
- `控制消息` 与 `全局环境消息` 必须直接进入主循环，不得被局部工作线程先行吞掉。
- `任务绑定且仅影响局部流程的外部反馈` 可先进入对应任务管理工作线程，再把升级后的控制面变化上报主循环。

补充固定：

1. 主循环每轮不是“只吃一条消息”，也不是“追到活队列彻底为空”为止。
2. 主循环每轮应处理完本轮冻结消息集，以及这些消息同步引发的内部闭包。
3. 本轮处理中异步新产生的消息，应进入下一轮。
4. `打断 / 收束 / 停止 / 强制优先级刷新 / 强制预算刷新` 这类高优先级控制消息，可按抢占规则中断本轮。
5. `步骤内部小进度 / 临时变量变化 / 内部小重试 / 纯中间状态` 不属于必须上报主循环的消息。
6. `当前步骤阻塞 / 解除阻塞 / 派生需求 / 局部失败 / 异常 / 当前任务完成 / 最终结果返回 / 需要父层任务判断的子结果` 属于应上报主循环的关键推进事件。

### 6.4 需求树、需求队列、派生需求与方法禁止项治理

主循环必须直接完成：

- 新需求登记
- 重复需求合并
- 需求树中父需求对子需求的权重传播与调制
- 需求队列生成
- 需求综合权重计算
- 已完成需求删除或降权
- 受影响需求分支的增量刷新
- 内部学习需求与派生需求的维护
- 派生需求是否上升为新任务的判定
- 方法禁止项生成
- 方法禁止项更新
- 方法禁止项解除或降级

必须固定：

- `需求树` 是需求真值面，`需求队列` 只是当前轮次的运行时排序视图。
- `需求列表` 是 `需求队列` 的线程持有形态；它由主循环从 `需求树` 的 `链上 / 链下` 扫描生成。
- 生成 `需求列表` 时，必须先按 `需求类型` 聚拢，使相同需求类型连续排列，再统计各类型组的 `权重汇总`。
- `需求列表` 自身必须按权重排序；工程推荐口径为：`先按类型组权重汇总排序，再在组内按节点权重排序`。
- `任务管理任务` 可以直接更新 `需求树`，但不得直接决定 `需求队列` 的最终顺序、当前主需求或最终 `方法禁止项集`。
- `需求队列` 中需求的综合权重，既影响 `安全值 / 服务值` 的变化趋势判断，也构成派生任务优先级的重要组成部分。
- 任一需求节点权重变化后，主循环必须同步刷新其所属 `需求列表` 类型组的权重汇总，并在必要时重排整个 `需求列表`。
- `消息驱动的需求增量更新`，由主循环直接完成；它是主循环内部最重的控制面工作，但必须保持为有边界的轻量治理。
- `需求树 / 需求队列的定时全局刷新`，不得在主循环内长时间执行，而必须由主循环创建 `刷新任务入口`，交给任务管理工作线程执行。
- 主循环对全局刷新任务只负责：`判断是否需要刷新 -> 创建任务入口 -> 赋予优先级和预算 -> 接收刷新结果 -> 决定是否替换当前需求列表/需求队列`。
- `方法禁止项` 只禁止方法，不禁止根方向；它属于运行时约束，不属于永久真理。
- 每个 `方法禁止项` 至少必须明确：`禁止方法 / 触发条件 / 禁止原因 / 证据来源 / 生效范围 / 解禁条件 / 当前状态`。
- `派生需求` 可以由任务管理任务提出，但只能由主循环决定是否实例化为新任务。
- 需求治理属于统一主体内部整理面，不应放到叶子执行器里各自处理。

必须固定：

- 必须显式结合需求位、需求队列综合权重、方法禁止项命中情况与结果方向。

### 6.8 最终去向决定

主循环必须统一决定：

- `继续提交`
- `回到重筹办`
- `转入学习`
- `进入收束`
- `待机`
- `停止`

必须固定：

- 这是最终裁决层的工程承接动作。
- 动作线程和学习线程只能提供证据，不能越权代决。
- 即使事件来自 `mailbox`，也必须先回到主循环统一拍板，不得让同一宿主越权连续推进两轮。


## 7. 主循环不直接做的事

以下事项不应直接放在主循环里长期阻塞执行：

1. 叶子方法长时间执行
2. 复杂任务树细节推进
3. 大块学习搜索与验证
4. 阻塞式外设或宿主 I/O

必须固定：

- 主循环负责“决定与归并”，不负责长期占用执行时间片的重活。


## 8. 与因果的边界

必须固定：

1. 主循环不直接凭空生成因果。
2. 主循环负责基于以下材料刷新或引用因果：
   - 动作后的状态变化
   - 方法返回结果
   - 执行线程回流消息
   - 前后状态比较结果
3. 没有变化，就没有新的变化因果；但动作事实动态仍可存在。
4. 因果仍然只作为主循环后续判断与组织的证据面，不直接替代根层裁决。


## 9. 收口结论

后续统一按以下口径执行：

1. 自我线程主循环必须同时承担：消息归并、固定机制、根层重判、需求治理、治理任务存在维护、上下文装配、双值结算和最终去向决定。
2. 若这些职责没有在规范层被统一收口，就不能宣称最小治理闭环已经成立。
3. 主循环是“统一主体”的运行核心，不是简单定时器。

最终统一收口为：

`自我线程主循环真正要做的，不只是选需求和收消息，而是把回流结果重新组织成下一轮可裁决、可结算、可治理的统一输入。`
