# 子任务完成触发父任务重筹办规范 20260508

## 1. 总原则

子任务完成不是父任务完成，而是父任务重新筹办的触发器。

这里的“完成”按单个子需求 / 子任务计算。父需求下任意一个活动子需求被满足、截止或明确收束后，父任务都必须重新筹办一次，不等待同层子需求“整批完成”。

固定流程：

```text
子任务完成
    ↓
父任务标记为待重筹办
    ↓
父任务阶段回到筹办
    ↓
重新判断父需求状态、活动子需求、方法可执行性、参数配对和新缺口
    ↓
进入等待、继续派生子需求、执行目标方法、收束、失败或待机
```

一句话：

```text
每个子任务完成，都是父任务重新筹办的时钟脉冲。
```

父任务每次被这个时钟脉冲唤醒后，都重新判断自己是继续补条件，还是终于可以执行自身目标。

## 2. 普通任务的两轮结构

普通任务不是直接执行业务目标。每个普通任务在执行自身目标前，必须先完成自己的可执行性治理。

```text
第一轮：
    让任务变得可执行。

第二轮：
    在任务已经可执行之后，重新筹办并执行真正的目标方法。
```

任务管理任务和普通任务不是两个割裂系统。普通任务内部自带治理阶段：

```text
筹办_检查可执行性
等待_可执行性子任务
筹办_重新检查
执行_目标方法
回流_结果归并
完成 / 未完成
```

## 3. 子任务完成后的父任务状态

子任务完成后，父任务必须写成：

```text
父任务状态 = 待重筹办
父任务阶段 = 筹办
父任务最近唤醒原因 = 子任务完成
```

不得直接写成：

```text
父任务完成
父需求满足
父任务执行中
```

子任务完成只说明某个阻塞点被处理了，不说明父需求目标已经达成。

如果一个父需求同时有多个子需求，任意一个子需求状态发生“已满足 / 已截止 / 已取消 / 已收束”变化时，都要触发父任务进入待重筹办。重筹办后若仍存在活动直接子需求，父任务继续等待或调度下一项；若不存在活动直接子需求，父任务回到自身目标进行方法查找和可执行性检查。

## 4. 父任务重新筹办顺序

父任务被子任务唤醒后，必须按以下顺序重新筹办：

```text
1. 刷新父需求当前状态。
2. 检查父需求是否已经满足。
3. 检查是否还有活动直接子需求。
4. 如果有活动子需求，继续等待或调度下一个子任务。
5. 如果没有活动子需求，回到父需求自身目标。
6. 重新查找能满足父需求目标的方法。
7. 检查方法是否可执行。
8. 如果方法仍有缺口，生成下一轮可执行性子需求 / 子任务。
9. 如果方法可执行，进入父任务目标执行。
```

父任务完成只能由可比较父任务目标结果符合确认，并交自我线程检查父需求满足；不能由子任务完成事件直接确认。

## 5. 活动子需求判断

判断父需求是否仍处于管理 / 等待状态，不能使用：

```text
父需求->子节点数量 > 0
```

必须使用：

```text
是否存在活动直接子需求
```

活动直接子需求指：

```text
未满足
未截止
未取消
仍然阻塞父任务执行
```

伪代码：

```cpp
bool 存在活动直接子需求(需求节点* 父需求) {
    for (auto* 子 : 直接子需求列表(父需求)) {
        if (!子) {
            continue;
        }
        if (子->主信息.需求有效截止 != 0) {
            continue;
        }
        if (!子->主信息.是否阻塞父任务执行) {
            continue;
        }
        if (需求已满足(子)) {
            continue;
        }
        return true;
    }
    return false;
}
```

历史子需求可以留在树上，但不得阻止父任务回到自身目标进行第二轮筹办。

## 6. 可执行性子需求不能复制父需求目标

第一轮子需求承接的是父任务可执行性缺口，不是父需求业务目标。

父需求可能是：

```text
目标对象.X特征 = 目标状态
```

子需求不能继续是：

```text
目标对象.X特征 = 目标状态
```

子需求应该是：

```text
目标方法头已经存在（由筹办阶段创建或复用，不作为子需求目标）
方法动作骨架状态 = 已形成
方法可执行入口状态 = 已具备
方法条件节点存在状态 = 已存在
方法结果节点存在状态 = 已存在
方法条件结果配对状态 = 已配对
参数配对状态 / 任务参数绑定状态 = 已配对
方法运行账完整性 = 完整
方法稳定状态 = 达标
```

第一轮子需求的目标主体通常是：

```text
任务
任务虚拟存在
方法
方法虚拟存在
参数绑定场景
```

而不是父需求的业务目标对象。

筹办或执行回执上报结构缺口时，必须显式携带缺口目标主体。自我线程创建子需求时以 `缺口目标主体 + 缺口特征类型` 为权威目标键，不能从父需求复制目标主体、当前状态或目标状态。

方法结构补齐任务还必须区分“被补方法”和“当前执行方法”：

```text
被补方法 = 缺动作骨架 / 条件结果对 / 运行账的目标方法头
当前执行方法 = 能补齐该缺口的学习 / 练习 / 尝试学习方法
```

不得把被补方法本身当成当前执行方法；否则会把“补齐方法结构”的子任务误绑定成“执行父需求目标”的业务方法。

## 7. 多轮筹办模型

第一轮、第二轮只是解释口径，实现必须支持 N 轮：

```text
第 N 轮筹办：
    检查当前是否可执行

如果不可执行：
    派生第 N 轮子需求 / 子任务
    等待子任务

子任务完成：
    进入第 N+1 轮筹办
```

可能过程：

```text
第一轮：方法查找未命中，筹办创建目标方法头并待重筹办
第二轮：目标方法头缺动作骨架
第三轮：目标方法头缺条件结果对
第四轮：目标方法头缺参数绑定
第五轮：目标方法可执行，执行目标方法
```

每一轮都是：

```text
筹办 -> 缺口子任务 -> 子任务完成 -> 重新筹办
```

## 8. 执行桥缺失事实到子需求映射

执行桥或筹办检查发现的可执行性缺失，必须转换成明确原子需求：

| 缺失事实 | 原子子需求目标 |
| --- | --- |
| 方法查找未命中 | 不生成子需求；筹办创建或复用待学习目标方法头，并让父任务待重筹办 |
| 缺动作骨架 | 方法动作骨架状态 |
| 缺可执行入口 | 方法可执行入口状态 |
| 缺条件节点 | 方法条件节点存在状态 |
| 缺结果节点 | 方法结果节点存在状态 |
| 缺条件结果对 | 方法条件结果配对状态 |
| 实际场景缺少参数特征 | 参数配对状态 / 任务参数绑定状态 |
| 缺运行账完整性 | 方法运行账完整性 |
| 结果复现不稳定 | 方法稳定状态 |

不得只返回泛泛的“任务缺条件需自我线程”。自我线程必须能据此创建或复用明确的原子子需求。

“没有方法头”不是子需求入口。筹办先让目标方法头出生；只有方法头存在但不可执行时，才把动作骨架、入口、条件结果对、方法可用性、运行账和稳定性等目标展开成子需求。学习 / 练习是否承接这些目标，由下一轮筹办按结果能力匹配决定。

## 9. 去重规则

添加可执行性子需求前必须检查：

```text
1. 子需求目标不能和父需求目标重叠。
2. 同父直接兄弟子需求目标不能重叠。
3. 如果已有活动兄弟子需求目标重叠，只更新归因和统计，不新增节点。
4. 如果已有历史兄弟子需求已经满足或截止，可按策略复用记录，但不得阻止父任务重新筹办。
```

父子同目标必须作为结构错误处理：

```text
候选子需求目标
    vs
父需求自身目标
```

如果目标主体、目标特征类型和方向掩码重叠，则不得新增子需求，也不得返回一个“等价子需求”来继续执行。正确处理是拒绝入树并记录派生需求逻辑错误。

原因是：

```text
父需求 -> 同目标子需求
```

不是目标分解，而是把同一个目标复制到下一层，会破坏“第一轮让任务可执行，第二轮执行父目标”的多轮筹办模型。

去重检查顺序固定为：

```text
1. 候选子需求 vs 父需求自身目标：
       重叠则拒绝入树。
2. 候选子需求 vs 同父直接兄弟子需求：
       重叠则返回已有兄弟子需求并更新归因 / 统计。
3. 均不重叠：
       才允许新增子需求。
```

## 10. 最终规则

```text
1. 子任务完成后，父任务必须标记为待重筹办。
2. 父任务重筹办时先检查父需求是否已满足。
3. 再检查是否还有活动直接子需求。
4. 没有活动子需求时，父任务回到自身目标，重新查找方法。
5. 如果方法仍不可执行，生成下一轮可执行性子需求。
6. 如果方法可执行，才进入父任务目标执行。
7. 子任务完成不等于父任务完成。
8. 父任务完成只能由可比较父任务目标结果符合确认，并交自我线程检查父需求满足。
```

一句话定型：

```text
第一轮让任务能执行；
第二轮才让任务满足需求。
```

再固定一条：

```text
每个子需求满足一次，父任务就重筹办一次。
```
