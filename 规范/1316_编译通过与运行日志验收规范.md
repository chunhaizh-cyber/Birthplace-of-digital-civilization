# 1316 编译通过与运行日志验收规范

本文用于统一后续代码修改后的最小验收口径，回答两个问题：

1. `代码本身是否达到可交付底线`
2. `逻辑是否按规范真实命中`

本文是 [0000_规范总览与优先级.md](D:/鱼巢/规范/0000_规范总览与优先级.md) 在验收方式上的展开约束；若与更高层总纲冲突，以 `0000` 为准。

## 1. 总原则

后续验收统一拆成两层：

1. `代码验收`
2. `逻辑验收`

并且必须按顺序执行：

1. 先过 `代码验收`
2. 再做 `逻辑验收`

`代码验收` 不通过时，不得宣称逻辑已验收通过。

## 2. 代码验收口径

后续代码验收的最小标准固定为：

- 工程在目标配置下 `不报编译错误`
- 若涉及链接阶段，也不得有 `链接错误`

当前默认口径先固定到这里：

- `编译通过` 是硬门槛
- `warning` 是否阻断，不在本文强制定义；若上位任务单独要求，再额外收紧

也就是说，后续“代码已验收”的最小含义不是“看起来合理”，而是：

- `项目能实际编过去`

## 3. 逻辑验收口径

后续逻辑验收不再以“新增验收场景”作为默认方法，而统一改为：

- 先预测关键点应该生成什么运行时数据
- 再在正式运行链上插入日志
- 最后通过日志分析是否达到要求

这里的关键不是“有没有日志”，而是：

- `日志前有预测`
- `日志里有真实运行时数据`
- `日志后有对照分析`

## 4. 标准流程

后续逻辑验收必须优先按下面流程走：

1. `先读规范`
   明确本次修改要命中的规则、边界、阶段和字段。

2. `先写预测`
   在动代码前，先列出本次最关键的 `3~7` 个关键点，说明每个点按规范应该出现什么。

3. `再插日志`
   把日志插到正式运行链、正式事件链、正式调度链上，而不是另造一条专门验收链。

4. `再真实运行`
   通过主程序入口、正式线程、正式任务链、正式学习链触发逻辑。

5. `再读日志`
   对照预测关键点，检查实际运行时数据是否命中。

6. `最后下结论`
   明确写出：
   - 哪些点已命中
   - 哪些点未命中
   - 未命中的原因是逻辑不对、日志不够，还是触发链没跑到

## 5. 关键点预测要求

“预测关键点应该生成什么”是正式验收的一部分，不是附属说明。

每次逻辑验收前，至少应预测：

1. `入口点`
   例如：事件类型、宿主任务、当前方法、当前门控结果。

2. `分支点`
   例如：功能域切换、缺口判定、方法需求位、允许执行判断、学习承接判断。

3. `产出点`
   例如：生成了哪个步骤、哪个结果、哪个子任务、哪个长期策略写回。

4. `收口点`
   例如：当前下一步去向、最近反馈摘要、最终去向、恢复快照摘要。

预测时不要求先知道所有值，但至少要知道：

- `应该出现哪一类数据`
- `应该在哪个阶段出现`
- `应该是什么方向或关系`

## 6. 日志最小要求

用于逻辑验收的日志，至少应显式包含：

1. `触发来源`
   说明这轮逻辑是由什么正式入口触发的。

2. `关键输入`
   例如当前宿主、当前方法、当前缺口、根层结果、门控结果、事件类型。

3. `关键中间态`
   例如功能域、方法需求位、本能能力缺口、步骤位、步骤语义、恢复点。

4. `关键产出`
   例如新建/复用的方法、步骤、结果、补条件子任务、学习子任务、写回结果。

5. `关键结论`
   例如当前下一步去向、反馈类型、总控结果、最终去向。

日志内容必须是 `真实运行时数据`，不能只写抽象结论词，例如单独写“这里通过了”是不够的。

## 7. 日志不足时的处理规则

若逻辑是否正确无法判断，默认先做：

- `补日志`

而不是先做：

- `补验收场景`
- `补额外注入链`
- `补专门的模拟主链`

也就是说：

- 看不清，就先把正式链路上的中间态打亮
- 不允许用另造一条验收链来替代真实主链

## 8. 禁止作为默认正式口径的方法

后续默认禁止把下列方式作为正式验收主路径：

1. `场景化验收`
   不再以新增“场景1/场景2/场景3”这一类固定场景函数作为主验收手段。

2. `额外注入链优先`
   不再优先通过额外控制、额外触发源、构造上下文来证明逻辑正确。

3. `只看最终结果`
   不能只看“最后状态对了”，而不核对中间态是否按规范生成。

4. `只给口头判断`
   不能只说“应该是对的”，必须给出日志中的真实数据来支撑判断。

## 9. 完成标准

一次修改若要被视为“完成了最小验收”，至少要满足：

1. `编译通过`
   即工程不报编译错误。

2. `关键点预测已写出`
   即本轮想验证的关键节点和预期产出已经明确。

3. `正式链路日志已产出`
   即不是靠专门验收场景，而是靠正式运行链得到日志。

4. `日志对照已完成`
   即已经把预测和实际日志做过对照分析。

5. `结论已明确`
   即能明确回答“达到要求 / 未达到要求 / 还缺哪条日志或哪段逻辑”。

## 10. 一句话收口

后续统一按这条线执行：

`先保证工程编译不报错，再预测关键点应该生成什么运行时数据，再把这些点打到正式运行日志里，最后靠日志分析是否真正达到规范要求。`
