## 3. 核心结论

必须固定以下结论：

1. `任务管理任务` 是任务系统中的特殊治理任务，但它仍然是任务，不是系统外的超级管理员。
2. 在架构层上，`任务管理任务` 应定义为：`某个任务的局部治理执行体`。
3. `任务管理任务` 与普通任务遵循同一条最小主流程：`筹办条件 -> 条件满足 -> 运行方法 -> 比对结果与目标 -> 满足则完成，不满足则继续完善任务树`。
4. 在实现层上，`任务管理任务` 应定义为：`由一组最小原语构成，并基于任务特征信息执行步骤推进、条件检查、局部等待处理、派生需求上报和结果汇总的受边界约束治理体`。
5. `任务管理任务` 的唯一终局目标，是推动 `宿主任务状态进入完成`；任务树补齐、方法补齐、学习承接、因果沉淀，都是为该目标服务的中间过程。
6. `任务管理任务` 的职责是：在 `根层已给出本轮允许边界` 的前提下，承接宿主任务的一步局部治理推进。
7. `任务管理任务` 改写的是 `宿主任务虚拟存在` 中的内部治理特征；普通任务改写的是 `世界树` 中的对象/场景特征。
8. 为区分承载域，`任务管理任务` 使用的最小内部方法统一称 `原语`；普通任务使用的外部最小方法统一称 `本能函数`。
9. `任务管理任务` 不负责重新定义根方向，不负责根层最终裁决，也不负责直接结算 `安全值 / 服务值`。
10. `任务管理任务` 不负责直接改写正式需求树，不负责直接实例化正式子任务，只能提交 `派生需求消息 / 结构候选 / 治理建议`，由外层决定是否正式落账。
11. `任务管理任务` 负责的是：围绕宿主任务，生成管理事实、管理建议、管理证据，并落实一步合法治理状态迁移。
12. `任务管理任务` 的产出，优先属于 `条件判定结果层` 与 `管理上下文层`，不是最终裁决本身。
13. `任务管理任务` 每轮最多只推进一步，推进结果必须回到外层执行 `根层重判 -> 上下文刷新 -> 再决定是否继续`。
14. `任务管理任务` 可以提出 `继续执行 / 补条件 / 转学习 / 挂起等待 / 回到重筹办 / 进入收束` 等去向候选，但这些候选不得越过外层重判直接冒充最终结论。
15. `任务管理任务` 可以补齐局部步骤结构、方法挂点、等待原因、学习候选和结果证据，但不能伪造叶子执行成功、外部结果成功或需求已满足结论。
16. `任务管理任务` 默认优先沿当前宿主任务树继续治理，不应每轮都对根任务和宿主任务从头重梳；只有当前已无可执行/可继续推进子任务，或系统刚重启/休眠恢复后，才允许重头梳理。
17. `任务管理任务` 可以依附 `任务管理主体` 做跨宿主、跨候选路径和跨学习反馈的成长汇总，但常规治理轮仍然只围绕一个宿主推进一步；主体级多分身不等于同宿主并发多实例。
18. 若需要做主体级的全局宿主筛选、压力重排或巡视治理，必须由主循环显式提交 `全局巡视治理请求` 或等价请求类型，不得在普通宿主治理轮中无边界盲扫。
19. `任务完成` 是任务管理主体的局部运行目标，`安全值 / 服务值` 仍然是上位方向约束。
20. 从自我线程一级模块层级看，真正的一级治理模块应称为 `任务管理中轴`。
21. `任务管理任务` 是 `任务管理中轴` 在本轮承接或生成的运行态治理实体。
22. `策略沙盘与方案演练` 不是和自我线程并列的一级模块，也不是 `任务管理任务` 本身，而是 `任务管理中轴 / 任务管理任务` 内部可调用的方案比较子模块。
23. 文档层统一称 `任务管理中轴`，运行态实体继续称 `任务管理任务`，运行态产物称 `一步治理任务包`；若代码里落类，统一建议称 `任务管理模块类`。
24. `任务管理任务` 当前可正式收成为：`一颗任务树 + 一个原语函数“推进一步” + 一份局部运行态 + 三类上行消息`。
25. `任务推进` 的最小解释单元，不是“为了做某个动作”，而是“为了让某个存在的某个特征进入某个目标状态”。
26. 因此：`任务` 是让某个关键特征达到目标状态；`步骤` 是让一个中间特征达到一个中间目标状态；`方法` 是把当前状态集推进到目标状态集的一种可复用动作路径；`派生需求` 是当前步骤要达成目标状态、但所需前置状态无法在本地满足时上报出去的缺口；`结果` 是执行后哪些特征真的达到了目标状态、哪些没达到，以及过程中出现了哪些关键中间状态。
27. `任务管理任务` 的最小工作面，可进一步压成：`找目标状态 -> 查前置状态 -> 选动作推进`。
28. `任务管理界面线程` 不是 `任务管理任务` 的推进者；它只负责读取 `任务管理任务界面信息`、`工作线程列表`、`消息汇总` 等观察快照，并把控制意图发回运行时承接层。
29. `任务管理任务` 的运行推进权只允许由 `任务管理实例/工作线程` 持有；界面线程不得直接改写宿主任务事实、步骤树、结果树或外层根裁决。

可收敛为一句话：

`任务管理任务的本质是“局部治理执行体”，它的实现方式是“最小原语集 + 受边界约束的局部增强”，它的终局目标是推动宿主任务状态进入完成。`

更细的 `根任务 / 宿主任务 / 完成树保留 / 结果回挂` 规则，按 [0975_根任务宿主任务与任务树保留复用规范.md](D:\鱼巢\规范\0975_根任务宿主任务与任务树保留复用规范.md) 执行。

### 3.1 架构定义与实现定义

当前应固定两层定义，不再混写成一句漂浮表述：

- 架构定义：`任务管理任务是某个任务的局部治理执行体。`
- 实现定义：`任务管理任务由一组最小原语组成，这组原语基于任务特征信息，对当前任务进行步骤推进、条件检查、局部等待处理、派生需求上报和结果汇总；并可在边界内通过组合、参数化、模板替换和经验沉淀，逐步完善任务管理能力。`

其中必须明确：

- 架构定义回答的是“它在系统里扮演什么角色”。
- 实现定义回答的是“它内部靠什么工作”。
- 以后凡出现“自行完善和增强”之类表述，默认只属于实现层，不得拿来改写架构边界。

### 3.3 允许增强与禁止越权

“逐步完善和增强”当前只能解释为：`在既有边界内提升局部推进能力`。

允许增强的方式，当前只限以下 4 类：

1. `组合增强`：同一组原语可因任务特征不同而改变组合顺序。
2. `参数增强`：阈值、超时、重试次数、等待窗口等可做局部适配。
3. `模板增强`：步骤模板、检查模板、结果汇总模板可逐步沉淀。
4. `经验增强`：可基于历史执行和因果样本提升诊断、分流与摘要质量。

但“增强”绝不能越过以下边界：

1. 不能改 `任务方向`、`完成标准`、`来源需求`。
2. 不能做 `根层判断`，包括 `服务/安全主导权`、`全局优先级`、`系统是否停止`。
3. 不能直接写 `正式需求树`。
4. 不能直接创建 `正式子任务`；只能上报派生需求或结构候选，由外层决定是否实例化。
5. 不能直接改 `服务值 / 安全值` 总账。
6. 不能无审计地自改 `原语集`；若以后允许扩充，必须同时具备 `来源`、`验证`、`回滚`、`审计`。

### 3.4 最小结构式

当前可把 `任务管理任务` 的最小结构正式收成：

`任务管理任务 = 一颗任务树 + 一个原语函数“推进一步” + 一份局部运行态 + 三类上行消息`

其中四部分分别回答：

- `任务树`：这个任务是什么、现在推进到哪一步。
- `推进一步`：本轮最小允许做哪一件推进动作。
- `局部运行态`：本轮前沿、等待、预算、缓存、依赖回流等驱动信息。
- `三类上行消息`：如何把局部事实规范化回传给自我线程。

必须进一步固定：

1. `任务树` 是主体，但不是全部运行时。
2. `局部运行态` 允许保留在任务管理任务自己的运行时承载里，不要求全部硬塞成树节点。
3. `推进一步` 是最小可调度原语；自我线程若要继续推进，批准的也应是“再推进一步”，而不是放任内部长时间自循环。
4. `线程状态消息 / 派生需求消息 / 结果消息` 是当前三类正式上行消息，不再额外膨胀新的一级上行大类。

### 3.5 任务推进最小原理

当前还应再固定一层最小原理：

`任务推进` 统一围绕“获取某个存在的某个特征的某个目标状态”来组织。

最小模板应理解为：

```text
目标存在 E
目标特征 F
目标状态 G

当前状态 C
前置状态集 P
可选动作集 A

若 P 满足：
    选择 A 中的一个或组合去推进
若 P 不满足：
    判断是局部等待，还是派生需求

执行后得到结果状态 R
再检查 F 是否已经达到 G
```

因此必须固定：

1. `动作` 是手段，不是步骤的第一定义。
2. `步骤` 应先解释成“目标特征状态缺口”，再解释成“为弥补缺口可选哪些动作路径”。
3. `派生需求` 的本质，是“当前目标状态所需前置状态无法在本地满足”的缺口上报。
4. `结果` 的本质，不只是“做过什么动作”，而是“目标特征状态是否真的成立，以及过程中出现了哪些关键中间状态”。
5. 每个步骤都必须先能回答两个问题：`想拿到哪个特征的哪个目标状态`，以及 `为了拿到它，需要哪些状态和动作`。

步骤节点的正式字段合同，当前按 [1311_任务步骤结构规范_v0.1.md](D:\鱼巢\规范\1311_任务步骤结构规范_v0.1.md) 执行；本文只负责冻结职责与边界，不单独再发明另一套步骤字段表。


## 4. 在统一主链中的位置

当前统一主链应固定为：

```text
自我线程主循环
-> 消费消息并刷新根层输入
-> 根层重判
-> 整理需求与双值
-> 形成任务管理上下文
-> 调用任务管理中轴
-> 任务管理任务推进宿主任务前进一步
-> 返回一步结果、治理证据与去向候选
-> 自我线程再次做根层重判与主链调度
```

因此必须固定：

1. `自我线程主循环` 是总协调者。
2. `任务管理任务` 是治理承接者。
3. `任务执行器` 是叶子执行承载者。
4. `学习子任务` 是缺口补齐承载者。
5. `外层根层重判` 才是最终是否继续本轮主链的上位约束输入。
6. `自我线程` 不直接展开任务树，只负责给不给这颗树继续 `推进一步` 的资格。
7. `任务管理界面线程` 只观察和发控制意图，不直接推进任务树。

一句话收口：

`线程管节拍，任务管理任务管步子。`


## 5. 最小职责表

### 5.1 最小职责规范表

### 5.2 当前阶段最少必须做到的职责

当前阶段若要承认 `任务管理任务` 已最小成立，至少必须能做到：

1. 明确接收 `宿主任务 + 根层允许边界 + 最小任务管理上下文`。
2. 明确给出 `当前功能域判定结果`。
3. 明确给出 `当前缺口类型` 与 `当前方法需求位`。
4. 明确给出 `当前下一步去向候选` 与 `当前本能能力缺口类型`。
5. 在需要时能推进以下任一治理动作：
   - 出生或切换一步结构
   - 绑定或切换一个方法挂点
   - 下发一次执行
   - 进入一个等待态
   - 生成一个学习子任务候选
6. 本轮结束后能至少规范化回传 `线程状态消息 / 派生需求消息 / 结果消息` 三类上行消息中的必要子集。
7. 本轮结束后能返回 `一步治理结果` 给外层主循环，而不是在内部继续连跑多轮闭环。

### 5.3 最小工作面

若继续向下压缩，`任务管理任务` 当前最少只需要稳定完成三件事：

1. `找目标状态`：明确当前步骤想让谁的哪个特征进入什么目标状态。
2. `查前置状态`：检查前序、资源、确认、方法可调用性、依赖结果和风险门控是否满足。
3. `选动作推进`：若前置状态满足，则选择动作路径推进；若不满足，则先判断应局部等待还是上报派生需求；若已执行后仍未达成，则再判断应本地继续推进，还是失败/收束/回退。

当前这三件事在结构上的正式承载，统一落在 [1311_任务步骤结构规范_v0.1.md](D:\鱼巢\规范\1311_任务步骤结构规范_v0.1.md) 定义的 `步骤标识 / 目标特征状态 / 前置状态集 / 动作集 / 达成判据 / 失败去向 / 结果写回锚点 / 一步治理返回映射` 八组字段上。

这三件事不是替代前面的职责表，而是它的最小压缩面。


## 6. 留在自我线程主循环的职责

以下职责必须留在 [0014_自我线程主循环最小职责表规范.md](D:\鱼巢\规范\0014_自我线程主循环最小职责表规范.md) 所定义的 `自我线程主循环`，不得下沉给 `任务管理任务`：

1. `根层重判`
2. `安全值 / 服务值` 统一结算
3. `需求整理、需求合并、需求权重调整`
4. `根任务与常驻治理任务存在维护`
5. `整个主链最终去向决定`
6. `正式需求树写入与正式子任务实例化审批`
7. `待机 / 停机边界处理`

原因是：

- 这些职责都直接涉及根层不变量或主链统一协调权。
- `任务管理任务` 只能消费这些结果，不能反向改写这些结果。


## 7. 不属于任务管理任务的职责

以下职责不应归到 `任务管理任务`：

1. 长时间叶子方法执行
2. 外设阻塞式等待
3. 外部世界原始事实采集线程本身
4. 根层安全/服务总裁决
5. 方法最终升格裁决
6. 学习反馈后的最终允许裁决
7. 需求是否满足的最终判断
8. 正式学习任务或补条件子任务的最终实例化审批

这些职责要么属于：

- `叶子执行器`
- `学习子任务`
- `自我线程主循环`
- `外层重判`


## 8. 与方法、学习、筹办成长的关系

必须固定：

1. `任务树结构成长` 主要靠 `任务筹办` 完成。
2. `任务管理任务` 负责触发和承接筹办，不负责把所有成长都压成执行器内部动作。
3. `方法头出生` 不要求当前就可执行；只要当前方法已能被结构化表达，就允许作为候选方法挂点进入管理链。
4. `方法可用 / 可执行 / 成熟度` 属于方法后续状态，不属于方法存在本身。
5. `学习` 主要负责补方法存在面、补可用性、补可靠性、补条件面；`任务管理任务` 负责在需要时把这种补齐诉求转成学习子任务候选。
6. `方法升格` 和 `学习反馈后的使用决定` 都不是任务管理任务可以单方定论的；任务管理任务只能提交候选、回收结果并把证据返回外层。
7. `方法需求位` 是成长需求与具体方法枝之间的正式桥层，不得跳过。
8. 当命中的 `方法需求位` 仍缺少最小动作原语时，`任务管理任务` 应显式返回 `本能能力缺口`，并允许把该缺口转成学习承接输入。
