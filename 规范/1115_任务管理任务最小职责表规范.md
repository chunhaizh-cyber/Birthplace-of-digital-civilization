# 1115 任务管理任务最小职责表规范

## 1. 目标

本文用于固定 `任务管理任务` 的最小职责。

当前唯一正式口径是：

- 标的：`任务`
- 目标：`使任务达到完成状态`
- 主逻辑：`目标 -> 筹办 -> 执行`


## 2. 核心结论

必须固定以下结论：

1. `任务管理任务` 仍然是任务，不是系统外的管理超体。
2. `任务管理任务` 的标的是 `宿主任务`。
3. `任务管理任务` 的目标是 `使宿主任务达到完成状态`。
4. `任务管理任务` 的筹办职责，是完善 `任务管理任务树` 和 `宿主任务树`。
5. `任务管理任务` 的执行职责，是执行当前治理步骤，并把治理结果写入结果节点。
6. `学习承接`、`方法补齐`、`结构补齐` 都只是服务于“宿主任务完成”的中间手段。
7. `任务管理任务` 不直接本地裁定“需求一定可满足”；它只形成需求并等待 `自我线程` 回执。
8. `可执行判断`、`允许执行判断` 不再是一级主阶段，只是统一称为 `执行入口闸门` 的筹办闸门。


## 3. 最小职责表

| 段 | 最小职责 |
| --- | --- |
| 目标 | 明确当前宿主任务和完成判据 |
| 筹办 | 识别当前缺口，形成筹办需求，提交给自我线程，并把回执返回的步骤、结果、方法、子任务挂回任务树 |
| 执行 | 消费已回执的当前治理步骤与筹办闸门，执行当前治理步骤，写回治理结果，判断宿主任务是否更接近完成 |


## 4. 任务管理任务的树

`任务管理任务` 自己必须显式长成一棵真实任务树：

```text
任务管理任务头
-> 治理步骤
-> 治理结果
-> 下一治理步骤
```

这棵树表达的是：

- 这一轮治理到底做了哪些治理步骤
- 每个治理步骤产出了什么结果

它不再退回为隐藏在代码里的“五模块流程说明”。


## 5. 筹办的特别职责

对 `任务管理任务` 来说，筹办必须额外负责：

1. 完善自己的治理步骤树。
2. 完善宿主任务树。
3. 把“少什么”统一收成筹办需求集合。
4. 把需求提交给自我线程，并等待“是否可满足 + 返回节点”的回执。
5. 当现有特征类型不足以表达当前治理缺口时，为宿主虚拟存在创建新的特征类型。

因此 `任务管理任务` 的成长，首先表现为：

- 会根据需求与虚拟存在特征继续生长新的治理特征类型
- 会根据这些特征继续完善任务树


## 6. 执行的特别职责

对 `任务管理任务` 来说，执行至少包括：

1. 执行当前治理步骤。
2. 消费筹办回给执行步骤的 `执行入口闸门`。
3. 推进宿主任务的当前步骤。
4. 当宿主任务缺方法时，转入学习任务筹办，而不是掉入断头路。


## 7. 禁止项

禁止以下做法：

1. 把 `任务管理任务` 的主流程写成 `四域`。
2. 把 `任务管理任务` 的主流程写成 `五模块`。
3. 把“学习承接”“反馈收口”升格成和目标、筹办、执行并列的一级主段。
4. 让 `任务管理任务` 继续本地硬编码决定“补条件 / 补学习 / 补结构一定能满足”。
