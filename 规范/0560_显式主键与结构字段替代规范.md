# 0560 显式主键与结构字段替代规范

## 1. 目标

本文用于统一回答以下问题：

- 是否可以在规范层彻底删除旧的压缩字段术语
- 删除旧术语后，哪些职责必须由 `主键 / 条件主键 / 结果节点 / 稳定ID / 显式字段组合` 接管
- 为什么不能把“删掉旧压缩字段术语”误解成“删除身份、索引、归桶、摘要能力”
- 新增规范和新增字段以后应如何命名

## 2. 可行性结论

在规范层删除旧的压缩字段术语，是可行的。

但必须同时固定三点：

1. 可以删除的是旧术语，不是它原先承载的 `身份定位 / 索引 / 归桶 / 候选聚类 / 摘要提示 / 稳定标识` 职责。
2. 不允许把这些职责简单偷换成 `名称字符串`。
3. 复合对象、复合条件、复合结果、复合路径，仍然需要 `显式主键`、`稳定ID` 或 `字段组合后的结构化主键`。

因此本文的最终结论不是：

- `以后什么键都不要`

而是：

- `以后不再使用旧压缩字段术语`
- `改用显式字段、显式主键、条件主键、结果节点主键、路径主键、稳定ID`


## 3. 统一定位

后续规范统一不再使用旧的压缩词作为领域术语。

统一改用以下概念：

- `主键`
  - 用于唯一定位一个实体、节点、对象或关系。
- `条件主键`
  - 用于定位一组已裁剪、可复用、可比较的输入条件。
- `结果主键`
  - 用于定位结果节点、结果模板或结果桶。
- `路径主键`
  - 用于定位步骤链、动态路径、稳定子树或组合动作序列。
- `稳定ID`
  - 用于跨线程、跨模块、跨快照、跨重启仍需稳定引用的对象。
- `摘要字段`
  - 用于展示、审计、恢复提示和候选检索，不充当最终语义真值。
- `显式字段组合`
  - 当单一字段不足以表达身份时，必须用 `动作键 + 条件主键 + 主结果特征类型 + 主结果方向/语义` 这类显式组合表达。

统一必须固定：

- `主键 / 条件主键 / 路径主键 / 稳定ID != 语义真值`
- `摘要字段 != 方向真值`
- `名称 != 身份`
- `显式字段组合` 只负责稳定定位、去重、归桶和候选组织，不直接替代比较和裁决


## 4. 与二次特征判定的关系

必须固定：

1. 一次特征、原始状态、原始返回值、原始主键、原始摘要字段，都只是真值输入面或辅助定位面。
2. 任何需要回答 `是否成立 / 是否满足 / 是否继续 / 是否同向` 的场景，都必须经过比较。
3. 比较后的产物应落成 `二次特征` 或等价的条件判定结果。
4. 因此：
   - `显式主键 / 条件主键 / 摘要字段` 只能帮助系统更快找到“值得比较的候选”
   - `二次特征` 才负责给出“最后是否成立”

统一主链应理解为：

`一次特征 / 状态 / 返回值 / 主键或摘要字段`
-> `候选检索与归桶`
-> `真实比较`
-> `二次特征`
-> `最终裁决`


## 5. 替代原则

### 5.1 能用现成主键就不用额外摘要字段

若当前已经有：

- `任务主键`
- `方法主键`
- `锚点主键`
- `条件节点主键`
- `结果节点主键`
- `学习任务ID`

则应优先直接使用这些主键，不再额外发明平行摘要字段或平行压缩字段。

### 5.2 能用显式字段组合就不用名称压缩

若某类身份可直接由以下字段稳定表达：

- `动作键`
- `动作句柄主键`
- `主结果特征类型`
- `主结果方向或主结果语义`
- `锚点主键`
- `来源任务主键`
- `来源方法主键`

则应优先使用显式字段组合，而不是先压成一个模糊缩写字段再回读。

### 5.3 名称只作解释，不作身份

必须固定：

- `名称` 可用于解释、展示、命名、人工阅读。
- `名称` 不能单独承担身份、唯一性、并头、归桶职责。
- 若只有名称，没有可回溯主键、证据链或结构字段组合，则该对象不得视为已稳定成立。

### 5.4 哈希只能降为实现细节

若当前实现为了性能仍需：

- 文本哈希
- 条件哈希
- 结果哈希
- 路径哈希

则它们只能作为：

- 私有缓存
- 热路径索引优化
- 非规范化的内部实现细节

统一不得：

- 进入规范主术语
- 作为跨模块暴露字段名
- 作为领域判定依据


## 6. 常见职责与新口径替代表

| 职责 | 新规范口径 | 优先替代字段 |
| --- | --- | --- |
| 需求身份与粗召回 | 需求身份与粗召回 | `需求主键`、`需求类型`、`需求主体主键`、`被需求状态主键` |
| 目标面定位 | 目标面定位 | `目标状态主键`、`目标特征类型列表`、`被需求状态主键` |
| 方法影响面定位 | 方法影响面定位 | `主结果特征类型`、`影响特征类型列表`、`方法作用域` |
| 条件归桶与条件定位 | 条件归桶与条件定位 | `输入条件主键`、`条件节点主键`、`条件状态列表` |
| 结果归桶与结果摘要 | 结果归桶与结果摘要 | `结果节点主键`、`最近实际结果节点`、`结果摘要原文`、`主结果特征类型 + 结果方向` |
| 过程面定位 | 过程面定位 | `动态稳定键`、`动作序列主键`、`过程动态列表` |

必须额外固定：

- 上表中的“替代字段”若单个不足，应采用显式字段组合。
- 不得再新建名字里带旧压缩词的字段、接口或规范条目。


## 7. 各域落地要求

### 7.1 需求与任务

需求和任务侧后续统一优先使用：

- `需求主键`
- `被需求状态主键`
- `目标特征类型`
- `任务主键`
- `最近实际结果节点`

这里不再保留“压缩值命中即成立”这类说法。

统一改为：

- `同需求主键`
- `同目标状态主键`
- `同目标特征类型`
- `同任务主键`

### 7.2 方法

方法系统后续统一优先使用：

- `动作键`
- `动作句柄主键`
- `输入条件主键`
- `主结果特征类型`
- `主结果方向或主结果语义`
- `方法首节点主键`

方法树唯一性口径统一表达为：

`动作键 + 输入条件主键 + 主结果特征类型 + 主结果方向或主结果语义`

不再表述为“压缩组合值”或“压缩结果值”。

### 7.3 因果

因果系统后续统一优先使用：

- `锚点主键`
- `来源任务主键`
- `来源方法主键`
- `条件状态列表`
- `过程动态列表`
- `结果状态列表`
- `条件主键`
- `结果主键`
- `稳定ID`

若当前实现内部仍保留哈希键，只能作为运行时缓存，不得上浮为规范主词。

### 7.4 学习与路径

学习对象和路径对象后续统一优先使用：

- `学习任务ID`
- `来源方法主键`
- `来源任务主键`
- `路径主键`
- `子树主键`
- `步骤序列`

不再表述为“对象压缩值”“路径压缩值”“子树结构压缩值”。

### 7.5 概念

概念系统后续统一优先使用：

- `概念主键`
- `概念域`
- `作用对象主键`
- `被作用对象主键`
- `关系类型`
- `适用条件主键`
- `概念锚点节点`

不再把“概念主键命中即概念成立”作为规范话语。


## 8. 禁止事项

以下做法统一视为违规：

1. 把 `名称` 直接当身份。
2. 把 `主键命中 / 条件主键命中 / 结果主键命中 / 摘要命中` 直接当最终判定。
3. 把 `同条件主键` 直接等价成“条件语义完全相同”。
4. 把 `同结果主键或同结果节点` 直接等价成“目标已经完成”。
5. 把 `同状态面类型 + 同锚点主键` 直接等价成“当前局面完全等价”。
6. 跳过一次特征比较，直接拿任意主键、稳定ID或摘要字段做 `继续 / 停止 / 成立 / 失败` 的最终裁决。

必须固定：

- `命中` 最多只能把对象推进到 `候选`
- 不得把 `候选` 直接冒充 `结论`


## 9. 当前阶段的迁移要求

后续规范与实现都应统一满足：

1. 新增规范禁止再引入旧压缩术语。
2. 新增字段、接口、特征类型命名禁止再使用旧压缩词。
3. 旧字段若暂时仍保留历史命名，只能视为兼容字段，规范解释必须按对应 `主键 / 条件主键 / 摘要字段 / 稳定ID` 口径理解。
4. 需要跨模块长期暴露的键，优先升级为显式主键或稳定ID，不再暴露哈希压缩值。
5. 需要恢复、审计、展示的字段，优先保留 `原文摘要` 或 `节点主键`，不再只保留压缩值。


## 10. 最终结论

规范层可以彻底去掉旧的压缩字段概念。

但必须同时保留并显式化以下能力：

- `主键定位`
- `条件主键归桶`
- `结果节点或结果主键引用`
- `路径主键定位`
- `稳定ID引用`
- `摘要字段提示`

一句话固定为：

`以后统一改为“显式主键、条件主键、结果节点和摘要字段负责定位候选，二次特征负责下结论”。`
