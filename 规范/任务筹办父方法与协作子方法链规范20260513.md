## 1. 总原则

2026-05-13 修订：

```text
任务筹办不再是本能方法，也不进入方法树候选集。
任务筹办改为任务管理生命周期普通函数，并保持单步循环执行。
每一轮任务筹办只根据当前任务状态、来源需求、候选方法和方法状态形成一个筹办过程回执。
```

核心原则：

```text
任务筹办是任务管理生命周期普通函数。
任务筹办的语义目标是让任务从初始状态逐步具备就绪条件。

任务筹办不独占所有中间动作。
凡是中间过程改变了任务、自我、方法、世界中的某个特征，
都必须由对应本能方法完成，并由该方法生成动作动态。
```

任务筹办本身只形成筹办过程回执、候选选择结果和缺口提示；任务状态变化仍由 `提交任务状态变化` 本能方法提交并生成任务状态动作动态。

任务筹办不能在自己内部递归执行子任务、学习链或执行链。它发现缺口后只返回等待 / 派生 / 重筹办提示；后续由自我线程生长需求、任务管理线程创建或推进子任务，再在下一轮筹办中读取已经回流的事实。

## 2. 任务筹办单步循环

### 2.1 任务管理生命周期普通函数_任务筹办

定位：

```text
单步筹办循环函数。
```

它统领：

```text
任务初始状态
-> 任务具备就绪条件
```

直接职责：

```text
1. 读取任务虚拟存在和任务状态。
2. 根据任务状态判断本轮是否需要筹办。
3. 读取任务来源需求。
4. 解析目标宿主、目标特征类型、当前状态、目标状态、满足关系。
5. 先填充任务方法候选集。
6. 再从候选集中选择当前方法。
7. 根据当前方法的方法状态、动作入口、条件结果对和参数绑定情况判断是否可就绪。
8. 如果当前方法不可用，形成 `方法状态 = 可用` 等缺口回执。
9. 如果当前方法可用且参数可绑定，形成可就绪回执。
10. 生成任务筹办运行存在或等价过程结果。
11. 在运行存在中写入筹办回执。
```

输出位置：

```text
运行存在：
    筹办回执
    筹办状态段轨迹
    当前方法候选
    筹办缺口类型
    派生需求提示
    已回流子任务结果引用
    已回流子方法结果引用
    建议任务状态

输出结果场景：
    一般不写任务状态。
    如果子方法产生世界事实，由子方法写入。

方法虚拟存在：
    不写。任务筹办不是方法，没有自己的方法虚拟存在。
```

任务筹办不生成方法动作动态。它可以形成这些过程回执事实：

```text
筹办运行事实已形成
筹办回执已生成
候选方法已确定
筹办缺口已形成
可就绪依据已形成
```

这些事实只是任务管理过程回执，不是方法动作动态。若要把候选方法集、当前方法、可执行输入参数场景等写入任务虚拟存在，必须由 `同步筹办回执到任务` 完成。

任务筹办不直接生成：

```text
任务状态：未启动 -> 就绪
```

该任务状态动作动态来源必须是：

```text
本能方法_提交任务状态变化
```

### 3.2 本能方法_提交任务状态变化

职责：

```text

```




## 4. 需求树协作方法

任务筹办无法一步就绪时，不是失败，而是需求树继续细化。
这里的“一步”指当前一轮任务筹办函数调用。
任务筹办本轮只返回缺口和派生提示，不在本函数内继续调用学习方法、创建子任务或执行子任务。

### 6.2 本能方法_选择当前方法

职责：


输入：

```text

```


动作动态：

```text

主体 = 任务筹办运行存在 / 任务虚拟存在
特征 = 当前方法
事实 = 当前方法已确定
```

如果当前方法最终写入任务虚拟存在，应由 `同步筹办回执到任务` 完成。

选择当前方法必须采用分层排序，不得把“已经具备条件”和“将来可能具备条件”混为同一优先级。

第一优先级是直接满足全部条件的方法：

```text
方法状态可用；
有执行入口；
有条件结果对；
必需条件全部满足；
参数全部可绑定；
禁止项未命中；
结果能力能确定性命中来源需求目标。
```

如果存在多个直接满足全部条件的方法，继续按以下顺序排序：

```text
1. 禁止项少者优先；
2. 风险约束少者优先；
3. 前置冲突少者优先；
4. 预计效率高者优先；
5. 预计耗时少者优先；
6. 能耗和资源占用少者优先；
7. 副作用少者优先；
8. 最近成功率和稳定性高者优先。
```

只有当不存在直接满足全部条件的方法时，才进入过程型 / 可补齐候选：

```text
缺少条件但有明确补齐路径；
缺少参数但可由已知方法获得；
方法不可用但已有学习链可推进；
禁止项可通过子需求解除；
需要任务筹办继续拆解缺口。
```

再后才进入不确定候选：

```text
可能可学习；
可能可探索；
证据不足但有候选线索；
需要外界反馈或观察。
```

因此，`任务筹办` 不声明方法结果能力，也不作为候选方法参与选择。

```text
任务筹办是任务管理生命周期普通函数；
它负责组织候选填充、选择、缺口识别和回执形成；
它每轮只执行一次筹办判断；
真正能改变 方法状态 / 条件结果对 / 参数 / 世界特征 的，必须是对应方法。
```

对于 `方法状态 = 可用` 这类目标，候选集不应写死为某一个学习方法，而应先包含所有结果能力相关候选，再按当前目标方法的缺口阶段选择：

```text
条件结果关系未知 / 未证实
    -> 选择 尝试学习；

已有条件结果证据但未入账
    -> 选择 建立条件结果对；

条件结果对已入账且达到可用判据
    -> 选择 提交方法状态变化；

已可用但稳定性 / 好用度不足
    -> 选择 练习；

缺口尚未能直接提交
    -> 由任务筹办普通函数继续拆解子需求，不把任务筹办当候选方法。
```

## 11. 推荐执行链

任务筹办从初始到就绪：

```text
任务筹办
-> 查找候选方法
-> 选择当前方法
-> 检查方法条件是否满足
-> 配对参数
-> 同步筹办回执到任务
-> 提交任务状态变化
-> 任务状态进入就绪
```

缺方法：

```text
任务筹办
-> 派生需求入树
-> 登记任务发起意图
-> 承接需求创建任务
-> 子任务筹办 / 子任务执行
-> 回写需求满足
-> 归并子结果到父任务筹办
-> 检查父任务是否可重筹办
-> 唤醒任务重筹办
-> 任务筹办再次运行
```

方法不可用：

```text
任务筹办
-> 派生需求入树：方法状态 = 可用
-> 定向学习 / 尝试学习 / 练习
-> 建立条件结果对
-> 提交方法状态变化
-> 归并子结果到父任务筹办
-> 唤醒任务重筹办
-> 任务筹办再次运行
```

参数缺失：

```text
任务筹办
-> 派生需求入树：缺失参数已获得 / 已绑定
-> 子任务处理参数
-> 配对参数
-> 归并子结果到父任务筹办
-> 唤醒任务重筹办
-> 任务筹办再次运行
```

## 12. 最终定稿

任务筹办要完成“初始状态 -> 就绪状态”这一整段，需要一组协作方法。

```text
任务筹办负责单步读取任务状态、填充候选、选择当前方法、识别缺口并生成筹办回执；
同步筹办回执到任务负责写任务筹办事实；
提交任务状态变化负责真正推进任务状态；
派生需求入树负责让缺口变成更细目标；
学习类方法负责让方法从不会变会；
归并子结果负责把子任务、子方法结果纳入父筹办结果；
唤醒重筹办负责让父任务继续前进。
```

一句话：

```text
任务筹办是任务管理生命周期中的单步筹办循环；
每轮只产出本轮回执，缺口通过需求树和子任务在后续轮次回流；
每个中间特征变化都必须交给对应本能方法或动作骨架入口落账。
```
