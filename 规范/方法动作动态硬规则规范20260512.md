# 方法动作动态硬规则规范 20260512

## 1. 权威结论

凡是能生成“方法动作动态”的变化，都必须对应一个明确动作。

```text
动作 = 方法。
动作函数 = 本能函数或动作骨架入口。
```

方法动作动态不是普通日志。它表示系统承认：

```text
某个方法在某个场景下，
读取输入参数场景，
对某个主体的某个特征执行了动作，
并产生了可记录的运行事实。
```

因此：

```text
方法动作动态存在
    -> 系统承认某方法执行了某动作
    -> 自我具备这个动作能力
    -> 该动作原则上可由自我有意识调用
```

## 2. 成功 / 失败不是根概念

当前系统不以“成功 / 失败”作为动作动态的根概念。

动作动态记录的是：

```text
方法运行造成的事实动态。
```

不是：

```text
成功日志
失败日志
```

只要某个方法真实运行，并造成了某种可记录事实，就应该生成动作动态。除非逻辑崩溃、运行中断到连动态都无法生成，否则不应因为“没有达到最终目标”而放弃记录。

`成功 / 失败` 只能作为普通结果特征、兼容字段或派生判断使用，例如：

```text
目标达成情况
筹办结果类型
执行结果类型
需求满足情况
中断原因
```

不得把 `成功 / 失败` 当成动作动态本体的分类轴。

建议后续将动作动态核心结果改为：

```cpp
enum class 枚举_动作事实类型 : std::uint8_t {
    未定义 = 0,

    特征已改变,
    状态已推进,
    需求已派生,
    任务已创建,
    条件已观测,
    回执已生成,
    方法已运行,
    事实已提交,
    运行中断
};
```

如果底层签名暂时仍保留：

```cpp
bool 成功
```

则只能解释为：

```text
动作动态是否已生成 / 方法运行事实是否成立
```

不能解释为：

```text
父目标是否达成
```

## 3. 外部变化边界

只有“方法动作动态”表示自我能做。

外部发生的变化不能生成自我方法动作动态，只能落成：

```text
观察动态
外部事件动态
环境动态
未知来源动态
```

如果没有对应方法，不能叫方法动作动态。

## 4. 可有意识调用

“可有意识调用”不是无条件调用。

它表示自我线程或任务管理线程能够基于需求、目标、缺口、调度意图，构造输入参数场景，然后调用某个方法。

一个动作至少需要：

```text
1. 方法头。
2. 动作函数 / 本能函数入口。
3. 输入形参规格。
4. 条件节点。
5. 结果规格 / 回执规格。
6. 可构造输入参数场景。
7. 调用后能返回运行存在 / 输出结果场景。
8. 事实提交后能生成方法动作动态。
```

当前是否允许调用，还要受这些条件限制：

```text
条件节点是否满足。
参数是否可绑定。
权限是否允许。
所属线程是否有写入权。
目标对象是否存在。
状态迁移是否合法。
```

## 5. 线程不是动作来源

线程只能作为调用者或编排者，不能作为方法动作动态的动作来源。

禁止表达：

```text
任务管理线程把任务改成完成。
自我线程把需求改成满足。
学习线程把方法改成可用。
```

正确表达：

```text
任务管理线程调用 本能方法_提交任务状态变化。
自我线程调用 本能方法_回写需求满足。
学习 / 方法管理流程调用 本能方法_提交方法状态变化。
```

方法才是动作来源，动作动态记录的是方法动作事实。

## 6. 提交类逻辑必须方法化

凡是会落成方法动作动态的提交类变化，都必须有对应本能方法或动作骨架入口。

第一批必需动作来源：

```text
本能方法_承接需求创建任务
本能方法_提交任务状态变化
本能方法_同步筹办回执到任务
本能方法_同步执行回执到任务
本能方法_回写需求满足
本能方法_派生需求入树
本能方法_提交方法状态变化
本能方法_建立条件结果对
本能方法_改变特征值
```

线程可以决定调用这些方法，但不能绕过方法直接生成方法动作动态。

## 7. 任务完成动作动态

任务完成动作动态的动作来源是：

```text
本能方法_提交任务状态变化
```

标准链路：

```text
任务执行普通函数
    -> 运行回执：
        建议任务状态 = 完成
        目标是否达成 = true

任务管理线程
    -> 调用 本能方法_提交任务状态变化

本能方法_提交任务状态变化
    -> 写任务状态：
        执行中 -> 完成
    -> 生成任务状态方法动作动态
```

任务完成动作动态的动作来源不是任务管理线程，也不直接是被执行的业务方法。

业务方法负责世界事实动作动态；任务状态提交方法负责任务状态动作动态。

## 8. 需求满足动作动态

需求满足动作动态的动作来源是：

```text
本能方法_回写需求满足
```

标准链路：

```text
任务完成动作动态
    -> 自我线程收到任务完成回执
    -> 自我线程判断来源需求满足
    -> 调用 本能方法_回写需求满足
    -> 需求状态 / 被需求当前状态发生变化
    -> 生成需求状态方法动作动态
```

自我线程只是决定调用该方法，不是动作来源。

## 9. 动作动态链

不新增独立因果记录结构。方法动作动态之间通过来源特征串成因果链。

标准链：

```text
D0：世界事实动作动态
    方法 = 改变特征值 / 具体执行方法
    主体 = 世界存在
    特征 = 目标特征
    旧值 -> 新值

D1：任务状态动作动态
    方法 = 提交任务状态变化
    主体 = 任务
    特征 = 任务状态
    执行中 -> 完成
    来源动作动态 = D0

D2：需求状态动作动态
    方法 = 回写需求满足
    主体 = 需求
    特征 = 需求状态 / 被需求当前状态
    未满足 -> 已满足
    来源动作动态 = D1
    满足证据动态 = D0
```

## 10. 禁止项

```text
1. 没有方法来源时生成方法动作动态。
2. 线程直接作为方法动作动态来源。
3. 外部变化伪装成自我方法动作动态。
4. 任务状态、需求状态、方法状态变化绕过提交类本能方法。
5. 把观察动态 / 外部事件动态当成自我动作能力证据。
6. 把“可有意识调用”理解成无条件乱调。
7. 把“未达成父目标”直接记成动作动态失败。
8. 用成功 / 失败替代事实类型、相位、缺口、等待、派生、细化等真实事实。
```

## 11. 最终定稿

```text
动作动态是“自我可调用动作”的事实痕迹。
凡能落成方法动作动态的变化，都必须有一个对应的本能方法或动作骨架入口作为动作来源。
动作动态不是胜负判定，而是方法运行造成的事实动态。
```
