# 0570 二次特征信息生成与用途和使用方法规范

## 1. 目标

本文用于固定以下问题：

- `二次特征信息` 从哪里生成
- `二次特征` 在系统里到底属于什么层
- `二次特征` 能不能成为任务或方法的目标
- 哪些二次特征适合当目标，哪些不适合
- “二次特征本体”与“任务/方法对它的使用角色”应如何分开
- 若任务或方法要以二次特征为目标，最小应如何绑定

## 3. 核心结论

必须固定以下结论：

1. `二次特征` 不是备注，也不是纯显示标签；它是正式真值层中的结构化关系结果。
2. `二次特征` 可以成为某些任务或方法的目标。
3. 但“是否成为目标”不是二次特征本体的固有属性，而是任务/方法上下文中的使用角色。
4. 不是所有二次特征都适合直接当目标；只有那些表达 `关系状态` 或 `过程性质`，并且可被动作真正改变、维持或消除的二次特征，才适合当目标。
5. `特征类` 继续负责不同特征类型的比较方式、稳态、误差与值域处理。
6. `动态类` 继续负责起止状态、时间、路径与过程事实。
7. `二次特征类` 只负责在这些上游结果之上生成二次语义，并承接其使用。

一句话收口：

`二次特征可以成为目标，但“目标”属于任务/方法对它的使用角色，不属于二次特征本体的固有属性。`


## 4. 当前结构边界

从当前结构看，边界已经基本成立：

1. `特征节点主信息类`
   已承载 `当前值 / 区间 / 稳态列表 / 连续测量误差区间`。

2. `动态节点主信息类`
   已承载 `初始状态 / 结果状态 / 动态特征 / 时间 / 动态路径签名`。

3. `二次特征主信息类`
   当前主要承载：
   - `形态`
   - `种类`
   - `生成母式字段组`
   - `度量签名_链键`
   - `概念名称`
   - `主体 / 客体 / 左对象 / 右对象`
   - `标量值`
   - `是否满足 / 置信度`

4. `场景节点主信息类`
   已把二次特征作为 `关系索引` 保存；后续推荐新名为 `二次特征索引`。

5. `存在节点主信息类`
   则单独挂有 `需求根节点 / 任务根节点 / 方法根节点`。

6. `因果模板主信息类`
   已把二次特征放进 `条件模板` 与 `结果模板`。

7. `因果实例主信息类`
   已保留 `来源任务主键` 与 `来源方法主键`。

因此必须固定：

- 二次特征已经进入 `任务 / 方法 -> 因果模板 -> 结果` 这条链。
- 但二次特征更像“世界中的关系事实”与“判定层中的结构结果”。
- `目标` 仍应属于控制层、任务层、方法层的角色，不应直接焊回二次特征本体层。
- `来源状态 / 来源动态 / 来源场景 / 来源层级` 应被视为二次特征的合法来源链，不得长期停留在口头层。


## 5. 二次特征的生成主链

推荐固定主链为：

```text
特征比较
-> 状态
-> 动态
-> 生成二次特征
-> 写入场景关系索引 / 二次特征索引 / 存在引用 / 条件模板 / 结果模板 / 判定层
-> 被任务、方法、因果、学习继续消费
```

必须固定：

1. 二次特征只能建立在上游真值之上。
2. 二次特征不得私自复制一份上游比较核。
3. 二次特征只保存“比较或归纳之后的关系结果”，以及“它怎么来的”的显式生成语义。
4. 若需要追更深的历史路径，应优先回到 `动态类` 的时间与路径层，不要把二次特征重新做成万能抽屉。

### 5.1 阈值硬规则

必须固定：

- 没过阈值，连动态都不生成。
- 未形成合法动态，就不得生成 `动态型二次特征`。
- 不允许后续模块通过“微差累计补算”的方式，在高层偷偷生成本应不存在的动态型二次特征。

一句话收口：

`动态型二次特征只能建立在已经成立的动态之上。`

### 5.2 非相邻规则

必须固定：

`非相邻先降动态，再聚合生成多级动态，再在同级相邻上继续获取二次特征。`

这条规则是当前唯一合法路径。

因此必须继续固定：

1. 不允许对非相邻状态直接做首尾差分，就宣称得到了完整动态型二次特征。
2. 若确实需要跨更长过程取二次特征，必须先生成：
   - 原子动态
   - 聚合动态
   - 同级相邻动态对
3. 路径语义、回返语义、闭环语义应优先建立在聚合动态之上，而不是越过动态层直接从裸状态序列硬算。


## 6. 二次特征的三条正交轴

二次特征的语义，推荐至少拆成三条正交轴。

### 6.1 第一条轴：所属域

这条轴回答：

- 这条二次特征主要描述的是哪类上游对象

推荐口径：

```cpp
enum class 枚举_二次特征所属域 : std::uint8_t {
    未定义 = 0,
    存在型 = 1,
    动态型 = 2,
};
```

解释：

- `存在型二次特征` 更偏静态关系、占用关系、距离关系、包含关系、冲突关系等。
- `动态型二次特征` 更偏恢复速度、衰减速度、波动程度、压缩程度、扩散趋势、闭环路径等。

### 6.2 第二条轴：语义种类

这条轴回答：

- 这条二次特征在语义上属于什么关系或什么过程性质

例如：

- `冲突`
- `接近`
- `包含`
- `占用`
- `恢复`
- `扩散`
- `波动`
- `压缩`

必须固定：

- 这条轴属于“它是什么”。
- 不负责回答“它当前在任务里是条件还是目标”。

### 6.3 第三条轴：任务/方法使用角色

这条轴回答：

- 任务或方法当前把这条二次特征当成什么来用

推荐角色固定为：

- `观测`
- `条件`
- `结果`
- `目标`
- `评估依据`

必须固定：

- 这条轴不属于二次特征本体。
- 这条轴属于任务/方法上下文。
- 同一条二次特征概念，今天可以被当条件，明天可以被当结果，后天也可以被当目标。

一句话收口：

`所属域`、`语义种类`、`任务角色` 三条轴必须分开，不得混成一个字段。


## 7. 二次特征能否作为目标

### 7.1 可以成为目标

必须固定：

`二次特征可以成为某些任务或方法的目标。`

原因是：

1. 二次特征已经正式进入条件模板和结果模板。
2. 它已经能表达“系统真正关心的关系状态或过程性质”。
3. 很多任务与方法的真实目的，并不是“把某个一次值改成某个数”，而是“让某种关系成立/消失/减弱/增强”。

典型例子包括：

- `消除占用冲突`
- `降低保障时间压缩程度`
- `提高恢复速度`
- `抑制波动`
- `维持接近关系`
- `维持包含关系`
- `维持占用关系`

### 7.2 不是所有二次特征都适合直接当目标

必须固定：

以下类型更适合直接成为目标：

1. `关系状态型`
   例如冲突、接近、包含、占用、偏离、对齐。

2. `过程性质型`
   例如恢复速度、衰减速度、波动程度、压缩程度、扩散趋势。

以下类型通常不应直接成为目标：

1. 纯审计型标签
2. 纯来源型标记
3. 纯链路缓存或纯索引键
4. 缺少可度量落点的黑箱摘要
5. 不能被动作真实影响、维持或消除的临时中间量

### 7.3 可目标化判定

一个二次特征要想成为任务或方法目标，至少应满足：

1. 能明确回答“希望它达到/维持/消除/降低/提高什么”。
2. 能被某类方法真实作用。
3. 作用后能重新观测与验证。
4. 能落成可比较区间、可比较符号或可比较类别。
5. 不违反根约束与安全边界。


## 8. 目标角色不应写死在二次特征本体中

必须固定：

- `目标` 是任务/方法上下文中的使用角色。
- `目标` 不应做成 `二次特征主信息类` 里的固有常驻属性。

原因是：

1. 二次特征作为世界事实，首先应挂在场景或存在的关系面中。
2. 任务和方法则属于控制层、推进层、治理层。
3. 若把“目标”焊成二次特征本体属性，会把本体层和控制层拧死。
4. 同一条二次特征可能在不同上下文中切换角色，若写死为“目标”，后续会发硬。

因此必须固定：

- `二次特征侧` 只负责回答“它是什么”。
- `任务/方法侧` 负责回答“当前把它当什么用”。

### 8.1 过渡方案

若当前工程确实缺少任务/方法层的正式承载槽位，可以临时把 `二次特征主信息类::形态` 扩成包含 `目标`。

但必须明确：

- 这只属于过渡方案。
- 不应视为长期真值结构。
- 一旦任务目标绑定结构落成，应把“目标角色”迁回任务/方法层。


## 9. 推荐的目标绑定结构

推荐把“二次特征作为目标”的正式承载放在任务/方法层，最小结构可固定为：

```cpp
enum class 枚举_目标对象类型 : std::uint8_t {
    未定义 = 0,
    状态 = 1,
    动态 = 2,
    二次特征 = 3,
};

enum class 枚举_目标作用 : std::uint8_t {
    未定义 = 0,
    达到 = 1,
    维持 = 2,
    消除 = 3,
    降低 = 4,
    提高 = 5,
};

struct 任务目标绑定 {
    枚举_目标对象类型 目标对象类型 = 枚举_目标对象类型::未定义;
    可解析引用<基础信息节点类> 目标对象{};
    枚举_目标作用 目标作用 = 枚举_目标作用::未定义;
    std::optional<I64区间> 目标区间{};
    bool 硬约束 = false;
    std::int32_t 优先级 = 0;
};
```

必须固定：

1. `目标对象类型` 负责说明当前目标是状态、动态还是二次特征。
2. `目标对象` 负责回指具体目标节点。
3. `目标作用` 负责说明是达到、维持、消除、降低还是提高。
4. `目标区间` 负责承接可比较目标范围。
5. `硬约束` 负责说明这是必须满足还是可退化权衡。
6. `优先级` 负责说明在多目标并存时的先后与权重。

### 9.1 方法侧复用

方法若需要直接以二次特征为目标，可复用同一结构，或在方法侧使用同构结构。

推荐理解为：

- `任务目标绑定` 更偏治理目标与推进目标
- `方法目标绑定` 更偏期望结果与执行意图

但二者的语义骨架应尽量一致。


## 10. 二次特征的标准用途

二次特征当前至少有五种正式用途。

### 10.1 观测用途

回答：

- 当前世界里出现了什么关系状态或过程性质

典型承接：

- 场景的 `关系索引`
- 场景的 `二次特征索引`
- 存在上的局部引用
- 控制面板的关系显示

### 10.2 条件用途

回答：

- 方法或任务当前是否满足进入条件

典型承接：

- 因果模板的 `条件模板`
- 方法可执行条件
- 外层门控输入

### 10.3 结果用途

回答：

- 某次动作、方法或任务之后形成了什么关系结果或过程结果

典型承接：

- 因果模板的 `结果模板`
- 因果实例的结果桥接
- 回流学习的结果摘要

### 10.4 目标用途

回答：

- 当前任务或方法希望把哪个关系状态、过程性质推进到什么方向

典型承接：

- 冲突消除
- 波动抑制
- 恢复速度提高
- 压缩程度降低
- 占用关系维持

### 10.5 评估依据用途

回答：

- 当前某条二次特征是否可被用作方法评分、规避过滤、执行后验收或长期策略评估的依据

典型承接：

- 方法评分项
- 方法规避过滤
- 执行后结果验收
- 长期策略回填


## 11. 任务与方法如何使用二次特征

### 11.1 任务侧使用顺序

推荐顺序固定为：

```text
读取当前状态 / 动态 / 二次特征
-> 识别当前关键二次特征
-> 判断哪些二次特征适合转成任务目标
-> 为目标二次特征创建任务目标绑定
-> 按目标绑定派生任务或调整已有任务
```

### 11.2 方法侧使用顺序

推荐顺序固定为：

```text
读取方法条件模板
-> 读取当前二次特征
-> 判断是否命中条件
-> 选择方法
-> 执行后重新生成二次特征
-> 与期望目标或结果模板比较
```

### 11.3 执行后的验收方式

若任务或方法以二次特征为目标，执行后至少要重做以下事情：

1. 重新生成目标相关二次特征。
2. 判断其是否落入目标区间或目标类别。
3. 判断它是被达到、维持、消除、降低还是提高。
4. 若未命中，则回写为失败证据、偏差证据或待学习缺口。


## 12. 与因果模板的关系

必须固定：

1. 二次特征可以进入 `条件模板`。
2. 二次特征可以进入 `结果模板`。
3. 若任务或方法以二次特征为目标，则该目标最终仍应通过：
   - 结果二次特征
   - 条件/结果模板命中
   - 实例因果或抽象因果支撑
   来验证是否成立。

但必须继续固定：

- `来源任务主键 / 来源方法主键` 只是来源追溯字段，不自动等于目标语义。
- “某次因果来自哪个任务/方法”与“这个二次特征当前是不是目标”不是同一回事。


## 13. 不允许的做法

以下做法必须禁止：

1. 把所有二次特征一律视为可目标化。
2. 把 `目标` 永久焊死进 `二次特征主信息类`，让它变成本体常驻属性。
3. 把 `关系索引` 与 `任务目标绑定` 混成同一个字段。
4. 把 `来源任务主键 / 来源方法主键` 误当成“当前目标角色”的证明。
5. 只靠 `度量签名_链键` 或自然语言摘要判断目标是否成立。
6. 用黑箱标签类二次特征直接驱动任务，而不给出可验证的作用区间或方向。


## 14. 一句话总括

可收敛为一句话：

`二次特征是从特征、状态、动态之上生成出的正式结构化关系结果；它既可以被当作观测、条件、结果、评估依据，也可以在合适情况下成为任务或方法的目标，但“目标”必须作为任务/方法层对二次特征的使用角色来承载，而不应被做成二次特征本体的固有属性。`
