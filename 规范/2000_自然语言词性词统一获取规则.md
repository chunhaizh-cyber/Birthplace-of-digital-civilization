# 2000 自然语言词性词统一获取规则

## 1. 目标

本文用于固定当前工程里 `自然语言名称 / 动作名称 / 描述性短语` 的统一获取方式，避免以下混乱：

- 同一自然语言名称在不同模块里走不同创建路径
- 为了“提速”临时再造一层独立缓冲区，导致词性词指针来源不一致
- 动态记录、因果记录、窗口显示各自生成一套名字，最后无法对齐


## 2. 统一规则

自本规范生效起，凡是工程内 **需要新增、补全、生成、映射自然语言名称** 的地方，统一遵守：

1. 必须通过 `语素集.添加词性词(...)` 获取 `词性词指针`。
2. 不允许绕开 `语素集` 自己维护第二套“自然语言词性词对象池”。
3. 不允许为了局部显示或局部动态记录，直接 new 一个独立词对象替代 `语素集` 返回结果。
4. 若需要一个自然语言名称，应在“真正需要落记录、落状态、落显示”的当下按需获取，不要求预先全部生成。

一句话固定：

`自然语言名称的统一入口，就是 语素集.添加词性词(...)。`


## 3. 为什么当前不建立额外缓冲区

当前默认不建立额外的“自然语言词性词缓冲区”，原因固定如下：

1. `语素集.添加词性词(...)` 本身已经承担统一入口职责。
2. `语素类` 当前实现已包含“先查后加”的复用语义，而不是无条件重复创建。
3. 如果再在外面包一层通用缓冲区，会引入第二套命中规则、第二套一致性边界、第二套失效策略。
4. 这类额外复杂度，只有在后续确实出现性能瓶颈时才值得引入。

因此当前结论是：

- `先不建立独立缓冲区`
- `先统一入口`
- `后续若要优化，优先优化语素类内部或补一个极小范围的专用显示名缓存`


## 4. 适用范围

本规则适用于以下所有需要自然语言名的场景：

- `动作动态` 的来源动作名
- `实例因果` 的动作语义名
- `状态摘要` 中的自然语言标签
- `任务 / 方法 / 需求 / 场景 / 存在` 的运行时补充名称
- 任何窗口显示、调试输出、运行时镜像中需要补出的可读词名


## 5. 动作动态的专门约束

对 `自我动作动态`、`方法内部动作动态`、`任务内部动作动态`，额外固定：

1. 动态记录时若需要“更完整、更自然语言化”的动作名称，应在记录当下补全。
2. 补全后的名字，仍必须通过 `语素集.添加词性词(...)` 获取词性词指针。
3. 不允许把补全后的名字只当普通字符串保存而绕过语素集。
4. 不允许先全量预生成一批动作自然语言名称，再在运行期硬编码分发。

也就是说：

- 允许 `按需补全`
- 不允许 `绕开语素集`
- 不要求 `提前全量生成`


## 6. 性能优化边界

如果后续真的出现自然语言名称生成成为热点，优化顺序固定如下：

1. 先确认热点是否真实存在，而不是凭感觉预优化。
2. 优先优化 `语素类` 内部查找/索引。
3. 若仍不足，再考虑增加“小范围专用缓存”。
4. 即使增加缓存，也只能缓存 `语素集` 返回的词性词指针，不得形成第二套词对象来源。

当前明确不建议：

- 建立独立于 `语素集` 的通用自然语言缓冲区
- 为每个模块各自维护一份自然语言词对象表
- 在动态记录层直接持有裸字符串并跳过词性词指针
