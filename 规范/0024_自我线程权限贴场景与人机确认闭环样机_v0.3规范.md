## 3. v0.3 的定位与硬边界

### 3.1 v0.3 的本质

必须固定：

`v0.3` 的本质不是再扩场景，而是把权限和确认真正挂到既有骨架上。

它当前主要验证：

- 局部执行链能否区分 `可见 / 可建议 / 可确认 / 可改写`
- 高风险动作在未确认前能否真正停在闸门前
- 拒绝、撤销、超时是否都能正式结算
- 安全接管是否能覆盖尚未执行的确认链

### 3.2 v0.3 不做什么

本版明确不做：

- 复杂策略引擎
- 第四个全局确认包
- 多层审批流
- 复杂组织角色树
- 自动跳过确认直接执行

### 3.3 一句硬边界

必须固定：

`能看见不等于能处置，能处置不等于能改写。`


## 4. 当前只做六个工程工作项

### 4.1 角色-对象-动作权限矩阵

先只做最小矩阵，不做花哨策略引擎。

固定维度：

- 作用层级：`对象 / 属性 / 关系 / 动作`
- 权限级别：`可见 / 可建议 / 可确认 / 可改写`

### 4.2 确认闸门

所有可能改变外部世界、关键状态、关键关系的动作，都不得从 `建议` 直接滑进 `执行`。

确认闸门最少要回答：

- 谁可以发起
- 谁必须确认
- 是否单签或双签
- 超时怎么办
- 被拒绝后怎么结算
- 被安全接管后怎么撤销

### 4.3 不新增第四个大包，只给三包补确认字段

必须固定：

`v0.3` 不新增第四个全局确认包，只在三包中追加确认与权限字段。

### 4.4 人机确认状态机

把确认过程单独立成状态机，不和执行状态混写。

### 4.5 审计与责任回写

每一次动作至少留下四个责任钩子：

- 谁发起
- 谁确认
- 谁执行
- 谁回写

### 4.6 失败路径测试先于成功路径扩展

`v0.3` 当前主角不是“建议多聪明”，而是：

`没确认就绝不越线。`


## 5. 角色-对象-动作权限矩阵

### 5.1 最小矩阵维度

当前最小矩阵固定为：

| 作用层级 | 说明 |
| --- | --- |
| `对象` | 能否看见某个对象、能否对对象发起建议或确认 |
| `属性` | 能否读取或建议修改某个状态/值 |
| `关系` | 能否查看或建议变更关键关系 |
| `动作` | 能否发起、确认、执行某种动作 |

| 权限级别 | 说明 |
| --- | --- |
| `可见` | 允许读取与理解 |
| `可建议` | 允许提出建议，但不能落地 |
| `可确认` | 允许批准或拒绝待执行动作 |
| `可改写` | 允许在确认完成后执行正式改写 |

### 5.2 最小角色面

当前样机最少只需要三类角色：

1. `观察/建议角色`
2. `确认角色`
3. `执行/回写角色`

必须固定：

- 同一角色可以兼任多个身份
- 但权限级别必须显式写清
- 默认不允许隐式越权


## 6. 确认闸门

### 6.1 硬纪律

必须固定：

1. 任一高风险动作，没有确认不得落地。
2. 任一确认流程被拒绝、撤销或超时，都必须生成结算包。
3. 安全接管一旦触发，可以覆盖尚未执行的确认链。

### 6.2 最小确认规则

确认闸门当前只需处理：

- `单签`
- `双签`
- `超时失效`
- `安全接管撤销`

### 6.3 最小确认结果

确认结论最少只有：

- `已确认`
- `已拒绝`
- `已撤销`
- `已超时`

## 8. 人机确认状态机

确认过程最小状态固定为：

- `待确认`
- `已确认`
- `已拒绝`
- `已撤销`
- `已超时`
- `已执行`
- `已结算`

### 8.1 最小转移

```text
待确认
-> 已确认
-> 已执行
-> 已结算

待确认
-> 已拒绝
-> 已结算

待确认
-> 已超时
-> 已结算

待确认 / 已确认
-> 已撤销
-> 已结算
```

### 8.2 状态机硬规则

必须固定：

1. `待确认` 之前不得进入正式执行。
2. `已拒绝 / 已撤销 / 已超时` 都必须走正式结算。
3. `已执行` 不等于闭环完成，仍必须进入 `已结算`。


## 9. 审计与责任回写

每一次动作至少要留下：

- `发起者`

- `回写者`

### 9.2 审计回写最小面

`自我线程审计回写器.ixx` 当前至少要写回：

- 审计签名
- 发起者主键
