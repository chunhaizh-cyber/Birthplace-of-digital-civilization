## 1. 总体评估

大部分未实现项可以实现，但要分阶段。

当前已经有以下基础：

1. 状态节点、动态节点、二次特征节点、因果模板节点。
2. 状态前后对生成动态的基础入口。
3. 动作动态基础入口。
4. 动态二次特征生成入口。
5. 连续同向段、同类事件段聚合入口。
6. 动态转抽象因果模板入口。
7. 状态活动链清理第一版入口。

当前主要缺口不是“完全没有动态系统”，而是：

1. 分层观察边界没有被硬约束。
2. 状态写入后没有统一进入动态生成。
3. 存在动态、场景动态还没有独立入口。
4. 自我方法动作动态没有统一强制出口。
5. 第三方外显状态、第三方动作推断、第三方因果转方法还缺数据源和转换链。
6. 因果稳定性升格还没有形成完整循环。

## 3. 功能设计

### 3.1 分层动态入口

目的：

把动态生成从一个通用入口拆成明确层级，避免跨层。

第一版入口：

| 入口 | 所属模块 | 生成动态 | 观察关系 |
| --- | --- | --- | --- |
| `存在观察实例特征状态变化` | `存在类/状态类/动态类` | 状态动态 | 存在观察自身实例特征 |
| `场景观察存在状态变化` | `场景类/动态类` | 存在动态 | 场景观察直接子存在 |
| `父场景观察子场景状态变化` | `场景类/动态类` | 场景动态 | 父场景观察直接子场景 |
| `方法执行生成动作动态` | `自我_执行桥模块/动态类` | 自我方法动作动态 | 自我真实调用方法 |

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 观察者 | 节点引用 | 存在、场景或父场景 |
| 被观察对象 | 节点引用 | 实例特征宿主、存在或子场景 |
| 前状态 | 状态节点引用 | 可为空，空则只记录当前状态 |
| 后状态 | 状态节点引用 | 必须有效 |
| 时间 | I64 | 发生或收到时间 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 是否生成动态 | 布尔 0/1 | 1 表示生成动态 |
| 动态节点 | 节点引用 | 新建或复用动态 |
| 拒绝原因 | 枚举/状态词 | 无前状态、跨层、主体不一致、特征不一致、无变化 |

方向：

- 生成动态数量越大，观察链越活跃。
- 拒绝次数越大，说明入口调用或数据层级存在问题。

### 3.2 跨层校验

目的：

任何动态生成前都先确认观察者只观察直接子层。

校验规则：

| 动态类型 | 合法观察关系 |
| --- | --- |
| 状态动态 | 状态主体等于观察存在、场景或虚拟存在 |
| 存在动态 | 被观察存在的父节点等于观察场景 |
| 场景动态 | 被观察子场景的父节点等于观察父场景 |
| 方法动作动态 | 动态主体为自我存在或方法虚拟存在，来源动作是真实方法调用 |

输出：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 是否允许生成 | 布尔 0/1 | 0 时拒绝创建动态 |
| 层级关系方向 | 枚举/状态词 | 直接父子、同层、跨层、无关系 |
| 拒绝原因 | 枚举/状态词 | 非直接子层、状态主体不一致、方法未真实调用 |

### 3.3 状态写入后自动生成状态动态

目的：

状态落账后，自动尝试生成状态动态，而不是依赖调用方手动传后处理。

流程：

```text
创建状态
-> 写入场景状态索引
-> 状态写入后整理旧状态
-> 查找同主体同特征的相邻前状态
-> 通过跨层校验
-> 生成状态动态
-> 生成动态二次特征
-> 尝试聚合
```

第一版规则：

1. 创建状态时只对同一场景、同一状态主体、同一状态特征找前状态。
2. 前后状态值完全相同，可选择不生成动态，或生成零变化动态；第一版建议不生成动态。
3. 新状态是创建事件且无前状态时，不生成状态动态。
4. 生成动态后，不物理删除来源状态。

### 3.4 存在动态

目的：

场景观察直接子存在的状态变化后，生成存在动态。

注意：这里不使用子层整理信息作为输入。存在动态来自场景对直接子存在状态变化的观察。

输入：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 场景 | 场景节点 | 观察者 |
| 存在 | 存在节点 | 必须是场景直接子存在 |
| 前存在状态 | 状态节点 | 可为空 |
| 后存在状态 | 状态节点 | 必须有效 |

输出：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 存在动态 | 动态节点 | 动态主体为存在 |
| 动态层级 | I64 | 状态动态为 0，存在动态为 1 |
| 聚合方式 | 枚举 | 第一版可用原子相邻 |

第一版实现策略：

1. 复用 `动态节点主信息类`。
2. `动态主体 = 被观察存在`。
3. `动态特征 = 后状态中的状态特征`，表示此存在某口径发生变化。
4. `动态层级 = 1`。
5. `来源低层动态` 可选；若存在状态动态已经生成，则挂入来源低层动态。

### 3.5 场景动态

目的：

父场景观察直接子场景的状态变化后，生成场景动态。

输入：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 父场景 | 场景节点 | 观察者 |
| 子场景 | 场景节点 | 必须是父场景直接子场景 |
| 前场景状态 | 状态节点 | 可为空 |
| 后场景状态 | 状态节点 | 必须有效 |

第一版实现策略：

1. 先要求子场景本身有状态节点。
2. 不允许父场景直接读取子场景内部存在或特征。
3. 动态主体为子场景。
4. 动态层级为 2 或上层动态层级加一。

不确定点：

- 当前代码中场景自身状态来源不充分，可能需要先实现“场景状态记录入口”。

### 3.6 多特征协同动态

目的：

在共同父层内发现不同特征变化之间的关系。

第一版关系：

| 关系 | 判断 |
| --- | --- |
| 同步变化 | 两个状态动态时间窗口重叠 |
| 前置变化 | A 的变化结束早于 B 的变化开始 |
| 滞后变化 | B 在 A 之后固定窗口内出现 |
| 抑制关系 | A 出现后 B 的预期变化不出现 |

实现方式：

1. 不新增结构体。
2. 使用二次特征节点表达协同关系。
3. 二次特征来源绑定两个动态或两个状态。
4. 若关系稳定，再进入因果候选。

### 3.7 自我方法动作动态统一出口

目的：

所有真实方法调用都必须生成动作动态，非真实执行不能伪造动作动态。

统一入口：

```text
真实方法执行开始
-> 记录输入场景、输出场景、方法首节点、动作名
-> 方法执行完成
-> 创建自我方法动作动态
-> 写入来源执行成功、错误码、相位
-> 绑定来源方法和来源场景
```

禁止入口：

1. 筹办查找。
2. 缺方法报错。
3. 缺条件报错。
4. 线程消息落账。
5. 任务状态推进。

这些只能生成治理记录或状态动态，不能生成自我方法动作动态。

### 3.8 动作动态与治理动态边界

当前代码中存在任务最小回写器创建动态并标记 `方法调用` 的情况。后续需要统一收口：

| 情况 | 动态类型 |
| --- | --- |
| 真实调用方法函数 | 自我方法动作动态 |
| 任务筹办、调度、消息处理 | 治理过程动态 |
| 失败边界、挂起、收束 | 边界结束动态 |
| 内部修复、回归、保护 | 内部机制动作或被动记账动态 |

第一版处理：

1. 增加动作动态创建前置判定。
2. 治理动态不得写 `来源类型 = 方法调用`。
3. 真实方法执行入口统一写 `来源类型 = 方法调用`。

### 3.9 第三方推断动作动态

可先实现骨架，完整闭环依赖外显状态来源。

骨架流程：

```text
观察到目标外显状态 S0 -> S1
-> 读取 t0 到 t1 中间状态集合
-> 排除目标结果状态自身
-> 生成候选原因集合
-> 多次样本做语义交集
-> 找到候选原因存在
-> 生成第三方动作动态候选
```

第一版不新增复杂结构：

1. 候选原因用状态节点和因果候选表达。
2. 稳定前不视为可信动作事实。
3. 稳定后生成因果信息。
4. 第三方动作具体方法未知时，只生成抽象方法。

不确定点：

- 外显状态由谁提供。
- 外显状态是否仍使用普通状态节点表达。
- 第三方候选动作动态是否需要专门状态词表示阶段：候选、临时、待验证、稳定。

### 3.10 外显状态确认链

目的：

把自我内部状态和外部观察状态进行比较确认。

输入：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 自我内部状态 | 状态节点 | 自我自己记录 |
| 外显观察状态 | 状态节点 | 另外途径观察 |
| 对齐特征口径 | 特征节点 | 同一特征类型或可比较口径 |
| 时间窗口 | I64 | 允许的时间差 |

输出：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 是否确认一致 | 布尔 0/1 | 1 表示内部状态被外部观察确认 |
| 差值 | I64 | 外显值 - 内部值 |
| 确认动态 | 动态或二次特征 | 表达确认结果 |

### 3.11 因果动态升格

目的：

把多个动态样本升格为因果信息。

第一版可复用 `因果主信息类` 字段：

| 字段 | 用途 |
| --- | --- |
| `证据动态样本` | 来源动态样本集合 |
| `成立次数` | 正样本次数 |
| `失败次数` | 反例次数 |
| `置信度` | 稳定程度 |
| `已验证` | 是否可作为稳定因果 |
| `最近命中时间` | 最近成立时间 |
| `最近失败时间` | 最近反例时间 |

升格规则：

| 阶段 | 判定 |
| --- | --- |
| 候选因果 | 有至少 1 个动态样本 |
| 临时因果 | 成立次数大于 1 且语义相同 |
| 待验证因果 | 成立次数明显高于失败次数 |
| 稳定因果 | 达到稳定阈值，`已验证 = true` |

第一版阈值可使用 I64，不使用 Q10000。

### 3.13 第三方因果转方法

目的：

第三方因果稳定后，形成第三方方法列表中的抽象方法。

规则：

1. 第三方因果不能直接加入自我可执行方法列表。
2. 抽象方法表达“第三方存在某种方法可以达成结果”。
3. 抽象方法不带自我可执行动作函数。
4. 只有补齐自我动作函数、输入条件、输出场景、目标特征和稳定性后，才能转换为自我方法。

不确定点：

- 第三方方法列表挂在世界树哪里。
- 抽象方法是否复用现有方法树节点，还是挂在第三方存在的虚拟方法空间。

3. 活动状态限量清理入口。
