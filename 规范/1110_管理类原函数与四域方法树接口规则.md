## 3. 核心结论

必须固定以下结论：

1. `任务管理任务` 自身也是任务，不是任务系统外的额外管理员。
2. `任务管理任务` 是任务系统中的特殊任务，也是任务控制链的工程起点。
3. 管理类原函数是任务管理任务在四域方法树中的先天执行原语。
4. `任务管理任务` 由一组管理类本能函数构成，这组本能函数共同承担任务的 `筹办` 与 `执行承接`。
5. 这些本能函数运行时，必须能够确保或生成其对应的 `方法首节点`，作为后续方法树复用、生长和调度的挂点。
6. 四域方法树是本域优先复用的经验树，本域执行器是未命中时才接管的兜底器。
7. 输入场景、输出场景只是统一容器，容器中既可以放现实世界存在，也可以放任务虚拟存在、方法虚拟存在、需求目标状态和当前步骤指针。
8. `任务虚拟存在` 与 `方法虚拟存在` 是特征承载存在，不是任务或方法本体；其中主要存储运行特征、统计特征、派生特征与二次特征。
9. 四域方法树运行所需信息，必须拆成 `前置上下文特征` 与 `原函数产出特征` 两层，不允许混成“一次返回全包”。
10. 新增管理类原函数的职责，是产出本职责范围内的管理事实、管理建议和管理证据，而不是重复返回全部四域上下文。
11. 管理类原函数的输出，默认先落到输出场景和必要的虚拟存在特征中，不直接等同于调度器消息。
12. 任务状态推进、消息分发、队列流转，仍由任务管理器、总控与调度器负责。
13. 管理类原函数允许输出二次特征、评分、原因码、指针句柄和子场景证据，但不得借名发明第二套任务状态机。

可收敛为一句话：

`任务管理任务是任务控制链的特殊起点任务；管理类原函数是它的先天执行原语，并通过方法首节点把筹办与执行承接进四域方法树。`

### 3.1 最新体系下的最终定位

按当前最新逻辑，四域方法树必须重新固定其层级位置：

1. 四域方法树 `仍然可用`，但其定位应固定为 `任务管理的工程方法树 / 经验复用树 / 本域兜底写回树`。
2. 四域方法树 `不是` 根语义裁决树，不负责定义根目的，不负责改写安全与服务的根因果关系，也不负责替代需求方向判断。
3. 根层是否允许执行、结果方向是否成立，仍必须先由 `需求值 / 当前值 / 返回值` 的比较、由此生成的方向，以及安全/服务根约束裁决。
4. 只有在根层已允许继续推进的前提下，四域方法树才负责回答：`当前域里优先复用什么经验路径、未命中时走什么兜底路径、成功后如何写回成长`。
5. 四域方法树输出的本质，应继续限定为：`管理事实 / 管理建议 / 管理证据`，而不是最终真理。
6. 四域方法树当前允许继续以工程代理量运行，但后续应逐步把输入重心转向 `任务管理任务最小一次特征组`，减少长期依赖纯布尔拼接和优先级代理量。

当前阶段可收敛为以下分层：

```text
根需求与根方向裁决
-> 任务管理任务的一次特征与管理上下文
-> 四域方法树做本域复用 / 兜底 / 写回
-> 外层任务管理器与调度器真正落地状态迁移和结构改写
```

因此必须固定：

1. 四域方法树不得跳过根方向判断，直接决定“什么是最终正确”。
2. 四域方法树可以决定“当前域中先走哪条管理路径”，但不能决定“根层是否允许违背安全/服务去换取局部成功”。
3. 四域方法树命中与否，只能作为工程复用层证据，不能直接等同于语义正确性成立。
4. 四域方法树的自动生长，只能沉淀已被外层确认有效的管理路径，不能把尚未经过根层方向确认的路径直接升格为稳定经验。
5. `当前缺口类型`、`当前下一步去向`、`当前功能域建议` 等管理判定，最终都只能建立在比较后形成的 `二次特征`、条件判定结果和证据之上。
6. 一次特征、原始状态、原始布尔量、原始数值只能作为管理输入真值，不得直接跳过比较过程充当最终管理裁判依据。


## 4. 分层定义

### 4.1 结构层

结构层负责回答“谁是谁、挂在哪里、彼此怎样关联”。

包括：

- 任务信息节点
- 方法信息节点
- 需求节点
- 步骤节点
- 结果节点

### 4.2 虚拟存在层

虚拟存在层负责承载“可读写、可比较、可被方法消费”的运行特征。

包括：

- 任务虚拟存在
- 方法虚拟存在

### 4.3 场景层

场景层负责一次调用的容器化装载。

包括：

- 输入场景
- 输出场景
- 子场景
- 场景中的状态、关系、指针与证据


## 5. 四域所需信息的两层来源

### 5.1 前置上下文特征

以下信息属于进入四域方法树匹配前就应具备的上下文，不要求由单个原函数重复返回：

- 当前功能域
- 任务虚拟存在
- 方法虚拟存在
- 当前任务头结点 / 当前步骤结点 / 当前方法首节点
- 当前现实场景
- 需求目标状态
- 安全值
- 服务值
- 需求值
- 当前预算
- 最近结果稳定
- 最近执行失败
- 风险变坏
- 检查点通过
- 依赖缺失
- 需要新子任务或新步骤
- 关键方法不再可用
- 已满足完成判据

### 5.2 原函数产出特征

以下信息适合作为管理类原函数的直接产出：

- 条件缺口分类
- 可执行性判断
- 推荐方法指针
- 候选方法评分
- 推荐分解方向
- 是否需要补条件任务
- 是否需要观察任务
- 是否需要恢复任务
- 是否建议等待学习
- 是否建议回到筹办
- 失败原因码
- 评分证据
- 二次特征证据
- 结果主键

### 5.3 分层硬约束

必须固定：

1. 原函数不负责重复生成全部四域上下文。
2. 四域匹配器消费的是完整上下文，而不是某个原函数的单一返回值。
3. 原函数必须完整覆盖其职责范围内的产出特征。
4. 前置上下文缺失时，应由外层先补齐输入场景，而不是要求原函数自造上下文真值。

### 5.4 任务管理任务最小一次特征组

本组只讨论 `一次特征`，不讨论评分、二次特征、可执行度、可靠度和综合裁决。

必须固定：

1. `任务管理任务` 只持有“我当前在管谁、准备怎么管”的管理上下文真值。
2. `具体任务` 持有“我当前处在哪里、缺什么、刚产出了什么”的任务侧真值。
3. 同一个特征不得同时存在两个真值归属点。
4. 若管理侧存在镜像，而任务侧真值与管理侧镜像不一致，必须以任务侧真值为准，管理侧镜像立即进入修订状态。

| `当前下一步去向` | 指出当前轮次管理后要继续执行、补条件、重筹办、转学习、挂起等待还是收束 | `未定义 -> 某去向`、`去向A -> 去向B`、`去向 -> 未定义` | 当前主要落在 `结构_四域执行上下文::建议动作 / 下一步骤 / 需要重筹办`，尚未沉淀为独立虚拟存在特征。 |

#### 5.4.2 真值归属点与管理侧镜像表

| 特征名 | 真值归属点 | 是否允许管理侧镜像 | 归属说明 |
| --- | --- | --- | --- |
| `当前管理任务头指针` | `任务管理任务 / 当前调度上下文` | `不涉及（自身即管理真值）` | 这是“我现在在管谁”的管理事实，不属于被管理任务本体。 |
| `当前承接需求指针` | `具体任务头` | `是` | 需求属于任务来源，不属于管理者本体；管理侧若要快速裁决，可保留镜像，但不得反客为主。 |
| `当前步骤指针` | `具体任务头` | `是` | 当前步骤描述的是任务推进位置；管理侧允许镜像，但任务头上的当前步骤才是真值。 |
| `当前方法首节点指针` | `具体任务头 / 当前步骤` | `是` | 当前选中方法服务于具体任务推进；管理侧可暂存镜像用于分流和解释。 |
| `当前缺口类型` | `具体任务头 / 当前步骤` | `是` | 缺口是任务推进缺口，不是管理者私有属性；管理侧可保留本轮镜像。 |
| `当前下一步去向` | `任务管理任务 / 当前调度上下文` | `不涉及（自身即管理真值）` | 这是本轮管理裁决结果，首先属于管理上下文。若后续需要持久化，可再映射为任务侧状态或结果。 |
| `最近结果节点指针` | `具体任务头` | `是` | 最近结果是任务真实产出，应由任务头持有；管理侧可镜像用于复盘与收束判定。 |

#### 5.4.3 当前阶段的工程结论

当前应视为：

1. `当前管理任务头指针` 与 `当前下一步去向` 还主要停留在执行上下文字段层。
2. `当前承接需求指针 / 当前步骤指针 / 当前方法首节点指针 / 最近结果节点指针` 已经基本有了任务侧真值归属点。
3. `当前缺口类型` 仍缺正式的一次特征承载位，现阶段主要靠分析结果和布尔上下文字段表达。
4. 后续若要把任务管理任务进一步方法化，优先补的是 `当前缺口类型` 与 `当前下一步去向` 的稳定特征承载。

#### 5.4.4 `当前缺口类型` 取值表

`当前缺口类型` 当前应先被理解为“任务当前不能直接推进时，最主要缺口属于哪一类”。

必须固定：

1. `当前缺口类型` 是一次特征，不是评分，不是可执行度。
2. 当前阶段缺口类型必须优先从条件判定结果中提取，不得跳过底层判定直接写抽象口号。
3. 若同时存在多个缺口，当前工程阶段只允许提炼一个 `主缺口类型`；详细缺口清单仍保留在分析结果中。
4. 主缺口提炼的工程优先级当前固定为：`无权限 > 不可获取 > 值未知需观察 > 未满足但可补 > 无缺口`。
5. `当前缺口类型` 的分层定位，统一属于 [0566_条件判定结果层与条件裁决边界规范.md](D:\鱼巢\规范\0566_条件判定结果层与条件裁决边界规范.md) 中的 `主缺口类型提炼结果`。

| 缺口类型值 | 含义 | 允许的最小变化 | 当前代码来源 |
| --- | --- | --- | --- |
| `无缺口` | 当前条件已满足，可直接执行 | `某缺口 -> 无缺口`、`无缺口 -> 某缺口` | 当前对应 `结构_条件展开结果::当前可直接执行 = true`，以及 `枚举_条件判定类别::已满足`。 |
| `待观察` | 当前值未知，需要先观察再继续判断 | `未定义 -> 待观察`、`待观察 -> 其他缺口/无缺口` | 当前对应 `枚举_条件判定类别::值未知需观察`，分析入口见 `任务执行类::判定目标状态可用性(...)`。 |
| `可补条件` | 当前条件未满足，但可以通过补条件、补参数或重筹办继续推进 | `未定义 -> 可补条件`、`可补条件 -> 其他缺口/无缺口` | 当前对应 `枚举_条件判定类别::未满足但可补`。 |
| `不可获取` | 需要的条件当前无法获取或目标状态不完整，不能直接补出 | `未定义 -> 不可获取`、`不可获取 -> 其他缺口/无缺口` | 当前对应 `枚举_条件判定类别::不可获取`。 |
| `无权限` | 目标状态或动作当前不允许直接使用 | `未定义 -> 无权限`、`无权限 -> 其他缺口/无缺口` | 当前对应 `枚举_条件判定类别::无权限`。 |

当前工程上应视为：

- `当前缺口类型` 的正式一次特征位仍待补。
- 现阶段可先由 `任务执行类::分析方法条件缺口(...)` 的输出提炼主缺口，并作为管理侧镜像使用。
- 以后若写回任务侧真值，应写入任务虚拟存在或步骤虚拟存在，不得只停留在局部栈变量里。

#### 5.4.5 `当前下一步去向` 取值表

`当前下一步去向` 当前应先被理解为“任务管理任务在本轮管理后，要求系统进入的下一种推进路径”。

必须固定：

1. `当前下一步去向` 首先属于管理上下文真值。
2. 当前阶段下一步去向必须优先复用现有 `任务反馈动作 / 分支选择动作 / 下一步骤指针 / 需要重筹办` 的组合结果，不再新造第三套状态词。
3. 若将来需要长期沉淀为任务侧一次特征，必须保持与管理上下文可逆映射。
4. `当前下一步去向` 的分层定位，统一属于 [0566_条件判定结果层与条件裁决边界规范.md](D:\鱼巢\规范\0566_条件判定结果层与条件裁决边界规范.md) 中的 `下一步去向候选结果`。

| 下一步去向值 | 含义 | 允许的最小变化 | 当前代码来源 |
| --- | --- | --- | --- |

当前工程上应视为：

1. `继续下一步 / 回跳重试 / 补条件后重试 / 转入尝试学习 / 挂起等待` 已经有较稳定的代码值域。
2. `回到重筹办` 与 `进入收束` 目前仍主要通过组合字段表达，还不是独立单值枚举。
3. 后续若要把 `当前下一步去向` 写成正式一次特征，建议优先把 `回到重筹办` 与 `进入收束` 也补成稳定值域，避免长期依赖布尔组合判定。


## 6. 输入输出场景规范

### 6.1 输入场景允许装载的对象

管理类原函数的输入场景允许直接装载：

- 任务虚拟存在指针
- 方法虚拟存在指针
- 需求目标状态
- 当前任务头结点指针
- 当前步骤结点指针
- 当前方法首节点指针
- 最近反馈摘要
- 当前控制预算
- 当前风险与检查点状态

### 6.2 输出场景允许写入的对象

管理类原函数的输出场景允许写入：

- I64 特征值
- 指针特征值
- 状态节点
- 二次特征节点
- 原因码
- 推荐方法 / 推荐步骤 / 推荐子任务的指针
- 子场景形式的证据包

### 6.3 写回边界

必须固定：

1. 即时供调用方消费的信息优先写输出场景。
2. 需要长期沉淀的统计与组织信息，允许写回任务虚拟存在或方法虚拟存在。
3. 原函数不应直接冒充调度器去提交 `结构_任务返回消息`。
4. 原函数若需要推动状态机，必须通过外层适配器解释其输出后再落地。


## 7. 七个管理类原函数的职责表

### 7.1 第一批最小闭环

当前最少应先落成以下 4 个：

- `管理_筹办器(4000)`：负责读取需求目标、任务虚拟存在与当前场景，输出候选方法、条件缺口与可执行筹办结论。
- `管理_门控器(4004)`：负责在维护域判断继续维护、转筹办、转纠偏或转收束。
- `管理_复盘器(4005)`：负责在收束域判断是否满足完成/失败闭环，并输出收束证据与经验摘要。
- `管理_失败策略器(4006)`：负责在纠偏域输出替代方法、回退、收缩、恢复任务与等待学习建议。

### 7.2 第二批补全能力

为了让任务管理任务更完整地转为原函数驱动，还应补齐：

- `管理_选择器(4001)`：在候选方法、候选步骤、候选子任务之间做选择。
- `管理_分解器(4002)`：把父任务分解成步骤、阶段任务、补条件任务和恢复任务建议。
- `管理_求证器(4003)`：验证需求是否满足、预测是否命中、完成判据是否成立。

### 7.3 各函数的最小输出责任

必须固定：

- `管理_筹办器` 至少输出：候选方法、条件缺口、可直接执行标记、推荐分解方向。
- `管理_选择器` 至少输出：被选对象指针、排序分、选择原因码。
- `管理_分解器` 至少输出：新步骤建议、新子任务建议、并发/串行关系建议。
- `管理_求证器` 至少输出：需求满足标记、命中证据、完成判据标记。
- `管理_门控器` 至少输出：下一域建议、继续/回退/等待标记。
- `管理_复盘器` 至少输出：终态结论、经验摘要、收束原因码。
- `管理_失败策略器` 至少输出：替代方法、恢复动作、等待学习或无法继续建议。


## 8. 与四域方法树的优先级关系

必须固定运行顺序：

1. 先判定当前功能域。
2. 再匹配对应的本域方法树。
3. 本域树命中时，优先使用本域树建议方法。
4. 只有本域树未命中时，才允许进入本域兜底执行器。
5. 兜底执行器走通后的成功路径，允许写回本域方法树。

因此：

- 管理类原函数不是树外兜底补丁。
- 管理类原函数首先是四域树根上的先天动作来源。
- 本域执行器应视为“域树未长出足够枝条时”的保底承接层。


## 9. 当前实现对齐结论

按当前代码口径，应视为：

1. `4000~4006` 的管理类原函数 ID 已预留。
2. 四域方法树根已绑定 `管理_筹办器 / 管理_门控器 / 管理_失败策略器 / 管理_复盘器` 作为根动作。
3. 当前主要任务管理能力仍大量落在总控模块、任务执行器、缺方法分流器与各域执行器中。
4. 因此当前系统已经具备“四域树优先、未命中才兜底、成功路径可写回”的框架，但管理类原函数执行体仍未完全落成。


## 10. 筹办兜底方法化边界

当前筹办兜底逻辑允许逐步方法化，但必须固定为“分析与决策方法化，结构与状态落地留在外层”。

### 10.1 适合先方法化的部分

以下能力适合作为第一批筹办管理类原函数沉入方法树：

- 候选方法召回
- 方法结果满足需求的评分
- 条件缺口分析
- 候选方法可执行性判断
- 补条件 / 观察 / 等待学习 / 无法继续的分流判定
- 候选步骤排序与首选步骤推荐

这些部分的共同特点是：

- 输入可以主要来自任务虚拟存在、方法虚拟存在、需求目标状态和当前场景
- 输出可以自然落成评分、原因码、推荐指针、缺口分类和证据二次特征
- 它们本质上属于“算什么、判什么、推荐什么”

### 10.2 不应直接方法化的部分

以下能力不宜直接下沉为管理类原函数主体，而应继续由任务管理器、总控或调度器负责：

- 确保任务头结点存在
- 创建初始步骤结点
- 创建初始结果结点
- 把方法绑定到步骤
- 写任务状态 `就绪 / 待重筹办 / 无法执行 / 等待中`
- 提交任务返回消息
- 进入队列、挂起、等待学习唤醒

这些部分的共同特点是：

- 它们直接改写任务结构或状态机
- 它们承担调度与生命周期责任
- 它们不只是“给建议”，而是在真正修改系统运行结构

### 10.3 固定分工

因此，筹办兜底方法化后必须保持以下分工：

1. 管理类原函数负责输出管理事实、管理建议和管理证据。
2. 任务管理器负责按这些输出真正创建步骤、结果、子任务和绑定关系。
3. 总控与调度器负责把外层解释结果落成状态迁移与消息推进。

可收敛为一句话：

`筹办兜底可以方法化，但应方法化的是“筹办分析与决策”；真正的任务结构创建、状态写回和消息推进仍留在外层控制器。`


## 11. 硬约束

1. 管理类原函数必须被视为四域方法树中的先天执行原语。
2. 任务管理任务使用的输入场景，允许直接携带任务虚拟存在与方法虚拟存在。
3. 虚拟存在存储的是任务或方法的运行特征、统计特征、派生特征与二次特征，而不是结构关系本体。
4. 四域方法树运行信息必须拆成前置上下文特征与原函数产出特征两层。
5. 新增原函数不得被要求单次返回四域全部所需特征。
6. 新增原函数必须完整覆盖其职责范围内需要产出的管理事实、管理建议与管理证据。
7. 运行时必须先优先匹配当前域方法树，未命中后才允许进入本域兜底执行器。
8. 兜底成功路径允许写回本域方法树，作为临时枝继续生长。
9. 管理类原函数不得绕开外层适配器直接冒充调度器消息。
10. 第一批实现最少应补齐 `管理_筹办器 / 管理_门控器 / 管理_复盘器 / 管理_失败策略器`。
