# 0579 特征使用账本与生命周期规范

## 1. 目标

本文用于固定以下问题：

- 特征是否值得保留，到底看什么
- 什么叫“被使用”，什么不算“被使用”
- 特征生命周期如何按使用事实推进
- 为什么不能只按抽象价值函数保留特征
- 为什么“低频高危特征”不能按频率误删

说明：

- 本文中的代码块或流程块都只表示语义，不表示现有实现中的直接接口或函数签名。


## 2. 核心结论

必须固定：

1. `特征是否值得保留`，不看它抽象上“看起来有没有价值”；优先只看它是否形成过 `有效使用事实`。
2. `被读取`、`被计算`、`被展示`，都不自动等于 `被使用`。
3. 真正的 `被使用`，指的是：
   - 某个特征参与了决策、排序、过滤、风险闸门、高阶派生、需求显现、验证、解释或学习闭环
   - 并且对结果产生了影响，或仍被下游正式依赖
4. 一个特征类型值得保留，当且仅当：
   - 它已经形成过至少一条 `有效治理边`
   - 或者它还处在 `试用暴露期`
5. `被使用但经常误导` 的特征，不应直接删除；应进入 `修复` 或 `学习`。

一句话收口：

`特征保留不看抽象价值，只看有效参与治理闭环的使用事实。`


## 3. 什么不算使用

必须明确区分：

```text
被读取 ≠ 被使用
被计算 ≠ 被使用
被展示 ≠ 被使用
```

因此以下情况默认都不构成正式使用：

1. 某个模块只是把特征值读出来，但没有参与后续选择、约束、派生、验证或解释。
2. 某个特征只是被重算或刷新，但没有进入任何决策或高阶依赖链。
3. 某个特征只是被控制面板、日志、摘要或调试页面展示。


## 4. 什么算有效使用

`有效使用` 指的是：

- 该特征参与了某条治理闭环
- 并且在该闭环中形成了 `有效治理边`

### 4.1 有效治理边

有效治理边至少包括：

```text
feature -> decision
feature -> step
feature -> demand
feature -> learning_task
feature -> method
feature -> risk_gate
feature -> validation
feature -> higher_order_feature
feature -> explanation
```

必须固定：

1. 没有形成过治理边的特征，不是活特征。
2. 形成了治理边但没有改变任何结果的特征，不自动算 `有效使用`，除非它仍被正式下游依赖。
3. 作为高阶特征父节点被使用，也算正式使用。


## 5. 使用分层

### 5.1 直接使用

直接使用指：

- 特征直接参与下一步选择、排序、过滤、风险闸门或完成判定。

例如：

- `step_executability` 参与下一步排序
- `irreversible_change_risk` 阻断高风险动作

### 5.2 派生使用

派生使用指：

- 特征本身没有直接决定最终动作，但作为高阶特征的父节点参与了组合
- 最终经由高阶特征影响了决策、学习任务、方法选择或需求显现

例如：

```text
task_has_learning_signal
task_method_availability
task_previous_failure_count
    ↓
task_learning_gap_pressure
    ↓
task_self_growth_priority
    ↓
创建学习任务
```

这里父特征也算被正式使用。

### 5.3 显现使用

显现使用指：

- 特征本身不直接选步骤
- 但它让一个以前不可见的需求、风险或学习任务显现出来

例如：

- `system_governance_stagnation_index` 触发系统成长需求
- `task_self_growth_priority` 触发学习任务

必须固定：

1. `显现使用` 不是弱使用，它是正式使用的一类。
2. 某些低频高阶特征主要通过显现使用发挥价值，不得因为频率低而误删。


## 6. 使用事件与暴露事件

后续应至少区分两类账本：

1. `使用事件`
2. `暴露事件`

### 6.1 使用事件

`特征使用事件` 记录：

- 谁用了该特征
- 怎么用的
- 用在谁身上
- 是否改变了结果
- 改变了什么结果

必须固定：

1. 判断是否“被正式使用”，优先看 `使用事件`。
2. 其中最关键的是：
   - 是否真的形成了治理边
   - 是否真的改变了结果

### 6.2 暴露事件

`特征暴露事件` 记录：

- 某个特征曾经被暴露给哪个决策器、组合器、需求显现器、风险闸门或学习触发器
- 该轮是否真的被用上

必须固定：

1. `没有暴露机会 + 没有使用`，不能证明特征无用。
2. `有暴露机会 + 仍长期没有使用`，才构成归档证据。


## 7. 生命周期

特征生命周期统一按使用账本驱动：

```text
candidate
-> trial
-> active
-> cold
-> archived
-> deprecated
```

各态含义固定为：

1. `candidate`
   - 新发现，尚未充分暴露

2. `trial`
   - 已进入试用暴露窗口，允许被决策链、组合链、需求显现链、风险链消费

3. `active`
   - 已经形成过有效使用边

4. `cold`
   - 历史上被使用过，但近期未再形成有效使用

5. `archived`
   - 试用期结束且仍没有形成有效使用，也没有正式下游依赖

6. `deprecated`
   - 已明确不再建议默认使用，只允许兼容或历史回放

### 7.1 生命周期推进规则

最小规则固定为：

1. 新发现特征先进入 `candidate -> trial`
2. 试用期内一旦形成有效使用边，进入 `active`
3. 历史上用过但近期未再使用，进入 `cold`
4. 试用期结束且没有任何有效使用，也没有下游依赖，进入 `archived`
5. 若特征已明确不再建议默认接入主链，则进入 `deprecated`


## 8. 保留与归档规则

特征保留的最小判断，统一按以下语义理解：

```text
已经被有效使用，保留；
仍在试用期，暂存；
试用期已过且没有形成有效使用，归档；
长期归档且仍无任何链路引用，可以废弃。
```

必须固定四条最小规则：

1. 从未被使用，且试用期结束：`归档`
2. 被用于决策、派生、风险、需求显现、验证之一：`保留`
3. 被高阶特征依赖：`保留`
4. 被使用但结果经常不好：`不删除，转修复/学习`

### 8.1 低频高危特征

必须额外固定：

1. 不允许只按使用频率判断保留。
2. `低频高危特征`、`低频风险闸门特征`、`低频需求显现特征` 只要曾经改变过结果，就应继续保留。

例如：

- `schema_migration_risk`
- `learning_memory_pollution_risk`
- `irreversible_change_risk`
- `critical_path_touch`


## 9. 特征使用类别

为避免误删，允许把特征的使用类别进一步分成：

```text
frequent_operational
rare_guard
composition_parent
diagnostic
demand_revealer
historical
```

必须固定：

1. `rare_guard` 不得因为低频被自动归档。
2. `composition_parent` 只要仍被高阶特征依赖，就不能视为未使用。
3. `demand_revealer` 只要曾经显现过正式需求，就应继续保留。


## 10. 坏使用与修复

必须再区分一层 `健康状态`：

```text
healthy
unused
used_but_suspect
used_but_unstable
obsolete
```

其中：

1. `unused`
   - 尚未形成有效使用

2. `used_but_suspect`
   - 被正式使用过，但经常与后续结果不一致

3. `used_but_unstable`
   - 被使用过，但值本身摇摆过大、含义不稳定

4. `obsolete`
   - 已经被更新特征或更稳定特征替代

必须固定：

1. `使用但误导` 不等于 `应该删除`
2. 更合理的处理是：
   - 修正提取策略
   - 修正组合公式
   - 拆分特征类型
   - 降低权重
   - 增加验证信号
   - 派生学习任务或修复任务


## 11. 与抽象价值函数的边界

以下旧口径应降级：

```text
coverageGain
+ explanationGain
+ decisionValue
+ predictionGain
- extractionCost
- redundancyPenalty
```

必须固定：

1. 抽象价值函数可以作为观察指标、研究指标或辅助排序指标保留。
2. 但它不再构成 `是否保留一个特征类型` 的主裁决依据。
3. 正式保留与归档，优先看 `使用账本`、`有效治理边` 与 `试用暴露期`。


## 12. 最终收口

`一个特征类型值得保留，当且仅当：它已经形成过至少一条有效使用边，或者它仍处在试用暴露期。`

更工程化地说：

`特征保留不看抽象价值，只看使用事实；但“使用”必须指有效参与治理闭环，而不是被计算、被读取、被展示。`
