# 0041 一次特征输入到二次特征最小实现表 v0.1

## 1. 目标

本文用于把 `G2 二次特征生成` 这一步，压成一张可直接接代码的最小实现表。

本文只回答四件事：

1. `自我线程` 在入账与对账之后，二次特征应吃哪些输入。
2. `状态型二次特征` 与 `动态型二次特征` 分别先做哪几项。
3. 每项二次特征的 `原因摘要模板` 应如何统一。
4. 三类上行消息分别触发哪些二次特征重算。

本文只负责：

`把“消息入账后的哪些一次事实，会触发哪些二次特征，以及摘要怎么统一拼”冻结成 G2 代码实现表。`


## 2. 核心结论

必须固定以下结论：

1. `二次特征` 至少分两类：
   - `状态型二次特征`
   - `动态型二次特征`
2. `状态型二次特征` 主要回答 `现在是否成立`。
3. `动态型二次特征` 主要回答 `过程正在往哪个方向演化`。
4. `三类消息` 只决定 `重算触发范围`，不直接替代二次特征输入真值。
5. `G2` 的正式输入必须来自：
   - `一次特征变化集`
   - `关键中间状态变化项`
   - `治理侧动作摘要`
   - `执行尝试对账结果`
   - `当前任务/控制态/等待态镜像`
6. 对动态型二次特征，若下层 [二次特征生成模块.cpp](D:/鱼巢/二次特征生成模块.cpp) 已能产出统一 `V1` 结果，则线程侧优先消费这些结果；当前 `V1` 最小动态输出固定为：
   - `变化量`
   - `变化方向`
   - `持续时间`
   - `变化率`
7. 每个二次特征都至少应统一产出：
   - `是否满足或是否成立`
   - `标量值或离散值`
   - `置信度`
   - `原因摘要`
8. `原因摘要` 必须由统一模板生成，不得在 `线程状态消息 / 派生需求消息 / 结果消息` 的分支里各自临时拼装。


## 3. G2 最小输入面

| 输入类 | 说明 | 当前主要来源 |
| --- | --- | --- |
| `一次特征变化集` | 已落账的状态变化真值 | `线程状态消息 / 派生需求消息 / 结果消息` |
| `关键中间状态变化项` | 过程里的阻塞、解阻、副作用、风险变化、恢复等中间事实 | 以 `结果消息` 为主 |
| `治理侧动作摘要` | 谁做了什么、是否成功、错误码、路径签名、动作语义键 | 三类消息统一可提供 |
| `执行尝试对账结果` | 重复、冲突、缺结果、缺结束消息、时序异常等 | `线程状态消息 + 结果消息` |
| `任务/控制态/等待态当前镜像` | 当前步骤前沿、前序完成、条件满足、等待原因、控制意图、运行状态 | 自我线程当前世界镜像 |
| `下层动态 V1 结果` | 变化量、方向、持续时间、变化率 | [二次特征生成模块.cpp](D:/鱼巢/二次特征生成模块.cpp) |

必须固定：

1. `消息名字` 不能直接当二次特征输入。
2. `触发消息源` 只回答“为什么要重算”，不回答“最终真值是什么”。
3. 动态型二次特征优先吃 `关键中间状态 + V1 动态结果`，而不是只看单条结果消息摘要。


## 4. 状态型二次特征实现表

| 二次特征 | 一次特征输入 | 原因摘要模板 | 主要触发消息源 |
| --- | --- | --- | --- |
| `依赖满足` | `当前等待原因`、`依赖等待状态`、`子任务结果状态`、`外部条件满足状态（占位输入）` | `等待原因={等待原因}; 子结果={子结果状态}; 外部条件={外部条件状态}; 依赖满足={是否满足}; 主因={主因}` | `结果消息` 为主，`派生需求消息` 次之 |
| `当前步骤可执行` | `当前步骤前沿`、`前序完成状态`、`条件满足状态`、`依赖满足`、`任务控制态.是否允许继续派发`、`工作线程.控制意图`、`工作线程.实际运行状态` | `步骤={步骤前沿}; 前序={前序状态}; 条件={条件状态}; 依赖={依赖满足}; 控制={控制态}; 线程={线程状态}; 可执行={是否满足}; 首阻断={主因}` | 三类消息都可触发 |
| `当前任务是否阻塞` | `当前运行状态`、`当前等待原因`、`派生需求类型`、`是否阻塞当前步骤`、`是否影响父任务推进`、`结果状态` | `运行={当前运行状态}; 等待={当前等待原因}; 缺口={派生需求类型}; 父影响={是否影响父任务推进}; 阻塞={是否满足}; 主阻断={主因}` | `线程状态消息`、`派生需求消息` |
| `当前任务是否解阻` | `当前运行状态`、`前一等待原因`、`依赖满足`、`结果状态`、`关键中间状态中的解阻事实` | `前阻断={前一等待原因}; 当前={当前运行状态}; 依赖={依赖满足}; 结果={结果状态}; 解阻={是否满足}; 解除原因={主因}` | `线程状态消息`、`结果消息` |
| `方法是否可调用` | `当前方法主键`、`条件满足状态`、`依赖满足`、`任务控制态.是否允许继续派发`、`工作线程控制意图`、`风险态`、`派生需求类型` | `方法={方法主键}; 条件={条件状态}; 依赖={依赖满足}; 控制={控制态}; 风险={风险态}; 可调用={是否满足}; 主因={主因}` | `派生需求消息` 为主，`线程状态消息 / 结果消息` 次之 |
| `是否允许继续推进` | `当前步骤可执行`、`当前任务是否阻塞`、`任务控制态.是否允许继续派发`、`工作线程实际运行状态`、`执行尝试对账结果` | `可执行={当前步骤可执行}; 阻塞={当前任务是否阻塞}; 控制={控制态}; 线程={线程状态}; 对账={对账结论}; 允推={是否满足}; 主因={主因}` | 三类消息都可触发 |

必须固定：

1. `依赖满足` 应先于 `当前步骤可执行` 重算。
2. `当前步骤可执行` 的原因摘要必须输出 `首阻断原因`，不能把多个阻断原因并列当主因。
3. `是否允许继续推进` 是线程治理口径，不等于单步 `当前步骤可执行`。


## 5. 动态型二次特征实现表

| 二次特征 | 一次特征输入 | 原因摘要模板 | 主要触发消息源 |
| --- | --- | --- | --- |
| `反向推进标记` | `关键中间状态变化项`、`结果状态`、`未满足项`、`V1.变化方向`、`V1.变化量`、`已恢复` | `目标方向={目标方向}; 当前方向={变化方向}; 幅度={变化量}; 恢复={已恢复}; 结果={结果状态}; 反向推进={是否满足}; 主因={主因}` | `结果消息` |
| `风险升高` | `派生需求阻塞级别`、`关键中间状态中的风险变化`、`错误码`、`V1.变化方向`、`V1.持续时间`、`V1.变化率` | `风险源={风险源}; 阻塞级别={阻塞级别}; 方向={变化方向}; 时长={持续时间}; 速率={变化率}; 风险升高={是否满足}; 主因={主因}` | `派生需求消息`、`结果消息` |
| `风险下降` | `关键中间状态中的恢复事实`、`结果状态`、`已恢复`、`V1.变化方向`、`V1.持续时间`、`V1.变化率` | `恢复源={恢复源}; 方向={变化方向}; 时长={持续时间}; 速率={变化率}; 已恢复={已恢复}; 风险下降={是否满足}; 主因={主因}` | `结果消息`、`线程状态消息` |
| `有效优先级候选` | `主导需求压力`、`是否允许继续推进`、`当前任务是否阻塞`、`风险升高/下降`、`结果完成度`、`服务/安全影响提示`、`V1.变化率` | `需求压力={需求压力}; 允推={是否允许继续推进}; 阻塞={当前任务是否阻塞}; 风险={风险态}; 完成度={完成度}; 速率={变化率}; 优先级候选={标量值}; 主因={主因}` | 三类消息都可触发，`结果消息` 权重最高 |

必须固定：

1. `反向推进标记`、`风险升高/下降` 属于线程治理的动态型判断，不等于下层 `V1` 动态结果本身。
2. `V1.变化量 / 变化方向 / 持续时间 / 变化率` 在这里主要承担 `动态判断上游材料`，不直接替代线程治理结论。
3. `有效优先级候选` 必须建立在前置二次特征已完成重算的基础上，不得直接从原始消息字段硬算。


## 6. 三类消息到重算范围的最小映射

| 触发消息源 | 状态型主重算项 | 动态型主重算项 |
| --- | --- | --- |
| `线程状态消息` | `当前任务是否阻塞`、`当前任务是否解阻`、`当前步骤可执行`、`是否允许继续推进` | `风险下降（若出现恢复/收束）`、`有效优先级候选` |
| `派生需求消息` | `依赖满足`、`当前步骤可执行`、`当前任务是否阻塞`、`方法是否可调用`、`是否允许继续推进` | `风险升高`、`有效优先级候选` |
| `结果消息` | `依赖满足`、`当前步骤可执行`、`当前任务是否解阻`、`方法是否可调用`、`是否允许继续推进` | `反向推进标记`、`风险升高/下降`、`有效优先级候选` |

必须固定：

1. `结果消息` 是动态型二次特征的主来源，但不是唯一来源。
2. `派生需求消息` 是阻塞、风险与外部支撑判断的重要前置来源。
3. `线程状态消息` 主要完善控制态与推进态，但在恢复、收束、停机等场景下也会触发动态型判断。


## 7. 代码落地约束

`G2` 代码实现阶段，必须满足以下约束：

1. 先根据受影响的一次特征，选出受影响的二次特征集合，再逐项重算。
2. 统一实现 `原因摘要模板` 生成函数，不在每个消息分支各自拼摘要。
3. 动态型判断优先消费下层 `V1` 输出；若缺少 `V1` 结果，再回退到 `关键中间状态变化项 + 动作摘要`。
4. 同一轮里先产出 `状态型二次特征`，再产出依赖它们的 `动态型二次特征` 和 `有效优先级候选`。
5. 本表只冻结 `G2` 第一批最小项；更高阶项如 `恢复趋势 / 衰减趋势 / 重复失败趋势 / 收束趋势`，后续再扩。


## 8. 一句话收口

`G2` 的实现不该再围着消息名字写分支，而应围着这张表做：先找受影响的一次特征，再重算对应的状态型/动态型二次特征，最后统一生成原因摘要并交给后续影响项与根层治理继续消费。
