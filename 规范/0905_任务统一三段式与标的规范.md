# 0905 任务统一三段式与标的规范

## 1. 目标

本文用于把当前所有任务统一收口到同一条主逻辑上：

`目标 -> 筹办 -> 执行`

本文回答的是：

- 什么叫任务的 `标的`
- 什么叫任务的 `目标`
- `筹办` 的唯一职责是什么
- `执行` 的唯一职责是什么
- `任务管理任务`、`学习任务`、普通任务分别在同一逻辑里对应什么对象

本文是后续任务规范的总约束；凡仍把任务主逻辑写成 `四域`、`五模块` 或其他并列主流程的口径，都应以下文为准。


## 2. 核心结论

必须固定以下结论：

1. 所有任务都遵循同一逻辑：`目标 -> 筹办 -> 执行`。
2. `目标` 不是运行时附属说明，而是任务头结点的第一定义。
3. `筹办` 的唯一职责，是完善任务树，使当前任务拥有可执行的步骤、结果、子任务和方法实例。
4. `执行` 的唯一职责，是执行当前步骤，把结果写入结果节点，并据此判断目标是否达成。
5. `筹办` 不直接代替执行；`执行` 不直接代替筹办。
6. 若执行后目标未达成，下一轮应回到 `筹办`，继续完善任务树，而不是发明平行主流程。
7. `任务管理任务`、`学习任务`、普通任务，只是 `标的` 不同，不是主逻辑不同。
8. 任务与方法都必须拥有各自的 `虚拟存在`，其特征才是主逻辑读写的权威运行特征面。
9. `步骤任务` 由筹办阶段建立，属于主步骤链；`前置任务` 由执行方法运行中发现缺口后提出，由自我线程实例化并挂回当前步骤。


## 3. 三类任务的统一解释

### 3.1 普通任务

- 标的：`现实世界中的存在`
- 目标：`使该存在的某些特征变为目标特征值`

### 3.2 任务管理任务

- 标的：`任务`
- 目标：`使该任务达到完成状态`

### 3.3 学习任务

- 标的：`方法`
- 目标：`使该方法变为可用`

因此三类任务统一收口为：

| 任务类型 | 标的 | 目标 |
| --- | --- | --- |
| 普通任务 | 现实世界存在 | 让目标特征达到目标值 |
| 任务管理任务 | 任务 | 让宿主任务达到完成 |
| 学习任务 | 方法 | 让目标方法变为可用 |


## 4. 目标

目标必须固定写在任务头结点上，至少回答：

1. 我在处理谁
2. 我想让它变成什么状态
3. 我以什么判据视为达成

因此：

- 普通任务头必须能指出 `目标存在 + 目标特征 + 目标值`
- 任务管理任务头必须能指出 `宿主任务 + 完成判据`
- 学习任务头必须能指出 `目标方法 + 可用判据`


## 5. 筹办

`筹办` 的唯一职责固定为：

`完善任务树`

具体只允许做以下几类事：

1. 明确当前目标对应的当前步骤。
2. 为步骤补齐结果节点。
3. 为步骤绑定或创建方法头。
4. 为方法创建任务实例条件节点和结果节点。
5. 必要时把当前可见的推进需求长成新的主步骤任务；这些步骤任务属于 `头 -> 步骤 -> 结果 -> 下一步骤` 主链。
6. 必要时触发 `特征空间生长请求`，让外层把当前反复出现却尚无法表达的差异升格成新的虚拟存在特征类型。
7. 必要时为结果节点显式写入 `下一跳步骤节点`，把“下一步走哪一条枝”固定到树上。
8. 对于“当前缺口是否已能被满足”这类判断，只允许形成筹办需求并提交给 `自我线程`，不得在任务本地直接定案。
9. 不直接建立执行方法运行中才会显现的前置任务头；这类前置缺口只能由执行方法提出，再由自我线程实例化并挂回当前步骤。

必须固定：

- `筹办` 可以长树，但不直接替代步骤执行。
- `筹办` 可以补方法骨架，但不直接把“方法存在”冒充“方法已执行”。
- `筹办` 的产物是树结构和运行实例准备度，不是最终结果。
- `筹办` 消费的是当前特征空间里已经可见的缺口；它不是新特征的根来源。
- `筹办` 可以把 `可执行判断 / 允许执行判断` 作为执行入口闸门交给执行步骤，但它们不再升格为一级主流程。
- `筹办` 长的是主步骤链，不长执行中临时发现的前置任务链。


## 6. 执行

`执行` 的唯一职责固定为：

`执行当前步骤，并把结果写入当前结果节点`

执行至少包含：

1. 消费筹办交给当前步骤的执行入口闸门。
2. 读取当前步骤绑定的方法。
3. 读取步骤上的方法实例条件节点。
4. 调用对应方法执行。
5. 把产物写入当前结果节点的方法实例结果节点。
6. 若当前结果节点是预测结果，则新生成实际结果节点，不得直接覆写预测结果。
7. 比对结果与目标。
8. 若实际结果与命中的预测结果不一致，则当前任务回到 `筹办`，并进入 `待重筹办`。
9. 若执行方法运行中发现额外前置缺口，只能提出前置任务需求；由自我线程实例化前置任务头并挂回当前步骤，不得由执行直接篡改主步骤链。

执行结束后只允许有三种后续：

1. 目标已达成：当前任务完成。
2. 目标未达成但路径已明确：回到筹办，继续长下一步。
3. 当前路径不可继续：回到筹办，改写任务树。


## 7. 任务树生长闭环

所有任务的统一闭环固定为：

```text
确定目标
-> 筹办完善任务树
-> 执行当前步骤
-> 结果写入结果节点
-> 比对目标
-> 达成则完成
-> 未达成则回到筹办
```

其中：

- 普通任务的树，围绕现实存在特征值生长
- 任务管理任务的树，围绕宿主任务完成度生长
- 学习任务的树，围绕目标方法可用度生长


## 8. 对旧口径的处理

以下旧口径不再作为任务主逻辑保留：

1. 把 `筹办 / 维护 / 纠偏 / 收束` 当成任务统一主流程。
2. 把 `筹办 -> 可执行判断 -> 允许执行判断 -> 执行 -> 反馈` 当成任务统一主流程。
3. 把学习任务理解成独立于任务树的另一套长期状态机主流程。

这些内容若继续保留，只能退为：

- 步骤语义
- 执行细分
- 日志观察口径

不得再作为任务总流程定义。


## 9. 约束下游规范

后续与任务相关的规范，必须满足：

1. 先说明该任务的 `标的` 与 `目标`。
2. 再说明 `筹办` 如何完善这类任务树。
3. 最后说明 `执行` 如何消费当前步骤。

若某份规范无法映射到这三段式，应视为旧口径，必须删改。
