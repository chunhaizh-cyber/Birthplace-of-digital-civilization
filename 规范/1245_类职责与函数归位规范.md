## 4. 核心结论

必须固定以下结论：

1. `通用函数` 只收容不依赖业务归属、可跨模块复用的函数。
2. 新建通用函数前，必须先到 `通用函数模块` 查找现有实现；没有才允许新增。
3. 通用函数一旦新增或新增调用，必须同步更新注释中的用途与调用登记信息。
4. 默认不新建业务结构体来承载程序能力；确有需要时，优先判断能否通过 `虚拟存在 + 特征类型` 落地。
5. 数据操作函数必须在各自归属类中实现，禁止在无关类中越权操作别人的数据。
6. 类必须先明确“自己是哪一型”，再决定能做什么；不得一边做数据真值、一边做调度、一边做线程管理。
7. 任何跨域流程都应由编排类承接，而不是让底层数据类互相越权写对方内部状态。

可收敛为一句话：

`通用函数归通用，数据操作归属地实现，跨域流程走编排，默认不加新业务结构体，新增前先查现有。`

## 5. 类的角色分型

### 5.1 信息定义类

`信息定义类` 指的是：

- 主要承载主信息、枚举、最小比较、序列化、反序列化、轻量帮助函数的类或模块。

当前典型入口：

- [主信息定义模块.ixx](D:\鱼巢\主信息定义模块.ixx)

职责边界：

- 负责定义数据长什么样。
- 负责定义最小比较和最小承载约束。
- 不负责跨域业务流程。
- 不负责线程调度。
- 不负责把多个领域对象编排成闭环。

### 5.2 领域操作类

`领域操作类` 指的是：

- 以某一明确领域对象为主语义归属，负责该对象的增删改查、比较、写回、局部约束和局部流程的类。

当前典型入口：

- [特征类.ixx](D:\鱼巢\特征类.ixx)
- 自我相关模块（当前以 [自我类.ixx](D:\鱼巢\自我类.ixx) 为主，后续拆分模块统一以 `自我_` 开头）
- `状态类.ixx`
- `动态类.ixx`
- `二次特征类.ixx`
- `因果信息类.ixx`

职责边界：

- `特征类` 负责特征增删改查、特征写入、稳态策略、特征集合比较。
- `自我相关模块` 负责自我状态、自我动作、自我内部需求与自我内部治理入口；其中各模块统一以 `自我_` 开头，并共同承接自我的正式能力。
- `动态类` 负责动态记录、起止状态绑定、时间与路径承载、层级与来源链承载。
- `二次特征类` 负责二次特征节点创建、挂接、引用回写和值结果承载。
- `因果信息类` 负责因果信息创建、累计、查询、模板整理。

必须固定：

- 某领域对象的数据操作函数，必须优先放在该领域的归属类中。
- 不允许把 `特征增删改查` 写进 `自我相关模块`。
- 不允许把 `自我状态更新` 写进 `特征类`。
- 不允许把 `特征类型比较方式` 复制进 `二次特征主信息类` 充当平行真值。
- `特征类` 继续负责 `比较方式 / 稳态 / 误差`。
- `动态类` 继续负责 `起止状态 / 时间 / 路径`。
- `二次特征类` 只负责这些上游结果之上的 `二次语义与引用结果`，不负责重新定义上游比较核。
- 不允许把 `因果查询与整理` 塞进不相干的工具类。

### 5.2a 生成器/规则服务类

`生成器/规则服务类` 指的是：

- 负责把既有真值层对象加工成另一层结构化结果，但自身不承担长期真值承载的类或模块。

典型职责包括：

- 二次特征生成
- 动态聚合
- 模板匹配
- 规则分类映射
- 决策前评分或过滤映射

推荐理解为：

- `特征类` 保存值语义、区间、稳态、误差
- `状态类` 保存主体、特征、时点值
- `动态类` 保存状态变化、时间、路径、来源
- `二次特征类` 保存语义结果与引用
- `生成器/规则服务` 负责 `生成 / 聚合 / 匹配 / 分类映射`

必须固定：

1. 二次特征的生成、聚合、匹配，应由专门生成器/规则服务承担，不应塞回数据类。
2. `动态类` 不应偷偷长出跨场景、跨批次的二次特征批量生成逻辑。
3. `二次特征类` 不应偷偷长出模板推理、方法评分或因果匹配主逻辑。
4. `场景类` 不应承担大批量刷新、聚合、匹配的主逻辑，它只负责场景侧承载与引用组织。

### 5.3 编排/管理类

`编排/管理类` 指的是：

- 需要跨多个领域类调度流程、组织顺序、组合结果，但不应直接定义底层真值的类。

当前典型入口：

- [世界树类.ixx](D:\鱼巢\世界树类.ixx)
- `方法管理模块_v0.ixx`
- `任务管理模块_v0.ixx`

职责边界：

- 负责组织多个领域对象协同工作。
- 负责跨类流程编排。
- 不负责篡改各领域类的内部真值定义。
- 不负责把本应放进归属类的 CRUD 全搬出来重写一遍。

### 5.4 线程/运行时类

`线程/运行时类` 指的是：

- 负责线程生命周期、心跳、消息处理、运行时队列和 stop/start/join 边界的类。

当前典型入口：

- `自我线程模块.ixx`
- `本能线程类.ixx`
- `动作线程类.ixx`

职责边界：

- 负责“什么时候运行、何时停止、如何调度节拍”。
- 不负责重新定义业务语义。
- 不负责把领域数据真值私有化成另一套平行结构。

### 5.5 环境/入口类

`环境/入口类` 指的是：

- 提供共享单例、模块入口、上下文绑定、初始化桥接的类或模块。

职责边界：

- 可以提供访问入口。
- 可以维护环境句柄。
- 不应成长为无边界巨型业务类。

## 6. 通用函数模块规范

### 6.1 什么函数才算通用函数

只有同时满足以下条件，函数才允许进入 `通用函数模块`：

1. 不依赖某个特定业务归属类的内部状态。
2. 不依赖某个专有业务结构体或专有节点语义才能成立。
3. 有较高概率被两个及以上模块复用。
4. 提取出来后，不会破坏领域归属边界。

典型可进入项：

- 数学/数值工具
- 位运算、编码解码工具
- 纯字符串/时间/哈希帮助函数
- 无业务归属的轻量容器遍历帮助函数
- 异常保护包装、通用格式化帮助函数

典型不得进入项：

- 自我安全值更新
- 特征节点写入策略
- 因果模板归并
- 方法召回排序
- 任务状态迁移

### 6.2 新建前先查现有

在新建通用函数前，必须：

1. 先到 [通用函数类.ixx](D:\鱼巢\通用函数类.ixx) 搜索是否已有同类函数。
2. 若已有可复用函数，优先复用，不得重复造轮子。
3. 若现有函数只差少量参数或命名不清，优先改造现有函数，而不是平行再建一份。

必须固定：

- `先查现有，再谈新增。`

### 6.3 新增函数的落点

若确认当前没有可复用实现，新增时必须：

- 放进 `通用函数模块`。
- 写清楚函数头注释。
- 说明适用范围与禁止用途。

### 6.4 调用登记与注释纪律

通用函数必须写注释，且后续新增调用时必须同步更新调用登记信息。

推荐固定注释模板：

```cpp
/// 用途：
/// 输入：
/// 输出：
/// 副作用：
/// 适用范围：
/// 禁止用途：
/// 当前调用处数：
/// 当前主要调用点：
```

必须固定：

- 新增通用函数时，必须填写完整注释。
- 每次新增调用时，必须同步修改注释中的 `当前调用处数` 与 `当前主要调用点`。
- 若调用点很多，可写“主要调用模块 + 最近新增调用点”，但不能完全不登记。

### 6.5 通用函数模块不负责

通用函数模块不负责：

- 保存业务真值。
- 直接操作归属类内部私有状态。
- 承担跨线程所有权。
- 变成“任何不好放地方的函数垃圾桶”。

## 7. 数据操作函数归位规则

### 7.1 总规则

数据操作函数必须在各自归属类中实现。

可收敛为：

- `谁的数据，谁的类负责增删改查和局部规则。`

### 7.2 归位示例

- 特征信息的增删改查、比较、稳态处理：放在 [特征类.ixx](D:\鱼巢\特征类.ixx)
- 自我相关状态、自我动作、自我内部需求：放在自我相关模块（模块名统一以 `自我_` 开头；当前以 [自我类.ixx](D:\鱼巢\自我类.ixx) 为主）
- 动态记录与聚合：放在 `动态类.ixx`
- 因果信息的创建、累计、查询：放在 `因果信息类.ixx`
- 节点主信息定义、序列化、最小比较：放在 [主信息定义模块.ixx](D:\鱼巢\主信息定义模块.ixx)

### 7.3 跨域流程怎么处理

若一个流程同时涉及多个归属类，不应把它硬塞进某一个底层归属类。

正确做法是：

- 由编排/管理类承接。
- 由编排类调用各归属类公开接口。
- 不允许编排类绕过归属类，直接改它的内部不变量。

### 7.4 禁止写法

以下写法默认禁止：

- 在 `自我类` 中直接重写一套特征 CRUD。
- 在 `通用函数模块` 中写“顺手更新自我状态”的函数。
- 在无关类里直接操作别的领域类内部链表、缓存和节点状态。
- 为了图快，把跨域流程临时塞进当前正在编辑的类里。

## 8. 新增结构体与新类的准入规则

### 8.1 默认禁止新增业务结构体

必须固定：

- 不新建结构体来用于程序能力的实现。
- 确有需要时，优先判断是否可以通过增加 `虚拟存在的特征类型` 来实现。

这里的“程序能力实现”包括：

- 业务状态承载
- 成长状态承载
- 权限与成熟度承载
- 运行结果账本承载

### 8.2 优先替代路径

新增业务承载前，必须按以下顺序检查：

1. 能否复用现有字段或特征类型。
2. 能否通过 `虚拟存在 + 特征类型` 承载。
3. 能否作为现有归属类中的局部字段解决。
4. 能否作为已有结果结构的补字段解决。
5. 以上都不成立，才讨论新增结构体或新类。

### 8.3 允许的有限例外

以下情况允许新增轻量结构体，但必须控制边界：

- 配置参数结构
- 纯返回结果结构
- 只在单文件内部使用、不会扩散成新真值层的局部组装结构

必须固定：

- 这些例外结构不得成长为新的业务真值仓库。
- 这些例外结构不得替代 `虚拟存在 + 特征类型` 的长期承载职责。

## 9. 注释与接口规范

### 9.1 文件头注释

每个主要类文件建议在文件头至少写清四项：

- 负责什么
- 不负责什么
- 主要依赖谁
- 主要服务谁

### 9.2 公开函数注释

公开函数至少应说明：

- 用途
- 输入
- 输出
- 副作用
- 是否可能创建节点 / 写回 / 加锁 / 启动线程

### 9.3 命名纪律

建议固定：

- 查询函数尽量用 `获取 / 读取 / 查找 / 判断`
- 写操作函数尽量用 `创建 / 写入 / 更新 / 删除 / 同步 / 记录`
- 已加锁路径继续使用 `*_已加锁` 明示语义
- 自我相关模块统一使用 `自我_` 前缀命名，不再旁路新建无前缀的平行“自我小类”

### 9.4 一类数据一个真值归属点

必须固定：

- 同一类业务真值只能有一个归属点。
- 缓存可以有，但缓存不得与真值并列成长。
- 若存在缓存、副本、镜像，必须明确谁是真值、谁是派生物。

## 10. 新增函数/类/结构体前检查表

新增前至少检查以下问题：

1. 这个能力是否已经在现有类或现有通用函数中存在？
2. 它是否真的通用，还是只是当前类的局部能力？
3. 它操作的是谁的数据，归属类是否明确？
4. 它能否通过现有 `虚拟存在 + 特征类型` 落地？
5. 新增后是否会造成真值分叉、职责漂移或跨域越权？
6. 注释、调用登记、边界说明是否已经补齐？

## 12. 硬约束

1. 通用函数只允许收容不依赖业务归属、可跨模块复用的函数。
2. 新建通用函数前，必须先在 `通用函数模块` 中查找现有实现。
3. 新增通用函数和新增通用函数调用时，必须同步更新注释中的用途与调用登记信息。
4. 默认不新建业务结构体来承载程序能力；优先考虑 `虚拟存在 + 特征类型`。
5. 数据操作函数必须在各自归属类中实现，不得在无关类中越权重写。
6. 跨域流程必须优先由编排/管理类承接，不得让底层归属类互相越权。
7. 每个类都应能明确回答：自己是哪一型、负责什么、不负责什么。
8. 同一类业务真值只能有一个归属点，缓存不得与真值并列成长。
9. 自我相关能力应优先归位到自我相关模块，且模块名统一以 `自我_` 开头；不得旁路新建平行“自我小类”。
10. 二次特征的 `生成 / 聚合 / 匹配 / 分类映射` 应优先归位到生成器/规则服务，不得把数据类写成一锅互相借厨房的业务类。
