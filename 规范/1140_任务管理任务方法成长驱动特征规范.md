# 1140 任务管理任务方法成长驱动特征规范

## 1. 目标

本文用于固定以下问题：

- `任务管理任务` 的方法成长，到底主要由哪些特征驱动
- 哪些特征属于 `成长输入`
- 哪些特征属于 `方法运行后的派生产出`
- 哪些特征属于 `成长证据`
- 这些成长驱动特征在统一主链中的位置是什么
- 哪些统计特征必须继续补齐，才能让方法成长进入稳定闭环

本文直接服务于：

- `任务控制总控模块_v0`
- `四域方法树匹配 / 写回 / 生长判定`
- `任务管理任务` 的后续任务化与方法化
- 任务虚拟存在、方法虚拟存在上的治理特征收口

其中“成长需求如何先压缩成方法需求位，再由方法需求位倒推出所需因果证据”，以 [1195_任务管理方法需求位与因果到方法映射规范.md](D:\鱼巢\规范\1195_任务管理方法需求位与因果到方法映射规范.md) 为准。

其中“任务管理特征如何进一步上浮出扩选、健壮、覆盖、恢复等衍生需求”，以 [1206_任务管理衍生需求推导特征映射规范.md](D:\鱼巢\规范\1206_任务管理衍生需求推导特征映射规范.md) 为准。

其中“这些成长驱动特征如何在主体级汇总、分身继承、本能能力缺口上继续承载”，以 [1228_任务管理主体虚拟存在与分身继承规范.md](D:\鱼巢\规范\1228_任务管理主体虚拟存在与分身继承规范.md) 为准。


## 2. 基本判断

必须先固定以下 7 条：

1. `任务管理任务` 自身也是任务，不是系统外侧的额外管理者。
2. `任务管理任务` 的方法成长，主要由 `内部治理特征` 驱动，而不是由外部设备动作直接驱动。
3. 这些驱动特征，大多属于 `自我内部动作` 所读写的内部治理特征。
4. 若其中某些特征只能在运行时钩子上被动更新，则它们进一步属于 `先天记账动作` 对应的内部特征写回。
5. 外部场景事实、外设事实可以作为上下文，但通常不是 `任务管理方法成长` 的主驱动轴。
6. 方法成长的真正闭环，不是“成功过一次”，而是“在相似输入条件下稳定推动任务前进并留下闭环证据”。
7. 方法成长只能建立在 `根层已允许推进`、且任务管理上下文已经形成的治理运行上。

### 2.1 在统一主链中的位置

按当前最新体系，方法成长驱动特征必须落在下面这条链上：

```text
根层方向裁决
-> 形成任务管理上下文
-> 单步治理执行
-> 产出结构推进、管理派生和成长证据
-> 多轮累计形成成长驱动特征
-> 支撑方法出生 / 验证 / 升格
```

必须固定：

1. 成长驱动特征不属于根层方向本体。
2. 它们是任务管理层在多轮治理后沉淀出来的成长原料。
3. 它们最终服务于方法成长，不服务于改写根约束。
4. 当系统启用了主体层，这些成长原料允许先在 `宿主任务虚拟存在` 沉淀，再回并到 `任务管理主体虚拟存在`；但主体级聚合不替代本文的五层成长特征口径。


## 3. 成长闭环主链

`任务管理任务` 的方法成长，后续统一按以下主链理解：

```text
根层允许继续推进
-> 输入条件与任务管理上下文
-> 进入某一治理域
-> 命中或调用某个管理方法
-> 推动任务结构发生可检索推进
-> 宿主任务进入更前进或更明确的状态
-> 产出结果节点或可结算叶子结果
-> 在相似条件下稳定复现
-> 方法树沉淀或升格
```

因此，驱动成长的特征不能只盯着“方法本身”，而必须覆盖：

- 进入方法前的上下文
- 方法执行时的治理判断
- 方法执行后的结构推进结果
- 运行后的成长证据
- 方法自身的稳定度与可靠度统计


## 4. 驱动特征的五层口径

## 4.1 前置上下文特征

这类特征是在进入四域方法树匹配前就应具备的上下文。

它们是 `方法是否命中当前条件` 的第一层驱动特征。

最小集合固定为：

- `最近根层裁决结果`
- `当前功能域`
- `任务虚拟存在`
- `方法虚拟存在`
- `当前任务头结点 / 当前步骤结点 / 当前方法首节点`
- `当前现实场景`
- `需求目标状态`
- `安全值`
- `服务值`
- `需求值`
- `当前预算`
- `最近结果稳定`
- `最近执行失败`
- `风险变坏`
- `检查点通过`
- `依赖缺失`
- `需要新子任务或新步骤`
- `关键方法不再可用`
- `已满足完成判据`

说明：

- 这层特征的职责是回答“现在像不像一个已知治理条件”。
- 这层特征不负责直接证明方法已经成长成功。
- 这层特征缺失时，不应强迫单个原函数伪造上下文真值。
- 其中 `最近根层裁决结果` 用于区分“本轮属于普通治理路径”还是“本轮已进入收束/停机边界处理”。


## 4.2 任务结构推进特征

这类特征回答的是：

- 这个管理方法有没有真的推动任务树前进
- 它推进到了哪里
- 推进后留下了什么结构痕迹

最关键的特征包括：

- `任务_状态`
- `任务_当前步骤ID`
- `任务_当前子任务头指针`
- `任务_当前分支动作`
- `任务_当前方法首节点指针`
- `任务_最近方法首节点指针`
- `任务_最近结果节点指针`
- `任务_运行次数`
- `任务_成功次数`
- `任务_失败次数`
- `任务_连续失败次数`
- `任务_转入学习次数`
- `任务_最近错误码`
- `任务_最近开始时间`
- `任务_最近结束时间`
- `任务_最近调度时间`
- `任务_基准优先级`
- `任务_局部优先级偏移`
- `任务_调度优先级`
- `任务_绑定全局容器指针`

说明：

- 这层特征不是简单的运行日志，而是 `方法是否推动任务结构前进` 的可检索证据面。
- 对 `任务管理任务` 来说，能否让步骤、子任务、结果节点、当前指针发生合理推进，是成长判断里的高权重信号。
- 这层特征大多属于内部治理特征，不属于外部动作特征。


## 4.3 管理产出与派生特征

这类特征是管理类原函数或域内方法在一次运行后直接产出的治理判断。

它们不是最初输入，但会成为：

- 后续写回方法树的派生信息
- 下一轮复用时的二次特征
- 方法树成长与分化的重要依据

最小集合固定为：

- `条件缺口分类`
- `可执行性判断`
- `推荐方法指针`
- `候选方法评分`
- `推荐分解方向`
- `是否需要补条件任务`
- `是否需要观察任务`
- `是否需要恢复任务`
- `是否建议等待学习`
- `是否建议回到筹办`
- `失败原因码`
- `评分证据`
- `二次特征证据`
- `结果主键`

说明：

- 这层特征是 `管理方法对当前上下文做出的治理解释`。
- 它们决定同一个输入条件下，该方法到底是“继续维持”“回退重筹办”“进入纠偏”“等待学习”还是“进入收束”。
- 它们也是未来形成 `输入条件 -> 治理结论` 经验枝的直接材料。


## 4.4 成长证据特征

这类特征不负责帮助方法“被选中”，而负责帮助方法“被认定已经成长出价值”。

最小成长证据固定为：

- `治理所属域`
- `输入条件主键`
- `推动宿主任务进入的状态`
- `是否最终产出可结算叶子结果`

若进入价值成功证据层，还应至少补齐：

- `所属根树`
- `结算增量`

说明：

- 没有 `输入条件主键` 的成功，不能稳定沉淀成可复用治理经验。
- 没有 `宿主任务进入状态` 的成功，无法判断该方法是否真的推动了任务闭环。
- 没有 `是否最终产出可结算叶子结果`，就只能算运行成功，不能算价值成功。


## 4.5 方法可靠度与学习反馈特征

这类特征回答的是：

- 方法在相似条件下是不是稳定
- 方法是不是还在观察期
- 方法结果有没有漂移
- 方法是否值得在后续更积极复用

当前最关键的特征包括：

- `历史成功率`
- `方法连续稳定次数`
- `方法结果分裂次数`
- `方法尝试状态`

当前已经补成标准特征类型的包括：

- `方法尝试次数`
- `方法稳定出现次数`
- `方法可靠度_Q10000`

与学习反馈直接相关、也应纳入成长驱动面的，还有：

- `学习阶段码`
- `学习目标方法指针`
- `尝试学习_待处理`
- `尝试学习_请求时间`
- `尝试学习_来源优先级`
- `尝试学习_错误码`
- `尝试学习_来源任务头指针`
- `尝试学习_来源步骤指针`
- `尝试学习_来源方法指针`
- `尝试学习_来源结果指针`

说明：

- 对任务管理方法来说，`成长` 不只来自正常治理链，也来自失败后的等待学习与反馈验证链。
- 因此学习来源信息不是外围附属信息，而是成长驱动的一部分。


## 5. 最小驱动总表

| 特征层 | 代表特征 | 主要回答的问题 | 是否属于成长核心 |
| --- | --- | --- | --- |
| 前置上下文特征 | 当前功能域、任务虚拟存在、当前预算、最近结果稳定 | 这个条件像不像某类已知治理场景 | 是 |
| 任务结构推进特征 | 当前步骤、当前子任务、最近结果节点、任务状态 | 方法有没有真的把任务推进 | 是 |
| 管理产出与派生特征 | 条件缺口、推荐方法、评分、分解方向、结果主键 | 方法对当前场景做了什么治理判断 | 是 |
| 成长证据特征 | 治理域、输入条件主键、宿主进入状态、可结算叶子结果 | 这次运行是否值得沉淀为成长证据 | 是 |
| 方法可靠度与学习反馈特征 | 连续稳定次数、结果分裂次数、尝试状态、等待学习来源 | 这个方法后续该多大胆地复用 | 是 |
| 单次表面成功 | 仅一次成功回执 | 这次运行是否没报错 | 否，单独使用不足以驱动成长 |
| 外设局部参数 | 相机局部参数、一次性设备状态 | 外设是否正常 | 通常不是任务管理方法成长主驱动 |


## 6. 哪些不是主驱动特征

以下信息可以参与上下文，但不应被误当成 `任务管理方法成长` 的主驱动：

- 没有条件主键绑定的一次性成功
- 只有“执行过”但没有任务结构推进痕迹的运行记录
- 没有宿主状态推进的局部回执
- 纯外设参数或纯设备读数
- 直接把 `安全值`、`服务值` 的变化当成任务管理方法自己的产出

必须注意：

1. `任务管理任务` 不能直接写 `安全值`、`服务值`。
2. 但它可以把 `安全值 / 服务值 / 需求值 / 预算` 当成上下文。
3. 因此根值更像 `治理输入`，而不是 `治理产出`。


## 7. 当前代码落点对齐

当前仓库中，这套特征已经分散落在以下位置：

1. `四域前置上下文特征 + 原函数产出特征`
- 已在 `1110` 中固定口径。

2. `任务结构推进特征`
- 已在 `任务类` 中形成一批稳定字段与动作动态写回口。

3. `四域判定输入`
- 已在 `自我线程类::私有_组装根任务控制判定输入` 中组装。
- 已在 `任务控制总控模块_v0` 中作为最小输入结构消费。

4. `成长证据`
- 已在 `任务控制总控模块_v0` 中生成并同步到任务事实。
- 已在 `四域方法树生长判定器_v0` 中用于记录治理成长证据与价值成功证据。

5. `方法可靠度与学习反馈特征`
- 当前已有 `方法尝试状态 / 方法结果分裂次数 / 方法连续稳定次数`。
- 现在也已补齐 `方法尝试次数 / 方法稳定出现次数 / 方法可靠度_Q10000`。


## 8. 后续补齐优先级

若下一步要开始改代码，建议按以下顺序补齐：

1. 先保证 `任务管理任务` 的每次治理运行都能产出完整的 `成长证据`。
2. 再保证每次治理运行后，都能在任务树上留下 `步骤推进 / 结果节点 / 宿主状态变化` 之一。
3. 再补更细的评分证据、二次特征证据和价值成功证据统计。
4. 最后把任务管理方法专用实例因果与轻量因果链正式接上。


## 9. 一句话收口

可收敛为一句话：

`任务管理任务的方法成长，主要由内部治理特征驱动；最小必须同时具备“前置上下文 + 任务结构推进 + 管理派生产出 + 成长证据 + 方法可靠度回流”五层特征，才足以形成稳定的自成长闭环。`
