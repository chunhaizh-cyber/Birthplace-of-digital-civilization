## 3. 定位

当前工程中应固定：

`自我 = 根约束之下持续运行的统一主体在实现层的运行时承载`

它的总定义、服务与安全的分权关系、差额判定核心、任务管理中轴地位，以及回流必须回根层结算的总口径，以上位 [0011_自我架构总定义规范.md](D:\鱼巢\规范\0011_自我架构总定义规范.md) 为准；本文继续只展开“这套总纲在实现层如何闭环落地”。

它至少同时承担四个角色：

1. `状态承载者`
   承载自我的核心状态、自我存在、自我内部世界以及关键特征入口。

2. `方向维持者`
   承载与维护 `服务值 / 安全值 / 待机 / 时序步长 / 主方向锚点` 等最小闭环控制面。

3. `需求-任务-方法-学习闭环组织者`
   负责从需求选择、任务落实、方法召回、方法执行、结果反馈、学习补缺，到下一轮调度的最小运行链。

4. `内部治理入口`
   负责内部尝试学习、内部保底维护、自我动作动态登记等治理动作。

因此必须明确：

- `自我` 不是新的根目的定义器。
- `自我` 不是任意跨域黑盒管理器。
- `自我` 不是平行于世界树、需求树、方法树之外的第二真值系统。


## 4. 自我的实现总闭环

当前自我闭环的起点必须固定为两个一级根需求：

- `服务需求`
- `安全需求`

它们分别来源于：

- `服务值的目标值 - 当前值`
- `安全值的目标值 - 当前值`

当前实现逻辑必须统一理解为以下闭环：

1. `初始化自我`
   建立自我存在、自我内部世界、自我特征承载和默认运行面。

2. `进入运行态`
   启动自我主循环和工作者承载能力。

3. `主循环统一消费回流`
   消费任务执行线程、学习线程、宿主线程与外部回流消息，并先做结果归并。

4. `执行固定先天机制`
   统一触发服务衰减、待机保护、风险回归等固定镶嵌机制。

5. `根层重判`
   基于当前值、需求值、预测或实际结果方向、禁止边界形成 `最近根层重判结果`。
   这里直接生成的是 `根层重判结果` 与上位允许边界，不是凭空直接生成因果。

6. `需求治理`
   整理需求、合并重复需求、刷新权重，并确定当前主需求。

7. `确保根任务与常驻治理任务存在`
   至少确保：
   - `安全根任务`
   - `服务根任务`
   - `任务管理任务`
   - 必要时的 `学习承接任务`

8. `装配任务管理上下文`
   把 `最近根层重判结果`、`当前主需求`、`当前管理对象`、`当前缺口类型`、`当前下一步去向候选` 等最小稳定字段装配成承接层输入。

9. `提交任务管理任务做一步治理`
   由任务管理任务承接宿主任务的一步推进，产出：
   - 一步治理结果
   - 治理证据
   - 去向候选
   - 必要时的执行请求或学习候选

10. `叶子执行与学习承接`
    若本轮进入执行链、等待链或学习链，则由对应承载层完成一步消费，并把结果以消息形式回流主循环。

11. `结果回收与双值结算`
    主循环统一把实际结果落成带值特征，生成实际结果方向，与需求方向逐特征比较，并据此完成双值结算、经验项后果回写、禁止项后果回写与需求刷新。

12. `外层再重判与最终去向决定`
    在停止条件或死亡边界未触发前，主循环统一决定：
    - `继续提交`
    - `回到重筹办`
    - `转入学习`
    - `进入收束`
    - `待机`
    - `停止`

必须补充固定：

- `任务管理任务` 是当前自我闭环中的治理承接中轴，不再允许把“需求直接连到方法执行”当成默认正式主链。

## 5. 初始化阶段规范

### 5.1 初始化必须完成的承载面

`初始化自我` 至少必须完成以下动作：

1. 建立当前活动自我锚点。
2. 建立自我内部世界与自我存在。
3. 初始化自我相关特征类型与默认承载。
4. 初始化时序步长默认值。
5. 注册最小可运行所需的默认模板或等价基础依赖。

必须固定：

- 自我在启动前，必须已经具有 `可读写` 的自我存在承载面。
- `安全值 / 服务值 / 物理安全 / 风险安全 / 当前主需求 / 当前主任务 / 当前主方法 / 待机状态 / 时序步长` 等核心入口，必须已经有稳定特征承载。
- 初始化阶段负责“确保承载已经存在”，不负责在此阶段把所有运行期任务都提前执行完。

### 5.2 初始化的真值口径

必须固定：

- 初始化阶段只负责把默认真值写到 `自我存在特征` 上。
- 运行期的正式读写，应继续以 `世界树中的自我特征` 为准。
- 不允许把结构体缓存扩张成与世界树并列的长期真值层。

因此：

- 结构体、原子变量、临时缓存可以服务运行时效率。
- 但正式业务真值必须能追溯回 `自我存在` 及其特征。

### 5.3 初始化不固定的事项

本文不固定以下内容：

- 具体默认数值比例
- 具体模板细节
- 具体日志格式

这些应由上位语义或下游实现细则决定，不能在本文中反向发明。


## 6. 双值核、时序与待机规范

### 6.1 双值核的收口地位

`自我` 必须以 `安全值 + 服务值` 作为最小双值核。

必须固定：

- `安全值` 直接决定是否还活着。
- `服务值` 直接决定当前服务成立程度。
- `自我` 的最初直接控制目标，是让 `安全值` 与 `服务值` 分别持续逼近各自目标最大值。
- `服务值` 可以通过时序步长间接塑造 `安全值` 的未来演化。
- `物理安全 / 风险安全` 属于安全面的分解观察口，不替代根安全值本身。

### 6.2 自我中的统一更新入口

凡是对 `安全值 / 服务值` 的更新，都必须通过自我的统一入口落地。

必须固定：

- 对双值的增加、减少、钳制、归零保护，应统一收束到自我归属入口。
- 不允许在外部模块中绕过自我归属类，直接把本应属于自我控制面的更新写成散落逻辑。
- 数值写回后，应同步保持自我时序和待机状态的一致性。

### 6.3 时序规则

自我的时序规则至少包含以下三类：

1. `服务时序衰减`
   在没有外部推进、没有需求闭环收益时，服务能力应随时间持续衰减。

2. `风险安全回归`
   风险安全属于可回归面，允许按规则向目标值回归。

3. `服务归零待机保护`
   当服务值归零时，自我进入待机保护，安全值进入最小非零保底线，而不是直接死亡。

必须固定：

- `服务值 = 0` 表示进入待机边界，不等于死亡。
- `安全值 = 0` 才是死亡边界。
- 时序维护动作属于 [1120_自我动作分类与调用边界规范.md](D:\鱼巢\规范\1120_自我动作分类与调用边界规范.md) 中的 `内部先天动作`，不得伪装成普通任务叶子动作。


## 7. 需求、任务管理、方法与执行主链

### 7.1 需求治理与主需求刷新

每一轮主循环中，自我必须先完成 `需求治理`，再形成当前主需求。

当前应固定：

- 自我可以在 Tick 内进行轻量需求整理，例如去重、合并、权重维护。
- 自我随后应从需求集中刷新 `当前主需求`。
- `安全需求` 与 `服务需求` 作为一级根需求必须常驻；正常运行态下，`当前主需求` 不得为空。
- 若运行态里出现 `当前主需求 = 空`，应优先视为低级代码逻辑错误、需求治理断链或上下文装配错误，先进入诊断/纠错路径，不得把它当成正常治理分支使用。
- `需求` 本身自带 `目标值 / 当前值 / 需求方向`；需求方向必须来自比较，不得直接凭标签拍定。

### 7.2 任务管理任务承接

当存在待推进需求且根层允许继续承接时，自我不得直接跳过治理中轴，而必须先装配 `任务管理上下文` 并提交 `任务管理任务`。

必须固定：

- 当前正式主链是：
  `需求治理 -> 任务管理上下文 -> 任务管理任务一步治理 -> 一步治理结果 -> 外层再重判`
- `任务管理任务` 负责：
  - 判当前治理域
  - 提炼主缺口类型
  - 生成下一步去向候选
  - 组织任务树结构
  - 组织方法挂点
  - 下发执行或进入等待/学习承接
- `任务管理任务` 不负责：
  - 根层重判
  - 双值统一结算
  - 需求整理与权重合并
  - 最终去向拍板

因此必须明确：

- `自我主循环` 是总协调者。
- `任务管理任务` 是治理承接者。
- `任务执行器` 是叶子执行承载者。
- `学习承接链` 是缺口补齐承载者。
- `根任务` 与 `宿主任务` 默认优先沿现有树继续推进，而不是每轮从头梳理；更细规则按 [0975_根任务宿主任务与任务树保留复用规范.md](D:\鱼巢\规范\0975_根任务宿主任务与任务树保留复用规范.md) 执行。

更细的职责与字段表，按 [1115_任务管理任务最小职责表规范.md](D:\鱼巢\规范\1115_任务管理任务最小职责表规范.md)、[1213_任务管理一步治理结果最小返回面规范.md](D:\鱼巢\规范\1213_任务管理一步治理结果最小返回面规范.md)、[1214_任务管理上下文最小字段表规范.md](D:\鱼巢\规范\1214_任务管理上下文最小字段表规范.md) 执行。

### 7.3 方法挂点、方法存在与执行承接

方法不再作为“主循环直接按主键字段粗召回后就跑”的默认主链节点，而是应先由任务管理任务组织为 `当前方法挂点` 或 `方法候选`。

必须固定：

- `方法存在 / 方法可用 / 方法可执行 / 方法成熟(升格)` 是四层概念，不得混写。
- 方法头出生不要求当前一定可执行；只要当前已能把方法结构化表达出来，就允许承认其存在。
- `需求主键 / 主作用特征类型与作用方向 / 结果主键` 只负责粗召回、归桶和候选检索，不得直接当成方法正确、需求满足或允许执行的结论。
- 真正的 `当前可执行方法`，必须建立在：
  - 根层已放行
  - 上下文已到可执行态
  - 局部条件判定已成立
  - 执行前门控未拒绝
  这几层共同成立之上。
- 若当前没有可执行方法，自我必须把它显性视为 `方法缺口 / 条件缺口 / 学习缺口`，而不是静默吞掉。
- 即使当前没有可执行方法，也不等于“没有方法存在”；自我应允许知道“某方法存在，但当前不可用/不可执行”。
- `方法存在 / 方法可用 / 方法可执行 / 方法成熟(升格)` 的统一边界，按 [0568_方法存在可用可执行成熟分层规范.md](D:\鱼巢\规范\0568_方法存在可用可执行成熟分层规范.md) 执行。
- 若存在更好的方法候选，允许切换当前方法挂点；切换不等于删除旧方法。
- `无方法` 或 `方法不可用` 默认不应触发“学习成功回流 -> 再次缺方法 -> 再学习”的自旋链；应先把对应 `待学习方法` 或 `方法需求位` 挂起，等待再次观察、条件变化、或记录到该方法被真实用到后再重判。
- 被挂起的 `待学习方法` 必须至少具备 `动作骨架`，以及一份 `条件侧`；其中 `条件侧` 可以来自现有上下文，也可以由学习任务显式补给。
- 若连 `动作骨架` 都不存在，则此时只能上浮 `方法缺口 / 扩选压力`，不得直接把“无方法”当成可执行学习任务。
- 挂起中的 `待学习方法` 应进入可观察状态，并在控制面板或等价任务信息窗口中显式标出，避免被静默吞掉。

### 7.4 执行与结果回收主链

当前最小闭环中，自我可以通过工作者承载具体方法执行，但执行前后都必须回到统一规则链。

必须固定：

1. 自我主循环负责：
   - 外层根层重判
   - 上下文装配
   - 承接一步治理结果
   - 最终是否继续的决定
2. `任务管理任务` 负责：
   - 给出本轮执行候选或等待/学习候选
3. 工作者、执行器和学习承载层负责：
   - 消费叶子执行
   - 返回实践结果消息
4. 超时、异常、失败、无推进、反向都必须回到主循环统一回收，不得遗失。

在进入普通执行前，还必须完成 `执行前门控`：

- 只有 `根层重判结果` 允许、上下文允许、局部条件允许时，候选才允许进入普通执行链。
- 方向相反、会扩大差值、或会破坏安全边界的候选，不得执行。
- 若候选动作直接作用于另一种特征类型，则必须先证明其结果方向能够通过因果回投到 `安全 / 服务` 两根需求之一，才允许执行。
- `局部可执行` 不等于 `根层允许执行`；前者永远不能替代后者。
- 不得以权重、经验、临时偏好或实现方便，跳过方向裁决直接放行。

当前允许的最小实现是：

- 先只执行可直接调用的本能函数方法
- 以后再扩展为更完整的方法执行形态

但不允许：

- 把“当前实现简化”为“可以跳过方法层”
- 把“任务做过一次”直接冒充正式结算真值

## 8. 内部学习与内部治理规范

### 8.1 内部学习的地位

自我中的 `内部学习承接` 必须固定为：

`任务管理内部治理子链在自我实现层的正式承接入口`

它的作用是：

- 承接方法缺口、条件缺口、观察缺口和可靠性缺口
- 组织学习对象、参数、适配器与最小试探
- 把学习产出回流为候选能力、候选方法、候选条件或继续治理信号
- 重新补齐 `需求 / 任务管理任务 / 方法 / 执行` 闭环中当前缺失的那一段

它不是：

- 第三根目的
- 脱离服务与安全的独立长期总任务
- 可以绕过根约束的自由实验区
- 挂在主闭环外部、只靠人工感觉触发的外挂流程

### 8.2 内部学习的出生条件

内部学习承接至少可以由以下条件触发：

- 当前主缺口类型指向 `方法 / 条件 / 观察 / 解释 / 可靠性` 缺口
- 当前候选方法都不可执行
- 某需求长期得不到推进
- 一步治理结果提出 `学习子任务候选`
- 学习压力升高，表明当前存在明确补缺压力

并且必须补充固定：

- 只有当某个 `待学习方法` 已具备 `动作骨架`，且 `条件侧` 已存在或可由学习任务补给时，才允许把它升成正式学习承接对象。
- `当前只是无方法 / 方法不可用` 本身，只能先形成 `挂起等待 / 待学习方法记录 / 扩选压力`，不自动等于立即进入学习执行态。

### 8.3 内部学习的最小闭环

当前自我中的内部学习闭环可固定为：

1. 主循环或任务管理任务显性化学习缺口。
2. 生成学习承接任务或学习子任务候选。
3. 组织学习对象、参数场景与适配器。
4. 将学习请求入队，由学习承载层消费。
5. 学习结果以消息回流主循环。
6. 主循环统一决定：
   - 是否形成方法存在面补齐
   - 是否形成可用性/可靠性候选
   - 是否回写学习压力或更保守治理路径
   - 是否进入下一轮再重判

必须固定：

- 内部学习是 `渐进治理`，不是每轮暴力穷举。
- 学习结果是否能正式沉淀，仍要服从 `1207–1227` 与 `1270` 的上位写回规则。
- 学习反馈本身不是最终裁决；反馈被消费后仍必须回到主循环做外层重判。
- 自我内部学习的主结果首先落在内部治理域，而不是直接冒充对外服务结果。
- 每次学习尝试的结果允许先直接写入方法与记忆整理，但这不自动等于默认执行或成熟度升格。


## 9. 自我动作动态与可追溯性规范

### 9.1 为什么必须记录

自我的关键内部动作如果没有动作边界和结果记录，就会失去后续因果组织、问题追踪和学习反馈的真值基础。

因此必须固定：

- 自我发起的关键内部先天动作和关键内部动作，应尽量形成 `动作动态记录`。
- 至少要能追溯：动作名、开始/结束时间、成功与否、输入/输出场景、初始状态、结果状态。
- `内部先天动作` 不属于自我可主动点名调用的方法；自我只能读取其运行结果、结果状态与动作动态，并据此组织后续因果信息。
- 纳入固定先天机制清单的内部先天动作，必须由 `自我线程` 每个周期自动调用一次。
- 每次调用都必须生成一条对应的 `本能动作动态`，并把本轮主结果挂到这条动态上。

### 9.2 哪些动作应优先记录

当前至少以下动作应优先具备动态记录能力：

- 服务时序衰减
- 按双值重算服务衰减步长
- 按服务值重算安全增加步长
- 按服务值重算安全减少步长
- 安全低位回升
- 安全高位回落
- 风险安全回归
- 服务归零待机保护
- 内部学习触发与学习结果反馈
- 后续所有会直接改写自我核心特征的关键内部动作

### 9.3 动作分类边界

本文中的自我动作动态，必须服从 [1120_自我动作分类与调用边界规范.md](D:\鱼巢\规范\1120_自我动作分类与调用边界规范.md)：

- 内部先天动作不能被普通任务误当成主动方法调用

- 同一个原子动作不得同时承担内部主结果写入与外部主结果写入


## 10. 启停与停止边界

### 10.1 启动前提

`启动` 之前，至少必须确保：

- 自我已经初始化
- 核心特征承载已建立
- 最小可运行依赖已具备

不得在未完成初始化时直接进入运行态。

### 10.2 运行中停止

自我的停止至少分为两步：

1. `请求停止`
   发出停止意图，并向工作者传播停止信号。

2. `等待停止`
   回收自我主线程和工作者线程，收束生命周期。

### 10.3 退出边界

必须固定：

- `安全值 = 0` 时，自我进入死亡退出边界。
- `服务值 = 0` 时，只进入待机保护，不直接死亡。
- 外部显式停止请求、调试上限、不可恢复的运行态约束，也可以构成退出条件。

### 10.4 异常路径

异常路径必须固定为：

- 异常不是“什么都没发生”
- 异常应至少形成失败反馈和风险后果
- 异常不得悄悄吞掉后继续冒充正常闭环

更细的 `失败反馈` 与 `风险后果` 分层，按 [0324_零增量与负面后果边界规范.md](D:\鱼巢\规范\0324_零增量与负面后果边界规范.md) 执行。

## 13. 硬约束

1. `自我` 只能在根目的与安全边界之下运行，不能自创新根目的。
2. `安全值 = 0` 是死亡边界，`服务值 = 0` 只是待机边界，两者不得混淆。
3. 自我闭环的起点固定为 `安全需求 / 服务需求` 两个一级根需求，不允许并列新增第三根需求树。
4. 自我的正式业务真值必须可追溯到 `自我存在及其特征`，不得扩张出平行真值系统。
5. 内部先天动作不得伪装成普通任务叶子动作或对外服务动作。
6. 需求推进必须优先经过 `需求 -> 任务 -> 方法 -> 执行 -> 反馈 -> 学习` 主链，不能长期绕开方法层或学习补缺链。
7. 只有方向一致，或能通过合法因果回投到根方向的候选，才允许执行；方向相反者不得执行。
8. `因果` 是不同特征类型之间方向能够合法对齐的唯一桥接准则，不得用名称相似、权重高低或人工希望替代。
9. 当前任何局部删除需求、局部奖励与负面后果处理、局部简化执行，都不得反向改写 `0310` 的正式结算规则。
10. 内部尝试学习只能是内部治理子链，不能升格为第三根需求树。
11. 自我中的关键内部动作必须逐步具备动作动态与结果可追溯能力。
12. 新增自我能力时，默认优先通过 `虚拟存在 + 特征类型 + 归属类函数` 落地，不默认新增业务结构体。
13. 任何实现方便性都不得用来推翻 `0100–0500` 的上位根约束。
