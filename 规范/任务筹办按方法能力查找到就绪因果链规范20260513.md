## 1. 标准因果句式

任务筹办按方法能力查找到任务就绪，使用以下因果链模板：

```text
要任务 T 就绪，
需要用方法 M。

因为方法 M 的结果能力能够改变：
    目标宿主 X 的目标特征 F

但方法 M 要可用于任务 T，
还需要条件 C1 / C2 / C3 成立。

所以需要派生更细需求 D_child，
让这些条件成立。

D_child 被子任务满足后，
父任务 T 重新筹办，
再次查找 / 绑定方法 M，
最终形成可就绪回执，
再由提交任务状态变化方法把任务状态推进到就绪。
```

## 2. 父任务目标

父任务 `T` 来源于需求 `D`：

```text
D 的目标：

    目标宿主 = X
    目标特征类型 = F
    目标状态 = V_target
    满足关系 = R
```

也就是：

```text
需要让 X.F 达到 V_target。
```

任务 `T` 要就绪，首先要找到一个方法：

```text
M.结果能力 能够产生 X.F -> V_target 相关结果。
```

这一步叫：

```text
按方法结果能力查找候选方法。
```

查找依据不是方法名字，而是：

```text
哪个方法声明自己能够改变目标特征类型 F。
```

方法头只用于候选召回。真正执行仍必须看：

```text
条件节点
结果节点
条件结果对
方法虚拟存在
方法状态
```

候选方法选择必须先区分“可直接达成”与“可补齐推进”：

```text
可直接达成：
    方法自身条件全部满足，参数可绑定，禁止项未命中，结果能力确定命中目标。

可补齐推进：
    方法自身不能直接完成父任务目标，但能够推进目标方法的结构、条件结果对、参数或方法状态。
```

直接达成方法优先于可补齐推进方法。多个直接达成方法同时存在时，按：

```text
禁止项少
风险少
前置冲突少
效率高
耗时少
能耗少
资源占用少
副作用少
稳定性高
```

依次排序。

`任务筹办` 不能作为过程推进方法声明，也不能进入候选方法集：

```text
任务筹办不是本能方法；
任务筹办不是候选方法；
任务筹办不声明“目标方法.方法状态 -> 可用”的方法结果能力。
```

当来源需求目标是：

```text
目标方法.方法状态 -> 可用
```

任务筹办只负责单步查找、填充候选集、选择当前方法并形成回执。
候选方法应来自能够推进方法可用性的真实方法，例如：

```text
尝试学习
定向学习
练习
建立条件结果对
提交方法状态变化
```

其中能够直接推进当前阶段、且条件满足的方法优先。
`任务筹办` 本身不参与这次选择；它只在下一轮读取这些方法产生并回流的事实。

## 3. 直达就绪路径

如果存在方法 `M`：

```text
M.结果能力.结果项集 包含：
    特征类型 = F

并且：

M.方法状态 = 可用
M.有执行入口
M.有条件结果对
M.条件节点可由任务输入场景满足
M.参数可绑定
```

则因果链是：

```text
要任务 T 就绪，
需要用方法 M。

因为方法 M 的结果能力能够作用于：
    X.F

并且方法 M 的条件已经满足：
    方法状态 = 可用
    执行入口存在
    条件结果对存在
    参数可绑定
    可执行输入参数场景可构造

所以任务筹办生成可就绪回执：
    当前方法 = M
    可执行输入参数场景 = S_exec
    筹办缺口类型 = 无
    建议任务状态 = 就绪
```

提交链：

```text
任务筹办
    -> 运行存在中形成“可就绪回执”

同步筹办回执到任务
    -> 写入任务虚拟存在：
        当前方法 = M
        可执行输入参数场景 = S_exec
        筹办缺口类型 = 无

提交任务状态变化
    -> 写入：
        T.任务状态 = 就绪
    -> 生成任务状态动作动态
```

## 4. 无候选方法路径

如果任务筹办按能力查找后发现：

```text
不存在结果能力命中 F 的方法
```

则不是失败，而是需求树继续生长：

```text
要任务 T 就绪，
需要用方法 M。

因为任务目标是：
    X.F -> V_target

所以需要存在一个方法 M，
其结果能力能够改变特征类型 F。

但当前方法库中没有确定候选方法。

所以需要派生子需求 D1：
    存在一个可改变目标特征类型 F 的方法。
```

动作链：

```text
任务筹办
    -> 形成筹办缺口回执：
        筹办缺口类型 = 无确定候选方法
        派生需求提示 = 学习或生成可改变 F 的方法

派生需求入树
    -> 新增子需求 D1：
        被需求存在 = 方法仓库 / 方法虚拟存在
        目标特征类型 = 方法结果能力
        目标状态 = 存在可改变 F 的方法

承接需求创建任务
    -> 创建子任务 T1

子任务 T1 通过学习链形成方法 M

提交方法状态变化 / 建立条件结果对
    -> 方法 M 达到可用条件

回写需求满足
    -> 登记 D1 已满足

唤醒任务重筹办
    -> 父任务 T 进入待重筹办

任务筹办再次运行
    -> 这次能查到 M
    -> 形成可就绪回执

同步筹办回执到任务
提交任务状态变化
    -> T.任务状态 = 就绪
```

## 5. 候选方法存在但不可用路径

如果查到方法 `M`，但：

```text
M.方法状态 != 可用
```

则因果链是：

```text
要任务 T 就绪，
需要用方法 M。

因为方法 M 的结果能力命中了：
    X.F

但方法 M 要被用于任务 T，
需要：
    M.方法状态 = 可用

当前：
    M.方法状态 = 待学习 / 待补齐 / 未可用

所以需要派生子需求 D2：
    M.方法状态 = 可用
```

为了让 `M.方法状态 = 可用`，又需要：

```text
M 有执行入口
M 有条件节点
M 有结果节点
M 有至少一组有效条件结果对
```

如果缺条件结果对：

```text
要 M 可用，
需要建立条件结果对。

因为方法可用最低条件是：
    至少存在一组有效条件结果对。

所以需要调用：
    尝试学习
    建立条件结果对
    提交方法状态变化
```

动作链：

```text
任务筹办
    -> 筹办缺口类型 = 候选方法不可用
    -> 派生需求提示 = 方法状态 = 可用

派生需求入树
    -> D2：M.方法状态 = 可用

承接需求创建任务
    -> 子任务 T2

尝试学习
    -> 形成条件结果对候选

建立条件结果对
    -> 写入 M.条件结果对集合
    -> 生成条件结果对动作动态

提交方法状态变化
    -> M.方法状态 = 可用
    -> 生成方法状态动作动态

回写需求满足
    -> D2 满足

唤醒任务重筹办
    -> 父任务 T 待重筹办

任务筹办再次运行
    -> M 已可用
    -> 继续检查参数和条件
```

候选方法不可用时，派生需求应表达为：

```text
方法状态 = 可用
```

不得在缺口不明确时直接派生：

```text
方法动作骨架状态
```

## 6. 候选方法缺执行入口路径

如果查到 `M`，且结果能力命中，但：

```text
M 缺执行入口
```

则因果链是：

```text
要任务 T 就绪，
需要用方法 M。

因为 M 的结果能力能够改变 X.F。

但方法 M 要用于任务 T，
需要 M 有可执行入口。

当前 M 缺执行入口。

所以需要派生子需求 D3：
    M.执行入口状态 = 已形成
```

执行链：

```text
D3 被承接为子任务 T3

T3 调用：
    定向学习
    或 补齐方法结构
    或 建立动作骨架

完成后：
    M.执行入口状态 = 已形成

若执行入口和条件结果对都满足可用条件：
    提交方法状态变化
    -> M.方法状态 = 可用

父任务 T 被唤醒重筹办
任务筹办再次运行
```

随后闭合到：

```text
M 可用 + 条件满足 + 参数可绑定
    -> 形成可就绪回执
    -> 提交任务状态变化
    -> T 就绪
```

## 7. 候选方法参数无法绑定路径

如果 `M` 可用，但任务上下文无法构造 `M` 的输入参数场景：

```text
缺目标宿主
缺目标特征类型绑定
缺目标值
缺工具存在
缺环境条件
缺中间对象
```

则因果链是：

```text
要任务 T 就绪，
需要用方法 M。

因为方法 M 的结果能力能作用于 X.F。

但方法 M 要执行，
需要输入参数场景 S_exec。

S_exec 需要参数：
    P1, P2, P3 ...

当前缺参数 P2。

所以需要派生子需求 D4：
    参数 P2 已获得 / 已绑定 / 已识别。
```

动作链：

```text
任务筹办
    -> 筹办缺口类型 = 参数配对失败
    -> 派生需求提示 = 获取或绑定缺失参数

派生需求入树
    -> D4：缺失参数 P2 已绑定

承接需求创建任务
    -> 子任务 T4

子任务 T4
    -> 查找 / 识别 / 构造参数 P2

回写需求满足
    -> D4 满足

父任务 T 重筹办
    -> 任务筹办再次调用配对参数

配对参数
    -> 生成 S_exec

任务筹办
    -> 可就绪回执

同步筹办回执到任务
提交任务状态变化
    -> T 就绪
```

## 8. 候选方法条件节点不满足路径

如果 `M` 可用，参数也基本存在，但 `M` 的条件节点不满足：

```text
M.条件节点要求 C
当前任务上下文不满足 C
```

则因果链是：

```text
要任务 T 就绪，
需要用方法 M。

因为 M 的结果能力命中 X.F。

但 M 要运行，
需要条件 C 成立。

当前条件 C 不成立。

所以需要派生子需求 D5：
    条件 C 成立。
```

例如：

```text
M = 加热方法
C = 有可用加热器
```

则派生子需求：

```text
D5：可用加热器存在
```

满足后：

```text
D5 满足
    -> 父任务重筹办
        -> 条件节点匹配成功
            -> 构造可执行输入参数场景
                -> 任务进入就绪
```

## 9. 统一因果模板

完整模板：

```text
要任务 T 就绪，
需要方法 M。

因为任务 T 的来源需求要求：
    X.F 满足目标状态 V_target

所以方法 M 必须具有结果能力：
    能产生 / 改变 特征类型 F

如果 M 不存在：
    需要需求 D1：存在可改变 F 的方法。

如果 M 存在但不可用：
    需要需求 D2：M.方法状态 = 可用。

如果 M 缺执行入口：
    需要需求 D3：M.执行入口状态 = 已形成。

如果 M 缺条件结果对：
    需要需求 D4：M.条件结果对 = 已建立。

如果 M 参数无法绑定：
    需要需求 D5：缺失参数已获得 / 已绑定。

如果 M 条件节点不满足：
    需要需求 D6：方法前置条件成立。

当这些子需求被满足：
    父任务 T 被唤醒重筹办。

父任务 T 再次任务筹办：
    能找到 M
    M 可用
    参数可绑定
    条件满足
    可执行输入参数场景可构造

所以任务筹办形成：
    可就绪回执。

同步筹办回执到任务：
    写入当前方法、可执行输入参数场景、筹办缺口类型。

提交任务状态变化：
    写入任务状态 = 就绪
    生成任务状态动作动态。
```

## 10. 闭合点

### 10.1 方法能力查找闭合

```text
需求目标 X.F
    -> 查找结果能力包含 F 的方法 M
```

若找到：

```text
进入方法可用性检查
```

若找不到：

```text
派生需求：存在可改变 F 的方法
```

### 10.2 方法可用性闭合

```text
M 不可用
    -> 派生需求：M.方法状态 = 可用
    -> 学习 / 补齐
    -> 提交方法状态变化
    -> M 可用
```

### 10.3 参数和条件闭合

```text
参数缺失 / 条件不满足
    -> 派生参数需求 / 条件需求
    -> 子任务满足
    -> 父任务重筹办
```

### 10.4 父任务重筹办闭合

```text
子需求满足
    -> 自我线程发父任务重筹办请求
    -> 任务管理线程调用相关任务域方法
    -> 父任务状态进入待重筹办
    -> 再次调用任务筹办
```

自我线程不得直接写任务状态。

### 10.5 任务状态落账闭合

```text
任务筹办形成可就绪回执
    -> 同步筹办回执到任务
        -> 提交任务状态变化
            -> 任务状态 = 就绪
```

任务状态动作动态绑定：

```text
本能方法_提交任务状态变化
```

不能绑定线程，也不能由任务筹办裸生成。

## 11. 完整链条

```text
D0：需求 D
    目标 = X.F -> V_target

D1：承接需求创建任务
    生成任务 T
    T.状态 = 未启动

D2：任务筹办
    按 F 查找方法能力
    形成候选方法 M 或缺口

若 M 不存在：
    D3：派生需求入树
        子需求 = 存在可改变 F 的方法

    D4：承接需求创建任务
        子任务 T_child

    D5：学习 / 建立条件结果对 / 提交方法状态变化
        方法 M 形成并可用

    D6：回写需求满足
        子需求满足

    D7：唤醒任务重筹办
        父任务 T 可重新筹办

    回到 D2

若 M 可用且参数条件满足：
    D8：任务筹办
        形成可就绪回执

    D9：同步筹办回执到任务
        写入当前方法 M
        写入可执行输入参数场景 S_exec

    D10：提交任务状态变化
        T.状态 = 就绪
        生成任务状态动作动态
```

## 12. 实现前提

按照方法能力查找方法，能够形成任务推进到就绪的闭环。前提是：

```text
1. 任务筹办必须从来源需求解析出：
   目标宿主 + 目标特征类型 + 目标状态 + 满足关系。

2. 方法能力查找必须按：
   方法结果项.特征类型
   或 特征类型由参数决定并绑定到目标特征类型。

3. 查不到方法时，不进入失败，而是派生“存在可改变目标特征的方法”的子需求。

4. 方法不可用时，派生“方法状态 = 可用”的子需求。

5. 参数 / 条件缺失时，派生对应参数或条件需求。

6. 子需求满足后，必须触发父任务重筹办。

7. 任务筹办形成可就绪回执后，必须通过：
   同步筹办回执到任务
   -> 提交任务状态变化
   把任务状态真正落到就绪。

8. 所有状态、需求、方法变化的动作动态必须绑定对应本能方法，不能绑定线程。
```

## 13. 最终定稿

```text
要任务就绪，需要找到能改变目标特征的方法；
要方法可用，需要满足方法自己的状态、入口、条件、参数；
这些缺口会长成子需求；
子需求满足后回流父任务重筹办；
最终形成可就绪回执；
再由状态提交方法把任务推进到就绪。
```
