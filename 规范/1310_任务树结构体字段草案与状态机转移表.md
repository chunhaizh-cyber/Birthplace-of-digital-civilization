# 1310 任务树结构体字段草案与状态机转移表

## 1. 目标

本文在 [1300_任务树数据结构与步骤子任务执行规范.md](D:/鱼巢/规范/1300_任务树数据结构与步骤子任务执行规范.md) 的基础上，继续收束出一版可直接指导实现的数据结构草案，重点回答三个问题：

- 任务头结点、步骤节点、结果节点分别需要哪些字段
- 步骤、子任务、执行、等待、回接在状态机上如何表达
- 任务管理任务后续接管根任务分解时，应当向任务树写入哪些标准信息

本文不直接替换现有类定义，而是给出与当前工程实现对齐的“规范口径”。后续新增字段、重构代码、补适配层时，都以本文为准。

## 2. 设计口径

先固定四条总原则：

1. 任务树只承认三类主节点：`头结点`、`步骤节点`、`结果节点`。
2. 子任务不是第四类节点。子任务本体仍然是一个新的任务头结点。
3. 真正执行的最小单元固定为：`步骤节点 + 当前方法首节点`。
4. 任务状态不直接修改安全值或服务值；任务树只负责把治理过程沉淀为可回溯的执行、结果、回接记录。

### 2.1 对任务管理任务的三层承载理解

从 `任务管理任务` 自身的结构看，当前应再补一层正式口径：

1. `任务根节点`：偏只读，承载来源需求、方向、服务对象、完成标准、父任务引用等语义信息；它回答“这是什么任务”。
2. `任务树主体`：偏可变，主要由步骤节点组成；它回答“这个任务现在推进到哪了”。
3. `局部运行态`：不要求全部塞进树节点，负责承载当前前沿、等待原因、局部预算、最近结果缓存、已上报派生需求、已收到依赖结果等驱动信息。

可收成一句话：

`任务管理任务以任务树为主体，以局部运行态为驱动。`

### 3.2 公共基类

这部分直接对应当前项目里的 `任务信息基类`，是三类节点共享的公共头。

```cpp
struct 结构_任务节点公共信息 {

    const 自然句节点类* 描述信息 = nullptr;
    const 词性节点类* 类型 = nullptr;

    场景节点类* 场景 = nullptr;
    存在节点类* 任务虚拟存在 = nullptr;
    枚举_任务树类型 任务树类型 = 枚举_任务树类型::未定义;

    std::int64_t 运行次数 = 0;
    std::int64_t 成功次数 = 0;
    std::int64_t 失败次数 = 0;
    std::int64_t 连续失败次数 = 0;
    std::int64_t 转入学习次数 = 0;
    std::int64_t 最近错误码 = 0;
    时间戳 启动时间 = 0;
    时间戳 完成时间 = 0;
    时间戳 最近调度时间 = 0;
    方法节点类* 执行方法 = nullptr;

```

字段职责固定如下：

- `状态` 是运行期镜像；权威写回在 `任务虚拟存在`。
- `场景` 是当前节点语义的锚点，头结点挂全局条件，步骤节点挂局部执行场景，结果节点挂结果场景。
- `基准优先级 + 局部优先级偏移 -> 调度优先级`，不允许只写最终值不写来源。
- 执行流水账统一写在公共头，避免步骤、头结点、结果节点各自分裂出不同记账口径。

### 3.3 头结点结构

头结点表示一个独立任务单元，也是“子任务”的真实结构载体。

对 `任务管理任务` 而言，头结点在语义上更接近 `任务根节点` 或 `根面`：

- 其中 `需求 / 来源 / 服务对象 / 完成标准` 这类字段应按只读或读多写少理解
- `当前步骤节点 / 当前子任务头结点 / 最近实际结果节点` 这类字段，更适合作为挂在头结点上的运行索引面，而不是任务根语义本体

```cpp
struct 结构_任务头结点信息 : 结构_任务节点公共信息 {
    需求节点类* 需求 = nullptr;
    bool 是否真根任务 = false;
    bool 是否常驻任务 = false;
    bool 等待学习唤醒 = false;
    方法节点类* 等待学习方法首节点 = nullptr;

    任务节点类* 父任务头结点 = nullptr;
    任务节点类* 来源父结果节点 = nullptr;
    任务节点类* 来源父步骤节点 = nullptr;

    任务节点类* 当前步骤节点 = nullptr;
    任务节点类* 当前子任务头结点 = nullptr;
    任务节点类* 最近实际结果节点 = nullptr;
    std::vector<任务节点类*> 直属子任务头结点列表{};

    std::vector<std::string> 目标特征类型{};
};
```

头结点必须承载四类信息：

- 目标来源：`需求`
- 纵向来源：`父任务头结点 / 来源父步骤节点 / 来源父结果节点`
- 当前执行位：`当前步骤节点 / 当前子任务头结点 / 最近实际结果节点`
- 横向分解面：`直属子任务头结点列表`

### 3.4 步骤节点结构

步骤节点不应先被解释为“当前要做哪个动作”，而应先被解释为：

`为了让某个存在的某个特征进入某个目标状态，而设置的一个局部推进单元。`

换句话说，每个步骤都应先能回答两句：

- `为什么要这么做`：因为当前还没有得到某个关键特征的目标状态。
- `这么做需要什么`：需要一组前置状态满足，再执行一组动作路径。

步骤节点的正式字段分组与返回合同，当前以 [1311_任务步骤结构规范_v0.1.md](D:\鱼巢\规范\1311_任务步骤结构规范_v0.1.md) 为准；本文只保留任务树视角下的承载口径。

```cpp
struct 结构_任务步骤节点信息 : 结构_任务节点公共信息 {
```

#### 3.4.1 步骤节点的八组结构与六个解释问题

步骤节点的正式字段，当前统一收成下面 `8` 组：

| 字段组 | 最小承载 |
| --- | --- |
| `步骤标识` | `步骤主键 / 所属任务 / 所属域 / 当前方法需求位 / 步骤语义` |
| `目标特征状态` | `目标存在 / 目标特征 / 目标状态 / 当前状态摘要 / 状态差距摘要` |
| `前置状态集` | `前序要求 / 前置状态集 / 来源层 / 是否硬前置 / 是否可部分满足` |
| `动作集` | `候选动作集 / 主动作 / 备用动作 / 当前方法 / 备用方法` |
| `达成判据` | `达成判据 / 主达成判据 / 次达成判据 / 结果确认方式` |
| `步骤分流与失败去向` | `步骤分流类型列表 / 细分失败去向列表` |
| `结果写回锚点` | `结果状态写回锚点 / 结构推进写回锚点 / 动作动态写回锚点 / 因果实例写回锚点 / 外部依赖引用` |
| `一步治理返回映射` | `一步治理返回字段映射` |

每个步骤节点，最少应能解释成下面 6 件事：

| 解释面 | 最小回答 |
| --- | --- |
| `目标是什么` | 让谁的哪个特征，变成什么目标状态 |
| `当前为什么还没达到` | 当前状态与目标状态之间还差什么 |
| `需要哪些前置状态` | 前序、资源、确认、方法可调用、依赖结果、风险门控等是否满足 |
| `需要哪些动作` | 当前有哪些可选动作或方法路径可把状态推进过去 |
| `怎样判定达成` | 看哪个特征到什么值算完成 |
| `如果没达成怎么办` | 本地继续推进、局部等待、上报派生需求、失败/收束/回退中的哪一类 |

步骤节点的规范职责是：

- 承载当前步骤的 `目标存在 / 目标特征 / 目标状态 / 当前差距`
- 承载当前步骤的 `前置状态集 / 条件面 / 达成判据 / 步骤分流`
- 挂载条件面，而不强迫先把所有条件都独立长成新节点
- 绑定当前可执行方法与候选动作路径
- 同层候选路径通过并列步骤表达；同一步骤内只保留当前方法与少量备用方法挂载
- 记录是否已经进入执行
- 表达是否正在等待依赖任务或外部条件
- 通过 `外部依赖任务引用` 表达“已获批准并回挂的外部支撑”，而不是授予本步骤直接创建外部子任务的权力
- 承载重试、超时、换方法等局部控制
- 提前声明结果写回锚点与一步治理返回映射，而不是在执行结束后临时拼字段

补充固定：

- `前序要求` 与 `前置状态集` 共同回答“当前还缺哪些状态”；其更细来源层、硬前置和部分满足口径，按 [1311_任务步骤结构规范_v0.1.md](D:\鱼巢\规范\1311_任务步骤结构规范_v0.1.md) 执行。
- `候选动作集` 负责表达“当前有哪些状态-动作路径可把当前状态推进到目标状态”；若这些路径对应不同条件链或不同结果链，应先拆成并列步骤。
- `达成判据` 是步骤完成的正式判断面，优先于“动作是否执行过”这类过程性痕迹。
- `步骤分流类型列表` 顶层当前至少应覆盖：`本地继续推进 / 局部等待 / 上报派生需求 / 失败或收束或回退`。
- 若实现层需要更细分，可继续在 `细分失败去向列表` 中表达：`切备用动作 / 回筹办 / 转纠偏 / 转收束` 等更细路由。
- 对 `任务管理任务` 的步骤状态语义，当前至少应能稳定映射为：`未开始 / 可推进 / 推进中 / 等待中 / 已完成 / 已失败 / 已收束`。
- `可推进` 可以在实现上暂映射为 `就绪` 或等价状态，但语义上必须保留“已具资格进入下一次推进一步”的独立含义。

### 3.5 结果节点结构

结果节点不是装饰节点，而是任务树里的标准回接点、结算前置点和派生点。

```cpp
struct 结构_任务结果节点信息 : 结构_任务节点公共信息 {
    枚举_任务结果角色 结果角色 = 枚举_任务结果角色::未定义;
    任务节点类* 所属任务头结点 = nullptr;
    任务节点类* 来源步骤节点 = nullptr;
    任务节点类* 下一跳步骤节点 = nullptr;

    std::vector<任务节点类*> 派生子任务头结点列表{};

    方法节点类* 来源方法首节点 = nullptr;
    方法节点类* 对应方法结果节点 = nullptr;
    二次特征节点类* 结果方向二次特征 = nullptr;
    任务节点类* 命中预测结果节点 = nullptr;
    枚举_分支选择动作 命中后动作 = 枚举_分支选择动作::未定义;

    bool 是否预测外结果 = false;
    std::int64_t 置信度_Q10000 = 0;
    std::int64_t 错误码 = 0;
    存在节点类* 评估摘要存在 = nullptr;
};
```

结果节点至少承担三件事：

- 记录实际执行产物
- 作为“下一步”或“子任务派生”的挂接点
- 为后续“叶子需求满足判定”和“价值结算”提供可回溯的证据锚点

补充固定：

- 步骤节点建立后，应立即至少创建一个结果节点，作为初始化结果锚点。
- 初始化结果锚点默认按 `预测结果` 或 `占位结果` 理解，后续可被补全、命中或派生实际结果。
- 初始化结果锚点必须给出 `结果方向二次特征`；它是结果特征状态的二次特征，初始化时按 `步骤目标状态` 对 `当前状态` 生成，后续再按更明确的预测/实际结果状态刷新。
- 结果节点允许多实例；预测结果与实际结果必须允许并存，不得直接把预测结果覆写成实际结果。
- 结果节点的下一步枝路只允许写在 `下一跳步骤节点`；`后续任务` 不再承载结果节点到下一步骤的语义。

## 4. 结构关系规范

### 4.1 最小主链

任务树的最小闭环固定为：

```text
任务头结点
-> 步骤节点
-> 结果节点
```

原因有两点：

- 执行必须落在步骤上，不能让头结点直接执行
- 回接必须落在结果上，不能让下一步直接跳过结果

### 4.2 子任务的真实位置

任务头结点形式的外部分支有两种常见挂法：

```text
步骤节点
-> 并发前置任务头结点列表

结果节点
-> 派生子任务头结点列表
```

语义区分如下：

- 挂在步骤上：表示“执行方法运行中发现缺口，自我线程已实例化并回挂的前置任务”
- 挂在结果上：表示“某个结果导致系统决定再派生一个独立任务”

但对 `任务管理任务` 还必须补一条：

- 在外层尚未批准之前，步骤上更稳的写法不是直接长出正式子任务，而是先记录 `需要外部支撑` 与 `外部依赖引用待定`
- 只有自我线程批准并创建了新的任务入口之后，才把对应任务头引用正式挂回 `外部依赖任务引用列表` 或等价依赖槽位

补充固定：

1. `并发前置任务头结点列表` 只承载前置任务，不承载主步骤链上的下一步。
2. 主步骤链的下一步只允许通过结果节点的 `下一跳步骤节点` 或结果节点的子步骤表达。
3. 筹办阶段负责长主步骤；执行方法负责产生命令，再由自我线程回挂前置任务。

### 4.3 当前指针

所有运行期调度都优先读取以下三个指针：

- 头结点：`当前步骤节点`
- 头结点或步骤节点：`当前子任务头结点`
- 头结点：`最近实际结果节点`

没有这三个指针，任务树虽然能遍历，但不能稳定驱动执行。

## 5. 步骤、子任务、执行的判定规范

### 5.1 什么时候长步骤

满足以下任一条件时，默认长步骤：

- 只是当前任务内部的一个动作
- 只是一次方法执行
- 只是等待某个子任务完成
- 只是对已有结果做检查、确认、回接
- 只是一次重试、改方法、继续下一步

### 5.2 什么时候必须长子任务

满足以下任一条件时，必须长子任务：

- 该工作体有自己的需求来源
- 该工作体需要独立筹办、维护、纠偏、收束
- 该工作体会继续分解出自己的步骤或子任务
- 该工作体完成后需要作为独立结果回接父任务
- 该工作体需要被单独统计、重试、挂起、取消

### 5.3 执行的唯一口径

执行只承认下面这一个最小单元：

```text
执行单元 = 步骤节点 + 当前方法首节点
```

因此：

- 头结点不直接执行
- 子任务不是执行动作，它是独立任务单元
- 结果节点不执行，它只记录执行产物并驱动后续分支

### 5.4 等待的表达

等待不是执行失败，也不是空转。等待应建模为显式步骤状态：

```text
步骤A：发起子任务B
步骤A：进入等待中
子任务B：独立推进
子任务B：完成
步骤A：接收子任务完成消息
步骤A：生成结果节点
父任务：进入下一步
```

因此必须固定一句话：

`等待子任务完成可以作为步骤，但被等待的独立工作仍然必须是子任务。`

## 6. 状态机转移表

### 6.1 头结点状态机

头结点的状态代表整条任务主链当前所处的治理阶段。

| 当前状态 | 可转向状态 | 说明 |
| --- | --- | --- |
| 未启动 | 就绪 | 任务已创建但未进入运行 |
| 就绪 | 筹办中, 排队中, 取消 | 已可被治理器接管 |
| 筹办中 | 排队中, 等待中, 无法执行, 待重筹办, 取消 | 补条件、补方法、补子任务 |
| 排队中 | 运行中, 等待中, 无法执行, 取消 | 等待执行器领取 |
| 运行中 | 等待中, 完成, 失败, 挂起, 待重筹办, 取消 | 主链正在推进 |
| 等待中 | 就绪, 运行中, 挂起, 超时, 取消 | 等子任务、等外部输入、等学习 |
| 无法执行 | 待重筹办, 挂起, 取消 | 条件不足或方法缺失 |
| 待重筹办 | 筹办中, 挂起, 取消 | 等待再次治理 |
| 挂起 | 就绪, 取消 | 人工或系统暂停 |
| 完成 | 无 | 已收束 |
| 失败 | 待重筹办, 挂起, 取消 | 允许回到治理侧 |
| 超时 | 待重筹办, 失败, 取消 | 超时后不直接算成功 |

### 6.2 步骤节点状态机

步骤节点的状态代表一个局部执行位当前是否可运行、在运行、在等待、或已产出结果。

| 当前状态 | 可转向状态 | 说明 |
| --- | --- | --- |
| 未启动 | 就绪 | 步骤已创建 |
| 就绪 | 排队中, 执行中, 等待中, 取消 | 已具备执行条件 |
| 排队中 | 执行中, 等待中, 取消 | 等待调度 |
| 执行中 | 完成, 失败, 无法执行, 超时, 等待中 | 方法正在运行 |
| 等待中 | 就绪, 执行中, 超时, 取消 | 等子任务、等外部信号 |
| 失败 | 就绪, 待重筹办, 取消 | 允许重试或切方法 |
| 无法执行 | 待重筹办, 挂起, 取消 | 当前位缺前置条件 |
| 超时 | 就绪, 失败, 取消 | 由重试策略决定 |
| 挂起 | 就绪, 取消 | 局部暂停 |
| 完成 | 无 | 应立即配套有结果节点 |

对 `任务管理任务` 的步骤语义，还应补充按如下方式理解：

- `就绪` 在步骤推进语义上，默认等价于 `可推进`
- `执行中` 在步骤推进语义上，默认等价于 `推进中`
- 若因控制要求、主动收口或局部裁剪结束当前步骤，应允许在实现上附带 `已收束` 的等价语义，而不强迫混成普通 `失败`

### 6.3 结果节点状态口径

结果节点不作为长期运行态节点使用，建议遵守以下口径：

- 创建时可为 `就绪`
- 一旦完成评估和回接，置为 `完成`
- 如结果无效或判定失败，可置为 `失败`

结果节点不承担排队、执行、等待语义。

## 7. 写回规范

### 7.1 创建头结点时必须写

- `需求`
- `任务树类型`
- `父任务头结点 / 来源父步骤节点 / 来源父结果节点`
- `目标特征类型`
- `基准优先级 / 局部优先级偏移 / 调度优先级`

### 7.2 创建步骤节点时必须写

- `所属任务头结点`
- `步骤主键 / 父步骤主键 / 步骤语义`
- `所属域 / 当前方法需求位`
- `目标存在标识 / 目标特征类型 / 目标状态`
- `当前状态摘要 / 状态差距摘要`
- `前序要求 / 前置状态集 / 前置状态来源层`
- `硬前置标记 / 可部分满足标记`
- `候选动作集 / 主动作标识 / 备用动作标识`
- `达成判据 / 主达成判据 / 结果确认方式`
- `步骤分流类型列表 / 细分失败去向列表`
- `结果状态写回锚点 / 结构推进写回锚点 / 动作动态写回锚点 / 因果实例写回锚点`
- `一步治理返回字段映射`
- `步骤序号`
- `当前选中方法首节点`
- `备用方法首节点列表`
- `允许重试次数`
- `超时截止时间`

并且必须同步：

- 创建至少一个结果节点
- 初始化该结果节点的 `结果角色 / 所属任务头结点 / 来源步骤节点 / 来源方法首节点 / 对应方法结果节点 / 结果方向二次特征 / 命中后动作 / 是否预测外结果 / 置信度_Q10000 / 错误码`

### 7.3 创建结果节点时必须写

- `结果角色`
- `所属任务头结点`
- `来源步骤节点`
- `来源方法首节点`
- `对应方法结果节点`
- `结果方向二次特征`
- `命中预测结果节点`
- `命中后动作`
- `是否预测外结果`
- `置信度_Q10000`
- `错误码`
- `评估摘要存在`

### 7.4 进入执行前必须写

- `当前步骤节点`
- `执行方法`
- `运行次数`
- `启动时间`
- `最近调度时间`

### 7.5 执行结束后必须写

- `成功次数 / 失败次数 / 连续失败次数`
- `最近错误码`
- `完成时间`
- `最近结果节点`
- 若转入学习，则写 `转入学习次数`

### 7.6 绑定前置任务或派生任务时必须写

如果前置任务由执行方法运行中提出并经自我线程实例化：

- 写入步骤节点的 `并发前置任务头结点列表`
- 写入步骤节点的 `当前前置任务头结点`
- 写入父头结点的 `直属子任务头结点列表`
- 写入父头结点的 `当前子任务头结点`
- 回填前置任务的 `父任务头结点 / 来源父步骤节点`

如果任务由结果派生：

- 写入结果节点的 `派生子任务头结点列表`
- 不得把它回写成步骤节点的 `并发前置任务头结点列表`
- 回填派生任务的 `父任务头结点 / 来源父结果节点`

## 8. 给任务管理任务的接口约束

后续任务管理任务接管根任务分解时，至少需要遵守以下接口约束：

1. 任务管理任务只能直接创建或调整：
   - 步骤节点
   - 结果节点
   - 当前指针与状态
   - 局部结构候选与结构写回摘要
2. 若需要新的正式子任务，`任务管理任务` 只能上报 `派生需求消息 / 结构候选`，由外层批准后再实例化 `子任务头结点`。
3. 任务管理任务不能直接写安全值、服务值。
4. 任务管理任务若要宣称“治理有效”，必须在任务树上留下至少一条可检索证据：
   - 任务进入可执行或前进态
   - 上报了新的派生需求或结构候选，并被外层采纳
   - 产出了新的结果节点
   - 触发了下一步或回接
5. 任务管理方法的成长证据，应优先从以下链条回溯：

```text
方法应用
-> 步骤状态推进
-> 结果节点产生
-> 子任务完成或叶子需求满足
-> 根任务账本变化
```

## 9. 最小实现建议

如果下一步要开始改代码，最小改造顺序建议固定为：

1. 先让根任务分解统一经过任务管理任务入口。
2. 让任务管理任务只通过本文定义的结构写口去改任务树；正式子任务头结点仍由外层审批后落账。
3. 让显式规则分解退化为“筹办域的兜底方法”。
4. 再把“结果节点 -> 叶子需求满足 -> 安全值或服务值变化”的回溯链挂到方法成长判定上。

## 10. 一句话收束

任务树的数据结构规范可以收束为一句话：

`头结点负责承载独立任务，步骤节点负责承载执行与等待，结果节点负责承载产物与回接，子任务始终是新的头结点，执行始终落在步骤加方法上。`
