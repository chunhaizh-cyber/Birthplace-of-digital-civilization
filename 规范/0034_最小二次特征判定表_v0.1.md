# 0034 最小二次特征判定表 v0.1

## 1. 目标

本文只做当前阶段最值得先冻结的两项二次特征：

- `依赖满足`
- `当前步骤可执行`

目标不是一次把所有判断都铺开，而是先给后续代码落地提供一套最小、稳定、可复用的判定基座。

本文要回答：

- 这两个二次特征分别由哪些一次特征输入得出
- 它们的最小判定公式是什么
- 结果应以什么形式落回 `二次特征`
- 何时重算、谁负责重算
- 三条样板链如何从 `动作 -> 一次特征变化 -> 二次特征判断` 跑通

## 3. 范围与硬边界

本版只做两个二次特征：

- `依赖满足`
- `当前步骤可执行`

本版先不展开：

- 父任务阻塞
- 父任务解阻
- 风险升高
- 预算紧张
- 是否需要收束
- 有效优先级

必须固定：

1. 二次特征只能建立在 `合法一次特征真值` 之上。
2. 二次特征不能直接从消息名字推断。
3. 二次特征结果至少要能落成：`是否满足 + 标量值 + 置信度 + 原因摘要`。
4. 若需要引用尚未进入 6 类核心存在字典的对象，例如 `资源`，当前只允许把它当作 `外部条件输入` 使用，不得绕过 `0033` 另造一套正式核心存在字典。


## 4. 核心结论

必须固定以下结论：

1. `依赖满足` 是“等待对象是否已经满足继续推进所需条件”的结构化判断。
2. `当前步骤可执行` 是“当前步骤在前序、条件、依赖和控制边界上是否允许推进”的结构化判断。
3. `当前步骤可执行` 允许依赖 `依赖满足`，但不允许反过来循环依赖。
4. `是否满足` 是二次特征的主输出；`标量值` 是便于排序和比较的辅输出；`置信度` 是便于审计和学习的补输出。
5. 后续 `父任务解阻 / 回可运行队列 / 主派发决议` 等逻辑，不得直接裸判本文的二次特征结果；应先把它们转成正式原语输出，再进入选择函数或外层裁决。
6. 本文这两个二次特征的输出，应继续作为 `0038` 中影响项生成的直接输入，而不是被直接当成 `服务值 / 安全值` 最终结论。
7. 与任务相关的一次特征输入，必须遵守 `0039` 的三层归属，不得再把 `任务控制态` 和 `任务管理任务局部推进态` 混成一团。

## 6. 两个二次特征的最小字段合同

### 6.1 依赖满足

| 字段 | 固定要求 |
| --- | --- |
| 主体 | 父任务 |
| 客体 | 子任务或外部条件对象 |
| 左对象 | 当前等待原因对应对象 |
| 右对象 | 最近结果或最近确认条件 |
| 是否满足 | `true / false` |
| 标量值 | `1` 表示满足，`0` 表示不满足 |
| 置信度 | 默认允许取 `0-10000` |
| 原因摘要 | 必须指出“为什么满足或为什么不满足” |

### 6.2 当前步骤可执行

| 字段 | 固定要求 |
| --- | --- |
| 主体 | 任务管理任务 |

| 左对象 | 宿主任务 |

| 是否满足 | `true / false` |
| 标量值 | `1` 表示可执行，`0` 表示不可执行 |
| 置信度 | 默认允许取 `0-10000` |
| 原因摘要 | 必须指出“不满足的第一阻断原因”或“满足的成立原因” |


## 7. 最小判定顺序

### 7.1 依赖满足判定顺序

固定顺序为：

1. 先检查 `任务管理任务.当前等待原因` 是否存在。
2. 若不存在等待原因，则 `依赖满足 = true`，因为当前没有依赖阻断。
3. 若等待的是子任务结果，则检查 `子任务.结果状态` 是否进入成功或可接受完成态。
4. 若等待的是外部条件，则检查 `外部条件满足状态（占位输入）` 是否为真。
5. 若 `任务管理任务.依赖等待状态` 仍显式为等待，则优先压低结果。
6. 产出 `是否满足 + 标量值 + 置信度 + 原因摘要`。

### 7.2 当前步骤可执行判定顺序

固定顺序为：

1. 先检查 `任务管理任务.当前步骤前沿` 是否有效。
2. 再检查 `任务管理任务.前序完成状态`。
3. 再检查 `任务管理任务.条件满足状态`。
4. 再消费 `依赖满足` 判定结果。
5. 再检查 `任务控制态.是否允许继续派发` 是否为真。
6. 再检查 `工作线程.控制意图` 是否已进入 `请求暂停 / 请求打断 / 请求收束 / 请求停止`。
7. 再检查 `工作线程.实际运行状态` 是否允许继续。
8. 输出第一阻断原因；若无阻断，则 `当前步骤可执行 = true`。

必须固定：

1. `当前步骤可执行` 的阻断原因应按顺序早停，不得在多个原因之间随意摇摆。
2. `依赖满足` 应优先于 `当前步骤可执行` 计算。


## 8. 推荐最小伪公式

### 8.1 依赖满足

```text
if 当前等待原因 == 无:
    满足 = true
else if 等待原因 == 等待子结果 and 子任务结果状态 in {成功, 可接受完成}:
    满足 = true
else if 等待原因 == 等待外部条件 and 外部条件满足状态 == true:
    满足 = true
else:
    满足 = false
```

### 8.2 当前步骤可执行

```text
if 当前步骤前沿无效:
    可执行 = false
else if 前序完成状态 != 已完成:
    可执行 = false
else if 条件满足状态 != 已满足:
    可执行 = false
else if 依赖满足 != true:
    可执行 = false
else if 任务控制态.是否允许继续派发 != true:
    可执行 = false
else if 控制意图 in {请求暂停, 请求打断, 请求收束, 请求停止}:
    可执行 = false
else if 工作线程实际运行状态 in {已暂停, 已打断, 收束中, 已收束, 已停止, 故障中}:
    可执行 = false
else:
    可执行 = true
```


## 9. 三条样板链

### 9.1 样板 1：控制链

```text
自我线程
-> 执行动作“下发收束指令”
-> 工作线程.控制意图 = 请求收束
-> 工作线程执行后回报“完成收束”
-> 工作线程.实际运行状态 = 已收束
-> 任务控制态.是否允许继续派发 = false
-> 当前步骤可执行 = false
```

说明：

- 这条链先改的是 `控制意图`，后改的是 `实际运行状态`。
- `当前步骤可执行` 由控制域和运行状态共同压成 `false`。

### 9.2 样板 2：结果回流链

```text
子任务
-> 执行动作“提交结果”
-> 子任务.结果状态 = 成功
-> 父任务管理任务.当前等待原因 = 等待子结果
-> 依赖满足 = true
-> 当前步骤可执行 = true
```

说明：

- 这条链里 `依赖满足` 先成立。
- `当前步骤可执行` 在 `前序完成 / 条件满足 / 控制域允许继续` 的前提下再成立。

### 9.3 样板 3：环境链

```text
环境监测器
-> 上报“外部条件满足”
-> 外部条件满足状态 = true（占位输入）
-> 任务管理任务.当前等待原因 = 等待外部条件
-> 依赖满足 = true
-> 当前步骤可执行 = true
```

说明：

- 本版先不把 `资源` 展开成正式核心存在。
- 环境链仍可通过 `外部条件满足状态（占位输入）` 接入 `依赖满足` 判定。


## 10. 完成标准

达到以下 5 条，即可视为当前轮站稳：

1. `依赖满足` 和 `当前步骤可执行` 都有正式输入表。
2. 两者都有固定判定顺序和最小公式。
3. 两者都能稳定产出 `是否满足 + 标量值 + 置信度 + 原因摘要`。
4. 三条样板链都能从 `动作 -> 一次特征变化 -> 二次特征判断` 跑通。
5. 后续消息字段表、变化项表和线程门控，均可直接引用本文，不再靠消息名字硬猜。
6. 若后续要把“这些二次特征究竟如何影响服务与安全”正式写成模板，应按 [0038_关键特征变化影响项与服务安全聚合规范_v0.1.md](D:\鱼巢\规范\0038_关键特征变化影响项与服务安全聚合规范_v0.1.md) 执行。


## 11. 对后续实现的直接约束

必须固定：

1. 若后续代码里还存在“只看消息子类就直接决定任务可不可执行”的写法，应逐步迁到本文定义的二次特征判定上。
2. `依赖满足` 应优先做成独立判定器或独立函数。
3. `当前步骤可执行` 应明确依赖 `依赖满足` 的输出，不得在内部重复拼一份异构判断。
4. 新增二次特征前，应先保证这两个最小二次特征可观测、可回写、可在日志中定位。

一句话收口：

`0034` 先把“依赖有没有满足、当前步骤能不能执行”这两个最小判断定成正式中间层，后续所有更复杂的阻塞、解阻、派发和收束判断都应建立在它们之上。
