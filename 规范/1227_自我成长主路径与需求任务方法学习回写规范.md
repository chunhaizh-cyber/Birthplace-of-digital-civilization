## 3. 基本判断

必须先固定以下 12 条：

1. `真正成长` 不是任务节点变多，也不是治理分支变多。
2. `真正成长` 的主结果，必须落在 `方法资产` 增长上，而不是只落在 `任务运行痕迹` 上。
3. `需求` 负责显性化差额，回答“现在缺什么、为什么要补”。
4. `任务` 负责承接差额，回答“这一轮由谁推进、推进到哪一步”。
5. `方法` 负责形成可复用能力，回答“同类缺口下以后可重复怎么做”。
6. 没有 `目标 - 结果差额`，系统无法知道自己是否真的在前进。
7. 没有 `状态 / 动态 / 因果` 证据链，系统无法把一次运行沉成可复用经验。
8. 没有 `学习回写 -> 方法树进入正式资产面`，系统只能维持运行，不能算真正成长。
9. `任务管理任务` 的职责是治理承接，不是无限细分业务内部步骤。
10. `双值维持 / 待机保护 / 根层重判 / 执行前门控` 是成长的生存前提，不是成长本身。
11. 成长必须始终服从 `服务为主、安全为辅、安全优先` 的边界，不得因成长冲动越权。
12. 同类需求再次出现时，若系统能凭已沉淀的方法更快、更稳、更低代价地缩小差额，才应认定为真实成长。

可收敛为一句话：

`成长不发生在任务树节点数上，而发生在方法树和方法账本能否因为证据沉淀而扩张。`


## 4. 为什么当前要先做这些基础设施

必须固定：

当前工程先补 `双值维持 / 自我线程 / 任务管理 / 因果 / 抽象因果 / 控制面板与日志`，不是为了堆模块，而是为了先建立成长的最小前提面。

### 4.1 双值与自我线程

作用：

- 让系统持续存在
- 让系统有统一周期
- 让系统能长期累积证据而不是单次运行后消散

结论：

- 没有稳定运行面，就没有成长时长。
- 没有统一线程节拍，就没有多轮对比与回写窗口。

### 4.2 任务管理任务

作用：

- 把需求差额承接成当前宿主任务、当前方法、当前步骤、最近结果
- 形成统一的治理中间态
- 把“要补什么”翻译成“本轮实际在推进什么”

结论：

- 没有承接层，需求会直接跳到方法执行，成长链会断。
- 没有宿主、步骤、结果双面，系统无法知道哪一轮真的缩小了差额。

### 4.3 状态 / 动态 / 因果 / 抽象因果

作用：

- 把运行事实沉成可比较、可复用的证据
- 把“做了什么、发生了什么、结果怎样、通常会怎样”分层记录

结论：

- 没有实例因果，无法知道这次为什么成功或失败。
- 没有抽象因果，无法把多次实例压成可学习原料。

### 4.4 控制面板、日志、自检

作用：

- 让成长链可观察、可复核、可定位缺口

结论：

- 没有观察面，系统即使长出方法，也无法确认是不是有效成长。

因此必须继续固定：

`前面这些补建动作，都是成长前提，不是绕远路。`


## 5. 需求、任务、方法三层在成长主链中的职责

### 5.1 需求层

`需求` 的职责是显性化差额。

它主要回答：

- 缺什么
- 为什么要补
- 缺口对服务和安全的关系是什么

因此必须固定：

1. 需求层不直接决定具体方法。
2. 需求层首先只负责把 `目标值 - 当前值` 变成可推进缺口。
3. 学习动机也必须首先来源于需求差额或由治理压力上浮的衍生需求。

### 5.2 任务层

`任务` 的职责是承接差额。

它主要回答：

- 当前由哪个宿主承接
- 当前推进到哪一步
- 当前缺的是方法、条件、结果，还是已经可以等待

因此必须固定：

1. 任务层负责“本轮怎样推进”，不负责“长期能力怎样成长”。
2. 任务层可以产出业务步骤、等待步骤、补条件子任务、学习子任务候选。
3. 任务层的真正价值，在于为成长准备结构化上下文与结果面。

### 5.3 方法层

`方法` 的职责是承载可复用能力。

它主要回答：

- 在什么条件下能做
- 做了以后通常会产生什么结果
- 下次类似任务出现时是否值得优先复用

因此必须固定：

1. 只有方法资产增长，才构成真正成长。
2. 任务树主要记录“这次怎样做过”；方法树主要承载“以后怎样重复更好地做”。
3. 同类需求再来时，如果仍只能靠默认本能兜底，就不能算成长已经成立。


## 6. 真正自我成长的主路径

当前工程中，真正的成长主路径必须统一理解为：

```text
根层允许继续推进
-> 需求显性化差额
-> 任务管理承接当前宿主任务
-> 选择/复用/缺失当前方法
-> 执行并产生结果
-> 结果落成状态/动态/实例因果
-> 多轮实例归并为抽象因果或稳定任务原料
-> 上浮学习需求或直接形成待学习方法
-> 学习任务消费原料并形成学习产出
-> 方法写回方法树/方法账本/方法虚拟存在
-> 外层重判
-> 下一轮同类需求优先复用更合适的方法
```

必须固定：

1. 这条链是 `成长主链`，不是“每轮都必须完整跑完”的强制同步链。
2. 某一轮可以只走到 `维持运行`。
3. 只有当结果真正回写到方法资产面，并在后续治理轮被再次复用时，成长主链才算向前推进。


## 7. 什么算成长，什么不算成长

### 7.1 不算成长

以下情况默认都不应直接算“真正成长”：

- 只是多生成了任务节点
- 只是多走了一条治理分支
- 只是多记录了状态、动态、因果
- 只是抽出了抽象因果，但没有后续方法写回
- 只是把任务管理任务继续拆得更细
- 只是一次成功，没有后续复用

### 7.2 算成长的最小判据

以下条件至少应同时满足：

1. 某类需求差额被显性化。
2. 某轮任务推进留下了可追溯证据。
3. 证据被整理成 `待学习方法 / 候选方法 / 条件补全 / 结果补全 / 方法可靠度增量` 之一。
4. 这些结果正式写回方法资产面。
5. 后续相似需求再次出现时，系统能因这份资产而减少默认兜底、减少重筹办、减少无效试探，或提高稳定完成率。

可收敛为一句话：

`成长 = 同类需求再次出现时，系统因为已有方法资产而更会做。`


## 8. 成长结果的最小写回位置

成长结果必须优先写回以下位置之一：

- 方法树正式方法首节点
- 方法条件节点 / 结果节点
- 方法虚拟存在特征
- 方法账本中的可靠度 / 稳定度 / 来源 / 最近证据
- 待学习方法或候选方法承载面

必须固定：

1. 成长结果不得只停留在任务树步骤痕迹里。
2. 成长结果不得只停留在日志或控制面板摘要里。
3. 成长结果不得另起平行真值层。
4. 任务树保留的是运行实例；方法树与方法账本保留的是可复用能力。


## 9. 从因果到方法成长的最小桥接

当前最值得固定的桥接规则是：

1. `实例因果` 提供这次运行的事实证据。
2. `抽象因果` 提供同类运行的稳定公共结构。
3. `学习任务` 负责把这些原料收口成正式可写回的学习产出。
4. `方法来源判定与成熟度阶段` 负责把写回结果安放到正确的方法存在面上。

因此必须固定：

- 抽象因果本身不是成长终点。
- 抽象因果只有在进入 `待学习方法 -> 方法写回 -> 后续复用` 这条链后，才开始转化为真正成长。


## 10. 当前工程下一段真正值得推进的方向

若以上主路径成立，则当前工程后续优先方向不应再是“继续细拆任务管理任务”，而应是：

1. `方法缺口显性化`
   - 当没有可用方法或目标-结果差额长期存在时，显性生成学习压力与待学习方法。

2. `学习任务正式接入`
   - 让学习任务真正消费实例因果、抽象因果、稳定任务路径与稳定任务子树。

3. `方法写回`
   - 把学习产出正式写回方法树、方法账本与方法虚拟存在。

4. `后续复用验证`
   - 在下一轮相似需求出现时，验证是否减少默认兜底、减少重筹办、提高稳定推进。

必须固定：

- 这四步中，真正的成长拐点是第 3 步，不是第 1 步和第 2 步。
- 第 4 步是成长成立的验收面。


## 11. 最小验收口径

后续若要判断“自我是否开始真正成长”，至少应检查以下四条：

1. 是否已有从需求差额上浮出的正式学习需求或待学习方法。
2. 是否已有学习产出写回方法树或方法账本。
3. 是否已有至少一类相似任务不再主要依赖默认本能兜底。
4. 是否能在后续轮次中观察到：更少步骤、更少失败、更低代价，或更高稳定完成率。

若以上都没有，则当前系统仍只应视为：

`会维持运行、会治理、会记账、会解释`

而不应过早宣称：

`已经开始真正成长`


## 12. 一句话收口

当前工程里的真正自我成长路径，应统一固定为：

`需求显性化差额 -> 任务承接推进 -> 方法执行留证 -> 因果沉淀经验 -> 学习把经验收口成方法产出 -> 方法写回 -> 下一轮同类需求更好地被完成`

并且必须继续固定：

`任务树负责记录这次怎样做过，方法树负责承载以后怎样更会做；只有方法资产真的增长了，才算自我开始真正成长。`
