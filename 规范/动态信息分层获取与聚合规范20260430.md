## 1. 总原则

动态信息不是子层自己给自己的属性，而是父层观察直接子层状态变化后形成的信息。

动态获取必须逐层进行，不能跨层。任何父层只能观察自己的直接子层，不能绕过中间层直接抓取孙层或更深层的状态变化。

动态信息天然是信息聚合过程：

```text
底层状态事实
-> 直接父层生成原子动态
-> 更高父层观察直接子层状态变化
-> 聚合成更高层动态
```

底层动态保留细节，上层动态保留聚合结果、方向、强度、持续时间和来源链。

## 2. 与特征规范的关系

特征系统分为抽象特征和实例特征。

抽象特征是定义层，不保存当前值。实例特征是运行层，挂在存在、场景或虚拟存在下，保存当前值、观测和稳态。

硬规则：

1. 动态只能围绕实例特征、存在、场景这些运行事实生成。
2. 抽象特征不直接生成动态；抽象特征只提供特征类型、区间和解释规则。
3. 状态动态只能由该实例特征所属宿主获取；该宿主就是特征的父级。
4. 任务、方法、线程不能直接作为特征宿主；它们需要特征时，必须先通过虚拟存在表达。
5. 虚拟存在下的实例特征变化，按普通存在的状态动态规则处理。
6. 特征子链只表示特征口径的细化描述，不形成动态观察层级。
7. 同一宿主下的实例特征在运行层视为一层；父特征是粗略口径，子特征是详细口径。
8. 使用特征时优先使用可用的子特征；子特征缺失或不可用时，才用父特征补位。

因此：

```text
方法是否可用 是方法虚拟存在的实例特征
方法是否可用: 0 -> 1
=> 由方法虚拟存在获取特征状态和状态动态
=> 方法虚拟存在的状态变化再由所属场景观察
```

而不是由方法节点本身直接挂特征、直接生成动态。

## 3. 与方法规范的关系

方法是为了让某个存在的某类特征状态发生目标变化而可被执行的能力单元。

方法执行和动态之间必须分清两条线：

| 信息 | 来源 | 落点 | 含义 |
| --- | --- | --- | --- |
| 动作动态 | 真实调用方法函数 | 自我所在场景 | 自我主动调用了某个方法 |
| 状态变化动态 | 父层观察子层状态变化 | 被观察对象的直接父层 | 某对象状态发生了变化 |
| 方法运行状态动态 | 方法虚拟存在特征变化 | 方法虚拟存在 | 方法自身运行状态发生变化 |
| 方法功能结果动态 | 目标存在特征变化 | 目标存在 | 方法实际影响了目标对象 |

每次真正调用方法函数时，都必须生成动作动态。动作动态记录“自我主动调用某个方法”的事实。

动作动态不能伪造。以下情况不得生成真实动作动态：

| 情况 | 原因 |
| --- | --- |
| 方法不可用 | 尚未真实调用方法函数。 |
| 缺条件节点 | 尚未完成输入实例化。 |
| 只做筹办查找 | 筹办不是被筹办方法的真实执行。 |
| 只处理线程消息 | 消息落账是治理记录，不是方法动作。 |

方法执行后的功能事实应写入结果场景；方法自身运行情况应写入方法虚拟存在。二者后续各自按分层动态规则被观察。

## 4. 分层获取规则

### 4.1 固定层级表

| 被观察对象 | 动态获取层 | 动态含义 |
| --- | --- | --- |
| 实例特征 | 所属宿主 | 某宿主的某个特征值如何变化 |
| 存在 | 所属场景 | 某场景中的某个存在如何变化 |
| 子场景 | 父场景 | 某子场景的整体态势、阶段或结构如何变化 |
| 任务虚拟存在 | 所属场景 | 任务虚拟存在的存在级变化 |
| 方法虚拟存在 | 所属场景 | 方法虚拟存在的存在级变化 |
| 线程虚拟存在 | 所属场景 | 线程虚拟存在的存在级变化 |

实例特征若挂在存在或虚拟存在下，由该存在或虚拟存在获取状态和动态；实例特征若挂在场景下，由该场景获取状态和动态。

### 4.2 禁止跨层

必须禁止以下行为：

1. 场景直接读取某个存在内部特征的两次状态并生成状态动态。
2. 父场景直接读取子场景内某个存在或特征的变化并生成存在动态。
3. 任务、方法、线程节点直接承载特征并直接生成动态。
4. 因果或二次特征模块绕过动态层级，直接抓取底层状态生成高层结论。
5. 任意接口要求调用方传入“调用点”这类自由文本来解释动态来源。

正确链条为：

```text
特征状态变化
-> 存在观察特征并生成状态动态
-> 场景观察存在状态变化并生成存在动态
-> 父场景观察子场景状态变化并生成场景动态
```

### 4.3 状态信息落点

状态信息也遵守父级观察规则。

实例特征本身不承载状态节点；状态信息应落在该实例特征所属宿主中，并通过结构化引用说明：

| 字段 | 含义 |
| --- | --- |
| 状态主体 | 拥有该特征的存在、场景或虚拟存在 |
| 状态特征 | 本次采用的实例特征口径 |
| 状态值 | 该口径下的当前值 |

因此：

```text
存在 A
  实例特征: 健康状态
  实例特征: 血压

状态信息:
  状态主体 = 存在 A
  状态特征 = 血压
  状态值 = 120
  落点 = 存在 A
```

这里 `血压` 可以是 `健康状态` 的细化特征，但它不是动态层级中的子层。存在 A 的特征在运行层仍是一层。

### 4.4 特征口径选择

同一宿主存在父特征和子特征时，使用规则固定为：

1. 优先选择满足目标语义的最细子特征。
2. 子特征没有当前值、不可比较或缺失时，使用最近可用父特征补位。
3. 父特征补位只能提供粗略描述，不能伪装成子特征的精确值。
4. 动态和状态都必须记录实际采用的特征口径。
5. 多个子特征同时可用时，应按需求目标、方法输入要求或明确的组合规则选择；不得随意混合。

### 4.5 内部状态与外显状态

当前实现条件下，已经可靠存在的是自我的状态信息。

自我的状态信息属于自我内部状态，仍然落在自我信息内部，现有内部状态挂载点不变。

外显状态指外界能观察到的状态，不等同于自我内部状态。自我内部记录的状态事实，只能说明自我自身已经记录到某个内部变化；其中部分动作可能导致外显状态改变，但外显状态是否真的改变，需要由另外途径观察到的信息进行比较和确认。

因此，自我动作与外显结果之间的确认链固定为：

```text
自我内部状态记录
-> 自我方法动作动态
-> 另外途径观察到的外显状态
-> 内部状态与外显状态比较
-> 因果候选或稳定因果
```

当前阶段不得为了表达外显状态而刻意新增特征类型。外显状态只能使用已有可识别状态事实表达；新的特征类型应在后续观察和认知过程中自动形成。

## 5. 动态类型

动态信息按用途分为三类：

| 类型 | 本质 | 主要用途 |
| --- | --- | --- |
| 动作动态 | 描述某个存在执行动作的过程事实 | 查找同一种特征类型变化的原因 |
| 状态动态 | 只描述状态变化过程的动态 | 查找不同特征类型之间的变化关系和因果链 |
| 因果动态 | 从动态样本中抽象出的稳定关系 | 作为因果类承载的动态抽象信息 |

动作动态回答“谁做了什么，可能导致同类特征变化”。

状态动态回答“哪些状态变化彼此相关，尤其是不同特征类型之间如何联动”。

因果动态回答“这些动态样本已经抽象成什么稳定关系”。

### 5.1 状态动态

状态动态是只描述状态变化过程的动态。

状态动态不声明动作主体，不声明“谁导致了变化”，只记录状态从前状态到后状态的变化事实。

状态动态的主要用途不是直接证明动作原因，而是查找因果链，特别是查找不同特征类型之间的变化关系。

同一特征口径内的状态动态可以形成：

- 变化量
- 变化方向
- 持续时间
- 稳定阶段链

不同特征口径之间的状态动态可以形成：

- 前置变化关系
- 同步变化关系
- 滞后变化关系
- 抑制或阻断关系
- 跨特征因果候选

特征所属宿主观察同一有效特征口径前后状态后形成的动态，是状态动态的基础形式。

输入必须至少包含：

| 输入 | 要求 |
| --- | --- |
| 宿主 | 同一存在、场景或虚拟存在 |
| 特征口径 | 同一实例特征，或同一父特征下按子特征优先规则选出的有效口径 |
| 前状态 | 可比较状态 |
| 后状态 | 可比较状态 |
| 时间 | 前后状态的发生时间或收到时间 |

输出至少包括：

- 基础动态事实
- 变化量
- 变化方向
- 持续时间
- 来源状态引用

状态动态和动作动态的区别固定为：

| 对比项 | 状态动态 | 动作动态 |
| --- | --- | --- |
| 描述对象 | 状态变化过程 | 动作过程事实 |
| 是否有动作主体 | 没有或不声明 | 有 |
| 主要用途 | 查找跨特征变化关系 | 查找同类特征变化原因 |
| 可信性 | 状态变化可信，原因未定 | 自我动作事实可直接可信，第三方动作需推断 |

### 5.2 存在动态

存在动态是场景观察某个存在的状态变化后形成的动态。

存在动态不得直接复述所有特征细节，而应回答：

```text
这个存在在本场景中发生了什么变化？
```

输入可以来自：

- 存在自身状态
- 关系变化
- 空间变化
- 行为阶段变化
- 虚拟存在运行状态变化

### 5.3 场景动态

场景动态是父场景观察子场景的状态变化后形成的动态。

场景动态不得绕过子场景直接读取子场景内部存在和特征。

场景动态应回答：

```text
这个子场景的整体态势、阶段或结构发生了什么变化？
```

### 5.4 动作动态

动作动态记录某个存在执行动作的过程事实。

动作动态不是普通状态转换动态的别名。

动作动态按来源分为两类：

| 类型 | 来源 | 可信性 | 稳定性 |
| --- | --- | --- | --- |
| 自我方法动作动态 | 自我真实调用方法函数时直接生成 | 动作事实可信 | 方法效果和结果稳定性仍需继续观察 |
| 第三方推断动作动态 | 从场景中的状态信息和中间原因交集推断生成 | 初始只是候选 | 需要多次观察、交集收敛和反例过滤后升格 |

动作动态最少应回答：

| 问题 | 数据来源 |
| --- | --- |
| 谁在动作 | 动态主体，自我动作通常是自我存在，第三方动作是被推断的原因存在 |
| 做了什么 | 来源动作、来源动作名、方法首节点或推断动作类型 |
| 何时发生 | 动态时间段 |
| 输入来自哪里 | 来源输入场景 |
| 输出写到哪里 | 来源输出场景 |
| 是否成功 | 来源执行成功 |
| 为什么失败 | 来源错误码 |

自我方法动作动态的落点固定为自我所在场景。

第三方推断动作动态的落点固定为观察到外显状态变化的场景。

### 5.5 自我方法动作动态

自我方法动作动态来自自我的方法。

当自我方法真实执行时，必须直接产生动作动态。这条动态记录的是“自我确实执行了某个方法”的动作事实，因此动作事实可信。

但是：

1. 动作事实可信，不代表该方法稳定。
2. 动作事实可信，不代表该动作一定导致了目标结果。
3. 方法效果、结果方向、适用条件和稳定性仍需要通过后续状态变化、动态二次特征和因果稳定性继续观察。

自我方法动作动态不得由筹办、消息处理、任务状态推进等非真实方法执行流程伪造。

### 5.6 第三方推断动作动态

第三方推断动作动态来自场景中的状态信息。

当场景中某个存在的外显特征发生改变时，会形成前后两个状态信息：

```text
目标存在 E
外显特征 F
前状态 S0: E.F = v0, 时间 t0
后状态 S1: E.F = v1, 时间 t1
```

`t0` 到 `t1` 之间的状态信息，都可能是导致该外显特征变化的原因候选。

第三方动作推断主链固定为：

```text
外显特征状态变化 S0 -> S1
-> 取 t0 到 t1 之间的中间状态信息集合
-> 多次同类变化样本分别取中间状态集合
-> 对中间状态集合做语义交集
-> 得到稳定出现的候选原因信息
-> 找到候选原因信息所属存在
-> 生成该存在导致目标变化的第三方动作动态候选
```

这里的“交集”必须按语义做，不按实时主键做。

推荐语义键至少包含：

| 语义项 | 用途 |
| --- | --- |
| 原因存在 | 谁可能是原因 |
| 原因存在类型 | 原因主体的稳定类别 |
| 原因特征口径 | 哪个外显或可观察特征发生变化 |
| 原因方向 | 增加、减少、保持或其他可比方向 |
| 目标存在类型 | 被影响对象的稳定类别 |
| 目标外显特征口径 | 哪类外显特征被改变 |
| 目标方向 | 目标状态变化方向 |
| 时间窗口关系 | 原因状态是否稳定落在目标变化前后合理窗口内 |

第三方推断动作动态的主体是候选原因信息所属存在。

第三方推断动作动态的客体或结果指向被改变外显特征的目标存在。

第三方推断动作动态必须保留来源链：

- 目标前状态
- 目标后状态
- 中间状态信息集合
- 交集后的候选原因信息
- 观察场景
- 时间窗口
- 样本次数
- 反例次数

第三方推断动作动态的升格规则：

| 阶段 | 条件 | 含义 |
| --- | --- | --- |
| 候选动作动态 | 单次外显变化中发现中间原因信息 | 只能说明可能相关 |
| 临时动作动态 | 多次样本中交集保留同一原因语义 | 说明有重复共现 |
| 待验证动作动态 | 反例较少，时间窗口稳定 | 可进入因果验证 |
| 稳定第三方动作动态 | 多次复现且能解释目标变化 | 可作为场景中第三方动作事实使用 |

硬规则：

1. 第三方动作动态不能单次直接视为可信动作事实。
2. 第三方动作动态的可信度来自多次观察的交集收敛。
3. 中间状态信息的交集必须排除目标结果状态自身，避免把结果当原因。
4. 找不到原因存在时，只能保留原因候选状态，不能生成第三方动作动态。
5. 若多个原因存在长期共同出现，应保留候选集合，不得强行只选一个。
6. 若原因存在出现但目标变化不出现，应计入反例并降低可信度。
7. 第三方动作动态只能由观察该外显变化的场景生成，不得由父场景跨层生成。

第三方原因不要做复杂结构表达。稳定前使用来源状态集合、候选原因信息和来源链表达；稳定后使用因果信息表达。

第三方动作的具体方法可能对自我不可见。此时只能抽象为“某个第三方动作或方法可以达成该结果”，不得伪造成自我已经掌握的具体方法。

### 5.7 因果动态

因果动态实际上就是因果类。

它可以看作动态信息的抽象信息：多个动作动态、状态动态或聚合动态经过重复观察、交集过滤、反例统计和稳定性验证后，升格成因果信息。

因果动态不再只描述某一次变化，而是描述一类稳定关系。

因果动态至少应回答：

| 问题 | 含义 |
| --- | --- |
| 哪类前置动态 | 动作动态、状态动态或聚合动态中的哪类变化作为前置 |
| 哪类结果动态 | 哪类变化稳定出现在之后 |
| 适用对象 | 适用于哪个存在类型、场景类型或虚拟存在类型 |
| 适用特征口径 | 适用于哪些特征类型或细化口径 |
| 稳定程度 | 样本次数、反例次数、置信度或稳定阶段 |
| 抽象层级 | 是实例因果、类型因果还是更高层模板 |

因果动态的生成链固定为：

```text
动作动态 / 状态动态 / 聚合动态样本
-> 语义归并
-> 反例统计
-> 稳定性验证
-> 因果类实例
-> 因果模板或抽象因果
```

硬规则：

1. 因果动态只能从已有动态信息抽象出来，不能越过动态直接从底层状态生成。
2. 因果动态必须保留来源动态样本集合。
3. 因果动态不是动作动态，也不是状态动态；它是对动态关系的抽象。
4. 因果类是因果动态的正式结构承载。

从使用角度看，自我因果信息和第三方因果信息必须分开处理：

| 因果来源 | 含义 | 是否可直接作为自我方法使用 |
| --- | --- | --- |
| 自我因果信息 | 来自自我已执行方法、内部状态和结果确认链 | 可以作为自我方法筹办和复用依据 |
| 第三方因果信息 | 来自场景观察、外显状态变化和中间原因交集 | 不能直接作为自我方法使用，必须先转换 |

第三方因果信息更接近外界真实因果，但它只说明“某类原因能导致某类结果”，不保证自我知道具体动作函数、输入条件或执行路径。

第三方因果信息转换为自我方法前，至少要补齐：

- 自我可执行的动作函数或本能方法指针
- 自我可满足的输入条件
- 自我可写入的结果场景
- 目标特征类型和目标方向
- 稳定性和反例信息

在转换完成前，可以在第三方方法列表中建立抽象方法。抽象方法只表达“第三方存在某种方法可以达成该结果”，不得加入自我可执行方法列表，也不得让自我线程直接执行。

## 6. 多动态特征类型并存

动态通常至少同时涉及两类基础变化：

1. 时间变化
2. 被观察对象的值变化

同一父层可以观察多个直接子层动态，并生成同层协同动态或派生动态指标。

示例：

```text
存在层输入:
  特征A变化量
  特征B变化量
  持续时间

存在层输出:
  A 与 B 的协同变化
```

硬规则：

1. 多特征协同只允许在共同父层内生成。
2. “一个特征单位变化对应另一个特征多少变化”不能绕过父层直接从底层取值。
3. 涉及除法、比值、单位变化率、加权结果时，应标记为派生动态指标或更高层动态结果。
4. 派生指标必须保留来源动态集合，不能只保存孤立数值。
5. `动态主信息.动态特征` 只保存该动态直接描述的基础特征，不保存动态二次特征集合或派生指标。

## 7. 动态二次特征

动态二次特征是动态生成后的结果集合。

动态二次特征可以包括：

| 类型 | 含义 |
| --- | --- |
| 变化量 | 后状态值减前状态值 |
| 变化方向 | 大于 0、小于 0、等于 0 的方向结果 |
| 持续时间 | 动态起止时间差 |
| 幅度分类 | 微小、小、中、大等区间映射 |
| 趋势结果 | 回升、下降、持平、波动等聚合结果 |
| 聚合段结果 | 多个低层动态聚合后的段级结果 |

动态二次特征必须跟随动态层级。

禁止：

1. 场景直接为存在内部特征生成状态动态二次特征。
2. 父场景直接为子场景内部存在生成存在动态二次特征。
3. 把二次特征或派生指标塞回 `动态主信息.动态特征`。

## 8. 聚合规则

### 8.1 从细节到聚合

动态向上聚合时，应逐层减少底层细节，保留对父层有用的聚合结果。

示例：

```text
状态动态：安全值 +10，持续 3 秒
存在动态：自我安全状态回升
场景动态：内部治理场景风险下降
父场景动态：长期运行态进入恢复趋势
```

### 8.2 来源链

聚合动态必须能回溯来源，但上层不应重复持有全部底层字段。

应保留：

- 来源低层动态
- 来源状态或来源场景
- 动态层级
- 聚合方式
- 时间段
- 方向或聚合结果

### 8.3 语义收敛

上层动态必须回答本层问题：

| 层 | 需要回答的问题 |
| --- | --- |
| 存在层 | 这个存在自身发生了什么变化？ |
| 场景层 | 这个场景中的存在、关系或态势发生了什么变化？ |
| 父场景层 | 这个子场景整体发生了什么变化？ |

## 9. 与状态和因果的关系

状态是某一时刻的事实。

动态是父层比较直接子层前后状态后得到的变化事实。

因果只能使用已经形成的动态作为证据。因果不能替代动态获取过程，也不能跨层抓底层状态直接生成高层因果。

因果动态就是因果类承载的动态抽象信息。

状态动态主要为跨特征因果链提供样本；动作动态主要为同特征变化原因提供样本；因果动态则把这些样本抽象为稳定关系。

正确链条为：

```text
状态
-> 动作动态 / 状态动态
-> 动态二次特征
-> 因果候选
-> 因果动态 / 稳定因果
```

## 10. 状态信息清理规则

状态信息会高频产生，不能长期滞留在活动链中。

状态信息清理的目标不是删除事实，而是把已经完成处理的原始状态从活动处理链移出，避免压垮系统；需要长期保留的内容应进入当前状态、动态事实、因果候选、因果信息或事件记录。

状态信息清理不采用单纯限量或单纯限时。固定规则为：

```text
状态清理 = 生命周期完成清理 + 限时窗口 + 限量兜底
```

### 10.1 三种清理机制

| 机制 | 用途 | 优先级 |
| --- | --- | --- |
| 生命周期完成清理 | 状态完成处理后立即移出活动链 | 主机制 |
| 限时窗口 | 保留前后状态比较、中间状态交集和因果候选需要的短期窗口 | 保护机制 |
| 限量兜底 | 高频状态爆发时防止活动链无限增长 | 兜底机制 |

限时和限量都不能越过生命周期规则强行清理未处理状态。

### 10.2 状态分类与清理策略

| 状态类别 | 清理策略 |
| --- | --- |
| 当前状态 | 不清理，只更新同一宿主、同一特征口径的当前值 |
| 上一状态 | 短期保留，用于下一次比较；比较完成后可进入已处理状态 |
| 未处理状态 | 禁止清理 |
| 已生成状态动态的原始状态 | 可清理，前提是动态事实和来源引用已经落账 |
| 因果窗口中的状态 | 窗口关闭前禁止清理 |
| 已进入因果候选或因果信息的状态 | 原始状态可清理，但必须保留来源链或可回溯引用 |
| 异常状态 | 对应任务、自检或错误处理完成前禁止清理 |

### 10.3 处理完成判定

状态信息满足以下条件后，才算处理完成：

1. 状态已经被所属宿主读取。
2. 同一宿主、同一特征口径的当前状态已经更新。
3. 必要的状态动态已经生成，或已经确认无变化。
4. 必要的动作动态、外显状态比较或因果候选窗口已经处理完成。
5. 没有任务、方法、自检、动态生成、因果验证正在引用该状态。
6. 需要保留的统计或来源关系已经进入动态、因果或事件信息。

处理完成后，原始状态信息应从活动状态链清理。

### 10.4 禁止清理条件

以下状态不得清理：

1. 尚未被读取的状态。
2. 当前状态。
3. 正在作为前状态、后状态或中间状态参与动态生成的状态。
4. 正在参与第三方动作推断中间状态交集的状态。
5. 正在被因果候选、因果验证、自检需求或任务处理引用的状态。
6. 尚未生成必要动态事实或事件记录的异常状态。

### 10.5 限时窗口

限时窗口用于保护动态和因果推断，不用于替代处理完成判断。

限时窗口至少覆盖：

- 前后状态比较窗口
- 动作动态结果观察窗口
- 外显状态确认窗口
- 第三方原因中间状态交集窗口
- 异常状态处理窗口

窗口关闭后，若状态已经处理完成，则应清理；若状态仍有未完成引用，则继续保留并由自检或任务线程发现滞留问题。

### 10.6 限量兜底

限量兜底用于高频状态爆发。

限量应按最小有效范围统计，优先级为：

1. 同一宿主、同一特征口径的活动状态数量。
2. 同一宿主的活动状态总量。
3. 同一场景的活动状态总量。
4. 全局活动状态总量。

超过限量时，只允许清理已经处理完成的状态。若超过限量但没有可清理状态，应生成事件信息或自检需求，不能强行删除未处理状态。

### 10.7 清理入口

状态清理入口固定为：

| 入口 | 触发条件 |
| --- | --- |
| 状态写入后处理 | 新状态落账后，清理同口径已处理旧状态 |
| 状态动态生成后 | 前后状态已经生成动态或确认无变化 |
| 因果窗口关闭后 | 中间状态交集、反例统计或因果候选已完成 |
| 休眠期整理 | 系统低负载时清理已处理滞留状态 |
| 限量保护 | 活动状态数量超过上限时触发 |

清理入口只能清理已处理状态。休眠期整理可以做更大范围扫描，但不得反复全量扫描整个休眠期；每次休眠只按需要执行一次或有限批次。

### 10.8 清理后的保留信息

原始状态清理后，必须至少保留下列一种长期信息：

- 当前状态值
- 状态动态事实
- 动作动态结果
- 因果候选来源链
- 因果信息来源链
- 异常事件记录

如果没有任何长期信息可保留，说明该状态没有形成有效后续信息；清理时只记录事件即可。

## 11. 与现有实现的关系

当前 `动态类` 可以继续作为底层创建服务，但调用者必须遵守观察层边界。

后续实现应逐步拆分为：

| 入口 | 职责 |
| --- | --- |
| 存在层动态入口 | 存在观察自身实例特征状态变化，生成状态动态 |
| 场景层动态入口 | 场景观察存在状态变化，生成存在动态 |
| 父场景动态入口 | 父场景观察子场景状态变化，生成场景动态 |
| 自我方法动作动态入口 | 真实方法调用时直接生成自我方法动作动态 |
| 第三方动作推断入口 | 场景观察外显状态变化和中间状态交集，生成第三方动作动态候选 |
| 第三方抽象方法入口 | 稳定第三方因果形成后，在第三方方法列表中建立不可直接执行的抽象方法 |

在完成结构拆分前，统一动态创建接口必须显式保存：

- 观察层
- 被观察子层
- 来源状态或来源动态
- 动态层级
- 聚合方式

这些信息应通过结构化字段、引用或特征表达，不能通过自由文本调用点表达。

状态清理实现应优先放在状态与动态后处理路径中，不能让任务、方法、线程节点直接承载清理状态特征。若清理过程需要记录运行情况，应通过所属存在、场景或虚拟存在的特征、事件或日志表达。

## 12. 验收规则

后续代码实现应满足：

1. 状态动态不会由跨层观察直接生成。
2. 存在动态不会由父场景直接生成。
3. 场景动态不会绕过子场景直接读取子场景内部存在或特征。
4. 任务、方法、线程动态均通过虚拟存在特征进入分层链条。
5. 真实方法调用必须生成动作动态；非真实调用不得伪造动作动态。
6. 方法功能结果与方法运行状态分别落到结果场景和方法虚拟存在。
7. 动态二次特征能按来源动态和层级回溯。
8. 多特征协同变化能保留来源动态集合。
9. 新动态接口不再要求调用方传入调用点自由文本。
10. 第三方动作动态必须保留目标前后状态、中间状态集合、原因交集和反例统计。
11. 第三方动作动态在稳定前只能作为候选或待验证动态使用。
12. 因果动态必须由因果类承载，并保留来源动态样本集合。
13. 当前阶段不得为外显状态手工新增特征类型，新特征类型应由后续观察自动认知形成。
14. 自我内部状态挂载点保持不变，外显状态必须通过另外途径观察并与内部状态比较确认。
15. 第三方因果信息不能直接作为自我可执行方法；转换完成前只能进入第三方方法列表的抽象方法。
16. 状态信息处理完成后必须从活动状态链清理。
17. 限时和限量只能清理已处理状态，不能强行清理未处理状态、当前状态或仍被引用的状态。
18. 状态清理后必须保留当前状态、动态事实、因果来源链或事件记录中的至少一种长期信息。

## 13. 最终收口

动态信息的正式定义为：

```text
动态 = 父层观察直接子层状态变化后形成的变化信息。
```

动态获取必须逐层进行，动态聚合必须保留来源链。特征规范规定动态的底层事实来自实例特征，方法规范规定真实方法调用必须生成自我方法动作动态；当前阶段自我状态仍落在自我内部，外显状态只能通过另外途径观察确认，不为此手工新增特征类型。状态动态用于发现不同特征类型之间的变化关系，场景外显状态变化可以通过中间状态交集推断第三方动作动态，因果动态则由因果类承载动态抽象信息。自我因果可以作为自我方法筹办依据，第三方因果需要转换后才能成为自我方法；转换前只能形成第三方方法列表中的抽象方法。处理完成的状态信息必须及时从活动链清理，限时和限量只是保护机制。本规范把这些来源收束为同一条分层动态链。
