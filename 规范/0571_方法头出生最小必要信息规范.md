# 0571 方法头出生最小必要信息规范

## 1. 目标

本文用于固定以下问题：

- 一个方法什么时候可以被承认为“已存在”
- `方法头出生` 最少必须具备哪些信息
- 哪些信息属于出生必需，哪些信息可以后补
- 为什么“当前不可用 / 不可执行 / 不成熟”都不阻止方法头出生

本文只收口 `方法头出生` 的最小必要信息，不讨论方法成熟度、当前可用性、当前可执行性和最终裁决。

## 3. 核心结论

必须固定以下 14 条：

1. `方法头出生` 的门槛，只回答“这个方法能不能被结构化承认”，不回答“它现在能不能用、能不能执行、是不是成熟”。
2. 一个方法要被承认为“已存在”，必须至少具备一组最小必要信息。
3. 这组最小必要信息的目标，是让该方法能被区分、能被挂来源、能被挂到方法树、能被后续继续补全。
4. 当前可用性不是方法头出生的必要条件。
5. 当前可执行性不是方法头出生的必要条件。
6. 当前成熟度不是方法头出生的必要条件。
7. 完整因果链不是方法头出生的必要条件。
8. 完整条件模板和完整结果模板都不是方法头出生的必要条件。
9. 方法头出生后，未知信息允许以后补；但最小必要信息缺失时，不得把模糊印象直接冒充正式方法头。
10. 方法头出生的最小必要信息，应优先落到现有 `方法首节点主信息 + 方法虚拟存在` 承载面，而不是另造独立结构。
11. 当前工程允许存在一种最低配的 `需求位方法头`：它只表达 `需求信息 / 目标特征类型 / 主作用方向`，通常在任务分解时生成，用于显性化“当前需要什么方法”。
12. `需求位方法头` 只解决“方法位已显性化”，不自动等于“可执行方法头”；它不得直接进入执行，也不得直接进入正式学习，必须待动作骨架与条件侧补齐后，才可进一步转成可执行方法头。
13. 对可执行方法树，若 `动作不同` 或 `条件组合不同`，则应独立生成另一棵方法树，不再强求把多条件同动作揉进同一个头下。
14. `结果特征类型` 主要承担 `索引 / 粗召回 / 候选归桶` 的职责，不单独承担“方法头并头”职责；是否为同一棵方法树，仍须回到 `动作 + 条件组合 + 主结果语义` 判定。

一句话收口：

`方法头出生只要求“足够被结构化承认”，不要求“已经足够好用”；出生看最小必要信息，后续好不好用再分别判。`


## 4. 最小必要信息组

一个方法要被承认为“已存在”，最少应具备以下 `3+1` 组信息：

### 4.1 身份锚点

回答的是：

- 这到底是哪一个方法
- 怎样把它和别的方法区分开

允许的承载形式至少包括：

- 动作句柄类型 + 动作句柄主键
- 外部定义键
- 路径骨架键
- 当前工程允许的临时方法头键

必须固定：

- 没有身份锚点，就不能正式建立方法头。
- 身份锚点只负责“区分是谁”，不负责证明它当前可执行。
- 对可执行方法树，工程上的最小唯一性检查，必须至少覆盖：`动作键 + 条件主键 + 主结果特征类型 + 主结果方向或主结果语义`。

### 4.2 主作用表达

回答的是：

- 这个方法主要是干什么的
- 它主要想产出什么结果或承担什么结构作用

允许的最小表达形式至少包括：

- 至少一项目标结果特征
- 至少一项主结果方向
- 至少一个结构作用类别，如 `观察 / 补条件 / 执行 / 组合执行 / 收束`
- 对 `需求位方法头`，允许当前只落 `目标特征类型` 或等价目标结果特征

必须固定：

- 如果连“它大致干什么”都无法表达，就还不应冒充正式方法头。
- 主作用表达允许粗，但不能完全空白。
- 若当前仅有 `目标特征类型`，则该头默认定位为 `需求位方法头`，而不是可直接执行或可直接学习的方法头。

### 4.3 来源归因

回答的是：

- 这个方法主要怎么来的

允许的写法：

- 正式来源值：`本能 / 外部导入 / 因果抽象 / 观察抽象 / 路径提炼`
- 过渡兼容值：`运行时学习沉淀`

必须固定：

- 来源归因在出生时就应可挂接。
- 若一时无法精确分类，允许用旧兼容值暂挂，但后续应进入精确化修订。

### 4.4 最小结构承载

回答的是：

- 这个方法至少以什么结构形式被承认

允许的最小承载形式至少包括其一：

- 动作原语或动作句柄壳
- 条件-结果骨架
- 观察-结果骨架
- 路径骨架或任务子树骨架
- 外部定义映射壳

必须固定：

- 这里要求的是“至少有一种结构壳”，不是要求已经完整可执行。
- 没有最小结构承载，就还是停留在印象、候选或描述层，不应直接出生正式方法头。
- 对 `需求位方法头`，允许当前最小结构承载只是 `需求位壳 / 目标特征类型壳`；但这种结构壳不足以直接进入学习执行。
- 对可执行方法树，真实条件仍应落在其条件节点场景中；头节点不强制保存条件并集/交集，后续若有需要，只允许把这类集合做成查询优化缓存，不得替代条件真值。


## 5. 哪些信息不是出生必需

以下信息可以后补，不属于方法头出生的最小必要信息：

1. 当前是否可用
2. 当前是否可执行
3. 当前是否已稳定复现
4. 当前是否已进入高成熟度阶段
5. 完整条件模板
6. 完整结果模板
7. 完整因果链
8. 当前默认召回权限
9. 当前默认执行权限

必须固定：

- 这些信息很重要，但属于出生后的补齐面，不得反向变成出生门槛。


## 6. 与相邻层的边界

### 6.1 与方法存在层

必须固定：

- 本文回答的是“方法存在层的最小落地条件”。
- 一旦满足本文要求，就允许进入 `方法存在` 层。

### 6.2 与方法可用层

必须固定：

- `可用` 回答“当前能不能进入候选面”。
- 即使当前不可用，只要满足本文最小必要信息，也可以先出生方法头。

### 6.3 与方法可执行层

必须固定：

- `可执行` 回答“当前这一轮能不能进执行闭环”。
- 即使当前不可执行，只要满足本文最小必要信息，也可以先出生方法头。

### 6.4 与方法成熟层

必须固定：

- `成熟` 回答“这个方法是否已积累足够证据值得高信任”。
- 方法头可以先出生，成熟度以后再补。

### 6.5 与学习执行层

必须固定：

- `方法头已出生` 不等于 `已经允许进入学习执行`。
- 只有目标特征类型、还没有动作骨架与条件侧的 `需求位方法头`，不得直接作为学习的直接对象。
- 学习可直接消费的，至少应是 `已具动作骨架且条件侧已存在或可补给` 的方法头或待学习方法骨架。
- 因此，`任务分解 -> 需求位方法头出生 -> 补动作/补条件 -> 再进入学习` 是允许路径；`需求位方法头一出生就直接进入学习` 则不允许。
- 若 `动作相同` 但 `条件组合不同`，则补动作/补条件后的转化应独立生成另一棵方法树，不得强制并回旧树。
- 若 `动作相同`、`条件组合也相同`，则补动作/补条件后的转化才应优先并回旧树，不得再生成第二个同键方法树。


## 7. 当前工程对齐口径

当前工程里，方法头出生的最小必要信息应优先对齐到：

1. `方法首节点主信息`
   - 身份锚点
   - 来源归因
   - 能表达主作用的最小字段
2. `方法虚拟存在`
   - 出生后的运行期未知项和后补项

必须固定：

- 新生方法头时，优先先把“最小必要信息”建上。
- 成熟度、权限、可靠度等后补项仍写到方法虚拟存在。
- 不得为了“出生时信息还不完整”就另造一套平行候选方法结构，长期绕开正式方法头。


## 8. 最终结论

后续统一按以下口径执行：

1. 方法头出生至少要有：`身份锚点 + 主作用表达 + 来源归因 + 最小结构承载`。
2. 当前可用、可执行、成熟都不是出生必要条件。
3. 允许先出生只表达 `需求信息 / 目标特征类型` 的 `需求位方法头`，但它不是可执行方法头。
4. `需求位方法头` 不得直接进入学习；只有补到至少具备 `动作骨架 + 条件侧` 后，才允许进入正式学习执行。
5. 对可执行方法树，`动作不同` 或 `条件组合不同` 时，应独立生成方法树；不得为了合并而把多个真实条件组合强塞进同一个头下。
6. `结果特征类型` 主要用于索引桶和快速查找，不单独承担并头职责；真实条件真值仍留在条件节点场景中。
7. 未知信息允许后补，但模糊印象不能直接冒充正式方法头。

最终统一收口为：

`只要已经能把一个方法区分出来、说明它大致干什么、说明它主要怎么来，并给它一个最小结构壳，就可以先承认它存在；至于能不能用、能不能执行、值不值得信任，后面再分层判断。`
