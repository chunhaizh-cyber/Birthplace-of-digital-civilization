# 0575 二次特征泛称与n次特征递推生成规范

## 1. 目标

本文用于把 `二次特征` 的泛称含义、严格层级含义，以及 `1次 -> 2次 -> 3次 -> ... -> n次` 的递推生成规则单独固定下来，重点回答以下问题：

- `二次特征` 到底是严格二阶，还是高次特征的泛称
- `1次特征` 转成 `2次特征` 时，为什么必须先知道 `比较方法`
- 为什么 `2次特征` 先负责判方向，而 `3次 / n次特征` 负责继续排序
- 为什么严格 `3次特征` 不是乘除结果，而是 `同需求轴上两个同向2次特征候选的排序结果`
- 乘法 / 除法 / 归一化 / 加权评分 这类结果为什么不再属于 `2次- n次特征族`

本文只讨论 `二次特征的命名口径` 与 `n次递推生成规则`，不单独讨论自然语言显示、UI 命名和具体类实现细节。

## 3. 核心定义

### 3.1 泛称与严格层级

必须固定：

1. 工程口语里的 `二次特征`，可以作为 `高次特征族` 的泛称，统指 `2次特征 / 3次特征 / ... / n次特征`。
2. 严格落文时，必须尽量写清楚具体阶次：
   - `2次特征`：由 `1次特征` 经过比较后生成的第一层高次特征，主要负责判方向
   - `3次特征`：在同一需求轴上，对两个同向 `2次特征` 候选继续排序，判谁更接近需求
   - `n次特征`：在上一层仍不足以完成优选时，继续对同向候选做更细排序
3. 若文中只写 `二次特征` 而未说明阶次，默认优先按“高次特征族泛称”理解。
4. 若文中在讲 `状态来源 / 动态来源 / 比较方法`，默认优先指向最底层的 `1次 -> 2次` 生成。

一句话收口：

`二次特征` 在工程里可以是泛称；严格计算时，必须拆成 `2次 / 3次 / ... / n次` 来理解。

### 3.2 严格2次特征的地位

必须固定：

1. 严格 `2次特征` 不是原始一次值。
2. 严格 `2次特征` 是 `1次特征` 比较后的第一层结构化结果。
3. 严格 `2次特征` 生成完成后，其正式数值承载应统一收口到 `int`。
4. 从这一层开始，后续更高层递推不再回到原始一次值重新发明算术，而是进入“同向候选排序”链。


## 4. 阶次递推生成规则

### 4.1 1次特征生成2次特征

`1次 -> 2次` 时，必须先知道 `比较方法`。

这里的 `比较方法` 负责回答：

- 这个一次特征按什么规则可比
- 左值和右值该如何对齐
- 哪个字段是正式参与相减的比较基值
- 相减顺序应固定成什么方向

正式母式固定为：

```text
比较方法 = M(F)
左基值 = M.提取可减基值(1次左值)
右基值 = M.提取可减基值(1次右值)
2次值 = 右基值 - 左基值
```

必须固定：

1. `比较方法` 只在 `1次 -> 2次` 这一步是强制前提。
2. 不同一次特征类型可以有不同 `比较方法`，但一旦进入正式生成，最终都必须落成 `int` 型 `2次值`。
3. 若没有明确 `比较方法`，就不得宣称已经正式生成了严格 `2次特征`。
4. `0574` 中的 `状态信息来源 / 动态信息来源`，都是在约束这一步怎么取左值、右值与比较窗口。

### 4.2 2次以后继续递推到n次

必须固定：

1. 从严格 `2次特征` 开始，`3次特征`、`4次特征` 直到 `n次特征`，不再承担“重新判方向”的职责，而承担“在同向候选里继续排序”的职责。
2. `2次` 以后的递推不再要求重新读取一次特征的 `比较方法`，但必须显式绑定 `需求轴` 与 `候选集`。
3. `2次` 以后的递推规则固定为：`对同需求轴上的同向候选做更细优选`。

正式递推母式固定为：

```text
2次特征 = 先判断方向
3次特征 = 在同方向的2次候选里判断谁更接近需求
4次特征 = 在3次仍不足以区分时继续细排
...
n次特征 = 在上一层排序仍不足以稳定选择时继续递推
```

必须固定：

1. 参与递推的输入候选必须属于 `同一需求轴`。
2. 参与递推的输入候选必须已经在低一层被判成 `同向`。
3. 更高阶特征应优先输出 `更接近 / 次优 / 等效 / 过头` 这类排序关系，而不是重新输出一次方向。
4. 更高阶特征的“来源解释”可以追溯到更早层，但其“运行时职责”已经转成排序与优选。

### 4.3 从2次继续升到n次的使用时机

必须固定：

1. 严格 `2次特征` 的主要用途之一，是先把“不同”找出来。
2. 若 `2次特征` 已足以完成区分、比较、定位，就不必为了形式继续升阶。
3. 若 `2次特征` 已能给出方向，但同方向上出现多个候选且无法直接优选，则允许继续生成 `3次 / 4次 / ... / n次特征`。
4. 因此，`n次特征` 的主要用途不是替代 `2次特征`，而是在 `2次特征` 已完成方向判定后继续完成同向候选排序。

可统一理解为：

```text
1次特征 -> 先比较 -> 得到2次特征 -> 先找不同
若同向候选还无法优选 -> 对2次特征继续做3次排序
若仍不够清楚 -> 继续递推 -> 直到足以支持选择
```

### 4.4 来源继承

必须固定：

1. `状态来源 / 动态来源` 主要约束严格 `2次特征` 的生成起点。
2. `3次` 及以上特征不再新增“第三种来源”“第四种来源”。
3. `3次` 及以上只继承下层来源链，不另外改写算术规则。
4. 换言之：
   - 来源语义从 `2次层` 往上继承
   - 排序语义从 `2次层` 往上统一为 `同向候选优选`


## 5. 算术边界

必须固定：

1. `2次特征` 到 `n次特征` 整个特征族，都禁止进入 `乘法`。
2. `2次特征` 到 `n次特征` 整个特征族，都禁止进入 `除法`。
3. `变化率 / 速率 / 比例 / 归一化差异 / 加权值 / 综合评分` 不属于严格 `3次特征`，也不属于更高 `n次特征`。
4. 这些结果若确有业务价值，应落到 `派生指标 / 统计结果 / 裁决辅助结构`，不得继续冒充 `n次特征`。
5. 因此，“乘除产物升格为三次特征”属于旧口径，现正式废止。
6. 其中 `除法` 尤其应被视为更高层派生概念；在当前阶段，正式高次特征链先不引入它。

一句话收口：

`严格3次特征` 不是乘除结果，而是 `同需求轴上两个同向2次特征候选的排序结果`；除法仍属于当前阶段不引入的更高层派生概念。


## 6. 字段建议

若系统需要稳定承接 `2次- n次特征`，建议至少保留：

- `阶次`
- `来源阶次`
- `比较方法`
- `左值`
- `右值`
- `差值`
- `方向区间`
- `来源链`

必须固定：

1. `比较方法` 在 `1次 -> 2次` 时必须明确。
2. `3次` 及以上若只是继续递推，可以复用继承下来的 `比较方法来源说明`，但不应再次把它当作算术核心。
3. `3次` 及以上应额外明确 `需求轴 / 候选组 / 排序结论`，避免把高阶排序误写成单值差分。


## 7. 审计清单

后续审计实现时，至少检查以下 8 项：

1. 是否把 `二次特征` 错写成只能指“严格二阶”。
2. 是否把 `1次 -> 2次` 的 `比较方法` 漏掉了。
3. 是否把 `2次值` 正式收口为 `int`。
4. 是否在 `2次 -> 3次` 时又重新回到一次特征比较。
5. 是否只在 `同需求轴 + 同向候选` 成立时才继续生成更高阶。
6. 是否把 `变化率 / 比例 / 归一化 / 加权值` 错叫成 `3次特征`。
7. 是否把乘法 / 除法 混进了 `2次- n次特征` 的正式母式。
8. 是否能从高阶结果回溯到其下层 `左值 / 右值 / 差值` 链。


## 8. 最终收口

`二次特征` 在工程里是从 `2次特征` 到 `n次特征` 的高次特征泛称；严格 `2次特征` 由 `1次特征` 依据固定比较方法相减得到并负责方向判定，严格 `3次特征` 及以上则统一用于同需求轴上的同向候选排序；凡乘除、归一化、加权、比例类结果，均不属于这条正式高次特征链。`
