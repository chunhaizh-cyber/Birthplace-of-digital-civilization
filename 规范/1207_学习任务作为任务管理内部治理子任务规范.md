## 3. 核心结论

### 3.0 2026-04 最新口径修正

若本文下文旧表述与本节冲突，以本节为准。

必须新增固定以下 7 条：

1. `学习任务` 在当前体系中优先固定为 `唯一可复用的学习任务执行壳`，不再优先理解为“多条长期并行学习子任务”。
2. 真正需要排队、暂停、恢复、切换的，是绑定到不同 `待学习方法首节点` 的 `学习工单`。
3. 学习工单不要求都实例化成独立任务节点；优先做法是把工单进度缓存在 `学习任务头节点` 的 `任务虚拟存在` 中。
4. 当某张工单被选中时，允许先把其缓存上下文复制回当前 `学习任务头节点`，再开始这一轮正式学习。
5. `任务管理任务` 负责管理学习工单队列与当前激活工单；`学习工作线程` 只负责执行当前激活工单。
6. `学习工作线程` 可以在方法 A 学到一半时暂停，切去方法 B；恢复时继续读取方法 A 的缓存工单上下文。
7. 学习结果首先服务于 `待学习方法首节点` 的补全与可用化，不再把“稳定度”作为主判词，统一改成 `有无 / 可用 / 优选`。

必须固定以下 10 条：

1. `学习任务` 不属于新的原生根任务。
2. `学习任务` 的直接来源，不是系统外部手工注入的新目标，而是 `任务管理任务` 从 `安全/服务根任务` 的真实推进过程中上浮出来的衍生需求。
3. `学习任务` 的主结果是 `内部治理结果`，例如：
   - `新增可用方法`
   - `把方法补到可执行`
   - `把方法推进到优选`
   - `补齐条件`
   - `补齐结果`
   - `补齐最小动作原语或治理本能组合`
4. 因为这些结果属于内部治理状态，所以 `学习任务` 属于 `任务管理任务提出治理需求、由外层批准挂载、再继续回到任务管理链消费的内部治理子任务执行壳`。
5. `学习任务` 的目标不是直接替代叶子执行，而是围绕 `学习目标 -> 学习需求 -> 候选学习程序` 这条链，完成对 `学习对象方法` 的完善，为原任务或同类任务扩充方法资产、修复治理缺口、扩大可选范围。
6. `学习任务` 可以由 `安全根任务` 派生，也可以由 `服务根任务` 派生；当多条派生需求相近时，允许共用一个共享学习执行器。
7. `学习任务` 即使由多个根任务共享执行，也必须保留可追溯的 `来源根任务 / 来源宿主任务 / 来源步骤 / 来源方法需求位 / 来源本能能力缺口类型 / 来源最小动作原语位图 / 来源主观察特征 / 方法服务目标 / 服务方法位或方法枝`。
8. `有目标学习`、`弱定向学习`、`泛化学习` 都属于学习任务的合法形态，只是目标约束强度不同。
9. `泛化学习` 也不是无需求学习；它仍然来源于任务管理任务上浮出的 `扩选 / 健壮 / 覆盖 / 恢复 / 冗余 / 可验证` 等内部治理压力。
10. 当前实现允许保留 `全局共享学习执行器`，但这不改变学习任务在语义上属于 `任务管理内部治理子任务` 的定位；当当前激活工单就绪时，真正执行并完善 `学习对象方法` 的，是学习任务当前轮选中的那条学习方法，其中本能函数 `尝试学习方法` 只是可选执行入口之一。

可收敛为一句话：

`学习任务不是第三根任务，而是由任务管理任务从安全/服务推进压力中上浮出派生需求、经外层批准挂载后再回到任务管理链消费的内部治理子任务执行壳。`


## 4. 为什么学习任务属于内部治理子任务

判断学习任务归属，不应看它是否运行在线程里，而应看它改变的对象是谁。

学习任务的主结果不是：

- 外部设备状态改变
- 现实世界事实直接改变
- 对外表达动作完成

学习任务的主结果是：

- 某需求位附近 `可用方法数` 增加
- 某方法的 `可执行性` 或 `优选资格` 提升
- 某类条件或结果模板被补齐
- 原宿主任务的 `等待学习缺口` 被弥补

因此，学习任务直接改变的是：

- 方法资产
- 方法树可用面
- 方法可用面与优选面
- 任务管理任务后续可选的治理路径

这些都属于 `内部治理状态`，所以学习任务应归类为：

`由任务管理任务提出并持续治理边界的内部治理子任务`


## 5. 学习任务的来源链

学习任务后续统一按以下来源链理解：

```text
安全/服务根需求
-> 派生真实业务任务
-> 任务管理任务推进业务任务
-> 识别出扩选/健壮/覆盖/恢复等治理压力
-> 命中方法需求位并识别必要时的本能能力缺口
-> 上浮任务管理衍生需求
-> 形成学习任务候选
-> 返回外层，由自我线程决定是否实例化学习任务
-> 学习任务进入正式任务体系
-> 再进入下一轮任务管理任务
-> 新的筹办阶段命中学习任务兜底方法
-> 生成可执行学习方法选项
-> 任务管理任务推进学习任务
-> 学习任务就绪
-> 选择并执行当前轮学习方法（`尝试学习方法` 只是其中一个选项）
-> 直接完善学习对象方法
-> 条件-结果对融合写入方法
-> 形成学习情况
-> 反馈给任务管理任务
-> 任务管理任务决定继续学习/挂起/使用该方法/再次上浮学习需求
```

可直接压成一句标准主链：

`任务管理任务上浮学习需求 -> 外层批准并挂载学习任务 -> 下一轮筹办中获得学习方法选项 -> 学习任务执行并完善学习对象方法 -> 学习情况反馈任务管理任务 -> 任务管理任务决定后续消费动作`

必须禁止以下错误理解：

- 不能把学习任务理解成“系统外部额外塞进来的成长任务”
- 不能把学习任务理解成“与安全、服务并列的第三根任务”
- 不能把学习任务理解成“与任务管理任务无关的旁路线程工作”


## 6. 三类学习任务的统一定位

## 6.1 定向学习

含义：

- 已经知道某个宿主任务当前缺哪类能力

特点：

- 来源任务和服务目标最明确
- 最容易在反馈后被任务管理任务用于原任务继续推进

在本文中的定位：

- `任务管理内部治理子任务`


## 6.2 弱定向学习

含义：

- 已经知道缺口大致在哪个需求位，但还不能锁定唯一方法

特点：

- 来源任务明确
- 目标能力片区明确
- 结果可能是原方法的变体，也可能是邻近替代方法

在本文中的定位：

- `任务管理内部治理子任务`


## 6.3 泛化学习

含义：

- 不是补某一个精确缺口，而是围绕某个需求位扩大可选范围、提升健壮性、扩展覆盖面

特点：

- 仍然来自任务管理衍生压力
- 结果默认先进入候选方法资产
- 不能直接宣称“已补齐当前任务缺口”

在本文中的定位：

- `任务管理内部治理子任务`


## 7. 学习任务与根任务的关系

学习任务对根任务的关系，后续固定成：

### 7.1 在语义上

- `学习任务` 是 `安全根任务 / 服务根任务` 的衍生子任务执行壳
- 更准确地说，是由这两条根任务的 `任务管理链` 派生出来的学习子任务

### 7.2 在执行上

- 允许多个来源根任务共用一个共享学习执行器
- 允许当前实现保留 `全局常驻学习任务 + 学习工单队列 + 尝试学习分支`

### 7.3 在反馈消费上

- 学习情况必须能追溯到来源根任务
- 同时也必须允许在反馈被消费后服务 `同需求位的同类任务`

因此，统一口径应为：

`学习任务可以物理共享执行器，但语义上仍属于由根任务治理压力派生出的内部治理子任务执行壳；真正切换的是学习工单。`


## 8. 任务管理任务对学习任务的治理边界

`任务管理任务` 至少应直接管理以下内容：

- 是否生成学习衍生需求
- 当前学习类型：`定向 / 弱定向 / 泛化`
- 来源根任务、来源宿主任务、来源步骤、来源方法需求位
- 是否进入等待学习
- 学习任务是否进入就绪
- 对现实影响型学习是否允许进入 `受控微试探`
- `受控微试探` 的允许范围与停止条件
- 学习完成后如何消费 `学习情况`
- 已完善的方法当前允许服务 `当前任务`、`当前根任务` 还是 `全局方法资产`
- `来源本能能力缺口类型`
- `来源最小动作原语位图`
- `来源主观察特征`
- 已完善的方法当前对准哪个 `方法位/方法枝`
- 当前轮选择哪条 `学习方法`
- 何时触发哪条 `学习方法`，其中 `尝试学习方法` 只是可选入口之一

`任务管理任务` 不一定需要亲自执行学习算法，但必须直接管理：

- 学习派生需求的提出
- 学习工单的推进与就绪判断
- 学习工单的目标边界
- 学习工单的受控边界
- 学习工单的执行触发
- 学习情况的消费条件
- 学习工单的结束条件

必须补充固定：

- `任务管理任务` 直接管理的是 `学习需求的上浮与治理边界`，不是 `正式学习任务实例化审批` 本身。
- `学习任务执行壳` 的创建、挂载与批准，仍由 `自我线程 / 学习调度层` 负责；工单排队与切换则由 `任务管理任务` 负责。

- 对 `现实影响型学习`，`任务管理任务` 负责先完成 `是否允许试探 / 允许作用到哪里 / 允许多大幅度 / 允许多久 / 何时必须停止` 的治理判定。
- 共享学习执行器承载的，是在上述边界内调度并运行被选中的学习方法，其中可以包含本能函数 `尝试学习方法`，而不是代替任务管理任务做受控裁决。
- 本能函数 `尝试学习方法` 作为尝试型学习方法选项之一，负责在上述边界内执行当前 `待学习方法` 的一轮完善，并把本轮得到的 `条件-结果对` 以融合写口直接写入 `方法账本 / 条件节点 / 结果节点 / 方法虚拟存在`。
- 这种直接写入首先只表示方法证据面和补全面已更新，不自动等于允许默认召回、默认执行或正式升格。
- 学习任务当前轮完成后，只向 `任务管理任务` 反馈 `学到了什么 / 方法补到了哪一层 / 是否还需要继续学习` 等学习情况；是否继续、挂起、切换到另一张工单或使用该方法，由 `任务管理任务` 决定。
- 当当前缺的不是“高层方法枝”，而是支撑某个 `方法需求位` 落地所需的最小动作原语时，学习任务也允许直接服务 `本能能力缺口`，而不强迫先伪造一条完整方法枝。


## 9. 当前实现的对齐判断

当前实现已经对齐到以下程度：

1. 任务管理总控已经能识别“无可用方法”并让宿主任务进入 `等待学习`。
2. 任务执行器已经能把来源任务/步骤/方法/结果/错误码写入尝试学习请求。
3. 学习线程已经能推进 `采样 / 更新 / 验证 / 提交 / 回滚`。
4. 学习结果已经能以 `可用方法增量 / 优选方法增量 / 可用证据增量` 这类内部治理结果回写。
5. 学习提交后已经能唤醒等待学习的原任务，并重新进入任务管理实例链。
6. 当前工程已经区分 `补方法` 与 `补原语/治理本能` 两条学习承接通道。
7. `影子验证 / 正式提交门槛 / 允许正式资产提交` 已进入正式字段，并已可通过运行日志、摘要与运行时快照持续核对。

因此，当前实现已经满足：

- `学习任务流已被任务管理任务管理`

但尚未完全满足：

- `学习反馈 / 外部反馈 / 执行结果 / 恢复重建` 已被统一纳入显式事件邮箱
- `学习待外层裁决 / 等待正式提交放行` 已被统一纳入延迟事实车道并参与主循环最终去向裁决

换句话说：

- 当前代码已经完成了学习承接显式化与正式提交门槛收口
- 下一阶段要补的重点，已经从“能否进入 mailbox”转成“事件优先级、延迟事实持久化与长期消费策略”


## 10. 后续收口方向

后续只允许沿以下两个方向之一继续收口：

### 10.1 共享执行器收口

- 保留全局学习执行器
- 但补齐 `来源根任务 / 来源宿主任务 / 目标需求位 / 来源本能能力缺口类型 / 来源最小动作原语位图 / 来源主观察特征 / 服务目标方法位或方法枝` 的统一状态面
- 继续由任务管理任务负责学习需求上浮、治理边界判定与反馈消费；正式学习任务实例化仍由外层批准

### 10.2 同构任务化收口

- 把学习任务也改造成标准宿主任务
- 让其进入任务管理实例队列

## 11. 收口结论

后续统一按以下判断执行：

1. `学习任务` 的来源是 `任务管理任务衍生需求`
2. `学习任务` 的结果是 `内部治理结果`
3. 因此 `学习任务` 属于 `任务管理任务提出需求并持续治理边界的内部治理子任务`
4. 当前实现已具备这条语义的大部分基础
5. 后续要做的主要是运行模型和状态面的继续收口，而不是重新发明新的根需求
