## 3. 核心结论

必须固定以下 10 条：

1. `学习任务` 不属于新的原生根任务。
2. `学习任务` 的直接来源，不是系统外部手工注入的新目标，而是 `任务管理任务` 从 `安全/服务根任务` 的真实推进过程中上浮出来的衍生需求。
3. `学习任务` 的主结果是 `内部治理结果`，例如：
   - `新增可用方法`
   - `提升方法稳定性`
   - `补齐条件`
   - `补齐结果`
   - `补齐最小动作原语或治理本能组合`
4. 因为这些结果属于内部治理状态，所以 `学习任务` 属于 `任务管理任务可直接管理的内部治理子任务`。
5. `学习任务` 的目标不是直接替代叶子执行，而是为原任务或同类任务扩充方法资产、修复治理缺口、扩大可选范围。
6. `学习任务` 可以由 `安全根任务` 派生，也可以由 `服务根任务` 派生；当多条派生需求相近时，允许共用一个共享学习执行器。
7. `学习任务` 即使由多个根任务共享执行，也必须保留可追溯的 `来源根任务 / 来源宿主任务 / 来源步骤 / 来源方法需求位 / 来源本能能力缺口类型 / 来源最小动作原语位图 / 来源主观察特征 / 回流目标 / 回流方法位或方法枝`。
8. `有目标学习`、`弱定向学习`、`泛化学习` 都属于学习任务的合法形态，只是目标约束强度不同。
9. `泛化学习` 也不是无需求学习；它仍然来源于任务管理任务上浮出的 `扩选 / 健壮 / 覆盖 / 恢复 / 冗余 / 可验证` 等内部治理压力。
10. 当前实现允许保留 `全局共享学习执行器`，但这不改变学习任务在语义上属于 `任务管理内部治理子任务` 的定位。

可收敛为一句话：

`学习任务不是第三根任务，而是由任务管理任务从安全/服务推进压力中派生出来、并可被其直接管理的内部治理子任务。`


## 4. 为什么学习任务属于内部治理子任务

判断学习任务归属，不应看它是否运行在线程里，而应看它改变的对象是谁。

学习任务的主结果不是：

- 外部设备状态改变
- 现实世界事实直接改变
- 对外表达动作完成

学习任务的主结果是：

- 某需求位附近 `可用方法数` 增加
- 某方法的 `稳定复现能力` 提升
- 某类条件或结果模板被补齐
- 原宿主任务的 `等待学习缺口` 被弥补

因此，学习任务直接改变的是：

- 方法资产
- 方法树可用面
- 方法可靠度与稳定度
- 任务管理任务后续可选的治理路径

这些都属于 `内部治理状态`，所以学习任务应归类为：

`任务管理任务可直接管理的内部治理子任务`


## 5. 学习任务的来源链

学习任务后续统一按以下来源链理解：

```text
安全/服务根需求
-> 派生真实业务任务
-> 任务管理任务推进业务任务
-> 识别出扩选/健壮/覆盖/恢复等治理压力
-> 命中方法需求位并识别必要时的本能能力缺口
-> 上浮任务管理衍生需求
-> 派生学习任务
-> 学习结果回流
-> 原任务或同类任务获得新增可用方法/稳定性提升/最小动作原语补齐
```

必须禁止以下错误理解：

- 不能把学习任务理解成“系统外部额外塞进来的成长任务”
- 不能把学习任务理解成“与安全、服务并列的第三根任务”
- 不能把学习任务理解成“与任务管理任务无关的旁路线程工作”


## 6. 三类学习任务的统一定位

## 6.1 定向学习

含义：

- 已经知道某个宿主任务当前缺哪类能力

特点：

- 来源任务和回流目标最明确
- 最容易直接回流到原任务

在本文中的定位：

- `任务管理内部治理子任务`


## 6.2 弱定向学习

含义：

- 已经知道缺口大致在哪个需求位，但还不能锁定唯一方法

特点：

- 来源任务明确
- 目标能力片区明确
- 结果可能是原方法的变体，也可能是邻近替代方法

在本文中的定位：

- `任务管理内部治理子任务`


## 6.3 泛化学习

含义：

- 不是补某一个精确缺口，而是围绕某个需求位扩大可选范围、提升健壮性、扩展覆盖面

特点：

- 仍然来自任务管理衍生压力
- 结果默认先进入候选方法资产
- 不能直接宣称“已补齐当前任务缺口”

在本文中的定位：

- `任务管理内部治理子任务`


## 7. 学习任务与根任务的关系

学习任务对根任务的关系，后续固定成：

### 7.1 在语义上

- `学习任务` 是 `安全根任务 / 服务根任务` 的衍生子任务
- 更准确地说，是由这两条根任务的 `任务管理链` 派生出来的学习子任务

### 7.2 在执行上

- 允许多个来源根任务共用一个共享学习执行器
- 允许当前实现保留 `全局常驻学习任务 + 尝试学习分支`

### 7.3 在回流上

- 学习结果必须能追溯到来源根任务
- 同时也必须允许回流到 `同需求位的同类任务`

因此，统一口径应为：

`学习任务可以物理共享执行器，但语义上仍属于由根任务治理压力派生出的内部治理子任务。`


## 8. 任务管理任务对学习任务的直接管理边界

`任务管理任务` 至少应直接管理以下内容：

- 是否生成学习衍生需求
- 当前学习类型：`定向 / 弱定向 / 泛化`
- 来源根任务、来源宿主任务、来源步骤、来源方法需求位
- 是否进入等待学习
- 对现实影响型学习是否允许进入 `受控微试探`
- `受控微试探` 的允许范围与停止条件
- 学习结果是否允许直接回流
- 学习结果是写到 `当前任务`、`当前根任务` 还是 `全局方法资产`
- `来源本能能力缺口类型`
- `来源最小动作原语位图`
- `来源主观察特征`
- `回流到哪个方法位/方法枝`

`任务管理任务` 不一定需要亲自执行学习算法，但必须直接管理：

- 学习任务的出生
- 学习任务的目标边界
- 学习任务的受控边界
- 学习任务的回流条件
- 学习任务的结束条件

必须补充固定：

- 对 `现实影响型学习`，`任务管理任务` 负责先完成 `是否允许试探 / 允许作用到哪里 / 允许多大幅度 / 允许多久 / 何时必须停止` 的治理判定。
- 学习执行器负责的，是在上述边界内做真实试探执行，而不是代替任务管理任务做受控裁决。
- 当当前缺的不是“高层方法枝”，而是支撑某个 `方法需求位` 落地所需的最小动作原语时，学习任务也允许直接服务 `本能能力缺口`，而不强迫先伪造一条完整方法枝。


## 9. 当前实现的对齐判断

当前实现已经对齐到以下程度：

1. 任务管理总控已经能识别“无可用方法”并让宿主任务进入 `等待学习`。
2. 任务执行器已经能把来源任务/步骤/方法/结果/错误码写入尝试学习请求。
3. 学习线程已经能推进 `采样 / 更新 / 验证 / 提交 / 回滚`。
4. 学习结果已经能以 `可用方法增量 / 稳定方法增量 / 稳定度总分增量` 这类内部治理结果回写。
5. 学习提交后已经能唤醒等待学习的原任务，并重新进入任务管理实例链。
6. 当前工程已经区分 `补方法` 与 `补原语/治理本能` 两条学习承接通道。
7. `影子验证 / 正式提交门槛 / 允许正式资产提交` 已进入正式字段，并已通过白盒与半白盒验收。

因此，当前实现已经满足：

- `学习任务流已被任务管理任务管理`

但尚未完全满足：

- `学习回流 / 外部反馈 / 执行结果 / 恢复重建` 已被统一纳入显式事件邮箱
- `学习待外层裁决 / 等待正式提交放行` 已被统一纳入延迟事实车道并参与主循环最终去向裁决

换句话说：

- 当前代码已经完成了学习承接显式化与正式提交门槛收口
- 下一阶段要补的重点，已经从“能否进入 mailbox”转成“事件优先级、延迟事实持久化与长期回流策略”


## 10. 后续收口方向

后续只允许沿以下两个方向之一继续收口：

### 10.1 共享执行器收口

- 保留全局学习执行器
- 但补齐 `来源根任务 / 来源宿主任务 / 目标需求位 / 来源本能能力缺口类型 / 来源最小动作原语位图 / 来源主观察特征 / 回流目标方法位或方法枝` 的统一状态面
- 继续由任务管理任务负责学习任务的出生与回流

### 10.2 同构任务化收口

- 把学习任务也改造成标准宿主任务
- 让其进入任务管理实例队列
- 把学习阶段 `采样 / 更新 / 验证 / 提交 / 回滚` 全部任务化

必须固定：

- 无论选哪条路线，都不允许把学习任务重新定义成“独立第三根任务”


## 11. 收口结论

后续统一按以下判断执行：

1. `学习任务` 的来源是 `任务管理任务衍生需求`
2. `学习任务` 的结果是 `内部治理结果`
3. 因此 `学习任务` 属于 `任务管理任务可直接管理的内部治理子任务`
4. 当前实现已具备这条语义的大部分基础
5. 后续要做的主要是运行模型和状态面的继续收口，而不是重新发明新的根需求
