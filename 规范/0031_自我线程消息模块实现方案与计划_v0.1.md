## 3. 当前代码现实

当前仓库里，消息面已经切到新的正式入口：

- `结构_治理消息`
- `治理mailbox_`
- `本轮冻结治理批次_`
- `投递治理消息(...)`
- `执行尝试汇总表_`

当前这套入口的优点是：

1. 已有 `冻结批次` 语义。
2. 已有 `优先级` 与 `幂等键`。
3. 已有 `需求树变更` 这类控制面载荷桥位。
4. 主循环已具备“冻结一批再处理”的基本骨架。
5. `任务管理任务 -> 自我线程` 的三类事实上推，已经有协议与桥接入口。

当前这套入口的不足是：

1. 当前代码虽然已能把 `线程状态消息 / 派生需求消息 / 结果消息` 转成治理消息，并在主循环第 `2` 步完成一次入账，但后续 `归并与对账 / 二次特征 / 影响项 / 根层重判 / 派发与路由 / 结算` 仍大多停在骨架。
2. `首收件人 / 上浮 / 路由父任务` 规则虽已有模块壳，但与当前代码实现的对应关系仍未完全收平。
3. 消息处理逻辑已从旧入口抽到模块与 helper，但“入账之后的治理七步”还没有独立落成稳定接口。
4. 当前最明显的缺口已不再是“有没有消息入口”，而是“落账之后如何继续判、算、派、结”。

因此当前最稳的路线不是继续补旧事件，而是：

`在现有三类消息转换链已接通的基础上，继续把自我线程主循环补成“入账后七步治理”：归并与对账、二次特征、影响项、根值/需求树/任务权重、根层重判、派发与路由、结算/否定项/学习触发。`


## 4. 实现总原则

必须固定以下 `8` 条：

1. 先切协议，不保留旧事件入口。
2. 先协议和路由，再下沉任务实体。
3. 先让 `自我线程类` 调四个消息模块，再慢慢削薄线程内部旧逻辑。
4. 公开入口、冻结批次与活邮箱统一只使用 `结构_治理消息`。
5. 消息模块只解决 `怎么说话 / 谁先收 / 怎么分发 / 哪些特征变了 / 何时重判`，不提前扩业务算法。
6. 控制面消息逻辑留在线程；局部工序语义继续留在工作线程。
7. 本轮仍必须遵守 `冻结一批，清完这一批，再做一次决议`。
8. 初始落地阶段不强制所有消费者改成 `import`，允许 `.ixx + .h` 双入口并存。

## 6. 现有结构的复用与替换

### 6.1 直接复用

当前建议直接复用：

- `治理mailbox_`
- `本轮冻结治理批次_`
- `私有_消息排在前面(...)`
- `需求树变更` 载荷桥位

原因：

- 已在线程主循环中稳定存在
- 已支持优先级和冻结批次
- 只需替换元素类型，不必重写邮箱与冻结机制

### 6.2 新增桥接层

当前建议新增并统一使用：

- `结构_治理消息`
- `结构_治理消息头`
- `结构_治理消息优先级`
- `投递治理消息(...)`

统一规则：

1. 自我线程对外入口只接受 `结构_治理消息`
2. 冻结批次与活邮箱都直接持有 `结构_治理消息`
3. 后续四模块全部只吃 `结构_治理消息`

### 6.3 后续替换面

当协议稳定后，再逐步把：

- `投递治理外部反馈事件(...)`
- `投递治理需求树变更事件(...)`

继续下沉为基于 `结构_治理消息` 的更细分投递器。


## 7. 自我线程内的调用骨架

当前目标骨架应固定为：

```cpp
void 自我线程类::执行主循环一轮_() {
    冻结本轮消息批次_();
    规范化消息并落一次特征账_();
    归并与对账_();
    生成受影响二次特征_();
    生成影响项_();
    更新根值需求树与任务权重_();
    根层重判_();
    执行派发与路由_();
    生成本轮结算与学习触发_();
}
```

但工程落地顺序应是：

1. `自我线程类` 先继续持有主循环
2. 主循环内部把“消息相关工作”逐步调用到四个模块
3. 线程内旧的分散 if/else 逻辑逐步缩减

必须固定：

`初始落地时应采用“线程调用消息模块”，而不是“线程被消息模块完全替代”。`
