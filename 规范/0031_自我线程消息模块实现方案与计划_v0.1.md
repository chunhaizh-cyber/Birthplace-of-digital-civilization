## 3. 当前代码现实

当前仓库里，消息面已经切到新的正式入口：

- `结构_治理消息`
- `治理mailbox_`
- `本轮冻结治理批次_`
- `投递治理消息(...)`
- `私有_归并冻结批次(...)`

当前这套入口的优点是：

1. 已有 `冻结批次` 语义。
2. 已有 `优先级` 与 `幂等键`。
3. 已有 `需求树变更` 这类控制面载荷桥位。
4. 主循环已具备“冻结一批再处理”的基本骨架。

当前这套入口的不足是：

1. 当前入口仍偏向“按路由大类/子类命名消息”，尚未升级成正式的 `什么存在做了什么动作，导致什么特征变了` 语义结构。
2. `首收件人 / 上浮 / 路由父任务` 还没有独立成模块。
3. 消息处理逻辑仍散在 `自我线程类.cpp` 中。
4. “会改谁、会不会触发重判/新任务/收束”还没有显式绑定到 `特征变化列表`。

因此当前最稳的路线不是继续补旧事件，而是：

`直接把自我线程公开入口和内部邮箱统一切换到“路由头 + 存在动作语义”的 结构_治理消息，再逐步把线程里的分发逻辑抽成四模块。`


## 4. 实现总原则

必须固定以下 `8` 条：

1. 先切协议，不保留旧事件入口。
2. 先协议和路由，再下沉任务实体。
3. 先让 `自我线程类` 调四个消息模块，再慢慢削薄线程内部旧逻辑。
4. 公开入口、冻结批次与活邮箱统一只使用 `结构_治理消息`。
5. 消息模块只解决 `怎么说话 / 谁先收 / 怎么分发 / 哪些特征变了 / 何时重判`，不提前扩业务算法。
6. 控制面消息逻辑留在线程；局部工序语义继续留在工作线程。
7. 本轮仍必须遵守 `冻结一批，清完这一批，再做一次决议`。
8. 第一阶段不强制所有消费者改成 `import`，允许 `.ixx + .h` 双入口并存。

## 6. 现有结构的复用与替换

### 6.1 直接复用

当前建议直接复用：

- `治理mailbox_`
- `本轮冻结治理批次_`
- `私有_消息排在前面(...)`
- `需求树变更` 载荷桥位

原因：

- 已在线程主循环中稳定存在
- 已支持优先级和冻结批次
- 只需替换元素类型，不必重写邮箱与冻结机制

### 6.2 新增桥接层

当前建议新增并统一使用：

- `结构_治理消息`
- `结构_治理消息头`
- `结构_治理消息优先级`
- `投递治理消息(...)`

统一规则：

1. 自我线程对外入口只接受 `结构_治理消息`
2. 冻结批次与活邮箱都直接持有 `结构_治理消息`
3. 后续四模块全部只吃 `结构_治理消息`

### 6.3 后续替换面

当协议稳定后，再逐步把：

- `投递治理外部反馈事件(...)`
- `投递治理需求树变更事件(...)`

继续下沉为基于 `结构_治理消息` 的更细分投递器。


## 7. 自我线程内的调用骨架

当前目标骨架应固定为：

```cpp
void 自我线程类::执行主循环一轮_() {
    冻结本轮消息批次_();
    处理抢占控制消息_();
    处理普通消息批次_();
    处理本轮同步闭包_();
    根层重判_();
    生成主派发决议_();
    生成本轮结算_();
    后置处理学习审计_();
}
```

但工程落地顺序应是：

1. `自我线程类` 先继续持有主循环
2. 主循环内部把“消息相关工作”逐步调用到四个模块
3. 线程内旧的分散 if/else 逻辑逐步缩减

必须固定：

`第一阶段是“线程调用消息模块”，不是“线程被消息模块完全替代”。`


## 8. 分阶段计划

### P0 结构冻结

目标：

- 固定 `0028 / 0029 / 0030` 对应的代码模块清单
- 固定现有代码锚点
- 固定桥接策略

本阶段只做：

- 新建 `0031` 计划文档
- 明确四模块文件名
- 明确新协议的唯一入口地位

完成标准：

1. 四模块职责无冲突
2. 协议切换策略已写死
3. 下一阶段能直接开工

### P1 消息协议模块落地

目标：

- 把 `枚举_消息大类 / 子类 / 首收件人 / 存在类型 / 动作类型 / 特征类型 / 消息头 / 特征变化项 / 存在动作语义 / 治理消息` 正式落码

本阶段只做：

- 新增 `自我线程消息协议模块.*`
- 定义 `结构_特征变化项`
- 定义 `结构_存在动作语义`
- 定义 `结构_治理消息` 的正式字段、默认优先级和下游适配函数
- 固定 `控制意图特征 / 运行状态特征` 两组控制域特征取值

完成标准：

1. 新协议结构可编译
2. 自我线程公开入口切换为 `投递治理消息(...)`
3. 冻结批次和活邮箱都直接持有新协议
4. 业务侧构造消息时，不再把 `启动 / 暂停 / 收束 / 停止` 当成裸消息原语，而是写成控制域特征变化

### P2 消息路由模块落地

目标：

- 把 `首收件人 / 上浮 / 回父任务 / 是否影响优先级 / 是否触发重判` 的规则独立落码

本阶段只做：

- 新增 `自我线程消息路由模块.*`
- 把 `0030` 的四张表先落成判定函数
- 新增 `从语义推导路由大类 / 子类` 的统一入口

完成标准：

1. 关键路由规则不再散在 `自我线程类.cpp`
2. `控制消息 / 全局环境消息 / 结果回流 / 派生需求` 的分流判断可单测
3. `发出者 / 动作主体 / 变化主体 / 记录者` 不同组合下的推导规则可单测

### P3 消息处理器模块落地

目标：

- 把七类路由消息的控制面处理从线程类里抽成七个入口

本阶段只做：

- 新增 `自我线程消息处理器模块.*`
- 处理器函数内部先只改：
  - 线程控制态
  - 工作线程状态表
  - 任务控制态摘要
  - 需求树更新输入
  - 结果缓存
  - 学习后置队列
  - 状态节点与动态节点所需的最小桥接输入

必须固定：

- 这阶段不展开任务实体三层
- 这阶段不展开方法内部逻辑

完成标准：

1. 线程类能调用七个处理入口
2. 处理器只做控制面，不吞局部工序
3. 处理器优先消费 `特征变化项`，不只靠 `子类` 猜语义

### P4 消息批次执行器落地

目标：

- 把“冻结一批，清完一批，再做一次决议”的轮次骨架收进独立模块

本阶段只做：

- 新增 `自我线程消息批次执行器模块.*`
- 管理：
  - 冻结批次
  - 抢占消息优先
  - 普通消息处理
  - 同步闭包处理
  - 一次重判
  - 一次主派发/收束决议

完成标准：

1. 主循环里的消息消费顺序不再散写
2. 批次边界清晰
3. 异步延后和控制抢占的语义已可验证

### P5 再进入任务实体三层

目标：

- 在消息模块稳定后，再进入：
  - `结构_任务根节点`
  - `结构_任务控制态`
  - `结构_任务局部运行态`

必须固定：

`任务实体应排在消息协议之后，不得倒序施工。`


## 9. 每阶段验收重点

### P1 验收

- 能定义并编译 `结构_治理消息`
- 自我线程公开入口与内部邮箱统一为 `结构_治理消息`

### P2 验收

- 能判定首收件人
- 能判定是否上浮线程
- 能判定是否路由父任务

### P3 验收

- 七类消息各有处理入口
- 处理器只改控制面状态，不直接推进局部步骤

### P4 验收

- 一轮先冻结批次
- 抢占控制消息可先行
- 普通消息与同步闭包处理完后才做一次决议
- 本轮异步新消息不追入当前轮次


## 10. 当前阶段最小实施顺序

1. `自我线程消息协议模块`
2. `自我线程消息路由模块`

4. `自我线程消息批次执行器模块`

不得倒序。

`先定义消息协议，再定义消息路由，再抽出消息处理器，再收口批次执行骨架，最后再进入任务实体三层。`

最终收口：

`0031 的意义，不是继续补消息概念，而是把 0028/0029/0030 三份规范翻成当前仓库可执行的模块拆分、桥接策略和阶段计划。`
