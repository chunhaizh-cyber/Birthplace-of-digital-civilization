# 任务筹办与执行边界规范 20260512

## 0. 权威修订

本规范受以下硬规则约束：

```text
规范/方法动作动态硬规则规范20260512.md
规范/自我线程与任务管理线程权责边界规范20260512.md
```

2026-05-13 修订：

```text
任务筹办和任务执行不再定义为本能方法。
它们是任务管理界面线程裁决、任务管理工作线程按工作包授权调用的任务生命周期普通函数。

任务筹办 / 任务执行不进入方法树，不进入候选方法集，不作为任务可选动作，也不生成方法动作动态。
```

任务筹办和任务执行只产出任务层过程回执，不直接写任务虚拟存在，也不直接生成任务状态方法动作动态。

凡是把任务回执同步进任务虚拟存在，或把任务状态推进成新状态，都必须通过任务域提交类本能方法完成。

## 1. 总原则

```text
筹办负责让任务可执行。
执行负责让任务按规则运行并获得结果。
```

最新口径：

```text
任务筹办：
    目标是把任务从初始状态推进到“就绪”。
    如果当前条件不足，返回缺口和派生提示，作为需求树继续生长入口。
    任务筹办保持单步循环执行：每轮只检查当前任务状态、来源需求、候选方法和方法状态，
    不在本函数内部递归执行学习链、子任务或执行链。

任务执行：
    在任务已就绪后按规则执行并获得结果。

任务管理线程：
    读取筹办 / 执行回执；
    调用同步 / 提交类本能方法；
    再把任务域变化落账。
```

它们必须遵守三通道边界：

```text
运行存在：
    写任务层回执，给任务管理层读取。

输出结果场景：
    写世界事实，只承载真实特征变化、状态节点、动态事实、结果状态。

方法虚拟存在：
    写本方法自身记忆，只记录运行账、最近运行、学习产物和条件结果对索引。
```

因此：

```text
任务筹办 / 任务执行普通函数不直接写任务虚拟存在。
任务筹办 / 任务执行普通函数不把任务状态写入输出结果场景。
任务筹办 / 任务执行普通函数不直接生成任务状态动作动态。
```

## 2. 筹办职责

筹办负责读取任务来源需求，并围绕需求目标建立可执行入口。

筹办的主结果不是“成功 / 失败”，而是：

```text
任务状态是否被推进到就绪。
```

如果暂时不能就绪，不是筹办失败，而是说明需求树需要继续细化。

筹办至少处理：

```text
1. 读取来源需求。
2. 提取需求目标：
   - 需求主体
   - 被需求存在
   - 目标特征类型
   - 当前状态
   - 目标状态 / 目标值 / 区间 / 方向
   - 满足关系

3. 根据需求目标查找候选方法：
   - 先按方法头结果能力粗查；
   - 查到的方法写入运行存在的候选回执；
   - 没查到时创建待学习方法头，并写入运行存在的候选回执。

4. 选择当前方法：
   - 候选只有一个时，直接作为当前方法；
   - 候选多个时，按匹配度、成熟度、可用性、稳定性等规则选择。

5. 判断当前方法是否已使任务具备可执行条件。
6. 若不可执行，返回结构化缺口。
7. 若可执行，返回当前方法和可执行输入条件场景。
```

任务虚拟存在中的：

```text
任务方法候选集
当前方法
可执行输入参数场景
任务状态
筹办缺口类型
```

由任务管理线程调用 `同步筹办回执到任务` 本能方法后同步，不由筹办本能函数直接写入。

## 3. 筹办发现的缺口

下列缺口属于“任务尚不可执行”，应由筹办返回：

```text
方法缺动作骨架
方法缺可执行入口
方法缺条件节点
方法缺结果节点
方法缺条件结果对
方法状态未达到可用
参数不可配
方法运行账不足
方法结果不稳定
```

这些缺口不代表父需求已满足，也不代表当前任务完成，更不代表筹办失败。它们只说明：

```text
当前任务已经找到候选方法，
但该方法还不能承接执行。
```

自我线程或任务管理线程应根据筹办缺口生成学习 / 补齐需求，让需求树继续生长。

标准链路：

```text
父需求
    -> 父任务
        -> 筹办发现缺口
            -> 派生更细需求
                -> 子任务
                    -> 子任务完成后支持父任务重筹办
```

筹办未直接到就绪：

```text
不是失败。
是需求树继续生长。
```

## 4. 执行职责

执行只在任务已具备可执行办法后运行。

执行至少处理：

```text
1. 读取筹办绑定好的当前方法。
2. 读取筹办准备好的输入条件场景。
3. 按方法条件节点绑定参数。
4. 调用本能函数 / 动作入口 / 方法体。
5. 生成执行结果场景。
6. 按方法结果节点校验结果。
7. 判断是否满足来源需求。
8. 写入任务层执行回执到运行存在。
9. 写入本任务执行方法自己的方法运行账。
10. 将真实状态变化留在输出结果场景，交由世界树提交层处理。
11. 返回运行存在。
```

执行阶段可以做防御性复核，但不应把方法结构补齐作为主要职责。

## 5. 执行必须遵守的规则

执行遇到的规则包括三层。

### 5.1 方法规则

```text
条件节点
参数绑定
动作入口
结果节点
条件结果对
```

### 5.2 任务规则

```text
当前任务是否仍有效
来源需求是否仍未满足
是否被活动子需求阻塞
任务是否超时
任务是否取消
任务是否挂起
```

### 5.3 自我限制性规则

```text
安全值下限
风险安全值
服务值消耗限制
待机保护
是否允许真实写回
是否允许外部动作
是否允许修改核心记忆
是否需要先进入低风险观察
```

因此：

```text
可执行方法 + 当前任务有效 + 自我限制允许
= 执行可以运行
```

若执行被自我限制性规则阻断，应返回：

```text
执行被自我规则阻断
需要安全评估
需要待机保护
需要更低风险方法
```

这类结果属于执行期治理结果，不应误写成方法结构缺口。

## 6. 方法结构职责

```text
方法头节点：
    大范围索引、候选召回、方法身份入口。

条件节点：
    参数槽、参数匹配、前置条件判断。

结果节点：
    输出结构、目标结果、结果校验。

条件结果对：
    条件节点和结果节点之间的可执行知识单元。

方法虚拟存在：
    方法语义记忆、统计、学习产物索引。
```

方法头不是具体执行接口。方法执行时，参数设置、参数匹配、前置条件判断，必须以条件节点为依据。方法结果判断，必须以结果节点为依据。

## 7. 阶段返回口径

### 7.1 筹办返回

筹办返回值是本轮运行存在。运行存在中的核心语义只有两类：

```text
1. 任务已经可执行：
   - 已找到并绑定当前方法；
   - 已准备输入条件场景；
   - 可进入执行阶段。

2. 任务尚不可执行：
   - 已找到候选方法；
   - 已明确缺口；
   - 需要学习 / 补齐后再重筹办。
```

筹办运行存在应至少能表达：

```text
任务层执行回执
建议任务状态
目标是否达成
筹办缺口类型
等待原因
派生需求提示
目标方法头节点
候选方法数量
可执行输入参数场景
方法运行结果
```

### 7.2 执行返回

执行返回值是本轮运行存在。运行存在中的核心语义包括：

```text
1. 执行成功，结果满足需求。
2. 执行成功，但结果不满足需求。
3. 执行失败。
4. 执行被任务规则阻断。
5. 执行被自我限制性规则阻断。
6. 执行产生真实状态变化，需要回写。
7. 执行产生派生需求。
```

执行运行存在应至少能表达：

```text
任务层执行回执
建议任务状态
目标是否达成
执行缺口类型
当前方法运行存在
当前方法执行情况
输出结果场景
需求满足证据
等待原因
派生需求提示
失败原因
```

执行阶段可以有“目标是否达成”等结果特征，但动作动态本体仍然记录事实，不以成功 / 失败为根概念。

任务完成判定必须使用：

```text
当前方法执行成功
+
来源需求满足
```

不能简化成：

```text
当前方法执行成功 -> 任务完成
```

## 8. 任务管理层同步口径

任务管理层负责读取筹办 / 执行返回的运行存在，并编排任务账提交。

筹办后：

```text
任务管理层调用 本能方法_同步筹办回执到任务：
    运行存在.建议任务状态 -> 任务虚拟存在.任务状态
    运行存在.目标方法头节点 -> 任务虚拟存在.当前方法
    运行存在.候选方法 -> 任务虚拟存在.任务方法候选集
    运行存在.可执行输入参数场景 -> 任务虚拟存在.可执行输入参数场景
    运行存在.筹办缺口类型 -> 任务虚拟存在.筹办缺口类型

如有任务状态变化，再调用 本能方法_提交任务状态变化。
```

如果筹办回执表示缺口，任务管理层不应把它当成失败。应根据回执调用对应方法：

```text
派生筹办缺口需求
挂起任务等待子需求
唤醒任务重筹办
提交任务状态变化
```

每个中间状态变化都应有自己的方法来源和动作动态。

执行后：

```text
任务管理层调用 本能方法_同步执行回执到任务：
    运行存在.建议任务状态 -> 任务虚拟存在.任务状态
    运行存在.当前方法运行存在 -> 任务虚拟存在.当前方法运行存在
    运行存在.执行缺口类型 -> 任务虚拟存在.执行缺口类型
    运行存在.目标是否达成 -> 任务虚拟存在.目标是否达成
    运行存在.失败原因 / 等待原因 / 派生需求提示 -> 任务虚拟存在对应特征

如有任务状态变化，再调用 本能方法_提交任务状态变化。
```

任务状态变化动作动态不得由任务管理层裸生成。任务管理层必须调用“提交任务状态变化”本能方法，由该方法作为动作来源生成任务状态动作动态。

## 9. 禁止项

筹办阶段禁止：

```text
1. 改变目标特征真实值。
2. 直接结算来源需求。
3. 将方法结构缺口伪装成执行结果。
4. 在没有候选方法时直接判定父需求失败。
5. 用筹办自己的方向替代来源需求目标。
6. 直接写任务虚拟存在的任务状态、当前方法或候选集。
7. 把任务状态写入输出结果场景。
```

执行阶段禁止：

```text
1. 自行查找新方法作为主要逻辑。
2. 绕过筹办绑定的当前方法。
3. 绕过方法条件节点直接拼参数。
4. 绕过自我限制性规则真实写回。
5. 把自我限制阻断误记为方法结构缺失。
6. 直接写任务虚拟存在的任务状态。
7. 把任务状态、执行缺口、等待原因写入输出结果场景。
```

任务管理层禁止：

```text
1. 绕过提交类本能方法生成任务状态动作动态。
2. 把任务状态动作动态的动作来源记为任务管理线程。
3. 把业务方法动作动态直接当成任务完成动作动态。
4. 把筹办缺口直接解释成失败。
5. 用成功 / 失败替代派生、等待、细化、重筹办等事实。
```

## 10. 废止旧口径

废止：

```text
任务筹办 / 任务执行注册为本能方法。
任务筹办 / 任务执行进入方法树或候选方法集。
任务筹办 / 任务执行被任务筹办过程选中为目标方法。
任务筹办 / 任务执行生成方法动作动态。
任务执行通过旧步骤节点壳直接提交任务状态。
工作线程直接修改任务虚拟存在任务状态。
任务筹办 / 执行把任务状态写进输出结果场景。
任务管理层绕过提交类本能方法直接生成任务状态动作动态。
任务筹办未就绪就直接记为失败。
```

保留：

```text
旧 自我_任务执行一步 废止。
旧 自我_任务执行 / 自我_任务筹办 本能方法入口废止，只允许作为历史残留 ID 被识别和排除。
任务筹办 是任务管理生命周期普通函数。
任务执行 是任务管理生命周期普通函数。
同步筹办回执到任务、同步执行回执到任务、提交任务状态变化仍是任务域提交本能方法。
自我线程主循环、任务管理界面线程承接、任务管理工作线程调度仍不是本能方法。
```

## 11. 最终定稿

```text
筹办问：
    这个任务怎样才能执行？
    如果不能立刻就绪，需要长出哪些更细需求？

执行问：
    在方法规则、任务规则和自我限制性规则下，
    现在能不能运行？
    运行后结果是什么？
```

一句话：

```text
筹办让任务可执行；
执行让任务受规则约束地发生结果。
两者都只返回任务层回执，真正编排任务账提交的是任务管理层；
凡生成方法动作动态的任务账提交，动作来源必须是提交类本能方法。
筹办未到就绪不是失败，而是需求树继续生长的入口。
```
