# 1000 任务执行类实现规范（当前顺序执行骨架）

本文用于固定当前 `任务执行类.ixx` 的实现边界，说明：
- 当前 `任务执行器` 已经实现了哪些能力
- 当前任务执行链如何组织
- 当前哪些策略已经落地
- 当前仍然保留哪些后续扩展空间

本文中的 `筹办` 部分，应以上位规范 [0970_任务筹办规则.md](D:\鱼巢\规范\0970_任务筹办规则.md) 为准。

## 1. 任务执行类的定位

当前 `任务执行器` 不是单纯的“步骤执行器”，而是四个角色的组合：

- `筹办器`：按任务需求召回方法，并生成候选步骤
- `条件展开器`：当方法条件不满足时，把条件缺口转成子需求并派生子任务
- `顺序执行器`：推动当前步骤执行，生成实际结果，并推进下一步
- `局部调度器`：在父任务、步骤、并发子任务之间按优先级做运行期选择

因此，当前 `任务执行器` 已经是：

`需求驱动筹办器 + 子任务展开器 + 顺序执行器 + 局部调度器`

但必须补充固定：

- `任务执行器` 不负责根层方向裁决。
- `任务执行器` 不替代任务管理总控，也不替代四域总判。
- `任务执行器` 的直接上游，应是 `根层方向已放行 + 任务管理上下文已具备 + 当前已判定进入某个工程承接路径`。

但它还不是：

- 完整全局任务调度中心
- 完整方法学习器
- 完整多目标决策器

### 1.1 最新统一主链中的位置

按当前最新体系，`任务执行器` 应处在下面这条链里：

```text
根层方向裁决
-> 形成任务管理上下文
-> 判定当前进入筹办 / 维护 / 纠偏 / 收束中的哪一路
-> 任务执行器负责其中需要落代码的局部筹办、局部展开、步骤执行与结果生成
-> 返回结果特征与执行反馈
-> 外层再重判根方向和当前功能域
```

因此必须固定：

1. `任务执行器` 处理的是工程落地，不是根层最终正确性。
2. `任务执行器` 输出的是步骤结果、任务结果、反馈和上下文更新，不是根语义真理。
3. `任务执行器` 每推进一步后，后续仍应回到外层做 `根层重判 -> 判域 -> 再决定是否继续调用执行器`。

## 2. 当前遵守的任务结构

当前执行器遵守的任务结构为：

`任务头结点 -> 步骤节点 -> 结果节点`

并且结合了并发子任务：

- `任务头结点.子链` 挂直属步骤节点
- `步骤节点.下节点` 表示同层并列步骤
- `步骤节点.子链` 挂该步骤下的结果节点
- `步骤节点.并发子任务头结点列表` 保存该步骤派生出的并发子任务
- `结果节点.下节点` 表示同一步下的不同预测结果
- `结果节点.子链` 表示命中该预测结果后的下一步骤

因此主执行链是：

`任务头 -> 当前步骤 -> 命中预测结果 -> 下一步骤`

如果该步骤存在补条件子任务，则顺序会先变成：

`任务头 -> 当前步骤 -> 并发子任务 -> 当前步骤继续执行 -> 结果 -> 下一步骤`

## 3. 当前已经实现的筹办规则

当前筹办应统一理解为：

- `先找方法`
- `再找条件`

其它诸如候选排序、步骤生成、补条件子任务等，都是这两步的结构化产物。

必须补充固定：

- 这里的筹办，只发生在 `根层已允许继续推进`、且 `任务管理上下文` 已要求进入筹办路径之后。
- 当前执行器中的筹办逻辑，属于四域承接层的落地实现，不得反向越权改写根层方向。

### 3.1 需求到方法的初选

当前自动筹办从任务头读取：

- `需求`
- `需求.被需求状态`
- `任务头全局条件场景`

这里必须先固定：

- `需求` 在可执行比较层上，不是抽象句子，而应落成 `一个或多个带值特征`。
- 每个需求特征项至少应能提供：`特征类型 / 需求值 / 当前值`。

然后遍历方法树中的方法首节点与方法结果节点。

一个方法能进入初选，当前要求是：

- 方法结果或作用路径能够把需求目标特征朝目标状态推进，或能补上达成该目标所需的关键中间状态

也就是：

`结果作用与目标同向，且能正向推进完成判据成立 -> 可以进入初选`

### 3.2 方向性判断

如果目标值和方法预测值是数值型，当前实现会带入：

- 当前场景值
- 需求目标值
- 方法预测结果值

并做方向性判断：

- 预测值不能比当前值离目标更远
- 预测方向不能与目标方向相反

这条规则用于避免：

`南辕北辙`

但当前正式口径应继续收紧为：

1. 方法自身不直接定义“任务成功”。
2. 方法只需要返回结果；返回结果必须是一组 `带值特征`。
3. 任务执行函数再基于 `需求特征` 与 `结果特征` 做逐特征方向判断。

也就是说，最小比较链应为：

- `需求方向 = sign(需求值 - 当前值)`
- `结果方向 = sign(返回值 - 当前值)`

然后只判断：

- 当 `需求方向 = 等于` 时，`结果方向` 只能为 `等于`
- 当 `需求方向 ≠ 等于` 时，`结果方向` 必须等于 `需求方向`

可以压成一句话：

`目标已完成时，只允许保持不变；目标未完成时，只允许向需求方向推进。`

更细的 `实际结果方向` 单特征对齐判定表，按 [0321_实际结果方向与需求方向对齐判定表规范.md](D:\鱼巢\规范\0321_实际结果方向与需求方向对齐判定表规范.md) 执行。

### 3.2.1 返回结果的最小要求

任务执行器后续不应主要依赖“方法自报成功”来判断推进成立，而应主要依赖方法返回的结果特征。

因此必须固定：

- 方法返回值必须是一组 `带值特征`。
- 不允许把纯诊断字符串、错误码、局部句柄直接冒充任务推进结果。
- 诊断信息可以存在，但任务推进判断优先看 `结果特征`。

### 3.2.2 多特征比较顺序

当一个需求由多个特征组成时，任务执行函数必须：

1. 先按 `安全特征` 与 `服务特征` 分组。
2. 再在组内逐特征比较 `需求方向` 与 `结果方向`。
3. 最后才把组内结果提交给更高层做综合裁决。

必须固定：

- 不得把安全特征和服务特征直接混在一起做单层黑箱总比较。
- 不得在跳过逐特征比较的前提下，直接给出“总体上成功”。

### 3.3 条件匹配与条件缺口分析

当前不会只判断“方法现在能不能做”，还会进一步分析：

- 当前二次特征是否命中方法条件索引
- 方法条件场景模板里有哪些目标状态尚未满足

这里的条件判断，应与上位规范中的 `自我可用 = 可获取 + 有权限使用` 保持一致。

因此当前会得到三类结论：

- `无条件，可直接执行`
- `条件满足，可直接执行`
- `条件不满足，但能明确列出缺哪些状态`

### 3.4 规避过滤

当前执行器提供：

- `设置方法规避过滤`

用于注入外部规避规则，排除：

- 法律禁止
- 人文伦理禁止
- 系统规则禁止
- 当前运行上下文明确不允许的方法

### 3.5 并列步骤生成

通过结果匹配、方向判断、条件分析、规避过滤之后，执行器会：

- 把候选推进路径按前置满足度、方向一致性、风险门控和优先级排序
- 对条件链、结果链或收口去向明显不同的候选，分别生成并列步骤节点
- 每个步骤只写自己的 `当前选中方法首节点`
- 只有同一目标、同一条件面下允许局部换法的方案，才保留为该步骤的 `备用方法首节点列表`
- 默认把当前最适合推进的步骤置为当前步骤

因此当前筹办产物是：

`多个并列候选步骤 + 一个当前首选步骤`

## 4. 当前已经实现的预测结果生成

对于每个步骤节点，当前执行器会：

- 在步骤建立后立即创建至少一个结果节点，作为本步骤的初始化结果锚点
- 枚举该方法树下的结果节点
- 为每个方法结果生成一个预测结果节点
- 如果方法没有显式结果节点，则生成占位预测结果节点

当前预测结果节点会记录：

- 结果角色（默认 `预测结果` 或占位结果）
- 所属任务头结点
- 来源步骤节点
- 来源方法首节点
- 对应方法结果节点
- 结果方向二次特征
- 结果场景
- 命中后动作

必须补充固定：

- 步骤节点一旦成立，不允许长期没有结果子节点。
- 若该步骤暂时还拿不到完整方法结果，也必须先生成一个占位结果节点，并把结果字段初始化到可回填状态。
- 初始化结果节点时，必须先根据 `步骤目标状态` 与 `当前状态` 生成 `结果方向二次特征`；这个方向属于结果特征状态的二次特征，不是自由文本。
- 后续若方法结果给出了更明确的预测结果状态，或执行后拿到了实际结果状态，应继续按该结果状态对同一比较基准刷新 `结果方向二次特征`。

因此，一个步骤下可以存在多个预测结果节点，用于后续命中和转步。

## 5. 当前已经实现的补条件子任务机制

这是本轮实现中最关键的新能力。

当步骤的当前方法不能直接执行，但能明确分析出未满足状态时，执行器会：

1. 把每个缺口状态转换成一个子需求
2. 为每个子需求创建一个子任务头结点
3. 把子任务绑定到当前步骤的并发子任务列表
4. 将当前步骤置为 `等待中`
5. 先推进这些子任务，待其完成后再恢复当前步骤执行

因此现在已经实现：

`方法条件缺口 -> 子需求 -> 子任务`

这意味着当前版本已经不只是“当前条件下直接可执行层”，而是已经进入了：

`一层前置条件展开`

## 6. 当前已经实现的执行推进主链

### 6.1 外层统一主链

当前 `任务执行类` 必须服从以下外层主链：

```text
根层方向裁决
-> 形成任务管理上下文
-> 判定当前功能域
-> 若当前路径需要进入任务执行器，则调用推进一步
-> 推进一步后返回结果特征与反馈
-> 外层重判根方向与当前功能域
```

也就是说：

- `推进一步` 不是整条链的起点
- `推进一步` 也不是最终裁决终点
- 它只是中间的工程执行节点

### 6.2 当前 `推进一步` 的内部主链

当前 `推进一步` 的主链如下：

1. 读取任务头状态，跳过终结任务
2. 选择当前步骤
3. 如果没有步骤，或步骤仍为空白待筹办步骤，则自动筹办，并立即创建该步骤的初始化结果节点
4. 把任务头置为 `运行中`
5. 若当前步骤存在补条件子任务，则优先推进子任务
6. 子任务全部完成后，补全或刷新步骤结果字段
7. 调用叶子执行回调执行当前步骤
8. 在步骤下补全当前结果节点，或追加一个实际结果节点
9. 将实际结果命中到预测结果节点
10. 若预测结果下挂有下一步骤，则切到下一步骤
11. 否则检查头结点完成判据是否已成立；成立则收口，否则按成功/失败/重试/挂起/学习继续分流

## 7. 当前已经实现的预测命中

当前预测命中优先走：

- `实际结果场景 -> 二次特征`
- `预测结果节点 -> 方法结果摘要索引`
- `结果摘要_匹配目标`

命中原则是：

`二次特征方向相同,区间包含或相同`

如果没有摘要可用，才退回到：

- 同来源方法首节点优先
- 再退到步骤下首个预测结果

因此当前已经具备：

`预测结果 -> 实际结果 -> 命中预测`

的完整闭环。

## 8. 当前已经实现的子任务推进规则

当步骤下存在并发子任务时，当前执行器会：

- 汇总子任务总数、完成数、失败数
- 找出当前最值得推进的子任务
- 将父步骤与父任务的“当前子任务”指针同步到该子任务
- 子任务未完成时，父步骤保持等待
- 任一子任务失败时，父步骤与父任务直接失败收口
- 全部子任务完成后，父步骤恢复为 `就绪`

因此当前父子任务关系是：

`父任务负责调度，子任务优先执行，父步骤等待子任务完成`

## 9. 当前已经实现的失败处置

当前执行器已经支持以下失败处置：

- `回跳重试`
- `切换方法重试`
- `转入尝试学习`
- `挂起等待`
- `任务失败`

其中：

- 步骤允许重试时，会累加 `已重试次数`
- 若同一步骤存在 `备用方法首节点列表`，允许在其内部切换到下一个方法；若替代方案带来不同条件链或结果链，则应切换到并列步骤而不是只切方法
- 配置了尝试学习任务时，可把失败转入学习兜底任务

## 10. 当前已经实现的优先级规则

当前调度优先级不再只是节点上静态保存的一个数，而是运行时动态计算。

当前规则为：

- `基准优先级` 统一追溯主任务头结点
- `局部优先级偏移` 保留步骤候选评分和局部调度修正
- 子任务头结点相对父任务头结点有明显加成
- 靠前步骤关联的任务、步骤、结果有顺序加成
- 近似单步、叶子、无并发子任务的最小任务有额外加成
- 父任务存在可推进直属子任务时会让位给子任务
- 等待中、挂起、终结状态不会进入正常调度队列

因此当前调度语义可以概括为：

`主任务给基础，子任务抢执行，靠前步骤先跑，最小叶子优先`

## 11. 当前公开接口的职责

当前 `任务执行器` 的主要公开接口包括：

- `设置叶子执行回调`
- `设置尝试学习任务`
- `设置方法规避过滤`
- `推进一步`
- `调度并推进一步`

## 12. 当前还没有实现的内容

虽然当前已经有了“一层条件缺口 -> 子任务”的展开能力，但以下能力还没有完全落地：

- 多层前置条件递归规划的收敛策略
- 时间、价格、风险等多维综合择优
- 预测外结果自动沉淀为方法补全任务
- 法律、人文、规则的内建策略层
- 更复杂的等待外部事件、循环回跳、阶段恢复机制

换句话说，当前版本已经实现的是：

`需求驱动筹办 + 一层条件展开 + 子任务优先推进 + 预测/实际闭环`

还没有完全实现的是：

`多层全局规划 + 完整学习补全 + 复杂策略控制`

## 13. 当前版本的一句话结论

当前 `任务执行类` 已经能够：

`按需求召回候选推进路径 -> 生成并列步骤 -> 依据条件选择当前步骤 -> 分析方法条件缺口 -> 把缺口转成子任务 -> 子任务优先推进 -> 执行步骤 -> 生成实际结果 -> 用二次特征命中预测结果 -> 按头结点完成判据推进下一步或收口任务`


## 14. V1 重构约束

后续按消息驱动重构时，应再固定以下规则：

### 14.1 执行器只做单步执行

执行器不应再把“一个任务连续推到多步完成”当作默认行为。

应固定为：

- 每次只执行一个当前步骤
- 当前步骤结束后，必须返回一条明确消息
- 在收到该步返回消息之前，不得自动进入下一步

### 14.2 执行器只管理执行现场状态

后续应把执行器职责收敛为：

- `筹办中`
- `执行中`
- `等待回复`
- `等待子任务回复`

也就是说：

- 执行器负责“当前这一步正在发生什么”
- 执行器不负责最终裁决需求是否满足
- 执行器不负责决定任务是否完成或进入下一轮重筹办
- 执行器不负责替代 `根层方向裁决 -> 任务管理上下文 -> 判域` 这条外层链

### 14.3 每一步都必须返回消息

V1 可先固定以下返回消息类型：

1. `步骤完成`
2. `步骤失败`
3. `步骤无法执行`
4. `生成子任务`
5. `子任务完成`
6. `子任务失败`
7. `子任务无法执行`

这些消息统一交由 `任务调度器` 消费。

### 14.4 所有任务共用同一阶段闭环

无论是真根任务、伪根任务还是普通子任务，都应遵守：

- `局部阶段结束 -> 检查对应需求 -> 已满足则完成 -> 未全满足则重新筹办 -> 生成下一阶段任务`

区别只在于：

- 真根任务对应的是持续逼近型根需求，因此结构上不会真正进入终局完成
- 普通任务与伪根任务在其绑定需求满足后可以进入完成状态

### 14.5 子任务清零后的行为

必须固定：

- `子任务清零 = 当前已没有可执行的子任务`
- 清零时子任务状态只允许落在 `完成 / 失败 / 无法执行`
- 若子任务清零后父任务需求仍未满足，则应回到 `重新筹办`
- 父任务不直接“执行父需求”，而是重新组织下一阶段任务
