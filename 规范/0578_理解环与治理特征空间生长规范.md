# 0578 理解环与治理特征空间生长规范

## 1. 目标

本文用于固定一条更底层的统一口径：

`特征先于需求；理解环先于需求环。`

本文重点回答：

- 特征到底是什么
- 新特征从哪里来
- 为什么 `需求 -> 缺口 -> 补字段` 不是最底层模型
- `0阶 / 1阶 / 2阶 / 3阶 / n阶` 特征应如何理解
- 特征空间生长、需求形成与任务行动之间的正式顺序是什么

说明：

- 本文中的代码块或流程块都只表示语义，不表示现有实现中的直接接口或函数签名。

配套规范：

- [0579_特征使用账本与生命周期规范.md](D:\鱼巢\规范\0579_特征使用账本与生命周期规范.md) 负责固定“特征是否保留”与“如何按使用账本推进生命周期”。


## 2. 核心结论

必须固定以下结论：

1. `特征` 不是字段，也不是日志标签；特征是 `系统发现的一种稳定可复用区分能力`。
2. `需求` 不是特征的唯一来源；需求只是特征空间中变得可行动的局部凸起。
3. 更准确的主链不是 `需求 -> 缺口 -> 新特征`，而是：

```text
经验/现象
-> 区分
-> 特征
-> 高阶特征
-> 可见缺口/可见可能
-> 需求
-> 行动
```

4. `理解环` 先于 `需求环`；需求环会反过来刺激理解环继续生长。
5. `从缺口反推特征` 只允许作为筹办层的局部表达；它不是系统总模型。
6. 任何“新特征类型”的正式生成，都应理解为 `特征空间生长`，不是“需求驱动的补字段”。
7. 特征是否长期保留，不看抽象价值评分，优先看它是否形成过 `有效使用边`，以及是否仍处在 `试用暴露期`。

一句话收口：

`特征不是需求的附属物，需求反而是特征空间里显现出来的可行动凸起。`


## 3. 理解环与需求环

### 3.1 理解环

理解环固定为：

```text
Raw Experience
-> Phenomenon Discovery
-> Feature Discovery
-> Higher-Order Feature Composition
-> Visible Gaps / Visible Possibilities
```

它处理的不是“下一步做什么”，而是：

- 发生了什么
- 哪些差异反复出现
- 哪些关系值得稳定记录
- 哪些模式正在成形
- 哪些新的区分维度已经值得被命名

### 3.2 需求环

需求环固定为：

```text
Visible Gaps / Visible Possibilities
-> Demand Formation
-> Task / Method / Learning Action
-> New Experience
```

它处理的是：

- 当前哪些可见差异已经值得转成行动
- 应由任务、方法还是学习承接
- 哪些执行反馈会回流到理解环

### 3.3 两环关系

必须固定：

1. 理解环先于需求环。
2. 需求环不得冒充理解环。
3. 需求环可以触发新特征候选，但不能宣称自己就是特征发明的唯一来源。
4. 任务筹办中的 `少什么就提什么需求`，只属于需求环内部的工作规则，不得扩写成整套系统的总起点。


## 4. 特征的正式定义

### 4.1 特征

`特征` 指的是：

- 系统发现的一种稳定可复用区分方式

它可以区分：

- 为什么这个任务卡住
- 为什么这个方法可复用
- 为什么这个步骤优先
- 为什么这个学习任务必要
- 为什么这个现象以前不可见
- 为什么现在需要形成新需求

### 4.2 特征类型

`特征类型` 指的是：

- 被正式命名并可重复写入、读取、组合、比较的一类区分能力

例如：

- `task_block_reason`
- `acceptance_clarity`
- `method_availability`
- `dependency_readiness`
- `learning_gap_pressure`
- `risk_blocking_level`
- `goal_clarity`

### 4.3 特征空间

`特征空间` 指的是：

- 当前系统已经能稳定区分、命名、组合与消费的全部特征类型及其关系集合

必须补充固定：

1. 某个特征被读取、被计算或被展示，不自动等于它已经进入活特征空间。
2. 只有形成过有效治理边，或仍处在试用暴露期的特征，才属于当前应维护的活特征空间。

### 4.4 需求

`需求` 指的是：

- 在当前特征空间中已经变得足够可见、足够可行动的一类差异或可能性

因此必须固定：

1. 需求是特征空间的消费结果，不是特征空间的上位来源。
2. 需求可以刺激新特征发现，但不垄断新特征发现。


## 5. 阶次定义

特征阶次统一按以下口径理解：

```text
0阶 = 原始经验信号
1阶 = 直接特征
2阶 = 二次特征
3阶 = 三次特征
n阶 = 更高阶治理特征
```

### 5.1 0阶：原始经验信号

0阶不是正式治理特征，只是材料。

至少包括：

- 用户消息
- 任务描述
- 任务状态变化
- 代码 diff
- commit 信息
- 方法调用结果
- 失败日志
- 学习结果
- 计划变更历史
- 执行反馈

### 5.2 1阶：直接特征

1阶特征直接从原始经验中提取。

例如：

- `task_has_acceptance_criteria`
- `task_mentions_missing_method`
- `task_has_external_dependency`
- `task_has_learning_signal`
- `method_exists_for_task_type`

### 5.3 2阶：二次特征

2阶特征由一个或多个低阶特征组合出来。

它开始承担 `治理解释` 的职责。

例如：

- `task_learning_gap_pressure`
- `task_dependency_readiness`
- `method_reuse_confidence`
- `goal_visibility`

必须固定：

1. `比较后相减` 生成的方向差值型特征，是正式二次特征的一类重要子族。
2. 但二次特征不只限于“两个值相减”；聚合、关系、反复共现、权重组合、稳定模式压缩，都是合法来源。

### 5.4 3阶及以上

3阶及以上特征由更低阶特征继续组合，开始承担：

- 策略判断
- 成长价值判断
- 系统治理态判断

例如：

- `task_self_growth_priority`
- `method_future_reuse_value`
- `system_governance_stagnation_index`

必须固定：

1. `3阶及以上` 不再被限定为“同需求轴同向候选排序”这一种形式。
2. 排序只是高阶特征的一种合法子类，不是高阶特征的全部定义。


## 6. 特征空间生长函数

后续总控函数不应再命名为：

- `deriveFeatureTypesFromGovernanceGaps()`

它应升级为类似：

- `growGovernanceFeatureSpace()`
- `discoverAndGrowFeatureSpace()`

其职责固定为：

1. 从当前经验中发现新的可区分维度。
2. 从已有特征中组合更高阶特征。
3. 从高阶特征中显现新的可见缺口与可见可能。
4. 再把这些结果转成需求、任务、方法或学习行动。


## 7. 最小输入与处理链

### 7.1 输入

至少允许读取：

- 当前需求，可选
- 当前任务图
- 方法图
- 学习任务图
- 宿主图
- 已有特征类型
- 执行历史
- 失败历史
- 计划变更历史
- 产物与反馈

### 7.2 处理

最小处理链固定为：

1. 收集经验材料
2. 发现未解释现象
3. 发现新的区分维度
4. 生成一阶候选特征
5. 组合出二阶、三阶、n阶候选特征
6. 把候选特征暴露给决策、组合、需求显现、风险与验证链路进入试用
7. 记录特征使用事件与暴露事件
8. 按使用账本更新候选、活跃、冷却、归档或修复状态
9. 把仍应保留的特征类型写入任务/方法/学习任务虚拟存在
10. 用更新后的特征图反过来发现新需求并合成下一步步骤


## 8. 与任务筹办的边界

必须固定：

1. `筹办` 负责消费已经可见的缺口与需求。
2. `特征空间生长` 负责让原本不可见的差异变得可见。
3. `特征使用账本` 负责决定哪些特征进入长期保留，哪些只停留在试用、冷却、归档或修复。
4. 因此：
   - `理解驱动的特征空间生长` 是上游
   - `按使用事实维护特征生命周期` 是中游维护环
   - `少什么就提什么需求` 是下游


## 9. 废止口径

以下旧口径自本文生效后降级：

1. `需求 -> 缺口 -> 新特征` 是总模型
2. 新特征只是“缺什么就补什么字段”
3. 高阶特征只能理解成比较结果或候选排序
4. 特征是否保留主要由抽象价值函数裁定

它们若继续保留，只能作为局部子链、实现分支或历史兼容说明，不得再作为总定义。
