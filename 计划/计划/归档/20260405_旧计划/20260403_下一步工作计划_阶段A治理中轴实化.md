# 20260403 下一步工作计划：阶段A 治理中轴实化

## 1. 核对结论

根据 [20260403_自我最小循环实现大纲](D:/鱼巢/计划/20260403_自我最小循环实现大纲.md) 核对，当前“下一步先做 `阶段 A：治理中轴实化`”这个方向是正确的，不需要改方向。

原因是：

- 总纲把 `治理承接` 列为自我最小循环的必需层之一
- 总纲的最小顺序中，`装配任务管理上下文 -> 做一步治理 -> 进入执行链/等待链/学习链` 是核心闭环中段
- 当前代码已经有根需求、双值核、主循环、执行挂接、学习兜底和结果结算基础
- 当前唯一最明显的结构性缺口，就是 `治理层还没有真正从 TickOnce_ 中抽出来`

所以，下一步工作计划应继续固定为：

`先把治理中轴实化，再继续学习子任务正式化和结果治理正规化。`

## 2. 与总纲对齐后的调整点

虽然方向正确，但按总纲核对后，下一步工作计划需要补 4 个明确约束：

### 2.1 必须显式覆盖最小治理上下文的全部核心字段

下一步不能只带：

- `最近根层重判结果`
- `当前主需求`
- `当前管理对象`
- `当前缺口类型`
- `当前下一步候选`

还必须显式覆盖：

- `当前主任务`
- `当前方法候选或当前方法挂点`
- `最近一步治理结果`

否则和总纲第 5.3 节还没完全对齐。

### 2.2 必须把“等待链”纳入治理层正式输出

按总纲第 6、9 节，最小合法分流不只有：

- 执行
- 学习
- 待机
- 停止

还必须把 `进入等待` 作为正式去向保留下来。

下一步治理入口输出里，必须明确允许：

- `继续治理`
- `进入执行`
- `转入学习`
- `进入等待`
- `进入待机`
- `进入停止`

### 2.3 必须把 TickOnce_ 收成“装配 -> 调治理 -> 消费结果”

当前下一步计划里已经有这个方向，但需要写得更硬：

- `TickOnce_()` 不能再直接散落多段治理判断
- 治理层必须成为单独的正式承接入口
- 执行链和学习链都只能消费治理结果，不能绕过治理层自判

### 2.4 当前阶段仍不得碰外部输入解封

按总纲第 10 节，下一步工作计划必须继续保留：

- 前端输入只记录日志
- 外部输入不进入自我闭环
- 验证以日志为主

所以阶段 A 里禁止夹带：

- 文本命令体系
- 外部输入转需求
- 人机交互解封

## 3. 核对后的下一步目标

下一步的正式目标应表述为：

`把自我最小循环中的“治理承接层”从主循环内联分支，收成可独立输入、独立输出、可日志验证的一步治理中轴。`

## 4. 下一步交付物

### 4.1 结构交付

新增或收口以下承载面：

- `一步治理输入结构`
- `一步治理输出结构`
- `治理证据摘要`
- `正式去向枚举`

### 4.2 逻辑交付

至少完成以下逻辑：

1. `TickOnce_()` 装配治理输入
2. 调用 `一步治理入口`
3. 返回结构化治理结果
4. 由治理结果统一决定：
   - 是否继续治理
   - 是否进入执行
   - 是否转入学习
   - 是否进入等待
   - 是否进入待机
   - 是否进入停止

### 4.3 日志交付

至少补齐以下日志：

- 本轮治理输入摘要
- 当前治理域
- 当前缺口类型
- 当前去向候选
- 一步治理输出摘要

## 5. 下一步不做什么

这一轮明确不做：

- 不做完整学习子任务正式化
- 不做元层学习写回接入
- 不做治理实例堆与恢复快照
- 不做外部输入解封
- 不做完整多特征正规裁决的终版

这些都属于后续阶段，不应挤进阶段 A。

## 6. 建议实施顺序

下一步代码工作建议严格按下面顺序推进：

1. 先定义 `一步治理输入/输出`
2. 再抽出 `一步治理入口`
3. 再把 `TickOnce_()` 改为 `装配 -> 调治理 -> 消费结果`
4. 再统一执行/学习/等待分流
5. 最后补治理日志验证

## 7. 阶段 A 验收标准

当以下条件满足时，阶段 A 可以视为完成：

- `TickOnce_()` 不再直接散落治理分支
- 每轮都显式装配治理输入
- 每轮都显式得到一步治理结果
- 一步治理结果能覆盖 `执行 / 学习 / 等待 / 待机 / 停止 / 继续治理`
- 日志能稳定看到治理中轴的输入和输出
- 外部输入仍未进入自我闭环

## 8. 一句话版下一步计划

`下一步继续做阶段 A，但要按总纲收严为：先把治理输入、治理输出和治理去向做全，再让 TickOnce_ 只负责装配和消费，不再内联散落治理判断。`
