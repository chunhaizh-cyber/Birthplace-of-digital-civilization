## 2. 当前基线

### 2.1 已经完成的部分

当前代码已经具备以下基础：

- `自我主循环骨架` 已经从“需求直连方法执行”改成了正式主链顺序。
  - 入口仍然在 `自我类::TickOnce_()`
  - 已有 `固定机制 -> 根层重判 -> 根需求与根任务确保 -> 学习回流 -> 主需求选择 -> 执行/学习分流 -> 结算 -> 镜像`
- `根需求 / 根任务 / 治理任务` 已经稳定承载。
  - `确保根需求与任务_()` 已接入主循环
- `最小治理上下文` 已存在。
  - 已有 `最近根层重判结果`
  - 已有 `当前主需求 / 当前管理对象 / 当前缺口类型 / 当前下一步去向`
- `执行链` 已挂到任务头，并带开始/结束记账、步骤节点、结果节点。
- `默认本能注册` 已收口到 `初始化自我()` 所在体系，不再依赖宿主入口手工安装。
- `运行时镜像字段` 已持续写回自我特征。
- `外部输入` 当前仍被明确封禁，只记录日志，不进入自我闭环。
- `P2` 已落下第一层结构增强：
  - `完整度更高的根层重判结果结构`
  - `一步治理结果结构`
  - `多特征方向结算最小版`

  - `一步治理结果` 的日志出口

### 2.2 当前仍未完成的关键缺口

按规范对照，当前还没有完成的核心点是：

- `任务管理任务一步治理` 还没有真正作为独立治理承接中轴落地。
  - 现在更像是“主循环内联模拟一步治理”
  - 规范要求的是 `需求治理 -> 任务管理上下文 -> 任务管理任务一步治理 -> 一步治理结果 -> 外层再重判`
- `学习子任务链` 还没有完成从“尝试学习参数队列”到“正式学习子任务”的升级。
- `经验项后果 / 禁止项后果` 目前只是最小回写与日志，不是完整治理面。
- `多特征方向裁决` 还不是严格的规范版。
  - 当前已经开始比较多个自我特征
  - 但还没有完整实现“否决项先裁、普通项再综合”的正式口径
- `治理面临时记忆 / 衰减 / 管理实例堆 / 恢复快照` 还没有落地。
- `学习写回 -> 元层沉淀 -> 外层允许回流` 这一段还没正式接上。

## 3. 总体阶段划分

后续实现按 6 个阶段推进：

1. `阶段 A：治理中轴实化`
2. `阶段 B：学习子任务正式化`
3. `阶段 C：结果比较与后果治理正规化`
4. `阶段 D：学习写回与元层沉淀接入`
5. `阶段 E：治理实例堆与恢复快照`
6. `阶段 F：外部输入解封前验收`

其中：

- `阶段 A-C` 属于当前最优先路径
- `阶段 D-E` 属于把“能运行”推进到“能沉淀、能恢复、能持续治理”
- `阶段 F` 不是立刻开放外部输入，而是定义“什么时候才允许开放”

## 4. 分阶段实施计划

### 阶段 A：治理中轴实化

#### 目标

把当前 `TickOnce_()` 里的“内联一步治理”收成正式的治理承接中轴，让 `任务管理任务` 不再只是一个被确保存在的头结点，而是成为当前主闭环中的正式治理执行位。

#### 规范依据

- `1265` 第 4、7 节
- `1266` 主循环最小职责表
- `1214` 上下文字段规范

#### 当前基础

- `结构_根层重判结果_最小`
- `结构_一步治理结果_最小`
- `recent_governance_result_`
- `提交一步治理结果_()`

这些都已经有了，但还没有把“治理决策逻辑”真正收成一层清晰的治理函数或治理承接入口。

#### 本阶段交付

- 抽出 `一步治理入口`
  - 输入：`最近根层重判结果 + 当前主需求 + 当前管理对象 + 当前缺口类型 + 当前下一步候选`
  - 输出：`一步治理结果 + 治理证据 + 执行候选/学习候选/等待候选`
- 明确 `治理域判定`
  - 根治理
  - 普通执行治理
  - 学习治理
  - 边界治理
- 明确 `缺口分类` 与 `去向分类`
  - 缺方法
  - 条件缺口
  - 学习缺口
  - 待机/停止
  - 执行/继续治理/转入学习
- 让 `TickOnce_()` 从“亲自分支判断”退到“装配输入 -> 调一步治理 -> 消费结果”

#### 验收标准

- `TickOnce_()` 不再直接散落多段执行/学习去向判断
- 每一轮都能产出结构化 `一步治理结果`
- 日志能稳定看到治理分支，而不只看到任务创建

#### 日志验收

至少出现：

- `一步治理结果`
- 当前治理域
- 当前缺口类型
- 当前下一步去向

### 阶段 B：学习子任务正式化

#### 目标

把当前“尝试学习参数队列”升级成符合规范的 `学习子任务治理链`。

#### 规范依据

- `1207_学习任务作为任务管理内部治理子任务规范.md`
- `1215_学习任务目标-对象-产出-回流规范.md`
- `1265` 第 8 节

#### 当前基础

当前已经有：

- `内部尝试学习需求`
- `内部尝试学习任务`
- `尝试学习参数队列`
- `消费一个尝试学习参数队列项_()`
- `触发尝试学习_()`

但这些还只是“最小兜底学习链”，不是正式的学习子任务体系。

#### 本阶段交付

- 给学习子任务补齐正式来源字段：
  - 来源根任务
  - 来源宿主任务
  - 来源步骤
  - 来源方法
  - 来源需求位
  - 当前学习缺口类型
- 把 `学习目标 -> 学习对象 -> 学习产出 -> 回流策略` 接全
- 至少支持 3 种回流模式：
  - `直接回流`
  - `匹配后回流`
  - `仅沉淀资产不回流`
- 让学习结果先成为承接层结果，再回到外层重判决定是否正式接回闭环

#### 验收标准

- 学习不再只是“拿到参数就试”
- 每次学习都有来源、目标、对象、产出、回流策略
- 学习结果不会直接越权唤醒原任务

#### 日志验收

至少出现：

- 学习子任务创建
- 学习目标判定
- 学习产出定义
- 回流策略判定
- 学习回流结果

### 阶段 D：学习写回与元层沉淀接入

#### 目标

把学习承接结果正式接到 `学习写回 -> 元层沉淀 -> 外层再裁决` 这条规范链上。

#### 规范依据

- `1270_元层核心与学习写回实现规范.md`
- `1215` 的学习产出与回流规则

#### 当前基础

工程里已经有：

- `元层核心模块.ixx`
- `学习写回模块_v0.ixx`

但当前自我主循环尚未把它们接成正式主链的一部分。

#### 本阶段交付

- 接上最小学习写回顺序：
  - 补建关系
  - 写任务统计
  - 写方法统计
  - 写经验实体
  - 记学习写回事务事件
- 给每次写回补失败摘要与可重试口径
- 保证写回不在热点锁内执行
- 让写回结果回到主循环，再决定是否允许正式回流

#### 验收标准

- 学习结果不再只是队列消费完成
- 学习结果有正式写回步骤和失败摘要
- 写回后仍要回到外层重判，不直接算默认能力升级

#### 日志验收

至少出现：

- 学习写回开始
- 学习写回成功/失败摘要
- 写回事件记录
- 回到外层重判

  - 当前治理域
  - 当前缺口
  - 当前候选
  - 最近结果摘要
  - 最近后果摘要

  - 当前实例

### 阶段 F：外部输入解封前验收

#### 目标

定义“什么时候才允许开放外部输入进入自我”，而不是现在就开放。

#### 当前约束

当前必须继续保持：

- 前端输入只记录日志
- 外部输入不进入自我闭环
- 验证主要通过日志完成

#### 解封前必须满足

- 阶段 A-E 全部通过
- 主循环已经有稳定治理中轴
- 学习链已经可追溯
- 后果治理与治理记忆已经可解释
- 至少完成一轮“执行 -> 结果回流 -> 后果回写 -> 再治理”的日志闭环验证

#### 解封顺序

如果未来开放，只能按下面顺序：

1. 先把外部输入作为 `宿主消息` 进入治理面
2. 再分类是否转成内部需求候选
3. 外层重判允许后，才进入正式主闭环

禁止：

- 前端文本直接写成自我需求
- 宿主输入绕过治理中轴直接进方法执行
