# 20260403 自我主循环重构计划

## 1. 目标

把当前 `自我类.ixx` 从“需求直接召回方法并执行”的实验性最小闭环，重构为符合规范主链的 `自我线程主循环`：

`回流归并 -> 固定机制 -> 根层重判 -> 需求治理 -> 根任务/治理任务确保 -> 上下文装配 -> 一步推进 -> 结果回流 -> 双值结算 -> 最终去向`

本次重构先追求：

- 主循环逻辑自洽
- 根需求与根任务正式接入
- 执行链挂到任务头并完成记账
- 双值结算从“固定奖惩”升级为“按需求方向结算”

本次不追求：

- 一次性做完整历史版任务控制系统回归
- 一次性补全所有 03xx/12xx 扩展细则
- 一次性重写学习系统与方法系统

## 2. 当前问题

当前代码的关键偏差如下：

1. `TickOnce_()` 仍是旧链：
   `扫描需求 -> 选下一需求 -> 召回方法 -> 执行 -> 成功/失败固定奖惩`
2. `确保自我安全服务根需求()` 已存在，但主循环没有正式接入。
3. `当前主需求 / 当前主任务 / 当前主方法` 只初始化，不持续维护。
4. `待机状态 / 时序步长 / 服务衰减步长` 只初始化或只保留在原子缓存，没有持续镜像回自我特征。
5. 固定机制没有统一在每轮主循环中执行。
6. 执行链没有正式挂到任务头，缺少 `任务执行开始/结束` 记账。
7. 双值结算仍然是固定成功奖励、固定失败惩罚，不符合按需求方向统一结算的规范。
8. 默认本能注册仍在 `鱼巢.cpp` 入口层，不在 `自我初始化` 体系内。

## 3. 设计原则

重构时固定遵守以下边界：

- 正式主链只保留在 `自我类.ixx`
- `任务类.ixx` 负责任务头、执行记账、状态承载，不代替外层最终裁决
- `需求类.ixx` 负责需求树治理，不直接越权成最终执行器
- `方法类.ixx` 与本能函数只提供执行能力，不直接结算双值
- 世界树中的自我特征仍是正式真值面，原子变量只作为运行时缓存

## 4. 改造范围

本次直接涉及：

- `D:\鱼巢\需求类.ixx`
- `D:\鱼巢\任务类.ixx`
- `D:\鱼巢\鱼巢.cpp`

## 5. 分阶段计划

### P0 主链成形

#### P0-1 重写 `TickOnce_()` 骨架顺序

把 `自我类::TickOnce_()` 改成正式主循环顺序：

1. 消费回流与轻量归并
2. 执行固定机制
3. 生成最近根层重判结果
4. 刷新与整理需求
5. 确保安全根需求、服务根需求
6. 确保安全根任务、服务根任务、治理任务
7. 装配最小任务管理上下文
8. 决定本轮主需求和主任务
9. 若允许执行，再进入执行或学习承接
10. 统一做结果结算与最终去向决策

交付标准：

- 主循环不再直接走“选需求后立即执行”
- 固定机制不再只在“无需求”分支触发
- 根需求与治理上下文在每轮都可见

#### P0-2 接入根需求与根任务

正式接入：

- `需求类::确保自我安全服务根需求`
- `任务类::确保需求头结点`

目标：

- 自我线程启动后，始终存在两棵根需求树
- 两棵根需求树分别有对应常驻根任务头
- 根任务头可复用，不每轮重建

交付标准：

- `安全根需求`
- `服务根需求`
- `安全根任务`
- `服务根任务`

都能由主循环稳定确保存在

#### P0-3 建立最小治理上下文

先不实现完整历史版管理实例，只先在 `自我类` 内建立最小稳定承载面，至少包含：

- 最近根层重判结果
- 当前主需求指针
- 当前管理对象指针
- 当前缺口类型
- 当前下一步去向

目标：

- 后续每轮都能用同一套中间态继续推进
- 不再依赖临时局部变量拼接闭环

交付标准：

- 有独立的上下文结构或等价稳定承载字段
- 主循环可以显式刷新和消费这些字段

#### P0-4 把执行链挂到任务头

当前直接调用：

- `执行方法_同步等待`

后续要改成：

1. 先给当前主需求确保任务头
2. 执行前调用 `标记任务执行开始`
3. 执行后调用 `标记任务执行结束`
4. 失败时按缺口类型决定是否转入学习承接

交付标准：

- 普通执行不再脱离任务头裸跑
- 当前任务/当前方法/最近结果可回挂

#### P0-5 替换固定成功失败奖惩

当前逻辑：

- 成功固定 `服务 +50`
- 失败固定 `服务 -10`
- 失败固定 `安全 -5`

改为：

- 先比较执行前后的需求差值
- 形成实际结果方向
- 与需求方向做对齐判断
- 再统一生成安全值/服务值结算增量

第一阶段允许采用“简化版对齐结算”：

- 需求差值缩小：正向结算
- 需求差值扩大：负向结算
- 需求差值不变：零增量或转学习

交付标准：

- 主循环中不再出现固定数值型成功/失败奖惩主逻辑
- 双值变化开始依赖需求方向和结果方向

### P1 运行时收口

#### P1-1 自我镜像字段持续写回

把以下字段从“只初始化”变成“运行时持续镜像”：

- 当前主需求
- 当前主任务
- 当前主方法
- 待机状态
- 时序正向步长
- 时序反向步长
- 服务时序衰减步长

目标：

- 世界树中的自我特征重新成为正式运行真值面

#### P1-2 固定机制统一入口化

统一封装为主循环固定调用：

- 服务归零待机保护
- 定时衰减服务值
- 风险安全回归

目标：

- 固定机制不散落在分支里
- 每轮调用顺序稳定

#### P1-3 默认本能注册收口到自我初始化

把默认本能注册从 `鱼巢.cpp` 入口层逐步迁回 `初始化自我()` 或其直接依赖链。

目标：

- 自我模块单独初始化后即可具备最小运行闭环

#### P1-4 初值口径统一

把初始化口径统一为更接近规范的版本：

- 初始安全值 = 中位值
- 初始服务值 = 中位值

同时复核：

- 风险安全回归目标值
- 时序步长默认值
- 待机切换边界

### P2 后续增强

以下内容进入第二阶段，不阻塞本次主闭环落地：

- 完整版根层重判结果结构
- 完整版一步治理结果返回面
- 多特征需求方向汇总
- 完整经验项后果/禁止项后果回写
- 完整学习子任务治理链
- 管理实例堆与恢复快照

## 6. 删改边界

### 可以替换的旧链

后续可直接整体替换 `自我类.ixx` 中这段旧逻辑：

- `TickOnce_()` 内从“选择下一需求”到“固定成功/失败奖惩”的主干

原因：

- 这段就是当前不符合规范的旧闭环核心

### 先保留复用的部分

以下部分先不删，优先接入新主链：

- worker 执行承载
- 方法召回能力
- 尝试学习参数队列
- 固定先天动作函数本体
- `任务类` 现有头结点和记账辅助函数

原因：

- 这些更适合复用，不值得在本轮重构中同时推倒

## 7. 实施顺序

建议严格按下面顺序推进：

1. 重写 `TickOnce_()` 骨架
2. 接入根需求与根任务确保
3. 补最小治理上下文
4. 接入任务头执行记账
5. 替换双值结算
6. 补运行时镜像写回
7. 收口默认本能注册
8. 统一初值与边界

## 8. 验收标准

完成本计划第一阶段后，至少满足：

- 自我启动后自动确保安全根需求和服务根需求
- 主循环每轮统一执行固定机制
- 主循环显式维护当前主需求/当前主任务/当前主方法
- 普通执行经过任务头并具备开始/结束记账
- 双值变化不再直接依赖“成功/失败固定奖惩”
- 学习承接成为执行链的正式分流，而不是裸分支兜底

## 9. 风险与注意事项

- 不要一开始就补完整历史版 `任务管理任务`，否则改造面会急剧扩大
- 不要先手工大删 `自我类`，应先搭好新主链再替换旧链
- 不要把原子缓存继续扩张成第二套真值层
- 不要让方法执行器越权做最终结算

按本计划执行时，第一刀从 `自我类.ixx` 的 `TickOnce_()` 下手，先把主循环骨架改正，再逐步把根需求、根任务、治理上下文和双值结算接上。
