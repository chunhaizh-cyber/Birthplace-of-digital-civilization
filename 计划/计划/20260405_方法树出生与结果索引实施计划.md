# 20260405 方法树出生与结果索引实施计划

## 1. 目的

本文用于根据当前最新规范与现行代码，实现并收口以下两条正式主线：

1. `任务分解 -> 需求位方法头出生`
2. `动作动态因果 -> 正式低成熟度方法树出生或回补旧树`

同时补上工程侧的新组织规则：

1. `结果特征类型` 作为方法召回的索引桶
2. `动作不同` 或 `条件组合不同` 时，独立生成另一棵方法树
3. 学习不负责方法头出生，只负责补条件、补结果、验证和升格

2. `需求位方法头` 只表达“当前需要什么方法”，不得直接执行，也不得直接进入正式学习。

## 4. 实施总原则

1. 先补统一出生与索引，再补具体业务出生入口。
2. 先保证 `需求位方法头` 与 `可执行方法树` 两层身份分开，再接学习链。
3. 先让 `结果特征类型` 成为可靠索引桶，再优化召回策略。
4. 学习链只允许消费：
   - 已具 `动作骨架 + 条件侧` 的方法头
   - 或已具对应最小条件的待学习方法骨架
5. 任何运行时新树出生前，都必须先走统一查重与回补判定。


## 5. 分阶段计划

### M0 基线冻结与旧口径清理

目标：

1. 冻结现有“方法出生”真实入口
2. 清理代码里剩余的“学习直接生头”旧口径

实施内容：

1. 扫描所有 `创建方法首(`、`注册方法首节点`、`查找或创建_本能方法首节点` 的实际调用点
2. 将调用点分成：
   - 初始化/本能注册
   - 任务分解出生
   - 动作动态因果出生
   - 外部导入
3. 确认学习执行链中不再直接新生方法头；若仍有残留，改成：
   - 仅整理原料
   - 或仅回补既有方法树

验收：

1. 全局能列出所有方法头出生入口
2. 学习执行链不再直接承担正式方法头创建


### M1 统一方法树查重与出生入口

目标：

建立一个统一入口，收口“命中旧树还是出生新树”。

实施内容：

1. 新增统一查重键：
   - `动作键`
   - `条件主键`
   - `主结果特征类型`
   - `主结果语义键`
2. 建立统一 helper：
   - `查找适配方法树(...)`
   - `查找或出生可执行方法树(...)`
   - `查找或出生需求位方法头(...)`
3. 统一回补判定：
   - 命中旧树则回补
   - 未命中才出生新树

建议落点：

- [高级信息类.ixx](D:/鱼巢/高级信息类.ixx)
- 新增 `方法树出生/索引` 承接模块，名称统一以 `方法_` 开头

验收：

1. 工程里不再各处散落“各自查重、各自生头”的重复逻辑
2. 本能注册、任务分解、动作动态因果三条链都能转到统一入口


### M2 结果特征类型索引桶

目标：

建立 `结果特征类型 -> 方法树列表` 的正式索引桶。

实施内容：

1. 为方法树建立按 `主结果特征类型` 的索引结构
2. 明确索引桶只做：
   - 粗召回
   - 候选归桶
   - 快速查找
3. 不在索引桶里做语义真值判定
4. 任务分解和方法选择时优先按索引桶取候选树，再做细匹配

建议落点：

- 新增 `方法_索引模块.ixx`
- 或纳入后续 `方法_出生索引模块.ixx`

验收：

1. 能按 `结果特征类型` 快速拿到候选方法树列表
2. 不同动作/不同条件组合的方法树可共处同一结果特征类型桶


### M3 任务分解出生需求位方法头

目标：

在任务分解阶段，正式显性化“当前需要什么方法”。

实施内容：

1. 在任务分解/方法筹办链上，识别当前需求目标特征类型
2. 若当前还没有适配方法树，则先出生 `需求位方法头`
3. `需求位方法头` 至少写入：
   - 目标特征类型
   - 主结果方向或主作用方向
   - `是否需求位方法头 = true`
4. 该头不得直接进入：
   - 执行
   - 正式学习
5. 仅允许它进入：
   - 继续补动作骨架
   - 继续补条件侧
   - 等待后续转化为可执行方法树

建议落点：

- [自我_主循环治理模块.ixx](D:/鱼巢/自我_主循环治理模块.ixx)
- [任务管理类.ixx](D:/鱼巢/任务管理类.ixx)
- 需要时补到 `步骤可用方法候选` 的承载链

验收：

1. 首轮任务分解时可观察到 `需求位方法头` 出生
2. 控制面板或日志能明确区分：
   - `需求位方法头`
   - `可执行方法树`
3. `需求位方法头` 不会被学习链直接消费


### M4 动作动态因果出生正式低成熟度方法树

目标：

让 `自我动作动态因果` 真正进入“先出生方法头，再交学习补全”的正式链。

实施内容：

1. 从动作动态和因果证据中提取：
   - 动作键
   - 条件主键
   - 主结果特征类型
   - 主结果语义键
2. 进入统一出生判定：
   - 若命中旧树，则回补旧树
   - 若未命中，则出生新低成熟度方法树
3. 新生树默认成熟度落在：
   - `运行期临时`
   - 或 `反推候选`
4. 学习后续只负责：
   - 补条件节点
   - 补结果节点
   - 写回账本
   - 验证和升格

建议落点：

- [任务方法世界树桥接类.ixx](D:/鱼巢/任务方法世界树桥接类.ixx)
- [因果信息类.ixx](D:/鱼巢/因果信息类.ixx)
- `方法_出生索引模块`

验收：

1. 有动作动态因果时，可以直接观察到：
   - `命中旧树回补`
   - 或 `新低成熟度方法树出生`
2. 学习链不再承担“先出生方法头”的责任


### M5 学习链适配新方法树规则

目标：

把学习链彻底适配到“只补全、不生头”的新规则。

实施内容：

1. 学习请求对象只允许是：
   - 可执行方法树
   - 待学习方法骨架
2. 若传入的是 `需求位方法头`：
   - 直接拒绝进入正式学习
   - 写明“待补动作/条件”
3. 学习正式写回仍按：
   - 条件主键
   - 结果主键
   精确命中节点
4. 学习不再承担：
   - 正式方法首节点出生

落点：

- [学习类.ixx](D:/鱼巢/学习类.ixx)
- [自我_学习承接模块.ixx](D:/鱼巢/自我_学习承接模块.ixx)

验收：

1. 需求位头进入学习时会被稳定拒绝
2. 可执行方法树进入学习时仍保持现有 `H1~H8` 全通过


### M6 方法选择与任务分解改走“结果索引桶 -> 方法树候选”

目标：

把主循环/任务管理的候选方法召回切到新索引规则上。

实施内容：

1. 在任务分解时，先按 `目标结果特征类型` 取索引桶
2. 再从桶中按：
   - 动作
   - 条件组合
   - 主结果语义
   - 当前条件匹配
   做候选树筛选
3. 未命中时：
   - 出生 `需求位方法头`
   - 或继续等待动作动态因果侧新树出生

落点：

- [自我_主循环治理模块.ixx](D:/鱼巢/自我_主循环治理模块.ixx)
- [任务管理类.ixx](D:/鱼巢/任务管理类.ixx)

验收：

1. 方法选择入口先按结果特征类型召回
2. 结果特征类型相同但动作/条件不同的多棵树可同时成为候选


### M7 日志、控制面板与验收

目标：

让新规则可观察、可审计、可验收。

实施内容：

1. 新增或统一日志阶段键：
   - `任务分解_需求位方法头出生`
   - `任务分解_命中结果索引桶`
   - `动作动态因果_回补旧方法树`
   - `动作动态因果_出生新方法树`
   - `学习拒绝_仅需求位方法头`
2. 控制面板补充统计：
   - 需求位方法头数
   - 可执行方法树数
   - 按结果特征类型的索引桶统计
3. 新增专项验收入口：
   - `--accept-method-tree-birth`

验收：

1. 新专项验收通过
2. 旧专项验收：
   - `--accept-learning-loop`
   - `--accept-self-runtime`
   - `--accept-log-mainline`
   均不被破坏


## 6. 推荐执行顺序

建议严格按以下顺序推进：

1. `M0`
2. `M1`
3. `M2`
4. `M3`
5. `M4`
6. `M5`
7. `M6`
8. `M7`

原因：

1. 若先做 `M3/M4` 而没有统一出生/查重入口，代码会再长出平行逻辑。
2. 若先做 `M5` 而没有 `需求位方法头` 真入口，就无法真实验证“学习拒绝需求位头”。
3. 若先做 `M6` 而没有结果索引桶，则候选召回仍会散落在旧逻辑里。
