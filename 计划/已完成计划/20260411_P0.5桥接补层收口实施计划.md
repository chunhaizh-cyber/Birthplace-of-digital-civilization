## 0. 计划定位

这份计划不是从零开始补 `P0.5`。

按当前代码现状，`方法需求位 / 本能能力缺口 / 请求 / 单步决策 / 写回结果 / 一步结果 / 主体层壳` 已经落进工程；真正还没收平的是：

1. 桥接三阶段还没有在 `执行一步治理(...)` 前半段被显式收成一个“薄而硬”的补层
2. `方法需求位 -> 本能能力缺口 -> 最小原语` 已经接进主流程，但还没有形成“命中后先裁决、再决定是否允许进入树写回与学习”的明确闸门
3. `结构_任务管理请求 / 单步决策 / 写回结果 / 一步结果` 已经存在，但仍有大量字段通过 `结构_任务管理结果` 和多组 `P0_覆盖...` 辅助函数来回镜像，壳已经有了，边界还不够薄
4. 学习承接已经提前接进来了，导致当前桥接补层还不够像一个“前半段治理消费层”

因此，这份新计划的目标不是“把桥接层加出来”，而是：

`把已经接进去的桥接层前移、收薄、硬化成真正的 P0.5 补层。`

## 2. 计划目标

这轮计划只做一件事：

`把桥接层收成一个显式、前置、可早停的 P0.5 补层。`

固定顺序应变成：

```text
受理请求
-> 宿主锚点确认
-> 当前方法锚点确认
-> 形成治理特征面
-> 命中当前方法需求位
-> 评估当前本能能力缺口
-> 桥接闸门裁决
-> 若允许，再进入树写回 / 学习承接 / 主体回并
-> 返回一步结果
```

这轮明确不做：

1. 不重做线程池或运行模型
2. 不扩大学习树语义
3. 不再新增一批主体化概念
4. 不把 `结构_任务管理结果` 一次性拆爆

这样做完之后，当前桥接层就会从“已经接进来了，但还混在大函数里”变成：

`一个真正能先消费原语、再决定后段是否继续的 P0.5 补层。`


## 7. 结论

结合当前代码现状、已有验收结果和这次建议，新的实施重点不再是：

- 先补桥接结构本身
- 先扩学习
- 先改线程池

而是：

`把已经存在的桥接层收成“前半段显式桥接三阶段 + 一个原语消费闸门 + 四层薄壳主写口”。`

等这轮做完，后续再拆 `请求 / 决策 / 写回 / 返回` 时，就不再是硬重构，而会更像机械搬家。
