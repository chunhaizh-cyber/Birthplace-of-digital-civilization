## 4. 完善红线

后续完善必须一直遵守以下红线：

1. 不得为了迁移功能修改现有任务、需求、方法、存在、自我等领域数据结构的语义边界。
2. 可以新增模块、运行时适配层、结果结构、日志检查结构，但不得另起与世界树并列的长期真值层。
3. 线程、学习、调度、日志、控制面板都只能作为承接层和观测层，不能反向改写根约束和根裁决边界。
4. 一切“继续 / 停止 / 收束 / 待机 / 转学习 / 再提交”的最终决定，必须继续回到外层主循环，不得由执行器、学习器或局部治理函数越权代决。

## 6. 推荐执行顺序

建议严格按以下顺序推进：

1. `P0 验收基线`
2. `P0-R1 双值规则解释收口与结算顺序固定`
3. `P1 主循环骨架`
4. `F-R1 自我线程最小闭环验证双值时间本能动作`
5. `P2 上下文与返回面`
6. `P3 任务筹办与执行`
7. `P4 学习模块`
8. `P5 方法树成长`
9. `P6 双值结算与最终裁决`

原因：

1. 没有 `P0`，后续每一轮都要重新人工判断差距。
2. 没有 `P0-R1`，双值时间规则会长期停在“方向对了但结算接口不清”的状态。
3. 没有 `P1/F-R1`，后面的执行、学习、方法成长都没有稳定挂接面，双值时间动作也缺少最小闭环验证。
4. `P3` 完成后，任务系统才不再只是一条治理展示链。
5. `P4` 必须晚于 `P1/P2/P3`，否则学习模块会先失去承接边界。
6. `P5/P6` 放后面，是为了避免在执行和学习承接没立稳前就过早做高层成长与裁决。
