状态：进行中（2026-04-11，`M7-1 / M7-2 / M8 / M9-1 / M9-2 / M10-1 / M10-2前两刀` 已完成，业务子任务派生待续）

## 目标

在 `M1-M6` 已完成基线上，按“先把事件变成有序可见，再谈持久化、自动化和内聚重构”的顺序推进：

- `M7-1`：事件优先级、幂等键、队列观测
- `M7-2`：事件持久化与恢复重放纪律
- `M8`：`--acceptance-report` 自动化回归门
- `M9`：任务管理模块内聚重构
- `M10`：更完整的业务步骤链与学习回流长期策略
  - `M10-1`：正式学习回流沉淀到主体长期策略
  - `M10-2`：更完整业务步骤链与长期策略深化

## 当前判断

当前代码已经具备：

- 单消费者 `mailbox`
- 五类统一事件源：`普通推进 / 执行结果 / 学习回流 / 外部反馈 / 恢复重建`
- 主循环统一延迟事实收集与最终去向裁决
- 恢复重建一次性消费语义
- `M1-M10` 共 22 个验收场景、重复执行入口、JSON 输出与本地 `pre-push` 门禁

当前最主要的缺口，不再是“事件能否回来”，而是：

- 如何把当前回归门继续沉到更稳定的 CI
- 如何继续切薄 `任务管理任务模块`
- 如何继续把业务步骤链和学习回流扩成更完整的长期策略

### M7-2 事件持久化与恢复重放纪律（已完成）

本轮已完成内容：

- 为治理恢复快照补齐 `待恢复治理事件列表 / 待恢复治理事件数 / 待恢复治理事件摘要 / 最近已消费治理事件签名`
- 明确事件恢复纪律：
  - `学习回流 / 外部反馈 / 执行结果` 可跨恢复保存
  - `普通推进` 仅允许内存态消费
  - `恢复重建` 继续由恢复快照重建承接，不写成第二份线程栈
- 线程侧补齐“待回判事件镜像”同步：
  - pending/in-flight 显式事件会投影到恢复快照
  - 恢复启动时只恢复为 `待回判事实`
  - 同幂等键事件结合 `最近已消费签名` 防止恢复后二次消费
- 控制面板与线程摘要增加 `可恢复事件数 / 待回判摘要`
- `--acceptance-report` 新增 `M7-2` 三个场景并通过：
  - `场景17 M7-2 学习回流恢复待回判`
  - `场景18 M7-2 外部反馈恢复待回判`
  - `场景19 M7-2 普通推进仅内存态`

验收结果：

1. 可恢复事件会以“待回判事实”而非旧线程栈形式恢复
2. `普通推进` 不会被持久化成恢复事件
3. 恢复后会记录最近已消费事件签名，且不会留下同标记残留事件
