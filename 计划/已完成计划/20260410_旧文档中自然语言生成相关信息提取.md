## 3. 旧资料中已经明确的总原则

### 3.1 自然语言是表达层，不是真值层

旧资料里反复强调：

- 自然语言名称、描述、解释，只能作为显示与解释层
- 世界树、特征、状态、动态、因果这些结构化对象才是真值层
- 名称只是附属解释字段，不是对象主键
- 身份以结构化签名为主，不应退化成“纯自然语言命名系统”

可直接转为当前规范的口径：

- `自然语言显示不得反向定义世界树真值`
- `自然语言只负责把现有结构翻译成可读表达`
- `显示名、解释句、摘要句都属于视图层派生物`


### 3.2 自然语言名称必须统一从语素集获取

旧规范 `1180` 已经给出明确规则：

- 一切自然语言名称、动作名称、描述性短语，都必须通过 `语素集.添加词性词(...)` 获取
- 不允许为显示层再造一套独立词对象池
- 不允许只持有字符串而绕开语素集
- 允许按需生成，不要求预先全量生成

可直接转为当前规范的口径：

- `世界树自然语言显示中的所有可读词名，统一从语素集取得`


## 4. 旧资料中已有的语言结构骨架

### 4.1 自然语言树的层级

旧代码已经定义出一套自然语言结构树：

- 文章
- 段落
- 自然句
- 子句
- 句子成分

其中最关键的是：

- `自然句主信息类` 保存整句词序列、原始文本、主语、谓语、宾语
- `子句主信息类` 保存子句区间、局部词序列、子句主语、子句谓语、子句宾语


### 4.2 子句与世界树对象可以直接对应

旧 `子句主信息类` 明确保留了以下落脚点：

- `对应场景`
- `对应状态`
- `对应动态`
- `对应因果`

这条很重要，它意味着旧设计不是“先生成一堆句子再看怎么挂世界”，而是：

- 先有结构化世界对象
- 再把它们绑定到自然语言子句

可直接转为当前规范的口径：

- `世界树自然语言显示的最小生成单位，不一定是整棵树，而可以是“节点 -> 子句”`

## 6. 旧资料中已有的语言模板模型

旧资料不是“直接拼字符串”，而是定义了模板化表达模型：

- `结构体_语言模板`
- `结构体_抽象场景模板`
- `结构体_语言知识条目`
- `结构体_线性化上下文`
- `结构体_LK生成输入`
- `结构体_LK生成输出`

这套模型里已经有几个可直接复用的思想：

### 6.1 模板不是句子文本，而是槽位序列

旧 `结构体_语言模板` 中的核心字段是：

- `用途`
- `模板ID`
- `词序列槽位`
- `线性化策略ID`

这说明旧设计想要的是：

- 先定义句子骨架
- 再按上下文绑定槽位
- 最后再线性化成自然语言


### 6.2 模板生成需要上下文绑定

旧 `结构体_线性化上下文` 使用：

- `场景`
- `绑定`

其中 `绑定` 实际就是：

- `"E1" -> 某存在节点`
- `"E2" -> 某存在节点`
- `"Fx" -> 某特征节点`

这对当前世界树显示的意义是：

- 显示句子不是“孤立句式”
- 它必须知道当前在说哪个节点、哪个场景、哪个对象对


### 6.3 输出不只是句子，还包括证据链

旧 `结构体_LK生成输出` 保留了：

- `词序列`
- `最终绑定`
- `证据链`
- `置信度`

这说明旧资料已经默认：

- 自然语言显示不只是“说出来”
- 还要知道“凭什么这么说”

这对世界树显示尤其有用，因为后续可以支持：

- 展示自然语言摘要
- 同时允许展开对应状态、动态、因果证据

## 10. 旧资料中已经明确的实现缺口

旧资料里“自然语言生成”方向很清楚，但实现本体大多还是壳：

- `自然语言树类` 只有最小接口，很多函数未完成
- `表达落地.ixx` 只有占位函数
- `语言知识库类` 大量融合函数处于注释或示意态
- `语言知识库_初始化.ixx` 已有模板，但注册代码是注释态

因此可提取出的有效结论是：

- `旧资料已经给出了可复用的表达模型`
- `旧资料没有给出现成可直接搬运的完整生成器`


## 11. 对当前工程最有价值的直接继承项

如果目标是“把当前世界树显示成自然语言”，从旧资料中最值得直接继承的是这 7 条：

1. 自然语言只做显示与解释，不得变成新的真值层。
2. 所有可读名称统一通过 `语素集.添加词性词(...)` 获取。
3. 最小生成单位应允许是“节点 -> 子句”，不必一开始就强求整篇文章。
4. 世界树显示优先分成六类用途：存在、特征、事件、关系、比较、因果。
5. 句子生成应优先走“模板槽位 + 上下文绑定 + 线性化”，而不是到处手拼字符串。
6. 因果显示默认口径应保持 `条件 + 动作 = 结果`。
7. 生成输出应尽量保留证据链与置信度，不只保留最终文本。
