## 0. 计划定位

这份计划用于纠正当前 `任务管理_*` 特征的宿主混层问题。

本计划启动时最核心的偏差只有一条：

`自我只能挂自我特征，但当时的代码仍把大量任务管理局部治理值写进了自我存在。`

因此，这份计划不追求“一步到位把所有新存在都正式节点化”，而是分两段推进：

1. 先把语义归属和运行写入从 `自我存在` 上剥离出来
2. 再把桥接类特征从过渡承载面继续下沉到 `任务管理主体虚拟存在 / 分身继承面 / 方法需求位特征面`

一句话收口：

`先止血，再迁宿主，最后拆细承载。`


## 1. 当前代码结论

### 1.1 计划启动时真正的问题

计划启动时，仓内已经形成以下边界：

- `任务目标状态 / 任务结果状态` 已经写入 `任务虚拟存在`
- `方法类型` 已经写入 `方法虚拟存在`
- `任务管理主体虚拟存在 / 分身继承面 / 方法需求位特征面 / 本能能力缺口面` 已经有结构层壳

但同时仍存在一个关键偏差：

- 大多数 `任务管理_*` 特征仍通过 `私有_记录治理证据_I64(...)` / `私有_记录治理证据_指针(...)` 写入 `自我存在`

这意味着计划启动时实现里混着两套口径：

1. 规范口径：`任务管理特征属于任务管理任务侧承载`
2. 代码口径：`任务管理特征当前先落在自我存在`

截至当前，上述写入偏差已经完成迁出；旧世界树、旧日志和旧状态节点这条兼容收口链，也已经补齐到“识别、归档、清理、自动审计、告警分级”这一层。

### 1.2 这会造成什么问题

若当时继续保持该写法，至少会带来以下 4 个问题：

1. `自我存在` 被迫承载大量非自我真值，语义边界被冲淡
2. 不同宿主任务、不同治理阶段、不同候选治理链的局部值容易被误读成“系统当前唯一真值”
3. 后续若要支持并列宿主、恢复重建、历史回看、主体成长汇总，会缺少清晰宿主锚点
4. 文档、面板、恢复链和代码写入宿主不一致，后续越改越难收口


## 2. 纠偏目标

本轮纠偏目标固定为以下 5 条：

1. `自我存在` 只保留自我本体特征
2. `任务管理_*` 特征全部从语义上脱离 `自我存在`
3. 短期先把运行写入宿主改到 `宿主任务虚拟存在` 或 `任务管理任务侧过渡承载`
4. 中期把桥接类特征继续下沉到 `任务管理主体虚拟存在 / 分身继承面 / 方法需求位特征面`
5. 全程保持兼容读取，避免一次性迁移击穿现有面板、恢复链和验收

本轮明确不做：

1. 不重做线程模型
2. 不重写双根树逻辑
3. 不一次性推翻当前 `结构_任务管理结果` 兼容壳
4. 不把所有主体层结构立刻强行改成正式世界树节点


## 3. 纠偏原则

后续统一按以下 6 条执行：

1. `自我只能挂自我特征`
2. 凡是随 `宿主不同 / 阶段不同 / 候选链不同` 而变化的治理值，都不得再按语义归入 `自我存在`
3. 已经有正式存在宿主的特征，优先直接落正式存在；不要再借道 `自我存在`
4. 尚未正式节点化的主体/分身桥接特征，允许先用过渡承载，但不得再解释成自我特征
5. 写入迁移必须采用 `目标宿主优先 + 旧宿主兼容回退读取` 的渐进方式
6. 验收标准以“语义边界正确 + 多宿主不串值 + 恢复链不断”为准

### P2 宿主任务治理字段先迁到宿主任务虚拟存在

- 把最主要的一批稳定治理字段从 `自我存在` 挪走

1. 先迁 `当前功能域 / 当前缺口类型 / 当前下一步去向 / 当前治理状态迁移 / 当前后果治理 / 最近总控结果 / 最近反馈摘要` 这类核心治理字段

   - 先读 `宿主任务虚拟存在`
   - 读不到再回退旧 `自我存在`

5. 对 `当前方法指针 / 当前步骤指针 / 最近结果指针` 增加“按当前宿主清洗”的失效规则：

   - `当前方法` 只有在属于当前宿主或等于宿主绑定方法时才保留

完成标准：

- 同一时刻两个不同宿主任务可以各自保留自己的治理状态，不再只能读出一套全局值
- 宿主切换或宿主丢失后，旧轮治理镜像不会残留并误导下一轮上下文拼装

## 6. 推荐实施顺序

不建议直接从 `P4` 开始。

最稳的顺序是：

1. `P0` 先冻结宿主映射
2. `P1` 先切写入口，阻止技术债继续扩散
3. `P2` 先迁稳定治理字段
4. `P3` 再迁桥接字段到过渡承载
5. `P4` 最后正式节点化主体/分身
6. `P5` 收口兼容与删除旧宿主

原因很简单：

- `宿主任务虚拟存在` 已经存在，迁移风险最低
- `任务管理主体虚拟存在 / 分身继承面` 目前还没有完全世界树节点化，不能把首轮计划压在它们上面
- 先把大头从 `自我存在` 上拿走，系统语义会立刻变干净很多


## 7. 验收标准

本计划完成至少应满足以下 8 条：

1. `自我存在` 上不再新增任何 `任务管理_*` 特征写入
2. `自我特征` 与 `任务管理特征` 的读取入口分开
3. 两个不同宿主任务的治理状态能并存，不互相覆盖
4. `任务完成 -> 宿主目标结果比较 -> 双根反馈` 链仍然成立
5. `任务目标状态 / 任务结果状态 / 方法类型` 旧链不回退
6. 恢复重建后能从任务侧宿主继续恢复，而不是回到自我侧找真值
7. 控制面板和验收报告按新宿主展示时，结论与旧语义不冲突
8. `msbuild D:\鱼巢\鱼巢.vcxproj /t:Build /p:Configuration=Debug /p:Platform=x64 /m:1 /nologo` 构建通过

2. 双宿主并列治理
   - 验证两个宿主各自保留独立治理状态
3. 当前方法切换
   - 验证 `当前方法指针 -> 方法类型` 仍能正确跨存在跳转
4. 任务完成回流
   - 验证 `任务结果状态=完成` 仍能回流到 `宿主目标结果比较`，再反馈双根
5. 恢复重建
   - 验证恢复后优先从任务侧承载读出治理真值


## 9. 主要风险与规避

### 9.1 风险：读写错位

如果先改写入、不改读取，面板会出现“值突然消失”。

规避：

- 必须采用 `目标宿主优先 + 旧宿主回退` 的双读期

### 9.2 风险：桥接字段提前下沉失败

如果还没正式节点化主体/分身，就强行把桥接字段全迁过去，容易把计划卡住。

规避：

- 先允许桥接字段过渡镜像在宿主任务虚拟存在上

### 9.3 风险：历史状态解释不一致

旧日志和旧状态节点仍然挂着 `自我存在` 宿主。

规避：

- 在计划收口阶段补一份“历史宿主兼容说明”，明确旧数据只是旧宿主实现，不代表语义归属


## 10. 本轮建议的第一步

如果紧接着要开始动代码，我建议先只做 `P0 + P1`：

1. 先把 `28` 个 `任务管理_*` 特征正式分组
2. 再把写入 helper 拆开
3. 暂时不急着批量迁状态节点

这样做的收益最大：

- 可以先阻止继续误写 `自我存在`
- 可以把后面的迁移从“散点修补”变成“按宿主分批移动”
- 不会因为主体层尚未节点化而卡住第一轮实施


## 11. 计划结论

这次纠偏的本质，不是“把一些字段从 A 挪到 B”。

真正要纠正的是一条更底层的边界：

`自我存在表达自我真值，任务管理任务侧承载表达局部治理真值，二者不得再混。`

因此后续统一按以下顺序推进：

`先冻结归属 -> 再拆写入口 -> 先迁宿主治理面 -> 再迁桥接面 -> 最后删掉自我侧兼容。`
