# 2026-04-12 抽象状态与抽象动态生成流程梳理

## 1. 先给结论

当前代码里，`抽象状态` 和 `抽象动态` 不是同成熟度的两条链。

- `抽象状态`：已经形成了比较清晰的“两段式”流程  
  先生成 `具体状态节点 / 具体动态节点 / 实例因果节点`，再由 `因果类` 把这些具体节点抽成条件短语、结果短语和自然语言。
- `抽象动态`：已经有稳定的 `动态实例生成` 和 `动态业务语义分类`，但还没有像 `抽象状态` 那样在主流程里形成一套成熟、统一、被广泛调用的“抽象动态表达单元”实现。

一句话收口：

`状态的抽象主要通过因果表达层完成；动态当前主要先完成实例记录与业务分类，抽象表达层还偏规范先行。`


## 2. 统一主链位置

当前系统里与这两个对象最贴近的主链，可以压成：

```text
特征当前值变化
-> 生成条件状态 / 结果状态
-> 生成动态节点
-> 生成实例因果
-> 从实例因果提取表达单元
-> 生成抽象状态短语 / 抽象结果短语 / 自然语言
```

对动态来说，当前真正落地到代码的是：

```text
动作或状态变化发生
-> 生成动态节点
-> 绑定主体 / 特征 / 来源方法 / 场景 / 相位
-> 自动推导业务语义类型
-> 供控制面板、因果桥接和后续语言层消费
```


## 3. 抽象状态生成流程

## 3.1 第一步：具体状态先出生

所有后续抽象状态，都不是直接从“特征当前值”硬拼出来的，而是先落成 `状态节点`。

统一底层创建口在：

- [自我类.cpp](/D:/鱼巢/自我类.cpp:128) `私有_创建状态节点(...)`
- [状态类.cpp](/D:/鱼巢/状态类.cpp:57) `状态类::创建状态(...)`

这一步做的事情是：

1. 指定 `场景`
2. 指定 `主体`
3. 指定 `特征`
4. 写入这次状态的值
5. 标记：
   - 是否变化
   - 原因类别
   - 原因说明

其中 `状态类::标记变化(...)` 会把：

- `是否变化`
- `变化原因类别`
- `变化原因说明`

一起挂到状态主信息上，[状态类.cpp](/D:/鱼巢/状态类.cpp:135)。


## 3.2 第二步：状态节点通常成对出现

在实际主流程里，状态通常按“前后镜像”成对生成：

- `条件状态`：变化前镜像
- `结果状态`：变化后镜像

典型写法见：

- [自我类.cpp](/D:/鱼巢/自我类.cpp:387) `私有_记录自我I64状态(...)`
- [自我类.cpp](/D:/鱼巢/自我类.cpp:285) `私有_记录自我指针状态(...)`
- [任务管理任务模块.cpp](/D:/鱼巢/任务管理任务模块.cpp:4103) `私有_记录宿主任务治理证据_I64(...)`
- [任务管理任务模块.cpp](/D:/鱼巢/任务管理任务模块.cpp:4228) `私有_记录宿主任务治理证据_指针(...)`
- [任务管理任务模块.cpp](/D:/鱼巢/任务管理任务模块.cpp:4366) `私有_记录主体桥接证据_I64(...)`
- [任务管理任务模块.cpp](/D:/鱼巢/任务管理任务模块.cpp:4491) `私有_记录主体桥接证据_指针(...)`

固定模式基本都是：

```text
读取旧值
-> 写入新特征当前值
-> 若有旧值，创建条件状态
-> 创建结果状态
```

因此：

- `抽象状态` 的输入真值，不是某个裸值
- 而是“已经结构化落成的状态节点实例”


## 3.3 第三步：实例因果消费状态时，先取具体状态短语

实例因果表达阶段，并不立刻做抽象归类，而是先把具体状态节点变成“状态短语”。

核心入口在：

- [因果类.cpp](/D:/鱼巢/因果类.cpp:382) `私有_状态短语(...)`
- [因果类.cpp](/D:/鱼巢/因果类.cpp:788) `生成实例因果表达单元(...)`

这里做的是：

1. 读取状态节点上的主体、特征、值
2. 生成类似：
   - `X的Y为Z`
   - `Y为Z`
   - `状态为Z`

这一步仍然是“实例表达”，不是高度抽象表达。


## 3.4 第四步：从实例状态抽成“抽象条件短语”

真正进入 `抽象状态` 的入口，是：

- [因果类.cpp](/D:/鱼巢/因果类.cpp:367) `私有_抽象条件短语(...)`
- [因果类.cpp](/D:/鱼巢/因果类.cpp:262) `私有_抽象数值条件短语(...)`

当前规则是：

1. 若状态值是 `I64`
2. 则按“特征类型 + 数值区间”压缩成更抽象的条件短语

例如当前已写死的几类：

- `安全类特征`
  - 低位
  - 高位
  - 稳定带
- `服务值`
  - 低位
  - 高位
  - 中位区

也就是：

```text
具体状态值
-> 识别特征类型
-> 按阈值区间归类
-> 形成抽象条件短语
```

这正符合规范里“特征状态描述的是特征值如何被抽象”的口径，[0550_核心术语与概念定义规范.md](/D:/鱼巢/规范/0550_核心术语与概念定义规范.md:432)。


## 3.5 第五步：从前后状态差分抽成“抽象结果短语”

结果侧的抽象入口在：

- [因果类.cpp](/D:/鱼巢/因果类.cpp:316) `私有_抽象结果短语(...)`

这里的规则比条件侧更强，因为它会同时看：

1. `结果状态`
2. 是否存在对应的 `条件状态`
3. 前后值有没有变大、变小或保持稳定
4. 动作语义键里有没有：
   - `回升`
   - `回落`
   - `归零`
   - `切换`
   - `迁移`
   - `推进`
   - `衰减`
   - `增加`
   - `下降`

所以结果抽象链大致是：

```text
结果状态
-> 查找同主体同特征的条件状态
-> 做前后比较
-> 若能直接比较，优先生成“上升/下降/回升/回落/切换”
-> 若不能直接比较，则退回按动作语义键推断方向
```

这一步本质上是在做：

`具体状态差分 -> 抽象方向结果`


## 3.6 第六步：抽象状态真正作为“抽象因果表达单元”的输入被消费

当前抽象状态在主链中的最终消费口，是：

- [因果类.cpp](/D:/鱼巢/因果类.cpp:822) `生成抽象因果表达单元(...)`

这里做的事情是：

1. 选一个证据实例因果
2. 把实例因果里的：
   - 条件状态
   - 结果状态
3. 分别送进：
   - `私有_抽象条件短语(...)`
   - `私有_抽象结果短语(...)`
4. 形成：
   - `条件短语`
   - `动作短语`
   - `结果短语`
   - `主结果短语`

所以可以明确说：

`当前抽象状态，不是独立先生成一棵正式状态概念树再回填主流程；而是先经由实例状态与实例因果，再在表达层被抽成抽象条件/结果短语。`


## 4. 抽象动态生成流程

## 4.1 第一步：先生成具体动态节点

动态的底层创建口在：

- [动态类.cpp](/D:/鱼巢/动态类.cpp:121) `动态类::创建动态(...)`
- [动态类.cpp](/D:/鱼巢/动态类.cpp:134) `动态类::创建动态(场景, 初始状态, 结果状态, 起, 止)`

当前最常见的动态来源有两类：

1. `状态变化驱动`
2. `动作边界驱动`

对应到代码里：

- 自我层本能动作动态：
  - [自我类.cpp](/D:/鱼巢/自我类.cpp:151) `私有_创建本能动作动态(...)`
- 任务管理治理动态：
  - [任务管理任务模块.cpp](/D:/鱼巢/任务管理任务模块.cpp:4763) `私有_记录最小原语边界动态(...)`
  - [任务管理任务模块.cpp](/D:/鱼巢/任务管理任务模块.cpp:4713) `私有_记录任务管理方法位专项动态(...)`
  - 以及各类治理证据写回函数内部的 `创建动态(...)`


## 4.2 第二步：动态节点补齐“主体 / 特征 / 来源方法 / 场景”

动态不是只存一个名字，它在创建后还会补充多类结构化信息：

- [动态类.cpp](/D:/鱼巢/动态类.cpp:198) `设置主体特征(...)`
- [动态类.cpp](/D:/鱼巢/动态类.cpp:209) `绑定来源方法(...)`
- [动态类.cpp](/D:/鱼巢/动态类.cpp:226) `绑定来源场景(...)`

也就是说，动态实例会被补齐：

1. 谁在动
2. 哪个特征在动
3. 来源方法是谁
4. 当前是开始、完成还是失败相位
5. 输入输出场景是什么


## 4.3 第三步：动态自动被压成业务语义类型

当前“抽象动态”最明确、最稳定的实现，不是自然语言，而是：

`动态业务语义类型`

推导逻辑在：

- [动态类.cpp](/D:/鱼巢/动态类.cpp:45) `私有_推导动态业务语义类型(...)`

当前会自动压成 5 类：

- `动作事实动态`
- `状态变化动态`
- `被动记账动态`
- `治理过程动态`
- `边界结束动态`

判定依据主要是：

1. 动作标题里有没有 `管理_`
2. 特征标题里有没有 `任务管理_`
3. 来源动作相位是不是 `失败`
4. 动作词里是不是含有：
   - `衰减`
   - `回归`
   - `回升`
   - `保护`
   - `重算`
   - `校正`
   - `锚点`
   - `进入收束`
   - `进入等待`
   - `停止`
   - `待机`

所以当前抽象动态主链可以压成：

```text
具体动态节点
-> 读取动作标题 / 特征标题 / 相位 / 来源类型
-> 自动分类到某个业务语义类型
```

这是现在最接近“抽象动态”的已落地实现。


## 4.4 第四步：动作边界动态允许独立存在，不依赖状态变化

规范和代码都明确支持：

`没有变化，不等于动作没有发生。`

因此抽象动态里有一条很重要的规则：

- 就算前后状态没变
- 只要动作确实发生
- 也可以保留 `动作事实动态`

这在规范里被反复强调：

- [0550_核心术语与概念定义规范.md](/D:/鱼巢/规范/0550_核心术语与概念定义规范.md:424)
- [1100_本能动作体系规范（任务方法本能方法函数）.md](/D:/鱼巢/规范/1100_本能动作体系规范（任务方法本能方法函数）.md:330)

代码上的体现主要是：

- 自我层本能动作动态可以单独创建，[自我类.cpp](/D:/鱼巢/自我类.cpp:151)
- 任务管理的最小原语开始/结束会单独记一条动态，[任务管理任务模块.cpp](/D:/鱼巢/任务管理任务模块.cpp:4763)

也就是说：

`抽象动态` 当前既承担“状态变化过程”的抽象，也承担“动作边界事实”的抽象。


## 4.5 第五步：动态当前更多是“分类完成”，语言抽象层尚未全面接通

规范里已经要求动态也应该走：

`动态节点 -> 动态表达单元 -> 自然语言`

见：

- [2060_动态信息自然语言表达规范.md](/D:/鱼巢/规范/2060_动态信息自然语言表达规范.md:176)

但从当前代码实现看：

- `动态类.cpp` 已经完成了：
  - 动态实例创建
  - 主体/特征/来源绑定
  - 业务语义分类
- 但还没有看到和 `因果类::生成抽象因果表达单元(...)` 对等成熟度的
  - `生成动态表达单元(...)`
  - `生成动态自然语言(...)`

所以必须明确：

`当前抽象动态更像“结构化分类已落地、统一表达层待补”。`


## 5. 两条链放在一起看

可以把当前实现压成下面这张对照表。

| 层级 | 抽象状态 | 抽象动态 |
| --- | --- | --- |
| 实例出生 | `创建状态(...)` | `创建动态(...)` |
| 结构补齐 | 主体、特征、值、变化原因 | 主体、特征、来源方法、相位、场景 |
| 主过程载体 | 状态节点 | 动态节点 |
| 当前抽象手段 | 区间归类 + 前后差分 + 动作语义方向 | 业务语义类型分类 |
| 当前主消费口 | `生成抽象因果表达单元(...)` | 控制面板 / 因果桥接 / 未来语言模块 |
| 当前成熟度 | 已有可运行抽象表达 | 已有可运行抽象分类，但表达层未完全收口 |


## 6. 当前真正的“生成顺序”

如果只看已经落代码的稳定顺序，可以固定为：

### 6.1 抽象状态

```text
特征值变化
-> 状态节点（前 / 后）
-> 动态节点
-> 实例因果节点
-> 抽象条件短语 / 抽象结果短语
-> 抽象因果表达单元
```

### 6.2 抽象动态

```text
动作发生或状态变化
-> 动态节点
-> 绑定主体 / 特征 / 来源动作 / 场景 / 相位
-> 自动推导业务语义类型
-> 供面板、因果和未来语言层消费
```


## 7. 还没完全打通的地方

当前最值得注意的缺口有 3 个。

### 7.1 抽象状态仍主要靠表达层即时抽象

当前虽然规范里已经有：

- `特征状态`
- `状态概念树`

但代码主流程里，真正稳定落地的是：

- 先生成实例状态
- 再在 `因果类` 里即时把它抽成短语

也就是说，`状态概念树` 还不是当前主治理链的主承载。

### 7.2 抽象动态还没有完整“表达单元”实现

动态目前有：

- 实例
- 分类
- 相位
- 场景

但还缺一条像因果那样明确、统一、被广泛调用的：

- `动态表达单元`
- `动态自然语言`

### 7.3 二次特征层还没有并入这次梳理的主链

规范里其实明确要求：

`状态 / 动态 -> 二次特征 -> 因果 / 判定 / 裁决`

但这次代码主链里最稳定可见的是：

- `状态 -> 动态 -> 实例因果 -> 抽象表达`

因此当前如果问“抽象状态、抽象动态从哪里来”，答案主要落在：

- `状态节点`
- `动态节点`
- `因果表达层`

而不是已经完全下沉到独立“状态概念树 / 动态表达模块 / 二次特征统一模块”。


## 8. 一句话总括

当前代码里：

- `抽象状态` 是“具体状态节点在实例因果表达阶段被区间化、方向化、短语化”的结果
- `抽象动态` 是“具体动态节点被压成业务语义类型，并等待统一语言表达层进一步收口”的结果

所以两者当前最准确的生成口径应写成：

`先生成实例，再生成抽象；其中状态的抽象已经能稳定落到因果表达层，动态的抽象目前主要先落到业务语义分类层。`
