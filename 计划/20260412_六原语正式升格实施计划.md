## 1. 目标

把当前任务管理所依赖的 6 个最小原语，从：

- `签名表 + 模块内 helper + 缺口评估默认视为具备`

推进到：

- `正式动作资产 + 可注册 + 可查询 + 可被缺口评估真实验证 + 后续可统一执行`

六原语固定为：

1. `确保单承载存在`
2. `读取单特征当前值`
3. `写入单特征当前值`
4. `写入单挂点指针`
5. `记录动作开始`
6. `记录动作结束`


## 2. 当前代码事实

当前已经具备的基础：

- 六原语已进入 `本能方法类` 的正式资产表，并拥有正式 `本能方法ID -> 元信息 -> 方法首节点注册` 机制
- `任务管理任务模块` 已有六原语 `ID / 签名表 / 位图摘要 / 缺口面`
- `任务管理最小原语模块` 已能提供统一执行壳，主写入口已切过去
- `自我线程` 与任务管理主链已能稳定消费这些正式治理最小动作

当前剩余的收口点：

1. 治理主摘要主链已完成 `原语=...` 首轮清理，但仍需继续统一 `派生摘要 / 派生能力` 的摘要层口径
2. 少量读取面虽然已走正式资产优先，但仍需继续做口径一致性审计，避免旧桥接表达从摘要侧漏回
3. `G5-G6` 的 helper 主路径已基本收口，剩余更偏向 `注册期兼容 helper` 与 `内部技术串` 的最后清理


## 3. 总体策略

不建议一步到位要求“六原语先能像普通本能方法一样完整调度再落地”。

最稳的方式是分两阶段：

### 阶段 A：先资产化

先解决：

- 有正式 ID
- 有正式元信息
- 有正式方法首节点
- 有正式调用壳
- 缺口评估能判断“正式资产是否存在”

这一阶段不强求：

- 所有调用点立刻改成统一调度器


### 阶段 B：再统一执行

等资产层稳定后，再把：

- `任务管理模块内直接 helper 调用`

逐步收口到：

- `六原语统一执行入口`

## 5. 分阶段实施

## P0 资产编号冻结

目标：

- 六原语有正式本能 ID

完成标准：

- `本能方法类.h` 中存在 6 个正式 ID
- `本能方法类.cpp` 中能查到对应元信息


## P1 注册与方法首节点出生

目标：

- 六原语和现有自我本能动作一样，能在方法树中出生正式首节点

完成标准：

- 调用默认注册后，六原语都能拿到 `动作句柄 = 本能函数ID`
- 能生成对应 `方法虚拟存在 / 条件场景 / 结果场景`


## P2 统一执行壳

目标：

- 把六原语从“散落 helper”包成正式调用壳

完成标准：

- 新增 `任务管理最小原语模块`
- 六原语至少各有一个稳定执行函数
- 旧 helper 暂时保留，但主调用点开始切换


## P3 缺口评估改真

目标：

- 缺口评估不再把六原语直接判成 `true`

完成标准：

- `评估任务管理本能能力缺口(...)` 读取正式资产存在性
- 未注册六原语时，评估结果能够真实落到：
  - `缺观察原语`
  - `缺写回原语`
  - `缺挂点原语`
  - `缺动作边界`


## P4 主调用点切换

目标：

- 任务管理主链改用正式六原语执行壳

完成标准：

- 固定派生写回、动作边界、关键挂点写回不再直接依赖散落 helper
- 任务管理本能缺口与真实执行壳口径一致


## P5 收口与验收

目标：

- 六原语完成真正意义上的正式升格

完成标准：

- 代码、缺口评估、学习承接、恢复快照、验收摘要、面板显示的原语口径一致
- “工程底层具备”和“正式资产具备”不再混用
- 治理主摘要中不再残留 `原语=...` 这类混合表达
- 六原语相关旧 helper 路径已有覆盖账本或禁旧路径审计结论
- `G5-G6` 的运行态旧 helper 不再回流主路径


## 6. 风险与规避

## 6.1 风险：一开始就追求统一调度器，改动面过大

规避：

- 先资产化，再统一执行


## 6.2 风险：六原语语义和现有 helper 语义发生偏移

规避：

- 第一轮执行壳只包裹旧逻辑，不改行为


## 6.3 风险：缺口评估被一次性改坏

规避：

- 先把 `true` 改成“查正式资产存在”
- 不立即把“本轮证据具备”也混进来


## 6.4 风险：与宿主迁移工程互相打架

规避：

- 六原语正式化优先关注“动作层”
- 宿主迁移继续按既有 `P0-P5` 计划推进
- 第一轮不强行同时完成真实宿主迁移

## 9. 本轮建议直接执行的最小切口

如果下一步立刻开工，推荐切口固定为：

1. 先给六原语分配正式 `本能方法ID`
