## 1. 目标

A4 的目标是让自我在发现自身结构或能力缺失时，形成这条边界清楚的链路：

```text
自检线程发现缺失
→ 报送自我线程
→ 自我线程按治理时机决策
→ 运行期记录 / 休眠期修复需求 / 待机保护 / 非正常逻辑错误报错
```

自检线程只负责发现缺失和报送事实，不生成修复需求，不直接写需求树。

因此 A4 不能实现为旧错误方案：

```text
自检线程发现什么 -> 自检线程直接改需求树或直接安排修复
```

必须由自我线程按缺失类型和修复时机分流。

## 2. 当前代码基础

已具备：

```text
1. 自检线程通过自检报告消息把缺口候选报送自我线程。
2. 自检候选消息已有报告型 / 需求型字段，但自检线程侧应只产生报告型。
3. 标准原子状态词典已存在：
   缺失 / 未就绪 / 已建立 / 已就绪。
4. 自检状态特征入口已存在：
   结构存在状态、需求根存在状态、任务根存在状态、方法根存在状态、
   内部世界存在状态、核心状态可读写状态、任务虚拟存在绑定状态。
5. 自我类::创建自检I64原子状态 已能显式创建状态节点。
```

本轮代码边界已补：

```text
1. 自检候选消息默认不建议生成需求。
2. 自检线程构造缺口候选默认报告型。
3. 自我线程收到自检报告中的需求型 / 建议生成需求候选时，记录错误日志并拒绝入树。
4. 自检报告落账不再把本轮标记为“有派生需求待实例化”。
```

## 3. 分流规则

### 3.1 自检报告型

适用：

```text
统计镜像不一致
可观察但不具备明确修复动作的差异
诊断信息不足的候选
运行期不允许立即修复的缺失
```

处理：

```text
写日志
进控制面板 / 自检摘要
不入需求树
等待自我线程后续治理窗口判断
```

### 3.2 自我线程修复决策型

适用：

```text
目标主体是基础信息节点或存在节点
当前状态和目标状态都明确
满足关系明确
修复动作边界明确
缺口属于可恢复业务缺口或显式自修复缺口
```

处理：

```text
自我线程接收自检报告；
自我线程判断缺口是否可恢复；
自我线程判断修复是否必须等待休眠期；
自我线程在允许时机内生成修复需求或进入保护态。
```

修复时机分层：

```text
运行期只记录：
    结构镜像不一致、统计差异、非阻塞缺口。

休眠期可修复：
    可能扰动任务 / 方法 / 状态索引的一致性修复；
    需要重建镜像、补账、重新扫描或批量整理的缺口。

立即报错 / 保护：
    当前自我、内部世界、核心状态读写等关键锚点不可用；
    请求创建结构根失败；
    任务建立完成后仍缺任务虚拟存在。
```

### 3.3 非正常逻辑错误

适用：

```text
当前自我实例缺失
自我内部世界缺失
请求创建需求根 / 任务根 / 方法根失败
任务建立流程完成后仍未绑定任务虚拟存在
目标主体不是基础信息节点，无法构造状态节点
自检线程越界提交需求型候选
```

处理：

```text
弹错 / 报错
记录错误日志
控制面板显示故障
拒绝自动入需求树
```

## 4. 已确认边界

```text
1. 自检线程不生成修复需求，只发现缺失并报送自我线程。
2. 需求根 / 任务根 / 方法根 是存在信息结构字段，默认可以为空；
   只有在请求创建或使用时创建失败，才属于非正常逻辑错误。
3. 任务虚拟存在应在建立任务时由程序建立；
   未建立任务时为空不是错误，任务建立完成后仍为空才是错误。
4. 有些修复只能在自我休眠期执行；运行期只能记录和等待修复窗口。
```

## 5. 后续实现计划

```text
1. 自我线程维护自检报告待处理队列。
2. 为每类缺失标注修复时机：运行期记录 / 休眠期修复 / 立即保护 / 非正常错误。
3. 自我进入休眠期时，只打开休眠期修复门控并统计允许休眠处理的报告。
4. 运行期不得因自检报告直接入需求树。
5. 控制面板显示：待休眠修复缺失数、最近被拒绝的越界需求化报告。
```

休眠期修复动作不在当前阶段实现。它不能凭空决定，必须等后续明确：

```text
1. 自检报告类型到治理动作的映射表。
2. 每类动作的目标主体、目标状态、当前状态和满足关系。
3. 哪些动作只能重扫 / 补镜像 / 补统计，哪些动作允许生成修复需求。
4. 执行动作时是否会扰动任务、方法、状态索引和当前主链。
5. 修复失败时的错误上报和保护策略。
```

## 5.1 第一版处置映射表

状态：已落地为“处置分类”，未开放自动修复动作。

```text
缺口类型值 1：方法结构
    处置：仅报告保留
    自动动作：无
    原因：方法结构缺口属于方法学习 / 补结构链路，不由自检报告直接修。

缺口类型值 3：方法运行账
    处置：仅报告保留
    自动动作：无
    原因：运行账不足属于经验样本不足，不由自检报告直接补账。

缺口类型值 4：方法虚拟存在
    处置：休眠期评估
    候选动作：待定义_方法虚拟存在一致性检查
    自动动作：未授权
    原因：该类可能涉及方法语义对象一致性，但具体补建动作需要单独确认。

缺口类型值 5：可用方法数量镜像
    处置：休眠期评估
    候选动作：重扫方法可用数量镜像
    自动动作：未授权
    原因：统计镜像差异适合休眠期重扫，不应需求化。

缺口类型值 6：状态清理阻塞
    处置：休眠期评估
    候选动作：重扫状态清理阻塞快照
    自动动作：未授权
    原因：清理阻塞可能扰动状态索引，只能在休眠期评估。

其他缺口类型：
    处置：待映射确认
    自动动作：未授权
```

实现边界：

```text
1. 映射表只决定报告处置分类，不生成修复需求。
2. “休眠期评估”不等于“休眠期执行修复”。
3. 需要映射确认的报告会进入待映射计数。
4. 控制面板显示待休眠评估、待映射确认、仅报告保留和最近报告处置。
```

## 5.2 待确认项

```text
1. 哪些自检报告类型允许在休眠期自动转为治理动作。
2. 哪些自检报告类型只能继续保留为报告，不允许自动修复。
3. 休眠期修复动作的第一批目标是否只做镜像 / 统计 / 索引类修复。
4. “核心状态可读写状态”目前作为验收 / 诊断状态存在，是否允许转为具体修复动作，需要先定义可执行动作。
5. 可用方法数量镜像不一致这类统计差异，是否应走“休眠期重扫镜像”而不是需求化。
```

在上述动作表确认前，自我线程只记录报告、统计休眠期门控，不生成修复需求或治理动作。
