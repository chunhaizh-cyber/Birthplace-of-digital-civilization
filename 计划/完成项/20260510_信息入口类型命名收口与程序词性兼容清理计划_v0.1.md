## 1. 背景

最新规范已经把原“程序词性”正式收口为：

```text
信息入口类型
```

新口径是：

```text
人类词性是语言数字化工具。
它负责把自然语言切分、标注、转成候选语义入口。

系统内部不依赖人类词性直接落树。
系统使用“信息入口类型”决定：
1. 对应的基础信息类型；
2. 默认存储树；
3. 默认根节点；
4. 是否需要上下文限定。

同一个词面可以有多个人类词性，也可以有多个信息入口类型。
裸词通常指向模板 / 概念入口。
带主体、场景、时间、任务上下文的词组，才指向实例入口。
```

当前代码和计划在上一轮已经完成 P1-P5 第一轮落地，但命名仍使用：

```cpp
枚举_程序词性
结构_程序词性定义
添加程序词性词(...)
选择程序词性默认根(...)
```

这些语义上已经是“信息入口类型”，需要统一命名，避免后续继续把它误读为人类语言词性。

## 2. 当前检查结论

### 已符合最新精神

```text
1. 已经存在权威规范《信息入口类型与信息存储树路由规范20260510》。
2. 旧《程序词性与信息存储树路由规范20260510》已经声明为历史入口。
3. 《基础信息语素信息关系整合规范20260509》已在开头提示“词性”拆分为人类词性和信息入口类型。
4. 当前实现已经不再只靠人类词性决定核心概念、任务 / 方法虚拟存在、方法动作名等入口。
5. 当前代码已经有信息入口类型所需的定义表、存储树、默认根路由雏形。
```

### 仍需规范化修订

```text
1. 当前主计划仍命名为“程序词性与信息存储树路由落地计划”。
2. 计划索引仍把当前主线描述为“程序词性决定基础信息类型、存储树和默认根”。
3. 代码类型名仍是 `枚举_程序词性`，容易与 `枚举_词性` 混淆。
4. 语素接口仍是 `添加程序词性词`、`查找程序词性节点`、`确保程序词性对应基础信息`。
5. `词性主信息类` 中字段仍叫 `程序词性`，应改为 `信息入口类型`。
6. 定义结构还缺少显式“是否需要上下文限定”的字段。
7. 裸词默认模板 / 概念入口、带上下文词组才实例入口的规则还没有成为代码硬约束。
```

---

## 3. 修订原则

```text
1. 权威名称使用“信息入口类型”。
2. “程序词性”只作为旧名、迁移名、兼容名，不再作为新增接口命名。
3. 人类词性只生成候选，不直接选择主信息类型或存储树。
4. 信息入口类型定义表是主信息类型、存储树、默认根、上下文需求的唯一权威。
5. 裸词入口优先落到模板 / 概念。
6. 实例入口必须有主体、场景、时间、任务、方法等上下文之一。
7. 旧接口可以短期保留为包装层，但新调用点必须使用信息入口类型命名。
```

| 风险 | 处理 |
| --- | --- |
| 大规模重命名造成编译抖动 | 先加新名并迁移调用点，再删旧名 |
| 旧计划仍引用“程序词性” | 完成项不逐份改历史正文，只在索引和新规范声明旧名含义 |
| 裸词实例入口误创建实例 | 信息入口定义增加上下文约束，实例入口缺上下文直接拒绝 |
| 控制面板展示仍叫程序词性 | P7 专门更新展示字段 |
| `词性节点类` 类名本身会误导 | 第一阶段不改类名，文档中固定解释为语素入口节点；后续可另起“语素入口节点重命名计划” |

---

## 7. 完成定义

```text
1. 权威规范和当前主计划都使用“信息入口类型”。
2. 代码主类型名迁移为 `枚举_信息入口类型`。
3. 定义表迁移为 `结构_信息入口定义`。
4. 词性主信息类保存 `人类词性` 和 `信息入口类型`。
5. 信息入口定义能返回主信息类型、存储树、默认根和上下文约束。
6. 裸词 / 上下文词组分别落到模板入口 / 实例入口。
7. 控制面板可观察信息入口类型。
8. 旧“程序词性”兼容包装被删除或仅保留历史说明。
9. Debug x64 构建通过。
```

---

## 8. 最终口径

```text
人类词性负责语言数字化；
信息入口类型负责系统语义入口；
基础信息类型负责节点结构；
存储树路由负责节点落点；
上下文决定模板入口还是实例入口。
```

一句话：

```text
词性不是世界树路由器；信息入口类型才是。
```
