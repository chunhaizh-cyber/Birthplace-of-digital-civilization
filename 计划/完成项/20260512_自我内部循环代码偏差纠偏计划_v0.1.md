## 1. 计划目标

按最新 `自我内部循环实现规范20260512.md`，把当前代码从“线程裸写任务域”的过渡形态，纠偏为：

```text
任务筹办 / 任务执行：
    任务管理生命周期普通函数，只生成过程结果和任务层回执。

同步筹办回执到任务 / 同步执行回执到任务：
    任务回执提交本能方法，写任务虚拟存在的回执镜像。

提交任务状态变化：
    任务状态提交本能方法，写任务状态并生成任务状态方法动作动态。
```

最终要求：

```text
线程只调度方法。
本能方法提交特征变化。
动作动态必须绑定具体方法。
任务状态、任务回执、需求满足不允许裸写成最终事实。
任务筹办 / 任务执行不作为本能方法注册、候选选择或动作动态来源。
```

## 3. 当前偏差基线

### D1. 提交类本能方法缺失

当前代码没有真正落地：

```text
同步筹办回执到任务
同步执行回执到任务
提交任务状态变化
```

这些只在规范、注释或动态词中出现，尚不是可被本能动作系统调用的本能方法。

### D2. 任务管理界面线程仍裸写任务状态

当前任务管理界面线程中仍存在直接提交：

```cpp
任务类::应用任务状态(...)
任务类::写入任务状态(...)
任务类::写入任务当前阶段(...)
```

对应位置：

```text
任务模块.管理界面线程.impl.cpp::私有_应用工作结果到任务壳
任务模块.管理界面线程.impl.cpp::提交任务状态推进请求
```

这与“任务状态变化必须由提交任务状态变化本能方法完成”冲突。

### D3. 任务状态动作动态不是方法动作动态

当前 `私有_输出任务状态动作动态` 使用普通动态创建，再手写：

```text
来源动作名 = 任务状态提交
```

这没有绑定真实方法节点，不满足：

```text
动作动态必须绑定具体方法。
```

### D4. 回执同步仍是普通函数

当前 `私有_同步工作回执方法到任务壳` 直接写：

```text
任务方法候选集
当前方法
当前方法虚拟存在
任务选中自我能力值
```

它应迁移为：

```text
同步筹办回执到任务 本能方法
同步执行回执到任务 本能方法
```

### D5. 工作线程生命周期函数入口曾与本能方法口径冲突

历史偏差曾认为工作线程应绕到：

```text
自我_任务筹办
自我_任务执行
```

最新口径已修订：

```text
任务模块::筹办::筹办任务(...)
执行任务(...)
```

才是任务管理生命周期普通函数入口。
工作线程可以调用它们，但不得直接写任务状态。

### D6. 旧执行桥仍直接写任务状态

`任务模块.执行.ixx` 中仍存在执行桥直写：

```cpp
任务类::写入任务状态(...)
```

只要工作线程还调用旧执行桥，这条路径就会绕过任务状态提交本能方法。

### D7. 需求满足回写仍是自我线程裸写

当前自我线程直接写：

```text
来源需求.被需求当前状态
需求动作动态
```

需求满足应改为需求域本能方法，例如：

```text
回写需求满足
```

自我线程只决定调用，不直接作为动作来源。

### D8. 任务完成判断仍有过渡兜底

`任务执行` 当前在缺少显式满足证据时，会用：

```text
本能动作运行成功
```

作为任务完成建议。该逻辑只能作为过渡闭环保底，不能作为最终内部循环成立证据。

## 4. 纠偏原则

```text
1. 不新增额外语义结构体承载任务结果，优先用运行存在、输出结果场景和方法虚拟存在三通道。
2. 不恢复旧“任务执行一步”步骤壳。
3. 不让工作线程直接写任务状态、任务阶段、任务虚拟存在最终事实。
4. 不让界面线程裸生成任务状态动作动态。
5. 不把输出结果场景当任务回执通道。
6. 不把方法运行成功直接等同任务完成。
7. 任何能生成方法动作动态的变化，必须对应一个明确本能方法。
```
