# 20260418 模糊型双落点细则前推为治理桥主裁决

## 本步目标

继续削弱 `自我线程` 在主派发后半段对“逐类路由细则”的本地补判，让 `自我_主循环治理模块` 开始原生消费更多覆盖后的细则输入，并直接对部分双落点/软分流场景给出主裁决。

## 本步完成内容

1. 在 `自我_主循环治理模块` 的 `结构_一步治理输入_最小` 中继续补强主派发细则输入：
   - `主派发细则允许继续提交级`
   - `主派发细则允许重筹办级`
   - `主派发细则允许待机级`
   - `主派发细则允许收束级`
   - `主派发逐类路由摘要`
   - `主派发逐类路由细则摘要`
   - `主派发多证据覆盖摘要`
   - `主派发多证据联合裁决摘要`
2. 在 `结构_一步治理结果_最小` 中新增：
   - `已消费主派发逐类路由细则`
   - `主派发逐类路由摘要`
   - `主派发逐类路由细则摘要`
3. 治理桥在 `当前去向 == 执行方法` 的主分支下，继续前推并主裁决这几类细则：
   - 仅允许 `重筹办级` 时，直接回 `继续治理`
   - 仅允许 `待机级` 时，直接落 `待机`
   - 仅允许 `收束级` 时，直接改入 `收束链`
4. 快照重读链也补上了同一批逐类路由细则字段，保证运行态与快照态对“桥层到底看见了什么证据”保持同口径。
5. 线程侧主派发输入装配继续瘦身：
   - 预判阶段开始正式喂入细则允许级与细则摘要
   - 最终治理输入阶段也统一喂入覆盖后的细则允许级、细则摘要、多证据覆盖摘要、多证据联合裁决摘要

- 治理桥不再只消费“主派发路由总摘要”，而是开始直接理解更细粒度的逐类路由细则。
- 对于 `只许重筹办 / 只许待机 / 只许收束` 这类细则，桥层已经能直接主裁决，不需要线程后段再做一层本地解释。
- 主派发运行态与桥层快照重读链已经喂入同一份细则证据面，后续继续前推模糊型双落点细则时，观察面不会先分叉。
