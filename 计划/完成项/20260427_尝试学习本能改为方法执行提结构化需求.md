# 2026-04-27 尝试学习本能改为方法执行提结构化需求

## 背景

最新任务执行口径中，任务只负责 `筹办 -> 执行` 两步。真正产生缺口需求的是当前步骤中的方法，而不是任务本身。

学习任务的目标是让待学习方法的虚拟存在满足：

```text
方法是否可用 >= 1
```

任务筹办首次命中学习任务时，只负责选择并绑定兜底方法 `学习_尝试学习`。执行阶段由 `学习_尝试学习` 根据待学习方法的真实结构返回结果或缺口。

## 已完成

- `学习_尝试学习` 不再从全局学习调度队列选择任务。
- `学习_尝试学习` 改为读取当前学习任务的 `等待学习方法首节点` 作为待学习方法。
- 当待学习方法缺动作骨架时，返回 `方法动作骨架数量: 0 -> 1` 的结构化需求。
- 当待学习方法缺条件节点时，返回 `方法条件节点数量: 0 -> 1` 的结构化需求。
- 当待学习方法缺结果节点时，返回 `方法结果节点数量: 0 -> 1` 的结构化需求。
- 当动作、条件、结果结构都具备时，写回待学习方法虚拟存在的 `方法是否可用=1`。
- 补齐执行桥中的 `方法动作骨架数量`、`方法结果节点数量` 特征和对应缺口返回函数。
