# 内部闭环本能函数九件套实现计划 v0.1

日期：2026-05-10

## 1. 定位

本计划用于把“能让自我内部闭环运行的本能函数”收口成最小生理反射集合。

内部闭环不是靠某一个学习函数完成，而是由以下链条协同：

```text
状态 / 需求判断
→ 方法查找
→ 参数配对
→ 方法补齐学习
→ 动作执行
→ 需求满足检查
→ 价值回写
→ 下一轮判断
```

当前已注册到本能动作执行闭环的 7 个函数：

```text
自我_查找特征类型
自我_配对参数
自我_改变特征值
自我_练习
自我_尝试学习
自我_定向学习
自我_检查需求是否满足
```

内部闭环还需要两个价值推进本能：

```text
自我_安全服务闭环安全推进
自我_安全服务闭环服务推进
```

这两个属于“自我价值推进本能”，可以先完成自身方法环境和条件节点匹配，不急于并入同一执行注册表。是否注册为任务可调度本能方法，后续按任务筹办 / 价值回写链路确认。

## 2. 总原则

所有本能函数统一遵循四段模式：

```text
第 1 段：确保自身方法环境。
第 2 段：用自身条件节点绑定输入。
第 3 段：执行本能核心逻辑。
第 4 段：写出结果、更新方法账、同步运行事实。
```

结构职责固定为：

```text
方法头节点 = 大范围索引、候选召回、身份入口。
条件节点 = 执行时参数设置、参数匹配、前置条件判断依据。
结果节点 = 输出槽、结果状态、完成判断依据。
方法虚拟存在 = 方法语义记忆、统计、学习产物索引。
本能函数 = 本能方法的 C++ 代码体。
```

禁止：

```text
1. 只因方法头存在就进入核心逻辑。
2. 直接把输入包当作执行规格权威。
3. 绕过条件节点读取参数。
4. 学习函数直接创建需求树节点。
5. 恢复任务筹办 / 任务执行一步为本能方法。
```

## 3. 和现有计划的先后关系

本计划排在以下工作之后：

```text
1. 本能方法自身环境与专属规格确认计划。
2. 方法能力签名细化。
3. 本能方法条件节点执行匹配。
4. 需求树生长本能转换规范。
```

本计划排在以下工作之前：

```text
1. 学习需求正式入树。
2. 外界探索需求生成。
3. 更复杂的候选方法排序。
4. 参数配对按抽象特征层级和类型覆盖深化。
```

原因：

```text
学习需求和探索需求必须建立在内部闭环本能稳定可诊断的基础上。
如果方法查找、参数配对、补齐学习、改变特征值、需求满足检查和价值推进仍未统一执行口径，
学习需求入树会把方法缺口、输入缺口和价值回写缺口混成一团。
```

```text
2. 配对参数输出与任务筹办的“参数不可配 / 条件不足”回执进一步对齐。
```

```text
3. 自我_改变特征值

```

作用：

```text
改得动特征；

```

当前状态：

```text
已注册。
```

```text

9. 自我_安全服务闭环服务推进
```

作用：

```text
让安全值 / 服务值真实变化；
```

```text
6. 是否注册到同一执行闭环，单独在 P4 决策。
```

## 5. 分阶段实施计划

### P0：现状复核与边界冻结

目标：

```text
确认 9 个本能函数的注册状态、执行入口、当前四段模式完成度。
```

动作：

```text
1. 复核 `注册本能函数执行闭环`。
2. 复核 7 个已注册函数是否均走统一自身环境和条件节点匹配。
3. 复核安全 / 服务推进函数是否仍为旧式入口。
4. 记录哪些函数不允许恢复为本能方法：任务筹办、任务执行一步、自我线程主循环、任务管理界面线程承接、任务管理工作线程调度。
```

验收：

```text
形成九件套状态矩阵。
```

### P1：方法补齐类别枚举化

目标：

```text
删除学习函数中的裸数字 `1 / 2 / 3`。
```

动作：

```cpp
enum class 枚举_方法补齐类别 : I64 {
    未定义 = 0,
    条件结果对补齐 = 1,
    候选动作补齐 = 2,
    实参与验证补齐 = 3,
};
```

替换位置：

```text
尝试学习：条件结果对补齐。
定向学习：候选动作补齐。
练习：实参与验证补齐。
条件结果对 / 运行存在 / 输出结果场景 / 方法虚拟存在中所有方法补齐类别写入点。
```

验收：

```text
代码中不再出现语义不明的 `方法补齐类别` 裸值写入。
```

### P2：安全 / 服务推进本能接入统一自身环境

目标：

```text
把两个价值推进本能从旧式入口迁到四段模式。
```

动作：

```text
1. 新增 `确认安全服务闭环安全推进规格`。
2. 新增 `确认安全服务闭环服务推进规格`。
3. 在 `确认本能方法专属规格` 中分发这两个 ID。
4. 两个函数开头改为 `确保本能方法自身环境`。
5. 用条件节点声明并匹配：
   - 安全推进：安全值运行参数。
   - 服务推进：服务值运行参数。
6. 保留其“记录运行事实 + 由自我类应用函数改变安全 / 服务值”的分层。
```

验收：

```text
安全 / 服务推进函数不再手写旧式方法环境确保。
无输入场景时仍可作为价值推进本能形成运行事实。
有输入场景时必须通过条件节点匹配。
```

### P3：价值推进结果事实与下一轮判断衔接

目标：

```text
让安全 / 服务推进本能结果可被自我内部循环诊断和下一轮需求选择稳定读取。
```

动作：

```text
1. 检查自我类应用函数：
   - 应用安全低位回升
   - 应用安全高位回落
   - 应用风险安全回归
   - 应用定时衰减服务值
   - 应用按双值重算服务衰减步长
   - 应用按服务值重算安全增加步长
   - 应用按服务值重算安全减少步长
2. 确认这些函数属于自我类对外功能函数 / 先天调控函数，不塞进任务工作线程。
3. 明确安全 / 服务推进本能只负责形成运行事实和触发应用函数，不替代需求结算。
4. 更新内部循环摘要或诊断字段，区分：
   - 方法执行事实
   - 自我价值应用事实
   - 需求结算事实
```

验收：

```text
服务值或安全值变化后，下一轮压力评估发生变化。
价值推进事实不会被误判为方法补齐学习产物。
```

### P4：是否注册安全 / 服务推进本能的决策

目标：

```text
决定安全 / 服务推进函数是否并入 `注册本能函数执行闭环`。
```

判断规则：

```text
可以注册的条件：
1. 已完成统一自身环境。
2. 已完成条件节点匹配。
3. 已明确输入场景为空时的合法执行边界。
4. 已明确输出运行事实和自我值应用之间的关系。
5. 不会绕过任务结果回写和需求结算。

暂不注册的条件：
1. 仍只作为自我类调控函数直接调用。
2. 缺少任务调用场景。
3. 注册后会和价值回写链重复记账。
```

验收：

```text
计划正文记录决策结果。
如果注册，构建和三条摘要命令通过。
如果不注册，保留为“自我价值推进本能直接入口”，并在索引中标明后续触发条件。
```

## 6. 当前待确认项

### 待确认 1：安全 / 服务推进是否进入同一注册表

当前建议：

```text
先迁移到统一自身环境和条件节点匹配；
暂不急于注册到 `注册本能函数执行闭环`。
```

原因：

```text
它们更像自我价值推进本能，可能由自我类调控函数和内部循环状态机直接触发。
若直接注册为任务可调度方法，必须先确认不会和任务反馈价值结算重复记账。
```

### 待确认 2：价值推进和需求结算的分账

必须区分：

```text
安全 / 服务推进本能改变自我状态；
任务反馈结算改变来源需求账本；
两者不能重复兑换同一份安全 / 服务价值。
```

### 待确认 3：九件套摘要入口位置

可选：

```text
1. 并入 `--self-loop-summary`。
2. 新增独立 `--instinct-loop-summary`。
```

建议：

```text
先并入 `--self-loop-summary`，若字段过多再拆独立命令。
```

## 7. 最终目标

完成后，内部闭环本能函数应达到：

```text
方法可找；
参数可配；
方法可补；
动作可执行；
结果可验；
自我状态可变；
需求可结算；
下一轮可重判。
```

一句话：

```text
内部闭环的本能函数不是学习函数集合，而是一组自我生理反射。
它们让自我能从状态长出需求，从需求形成行动，从行动改变自己，再用改变后的自己判断下一轮。
```
