## 1. 背景

`方法主信息类` 已经从旧的 `方法信息基类` 改为：

```cpp
class 方法主信息类 {
public:
    结构_方法节点公共信息 公共{};
    方法角色信息 角色{};
};
```

其中 `方法角色信息` 只允许一个角色有效：

```cpp
std::variant<
    std::monostate,
    结构_方法首节点信息,
    结构_方法条件节点信息,
    结构_方法结果节点信息>
```

这次改造已经物理删除了旧基类和三种派生提示类，但代码里还残留一些“旧大字段思路”的调用方式。刚刚启动失败就是其中一个后果：默认本能方法已经创建了条件子节点，但旧判断仍然从方法首节点读取 `条件节点信息().条件判定索引`，导致默认本能方法被误判为未注册。

因此现在需要把影响面系统收口，避免后续再出现“编译通过但语义写到空角色”的问题。

## 2. 当前已明确的影响

### 2.1 节点角色语义已经改变

现在一个方法节点只有一个有效角色：

```text
方法首节点：
    动作骨架
    方法虚拟存在
    来源 / 作用域 / 来源任务
    能力签名

方法条件节点：
    条件场景
    条件主键
    条件面 / 条件判定索引
    时间窗口

方法结果节点：
    结果场景
    结果主键
    结果包
    结果初始状态
```

旧语义中“首节点也能读条件场景、结果场景、结果初始状态”的做法已经不成立。

### 2.2 现有兼容访问器存在风险

`方法主信息类` 目前保留了：

```cpp
首节点信息()
条件节点信息()
结果节点信息()
```

当角色不匹配时，这些函数会返回静态空对象。这能降低迁移期的崩溃风险，但也会掩盖错误：

```text
从方法首节点写 条件节点信息()
    ↓
写入的是静态空对象
    ↓
编译通过
    ↓
真实方法树没有变化
```

这类代码必须逐步清掉，最终改成显式 API 或指针式访问。

### 2.3 方法类仍有旧首节点混合字段路径

当前重点残留在 `方法类.cpp`：

```text
取或创建_方法条件场景
取或创建_方法结果场景
初始化方法虚拟存在信息
创建条件节点
创建结果节点
补齐本能方法首节点
```

其中部分逻辑仍尝试从方法首节点读取或写入：

```text
条件节点信息().条件场景
条件节点信息().条件判定索引
结果节点信息().结果场景
结果节点信息().结果初始状态
```

这些要改成：

```text
首节点只读首节点信息；
条件场景从条件子节点或方法虚拟存在派生；
结果场景从结果子节点或方法虚拟存在派生；
条件/结果补齐只写对应子节点。
```

### 2.4 方法查找规范已经从单结果槽升级为集合覆盖

旧规范仍有大量表述：

```text
方法结果 = 主结果特征类型 + 结果方向索引
```

新规范应该固定为：

```text
方法能力签名 = 条件面 -> 结果包
方法查找 = 需求结果集 ⊆ 方法结果包
```

旧 `结果能力集` 已删除，不再作为索引或迁移入口。方法结果能力只由 `能力签名集.结果包` 表达。

### 2.5 文档和计划索引仍有旧口径

已发现旧口径分布：

```text
README.md
说明书/方法类说明.md
说明书/本能方法类说明.md
若干完成项历史计划
```

完成项历史计划不回改正文，但当前规范文档必须更新，避免后续开发继续按旧字段理解。

## 3. 修订原则

1. 不恢复 `方法信息基类`。
2. 不把条件字段、结果字段重新塞回方法首节点。
3. 不再把 `首节点信息() / 条件节点信息() / 结果节点信息()` 当成通用读写入口。
4. 首节点只能直接承载“方法是什么、怎么执行、能产生什么结果能力”。
5. 条件节点只承载“何时可用”。
6. 结果节点只承载“发生了什么结果、结果证据是什么”。
7. 方法查找只按结果集合覆盖命中，不按名称、动作句柄、本能 ID 命中。
8. 动作句柄只用于执行，不用于查找方法。
9. 找不到就是找不到，暴露闭环缺口或结果声明缺口。
10. 内部方法补齐闭环不生成定向学习需求；尝试学习应从内部因果样本直接形成候选方法。

## 4. 修订目标

### 4.1 主信息访问边界收口

新增或明确以下访问方式：

```cpp
取首节点信息()
取条件节点信息()
取结果节点信息()
```

指针返回用于安全读取。

新增方法树级辅助函数：

```text
取首个条件子节点
取首个结果子节点
查找条件子节点_按条件主键
查找结果子节点_按结果主键
方法首节点有可执行条件子节点
方法首节点有结果子节点
收集方法首节点条件索引
收集方法首节点能力签名结果项
```

这些辅助函数放在 `方法类` 或局部私有函数中，避免外部模块遍历细节散开。

### 4.2 删除静态空对象写入风险

分阶段处理：

```text
第一阶段：
    保留 const 兼容读取，减少显示和摘要路径破坏。

第二阶段：
    删除或限制非 const 的 首节点信息()/条件节点信息()/结果节点信息() 引用返回。

第三阶段：
    对写入路径强制使用：
        取首节点信息()
        取条件节点信息()
        取结果节点信息()
        构造首节点()
        构造条件节点()
        构造结果节点()
```

最终不允许角色不匹配时写入静态空对象。

### 4.3 方法类职责重新收口

`方法类` 需要重构，但不需要推倒整棵方法树。

修订方向：

```text
注册方法首节点：
    只创建首节点角色。

创建条件节点：
    只创建条件节点角色。

创建结果节点：
    只创建结果节点角色。

补齐本能方法首节点：
    写首节点动作和能力；
    创建条件子节点；
    创建结果子节点；
    不把条件/结果写回首节点。

初始化方法虚拟存在信息：
    只写首节点方法虚拟存在和方法级特征；
    条件/结果场景由对应子节点负责。
```

### 4.4 能力签名成为权威结构

当前只保留一个能力结构：

```text
能力面.能力签名集
```

目标关系应固定为：

```text
能力签名集 = 权威能力表达
```

后续所有本能方法结果声明应先写为：

```text
条件面 -> 结果包
```

不再派生旧 `结果能力集`。

### 4.5 规范文档修订

修订：

```text
README.md
说明书/方法类说明.md
说明书/本能方法类说明.md
必要时补充 方法主信息类说明.md
```

删除或改写旧说法：

```text
主结果特征类型
结果方向索引
方法首节点条件判定索引
方法首节点结果场景
```

改为：

```text
条件集合
结果集合
能力签名
结果包覆盖需求结果集
条件子节点 / 结果子节点
```

## 5. 执行步骤

### 阶段一：影响面清点与断言补强

1. 扫描所有 `条件节点信息()` 和 `结果节点信息()` 调用点。
2. 标记哪些调用对象一定是条件/结果节点，哪些调用对象可能是方法首节点。
3. 对方法首节点误读条件/结果角色的路径列清单。
4. 为启动链、本能注册链、摘要链加最小自检输出。
5. 保持当前启动可运行。

验收：

```powershell
鱼巢.exe --summary
鱼巢.exe
```

均返回 `EXIT_CODE=0`。

### 阶段二：方法类内部角色访问修订

1. 重构 `取或创建_方法条件场景`：
   - 输入首节点时，优先从条件子节点找场景。
   - 找不到时基于方法虚拟存在创建场景。
   - 不写首节点的条件角色。
2. 重构 `取或创建_方法结果场景`：
   - 输入首节点时，优先从结果子节点找场景。
   - 找不到时基于方法虚拟存在创建场景。
   - 不写首节点的结果角色。
3. 重构 `创建条件节点`：
   - 只写条件节点角色。
   - 不复制动作名、动作句柄、来源、作用域到条件节点。
   - 父方法引用如仍需要，另定挂接位置。
4. 重构 `创建结果节点`：
   - 只写结果节点角色。
   - 不复制动作名、动作句柄、来源、作用域到结果节点。
   - 结果事实写入结果节点结果包；方法查找能力只写首节点能力签名。
5. 重构 `补齐本能方法首节点`：
   - 首节点写动作和能力签名。
   - 条件写条件子节点。
   - 结果写结果子节点。
   - 可执行判断读取条件子节点。

验收：

```text

方法树结构缺口为 0。
能力包缺失为 0。
条件结果对缺失为 0。
```

### 阶段三：外部模块访问收口

重点模块：

```text

自我动作实现模块.ixx

因果类.cpp

```

收口方式：

```text

    只对首节点取首节点信息。

读取条件：
    从条件子节点或方法类辅助函数读取。

读取结果：
    从结果子节点或方法类辅助函数读取。

读取结果能力：
    从首节点能力签名的结果包读取。
```

验收：

```text

控制面板摘要仍能显示方法结构。

```

### 阶段四：能力签名权威化

1. 为本能结果声明生成 `结构_方法能力签名`。
2. 练习声明多个结果项：
   - 方法稳定状态更稳定
   - 方法运行账完整性更完整
   - 方法是否可用趋向可用
   - 方法成熟度阶段上升
   - 方法新用途发现状态形成候选
3. 尝试学习声明多个结果项：
   - 候选方法存在状态从无到有
   - 方法动作骨架状态已形成
   - 方法条件结果配对状态已配对
   - 方法结构是否完整更完整
4. 定向学习声明结果项，但条件限定外界信息上下文。
5. 删除 `结构_方法结果能力` 和 `结果能力集`。
6. 查找方法从 `能力签名.结果包` 做集合覆盖。

验收：

```text

多需求只在同一个结果包覆盖全部必需项时命中。
改变特征值作为根写入原语不被普通查找命中。

```

### 阶段五：删除兼容访问器或降级为只读

   - `首节点信息()`
   - `条件节点信息()`
   - `结果节点信息()`
2. 或将其改为只用于断言明确的角色访问。
3. 写入路径全部改成：
   - `取首节点信息()`
   - `取条件节点信息()`
   - `取结果节点信息()`
   - 构造函数返回的明确角色对象
4. 角色不匹配时不返回可写静态对象。
