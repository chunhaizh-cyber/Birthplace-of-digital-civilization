# 20260508 子任务完成触发父任务重筹办与活动子需求判定计划 v0.1

## 1. 背景

当前需求树已经收束为：

```text
需求 / 任务单目标
多目标由子需求 / 子任务展开
方法执行可产生多结果
```

但父任务和子任务回流之间还需要进一步固定：

```text
子任务完成不是父任务完成。
子任务完成是父任务重新筹办的触发器。
```

父任务每次被子任务唤醒后，都必须重新走一遍：

```text
筹办 -> 执行判定
```

这个唤醒按单个子需求 / 子任务计算。任意一个活动子需求满足、截止、取消或收束后，父任务都进入下一轮筹办，不等待同层子需求整批完成。

## 2. 目标

1. 子任务完成后，父任务进入 `待重筹办 + 筹办阶段`。
2. 父任务重筹办时先检查父需求是否已满足。
3. 父任务不再用 `子节点数量 > 0` 判断是否有阻塞子需求。
4. 新增或统一 `存在活动直接子需求` 判断。
5. 没有活动子需求时，父任务回到自身目标重新查找方法。
6. 方法不可执行时，生成下一轮可执行性原子子需求。
7. 可执行性子需求不得复制父需求目标。
8. 执行桥缺失事实必须映射成明确原子需求。
9. 添加子需求前必须先检查候选子需求目标是否与父需求自身目标重叠。

## 3. 非目标

1. 不恢复旧任务步骤节点和预测结果节点主链。
2. 不把任务管理任务重新做成独立长期结构。
3. 不让父任务直接因为子任务完成而结算。
4. 不引入持久化目标包。

## 4. 当前风险点

### 4.1 `子节点数量 > 0` 旧判断

需要扫描并替换以下逻辑：

```text
需求->子节点数量 > 0
```

在父任务筹办、目标选择、管理需求判断里，这个判断必须改为：

```text
存在活动直接子需求(需求)
```

历史子需求已经满足或截止时，不得阻止父任务进入第二轮筹办。

### 4.2 子任务完成回流不足

需要确认子任务完成后是否稳定执行：

```text
父任务状态 = 待重筹办
父任务阶段 = 筹办
父任务最近唤醒原因 = 子任务完成
```

如果目前只唤醒部分状态，需统一入口。

### 4.3 缺失事实过泛

执行桥返回缺失时，不能只返回：

```text
任务缺条件需自我线程
```

必须携带明确目标：

```text
候选方法存在状态
方法动作骨架状态
方法可执行入口数量
方法条件节点数量
方法结果节点数量
方法条件结果配对状态
参数配对状态
方法运行账完整性
```

### 4.4 父子同目标未被阻断

需求类已经有同父直接兄弟子需求目标重叠判断，但如果候选子需求直接复制了父需求目标，兄弟去重无法拦住：

```text
父需求 D：目标对象.X -> 目标状态
└── 子需求 D1：目标对象.X -> 目标状态
```

这不是可执行性治理，而是同目标下沉。它会让父任务每轮都在处理自己的复制品，破坏多轮筹办：

```text
子需求满足 -> 父任务重筹办 -> 回到父需求自身目标
```

所以新增子需求前必须先做：

```text
候选子需求目标 vs 父需求自身目标
```

若目标主体、目标特征类型和方向掩码重叠，则拒绝入树并记录派生需求逻辑错误。

## 5. 实施阶段

### 阶段一：规范和索引落地

1. 新增 `规范/子任务完成触发父任务重筹办规范20260508.md`。
2. 更新 `规范/方法规范20260425/README.md` 的两阶段任务说明。
3. 更新 `说明书/任务类说明.md` 的任务状态和重筹办口径。
4. 更新 `计划/计划索引.md`，登记本计划为当前主计划。

验收：

```text
规范中明确子任务完成不等于父任务完成。
规范中明确活动子需求替代子节点数量判断。
```

### 阶段二：活动子需求服务

1. 在需求服务或任务筹办侧新增统一 helper：

```cpp
bool 存在活动直接子需求(需求节点类* 父需求);
```

2. 活动子需求过滤：

```text
未满足
未截止
未取消
仍阻塞父任务执行
```

3. 统一直接子需求遍历，避免各模块复制链表遍历。

验收：

```text
rg -n "子节点数量 > 0|子节点数量" 任务/需求/自我线程活代码
```

所有用于管理/执行判定的 `子节点数量` 均已替换。

### 阶段三：父任务回流重筹办

1. 梳理子任务完成回流入口。
2. 找到父需求 / 父任务。
3. 如果父需求尚未满足，父任务写为：

```text
状态 = 待重筹办
阶段 = 筹办
最近唤醒原因 = 子任务完成
```

4. 如果父需求已经满足，父任务才能进入完成。
5. 子任务失败或无推进时，也触发父任务重筹办，但不得重复创建同目标子需求。
6. 任意一个活动子需求满足、截止、取消或收束后，都唤醒父任务重筹办一次。

验收：

```text
子任务完成后父任务不会直接完成。
父任务可重新进入筹办候选。
同层多个子需求时，不等待整批完成才重筹办。
```

### 阶段四：筹办目标选择修正

1. 父任务重筹办先刷新父需求当前状态。
2. 先检查父需求是否满足。
3. 再调用 `存在活动直接子需求`。
4. 有活动子需求时，推进活动子任务。
5. 无活动子需求时，任务目标回到父需求自身单目标：

```text
目标状态 = 父需求.需求状态
当前状态 = 父需求.被需求当前状态
```

验收：

```text
历史已满足子需求存在时，父任务仍可进入自身目标筹办。
```

### 阶段五：可执行性缺失映射

1. 梳理执行桥 / 筹办返回的缺失事实。
2. 建立缺失事实到原子需求目标的映射。
3. 自我线程创建子需求前检查：

```text
子需求目标不与父需求目标重叠
同父兄弟不重叠
```

4. 缺失事实生成的子需求目标主体应指向任务、方法、方法虚拟存在或参数绑定场景。

验收：

```text
缺动作骨架不会生成与父需求相同的业务目标。
缺条件结果对能生成方法条件结果配对状态需求。
```

### 阶段五补充：父子同目标硬阻断

1. 在需求树底层添加子需求入口增加父子目标重叠检查，作为全局兜底。
2. 在自我线程派生需求入树入口增加前置检查，命中时弹出 / 记录派生需求逻辑错误。
3. 判断键使用：

```text
目标主体主键
目标特征类型主键
方向掩码
```

4. 检查顺序固定：

```text
先查父子重叠，重叠则拒绝。
再查同父兄弟重叠，重叠则复用兄弟。
最后才新增子需求。
```

验收：

```text
父需求目标为 X 时，任何派生路径都不能新增同目标 X 子需求。
父子同目标时返回拒绝，而不是返回父节点或新增子节点。
```


## 6. 验收标准

1. 活代码不再用 `子节点数量 > 0` 判断父需求是否阻塞。
2. 子任务完成后父任务稳定进入待重筹办。
3. 父任务完成只由父需求目标满足确认。
4. 历史子需求不阻止父任务第二轮执行。
5. 可执行性子需求不复制父需求目标。
6. 缺失事实映射到明确原子目标。
7. 编译通过，摘要运行正常。
8. 任意单个子需求满足后，父任务都会进入待重筹办。
9. 父子同目标候选子需求会被拒绝入树。
