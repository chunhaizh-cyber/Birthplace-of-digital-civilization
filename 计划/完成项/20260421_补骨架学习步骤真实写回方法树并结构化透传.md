# 补骨架学习步骤真实写回方法树并结构化透传

## 本次完成

本次把预备学习步骤从“只推进学习账本状态”推进到了“真的修改待学习方法结构”。

具体完成了三件事：

1. 在 `D:\鱼巢\学习执行模块.cpp` 新增了 `私有_按学习步骤写回方法结构(...)`
2. 在学习执行结果中新增结构化布尔位：
   - `已写回学习方法结构`
   - `已补动作骨架`
   - `已补条件节点`
   - `已补结果节点`
3. 在任务管理学习推进结果中继续透传同一组布尔位，供上层后续消费

## 真正写回了什么

按当前学习步骤类型，补骨架步骤会真正改方法树：

- `学习步骤_候选动作生成`
  - 若待学习方法头还没有动作骨架，则从当前执行方法或兜底学习方法复制：
    - `动作名`
    - `动作句柄`
    - 必要时补 `来源/作用域`
- `学习步骤_共同条件提取`
  - 会补：
    - 方法头的 `条件场景`
    - `条件主键`
    - `条件场景模板主键`
    - `条件判定索引`
  - 然后再调用 `方法类::创建条件节点(...)` 真正生成/复用条件节点
- `学习步骤_结果预测`
  - 会补：
    - 方法头的 `主结果特征类型`
    - `结果场景`
    - `结果主键`
    - `结果摘要索引`
    - `结果初始状态`
  - 然后再调用 `方法类::创建结果节点(...)` 真正生成/复用结果节点

也就是说，这一步不再只是“说应该补什么”，而是会把缺的结构真落到待学习方法头和它的子节点上。

## 任务树侧同步

这次还顺手把学习子任务当前结果节点也对齐了：

- `来源方法首节点` 会回指当前待学习方法
- 如果本轮补出了方法结果节点，`对应方法结果节点` 也会挂上去

这样以后沿学习子任务树去看：

`学习子任务头 -> 学习步骤 -> 学习结果`

不仅能看见“当前执行到了哪一步”，还能直接看到“这一步产出的目标方法结构现在挂到了哪棵方法树上”。

## 为什么这一步重要

前一轮虽然已经让预备学习步骤能调度、能执行、能回到继续学习，
但如果它不真实修改方法树，下一轮任务管理重算时仍然会看到同样的缺口，
学习链容易在相同步骤上原地打转。

本次接完以后，下一轮任务管理在同步学习实体时，会直接读到已经补上的：

- 动作骨架
- 条件节点/条件侧
- 结果节点/结果侧

从而可以自然生成下一学习步骤，而不是重复上一学习步骤。
