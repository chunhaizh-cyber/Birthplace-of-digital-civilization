# 任务管理任务本体阶段步骤改由统一筹办函数显式生成

## 本次完成

这次把“给 `任务管理任务` 增加步骤”的职责，正式收回到了任务筹办兜底函数。

之前的实现是：

- 若干阶段 wrapper 一边执行治理，一边直接创建或同步 `任务管理任务` 本体步骤
- 文件里还留着一组旧的单步治理 helper
- 这会让“阶段判断”和“任务树生长”缠在一起

现在改成了两层：

1. 阶段 wrapper 只记录本轮经历过的本体阶段轨迹
2. 真正把步骤节点/结果节点落进 `任务管理任务` 树，统一由兜底函数完成

## 现在的主链

本体阶段枚举固定为：

1. `受理请求`
2. `宿主锚点确认`
3. `绑定宿主方法`
4. `可执行判断`
5. `允许执行判断`
6. `执行后段治理`
7. `学习承接与业务派生`
8. `写回结果`
9. `组装一步结果`

运行时流程变成：

1. `私有_执行筹办模块(...)`
2. `私有_执行可执行判断模块(...)`
3. `私有_执行允许执行判断模块(...)`
4. `私有_执行执行模块(...)`
5. `私有_执行反馈模块(...)`
6. `私有_为任务管理任务筹办阶段步骤链(...)`

前五步只负责：

- 执行本轮治理
- 记录 `本体阶段轨迹`

最后一步才负责：

- 读取 `本体阶段轨迹`
- 逐段创建或追加 `任务管理步骤_*`
- 为每个步骤补一个 `任务管理结果_*`
- 回写 `当前步骤节点`
- 回写 `最近实际结果节点`

这样后面再讨论“哪些阶段应该长新步骤、哪些阶段可以复用”，就只需要改这一处筹办兜底函数，不会再散在多个 wrapper 里。

## 现在的语义

现在 `任务管理任务` 本体树表达的是：

- 这轮治理依次经历了哪些本体阶段
- 每个阶段对应的结果是什么

也就是显式长成：

`任务管理任务头 -> 任务管理步骤_阶段A -> 任务管理结果_阶段A -> 任务管理步骤_阶段B -> 任务管理结果_阶段B -> ...`

这条链和宿主任务树、学习子任务树并列存在，但职责不同：

- 宿主任务树：记录宿主任务如何被推进
- 学习子任务树：记录学习步骤如何推进
- 任务管理任务树：记录任务管理本体本轮如何治理
