## 1. 目标

本计划把语素层从“人类词性直接推断基础信息类型”升级为：

```text
人类词性 -> 程序词性候选
程序词性 -> 主信息类型 + 存储树 + 默认根
```

最终形成：

```text
词面可以相同；
人类词性可以相同；
程序词性必须明确；
程序词性决定基础信息类型和存储树。
```

本轮不是重写语素系统，而是在保留现有 `词性节点类`、`词性主信息类`、`对应基础信息类型`、`对应基础信息` 的基础上，补一层系统内部真正使用的路由标签。

## 2. 当前代码基线

当前已经具备：

```text
1. 词节点保存词面。
2. 词性节点保存 `枚举_词性 词性`。
3. 词性节点保存 `对应基础信息类型` 和 `对应基础信息`。
4. `语素_推断词性对应基础信息类型` 可按人类词性推断默认主信息类型。
5. `语素_词性是否兼容基础信息类型` 已支持抽象特征 / 特征、关系等兼容。
6. 世界树已经有存在概念根、抽象特征根和核心存在概念初始化。
7. 任务信息节点、任务虚拟存在、方法信息节点、方法虚拟存在的边界已经基本稳定。
```

当前不足：

```text
1. 没有 `枚举_程序词性`。
2. 没有 `枚举_信息存储树`。
3. 没有程序词性定义表。
4. 词性主信息类尚未区分人类词性与程序词性。
5. 现有 `语素_推断词性对应基础信息类型` 仍容易被误用为最终判断。
6. 创建基础信息模板 / 实例时，默认根仍由调用点各自决定。
```

## 6. 风险与处理

| 风险 | 处理 |
| --- | --- |
| 同词面同人类词性旧查找只返回一个词性节点 | 新查找接口必须按程序词性区分；旧接口继续返回默认候选 |
| 当前 `枚举_主信息类型` 没有模板 / 实例细分 | 先由程序词性区分，主信息类型兼容映射；后续再决定是否拆枚举 |
| 任务信息节点不是基础信息节点 | 程序词性仍可描述路由，但对应基础信息引用要按节点类型实际能力处理 |
| 过早替换旧推断导致大面积编译风险 | 分阶段兼容，先新增入口，再迁移调用点 |
| 控制面板误把人类词性当最终分类 | 展示时明确区分“人类词性 / 程序词性 / 存储树” |

## 8. 最终口径

```text
人类词性负责语言解析候选；
程序词性负责系统语义入口和世界树路由。
```

一句话：

```text
以后创建信息节点时，先问“程序词性是什么”，再问“上下文在哪里”。
```
