## 1. 阶段目标

本计划只实现因果补完的第一层诊断能力：

```text
B0：内部循环结果形成因果候选记录。
B1：任务失败、派生、等待、回写缺失等情况被识别为因果缺口。
```

本阶段不生成学习需求，不生成外界探索需求，不直接写入因果模板。

## 2. 实现边界

允许：

```text
1. 从任务界面承接结果、推进事件、上行消息、状态回写、需求结算中形成因果候选快照。
2. 标记候选样本是否完整。
3. 标记是否存在因果缺口。
4. 将派生需求类型、执行失败、无结果样本等情况映射成最近缺口类型。
5. 控制面板展示最近因果补完诊断。
```

禁止：

```text
1. 直接生成学习需求。
2. 直接生成探索需求。
3. 绕过需求树原子目标闸门。
4. 在缺少内部学习不足判定和探索安全门控前对外探索。
```

## 3. B0 因果候选记录

候选来源：

```text
任务结果状态消息
任务推进事件
状态回写快照
需求结算快照
派生需求消息
冻结治理消息中的派生需求
```

候选判断：

```text
存在结果消息、执行事件、状态回写、需求结算、派生消息、派生事件之一，即形成候选。
```

样本完整判断：

```text
有结果消息，或有状态回写，或有需求结算，则认为本轮因果样本具备最小结果侧信息。
```

## 4. B1 因果缺口识别

缺口来源：

```text
派生需求消息
派生需求推进事件
冻结治理消息中的派生需求
候选存在但没有最小结果侧信息
```

第一版缺口类型：

```text
派生需求消息类型 = 直接作为缺口类型；
只有派生事件 = 派生需求缺口；
只有候选但样本不完整 = 因果样本不完整；
无缺口 = 无。
```

学习建议：

```text
本阶段只标记“建议学习”，不创建学习需求。
存在因果缺口时，建议学习 = true。
```
