## 1. 目标

本计划把“抽象存在”正式收口为：

```text
抽象存在 = 存在概念
```

不新增 `抽象存在主信息类`，不新增 `枚举_存在形态`。概念存在和实例存在继续共用：

```cpp
存在节点主信息类
```

区分方式只靠树位置和用途：

```text
存在概念根 下的存在节点 = 概念存在
世界根 / 场景根 / 内部世界 下的存在节点 = 实例存在
```

本计划的落地目标是形成最小启动链：

```text
存在概念根
→ 根概念“存在”
→ 核心概念分支
→ 概念默认特征槽
→ 名词性语素绑定到存在概念
→ 当前自我实例绑定 [存在, 数字生命, 自我]
→ 创建任务 / 方法虚拟存在时复用同一套概念实例机制
```

### 阶段二：默认特征槽初始化

```text

2. 给 “任务” 概念补任务特征槽：
   - 发起存在
   - 作用对象

   - 任务状态
   - 完成条件
   - 候选方法

3. 给 “方法” 概念补方法特征槽：

```

验收：


## 7. 风险点

```text
1. 概念默认特征槽如果写得太重，会把业务特征错误扩散到所有实例。
2. 任务 / 方法模块如果继续局部造概念，后续会出现同名不同节点。
3. 概念扩充当前只复制直接特征槽，暂不复制抽象特征绑定和复杂稳态规则。
4. 概念模板 是兼容字段，短期保留；后续若删除，需要清理语言表达、控制面板和若干旧引用。
5. 语素绑定要避免把专名实例误绑成概念，例如 “B-1234” 应能绑定实例存在。
```

## 9. 最终口径

```text
抽象存在不是新类。
抽象存在就是存在概念。

存在概念位于存在概念根下，用于语言入口和实例创建模板。
实例存在位于世界、场景、内部世界下，通过概念集获得多个概念归属。

概念扩充只负责给实例补第一批记忆槽。
任务、方法、需求仍各自保留运行壳或目标节点，不被虚拟存在吞掉。
```
