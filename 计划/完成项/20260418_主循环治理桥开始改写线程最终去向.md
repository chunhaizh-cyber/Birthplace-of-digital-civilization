# 20260418 主循环治理桥开始改写线程最终去向

## 本次目标

在上一轮“治理桥开始约束执行链和队列裁决”的基础上，再往前推一刀：

- 让 `自我_主循环治理模块` 不只约束 `是否执行 / 是否等待 / 是否学习 / 是否停止`
- 而是开始正式给出一份 `线程最终去向建议`
- 并让 `自我线程` 把这份建议写回 `当前最终去向`

## 本次完成

### 1. 治理桥增加“线程最终去向建议”

在 `D:\鱼巢\自我_主循环治理模块.ixx` 中新增：

- `推导线程最终去向_最小_v0(const 结构_一步治理结果_最小&)`

这条最小规则现在会把治理桥结果统一投影成线程最终去向：

- `停止`
- `转入学习`
- `回到重筹办`
- `继续提交`
- `待机`
- `进入收束`

其中最关键的一条是：

- `主循环去向=进入等待` 时，不再一律落到同一种线程去向；
- 若治理桥判定已经进入完成/需求满足，则会提升为 `进入收束`；
- 否则保持 `待机`。

### 2. 线程开始消费治理桥建议最终去向

在 `D:\鱼巢\自我线程模块.impl.cpp` 中：

- 主派发阶段拿到 `主循环治理桥结果` 后
- 立即调用 `推导线程最终去向_最小_v0(...)`
- 若治理桥建议的最终去向与线程当前去向不同，则直接回写：
  - `当前最终去向`
  - `任务管理当前去向`
  - `主派发决议.任务管理当前去向`

这意味着从这一刀开始，治理桥已经不只是“卡执行权限”，而是真正开始收敛线程最终去向。

### 3. 线程去向文本映射补齐

为了配合这次改写，在 `D:\鱼巢\自我线程模块.impl.cpp` 中新增了：

- `私有_映射线程最终去向到当前去向文本(...)`

这样线程在接受治理桥改写后，可以直接把：

- `继续提交 -> 进入运行`
- `回到重筹办 -> 回到重筹办`
- `转入学习 -> 转入学习`
- `进入收束 -> 进入收束`
- `待机 -> 保持等待`
- `停止 -> 停止`

统一回写到运行摘要、主派发快照和结果摘要里。

## 当前效果

到这一步为止，线程主派发链已经形成三层关系：

1. 原始批次裁决先给出线程基准去向；
2. 根层/联合证据链继续改写一次；
3. 治理桥再基于统一主循环口径，给出最终收敛建议并可再次改写。

也就是说，`当前最终去向` 已经开始逐步从“线程私有裁决产物”变成“线程 + 治理桥联合收敛产物”。

## 仍未完成

1. 当前还是“线程先裁决，再由治理桥收尾改写”，不是“治理桥前置产出主建议”。
2. 批次执行去向与线程最终去向之间仍存在双通道；真正的单通道还没完成。
3. `转入学习` 目前已能改写线程最终去向，但在派发动作层仍没有一条完全独立的显式动作分支。

## 下一步建议

1. 继续把 `批次执行去向 -> 线程最终去向` 这条老映射弱化，让治理桥先产建议，再由线程确认。
2. 把 `进入学习 / 进入收束 / 重筹办` 这些去向逐步从“摘要和裁决文本”推进成显式主派发动作。
3. 等这条线稳定后，再清理 `自我_主循环治理模块 / 自我_恢复快照模块 / 自我_学习承接模块` 对 `自我线程模块` 的反向依赖。
