# 20260418 双允许主派发细则前推为治理桥主裁决

## 本步目标

把 `自我线程` 主派发后半段里仍依赖本地候选去向补判的“双允许细则”继续前推进 `自我_主循环治理模块`，让治理桥开始直接主裁：

- `继续提交 / 重筹办`
- `重筹办 / 待机`

这两类双落点场景。

## 本步完成内容

1. 在 `自我_主循环治理模块` 的 `结构_一步治理输入_最小` 中新增双落点主裁所需的补充证据：
   - `主派发命中需求需求冲突`
   - `主派发命中禁止项重筹办建议`
   - `主派发命中联合需求主导`
   - `主派发命中联合收益恢复规则`
2. 治理桥执行判定阶段新增两类双允许细则识别：
   - `继续提交 + 重筹办`
   - `重筹办 + 待机`
3. 治理桥开始直接主裁这些双落点场景：
   - `重筹办 + 待机`：
     - 命中联合僵持，回治理
     - 命中联合需求主导且存在重筹办建议，回治理
     - 单纯存在重筹办建议，回治理
     - 否则进入待机
   - `继续提交 + 重筹办`：
     - 命中需求-需求冲突，回治理
     - 命中重筹办建议，回治理
     - 否则保留执行链继续推进
4. 线程向治理桥喂入的运行态输入与快照重读输入都同步补齐了上述证据，避免运行时和观察回放时出现不同主裁口径。
5. 线程在生成最终治理桥输入时，若当前命中“双允许细则主裁场景”，就不再把线程本地候选去向直接当成治理桥种子，而是改成：
   - 缺口种子 = `未定义`
   - 去向种子 = `执行方法`
   让治理桥真正进入双落点主裁分支。

## 主要改动文件

- `D:\鱼巢\自我_主循环治理模块.ixx`
- `D:\鱼巢\自我线程模块.ixx`
- `D:\鱼巢\自我线程模块.impl.cpp`

## 当前效果

- 双允许细则不再只是线程本地候选去向的尾部补判，治理桥已经可以直接主裁其中的关键分流。
- `继续提交/重筹办` 与 `重筹办/待机` 两类场景，开始真正由桥层消费“冲突 / 重筹办建议 / 联合需求主导 / 联合收益恢复”这些证据。
- 对于命中双允许细则的场景，线程不再先把本地候选去向塞给治理桥，而是把主裁权让出来。
