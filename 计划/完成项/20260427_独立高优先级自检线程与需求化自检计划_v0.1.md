# 20260427 独立高优先级自检线程与需求化自检计划 v0.1

## 1. 目标

按 `0580_自检发现缺口与需求落点规范` 落地自检主线。

目标不是做自检报告，而是建立一条独立高优先级自检线程：

`自检线程读取事实 -> 生成现有需求信息 -> 提交自我线程 -> 自我线程落需求树并建任务 -> 任务提交任务管理任务界面线程`

自检不直接修复，不直接建任务，不直接调用任务管理任务界面线程。

## 2. 约束

1. 自检返回结果必须是现有需求信息或等价派生需求消息。
2. 不新增与需求信息并行的 `自检问题` 结构。
3. 自检线程只读世界树、需求树、需求列表、任务树、任务列表、方法树、方法运行账、学习账本等基础数据。
4. 自检线程只向自我线程提交需求信息。
5. 检查项必须能落入需求树。
6. 没有需求表达、没有落点、没有后续任务闭环的检查项不实现。
7. 自检不得重复任务筹办、学习执行、初始化、归档、回写等既有能力。
8. 自检线程不得读取控制面板快照、控制面板摘要或展示层聚合快照。
9. 自检未发现问题时不提交消息，只记录事件信息。
10. 自检频率不得过高，不同检查项使用不同频率。
11. 休眠期自检每次休眠按需要检查一次或分批一次，不在整个休眠期重复扫同一项目。

## 3. 现有能力复用清单

自检线程后续处理依赖现有能力：

- `自我_任务筹办`
- `自我_任务执行一步`
- `自我_学习任务`
- `学习_尝试学习`
- `自我_增加可用方法数量`
- `自我_更新待学习方法数量`
- `管理_确保单承载存在`
- `管理_读取单特征当前值`
- `管理_写入单特征当前值`
- `管理_形成治理特征面`
- `管理_刷新方法需求位特征`
- `管理_刷新治理二次特征证据`
- `管理_评估本能能力缺口`
- `管理_执行桥接补层裁决`
- `管理_沉淀运行方法证据`

其中 `自我_更新待学习方法数量`、部分管理最小原语如果只有方法头而运行体未完成，第一版自检仍只生成对应需求；后续任务链遇到运行体缺口时，应继续按待建方法头/缺能力规则派生学习或实现需求。

## 3.1 特征值类型约束

本计划实现时必须遵守 `0581` 的值域口径：

1. 布尔型只用于天然二值事实，例如是否存在、是否有效、是否允许、是否完成。
2. 布尔型统一用 `0/1`。
3. 可靠度、质量、完成度、优先级、成本、次数、压力、收益、损失等可比较或可累计的值，统一使用 `I64`。
4. I64 度量值不使用 `Q10000`，不设固定上限。
5. 选择时直接按值最大或最小选择。
6. 更新时直接做 `当前值 + 增量` 或 `当前值 - 减量`。
7. 超过两个状态的值使用枚举或状态词，不压成 `0/1`。

第一版常用布尔特征：

- 方法结构是否完整。
- 方法运行账是否完整。
- 方法可参与可靠度提升。
- 方法可参与自动召回。
- 学习账本需重建。
- 引用是否有效。
- 是否可重定位。
- 是否阻塞当前步骤。
- 是否影响父任务推进。

第一版常用 I64 特征：

- 方法可靠度。
- 方法最近质量。
- 方法连续失败次数。
- 方法异常次数。
- 方法运行次数。
- 方法条件节点数量。
- 方法结果节点数量。
- 方法条件样本数量。
- 方法结果样本数量。
- 可用方法数量。
- 待学习方法数量。
- 任务完成度。

第一版常用枚举/状态词特征：

- 方法稳定状态：`稳定/不稳定/证据不足`。
- 方法运行账完整性：`完整/不完整/不可恢复`。
- 方法样本完整性：`完整/不完整/证据不足`。
- 学习账本引用有效性：`有效/部分无效/无效`。
- 学习账本重定位状态：`已重定位/部分重定位/失败`。
- 任务停滞原因：`待建方法头/等待学习/等待结果/连续失败/未知`。
- 任务推进状态：`停滞/可继续推进`。

## 4. 阶段划分

### 阶段 0：规范与字段对齐

目标：

- 确认 `0580` 中运行中自检和休眠期自检条目为第一版边界。
- 确认自检需求复用现有派生需求消息或需求树更新指令。
- 确认需求落点优先级：来源需求 -> 当前主需求 -> 安全根/服务根。
- 确认自检线程禁止读取控制面板快照，只能读取基础树、列表、运行账和学习账本。
- 确认未发现问题时只记事件，不提交空消息。

改动范围：

- 规范文件。
- 不改代码。

完成判据：

- 规范中每个检查项都有读取证据、需求表达、需求落点、后续本能函数。

### 阶段 1：自检线程壳

目标：

- 新增独立自检线程模块。
- 线程生命周期可观察。
- 支持运行中模式和休眠期模式。
- 支持启动、停止、唤醒、读取自检线程自身事件摘要。

建议文件：

- `自检线程模块.ixx`
- `自检线程模块.impl.cpp`
- 必要时新增 `自检线程模块.消息协议.ixx`
- `鱼巢.vcxproj`
- `鱼巢.vcxproj.filters`

实现要点：

- 线程只读世界树、需求树、需求列表、任务树、任务列表、方法树、方法虚拟存在、方法运行账、学习账本。
- 线程不得读取控制面板快照、控制面板摘要、控制面板 HTML 或展示层聚合快照。
- 线程不持有长期写锁。
- 线程不直接修改树。
- 线程内部仅保存去重账本、最近提交摘要、运行计数、休眠期游标、同类检查最近执行时间。
- 不同检查项配置不同频率，禁止按每个 Tick 全量检查。

完成判据：

- 程序启动后自检线程可随自我线程启动。
- 控制面板或摘要能看到自检线程生命周期和最近摘要。
- 停止程序时自检线程可正常收尾。
- 未发现问题时仅记录事件，不向自我线程提交消息。

### 阶段 2：需求化提交入口

目标：

- 自检线程能把发现项表达为现有需求信息。
- 自检线程能提交给自我线程。
- 自我线程按现有派生需求逻辑落需求树。

建议文件：

- `自我线程模块.消息协议.ixx`
- `自我线程模块.impl.cpp`
- `任务管理任务模块.消息协议.ixx` 如需复用派生需求字段则优先不改。

实现要点：

- 优先复用 `结构_任务管理派生需求消息` 或等价需求树更新指令。
- 如必须扩展消息来源，只新增来源枚举，不新增自检问题结构。
- 提交内容必须包含需求类型、当前状态、目标状态、来源任务、来源步骤、来源方法、父需求定位证据。
- 自我线程收到后走 `派生需求入树` 现有路径。

完成判据：

- 手工构造一个方法结构不完整或方法稳定性不足自检需求，可以落到来源需求下。
- 重复提交不会生成重复需求。
- 需求任务仍由自我线程建立，并提交任务管理任务界面线程。

### 阶段 2.6：第一版本能函数运行体

目标：

- 实现第一版自检需求进入任务闭环后真正可调用的本能函数运行体。
- 每个运行体都走 `本能动作管理类::运行` 统一记账。
- 每个运行体只处理自己的职责，不重复自检线程和任务筹办能力。

#### 2.6.1 `学习_补齐方法结构`

输入：

- 待学习方法。
- 方法动作骨架数量：`0..I64_MAX`。
- 方法条件节点数量：`0..I64_MAX`。
- 方法结果节点数量：`0..I64_MAX`。
- 方法是否可用：`0/1`。

输出：

- 缺动作时：`方法动作骨架数量 0 -> 1`。
- 缺条件时：`方法条件节点数量 0 -> 1`。
- 缺结果时：`方法结果节点数量 0 -> 1`。
- 三者具备时：`方法是否可用 = 1`。

实现要点：

- 可复用现有 `学习_尝试学习` 对动作/条件/结果缺口的判断逻辑。
- 不直接提升可靠度。
- 不直接增加可用方法数量，确认可用后的数量同步由镜像同步本能处理。

验收：

- 缺条件节点的方法进入该本能后，返回条件节点补齐需求。
- 结构完整的方法进入该本能后，返回 `方法是否可用=1`。

#### 2.6.2 `学习_验证方法稳定性`

输入：

- 方法可靠度：`I64`。
- 方法最近质量：`I64`。
- 方法连续失败次数：`0..I64_MAX`。
- 方法异常次数：`0..I64_MAX`。
- 方法运行次数：`0..I64_MAX`。
- 最近错误码：`I64`。

输出：

- 方法稳定状态：`稳定/不稳定/证据不足`。
- 方法可靠度：不足时 `当前值 -> 当前值 + 1`，仅作为目标需求方向。
- 方法允许默认执行：`0/1`。
- 方法允许自动召回：`0/1`。

实现要点：

- 运行次数过少返回 `证据不足`。
- 连续失败次数大于 0、异常次数大于 0、可靠度小于等于 0、最近质量小于等于 0 时，优先返回 `不稳定`。
- 可靠度和质量比较直接用 I64 大小。
- 不直接切换当前任务方法。

验收：

- 连续失败方法返回 `方法稳定状态=不稳定`、`方法允许默认执行=0`。
- 运行次数不足的方法返回 `证据不足`。

#### 2.6.3 `管理_检查方法运行账完整性`

输入：

- 方法运行次数：`0..I64_MAX`。
- 最近开始时间。
- 最近结束时间。
- 最近耗时：`0..I64_MAX`。
- 最近结果场景。
- 最近错误码：`I64`。
- 成功次数：`0..I64_MAX`。
- 失败次数：`0..I64_MAX`。
- 异常次数：`0..I64_MAX`。

输出：

- 方法运行账完整性：`完整/不完整/不可恢复`。
- 方法可参与可靠度提升：`0/1`。
- 方法可参与自动召回：`0/1`。

实现要点：

- 运行次数为 0 时可返回 `不完整` 或 `证据不足` 类状态，第一版用 `不完整`。
- 有运行次数但缺最近结果场景、开始/结束时间明显不一致时返回 `不完整`。
- 不伪造历史事实。
- 对不可恢复账返回不可参与可靠度提升和自动召回。

验收：

- 有运行次数但缺结果场景的方法返回 `方法运行账完整性=不完整`。
- 完整运行账返回 `完整` 和两个允许标志为 `1`。

#### 2.6.4 `学习_检查学习账本引用`

输入：

- 学习任务引用。
- 待学习方法引用。
- 来源任务引用。
- 来源需求引用。
- 当前学习状态。

输出：

- 学习账本引用有效性：`有效/部分无效/无效`。
- 学习任务状态：必要时返回 `挂起`。
- 学习账本需重建：`0/1`。

实现要点：

- 只检查引用是否可解析到当前树。
- 不直接重定位。
- 不直接创建新学习任务。
- 无法解析待学习方法时，应返回 `学习账本需重建=1`。

验收：

- 待学习方法引用失效时，返回 `无效` 和 `学习账本需重建=1`。
- 所有引用有效时，返回 `有效`。

#### 2.6.5 `自我_同步可用方法数量镜像`

输入：

- 当前可用方法数量：`0..I64_MAX`。
- 统计可用方法数量：`0..I64_MAX`。

输出：

- 可用方法数量：`当前值 -> 统计值`。
- 可用方法数量镜像一致性：`一致`。

实现要点：

- 使用绝对同步，不使用 `+1`。
- 不改变任何方法的可用性。
- 统计口径应以方法树中 `方法是否可用=1` 为准。

验收：

- 当前值与统计值不一致时，写回统计值。
- 当前值一致时不提交多余变化。

#### 2.6.6 `自我_同步待学习方法数量镜像`

输入：

- 当前待学习方法数量：`0..I64_MAX`。
- 统计活跃学习任务数量：`0..I64_MAX`。

输出：

- 待学习方法数量：`当前值 -> 统计值`。
- 待学习方法数量镜像一致性：`一致`。

实现要点：

- 使用绝对同步。
- 不直接创建、删除或挂起学习任务。
- 统计口径以学习账本活跃任务数量为准。

验收：

- 当前值与学习账本活跃数不一致时，写回账本活跃数。
- 一致时不提交多余变化。

### 阶段 3：运行中自检第一批（直接从原第二批开始）

目标：

先实现现有筹办、执行桥和任务管理治理没有直接重复覆盖的运行中检查。

第一批条目：

1. 方法结构不完整。
2. 方法稳定性不足。
3. 当前任务完成度停滞。
4. 学习任务可调度但未推进。
5. 方法运行账不完整。
6. 方法样本缺失一侧。
7. 学习账本引用失效。

实现要点：

- 每项只生成需求，不处理问题。
- 每项必须有去重键。
- 每项必须有父需求落点。
- 每类检查有独立频率，避免 Tick 噪声。
- 未发现问题时只记录自检事件。
- 不读取控制面板快照。

完成判据：

- 构造方法缺条件/结果节点时，自检能生成结构补齐需求。
- 构造方法连续失败或运行账质量不足时，自检能生成稳定性/可靠度需求。
- 构造学习账本可调度但长期未推进时，自检能生成学习推进需求。
- 任务管理后续仍按既有本能函数推进。

### 阶段 4：运行中自检扩展批

目标：

在不重复现有能力的前提下补充更多运行中检查。

候选条目：

1. 方法运行账与方法虚拟存在不一致。
2. 学习回流后方法仍未形成可召回样本。
3. 同一来源需求下重复生成同类学习需求。
4. 同一方法反复生成同类结构缺口。

实现要点：

- 方法稳定性只使用已有运行账字段，例如连续失败、异常次数、最近质量、可靠度。
- 没有明确阈值时，先采用保守阈值并写入规范补充。
- 学习停滞只生成学习推进需求，不直接调用学习执行。

完成判据：

- 不稳定方法能转成方法可靠度/稳定状态需求。
- 可调度学习项长期未推进时，能生成学习推进状态需求。

### 阶段 5：休眠期自检第一批

目标：

实现每次休眠按需执行的轻量数据完整性检查。

第一批条目：

1. 方法虚拟存在完整性。
2. 学习账本引用有效性。
3. 可用方法数量镜像一致性。
4. 待学习方法数量镜像一致性。
5. 历史宿主残留是否仍影响当前树。

实现要点：

- 只在进入待机、收束空闲或显式请求时按需运行。
- 同一次休眠期内，同一检查项完成后不重复执行。
- 只有树结构、账本、方法运行账发生相关变化，或外部显式请求，才允许同一休眠期重检。
- 每次可分批检查，避免长时间占用。
- 方法资产类问题挂服务根需求，能追溯来源时挂来源需求。
- 不读取控制面板快照。

完成判据：

- 方法缺虚拟存在时，生成方法虚拟存在状态需求。
- 学习账本引用无效时，生成学习账本引用有效性需求。
- 镜像数量不一致时生成对应数量需求。

### 阶段 6：休眠期自检第二批

目标：

补齐全局结构完整性检查，仍只做按需一次或分批一次。

第二批条目：

1. 自我锚点完整性。
2. 根需求完整性。
3. 根任务完整性。
4. 方法根完整性。
5. 当前主链引用有效性。

实现要点：

- 数量一致性只生成需求，不直接重算写回。
- 引用有效性优先按主键重定位的需求表达，不由自检直接重定位。
- 自我锚点、根需求、根任务、方法根这类已有初始化确保能力覆盖的事项，只在休眠期基础结构检查中作为兜底需求生成，不进入运行中第一批。

完成判据：

## 5. 风险与处理

### 5.1 自检需求噪声

风险：

- 每个 Tick 重复提交同类需求。

处理：

- 去重键必须包含父需求、需求类型、被需求对象、当前状态、目标状态、来源任务、来源步骤、来源方法。
- 运行中自检每轮提交数量设上限。
- 每类检查有独立频率，禁止全量 Tick 检查。

### 5.2 自检线程越权

风险：

- 自检直接修数据，破坏任务闭环。

处理：

- 自检模块不暴露写树接口。
- 只允许调用自我线程需求提交入口。

### 5.3 检查项没有处理闭环

风险：

- 发现问题但无法推进。

处理：

- 此类检查项不进入正式自检。
- 先补需求表达和处理闭环，再纳入自检。

### 5.4 休眠期检查过重

风险：

- 数据完整性检查拖慢系统。

处理：

- 分批扫描。
- 每次休眠只按需检查一次或分批一次。
- 有队列、有高优先级任务、有停止请求时立即暂停休眠期检查。
- 同一休眠期内不重复检查已完成项目，除非相关基础树或账本发生变化。

## 6. 第一版最小交付

第一版只做以下最小闭环：

1. 自检线程壳。
2. 自检需求提交到自我线程。
3. 方法结构不完整检查。
4. 方法稳定性不足检查。
5. 学习任务可调度但未推进检查。
6. 方法虚拟存在完整性休眠期检查。
7. 学习账本引用有效性休眠期检查。
8. 去重、频率控制和自检事件摘要。

这 8 项通过后，再扩展其余检查项。
