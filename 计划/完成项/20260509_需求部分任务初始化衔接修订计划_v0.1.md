# 需求部分任务初始化衔接修订计划 v0.1

日期：2026-05-09

## 1. 目标

本计划用于把需求部分和“任务信息节点 + 任务虚拟存在”的最新口径对齐。

最终结构固定为：

```text
存在 E
├─ 需求根节点
│  └─ D001：需求节点
│
└─ 任务根节点
   └─ N001：任务信息节点
      └─ T001：任务虚拟存在
```

职责边界：

```text
需求节点 D = 存在内部的目标记录 / 动因节点。
任务信息节点 N = 为满足需求创建的运行壳 / 管理壳。
任务虚拟存在 T = 任务信息节点内部引用的语义存在实例。
```

需求节点不直接变成存在，也不直接把 `对应任务` 指向任务虚拟存在。需求到任务的权威链路必须保持：

```text
D.对应任务 -> 任务信息节点 N
N.任务虚拟存在 -> 存在节点 T
```

## 2. 当前代码现状

当前 `需求主信息类.h` 的主方向已经正确：

```text
1. 需求表达一个原子目标。
2. 多目标由管理需求子链展开。
3. 需求树父子关系由节点树结构表达。
4. `对应任务` 已注释为该需求节点当前权威运行壳。
5. `任务列表` 已作为历史、候选、重建痕迹列表。
```

当前已具备的字段应保留：

```cpp
可解析引用<存在节点类> 需求主体{};
可解析引用<场景节点类> 需求场景{};
可解析引用<存在节点类> 被需求存在{};
可解析引用<状态节点类> 被需求当前状态{};
可解析引用<状态节点类> 需求状态{};
三向关系掩码 满足关系 = 关系_等于;

可解析引用<任务节点类> 对应任务{};
std::vector<可解析引用<任务节点类>> 任务列表{};
```

当前主要缺口：

```text
1. `结构_需求树更新指令` 没有显式输入需求主体、需求场景、被需求存在。
2. `合并需求主信息_已加锁` 尚未合并需求主体和需求场景。
3. `结构角色` 的刷新散落在自我线程，不在需求类内部形成统一规则。
4. `对应任务` / `任务列表` 的写入散落在多个模块。
5. 需求类还没有输出任务初始化上下文的公用接口。
6. `需求信息类型.h` 存在 `需求息节点类` 命名残留。
```

## 3. 修订原则

```text
1. 需求类只维护需求树和任务初始化材料。
2. 需求类不创建任务信息节点。
3. 需求类不创建任务虚拟存在。
4. 需求类不选择方法。
5. 需求类不执行任务。
6. 需求类不新增 `对应任务虚拟存在` 字段。
7. 任务系统根据需求初始化上下文创建任务信息节点 N 和任务虚拟存在 T。
```

允许需求类做的事情：

```text
1. 添加、更新、满足、删除需求。
2. 根据目标状态补齐被需求存在和目标特征缓存。
3. 判断需求目标是否重叠。
4. 合并重复需求。
5. 刷新管理需求 / 执行需求角色。
6. 绑定需求对应的任务信息节点。
7. 生成任务初始化上下文。
```

修改前先固定这些扫描项：

```text

2. 直接写 `需求->主信息.对应任务` 的位置。

6. 任务类中根据需求创建或复用任务的逻辑。
```

当前已检索到的重点文件：

```text
需求类.h
需求主信息类.h
需求信息类型.h
任务类.cpp
任务模块.管理工作线程.ixx
自我线程模块.impl.cpp
```

## 5. P1 扩展需求树更新指令

在 `结构_需求树更新指令` 中新增输入能力：

```cpp
std::uintptr_t 需求主体指针 = 0;
std::string 需求主体主键{};

std::uintptr_t 需求场景指针 = 0;
std::string 需求场景主键{};

std::uintptr_t 被需求存在指针 = 0;
std::string 被需求存在主键{};
```

新增需求时写入：

```text
主信息.需求主体 = 指令.需求主体
主信息.需求场景 = 指令.需求场景
主信息.被需求存在 = 指令.被需求存在
```

补齐规则：

```text
1. 指令显式给出时，优先使用指令。
2. 子需求未给需求主体时，继承父需求.需求主体。
3. 子需求未给需求场景时，继承父需求.需求场景。
4. 未给被需求存在时，继续沿用现有“从状态主体补齐”的逻辑。
5. 指令和状态主体冲突时，先保留显式指令，后续由需求一致性检查处理。
```

这一步的目的不是让需求变胖，而是保证后续任务初始化能找到：

```text
发起存在 E = 需求主体
任务挂载位置 = E.任务根节点
作用对象 = 被需求存在
所属上下文 = 需求场景
```

## 6. P2 合并需求主信息时保留归属

修改 `合并需求主信息_已加锁`，在目标需求缺失字段时补入候选字段：

```cpp
if (!目标.需求主体.有效() && 候选.需求主体.有效()) {
    目标.需求主体 = 候选.需求主体;
}

if (!目标.需求场景.有效() && 候选.需求场景.有效()) {
    目标.需求场景 = 候选.需求场景;
}
```

保留现有目标合并逻辑：

```text
1. 类型
2. 当前状态
3. 目标状态
4. 被需求存在
5. 满足关系
6. 目标特征类型缓存
7. 派生归因
```

验收点：

```text
同目标去重合并后，需求不能丢失“谁发起”和“在哪个场景”的信息。
```

## 7. P3 统一维护需求结构角色

把 `结构角色` 的维护收进需求类，形成统一接口：

```cpp
static void 刷新需求结构角色(节点类* 节点) noexcept;
static void 刷新需求及祖先结构角色(节点类* 节点) noexcept;
```

规则：

```text
有子需求 -> 管理需求
无子需求 -> 执行需求
```

触发点：

```text
1. 新增子需求后，刷新新节点、父节点和祖先。
2. 删除需求后，刷新原父节点和祖先。
3. 合并需求后，刷新目标节点和父节点。
4. 满足 / 截止需求后，刷新相关父节点。
```

后续清理目标：

```text
自我线程不再直接写 `需求->主信息.结构角色`，
改为调用需求类接口。
```

## 8. P4 增加绑定任务信息节点接口

需求侧新增公用函数：

```cpp
static bool 绑定对应任务(
    节点类* 需求,
    任务节点类* 任务
) noexcept;
```

职责：

```text
1. 校验需求和任务非空。
2. 写入 `需求->主信息.对应任务`。
3. 将任务引用加入 `需求->主信息.任务列表`。
4. 对任务列表去重。
5. 不接触任务虚拟存在。
```

迁移目标：

```text
1. `任务模块.管理工作线程.ixx` 中直接 push 任务列表的位置。
2. `任务类.cpp` 中直接 push 任务列表的位置。
3. `自我线程模块.impl.cpp` 中直接 push 任务列表和直接写对应任务的位置。
```

绑定后的权威语义：

```text
需求.对应任务 = 任务信息节点 N。
任务虚拟存在只能通过 N.任务虚拟存在 间接取得。
```

禁止新增：

```cpp
可解析引用<存在节点类> 对应任务虚拟存在{};
```

## 9. P5 增加任务初始化上下文

在需求类附近定义轻量上下文：

```cpp
struct 结构_任务初始化上下文 {
    需求节点类* 来源需求 = nullptr;

    可解析引用<存在节点类> 发起存在{};
    可解析引用<场景节点类> 所属场景{};
    可解析引用<存在节点类> 作用对象{};

    可解析引用<状态节点类> 当前状态{};
    可解析引用<状态节点类> 目标状态{};

    三向关系掩码 满足关系 = 关系_等于;
    const 词性节点类* 目标特征类型 = nullptr;

    std::int64_t 安全权重 = 0;
    std::int64_t 服务权重 = 0;

    bool 有效() const noexcept {
        return 来源需求
            && 发起存在.有效()
            && (作用对象.有效() || 目标状态.有效());
    }
};
```

需求类新增：

```cpp
结构_任务初始化上下文 生成任务初始化上下文(
    节点类* 需求
) const noexcept;
```

字段映射：

| 需求字段 | 上下文字段 |
| --- | --- |
| `需求主体` | `发起存在` |
| `需求场景` | `所属场景` |
| `被需求存在` | `作用对象` |
| `被需求当前状态` | `当前状态` |
| `需求状态` | `目标状态` |
| `满足关系` | `满足关系` |
| `目标特征类型缓存` | `目标特征类型` |
| `安全权重` | `安全权重` |
| `服务权重` | `服务权重` |

边界：

```text
生成上下文只读需求。
不创建任务信息节点。
不创建任务虚拟存在。
不写需求.对应任务。
```

## 10. P6 对接任务初始化流程

任务系统后续应使用需求上下文创建任务：

```text
1. 需求 D 触发任务创建。
2. 需求类生成任务初始化上下文 Ctx。
3. 任务系统读取 Ctx.发起存在 E。
4. 在 E.任务根节点 下创建任务信息节点 N。
5. 根据任务概念创建任务虚拟存在 T。
6. N.任务虚拟存在 = T。
7. 用 Ctx 填充 T 的任务语义特征。
8. 调用 需求类::绑定对应任务(D, N)。
```

任务虚拟存在初始特征由任务类负责，不由需求类负责。

建议任务侧调用关系：

```text
需求类::生成任务初始化上下文
    -> 任务类::根据需求初始化任务
        -> 创建任务信息节点 N
        -> 创建 / 绑定任务虚拟存在 T
        -> 需求类::绑定对应任务(D, N)
```

## 11. P7 管理需求与执行需求策略

第一阶段采用保守策略：

```text
执行需求：必须能够生成任务初始化上下文，并创建具体任务信息节点。
管理需求：优先只组织子需求；需要调度或汇总时再创建管理任务信息节点。
```

判断规则：

```text
结构角色 = 执行需求 -> 可直接创建任务。
结构角色 = 管理需求 -> 默认等待子需求任务，除非调用方明确要求创建管理任务。
```

这能避免管理需求和执行需求同时重复建任务。

## 12. P8 清理命名残留

`需求信息类型.h` 当前存在：

```cpp
using 需求息节点类 = 链表模板<需求信息基类*>::节点类;
```

计划清理为：

```cpp
using 需求信息节点类 = 链表模板<需求信息基类*>::节点类;
```

处理规则：

```text
1. 若 `需求息节点类` 无引用，直接删除或改名。
2. 若仍有引用，连引用点一起改为 `需求信息节点类`。
3. 不影响正式 `需求节点类 = 链表模板<需求主信息类>::节点类`。
```

## 13. 旧代码清理专项

本次修订必须同步清理旧路径，不能只新增接口后保留并行写法。否则后续会同时存在：

```text
需求类统一维护字段
外部模块继续直接改字段
```

这会让 `对应任务`、`任务列表`、`结构角色` 再次分裂。

### 13.1 清理原则

```text
1. 新接口落地后，旧写法必须迁移或删除。
2. 若旧函数只服务旧写法，连函数和引用一起删除。
3. 若旧函数仍承担有效业务，只保留业务判断，把字段写入改为调用新接口。
4. 不保留“兼容旧口径”的双入口。
5. 不新增需求到任务虚拟存在的直连入口。
6. 不新增需求虚拟存在。
```

### 13.2 需要清理的旧写法

#### A. 直接写 `需求.对应任务`

旧写法：

```cpp
需求->主信息.对应任务 = 引用;
```

新写法：

```cpp
需求类::绑定对应任务(需求, 任务);
```

处理要求：

```text
1. 所有外部模块不再直接写 `对应任务`。
2. 需求类内部可以作为唯一写入点。
3. 若某处只为同步 `对应任务` 而存在，迁移调用后删除旧辅助函数。
```

#### B. 直接 push `需求.任务列表`

旧写法：

```cpp
需求->主信息.任务列表.push_back(引用);
```

新写法：

```cpp
需求类::绑定对应任务(需求, 任务);
```

处理要求：

```text
1. 任务列表去重只能放在 `绑定对应任务` 内。
2. 外部模块可以读任务列表，但不能写任务列表。
3. 原有私有去重函数如果只服务任务列表写入，应删除或内聚到需求类。
```

#### C. 直接写 `需求.结构角色`

旧写法：

```cpp
需求->主信息.结构角色 = 枚举_需求结构角色::管理需求;
需求->主信息.结构角色 = 枚举_需求结构角色::执行需求;
```

新写法：

```cpp
需求类::刷新需求结构角色(需求);
需求类::刷新需求及祖先结构角色(需求);
```

处理要求：

```text
1. 自我线程不再自行判断并写结构角色。
2. 新增、删除、合并、截止、满足需求后统一刷新。
3. 若旧判断函数只用于写结构角色，应删除或改为只提供“是否存在活动子需求”的业务判断。
```

#### D. 自我线程中补需求主体 / 场景 / 被需求存在的散写逻辑

旧口径：

```text
自我线程在构造或入树后临时补 `需求主体`、`需求场景`、`被需求存在`。
```

新口径：

```text
构造 `结构_需求树更新指令` 时显式携带这些输入；
需求类在入树时统一解析、继承、兜底。
```

处理要求：

```text
1. 自我线程只负责把消息字段填入指令。
2. 父需求继承和状态主体兜底收进需求类。
3. 入树后再补字段的旧代码应删除。
```

#### E. 需求直接指向任务虚拟存在的旧入口

若发现以下字段、函数或注释语义：

```text
需求 -> 任务虚拟存在
需求 -> 任务存在
需求 -> 对应任务实例存在
```

处理要求：

```text
1. 不保留。
2. 改为 `需求 -> 任务信息节点 -> 任务虚拟存在`。
3. 查询便利需求通过只读辅助函数解决，不增加并列权威字段。
```

允许的只读辅助函数：

```cpp
存在节点类* 取需求对应任务虚拟存在(需求节点类* 需求) noexcept;
```

该函数内部必须走：

```text
需求.对应任务 -> 任务主信息.任务虚拟存在
```

#### F. 需求节点当基础信息存在使用的旧入口

第一阶段不引入需求虚拟存在。

若发现：

```text
需求节点被塞进存在 / 基础信息特征值
需求节点被当成自然语言语义对象
需求节点被作为基础信息节点查找
```

处理要求：

```text
1. 删除或隔离该入口。
2. 若确实需要长期语义引用，另开“需求虚拟存在”专项，不在本计划内兼容。
3. 本计划只保留需求信息节点自身和任务初始化上下文。
```

#### G. `需求信息类型.h` 命名残留

旧残留：

```cpp
using 需求息节点类 = 链表模板<需求信息基类*>::节点类;
```

处理要求：

```text
1. 若无引用，直接删除旧别名。
2. 若有引用，统一改为 `需求信息节点类`。
3. 不影响正式 `需求节点类`。
```

### 13.3 当前已知清理点

当前扫描已发现这些位置需要纳入迁移或删除清单：

```text
任务模块.管理工作线程.ixx
  - 存在直接维护 `任务列表` 的私有辅助逻辑。

任务类.cpp
  - 创建任务时存在直接 push 来源需求任务列表的逻辑。

自我线程模块.impl.cpp
  - 存在直接 push `任务列表` 的逻辑。
  - 存在直接写 `结构角色` 的逻辑。
  - 存在新增需求后补主体 / 场景 / 被需求存在的逻辑。

需求信息类型.h
  - 存在 `需求息节点类` 命名残留。
```

清理时不要求一次删除所有读取逻辑，但要求写入入口收敛：

```text
读取可以保留。
写入必须走需求类。
```

### 13.4 删除顺序

```text
1. 先新增需求类公用接口：
   - `绑定对应任务`
   - `刷新需求结构角色`
   - `生成任务初始化上下文`

2. 再迁移调用点：
   - 任务类创建任务后的需求绑定
   - 管理工作线程中的需求任务列表维护
   - 自我线程中的需求入树、任务绑定和角色刷新

3. 然后删除旧私有辅助：
   - 只负责任务列表去重 / push 的重复函数
   - 只负责直接写结构角色的重复函数
   - 只负责入树后补主体 / 场景的临时代码

4. 最后做全仓库扫描：
   - 确认无外部直接写 `对应任务`
   - 确认无外部直接 push `任务列表`
   - 确认无外部直接写 `结构角色`
   - 确认无需求到任务虚拟存在直连字段
```

## 14. 保留项边界

以下字段保留，不属于旧代码：

```text
1. `需求主信息类::对应任务`
2. `需求主信息类::任务列表`
3. `需求主信息类::需求主体`
4. `需求主信息类::需求场景`
5. `需求主信息类::被需求存在`
```

以下读取行为保留：

```text
1. 控制面板读取需求任务列表用于显示。
2. 任务管理线程读取需求任务列表用于查找可复用任务。
3. 自我线程读取需求任务列表用于判断需求是否已有运行壳。
```

保留条件：

```text
只读可以保留；
写入必须收敛到需求类。
```

## 15. 验收用例

至少覆盖：

```text
1. 新增根需求时，显式 `需求主体` 能写入主信息。
2. 新增子需求未给主体时，继承父需求主体。
3. 新增子需求未给场景时，继承父需求场景。
4. 未给被需求存在时，仍能从目标状态主体补齐。
5. 同目标需求去重合并后，需求主体和需求场景不丢失。
6. 叶子需求刷新为执行需求。
7. 有子需求的节点刷新为管理需求。
8. 绑定任务时，`对应任务` 写入任务信息节点。
9. 绑定任务时，`任务列表` 去重。
10. 生成任务初始化上下文时，发起存在、作用对象、目标状态、满足关系正确。
11. 需求类不创建任务虚拟存在。
12. 全仓库无新增 `对应任务虚拟存在` 字段。
```

## 16. 执行顺序

```text
P0：扫描影响面，确认所有直接写字段位置。
P1：扩展 `结构_需求树更新指令`。
P2：新增指令解析、父需求继承和状态主体兜底。
P3：补 `合并需求主信息_已加锁` 的主体 / 场景合并。
P4：新增并接入结构角色刷新接口。
P5：新增 `绑定对应任务` 并迁移直接写任务字段的调用点。
P6：新增 `结构_任务初始化上下文` 和生成接口。
P7：让任务初始化计划调用需求上下文，而不是自行猜需求字段。
P8：清理 `需求息节点类` 命名残留。
P9：编译和最小回归验证。
```

## 17. 完成标准

```text
1. 需求仍是轻量目标记录，不是存在实例。
2. `需求.对应任务` 仍指向任务信息节点。
3. 任务虚拟存在只能从任务信息节点取得。
4. 需求主体、需求场景、被需求存在能从更新指令进入需求主信息。
5. 合并需求不会丢失发起存在和场景。
6. 结构角色由需求类统一维护。
7. 需求到任务绑定由单一公用接口维护。
8. 任务初始化能从需求类获得稳定上下文。
```

一句话定稿：

```text
需求是动因和目标，任务信息节点是运行壳，任务虚拟存在是语义记忆体；需求类只负责把动因整理成任务初始化材料。
```
