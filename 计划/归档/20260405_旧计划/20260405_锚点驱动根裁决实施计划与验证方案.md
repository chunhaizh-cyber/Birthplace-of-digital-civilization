## 5. 分阶段实施计划

### 阶段 A：锚点读取正式化

目标：

- 给根裁决层提供统一的 `锚点快照读取` 入口
- 后续所有根裁决代码都从该入口取：
  - `服务当前值`
  - `服务目标值`
  - `服务差值`
  - `安全当前值`
  - `安全目标值`
  - `安全保障差值`
  - `主方向极性`
  - `保障方向极性`

实施点：

- 在 [自我_主循环治理模块.ixx](D:/鱼巢/自我_主循环治理模块.ixx) 内新增方向锚点快照结构
- 不再在根裁决逻辑里散落读取 `自我_安全值 / 自我_服务值`
- 对锚点缺失、类型不符、指针断裂做显式回退

完成判据：

- 根裁决层只保留一处方向读取入口
- 方向值与当前控制面板显示值一致


### 阶段 B：活动自我锚点合法性前置校验

目标：

- 在每轮根裁决前确认：
  - `活动自我锚点` 存在
  - 能回指 `自我存在`
  - 能回指 `自我内部世界`
  - 能回指 `宿主现实场景`
  - 能回指 `主方向锚点`

实施点：

- 若锚点链不完整：
  - 优先触发一次锚点重同步
  - 重同步失败时进入 `边界处理`
  - 不能直接继续进入普通执行链

完成判据：

- 锚点链缺失时不会静默继续治理
- 控制台摘要能看到“锚点失效/重建/回退”的显式反馈


### 阶段 C：主方向锚点驱动根任务选择

目标：

- 把 `安全根任务 / 服务根任务` 的选择从“直接比需求权重”改为“先读方向锚点，再决定当前主需求/主任务”

建议规则：

1. `安全值 <= 0`
   - 直接停止
   - 不再讨论主方向

2. `安全保障差值 > 0` 且达到当前主导边界
   - 优先选择 `安全需求 / 安全根任务`

3. `安全保障差值` 不主导，且 `服务差值 > 0`
   - 优先选择 `服务需求 / 服务根任务`

4. 两侧差值都为 `0`
   - 保持当前主任务
   - 若当前主任务为空，再退回 `服务根任务`

5. 两侧同时为正且接近
   - 安全优先
   - 但不改写“服务是总方向、安全是保障方向”的规范边界

说明：

- `需求权重` 仍可保留，但退化为镜像与展示层
- 真正的根选择依据改为 `方向锚点`

完成判据：

- 当前主任务切换可由方向差值解释
- 同一轮里“面板显示方向差值”和“主任务实际选择”能互相对上


### 阶段 D：根层结果与锚点分层收口

目标：

- 明确：
  - 什么写进方向锚点
  - 什么只写进根层重判结果

方向锚点只承载：

- 当前值
- 目标值
- 差值
- 极性
- 计算时间
- 来源活动锚点

根层重判结果继续承载：

- 当前缺口类型
- 当前功能域
- 当前下一步去向
- 最近总控结果
- 最近反馈类型

完成判据：

- 不再把根层结果字段混写进方向锚点
- 方向锚点与重判结果能分别解释“为什么选这条方向”和“本轮最终怎么判”


### 阶段 E：锚点切换规则预留口

目标：

- 先不做热切换实现
- 但把切换条件和接口留出来

预留触发来源：

- 宿主重建
- 恢复快照回接
- 多自我实例切换
- 锚点失效后重建

完成判据：

- 代码里有统一切换入口或占位接口
- 后续不会把“重新同步”和“切换活动锚点”混为一谈


## 6. 验证方案

### 6.1 验证原则

本轮验证不恢复旧烟测大包，只保留：

- 可复现
- 可脚本化
- 可通过正式宿主路径执行

验证入口统一优先使用：

`D:\鱼巢\x64\Debug\鱼巢.exe --console-only --no-webview`


### 6.2 验证组 A：初始化与回链合法性

目标：

- 验证锚点建得出来，而且链条完整

检查项：

1. `活动自我锚点` 非空
2. `主方向锚点` 非空
3. `活动自我锚点 -> 自我存在指针` 非空
4. `活动自我锚点 -> 自我内部世界指针` 非空
5. `活动自我锚点 -> 宿主现实场景指针` 非空
6. `活动自我锚点 -> 主方向锚点指针` 与 `自我存在 -> 主方向锚点` 一致

通过标准：

- 控制台摘要显示 `活动锚点: 已建立`
- 无锚点缺失类错误


### 6.3 验证组 B：方向重算正确性

目标：

- 验证方向锚点不是静态值，而是随自我状态变化重算

检查项：

1. 记录一轮的：
   - `服务值`
   - `安全值`
   - `主方向差值`
   - `安全保障差值`
2. 让自我线程继续运行若干 tick
3. 再读一次上述值
4. 校验：
   - `主方向差值 = 目标服务值 - 当前服务值`
   - `安全保障差值 = 目标安全值 - 当前安全值`

通过标准：

- 差值变化与值变化方向一致
- 不出现“值已变化但方向锚点未变”


### 6.4 验证组 C：根任务选择正确性

目标：

- 验证根裁决确实开始读锚点，而不是只靠旧权重逻辑

最小用例：

1. `安全保障差值 > 服务差值`
   - 应选 `安全根任务`

2. `服务差值 > 安全保障差值`
   - 应选 `服务根任务`

3. 两侧差值相近或同为正
   - 默认安全优先

4. `安全值 <= 0`
   - 应进入停止/边界处理

通过标准：

- 当前主任务与预期一致
- 控制台摘要或调试日志能解释选择原因


### 6.5 验证组 D：锚点失效回退

目标：

- 验证锚点链断裂时不会静默继续执行

最小用例：

1. 人工清空 `自我存在 -> 主方向锚点`
2. 触发一轮治理
3. 观察系统行为

允许结果：

- 自动重建成功并继续
- 重建失败后进入边界处理

不允许结果：

- 锚点缺失但仍直接进入普通执行


### 6.6 验证组 E：展示一致性

目标：

- 验证控制台、控制面板和实际决策对同一份锚点状态保持一致

检查项：

1. 控制台摘要显示的：
   - `活动锚点`
   - `主方向差值`
   - `保障差值`
2. 控制面板显示的同名字段
3. 当前主任务实际选择

通过标准：

- 三者互相不矛盾


## 7. 执行顺序

建议严格按以下顺序推进：

1. 阶段 A `锚点读取正式化`
2. 阶段 B `活动自我锚点合法性前置校验`
3. 阶段 C `主方向锚点驱动根任务选择`
4. 阶段 D `根层结果与锚点分层收口`
5. 阶段 E `锚点切换规则预留口`

不建议跳序：

- 若先改任务选择，再补合法性校验，很容易把坏锚点带进主循环
- 若先做锚点切换，再做根裁决接线，会把问题扩大成多实例边界问题


## 8. 本轮完成定义

本计划对应的“锚点真正应用起来”完成标准是：

1. 根裁决已正式读取方向锚点
2. 主任务选择已正式受方向锚点驱动
3. 活动自我锚点断裂时有明确回退策略
4. 控制台与控制面板可以解释当前选择
5. 根层结果字段与锚点字段保持分层

如果只做到：

- 已创建
- 已展示
- 已同步

但还没有影响主任务选择，则只能算“锚点承载层完成”，不能算“锚点应用完成”。
