## 3. 统一口径

必须固定以下 9 条：

1. `学习任务` 不是独立根任务，而是 `任务管理任务派生并直接管理的内部治理子任务`。
2. 学习对内服务于 `方法资产治理`，对外服务于 `更好完成当前任务及同类任务`。
3. 学习不是“有原料就学”，而是必须走：`目标 -> 对象 -> 产出 -> 回流`。
4. 正式学习对象统一收口为 `方法`；`因果信息 / 成功执行片段 / 稳定任务路径 / 稳定任务子树` 都只是 `方法原料`。
5. 学习结果先进入 `方法存在面 / 证据面 / 补全面 / 资产面`，是否允许正式回流仍由外层重判决定。
6. 学习执行必须按 `方法主影响面` 分型。
7. 若方法主结果只影响 `自我的真实特征值 / 自我运行真值`，学习阶段默认只能做 `影子验证 / 镜像验证 / 等价验证`。
8. 若方法主结果会影响 `现实对象 / 外设 / 对外输出`，学习阶段默认只能做 `受控微试探`，并显式限制 `作用对象 / 幅度 / 时长 / 次数 / 停止条件`。
9. `受控` 先由 `任务管理任务` 完成治理判定，`试探` 再由学习执行器做真实执行；两者不得混成一层。

一句话收口：

`学习是任务管理任务派生的内部治理子任务；它通过补出、修复、验证和提升方法资产，来提高任务完成的可行性、稳定性与覆盖面。`


## 4. 规范闭环判断

按现有规范，学习主链已经闭环，标准链固定为：

```text
根层方向裁决
-> 形成任务管理上下文
-> 识别方法缺口 / 方法需求位
-> 上浮学习需求
-> 生成学习目标
-> 选择学习对象
-> 把方法原料抽象成待学习方法
-> 进入独立学习任务实体
-> 学习调度器排序 / 去重 / 合并 / 分派
-> 共享学习执行器执行
-> 产出学习结果
-> 结果先写方法资产 / 证据面
-> 学习结果回到学习任务实体
-> 返回外层重判
-> 决定正式回流 / 匹配回流 / 仅沉淀资产
```

因此规范层面的结论是：

- `学习逻辑在规范上已经闭环`
- 当前不再缺“学习是什么”的定义
- 当前真正缺的是“把规范闭环完整落到代码”

## 7. 分阶段实施步骤

本计划拆成 `L0-L8` 九步，必须顺序推进。

交付：

- 无新增学习规则继续长回 `自我类`

### L1. 补全学习任务实体

目标：

- 让每条学习需求都成为完整可追溯的独立学习任务实体

实施点：

- 在 [学习类.ixx](D:/鱼巢/学习类.ixx) 补齐：
  - 来源根任务
  - 来源宿主任务
  - 来源步骤
  - 来源方法
  - 来源结果
  - 来源方法需求位
  - 学习类型
  - 学习目标类型
  - 目标约束强度
  - 回流策略
  - 结果归属范围
  - 影响面类型
  - 等待原因

交付：

- 每条学习任务都能单独回答“为谁学、学什么、回给谁”

### L2. 把 `目标 -> 对象 -> 产出 -> 回流` 做成统一入口

目标：

- 把规范中的四段正式落到代码里

建议新增入口：

- `构建学习任务实体(...)`
- `判定学习目标(...)`
- `选择学习对象(...)`
- `定义学习产出(...)`
- `判定学习回流策略(...)`

交付：

- 学习不再只是“有请求就进执行器”
- 无对象、无骨架、无可回流结果时都能显式记录原因

### L3. 把待学习方法骨架变成强前置门

目标：

- 没有 `动作骨架 + 条件侧` 的对象，不能进入正式学习执行

实施点：

- 正式区分：
  - `方法原料`
  - `待学习方法骨架`
  - `正式学习对象`
- 若只有缺方法结论但没有骨架：
  - 只允许进入 `挂起待观察`
  - 不允许进入 `可执行学习`

交付：

- 学习执行入口只接受方法或待学习方法骨架

### L4. 落实学习执行影响面分型

目标：

- 把新增规范边界落成正式执行路径

最少分两类：

1. `自我真值型学习执行`
- 只影响自我的真实特征值与运行真值
- 默认执行策略：`影子验证 / 镜像验证 / 等价验证`
- 正式执行载体固定为：`挂在世界树根链上的纯想象镜像世界`
- 该镜像世界可随时创建、销毁或复用
- 镜像世界中的状态变化不得直接带回现实世界
- 允许回带的只有 `方法资产与待验证方法`

2. `现实影响型学习执行`
- 会影响现实对象、外设、对外输出
- 默认执行策略：`受控微试探`
- `受控` 判定责任归 `任务管理任务`
- `试探` 执行责任归 `学习执行器`
- 受控评判输入固定为：`整次试探执行中产生并最终保留下来的结果状态集合`
- 不得只按 `最后一步返回状态` 判定是否可接受

实施点：

- 在 [学习类.ixx](D:/鱼巢/学习类.ixx) 新增影响面判定
- 在任务管理学习承接链补：`受控微试探放行判定`
- 为学习执行器新增：
  - 挂在世界树根链上的纯想象镜像世界
  - 基于镜像世界的验证路径
  - 受控微试探约束字段
  - 基于受控边界的真实试探执行路径

交付：

- 不再把所有学习执行都当成同一种普通执行

### L5. 把共享学习执行器补成真实五阶段

目标：

- 让 `采样 / 更新 / 验证 / 提交 / 回滚` 从阶段名变成真正业务阶段

实施点：

- 在 [学习类.ixx](D:/鱼巢/学习类.ixx) 为每一阶段补：
  - 输入
  - 中间结果
  - 失败条件
  - 阶段摘要
  - 下一阶段迁移条件
- 为 `受控微试探` 补：
  - 任务管理任务侧的受控放行/拒绝判定
  - 结果状态集合收集
  - 中间产生且最终保留状态的入集合规则
  - 综合分叠加规则
  - 基于综合分的可接受性判定
  - 受控边界内的真实试探执行

交付：

- 执行器不再是“有对象就直接完成、没对象就等待”的最小占位逻辑

### L6. 拆开“方法资产写入”和“正式回流生效”

目标：

- 让学习结果先写方法资产，再由外层决定是否正式回流

实施点：

- 先建立 `方法资产写入层`
  - 条件节点
  - 结果节点
  - 方法账本
  - 可靠度 / 稳定度
  - 候选方法 / 临时方法 / 方法虚拟存在
- 再建立 `正式回流判定层`
  - 是否建议直接回流
  - 是否匹配后回流
  - 是否仅沉淀资产

交付：

- 学习结果不再混成“一次完成消息”

### L7. 外层正式消费学习结果

目标：

- 把学习线程结果收成真正的承接层结果，再交主循环拍板

实施点：

- [自我_学习承接模块.ixx](D:/鱼巢/自我_学习承接模块.ixx)
  - 继续承接回流消息消费

  - 根层重判
  - 决定是否正式回流
  - 决定是否唤醒来源任务
  - 决定是否更新治理记忆

交付：

- 学习结果不再越权直接生效

### L8. 学习树与审计收口

目标：

- 把当前最小任务表/等待表，升级成规范里的结构挂载与审计面

- 建 `任务管理学习树`

  - 学习任务表
  - 就绪队列
  - 等待表
  - 当前阶段
  - 回流建议
  - 影响面类型

交付：

- 学习链能从结构、状态、日志三侧被完整审计


## 8. 详细实施顺序建议

推荐顺序固定为：

1. `L1 学习任务实体补全`
2. `L2 目标/对象/产出/回流统一入口`
3. `L3 待学习方法骨架强前置`
4. `L4 学习执行影响面分型`
5. `L5 共享学习执行器五阶段化`
6. `L6 资产写入层 / 正式回流层拆分`
7. `L7 外层正式消费学习结果`
8. `L8 学习树与审计`

原因：

- 前三步先收“语义闭环”
- 中三步再收“执行闭环”
- 后两步最后收“回流闭环”和“可审计闭环”


## 9. 阶段验收标准

### 9.1 语义闭环验收

必须同时满足：

- 每条学习任务都有独立实体
- 每条学习任务都有 `目标 / 对象 / 产出 / 回流` 四段摘要
- 学习原料不再直接当正式学习对象执行

### 9.2 执行闭环验收

必须同时满足：

- 学习执行器存在真实五阶段
- 自我真值型学习默认走影子/镜像/等价验证
- 现实影响型学习默认走受控微试探
- `受控` 与 `试探` 已分层：前者归任务管理任务，后者归学习执行器
- 受控微试探的可接受性按 `整次试探结果状态集合` 判定，而不是只看最后一步
- 方法资产允许正常写入

### 9.3 回流闭环验收

必须同时满足：

- 学习结果先回学习任务实体
- 学习结果先写方法资产，再由外层决定是否正式回流
- 外层仍执行 `根层重判 -> 上下文刷新 -> 决定回流`

### 9.4 审计闭环验收

必须同时满足：

- 控制面板能看见学习任务表 / 就绪 / 等待 / 完成 / 失败
- 日志能看见：
  - 学习任务创建
  - 学习对象判定
  - 影响面类型
  - 阶段推进
  - 资产写入
  - 回流建议
  - 外层正式回流判定

当前学习系统的判断应统一为：

- `规范上已闭环`
- `代码上已有最小闭环`
- `工程上尚未完成完整闭环`

后续真正要做的，不再是重新定义“学习是什么”，而是把现有规范已经定下来的：

- `目标`
- `对象`
- `产出`
- `回流`
- `执行影响面分型`
- `资产写入`
- `外层正式生效`

这 7 个关键环，按本计划逐步落到代码里。
