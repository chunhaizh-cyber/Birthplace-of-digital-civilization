## 4. 总体实现判断

这次实现不能只做“多打一条日志”，必须把线程状态变化正式纳入：

`线程子存在`
-> `线程特征变化`
-> `状态变化消息`
-> `自我线程统一消费`
-> `动作动态`
-> `自我内部世界证据`

后续实施必须坚持三条总原则：

1. `先立线程存在面，再立消息合同`
   - 线程先是自我的子存在，消息只是这个子存在变化的运输层。
2. `先立统一状态变化入口，再补各线程发件人`
   - 必须先有统一 helper，避免每个线程自己拼摘要字符串。
3. `先沉动作动态，再扩因果与面板`
   - 第一轮先把消息沉成动态和状态证据，不在这一轮强行扩成完整因果链。
4. `影响评估另立计划，不在本文硬并`
   - `二次特征 / 影响项 / 服务安全摘要 / 根层重判接线` 属于下一份更复杂的后续计划，不在本文当前范围里强行并进。

还必须追加一条硬约束：

5. `线程只发消息，不直接改自我的状态`
   - 线程侧只允许创建线程子存在、整理状态变化事实并投递消息。
   - 自我内部世界中的线程状态特征、状态节点、动作动态，只能由 `自我线程` 在消费消息时统一落账。


## 5. 目标架构

### 5.1 线程存在面

每个线程都必须先在 `自我的内部世界` 中拥有一个正式存在承载。

推荐最少覆盖：

1. `自我线程`
2. `任务管理工作线程`
3. `动作线程`
4. `学习线程`
5. 后续其它项目内治理线程
6. 需要线程池承载时，对应 `线程池存在`

线程子存在的父子关系应固定为：

- 非池化线程：父节点 = `自我内部世界`
- 线程池存在：父节点 = `自我内部世界`
- 池化工作线程：父节点 = `所属线程池存在`
- 子节点内部世界：可选；第一轮不要求每个线程都建独立内部世界
- 动态沉淀位置：统一先落在 `自我内部世界`

必须固定：

1. 线程状态变化的主体，不是“日志系统”，而是“某个线程子存在”。
2. 线程标签、线程种类、拥有者、运行句柄映射，都属于线程子存在的识别面。
3. 若某工作线程属于某个线程池，则该线程池本身也必须先有一个正式存在，不能只留工程壳对象。
4. 线程子存在必须可按 `主键 / 标签 / 线程种类` 三种方式回查。

### 5.2 线程特征面

每个线程子存在的特征面，应先拆成三层：

1. `核心标准特征`
   - `生命周期状态`
   - `控制意图`
   - `实际运行状态`
2. `通用扩展特征`
   - `健康状态`
   - `当前预算`
   - `绑定任务`
   - `阻塞状态`
3. `线程类型专属特征`
   - 例如 `自我线程` 可再加 `运行模式 / 邮箱深度`
   - 例如 `工作线程` 可再加 `心跳健康`

必须固定：

1. 每个线程子存在至少具备 `3` 类核心标准特征，不得再少。
2. `健康状态 / 当前预算 / 绑定任务 / 阻塞状态` 按适用性挂接，不要求所有线程挂满。
3. 当前阶段不设置线程特征理论上限，但新增特征不得混入 `任务 / 结果 / 场景 / 审计` 等非线程本体语义。
4. `时间 / 决策者 / 原因 / 旧状态 / 新状态` 属于 `状态变化记录字段` 与 `动作动态字段`，不直接冒充线程本体标准特征。
5. 若需要读取“最近一次状态变化摘要”，应优先读最近 `线程状态变化消息` 或对应 `线程状态变更动作动态`，而不是把这类审计信息硬塞回线程本体特征。

第一轮最重要的是把以下几项先接稳：

1. `生命周期状态`
2. `控制意图`
3. `实际运行状态`
4. 按适用性补 `健康状态`
5. 按适用性补 `绑定任务` 或 `阻塞状态`

### 5.3 统一线程状态变化消息

任一线程状态变化时，都应向自我线程发送一条统一治理消息。

推荐新增一段正式载荷：

`结构_线程状态变化消息段`

最少字段应包含：

1. `发生时间`
2. `决策者类型`
3. `决策者主键`
4. `决策者标签`
5. `触发动作`
6. `原因键`
7. `原因摘要`
8. `线程存在主键`
9. `线程标签`
10. `线程种类`
11. `直属拥有者主键`
12. `旧状态`
13. `新状态`
14. `是否故障触发`
15. `是否 stop 链触发`
16. `关联任务主键` 或 `关联线程主键`

你指定的核心字段必须无条件纳入：

1. 时间
2. 是谁做出决定
3. 什么原因
4. 哪个线程
5. 原来状态
6. 最新状态

### 5.4 动作动态沉淀规则

自我线程收到线程状态变化消息后，必须做三件事：

1. 由自我线程更新线程子存在当前状态特征
2. 为旧状态和新状态形成对应状态节点或状态镜像
3. 在 `自我内部世界` 创建一条 `线程状态变更动作动态`

这条动作动态的推荐语义固定为：

- 动作主体：线程子存在
- 记录者：自我线程
- 动作语义键：`线程状态变更`
- 条件状态：旧状态
- 结果状态：新状态
- 附加说明：决策者、原因、关联对象、发生时间

必须固定：

1. 线程状态变化消息不能只写摘要，不沉动态。
2. 动态不能只挂“线程名”，必须能回到线程子存在和状态节点。
3. 第一轮先沉 `动作动态`，因果实例可作为第二轮增强项。

## 8. 风险点与约束

这次实现最容易踩的坑有五个：

1. `只打日志，不建线程子存在`
   - 这样最终还是无法进入世界树证据面。
2. `只改状态，不发消息`
   - 这样会让部分线程继续静默切换。
3. `一个状态切换被父线程和子线程各报一次`
   - 必须统一由状态切换 helper 负责去重。
4. `把自我线程自己的状态变化也硬塞进普通治理主链`
   - 这会把 `关键日志 / 宿主级审计例外` 和“对自我有治理意义的其它线程状态变化”重新混成一锅。
5. `只沉摘要，不沉动态`
   - 这会让后续因果、自然语言与可观测性链条断掉。
