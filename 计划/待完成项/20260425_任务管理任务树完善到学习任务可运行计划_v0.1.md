# 2026-04-25 任务管理任务树完善到学习任务可运行计划 v0.1

## 1. 计划目标

本文只回答一件事：

`如何借鉴现有代码中的旧治理逻辑，把任务管理任务树完善到能够稳定派生需求、由自我线程建任务、由工作线程执行步骤，并最终让学习任务真正运行起来。`

本文不是新的规范，不负责改写上位职责边界。

本文把当前仓库中的旧逻辑拆成可施工的阶段，并按最新思路重新落到任务管理任务树：

1. 任务管理任务顶层仍只有 `筹办` 和 `执行` 两个步骤。
2. `筹办 / 执行 / 反馈` 是任务执行逻辑，也是任务管理任务的方法逻辑。
3. 自我线程负责同步需求并创建任务，任务一定只在自我线程新建。
4. 任务管理任务界面线程负责任务管理任务工作线程调度、消息收取和任务管理任务状态改变。
5. 工作线程只负责执行当前步骤的方法，并返回两类数据：`方法实际数据` 和 `任务管理任务数据`。
6. 任务管理任务的目标是让其管理的任务状态变为完成；普通任务的目标是达到需求；学习任务的目标是让方法虚拟存在变可用。
7. 任务管理任务的虚拟存在和其参数任务的虚拟存在，在治理该参数任务时应视为同一份任务状态观察面。


## 2. 上位边界

本计划必须服从当前最新口径：

1. 任务管理任务不能绕过自我线程直接创建普通子任务。
2. 筹办遇到缺项，只向自我线程提需求。
3. 执行方法时返回的任务，也只能作为需求或前置任务信息回报，由界面线程和自我线程处理，不由工作线程直接挂正式任务。
4. 筹办新增步骤时必须同步生成对应需求，否则需求树和任务树后续会错位。
5. 同类型、同来源、同状态语义的子需求，在同一子链上只能出现一次。
6. 当任务衍生出新需求后，父任务应暂停，等待新需求满足或对应子任务完成调度后再继续。
7. 新需求对应的子任务优先级应高于父任务；优先级使用现有选择器机制，不再回退到旧数值拼接口径。
8. 控制面板显示真实树结构和字段信息，不做旧摘要加工。

## 5. 分阶段施工计划

### 阶段 0：固定施工保护线

目标：

确保后续改动不会回到旧上下文旁路。

施工项：

1. 搜索并标记剩余直接创建业务子任务、补条件子任务、学习子任务的旧路径。
2. 将这些路径按结果分为三类：可直接删除、改为提需求、暂时保留但加日志警戒。
3. 对任务管理任务模块增加日志关键字：
   `筹办提需求`、`自我线程建任务`、`前置任务转步骤`、`学习任务启动`、`学习回流唤醒`。
4. 控制面板增加原始字段观察项，不再依赖旧摘要解释任务树。

验收：

1. 搜索 `创建或复用学习子任务头`、`已创建业务子任务`、`已创建补条件子任务` 等旧动作，能明确知道哪些仍在用、哪些应迁移。
2. `msbuild` 通过。
3. 摘要命令能看到日志链路，不再只看到任务数增长。


### 阶段 1：筹办输出结构化需求

目标：

让筹办遇到缺项时，只产生结构化需求，不直接建任务。

借鉴旧逻辑：

1. 用 `刷新任务管理方法需求位` 判断缺口属于补结构、选择推进、分解补条件、求证判定、门控转域、失败策略、收束复盘。
2. 用 `评估任务管理本能能力缺口` 判断缺观察原语、缺写回原语、缺挂点原语、缺动作边界、缺状态差分取证、缺实例因果取证、缺抽象因果归纳、需学习补齐。
3. 用 `任务治理二次特征证据` 填充等待、收束、重筹办、回接、可继续推进等判据。

施工项：

1. 定义 `结构_任务管理筹办需求输出`，至少包含：
   `需求类型`、`来源任务头节点`、`来源步骤节点`、`被需求当前状态`、`需求状态`、`方法需求位`、`本能能力缺口类型`、`主观察特征`、`候选方法节点`、`摘要`。
2. 筹办新增步骤时，必须同步产生对应需求输出。
3. 缺方法时产生 `补方法需求`，其需求状态来自方法结果特征类型对应的状态。
4. 缺任务步骤时产生 `任务步骤需求`，其需求状态来自任务状态特征。
5. 缺前置任务时产生 `前置任务需求`，并绑定来源步骤。
6. 筹办输出需求后，当前父任务进入暂停或等待状态，等待自我线程回执。

验收：

1. 需求节点字段尽量填满，不再只有类型和主键。
2. 同类同来源同状态语义需求不会在同一子链重复增长。
3. 任务管理摘要中需求增长可解释，每个新增需求都有来源步骤或来源任务头。


### 阶段 2：自我线程同步需求并创建任务

目标：

把 `需求 -> 任务头` 的唯一创建权固定到自我线程。

借鉴旧逻辑：

1. 自我线程已有消息处理、批次执行和需求候选治理的雏形。
2. 任务管理任务模块已有“相关任务为空则需要新建任务头”的设计口径。

施工项：

1. 在自我线程主循环增加或强化 `同步需求并创建任务` 步骤。
2. 扫描正式需求树，找到 `相关任务` 为空的需求节点。
3. 为每个未挂任务需求创建任务头。
4. 新任务主信息必须回填：
   `来源需求`、`父任务`、`来源任务头`、`来源步骤`、`目标状态`、`基准/选择器优先级`、`场景`。
5. 创建成功后，回填需求的 `相关任务` 字段。
6. 如果需求来自筹办新增步骤，则该任务后续应挂入任务树对应步骤链。
7. 如果需求来自执行方法返回的前置任务，则该任务先挂入来源步骤的前置任务字段。

验收：

1. 任务头只由自我线程创建。
2. 需求 `相关任务` 不再长期为空。
3. 同一需求不会重复创建任务头。
4. 新子任务优先级高于父任务，并使用现有选择器机制。


### 阶段 3：前置任务转步骤

目标：

让前置子任务最终成为任务树中的步骤，而不是游离的任务头。

借鉴旧逻辑：

1. 任务树已有步骤节点、结果节点、所属任务头、当前步骤、后续任务、前置任务字段。
2. 旧步骤结果子链可以借鉴，但不能加工成摘要树。

施工项：

1. 定义 `前置任务转步骤` 规则：
   任务树根链上的前置任务，在满足挂接条件后，转换为父任务树中的步骤节点。
2. 步骤节点字段至少填充：
   `来源需求`、`来源任务头`、`来源步骤`、`前置任务头`、`执行方法`、`目标状态`、`所属任务头结点`。
3. 父任务暂停期间，只允许调度其前置子任务或学习任务。
4. 前置子任务完成后，父任务重新进入筹办，重新判断是否还缺步骤、方法或条件。
5. 控制面板直接显示真实 `并发前置任务头结点列表`、`当前前置任务头结点`、`所属任务头结点` 等字段。

验收：

1. 前置任务不会只挂在根链上失联。
2. 父任务在前置任务未完成时不会继续空转执行。
3. 控制面板能沿字段看出：需求 -> 任务头 -> 前置任务 -> 步骤。

3. 自我线程根据学习类需求创建学习任务头。
4. 学习任务头必须关联：
   `来源需求`、`来源父任务`、`来源步骤`、`目标方法`、`方法需求位`、`本能能力缺口类型`、`学习类型`、`回流目标方法位`。
5. 学习任务同步到学习调度账本。
6. 父任务等待学习任务完成，不再重复派生同类学习需求。

验收：

1. 学习账本出现来自需求树的学习任务。
2. 学习任务可以被 `选择一条可执行学习任务` 选中。
3. 控制面板能从补方法需求追踪到学习任务和目标方法。


### 阶段 6：学习任务真正执行

目标：

让学习任务通过工作线程或学习执行模块真正推进方法虚拟存在。

借鉴旧逻辑：

1. `学习执行模块::执行一步学习任务` 已具备执行策略。
2. `私有_同步学习任务树状态` 已具备学习状态与任务树同步能力。
3. `提交学习方法资产` 已具备写回学习方法骨架、条件、结果、P2 证据索引和 M10 主体策略沉淀能力。

施工项：

1. 明确学习任务当前步骤的方法：
   `补动作骨架`、`补条件节点`、`补结果节点`、`执行沙箱验证`、`计算二次特征`、`正式资产提交`。
2. 学习任务执行时，修改方法虚拟存在特征状态，而不是修改普通任务状态。
3. 学习执行结果写回：
   `学习任务状态`、`学习方法结构`、`方法可用状态`、`学习反馈摘要`、`目标方法结果节点`。
4. 学习完成后，调用或复用 `提交学习方法资产`，正式补齐方法结构。
5. 学习完成后，向任务管理任务界面线程或自我线程发送学习回流消息。

验收：

1. `执行一步学习推进` 能实际执行学习任务。
2. 学习任务从待执行进入执行中，再进入完成或等待。
3. 方法节点出现动作骨架、条件节点、结果节点或证据索引变化。
4. 父任务能因学习回流被唤醒。


### 阶段 7：学习回流唤醒父任务并重新筹办

目标：

让学习任务完成后，父任务能重新选择方法并继续执行。

借鉴旧逻辑：

1. M10 的正式学习反馈沉淀可继续负责主体策略和方法资产回流。
2. M2 筹办链可继续负责重新判断方法是否可用。

施工项：

1. 学习回流消息必须带：
   `来源需求`、`学习任务`、`父任务`、`来源步骤`、`目标方法`、`方法可用状态`、`学习反馈摘要`。
2. 自我线程收到学习回流后，更新需求满足状态和相关任务状态。
3. 任务管理任务界面线程收到学习回流后，将父任务从等待/暂停转入待重筹办。
4. 下一轮筹办按任务目标状态重新查找方法。
5. 如果找到多个同方向方法，按任务状态出现顺序和更精准结果状态选择方法。

验收：

1. 父任务不会永久等待学习任务。
2. 补方法完成后，父任务能重新选中已补齐方法。
3. 普通任务执行不再因同一缺方法反复生成学习需求。


### 阶段 8：普通任务执行闭环

目标：

让“缺方法 -> 学习 -> 回流 -> 执行普通任务 -> 完成任务状态”形成闭环。

施工项：

1. 普通任务执行只改变目标需求对应的特征状态。
2. 任务管理任务根据目标状态和结果状态判断任务是否完成。
3. 如果普通方法执行又派生新需求，父任务暂停并回到阶段 1。
4. 如果普通方法执行成功，任务结果状态回填到任务虚拟存在。
5. 若任务状态达到完成，任务管理任务本轮执行结果为完成。

验收：

1. 从一个没有可用方法的需求开始，系统能自动产生学习需求。
2. 自我线程创建学习任务。
3. 学习任务执行并补齐方法。
4. 父任务重新筹办并执行普通方法。
5. 任务状态最终变为完成。

## 8. 风险与控制

### 8.1 需求继续无限增长

风险：

筹办每轮都生成语义相同的新需求。

控制：

需求去重必须按 `类型 + 来源任务头 + 来源步骤 + 被需求当前状态语义 + 需求状态语义` 判断。


### 8.2 任务绕过自我线程创建

风险：

旧函数仍直接创建学习子任务、业务子任务或补条件子任务。

控制：

所有创建任务头的入口必须收口到自我线程同步需求步骤。任务管理任务模块只能输出需求或前置任务数据。


### 8.3 父任务空转

风险：

派生新需求后父任务继续执行，导致任务数量或需求数量漂移。

控制：

一旦派生新需求或前置任务，父任务进入等待/暂停，直到需求满足、相关任务完成或学习回流。


### 8.4 学习任务只入账不执行

风险：

学习任务进入账本，但没有可执行步骤或方法。

控制：

学习任务建头时必须同步当前学习步骤和可执行学习方法；学习调度必须能选中该任务；学习执行必须能改变方法虚拟存在状态。


### 8.5 方法补齐后父任务不醒

风险：

学习任务完成后，父任务仍保持等待。

控制：

学习回流必须同时更新需求状态、学习任务状态、目标方法可用状态，并向任务管理任务界面线程触发父任务重筹办。


## 9. 最小闭环用例

建议用一个最小用例贯穿所有阶段：

1. 创建一个需求，其目标状态需要某种 `任务_状态` 或普通特征状态。
2. 自我线程发现该需求无相关任务，创建任务头。
3. 任务管理任务筹办该任务，发现无可用方法。
4. 筹办生成 `补方法需求`，父任务暂停。
5. 自我线程根据补方法需求创建学习任务。
6. 学习调度选中学习任务。
7. 学习执行补齐目标方法的动作骨架、条件和结果。
8. 学习回流更新方法虚拟存在状态。
9. 父任务被唤醒并重筹办，选中已补齐方法。
10. 工作线程执行父任务当前步骤方法。
11. 方法实际结果改变目标特征状态。
12. 任务管理任务判断父任务状态完成。


## 10. 完成判据

本计划完成时，应满足：

1. 任务管理任务顶层仍只有 `筹办` 和 `执行` 两个步骤。
2. 旧 P0/P1/P2/M2/M9/M10 中仍需要的函数，已经转为本能方法函数或被本能方法组织，不再以旧上下文旁路扩散。
3. 筹办所有缺项都转为结构化需求。
4. 任务只在自我线程创建。
5. 前置任务能转为父任务树步骤。
6. 工作线程只执行当前步骤方法并返回双结果。
7. 学习任务能由补方法需求创建、被调度、被执行、写回方法虚拟存在。
8. 学习回流能唤醒父任务重筹办。
9. 一个缺方法任务能最终完成任务状态。
