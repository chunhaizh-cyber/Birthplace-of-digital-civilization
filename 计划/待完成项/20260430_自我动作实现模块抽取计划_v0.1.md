## 1. 总目标

新增 `自我动作实现模块.ixx`，把当前散落在初始化流程中的默认本能动作函数实现集中管理，并同步调整方法执行前条件匹配逻辑。

拟新增文件：

- `D:\鱼巢\自我动作实现模块.ixx`

拟导出模块：

```cpp
export module 自我动作实现模块;
```

融合后的主线：

1. 将默认本能动作的具体函数体从 `自我初始化模块.cpp` 中抽出。
2. 初始化流程只负责调用注册入口，不再承载大段动作实现。
3. 动作实现统一适配 `本能动作函数` 返回 `存在节点类*` 的口径。
4. 动作实现支持“无参自描述”，用于补齐方法虚拟存在中的条件特征类型信息。
5. 方法虚拟存在初始化时同步写入通用特征信息和方法区分信息。
6. 方法虚拟存在成为方法条件类型信息的权威承载处。
7. 执行桥执行前读取条件特征类型信息，绑定本轮具体值，再调用本能动作管理类。
8. 本能动作管理类保持窄职责，不成为条件类型权威来源。
9. 模块名可以区分归属；函数名、动作名、特征类型名使用通用语义，不刻意添加主体前缀。

新的执行前条件链路：

```text
本能函数执行
-> 确保方法首节点、方法虚拟存在和本轮运行虚拟存在
-> 在方法虚拟存在中声明运行参数类型
-> 解析绑定本轮具体参数值
-> 执行功能或记录无法执行原因
-> 输出返回参数信息、返回场景信息和结果值
-> 返回本轮运行虚拟存在
```

## 2. 命名规则

- 函数、动作名、特征类型名、局部语义名优先使用通用词语，不刻意把主体前缀写进名称。

推荐：

- `自我_任务筹办`

说明：

- 旧枚举值 `枚举_本能方法ID::自我_...` 本轮不强制改名，避免扩大改动面。
- 如必须表达宿主主体，使用参数名或说明表达，例如 `主体对象`、`宿主存在`、`执行主体`，不放进动作名。

## 3. 当前问题

当前动作函数主要注册在：

- `D:\鱼巢\自我初始化模块.cpp`
- `D:\鱼巢\任务管理任务模块.impl.cpp`

其中 `自我初始化模块.cpp` 同时承担：

1. 初始化核心结构。
2. 注册默认本能方法。
3. 注册默认动作函数。
4. 实现任务筹办、任务执行、学习、安全值、服务值等动作函数。

当前条件匹配链路也存在问题：

1. 执行桥寻找方法条件节点。
2. 从方法条件节点或方法首节点读取条件场景。
3. 本能动作管理类使用 `输入参数场景` 匹配 `方法首节点.条件场景`。
4. 条件判定索引主要只参与“是否需要输入参数”的判断。

问题汇总：

- 初始化模块过宽。
- 部分注册 lambda 仍保留旧式 `结构_本能动作步骤结果` 返回思路。
- 条件类型信息散在方法条件节点和条件场景中，不够稳定。
- 方法虚拟存在没有明确承载“这个方法需要哪些输入特征类型”。
- 执行桥现在优先找方法条件节点；没有条件节点时，即使方法首节点已有条件场景，也可能被判为缺条件。
- 本能函数本身最清楚自己需要什么输入，但当前缺少“无参自描述并写入方法虚拟存在”的标准流程。
- 执行前匹配直接拿场景对场景做结构匹配，没有清晰区分“需要什么类型”和“本轮具体值是什么”。

## 4. 职责边界

### 4.1 `本能方法类`

继续负责：

- 本能方法 ID。
- 默认元信息表。
- 默认方法头注册。
- 方法树中的本能方法结构补齐入口。

不负责：

- 动作函数具体实现。
- 动作函数注册表。
- 动作执行调度。
- 本能函数无参自描述逻辑。

### 4.2 `方法虚拟存在`

负责保存方法自身的特征信息，其中包括通用特征信息、方法区分信息和条件类型信息。

初始化方法虚拟存在时必须同时写入：

| 信息组 | 用途 |
| --- | --- |
| 通用特征信息 | 支撑所有方法共用的运行、治理、学习、自检和统计链路。 |
| 方法区分信息 | 说明这个方法和其他方法有什么不同。 |
| 条件类型信息 | 说明执行前需要哪些输入特征类型。 |

通用特征信息至少包括：

| 信息 | 用途 |
| --- | --- |
| 方法是否可用 | 学习和筹办判断方法能否被使用。 |
| 方法可靠度 | 表示方法当前可靠程度。 |
| 方法类型 | 区分内部方法、外部方法等。 |
| 动作入口类型 | 区分本能函数、外部实现、动作序列、学习生成动作等入口来源。 |
| 成熟度阶段 | 表示候选、可学习、可召回、可默认执行等阶段。 |
| 允许自动召回 | 判断筹办是否可自动召回该方法。 |
| 允许默认执行 | 判断满足条件后是否可默认执行。 |
| 特征账本生命周期 | 支撑特征使用账本和淘汰保留策略。 |
| 最近运行状态 | 记录最近一次运行成功、失败、缺条件、缺方法等状态。 |
| 最近错误码 | 记录最近一次执行错误码。 |
| 最近输入场景 | 指向本轮输入或条件场景。 |
| 最近结果场景 | 指向本轮功能结果账本。 |
| 最近动作动态 | 指向本轮真实方法调用生成的动作动态。 |

至少应表达：

| 信息 | 用途 |
| --- | --- |
| 条件特征类型列表 | 该方法执行前需要读取哪些特征类型。 |
| 条件特征来源范围 | 特征值优先从哪里绑定，例如输入场景、任务虚拟存在、方法虚拟存在、现实目标存在。 |
| 条件是否必需 | 区分必需条件和可选条件。 |
| 条件默认值策略 | 缺值时是否允许默认值、保持值或失败。 |
| 条件结构标识 | 用于区分不同条件组合，使用结构标识口径。 |

### 4.3 `自我动作实现模块`

负责：

- 提供默认动作函数实现。
- 将具体动作实现转换为 `存在节点类*` 本轮运行虚拟存在。
- 支持本能函数自描述和运行参数声明。
- 写入方法虚拟存在上的条件特征类型信息。

不负责：

- 声明本能方法元信息。
- 管理动作注册表内部结构。
- 初始化世界树根骨架。
- 维护控制面板。
- 生成真实动作动态。

建议每个动作函数拆成两个私有 helper：

```text
描述条件类型(...)
执行功能动作(...)
```

### 4.4 `执行桥`

负责执行前条件绑定。

执行桥读取方法虚拟存在中的条件特征类型信息，然后按本轮上下文绑定具体值：

```text
方法虚拟存在.条件特征类型列表
-> 从输入场景 / 当前任务 / 目标存在 / 当前场景中查找同类型特征
-> 生成本轮输入参数场景
-> 传给本能动作管理类执行
```

执行桥不应只因为缺少方法条件节点就判定缺条件。真正缺条件应是：

```text
方法虚拟存在声明了必需条件类型
但执行桥无法从本轮上下文绑定出具体值
```

真实功能执行后，动作动态仍由执行桥生成。

### 4.5 `本能动作管理类`

继续负责：

- 注册动作函数。
- 查询动作函数。
- 检查动作是否可调用。
- 执行动作函数。
- 创建或读取标准运行虚拟存在。

不负责：

- 实现具体默认动作。
- 读取任务业务上下文并作任务裁决。
- 创建需求或推进任务。
- 生成动作动态。
- 成为条件类型信息的权威来源。

后续可把 `检查是否可调用(...)` 中的场景对场景匹配降级为：

1. 如果执行桥已经生成本轮输入参数场景，并标记绑定完成，则管理类信任执行桥。
2. 如果没有绑定完成标记，则管理类可做兼容场景匹配。

## 7. 返回口径整理

目标函数形态改为返回本轮运行虚拟存在：

```cpp
using 本能动作函数 = std::function<存在节点类*(
    方法类::节点类* 方法首节点,
    场景节点类* 输入参数场景,
    场景节点类* 输出结果场景)>;
```

动作实现内部可以继续先形成局部结算信息，但最终必须返回：

- 成功：本轮运行虚拟存在中写入运行状态 `0`。
- 失败：本轮运行虚拟存在中写入错误码。
- 特殊状态：本轮运行虚拟存在中写入已有 `本能方法结果_...` 结果码。
- 结构补齐完成：继续进入解析绑定、执行和输出，不把结构补齐当成成功出口。

本轮运行虚拟存在可看作任务虚拟存在的同类结构，承载结果值、错误码、输入场景、结果场景、返回参数信息、返回场景信息、开始时间、结束时间、耗时、动作动态入口等运行账本信息。函数功能结果仍写入结果场景。

## 8. 方法虚拟存在新增特征建议

新增特征类型使用通用命名，不加主体前缀。

本计划新增的条件类型特征必须和既有通用特征一起初始化，不得替代通用特征。

| 特征类型 | 值类型 | 说明 |
| --- | --- | --- |
| 方法条件特征类型集 | VecU 或引用列表 | 保存所需条件特征类型主键或引用。 |
| 方法必需条件特征类型集 | VecU 或引用列表 | 只保存必需条件。 |
| 方法可选条件特征类型集 | VecU 或引用列表 | 保存可选条件。 |
| 方法条件来源范围 | I64、文本或引用 | 表示输入场景、任务虚拟存在、方法虚拟存在、目标存在等来源优先级。 |
| 方法条件结构标识 | 文本或 VecU | 表示条件组合结构，用于复用和比较。 |
| 方法条件自描述状态 | I64 | 0 未描述，1 已描述，负值表示描述失败。 |
| 方法条件自描述时间 | 时间戳 | 最近一次本能函数写入条件类型信息的时间。 |

说明：

- “类型集”只保存需要哪些特征类型，不保存本轮具体值。
- 本轮具体值应进入执行桥生成的输入参数场景。
- 条件场景仍可作为可视化和样本承载，但不再作为唯一权威。

## 9. 无参自描述流程

### 9.1 触发时机

以下情况应触发本能函数无参自描述：

| 场景 | 说明 |
| --- | --- |
| 默认方法头注册后 | 确保方法虚拟存在具备条件类型信息。 |
| 执行桥发现条件类型信息为空 | 执行前先要求动作函数自描述一次。 |
| 学习补齐方法结构时 | 学习任务可调用目标动作的自描述路径补条件类型。 |
| 方法结构自检时 | 自检发现条件类型缺失时触发补齐。 |

### 9.2 执行步骤

```text
确保方法首节点
-> 确保方法虚拟存在
-> 本能函数写入条件特征类型信息
-> 写入方法条件自描述状态
-> 继续进入解析绑定、执行和输出
-> 返回本轮运行虚拟存在
```

### 9.3 边界

无参自描述不是一次真实功能动作。

因此：

- 不生成真实动作动态。
- 不改变目标状态。
- 不写功能结果事实。
- 可以写方法虚拟存在结构信息。
- 不能把结构补齐直接当成成功出口；仍要返回本轮运行虚拟存在。

## 10. 执行桥条件绑定流程

新的执行桥执行前流程：

```text
1. 确保方法首节点存在。
2. 确保方法虚拟存在存在。
3. 读取方法虚拟存在中的条件特征类型集。
4. 如果条件特征类型集为空：
   4.1 调用本能函数无参自描述。
   4.2 重新读取条件特征类型集。
5. 如果仍为空：
   5.1 若方法声明无条件，允许继续。
   5.2 若无法判断是否无条件，返回缺条件类型信息。
6. 根据条件来源范围，从本轮上下文绑定具体值。
7. 若必需条件绑定失败，返回缺条件。
8. 生成本轮输入参数场景。
9. 调用本能动作管理类执行功能函数。
10. 真实执行后由执行桥生成动作动态。
```

## 11. 条件绑定规则

执行桥绑定具体值时按来源优先级查找。

建议默认优先级：

| 顺序 | 来源 |
| --- | --- |
| 1 | 显式输入参数场景 |
| 2 | 当前任务绑定的任务场景 |
| 3 | 当前任务虚拟存在 |
| 4 | 来源需求指向的目标存在 |
| 5 | 当前方法虚拟存在 |
| 6 | 当前场景 |

绑定规则：

1. 对每个必需条件特征类型，必须找到一个同类型实例特征。
2. 对可选条件特征类型，找不到可以跳过。
3. 找到多个候选时，优先选择距离任务目标最近的来源。
4. 绑定结果应写入本轮输入参数场景。
5. 本轮输入参数场景只保存本轮具体值，不反写方法虚拟存在。

不新增：

- 具体条件类型语义判断。
- 本能函数自描述逻辑。

## 14. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 迁移后 include/import 变复杂 | 先只迁 `自我初始化模块.cpp` 中默认动作，不动任务治理动作 |
| 返回结构改为运行虚拟存在后丢失备注 | 本轮先保证结果值正确，备注迁入后续结果治理链路 |
| 旧枚举名仍带 `自我_` | 本轮不重命名枚举，避免大面积调用方变更 |
| 函数/特征命名继续过度主体化 | 模块名保留归属区分，函数名、动作名、特征类型名使用通用语义 |
| 动作实现仍读取主体对象 | 参数名使用 `主体对象`，动作名保持通用 |
| 无参自描述被误认为真实执行 | 明确不生成动作动态，不改变目标状态，不写功能结果事实 |
| 管理类继续承担条件权威 | 执行桥负责绑定，管理类只保留基础保护和兼容校验 |
| 条件类型为空导致无法判断无条件还是缺描述 | 增加 `方法条件自描述状态` 区分已描述无条件和未描述 |

## 15. 验收标准

必须满足：

- 新增 `自我动作实现模块.ixx`。
- 新模块提供默认动作注册入口。
- `自我初始化模块.cpp` 不再直接承载默认动作函数主体。
- 新增命名不使用刻意主体前缀。
- 新增命名统一使用“结构标识”等通用说法。
- 迁移范围内动作函数返回 `存在节点类*`。
- 方法头存在时，能确保方法虚拟存在存在。
- 方法虚拟存在初始化时能同时写入通用特征信息和方法区分信息。
- 对支持自描述的本能函数，无参执行后能写入条件特征类型信息。
- 条件特征类型信息存放在方法虚拟存在。
- 执行桥按条件特征类型绑定具体值。
- 缺少必需具体值时，执行桥返回缺条件。
- 无参自描述不生成真实动作动态。
- 真实功能执行仍由执行桥生成动作动态。
- `本能动作管理类` 职责不扩张。
- `本能动作管理类` 不成为条件类型权威来源。
- Debug x64 编译通过。
