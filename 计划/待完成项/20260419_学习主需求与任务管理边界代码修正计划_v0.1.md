# 20260419 学习主需求与任务管理边界代码修正计划 v0.1

## 1. 目标

本文用于把当前代码与最新规范之间的差异收口为一份可执行修正计划。

本轮修正只解决以下主线问题：

- `任务管理任务` 的目标、边界与完成判据，继续向最新规范收口。
- `学习任务` 的顶层需求，统一改为：`让当前待学习方法头可用`。
- `学习需求` 与 `学习动作` 分层，避免继续把动作名冒充顶层需求。
- `原语` 与 `本能函数` 的边界继续从“语义分开”推进到“数据结构与实现层分开”。
- `学习任务` 从“学习实体/阶段机为主”继续推进为：`唯一学习任务壳 + 工单/进度写入任务虚拟存在`。


## 2. 当前差异

### 2.1 已基本对齐

1. `任务管理工作线程` 与 `界面线程` 边界已基本明确。
2. `任务管理任务` 已形成 `治理证据 -> 原语解释 -> 单步决策 -> 写回` 主链。
3. `学习目标` 文案与摘要已经开始向 `让方法可用` 收口。

### 2.2 仍需修正

1. `学习需求类型` 仍使用旧动作型枚举：
   - `样本收集需求`
   - `差异比较需求`
   - `共同条件提取需求`
   - `候选动作生成需求`
   - `结果预测需求`
   - `沙箱验证需求`
   - `风险评估需求`
   - `边界标记需求`
   - `转正判断需求`

   这组名字不再适合作为顶层学习需求。

2. `学习目标类型` 仍保留旧口径：
   - `补全方法头节点`
   - `验证候选方法`

   它们应继续向 `让当前待学习方法头可用` 收口。

3. `任务管理最小原语` 在实现层仍复用 `枚举_本能方法ID`，边界尚未完全落成独立口径。

4. `任务管理任务` 的完成判据仍混合使用：
   - `宿主已完成`
   - `目标结果已对齐`

   还没有彻底收口为：
   - 终局目标：`宿主任务状态进入完成`
   - 中间判据：若未完成，则继续补齐结构/方法/学习/回接/等待。

5. `学习任务` 仍主要以 `学习实体表 + 阶段机` 为主，而不是：
   - `唯一学习任务壳`
   - `学习工单/进度`
   - `任务虚拟存在特征`


## 3. 修正原则

必须统一按以下口径修正：

1. `任务管理任务` 的唯一终局目标是：推动 `宿主任务状态进入完成`。
2. `任务管理任务` 和普通任务遵循同一主流程：

```text
筹办条件 -> 条件满足 -> 运行方法 -> 比对结果与目标 -> 满足则完成，不满足则继续完善任务树
```

3. `任务管理任务` 改写的是 `任务虚拟存在`；普通任务改写的是 `世界树`。
4. `任务管理任务` 使用的最小内部方法统一称 `原语`；普通任务使用的外部最小方法统一称 `本能函数`。
5. 顶层学习需求统一收口为：

```text
让当前待学习方法头可用
```

6. `样本收集/差异比较/...` 这组旧枚举，统一下沉为：
   - `学习动作`
   - 或 `原函数需求`
   - 或 `候选学习程序` 的内部步骤

7. `学习任务` 是唯一可复用执行壳；进度与切换信息应逐步迁移到 `任务虚拟存在`。


## 4. 分阶段修正

### 阶段 1：学习目标与学习需求重命名收口

目标：

- 把顶层学习目标、学习需求改成最新规范口径。

要改的核心：

- `枚举_学习目标类型`
- `枚举_学习需求类型`
- 所有名称映射、摘要映射、控制面板展示文案
- 学习任务实体中的对应字段解释

收口后口径：

- 顶层学习目标：
  - `让当前待学习方法头可用`
- 顶层学习需求：
  - `方法可用需求`

旧的 `样本收集/差异比较/...` 不再作为顶层需求，而转为学习动作层。


### 阶段 2：学习动作层下沉

目标：

- 将旧 `学习需求类型` 下沉为“学习动作类型”或“原函数需求类型”。

建议动作层保留：

- `样本收集`
- `差异比较`
- `共同条件提取`
- `候选动作生成`
- `结果预测`
- `沙箱验证`
- `风险评估`
- `边界标记`
- `可用层进入判断`
- `优选层进入判断`

要求：

- `任务管理任务` 先生成一个顶层主需求；
- 再根据方法头缺口，分解出动作层执行步骤。


### 阶段 3：完成判据收口

目标：

- 让 `任务管理任务` 的终局目标与中间补救分支彻底分开。

要求：

- 终局目标只保留：

```text
宿主任务状态进入完成
```

- 中间直接证据至少包括：
  - `目标结果已满足`
  - `当前方法有结果节点`
  - `当前方法可运行`
  - `当前结果与任务目标对齐`
  - `不存在待补条件/动作/迁移`
  - `不存在待处理关键子任务`

- 若上述证据不满足，只允许转入：
  - `补结构`
  - `补条件`
  - `补方法`
  - `转学习`
  - `等待`
  - `回接`
  - `继续业务推进`


### 阶段 4：原语 / 本能函数实现层边界继续拆开

目标：

- 不再只在规范和文案上区分 `原语` 与 `本能函数`，而要把底层实现层继续拆开。

要求：

- `任务管理最小原语模块` 内部对 `枚举_本能方法ID` 的复用逐步降级；
- 为 `原语` 建立更清晰的独立映射和执行入口；
- 保证：
  - `原语` 只改 `任务虚拟存在`
  - `本能函数` 只改 `世界树`


### 阶段 5：学习任务壳与任务虚拟存在权威化

目标：

- 让 `学习任务` 从“实体表主导”转向“任务虚拟存在主导”。

要求：

- `学习任务` 保持唯一执行壳；
- 当前学习目标、当前需求、当前候选程序、当前方法头、当前执行方法、当前抽象因果等，逐步以 `任务虚拟存在特征` 为权威；
- `学习实体表` 降为缓存/镜像。

优先迁移的权威字段：

- `当前待学习方法头`
- `当前学习目标`
- `当前学习需求数量`
- `当前学习程序裁决`
- `当前学习方法有动作骨架`
- `当前学习方法条件数量`
- `当前学习方法结果数量`
- `当前学习方法可用`
- `当前已切换到兜底可执行方法`


### 阶段 6：控制面板与观察面同步纠偏

目标：

- 避免界面继续暴露旧口径。

要求：

- 控制面板学习任务页改用：
  - `当前待学习方法`
  - `方法结构缺口`
  - `当前是否可用`
  - `当前学习动作`
  - `当前程序裁决`
- 任务管理页继续保持：
  - 任务管理任务汇总
  - 工作线程列表
  - 收发消息汇总


## 5. 执行顺序

按以下顺序推进：

1. 阶段 1：学习目标与学习需求重命名收口
2. 阶段 2：学习动作层下沉
3. 阶段 3：完成判据收口
4. 阶段 4：原语 / 本能函数边界继续拆开
5. 阶段 5：学习任务壳与任务虚拟存在权威化
6. 阶段 6：控制面板与观察面同步纠偏


## 6. 验收线

完成本轮修正后，至少满足：

1. 代码中不再把 `样本收集/差异比较/...` 当作顶层学习需求。
2. 学习主目标正式统一为：`让当前待学习方法头可用`。
3. `任务管理任务` 的完成判据正式以 `宿主任务状态进入完成` 为终局目标。
4. `原语` 和 `本能函数` 在代码层的边界比当前更清楚。
5. 学习任务的权威状态不再主要依赖内存表，而开始以 `任务虚拟存在` 为主。


## 7. 本轮结论

本轮不推翻现有代码主骨架，采用：

```text
保留主链
-> 修正学习目标/学习需求命名
-> 下沉旧动作型需求
-> 收口完成判据
-> 继续拆原语/本能函数边界
-> 逐步把学习状态权威迁入任务虚拟存在
```

一句话收口：

`当前修正的主目标，不是重写任务管理，而是把“任务完成、方法可用、原语边界、学习壳复用”这四条主线，在代码里真正收口。`
