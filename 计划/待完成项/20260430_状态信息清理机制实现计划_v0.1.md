# 20260430 状态信息清理机制实现计划 v0.1

状态：第一版已执行，后续保留物理删除与自检需求接入。

权威规范：

- `D:\鱼巢\规范\动态信息分层获取与聚合规范20260430.md`

目标：

在不破坏状态、动态、因果来源链的前提下，把处理完成的状态信息从活动状态链清理，避免状态信息长期堆积压垮系统。

固定规则：

```text
状态清理 = 生命周期完成清理 + 限时窗口 + 限量兜底
```

## 1. 总原则

1. 处理完成的状态信息必须清理。
2. 当前状态不清理，只更新同一宿主、同一特征口径的当前值。
3. 未处理状态、仍在因果窗口中的状态、仍被任务/动态/因果/自检引用的状态不得清理。
4. 限时和限量只是保护阀，不能替代处理完成判定。
5. 清理后必须保留当前状态、动态摘要、因果摘要或事件摘要中的至少一种长期信息。
6. 不新增调用点字段，不新增并行状态账本结构。
7. 若需要表达清理运行情况，优先使用现有事件、日志或虚拟存在特征。

## 2. 新增能力实现部分

本计划需要补充的是状态生命周期治理能力，不是新增一套状态账本。

新增能力必须尽量落在 `状态类`、`动态类`、`因果类` 的既有职责内；确实需要跨模块协调时，由调用入口组合已有能力完成。

### 2.1 状态活动链扫描能力

用途：

找到某个宿主、某个特征口径或某个场景下仍在活动链中的状态信息。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 宿主节点 | 节点引用 | 存在、场景或虚拟存在 |
| 特征口径 | 节点引用，可为空 | 为空时扫描宿主下全部状态 |
| 最大扫描数量 | I64，可为空 | 用于限量整理，空表示按调用方默认值 |
| 最早允许时间 | I64，可为空 | 用于限时窗口，空表示不按时间过滤 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 候选状态数量 | I64 | 扫描到的状态数量 |
| 已处理状态数量 | I64 | 可清理状态数量 |
| 禁止清理状态数量 | I64 | 当前状态、未处理状态或仍被引用状态数量 |

方向：

- 候选状态数量越大，清理压力越大。
- 已处理状态数量越大，可释放空间越大。
- 禁止清理状态数量越大，说明状态仍在有效窗口或引用链中。

### 2.2 当前状态识别能力

用途：

判断某个状态是否为同一宿主、同一特征口径的当前状态。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 宿主节点 | 节点引用 | 状态所属存在、场景或虚拟存在 |
| 状态节点 | 节点引用 | 待判断状态 |
| 特征口径 | 节点引用 | 状态采用的实例特征口径 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 是否当前状态 | 布尔 0/1 | 1 表示禁止清理 |

方向：

- `是否当前状态 = 1` 时，状态必须保留。
- `是否当前状态 = 0` 时，仍需继续检查是否已处理和是否被引用。

### 2.3 状态处理完成判定能力

用途：

判断状态是否已经完成消费，可以从活动状态链清理。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 状态节点 | 节点引用 | 待判断状态 |
| 宿主节点 | 节点引用，可由状态反查 | 状态所属宿主 |
| 特征口径 | 节点引用，可由状态反查 | 状态采用的实例特征口径 |
| 当前时间 | I64 | 用于限时窗口判断 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 是否处理完成 | 布尔 0/1 | 1 表示允许进入清理候选 |
| 禁止清理原因 | 枚举/状态词 | 当前状态、未读取、动态未生成、因果窗口中、仍被引用、异常未处理 |
| 未完成引用数量 | I64 | 仍在引用该状态的动态、因果、任务或自检数量 |

方向：

- 是否处理完成为 1 时，可以进入清理候选。
- 未完成引用数量越大，越不能清理。

### 2.4 状态引用保护判定能力

用途：

防止清理仍被动态、因果、任务或自检使用的状态。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 状态节点 | 节点引用 | 待检查状态 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 动态引用数量 | I64 | 来源动态、动态二次特征或聚合动态引用数 |
| 因果引用数量 | I64 | 因果候选、因果验证或稳定因果引用数 |
| 任务引用数量 | I64 | 任务、步骤或结果仍在读取该状态的数量 |
| 自检引用数量 | I64 | 自检需求或自检窗口引用数量 |
| 是否仍被引用 | 布尔 0/1 | 任一引用数量大于 0 时为 1 |

方向：

- 任一引用数量越大，清理风险越大。
- 是否仍被引用为 1 时，禁止清理。

### 2.5 状态清理执行能力

用途：

从活动状态链移除已经处理完成的原始状态。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 宿主节点 | 节点引用 | 清理范围 |
| 特征口径 | 节点引用，可为空 | 为空时按宿主清理 |
| 最大清理数量 | I64，可为空 | 单次最多清理多少条 |
| 当前时间 | I64 | 用于限时窗口 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 清理状态数量 | I64 | 实际清理数量 |
| 保留状态数量 | I64 | 因当前、未处理或引用保护保留的数量 |
| 清理失败数量 | I64 | 需要清理但因链路异常无法清理的数量 |

方向：

- 清理状态数量越大，活动链压力越小。
- 清理失败数量越大，越需要自检发现结构缺口。

硬规则：

1. 清理执行能力只能清理处理完成状态。
2. 清理前必须确认不是当前状态。
3. 清理前必须确认没有未完成引用。
4. 清理后必须保留当前状态、动态摘要、因果摘要或事件摘要中的至少一种。
5. 清理失败不得静默吞掉，应记录事件信息；达到自检条件时提交需求。

### 2.6 状态写入后整理能力

用途：

新状态落账后，立即整理同宿主、同特征口径的旧状态。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 新状态节点 | 节点引用 | 本次写入状态 |
| 宿主节点 | 节点引用 | 状态所属宿主 |
| 特征口径 | 节点引用 | 状态采用的实例特征口径 |
| 当前时间 | I64 | 整理时间 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 是否更新当前状态 | 布尔 0/1 | 当前状态是否已切换到新状态 |
| 生成动态数量 | I64 | 写入后触发的状态动态数量 |
| 清理旧状态数量 | I64 | 写入后清理的旧状态数量 |

方向：

- 生成动态数量越大，说明状态变化越活跃。
- 清理旧状态数量越大，说明活动链被及时释放。

### 2.7 动态生成后释放来源状态能力

用途：

状态动态生成后，尝试释放已经被动态摘要承接的前状态、后状态或中间状态。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 动态节点 | 节点引用 | 刚生成或刚完成摘要的动态 |
| 来源状态集合 | 节点引用集合 | 动态引用的状态 |
| 当前时间 | I64 | 整理时间 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 来源状态数量 | I64 | 参与整理的状态数量 |
| 释放状态数量 | I64 | 已清理来源状态数量 |
| 保留状态数量 | I64 | 因仍被引用或仍在窗口内保留数量 |

方向：

- 释放状态数量越大，说明动态摘要承接越充分。
- 保留状态数量越大，说明后续因果或任务仍在使用。

### 2.8 因果窗口关闭后整理能力

用途：

因果候选、第三方原因交集或反例统计完成后，释放窗口内不再需要的中间状态。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 因果或候选节点 | 节点引用 | 已完成窗口处理的因果相关节点 |
| 窗口状态集合 | 节点引用集合 | 窗口中的状态 |
| 当前时间 | I64 | 整理时间 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 窗口状态数量 | I64 | 窗口内状态数量 |
| 已摘要状态数量 | I64 | 已被因果摘要承接的状态数量 |
| 清理状态数量 | I64 | 实际清理数量 |
| 保留状态数量 | I64 | 仍需保留数量 |

方向：

- 已摘要状态数量越大，因果承接越完整。
- 保留状态数量越大，窗口仍未完全关闭。

### 2.9 状态清理自检需求能力

用途：

当状态超过限量但没有可清理状态，或清理失败持续出现时，由自检发现缺口并提交需求。

输入特征：

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| 活动状态数量 | I64 | 当前活动状态数量 |
| 可清理状态数量 | I64 | 已处理可清理状态数量 |
| 清理失败数量 | I64 | 最近清理失败数量 |
| 超限程度 | I64 | 活动状态数量减上限 |

输出特征：

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| 是否需要自检需求 | 布尔 0/1 | 1 表示需要提交需求 |
| 需求目标特征 | 特征类型 | 状态清理能力、状态引用完整性或状态摘要完整性 |
| 需求方向 | 枚举/方向 | 增加、减少或不变 |

方向：

- 活动状态数量越大，压力越大。
- 可清理状态数量越小，说明普通清理越难释放。
- 清理失败数量越大，越需要自检提交结构修复需求。

## 4. 阶段 1：处理完成判定

目标：

实现最小可用的状态处理完成判定。

第一版判定条件：

1. 状态已经被所属宿主读取。
2. 同口径当前状态已经更新。
3. 已生成必要状态动态，或确认无变化。
4. 不在动作结果观察窗口、外显状态确认窗口、第三方原因交集窗口中。
5. 没有动态、因果、任务、自检正在引用。
6. 已经落入必要摘要或事件信息。

实现要求：

- 优先利用现有字段、引用和链路判断。
- 不足以判断时，先保守保留，不强行删除。
- 缺少必要标记时，记录实现缺口，后续由需求或计划补齐。

完成判据：

- `已处理` 和 `禁止清理` 两类状态可以被稳定区分。
- 完成第 2.2、2.3、2.4 节能力的最小实现。

## 5. 阶段 2：清理入口

目标：

把清理挂入现有后处理路径，避免额外全量扫描成为常态。

第一版入口：

| 入口 | 行为 |
| --- | --- |
| 状态写入后处理 | 清理同宿主、同特征口径的已处理旧状态 |
| 状态动态生成后 | 清理已生成动态且无引用的前后状态 |
| 因果窗口关闭后 | 清理已完成交集和反例统计的中间状态 |
| 休眠期整理 | 按需要清理滞留已处理状态 |
| 限量保护 | 超过活动状态数量上限时清理已处理状态 |

完成判据：

- 高频状态写入不会导致活动链无限增长。
- 清理入口不会删除未处理状态。
- 完成第 2.5、2.6、2.7 节能力的最小实现。

## 6. 阶段 3：限时窗口

目标：

为状态比较、动作结果观察、外显状态确认和第三方原因交集保留必要窗口。

第一版窗口使用 I64 时间差或现有时间戳差值，不使用 Q10000。

窗口关闭规则：

1. 窗口自然到期。
2. 已生成目标动态或因果候选。
3. 已确认无有效变化。
4. 任务、自检或异常处理已完成。

完成判据：

- 窗口内状态不会被清理。
- 窗口关闭后，已处理状态可以被清理。
- 完成第 2.8 节能力的最小实现。

## 7. 阶段 4：限量兜底

目标：

在状态洪峰时保护系统。

限量层级：

1. 同一宿主、同一特征口径的活动状态数量。
2. 同一宿主的活动状态总量。
3. 同一场景的活动状态总量。
4. 全局活动状态总量。

处理规则：

- 超限时只清理已处理状态。
- 超限但没有可清理状态时，生成事件或自检需求。
- 不允许为了达成限量删除当前状态、未处理状态或仍被引用的状态。

完成判据：

- 活动状态数量能被压回目标范围。
- 不出现因清理导致的动态、因果、任务引用断裂。
- 完成第 2.1、2.9 节能力的最小实现。

## 8. 阶段 5：验证

验证场景：

1. 单一特征连续变化，旧状态被生成动态后清理。
2. 状态无变化，确认无动态后清理。
3. 状态仍在因果窗口中，不能清理。
4. 状态被任务或自检引用，不能清理。
5. 高频状态超过限量，只清理已处理状态。
6. 休眠期整理只执行有限批次，不反复全量扫描。

完成判据：

- 工程编译通过。
- 状态清理不会破坏当前状态读取。
- 状态清理不会破坏动态二次特征和因果来源摘要。
- 新增能力清单中的每项能力都有对应调用入口或明确后置原因。
