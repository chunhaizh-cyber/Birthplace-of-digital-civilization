# 2026-04-15 任务管理任务消息模块落实计划

## 0. 计划定位

本文用于把 `任务管理任务` 这一侧的消息世界，按现有规范压成一份可直接施工的落实计划。

本文只承接这条问题：

`任务管理任务` 如何把当前已经存在的 `显式事件控制 / 恢复重建控制 / 恢复事件镜像 / 最近反馈摘要`，进一步收口成正式的 `消息协议 + mailbox/消费骨架 + 返回面桥接`。

本文不负责：

1. 重写整个 `任务管理任务` 五模块业务算法。
2. 重写 `自我线程` 消息模块。
3. 一次性把全部任务调度器、执行器、学习反馈、外部反馈都改成统一新协议。


## 1. 目标

这份计划的目标只有一个：

`把任务管理任务当前分散在请求装配、显式事件控制、恢复快照和一步治理结果里的消息相关逻辑，收成正式模块边界与分阶段落地顺序。`

希望最后得到的是：

1. `任务管理任务` 有自己清晰的消息协议面。
2. `mailbox / 待消费事件 / 恢复事件镜像 / 最近反馈摘要` 之间的关系被正式写清。
3. 输入消息、消费过程、输出返回面和恢复快照桥接不再散落在一个超大实现文件里。

## 4. 硬边界

必须先固定以下边界：

1. `任务管理任务` 的消息模块，不是 `自我线程` 消息模块的简单复制。
2. `任务管理任务` 侧消息，首先服务于：
   - 装配 `任务管理请求`
   - 驱动单步治理推进
   - 产出 `一步治理结果`
   - 镜像到 `恢复快照`
3. `任务管理任务` 不负责根层重判；它只消费输入消息并给出候选、摘要和一步治理结果。
4. `最近反馈类型 / 最近反馈摘要` 只能表达最小稳定结果，不得冒充最终裁决。
5. 恢复快照只保存最小恢复信息，不保存物理线程态、锁态或整个原始消息现场。


## 5. 建议模块拆分

按当前仓库现实，推荐先拆成四块，而不是一步拆太细：

### 5.1 任务管理任务消息协议模块

建议文件：

- `任务管理任务模块.消息协议.ixx`

职责：

1. 定义 `任务管理任务` 侧正式输入消息类型。
2. 定义最小消息头、来源、摘要、幂等键、重放代数、消费结果承载。
3. 统一承接当前这些来源：
   - 显式事件控制
   - 恢复重建
   - 学习反馈
   - 学习回流
   - 外部反馈
   - 普通推进

### 5.2 任务管理任务消息装配/桥接模块

建议文件：

- `任务管理任务模块.消息桥接.ixx`

职责：

1. 把现有 `结构_任务管理显式事件控制` 桥接成正式消息。
2. 把 `治理恢复快照` 中的待恢复事件镜像桥接成恢复重建输入。
3. 把当前请求上的 `触发事件类型 / 触发事件摘要 / 恢复重建摘要` 统一改成从正式消息推导。

### 5.3 任务管理任务消息消费模块

建议文件：

- `任务管理任务模块.消息消费.ixx`

职责：

1. 负责消费当前轮次要处理的消息。
2. 统一落：
   - `最近反馈类型`
   - `最近反馈摘要`
   - `当前触发来源`
   - `来源最小原语位图`
   - `来源主观察特征`
   - `影子验证状态`
   - `允许正式资产提交`
3. 只生成候选和稳定摘要，不直接越权写最终裁决。

### 5.4 任务管理任务消息镜像/恢复模块

建议文件：

- 继续复用并逐步削薄 [任务管理任务模块_治理快照.cpp](D:/鱼巢/任务管理任务模块_治理快照.cpp)

职责：

1. 把已消费消息镜像到 `治理恢复事件镜像`。
2. 把最近消费结果、最近反馈摘要、最近反馈类型稳定写入 `治理恢复快照`。
3. 维持 `一步治理结果 -> 恢复快照` 的稳定桥接。


## 6. 消息最小对象面

第一轮建议先冻结以下最小字段，而不是一开始追求大而全：

1. `消息类型`
2. `触发来源`
3. `产生时间`
4. `摘要`
5. `来源主观察特征`
6. `来源最小原语位图`
7. `影子验证状态`
8. `允许正式资产提交`
9. `幂等键`
10. `重放代数`
11. `最后消费结果`

一句话收口：

`先把“谁送来什么消息、任务管理任务怎么消费、消费后最少留下什么稳定结果”固定下来。`


## 7. 实施顺序

### 阶段 A：冻结协议与桥接边界

目标：

- 先把现有显式事件控制、恢复重建、反馈摘要收成正式消息对象。

本阶段做什么：

1. 新增 `任务管理任务模块.消息协议.ixx`
2. 定义最小消息结构
3. 定义从 `显式事件控制 / 恢复快照 / 普通推进` 到正式消息的桥接 helper

完成标准：

1. `任务管理任务` 的正式消息不再只靠 `触发事件类型 + 摘要字符串` 隐式表达。
2. 现有请求装配逻辑可以从正式消息对象读字段。


### 阶段 B：把请求装配改成“先消费消息，再装配请求”

目标：

- 让 `任务管理请求` 的触发信息来自统一消息消费入口，而不是散落 helper。

本阶段做什么：

1. 收口 `私有_M9_消费请求触发态()`
2. 收口 `私有_M9_装配请求触发信息(...)`
3. 收口 `私有_M9_生成恢复重建请求摘要(...)`
4. 明确“本轮要消费哪一条/哪一批消息”

完成标准：

1. `任务管理请求` 的消息相关字段统一由一条入口生成。
2. 恢复重建与显式事件不再是两套平行拼接逻辑。


### 阶段 C：把消息消费结果正式写回稳定结果面

目标：

- 把消息消费对 `最近反馈类型 / 最近反馈摘要 / 来源桥接字段` 的更新，正式收口。

本阶段做什么：

1. 新增或抽出 `任务管理任务消息消费` helper
2. 统一更新：
   - `最近反馈类型`
   - `最近反馈摘要`
   - `最近上层反馈摘要`
   - `触发来源`
   - `来源最小原语位图`
   - `来源主观察特征`

完成标准：

1. 这些字段的生产位置明显收口。
2. `一步治理结果` 与 `任务管理上下文` 的消息来源更容易回查。


### 阶段 D：把消息消费结果稳定镜像到恢复快照

目标：

- 让 `任务管理任务` 的消息消费面，和恢复面真正接平。

本阶段做什么：

1. 继续复用 `结构_治理恢复事件镜像`
2. 把最近消费消息结果稳定写入 `治理恢复快照`
3. 明确哪些消息只留内存态，哪些消息进入待恢复事件镜像

完成标准：

1. 恢复快照不只是“结果摘要镜像”，还明确包含最小消息消费痕迹。
2. 但仍不越界保存完整原始消息现场。


### 阶段 E：最后再决定是否下沉成独立 mailbox 模块

目标：

- 只在前四步站稳后，再决定是否继续把实例级 mailbox 正式拆出来。

本阶段做什么：

1. 评估当前 `显式事件控制 + 恢复重建控制 + 待恢复事件镜像` 是否已经够用
2. 若不够，再单列“实例 mailbox 模块”计划

完成标准：

1. 不把 `mailbox` 物理实现提前变成第一步。
2. 先把协议、消费和稳定字段收口，再谈更重的实例级队列化。
