# 2026-04-15 任务管理任务消息模块落实计划

## 0. 计划定位

本文用于把 `任务管理任务` 这一侧的消息世界，按现有规范压成一份可直接施工的落实计划。

本文只承接这条问题：

`任务管理任务` 如何把当前已经存在的 `显式事件控制 / 恢复重建控制 / 恢复事件镜像 / 最近反馈摘要`，进一步收口成正式的 `消息协议 + mailbox/消费骨架 + 返回面桥接`。

按 `2026-04-15` 最新统一口径，`任务管理任务` 提交给 `自我线程` 的上行消息，也同步收口为两大类：

1. `运行状态消息`
2. `结果状态消息`

其中 `运行状态消息` 在实现上直接按 `线程状态消息` 落，不再另外发明一层新壳；`当前步骤 / 当前步骤前沿 / 等待原因 / 是否可继续推进 / 是否已响应控制意图` 等，只作为线程状态消息里的补充特征类型表达。

现有 `工作线程状态 / 任务推进关键事件 / 派生需求 / 结果与关键中间状态` 等旧细分，后续只作为兼容路由入口保留，不再作为长期一级大类继续扩张。

本文不负责：

1. 重写整个 `任务管理任务` 五模块业务算法。
2. 重写 `自我线程` 消息模块。
3. 一次性把全部任务调度器、执行器、学习反馈、外部反馈都改成统一新协议。


## 1. 目标

这份计划的目标只有一个：

`把任务管理任务当前分散在请求装配、显式事件控制、恢复快照和一步治理结果里的消息相关逻辑，收成正式模块边界与分阶段落地顺序。`

希望最后得到的是：

1. `任务管理任务` 有自己清晰的消息协议面。
2. `mailbox / 待消费事件 / 恢复事件镜像 / 最近反馈摘要` 之间的关系被正式写清。
3. 输入消息、消费过程、输出返回面和恢复快照桥接不再散落在一个超大实现文件里。

## 4. 硬边界

必须先固定以下边界：

1. `任务管理任务` 的消息模块，不是 `自我线程` 消息模块的简单复制。
2. `任务管理任务` 侧消息，首先服务于：
   - 装配 `任务管理请求`
   - 驱动单步治理推进
   - 产出 `一步治理结果`
   - 镜像到 `恢复快照`
3. `任务管理任务` 不负责根层重判；它只消费输入消息并给出候选、摘要和一步治理结果。
4. `最近反馈类型 / 最近反馈摘要` 只能表达最小稳定结果，不得冒充最终裁决。
5. 恢复快照只保存最小恢复信息，不保存物理线程态、锁态或整个原始消息现场。
6. `任务管理任务` 或 `工作线程` 上报的运行状态，只能是 `任务管理任务运行状态 / 工作线程运行状态`，不得替 `自我线程` 自身下运行状态结论。
7. `任务管理任务` 提交给 `自我线程` 的上行消息，长期口径只保留两大类：`运行状态消息` 与 `结果状态消息`。
8. `派生需求 / 未满足项` 不再作为第三大上行消息类；它应并入 `结果状态消息` 的 `未满足项 / 后续支撑请求段`。

## 6. 消息最小对象面

第一轮不再把上行消息无限细分，而是先冻结两大消息对象面。

### 6.1 运行状态消息字段表（实现上即线程状态消息）

| 段落 | 字段 | 最小要求 | 说明 |
| --- | --- | --- | --- |
| 身份段 | `发出者主键 / 任务管理任务主键 / 绑定任务主键 / 父任务主键 / 关联场景主键 / 时间戳 / 幂等键` | 必带 | 回答“谁在什么时候上报” |
| 运行状态段 | `当前运行状态 / 当前步骤主键 / 当前步骤前沿 / 当前等待原因 / 是否可继续推进 / 是否已响应控制意图` | 必带 | 回答“现在怎么跑、还能不能继续跑” |
| 资源与控制段 | `当前预算消耗 / 当前时间消耗 / 是否超时 / 是否触发异常 / 当前局部风险等级` | 建议首批落位 | 服务于控制面与调度判断 |
| 提示段 | `是否建议自我线程重判 / 是否建议收束 / 是否建议恢复重建` | 可选 | 只是提示，不是裁决 |

必须固定：

1. 这里的 `运行状态消息` 在实现上就是 `线程状态消息`；不再单独发明“任务管理任务运行状态消息”新壳。
2. `当前步骤 / 当前步骤前沿 / 当前等待原因 / 是否可继续推进 / 是否已响应控制意图` 等，只作为线程状态消息里的补充特征类型表达。
3. `任务推进关键事件消息` 若长期继续保留，也只保留为兼容路由入口；进入正式协议后应规范化为线程状态消息中的特征变化项。
4. `建议重判 / 建议收束 / 建议恢复重建` 只能作为提示位，不授予发件线程裁决权。

### 6.2 结果状态消息字段表

| 段落 | 字段 | 最小要求 | 说明 |
| --- | --- | --- | --- |
| 结果头 | `发出者主键 / 任务管理任务主键 / 任务主键 / 父任务主键 / 关联场景主键 / 时间戳 / 幂等键` | 必带 | 回答“谁在什么时点提交了这次结果事实” |
| 结果节点引用段 | `结果节点主键或指针 / 结果节点所属场景主键 / 是否已正式写入结果节点` | 必带 | 结果消息的主承载面；主要事实默认从 `结果节点` 的场景信息里读取 |
| 快速补充段 | `结果状态 / 完成度或结果值 / 是否成功 / 错误码 / 完成时间` | 必带 | 便于线程快速路由和结算，不必先完全展开结果节点 |
| 后续支撑补充段 | `是否存在关键中间状态 / 是否仍有未满足项 / 是否需要外部支撑 / 是否需要父任务处理 / 是否建议自我线程重判 / 是否疑似需要新任务入口` | 必带 | 只提供最小补充判断，不在消息层重复整包展开全部事实 |

必须固定：

1. `结果状态消息` 的核心不是“成没成功”，而是“这次执行到底留下了哪些事实”。
2. 这些事实的主承载面，默认是 `任务结果节点` 及其场景信息，而不是消息体里的重复平铺字段。
3. `服务对象当前状态 / 关键中间状态 / 未满足项` 这些主体信息，优先写进 `结果节点` 的场景信息；消息只回 `结果节点引用 + 少量补充信息`。
4. 若结果节点尚未稳定落成，才允许临时回退到消息体内展开更多细节。
5. `派生需求` 以后应按 `未满足项 / 后续支撑段` 理解，而不是独立第三大消息类。

### 6.3 自我线程处理动作表

| 消息大类 | 自我线程第一处理 | 后续处理 | 最终落点 |
| --- | --- | --- | --- |
| `运行状态消息` | 更新任务控制态与工作线程状态表 | 刷新全局队列位置、重算相关二次特征、必要时触发根层重判 | `控制面 / 队列态 / 线程状态账` |
| `结果状态消息` | 先解析 `结果节点引用段`，再从 `结果节点场景信息` 读取主体事实并结合补充字段落账 | 重算二次特征、抽取影响项、判断全局影响、路由父任务、统一结算 | `服务对象状态账 / 关键中间状态账 / 结果路由 / 总账结算` |

一句话收口：

`先把“任务管理任务向自我线程到底提交哪两类消息、每类消息最少带什么、线程收到后先做什么”固定下来。`


## 7. 实施顺序

### 阶段 A：冻结协议与桥接边界

目标：

- 先把现有显式事件控制、恢复重建、反馈摘要收成正式消息对象。

本阶段做什么：

1. 新增 `任务管理任务模块.消息协议.ixx`
2. 定义 `运行状态消息 / 结果状态消息` 两个最小消息结构
3. 定义从 `显式事件控制 / 恢复快照 / 普通推进 / 旧细分回流` 到正式消息的桥接 helper

完成标准：

1. `任务管理任务` 的正式消息不再只靠 `触发事件类型 + 摘要字符串` 隐式表达。
2. 现有兼容细分可以稳定映射到 `运行状态消息 / 结果状态消息` 两大上行对象。


### 阶段 B：把请求装配改成“先消费消息，再装配请求”

目标：

- 让 `任务管理请求` 的触发信息来自统一消息消费入口，而不是散落 helper。

本阶段做什么：

1. 收口 `私有_M9_消费请求触发态()`
2. 收口 `私有_M9_装配请求触发信息(...)`
3. 收口 `私有_M9_生成恢复重建请求摘要(...)`
4. 明确“本轮要消费哪一条/哪一批消息”

完成标准：

1. `任务管理请求` 的消息相关字段统一由一条入口生成。
2. 恢复重建与显式事件不再是两套平行拼接逻辑。


### 阶段 C：把消息消费结果正式写回稳定结果面

目标：

- 把消息消费对 `最近反馈类型 / 最近反馈摘要 / 来源桥接字段` 的更新，正式收口。

本阶段做什么：

1. 新增或抽出 `任务管理任务消息消费` helper
2. 统一更新：
   - `最近反馈类型`
   - `最近反馈摘要`
   - `最近上层反馈摘要`
   - `触发来源`
   - `来源最小原语位图`
   - `来源主观察特征`

完成标准：

1. 这些字段的生产位置明显收口。
2. `一步治理结果` 与 `任务管理上下文` 的消息来源更容易回查。


### 阶段 D：把消息消费结果稳定镜像到恢复快照

目标：

- 让 `任务管理任务` 的消息消费面，和恢复面真正接平。

本阶段做什么：

1. 继续复用 `结构_治理恢复事件镜像`
2. 把最近消费消息结果稳定写入 `治理恢复快照`
3. 明确哪些消息只留内存态，哪些消息进入待恢复事件镜像

完成标准：

1. 恢复快照不只是“结果摘要镜像”，还明确包含最小消息消费痕迹。
2. 但仍不越界保存完整原始消息现场。


### 阶段 E：最后再决定是否下沉成独立 mailbox 模块

目标：

- 只在前四步站稳后，再决定是否继续把实例级 mailbox 正式拆出来。

本阶段做什么：

1. 评估当前 `显式事件控制 + 恢复重建控制 + 待恢复事件镜像` 是否已经够用
2. 若不够，再单列“实例 mailbox 模块”计划

完成标准：

1. 不把 `mailbox` 物理实现提前变成第一步。
2. 先把协议、消费和稳定字段收口，再谈更重的实例级队列化。
