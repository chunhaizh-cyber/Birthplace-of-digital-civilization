# 需求满足闭环与权重分摊结算计划 v0.1

## 1. 目标

当前优先目标是先把需求闭环跑通，再在这个闭环上实现完整的“部分满足权重分摊算法”：

```text
任务失败
-> 派生补方法需求
-> 学习任务
-> 方法可用
-> 派生需求满足
-> 父任务回到待重筹办
-> 根任务继续执行真实目标方法
-> 根需求满足
-> 安全/服务值正式结算
```

完整权重分摊是本计划的最终交付目标。闭环稳定只是它的前置条件，不代表权重分摊可以无限后置。

对应详细设计：

- `详细设计/需求满足程度与权重分摊结算详细设计.md`

## 2. 当前代码现状

已经确认或部分修复：

1. 二次特征去重已按“特征类型 + 比较结果”做语义判断，不再依赖实时主键。
2. 任务结果节点回写已补，步骤统计和结果统计已正常。
3. 学习任务成功后，已经能看到需求满足结算。
4. 派生需求 `2`、`3` 已能被标记满足。
5. 已补“派生需求满足后唤醒父任务”的代码，但尚未编译验证。

暴露出的关键问题：

1. 派生需求满足后，父需求对应任务必须从 `等待中 / 挂起 / 等待学习唤醒` 回到 `待重筹办`。
2. 派生需求满足后，主链可能变成 `主导需求=安全根需求 | 主导任务=无`，说明旧根任务没有被复用或重新激活。
3. 补方法只满足中间需求，不能直接代表根需求满足。
4. 根任务必须重新执行真实目标方法，目标状态确实达到后，根需求才能满足并结算安全/服务值。
5. 当前学习路径主要是任务壳加 `学习_尝试学习` 本能，尚未正式接入 `学习执行模块::执行一步学习任务`。

## 3. 约束

1. 不新增“运行账本”层。
2. 状态、动态、因果、整理事件已经足够作为第一版最小事实来源。
3. 不使用 `Q10000`。
4. 可比较、可累计、可排序的值统一使用 `I64`。
5. 布尔值只用于天然二值事实，取值为 `0/1`。
6. 任务管理工作线程不直接写安全值或服务值。
7. 安全值和服务值只在自我线程最终结算链中变更。
8. 学习成功只能满足派生补方法需求，不能跳过父任务和根任务执行。
9. 派生需求满足必须推进父任务继续运转，而不是只把子需求打勾。
10. 学习执行模块接入后置，当前先让需求闭环跑通。

## 4. 第一优先级：需求闭环

### 4.1 验证派生需求满足后唤醒父任务

目标：

1. 编译验证已补代码。
2. 学习任务成功后，找到派生需求的父需求。
3. 找到父需求对应任务或父任务链。
4. 将处于 `等待中 / 挂起 / 等待学习唤醒` 的父任务恢复为 `待重筹办`。
5. 清理父任务上的等待学习标记，使其能重新筹办。

完成判据：

1. 学习任务成功后，派生需求被标记满足。
2. 父任务不再卡在等待学习状态。
3. 父任务进入 `待重筹办`。
4. 后续调度能重新看到这个父任务。

验证日志重点：

```text
派生需求满足
父任务等待学习标记清理
父任务状态=待重筹办
父任务重新进入任务候选
```

### 4.2 父任务/根任务重筹办闭环

目标：

1. 父任务回到 `待重筹办` 后，任务筹办本能重新检查方法可用性。
2. 如果补齐的方法已经可用，应绑定该方法。
3. 父任务不应再次派生同一个补方法需求。
4. 父任务应继续推进真实目标，而不是停在学习完成状态。

完成判据：

1. 同一个方法补齐后，父任务二次筹办能命中可用方法。
2. 任务状态从 `待重筹办` 进入可执行链。
3. 不重复生成同一派生需求。
4. 任务结果能继续回写到父步骤/结果节点。

需要检查的状态：

| 对象 | 字段/状态 |
| --- | --- |
| 父任务 | `状态` |
| 父任务 | `等待学习唤醒` |
| 父任务 | `等待学习方法首节点` |
| 父任务 | `执行方法` |
| 父任务 | `候选方法首节点` |
| 派生需求 | `需求状态` |
| 方法虚拟存在 | `方法是否可用` |

### 4.3 根需求重判与旧根任务复用

当前问题：

```text
主导需求 = 安全根需求
主导任务 = 无
```

目标：

1. 根需求重判时，如果根需求已有旧任务，应优先复用旧任务。
2. 旧任务如果处于等待学习、挂起或等待中，但阻塞派生需求已满足，应恢复到 `待重筹办`。
3. 旧任务如果已经终结但根需求未满足，应允许重新筹办或生成继任任务。
4. 只有根需求确实没有可复用任务时，才创建新任务。

完成判据：

1. 派生需求满足后，主导需求仍能找到对应主导任务。
2. 不出现 `安全根需求 | 主导任务=无` 的空转。
3. 旧根任务不会因为曾经等待学习而永久失联。
4. 新任务创建只发生在确实没有可复用任务时。

复用优先级：

```text
正在运行/可运行任务
-> 待重筹办任务
-> 等待学习但阻塞已解除的任务
-> 挂起但阻塞已解除的任务
-> 可恢复的旧终结任务或继任任务
-> 新建任务
```

### 4.4 根任务重新执行真实目标方法

目标：

1. 根任务恢复后，重新执行最初目标方法。
2. 执行结果必须回写真实目标状态。
3. 根需求满足判定必须基于真实目标状态，而不是基于“方法已补齐”。

完成判据：

1. 补方法需求满足后，根任务仍会继续执行。
2. 根任务执行的是根需求目标所需方法。
3. 根任务执行成功后，根需求目标状态达到。
4. 根需求被标记满足。

禁止：

1. 用“学习成功”直接满足根需求。
2. 用“方法可用”直接结算根需求安全/服务值。
3. 跳过根任务真实执行。

### 4.5 根需求最终满足结算

目标：

1. 根任务执行真实目标方法成功后，重新判断根需求是否满足。
2. 根需求首次满足时，按需求安全权重和服务权重结算。
3. 已满足根需求不得重复结算。

完成判据：

1. 根需求状态变为满足。
2. 安全值/服务值出现正式结算增量。
3. 重复回放同一结果不会重复加值。
4. 派生需求的满足结算不会冒充根需求结算。

## 5. 第二优先级：最小防重复结算

在闭环跑通后，补最小结算锚。

第一版最小字段：

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| 已结算状态值 | `I64` | 防止同一状态段重复结算 |
| 累计安全结算 | `I64` | 防止超过安全权重 |
| 累计服务结算 | `I64` | 防止超过服务权重 |
| 最近结算任务 | 任务引用 | 追踪贡献来源 |
| 最近结算时间 | 时间戳 | 追踪结算发生时间 |

第一版可以先按“首次满足一次性结算”实现。

完成判据：

1. 根需求首次满足时结算一次。
2. 已结算根需求再次进入主循环时不重复结算。
3. 派生需求满足只唤醒父链，不直接结算根需求权重。

## 6. 第三优先级：部分满足权重分摊

闭环稳定后，必须接入详细设计中的分摊算法。

这一阶段不是可选增强，而是需求满足机制的最终形态。没有权重分摊时，只能算“需求首次满足一次性结算”的临时闭环；实现权重分摊后，才算需求满足结算完成。

### 6.1 数值增加需求

适用：

```text
初始状态值 < 目标状态值
```

公式：

```text
总目标差额 = 目标状态值 - 初始状态值
已结算终点 = min(已结算状态值, 目标状态值)
当前可结算终点 = min(当前实际状态值, 目标状态值)
有效满足贡献 = max(0, 当前可结算终点 - 已结算终点)
```

结算：

```text
安全增量 = 安全权重 * 有效满足贡献 / 总目标差额
服务增量 = 服务权重 * 有效满足贡献 / 总目标差额
```

### 6.2 数值减少需求

适用：

```text
初始状态值 > 目标状态值
```

公式：

```text
总目标差额 = 初始状态值 - 目标状态值
已结算终点 = max(已结算状态值, 目标状态值)
当前可结算终点 = max(当前实际状态值, 目标状态值)
有效满足贡献 = max(0, 已结算终点 - 当前可结算终点)
```

结算同数值增加需求。

### 6.3 布尔需求

布尔型需求视为 `0/1`。

第一版只要求支持：

```text
0 -> 1
```

达到目标时一次性结算剩余权重。

### 6.4 有序枚举需求

有序枚举必须先映射为稳定 `I64` 阶梯，再复用数值结算。

无序枚举只用于路由，不参与分摊结算。

## 7. 实现顺序

### 阶段 0：编译验证当前父任务唤醒补丁

动作：

1. 编译当前代码。
2. 修复父任务唤醒代码的编译问题。
3. 保持改动只围绕派生需求满足后的父任务恢复。

完成判据：

1. 项目能编译。
2. 无新增结构层。
3. 不破坏已有派生需求满足逻辑。

### 阶段 1：验证学习成功到父任务待重筹办

动作：

1. 跑现有复现场景。
2. 观察派生需求 `2`、`3` 满足后父任务状态。
3. 若父任务仍未恢复，修正父任务定位和状态迁移条件。

完成判据：

1. 派生需求满足后，父任务进入 `待重筹办`。
2. 父任务等待学习标记被清理。

### 阶段 2：修复根需求有旧任务但主导任务为空

动作：

1. 检查主导任务选择逻辑。
2. 增加旧任务复用条件。
3. 对等待学习阻塞已解除的旧任务恢复为 `待重筹办`。
4. 必要时生成继任任务，但不得优先新建替代旧任务。

完成判据：

1. 不再出现根需求有旧任务但主导任务为空。
2. 根需求能持续拿到可推进任务。

### 阶段 3：根任务继续执行并回写真实目标结果

动作：

1. 验证根任务重筹办后能绑定补齐后的方法。
2. 验证真实目标方法执行。
3. 验证结果节点和需求当前状态回写。

完成判据：

1. 根任务能从补方法后的状态继续执行。
2. 根需求目标状态被真实结果推进。

### 阶段 4：根需求首次满足结算

动作：

1. 确认根需求满足判定只看真实目标状态。
2. 首次满足时结算安全/服务权重。
3. 增加最小防重复结算标记。

完成判据：

1. 根需求满足后安全/服务值更新。
2. 同一根需求不会重复结算。

### 阶段 5：补部分满足分摊算法

动作：

1. 引入 `I64` 差额分摊。
2. 支持数值增加、数值减少、布尔 `0 -> 1`。
3. 完全满足时补齐整数除法余数。

完成判据：

1. 部分满足能按差额结算。
2. 完全满足累计结算等于需求权重。
3. 超额完成不超额结算。

### 阶段 6：评估正式学习执行模块接入

动作：

1. 保留当前任务壳 + `学习_尝试学习` 最小路径。
2. 闭环稳定后，再评估是否切换到 `学习执行模块::执行一步学习任务`。
3. 若接入正式学习执行模块，必须保持“学习成功 -> 派生需求满足 -> 父任务待重筹办”的同一回流语义。

完成判据：

1. 正式学习执行模块不是闭环跑通的前置条件。
2. 接入后不改变需求满足与父任务唤醒语义。

## 8. 验证用例

### 8.1 最小闭环用例

```text
根任务缺方法
-> 生成派生补方法需求
-> 生成学习任务
-> 学习成功，方法可用
-> 派生需求满足
-> 父任务待重筹办
-> 根任务复用旧任务继续执行
-> 根需求满足
-> 安全/服务值结算
```

期望：

1. 派生需求满足。
2. 父任务恢复为 `待重筹办`。
3. 主导任务不为空。
4. 根任务重新执行。
5. 根需求满足。
6. 安全/服务值只在根需求满足后结算。

### 8.2 主导任务为空回归用例

初始现象：

```text
主导需求 = 安全根需求
主导任务 = 无
```

期望：

1. 根需求已有旧任务时复用旧任务。
2. 阻塞解除的等待任务恢复为 `待重筹办`。
3. 无旧任务时才创建新任务。

### 8.3 防重复结算用例

期望：

1. 根需求首次满足时结算一次。
2. 重复回放结果不重复结算。
3. 派生需求满足不直接增加根需求权重。
