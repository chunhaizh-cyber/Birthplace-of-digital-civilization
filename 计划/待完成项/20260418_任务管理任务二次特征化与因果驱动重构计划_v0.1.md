# 20260418 任务管理任务二次特征化与因果驱动重构计划 v0.1

## 1. 计划目标

本文只回答一件事：

`在不推倒当前工程骨架的前提下，如何把任务管理任务模块从“内含大量业务判断的大控制器”，重构成“任务树/方法树表达结构，二次特征表达证据，原语解释证据，任务管理只做编排”的稳定内核。`

本计划不主张整仓重写。

本计划主张：

- 保留当前已经成型的壳：
  - `请求`
  - `原语判定汇总`
  - `高阶排序请求/结果`
  - `选择函数`
  - `写回结果`
  - `一步结果`
- 重构真正需要替换的部分：
  - `任务管理任务模块.impl.cpp` 中的大量直接业务判断
  - `步骤结果子链` 的私有推导
  - `学习承接 / 运行包` 中仍直接依赖裸决策枚举的部分
- 新增真正缺失的主链：
  - `状态差分/二次特征`
  - `实例因果`
  - `因果模板`
  - `方法树/任务树自动沉淀`


## 2. 已确认实现原则

必须固定：

1. `任务树/方法树` 负责表达结构。
2. `二次特征` 负责表达当前判断证据。
3. `原语` 只负责解释这些证据。
4. `任务管理任务模块` 只做编排，不继续容纳大块业务判断。
5. 能用 `任务/方法信息` 表达的，不保留为 `impl.cpp` 私有推理。
6. 运行时业务判断必须逐步迁移为：
   - `任务/方法结构`
   - `二次特征证据`
   - `判断原语`
   三层之一。
7. `因果信息` 必须成为“特征变化 -> 方法树/任务树”的正式中间层，而不是只做事后补写。


## 3. 是否需要推倒重来

结论：

`不建议抛开当前代码整体重写。`

建议采用：

`保留骨架、切掉判断内核、逐段迁移证据源`

原因如下：

### 3.1 已经成熟的部分值得保留

当前工程已经具备以下正式结构：

- `原语判定接口 / 选择函数接口`
- `结构_原语判定结果 / 结构_原语判定汇总`
- `结构_高阶排序请求 / 结构_高阶排序结果 / 结构_高阶胜出绑定`
- `请求 -> 单步决策 -> 写回 -> 一步结果 -> 总结果`
- `控制承接`
- `学习承接`
- `运行包`

这些壳已经是可复用资产，不适合推倒重写。

### 3.2 真正的问题不在壳，而在证据源

当前最大问题是：

- `原语壳` 已经建立
- `高阶排序壳` 已经建立
- `选择函数壳` 已经建立

但很多原语和后段推进逻辑，仍直接读取：

- 裸上下文
- 裸状态枚举
- 裸布尔
- 裸节点归属判断
- 裸文本摘要

因此问题本质是：

`判断内核没有完成二次特征化`

而不是：

`工程骨架彻底错误`

### 3.3 直接重写的风险更大

如果完全抛开现有代码重做，会同时失去：

- 当前请求/决策/写回/结果四层壳
- 当前高阶排序链
- 当前控制承接链
- 当前学习承接链
- 当前运行包和治理快照观察面

这会让“结构重构”和“运行链重建”耦合在一起，风险过高。

因此本计划采用：

`局部重构 + 证据链替换 + 因果沉淀前推`


## 4. 当前代码的主要偏差

### 4.1 原语层仍大量读取裸上下文

典型位置：

- `D:\鱼巢\任务管理任务模块.impl.cpp`
  - `私有_P0_执行原语(...)`

当前仍直接读取：

- `最近功能域`
- `最近根层重判结果`
- `最近执行前门控结果`
- `影子验证状态` 文本
- `目标结果已对齐/待推进` 的即时计算结果

这说明当前原语还不是“纯证据解释器”。

### 4.2 步骤结果子链仍由私有逻辑直接驱动

典型位置：

- `D:\鱼巢\任务管理任务模块.impl.cpp`
  - `私有_推进步骤结果子链(...)`

当前仍直接判断：

- 最近结果是否可回接
- 当前步骤是否可复用
- 当前步骤位类型
- 当前步骤语义类型
- 结果节点是否匹配当前治理
- 业务子任务是否应转执行位

这些都应逐步迁移。

### 4.3 完成/等待/收束判据还没有收成正式证据层

典型位置：

- `私有_可进入等待分支(...)`
- `私有_业务子任务执行已完成(...)`
- `私有_结果节点匹配当前治理(...)`

这批函数本质上就是：

`运行时完成判据`

它们后续应收口为：

- 二次特征证据
- 原语解释

而不应长期保留为零散私有布尔函数。

### 4.4 因果信息还不是方法树/任务树生成主引擎

当前已有：

- 实例因果派生
- 轻量因果链组织
- 方法条件/结果节点

但现状更像：

- 运行后补因果
- 学习后补方法树

而不是：

- 特征变化
- 二次特征
- 因果实例
- 因果模板
- 方法树/任务树沉淀


## 5. 目标重构形态

最终目标结构固定为：

### 5.1 任务树/方法树

负责表达：

- 条件节点
- 结果节点
- 目标状态
- 当前结果方向
- 已绑定方法
- 控制意图绑定
- 高阶排序绑定
- 回流目标
- 服务范围
- 约束与等待位

### 5.2 二次特征层

负责表达：

- 条件满足证据
- 方向一致证据
- 结果符合证据
- 冲突证据
- 因果稳定证据
- 等待证据
- 回接证据
- 完成证据
- 收束证据
- 控制响应证据

### 5.3 原语层

只负责解释二次特征证据，例如：

- `条件满足`
- `方向一致`
- `结果符合`
- `冲突判定`
- `因果稳定`
- `升阶判定`
- `可回接`
- `可等待`
- `已完成`
- `需收束`

### 5.4 任务管理任务模块

只负责：

- 读取任务/方法结构
- 收集二次特征证据
- 调用原语
- 选择下一步
- 写回任务树/方法树/学习承接/运行包

### 5.5 因果层

负责把：

`特征变化 -> 二次特征 -> 实例因果 -> 因果模板 -> 方法条件/方法结果 -> 任务树调用`

真正串成主链。


## 6. 分阶段实现计划

### 阶段 0：建立判断迁移总表

目标：

先把当前 `任务管理任务模块.impl.cpp` 中的判断完整分拣，不急着改代码。

工作项：

1. 梳理所有“推进依据”和“完成判据”。
2. 为每一条判断标记归属：
   - `应变成任务/方法结构`
   - `应变成二次特征`
   - `应保留为最小判断原语`
3. 输出一张迁移表，作为后续所有重构的唯一入口清单。

验收标准：

- 不再新增未归档的私有业务判断。
- 后续所有迁移都以该表为依据。


### 阶段 1：建立任务治理二次特征证据层

目标：

把当前最核心的运行时判断，先立成正式二次特征证据。

第一批必须收口的证据：

1. `任务条件满足`
2. `目标结果方向一致`
3. `目标结果已落范围`
4. `当前存在显式冲突`
5. `影子验证稳定`
6. `控制请求待确认`
7. `控制请求已拒绝`
8. `高阶排序已完成`
9. `高阶排序胜出已绑定`
10. `当前需等待`
11. `当前需收束`
12. `当前需重筹办`
13. `当前可回接最近结果`
14. `业务子任务已完成`
15. `最近结果命中当前治理`
16. `当前步骤可复用`

工作项：

1. 在任务治理证据写回链中正式写入这些二次特征。
2. 建立统一命名与摘要。
3. 停止新增对应的裸布尔临时变量。

验收标准：

- 第一批关键推进依据都有对应二次特征节点或二次特征读取口。


### 阶段 2：把原语改成纯证据解释器

目标：

让原语只解释二次特征，不再直接读取裸上下文。

工作项：

1. 重构：
   - `私有_P0_执行原语(...)`
2. 移除它对以下内容的直接依赖：
   - `最近功能域`
   - `最近根层重判结果`
   - `最近执行前门控结果`
   - `影子验证状态` 文本
   - 直接现算的对齐/待推进布尔
3. 改为只消费阶段 1 输出的二次特征证据。

验收标准：

- 原语成为“纯解释器”。
- 同一业务判断不再在原语层和步骤子链层各算一遍。


### 阶段 3：收口步骤结果子链

目标：

把 `步骤/结果推进` 从私有逻辑驱动，改成“结构 + 证据 + 原语”驱动。

工作项：

1. 重构：
   - `私有_推进步骤结果子链(...)`
2. 逐步替换这些直接逻辑：
   - 最近结果可回接
   - 当前步骤可复用
   - 结果节点是否命中当前治理
   - 是否转执行位
   - 步骤位类型推导
   - 步骤语义类型推导
3. 优先把“结构表达”的部分从任务/方法信息上前移。

验收标准：

- 步骤结果子链不再是最大的私有判断黑箱。


### 阶段 4：把完成/等待/收束收成正式原语

目标：

把“任务什么时候算完成/等待/收束”从零散函数收成正式原语组。

建议新增或收口的原语：

1. `可等待`
2. `已完成`
3. `需收束`
4. `可回接`
5. `可继续推进`

首批改造对象：

- `私有_可进入等待分支(...)`
- `私有_业务子任务执行已完成(...)`
- `私有_结果节点匹配当前治理(...)`
- `私有_稳定等待态复盘学习(...)`

验收标准：

- 完成判据不再散落在各私有函数里。


### 阶段 5：前推因果驱动方法树/任务树生成

目标：

让因果层真正成为方法树和任务树的生成主链。

工作项：

1. 在状态差分之后，强制生成条件侧和结果侧二次特征。
2. 由这组二次特征生成或补全实例因果。
3. 当实例因果稳定复现时，补全或沉淀因果模板。
4. 由因果模板自动补：
   - 方法条件节点
   - 方法结果节点
5. 让任务树优先调用已有因果驱动的方法链，而不是继续写散乱判断。

验收标准：

- 方法树/任务树的新增节点，开始能追溯到因果沉淀链。


### 阶段 6：瘦身任务管理任务模块

目标：

把任务管理模块压回“薄编排层”。

工作项：

1. 清理 `impl.cpp` 中可以被替代的私有推理函数。
2. 保留仅有的三类逻辑：
   - 结构装配
   - 原语调用
   - 写回与同步
3. 禁止新增“直接吃裸上下文做业务判断”的 helper。

验收标准：

- `任务管理任务模块.impl.cpp` 不再继续膨胀为业务判断中心。


## 7. 近期施工顺序

近期不建议平均铺开。

建议按下面顺序推进：

1. `阶段 0：判断迁移总表`
2. `阶段 1：第一批任务治理二次特征`
3. `阶段 2：原语纯证据化`
4. `阶段 3：步骤结果子链收口`
5. `阶段 4：完成/等待/收束原语化`
6. `阶段 5：因果驱动方法树/任务树`
7. `阶段 6：模块瘦身`

原因：

- 先有迁移总表，后面才不会边改边乱。
- 先做证据层，再做原语层，顺序更稳。
- 先收口步骤结果子链，再推进因果主链，能减少回滚。


## 8. 第一批明确要迁移的判断

第一批建议直接列为施工对象：

1. `私有_任务节点目标结果已对齐(...)`
2. `私有_任务节点目标结果待推进(...)`
3. `私有_可进入等待分支(...)`
4. `私有_业务子任务执行已完成(...)`
5. `私有_结果节点匹配当前治理(...)`
6. `私有_推进步骤结果子链(...)`
7. `私有_P0_执行原语(...)`

原因：

这 7 个点基本就覆盖了：

- 推进依据
- 完成判据
- 回接判据
- 原语解释
- 子链推进


## 9. 完成判据

本计划是否算完成，只看下面 6 条：

1. `任务树/方法树` 结构表达已经替代一批原 impl 里的步骤/结果推理。
2. 关键运行时判断已经有正式二次特征证据。
3. 原语层不再直接读取裸上下文做业务判断。
4. 步骤结果子链不再自行私算主要推进逻辑。
5. 因果信息开始正式参与方法树/任务树生成。
6. `任务管理任务模块` 体积和判断复杂度明显下降，回到编排层定位。

本次不推倒重写。

本次采用：

`保留当前骨架，迁移判断证据源，收口原语解释层，前推因果驱动方法树/任务树生成，最终把任务管理任务模块压回薄编排层。`
