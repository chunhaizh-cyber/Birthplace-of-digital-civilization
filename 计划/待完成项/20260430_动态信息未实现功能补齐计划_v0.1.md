4. 跨层输入不生成动态。

## 4. 阶段 2：状态写入后自动生成状态动态

状态：已完成。

目标：

状态写入后自动查找同主体同特征前状态，并生成状态动态。

实现内容：

1. 在状态创建成功后进入统一整理入口。
2. 查找同场景、同主体、同特征的最近前状态。
3. 前后状态有变化时生成状态动态。
4. 生成后触发现有动态二次特征生成入口。
5. 生成后尝试进入连续同向段或同类事件段聚合。

输入特征：

| 输入 | 类型 | 范围 |
| --- | --- | --- |
| 新状态 | 状态节点引用 | 必须有效 |
| 前状态 | 状态节点引用 | 可空 |
| 状态主体 | 节点引用 | 存在、场景、虚拟存在 |
| 状态特征类型 | 特征类型节点引用 | 必须有效 |
| 状态值差 | I64 | 后值减前值 |

返回特征：

| 返回 | 类型 | 方向 |
| --- | --- | --- |
| 状态动态生成数量 | I64 | 越大表示有效变化越多 |
| 零变化跳过数量 | I64 | 越大表示重复状态越多 |
| 无前状态跳过数量 | I64 | 越大表示初始状态越多 |

验证：

1. 第一次状态写入不生成状态动态。
2. 同主体同特征值变化生成 1 个状态动态。
3. 同值重复写入不生成状态动态。
4. 不破坏现有状态清理逻辑。

## 5. 阶段 3：真实方法动作动态统一出口

状态：已完成。

目标：

把真实本能方法动作函数执行结果统一转为自我方法动作动态。

实现内容：

1. 梳理真实方法动作函数执行入口。
2. 方法函数执行前记录输入场景、方法首节点、目标特征。
3. 方法函数执行后记录输出场景、执行结果、错误码。
4. 成功或失败都可记录动作动态，但必须标明执行结果。
5. 非真实动作执行路径不得写 `来源类型 = 方法调用`。

输入特征：

| 输入 | 类型 | 范围 |
| --- | --- | --- |
| 方法首节点 | 方法节点引用 | 必须有效 |
| 动作函数指针或动作 ID | 指针/I64 | 必须存在其一 |
| 输入场景 | 场景节点引用 | 可空但要可解释 |
| 输出场景 | 场景节点引用 | 可空 |
| 执行结果 | 枚举 | 成功、失败、未执行 |

返回特征：

| 返回 | 类型 | 值或方向 |
| --- | --- | --- |
| 动作动态生成数量 | I64 | 越大表示真实执行记录越完整 |
| 方法执行成功 | 布尔 0/1 | 1 成功，0 失败 |
| 错误类别 | 枚举 | 无、缺动作函数、输入不满足、执行失败 |

验证：

1. 本能方法真实执行后生成动作动态。
2. 只有方法头和输入检查不生成动作动态。
3. 筹办、调度、消息处理不生成方法调用动作动态。

## 7. 阶段 5：因果样本与稳定性更新

状态：已完成。

目标：

让动态样本可以推进因果模板从候选到稳定。

实现内容：

| 成立次数 | I64 | 大于等于 0 |
| 失败次数 | I64 | 大于等于 0 |

返回特征：

| 返回 | 类型 | 值或方向 |
| --- | --- | --- |
| 因果阶段 | 枚举 | 候选、临时、待验证、稳定 |
| 稳定值 | I64 | 越大越稳定 |
| 是否已验证 | 布尔 0/1 | 1 可用于推理 |

验证：

1. 新样本可以创建或更新候选因果。

## 8. 阶段 6：存在动态与场景动态

状态：入口已实现，自动触发待后续确认。

目标：

在状态动态稳定后，补齐存在层和场景层动态。

实现内容：

2. 父场景观察直接子场景状态变化，生成场景动态。
3. 不允许父场景直接读取孙层状态。
4. 来源低层动态可以被引用，但不能作为唯一来源。

输入特征：

| 输入 | 类型 | 范围 |
| --- | --- | --- |
| 观察场景 | 场景节点引用 | 必须有效 |
| 被观察存在 | 存在节点引用 | 必须是直接子存在 |
| 父场景 | 场景节点引用 | 场景动态时必须有效 |
| 子场景 | 场景节点引用 | 必须是直接子场景 |

返回特征：

| 返回 | 类型 | 方向 |
| --- | --- | --- |
| 存在动态生成数量 | I64 | 越大表示场景对存在观察越充分 |
| 场景动态生成数量 | I64 | 越大表示父场景观察越充分 |
| 跨层拒绝数量 | I64 | 越大表示调用链仍有问题 |

验证：

1. 子存在变化可在场景层形成存在动态。
2. 子场景状态变化可在父场景层形成场景动态。
3. 非直接子层拒绝生成。

## 9. 阶段 7：多特征协同动态

状态：第一版已完成。

目标：

识别共同父层下不同特征变化之间的关系。

实现内容：

1. 以时间窗口收集同主体或同父层下多个状态动态。
2. 判断同步、前置、滞后、抑制关系。

4. 稳定后进入因果候选。

输入特征：

| 输入 | 类型 | 范围 |

| 动态 A | 动态节点引用 | 必须有效 |
| 动态 B | 动态节点引用 | 必须有效 |
| 时间差 | I64 | 可正可负可零 |
| 特征值变化量 A | I64 | 可正可负可零 |
| 特征值变化量 B | I64 | 可正可负可零 |

返回特征：

| 返回 | 类型 | 值 |
| --- | --- | --- |
| 协同关系 | 枚举 | 同步、前置、滞后、抑制、无关系 |
| 关系强度 | I64 | 越大关系越强 |
| 是否进入因果候选 | 布尔 0/1 | 1 进入 |

验证：

1. 时间变化和特征值变化可形成单位变化关系。
2. 多个特征类型都参与比较。
3. 未稳定关系不直接升格为因果。

## 10. 阶段 8：状态清理闭环补齐

状态：第一版已完成。

目标：

在已有活动索引清理基础上，增加清理失败后的自检需求生成。

实现内容：

1. 保持第一版只清活动索引，不物理删除状态节点。

输入特征：

| 输入 | 类型 | 范围 |
| --- | --- | --- |
| 活动状态数量 | I64 | 大于等于 0 |
| 限额 | I64 | 大于 0 |
| 可清理数量 | I64 | 大于等于 0 |
| 被引用数量 | I64 | 大于等于 0 |

返回特征：

| 返回 | 类型 | 值或方向 |
| --- | --- | --- |
| 清理成功数量 | I64 | 越大越好 |
| 清理失败数量 | I64 | 越大问题越严重 |

验证：

1. 最新状态不会被清掉。
2. 被动态或二次特征引用的状态不会被误清。
3. 超限且无法清理时形成可处理问题。
