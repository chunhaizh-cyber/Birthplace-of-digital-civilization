# 20260417_世界树相关模块迁移计划_v0.1

## 1. 目标与范围

本计划用于将 `D:\鱼巢20260404` 中以下模块的有效能力迁移到当前项目 `D:\鱼巢`：

- `特征类`
- `存在类`
- `场景类`
- `世界树类`

说明：

- 你的消息里 `特征类` 出现了两次，这里先按一次处理。
- 如果第二个 `特征类` 实际指的是 `特征值类` 或 `二次特征类`，可以在本计划执行时并入第 2 阶段和第 3 阶段。

本次迁移目标不是“原样拷贝旧工程 `.ixx` 文件”，而是把旧工程里已经验证过的结构与行为，按当前工程的分层方式回灌进现有 `.h/.cpp + .ixx` 体系。

已确认决策：

- 主信息结构以当前项目为主。
- 旧工程中的主信息字段、字段命名、字段组织方式，只能作为迁移参考，不能反向覆盖当前项目的数据模型。
- 旧工程优先迁移“行为逻辑、接口语义、门面能力”，不迁移“旧主信息定义体系”。

## 2. 现状判断

### 2.1 来源工程的特点

来源工程中，目标模块不是简单 CRUD，而是已经形成了“世界树统一编排入口”：

- `D:\鱼巢20260404\特征类.ixx`
  - 具备候选窗口、稳态原型、连续测量误差区间、特征集合比较、度量签名生成。
- `D:\鱼巢20260404\存在类.ixx`
  - 以世界链为核心，提供存在与特征节点的结构化 CRUD。
- `D:\鱼巢20260404\场景类.ixx`
  - 场景 CRUD 很薄，但依赖 `世界树` 做统一创建和默认特征初始化。
- `D:\鱼巢20260404\世界树类.ixx`
  - 负责默认世界、自我内部世界、按名称/类型取或创建、按宿主+特征类型写入特征、存在比较、二次特征编排。

### 2.2 当前工程的特点

当前工程已经不是旧工程的同构版本，而是新的承载方式：

- `D:\鱼巢\特征类.cpp`
  - 当前仅完成“写当前值 + 记录统计”的最小实现，还没有迁入旧版稳态写入、集合比较、度量签名逻辑。
- `D:\鱼巢\基础信息主信息类.h`
  - 已经保留了 `区间`、`稳态列表`、`连续测量误差区间` 等字段，但缺少旧版写入逻辑需要的 `当前稳态索引`、近帧候选缓存、近帧时间缓存等运行字段。
- `D:\鱼巢\存在类.h/.cpp`
  - 在旧版基础上新增了 `需求根节点 / 任务根节点 / 方法根节点`，已经和当前任务系统耦合，不能被旧版直接覆盖。
- `D:\鱼巢\场景类.h/.cpp`
  - 当前采用“索引引用”设计，保留 `状态索引 / 动态索引 / 二次特征索引 / 实例因果索引`，比旧版更贴近当前工程。
- `D:\鱼巢\世界树类.h/.cpp`
  - 当前更像服务聚合器，已经负责默认世界、自我存在、内部世界、特征值池接入，但还缺少旧版 `世界树` 的一批便捷门面和比较编排能力。

### 2.3 结论

不建议直接复制以下旧版文件进入当前项目：

- `D:\鱼巢20260404\特征类.ixx`
- `D:\鱼巢20260404\存在类.ixx`
- `D:\鱼巢20260404\场景类.ixx`
- `D:\鱼巢20260404\世界树类.ixx`

原因：

- 旧版依赖 `数据仓库模块`、`主信息定义模块`、`世界树环境模块`、`概念树模块`、`特征类型定义模块` 等模块化入口。
- 当前工程的等价承载点已经拆散到 `基础信息类`、`基础信息主信息类`、`世界树类`、`二次特征生成模块`、`自我初始化模块` 等文件中。
- 若整块替换，最容易破坏当前已经落地的任务管理、自我线程、学习模块接口。

因此应采用：

`数据结构补齐 -> 服务逻辑回灌 -> 世界树门面兼容 -> 回归验证`

这里的“数据结构补齐”特指：

- 仅在当前主信息结构无法承载旧逻辑时，向当前结构补最小必要字段。
- 不把当前项目的主信息类整体改造成旧工程版本。

## 3. 迁移原则

1. 保留当前工程的主信息模型，不回退当前项目里已经新增的任务/方法/需求根节点设计。
2. 保留当前 `场景类` 的索引化设计，不把旧版“对象副本列表”重新搬回来。
3. 旧版 `世界树类` 的价值主要迁移为“门面能力”和“编排能力”，不是迁成一个超大杂糅文件。
4. 旧版 `.ixx` 中可复用的是算法和接口语义，不是源码形态本身。
5. 每一阶段都必须先完成可编译，再继续叠加行为能力，避免一次性大迁移导致问题不可定位。
6. 若当前主信息结构已经有等价字段，则优先复用当前字段，不再为贴合旧版命名而新增重复字段。
7. 若旧版主信息字段与当前结构冲突，以当前项目语义为准，由旧逻辑做适配。

## 5. 分阶段迁移方案

### 阶段 0：建立迁移基线

目标：先确认当前工程的可编译基线和影响面。

工作项：

- 编译当前工程 `Debug|x64`，记录基线状态。
- 以这 4 个模块为起点，列出会直接受影响的文件：
  - `基础信息主信息类.h/.cpp`
  - `特征类.h/.cpp`
  - `存在类.h/.cpp`
  - `场景类.h/.cpp`
  - `世界树类.h/.cpp`
  - `二次特征生成模块.h/.cpp`
  - `自我初始化模块.cpp`
- 明确“兼容层优先”策略：
  - 旧调用语义优先通过新增重载/新增门面实现。
  - 现有调用点尽量不大规模改名。

完成标志：

- 当前工程可作为迁移前基线重新编译。
- 受影响文件列表固定。

### 阶段 1：补齐主信息与运行时结构

目标：先把旧版逻辑所依赖的数据结构补齐到当前工程。

工作项：

- 在 `D:\鱼巢\基础信息主信息类.h` 为 `特征节点主信息类` 补齐旧版运行字段：
  - `当前稳态索引`
  - `近帧候选`
  - `近帧时间`
  - 必要时补 `区间语义`、`区间代表点`
- 明确当前 `统计` 字段和旧版 `命中次数 / 可信度` 的职责关系：
  - 若能复用 `统计`，优先复用。
  - 若不能复用，再局部补字段。
- 确认 `存在节点主信息类`、`场景节点主信息类` 继续保留当前项目新增字段，不按旧版回退。
- 所有新增字段都必须满足“当前项目缺它就无法承载旧逻辑”，不能为了和旧版主信息长得更像而机械补齐。

风险点：

- 候选窗口类型如果直接照搬旧版，可能和当前头文件组织冲突。
- `特征值` 与旧版 `特征快照值` 的命名和字段语义需要统一。

完成标志：

- `特征节点主信息类` 具备承载旧版特征写入算法的最小字段集。
- 不破坏当前存在/场景主信息结构。

### 阶段 2：迁移 `特征类`

目标：把旧版 `特征类` 的“最小写入类”升级为“写入 + 比较 + 稳态”的正式服务。

工作项：

- 在 `D:\鱼巢\特征类.h/.cpp` 新增或补齐以下结构与接口：
  - `稳态策略`
  - `写入参数`
  - `特征写入结果`
  - `集合比较参数`
  - `特征比较明细`
  - `存在比较结果`
- 迁移旧版 `写入特征值` 核心逻辑：
  - 当前快照更新
  - 候选窗口写入
  - 连续稳定判定
  - 稳态命中/创建
  - 容量修剪/超时淘汰
- 迁移旧版 `比较特征集合`
  - 先实现 I64 / `VecU句柄` / `指针句柄` 的差异度比较
  - 保留缺失惩罚和归一化差异度
- 迁移旧版 `生成度量签名`
  - 当前即便先提供最小版本，也要把接口定下来
- 旧版算法在落地时统一适配当前 `特征节点主信息类` 与当前 `特征值` 表达，不为了算法直接改写当前主信息体系。

建议：

- 先保持当前 `写入特征值_I64 / VecU / 指针` 旧接口不动，再在其内部改走新核心逻辑。

完成标志：

- `特征类` 能独立支持旧版“写入、稳态、比较、签名”四类能力。

### 阶段 3：迁移 `存在类` 与 `场景类`

目标：在不破坏当前任务和索引体系的前提下，引入旧版便捷接口语义。

工作项：

- `D:\鱼巢\存在类.h/.cpp`
  - 保留当前 `需求根节点 / 任务根节点 / 方法根节点`
  - 可新增旧版风格的便捷方法，但底层仍调用当前实现
  - 重点考虑是否补两个接口：
    - `取或创建子存在_按类型`
    - `取或创建子存在_按类型并按特征I64`
- `D:\鱼巢\场景类.h/.cpp`
  - 保留当前 `状态索引 / 动态索引 / 二次特征索引`
  - 对齐旧版语义：
    - 创建场景时由 `世界树` 负责默认特征初始化
    - `绑定宿主`、`绑定来源`、`追加索引` 的职责不回退

注意：

- 这一阶段不要把旧版 `存在类 / 场景类` 里对世界链加锁、默认特征创建、全局门面行为直接搬进类内部。
- 这些行为更适合继续收口到 `世界树类`。
- 不得因为兼容旧版便捷接口而回退当前 `存在节点主信息类`、`场景节点主信息类` 的字段设计。

完成标志：

- `存在类` 和 `场景类` 成为“当前结构 + 旧版便捷语义”的兼容实现。

### 阶段 4：迁移 `世界树类` 门面与编排能力

目标：把旧版 `世界树` 最有价值的统一入口能力补到当前项目。

工作项：

- 在 `D:\鱼巢\世界树类.h/.cpp` 补齐以下门面能力：
  - `取或创建子场景_按名称`
  - `取或创建子存在_按类型`
  - `取或创建子存在_按类型并按特征I64`
  - `确保特征`
  - `按宿主 + 特征类型` 的写入/读取接口
  - `比较存在`
- 保留当前已经存在的能力：
  - 默认六大世界初始化
  - 自我存在初始化
  - 自我内部世界 / 存在内部世界创建
  - `特征值类` 证据池对 `VecU` 的托管
- 对旧版“创建场景时补默认特征”的行为做显式决策：
  - 若当前项目已有初始化入口，优先在 `世界树::创建场景` 里调公共初始化函数
  - 不建议把默认特征分散到 `场景类` 内部隐式创建

建议拆分：

- `世界树类` 只迁门面和编排。
- 若旧版 `由存在比较结果生成二次特征` 与当前 `二次特征应用模块` 语义重合，优先迁到 `D:\鱼巢\二次特征应用模块.ixx`，不要把生成算法硬塞回 `世界树类.cpp`。

完成标志：

- 旧版以 `世界树` 为入口的核心调用语义，当前项目已有等价门面。

### 阶段 5：联调与回归

目标：确认迁入后的结构能和当前任务、自我线程、学习链路共存。

最小验证清单：

- 世界初始化后，六大世界仍可正常创建。
- `自我存在 -> 自我内部世界` 链路仍可正常建立。
- `按类型取或创建存在` 行为正确。
- `按类型并按特征I64取或创建存在` 行为正确。
- 场景创建后默认特征初始化行为符合当前项目预期。
- 特征写入后：
  - `当前值` 正确
  - 候选窗口有记录
  - 稳态原型可形成
- 两个存在的特征集合可比较，输出差异度结果。
- 任务管理、自我线程相关模块能够继续编译通过。

## 6. 推荐实施顺序

推荐按下面顺序落地，不要并行乱改：

1. `基础信息主信息类.h/.cpp`
2. `特征类.h/.cpp`
3. `世界树类.h/.cpp`
4. `存在类.h/.cpp`
5. `场景类.h/.cpp`
6. `二次特征生成模块.*`
7. 编译与联调

原因：

- 真正的依赖核心在 `特征主信息结构 + 特征服务 + 世界树门面`。
- `存在类` 和 `场景类` 在当前工程里已经较稳定，应尽量后移，减少返工。

## 7. 主要风险

### 风险 1：主信息字段语义不完全同构

旧版 `特征快照值`、旧版稳态字段和当前 `特征值/统计` 不是 1:1 同名结构，直接照抄很容易导致混乱。

控制方式：

- 先统一字段语义，再写实现。
- 优先把旧版逻辑适配到当前命名，而不是把当前结构改回旧版命名。
- 若出现“一份当前字段 + 一份旧版同义字段”并存的倾向，应直接禁止，避免后续双轨维护。

### 风险 2：`世界树` 迁移过大，冲击当前任务系统

当前工程的 `存在类` 已经绑定任务/需求/方法根节点，`世界树` 一旦整体替换，最容易把这条链打断。

控制方式：

- 只迁门面和编排。
- 现有任务系统相关字段和接口不得删除。

### 风险 3：旧版二次特征编排与当前模块重复

来源项目把一部分比较后生成逻辑放在 `世界树类`，当前项目已有 `二次特征生成模块`。

控制方式：

- 先迁存在比较结果本身。
- 二次特征生成逻辑单独评估后再决定是否合并。

### 风险 4：工程形态差异导致“源码能贴上但编不过”

来源工程大量依赖 `.ixx` 模块导入，当前工程虽然也编译部分 `.ixx`，但主承载已转到 `.h/.cpp`。

控制方式：

- 迁逻辑，不迁整块模块依赖。
- 所有新增能力优先落在当前已被 `vcxproj` 管理的文件里。

## 8. 验收标准

满足以下条件即可认为迁移完成：

1. 当前工程可在 `Debug|x64` 下完整编译。
2. 旧版四个目标模块的核心能力在当前工程已有等价实现：
   - `特征类`：写入、稳态、比较、签名
   - `存在类`：CRUD + 取或创建便捷能力
   - `场景类`：CRUD + 当前索引化引用能力不丢失
   - `世界树类`：统一门面、默认世界、自我内部世界、存在比较
3. 不破坏当前任务管理、自我线程、学习调度模块编译和初始化链路。

## 9. 建议的第一次落地范围

建议第一轮只做“可编译、可联调、可验证”的最小闭环，不追求一次迁全：

- 先迁 `特征类` 的写入与比较
- 再补 `世界树类` 的 `取或创建` 门面
- 最后补 `存在类/场景类` 的兼容便捷接口

这样能最快拿到一个能跑的中间版本，也最利于后续继续迁 `二次特征`、`概念树`、`特征类型定义` 等旧版能力。
