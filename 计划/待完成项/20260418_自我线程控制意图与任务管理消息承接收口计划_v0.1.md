# 20260418 自我线程控制意图与任务管理消息承接收口计划 v0.1

## 1. 计划目标

本文只回答一件事：

`在“自我线程不直接改其它线程状态”的口径下，如何把线程控制链收口成“自我线程发控制意图 -> 任务管理任务模块承接 -> 工作线程执行并回报 -> 自我线程消费回报”的稳定工程闭环。`

本计划不是新规范正文，不直接改写上位规范。

本计划只把当前已讨论清楚的约束压成一份可施工计划，供当前仓库继续推进。


## 2. 已确认口径

必须固定：

1. `自我线程` 只允许直接切换自己的生命周期状态，不应直接更改其它线程的事实状态。
2. `自我线程` 对外线程只能发：
   - `控制意图`
   - `控制投影`
   - `治理消息 / 上行消息`
   不能直接把对方写成 `已暂停 / 已停止 / 已收束 / 已恢复`。
3. `任务管理任务模块` 必须成为这条控制链的正式消息承接方，而不是让控制意图直接散落在线程侧和工作线程侧之间。
4. `工作线程` 只负责：
   - 消费控制意图
   - 更新自己的局部运行状态 / 实际运行状态
   - 回报执行结果与状态变化
5. `自我线程` 最终只消费：
   - 控制意图是否已下发
   - 对方是否已响应
   - 对方实际运行状态
   - 对账与回流结果
6. `控制意图` 与 `实际运行状态` 必须继续分离，不得混写。


## 3. 当前代码判断

### 3.1 已有基础

当前工程已经具备这条链的一部分基础：

- `自我线程` 已有自己的生命周期控制口：
  - `D:\鱼巢\自我线程模块.ixx`
  - `请求停止 / 等待停止 / 停止`
- `自我线程消息协议` 已区分：
  - `控制意图`
  - `实际运行状态`
  - `生命周期状态`
- `任务管理任务模块.消息协议` 已具备：
  - `线程状态消息`
  - `结果消息`
  - `是否已响应控制意图` 这类补充特征位
- `任务模块.运行包` 已具备“高阶绑定字段”和“任务包/结算包校验”这类正式消息承载习惯，可继续作为控制投影的宿主容器。

### 3.2 当前主要缺口

当前最关键的缺口不是“有没有线程状态消息”，而是下面 6 件事还没有收口：

1. `任务管理任务模块` 还没有成为正式的“控制意图承接方”。
2. 代码里还没有一套正式的 `控制意图值枚举 / 控制命令合同`。
3. `切换线程状态并上报(...)` 这类通用口权限过宽，容易越过“只改自己”的边界。
4. `控制意图 -> 对方已响应 -> 对方已执行/拒绝/超时` 的闭环还没有形成正式主链。
5. `是否允许继续派发 / 当前步骤可执行 / 当前任务是否阻塞` 这类二次特征还没有完整接入“控制意图已下发但未执行”和“控制意图已响应”的证据面。
6. 控制链仍偏“线程侧摘要 + 状态回报摘要”，还不够对象化、可追踪、可回放。

### 3.3 当前最缺的模块

当前最缺的正式承接方，确实就是：

`任务管理任务模块`

原因：

- 它位于 `自我线程` 和 `任务管理工作线程` 之间，天然适合做控制链的中间承接层。
- 它已经承担：
  - 任务管理状态消息
  - 结果消息
  - 执行尝试对账
  - 一步治理任务包
- 如果控制意图不先进入 `任务管理任务模块`，后续就很难稳定挂上：
  - 控制请求序号
  - 响应状态
  - 超时
  - 拒绝原因
  - 二次特征重算触发点


## 4. 目标控制链

目标链路固定为：

`自我线程`
-> `任务管理任务模块（控制意图承接方）`
-> `任务管理工作线程/工作线程（控制执行方）`
-> `任务管理线程状态消息/结果消息`
-> `自我线程`

每层职责固定如下：

### 4.1 自我线程

只负责：

- 生成控制意图
- 下发控制请求
- 记录控制请求摘要
- 等待承接方和执行方回报
- 根据回报重算二次特征、影响项、根层裁决

不负责：

- 直接修改其它线程的事实状态
- 直接把其它线程写成 `已停止 / 已收束 / 已暂停 / 已恢复`

### 4.2 任务管理任务模块

作为正式承接方，负责：

- 接收来自自我线程的控制意图
- 写入任务控制态/控制投影
- 形成正式控制请求记录
- 把控制请求投递到工作线程
- 记录“是否已响应控制意图 / 是否超时 / 是否拒绝 / 是否已完成”

### 4.3 工作线程

只负责：

- 消费控制意图或控制投影
- 改自己的局部运行状态/实际运行状态
- 回报线程状态消息和结果消息

### 4.4 自我线程回收侧

只负责：

- 消费承接方和工作线程回报
- 更新对账、二次特征、影响项、根层结果
- 决定是否继续、待机、收束、恢复、再派发


## 5. 分阶段实现计划

### 阶段 0：先收边界，不改大逻辑

目标：

先把“谁能改谁”钉成正式边界。

工作项：

1. 梳理并标记当前所有会触发线程状态改写的入口：
   - `D:\鱼巢\自我线程模块.ixx`
   - `D:\鱼巢\自我线程模块.impl.cpp`
   - `D:\鱼巢\自我线程模块.消息协议.ixx`
2. 将接口分成两类：
   - `自我线程自改`
   - `他线程控制请求`
3. 明确“他线程控制请求”不得继续走直接状态改写口。

完成标志：

- 形成一张现有入口表，明确哪些要保留，哪些要改为“只发控制意图”。


### 阶段 1：建立正式控制意图合同

目标：

把规范里的“控制意图”落成正式代码合同。

工作项：

1. 在控制域或消息协议层新增正式值域：
   - `请求启动`
   - `请求暂停`
   - `请求继续`
   - `请求打断`
   - `请求收束`
   - `请求停止`
   - `请求恢复重建`
   - `请求刷新预算`
   - `请求刷新优先级`
2. 建立控制请求结构，至少包含：
   - 请求主键
   - 目标线程主键
   - 目标任务管理任务主键
   - 控制意图值
   - 发起者
   - 发生时间
   - 请求原因
   - 是否需要确认
3. 先不落事实状态改写，只形成合同和序列化/摘要能力。

建议落点：

- `D:\鱼巢\自我线程模块.消息协议.ixx`
- `D:\鱼巢\任务管理任务模块.消息协议.ixx`

完成标志：

- 控制意图不再只是规范概念，而是正式代码值域。


### 阶段 2：让任务管理任务模块成为正式承接方

目标：

把“控制意图承接”从摘要层变成正式对象面。

工作项：

1. 在 `任务管理任务模块` 中新增最小承接结构，例如：
   - `结构_控制请求承接记录`
   - `结构_控制投影`
   - `结构_控制响应摘要`
2. 记录至少这些字段：
   - 控制请求主键
   - 对应执行尝试或任务主键
   - 当前控制意图值
   - 是否已下发
   - 是否已响应
   - 是否已执行完成
   - 是否超时
   - 拒绝原因
   - 最近回报时间
3. 让 `任务管理任务模块` 具备：
   - 接收控制请求
   - 查询最新控制投影
   - 合并状态回报
   - 输出控制承接摘要

建议落点：

- `D:\鱼巢\任务管理任务模块.ixx`
- `D:\鱼巢\任务管理任务模块.impl.cpp`
- `D:\鱼巢\任务管理任务模块.消息协议.ixx`

完成标志：

- 控制意图不再直接浮在 `自我线程 -> 工作线程` 两端，而是先进入 `任务管理任务模块`。


### 阶段 3：收口自我线程发起侧

目标：

让 `自我线程` 不再直接改他线程状态，而是只发控制请求。

工作项：

1. 收紧：
   - `切换线程状态并上报(...)`
   - `上报线程状态变化(...)`
   这类通用口的使用边界。
2. 新增 `自我线程 -> 任务管理任务模块` 的控制请求发起入口。
3. 自我线程只记录：
   - 已发什么控制意图
   - 发给谁
   - 为什么发
   - 是否要求确认
4. 自我线程不再直接把工作线程写成某个事实状态。

建议落点：

- `D:\鱼巢\自我线程模块.ixx`
- `D:\鱼巢\自我线程模块.impl.cpp`

完成标志：

- `自我线程` 对他线程只保留控制请求能力，不再保留事实状态改写能力。


### 阶段 4：收口工作线程执行与回报侧

目标：

让工作线程自己消费控制意图并回报事实状态。

工作项：

1. 让 `任务管理工作线程/工作线程` 消费控制投影。
2. 明确区分：
   - 收到控制请求
   - 已响应控制请求
   - 已执行完成
   - 无法执行 / 拒绝执行
   - 超时未执行
3. 把这些信息回写到：
   - `线程状态消息`
   - `结果消息`
   - `是否已响应控制意图`

建议落点：

- `D:\鱼巢\任务模块.管理工作线程.ixx`
- `D:\鱼巢\任务管理工作线程.cpp`
- `D:\鱼巢\任务管理任务模块.消息协议.ixx`

完成标志：

- 控制链形成“请求 -> 响应 -> 执行 -> 回报”最小闭环。


### 阶段 5：把控制链接回二次特征与原语层

目标：

让控制意图不只是消息链，也成为正式治理证据。

工作项：

1. 把下面这些判断接入二次特征和原语：
   - `当前步骤可执行`
   - `是否允许继续推进`
   - `当前任务是否阻塞`
   - `是否需要收束`
   - `是否需要恢复`
2. 输入证据至少包含：
   - `控制意图`
   - `是否已响应控制意图`
   - `实际运行状态`
   - `是否允许继续派发`
   - `当前等待原因`
3. 形成“控制证据 -> 二次特征 -> 治理原语 -> 主派发/恢复裁决”闭环。

建议落点：

- `D:\鱼巢\自我线程模块.impl.cpp`
- `D:\鱼巢\二次特征应用模块.ixx`
- `D:\鱼巢\自我_主循环治理模块.ixx`

完成标志：

- 控制链不再只是消息和摘要，而是正式进入治理判据层。


### 阶段 6：最后再做观察面和学习承接

目标：

让控制链可观察、可复盘、可学习。

工作项：

1. 在快照层和控制面板显示：
   - 最近控制请求
   - 最近响应状态
   - 超时/拒绝原因
   - 当前控制意图
   - 当前实际运行状态
2. 在学习承接中记录：
   - 当时下发了什么控制意图
   - 对方是否及时响应
   - 后续是否验证这条控制更合适

完成标志：

- 控制链具备完整观察和回放能力。


## 6. 第一优先级施工顺序

如果只按最小闭环推进，优先顺序固定为：

1. `任务管理任务模块` 先变成正式承接方
2. 再补正式 `控制意图值`
3. 再改 `自我线程` 发起侧
4. 再补工作线程响应闭环
5. 最后把控制证据接入二次特征与原语

也就是说：

`当前第一刀不该先改工作线程，也不该先改展示层，而是先改任务管理任务模块。`


## 7. 验收标准

本计划完成时，至少满足以下 7 条：

1. `自我线程` 不再直接改其它线程的事实状态。
2. 其它线程状态变化必须来自：
   - 线程自身执行
   - 线程自身回报
   - 或其直属拥有者的合法控制请求链
3. `任务管理任务模块` 成为正式消息承接方。
4. 控制意图有正式代码值域，不再只是规范描述。
5. 控制请求至少形成：
   - 已下发
   - 已响应
   - 已执行
   - 已拒绝/超时
   的最小闭环。
6. `当前步骤可执行 / 是否允许继续推进 / 是否需要收束` 等治理判断能正式消费控制证据。
7. 观察面能追踪一条控制请求从发起到回报的完整链路。


## 8. 一句话收口

下一阶段的主线，不是继续让 `自我线程` 变得更像“全局状态写入器”，而是：

`让它退回控制请求发起者与结果汇总者的位置，并把任务管理任务模块补成正式消息承接方。`
