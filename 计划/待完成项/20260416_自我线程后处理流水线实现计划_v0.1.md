必须固定：

1. 现实世界的动作动态，仍由 `任务管理任务 / 任务 / 本能动作` 三类执行体直接管理；`自我线程` 不直接代管这些动作动态本体。
2. `消息 -> 状态变化 / 治理侧动作摘要` 只叫 `入账`，不等于线程已经完成判断。
3. 本计划从 `一次入账完成之后` 开始，不再重复定义“冻结批次”和“消息转治理消息”的前置步骤。
4. `派生需求消息` 只能先形成 `需求候选`，不能越过根层重判直接写正式需求树。
5. `结果消息` 不能裸着直接改 `服务值 / 安全值`；必须先经过 `二次特征 -> 影响项`。
6. 子结果必须先回自我线程，由自我线程先做全局判断，再路由给父层任务；父层任务只判断局部影响。
7. 本计划以 **现有规范** 为准；若未来要把 `回流结算 / 学习触发 / 否定项承接` 从自我线程职责面进一步外移，必须先改规范，再改计划与代码。

-> `需求树与权重更新`
-> `根层重判与主派发`

## 5. 六模块分解与施工顺序

### 模块 1：对账与归并模块

职责：

- 幂等去重
- 线程状态消息与结果消息对账
- 执行尝试汇总
- 冲突检测
- 关键中间状态归并

当前代码现状：

- 已部分具备，但还混在 `步骤_规范化消息并落一次特征账_` 里。

下一步应做：

1. 把 `私有_归并治理消息并对账(...)` 抽成稳定模块入口。
2. 把 `执行尝试汇总表_`、对账结论、重复跳过逻辑统一收口。
3. 明确 `派生需求消息` 不进入执行尝试闭环，只进入缺口事实归并链。

完成标准：

1. 同一执行尝试重复消息不再重复改状态。
2. `线程状态消息 + 结果消息` 的闭环不再混在一次落账分支里。
3. 三类消息都能稳定产出统一归并结果对象。

### 模块 2：二次特征生成模块

职责：

- 把一次特征变化转成可判断信息。

第一批固定只做：

- `依赖满足`
- `当前步骤可执行`
- `当前任务阻塞 / 解阻`
- `反向推进标记`
- `方法可调用`
- `有效优先级候选`

当前代码现状：

- 已有最小实现，对应 `步骤_重算受影响二次特征_`。

下一步应做：

1. 固定统一输入结构：`归并后的一次事实 + 关键中间状态 + 当前镜像`。
2. 固定统一输出结构：`概念名 / 主体 / 标量值 / 原因摘要 / 来源消息源`。
3. 把当前分散 helper 收成正式模块接口，不再直接从主循环里拼判断。

完成标准：

1. 三类消息都能稳定触发对应的二次特征重算。
2. 原因摘要模板不再由各消息分支临时拼接。
3. 状态型和动态型二次特征都能走统一增量入口。

### 模块 3：影响项生成模块

职责：

- 把二次特征进一步归结到根值方向。

固定四类：

- `直接服务影响`
- `准备性服务影响`
- `物理安全影响`
- `风险安全影响`

当前代码现状：

- 已有最小实现，对应 `步骤_刷新线程上下文并生成治理帧_` 前半段。

下一步应做：

1. 把当前摘要层提升成正式 `影响项池`。
2. 每个影响项至少稳定持有：
   - `方向`
   - `强度`
   - `是否即时`
   - `是否可逆`
3. 稳定产出 `服务净变化候选 / 安全净变化候选`，供后续模块消费。

完成标准：

1. 不再允许一次特征直接裸着改 `服务值 / 安全值`。
2. 结果消息与关键中间状态能稳定转出影响项。
3. 负向影响项可直接成为 `否定项候选` 的上游输入。
