# 2026-04-12 二次特征信息生成与用途落地实施计划

## 1. 目标

本文用于把 `二次特征` 从“带签名的结果盒子”补成一条完整链路，固定以下问题：

- 二次特征应如何在现有分层内继续完善，而不推翻现有骨架
- 二次特征如何区分 `存在型` 与 `动态型`
- 二次特征如何显式记录自己的生成母式、粒度、基准和时间归一方式
- 二次特征如何稳定接入 `因果模板 / 因果实例 / 任务 / 方法`
- 当前应先做什么，哪些内容必须后置

本文不是抽象设计说明，而是后续代码落地的正式施工计划。


## 2. 总判断

当前代码骨架已经具备继续完善二次特征的基础：

- `特征节点主信息类`
  - 已负责原始值、区间、稳态和连续测量误差
- `状态节点主信息类`
  - 已负责状态主体、状态特征、状态值和时间
- `动态节点主信息类`
  - 已负责起始状态、结果状态、动态主体、动态特征、时间、来源动作和路径签名
- `二次特征主信息类`
  - 已负责形态、种类、度量签名、主体/客体、标量值、是否满足、置信度
- `因果模板主信息类`
  - 已能直接把二次特征作为 `条件模板 / 结果模板`
- `因果实例主信息类`
  - 已保留 `来源任务主键 / 来源方法主键`

因此当前最缺的不是重造底盘，而是把以下语义缝补严：

1. `存在型 / 动态型` 二次特征的分域
2. `值形态` 的显式化
3. 生成母式字段组
4. 多级动态与上层二次特征之间的来源链
5. `度量签名_链键` 与语义字段的职责分离
6. 因果模板按显式字段匹配的规则
7. 二次特征作为任务/方法目标时的独立绑定结构

### 2.1 规范先钉的五颗钉子

在正式改代码前，必须把以下五件事作为先验硬约束看待：

1. `来源链`
   - 至少承认 `来源状态 / 来源动态 / 来源场景 / 来源层级`
2. `值形态`
   - 至少承认 `布尔 / 标量 / 区间 / 离散或签名`
3. `签名与语义分离`
   - `度量签名_链键` 是索引壳，不是语义真身
4. `上下文角色`
   - `观察 / 条件 / 结果 / 目标 / 评估依据` 属于使用角色，不属于二次特征本体
5. `生成器归位`
   - 生成、聚合、匹配应由生成器/规则服务承担，不应把数据类写胖


## 3. 总目标

本轮完善固定为 4 个总目标：

1. `区分存在型和动态型二次特征`
2. `让二次特征显式承认自己的值形态`
3. `让二次特征知道自己是怎么生成出来的`
4. `让二次特征稳定接入因果、任务、方法`
5. `让二次特征生成遵守既有规则`

其中第 4 条的硬边界必须保持一致：

- 不在二次特征层重复存 `比较方式`
- 未过阈值不生成动态，也就不生成动态型二次特征
- 非相邻先降动态、再聚合、再在同级相邻上取二次特征


## 4. 不做什么

### 4.1 不在二次特征类里重复存比较方式

比较方式、稳态和误差仍归 `特征节点主信息类` 负责，二次特征只记录：

- 自己是什么
- 自己来自哪里
- 自己按什么母式生成

### 4.2 不把目标写成二次特征的固有属性

同一条二次特征在不同上下文里可能充当：

- 观察事实
- 因果条件
- 因果结果
- 任务目标

因此 `目标` 必须是任务/方法层的使用角色，不进入 `二次特征主信息类` 固有字段。

### 4.3 不靠未感知微差做补算

当前规则已固定：

- 没过阈值，不形成动态
- 未形成动态，不生成动态型二次特征

因此禁止在更高层用微差补算出动态型二次特征。


## 5. 当前最明显的缺口

### 5.1 二次特征层缺少生成母式字段组

当前 `二次特征主信息类` 已能回答：

- 它是什么种类
- 它的值是多少
- 属于哪个场景
- 主体客体或左右对象是谁

但还回答不了：

- 它是 `存在型` 还是 `动态型`
- 它是按 `差分`、`路径`、`参照偏离` 还是 `状态转移` 生成
- 它是 `相邻分步`、`首尾整体` 还是 `完整路径`
- 它是相对 `相邻前态`、`起始态`、`稳态` 还是 `目标态`
- 它的时间归一到底是 `分步时长` 还是 `总时长`
- 它具体来自哪一层动态、哪条路径

### 5.2 动态层缺少层级与聚合来源

当前 `动态节点主信息类` 已有：

- 起始状态
- 结果状态
- 时间
- 来源动作
- 动态路径签名

但还没有：

- 动态层级
- 聚合方式
- 来源低层动态链

这会直接阻塞“非相邻先降动态再聚合”这条规则的代码化。

### 5.3 场景侧索引名称过窄

当前 `场景节点主信息类` 存的是：

- `关系索引 : std::vector<可解析引用<二次特征节点类>>`

当二次特征扩大到动态路径、变化率、回返类型等语义后，`关系索引` 这个名字会越来越窄，需要引入更准确的 `二次特征索引`。


## 6. 阶段总表

推荐阶段顺序固定为：

1. `P1 数据结构补槽位`
2. `P2 统一二次特征填充规范`
3. `P3 落地存在型 / 动态型二次特征生成器`
4. `P4 落地动态聚合与上层二次特征`
5. `P5 重做度量签名_链键职责`
6. `P6 接因果模板与因果实例`
7. `P7 接任务/方法目标绑定`
8. `P8 测试与闭环场景验收`

工程顺序必须固定：

- 不先写大而全的动态型种类
- 不先写复杂学习调参
- 不把 V2/V3 的路径、闭环、周期一次性灌进 V1

## 8. P2：统一二次特征填充规范

### 8.1 对象位规范

#### 存在型二次特征

- 有方向关系：用 `主体 / 客体`
- 无方向比较：用 `左对象 / 右对象`
- 单对象修饰：只填 `主体`

#### 动态型二次特征

- 单对象动态：`主体 = 动态主体`
- 动态比较型：`左对象 / 右对象`
- 相对稳态、目标态的恢复或偏离：仍以 `主体` 为主

#### 禁止

- 不允许同一条二次特征随意混填四个对象位

### 8.2 值位规范

#### 标量型

- 用 `标量值`
- 模板侧可配 `标量区间`

#### 布尔型

- 用 `是否满足`
- `值形态 = 布尔`

#### 离散枚举型

- 用 `离散编码 + 离散语义键`
- `概念名称` 只做显示，不做计算主键

#### 签名型

- 优先用 `离散语义键` 或专用签名字段承载

总原则固定为：

`计算靠结构字段，表达靠概念名称`

并且必须继续固定：

`度量签名_链键` 只负责索引、去重、检索与查询加速，不得替代本组显式值形态与来源字段。


## 9. P3：落地生成链路

### 9.1 存在型二次特征生成器

输入：

- 场景
- 场景中存在节点
- 当前状态 / 当前特征

输出：

- 占用
- 冲突
- 包含
- 接近

写回：

- `场景.二次特征索引`

字段要求：

- `域 = 存在`
- `主母式 = 存在关系`
- `粒度 = 当前场景`
- `来源动态` 留空

V1 第一批只做一小组稳定种类：

- `占用`
- `冲突`
- `包含 / 接近`

### 9.2 动态型二次特征生成器

输入：

- 动态节点

前提：

- 动态已经成立
- 阈值过滤已在状态变化 / 动态生成阶段完成

V1 第一批动态种类固定为：

- `变化量`
- `变化方向`
- `变化率`

字段要求：

- `域 = 动态`
- `主母式 = 差分` 或 `时间归一`
- `粒度 = 相邻分步` 或 `首尾整体`
- `基准类型 = 相邻前态` 或 `起始态`
- `时间归一方式 = 分步时长` 或 `总时长`
- `来源起始状态 / 来源结果状态 / 来源动态` 全部写入
- `来源动态路径签名 = 动态路径签名`


## 10. P4：动态聚合与高层二次特征

当前规则固定为：

`多节点相邻逐步比较；非相邻先降动态再聚合，再在同级相邻上取二次特征`

因此实现顺序固定为：

1. 从状态序列生成 `0 层原子动态`
2. 用聚合规则把 `0 层动态` 压成 `1 层动态`
3. 在 `1 层同级相邻动态` 之间生成新的动态型二次特征

V1 只落一种聚合方式：

- `连续同向段聚合`

路径型二次特征必须后置到高层动态之后，再补：

- 转折点数量
- 峰值/谷值偏移
- 回返类型
- 闭环类型
- 路径签名类特征


## 11. P5：重做 `度量签名_链键` 的职责

### 11.1 显式字段是真身

以下字段是真正的语义本体：

- `域`
- `种类`
- `主母式`
- `粒度`
- `基准类型`
- `时间归一方式`
- `主体 / 客体 / 左对象 / 右对象`
- `来源动态层级`
- `来源动态路径签名`

### 11.2 `度量签名_链键` 退回索引角色

`度量签名_链键` 由显式字段自动生成，只负责：

- 去重
- 索引
- 查询加速
- 排序辅助

必须继续固定：

- 模板匹配、任务判定、方法选择等需要语义真值的逻辑，不得只依赖 `度量签名_链键`

建议格式固定为：

```text
域|种类|主母式|粒度|基准|时间归一|主体|客体|左|右|层级|路径签名
```

### 11.3 拆出语义比较与模板匹配

不应再把模板匹配塞进 `比较()`。

建议新增：

```cpp
bool 语义相同(const 二次特征主信息类& 右) const;
bool 模板匹配(const 二次特征主信息类& 模板) const;
```

其中：

- `比较()` 负责排序
- `语义相同()` 负责去重与语义等价
- `模板匹配()` 负责因果和任务/方法的真正匹配逻辑


## 12. P6：接因果模板与因果实例

### 12.1 因果模板先不改结构

当前 `因果模板主信息类` 已有：

- `条件模板`
- `动作模板`
- `结果模板`
- `证据实例`

V1 不需要先重做结构，重点改匹配规则。

### 12.2 模板匹配规则

条件模板和结果模板的二次特征匹配应按以下字段进行：

- `种类`
- `域`
- `主母式`
- `粒度`
- `基准类型`
- `主体/客体角色`
- `值`

值匹配规则固定为：

- 标量：看 `标量区间`
- 枚举：看 `离散编码 / 离散语义键`
- 布尔：看 `是否满足`

### 12.3 第一批闭环

V1 先打通：

`状态变化 -> 动态 -> 动态型二次特征 -> 命中因果模板 -> 形成因果实例`


## 13. P7：任务/方法目标绑定

### 13.1 原则

二次特征可以成为任务或方法的目标，但：

- 目标绑定属于 `任务 / 方法 / 环境` 层
- 不进入 `二次特征主信息类` 本体

### 13.2 建议结构

```cpp
enum class 枚举_目标作用 : std::uint8_t {
    未定义 = 0,
    达到 = 1,
    维持 = 2,
    消除 = 3,
    降低 = 4,
    提高 = 5,
};

struct 任务目标绑定 {
    可解析引用<二次特征节点类> 目标二次特征{};
    枚举_目标作用 作用 = 枚举_目标作用::未定义;
    std::optional<I64区间> 目标区间{};
    bool 硬约束 = false;
    std::int32_t 优先级 = 0;
};
```

### 13.3 V1 第一批用法

- 消除冲突
- 降低占用压缩程度
- 提高恢复速度
- 维持接近安全边界内

## 15. V1 / V2 / V3 范围控制

### V1 最小闭环

数据结构：

- `二次特征主信息类`
  - 新增 `域 / 主母式 / 粒度 / 基准类型 / 时间归一方式 / 来源状态 / 来源动态 / 来源场景 / 来源层级 / 值形态`
- `动态节点主信息类`
  - 新增 `动态层级 / 聚合方式 / 来源低层动态`
- `场景节点主信息类`
  - 新增 `二次特征索引`

生成能力：

- 存在型二次特征一组
- 动态型二次特征一组：`变化量 / 方向 / 变化率`
- 动态聚合一组：`连续同向段聚合`

模板能力：

- 因果模板按新字段匹配
- 因果实例可回写

V1 不做：

- 周期相位
- 闭环类型
- 复杂结构变更
- 自动学习调参

### V2

- 路径型二次特征
- 参照偏离型二次特征
- 非标量值形态
- 任务目标绑定
- 方法选择时按二次特征目标筛选

### V3

- 周期 / 相位
- 状态转移 / 驻留
- 结构变更型二次特征
- 规则版本迭代和学习回写


## 16. 实际施工顺序

推荐按下面顺序落地：

1. 先改 `基础信息主信息类.h`

   - 拆出 `语义相同 / 模板匹配`
4. 再写两个生成器
   - 存在型二次特征生成器
   - 动态型二次特征生成器
5. 再写动态聚合器
   - 先支持 `连续同向段聚合`
6. 再改因果模板匹配逻辑
7. 最后在任务/方法侧补目标绑定结构
8. 选一个小场景跑通 `生成 -> 匹配 -> 回写` 闭环


## 17. 一句话收口

本计划的核心不是“给二次特征多加几个名字”，而是把它补成一层真正独立的语义层：

`特征类` 管原始值和比较核，`动态类` 管状态变化和层级路径，`二次特征类` 管存在/动态二阶语义，`因果类` 管条件与结果，`任务/方法类` 管目标和动作。

只有把这条链补完整，二次特征才会成为可以被 `生成 / 检索 / 匹配 / 推理 / 调度 / 回写` 的正式节点，而不是附带标签。
