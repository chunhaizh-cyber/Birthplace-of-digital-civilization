# 2026-04-12 全特征改值本能函数化总计划

## 0. 计划目标

本计划的总目标只有一条：

`实现所有特征的值的改变，最终都经由正式本能函数发生。`

这里的“经由正式本能函数发生”，口径固定为：

1. 每一种允许被改变的特征类型，都必须有明确的正式改值入口
2. 正式改值入口必须能映射到已注册的 `本能方法ID`
3. 非本能函数体之外，不再允许直接散写“特征当前值”
4. 所有改值都必须可追溯到：
   - `谁改了`
   - `为什么改`
   - `改前是什么`
   - `改后是什么`
   - `是否产生了状态差分 / 动态 / 因果`

一句话收口：

`以后不是“哪里方便就在哪里写值”，而是“所有值变化都要走正式动作资产”。`


## 1. 本计划回答什么

本计划只回答 5 件事：

1. 哪些特征已经部分本能函数化
2. 哪些特征仍主要靠模块内 helper 或直写
3. 应该先补哪一层地基
4. 应该按什么阶段推进到“全特征全覆盖”
5. 什么状态才能算真正完成

本计划不直接替代以下专项计划，而是把它们收编为总计划的子计划：

- `任务管理特征宿主纠偏`
- `任务管理六原语正式升格`
- `任务管理特征改值能力审计`

当前对子计划的状态判断固定为：

- `任务管理特征宿主纠偏`：已完成，后续只作为前置约束引用
- `任务管理六原语正式升格`：进行中，当前主任务是 `P5 收口与验收`
- `任务管理特征改值能力审计`：已完成，可直接作为覆盖账本输入
- `抽象状态与抽象动态接入任务管理主链`：路线有效，但需先按最新规范修订后再执行


## 2. 当前事实判断

按当前代码与文档，系统已经形成三个不同成熟度层级：

### 2.1 已基本完成本能函数化的层

主要是：

- `安全值 / 服务值`
- `时序正向步长 / 时序反向步长 / 服务时序衰减步长`
- `待机状态`
- `风险安全`
- `当前主需求 / 当前主任务 / 当前主方法`

这部分已经具备较明确的：

- 正式本能 ID
- 元信息
- 实际运行入口
- 稳定改值逻辑

### 2.2 已进入正式动作化收口阶段的层

主要是：

- `任务管理_*` 治理核心特征
- `任务目标状态 / 任务结果状态`
- `待学习方法数量 / 尝试学习状态`

这部分的问题不是“不会改值”，而是：

- 宿主纠偏虽已完成，但仍需继续清理旧读取口径、旧摘要和旧 helper 影子
- 六原语主写链已基本完成正式动作化，剩余重点转为 `P5` 的读取口径一致性与旧表达收口
- 缺口评估、动作资产、实际调用证据三层已基本拆开，但覆盖账本与禁旧路径审计尚未完成

### 2.3 仍偏占位或仅有通用承载能力的层

主要是：

- `情绪ID / 情绪强度`
- `外显情绪ID / 外显情绪强度`
- `外设可用性`
- `物理安全` 的部分来源口径

这部分当前多半只有：

- 类型定义
- 初始化默认值
- 通用写接口可承载

但尚未形成稳定的正式本能函数资产。


## 3. 总体策略

要实现“所有特征的值改变都用本能函数”，不能直接从每个特征各补一个函数开始。

正确顺序必须是：

1. 先补公共动作地基
2. 再补各特征组的正式改值入口
3. 再收口所有旧 helper 和直写后门
4. 最后做覆盖率验收

总体策略固定为：

`先地基，后分域；先正式入口，后禁旧路径；先主闭环，后侧翼。`


## 4. 统一架构约束

后续所有改造统一遵守以下 8 条。

### 4.1 一切改值都要有正式归口

每个“可变特征类型”都必须能回答：

- 对应哪个 `本能方法ID`
- 由哪个模块实现
- 允许谁调用
- 改值的主观察特征是什么

### 4.2 禁止在业务流程里直接散写特征当前值

允许直接调用世界树写入接口的地方，只能有两类：

1. 正式本能函数实现体内部
2. 受控的最小原语执行壳内部

除此之外，业务模块不得再直接把值写进特征当前值。

### 4.3 本能函数必须区分“原语动作”和“派生规则”

统一口径：

- `原语动作` 负责真正做变更
- `派生规则` 负责围绕变更产出状态、动态、因果、摘要

不得再把：

- `状态差分取证`
- `实例因果派生`
- `轻量因果链组织`

继续混成“最小原语”。

### 4.4 本能函数必须可被查询、注册、验证

每个正式改值入口至少要具备：

- `本能方法ID`
- 元信息
- 方法首节点或等价正式资产
- 能被缺口评估或覆盖率审计识别

### 4.5 宿主语义必须正确

本能函数化不等于全部写到 `自我存在`。

必须坚持：

- `自我特征写自我存在`
- `宿主任务治理特征写宿主任务侧宿主`
- `任务双面状态写任务虚拟存在`
- `方法资产特征写方法虚拟存在`
- `主体 / 分身 / 方法位桥接特征` 落到其语义宿主或过渡承载

### 4.6 每次改值都要保留最小证据

正式改值入口至少要能支持以下最小证据面：

- 动作开始
- 动作结束
- 主观察特征前值
- 主观察特征后值
- 是否发生变化

有变化时再进一步生成：

- 状态差分
- 二次特征方向
- 实例因果

### 4.7 能力判断必须拆三层

后续任何“是否具备某改值能力”的判断，都要拆成：

1. `底层工程能力具备`
2. `正式动作资产具备`
3. `本轮执行证据具备`

### 4.8 覆盖率必须能落成账本

最终必须存在一张正式覆盖表，至少能列清：

- `特征类型`
- `当前写法`
- `目标本能函数`
- `目标宿主`
- `是否已完成本能函数化`
- `是否已禁止旧路径`

## 7. 各阶段优先级

整体优先级固定为：

1. `P0 总账本`
2. `P1 六原语与固定派生规则`
3. `P2 双根与主链锚点`
4. `P3 任务双面与治理核心`
5. `P4 桥接与挂点`
6. `P5 学习回流`
7. `P6 侧翼支路`
8. `P7 禁旧路径与最终验收`

这样排序的原因是：

- 主闭环最先受益
- 公共动作层先稳定，后面改造成本最低
- 侧翼特征不阻塞“把任务推到完成”的主能力

## 9. 风险与控制

### 9.1 最大风险

最大的风险不是“少几个本能函数”，而是：

`一边新增正式动作，一边继续默许旧 helper 和直写路径扩散。`

如果不控制这个风险，最后会形成：

- 新旧两套入口长期并存
- 缺口评估失真
- 宿主语义继续混层
- 审计账本很快失效

### 9.2 控制办法

统一采取以下控制：

1. 每补一类正式动作，就同步标记对应旧入口为过渡
2. 新增代码禁止再默认直写特征当前值
3. 每个阶段结束前补一次覆盖账本更新
4. 编译通过不算完成，必须附带“入口是否收口”的审计结论


## 10. 最终验收标准

1. 全量特征清单已有正式覆盖账本
2. 每个可变特征都能映射到明确正式本能函数
3. 六原语和固定派生规则已经正式资产化
4. 各特征组的主要改值路径已切到正式动作入口

6. 业务模块不再直接散写世界树特征当前值
7. 缺口评估能区分底层能力、正式资产、执行证据
8. 工程经过编译与主链回归验证，且覆盖账本与实际代码一致
