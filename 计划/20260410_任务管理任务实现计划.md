# 2026-04-10 任务管理任务实现计划

## 1. 计划目标

本计划用于把当前工程中的 `任务管理任务模块` 从“最小一步治理骨架”继续推进到“可持续承接任务治理、筹办、执行、回流”的稳定模块。

本计划只回答三件事：

1. 当前 `任务管理任务` 已经做到哪里
2. 后续应该按什么顺序继续补
3. 每一阶段怎样算真正落地


## 2. 当前实现结论

按当前代码与运行结果，`任务管理任务模块` 已经具备以下最小闭环：

1. 能读取最小治理上下文：
   - `当前主需求`
   - `当前管理任务`
   - `当前宿主任务`
   - `当前方法`
   - `当前步骤`
   - `最近结果`

2. 能做一步最小治理：
   - 补齐治理任务
   - 选择宿主任务
   - 绑定宿主任务方法挂点
   - 推进宿主任务状态
   - 生成治理步骤节点
   - 生成治理结果节点

3. 能把治理中间态写回自我状态/动态/因果链：
   - `当前宿主任务指针`
   - `当前方法指针`
   - `宿主任务状态`
   - `当前功能域`
   - `当前缺口类型`
   - `当前下一步去向`
   - `当前步骤指针`
   - `最近结果指针`

4. 已接入两条运行链：
   - 初始化阶段做一次治理基线
   - 自我线程每轮做一步治理

5. 当前任务树已经能落成最小治理子链：
   - `宿主任务`
   - `治理步骤`
   - `治理结果`


## 3. 当前主要缺口

虽然最小骨架已跑通，但当前仍明显停留在“治理记账层”，还不是完整任务管理系统。当前缺口主要有：

1. `宿主任务选择` 还很粗
   - 目前主要在 `安全根任务 / 服务根任务 / 任务管理任务` 之间做简单回退
   - 没有真正的多任务竞争、排序和仲裁

2. `方法挂点` 还很粗
   - 目前主要绑定默认本能方法
   - 还没有按条件节点、结果节点、来源证据来选方法

3. `上下文与结果面` 还不够稳定
   - 目前偏“本轮做了什么”
   - 距离 `1214 / 1213` 所要求的稳定中间态与返回面还有差距

4. `步骤/结果子链` 还偏治理痕迹
   - 当前已经补入 `执行位 / 等待位 / 回接位 / 补条件位` 的最小位语义
   - 但真实业务步骤和子任务分解还不够细

5. `等待 / 无缺口 / 保持等待` 等分支还没真正进主流程
   - 当前已进入治理判定主线
   - 但还缺专项覆盖验收与更长时程验证

6. `目标状态` 还没真正进入任务治理主线
   - 当前已补入宿主任务、步骤节点、结果节点的 `目标状态` 写回
   - 但还没完全形成稳定的“目标 - 结果差额驱动”主线

7. `任务治理` 还没和更上层裁决完整闭合
   - 当前还没有把根层重判、执行前门控、反馈回流正式合入治理主线


## 4. 实现红线

后续推进必须一直遵守以下红线：

1. 不改现有 `需求 / 任务 / 方法 / 存在 / 自我` 的领域数据结构语义边界。
2. 可以新增模块、适配层、结果结构、日志与检查结构，但不得另起并行真值层。
3. `任务管理任务模块` 只能做承接层治理，不得越权替代：
   - 根层重判
   - 执行前门控
   - 最终允许边界
4. `能治理` 不等于 `能代决`。
5. 一切最终 `继续 / 收束 / 待机 / 停止 / 转学习` 结论，仍必须回到主循环外层裁决。
