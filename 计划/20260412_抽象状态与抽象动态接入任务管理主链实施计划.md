# 2026-04-12 抽象状态与抽象动态接入任务管理主链实施计划

## 1. 目标

本文用于把 `抽象状态`、`抽象动态` 与 `任务管理主链` 的实现顺序、模块拆分和接入方式一次性固定下来。

本文重点回答：

- 实现顺序为什么必须是 `抽象状态 -> 抽象动态 -> 主链接回`
- 哪些能力应走热路径，哪些能力应走冷路径
- 应新增哪些模块，以及哪些运行期信息优先复用现有承载
- 应在哪些主链位置插入抽象命中与摘要写回
- 第一刀最小切口应该先做什么

当前计划状态固定为：

- 总路线未过期
- 阶段顺序未过期
- 字段命名、生成字段和结构体准入口径，必须先按 `0561 / 0800 / 1245` 修订后才能正式开工


## 2. 总判断

总路线必须固定为：

```text
实例状态 / 实例动态
-> 二次特征
-> 抽象状态命中
-> 抽象动态命中
-> 概念统计更新
-> 命中摘要写回
-> 下一轮任务管理判域 / 方法需求位 / 长期策略消费
```

同时必须固定：

1. `抽象状态先于抽象动态`
- `0606` 已明确要求抽象动态建立在抽象状态之上
- 因此不得先做抽象动态概念树，再反推抽象状态

2. `热路径只做命中与摘要`
- 当前治理轮内部只负责：
  - 命中候选
  - 更新计数
  - 写回摘要
  - 记录最近命中时间
- 升格、衰减、淘汰留到冷路径

3. `抽象动态必须支持首尾同态`
- `初始状态 = 结果状态` 不等于 `无动态`
- 只要中间存在可识别的相位、转折、回返、节奏或局部片段，就仍是合法动态
- 这部分由 `过程路径二次特征` 承接


## 3. 实现边界

当前这条路线不应重造地基，而应复用已有底盘：

- `状态类`
  - 已能创建状态、写入主体/特征/值、记录变化原因
- `动态类`
  - 已能以初始状态/结果状态创建动态
  - 已能绑定业务语义、来源方法和来源场景
- `二次特征类`
  - 已能按度量创建二次特征、绑定场景、设置主体客体/左右对象、写入标量值
- `因果类`
  - 已能消费实例状态、生成抽象条件/结果短语
- `任务管理任务模块`
  - 已有单步治理、结果壳、主体虚拟存在和治理快照主链

因此当前最缺的不是基础设施，而是：

1. `抽象命中`
2. `概念统计`
3. `升格/衰减/淘汰`
4. `主链接入`


## 4. 阶段总表

推荐阶段顺序固定为：

1. `P1 单特征抽象状态最小闭环`
2. `P2 复合抽象状态`
3. `P3 简单动态片段与复杂动态聚合`
4. `P4 抽象动态候选命中`
5. `P5 抽象状态/抽象动态接回任务管理主链`
6. `P6 冷路径升格、衰减与淘汰`

工程上必须固定：

- 不直接从 `P3/P4` 开始
- 不先做完整概念树维护器
- 不先让抽象层直接改最终去向


## 5. P1：单特征抽象状态最小闭环

### 5.1 目标

先做 `单特征抽象状态` 的热路径闭环，不碰复合状态，不碰抽象动态。

### 5.2 新增模块

建议新增：

- `抽象状态生成模块.h`
- `抽象状态生成模块.cpp`

尽量不要继续把新逻辑塞进 `任务管理任务模块.cpp`。

### 5.3 少量改动文件

- `世界树类.h`
- `世界树类.cpp`
- `状态类.h`
- `状态类.cpp`
- `任务管理任务模块.cpp`
- `任务管理任务模块_治理快照.cpp`

### 5.4 优先复用的运行期承载与局部薄结构

第一刀优先只允许出现少量模块内局部薄结构：

- `结构_实例状态样本`
- `结构_单特征抽象状态候选身份`
- `结构_抽象状态命中结果`
- `结构_抽象概念统计更新`

必须固定：

- 这些结构优先视为模块内局部薄结构，不默认升格成新的持久业务结构体
- 若现有 `二次特征 / 治理快照 / 宿主任务虚拟存在` 已能承载对应信息，则优先复用，不额外新增平行持久结构

其中 `结构_单特征抽象状态候选身份` 至少应包含：

- `概念域 = 状态`
- `来源特征类型`
- `值域 / 区间 / 类别`
- `适用条件主键`
- `父状态概念主键`
- `主母式`
- `粒度`
- `基准类型`
- `时间归一方式`

### 5.5 最小函数组

第一刀先固定为这 5 个函数：

```cpp
结构_实例状态样本 私有_抽取实例状态样本(...);
结构_单特征抽象状态候选身份 私有_生成单特征抽象状态候选身份(...);
结构_抽象状态命中结果 私有_命中或创建单特征抽象状态候选(...);
void 私有_更新抽象状态统计(...);
void 私有_写回抽象状态命中摘要(...);
```

### 5.6 主链接入点

这一阶段的抽象状态不得插到判域前。

最稳的接法固定为：

```text
步骤/结果写回
-> 宿主目标状态/结果状态写回
-> 抽取实例状态样本
-> 命中或更新抽象状态候选
-> 写回抽象状态命中摘要
-> 继续后续治理收口
```

摘要至少应穿过：

- `宿主任务虚拟存在`
- `一步治理结果`
- `治理快照`

### 5.7 验收场景

建议先补：

- `场景26`：重复命中同类实例状态，生成单特征抽象状态候选
- `场景27`：候选命中次数、稳定度、活跃度持续增长
- `场景28`：抽象状态命中摘要能穿过一步结果、宿主虚拟存在和治理快照


## 6. P2：复合抽象状态

### 6.1 目标

复合抽象状态只能建立在 `单特征抽象状态` 已稳定命中的前提上。

### 6.2 新增结构

- `结构_复合抽象状态候选身份`
- `结构_复合抽象状态共现窗口`
- `结构_复合抽象状态命中结果`

### 6.3 实现规则

复合状态只做 `稳定共同命中`，不做自由组合爆炸。

推荐主链固定为：

```text
多个单特征抽象状态在同一证据窗口内稳定共同命中
-> 形成复合状态候选身份
-> 更新共现统计
-> 达阈值再升格
```

### 6.4 接入点

仍然放在“本轮状态写回之后”：

```text
单特征抽象状态命中
-> 检查本轮稳定共现
-> 若形成则更新复合状态候选
```

### 6.5 本阶段边界

本阶段不要做：

- 复合状态直接驱动方法需求位
- 复合状态直接改最终去向

它优先只作为：

- 摘要
- 统计
- 长期策略候选输入


## 7. P3：简单动态片段与复杂动态聚合

### 7.1 目标

先把动态从“首尾差值模型”拆成“过程模型”，再谈抽象动态。

### 7.2 为什么不能直接上抽象动态

当前 `动态类` 更像实例动态节点工厂，已具备：

- 初始状态 / 结果状态
- 业务语义类型
- 来源方法
- 来源场景

但还缺显式的：

- 中间相位序列
- 过程路径签名
- 转折与回返结构

因此若直接从当前动态节点硬上抽象动态，会自然退化成“首尾差值模型”。

### 7.3 新增结构

建议新增：

- `结构_简单动态片段`
- `结构_过程路径二次特征`
- `结构_复杂动态聚合`

其中 `结构_过程路径二次特征` 至少应包含：

- `中间抽象状态序列签名`
- `相位序列签名`
- `转折点数量`
- `峰值/谷值偏移`
- `回返类型`
- `闭环类型`
- `节奏/频率/时长签名`
- `主母式`
- `粒度`
- `基准类型`
- `时间归一方式`

并且必须固定：

- `过程路径签名 / 节奏签名` 只作为索引外壳，不替代显式生成字段
- 若现有 `动态类 / 二次特征类` 已能承载其中部分信息，则优先补字段，不先新增持久平行结构

### 7.4 最小函数组

```cpp
结构_简单动态片段 私有_记录简单动态片段(...);
结构_过程路径二次特征 私有_生成过程路径二次特征(...);
结构_复杂动态聚合 私有_聚合复杂动态(...);
```

### 7.5 片段来源

简单动态片段优先来自：

- `步骤位切换`
- `步骤语义切换`
- `结果节点落点`
- `业务子任务回接`
- `学习承接开始/结束`
- `动作开始/动作结束边界`


## 8. P4：抽象动态候选命中

### 8.1 新增模块

- `抽象动态生成模块.h`
- `抽象动态生成模块.cpp`

### 8.2 动态候选身份字段组

抽象动态身份至少由以下字段共同决定：

- `概念域 = 动态`
- `来源业务语义类型`
- `动态核心类型`
- `初始抽象状态主键`
- `结果抽象状态主键`
- `过程路径签名`
- `时间因素签名`
- `适用条件主键`
- `主母式`
- `粒度`
- `基准类型`
- `时间归一方式`

同时必须固定：

当 `初始抽象状态主键 = 结果抽象状态主键` 时，
`过程路径签名` 变成必要身份字段。

### 8.3 最小生成条件

抽象动态候选分两路：

#### A 路：首尾迁移型

- 存在 `初始抽象状态`
- 存在 `结果抽象状态`
- 首尾可形成 `状态迁移二次特征`
- 可形成 `时间因素二次特征`

#### B 路：过程闭环型

- 存在 `初始抽象状态`
- 存在 `结果抽象状态`
- 初始与结果允许相同
- 存在至少一个 `中间状态或中间相位`
- 可形成 `过程路径二次特征`
- 可形成 `时间因素二次特征`

### 8.4 动态双车道

本阶段必须明确两路输出：

1. `业务语义分类结果`
2. `抽象动态命中结果`

前者给：

- 控制面板
- 治理摘要
- 桥接快照

后者给：

- 概念域
- 长期策略
- 后续方法需求位消费

### 8.5 验收场景

建议补：

- `场景29`：简单动态片段可命中抽象动态候选
- `场景30`：多个简单动态可聚合成复杂动态
- `场景31`：`S0 -> S1 -> S0` 的首尾同态复杂动态仍被识别
- `场景32`：业务语义分类与抽象动态命中可同时存在


## 9. P5：接回任务管理主链

### 9.1 总原则

这一步不是重写主链，而是在现有单步治理里插入 3 个小阶段。

### 9.2 建议接入点

固定插到：

```text
步骤/结果写回
-> 宿主目标状态/结果状态写回
-> 生成状态二次特征与过程路径二次特征
-> 命中抽象状态候选与抽象动态候选
-> 更新抽象概念统计并写回命中摘要
-> 实例因果补强 / 方法需求位 / 长期策略回填
```

也就是：

- 晚于状态镜像稳定
- 早于方法需求位消费

### 9.3 为什么放这里

因为当前治理链已经固定：

- 一轮只推进一步
- 动态/因果必须晚于状态镜像
- 最终去向仍由主循环统一裁决

把抽象命中放在这里：

- 不会污染前置门控
- 不会沦为尾部调试摘要
- 又能成为下一轮上下文输入

### 9.4 首批消费对象

抽象层最先影响的不是最终去向，而是：

- `方法需求位命中`
- `本能能力缺口判定的上下文证据`
- `长期策略回填`
- `业务子任务派生/复用的复用键`


## 10. P6：冷热分离，做升格、衰减、淘汰

### 10.1 热路径只做

- 命中候选
- 更新计数
- 写回摘要
- 写回最近命中时间
- 写回最近路径签名

### 10.2 冷路径再做

- `候选 -> 正式概念` 升格
- 长期未命中后的衰减
- 降级为冷概念
- 最终清理

### 10.3 调度方式

第一阶段不额外引入新线程。

优先复用：

- `普通推进心跳`
- 维护轮

来承接概念维护冷路径。


## 11. 建议的文件改动顺序

最顺的落刀顺序建议固定为：

1. `世界树类.h/.cpp`
- 补概念域根、状态概念根、动态概念根的取或创建接口

2. `抽象状态生成模块.h/.cpp`

3. `任务管理任务模块.cpp`

4. `任务管理任务模块_治理快照.cpp`
- 把抽象状态命中摘要带进治理快照

5. `抽象动态生成模块.h/.cpp`
- 先做简单动态片段、过程路径二次特征、复杂动态聚合
- 再做抽象动态命中

6. `动态类.h/.cpp`
- 补最小扩展字段或辅助接口，让动态片段和聚合结果有落点

7. `鱼巢.cpp`
- 把场景 `26 ~ 32` 接进 `--acceptance-report`

8. `README.md`
- 补抽象状态/抽象动态实施与验收入口说明


## 12. 第一刀切口

第一刀必须固定为：

`单特征抽象状态最小闭环 + 命中摘要写回`

原因是：

1. 规范上，抽象动态依赖抽象状态
2. 代码上，当前最成熟可复用的基础设施是 `状态类 + 二次特征类 + 任务管理写回链`
3. 工程上，先让 `抽象状态命中摘要` 穿过一步治理、宿主虚拟存在和治理快照，下一刀做动态时就有稳定首尾与中间锚点

因此第一刀明确不做：

- 抽象动态候选
- 复杂动态聚合
- 升格/淘汰冷路径
- 直接影响最终去向


## 13. 一句话收口

最合理的实现路线不是直接上“抽象动态概念树”，而是：

```text
先用现有状态类、动态类、二次特征类和任务管理写回链，
做出抽象状态的热路径命中；
再把动态拆成简单片段和复杂聚合，补上过程路径二次特征；
最后把抽象状态/抽象动态命中摘要接回任务管理的一步治理、方法需求位和长期策略。
```

这条路线最稳，也最不返工。
